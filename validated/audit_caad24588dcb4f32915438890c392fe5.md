### Title
Linked Signer Can Overwrite Its Own Authorization via `LinkSigner` Sequencer Transaction — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In the Nado sequencer path, `LinkSigner` transactions are validated with `allowLinkedSigner = true`, meaning the currently-set linked signer can sign a `LinkSigner` transaction to replace itself with any arbitrary address. This is the direct analog to the SettV3 bug: a delegated actor (linked signer ≈ `transferFrom` caller) can act on behalf of the subaccount owner and mutate a state that should only be mutable by the owner, because the authorization check validates the wrong principal.

---

### Finding Description

In `EndpointTx.sol`, the sequencer path for `LinkSigner` calls `validateSignedTx` with `allowLinkedSigner = true`:

```solidity
// EndpointTx.sol ~577-590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` calls `validateSignature`, which calls `verifier.validateSignature` with the linked signer as an accepted signer:

```solidity
// EndpointTx.sol ~172-184
function validateSignature(
    bytes32 sender,
    bytes32 digest,
    bytes memory signature,
    bool allowLinkedSigner
) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

`verifier.validateSignature` accepts a signature from either the subaccount owner address or the linked signer:

```solidity
// Verifier.sol ~291-304
function validateSignature(...) public pure {
    address recovered = ECDSA.recover(digest, signature);
    require(
        (recovered != address(0)) &&
            ((recovered == address(uint160(bytes20(sender)))) ||
                (recovered == linkedSigner)),
        ERR_INVALID_SIGNATURE
    );
}
``` [3](#0-2) 

This means the currently-set linked signer can sign a `LinkSigner` transaction to overwrite `linkedSigners[subaccount]` with any address it chooses.

Contrast this with the **slow-mode path** for `LinkSigner`, which correctly enforces that only the subaccount owner can change the linked signer:

```solidity
// EndpointTx.sol ~232-239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);   // ← msg.sender must be subaccount owner
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

The slow-mode path uses `validateSender`, which checks that the Ethereum `msg.sender` is the address embedded in the subaccount bytes. The sequencer path skips this check and instead accepts the linked signer's signature, creating an inconsistency that is the root cause of the vulnerability.

---

### Impact Explanation

A linked signer (e.g., a trading bot key) can silently replace itself with an attacker-controlled address. Once replaced, the attacker's address becomes the new linked signer and can sign any sequencer transaction on behalf of the subaccount — including `WithdrawCollateral`, `BurnNlp`, `TransferQuote`, and further `LinkSigner` mutations — with full signing authority.

The subaccount owner's only recourse is a slow-mode `LinkSigner` transaction, which has a hardcoded 3-day delay (`SLOW_MODE_TX_DELAY = 3 * 24 * 60 * 60`): [5](#0-4) 

During those 3 days, the attacker retains full signing authority and can drain the subaccount. The attacker can also chain further `LinkSigner` mutations to maintain persistent access even after the owner submits a revocation.

**Corrupted state**: `linkedSigners[subaccount]` is set to an attacker-controlled address without the subaccount owner's consent.

---

### Likelihood Explanation

The attack requires only that the linked signer key be compromised or malicious. Linked signers are commonly API keys or hot-wallet keys used by trading bots, which are a realistic target for compromise. No admin access, governance capture, or sequencer compromise is needed. The attacker only needs to submit a single signed `LinkSigner` transaction through the normal sequencer path.

---

### Recommendation

For `LinkSigner` transactions in the sequencer path, set `allowLinkedSigner = false`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This aligns the sequencer path with the slow-mode path, which already correctly enforces owner-only authorization for `LinkSigner` via `validateSender`.

---

### Proof of Concept

1. User A owns subaccount `S = bytes32(abi.encodePacked(A, subaccountName))`.
2. A sets linked signer `L` (a trading bot key) via a slow-mode `LinkSigner` transaction. `linkedSigners[S] = L`.
3. Attacker obtains `L`'s private key (e.g., via API key leak).
4. Attacker uses `L` to sign a `LinkSigner` transaction: `{ sender: S, signer: bytes32(M), nonce: N }` where `M` is the attacker's address.
5. Attacker submits this transaction to the sequencer. The sequencer path validates the signature against `L` (the current linked signer), which passes because `allowLinkedSigner = true`.
6. `linkedSigners[S]` is now set to `M`. [6](#0-5) 
7. `M` signs a `WithdrawCollateral` transaction for `S`, draining all collateral.
8. A submits a slow-mode `LinkSigner` to revoke `M`, but the 3-day delay means the funds are already gone. [5](#0-4)
