### Title
`DelegateAction` (NEP-366/NEP-611) signing domain omits chain identity, enabling cross-network signature replay - (File: `core/primitives/src/action/delegate.rs`, `core/primitives/src/signable_message.rs`)

### Summary

The NEP-461 signing scheme used for meta-transaction `DelegateAction` and `DelegateActionV2` payloads computes a hash over a fixed `MessageDiscriminant` constant plus the Borsh-serialized action body. Neither the discriminant nor the body includes the NEAR chain ID (`"mainnet"`, `"testnet"`, etc.) or the genesis hash. A `SignedDelegateAction` produced on one NEAR network is cryptographically valid on any other NEAR network where the same account, key, and nonce state exist, allowing an attacker to replay a user's off-chain-signed meta-transaction on a different network.

### Finding Description

`DelegateAction.get_nep461_hash()` constructs the signed payload as:

```
SHA-256( borsh(MessageDiscriminant) || borsh(DelegateAction) )
``` [1](#0-0) 

The `MessageDiscriminant` is a compile-time constant derived solely from the NEP number:

```
discriminant = (1 << 30) + 366   // DelegateAction
discriminant = (1 << 30) + 611   // DelegateActionV2
``` [2](#0-1) [3](#0-2) 

The `DelegateAction` struct itself contains `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key` — no chain identifier of any kind: [4](#0-3) 

The same is true for `DelegateActionV2` / `VersionedDelegateActionPayload`: [5](#0-4) 

On-chain verification in `apply_delegate_action` checks only: signature validity against this chain-agnostic hash, nonce freshness relative to the sender's on-chain access key, `max_block_height` relative to the current block, and access key permissions. None of these checks bind the signature to a specific network. [6](#0-5) 

NEAR's `GenesisId` carries both `chain_id` and genesis block hash, but it is never consulted during delegate action signature verification: [7](#0-6) 

### Impact Explanation

A user who signs a `DelegateAction` on testnet (or any NEAR fork/mirror) produces a signature that is byte-for-byte valid on mainnet, provided:

1. The same account ID exists on mainnet with the same signing key.
2. The access key nonce on mainnet has not yet reached the value embedded in the `DelegateAction`.
3. The current mainnet block height is below `max_block_height`.

All three conditions are routinely satisfied: many users hold the same implicit or named account on both mainnet and testnet with identical key material; nonces on the two networks diverge independently; and `max_block_height` is a raw block number shared across networks (both are in the tens-of-millions range).

An attacker who intercepts or observes a `SignedDelegateAction` intended for testnet can submit it to a mainnet relayer. The mainnet runtime will accept it, executing the inner actions (token transfers, key additions/deletions, contract calls) as if authorized by the user, with the relayer paying gas. The user's mainnet assets are at risk without any action on their part.

### Likelihood Explanation

The attack requires no privileged access. Any party who receives a `SignedDelegateAction` off-chain (e.g., a relayer, an observer of a public relayer API, or anyone who intercepts the payload in transit) can attempt the replay. The preconditions (same account on both networks, nonce not consumed) are common for NEAR users who operate on both mainnet and testnet. The `max_block_height` window is typically set to hundreds of blocks (~minutes), giving an attacker a practical submission window.

### Recommendation

Include the chain identity in the signed payload. The canonical approach is to add the `GenesisId` (or at minimum the `chain_id` string) to the `DelegateAction` struct or to the `SignableMessage` wrapper before hashing. Alternatively, the `MessageDiscriminant` could be extended to encode a chain-specific salt derived from the genesis hash, mirroring how EIP-712 domain separators bind signatures to a specific `chainId` and `verifyingContract`.

Short term: add `chain_id: String` (or `genesis_id: GenesisId`) to `DelegateAction` and `DelegateActionV2`, include it in the Borsh-serialized payload, and validate it against the executing chain's genesis ID inside `apply_delegate_action`.

Long term: standardize the chain-binding requirement in NEP-461 so all future signable message types inherit it automatically via the `SignableMessage` wrapper.

### Proof of Concept

1. Alice holds account `alice.near` on both mainnet and testnet with the same ED25519 key. Her access key nonce on testnet is 42; on mainnet it is also 42 (or any value below 43).

2. Alice signs a testnet `DelegateAction` to call `ft_transfer` on a testnet token contract, with `nonce: 43` and `max_block_height: <current_testnet_height + 200>`.

3. The signed payload hash is:
   ```
   SHA-256( LE32(1073742190) || borsh(DelegateAction{sender="alice.near", receiver="ft.testnet", nonce=43, ...}) )
   ```
   This hash is identical on mainnet because the discriminant is a fixed constant and the struct contains no chain identifier.

4. An attacker submits the same `SignedDelegateAction` to a mainnet relayer, wrapped in a mainnet transaction. The mainnet runtime calls `SignedDelegateAction::verify()`: [8](#0-7) 

   Verification passes. Nonce 43 > 42 on mainnet — valid. Block height check passes. The `ft_transfer` executes on mainnet's `ft.near` (or whichever contract shares the receiver account ID), draining Alice's mainnet fungible token balance without her consent.

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }
