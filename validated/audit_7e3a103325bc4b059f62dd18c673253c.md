### Title
Zero `confirmations` Bypasses SPV Proof Finality Check — (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` (and `verify_transaction_inclusion_v2` which delegates to it) accepts a caller-supplied `confirmations: u64` with no minimum enforcement. Passing `confirmations = 0` makes the confirmation-depth check trivially true for every block in the mainchain, allowing any unprivileged NEAR caller to obtain a `true` SPV proof result for a transaction that has zero on-chain confirmations and could still be reorganized away.

### Finding Description

The confirmation-depth guard in `verify_transaction_inclusion` is:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [1](#0-0) 

Because `args.confirmations` is a `u64` and the left-hand side is also `u64`, when `args.confirmations == 0` the condition reduces to `(anything) >= 0`, which is always `true`. There is no `require!(args.confirmations > 0, ...)` guard anywhere in the function or in `verify_transaction_inclusion_v2`. [2](#0-1) 

The only other guard that touches `confirmations` is:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [3](#0-2) 

`gc_threshold` is a `u64` initialized from `InitArgs` with **no non-zero validation** in `init()`:

```rust
gc_threshold: args.gc_threshold,
``` [4](#0-3) 

`0 <= gc_threshold` is always satisfied, so this guard never blocks a zero `confirmations` value regardless of how `gc_threshold` was set.

`verify_transaction_inclusion_v2` inherits the same flaw because it converts its args and calls the deprecated function directly:

```rust
self.verify_transaction_inclusion(args.into())
``` [5](#0-4) 

### Impact Explanation

`verify_transaction_inclusion` is the primary trust anchor for downstream NEAR contracts that gate financial operations on BTC transaction finality (e.g., bridge unlock, cross-chain settlement). When called with `confirmations = 0`, the function returns `true` for a transaction whose containing block is the current chain tip — a block that carries no subsequent PoW and can be displaced by a natural reorg or a targeted 51% attack. A downstream contract that relies on this result would release funds for a transaction that may never achieve finality, enabling a double-spend.

### Likelihood Explanation

The entry path is fully unprivileged: any NEAR account can call `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` with `confirmations = 0`. No staking, role, or special permission is required. The `ProofArgs` struct is caller-supplied borsh data with no server-side minimum enforcement. [6](#0-5) 

### Recommendation

1. Add a non-zero guard at the top of `verify_transaction_inclusion`:

```rust
require!(args.confirmations > 0, "confirmations must be greater than zero");
```

2. Add a non-zero guard in `init()` for `gc_threshold`:

```rust
require!(args.gc_threshold > 0, "gc_threshold must be greater than zero");
``` [7](#0-6) 

### Proof of Concept

1. A relayer submits a single BTC block header to the contract via `submit_blocks`. The block is now the mainchain tip at height `H`.
2. An attacker (any NEAR account) calls `verify_transaction_inclusion` with:
   - `tx_block_blockhash` = hash of the block at height `H`
   - `confirmations = 0`
3. The guard `(H - H) + 1 >= 0` → `1 >= 0` → `true`. The function returns `true`.
4. A downstream bridge contract, trusting this result, releases locked assets.
5. The BTC block at height `H` is reorganized away (naturally or via a targeted attack). The original BTC transaction is double-spent.
6. The downstream contract has already released funds; the attacker profits. [8](#0-7)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
