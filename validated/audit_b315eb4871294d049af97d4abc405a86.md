### Title
Missing Early Return in `validate_delegate_action_key` Silently Advances Nonce on Failed `DepositWithFunctionCall` Delegate Action — (`runtime/runtime/src/actions.rs`)

---

### Summary

In `validate_delegate_action_key`, when a delegate action uses a function-call access key and the inner `FunctionCall` carries `deposit > 0` (which is forbidden), the code writes `result.result = Err(DepositWithFunctionCall)` but does **not** return early (pre-`FixDelegateActionDepositWithFunctionCallError`). If the subsequent `receiver_id` and `method_name` checks both pass, execution falls through to the unconditional nonce-update block, advancing the access-key nonce in trie state even though the action will ultimately fail. The caller (`apply_delegate_action`) then detects the error and suppresses receipt generation — but the nonce has already been consumed.

---

### Finding Description

`validate_delegate_action_key` in `runtime/runtime/src/actions.rs` performs three sequential permission checks for function-call access keys:

1. `deposit > 0` → set `DepositWithFunctionCall` error (no early return pre-fix)
2. `receiver_id` mismatch → set `ReceiverMismatch` error, **return**
3. `method_name` mismatch → set `MethodNameMismatch` error, **return** [1](#0-0) 

Because check (1) does not return, if the deposit is non-zero but the receiver and method name are both correct, execution falls through to the unconditional nonce-update block: [2](#0-1) 

The nonce is written to trie state via `set_access_key` / `set_gas_key_nonce`. The caller then inspects `result.result`, finds it is `Err`, and returns without emitting a receipt: [3](#0-2) 

The net effect: the action fails (no receipt, no token transfer), but the access-key nonce is permanently advanced — a state mutation that should not occur on a failed validation.

The fix, gated behind `ProtocolFeature::FixDelegateActionDepositWithFunctionCallError` (protocol version 85), adds the missing `return Ok(())` immediately after setting the error: [4](#0-3) 

The pre-fix code path is still present in the binary for backward-compatible replay of pre-85 blocks.

---

### Impact Explanation

**Protocol invariant violated**: a failed delegate-action validation must not mutate trie state. Pre-fix, a delegate action that carries `deposit > 0` with a function-call key whose `receiver_id` and `method_name` match the permission silently advances the nonce. The user's next delegate action must use `nonce + 1`; any relayer or wallet that retries with the original nonce will receive `DelegateActionInvalidNonce`. No funds are transferred, but the nonce slot is permanently consumed without the intended action executing — a broken atomicity guarantee at the protocol layer.

A secondary impact (also pre-fix): when `deposit > 0` **and** `receiver_id` mismatches, the `DepositWithFunctionCall` error is overwritten by `ReceiverMismatch`, returning the wrong error code to the user and to any off-chain tooling that branches on the error variant. [5](#0-4) 

---

### Likelihood Explanation

The trigger conditions are:
- Sender holds a **function-call** access key (not full-access).
- The delegate action's inner `FunctionCall` has `deposit > 0`.
- The `receiver_id` and `method_name` in the action match the key's permission exactly.

Any unprivileged user who signs such a delegate action (e.g., a wallet bug, a relayer that injects a deposit, or a deliberate test of the boundary) reaches this path. No validator or privileged role is required. The path is exercised on every shard that processes the receipt.

---

### Recommendation

The fix is already shipped as `ProtocolFeature::FixDelegateActionDepositWithFunctionCallError` at protocol version 85: [6](#0-5) 

No further action is needed for protocol versions ≥ 85. For completeness, the pre-fix code path (the `if ProtocolFeature::...enabled(...)` guard) should be collapsed once the minimum supported protocol version is raised above 85, removing the dead branch entirely.

---

### Proof of Concept

The existing test `test_delegate_deposit_with_function_call_reports_receiver_mismatch_before_fix` demonstrates the wrong-error-code half of the bug: [7](#0-6) 

To demonstrate nonce consumption, extend the test helper `deposit_with_function_call_and_receiver_mismatch` to use a **matching** `receiver_id` (instead of `"other.test.near"`). Pre-fix, `validate_delegate_action_key` returns `Ok(())` with `result.result = Err(DepositWithFunctionCall)`, and a subsequent read of the access key from `state_update` shows `nonce` advanced to `initial_nonce + 1` — confirming the trie mutation occurred despite the failed validation.
