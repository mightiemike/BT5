### Title
`tree_hash_from_stream` Silently Miscomputes Hash for Back-Reference-Compressed Serializations — (`File: src/serde/tools.rs`)

### Summary

`tree_hash_from_stream` does not handle the `BACK_REFERENCE` byte (`0xfe`) in the CLVM serialization format. When a back-reference-compressed blob is passed to this function, the back-reference path bytes are misinterpreted as an atom-size prefix, causing `decode_size` to either return an error or produce a hash over the wrong bytes. Meanwhile, `node_from_stream_backrefs` in `src/serde/de_br.rs` correctly resolves back-references. This creates a hash/metadata mismatch: two serialized blobs that represent the same CLVM tree — one compressed, one not — will produce different results from `tree_hash_from_stream`, breaking the invariant that the tree hash is a function of the tree's semantic content, not its serialization encoding.

### Finding Description

`tree_hash_from_stream` in `src/serde/tools.rs` handles three cases for each byte read from the stream:

- `0xff` (`CONS_BOX_MARKER`): push a pair parse operation
- `0x80` or `<= 0x7f` (`MAX_SINGLE_BYTE`): hash as atom
- anything else: call `decode_size(f, b[0])` and hash the following bytes as an atom [1](#0-0) 

`BACK_REFERENCE = 0xfe` is defined in the same file but is never checked in `tree_hash_from_stream`: [2](#0-1) 

`0xfe` (`11111110` binary) has seven leading `1`-bits. The CLVM atom-size encoding uses leading `1`-bits to determine prefix length (up to six bytes for `0xFC–0xFD`). `0xfe` is not a valid atom-size prefix — it is the reserved back-reference sentinel. Passing it to `decode_size` will either return `Err(EvalErr::SerializationError)` or decode a nonsensical atom length, causing the function to hash the back-reference path bytes as if they were atom payload.

By contrast, `node_from_stream_backrefs` in `src/serde/de_br.rs` explicitly checks for `BACK_REFERENCE`, resolves the path via `traverse_path_with_vec`, and returns the correct subtree: [3](#0-2) 

The `serialized_length_from_bytes_trusted` function in the same file as `tree_hash_from_stream` also explicitly handles `BACK_REFERENCE`: [4](#0-3) 

This confirms the omission in `tree_hash_from_stream` is not intentional design — the back-reference case is handled everywhere else in the same file except in the hash computation path.

### Impact Explanation

In Chia, puzzle hashes are the primary identity of a coin's spending condition. If `tree_hash_from_stream` is called on a back-reference-compressed program blob (which is a valid and accepted serialization format, as confirmed by `node_from_bytes_backrefs`), the computed puzzle hash will be wrong. This creates a consensus divergence: a node computing the hash via `tree_hash_from_stream` will disagree with a node that deserializes first and then hashes via `ObjectCache::treehash`. An attacker who can submit a back-reference-compressed program can cause the two code paths to produce different puzzle hashes for the same semantic program, potentially bypassing puzzle-hash-based spend validation or causing block rejection.

### Likelihood Explanation

The function is `pub` and accepts a raw `Cursor<&[u8]>` with no precondition that the input be free of back-references. Back-reference compression is a supported and documented serialization format in this codebase. Any caller — including Python bindings in `wheel/` — that passes a compressed blob to `tree_hash_from_stream` will trigger the bug. The likelihood depends on whether any production call site passes compressed bytes directly to this function rather than decompressing first.

### Recommendation

Add an explicit `BACK_REFERENCE` arm to the `ParseOp::SExp` match in `tree_hash_from_stream`. When `0xfe` is encountered, parse the path bytes (as `parse_path` does in `de_br.rs`), resolve the referenced subtree's already-computed hash from a side-table (similar to how `node_from_stream_backrefs` maintains a value stack), and push that hash onto `values`. Alternatively, document and enforce that `tree_hash_from_stream` must only be called on non-compressed serializations, and add a guard that returns `Err(EvalErr::SerializationError)` on `0xfe` with a clear error message.

### Proof of Concept

```
Compressed blob:   ff 86 666f6f626172 fe 01
                   ^  ^               ^  ^
                   |  "foobar"        |  path=1 (back-ref to "foobar")
                   CONS               BACK_REFERENCE

Uncompressed blob: ff 86 666f6f626172 86 666f6f626172
                   (foobar . foobar)
```

`node_from_bytes_backrefs` + `ObjectCache::treehash` on the compressed blob produces hash `9148834131750904c023598bed28db269bdb29012514579e723d63e27829bcba` (confirmed by the test in `de_br.rs`).

`tree_hash_from_stream` on the same compressed blob encounters `0xfe`, calls `decode_size(f, 0xfe)`, and either returns `Err(EvalErr::SerializationError)` or hashes the path byte `0x01` as a 6-byte atom payload — producing a completely different result than the correct tree hash, breaking the hash-identity invariant. pub fn tree_hash_from_stream(f: &mut Cursor<&[u8]>) -> Result<[u8; 32]> {
    let mut values: Vec<[u8; 32]> = Vec::new();
    let mut ops = vec![ParseOp::SExp];

    let mut b = [0; 1];
    while let Some(op) = ops.pop() {
        match op {
            ParseOp::SExp => {
                f.read_exact(&mut b)?;
                if b[0] == CONS_BOX_MARKER {
                    ops.push(ParseOp::Cons);
                    ops.push(ParseOp::SExp);
                    ops.push(ParseOp::SExp);
                } else if b[0] == 0x80 {
                    values.push(hash_atom(&[]));
                } else if b[0] <= MAX_SINGLE_BYTE {
                    values.push(hash_atom(&b));
                } else {
                    let blob_size = decode_size(f, b[0])?;
                    let blob = &f.get_ref()[f.position() as usize..];
                    if (blob.len() as u64) < blob_size {
                        return Err(EvalErr::SerializationError);
                    }
                    f.set_position(f.position() + blob_size);
                    values.push(hash_atom(&blob[..blob_size as usize]));
                }
            }
            ParseOp::Cons => {
                // cons
                let v2 = values.pop();
                let v1 = values.pop();
                values.push(hash_pair(&v1.unwrap(), &v2.unwrap()));
            }
        }
    }
    Ok(values.pop().unwrap())
}
