### Title
Governance Can Set Unbounded `ExitDuration` on Tiers, Trapping Locked User Funds — (`x/tieredrewards/types/tier.go`)

---

### Summary

The `Tier.Validate()` function in `x/tieredrewards/types/tier.go` enforces only a lower bound (`ExitDuration > 0`) on the tier exit duration, with no upper bound. A governance majority can pass a proposal to update a tier's `ExitDuration` to an arbitrarily large value (e.g., 100 years). Any user who subsequently calls `MsgTriggerExitFromTier` will have their `ExitUnlockAt` timestamp set to `blockTime + ExitDuration`, making their locked funds inaccessible for an unreasonably long period.

---

### Finding Description

`Tier.Validate()` checks that `ExitDuration` is strictly positive but imposes no maximum:

```go
// x/tieredrewards/types/tier.go
func (t Tier) Validate() error {
    if t.ExitDuration <= 0 {
        return fmt.Errorf("exit duration must be positive")
    }
    ...
}
``` [1](#0-0) 

The keeper's `SetTier` is explicitly documented as the write path for governance messages:

```go
// SetTier writes a tier after validation. Used by governance messages, genesis, and chain upgrades.
func (k Keeper) SetTier(ctx context.Context, tier types.Tier) error {
    if err := tier.Validate(); err != nil {
        return err
    }
    ...
}
``` [2](#0-1) 

When a user calls `MsgTriggerExitFromTier`, the message server reads the **current** tier's `ExitDuration` and stamps `ExitUnlockAt` on the position:

```go
// x/tieredrewards/keeper/msg_server.go
tier, err := ms.getTier(ctx, pos.TierId)
...
pos.TriggerExit(sdkCtx.BlockTime(), tier.ExitDuration)
``` [3](#0-2) 

Because `ExitDuration` is read at trigger time (not at lock time), a governance-updated value immediately affects all future exit triggers on that tier. There is no cap analogous to the `BonusApy` cap at `1.0` that was deliberately added for the rewards pool:

```go
// Cap BonusApy at 100% to prevent governance from draining the rewards pool.
if t.BonusApy.GT(math.LegacyOneDec()) {
    return fmt.Errorf("bonus apy must not exceed 1.0 (100%%): got %s", t.BonusApy)
}
``` [4](#0-3) 

No equivalent upper-bound guard exists for `ExitDuration`.

---

### Impact Explanation

A user who has locked funds in a tier and calls `TriggerExitFromTier` after a malicious governance update will have `ExitUnlockAt` set to `blockTime + <unreasonably large ExitDuration>`. The `WithdrawFromTier` and `ExitTierWithDelegation` paths both require the exit window to have elapsed before funds can be recovered. The user's principal (locked tokens and associated delegation) is inaccessible for the duration. This is a direct, concrete loss of fund accessibility — not merely a reward dilution or gas issue.

---

### Likelihood Explanation

Requires a governance majority to pass a malicious parameter-update proposal. This is the same threat model as the NounsDAO report (malicious majority). Likelihood is low, but the impact (permanent or near-permanent fund lock for all users who have not yet triggered exit on the affected tier) is high. **Low likelihood + High impact = Medium severity.**

---

### Recommendation

1. Enforce a `MAX_EXIT_DURATION` constant (e.g., 365 days) in `Tier.Validate()`, mirroring the `BonusApy` cap pattern already present in the same function.
2. Consider snapshotting `ExitDuration` into the position at lock time (or at trigger time from the position's stored tier snapshot) so that a governance change cannot retroactively affect users who are already in the exit process.
3. Consider preventing a decrease of `ExitDuration` while positions with a pending triggered-exit exist on that tier, analogous to the NounsDAO recommendation to prevent `forkPeriod` decrease during an active escrow period.

---

### Proof of Concept

1. Alice locks 1000 CRO in Tier 1 (currently `ExitDuration = 7 days`).
2. Governance passes `MsgUpdateTier` setting Tier 1's `ExitDuration = 36500 days` (100 years). `Tier.Validate()` accepts this because `36500 days > 0`.
3. Alice calls `MsgTriggerExitFromTier`. The keeper reads the current tier: `tier.ExitDuration = 36500 days`. `pos.TriggerExit(blockTime, 36500 days)` sets `pos.ExitUnlockAt = blockTime + 100 years`.
4. Alice's 1000 CRO and the underlying delegation are inaccessible until the year ~2126. `WithdrawFromTier` and `ExitTierWithDelegation` both enforce the `ExitUnlockAt` guard before releasing funds.

The root cause is the missing upper-bound check at [5](#0-4)  and the live tier read at trigger time at [3](#0-2) .
