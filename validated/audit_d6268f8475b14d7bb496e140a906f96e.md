Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` gates the router address instead of the end user, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` = pool's `msg.sender`, which resolves to the `MetricOmmSimpleRouter` contract address for all router-mediated swaps. Because allowlisted users cannot use the router unless the router itself is allowlisted, a pool admin is forced to add the router to `allowedSwapper[pool]`. Once the router is allowlisted, every address on-chain can call any router entry point against the restricted pool and the `beforeSwap` guard passes unconditionally, fully neutralizing the swap allowlist.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // pool's msg.sender — the router when called via MetricOmmSimpleRouter
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks that value as the identity to gate:

```solidity
// SwapAllowlistExtension.sol L37-38
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

When `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap`, the pool's `msg.sender` is the router contract. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][endUser]`.

**Forced precondition:** An allowlisted user (e.g., alice) who tries to swap through the router will have their call revert because `allowedSwapper[pool][router]` is false. The pool admin must call `setAllowedToSwap(pool, router, true)` to let alice use the router. This is not optional — it is the only way to enable router access for any approved user.

**Bypass:** Once the router is allowlisted, any address (bob, charlie, …) can call `MetricOmmSimpleRouter.exactInputSingle` targeting the restricted pool. The router calls `pool.swap` → pool calls `_beforeSwap(sender=router, ...)` → extension checks `allowedSwapper[pool][router]` → `true` → swap executes. The end user's identity is never checked.

**Contrast with `DepositAllowlistExtension`:** The deposit extension correctly gates `owner` (the second argument, the economically relevant actor), not `sender` (the immediate caller):

```solidity
// DepositAllowlistExtension.sol L38
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
```

The swap extension has no equivalent forwarding of the end user's identity.

## Impact Explanation
The swap allowlist is the sole mechanism for restricting who may trade in a `SwapAllowlistExtension`-gated pool. Once the router is allowlisted (a forced operational step), the allowlist is completely ineffective for any router-mediated call. Any unauthorized actor can execute swaps at oracle-derived prices, drain liquidity from bins, or trade in a pool explicitly configured to be access-controlled. This constitutes broken core pool functionality and an admin-boundary break where an unprivileged path bypasses the pool admin's access control. Severity: High.

## Likelihood Explanation
Every pool admin who deploys a `SwapAllowlistExtension` pool and wants their approved users to benefit from the router's slippage protection or multi-hop routing must allowlist the router — there is no alternative. The bypass is then reachable by any EOA or contract that calls the router with no special privileges, no flash loans, and no multi-block setup. There is no documentation warning against this pattern.

## Recommendation
The extension must gate the actual end user, not the intermediate caller. Two sound approaches:

1. **Forward the end user via `extensionData`:** Require the router to ABI-encode `msg.sender` (the end user) into `extensionData`. The extension decodes and checks that address when `sender` is a known trusted forwarder (the router). The pool admin configures the router as a trusted forwarder.

2. **Add a `recipient`/`originator` parameter to the hook:** Pass the end user's address as a distinct argument through `_beforeSwap` so the extension can gate on it directly, mirroring how `beforeAddLiquidity` receives `owner` separately from `sender`.

Either fix must ensure `SwapAllowlistExtension` never treats the router address as the identity to gate.

## Proof of Concept
```
Setup:
  pool admin deploys pool with SwapAllowlistExtension
  pool admin calls setAllowedToSwap(pool, alice, true)           // alice is approved
  pool admin calls setAllowedToSwap(pool, address(router), true) // required so alice can use the router

Attack:
  bob (not allowlisted) calls:
    MetricOmmSimpleRouter.exactInputSingle({
        pool: restrictedPool,
        tokenIn: token0,
        amountIn: X,
        recipient: bob,
        ...
    })

  router calls pool.swap(bob, ...)         // msg.sender = router
  pool calls _beforeSwap(sender=router, ...)
  SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  bob's swap executes — allowlist bypassed

Verification: repeat for exactInput, exactOutputSingle, exactOutput — all bypass identically.
```
