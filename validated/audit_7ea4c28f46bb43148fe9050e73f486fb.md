### Title
Stale Slow-Mode `LinkSigner` Transaction Re-Triggers Revoked Linked Signer Authorization — (`core/contracts/EndpointTx.sol`)

---

### Summary

A pending slow-mode `LinkSigner` transaction can re-set `linkedSigners[subaccount]` to a previously revoked signer address after the subaccount owner has already cleared it via the faster sequencer path. Because there is no cancellation mechanism for queued slow-mode transactions and no staleness guard in the slow-mode `LinkSigner` handler, the revocation is silently overwritten when the queued transaction eventually executes.

---

### Finding Description

Nado supports two distinct paths for updating `linkedSigners[subaccount]`:

**Path 1 — Sequencer (fast, immediate):** `processTransactionImpl` handles `TransactionType.LinkSigner` by calling `validateSignedTx` (which enforces a nonce) and then immediately writing:

```solidity
linkedSigners[signedTx.tx.sender] = address(uint160(bytes20(signedTx.tx.signer)));
``` [1](#0-0) 

**Path 2 — Slow mode (delayed, 3-day queue):** `processSlowModeTransactionImpl` handles `TransactionType.LinkSigner` by calling only `validateSender` (address match, no nonce) and then writing:

```solidity
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [2](#0-1) 

The slow-mode queue is strictly FIFO with a hardcoded 3-day delay and **no cancellation mechanism**:

```solidity
slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY,
    ...
});
``` [3](#0-2) 

Transactions are consumed in order and deleted on execution:

```solidity
SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
delete slowModeTxs[_slowModeConfig.txUpTo++];
``` [4](#0-3) 

**The desynchronization:** When a user submits a slow-mode `LinkSigner` (linking signer B) and later revokes B via the sequencer path (setting `linkedSigners` to `address(0)`), the pending slow-mode transaction is still in the queue. When it eventually executes — up to 3 days later, and triggerable by anyone via the public `executeSlowModeTransaction()` — it unconditionally overwrites `linkedSigners[subaccount]` back to signer B, restoring the revoked authorization. [5](#0-4) 

The slow-mode `LinkSigner` handler performs no check against the current value of `linkedSigners[txn.sender]` and has no nonce guard, so it cannot detect that the authorization was already superseded. [2](#0-1) 

---

### Impact Explanation

`linkedSigners` is the authorization gate for all sequencer-submitted signed transactions on behalf of a subaccount, including `WithdrawCollateral`, `MintNlp`, `BurnNlp`, `TransferQuote`, and order matching. A re-linked signer can:

- Submit withdrawal transactions draining the subaccount's collateral.
- Place or cancel orders on behalf of the victim.
- Trigger NLP mint/burn operations affecting the victim's position.

The `getLinkedSigner` function also propagates the linked signer to all isolated subaccounts of the parent, so the blast radius extends to every isolated position the victim holds. [6](#0-5) 

---

### Likelihood Explanation

The scenario is realistic:

1. A user without direct sequencer access submits a slow-mode `LinkSigner` to authorize a trading bot (signer B).
2. Signer B's private key is later compromised, or the user simply wants to revoke access.
3. The user signs a `LinkSigner(signer=bytes32(0))` and submits it to the sequencer (fast path) to immediately clear B.
4. The user believes B is revoked.
5. The pending slow-mode transaction executes (callable by anyone after 3 days via `executeSlowModeTransaction()`), re-writing `linkedSigners[subaccount] = B`.
6. The attacker, holding B's key, now has full signing authority over the subaccount.

There is no on-chain signal to the user that the revocation will be overwritten, and no way to cancel the queued transaction.

---

### Recommendation

Add a staleness guard in the slow-mode `LinkSigner` handler that rejects the update if the current `linkedSigners[txn.sender]` value differs from what the transaction was submitted to set — or, more robustly, include a nonce in the slow-mode `LinkSigner` struct (mirroring the sequencer path) so that a subsequent sequencer-path update invalidates any pending slow-mode transaction for the same subaccount. Alternatively, provide a cancellation path for slow-mode transactions that allows the original submitter to remove their own queued entry.

---

### Proof of Concept

```
1. Alice (subaccount owner) submits:
       submitSlowModeTransaction(LinkSigner{ sender: aliceSubaccount, signer: bytes32(bobAddress) })
   → Queued at slowModeTxs[N], executableAt = now + 3 days.

2. Bob's key is compromised. Alice immediately signs and submits to the sequencer:
       LinkSigner{ sender: aliceSubaccount, signer: bytes32(0), nonce: currentNonce }
   → Sequencer processes it: linkedSigners[aliceSubaccount] = address(0). Bob is revoked.

3. After 3 days, attacker calls:
       endpoint.executeSlowModeTransaction()
   → processSlowModeTransactionImpl executes slowModeTxs[N]:
       validateSender(aliceSubaccount, aliceAddress)  // passes
       requireSubaccount(aliceSubaccount)              // passes
       linkedSigners[aliceSubaccount] = bobAddress     // Bob re-linked

4. Attacker (holding Bob's key) signs:
       WithdrawCollateral{ sender: aliceSubaccount, productId: USDC, amount: fullBalance }
   → Sequencer validates: getLinkedSigner(aliceSubaccount) == bobAddress ✓
   → Collateral drained.
```
