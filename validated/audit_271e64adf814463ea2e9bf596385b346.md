Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing non-allowlisted callers to deposit into curated pools — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the address that actually calls `addLiquidity` and provides tokens via the swap callback) and checks only the caller-supplied `owner` argument (the position recipient). Any unprivileged address can bypass the allowlist by calling `pool.addLiquidity(allowlisted_address, …)`, satisfying the check with an allowlisted `owner` while the non-allowlisted `sender` provides the tokens.

## Finding Description

**Root cause — wrong address checked:**

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the position recipient:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both:

```solidity
// ExtensionCalling.sol lines 95-98
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but names it `address` (unnamed, discarded) and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol lines 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

**Exploit path:**
1. Pool is deployed with `DepositAllowlistExtension`; only `alice` is allowlisted.
2. Attacker (non-allowlisted `bob`) calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
3. Pool calls `_beforeAddLiquidity(bob /*sender*/, alice /*owner*/, …)`.
4. Extension checks `allowedDepositor[pool][alice]` → `true` → passes.
5. `bob` provides tokens via the swap callback; LP shares are minted to `alice`.
6. `bob` has deposited into a restricted pool despite not being allowlisted.

**Why existing guards fail:** There is no secondary check on `sender`. The `removeLiquidity` path enforces `msg.sender == owner`, but `addLiquidity` has no such constraint, making the `owner` parameter fully attacker-controlled.

## Impact Explanation
The allowlist is the pool admin's primary access-control mechanism for curated pools. Bypassing it breaks the core invariant that only approved depositors can add liquidity. The NatSpec of the contract explicitly states it "Gates `addLiquidity` by depositor address, per pool" — the depositor (`sender`) is never checked. This is broken core pool functionality: the allowlist provides no protection against non-allowlisted callers.

## Likelihood Explanation
Exploitation requires no special privileges, no malicious setup, and no off-chain coordination. Any unprivileged address can call `addLiquidity` with any allowlisted address as `owner`. The attack is repeatable at will and costs only gas plus the deposited tokens (which go to the allowlisted `owner`'s LP position). The attacker does not profit directly, but the pool admin's curation invariant is permanently violated.

## Recommendation
Replace the unnamed first parameter with `sender` and check it instead of (or in addition to) `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

    // bob must implement IMetricOmmAddLiquidityCallback to provide tokens
    pool.addLiquidity(alice, 0, deltas, callbackData, "");
    vm.stopPrank();

    // Verify: bob (non-allowlisted) successfully deposited; alice received LP shares
    // The allowlist check passed because alice (owner) is allowlisted, not bob (sender)
