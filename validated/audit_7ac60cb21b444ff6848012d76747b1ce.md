### Title
Duplicate Position IDs in `MsgClaimTierRewards` Enable Double-Claim of Bonus Rewards — (File: `x/tieredrewards/keeper/msg_server.go`)

---

### Summary

`ClaimTierRewards` loads all requested positions from the store into a slice **before** processing any of them. There is no deduplication of `msg.PositionIds`. If the same position ID appears twice in the list, the position is loaded twice with identical pre-claim state, and `processEventsAndClaimBonus` pays out bonus coins twice while consuming validator event reference counts that belong to other positions.

---

### Finding Description

In `ClaimTierRewards`, positions are collected in a loop and then processed in batch: [1](#0-0) 

```go
positions := make([]types.PositionState, 0, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    pos, err := ms.getPositionState(ctx, posId)
    ...
    if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
        return nil, err
    }
    positions = append(positions, pos)
}
totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
```

`validateClaimRewards` only checks ownership — it does not detect duplicate IDs. If `msg.PositionIds = [P1, P1]`, the same position is appended twice with identical pre-claim state (`LastEventSeq`, `LastBonusAccrual`).

`claimRewardsAndUpdatesPositions` then iterates over the slice: [2](#0-1) 

For the **first copy** of P1:
- `claimBaseRewards` withdraws base rewards (distribution module state updated → 0 remaining).
- `processEventsAndClaimBonus` queries events since `OLD_LastEventSeq`, decrements each event's reference count, sends bonus coins `B`, and updates the in-memory `pos` (LastEventSeq, LastBonusAccrual).
- `setPosition` persists the updated state to the store.

For the **second copy** of P1 (still holding the old pre-claim state):
- `claimBaseRewards` returns 0 (distribution already drained — no double base reward).
- `processEventsAndClaimBonus` queries events since `OLD_LastEventSeq` again. [3](#0-2) 

The critical condition: validator events are reference-counted across all positions delegating to the same validator. After the first processing, each event's ref count is decremented by 1. If any other position also delegates to the same validator (ref count was ≥ 2), the events still exist in the store with ref count ≥ 1. The second copy therefore finds the same events, computes the same bonus `B`, and calls: [4](#0-3) 

```go
k.bankKeeper.SendCoinsFromModuleToAccount(ctx, types.RewardsPoolName, ownerAddr, bonusCoins)
```

The attacker receives `2B` instead of `B`. The extra `decrementEventRefCount` calls also consume the reference counts belonging to other positions, permanently destroying their ability to claim bonus rewards for those events.

---

### Impact Explanation

- **Direct fund loss**: bonus coins are sent twice from `RewardsPoolName` to the attacker's address.
- **Collateral damage**: other positions delegating to the same validator lose their event references, making their future bonus claims return 0 for the affected events.
- **Drainable**: by submitting repeated transactions with `PositionIds = [P1, P1, P1, ...]` (up to the pool balance), the attacker can drain the entire bonus rewards pool.

The corrupted value is the `RewardsPoolName` module account balance and the validator event reference counts stored in the tiered rewards module.

---

### Likelihood Explanation

- Entry path: standard `MsgClaimTierRewards` transaction, no privileges required.
- Precondition: attacker needs ≥ 2 positions delegating to the same validator (to ensure event ref count ≥ 2). This is a normal user action (`MsgLockTier` or `MsgCommitDelegationToTier`).
- No governance, admin key, or social engineering required.

---

### Recommendation

Deduplicate `msg.PositionIds` before processing, either in `msg.Validate()` or at the start of `ClaimTierRewards`. A simple approach:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    if _, dup := seen[posId]; dup {
        return nil, errorsmod.Wrapf(sdkerrors.ErrInvalidRequest, "duplicate position id: %d", posId)
    }
    seen[posId] = struct{}{}
    ...
}
```

---

### Proof of Concept

1. Attacker creates position **P1** delegating to validator **V** via `MsgLockTier`.
2. Attacker creates position **P2** delegating to the same validator **V** (ensures each validator event has ref count ≥ 2).
3. Validator events accumulate for V (e.g., a bond/unbond cycle).
4. Attacker submits `MsgClaimTierRewards{Owner: attacker, PositionIds: [P1, P1]}`.
5. Both copies of P1 are loaded with identical pre-claim state.
6. First processing: bonus `B` sent to attacker; event ref counts decremented (e.g., 2 → 1).
7. Second processing (old state): same events found (ref count 1 > 0); bonus `B` sent again.
8. Attacker receives `2B`; P2's event references are consumed, permanently zeroing P2's future bonus for those events.

func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
}
