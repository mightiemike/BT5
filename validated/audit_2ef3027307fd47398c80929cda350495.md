### Title
Spice (V6) Chunk Header Round-Trips Through `ChunkHeaderView` as V3, Producing a Divergent Chunk Hash — (File: `core/primitives/src/views.rs`)

### Summary

The `From<ChunkHeaderView> for ShardChunkHeader` conversion in `core/primitives/src/views.rs` has no arm for `ShardChunkHeaderInnerV6SpiceTxOnly`. When a Spice chunk header (V6 inner, Borsh discriminant `5`) is serialized to `ChunkHeaderView` and then reconstructed, the reverse conversion silently produces a `ShardChunkHeaderInner::V3` (Borsh discriminant `2`). The chunk hash is recomputed from the wrong Borsh encoding, so the reconstructed header carries a hash that is provably different from the original on-chain hash. Any consumer that calls `.into::<ShardChunkHeader>()` on a `ChunkHeaderView` received from the RPC — including light-client verifiers and indexers — will compute a wrong chunk hash for every Spice chunk.

### Finding Description

**Serialization path — `From<ShardChunkHeader> for ChunkHeaderView`** (lines 1132–1167):

When the inner variant is `ShardChunkHeaderInner::V6`, the accessor methods return:
- `congestion_info()` → `CongestionInfo::default()` (line 219 of `shard_chunk_header_inner.rs`)
- `bandwidth_requests()` → `None` (line 231)
- `has_proposed_split_field()` → `false` (line 265)

So the resulting `ChunkHeaderView` has `congestion_info = Some(default)`, `bandwidth_requests = None`, `proposed_split = None`.

**Deserialization path — `From<ChunkHeaderView> for ShardChunkHeader`** (lines 1170–1265):

The match at line 1176 dispatches on `(proposed_split, bandwidth_requests, congestion_info)`. The V6 combination `(None, None, Some(_))` falls into the V3 arm at line 1216:

```rust
(None, None, Some(congestion_info)) => {
    ShardChunkHeaderInner::V3(ShardChunkHeaderInnerV3 { ... })
}
```

`ShardChunkHeaderV3::init()` at line 1263 then recomputes the hash from the V3 Borsh encoding. The Borsh discriminant for V3 is `2`; for V6 it is `5`. Because the discriminant is part of the hashed bytes, the resulting `ChunkHash` is different from the original V6 chunk hash. The `chunk_hash` field preserved in `ChunkHeaderView` (line 1139) is the original V6 hash, so after the round-trip the view's `chunk_hash` and the reconstructed header's `hash` field are inconsistent.

### Impact Explanation

The `ChunkHeaderView` is the canonical JSON representation of chunk headers returned by the `get_block` and `get_chunk` RPC endpoints. Any consumer that reconstructs a `ShardChunkHeader` from a `ChunkHeaderView` — light-client proof verifiers, indexers, cross-chain bridges — will compute a chunk hash that does not match the on-chain hash for every Spice chunk. Because the chunk hash is the input to the chunk producer's signature (`signer.sign_bytes(hash.as_ref())`), signature verification over the reconstructed header will also fail. The `chunk_hash` field in the view is the correct V6 hash, but the reconstructed header's `hash` field is a V3 hash, making the two irreconcilable without access to the original Borsh bytes. [5](#0-4) [6](#0-5) 

### Likelihood Explanation

The `Spice` protocol feature gates V6 chunk header production. Once `ProtocolFeature::Spice` is enabled on a network, every chunk produced by a Spice-enabled chunk producer carries a `ShardChunkHeaderInner::V6` inner. The `From<ChunkHeaderView> for ShardChunkHeader` conversion is a public, stable Rust trait impl in `near-primitives` that any downstream crate or tool can invoke. The `validate_version` check on `ShardChunkHeader` confirms V6 is the only valid inner for Spice-enabled epochs, so the divergence is systematic, not edge-case. [7](#0-6) [8](#0-7) 

### Recommendation

Add a V6 arm to `From<ChunkHeaderView> for ShardChunkHeader`. The `ChunkHeaderView` struct should carry an explicit version discriminant (or an `is_spice_chunk: bool` flag) so the reverse conversion does not have to infer the inner version from optional field presence. Until the struct is extended, a short-term fix is to detect the Spice case by checking that `congestion_info` is `Some(default)` and `bandwidth_requests` is `None` and `proposed_split` is `None` while `gas_limit == 0` (the sentinel set by `is_spice_chunk` branches), and reconstruct `ShardChunkHeaderInner::V6` in that case. The existing TODO comment at line 1174 already identifies the correct long-term fix. [9](#0-8) 

### Proof of Concept

```rust
use near_primitives::sharding::{ShardChunkHeader, ShardChunkHeaderV3, ShardChunkHeaderInner};
use near_primitives::views::ChunkHeaderView;

// 1. Produce a V6 Spice chunk header (requires Spice feature enabled).
let v6_header = ShardChunkHeader::V3(ShardChunkHeaderV3::new_for_spice(
    prev_block_hash,
    encoded_merkle_root,
    encoded_length,
    height,
    shard_id,
    prev_outgoing_receipts_root,
    tx_root,
    &signer,
));
let original_hash = *v6_header.chunk_hash();

// 2. Serialize to ChunkHeaderView (as the RPC does).
let view: ChunkHeaderView = v6_header.into();
assert_eq!(view.chunk_hash, original_hash.0); // preserved in view

// 3. Reconstruct ShardChunkHeader from the view (as a light client would).
let reconstructed: ShardChunkHeader = view.into();

// 4. The reconstructed header is V3, not V6.
match reconstructed.take_inner() {
    ShardChunkHeaderInner::V3(_) => println!("BUG: got V3, expected V6"),
    ShardChunkHeaderInner::V6(_) => println!("OK"),
    _ => unreachable!(),
}

// 5. The chunk hash is different.
let reconstructed_hash = *reconstructed.chunk_hash();
assert_ne!(original_hash, reconstructed_hash, "chunk hash diverged");
// original_hash  = sha256(borsh_discriminant=5 || v6_fields || ...)
// reconstructed  = sha256(borsh_discriminant=2 || v3_fields || ...)
