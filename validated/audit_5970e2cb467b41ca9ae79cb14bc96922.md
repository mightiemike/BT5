### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling Merkle Proof Forgery via Unvalidated `tx_id` Input — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` accepts a caller-supplied `tx_id` and embeds it directly into the Merkle root computation without validating that it is a leaf node (an actual transaction hash). An unprivileged NEAR caller can substitute an internal Merkle tree node hash for `tx_id`, supply a crafted `merkle_proof` and `tx_index`, and cause the function to return `true` for a transaction that was never included in the block. The function is deprecated but not removed and carries no access restriction, so any account can invoke it.

---

### Finding Description

`verify_transaction_inclusion` is still a `pub` NEAR method with only a `#[pause]` guard (active by default): [1](#0-0) 

The function's own doc-comment acknowledges the flaw: [2](#0-1) 

Inside the function, `args.tx_id` is passed directly to `compute_root_from_merkle_proof` with no check that it represents a leaf: [3](#0-2) 

`compute_root_from_merkle_proof` is a pure hash-chaining loop that treats whatever 32-byte value it receives as the starting node: [4](#0-3) 

Because the loop only uses `current_position % 2` to decide left/right ordering and then divides by 2, any internal node `N` at depth `d` in the real Merkle tree can be used as `tx_id` with a proof of length `d` (the path from `N` to the root) and a matching `tx_index` whose lower `d` bits encode that path. The computation will correctly reconstruct the Merkle root and the function returns `true`.

The `verify_transaction_inclusion_v2` fix that was introduced to mitigate this (by requiring a coinbase proof of equal length) does not protect callers who invoke the deprecated endpoint directly: [5](#0-4) 

---

### Impact Explanation

Any NEAR smart contract that calls `verify_transaction_inclusion` and acts on a `true` result — for example, a bridge that releases funds upon proof of a Bitcoin deposit transaction — can be deceived into accepting a forged inclusion proof. The attacker does not need to produce a valid Bitcoin transaction; they only need to know the real transactions in a confirmed block (all public on-chain data) to derive an internal node hash and craft the matching proof inputs.

---

### Likelihood Explanation

The attack requires no privileged role, no leaked key, and no social engineering. Bitcoin transaction data is fully public. The attacker constructs the forged call entirely from on-chain data and submits it as a normal NEAR function call. The only prerequisite is that the target block has been submitted to the light client (which is the normal operating state of the contract).

---

### Recommendation

Remove `verify_transaction_inclusion` entirely rather than leaving it deprecated-but-callable. All callers must migrate to `verify_transaction_inclusion_v2`. If backward compatibility is required during a transition window, gate the deprecated function with an explicit access-control role so that only trusted accounts can invoke it, and add a runtime check that `args.merkle_proof.len()` equals the expected full tree depth derived from the stored header.

---

### Proof of Concept

Given a confirmed block with four transactions `[t1, t2, t3, t4]` and Merkle tree:

```
        root
       /    \
      A      B
     / \    / \
    t1  t2 t3  t4
```

1. Compute `A = double_sha256(t1 || t2)` from public transaction data.
2. Call `verify_transaction_inclusion` with:
   - `tx_id = A`
   - `tx_index = 0`
   - `merkle_proof = [B]`
   - `tx_block_blockhash` = hash of the confirmed block
   - `confirmations = 1`
3. Inside `compute_root_from_merkle_proof`: `current_hash = A`, `current_position = 0`, `0 % 2 == 0` → `current_hash = hash(A, B) = root`.
4. `root == header.block_header.merkle_root` → function returns `true`.

`A` is an internal node, not a real transaction. The function has certified a non-existent transaction as included. [6](#0-5) [3](#0-2)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
