### Title
`authorized_declarer_accounts` / `block_declare` admission restriction bypassed via consensus proposal validation path — (`crates/apollo_gateway/src/gateway.rs`, `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The gateway enforces a declare-permission restriction (`authorized_declarer_accounts` / `block_declare`) exclusively inside `check_declare_permissions`, which is called only on the public/P2P `add_tx` path. The consensus proposal-validation path (`validate_proposal.rs`) converts incoming `ConsensusTransaction::Declare` transactions directly to internal form via `convert_consensus_tx_to_internal_consensus_tx` and forwards them to the batcher — with no call to `check_declare_permissions` at any point. A malicious consensus proposer can therefore inject an unauthorized declare transaction into a block proposal; every honest validator node will accept and execute it, committing the unauthorized class to chain state.

---

### Finding Description

**Enforced path — gateway `add_tx`:**

`add_tx_inner` calls `check_declare_permissions` before any conversion:

```
if let RpcTransaction::Declare(ref declare_tx) = tx {
    if let Err(e) = self.check_declare_permissions(declare_tx) { … return Err(e); }
}
```

`check_declare_permissions` enforces two independent restrictions:
1. `block_declare = true` → all declares rejected unconditionally.
2. `authorized_declarer_accounts = Some([…])` → only listed senders allowed. [1](#0-0) [2](#0-1) 

Both flags live exclusively in `GatewayStaticConfig`; they are not propagated to the blockifier or the OS. [3](#0-2) 

**Bypassed path — consensus proposal validation:**

When a validator node receives a proposal, `handle_proposal_part` in `validate_proposal.rs` processes each `ProposalPart::Transactions` batch by calling:

```rust
transaction_converter.convert_consensus_tx_to_internal_consensus_tx(tx)
``` [4](#0-3) 

`convert_consensus_tx_to_internal_consensus_tx` dispatches `ConsensusTransaction::RpcTransaction` directly to `convert_rpc_tx_to_internal`: [5](#0-4) 

`convert_rpc_tx_to_internal` performs Sierra compilation and hash calculation but contains **no call to `check_declare_permissions`** and no reference to `authorized_declarer_accounts` or `block_declare`: [6](#0-5) 

The resulting `InternalConsensusTransaction` is then sent directly to the batcher via `send_txs_for_proposal`: [7](#0-6) 

The batcher feeds it to the blockifier for execution with no further admission check. [8](#0-7) 

The grep confirms `authorized_declarer|block_declare|check_declare` appears **only** in `gateway.rs`, `gateway_test.rs`, `gateway_config`, deployment configs, and integration-test utils — never in the batcher, blockifier, transaction converter, or consensus orchestrator.

---

### Impact Explanation

An unauthorized `Declare` transaction injected by a malicious proposer into a consensus proposal will be:
1. Converted to internal form by every honest validator (no permission check).
2. Forwarded to each validator's batcher and executed by the blockifier.
3. Committed to chain state if the proposal reaches consensus.

This produces **wrong state**: a contract class declared by an address that the operator explicitly prohibited, or during a period when `block_declare = true` was intended to halt all declarations. The `block_declare` flag is specifically described as a temporary operational kill-switch; bypassing it is a direct violation of the operator's intended invariant.

Matching impact scope: *Critical — Wrong state … class hash … or revert result from blockifier/syscall/execution logic for accepted input.*

---

### Likelihood Explanation

Requires a malicious consensus proposer — one of the BFT validator set. In a permissioned network where `authorized_declarer_accounts` is configured (the exact scenario the feature targets), the validator set is known and bounded. A single dishonest proposer in a round can craft and broadcast the offending proposal; honest validators will accept it because the conversion path has no gate. The `block_declare` bypass is reachable by any proposer during an emergency maintenance window, which is precisely when the flag is most operationally critical.

---

### Recommendation

Apply the declare-permission check at the conversion boundary that is shared by all ingestion paths, not only the public gateway. Concretely:

1. Move `check_declare_permissions` logic (or an equivalent predicate) into `convert_rpc_tx_to_internal` inside `TransactionConverter`, or expose it as a separate validation step called by both `add_tx_inner` and `handle_proposal_part` before forwarding to the batcher.
2. Alternatively, propagate `block_declare` and `authorized_declarer_accounts` into the batcher/blockifier layer so that the restriction is enforced at execution time regardless of ingestion path.
3. Add a test that constructs a `ConsensusTransaction::RpcTransaction(RpcTransaction::Declare(…))` with an unauthorized sender and verifies it is rejected when processed through the consensus proposal validation path.

---

### Proof of Concept

1. Operator sets `gateway_config.static_config.authorized_declarer_accounts = "0xAUTHORIZED"` (or `block_declare = true`).
2. Attacker controls one consensus validator that becomes the proposer for a round.
3. Attacker constructs `ConsensusTransaction::RpcTransaction(RpcTransaction::Declare(RpcDeclareTransaction::V3(…)))` with `sender_address = 0xUNAUTHORIZED`.
4. Attacker broadcasts this transaction as part of a `ProposalPart::Transactions` batch to all peers.
5. Each honest validator's `handle_proposal_part` calls `convert_consensus_tx_to_internal_consensus_tx` → `convert_rpc_tx_to_internal` (line 391: `tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?`). No permission check is performed.
6. The converted `InternalConsensusTransaction` is sent to the batcher via `send_txs_for_proposal` and executed by the blockifier.
7. If the proposal reaches the 2/3+ quorum, the unauthorized class is committed to chain state — bypassing the `authorized_declarer_accounts` / `block_declare` restriction entirely. [9](#0-8) [10](#0-9)

    pub async fn send_txs_for_proposal(
        &mut self,
        send_txs_for_proposal_input: SendTxsForProposalInput,
    ) -> BatcherResult<SendTxsForProposalStatus> {
        let SendTxsForProposalInput { proposal_id, txs: new_txs } = send_txs_for_proposal_input;
        self.ensure_validate_proposal_exists(proposal_id)?;

        if self.is_active(proposal_id).await {
            // The proposal is active. Send transactions through the tx provider.
            let tx_provider_sender = &self
                .validate_tx_streams
                .get(&proposal_id)
                .expect("Expecting tx_provider_sender to exist during batching.");
            for tx in new_txs {
                tx_provider_sender.send(tx).await.map_err(|err| {
                    error!("Failed to send transaction to the tx provider: {}", err);
                    BatcherError::InternalError
                })?;
            }
            return Ok(SendTxsForProposalStatus::Processing);
        }

        // The proposal is no longer active, cannot accept more transactions.
        let proposal_result =
            self.get_completed_proposal_result(proposal_id).await?.expect("Proposal should exist.");
        match proposal_result {
            Ok(_) => panic!("Proposal finished validation before all transactions were sent."),
            Err(err) => match proposal_status_from(err)? {
                ProposalStatus::InvalidProposal(err) => {
                    Ok(SendTxsForProposalStatus::InvalidProposal(err))
                }
                status => {
                    error!("Unexpected batcher status for send txs for proposal: {status:?}");
                    Err(BatcherError::InternalError)
                }
            },
        }
    }
