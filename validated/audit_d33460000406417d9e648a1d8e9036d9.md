Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` gates on caller-supplied `owner` instead of `sender`, allowing full allowlist bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards its first argument (`sender`, the actual `msg.sender` of `addLiquidity`) and instead checks `owner`, a value freely supplied by the caller. Because `MetricOmmPool.addLiquidity` imposes no restriction on who may pass an arbitrary `owner`, any unprivileged address can bypass the deposit allowlist by setting `owner` to any already-allowlisted address. This breaks the admin-enforced access boundary the extension is designed to enforce.

## Finding Description

`MetricOmmPool.addLiquidity` calls `_beforeAddLiquidity(msg.sender, owner, ...)`, passing the real caller as `sender` and the caller-controlled `owner` parameter as the second argument: [1](#0-0) 

Inside `DepositAllowlistExtension.beforeAddLiquidity`, the first argument (`sender`) is unnamed and discarded. The allowlist check is performed exclusively on `owner`: [2](#0-1) 

The storage mapping and setter both use the name `depositor`, making the intended subject of the check unambiguous — it is supposed to vet the caller, not the position recipient: [3](#0-2) 

The sibling `SwapAllowlistExtension.beforeSwap` demonstrates the correct pattern — it checks `sender` (the swap caller) and discards the second argument (`recipient`): [4](#0-3) 

Because `addLiquidity` accepts any `owner` value from the caller with no restriction, the exploit path is trivial: an unprivileged `bob` calls `pool.addLiquidity(alice, ...)` where `alice` is allowlisted. The extension evaluates `allowedDepositor[pool][alice]` → `true` and passes. `bob` pays the tokens; `alice` receives the LP position. The wrong value checked is `allowedDepositor[msg.sender][owner]` — it should be `allowedDepositor[msg.sender][sender]`.

## Impact Explanation

This is an admin-boundary break. The pool admin's deposit allowlist — intended to curate who may provide liquidity — is fully nullified. Any unprivileged caller can deposit into a restricted pool by naming any allowlisted address as `owner`. Consequences include unauthorized liquidity provision to regulatory- or manipulation-sensitive pools, forced LP positions on allowlisted addresses without their consent (griefing), and complete defeat of any pool-depth control the admin intended to enforce.

## Likelihood Explanation

Exploitation requires a single direct call to `pool.addLiquidity(allowlistedAddress, ...)` with no special privileges, flash loans, or multi-step setup. The attacker only needs to know one allowlisted address (e.g., the pool admin themselves, who is typically allowlisted). The attack is repeatable in every block at zero marginal cost beyond gas.

## Recommendation

Change `beforeAddLiquidity` to name and check `sender` (the first argument) instead of `owner`, matching the documented intent and the `SwapAllowlistExtension` pattern:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` attached to `beforeAddLiquidity`.
2. Pool admin calls `depositExtension.setAllowedToDeposit(pool, alice, true)` — only `alice` is allowlisted.
3. `bob` (not allowlisted) calls `pool.addLiquidity(alice, salt, deltas, callbackData, "")`.
   - Pool passes `sender = bob`, `owner = alice` to the extension.
   - Extension evaluates `allowedDepositor[pool][alice]` → `true` and does not revert.
   - `bob` pays the input tokens; `alice` receives the LP position.
4. `bob` has deposited into a curated pool without being allowlisted, violating the admin's access policy.

A Foundry test reproducing this: set up a pool with the extension, allowlist only `alice`, then `vm.prank(bob); pool.addLiquidity(alice, ...)` and assert it does **not** revert — confirming the bypass. The existing test suite misses this because every allowlist test uses a setup where `sender == owner` (the `TestCaller` is both caller and position holder), so the wrong field coincidentally produces the correct result.
