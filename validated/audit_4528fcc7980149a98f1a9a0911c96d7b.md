### Title
Sanctioned User Can Complete Slow-Mode Collateral Withdrawal After Being Sanctioned — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `requireUnsanctioned` check is enforced at slow-mode **submission** time but is absent at slow-mode **execution** time. A user who submits a `WithdrawCollateral` slow-mode transaction while unsanctioned, then becomes sanctioned during the mandatory 3-day delay, can still receive their collateral when the transaction executes. This is a direct analog to the external report: deposits enforce the restriction, withdrawals do not.

---

### Finding Description

Nado uses a `requireUnsanctioned` guard (defined in `EndpointStorage.sol`) to enforce compliance with the Chainalysis sanctions list. This guard is applied symmetrically on the **deposit** side: `depositCollateralWithReferral` in `Endpoint.sol` calls `requireUnsanctioned` on both the caller and the subaccount owner before accepting funds. [1](#0-0) 

On the **slow-mode withdrawal** side, `submitSlowModeTransactionImpl` in `EndpointTx.sol` calls `requireUnsanctioned(sender)` at submission time: [2](#0-1) 

However, when the slow-mode transaction is later **executed** via `processSlowModeTransactionImpl`, the `WithdrawCollateral` branch performs no `requireUnsanctioned` check whatsoever: [3](#0-2) 

The execution path is `executeSlowModeTransaction` → `_executeSlowModeTransaction` → `processSlowModeTransaction` → `processSlowModeTransactionImpl`. `executeSlowModeTransaction` has no access control — any caller, including the sanctioned user themselves, can trigger it: [4](#0-3) 

The `sendTo` in the slow-mode `WithdrawCollateral` path is hardcoded to `address(0)`, which resolves to the subaccount owner's address inside `withdrawCollateral`, so the funds flow directly to the sanctioned user: [5](#0-4) 

---

### Impact Explanation

A user who is added to the Chainalysis sanctions list after submitting a slow-mode withdrawal can still receive their collateral. The protocol's compliance guarantee — that sanctioned addresses cannot receive funds — is broken for the slow-mode withdrawal path. The asset delta is the full collateral amount the user had on deposit, transferred to a sanctioned address on-chain.

---

### Likelihood Explanation

The slow-mode delay is hardcoded to 3 days (`SLOW_MODE_TX_DELAY`). Sanctions events (e.g., OFAC designations) are common and can happen at any time. A user anticipating sanctions could pre-submit a slow-mode withdrawal before the designation, then wait for execution. Alternatively, a user sanctioned for unrelated reasons during an active slow-mode window would trigger this inadvertently. The attacker-controlled entry point is `executeSlowModeTransaction`, which is permissionless.

---

### Recommendation

Add a `requireUnsanctioned` check inside `processSlowModeTransactionImpl` for the `WithdrawCollateral` branch, mirroring the check already present at submission time. The check should validate the resolved recipient address (the subaccount owner, i.e., `address(uint160(bytes20(txn.sender)))`):

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.WithdrawCollateral memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.WithdrawCollateral)
    );
    validateSender(txn.sender, sender);
+   requireUnsanctioned(address(uint160(bytes20(txn.sender))));  // re-check at execution
    clearinghouse.withdrawCollateral(
        txn.sender,
        txn.productId,
        txn.amount,
        address(0),
        nSubmissions
    );
``` [3](#0-2) 

---

### Proof of Concept

1. Alice holds collateral in Nado and is not sanctioned.
2. Alice calls `submitSlowModeTransaction` with a `WithdrawCollateral` transaction. `requireUnsanctioned(msg.sender)` passes. The transaction is queued with `executableAt = block.timestamp + SLOW_MODE_TX_DELAY`.
3. Within the 3-day window, Alice is added to the Chainalysis sanctions list.
4. After the delay, Alice (or anyone) calls `executeSlowModeTransaction`. `processSlowModeTransactionImpl` handles the `WithdrawCollateral` branch with no `requireUnsanctioned` check.
5. `clearinghouse.withdrawCollateral` executes, transferring Alice's collateral to her address — a now-sanctioned address — via `handleWithdrawTransfer`. [6](#0-5) [7](#0-6)
