Audit Report

## Title
`DepositAllowlistExtension` checks position `owner` instead of actual `sender`, allowing any unauthorized address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual caller who pays tokens) and gates access on `owner` (the position recipient), which is a free caller-supplied argument. Any unprivileged address can bypass the deposit allowlist by naming an allowlisted address as `owner`, while the pool admin's intended access restriction is fully circumvented.

## Finding Description
`ExtensionCalling._beforeAddLiquidity` encodes and forwards both `sender` (`msg.sender` of `pool.addLiquidity`, the actual payer) and `owner` (the position recipient) to every configured extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L95-98
_callExtensionsInOrder(
  BEFORE_ADD_LIQUIDITY_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
);
```

`DepositAllowlistExtension.beforeAddLiquidity` receives both but names the first parameter `address` (unnamed/discarded) and checks only `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
  }
```

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` argument as `owner` to `_beforeAddLiquidity`. The `owner` argument is only validated to be non-zero in `MetricOmmPoolLiquidityAdder._validateOwner`. The actual payer stored in transient context is always `msg.sender` of `addLiquidityExactShares` (the unauthorized caller `B`), and the callback pulls tokens from that payer:

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol L172-176
if (amount0Delta > 0) {
  pay(token0, payer, msg.sender, amount0Delta);
}
if (amount1Delta > 0) {
  pay(token1, payer, msg.sender, amount1Delta);
}
```

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual swap initiator), confirming the deposit extension's check is the anomaly:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

The wrong value is the extension decision: `allowedDepositor[pool][owner]` is evaluated instead of `allowedDepositor[pool][sender]`, causing the guard to pass for any caller who supplies an allowlisted address as `owner`.

## Impact Explanation
The deposit allowlist is an admin-configured guard intended to restrict which addresses may provide liquidity (KYC/AML compliance, private pool membership, manipulation prevention). Because the guard checks the position recipient rather than the actual depositor, any unprivileged address can add liquidity to a pool they are explicitly excluded from, manipulate the pool's bin liquidity distribution (affecting LP returns and swap prices for existing LPs), and undermine regulatory or operational restrictions the pool admin intended to enforce. This constitutes an admin-boundary break where an unprivileged path bypasses a pool admin access control, a permitted impact category under contest rules.

## Likelihood Explanation
Exploitation requires no special privilege, no flash loan, and no oracle manipulation. Any EOA or contract can call `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` with `owner` set to any known allowlisted address. Allowlisted addresses are discoverable on-chain via `AllowedToDepositSet` events emitted by `setAllowedToDeposit`. The attack is repeatable and stateless.

## Recommendation
Change the `beforeAddLiquidity` check to gate on `sender` (the actual depositor/payer) rather than `owner`, consistent with how `SwapAllowlistExtension` handles `beforeSwap`:

```solidity

  external view override returns (bytes4)

// After (correct):
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
  external view override returns (bytes4)
{
```

## Proof of Concept
1. Pool is deployed with `DepositAllowlistExtension`; only address `A` is allowlisted via `setAllowedToDeposit(pool, A, true)`.
2. Unauthorized address `B` calls `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, A, salt, deltas, max0, max1, extData)`.
3. `_validateOwner(A)` passes (A != address(0)); `_addLiquidity` stores `payer = B` in transient context.
4. Pool calls `_beforeAddLiquidity(LiquidityAdder, A, ...)` → extension evaluates `allowedDepositor[pool][A]` = `true` → **passes**.
5. Pool executes liquidity addition, then calls `metricOmmModifyLiquidityCallback` on the LiquidityAdder.
6. Callback reads `payer = B` from transient storage and pulls tokens from `B` via `pay(token, B, pool, amount)`.
7. LP position is minted for owner `A`; `B` has paid tokens and bypassed the allowlist entirely.
