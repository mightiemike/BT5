### Title
`deser_auto` Silently Drops `max_atom_len` Guard for Classic/Backrefs Blobs — (`File: wheel/src/api.rs`)

---

### Summary

`deser_auto` is a composite Python-facing function that auto-detects the serialization format (serde_2026 vs. classic/backrefs) and dispatches accordingly. It accepts `max_atom_len` and `strict` parameters, but only forwards them to the serde_2026 branch. When the input blob is in classic or backrefs format, both parameters are silently ignored and `node_from_bytes_backrefs` is called with no atom-size cap. An attacker who controls the serialization format of the input blob can bypass the caller's intended `max_atom_len` restriction entirely.

---

### Finding Description

`deser_auto` is defined as:

```rust
#[pyfunction]
#[pyo3(signature = (blob, *, max_atom_len=PY_DEFAULT_MAX_ATOM_LEN, strict=true))]
fn deser_auto(blob: &[u8], max_atom_len: usize, strict: bool) -> PyResult<LazyNode> {
    let mut a = Allocator::new();
    let node = if let Some(body) = blob.strip_prefix(SERDE_2026_MAGIC_PREFIX.as_slice()) {
        deserialize_2026_body_from_stream(&mut a, &mut Cursor::new(body), max_atom_len, strict)
            .map_err(eval_to_py)?
    } else {
        node_from_bytes_backrefs(&mut a, blob).map_err(eval_to_py)?   // ← max_atom_len and strict dropped
    };
    Ok(LazyNode::new(Rc::new(a), node))
}
``` [1](#0-0) 

The serde_2026 branch correctly passes `max_atom_len` and `strict` to `deserialize_2026_body_from_stream`. The classic/backrefs branch calls `node_from_bytes_backrefs` with no such parameters. The function signature implies to callers that `max_atom_len` is always enforced, but it is only enforced for one of the two code paths.

By contrast, `deser_2026` correctly enforces `max_atom_len` on every call:

```rust
fn deser_2026(blob: &[u8], max_atom_len: usize, strict: bool) -> PyResult<LazyNode> {
    ...
    let node = deserialize_2026(&mut a, blob, max_atom_len, strict)...
``` [2](#0-1) 

And `deser_backrefs` never had a `max_atom_len` parameter at all:

```rust
fn deser_backrefs(blob: &[u8]) -> PyResult<LazyNode> {
    let mut a = Allocator::new();
    let node = node_from_bytes_backrefs(&mut a, blob).map_err(eval_to_py)?;
``` [3](#0-2) 

`deser_auto` is the composite that combines both, but it inherits the uncapped behavior of `deser_backrefs` for the non-serde_2026 path while advertising the `max_atom_len` guard of `deser_2026`.

The default cap is `PY_DEFAULT_MAX_ATOM_LEN = 1 << 20` (1 MiB), described explicitly as a "don't OOM the parser" default: [4](#0-3) 

---

### Impact Explanation

A Python caller that uses `deser_auto(blob, max_atom_len=4096)` to bound memory usage receives no protection when the attacker supplies a classic-format blob. The classic CLVM serialization format encodes atom length in the header bytes; a blob claiming a multi-megabyte atom forces the parser to allocate that memory unconditionally. Without `max_atom_len` enforcement on the backrefs path, the allocator is unbounded, and a crafted blob can exhaust process memory. The corrupted invariant is the caller's assumption that `max_atom_len` is always respected — it is not, and the bypass is format-controlled by the attacker.

---

### Likelihood Explanation

Any Python consumer that calls `deser_auto` with a reduced `max_atom_len` (e.g., for mempool or RPC input validation) is affected. The attacker's only requirement is to send a blob that does not start with `SERDE_2026_MAGIC_PREFIX` (`fd ff 32 30 32 36`), which is trivially achievable since classic CLVM blobs never carry this prefix. No special privileges or insider access are required.

---

### Recommendation

Forward `max_atom_len` and `strict` to the backrefs branch as well. Since `node_from_bytes_backrefs` does not currently accept these parameters, either:

1. Add `max_atom_len` support to `node_from_bytes_backrefs` (and the underlying classic parser), or
2. After parsing, walk the resulting tree and reject any atom whose length exceeds `max_atom_len`.

Option 1 is preferred for early rejection. At minimum, document clearly that `max_atom_len` and `strict` are **not** enforced for classic/backrefs blobs, so callers are not misled by the function signature.

---

### Proof of Concept

```python
import clvm_rs

# Attacker crafts a classic-format CLVM blob with a 2 MiB atom.
# Classic encoding: 0xFE <3-byte big-endian length> <data>
atom_size = 2 * 1024 * 1024          # 2 MiB — exceeds max_atom_len below
header = bytes([0b11111110])          # 3-byte length follows
length_bytes = atom_size.to_bytes(3, "big")
payload = b"\xAA" * atom_size
classic_blob = header + length_bytes + payload

# Caller believes max_atom_len=4096 protects them.
# Because the blob is NOT serde_2026, deser_auto silently ignores max_atom_len
# and parses the full 2 MiB atom into the Allocator.
node = clvm_rs.deser_auto(classic_blob, max_atom_len=4096)
# No exception raised; 2 MiB atom is in memory despite the 4096-byte cap.
```

The same call via `deser_2026` with a serde_2026-prefixed blob would correctly raise an error when the atom exceeds `max_atom_len`. The bypass is entirely controlled by the format prefix chosen by the attacker.

fn deser_2026(blob: &[u8], max_atom_len: usize, strict: bool) -> PyResult<LazyNode> {
    let mut a = Allocator::new();
    let node = deserialize_2026(&mut a, blob, max_atom_len, strict).map_err(|e| {
        // Translate the prefix-missing error into a friendlier ValueError.
        if !blob.starts_with(SERDE_2026_MAGIC_PREFIX.as_slice()) {

        }
    })?;
