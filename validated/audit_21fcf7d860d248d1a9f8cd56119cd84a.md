### Title
`u64` Intermediate Overflow in `get_validator_online_ratio` Uptime Numerator/Denominator Before `U256` Widening Causes Epoch-Transition Panic - (`File: chain/epoch-manager/src/validator_stats.rs`)

### Summary

`get_validator_online_ratio` in `chain/epoch-manager/src/validator_stats.rs` computes cross-products of three `u64` validator-stat counters (`expected_blocks`, `expected_chunks`, `expected_endorsements`) entirely in `u64` arithmetic before passing the result to `U256::from(…)`. This is the exact structural analog of the Aloe-II Oracle overflow: the widening cast is applied *after* the multiplication, not before, so the intermediate product can silently overflow the narrower type. Because nearcore enables overflow-panicking in all build modes, the overflow manifests as a node crash during epoch transition rather than silent corruption.

### Finding Description

In the "Validator did all the things" branch and in every two-component branch of `get_validator_online_ratio`, the numerator and denominator are computed as native `u64` expressions before being handed to `U256::from`:

```rust
// chain/epoch-manager/src/validator_stats.rs  lines 84-90
let numer = U256::from(
    produced_blocks * expected_chunks * expected_endorsements   // u64 * u64 * u64
        + produced_chunks * expected_blocks * expected_endorsements
        + produced_endorsements * expected_blocks * expected_chunks,
);
let denom =
    U256::from(3 * expected_chunks * expected_blocks * expected_endorsements); // u64 * u64 * u64
```

`ValidatorStats::produced` and `ValidatorStats::expected` are both `NumBlocks = u64`. The chunk stats are accumulated across all shards before being passed here:

```rust
// chain/epoch-manager/src/lib.rs  lines 452-459
for (_, tracker) in chunk_stats_tracker {
    if let Some(stat) = tracker.get(&(i as u64)) {
        *chunk_stats.expected_mut() += stat.expected();
        chunk_stats.endorsement_stats_mut().expected += stat.endorsement_stats().expected;
    }
}
```

So `expected_chunks` and `expected_endorsements` grow linearly with the number of shards. The denominator expression `3 * expected_chunks * expected_blocks * expected_endorsements` is evaluated entirely in `u64` before the `U256::from` call. With epoch_length = 43 200 and `N` shards assigned to a single validator:

```
denom ≈ 3 × (43200 × N) × 43200 × (43200 × N)
      = 3 × 43200³ × N²
      ≈ 2.42 × 10¹⁴ × N²
```

`u64::MAX ≈ 1.84 × 10¹⁹`, so overflow occurs when `N² > 7.6 × 10⁴`, i.e. `N ≳ 276` shards assigned to one validator. The same threshold applies to the numerator terms.

The two-component branches have the same structural defect (e.g. `U256::from(2 * expected_chunks * expected_blocks)`) but overflow at a higher shard count.

The correct pattern—used everywhere else in the reward path—is to widen each operand individually before multiplying:

```rust
// reward_calculator.rs lines 134-139 (correct pattern)
U512::from(epoch_validator_reward.as_yoctonear())
    * U512::from(uptime_numer)
    * U512::from(stake.as_yoctonear())
```

### Impact Explanation

`get_validator_online_ratio` is called from `calculate_reward` for every validator at every epoch boundary. A panic inside this call aborts the epoch-transition logic. Because nearcore enforces overflow-panicking in all build modes (release included), the overflow does not silently corrupt the reward ratio—it crashes the node. If the crash is reproducible across the validator set (e.g. triggered by a protocol upgrade that increases shard count), it can halt the chain.

### Likelihood Explanation

With current mainnet parameters (~6 shards, 43 200-block epochs, ~100 validators) the maximum per-validator `expected_chunks` is on the order of 2 600, making the triple product ≈ 2.9 × 10¹¹—far below `u64::MAX`. The overflow threshold of ~276 shards is not reachable today. However, NEAR's dynamic resharding roadmap targets hundreds of shards; the bug is latent and will become reachable as resharding progresses. Severity is **Medium** (low current likelihood, high impact when triggered).

### Recommendation

Widen each operand to `U256` before multiplying, mirroring the pattern already used in `reward_calculator.rs`:

```rust
// Validator did all the things
(expected_blocks, expected_chunks, expected_endorsements) => {
    let produced_blocks = stats.block_stats.produced;
    let produced_chunks = stats.chunk_stats.produced();

    let numer = U256::from(produced_blocks) * U256::from(expected_chunks) * U256::from(expected_endorsements)
        + U256::from(produced_chunks) * U256::from(expected_blocks) * U256::from(expected_endorsements)
        + U256::from(produced_endorsements) * U256::from(expected_blocks) * U256::from(expected_chunks);
    let denom = U256::from(3u64)
        * U256::from(expected_chunks)
        * U256::from(expected_blocks)
        * U256::from(expected_endorsements);
    (numer, denom)
}
```

Apply the same fix to every two-component branch (`2 * expected_chunks * expected_blocks`, etc.).

### Proof of Concept

Construct a `BlockChunkValidatorStats` with all three stat categories non-zero and values large enough to overflow `u64` in the triple product:

```rust
// expected_blocks = 43_200, expected_chunks = 4_320_000, expected_endorsements = 4_320_000
// 3 * 4_320_000 * 43_200 * 4_320_000 ≈ 2.42e18  (within u64 — safe today)
// With N=300 shards: 3 * 12_960_000 * 43_200 * 12_960_000 ≈ 2.18e19 > u64::MAX — PANIC
let stats = BlockChunkValidatorStats {
    block_stats: ValidatorStats { produced: 43_200, expected: 43_200 },
    chunk_stats: ChunkStats {
        production:   ValidatorStats { produced: 12_960_000, expected: 12_960_000 },
        endorsement:  ValidatorStats { produced: 12_960_000, expected: 12_960_000 },
    },
};
// This call panics in release builds with overflow checks enabled:
let _ = get_validator_online_ratio(&stats, None);
```
