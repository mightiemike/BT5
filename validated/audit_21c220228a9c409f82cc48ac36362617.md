### Title
Missing Slippage Protection on `burnNlp()` Allows Users to Receive Less Quote Than Expected - (`core/contracts/Clearinghouse.sol`)

---

### Summary

The `burnNlp()` function in `Clearinghouse.sol` burns a user's NLP tokens and credits them quote tokens at a price (`oraclePriceX18`) supplied by the sequencer at execution time. The user-signed `BurnNlp` struct contains no `minQuoteAmount` field, so the user cannot enforce a minimum acceptable return. This is the direct analog of the missing slippage protection on `sellVotes()` reported in M-2.

---

### Finding Description

When a user initiates an NLP burn, they sign a `BurnNlp` struct containing only `sender`, `nlpAmount`, and `nonce`: [1](#0-0) 

The `oraclePriceX18` used to compute the quote payout is **not** part of the signed user payload — it is injected by the sequencer into `SignedBurnNlp` at submission time: [2](#0-1) 

Inside `burnNlp()`, the quote credited to the user is computed directly from this sequencer-supplied price with no floor check: [3](#0-2) 

There is no `require(quoteAmount >= txn.minQuoteAmount, ...)` guard anywhere in `burnNlp()`. The `ERR_SLIPPAGE_TOO_HIGH` error constant is defined in `Errors.sol` but is never referenced in any contract, confirming the slippage check was never implemented: [4](#0-3) 

By contrast, `mintNlp()` is naturally protected because the user signs the exact `quoteAmount` they spend — they know precisely what they pay: [5](#0-4) 

This asymmetry mirrors the `buyVotes()`/`sellVotes()` asymmetry in M-2 exactly.

---

### Impact Explanation

A user burning NLP tokens can receive materially less quote than they expected. The quote output is `nlpAmount × oraclePriceX18 − burnFee`. If `oraclePriceX18` is lower than the price the user observed when signing (due to price movement or sequencer delay), the user silently absorbs the loss with no ability to revert. There is no on-chain mechanism for the user to express "revert if I receive less than X quote."

---

### Likelihood Explanation

NLP oracle prices can move between the time a user signs the `BurnNlp` transaction and the time the sequencer includes it. The sequencer processes transactions asynchronously via the slow-mode queue. Any price decline in that window directly reduces the user's payout. This is a realistic, unprivileged scenario requiring no special access — any NLP holder who calls `burnNlp` is exposed.

---

### Recommendation

Add a `minQuoteAmount` field to the `BurnNlp` struct so it is part of the user's signed payload:

```solidity
struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint64  nonce;
    uint128 minQuoteAmount; // slippage guard
}
```

Then in `burnNlp()`, after computing `quoteAmount`, add:

```solidity
require(quoteAmount >= int128(txn.minQuoteAmount), ERR_SLIPPAGE_TOO_HIGH);
```

This mirrors the existing `ERR_SLIPPAGE_TOO_HIGH` error already defined in `Errors.sol` and makes the protection consistent with the `mintNlp` direction where the user controls the exact quote spent.

---

### Proof of Concept

1. NLP oracle price is `1.00` USDC per NLP at signing time. User signs `BurnNlp { nlpAmount: 1000e18, nonce: N }` expecting ~`999` USDC (after 0.1% burn fee).
2. Before the sequencer processes the transaction, the NLP oracle price drops to `0.80`.
3. Sequencer submits `SignedBurnNlp` with `oraclePriceX18 = 0.80e18`.
4. `burnNlp()` executes: `quoteAmount = 1000e18 × 0.80 = 800 USDC`, fee ≈ `0.8 USDC`, net ≈ `799.2 USDC`.
5. User receives ~`799` USDC instead of ~`999` USDC — a ~20% loss — with no revert path. [6](#0-5)

```text
    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
```
