### Title
`verify_transaction_inclusion` (v1) Accepts Internal Merkle Tree Nodes as Valid Transaction IDs — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) passes the caller-supplied `tx_id` directly as the starting hash into `compute_root_from_merkle_proof` without any leaf/internal-node type distinction. An unprivileged NEAR caller can supply an internal Merkle tree node hash as `tx_id` with a correspondingly shorter sibling proof, causing the contract to reconstruct the correct Merkle root and return `true` — falsely confirming a transaction that was never a leaf in the Bitcoin block.

---

### Finding Description

Bitcoin's Merkle tree uses `double_sha256` for both leaf nodes (txids) and internal nodes (hashes of two child hashes). Both are 32-byte values. There is no type byte or domain separator to distinguish them.

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` takes the caller-supplied `transaction_hash` and uses it verbatim as the starting accumulator:

```rust
// merkle-tools/src/lib.rs:39
let mut current_hash = transaction_hash;   // used raw, no pre-hashing
for proof_hash in merkle_proof {
    if current_position % 2 == 0 {
        current_hash = compute_hash(&current_hash, proof_hash);
    } else {
        current_hash = compute_hash(proof_hash, &current_hash);
    }
    current_position /= 2;
}
``` [1](#0-0) 

`verify_transaction_inclusion` (v1) feeds `args.tx_id` directly into this function with no additional validation beyond a non-empty proof check:

```rust
// contract/src/lib.rs:315-322
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,                                    // raw, caller-controlled
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [2](#0-1) 

Because internal nodes and leaf nodes are indistinguishable 32-byte hashes, an attacker can substitute any internal node for a leaf and provide a shorter sibling path that still reconstructs the root.

**Concrete attack for a block with 4 transactions `[T0, T1, T2, T3]`:**

```
H01  = double_sha256(T0 || T1)   ← internal node
H23  = double_sha256(T2 || T3)   ← internal node
Root = double_sha256(H01 || H23)
```

Attacker submits:
- `tx_id      = H01`  (an internal node, not a real txid)
- `tx_index   = 0`
- `merkle_proof = [H23]`  (length 1 instead of the honest length 2)

`compute_root_from_merkle_proof(H01, 0, [H23])`:
- position 0 is even → `double_sha256(H01 || H23) = Root` ✓

The function returns `true`. The contract has confirmed a transaction that does not exist as a leaf.

The developers acknowledge this in the docstring but rely on an unenforced assumption:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash. We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification." [3](#0-2) 

No such enforcement exists in the contract itself.

---

### Impact Explanation

Any downstream NEAR contract or off-chain application that calls `verify_transaction_inclusion` (v1) and acts on a `true` result — e.g., releasing bridged funds, minting tokens, or unlocking collateral upon "confirmed" Bitcoin payment — can be deceived. The attacker fabricates proof of a Bitcoin transaction that never occurred as a leaf, and the contract certifies it as valid. The corrupted value is the proof result returned by `verify_transaction_inclusion`.

---

### Likelihood Explanation

The attack requires only:
1. Knowledge of any confirmed Bitcoin block's Merkle tree structure (all data is public on-chain).
2. The ability to call `verify_transaction_inclusion` on NEAR — no privileged role is required. The function carries only `#[pause]` (pausable by a manager) and no access-control gate.
3. A block with ≥ 4 transactions (satisfied by virtually every real Bitcoin block). [4](#0-3) 

The function is deprecated since v0.5.0 but remains deployed and callable. Any integrator that has not yet migrated to v2 is exposed.

---

### Recommendation

**Short-term:** Remove or gate `verify_transaction_inclusion` (v1) behind an access-control role so it cannot be called by arbitrary accounts. The `#[deprecated]` attribute is a compiler hint only and does not prevent on-chain invocation.

**Long-term:** In `compute_root_from_merkle_proof`, enforce a minimum proof length equal to `ceil(log2(tree_size))` so that a proof shorter than the tree depth is rejected. Alternatively, adopt a domain-separated leaf hash (e.g., `double_sha256(0x00 || txid)` for leaves vs. `double_sha256(0x01 || left || right)` for internal nodes) so that an internal node hash can never be a valid leaf input. [5](#0-4) 

---

### Proof of Concept

For a real Bitcoin block with transactions `[T0, T1, T2, T3]` (all data publicly available from any block explorer):

```
H01  = double_sha256(T0 || T1)
H23  = double_sha256(T2 || T3)
Root = double_sha256(H01 || H23)   ← stored in the contract's headers_pool
```

NEAR call (pseudocode):
```json
{
  "tx_id":             "<H01 bytes>",
  "tx_block_blockhash":"<block hash already in headers_pool>",
  "tx_index":          0,
  "merkle_proof":      ["<H23 bytes>"],
  "confirmations":     1
}
```

`compute_root_from_merkle_proof(H01, 0, [H23])` computes `double_sha256(H01 || H23) = Root`, which equals `header.block_header.merkle_root`. The function returns `true`. [6](#0-5) [7](#0-6)

}
