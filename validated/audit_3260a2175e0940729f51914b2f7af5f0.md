### Title
Linked Signer Can Permanently Hijack Subaccount Authority via `SignedLinkSigner` — (`core/contracts/EndpointTx.sol`)

---

### Summary

`processTransactionImpl` handles `TransactionType.LinkSigner` with `allowLinkedSigner = true`, meaning the currently-linked signer is accepted as a valid authorizer to overwrite `linkedSigners[sender]`. A linked signer can therefore replace themselves with an attacker-controlled address, permanently seizing signing authority over the subaccount.

---

### Finding Description

In `EndpointTx.sol`, the `LinkSigner` branch of `processTransactionImpl` calls `validateSignedTx` with the fifth argument hardcoded to `true`: [1](#0-0) 

`validateSignedTx` forwards that flag to `validateSignature`, which in turn passes `getLinkedSigner(sender)` as the accepted alternate signer to `Verifier.validateSignature`: [2](#0-1) 

`Verifier.validateSignature` accepts the recovered address if it equals **either** the subaccount owner **or** the linked signer: [3](#0-2) 

After validation passes, the mapping is unconditionally overwritten: [4](#0-3) 

There is no check that the signer of the `LinkSigner` transaction is the subaccount owner (the address in the high 20 bytes of `sender`). A linked signer satisfies the signature check and can set `signedTx.tx.signer` to any arbitrary address.

---

### Impact Explanation

Once `linkedSigners[subaccountA]` is set to `attackerAddr`, the attacker can sign any transaction that accepts a linked signer — including `WithdrawCollateral`, `TransferQuote`, `LiquidateSubaccount`, `MintNlp`, `BurnNlp` — on behalf of `subaccountA`. The original owner loses exclusive control. All assets in the subaccount are at risk of immediate withdrawal or liquidation.

---

### Likelihood Explanation

Any party that has ever been granted linked-signer status (e.g., a trading bot, a delegated key) can execute this unilaterally. No admin access, no sequencer compromise, and no leaked keys are required beyond the linked signer's own key. The attack is a single sequencer-submitted transaction.

---

### Recommendation

Change the `allowLinkedSigner` argument for the `LinkSigner` branch to `false`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may reassign the linked signer
);
```

This ensures only the address encoded in `sender` (the subaccount owner) can authorize a change to `linkedSigners[sender]`, restoring the intended invariant. [5](#0-4) 

---

### Proof of Concept

```
State setup:
  subaccountA = bytes32(abi.encodePacked(ownerAddr, subaccountName))
  linkedSigners[subaccountA] = signerA   // set by ownerAddr previously

Attack:
  1. signerA constructs:
       LinkSigner { sender: subaccountA, signer: bytes32(attackerAddr), nonce: N }
  2. signerA signs the EIP-712 digest for this struct.
  3. Sequencer submits processTransactionImpl(abi.encode(TransactionType.LinkSigner, SignedLinkSigner{tx, sig}))
  4. validateSignedTx passes because recovered == signerA == getLinkedSigner(subaccountA)
  5. linkedSigners[subaccountA] = attackerAddr

Post-state assertion:
  linkedSigners[subaccountA] == attackerAddr  ✓
  attackerAddr signs WithdrawCollateral{sender: subaccountA, ...} → passes validateSignedTx(allowLinkedSigner=true) ✓
  ownerAddr can no longer sign on behalf of subaccountA without first reclaiming the linked signer slot ✓
```

    function validateSignature(
        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
