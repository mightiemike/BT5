### Title
Meta-Transaction Deposit Refund Routed to Sender Instead of the Relayer Who Paid — (File: `runtime/runtime/src/actions.rs`)

---

### Summary

In `apply_delegate_action`, the inner action receipt is constructed with `predecessor_id` set to `sender_id` (Alice, the meta-tx signer), not the relayer. The protocol's deposit-refund rule sends failed-receipt deposits to `predecessor_id`. Consequently, when an inner action fails on the receiver's shard, the deposit the **relayer** paid is refunded to **Alice**, not the relayer. An unprivileged user can deliberately craft a meta-transaction whose inner action will fail after the relayer has already committed the deposit, draining the relayer's balance.

---

### Finding Description

**Step 1 — Inner receipt is stamped with Alice as predecessor, not the relayer.**

In `apply_delegate_action`, the new inner receipt is built as:

```rust
let new_receipt = Receipt::V0(ReceiptV0 {
    predecessor_id: sender_id.clone(),   // ← Alice, NOT the relayer
    receiver_id: delegate_action.receiver_id().clone(),
    receipt: ReceiptEnum::Action(ActionReceipt {
        signer_id: action_receipt.signer_id().clone(), // relayer's signer_id
        ...
    }),
});
``` [1](#0-0) 

The comment immediately below this block acknowledges the invariant break:

> "If something goes wrong, deposit is refunded to the predecessor, this is sender_id/Sender in DelegateAction." [2](#0-1) 

**Step 2 — Deposit refund is sent to `balance_refund_receiver()`, which resolves to `predecessor_id` = Alice.**

When the inner receipt fails on the receiver's shard, `refund_unspent_gas_and_deposits` emits:

```rust
if deposit_refund > Balance::ZERO {
    result.new_receipts.push(Receipt::new_balance_refund(
        receipt.balance_refund_receiver(),   // ← resolves to predecessor_id = Alice
        deposit_refund,
    ));
}
``` [3](#0-2) 

`balance_refund_receiver` is defined as:

```rust
pub fn balance_refund_receiver(&self) -> &AccountId {
    self.refund_to().as_ref().unwrap_or_else(|| self.predecessor_id())
}
``` [4](#0-3) 

Because the inner receipt carries no `refund_to` override (it is a plain `ReceiptEnum::Action`, not `ActionV2`), the fallback is always `predecessor_id` — Alice. [5](#0-4) 

**Step 3 — The relayer paid the deposit, not Alice.**

The outer transaction is signed and submitted by the relayer. All gas and attached deposits are deducted from the relayer at `verify_and_charge_transaction` time. The spec confirms this:

> "All gas and balance costs for submitting `delegate_action.actions` are subtracted from the relayer." [6](#0-5) 

The architecture doc explicitly names the resulting invariant break:

> "If an inner action requires an attached balance … this balance is taken from the relayer … But what if the transaction fails execution on Bob's shard? At this point, the predecessor is `Alice` and therefore she receives the token balance refunded, not the relayer." [7](#0-6) 

---

### Impact Explanation

The relayer pays the deposit; Alice receives it back on failure. Alice can deliberately engineer a failure after the relayer has already committed:

- Alice holds 1 000 FT and crafts a `DelegateAction` with a 10 NEAR deposit attached to `ft_transfer`.
- Relayer verifies Alice's FT balance off-chain and submits, paying 10 NEAR.
- Alice front-runs the execution by transferring her FT to another account.
- The inner `ft_transfer` fails (insufficient balance).
- The 10 NEAR deposit refund is sent to Alice, not the relayer.

The relayer loses 10 NEAR; Alice gains 10 NEAR. This is repeatable with no protocol-level rate limit. The financial loss scales linearly with the deposit size Alice can convince the relayer to attach.

---

### Likelihood Explanation

- Any unprivileged user who can submit a `SignedDelegateAction` to a relayer can trigger this.
- The race condition (state change between relayer verification and on-chain execution) is inherent to the asynchronous receipt model and cannot be closed by the relayer alone.
- The protocol version gate for `refund_to` on `ActionV2` receipts exists but is not applied to the inner receipt generated by `apply_delegate_action`.

---

### Recommendation

When `apply_delegate_action` constructs the inner receipt, it should either:

1. Set `predecessor_id` to the relayer's account (the outer receipt's `predecessor_id`) so that deposit refunds flow back to the party that paid, **or**
2. Populate the `refund_to` field of an `ActionV2` inner receipt with the relayer's account ID, overriding the default `predecessor_id`-based refund routing.

The `set_refund_to` API already exists on `ReceiptManager` and is used in other contexts: [8](#0-7)

### Proof of Concept

```
Relayer (alice.near) submits:
  Transaction {
    signer: alice.near,
    receiver: bob.near,          // DelegateAction target = bob.near
    actions: [
      Delegate {
        sender_id: bob.near,
        receiver_id: ft.near,
        actions: [
          FunctionCall {
            method: "ft_transfer",
            deposit: 10_000_000_000_000_000_000_000_000,  // 10 NEAR
          }
        ]
      }
    ]
  }

Inner receipt created by apply_delegate_action:
  Receipt {
    predecessor_id: bob.near,    // ← Alice's account, NOT alice.near (relayer)
    receiver_id: ft.near,
    signer_id: alice.near,       // relayer
    actions: [FunctionCall { deposit: 10 NEAR }]
  }

bob.near drains FT balance before execution → ft_transfer panics.

Deposit refund receipt:
  Receipt::new_balance_refund(
    receiver_id: bob.near,       // ← predecessor_id of failed inner receipt
    deposit: 10 NEAR
  )

Result: alice.near (relayer) loses 10 NEAR; bob.near gains 10 NEAR.
```

    pub fn refund_to(&self) -> &Option<AccountId> {
        match self.receipt() {
            ReceiptEnum::Action(_)
            | ReceiptEnum::Data(_)
            | ReceiptEnum::PromiseYield(_)
            | ReceiptEnum::PromiseResume(_)
            | ReceiptEnum::GlobalContractDistribution(_) => &None,
            ReceiptEnum::ActionV2(action_receipt_v2)
            | ReceiptEnum::PromiseYieldV2(action_receipt_v2) => &action_receipt_v2.refund_to,
        }
    }
