### Title
Protobuf `ValidResourceBounds` Reconstruction Heuristic Rejects Valid V3 Transactions with Zero L2/Data Gas Bounds via P2P - (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion uses a **value-based heuristic** to decide between `L1Gas` and `AllResources` variants. When a valid V3 transaction carries `AllResourceBounds` with `l2_gas = 0` and `l1_data_gas = 0`, the heuristic misclassifies it as `ValidResourceBounds::L1Gas`. The downstream conversion `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` then hard-rejects the `L1Gas` variant, causing the transaction to be dropped by every receiving P2P peer. The same misclassification produces a **different transaction hash** in `get_tip_resource_bounds_hash`, breaking the canonicalization invariant that a transaction's hash must be stable across serialization boundaries.

### Finding Description

**Step 1 — Gateway accepts the transaction with `AllResources`.**

`RpcInvokeTransactionV3` stores `resource_bounds: AllResourceBounds` (never `L1Gas`). A user may legitimately submit a V3 invoke with `l2_gas = {0, 0}` and `l1_data_gas = {0, 0}` (e.g., a pure L1-gas-only V3 transaction). The gateway accepts it and computes `tx_hash` via `get_tip_resource_bounds_hash` with the `AllResources` branch, which hashes **four** elements: `[tip, l1_gas_packed, l2_gas_packed_zero, l1_data_gas_packed_zero]`. [1](#0-0) [2](#0-1) 

**Step 2 — Serialization to protobuf.**

When the transaction is propagated via P2P (mempool or consensus), `From<ValidResourceBounds> for protobuf::ResourceBounds` serializes both `l2_gas` and `l1_data_gas` as zero-valued `ResourceLimits` messages. [3](#0-2) 

**Step 3 — Heuristic misclassification on deserialization.**

On the receiving peer, `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` applies:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)   // ← wrong branch for a post-0.13.3 AllResources tx
} else {
    ValidResourceBounds::AllResources(...)
})
```

Because both fields are zero, the heuristic produces `ValidResourceBounds::L1Gas`, even though the original transaction was `AllResources`. [3](#0-2) 

This heuristic is invoked inside `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3`: [4](#0-3) 

**Step 4 — Hard rejection of `L1Gas` variant.**

`TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3` calls `snapi_invoke.try_into()`, which invokes `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3`. That conversion explicitly rejects the `L1Gas` variant:

```rust
resource_bounds: match value.resource_bounds {
    ValidResourceBounds::AllResources(bounds) => bounds,
    _ => { return Err(StarknetApiError::OutOfRange { ... }); }
},
```

The error is mapped to `DEPRECATED_RESOURCE_BOUNDS_ERROR` and the transaction is dropped. [5](#0-4) [6](#0-5) 

**Step 5 — Hash canonicalization break.**

`get_tip_resource_bounds_hash` produces **structurally different** outputs for the two variants even when the numeric values are identical:

- `L1Gas(l1_gas)` → `poseidon([tip, l1_gas_packed, l2_zero_packed])` — **3 elements**
- `AllResources { l2_gas: ZERO, l1_data_gas: ZERO }` → `poseidon([tip, l1_gas_packed, l2_zero_packed, l1_data_zero_packed])` — **4 elements**

`poseidon([a, b, c]) ≠ poseidon([a, b, c, 0])` in general, so the hash computed at the gateway and the hash that would be computed after P2P deserialization diverge. The `tx_hash` stored in `InternalRpcTransaction` at admission time is therefore not reproducible from the wire representation. [1](#0-0) 

A parallel, correct `TryFrom<protobuf::ResourceBounds> for AllResourceBounds` exists in `rpc_transaction.rs` and does **not** apply the heuristic — it requires all three fields to be present. The bug is that the `InvokeV3` → `InvokeTransactionV3` path uses the wrong converter. [7](#0-6) 

### Impact Explanation

**High — Mempool/gateway/RPC admission rejects valid transactions before sequencing.**

Any V3 invoke, declare, or deploy-account transaction with `l2_gas.max_amount = 0`, `l2_gas.max_price_per_unit = 0`, `l1_data_gas.max_amount = 0`, `l1_data_gas.max_price_per_unit = 0` is accepted by the gateway but silently dropped by every P2P peer that receives it. In the consensus path, a proposer that includes such a transaction in a block proposal will have that proposal rejected by all validators, stalling block production for that round.

### Likelihood Explanation

**Low.** The trigger requires a user to submit a V3 transaction with all-zero L2 and data gas bounds. This is an unusual but valid configuration (e.g., a user who only cares about L1 gas and sets the other bounds to zero). No privileged access is required; any unprivileged account can submit such a transaction via the public RPC endpoint.

### Recommendation

Replace the value-based heuristic with a **presence-based** check. The `l1_data_gas` field in the protobuf `ResourceBounds` message is declared `optional`; its absence (not its zero value) is the correct signal for a pre-0.13.3 `L1Gas` transaction:

```rust
// In TryFrom<protobuf::ResourceBounds> for ValidResourceBounds:
let l1_data_gas_opt = value.l1_data_gas; // keep as Option before unwrap_or_default
let l1_data_gas: ResourceBounds = l1_data_gas_opt.unwrap_or_default().try_into()?;
Ok(if l1_data_gas_opt.is_none() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

Alternatively, the `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3` conversion should use `AllResourceBounds::try_from(resource_bounds_proto)?` directly and wrap the result in `ValidResourceBounds::AllResources`, bypassing the heuristic entirely for transaction types that are always post-0.13.3. [3](#0-2) [8](#0-7) 

### Proof of Concept

1. Construct a V3 invoke transaction with:
   ```
   resource_bounds = AllResourceBounds {
       l1_gas:      { max_amount: 1000, max_price_per_unit: 1 },
       l2_gas:      { max_amount: 0,    max_price_per_unit: 0 },
       l1_data_gas: { max_amount: 0,    max_price_per_unit: 0 },
   }
   ```
2. Submit it to the gateway via `starknet_addInvokeTransaction`. The gateway accepts it and stores `tx_hash_A = poseidon([tip, l1_gas_packed, 0, 0])` (4-element hash, `AllResources` branch).
3. The mempool propagates the transaction as a `protobuf::MempoolTransaction` with `InvokeV3WithProof` carrying `resource_bounds.l2_gas = {0, 0}` and `resource_bounds.l1_data_gas = {0, 0}`.
4. A receiving peer deserializes: `protobuf::InvokeV3` → `InvokeTransactionV3` with `resource_bounds = ValidResourceBounds::L1Gas(l1_gas)` (heuristic fires because both are zero).
5. `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` returns `Err(DEPRECATED_RESOURCE_BOUNDS_ERROR)`.
6. The transaction is dropped. The receiving peer never adds it to its mempool.
7. If the proposer includes this transaction in a consensus proposal, all validators reject the proposal at the `convert_consensus_tx_to_internal_consensus_tx` step, causing a consensus round failure. [9](#0-8) [10](#0-9)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }

impl TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3 {
    type Error = ProtobufConversionError;
    fn try_from(mut value: protobuf::InvokeV3WithProof) -> Result<Self, Self::Error> {
        // Extract proof first, since `starknet_api::transaction::InvokeTransactionV3` does not
        // carry a `proof` field.
        let proof = Proof::from(std::mem::take(&mut value.proof));

        let snapi_invoke: InvokeTransactionV3 = value
            .invoke
            .ok_or(ProtobufConversionError::MissingField {
                field_description: "InvokeV3WithProof::invoke",
            })?
            .try_into()?;

        // This conversion can fail only if the resource_bounds are not AllResources.
        Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
    }
