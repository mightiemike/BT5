### Title
`get_tip_resource_bounds_hash` Produces Divergent Hash Preimage for V3 Transactions with Zero L2/Data-Gas Bounds — (`crates/starknet_api/src/transaction_hash.rs`)

### Summary

`get_tip_resource_bounds_hash` encodes a different number of resource-bound felts depending on whether `ValidResourceBounds` is the `L1Gas` variant (2 felts: L1_GAS, L2_GAS) or the `AllResources` variant (3 felts: L1_GAS, L2_GAS, L1_DATA_GAS). The RPC/internal submission path always wraps resource bounds as `AllResources` (3-felt hash), while the P2P historical-sync protobuf converter (`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`) silently downgrades to `L1Gas` (2-felt hash) whenever `l2_gas` and `l1_data_gas` are both zero. A valid V3 invoke transaction with only `l1_gas` set therefore receives a different transaction hash depending on which code path processes it, breaking the canonical hash invariant.

### Finding Description

**Step 1 — Hash function branches on variant tag, not on field values.**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` always emits two felts for `L1Gas` and three felts for `AllResources`, even when the extra `l1_data_gas` felt is zero:

```rust
// L1 and L2 gas bounds always exist.
let mut resource_felts = vec![
    get_concat_resource(&l1_resource_bounds, L1_GAS)?,
    get_concat_resource(&l2_resource_bounds, L2_GAS)?,
];
// For new V3 txs, need to also hash the data gas bounds.
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 felts
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 felts
    }
});
``` [1](#0-0) 

**Step 2 — Submission path always produces a 3-felt hash.**

`InternalRpcInvokeTransactionV3` stores `resource_bounds: AllResourceBounds` and its `InvokeTransactionV3Trait` implementation unconditionally wraps it as `ValidResourceBounds::AllResources`:

```rust
impl InvokeTransactionV3Trait for InternalRpcInvokeTransactionV3 {
    fn resource_bounds(&self) -> ValidResourceBounds {
        ValidResourceBounds::AllResources(self.resource_bounds)   // always AllResources
    }
``` [2](#0-1) 

The gateway accepts V3 transactions with only `l1_gas` non-zero (zero `l2_gas` and `l1_data_gas`), as confirmed by the stateless validator test cases: [3](#0-2) 

The hash is then computed via `InternalRpcInvokeTransactionV3::calculate_transaction_hash` → `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash(AllResources(...))` → **3-felt hash H₁**. [4](#0-3) 

**Step 3 — P2P historical-sync path silently downgrades to `L1Gas`, producing a 2-felt hash.**

The protobuf converter for historical transactions (`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`) in `crates/apollo_protobuf/src/converters/transaction.rs` applies a zero-check and emits `L1Gas` when both `l2_gas` and `l1_data_gas` are zero:

```rust
let l1_data_gas = value.l1_data_gas.unwrap_or_default();   // silently defaults to zero
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)                      // 2-felt hash path
} else {
    ValidResourceBounds::AllResources(...)
})
``` [5](#0-4) 

When the syncing node deserializes the same transaction as `InvokeTransactionV3` with `ValidResourceBounds::L1Gas(X)`, `get_tip_resource_bounds_hash` emits only 2 felts → **hash H₂ ≠ H₁**.

`InvokeTransactionV3`'s trait implementation returns `self.resource_bounds` directly (not forced to `AllResources`), so the variant tag drives the hash branch: [6](#0-5) 

**Step 4 — `validate_transaction_hash` will reject the block or store a wrong hash.**

`validate_transaction_hash` recomputes the hash from the deserialized `Transaction` and compares it to the expected hash. For a V3 transaction with zero `l2_gas`/`l1_data_gas`, the recomputed hash (H₂, 2-felt) will not match the stored hash (H₁, 3-felt), causing the syncing node to reject the transaction or the containing block. [7](#0-6) 

The analog to the Sherlock `bps()` bug is exact: the submission path reads `resource_bounds` from one representation (`AllResources`, 3 felts) while the sync path reads the same bytes from a different representation (`L1Gas`, 2 felts), causing the system to compute on one hash but commit/validate against another.

### Impact Explanation

A valid V3 invoke transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is accepted by the gateway and sequenced into a block with hash H₁. When a peer syncs the block via P2P, it deserializes the transaction as `ValidResourceBounds::L1Gas(X)` and recomputes hash H₂ ≠ H₁. The peer either rejects the block (causing a consensus/sync split) or stores the transaction under the wrong hash (causing RPC to serve an authoritative-looking wrong transaction hash). This matches the impact scope: **Wrong state, receipt, or revert result from blockifier/syscall/execution logic for accepted input** and **RPC returns an authoritative-looking wrong value**.

### Likelihood Explanation

The gateway explicitly allows V3 transactions with only `l1_gas` non-zero (zero `l2_gas` and `l1_data_gas`). Any user who submits such a transaction — intentionally or not — triggers the divergence. The protobuf downgrade is unconditional and requires no special privilege.

### Recommendation

1. **Fix the protobuf converter**: In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, do not downgrade to `L1Gas` for V3 transactions. Either require `l1_data_gas` to be present (removing the `unwrap_or_default`), or always produce `AllResources` for V3 transactions regardless of zero values.

2. **Fix `get_tip_resource_bounds_hash`**: Always include `l1_data_gas` in the hash for V3 transactions, even when it is zero, so the hash is determined by field values rather than the Rust enum variant tag.

3. **Add a cross-path hash consistency test**: Verify that `InternalRpcInvokeTransactionV3` and `InvokeTransactionV3` (deserialized from protobuf) produce identical hashes for the same transaction data.

### Proof of Concept

```rust
use starknet_api::transaction::fields::{
    AllResourceBounds, ResourceBounds, ValidResourceBounds,
};
use starknet_api::transaction_hash::get_tip_resource_bounds_hash;
use starknet_api::block::GasPrice;
use starknet_api::execution_resources::GasAmount;
use starknet_api::transaction::fields::Tip;

let l1_gas = ResourceBounds { max_amount: GasAmount(100), max_price_per_unit: GasPrice(1) };
let zero   = ResourceBounds::default();

// Submission path: InternalRpcInvokeTransactionV3 always uses AllResources
let all_resources = ValidResourceBounds::AllResources(AllResourceBounds {
    l1_gas,
    l2_gas: zero,
    l1_data_gas: zero,
});

// P2P sync path: TryFrom<protobuf::ResourceBounds> downgrades to L1Gas
let l1_gas_only = ValidResourceBounds::L1Gas(l1_gas);

let tip = Tip(0);
let h1 = get_tip_resource_bounds_hash(&all_resources, &tip).unwrap();
let h2 = get_tip_resource_bounds_hash(&l1_gas_only,   &tip).unwrap();

// h1 ≠ h2 — same transaction data, different hash preimage
assert_ne!(h1, h2, "Hash divergence: AllResources(zero l2/data) ≠ L1Gas");
```

The `AllResources` path hashes three felts `[L1_GAS_concat, L2_GAS_concat(0), L1_DATA_GAS_concat(0)]`; the `L1Gas` path hashes only two felts `[L1_GAS_concat, L2_GAS_concat(0)]`. The Poseidon hash of these two sequences is different, so H₁ ≠ H₂ for any non-trivial `l1_gas` value.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
