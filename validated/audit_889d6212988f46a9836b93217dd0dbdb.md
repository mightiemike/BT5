### Title
Zero `builderFeeRate` Bypasses Builder Fee While Still Granting Fee Tier Elevation — (`core/contracts/OffchainExchange.sol`)

### Summary

When a builder is configured with `lowestFeeRate = 0`, a user can craft an order appendix with `builderId != 0` and `builderFeeRate = 0`. The validation guard passes (`0 < 0` is false), the builder's `defaultFeeTier` is applied to the user's fee tier via the `max` operation, but zero builder fee is collected. The user obtains a privileged fee tier without compensating the builder, and if that tier carries a negative maker rate (rebate), funds are extracted from the protocol fee pool.

---

### Finding Description

**Step 1 — Appendix parsing (`_builderInfo`)** [1](#0-0) 

`builderFeeRate` is decoded from bits 38–47 of the appendix. When those bits are all zero, `builderFeeRate = 0 * 10^13 = 0`. The user controls the appendix (it is part of the EIP-712 signed order).

**Step 2 — Validation in `getUserFeeRateWithBuilder`** [2](#0-1) 

The guard rejects a rate only when `builderFeeRate < builder.lowestFeeRate`. When `builder.lowestFeeRate = 0` and `builderFeeRate = 0`, the condition evaluates to `0 < 0` → **false**. The builder is accepted as valid.

**Step 3 — Unconditional tier elevation** [3](#0-2) 

The `max` operation elevates the user's `feeTier` to `builder.defaultFeeTier` regardless of whether `builderFeeRate` is zero. There is no guard that skips tier elevation when the builder fee rate is zero.

**Step 4 — Zero builder fee computed and not collected** [4](#0-3) 

`orderInfo.builderFee = matchQuote.abs().mul(0) = 0`. The `if (orderInfo.builderFee > 0)` branch is never entered, so `collectedBuilderFee` is not incremented and the builder receives nothing.

**Step 5 — `updateBuilder` has no floor on `lowestFeeRate`** [5](#0-4) 

No validation prevents setting `lowestFeeRate = 0` when registering a builder. The configuration is fully reachable.

---

### Impact Explanation

A user who references such a builder obtains the fee rates of `builder.defaultFeeTier` without paying any builder fee. If that tier is configured with a negative `makerRateX18` (a rebate), `keepRateX18 = ONE - feeRate > ONE`, causing `newMeteredQuote > meteredQuote` and `orderInfo.fee < 0` — the protocol pays the user a rebate from `market.collectedFees`. This constitutes direct extraction of funds from the protocol fee pool. Even without rebates, users systematically underpay taker fees relative to their true tier, reducing protocol revenue. [6](#0-5) 

---

### Likelihood Explanation

- The attacker controls the appendix bits (user-signed order); no privileged access is needed.
- `lowestFeeRate = 0` is a valid and unguarded builder configuration.
- The path is reachable through the normal `matchOrders` → `applyFee` → `getUserFeeRateWithBuilder` flow, which is the standard production trade execution path.
- The only prerequisite is that at least one registered builder has `lowestFeeRate = 0` and `defaultFeeTier > 0`.

---

### Recommendation

Add an explicit guard in `getUserFeeRateWithBuilder` that skips tier elevation when `builderFeeRate == 0`:

```solidity
if (builderId != 0 && builderFeeRate > 0 && feeTier < builder.defaultFeeTier) {
    feeTier = builder.defaultFeeTier;
}
```

Alternatively, enforce `lowestFeeRate > 0` in `updateBuilder` to prevent builders from being registered with a zero floor rate.

### Proof of Concept

1. Register builder ID 1 with `owner = attacker`, `defaultFeeTier = 10`, `lowestFeeRate = 0`, `highestFeeRate = ONE`.
2. Configure tier 10 with `takerRateX18 = 0` (or a negative `makerRateX18`).
3. Craft an order appendix with `builderId = 1` and the 10 builder-fee-rate bits all zero (`builderFeeRate = 0`).
4. Submit the order through the sequencer via `matchOrders`.
5. Assert: `feeTier` applied equals 10, `collectedBuilderFee[quoteId][1]` remains 0, and the user's `quoteDelta` reflects tier-10 rates rather than their actual tier-0 rates.
