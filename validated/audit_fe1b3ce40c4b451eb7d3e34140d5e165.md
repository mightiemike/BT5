### Title
Missing Median-Time-Past Timestamp Validation Allows Relayer to Submit Consensus-Invalid Headers and Corrupt Difficulty Retargeting — (File: `contract/src/utils.rs`, `contract/src/lib.rs`)

---

### Summary

`get_median_time_past` is defined in `contract/src/utils.rs` but is never called in the production code path — it is suppressed with `#[allow(unused)]`. The `submit_block_header` function validates PoW but never checks that a submitted header's `time` field exceeds the Median Time Past (MTP) of the previous 11 blocks, violating Bitcoin's consensus rule. A malicious trusted relayer can submit headers with manipulated timestamps to corrupt the difficulty retargeting calculation, enabling construction of a lower-difficulty fake chain that `verify_transaction_inclusion` will accept as canonical.

---

### Finding Description

**Root cause — dead MTP function:**

`get_median_time_past` is defined in the private `utils` module: [1](#0-0) 

The `#[allow(unused)]` attribute on a `pub fn` inside a private module (`mod utils;`) suppresses the Rust `dead_code` lint, which fires precisely because the function is never called anywhere in the crate. This is the direct evidence that MTP validation is absent from the production path.

**Call chain that skips the check:**

`submit_blocks` iterates headers and calls `submit_block_header`: [2](#0-1) 

`submit_block_header` (non-Dogecoin path) calls `check_target` when PoW verification is enabled: [3](#0-2) 

`check_target` only delegates to the chain-specific `check_pow`: [4](#0-3) 

At no point between `submit_blocks` and `check_target` is `get_median_time_past` invoked. The `time` field of the submitted header is stored verbatim into `ExtendedHeader` and used in difficulty retargeting without any lower-bound (MTP) or upper-bound (network time + 2 h) validation.

**What Bitcoin consensus requires but the contract omits:**

Bitcoin's consensus rule mandates `header.time > MTP(prev 11 blocks)`. The contract stores the `time` field directly: [5](#0-4) 

The `block_height` is derived from the previous block's height + 1, and `chain_work` is accumulated from `bits`. Both are correct. But `time` is accepted without any consensus-level timestamp check.

**Difficulty retargeting impact:**

Bitcoin adjusts difficulty every 2016 blocks using `(time[2016n] − time[2016(n−1)])`. A relayer who controls the `time` fields of submitted headers can make this span appear longer than reality (e.g., 4 weeks instead of 2 weeks), causing the protocol to halve the required difficulty at the next retargeting boundary. Subsequent headers then require only half the real-world PoW to satisfy `U256::from_le_bytes(&pow_hash.0) <= target_from_bits(header.bits)`: [6](#0-5) 

With a sufficiently reduced difficulty, the relayer can build a fake chain whose accumulated `chain_work` exceeds the honest chain, triggering a reorg: [7](#0-6) 

Once the fake chain is canonical, `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` will return `true` for transactions that exist only in the fake chain: [8](#0-7) 

---

### Impact Explanation

Any downstream NEAR dApp that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` to gate asset releases or state transitions based on Bitcoin transaction confirmation will accept fabricated Bitcoin transactions as proven. This is a direct integrity failure in the cross-chain trust model: the light client's canonical chain and its SPV proofs can be made to attest to events that never occurred on the real Bitcoin network.

---

### Likelihood Explanation

The attacker must be a trusted relayer (gated by `#[trusted_relayer]`): [9](#0-8) 

If the trusted-relayer role is open to any staker (permissionless bonding), the precondition is reachable without privileged keys. The computational cost of the attack scales with the current chain difficulty: on Bitcoin mainnet the cost is prohibitive; on Litecoin, Dogecoin, or Zcash (all supported by the same contract via feature flags) the difficulty is orders of magnitude lower, making the attack realistic. The `skip_pow_verification = true` path (used in testing but potentially misconfigured in deployment) eliminates the PoW cost entirely.

---

### Recommendation

1. **Enforce MTP lower bound**: After computing `prev_block_header` in `submit_block_header`, call `get_median_time_past` and require `header.time > mtp`. Remove the `#[allow(unused)]` suppression once the call is added.
2. **Enforce upper bound**: Require `header.time < env::block_timestamp() / 1_000_000_000 + 7200` (NEAR timestamp is in nanoseconds; 7200 s = 2 h).
3. **Apply to all chain variants**: The same gap exists in the Dogecoin path (`submit_block_header` with `AuxData`) and any other chain-specific `check_pow` implementations.

---

### Proof of Concept

1. Register as a trusted relayer and stake the required bond.
2. Construct 2016 valid Bitcoin headers (satisfying current PoW) but set each header's `time` field to be spaced 20 minutes apart instead of the real ~10 minutes — making the apparent elapsed time for the epoch 4 weeks instead of 2 weeks.
3. Call `submit_blocks` with these 2016 headers. The contract accepts them because `check_target` only verifies PoW, not timestamps.
4. At the retargeting boundary the contract computes `new_target = old_target * (actual_span / expected_span)` using the manipulated timestamps, halving the required difficulty.
5. Mine the next 2016 headers at the halved difficulty (half the real-world cost). Repeat to further reduce difficulty.
6. After enough epochs, the fake chain's `chain_work` exceeds the honest chain's, triggering `reorg_chain` and replacing the canonical chain.
7. Call `verify_transaction_inclusion` with a Merkle proof for a fabricated transaction in one of the fake blocks. The function returns `true`.
