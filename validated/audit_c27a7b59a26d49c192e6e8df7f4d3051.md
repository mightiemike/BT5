Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` gates on the router address instead of the end user, allowing complete allowlist bypass via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `sender` is the value passed from `MetricOmmPool.swap` as `msg.sender` — which is the router contract, not the actual end user. When `MetricOmmSimpleRouter` is allowlisted (the natural periphery setup), every user — including those explicitly excluded — can bypass the per-user swap allowlist by calling `exactInputSingle` through the router. The allowlist is rendered completely ineffective.

## Finding Description
**Root cause:** `MetricOmmPool.swap` passes `msg.sender` (the immediate caller) as `sender` to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // <-- this is the router, not the end user
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to every configured extension via `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))`.

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

Here `msg.sender` is the pool and `sender` is the router — the actual end user is never checked.

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly with no originator forwarding:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
  );
```

**Exploit path:**
1. Pool admin deploys pool with `SwapAllowlistExtension` to restrict trading to KYC'd addresses.
2. Pool admin allowlists the router: `swapExtension.setAllowedToSwap(pool, router, true)` — required for any periphery routing to work.
3. Attacker (not individually allowlisted) calls `router.exactInputSingle(...)`.
4. Router calls `pool.swap(...)` with `msg.sender = router`.
5. Pool calls `_beforeSwap(router, ...)`.
6. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
7. Attacker receives output tokens; their address was never checked.

**Why existing guards fail:** The check `allowedSwapper[msg.sender][sender]` is structurally correct in form but wrong in substance — `sender` is the intermediate router contract, not the economically responsible actor. There is no mechanism in the current call path to propagate the original `msg.sender` from the router to the extension.

**Contrast with `DepositAllowlistExtension`:** `beforeAddLiquidity` correctly gates on `owner` (the position owner explicitly passed by the caller), not on `sender`. This works because `addLiquidity` takes an explicit `owner` parameter that the router can set to the actual user. No equivalent parameter exists in the swap path.

## Impact Explanation
A curated pool deploying `SwapAllowlistExtension` to restrict trading to approved addresses (KYC, whitelist, institutional) loses that restriction entirely for any user who routes through the public `MetricOmmSimpleRouter`. An unauthorized user executes swaps against the pool's LP liquidity at oracle-anchored prices. This is a direct broken-access-control impact on LP assets and pool curation policy — the allowlist invariant is completely violated. Severity: High (broken core pool access control causing unauthorized trade against LP assets).

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary public swap entrypoint. Any user who calls `exactInputSingle` triggers the bypass — no special privileges, flash loans, or unusual token behavior required. The router must be allowlisted for the pool to support periphery routing at all, making the precondition the natural operational state. The bypass is triggered by a standard router call.

## Recommendation
Pass the original end-user address through the swap path so the extension can gate on the economically relevant actor:

1. **Preferred — encode originator in `extensionData`:** Have the router encode `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check it. This keeps the core pool generic and requires no interface changes.
2. **Alternative — add an `originator` parameter to `pool.swap`:** The pool accepts an explicit originator address (validated against `msg.sender` or a trusted router list) and passes it as `sender` to extensions. The router fills this with its own `msg.sender`.

Either way, `SwapAllowlistExtension.beforeSwap` must check the address that is economically responsible for the swap, not the intermediate contract that called the pool.

// Pool admin allowlists the router (standard periphery setup).
// Pool admin does NOT allowlist `attacker`.

        deadline: block.timestamp,
