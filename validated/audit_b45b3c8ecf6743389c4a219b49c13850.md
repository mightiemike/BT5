### Title
Sanctioned-After-Registration Users' Allocations Permanently Locked With No Recovery Path — (`core/contracts/Airdrop.sol`)

---

### Summary

`Airdrop.sol` performs the sanctions check at claim time, not at merkle root registration time. If a user is added to the sanctions list after `registerMerkleRoot` commits their allocation but before they call `claim`, their tokens are permanently locked in the contract. The contract contains no owner withdrawal, token recovery, or administrative rescue function, so the allocated tokens can never be moved.

---

### Finding Description

`_verifyProof` enforces the sanctions check unconditionally on every call: [1](#0-0) 

If `isSanctioned(sender)` returns `true`, the function reverts with `"address is sanctioned."` The `claimed[week][sender]` mapping is only written on a successful path: [2](#0-1) 

So for a sanctioned user it remains `0` forever — the user can never retry successfully, and no other code path sets it. The only token egress in the entire contract is the `safeTransfer` inside `_claim`: [3](#0-2) 

The contract inherits `OwnableUpgradeable` and exposes `registerMerkleRoot` as the sole owner function: [4](#0-3) 

There is no `withdraw`, `recover`, `rescue`, or `sweep` function anywhere in `Airdrop.sol`. The grep across all production contracts confirms zero such functions exist in this file. Tokens deposited to cover a sanctioned user's allocation have no exit path.

---

### Impact Explanation

Every token pre-funded to the contract to cover a sanctioned user's weekly allocation is permanently irrecoverable. The asset delta is exact: `totalAmount` tokens per affected `(week, address)` pair are destroyed. This directly breaks the invariant that all tokens held by the Airdrop contract are either claimable by their rightful recipient or recoverable by the protocol.

---

### Likelihood Explanation

Sanctions lists (e.g., Chainalysis OFAC) are updated continuously. Any user included in a weekly merkle snapshot who is subsequently sanctioned before claiming — a realistic, non-negligible window — triggers this condition. The owner has no on-chain tool to mitigate it without a contract upgrade.

---

### Recommendation

Add an owner-only token recovery function scoped to unclaimed allocations, for example:

```solidity
function recoverUnclaimed(uint32 week, address user, address to) external onlyOwner {
    require(ISanctionsList(sanctions).isSanctioned(user), "user not sanctioned");
    uint256 amount = /* totalAmount from off-chain source or a stored mapping */;
    require(claimed[week][user] == 0, "already claimed");
    claimed[week][user] = amount; // mark as resolved
    SafeERC20.safeTransfer(IERC20(token), to, amount);
}
```

Alternatively, store `totalAmount` per `(week, address)` at registration time so the contract can compute the recoverable amount on-chain.

---

### Proof of Concept

```solidity
// 1. Owner registers merkle root with user Alice included for week 1, totalAmount = 1000e18
airdrop.registerMerkleRoot(1, root);

// 2. Alice is added to the Chainalysis sanctions list (external, no contract change needed)
// sanctionsList.addToSanctions(alice);

// 3. Alice attempts to claim
vm.prank(alice);
vm.expectRevert("address is sanctioned.");
airdrop.claim([ClaimProof({week: 1, totalAmount: 1000e18, proof: aliceProof})]);

// 4. claimed[1][alice] == 0 — tokens never leave the contract
assertEq(airdrop.getClaimed(alice)[1], 0);

// 5. No owner function exists to recover the 1000e18 tokens
// airdrop.withdraw(...) → does not exist → permanent lock confirmed
```
