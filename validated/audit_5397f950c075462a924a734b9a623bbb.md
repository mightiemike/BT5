### Title
Softfork Cost Atom with Redundant Leading Zeros Creates Mempool/Consensus Split — (`src/run_program.rs`, `src/op_utils.rs`)

### Summary
A softfork invocation whose cost argument is a Buffer atom with redundant leading zero bytes (e.g., `[0x00, 0x00, 0x03, 0x97]` for cost 919) is accepted by consensus-mode execution but rejected by mempool-mode execution, because `uint_atom::<8>` in `apply_op` strips leading zeros without `CANONICAL_INTS` and rejects them with it. This creates a concrete mempool/consensus split.

### Finding Description
In `apply_op` (`src/run_program.rs:385`), the softfork cost is extracted via `uint_atom::<8>(allocator, first(operand_list), "softfork", self.dialect.flags())`. The `?` operator means any error propagates immediately with no fallback.

In `uint_atom` (`src/op_utils.rs:67–86`), behavior diverges on the `CANONICAL_INTS` flag:

- **Without `CANONICAL_INTS`** (consensus): all leading zero bytes are stripped in a `while` loop. `[0x00, 0x00, 0x03, 0x97]` becomes `[0x03, 0x97]` = 919. Execution proceeds normally.
- **With `CANONICAL_INTS`** (mempool, per `MEMPOOL_MODE` at `src/chia_dialect.rs:72–76`): if `buf[0] == 0` and `(buf[1] & 0x80) == 0`, an `InvalidOpArg` error is returned immediately. For `[0x00, 0x00, 0x03, 0x97]`, `buf[1] = 0x00` has no high bit, so the error fires. The transaction is rejected.

The consensus-mode fallback for unknown softfork extensions (lines 400–413) does not apply here because the cost parsing error occurs before `parse_softfork_arguments` is ever called.

### Impact Explanation
A transaction containing a softfork operator with a non-canonical cost atom is:
- **Accepted by consensus nodes** (no `CANONICAL_INTS`): the program runs, cost is charged correctly, the block is valid.
- **Rejected by mempool nodes** (`CANONICAL_INTS` present): the transaction is dropped as invalid before it can be relayed.

This is a mempool/consensus split. Such a transaction can only reach a farmer via a direct out-of-band submission (bypassing the mempool entirely). Any farmer running consensus-mode validation would include it in a block, and all consensus nodes would accept that block. Mempool nodes would never have seen or relayed the transaction, creating a relay manipulation primitive.

### Likelihood Explanation
The attacker only needs to serialize a CLVM program where the softfork cost atom has one or more leading zero bytes. CLVM serialization does not normalize integer atoms; raw bytes are stored as-is. This is trivially constructable by any party who can submit transactions.

### Recommendation
In `apply_op`, normalize the cost atom before applying the `CANONICAL_INTS` check, or enforce that the cost atom must be canonical in both modes (i.e., apply the same rejection logic regardless of `CANONICAL_INTS`). The simplest fix is to always strip leading zeros from the softfork cost atom in both modes, or to always reject non-canonical cost atoms in both modes, so the accept/reject decision is identical across consensus and mempool.

### Proof of Concept

The two assertions cannot both hold for a valid invariant. The split is real.
