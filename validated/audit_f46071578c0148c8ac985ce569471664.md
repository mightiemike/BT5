### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable Without Coinbase Proof Validation, Enabling Merkle Proof Forgery — (File: contract/src/lib.rs)

### Summary

The contract exposes two transaction-inclusion verification endpoints: the deprecated `verify_transaction_inclusion` (v1) and the current `verify_transaction_inclusion_v2`. The v2 function adds a coinbase Merkle proof check specifically to block the 64-byte internal-node forgery attack. However, v1 is still a live, unprivileged-callable public function. Any NEAR caller can invoke v1 directly, bypassing the coinbase guard entirely, and obtain a `true` result for a crafted proof whose `tx_id` is an internal Merkle tree node rather than a real transaction hash. External DApps that consume this result are deceived into accepting a forged inclusion proof.

### Finding Description

**Root cause — missing coinbase proof validation in a still-callable public function.**

`verify_transaction_inclusion` (v1) performs only two checks before returning a result:

1. The block is in the current main chain (`mainchain_header_to_height` lookup).
2. The supplied Merkle proof, rooted at `tx_id` and position `tx_index`, produces the block's stored `merkle_root`. [1](#0-0) 

There is no check that `tx_id` is a leaf node (an actual transaction hash) rather than an internal Merkle tree node. The function's own documentation acknowledges this: [2](#0-1) 

`verify_transaction_inclusion_v2` closes this gap by first verifying that the coinbase transaction (always at index 0) produces the block's `merkle_root`: [3](#0-2) 

Because the coinbase is a real leaf, any internal node used as `tx_id` cannot simultaneously satisfy both the coinbase check and the tx check. But v1 has no such guard, and v1 is still decorated only with `#[pause]` — not with any access-control role — making it callable by any unprivileged account. [4](#0-3) 

**Exploit path (concrete, no privilege required):**

Consider a main-chain block with four transactions `[T0, T1, T2, T3]`. Its Merkle tree is:

```
Root = hash(H01 ‖ H23)
       H01 = hash(T0 ‖ T1)    H23 = hash(T2 ‖ T3)
       T0   T1                 T2   T3
```

An attacker calls `verify_transaction_inclusion` with:
- `tx_id      = H01`  (internal node, not a real transaction)
- `tx_index   = 0`
- `merkle_proof = [H23]`
- `tx_block_blockhash = <any main-chain block hash with this structure>`
- `confirmations = 1`

`compute_root_from_merkle_proof(H01, 0, [H23])` computes `hash(H01 ‖ H23)` = `Root`, which equals the stored `merkle_root`. The function returns `true`. [5](#0-4) 

The same call to v2 would fail: `compute_root_from_merkle_proof(T0, 0, [H23])` = `hash(T0 ‖ H23)` ≠ `Root` (since `H01 ≠ T0`), so the coinbase guard rejects it. [6](#0-5) 

### Impact Explanation

Any external NEAR DApp or smart contract that calls `verify_transaction_inclusion` (v1) and acts on a `true` result can be deceived into believing a transaction was confirmed in a Bitcoin block when no such transaction exists. Concrete consequences include:

- Fraudulent release of bridged assets or cross-chain state transitions triggered by a forged SPV proof.
- Bypass of any on-chain guard that relies on this function's return value as a trust anchor.

The broken invariant is: *a `true` result from `verify_transaction_inclusion` must mean the supplied `tx_id` is an actual transaction included in the specified block.* v1 does not enforce this invariant.

**Impact: High** — direct proof forgery enabling unauthorized state changes in consuming contracts.

### Likelihood Explanation

The attack requires no privileged keys, no cryptographic work, and no special tooling. The attacker only needs:
1. Knowledge of any main-chain block's Merkle tree (publicly available from any Bitcoin node or block explorer).
2. The ability to call a public NEAR contract function.

Any DApp that has not migrated from v1 to v2 is immediately exploitable. The deprecated function has been present since before v0.5.0 and remains live.

**Likelihood: Medium** — trivially executable by anyone aware of the vulnerability; gated only by whether a consuming DApp still calls v1.

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the contract entirely, or gate it behind an access-control role (e.g., `Role::DAO`) so it cannot be called by unprivileged accounts. Deprecation annotations in Rust are compile-time hints for library consumers; they do not restrict on-chain callability. As long as the function exists as a `pub` method, it is a live attack surface.

If removal is not immediately feasible, add the same coinbase proof guard that v2 uses, or add a `require!(false, "use verify_transaction_inclusion_v2")

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
