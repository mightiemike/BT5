### Title
Unchecked `transferFrom` Return Value in `ContractOwner::replaceUsdcEWithUsdc()` Enables usdcE Drain Without Providing USDC — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the boolean return value. Every other token transfer in the Nado codebase routes through `ERC20Helper.safeTransferFrom`, which enforces the return value check. This single raw call is the only place in the production contracts where the return value is silently discarded. If the USDC token's `transferFrom` returns `false` instead of reverting, an attacker can receive usdcE from a Direct Deposit Account (DDA) without supplying any USDC.

---

### Finding Description

`ERC20Helper.safeTransferFrom` is the protocol-wide safe wrapper:

```solidity
// core/contracts/libraries/ERC20Helper.sol
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [1](#0-0) 

All deposit and withdrawal paths in `EndpointStorage`, `BaseWithdrawPool`, and `EndpointTx` use this wrapper via `using ERC20Helper for IERC20Base`. [2](#0-1) [3](#0-2) 

However, `replaceUsdcEWithUsdc` in `ContractOwner.sol` bypasses this wrapper entirely:

```solidity
// core/contracts/ContractOwner.sol  line 608-620
function replaceUsdcEWithUsdc(bytes32 subaccount) external {
    require(block.chainid == 57073, ERR_UNAUTHORIZED);
    ...
    uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
    if (balance > 0) {
        IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // @audit unchecked
        DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
        IERC20Base(usdcE).safeTransfer(msg.sender, balance);
    }
}
``` [4](#0-3) 

The raw `transferFrom` call at line 616 discards the boolean return value. If it returns `false`, execution continues: the DDA's usdcE is withdrawn to `ContractOwner` (via `DirectDepositV1.withdraw`, which sends to `msg.sender` of the call — i.e., `ContractOwner`), and then `safeTransfer` sends that usdcE to the attacker. [5](#0-4) 

---

### Impact Explanation

An attacker who calls `replaceUsdcEWithUsdc` with a valid `subaccount` that has a DDA holding usdcE, while having zero USDC allowance or balance, can drain the usdcE from that DDA and receive it for free — provided the USDC token returns `false` on failure rather than reverting. The corrupted state delta is: usdcE balance of the DDA decreases to zero, while the attacker's usdcE balance increases by `balance`, with no corresponding USDC deposited into the DDA.

---

### Likelihood Explanation

The function is `external` with no access control beyond a chain ID check (`block.chainid == 57073`, the Ink chain), making it reachable by any unprivileged caller. The hardcoded USDC address (`0x2D270e6886d130D724215A266106e6832161EAEd`) on Ink chain is likely a standard Circle USDC implementation that reverts on failure, which reduces practical exploitability. However, the code pattern is wrong and inconsistent with the rest of the codebase. If the token is upgraded, replaced, or if a non-standard USDC variant is deployed at that address, the vulnerability becomes directly exploitable. [6](#0-5) 

---

### Recommendation

Replace the raw `transferFrom` call with the protocol-standard `ERC20Helper.safeTransferFrom`:

```diff
- IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
+ IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

Since `ContractOwner` already has `using ERC20Helper for IERC20Base` declared, this is a one-line fix consistent with every other transfer in the codebase. [7](#0-6) 

---

### Proof of Concept

1. A DDA exists for some `subaccount` with `balance > 0` usdcE.
2. Attacker calls `replaceUsdcEWithUsdc(subaccount)` on Ink chain with zero USDC allowance.
3. `IERC20Base(usdc).transferFrom(attacker, dda, balance)` returns `false` (no revert).
4. `DirectDepositV1(dda).withdraw(usdcE)` executes — DDA sends all usdcE to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, balance)` executes — attacker receives usdcE.
6. Result: attacker holds `balance` usdcE; DDA holds zero usdcE; no USDC was deposited. [8](#0-7)
