Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual User, Enabling Allowlist Bypass via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed by the pool, which is the pool's `msg.sender` — the router contract when a user routes through `MetricOmmSimpleRouter`. If the pool admin allowlists the router to enable the standard periphery path, any unprivileged user can bypass the per-user allowlist by routing through the router. Conversely, if the router is not allowlisted, every legitimately allowlisted user is blocked from using the router path.

## Finding Description
In `MetricOmmPool.swap`, the pool passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ← pool's msg.sender, not the original user
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension via `_callExtensionsInOrder`. `SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

Here `msg.sender` is the pool and `sender` is the first argument — the router address when the user entered through the router.

In `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap` directly, making itself the pool's `msg.sender`. The actual user (`msg.sender` of the router call) is stored only in transient callback context via `_setNextCallbackContext` for payment settlement and is never forwarded to the pool or to any extension:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    ...
  );
```

The same wrong-actor binding applies to `exactInput` (L92-125), `exactOutputSingle` (L130-147), and `exactOutput` (L154-188) — all router entry points call `pool.swap` with `msg.sender = router`.

**Bypass path:** Admin allowlists the router so allowlisted users can reach the pool through the standard periphery path → `allowedSwapper[pool][router] == true` → any unprivileged user calls `router.exactInputSingle` → extension sees `sender = address(router)`, finds it allowlisted, returns success selector → swap executes despite the user never being individually allowlisted.

**Broken functionality path:** Admin does not allowlist the router → every legitimately allowlisted user who tries to swap through the router is blocked with `NotAllowedToSwap`.

No existing guard compensates for this: the extension has no mechanism to recover the original initiator, and the pool interface provides no initiator field beyond `msg.sender`.

## Impact Explanation
**High — direct bypass of curation policy:** A pool admin who allowlists the router (the only way to let allowlisted users reach the pool through the standard periphery path) inadvertently opens the allowlist to every address on the network. Any address can call `router.exactInputSingle` and the extension will pass because `allowedSwapper[pool][router] == true`. The curated pool's per-user gate is completely defeated; non-KYC'd or otherwise excluded users can trade freely. This constitutes broken core pool functionality causing loss of the curation invariant and falls under the "Allowlist path" audit pivot: swap allowlist checks must cover the exact actor and cannot be bypassed through the router.

**Medium — unusable core flow:** If the admin does not allowlist the router, every legitimately allowlisted user who tries to swap through the router is blocked, rendering the standard periphery path unusable for the pool's intended participants.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing entry point for swaps. Pool admins who configure a `SwapAllowlistExtension` will naturally need to decide whether to allowlist the router; allowlisting it is the only way to let their allowlisted users reach the pool through the router, making the bypass scenario a predictable operational outcome. The attack requires no special privilege — any address can call the public router functions.

## Recommendation
The pool must forward the original user identity to extensions. Two complementary fixes:

1. **Pass the original initiator through the router.** Add an `initiator` field to the swap call so the pool can forward the real user to extensions instead of `msg.sender`. The router would pass `msg.sender` as the initiator, and the pool would forward it to `_beforeSwap` alongside `sender`.

2. **Short-term mitigation.** Until the pool interface is updated, `SwapAllowlistExtension` should not be used with pools that are also reachable through the router, or the router must not be allowlisted (accepting that allowlisted users cannot use the router path).

## Proof of Concept
```solidity
// Setup
SwapAllowlistExtension ext = new SwapAllowlistExtension(factory);
// Admin allowlists the router so that allowlisted users can reach the pool
ext.setAllowedToSwap(pool, address(router), true);
// Alice is NOT individually allowlisted
assert(!ext.isAllowedToSwap(pool, alice));

// Attack: Alice bypasses the allowlist by going through the router
vm.prank(alice);
router.exactInputSingle(ExactInputSingleParams({
    pool:            pool,
    recipient:       alice,
    tokenIn:         token0,
    zeroForOne:      true,
    amountIn:        1e18,
    amountOutMinimum: 0,
    priceLimitX64:   0,
    deadline:        block.timestamp,
    extensionData:   ""
}));
// ✓ swap succeeds — extension saw sender=router, which is allowlisted
// Alice traded on a curated pool she was never authorized to access
```

The extension receives `sender = address(router)`, finds `allowedSwapper[pool][router] == true` at `SwapAllowlistExtension.sol:37`, and returns the success selector. Alice's swap executes in full despite never being individually allowlisted.
