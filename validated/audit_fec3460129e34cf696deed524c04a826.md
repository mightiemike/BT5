### Title
Isolated Subaccount Collateral Permanently Locked After Full Perp Position Liquidation Without Finalization — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

When a liquidator fully zeros out an isolated subaccount's perp position via a regular (non-finalization) liquidation, the isolated subaccount's remaining quote collateral becomes permanently locked. The `_tryCloseIsolatedSubaccount` path is only triggered on the finalization branch, but finalization requires the subaccount to still be under maintenance — a condition that is no longer true once all positions are zeroed. The user has no other path to recover their collateral.

---

### Finding Description

In `ClearinghouseLiq.liquidateSubaccountImpl`, after a regular liquidation (`productId != type(uint32).max`), `tryCloseIsolatedSubaccount` is only called when `_finalizeSubaccount` returns `true`:

```solidity
if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
    if (RiskHelper.isIsolatedSubaccount(txn.liquidatee)) {
        IOffchainExchange(...).tryCloseIsolatedSubaccount(txn.liquidatee);
    }
    return;
}
```

`_finalizeSubaccount` returns `true` only when `txn.productId == type(uint32).max`. For any other `productId`, it returns `false` immediately and `tryCloseIsolatedSubaccount` is never called. [1](#0-0) 

A liquidator can submit a regular liquidation (`productId != type(uint32).max`) with `amount` equal to the isolated subaccount's entire perp position, zeroing it out in one step. After this:

- `balance.amount == 0` for the perp product
- The isolated subaccount retains a positive quote balance (remaining collateral)
- `isUnderMaintenance` returns `false` (no positions → health ≥ 0)

Now the user is stuck in a dead state:

**Path 1 — `withdrawCollateral`**: Blocked unconditionally for isolated subaccounts:
```solidity
require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
``` [2](#0-1) 

**Path 2 — Finalization liquidation** (`productId == type(uint32).max`): Blocked because `isUnderMaintenance` returns `false`:
```solidity
require(isUnderMaintenance(txn.liquidatee), ERR_NOT_LIQUIDATABLE);
``` [3](#0-2) 

**Path 3 — Direct `tryCloseIsolatedSubaccount`**: Only callable by the endpoint or clearinghouse address, not by the user:
```solidity
require(
    msg.sender == getEndpoint() || msg.sender == address(clearinghouse),
    ERR_UNAUTHORIZED
);
``` [4](#0-3) 

There is no transaction type in `EndpointTx` that allows a user to directly invoke `tryCloseIsolatedSubaccount` for an already-zeroed isolated subaccount that is no longer under maintenance. The quote collateral is permanently locked.

`_tryCloseIsolatedSubaccount` itself correctly handles the zero-position case — it transfers `vQuoteBalance` and `quoteBalance` to the parent — but it is simply never reached: [5](#0-4) 

---

### Impact Explanation

The isolated subaccount's remaining quote collateral (USDC) is permanently locked inside the protocol. The user loses all excess margin that was not consumed by the liquidation. The asset delta is: `quoteBalance` of the isolated subaccount, which is credited to no one and cannot be recovered by the owner.

---

### Likelihood Explanation

**Medium.** Isolated subaccounts are a supported feature for perp trading. A liquidator can zero out a small or medium-sized isolated position in a single liquidation call. This is a normal, incentivized action. The victim does not need to do anything unusual — simply having an isolated perp position that gets fully liquidated in one step is sufficient to trigger the lock.

---

### Recommendation

In `liquidateSubaccountImpl`, after a regular (non-finalization) liquidation of an isolated subaccount, check whether the perp position has been reduced to zero and, if so, call `tryCloseIsolatedSubaccount`. Specifically, after `_handleLiquidationPayment` completes, add:

```solidity
if (RiskHelper.isIsolatedSubaccount(txn.liquidatee)) {
    IPerpEngine.Balance memory bal = perpEngine.getBalance(
        txn.productId, txn.liquidatee
    );
    if (bal.amount == 0) {
        IOffchainExchange(...).tryCloseIsolatedSubaccount(txn.liquidatee);
    }
}
```

This mirrors the existing finalization path and ensures collateral is always returned to the parent when the isolated position reaches zero, regardless of how it got there.

---

### Proof of Concept

1. User opens an isolated perp position on product `P` with 100 USDC margin. Isolated subaccount `ISO` is created with `quoteBalance = 100`.
2. Price moves adversarially; `ISO` falls under maintenance health.
3. Liquidator calls `LiquidateSubaccount` with `productId = P` (not `type(uint32).max`) and `amount` equal to the full position. `_handleLiquidationPayment` zeros `balance.amount` for `ISO`. Suppose 20 USDC remains in `ISO.quoteBalance` after the liquidation payment.
4. `_finalizeSubaccount` returns `false` (because `productId != type(uint32).max`). `tryCloseIsolatedSubaccount` is **not** called.
5. `ISO.balance.amount == 0`, so `isUnderMaintenance(ISO)` returns `false`.
6. User attempts finalization: `LiquidateSubaccount` with `productId = type(uint32).max` → reverts `ERR_NOT_LIQUIDATABLE`.
7. User attempts `WithdrawCollateral` from `ISO` → reverts `ERR_UNAUTHORIZED` (isolated subaccount check).
8. The 20 USDC in `ISO` is permanently locked. [6](#0-5) [7](#0-6)

    function _tryCloseIsolatedSubaccount(bytes32 subaccount) internal {
        uint32 productId = RiskHelper.getIsolatedProductId(subaccount);
        if (productId == 0) {
            return;
        }
        IPerpEngine.Balance memory balance = perpEngine.getBalance(
            productId,
            subaccount
        );
        if (balance.amount == 0) {
            uint8 id = RiskHelper.getIsolatedId(subaccount);
            address addr = address(uint160(bytes20(subaccount)));
            bytes32 parent = parentSubaccounts[subaccount];
            if (balance.vQuoteBalance != 0) {
                perpEngine.updateBalance(
                    productId,
                    subaccount,
                    0,
                    -balance.vQuoteBalance
                );
                perpEngine.updateBalance(
                    productId,
                    parent,
                    0,
                    balance.vQuoteBalance
                );
            }
            int128 quoteBalance = spotEngine
                .getBalance(QUOTE_PRODUCT_ID, subaccount)
                .amount;
            if (quoteBalance != 0) {
                spotEngine.updateBalance(
                    QUOTE_PRODUCT_ID,
                    subaccount,
                    -quoteBalance
                );
                spotEngine.updateBalance(
                    QUOTE_PRODUCT_ID,
                    parent,
                    quoteBalance
                );
            }
            isolatedSubaccountsMask[addr] &= ~uint256(0) ^ (1 << id);
            isolatedSubaccounts[parent][id] = bytes32(0);
            parentSubaccounts[subaccount] = bytes32(0);

            emit CloseIsolatedSubaccount(subaccount, parent);
        }
