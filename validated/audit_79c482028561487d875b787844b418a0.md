### Title
Sanctioned EOA Bypasses Sanctions Check via Proxy Contract — (`core/contracts/Airdrop.sol`)

### Summary

The `_verifyProof` function gates claims with `ISanctionsList.isSanctioned(sender)` where `sender` is always `msg.sender`. Because `msg.sender` is the **immediate caller** — not the ultimate beneficial owner — a sanctioned EOA can deploy a non-sanctioned proxy contract, have that contract's address included in the Merkle tree, and claim the full allocation through the proxy. The sanctions check passes, the Merkle proof verifies, and tokens are transferred to the proxy, which the sanctioned EOA controls.

And the transfer: [2](#0-1)

Every check — sanctions, leaf construction, claimed mapping, and transfer destination — is keyed to `msg.sender`. There is no `tx.origin` check and no `require(msg.sender == tx.origin)` EOA-only guard.

**Concrete attack path:**

1. Sanctioned EOA pre-computes a `CREATE2` contract address (deterministic).
2. The contract (or the EOA acting through it) interacts with the Nado protocol, accumulating off-chain airdrop eligibility under the **contract address**.
3. The off-chain system builds the Merkle tree using on-chain addresses; it includes the contract address with its earned allocation. The off-chain system has no obligation to check whether a contract is *controlled by* a sanctioned EOA — it only sees an address.
4. Owner calls `registerMerkleRoot(week, root)`.
5. Sanctioned EOA deploys the proxy contract (now at the pre-computed address) and calls `claim()` from it.
6. `isSanctioned(contractAddress)` → `false` ✓
7. Leaf matches → `MerkleProof.verify` passes ✓
8. `claimed[week][contractAddress] = totalAmount` set ✓
9. `safeTransfer(token, contractAddress, totalAmount)` executes ✓
10. Sanctioned EOA calls a withdrawal function on the proxy → receives tokens.

---

### Impact Explanation

The invariant **"sanctioned principals must never receive airdrop tokens"** is broken. The sanctioned actor receives their full weekly allocation routed through a one-hop proxy. The only on-chain barrier — `isSanctioned(msg.sender)` — is trivially circumvented because it checks the proxy's address, not the controlling EOA. [3](#0-2) 

---

### Likelihood Explanation

- Contracts routinely interact with DeFi protocols and accumulate rewards under their own address.
- `CREATE2` makes the proxy address deterministic and pre-computable before deployment, so the EOA can arrange for the address to earn eligibility before the contract even exists.
- The off-chain Merkle tree construction has no in-protocol mechanism to trace contract ownership back to a sanctioned EOA.
- No special privilege or admin compromise is required — only a public `claim()` call.

---

### Recommendation

Add an EOA-only guard at the top of `_claim` or `_verifyProof`:

```solidity
require(msg.sender == tx.origin, "Only EOAs may claim");
```

Or, if contract callers must be supported, check both the immediate caller and the transaction origin:

```solidity
require(
    !ISanctionsList(sanctions).isSanctioned(msg.sender) &&
    !ISanctionsList(sanctions).isSanctioned(tx.origin),
    "address is sanctioned."
);
```

The `tx.origin` check ensures that even if a non-sanctioned proxy is used, the originating EOA is still screened.

    function _verifyProof(
        uint32 week,
        address sender,
        uint256 totalAmount,
        bytes32[] calldata proof
    ) internal {
        require(claimed[week][sender] == 0, "Already claimed.");
        require(
            merkleRoots[week] != bytes32(0),
            "Week hasn't been registered."
        );
        require(
            !ISanctionsList(sanctions).isSanctioned(sender),
            "address is sanctioned."
        );
        bytes32 leaf = keccak256(
            bytes.concat(keccak256(abi.encode(sender, totalAmount)))
        );
        bool isValidLeaf = MerkleProof.verify(proof, merkleRoots[week], leaf);
        require(isValidLeaf, "Invalid proof.");
        claimed[week][sender] = totalAmount;
    }

    function _claim(
        uint32 week,
        uint256 totalAmount,
        bytes32[] calldata proof
    ) internal {
        _verifyProof(week, msg.sender, totalAmount, proof);
        SafeERC20.safeTransfer(IERC20(token), msg.sender, totalAmount);
        emit Claim(msg.sender, week, totalAmount);
    }
