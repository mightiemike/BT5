### Title
`coinbase_tx_id` Is Not Committed to in the Block Header, Rendering the 64-Byte Forgery Mitigation in `verify_transaction_inclusion_v2` Ineffective - (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to mitigate the 64-byte transaction Merkle proof forgery vulnerability by requiring a valid coinbase Merkle proof. However, the `coinbase_tx_id` field is entirely caller-supplied and is never validated against any on-chain commitment. Because the Bitcoin block header only commits to `merkle_root` — not to the coinbase transaction hash — any unprivileged NEAR caller can supply an internal Merkle tree node as `coinbase_tx_id`, satisfy the coinbase check with a shorter proof, and then prove inclusion of another internal node as a fake `tx_id`. The mitigation is therefore a no-op.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces two checks before delegating to the deprecated `verify_transaction_inclusion`:

1. `merkle_proof.len() == coinbase_merkle_proof.len()` — both proofs must have the same depth.
2. `compute_root_from_merkle_proof(coinbase_tx_id, 0, coinbase_merkle_proof) == merkle_root` — the coinbase proof must reconstruct the block's Merkle root. [1](#0-0) 

The intent is to anchor the proof to the coinbase transaction (always at leaf position 0), preventing an attacker from substituting an internal node for a leaf. However, `coinbase_tx_id` is a free parameter supplied by the caller: [2](#0-1) 

The block header stored on-chain commits only to `merkle_root`: [3](#0-2) 

There is no stored coinbase transaction hash, no commitment to the coinbase `tx_id`, and no on-chain way to distinguish a leaf hash from an internal node hash. The coinbase check therefore only verifies that the caller-chosen value, treated as a node at position 0, reconstructs the root — which an attacker can trivially satisfy using any internal node that sits on the left spine of the tree.

`compute_root_from_merkle_proof` uses `transaction_position` purely to decide left/right ordering at each level; it has no awareness of tree depth or whether the starting value is a leaf: [4](#0-3) 

---

### Impact Explanation

Consider a block whose Merkle tree has depth 2:

```
        R
       / \
      A   B
     / \ / \
    T1 T2 T3 T4
```

An attacker calls `verify_transaction_inclusion_v2` with:

| Field | Value |
|---|---|
| `coinbase_tx_id` | `A` (internal node, left child of root) |
| `coinbase_merkle_proof` | `[B]` (length 1) |
| `tx_id` | `A` (same internal node) |
| `tx_index` | `0` |
| `merkle_proof` | `[B]` (length 1) |

Step-by-step:
- Length check: `1 == 1` ✓
- Coinbase check: `compute_root_from_merkle_proof(A, 0, [B])` → position 0 is even → `hash(A, B) = R` ✓
- Transaction check (inside deprecated call): `compute_root_from_merkle_proof(A, 0, [B]) = R` ✓
- **Returns `true`** for `tx_id = A`, which is an internal node, not a real transaction.

Any dApp consuming `verify_transaction_inclusion_v2` as a trust anchor for cross-chain settlement, bridge unlocking, or SPV-based authorization will accept a proof for a value that was never a Bitcoin transaction. The attacker needs only public knowledge of the Merkle tree of any confirmed block.

---

### Likelihood Explanation

- The attack requires no privileged role, no leaked key, and no social engineering.
- The Bitcoin Merkle tree of every block is publicly derivable from block data available via any Bitcoin node or block explorer.
- The attacker only needs to call `verify_transaction_inclusion_v2` on NEAR with crafted arguments — a standard, permissionless NEAR transaction.
- The function is not paused by default and is the recommended replacement for the deprecated v1. [5](#0-4) 

---

### Recommendation

The contract must validate that `coinbase_tx_id` is the actual coinbase transaction hash of the target block. Since the block header does not commit to the coinbase hash directly, one of the following approaches is needed:

1. **Store the coinbase hash at submission time.** When a block header is submitted via `submit_blocks`, require the relayer to also supply the coinbase transaction hash and store it in `ExtendedHeader`. `verify_transaction_inclusion_v2` then compares `coinbase_tx_id` against the stored value before running the proof.

2. **Require the raw coinbase transaction bytes.** Accept the full serialized coinbase transaction in `ProofArgsV2`, compute its `double_sha256` on-chain, and use that as the anchor — eliminating the free `coinbase_tx_id` parameter entirely.

3. **Enforce minimum proof length relative to `tx_index`.** At minimum, reject proofs whose length is inconsistent with the claimed `tx_index` (i.e., `proof.len() < ceil(log2(tx_index + 1))`), which prevents using high-level internal nodes as leaves. This is a partial mitigation only.

---

### Proof of Concept

Given any confirmed Bitcoin block with at least 2 transactions, let `R` be its Merkle root, `A = hash(T1, T2)` the left subtree root, and `B = hash(T3, T4)` the right subtree root.

```
coinbase_tx_id      = A          // internal node, not a real tx
coinbase_merkle_proof = [B]      // length 1
tx_id               = A
tx_index            = 0
merkle_proof        = [B]        // length 1
tx_block_blockhash  = <any mainchain block hash with this Merkle root>
confirmations       = 1
```

Call `verify_transaction_inclusion_v2` with these arguments. The function returns `true` despite `A` never appearing as a transaction in the block. [6](#0-5) [7](#0-6)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
