Audit Report

## Title
DepositAllowlistExtension Checks `owner` Instead of Caller, Allowing Allowlist Bypass via Owner/Salt Separation - (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary
The `beforeAddLiquidity` hook in `DepositAllowlistExtension` checks the `owner` parameter (the position recipient) rather than the actual caller (the first unnamed parameter, which is `msg.sender` of the `addLiquidity` call). Any unprivileged address can bypass the deposit allowlist by calling `addLiquidity` with an allowlisted address as `owner`, providing tokens via the swap callback, and crediting the position to the allowlisted address without being on the allowlist themselves.

## Finding Description
In `MetricOmmPool.addLiquidity` (line 191), the extension hook is invoked as:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

The first argument is the actual caller of `addLiquidity`; the second is the caller-supplied `owner`. In `DepositAllowlistExtension.beforeAddLiquidity` (lines 32–42):

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The first parameter (the actual depositing caller) is silently discarded. The guard only checks whether `owner` is allowlisted. Because `owner` is a free caller-supplied argument with no binding to `msg.sender`, any address can pass the check by supplying any allowlisted address as `owner`.

**Exploit path:**
1. Attacker (not on allowlist) calls `pool.addLiquidity(allowlistedAddress, salt, deltas, callbackData, extensionData)`.
2. Extension receives `(attacker, allowlistedAddress, …)` but checks only `allowedDepositor[pool][allowlistedAddress]` → passes.
3. `LiquidityLib.addLiquidity` executes; the attacker's callback provides tokens.
4. LP shares are minted to `allowlistedAddress`; the attacker has deposited into a restricted pool.

The `removeLiquidity` guard (`if (msg.sender != owner) revert NotPositionOwner()`) means the attacker cannot reclaim the tokens — they are permanently credited to `allowlistedAddress` — but the allowlist restriction is fully circumvented.

## Impact Explanation
The deposit allowlist is an admin-configured access-control boundary. Bypassing it via owner/salt separation means any unprivileged caller can inject liquidity into a restricted pool, violating the pool admin's intended access policy. The exact corrupted value is `allowedDepositor[pool][caller]`: the check that should gate the depositing actor is never evaluated. This constitutes an admin-boundary break where a pool admin-configured restriction is bypassed by an unprivileged path, matching the allowed impact gate.

## Likelihood Explanation
Exploitation requires no special privilege: any EOA or contract can call `addLiquidity` on a pool with this extension. The attacker only needs to know one allowlisted address (publicly readable from `allowedDepositor` or emitted events). The cost is the deposited tokens, which are irrecoverably credited to the allowlisted address. The attack is repeatable every block.

## Recommendation
Name and check the first parameter (the actual caller) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This ensures the address actually providing tokens (and initiating the call) is the one validated against the allowlist.

    // Alice now holds LP shares she did not request
