### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the 64-Byte Transaction Forgery Mitigation — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle-proof forgery vulnerability by requiring a coinbase merkle proof. However, the original `verify_transaction_inclusion` (v1) was only annotated `#[deprecated]` — it was never removed, gated, or blocked. Any unprivileged NEAR caller can invoke v1 directly, skipping the coinbase proof check entirely and obtaining a `true` verification result for a forged proof.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces a coinbase merkle proof check before delegating to v1: [1](#0-0) 

The coinbase check is the entire security addition: [2](#0-1) 

The v1 function is still a live, public, `#[pause]`-only endpoint with no role guard and no on-chain flag preventing its use: [3](#0-2) 

The function's own doc comment acknowledges the forgery risk explicitly: [4](#0-3) 

The v1 function computes the merkle root from the attacker-supplied `tx_id` and `merkle_proof` and compares it to the stored `merkle_root`, with no coinbase anchor: [5](#0-4) 

---

### Impact Explanation

An attacker crafts a 64-byte payload that is simultaneously a valid serialized internal Merkle tree node and a plausible `tx_id`. They supply a `merkle_proof` path that reconstructs the block's real `merkle_root` from this internal node. Calling `verify_transaction_inclusion` (v1) directly returns `true` for a transaction that was never included in the block. Any recipient NEAR contract that calls v1 to gate a withdrawal, bridge release, or cross-chain action is deceived into executing it. The corrupted value is the **proof result** (`true`) and the **canonical-chain inclusion decision** for a non-existent transaction.

---

### Likelihood Explanation

The 64-byte transaction attack is publicly documented (referenced in the contract's own comments and the BitMEX blog post cited in the code). The v1 function is a named, public NEAR method discoverable via any ABI or SDK call. No privileged role, leaked key, or social engineering is required — any NEAR account can call it. Recipient contracts that integrated before v2 was released, or that call v1 by name, are directly exposed.

---

### Recommendation

Remove `verify_transaction_inclusion` entirely, or add an unconditional `env::panic_str` / `require!(false, ...)` at its entry point so it can never return a result. A Rust `#[deprecated]` attribute is a compile-time lint for Rust callers only; it provides zero on-chain enforcement. The `#[pause]` guard is insufficient because it only blocks the function when an admin has explicitly paused it — it does not block it by default.

---

### Proof of Concept

1. Deploy the contract (Bitcoin feature flag).
2. Submit a real block header so `headers_pool` contains a known `tx_block_blockhash` with a known `merkle_root`.
3. Construct a 64-byte value `fake_tx_id` that is an internal Merkle node whose hash, combined with a crafted `merkle_proof`, reconstructs the block's real `merkle_root`.
4. Call `verify_transaction_inclusion` directly (v1) with `tx_id = fake_tx_id`, the crafted `merkle_proof`, and `confirmations = 1`.
5. The function returns `true` — no coinbase proof was required, no `tx_id` validity check is performed on-chain.
6. A recipient contract gating on this return value executes its protected action for a transaction that does not exist.

The root cause is the same vulnerability class as the GenesisGroup report: a function that should be blocked after a protocol-level state transition (the v2 security upgrade) is not blocked on-chain, because the only enforcement mechanism (`#[deprecated]`) is a compile-time Rust annotation with no runtime effect on NEAR.

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
