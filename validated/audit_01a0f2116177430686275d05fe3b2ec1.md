### Title
`burnNlp` Uses Weaker `MAINTENANCE` Health Check Instead of `INITIAL`, Allowing Collateral Extraction When Position Is Undercollateralized — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`burnNlp` in `Clearinghouse.sol` enforces only a `MAINTENANCE` health check after burning NLP tokens for quote collateral, while the directly analogous `withdrawCollateral` function enforces an `INITIAL` health check. This inconsistency allows a user whose `INITIAL` health is negative (undercollateralized for new positions) but whose `MAINTENANCE` health is non-negative (not yet liquidatable) to extract quote collateral through NLP burns — a path that is explicitly blocked for direct collateral withdrawals.

---

### Finding Description

`withdrawCollateral` enforces `INITIAL` health after reducing a user's spot balance:

```solidity
IProductEngine.HealthType healthType = sender == X_ACCOUNT
    ? IProductEngine.HealthType.PNL
    : IProductEngine.HealthType.INITIAL;
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
``` [1](#0-0) 

`burnNlp`, which also results in the user receiving quote tokens (collateral), enforces only `MAINTENANCE` health:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [2](#0-1) 

The in-code comment explicitly acknowledges the risk: *"Burning NLP can decrease health if the burn fee exceeds the health improvement from the withdrawal. This check prevents malicious actors from deliberately creating unhealthy subaccounts through NLP burns."* Yet the chosen threshold (`MAINTENANCE`) is weaker than what is needed to satisfy that stated goal. `INITIAL` health is the stricter threshold (lower weights applied to positions), so a subaccount can have `INITIAL` health < 0 while `MAINTENANCE` health ≥ 0. The `MAINTENANCE` check does not prevent the creation of subaccounts with negative `INITIAL` health. [3](#0-2) 

The health type definitions confirm the ordering: `INITIAL` uses `longWeightInitialX18` / `shortWeightInitialX18` (more conservative), while `MAINTENANCE` uses `longWeightMaintenanceX18` / `shortWeightMaintenanceX18` (less conservative). [4](#0-3) 

---

### Impact Explanation

A user holding NLP tokens whose `INITIAL` health is already negative (e.g., due to an open leveraged position that has moved against them) can call `burnNlp` to receive quote tokens back. Because only `MAINTENANCE` health is checked, the burn succeeds as long as `MAINTENANCE` health ≥ 0 post-burn. The user extracts collateral from the protocol in a state where `withdrawCollateral` would unconditionally revert. Each successive burn can push `INITIAL` health further negative while the `MAINTENANCE` check continues to pass, allowing the user to drain their NLP collateral incrementally until the position approaches the liquidation boundary. This creates an asymmetric collateral-extraction path that bypasses the protocol's primary health gate for collateral operations.

---

### Likelihood Explanation

The condition is reachable by any unprivileged user who holds NLP tokens and has an open position whose `INITIAL` health has gone negative (a common scenario during moderate market moves on leveraged accounts). The `burnNlp` path is a standard user-facing transaction type routed through the `Endpoint`, requiring no special privileges. The gap between `INITIAL` and `MAINTENANCE` health thresholds is a normal operating band in the protocol, making the trigger realistic. [5](#0-4) 

---

### Recommendation

Replace the `MAINTENANCE` health check in `burnNlp` with `INITIAL` health, consistent with `withdrawCollateral`:

```diff
- require(
-     getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
-     ERR_SUBACCT_HEALTH
- );
+ require(
+     getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
+     ERR_SUBACCT_HEALTH
+ );
``` [2](#0-1) 

---

### Proof of Concept

1. User opens a leveraged perp position. Market moves against them; `INITIAL` health drops to −ε (slightly negative), `MAINTENANCE` health remains +δ (positive).
2. User holds NLP tokens minted earlier.
3. User submits a `BurnNlp` transaction through the `Endpoint`.
4. `burnNlp` executes: NLP balance decreases, quote balance increases by `quoteAmount - burnFee`.
5. Post-burn `MAINTENANCE` health check passes (≥ 0); `INITIAL` health is now more negative.
6. User has successfully extracted quote collateral despite having negative `INITIAL` health — a withdrawal that `withdrawCollateral` would have rejected at step 5 with `ERR_SUBACCT_HEALTH`.
7. Steps 3–6 can be repeated until NLP balance is exhausted or `MAINTENANCE` health approaches zero. [6](#0-5)
