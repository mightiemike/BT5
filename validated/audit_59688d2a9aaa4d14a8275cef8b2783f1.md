### Title
Malicious Linked Signer Can Irrevocably Maintain Access by Re-Linking via `LinkSigner` with `allowLinkedSigner = true` — (`core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`. This means the **current linked signer** — not just the subaccount owner — can sign a valid `LinkSigner` transaction that overwrites `linkedSigners[subaccount]` to any address. A malicious linked signer can exploit this to perpetually maintain signing authority over a victim's subaccount even after the owner attempts to revoke it, enabling ongoing theft of funds.

---

### Finding Description

`EndpointTx.sol` stores one linked signer per subaccount in `linkedSigners[subaccount]`. The linked signer is treated as a fully trusted delegate: it can sign withdrawals, trades, transfers, and any other sequencer-submitted transaction on behalf of the subaccount owner.

The fast-path handler for `LinkSigner` in `processTransactionImpl` is:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
```

`validateSignedTx` with `allowLinkedSigner = true` calls `validateCompactSignature`, which in turn calls `verifier.validateCompactSignature` passing `getLinkedSigner(sender)` as the permitted alternate signer. The verifier accepts the signature if it was produced by **either** the subaccount owner **or** the current linked signer:

```solidity
// Verifier.sol lines 312–318
address recovered = ECDSA.recover(digest, signature.r, signature.vs);
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
```

Therefore, the current linked signer can produce a valid `LinkSigner` transaction that replaces `linkedSigners[subaccount]` with any address — including a fresh address the attacker controls.

The slow-mode path for `LinkSigner` (lines 232–239) is **not** vulnerable because it uses `validateSender`, which checks `msg.sender == address(uint160(bytes20(txn.sender)))` — i.e., only the subaccount owner's EOA can submit it. The vulnerability is exclusive to the sequencer-submitted fast path.

---

### Impact Explanation

A malicious linked signer retains the ability to:

- Sign `WithdrawCollateral` transactions to drain the subaccount.
- Sign `TransferQuote` transactions to move funds to an attacker-controlled subaccount.
- Sign any other sequencer-submitted transaction on behalf of the victim.

Because the linked signer can keep re-linking themselves (or a chain of colluding addresses) every time the owner attempts to revoke, the owner cannot reliably terminate the attacker's access. The attacker can cause **indefinite, ongoing loss of funds** from the victim's subaccount.

---

### Likelihood Explanation

The attack requires only that:

1. The owner has previously linked a signer (a normal, documented use case).
2. That signer becomes malicious.

No privileged access, sequencer compromise, or governance capture is needed. The malicious linked signer submits a `LinkSigner` transaction to the sequencer before the owner's revocation transaction is processed. Because the sequencer processes transactions in submission order, and both the owner and the linked signer can submit transactions independently, the attacker can race the owner's revocation. The nonce is per-subaccount (not per-signer), so whichever `LinkSigner` transaction the sequencer processes first consumes nonce N; the other is rejected. The attacker can repeat this for every subsequent nonce the owner uses.

---

### Recommendation

Process `LinkSigner` transactions with `allowLinkedSigner = false` in the fast path, so only the subaccount owner's key can authorize a signer change:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

If signing by a linked signer is intentionally desired for key-rotation use cases, add a dedicated `RotateSigner` transaction type that is explicitly scoped and cannot be used to re-link an already-revoked signer.

---

### Proof of Concept

**State before attack:**
- Alice owns subaccount `aliceSub`. Nonce = N.
- Bob is `linkedSigners[aliceSub]`.

**Attack sequence:**

1. Alice signs and submits to the sequencer: `LinkSigner { sender: aliceSub, signer: bytes32(0), nonce: N }` (signed by Alice's key).
2. Bob, observing Alice's pending transaction, signs and submits to the sequencer: `LinkSigner { sender: aliceSub, signer: Eve, nonce: N }` (signed by Bob's key — accepted because `allowLinkedSigner = true` and Bob is currently `linkedSigners[aliceSub]`).
3. Sequencer processes Bob's transaction first. `linkedSigners[aliceSub] = Eve`. Nonce advances to N+1.
4. Alice's transaction is rejected (`ERR_WRONG_NONCE`).
5. Alice resubmits: `LinkSigner { sender: aliceSub, signer: bytes32(0), nonce: N+1 }`.
6. Eve (now the linked signer) signs and submits: `LinkSigner { sender: aliceSub, signer: Bob, nonce: N+1 }`.
7. Sequencer processes Eve's transaction first. `linkedSigners[aliceSub] = Bob`. Nonce advances to N+2.
8. Steps 5–7 repeat indefinitely.

Throughout this loop, Bob/Eve retain full signing authority over `aliceSub` and can execute `WithdrawCollateral` or `TransferQuote` transactions to steal Alice's funds.
