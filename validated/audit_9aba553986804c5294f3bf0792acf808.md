Audit Report

## Title
`OracleValueStopLossExtension` Uses Arithmetic Mean Instead of Geometric Mean for Mid Price, Miscalibrating the Stop-Loss Guard — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid price as the arithmetic mean of bid and ask at line 218, while the pool's canonical mid price is the geometric mean as defined by `SwapMath.midAndSpreadFeeX64FromBidAsk`. Because AM ≥ GM for all positive bid/ask pairs, the stop-loss always uses an inflated mid price relative to the pool's own swap execution price, causing watermarks to be set at wrong levels and the token1-outflow guard to be systematically less sensitive than the configured `drawdownE6` threshold.

## Finding Description
In `_afterSwapOracleStopLoss` (line 218):
```solidity
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;  // arithmetic mean
```
The pool's canonical mid price, used for all swap execution, is the geometric mean defined in `SwapMath.midAndSpreadFeeX64FromBidAsk` (line 70):
```solidity
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);
```
The sibling extension `PriceVelocityGuardExtension` (line 48) correctly calls `SwapMath.midAndSpreadFeeX64FromBidAsk` and even comments *"geometric mid of two uint128 bid/ask quotes"*.

The inflated AM mid price propagates into `_metrics` (lines 254–255):
- `metricT0`: the t1 component is divided by the inflated mid → deflated value
- `metricT1`: the t0 component is multiplied by the inflated mid → inflated value

In `_checkAndUpdateWatermarks` (lines 270–284):
- `hwm0` is ratcheted to the deflated `metricT0` → the drawdown floor `hwm0 * floorMultiplier / E6` is lower than it should be → the stop-loss for `zeroForOne == true` (token1 outflow) fires later than configured
- `hwm1` is ratcheted to the inflated `metricT1` → the floor is higher → the stop-loss for `zeroForOne == false` (token0 outflow) fires earlier than configured

The miscalibration is active on every swap where `bidPriceX64 ≠ askPriceX64` (any non-zero spread). No existing guard checks or corrects the mid price formula used in this extension.

## Impact Explanation
The `OracleValueStopLossExtension` is the primary on-chain mechanism protecting LP principal from value drain. Its stated invariant is that value per share at oracle marks cannot fall faster than `drawdownE6 + decay × t`. With the arithmetic mean, the effective drawdown threshold for token1 outflow is approximately `drawdownE6 + s² · E6`, where `s` is the oracle spread fraction. For a 5% configured drawdown and 10% oracle spread, the guard fires at ~5.25% actual drawdown instead of 5%, allowing additional token1 extraction beyond the configured limit. The discrepancy scales with `s²` and worsens during high-volatility periods when oracle spreads widen — exactly when stop-loss protection is most critical. For a $10M LP pool, this represents ~$25,000 of additional token1 that can be extracted before the guard triggers. The miscalibration is permanent and compounds across every watermark update.

## Likelihood Explanation
The bug is always active whenever `bidPriceX64 ≠ askPriceX64`, which is the normal operating condition for any pool with a non-zero oracle spread. No special attacker action is required; any ordinary swap through a pool with this extension attached will set watermarks using the wrong mid price. Any unprivileged trader executing a swap triggers the miscalibrated watermark update. The miscalibration is permanent and accumulates across every watermark update.

## Recommendation
Replace the arithmetic mean with `SwapMath.midAndSpreadFeeX64FromBidAsk`, consistent with the pool's canonical mid price and with `PriceVelocityGuardExtension`:
```solidity
// Before (wrong):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (correct):
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(
    uint256(bidPriceX64),
    uint256(askPriceX64)
);
```
This ensures the stop-loss evaluates bin value at the same mid price the pool uses for swap execution, making the configured `drawdownE6` threshold exact.

## Proof of Concept
**Setup:** Pool with 1,000 token0 and 1,000 token1 per bin, 1,000 shares. Oracle: `bid = 0.9·Q64`, `ask = 1.1·Q64` (10% spread). Configured drawdown = 5% (`drawdownE6 = 50_000`).

**Step 1 — First swap sets watermarks:**
```
AM mid  = (0.9 + 1.1)/2 · Q64 = 1.0 · Q64
GM mid  = √(0.9 · 1.1) · Q64 ≈ 0.9950 · Q64

metricT0 (AM) = 1000 + 1000·Q64/(1.0·Q64) = 2000   ← set too low
metricT0 (GM) = 1000 + 1000·Q64/(0.995·Q64) ≈ 2005  ← correct

hwm0 (AM) = 2000
hwm0 (GM) = 2005
```

**Step 2 — Token1 drains; stop-loss check:**
```
floor (AM) = 2000 · 0.95 = 1900
floor (GM) = 2005 · 0.95 = 1904.75
```
A `metricT0` of 1902 triggers the GM-based guard but **not** the AM-based guard. The pool continues to allow token1 outflow beyond the 5% drawdown limit.

A Foundry unit test can reproduce this by: (1) deploying the extension with `drawdownE6 = 50_000`, (2) calling `afterSwap` with `bid = 0.9·Q64`, `ask = 1.1·Q64` to set watermarks, (3) reducing t1 to bring `metricT0` to 1902, (4) calling `afterSwap` again with `zeroForOne = true` and verifying no revert occurs despite the breach exceeding the GM-correct floor.
