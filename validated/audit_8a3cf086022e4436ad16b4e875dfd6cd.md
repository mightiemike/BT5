### Title
`ValidResourceBounds` Variant Collapse in Protobuf Deserialization Silently Rejects Valid Consensus Transactions — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

`ValidResourceBounds::try_from(protobuf::ResourceBounds)` collapses an `AllResources` variant into `L1Gas` whenever `l2_gas` and `l1_data_gas` are both zero. A V3 transaction with `AllResourceBounds { l1_gas: non-zero, l2_gas: 0, l1_data_gas: 0 }` is accepted by the gateway and hashed as `AllResources` (L1_DATA_GAS included in the Poseidon preimage), but when the proposer re-serialises it over the consensus protobuf channel the receiving validator's deserialisation path collapses the variant to `L1Gas`, causing `RpcInvokeTransactionV3::try_from(InvokeTransactionV3)` to return an error. The validator therefore rejects the block proposal, breaking consensus liveness.

---

### Finding Description

**Step 1 — Gateway accepts the transaction and computes the hash as `AllResources`.**

`convert_rpc_tx_to_internal` in `crates/apollo_transaction_converter/src/transaction_converter.rs` converts the user-supplied `RpcInvokeTransactionV3` (which carries `resource_bounds: AllResourceBounds`) into `InternalRpcTransactionWithoutTxHash::Invoke`, then calls:

```rust
let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
``` [1](#0-0) 

`InternalRpcInvokeTransactionV3::calculate_transaction_hash` delegates to `get_invoke_transaction_v3_hash`, which calls `get_tip_resource_bounds_hash`:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
``` [2](#0-1) 

Because the internal type always wraps `AllResourceBounds` in `ValidResourceBounds::AllResources`, the L1_DATA_GAS term is included in the hash even when its value is zero.

**Step 2 — Proposer serialises the transaction to protobuf.**

`From<ConsensusTransaction> for protobuf::ConsensusTransaction` routes an Invoke V3 through:

```
RpcInvokeTransactionV3 → InvokeTransactionV3 → protobuf::InvokeV3
``` [3](#0-2) 

`From<InvokeTransactionV3> for protobuf::InvokeV3` serialises `resource_bounds` via `protobuf::ResourceBounds::from(value.resource_bounds)`. [4](#0-3) 

The proto schema marks `l1_data_gas` as `optional`:

```proto
message ResourceBounds {
    ResourceLimits l1_gas = 1;
    optional ResourceLimits l1_data_gas = 2;
    ResourceLimits l2_gas = 3;
}
``` [5](#0-4) 

**Step 3 — Validator deserialises and the variant collapses.**

`TryFrom<protobuf::InvokeV3> for InvokeTransactionV3` calls:

```rust
let resource_bounds = ValidResourceBounds::try_from(
    value.resource_bounds.ok_or(missing("InvokeV3::resource_bounds"))?,
)?;
``` [6](#0-5) 

`ValidResourceBounds::try_from(protobuf::ResourceBounds)` contains the collapsing logic:

```rust
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← collapse
} else {
    ValidResourceBounds::AllResources(...)
})
``` [7](#0-6) 

When `l2_gas = 0` and `l1_data_gas = 0` (or absent), the result is `L1Gas`.

**Step 4 — Conversion to `RpcInvokeTransactionV3` fails.**

`TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3` then calls `snapi_invoke.try_into()`: [8](#0-7) 

`TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` rejects anything that is not `AllResources`:

```rust
resource_bounds: match value.resource_bounds {
    ValidResourceBounds::AllResources(bounds) => bounds,
    _ => {
        return Err(StarknetApiError::OutOfRange { string: "resource_bounds".to_string() });
    }
},
``` [9](#0-8) 

The error propagates up through `convert_consensus_tx_to_internal_consensus_tx`, causing the validator to reject the block proposal.

---

### Impact Explanation

A single unprivileged user submitting a V3 invoke transaction with `AllResourceBounds { l1_gas: non-zero, l2_gas: 0, l1_data_gas: 0 }` causes the proposer's block to be rejected by every validator that deserialises the transaction over the consensus channel. The gateway's own stateless validator explicitly accepts this configuration (`valid_l1_gas` test case). [10](#0-9) 

This matches: **High — Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

---

### Likelihood Explanation

The trigger is a standard, well-formed V3 transaction. No special privileges, no malformed bytes, no out-of-range values. Any wallet or script that sets only `l1_gas` and leaves the other two bounds at zero (a common pattern for pre-0.13.3 style usage under the new type) will produce the triggering input. The gateway's own test suite confirms acceptance.

---

### Recommendation

1. **Fix the collapsing logic.** In `ValidResourceBounds::try_from(protobuf::ResourceBounds)`, do not infer the variant from zero values. Instead, use the presence/absence of the `optional l1_data_gas` field as the discriminant: if `l1_data_gas` is `Some(...)`, always produce `AllResources`; if it is `None`, produce `L1Gas`. This preserves the original intent of the `optional` annotation.

2. **Add a round-trip test** that serialises `AllResources { l1_gas: non-zero, l2_gas: 0, l1_data_gas: 0 }` through `protobuf::ConsensusTransaction` and back, asserting the variant is preserved and the hash is unchanged.

---
