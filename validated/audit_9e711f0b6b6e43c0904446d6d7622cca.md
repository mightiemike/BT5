Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any unprivileged caller to bypass the deposit allowlist â€” (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` who pays tokens) and checks only `owner` (the position beneficiary). Because `MetricOmmPool.addLiquidity` explicitly supports an operator pattern where `msg.sender â‰  owner`, any address not on the allowlist can deposit into a curated pool by passing an allowed address as `owner`, bypassing the pool admin's access control entirely.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` parameter as `owner` to `_beforeAddLiquidity`:

```solidity
// MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`removeLiquidity` enforces `msg.sender == owner` (L206), but `addLiquidity` has no such constraint, confirming the operator pattern is intentional for deposits. `ExtensionCalling._beforeAddLiquidity` faithfully forwards both actors to the extension (L95â€“98). However, `DepositAllowlistExtension.beforeAddLiquidity` discards `sender` (unnamed first parameter) and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol L32â€“38
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
```

An attacker who knows any address `allowedOwner` that is on the allowlist calls `pool.addLiquidity(allowedOwner, salt, deltas, callbackData, extensionData)`. The extension receives `(unauthorizedUser, allowedOwner, ...)`, evaluates `allowedDepositor[pool][allowedOwner] == true`, and does not revert. The unauthorized caller pays tokens via the `IMetricOmmModifyLiquidityCallback` callback; the position is attributed to `allowedOwner`. The inconsistency is confirmed by `SwapAllowlistExtension.beforeSwap`, which correctly checks `sender` (L37).

## Impact Explanation
The deposit allowlist is a pool-admin-configured guard controlling who can provide liquidity to a curated pool. By checking `owner` instead of `sender`, any unprivileged address can bypass this guard with zero special privileges. The pool admin's curation boundary is broken: the allowlist no longer controls who actually deposits tokens. This is a confirmed admin-boundary break â€” an unprivileged path bypasses a pool-admin-configured access control. The NatSpec of the contract itself states *"Gates `addLiquidity` by depositor address"* (L11), where the depositor is the token payer (`sender`), not the position owner.

## Likelihood Explanation
The operator pattern (`msg.sender â‰  owner`) is explicitly supported and documented in `IMetricOmmPoolActions`. Any user who can identify a single allowed address (discoverable on-chain from past `setAllowedToDeposit` events) can exploit this immediately with no special privileges, no oracle manipulation, and no complex setup. The exploit is repeatable and requires only a standard `addLiquidity` call.

## Recommendation
Check `sender` (the actual depositor/caller) instead of `owner`, consistent with `SwapAllowlistExtension` and the NatSpec intent:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
1. Pool admin deploys a pool with `DepositAllowlistExtension` on `beforeAddLiquidity`.
2. Pool admin calls `setAllowedToDeposit(pool, allowedOwner, true)`.
3. Unauthorized user (not on allowlist) calls `pool.addLiquidity(allowedOwner, salt, deltas, callbackData, extensionData)`.
4. Pool calls `_beforeAddLiquidity(msg.sender=unauthorizedUser, owner=allowedOwner, ...)` â†’ extension receives `(unauthorizedUser, allowedOwner, ...)`.
5. Extension evaluates `allowedDepositor[pool][allowedOwner]` â†’ `true` â†’ no revert.
6. Unauthorized user pays tokens via callback; position is attributed to `allowedOwner`.
