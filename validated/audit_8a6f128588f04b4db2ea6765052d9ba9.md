### Title
`ENABLE_SECP_OPS` Flag Check Missing from 4-Byte Opcode Dispatch Path — (`File: src/chia_dialect.rs`)

---

### Summary

`ChiaDialect::op()` contains two separate dispatch paths for secp signature-verification operators. The 1-byte opcode path (opcodes 64 and 65) correctly gates execution behind the `ENABLE_SECP_OPS` flag. The 4-byte opcode path (`0x13d61f00` / `0x1c3a8f00`) dispatches to the same underlying secp functions **without checking `ENABLE_SECP_OPS` at all**, making the flag bypassable by any attacker-controlled CLVM program.

---

### Finding Description

`src/chia_dialect.rs` implements two distinct dispatch paths inside `ChiaDialect::op()`.

**Path A — 1-byte opcodes (correctly gated):** [1](#0-0) 

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

Without the flag, opcodes 64 and 65 fall through to `unknown_operator`, which either errors (`NO_UNKNOWN_OPS`) or returns a no-op cost.

**Path B — 4-byte opcodes (flag check absent):** [2](#0-1) 

```rust
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => { return unknown_operator(...); }
};
return f(allocator, argument_list, max_cost, flags);
```

No `ENABLE_SECP_OPS` check exists here. The secp functions are called unconditionally for these two 4-byte opcodes, regardless of the dialect flags the caller configured.

The flag is defined with the explicit intent to gate secp: [3](#0-2) 

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
```

The companion guard that is present in Path A is entirely absent from Path B — a direct structural analog to the external report's missing `nominateNewDependencyOwner()` call.

---

### Impact Explanation

Any caller that constructs a `ChiaDialect` **without** `ENABLE_SECP_OPS` — for example, a node running before the secp soft-fork has activated, or a mempool validator that deliberately omits the flag — still executes full secp signature verification when a CLVM program uses the 4-byte opcode encoding. `MEMPOOL_MODE` does not include `ENABLE_SECP_OPS`: [4](#0-3) 

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
```

A CLVM program using `0x13d61f00` / `0x1c3a8f00` will execute secp verification in mempool mode even though the 1-byte equivalents (64, 65) would be rejected. This produces **consensus divergence**: the same spend bundle is accepted by one code path and rejected by another, depending solely on which opcode encoding the attacker chose. The Python API exposes `ENABLE_SECP_OPS` as a user-facing flag, so any Python caller relying on it to disable secp is silently unprotected. [5](#0-4) 

---

### Likelihood Explanation

The trigger is entirely attacker-controlled CLVM bytes. An attacker needs only to encode the secp opcode as a 4-byte atom (`\x13\xd6\x1f\x00` or `\x1c\x3a\x8f\x00`) instead of the 1-byte form (`\x40` / `\x41`). No special privileges, configuration, or social engineering are required. The 4-byte encoding is valid CLVM and passes all serialization checks.

---

### Recommendation

Add the `ENABLE_SECP_OPS` guard to the 4-byte dispatch path, mirroring the 1-byte path:

```rust
// in the op_len == 4 branch:
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures both opcode encodings are governed by the same flag, eliminating the bypass.

### Proof of Concept

```

; 4-byte opcode 0x13d61f00 → executes op_secp256k1_verify unconditionally

; Attacker-crafted program using 4-byte opcode atom:
(0x13d61f00 pubkey msg sig)
```

A CLVM program whose operator atom is the 4-byte value `\x13\xd6\x1f\x00` reaches `ChiaDialect::op()`, enters the `op_len == 4` branch at line 157, matches `0x13d61f00`, and calls `op_secp256k1_verify` directly — bypassing the `ENABLE_SECP_OPS` check that would have blocked the equivalent 1-byte opcode 64.
