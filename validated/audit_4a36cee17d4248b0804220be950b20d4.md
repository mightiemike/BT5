### Title
`DepositAllowlistExtension` gates on LP position `owner` instead of the actual depositor `sender`, allowing any unprivileged user to bypass the deposit allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary
`DepositAllowlistExtension.beforeAddLiquidity()` silently discards the `sender` parameter (the actual depositor/caller) and checks only the `owner` parameter (the LP position recipient). Any user can bypass the deposit allowlist by routing through `MetricOmmPoolLiquidityAdder` with an allowlisted address as `owner`, breaking the pool admin's curation policy via an unprivileged path.

---

### Finding Description

The contract is documented as *"Gates `addLiquidity` by depositor address, per pool."* However, the `beforeAddLiquidity` implementation discards the first parameter (`sender`) entirely and checks only `owner`: [1](#0-0) 

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The first `address` parameter — `sender` — is unnamed and completely ignored.

In `MetricOmmPool.addLiquidity()`, the pool passes `msg.sender` as `sender` and the caller-supplied `owner` as the LP position owner: [2](#0-1) 

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

Because the extension ignores `sender` and only checks `owner`, any non-allowlisted user B can:

1. Call `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner=A, ...)` where A is an allowlisted address.
2. The adder calls `pool.addLiquidity(owner=A, ...)`. [3](#0-2) 

3. The pool calls `_beforeAddLiquidity(sender=adder, owner=A, ...)`.
4. The extension checks `allowedDepositor[pool][A]` → `true` → passes.
5. B pays the tokens (via the adder's callback pulling from `payer = msg.sender = B`), A receives the LP position. [4](#0-3) 

The `addLiquidityExactShares(pool, owner, ...)` overload explicitly supports a distinct `owner` from `msg.sender`, and the test suite confirms this path is reachable: [5](#0-4) 

The allowlist is completely bypassed. No admin cooperation is required.

---

### Impact Explanation

The pool admin's deposit allowlist policy — intended to enforce KYC, regulatory compliance, or LP curation — is bypassed by any unprivileged user. A non-allowlisted user can add liquidity to a curated pool by specifying any allowlisted address as `owner`. The non-allowlisted user pays the tokens; the allowlisted user receives the LP position and can subsequently withdraw the deposited assets. This is an admin-boundary break: a guard configured by the pool admin is bypassed via a supported periphery path without any privileged action.

---

### Likelihood Explanation

High. The `MetricOmmPoolLiquidityAdder` is the standard periphery deposit path and explicitly supports an `owner` distinct from `msg.sender`. Any user who knows an allowlisted address (which may be publicly observable on-chain) can execute the bypass immediately.

---

### Recommendation

Check `sender` (the actual depositor/caller) instead of `owner` (the LP position recipient):

```solidity
// In DepositAllowlistExtension.sol
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This matches the documented intent ("Gates `addLiquidity` by depositor address") and mirrors the correct actor binding used in `SwapAllowlistExtension`, which checks `sender`: [6](#0-5)

### Proof of Concept

```
Setup:
  - Pool deployed with DepositAllowlistExtension
  - Pool admin calls setAllowedToDeposit(pool, alice, true)
  - Bob (not allowlisted) holds token0 and token1, approved to MetricOmmPoolLiquidityAdder

Attack:
  vm.prank(bob);
  liquidityAdder.addLiquidityExactShares(
      pool,
      alice,   // owner = allowlisted address
      salt,
      deltas,
      maxAmount0,
      maxAmount1,
      ""
  );
  // Extension checks allowedDepositor[pool][alice] == true → passes
  // Bob pays tokens, alice receives LP shares
  // Allowlist bypassed without any admin action
