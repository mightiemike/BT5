### Title
Kickout-Exemption Sorting Uses Continuous Endorsement Ratio While Reward Calculation Uses Binary Cutoff-Gated Ratio — (`chain/epoch-manager/src/validator_stats.rs`)

---

### Summary

`get_sortable_validator_online_ratio` — the function that determines which validators are **protected from kickout** — always calls `get_validator_online_ratio(stats, None)`, hardcoding the `endorsement_cutoff_threshold` to `None`. However, `calculate_reward` calls `get_validator_online_ratio(stats, online_thresholds.endorsement_cutoff_threshold)` with the actual configured threshold. When `endorsement_cutoff_threshold` is set to a non-`None` value, the two functions compute fundamentally different online ratios for the same validator, causing the wrong validators to be exempted from kickout.

---

### Finding Description

The epoch-end validator lifecycle has two distinct computations that must agree on what "online ratio" means:

1. **Kickout-exemption sorting** (`compute_validators_to_reward_and_kickout`, line 500–506): sorts validators by online ratio to determine which ones are shielded from kickout (up to `validator_max_kickout_stake_perc` of total stake). It calls `get_sortable_validator_online_ratio`, which internally calls `get_validator_online_ratio(stats, None)`.

2. **Reward calculation** (`calculate_reward`, line 96): computes each validator's uptime multiplier using `get_validator_online_ratio(stats, online_thresholds.endorsement_cutoff_threshold)`.

When `endorsement_cutoff_threshold` is `Some(T)`, the two functions diverge:

- **Sorting path** (`None`): endorsement contribution = `produced / expected` (continuous, e.g. 49/100 = 0.49)
- **Reward path** (`Some(T)`): endorsement contribution = `0` if ratio < T, else `1` (binary)

A validator with endorsement ratio just below the cutoff (e.g. 49% with cutoff 50%) has a **higher** continuous ratio than a validator with lower block/chunk production but endorsement ratio just above the cutoff (e.g. 51%). The first validator is sorted higher and exempted from kickout, while the second is kicked out — even though the second validator has a higher effective online ratio for reward purposes.

The exact divergent line is in `get_sortable_validator_online_ratio`:

```rust
pub(crate) fn get_sortable_validator_online_ratio(stats: &BlockChunkValidatorStats) -> BigRational {
    let ratio = get_validator_online_ratio(stats, None);  // cutoff always ignored
    ...
}
``` [1](#0-0) 

While `calculate_reward` uses the actual threshold:

```rust

    get_validator_online_ratio(&stats, online_thresholds.endorsement_cutoff_threshold);
``` [2](#0-1) 

The sorted list is then fed directly into `compute_exempted_kickout`: [3](#0-2)

### Impact Explanation

The kickout-exemption mechanism is designed to protect the validators with the **highest effective online ratio** from being kicked out during network instability. When `endorsement_cutoff_threshold` is active, the exemption ordering is computed with a different metric than the reward ordering. This means:

- A validator with endorsement ratio just below the cutoff (e.g. 49%) is sorted **higher** (continuous ratio 0.49 > 0) and exempted from kickout, yet receives **zero** endorsement contribution to their reward.
- A validator with endorsement ratio just above the cutoff (e.g. 51%) may be sorted **lower** (if their block/chunk production is lower) and kicked out, yet would have received **full** endorsement contribution to their reward.

The protocol's validator selection invariant is broken: the validators most deserving of protection (highest effective online ratio) are not necessarily the ones protected. A validator can deliberately keep their endorsement ratio just below the cutoff to maximize their sorting score (continuous ratio) while minimizing their actual contribution, staying in the validator set while contributing less.

**Severity: High** — breaks the deterministic validator kickout/exemption invariant at every epoch boundary where `endorsement_cutoff_threshold` is non-`None`.

---

### Likelihood Explanation

`endorsement_cutoff_threshold` is a configurable field in `ValidatorOnlineThresholds`, which is derived from `EpochConfig`. The test `test_reward_stateless_validation_with_endorsement_cutoff` demonstrates it being set to `Some(50)`. Any deployment that enables this threshold (e.g. for stateless validation epochs) triggers the divergence. No privileged action beyond normal validator operation is required to exploit the ordering mismatch — a validator simply needs to maintain an endorsement ratio just below the configured cutoff. [4](#0-3) 

---

### Recommendation

Pass the `endorsement_cutoff_threshold` through to `get_sortable_validator_online_ratio` so that the sorting metric matches the reward metric:

```rust
pub(crate) fn get_sortable_validator_online_ratio(
    stats: &BlockChunkValidatorStats,
    endorsement_cutoff_threshold: Option<u8>,
) -> BigRational {
    let ratio = get_validator_online_ratio(stats, endorsement_cutoff_threshold);
    ...
}
```

Update the call site in `compute_validators_to_reward_and_kickout` to pass the configured threshold, and update `get_sortable_validator_online_ratio`'s internal call to `get_validator_online_ratio` accordingly. This ensures the exemption ordering is consistent with the reward ordering. [5](#0-4)

### Proof of Concept

Assume `endorsement_cutoff_threshold = Some(50)`, `online_min_threshold = 0.9`, `validator_max_kickout_stake_perc = 30` (so 70% of stake is exempted):

**Validator A**: blocks=100%, chunks=100%, endorsements=49% (below cutoff), stake=500
- Sorting ratio (no cutoff): `(1.0 + 1.0 + 0.49) / 3 = 0.830` → **sorted higher, exempted**
- Reward ratio (with cutoff): `(1.0 + 1.0 + 0.0) / 3 = 0.667`

**Validator B**: blocks=70%, chunks=70%, endorsements=51% (above cutoff), stake=500
- Sorting ratio (no cutoff): `(0.70 + 0.70 + 0.51) / 3 = 0.637` → **sorted lower, kicked out**
- Reward ratio (with cutoff): `(0.70 + 0.70 + 1.0) / 3 = 0.800`

Outcome: Validator A (effective reward ratio 0.667) is exempted from kickout. Validator B (effective reward ratio 0.800) is kicked out. The protocol's intent — protect the most productive validators — is inverted. Validator A remains in the set while contributing less to the network's security and liveness than Validator B would have. [6](#0-5) [1](#0-0)

/// Computes the overall online (uptime) ratio of the validator for sorting.
/// The reason for this function is that U256 used in the core implementation
/// cannot be used with `Ratio<U256>` for sorting since it does not implement `num_integer::Integer`.
/// Instead of having a full-blown implementation of `U256`` for `num_integer::Integer`
/// we wrap the value in a `BigInt` for now.
/// TODO: Implement `num_integer::Integer` for `U256` and remove this function.
/// cspell:words bigdenom bignumer
pub(crate) fn get_sortable_validator_online_ratio(stats: &BlockChunkValidatorStats) -> BigRational {
    let ratio = get_validator_online_ratio(stats, None);
    let mut bytes: [u8; size_of::<U256>()] = [0; size_of::<U256>()];
    ratio.numer().to_little_endian(&mut bytes);
    let bignumer = BigUint::from_bytes_le(&bytes);
    ratio.denom().to_little_endian(&mut bytes);
    let bigdenom = BigUint::from_bytes_le(&bytes);
    BigRational::new(bignumer.try_into().unwrap(), bigdenom.try_into().unwrap())
}
```
