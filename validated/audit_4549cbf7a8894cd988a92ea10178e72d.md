### Title
Proposer-controlled L1 gas price ratio produces an unvalidated `l2_gas_price_wei` committed to the block hash — (`crates/apollo_consensus_orchestrator/src/utils.rs`)

### Summary

`convert_to_sn_api_block_info` derives `l2_gas_price_wei` by back-calculating the ETH→STRK rate from the proposer-supplied `l1_gas_price_fri` and `l1_gas_price_wei` fields, then applying that rate to `l2_gas_price_fri`. The validator independently checks each of the four L1 price fields within a percentage margin, but never validates the **ratio** between them, nor the resulting `l2_gas_price_wei`. A malicious proposer can therefore skew the implied ETH→STRK rate by simultaneously pushing `l1_gas_price_fri` to the top of its allowed band and `l1_gas_price_wei` to the bottom, producing a `l2_gas_price_wei` that is systematically wrong. This wrong value is then committed into the block hash via `gas_prices_to_hash` and used as `eth_gas_prices.l2_gas_price` in `BlockInfo` for execution.

### Finding Description

**Step 1 – Rate back-calculation from proposer-supplied fields.**

`convert_to_sn_api_block_info` calls `calculate_eth_to_fri_rate` on a `PreviousProposalInitInfo` built directly from the incoming `ProposalInit`:

```
eth_to_fri_rate = l1_gas_price_fri * WEI_PER_ETH / l1_gas_price_wei
``` [1](#0-0) 

Both `l1_gas_price_fri` and `l1_gas_price_wei` come from the proposer's wire message.

**Step 2 – `l2_gas_price_wei` derived from that rate.**

```
l2_gas_price_wei = l2_gas_price_fri.fri_to_wei(eth_to_fri_rate)
                 = l2_gas_price_fri * WEI_PER_ETH / eth_to_fri_rate
                 = l2_gas_price_fri * l1_gas_price_wei / l1_gas_price_fri
``` [2](#0-1) 

**Step 3 – Validator checks each L1 price field independently, not their ratio.**

`is_proposal_init_valid` calls `get_l1_prices_in_fri_and_wei` to obtain the validator's own oracle values, then checks all four fields (`l1_gas_price_fri`, `l1_data_gas_price_fri`, `l1_gas_price_wei`, `l1_data_gas_price_wei`) independently within `l1_gas_price_margin_percent`. There is no check on the ratio `l1_gas_price_fri / l1_gas_price_wei`, and `l2_gas_price_wei` is never validated at all. [3](#0-2) 

**Step 4 – Wrong `l2_gas_price_wei` committed to block hash.**

`PartialBlockHashComponents::new` reads `block_info.gas_prices.l2_gas_price_per_token()`, which includes the manipulated `price_in_wei`. `gas_prices_to_hash` (for versions ≥ 0.13.4) chains `l2_gas_price.price_in_wei` into the Poseidon hash that becomes part of the canonical block hash. [4](#0-3) [5](#0-4) 

**Quantifying the manipulation.**

With `l1_gas_price_margin_percent = 10` (confirmed in test comments):

| Proposer action | Effect on derived rate |
|---|---|
| `l1_gas_price_fri` at +10 % | rate × 1.10 |
| `l1_gas_price_wei` at −10 % | rate × (1/0.90) |
| Combined | rate × 1.10/0.90 ≈ ×1.222 |

This makes `l2_gas_price_wei` ≈ **18 % lower** than the honest value (1/1.222 of the correct amount). The proposer can also go the other direction to inflate it by the same factor.

### Impact Explanation

`l2_gas_price_wei` is committed into the block hash for every block at Starknet version ≥ 0.13.4. A malicious proposer can cause the canonical block hash to commit a `l2_gas_price_wei` that diverges from the true market rate by up to ~22 %. This constitutes **wrong state committed on-chain** (wrong value in the block hash). Additionally, `eth_gas_prices.l2_gas_price` is available to the blockifier and to contracts via `get_execution_info` syscall; any contract or fee logic that reads the ETH-denominated L2 gas price will receive the manipulated value, enabling incorrect fee accounting for ETH-paying transactions that consume L2 gas.

### Likelihood Explanation

Any consensus participant that wins a proposer slot can trigger this. No special privilege beyond being a scheduled proposer is required. The manipulation is bounded by the margin but is systematic and repeatable every block the attacker proposes.

### Recommendation

Validate `l2_gas_price_wei` directly in `is_proposal_init_valid` rather than deriving it from the proposer-supplied L1 price ratio. The validator already has its own oracle-derived `eth_to_fri_rate`; it should compute the expected `l2_gas_price_wei` as `l2_gas_price_fri.fri_to_wei(validator_eth_to_fri_rate)` and check the proposed value is within the same margin. Alternatively, remove `l2_gas_price_wei` from `ProposalInit` entirely and have every node derive it locally from its own oracle rate, so the proposer cannot influence it at all.

### Proof of Concept

1. Validator's oracle returns: `l1_gas_price_fri = 1000`, `l1_gas_price_wei = 100`, `l2_gas_price_fri = 500`. Honest `eth_to_fri_rate = 1000 × 10^18 / 100 = 10^19`. Honest `l2_gas_price_wei = 500 × 10^18 / 10^19 = 50`.

2. Malicious proposer sends `ProposalInit` with `l1_gas_price_fri = 1100` (+10 %, within margin), `l1_gas_price_wei = 90` (−10 %, within margin), `l2_gas_price_fri = 500` (exact match, required).

3. `is_proposal_init_valid` passes: all four L1 price fields are within the 10 % band.

4. `convert_to_sn_api_block_info` computes `eth_to_fri_rate = 1100 × 10^18 / 90 ≈ 1.222 × 10^19`. Then `l2_gas_price_wei = 500 × 10^18 / (1.222 × 10^19) ≈ 40.9` instead of the honest `50`.

5. `gas_prices_to_hash` commits `l2_gas_price.price_in_wei = 40` (truncated) into the block hash. The canonical block hash now encodes a `l2_gas_price_wei` that is ~18 % below the true market rate. Any contract reading `get_execution_info` for the ETH-denominated L2 gas price, or any fee calculation using `eth_gas_prices.l2_gas_price`, receives the manipulated value.
