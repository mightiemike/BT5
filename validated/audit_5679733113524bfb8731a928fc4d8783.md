### Title
CEI Violation in `withdrawCollateral` Enables Reentrancy-Based Double Withdrawal — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` performs the full token transfer to the recipient **before** updating the subaccount balance in `SpotEngine`. This violates the checks-effects-interactions (CEI) pattern. If the collateral token supports transfer callbacks (e.g., ERC777 `tokensReceived`), a malicious recipient contract can re-enter the withdrawal flow during the callback window, observe a stale (pre-deduction) balance, pass the health check, and drain a second withdrawal — while the first call's `updateBalance` has not yet executed.

---

### Finding Description

In `Clearinghouse.withdrawCollateral` (lines 391–421), the execution order is:

1. **Line 408** — `handleWithdrawTransfer(token, sendTo, amount, idx)` is called. This internally calls `token.safeTransfer(withdrawPool, amount)` and then `BaseWithdrawPool.submitWithdrawal(...)`, which in turn calls `handleWithdrawTransfer(token, sendTo, amount)` → `token.safeTransfer(sendTo, amount)`. Tokens reach the user here.

2. **Line 412** — `spotEngine.updateBalance(productId, sender, amountRealized)` is called. The subaccount balance is decremented only after the transfer completes.

3. **Line 419** — `require(getHealth(sender, healthType) >= 0, ...)` — health check runs on the now-updated balance. [1](#0-0) 

The `handleWithdrawTransfer` override in `Clearinghouse` routes through `BaseWithdrawPool.submitWithdrawal`, which calls `handleWithdrawTransfer(token, sendTo, amount)` — a `virtual` internal function that executes `token.safeTransfer(to, amount)` directly to the user-controlled `sendTo` address. [2](#0-1) [3](#0-2) [4](#0-3) 

The `safeTransfer` implementation uses a low-level `.call(...)`, which forwards all gas and will invoke any `receive`/fallback or ERC777 `tokensReceived` hook on the recipient. [5](#0-4) 

The `WithdrawCollateralV2` transaction type allows the user to specify an arbitrary `sendTo` address, making the recipient fully attacker-controlled. [6](#0-5) 

There is no `nonReentrant` guard visible on `withdrawCollateral` or on the Endpoint transaction processing functions (`processTransactionImpl`, `processSlowModeTransactionImpl`). [7](#0-6) 

---

### Impact Explanation

During the token transfer callback window (step 1 above), `spotEngine` still holds the **original pre-withdrawal balance** for `sender`. A re-entrant call to `withdrawCollateral` (routed through the Endpoint) will:

- Read the stale balance from `SpotEngine`
- Pass the health check (`getHealth >= 0`) using the un-decremented balance
- Execute a second full token transfer to the attacker

When control returns to the outer call, `spotEngine.updateBalance` executes once, decrementing the balance by only one withdrawal amount. The attacker has received two withdrawals but the accounting reflects only one deduction — a net theft of `amount` tokens per re-entry cycle.

**Corrupted state delta**: `SpotEngine` balance for `sender` is decremented by `amountRealized` once, but `amount` tokens have been transferred out twice. The protocol's token holdings are drained by `amount` beyond what accounting permits.

---

### Likelihood Explanation

- The `WithdrawCollateralV2` path allows any user to specify a custom `sendTo` contract address, making the callback trigger fully attacker-controlled.
- ERC777 tokens and tokens with transfer hooks are realistic collateral types in a DEX supporting multiple spot products.
- The attacker only needs: (a) a collateral balance sufficient to pass the health check once, (b) a `sendTo` contract with a `tokensReceived` hook, and (c) a second withdrawal transaction queued (slow mode) or a pre-signed fast withdrawal.
- No privileged access, governance capture, or sequencer compromise is required.

---

### Recommendation

Apply the CEI pattern strictly in `withdrawCollateral`: perform all state updates (`spotEngine.updateBalance`, `assertUtilization`, health check) **before** calling `handleWithdrawTransfer`. Additionally, add a `nonReentrant` modifier (OpenZeppelin `ReentrancyGuardUpgradeable`) to `withdrawCollateral` and to the Endpoint's transaction processing entry points as defense-in-depth.

```solidity
// Corrected order:
spotEngine.updateBalance(productId, sender, amountRealized);
spotEngine.assertUtilization(productId);
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
handleWithdrawTransfer(token, sendTo, amount, idx); // external call last
```

---

### Proof of Concept

1. Attacker deploys `MaliciousRecipient` with a `tokensReceived` hook that calls `Endpoint.processSlowModeTransaction` for a second pre-queued withdrawal.
2. Attacker deposits collateral sufficient to pass one health check.
3. Attacker submits two slow-mode `WithdrawCollateralV2` transactions specifying `sendTo = address(MaliciousRecipient)`.
4. Sequencer (or attacker after delay) processes the first slow-mode withdrawal.
5. `Clearinghouse.withdrawCollateral` → `handleWithdrawTransfer` → `BaseWithdrawPool.submitWithdrawal` → `token.safeTransfer(MaliciousRecipient, amount)`.
6. `MaliciousRecipient.tokensReceived` fires; calls `Endpoint.processSlowModeTransaction` for the second queued withdrawal.
7. Re-entrant `Clearinghouse.withdrawCollateral` executes: `SpotEngine` balance is still the original value → health check passes → second `amount` transferred out.
8. Re-entrant call returns; outer call resumes at line 412, decrementing balance by `amountRealized` once.
9. Net result: attacker received `2 × amount` tokens; `SpotEngine` balance decremented by only `1 × amountRealized`. [8](#0-7) [3](#0-2)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
