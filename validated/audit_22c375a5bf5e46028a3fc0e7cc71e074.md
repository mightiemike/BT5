### Title
`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` Silently Downgrades `AllResources` to `L1Gas`, Causing Transaction Hash Divergence Across the Protobuf Boundary - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a zero-value heuristic to decide between `L1Gas` and `AllResources`. When a V3 Invoke transaction carries `AllResources` bounds with `l2_gas = 0` and `l1_data_gas = 0`, the deserializer silently reconstructs it as `ValidResourceBounds::L1Gas`. Because `get_tip_resource_bounds_hash` hashes a different number of resource elements for `L1Gas` (2) versus `AllResources` (3), the transaction hash recomputed on the receiving side of the protobuf boundary differs from the hash computed on the originating side. In the consensus path, the validator recomputes the hash from the deserialized transaction; the resulting hash diverges from the proposer's hash, binding the wrong hash to the executable payload.

### Finding Description

**Step 1 – Lossy protobuf deserializer.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` at line 431 applies a heuristic:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

Any `AllResources` transaction whose `l2_gas` and `l1_data_gas` are both zero is reconstructed as `L1Gas`. The original variant is not encoded in the wire format, so the information is permanently lost.

**Step 2 – Hash function branches on the variant.**

`get_tip_resource_bounds_hash` conditionally appends the `L1_DATA_GAS` element only for `AllResources`:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
``` [2](#0-1) 

`L1Gas` hashes `[tip, L1_GAS_packed, L2_GAS_packed(0)]` — two resource elements.  
`AllResources` with zero l2/l1_data hashes `[tip, L1_GAS_packed, 0, 0]` — three resource elements.  
These produce distinct Poseidon digests even when all numeric values are identical.

**Step 3 – Originating side always uses `AllResources`.**

`RpcInvokeTransactionV3` and `InternalRpcInvokeTransactionV3` both carry `resource_bounds: AllResourceBounds` (never `ValidResourceBounds`). Their `InvokeTransactionV3Trait::resource_bounds()` always returns `ValidResourceBounds::AllResources(...)`:

```rust
fn resource_bounds(&self) -> ValidResourceBounds {
    ValidResourceBounds::AllResources(self.resource_bounds)
}
``` [3](#0-2) 

So the hash computed at the gateway and stored in `InternalRpcTransaction.tx_hash` always uses the `AllResources` branch.

**Step 4 – Consensus path recomputes the hash after protobuf deserialization.**

`convert_rpc_tx_to_internal` (called by `convert_consensus_tx_to_internal_consensus_tx`) recomputes the hash from the deserialized fields:

```rust
let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
Ok((InternalRpcTransaction { tx: tx_without_hash, tx_hash }, proof_data))
``` [4](#0-3) 

If the deserialization path for `ConsensusTransaction` passes through `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` and then through `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` (which rejects `L1Gas` bounds), the validator either rejects the transaction outright or computes a divergent hash — neither of which matches the proposer's `tx_hash`.

**Step 5 – `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` rejects `L1Gas`.**

```rust
resource_bounds: match value.resource_bounds {
    ValidResourceBounds::AllResources(bounds) => bounds,
    _ => {
        return Err(StarknetApiError::OutOfRange {
            string: "resource_bounds".to_string(),
        });
    }
},
``` [5](#0-4) 

A transaction that was valid at the gateway (accepted with `AllResources` bounds, zero l2/l1_data) is rejected or mis-hashed when the validator reconstructs it from the proposer's protobuf stream.

### Impact Explanation

**High — Transaction conversion or signature/hash logic binds the wrong hash or executable payload.**

A proposer includes a valid V3 Invoke transaction (zero l2_gas, zero l1_data_gas, non-zero l1_gas) in a proposal. The proposer's `tx_hash` is computed with `AllResources` (3-element resource hash). The validator deserializes the same transaction from protobuf; the `ValidResourceBounds` heuristic reconstructs it as `L1Gas` (2-element resource hash). The validator either rejects the transaction (causing the proposal to be marked invalid) or stores a different `tx_hash` than the proposer, breaking consensus agreement on the transaction set.

### Likelihood Explanation

Any V3 Invoke transaction that sets `l2_gas = 0` and `l1_data_gas = 0` (a common pattern for transactions that only consume L1 gas) triggers this path. No special privileges or malformed bytes are required — the transaction is accepted by the gateway and enters the mempool normally. The divergence is triggered automatically during consensus proposal propagation.

### Recommendation

Remove the zero-value heuristic from `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. The protobuf schema should encode the variant explicitly (e.g., a boolean `is_all_resources` flag, or by making `l1_data_gas` a required field for `AllResources`). Alternatively, the consensus deserialization path for `RpcInvokeTransactionV3` should bypass `ValidResourceBounds` entirely and always reconstruct `AllResourceBounds` directly from the three wire fields, since the RPC/consensus layer only accepts V3 transactions with `AllResources` bounds.

### Proof of Concept

```

       l1_gas: { max_amount: 1000, max_price_per_unit: 1 },
       l2_gas: { max_amount: 0, max_price_per_unit: 0 },
       l1_data_gas: { max_amount: 0, max_price_per_unit: 0 },

2. Gateway computes tx_hash H1 using AllResources path:
   resource_felts = [L1_GAS_packed, L2_GAS_packed(0), L1_DATA_GAS_packed(0)]  // 3 elements

3. Proposer serializes to protobuf::ResourceBounds:
   { l1_gas: ..., l2_gas: {0,0}, l1_data_gas: {0,0} }

5. Validator recomputes hash H2 using L1Gas path:
   resource_felts = [L1_GAS_packed, L2_GAS_packed(0)]  // 2 elements
   tip_resource_bounds_hash = poseidon([tip, L1_GAS_packed, 0])

6. H1 ≠ H2 → validator rejects the transaction or stores wrong tx_hash

// An implementation of the SNIP: https://github.com/EvyatarO/SNIPs/blob/snip-8/SNIPS/snip-8.md
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
