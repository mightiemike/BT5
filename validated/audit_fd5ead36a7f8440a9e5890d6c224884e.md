### Title
Stale-pending-data fallback zeroes `l1_data_gas_price` and `l2_gas_price`, causing `estimate_fee`/`simulate_transactions` to return underestimated fees for Pending block — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary
`read_pending_data` constructs a `DeprecatedPendingBlock` fallback that omits `l1_data_gas_price` and `l2_gas_price` from the latest accepted header. Both accessors return zero for the `Deprecated` variant. Any unprivileged client calling `estimate_fee` or `simulate_transactions` with `BlockId::Tag(Tag::Pending)` during the normal inter-block window receives a fee estimate computed with zero data-gas and L2-gas prices.

### Finding Description
`read_pending_data` (api_impl.rs 1553-1595) has two branches:

- **Fresh path** (line 1570): returns the live `PendingData` as-is; `l1_data_gas_price` and `l2_gas_price` come from the actual pending block.
- **Stale fallback** (lines 1572-1594): constructs `PendingBlockOrDeprecated::Deprecated(DeprecatedPendingBlock { eth_l1_gas_price, strk_l1_gas_price, … })`. `DeprecatedPendingBlock` has no `l1_data_gas_price` or `l2_gas_price` fields. The `l1_data_gas_price()` and `l2_gas_price()` methods on the `Deprecated` arm return `GasPricePerToken::default()` (zero) by design for old-protocol blocks, but the fallback is also used for modern (≥ 0.13.3) chains.

The stale condition is triggered whenever the pending-data feed lags behind the latest accepted block — a routine occurrence between block finalization and the next pending-block broadcast.

### Impact Explanation
Under Starknet ≥ 0.13.3, `l1_data_gas_price` and `l2_gas_price` are nonzero components of the total fee. When both are zero in the block context passed to the blockifier's fee estimator, the data-gas and L2-gas cost components of the returned fee estimate are zero. Clients that rely on `estimate_fee` to set resource bounds will submit transactions with insufficient bounds, causing them to be rejected or under-charged depending on sequencer policy. This matches the allowed impact: **"High. RPC execution, fee estimation … returns an authoritative-looking wrong value."**

### Likelihood Explanation
The stale window is a normal, recurring operational state on every node. No special privileges, no malformed input, and no race condition beyond the ordinary block-production cadence are required. Any client polling `estimate_fee(…, BlockId::Tag(Tag::Pending))` at moderate frequency will hit the fallback regularly.

### Recommendation
In the stale fallback, copy `l1_data_gas_price` and `l2_gas_price` from `latest_header.block_header_without_hash` into the synthetic block, exactly as `l1_gas_price` is already copied. Alternatively, construct a `PendingBlockOrDeprecated::Current(PendingBlock { … })` fallback that carries all three price fields, gated on the latest header's `starknet_version` being ≥ 0.13.3.

### Proof of Concept
1. Seed storage with a block whose header has nonzero `l1_data_gas_price` (e.g., `price_in_wei = 1000`) and `l2_gas_price` (e.g., `price_in_wei = 2000`).
2. Set the in-memory `PendingData` to a block whose `parent_block_hash` does **not** match the stored block's hash (simulating stale data).
3. Call `estimate_fee` with `BlockId::Tag(Tag::Pending)` for a transaction that consumes data gas.
4. Assert that the returned fee accounts for nonzero data-gas cost — the assertion will fail because the fallback supplies zero for both `l1_data_gas_price` and `l2_gas_price`, producing a fee estimate that omits those components entirely.
