### Title
Linked Signer Can Overwrite `linkedSigners[sender]` to Attacker-Controlled Address — (`core/contracts/EndpointTx.sol`)

---

### Summary

`processTransactionImpl` handles `LinkSigner` transactions with `allowLinkedSigner=true`, meaning the currently registered hot wallet can sign a `LinkSigner` transaction that replaces itself with an attacker-controlled address. The invariant that only the subaccount owner (`address(uint160(bytes20(sender)))`) may mutate `linkedSigners[sender]` is broken.

---

### Finding Description

In `processTransactionImpl`, the `LinkSigner` branch decodes a `SignedLinkSigner` and calls `validateSignedTx` with `allowLinkedSigner=true`: [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner=true` routes through `validateCompactSignature`, which passes `getLinkedSigner(sender)` (the current hot wallet) as the accepted signer to the verifier: [2](#0-1) 

`verifier.validateCompactSignature` accepts the signature if the recovered address equals either the subaccount owner **or** the current linked signer: [3](#0-2) 

The EIP-712 digest for `LinkSigner` commits to `signedTx.tx.signer` (the new signer to be installed): [4](#0-3) 

Because the hot wallet is an accepted signer for this transaction type, it can craft and sign a `LinkSigner` struct where `tx.signer = attacker_address`. The digest is valid (it commits to the attacker address), the signature check passes (hot wallet == current linked signer), and the storage write executes: [5](#0-4) 

---

### Impact Explanation

After `linkedSigners[victim]` is overwritten to the attacker's address, the attacker can sign any transaction that accepts `allowLinkedSigner=true`, including:

- `WithdrawCollateral` — drain victim's collateral
- `TransferQuote` — move quote balance to attacker-controlled subaccount
- `LiquidateSubaccount` — trigger liquidation of victim
- `MintNlp` / `BurnNlp` — manipulate NLP positions

The takeover is permanent unless the subaccount owner (cold wallet) signs a corrective `LinkSigner`. If the attacker continuously re-signs `LinkSigner` transactions pointing back to themselves, the owner cannot recover without the sequencer refusing to process the attacker's transactions.

---

### Likelihood Explanation

Any party that has been granted linked-signer access (API key, hot wallet, third-party trading bot) can execute this unilaterally. No additional privileges, no sequencer compromise, and no governance capture are required. The sequencer processes user-submitted transactions normally — it has no reason to reject a `LinkSigner` transaction signed by the registered hot wallet. The precondition (a registered linked signer exists) is the normal operating state for active trading accounts.

---

### Recommendation

Change `allowLinkedSigner` to `false` for `LinkSigner` transactions in `processTransactionImpl`:

```solidity
// core/contracts/EndpointTx.sol
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only subaccount owner may change the linked signer
);
```

This ensures only `address(uint160(bytes20(sender)))` (the cold wallet / subaccount owner) can mutate `linkedSigners[sender]`, which is the intended invariant. The same fix should be applied to the slow-mode `LinkSigner` path if it is ever changed to accept linked signers. [6](#0-5) 

---

### Proof of Concept

```
1. Deploy Endpoint + Verifier + supporting contracts.
2. victim = bytes32(abi.encodePacked(coldWallet, bytes12(0)));
3. hotWallet registers as linked signer:
     coldWallet signs LinkSigner{sender=victim, signer=hotWallet, nonce=0}
     → linkedSigners[victim] == hotWallet  ✓
4. Attacker (controlling hotWallet) crafts:
     payload = LinkSigner{sender=victim, signer=attackerAddr, nonce=1}
     sig = hotWallet.sign(EIP712(payload))
5. Submit via sequencer → submitTransactionsChecked → processTransactionImpl(LinkSigner)
     validateSignedTx(victim, 1, txn, sig, true)
     → verifier.validateCompactSignature(victim, hotWallet, digest, sig)
     → recovered == hotWallet == linkedSigner  → PASSES
     → linkedSigners[victim] = attackerAddr
6. Assert: linkedSigners[victim] == attackerAddr  ✓
7. Attacker signs WithdrawCollateral{sender=victim, ...} with attackerAddr
     → validateCompactSignature accepts (recovered == linkedSigner == attackerAddr)
     → victim's collateral drained  ✓
```

```text
    function validateCompactSignature(
        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        IEndpoint.CompactSignature memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature.r, signature.vs);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
```
