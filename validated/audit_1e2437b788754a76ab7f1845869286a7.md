Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User — Allowlist Bypassed or Broken for All Router-Mediated Swaps - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap()` always sets to its own `msg.sender` — the immediate caller of the pool. When users route through `MetricOmmSimpleRouter`, the router is `msg.sender` at the pool, so the extension evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][end_user]`. This produces two fund-impacting failure modes: allowlist bypass if the router is allowlisted, or complete denial of service for legitimate users if it is not.

## Finding Description
`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`MetricOmmPool.swap()` populates `sender` with its own `msg.sender`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // ← always the immediate caller of pool.swap()
    ...
);
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` at the pool for every swap:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

The same pattern applies to `exactInput` (all hops), `exactOutputSingle`, and `exactOutput`. The transient-storage context (`_setNextCallbackContext`) stores the original `msg.sender` only for the payment callback — it is never forwarded to the extension. No existing guard in the extension, pool, or router resolves the true end user for the allowlist check. [1](#0-0) [2](#0-1) [3](#0-2) 

## Impact Explanation
**Bypass path (High):** A pool admin deploys `SwapAllowlistExtension` to restrict swaps to a curated set of addresses. To allow those users to access the pool via the router, the admin calls `setAllowedToSwap(pool, router, true)`. Because the extension checks `sender = router`, every public user who calls any `exact*` function on the router now passes the gate. The allowlist is entirely ineffective. In a pool with oracle-anchored pricing or subsidized rates, unauthorized traders can extract LP value at will — direct loss of LP principal.

**Broken-functionality path (Medium):** If the admin does not allowlist the router, every legitimately allowlisted user who calls the router is blocked with `NotAllowedToSwap`. The only working path is a direct `pool.swap()` call requiring the user to implement their own callback — the supported periphery is unusable for the pool's intended participants, constituting broken core swap functionality. [4](#0-3) [5](#0-4) 

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool that deploys `SwapAllowlistExtension` and expects users to interact through the router will immediately hit one of the two failure modes. The admin's natural remediation attempt (allowlisting the router) triggers the bypass. No special timing, flash loan, or oracle manipulation is required — any public user calling `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` triggers the condition. [6](#0-5) [7](#0-6) 

## Recommendation
The extension must resolve the actual end user, not the proximate caller. The simplest safe fix: store `msg.sender` of the router call in transient storage before calling the pool (analogous to how the router already stores the payer via `_setNextCallbackContext`), expose it via a view function, and have the extension read it when `sender` is a known router address. Alternatively, encode the verified end-user address in `extensionData` within the router and have the extension decode and verify it when present, falling back to `sender` for direct calls. [8](#0-7) 

## Proof of Concept
```
Setup:
  pool = deploy MetricOmmPool with SwapAllowlistExtension as beforeSwap hook
  admin calls setAllowedToSwap(pool, router, true)   // intending to enable router access for allowlisted users

Attack:
  attacker (not in allowlist) calls:
    router.exactInputSingle({pool: pool, recipient: attacker, ...})

Trace:
  router.exactInputSingle()
    → _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, attacker_eoa, tokenIn)  // stored for payment only
    → pool.swap(recipient=attacker, ...)          // msg.sender = router
      → _beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true  // ✓ passes — router is allowlisted
      → swap executes, attacker receives tokens

Result:
  Non-allowlisted attacker completes swap in a curated pool.
  LP principal is exposed to unauthorized traders.

Broken-functionality variant:
  admin does NOT allowlist router
  legitimate_user (in allowlist) calls router.exactInputSingle(...)
    → pool.swap() → _beforeSwap(sender=router, ...)
      → allowedSwapper[pool][router] == false → revert NotAllowedToSwap
  Result: legitimate user cannot use the router despite being allowlisted.
``` [1](#0-0) [9](#0-8) [6](#0-5)
