### Title
Corruptible Upgradability Pattern: Missing Storage Gaps in Multi-Level Inherited Upgradeable Contracts - (`core/contracts/EndpointGated.sol`, `core/contracts/BaseEngine.sol`, `core/contracts/ClearinghouseStorage.sol`, `core/contracts/EndpointStorage.sol`, `core/contracts/BaseWithdrawPool.sol`, `core/contracts/BaseProxyManager.sol`)

---

### Summary

Every upgradeable contract in the Nado protocol inherits one or more abstract parent contracts that define state variables but declare no `__gap` storage reserve. A confirmed `grep` across the entire codebase returns zero matches for `__gap`. Because the protocol's `ProxyManager` actively manages and executes implementation upgrades for `Clearinghouse`, `Endpoint`, `SpotEngine`, `PerpEngine`, `OffchainExchange`, `Verifier`, `WithdrawPool`, and `ProxyManager` itself, any future upgrade that adds a new state variable to any of these parent contracts will silently shift the storage layout of every inheriting child contract, corrupting all downstream state.

---

### Finding Description

The Nado protocol uses transparent proxy upgrades managed by `ProxyManager` / `BaseProxyManager`. The following inheritance chains are affected:

**`Clearinghouse` and `ClearinghouseLiq`**
```
Clearinghouse
  └─ EndpointGated        ← has `address private endpoint`, NO __gap
  └─ ClearinghouseStorage ← has 10 state variables, NO __gap
```

**`SpotEngine`**
```
SpotEngine
  └─ SpotEngineState      ← has 4 state variables, NO __gap
       └─ BaseEngine      ← has 5 state variables, NO __gap
            └─ EndpointGated ← has `address private endpoint`, NO __gap
```

**`PerpEngine`**
```
PerpEngine
  └─ PerpEngineState      ← has 3 state variables, NO __gap
       └─ BaseEngine      ← has 5 state variables, NO __gap
            └─ EndpointGated ← has `address private endpoint`, NO __gap
```

**`Endpoint` / `EndpointTx`**
```
Endpoint / EndpointTx
  └─ EndpointStorage      ← has 20+ state variables, NO __gap
```

**`WithdrawPool`**
```
WithdrawPool
  └─ BaseWithdrawPool     ← has 5 state variables, NO __gap
```

**`ProxyManager`**
```
ProxyManager
  └─ BaseProxyManager     ← has 7 state variables, NO __gap
```

None of these parent contracts define a `__gap` array. The `ProxyManager.migrateAll()` function is the live upgrade path that can trigger this corruption.

### Title
Missing Storage Gaps in Upgradeable Parent Contracts Corrupt Storage of All Core Protocol Contracts on Upgrade - (File: `core/contracts/EndpointGated.sol`, `core/contracts/BaseEngine.sol`)

---

### Summary

No `__gap` storage variable exists anywhere in the Nado codebase. Multiple parent contracts that hold storage variables and are inherited by upgradeable proxy contracts (`Clearinghouse`, `SpotEngine`, `PerpEngine`, `OffchainExchange`, `Endpoint`) lack the required storage gap. The most critical are `EndpointGated` and `BaseEngine`. Adding any new storage variable to either of these parent contracts during a future upgrade will shift all subsequent storage slots in every child contract, corrupting user balances, authorization mappings, insurance accounting, and order fill state.

---

### Finding Description

A grep for `__gap` across all Solidity files returns zero results: [1](#0-0) 

`EndpointGated` is an abstract upgradeable parent that declares one storage variable (`address private endpoint`) and no `__gap`. It is inherited by every major upgradeable contract in the protocol:

- `Clearinghouse is EndpointGated, ClearinghouseStorage`
- `OffchainExchange is IOffchainExchange, EndpointGated, EIP712Upgradeable`
- `BaseEngine is IProductEngine, EndpointGated` → inherited by `SpotEngine` and `PerpEngine`
- `ClearinghouseLiq is EndpointGated, ClearinghouseStorage` (used via `delegatecall` from `Clearinghouse`) [2](#0-1) 

`BaseEngine` is an abstract parent that declares four sequential storage variables (`_clearinghouse`, `productIds`, `canApplyDeltas`, `nonZeroBalances`) and no `__gap`. It is inherited by `SpotEngineState` → `SpotEngine` and `PerpEngineState` → `PerpEngine`.

The concrete storage layout for `SpotEngine` (following C3 linearization) is:

| Slot | Variable | Source |
|------|----------|--------|
| 0 | `_owner` | `OwnableUpgradeable` |
| 1–49 | `__gap[49]` | `OwnableUpgradeable` ✓ |
| 50 | `endpoint` | `EndpointGated` — **no gap** |
| 51 | `_clearinghouse` | `BaseEngine` — **no gap** |
| 52 | `productIds` | `BaseEngine` |
| 53 | `canApplyDeltas` | `BaseEngine` |
| 54 | `nonZeroBalances` | `BaseEngine` |
| 55 | `configs` | `SpotEngineState` — **no gap** |
| 56 | `states` | `SpotEngineState` |
| 57 | `balances` | `SpotEngineState` |
| 58 | `nlpLockedBalanceQueues` | `SpotEngineState` | [3](#0-2) 

If a single new `address` variable is appended to `EndpointGated`, every slot from 51 onward shifts by one. The `_clearinghouse` address reads from the wrong slot, `balances` reads from the wrong slot, and so on across the entire engine.

The same pattern applies to `Clearinghouse`: [4](#0-3) 

`ClearinghouseStorage` holds `quote`, `clearinghouse`, `clearinghouseLiq`, `productToEngine`, `engineByType`, `insurance`, `lastLiquidationFees`, `spreads`, `withdrawPool` — all without a `__gap`. Adding a variable to `EndpointGated` shifts all of these.

The protocol explicitly supports and performs upgrades. `BaseProxyManager.migrateAll` upgrades all registered proxies, and `bootstrapEndpointTx` is already a `reinitializer(2)`, confirming that at least one upgrade cycle has occurred: [5](#0-4) 

`EndpointStorage` and `ClearinghouseStorage` also lack `__gap` and are inherited by `Endpoint`/`EndpointTx` and `Clearinghouse`/`ClearinghouseLiq` respectively, compounding the exposure. [6](#0-5) 

---

### Impact Explanation

If any new storage variable is added to `EndpointGated` or `BaseEngine` during a future upgrade:

- **`SpotEngine.balances`** reads from the wrong slot → user spot balances are corrupted; withdrawals drain wrong accounts or revert; deposits credit wrong subaccounts.
- **`PerpEngine.balances`** reads from the wrong slot → perpetual positions and vQuote balances are corrupted; liquidation health checks use wrong values.
- **`Clearinghouse.insurance`** reads from the wrong slot → insurance fund accounting is broken; socialization and liquidation finalization use wrong values.
- **`Clearinghouse.productToEngine`** reads from the wrong slot → product-to-engine routing is broken; all deposits, withdrawals, and liquidations fail or route to wrong engines.
- **`OffchainExchange.filledAmounts`** reads from the wrong slot → order fill tracking is corrupted; orders can be double-filled or blocked.

The corrupted state is permanent post-upgrade and affects all users of the protocol. Assets can be stuck or stolen.

---

### Likelihood Explanation

The protocol has an active upgrade mechanism (`ProxyManager`, `migrateAll`) and has already performed at least one upgrade (`reinitializer(2)` in `bootstrapEndpointTx`). Future upgrades that extend `EndpointGated` or `BaseEngine` with new fields (e.g., adding a new access-control address, a new flag, or a new configuration parameter) are a normal part of protocol evolution. The risk is latent but will materialize on the next upgrade that touches any of these parent contracts.

---

### Recommendation

Add a `__gap` array at the end of every upgradeable parent contract that holds storage variables. The gap size should reserve enough slots to allow future additions without shifting child storage:

```solidity
// EndpointGated.sol
abstract contract EndpointGated is OwnableUpgradeable, IEndpointGated {
    address private endpoint;
    // ... existing functions ...
    uint256[49] private __gap; // reserve 49 slots (endpoint uses 1 of 50)
}

// BaseEngine.sol
abstract contract BaseEngine is IProductEngine, EndpointGated {
    IClearinghouse internal _clearinghouse;
    uint32[] internal productIds;
    mapping(address => bool) internal canApplyDeltas;
    mapping(bytes32 => mapping(uint32 => uint256)) internal nonZeroBalances;
    bytes32 internal constant RISK_STORAGE = keccak256("nado.protocol.risk");
    // ... existing functions ...
    uint256[46] private __gap; // reserve remaining slots
}
```

Apply the same pattern to `ClearinghouseStorage`, `EndpointStorage`, `SpotEngineState`, `PerpEngineState`, `BaseWithdrawPool`, `BaseProxyManager`, `Verifier`, `Airdrop`, and `ContractOwner`.

---

### Proof of Concept

1. Deploy `SpotEngine` behind a `TransparentUpgradeableProxy`. Record the current storage slot 51 value (should be `_clearinghouse` address).
2. Deploy a new `EndpointGated` implementation that adds `address private newField` after `endpoint`.
3. Deploy a new `BaseEngine` implementation that inherits the updated `EndpointGated`.
4. Deploy a new `SpotEngine` implementation that inherits the updated `BaseEngine`.
5. Call `ProxyManager.migrateAll` to upgrade `SpotEngine` to the new implementation.
6. Read storage slot 51 from the `SpotEngine` proxy — it now contains `newField` (zero/uninitialized) instead of `_clearinghouse`.
7. Call `SpotEngine.getClearinghouse()` — it returns `address(0)`, breaking all authorization checks that gate balance updates via `canApplyDeltas`.
8. All subsequent calls to `updateBalance`, `updateStates`, and `assertUtilization` either revert or operate on corrupted state, locking all user funds. [7](#0-6) [8](#0-7)
