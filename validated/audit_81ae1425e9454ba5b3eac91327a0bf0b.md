### Title
No Minimum Confirmation Depth Enforced in SPV Proof Verification â€” (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations` value with only an upper-bound check (`<= gc_threshold`) and no minimum. Any unprivileged NEAR caller can pass `confirmations = 0`, causing the contract to return `true` for a transaction in any block currently on the main chain â€” including blocks that have not yet accumulated enough PoW depth to resist a reorg. A downstream contract consuming this result and releasing assets would be vulnerable to a double-spend via a subsequent chain reorganization.

### Finding Description

Both SPV verification entry points delegate confirmation enforcement entirely to the caller:

```rust
// contract/src/lib.rs:289-292
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [1](#0-0) 

This is the **only** bounds check on `confirmations`. There is no lower bound. The subsequent confirmation-depth check is:

```rust
// contract/src/lib.rs:304-308
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [2](#0-1) 

When `args.confirmations = 0`, the right-hand side is `0` and the left-hand side is a `u64` that is always `>= 0`, so the check is trivially satisfied for **any** block on the main chain. The function then proceeds to verify the Merkle proof and returns `true`.

`verify_transaction_inclusion_v2` delegates directly to `verify_transaction_inclusion` after its coinbase proof check, so it inherits the same flaw: [3](#0-2) 

The `confirmations` field is a plain `u64` in `ProofArgs` and `ProofArgsV2` with no validation annotation: [4](#0-3) 

The relayer's default `max_fork_len` is **500 blocks**, meaning the system is explicitly designed to handle reorgs of up to 500 blocks deep: [5](#0-4) 

Any transaction confirmed by fewer blocks than the realistic reorg depth is at risk.

### Impact Explanation

**Impact: High** â€” A downstream NEAR contract that calls `verify_transaction_inclusion` (or `_v2`) with `confirmations = 0` or a dangerously small value will receive `true` for a transaction in a block that has not yet accumulated sufficient PoW depth. If that contract releases funds, mints tokens, or records an irreversible state change based on this result, and the underlying Bitcoin block is subsequently reorged out, the attacker receives assets for a transaction that no longer exists on the canonical chain. This is a direct loss-of-funds path for any protocol built on top of this light client.

### Likelihood Explanation

**Likelihood: Low-to-Medium** â€” The contract's own documentation notes that the genesis block should be "at least 144 blocks earlier than the last block" (one day of Bitcoin blocks), demonstrating awareness of confirmation-depth significance, yet no minimum is enforced for SPV proofs. [6](#0-5) 

A downstream integrator who does not independently research Bitcoin's reorg risk model (6 confirmations for standard finality; up to 500 blocks per the relayer's own `max_fork_len`) could easily pass `confirmations = 0` or `confirmations = 1`. The attack is most realistic during periods of low hashrate (e.g., early chain launch, a hashrate drop event) when reorgs are cheaper to execute, directly mirroring the M-10 scenario of limited liquidity at protocol launch.

### Recommendation

Enforce a protocol-level minimum confirmation depth inside `verify_transaction_inclusion`. A reasonable floor for Bitcoin is **6 confirmations**; for chains with faster block times (Litecoin: 12, Dogecoin: 40) the floor should be calibrated to equivalent PoW depth. Add a constant and a `require!` guard:

```rust
pub const MIN_CONFIRMATIONS: u64 = 6; // adjust per chain feature flag

require!(
    args.confirmations >= MIN_CONFIRMATIONS,
    format!("Confirmations must be at least {}", MIN_CONFIRMATIONS)
);
```

Additionally, document the minimum safe confirmation depth prominently in the public API so downstream integrators are not left to discover it independently.

### Proof of Concept

1. Deploy the contract with any valid genesis block and `gc_threshold = 1000`.
2. Submit one Bitcoin block header via `submit_blocks`.
3. Call `verify_transaction_inclusion` with:
   - `tx_id` = a valid transaction hash in that block
   - `tx_block_blockhash` = the hash of the just-submitted block
   - `tx_index` = correct index
   - `merkle_proof` = valid Merkle path
   - **`confirmations = 0`**
4. The contract returns `true` immediately â€” the block has zero depth above it and could be reorged out by any competing chain with more cumulative work.
5. A downstream contract acting on this `true` result and releasing funds is now exposed to a double-spend if a longer competing chain is later submitted via `submit_blocks`, triggering `reorg_chain` and removing the block from the canonical mapping. [7](#0-6)
