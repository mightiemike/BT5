### Title
ML-DSA-65 Gas-Key Host Functions Use Wire-Length Instead of Trie-ID Length for Fee Calculation, Causing ~59× Overcharge vs. Transaction Path — (`runtime/near-vm-runner/src/logic/logic.rs`)

---

### Summary

The host functions `promise_batch_action_transfer_to_gas_key` and `promise_batch_action_add_gas_key_with_full_access` pass the raw `public_key_len` (the borsh wire length of the public key as supplied by the contract) to `gas_key_transfer_send_fee`, `gas_key_transfer_exec_fee`, and `gas_key_add_key_exec_fee`. For ML-DSA-65 keys the wire length is **1953 bytes**, but the on-trie identifier is only **33 bytes** (a SHA3-256 hash). The transaction path in `runtime/runtime/src/config.rs` correctly calls `action.public_key.trie_id_len()` (33 bytes). This divergence causes contracts using ML-DSA-65 gas keys via host functions to be charged ~59× more gas per byte than the equivalent transaction action.

---

### Finding Description

The `PostQuantumSignatures` feature (protocol version 85) introduced ML-DSA-65 access keys. A deliberate design decision stores ML-DSA-65 access keys in the trie by their SHA3-256 hash (33 bytes) rather than the full 1952-byte pubkey. `PublicKey::trie_id_len()` returns 33 for ML-DSA-65, while `PublicKey::len()` returns 1953. [1](#0-0) 

The **transaction path** correctly uses `trie_id_len()` for both send and exec fee helpers: [2](#0-1) [3](#0-2) 

The **host function path** passes `public_key_len as usize` — the raw borsh wire length provided by the calling contract — to the same fee helpers: [4](#0-3) [5](#0-4) 

The same divergence exists in the wasmtime runner: [6](#0-5) [7](#0-6) 

The fee helpers themselves are correct — they accept a `public_key_len` parameter and use it for per-byte pricing: [8](#0-7) 

The exact divergent values for ML-DSA-65:
- Wire length (`public_key_len`): **1953 bytes** (1 tag + 1952-byte full pubkey)
- Trie-id length (`trie_id_len()`): **33 bytes** (1 tag + 32-byte SHA3-256 hash)

The per-byte send fee is multiplied by `public_key_len` directly; the per-byte exec fee is multiplied by `access_key_key_len(account_id_len, public_key_len)`. Both are ~59× larger than they should be for ML-DSA-65 keys.

The codebase documentation explicitly flags this as a known contract that all callers must respect: [9](#0-8) 

The storage-stake and runtime config paths were updated to use `trie_id_len()`: [10](#0-9) 

But the host function fee-charging paths were not.

---

### Impact Explanation

For a contract calling `promise_batch_action_transfer_to_gas_key` with an ML-DSA-65 public key:

- **Send fee per-byte component**: charged on 1953 bytes instead of 33 bytes → ~59× overcharge
- **Exec fee per-byte component**: charged on `access_key_key_len(account_id_len, 1953)` instead of `access_key_key_len(account_id_len, 33)` → ~43× overcharge for a 10-byte account ID

If the `gas_key_byte` exec fee is on the order of typical storage-write-key-byte costs (~14 Mgas/byte), the overcharge on the exec per-byte component alone is approximately `(1965 - 45) × 14 Mgas ≈ 26.9 Tgas` per call. A contract with a 300 Tgas budget that calls this host function multiple times, or that also pays other fees, will exhaust its gas budget and fail — even though the equivalent `TransferToGasKey` transaction succeeds within normal limits.

The fee invariant that "the same protocol action costs the same gas whether submitted as a transaction or via a host function" is broken for ML-DSA-65 gas keys. Contracts that compute their gas budget based on the transaction fee schedule will systematically fail when using the host function path.

---

### Likelihood Explanation

- `PostQuantumSignatures` is stabilized at protocol version 85 and active on mainnet.
- Any unprivileged contract can call `promise_batch_action_transfer_to_gas_key` or `promise_batch_action_add_gas_key_with_full_access` with an ML-DSA-65 public key.
- The overcharge is deterministic and reproducible — it affects every such call, not just edge cases.
- Contracts that manage gas keys programmatically (e.g., wallet contracts, key-rotation contracts) are the natural users of these host functions and will encounter this immediately upon adopting ML-DSA-65 keys.

---

### Recommendation

After decoding the public key in the host function, use `decoded_key.trie_id_len()` instead of `public_key_len as usize` for all fee helper calls. For example, in `promise_batch_action_transfer_to_gas_key`:

```rust
let public_key = self.get_public_key(public_key_ptr, public_key_len)?;
let decoded_key = public_key.decode()?;  // decode early
// ...
let send = gas_key_transfer_send_fee(&self.fees_config, sir, decoded_key.trie_id_len());
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    decoded_key.trie_id_len(),
);
// ...
self.ext.append_action_transfer_to_gas_key(receipt_idx, decoded_key, amount);
```

Apply the same fix to `promise_batch_action_add_gas_key_with_full_access` and the wasmtime runner equivalents.

---

### Proof of Concept

1. Deploy a contract that calls `promise_batch_action_transfer_to_gas_key` with an ML-DSA-65 public key (borsh-encoded, 1953 bytes).
2. Submit the equivalent `TransferToGasKey` transaction with the same ML-DSA-65 public key.
3. Compare `gas_burnt` on the action execution receipt:
   - Transaction path: fee computed with `trie_id_len() = 33`
   - Host function path: fee computed with `public_key_len = 1953`
4. The host function call burns ~59× more gas on the per-byte component than the transaction, confirming the divergence.

The existing test `test_gas_key_fee_parity` in `test-loop-tests/src/tests/gas_keys.rs` only exercises ED25519 keys (33-byte wire = 33-byte trie-id, so no divergence). Adding an ML-DSA-65 variant would expose the failure. [11](#0-10)

    /// entry owned by this public key. For ed25519/secp256k1 this matches

            Self::SECP256K1(_) => 1 + 64,
            Self::MLDSA65(_) => 1 + ML_DSA_65_HASH_LENGTH,

/// Send fee for TransferToGasKey / WithdrawFromGasKey actions.
/// Based on the public key length (what the sender sees).
pub fn gas_key_transfer_send_fee(
    cfg: &RuntimeFeesConfig,
    sender_is_receiver: bool,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).send_fee(sender_is_receiver);
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .send_fee(sender_is_receiver)
        .checked_mul(public_key_len as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}

/// Exec fee for TransferToGasKey / WithdrawFromGasKey actions.
/// Based on the access key trie key length + estimated value length (what the
/// receiver needs to read/write in the trie).
pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}

fn access_key_storage_usage(
    fee_config: &RuntimeFeesConfig,
    public_key: &PublicKey,
    access_key: &AccessKey,
) -> StorageUsage {
    let storage_usage_config = &fee_config.storage_usage_config;
    // Use the on-trie identifier length, not the borsh-serialized pubkey
    // length: ML-DSA-65 access keys live in the trie as a SHA3-256 hash
    // (33 bytes incl. type tag), not as a 1953-byte full pubkey.
    public_key.trie_id_len() as u64
        + borsh::object_length(access_key).unwrap() as u64
        + storage_usage_config.num_extra_bytes_record
}

/// Verify that adding, funding, and deleting a gas key via transaction vs host function
/// produces identical gas_burnt and tokens_burnt on the action execution receipt.
fn test_gas_key_fee_parity(mode: GasKeyKind) {
    let mut setup = setup_host_function_test();
    let account = setup.account.clone();

    let num_nonces: NonceIndex = 4;
    let gas_key_a_signer: Signer =
        InMemorySigner::from_seed(account.clone(), KeyType::ED25519, "gas_key_tx").into();
    let gas_key_b_signer: Signer =
        InMemorySigner::from_seed(account.clone(), KeyType::ED25519, "gas_key_host_fn").into();

    let public_key_b_base64 = near_primitives_core::serialize::to_base64(
        &borsh::to_vec(&gas_key_b_signer.public_key()).unwrap(),
    );

    // Add gas key A via transaction, B via host function
    let add_a_outcome = setup.run_actions(vec![Action::AddKey(Box::new(AddKeyAction {
        public_key: gas_key_a_signer.public_key(),
        access_key: mode.access_key(num_nonces, &account),
    }))]);
    let add_b_outcome = setup.run_call_promise(serde_json::json!([
        {"batch_create": {"account_id": account.as_str()}, "id": 0},
        mode.add_action_json(num_nonces, &account, &public_key_b_base64),
    ]));
    assert_eq!(add_a_outcome.gas_burnt, add_b_outcome.gas_burnt);
    assert_eq!(add_a_outcome.tokens_burnt, add_b_outcome.tokens_burnt);
