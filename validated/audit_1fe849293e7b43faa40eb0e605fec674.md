Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address as `sender` instead of the originating user, allowing any user to bypass the per-user swap allowlist via `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is the `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so the extension checks whether the **router** is allowlisted rather than the originating user. If the pool admin allowlists the router to enable router-based swaps for their curated users, every non-allowlisted user can bypass the per-user allowlist by routing through the router.

## Finding Description

**Confirmed call chain:**

`MetricOmmPool.swap()` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards `sender` unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly from the router contract, making the router `msg.sender` to the pool: [4](#0-3) 

The same substitution occurs in `exactInput`, `exactOutputSingle`, `exactOutput`, and `_exactOutputIterateCallback`, all of which call `pool.swap()` from within the router: [5](#0-4) [6](#0-5) [7](#0-6) 

**Broken invariant:**

The allowlist is keyed `allowedSwapper[pool][swapper]`. When a user swaps directly, `swapper = user` — the check is correct. When the same user routes through the router, `swapper = router` — the check is against the router's entry, not the user's. This creates an irreconcilable dilemma: not allowlisting the router blocks allowlisted users from using it; allowlisting the router opens the pool to every user.

**Contrast with `DepositAllowlistExtension`:**

`DepositAllowlistExtension.beforeAddLiquidity` correctly checks `owner` (the position owner, the economically relevant actor), not `sender` (the immediate caller): [8](#0-7) 

`SwapAllowlistExtension` does not apply the same pattern.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of users (e.g., KYC-gated, institutional-only, or whitelist-only pools) can be fully bypassed by any unprivileged user routing through `MetricOmmSimpleRouter`. The bypassing user trades at the oracle-anchored price against LP positions deposited under the assumption that only vetted counterparties would trade against them. This constitutes a direct loss of LP principal and fee revenue to unauthorized counterparties — an allowlist bypass with direct fund impact on LPs.

## Likelihood Explanation

The bypass requires the pool admin to have allowlisted the router address. This is a natural and expected administrative action: any pool admin who wants their allowlisted users to use the standard periphery router must allowlist it. The router is a trusted, factory-deployed contract, so allowlisting it is not an obviously dangerous act. Once the router is allowlisted, the bypass is trivially reachable by any user with no special privileges, requiring only a standard router call.

## Recommendation

1. **Pass the originating user through `extensionData`**: The router should encode `msg.sender` (the originating user) into `extensionData`, and `SwapAllowlistExtension` should decode and check it. This requires a convention between the router and the extension.
2. **Check `recipient` instead of `sender`**: Redesign `SwapAllowlistExtension` to check `recipient` (the address receiving swap output) rather than `sender`, mirroring how `DepositAllowlistExtension` checks `owner` rather than `sender`.
3. **Document the limitation**: At minimum, document clearly that `SwapAllowlistExtension` cannot gate router-mediated swaps by originating user, and that allowlisting the router opens the pool to all users.

## Proof of Concept

```solidity
// Setup:
// 1. Deploy pool with SwapAllowlistExtension on beforeSwap order.
// 2. Admin allowlists `alice` (a legitimate user) and the router
//    (so alice can use the router).
// 3. `charlie` (not allowlisted) calls the router.

function test_swapAllowlistBypassViaRouter() public {
    // Admin allowlists alice and the router
    swapExtension.setAllowedToSwap(address(pool), alice, true);
    swapExtension.setAllowedToSwap(address(pool), address(router), true);

    // charlie is NOT allowlisted
    assertFalse(swapExtension.isAllowedToSwap(address(pool), charlie));

    // charlie routes through the router — extension sees sender=router, not charlie
    vm.prank(charlie);
    // Succeeds because allowedSwapper[pool][router] == true
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool),
        recipient: charlie,
        zeroForOne: false,
        amountIn: 1000,
        amountOutMinimum: 0,
        priceLimitX64: type(uint128).max,
        tokenIn: address(token1),
        extensionData: "",
        deadline: block.timestamp
    }));
    // charlie successfully swapped on a pool he was not allowlisted for
}
```

The root cause is `SwapAllowlistExtension.beforeSwap` at line 37, which checks `allowedSwapper[msg.sender][sender]` where `sender` is the router address when the swap is router-mediated, rather than the originating user. [9](#0-8)
