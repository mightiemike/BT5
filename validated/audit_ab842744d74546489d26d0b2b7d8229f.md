### Title
Linked Signer Can Unilaterally Replace Itself via `LinkSigner` Transaction, Enabling Permanent Subaccount Takeover — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In `processTransactionImpl`, the `LinkSigner` transaction is validated with `allowLinkedSigner = true`, meaning the **current linked signer** is accepted as a valid authorizer to change the linked signer to any new address. Because the linked signer is a hot-wallet key held by off-chain services (the exact key material described in the external report as stored in environment variables), a compromised or malicious linked signer can permanently replace itself with an attacker-controlled address and subsequently drain all subaccount collateral.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` case calls `validateSignedTx` with the fifth argument hardcoded to `true`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true          // allowLinkedSigner
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which in turn calls `verifier.validateSignature(sender, getLinkedSigner(sender), digest, signature)`:

```solidity
function validateSignature(...) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

`verifier.validateSignature` accepts the signature if it recovers to **either** the subaccount owner address **or** the linked signer address:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

This means the currently registered linked signer can sign a `LinkSigner` transaction that overwrites `linkedSigners[subaccount]` with any arbitrary address — including one the attacker controls. The `linkedSigners` mapping is the sole on-chain record of delegated signing authority: [4](#0-3) 

---

### Impact Explanation

Once the linked signer is replaced with an attacker-controlled address, the attacker holds full signing authority over the victim subaccount. They can immediately sign and submit (via the sequencer) any of the following transactions:

- `WithdrawCollateral` / `WithdrawCollateralV2` — drain all deposited collateral
- `LiquidateSubaccount` — trigger liquidation of the victim's positions
- `TransferQuote` — transfer quote balance to an attacker-controlled subaccount
- `MintNlp` / `BurnNlp` — manipulate NLP positions

All of these transaction types also use `allowLinkedSigner = true`: [5](#0-4) [6](#0-5) 

The original subaccount owner cannot recover control without the sequencer's intervention, because the attacker's new linked signer can keep re-linking to itself. The corrupted state is `linkedSigners[subaccount]` — a direct mapping that governs all subsequent signature validation for that subaccount.

---

### Likelihood Explanation

The external report explicitly identifies that off-chain services hold private keys (e.g., `STRATEGY_EXECUTOR_PRIVATE_KEY`) in environment variables and that these keys can leak through crash reports exported to third-party analytics. The linked signer key is exactly this type of hot-wallet key — held by an automated off-chain service and used to sign sequencer-submitted transactions. The objective explicitly lists "linked signer" as a valid attacker role. Any user who has linked a signer (a common pattern for automated trading bots) is exposed. The sequencer includes all valid signed transactions without filtering by transaction type, so no additional privilege is needed beyond possession of the linked signer key.

---

### Recommendation

Restrict `LinkSigner` transactions so that only the **subaccount owner** (not the current linked signer) can authorize a change. Change the `allowLinkedSigner` argument to `false` for the `LinkSigner` case in `processTransactionImpl`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
``` [1](#0-0) 

This mirrors the slow-mode path, which already enforces that only the subaccount owner (`validateSender`) can set the linked signer: [7](#0-6) 

---

### Proof of Concept

1. Alice deposits collateral and links hot-wallet key `K` (address `0xHOT`) to her subaccount `S` via a `LinkSigner` transaction signed by herself.
2. `linkedSigners[S] = 0xHOT` is written on-chain.
3. Attacker obtains `K` (e.g., via the env-var leak described in the external report, or via a malicious linked-signer service).
4. Attacker constructs a `SignedLinkSigner` struct: `{ tx: { sender: S, signer: 0xATTACKER, nonce: currentNonce }, signature: sign(K, digest) }`.
5. The digest is computed by `Verifier.computeDigest` over `(LINK_SIGNER_SIGNATURE, S, 0xATTACKER, nonce)` and wrapped with `_hashTypedDataV4`. [8](#0-7) 

6. Attacker submits the transaction to the sequencer (standard API call). The sequencer calls `submitTransactionsChecked`, which calls `processTransactionImpl`.
7. `validateSignedTx` recovers `0xHOT` from the signature — equal to `getLinkedSigner(S)` — so validation passes.
8. `linkedSigners[S]` is overwritten with `0xATTACKER`.
9. Attacker signs a `WithdrawCollateral` transaction with `0xATTACKER`'s key and submits it. All of Alice's collateral is withdrawn to an attacker-controlled address. [9](#0-8)
