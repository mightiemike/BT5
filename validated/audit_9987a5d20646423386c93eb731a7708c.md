Audit Report

## Title
SwapAllowlistExtension Checks Router Address as `sender` Instead of Actual User, Allowing Any User to Bypass the Per-User Swap Allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the pool's `msg.sender`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the original user. If the router is allowlisted (the natural admin action to enable router-based swaps), every user can bypass the per-user allowlist entirely. Conversely, if only individual users are allowlisted, those users cannot swap through the router at all.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // router address when called via router
  recipient,
  ...
  extensionData
);
```

`ExtensionCalling._beforeSwap` (L149-177) forwards this value unchanged via `abi.encodeCall` to every configured extension. `SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the router. The check becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

`MetricOmmSimpleRouter.exactInputSingle` calls the pool directly with no user identity forwarding:

```solidity
// MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
  );
```

The router stores the real `msg.sender` only in transient callback context for payment purposes; it is never encoded into `extensionData` or otherwise communicated to the extension. The same structural issue applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

Two mutually exclusive failure modes result:
1. Admin allowlists the router → `allowedSwapper[pool][router] == true` → every unprivileged EOA passes the check.
2. Admin allowlists individual users but not the router → `allowedSwapper[pool][router] == false` → all allowlisted users are blocked from using the router.

## Impact Explanation

**Allowlist bypass (primary):** If the admin allowlists the router to enable router-based swaps, every address on the internet can swap on a pool intended to be restricted (e.g., KYC-gated, private market-making). Unauthorized swaps drain LP value at oracle prices without restriction — direct loss of LP principal.

**Broken core swap flow:** If the admin allowlists only specific user addresses, those users cannot use the router at all, reverting with `NotAllowedToSwap` despite being individually permitted. This breaks the primary swap UX path for legitimate participants.

Both outcomes violate the core invariant that the allowlist gates the exact actor the pool designer intended, constituting broken core pool functionality causing loss of funds or an unusable swap flow.

## Likelihood Explanation

- Trigger is fully unprivileged: any EOA calls `MetricOmmSimpleRouter.exactInputSingle`.
- The router is a public, deployed periphery contract with no access control.
- Allowlisting the router is the expected operational step to enable router-based swaps; the bypass is therefore the default outcome of normal configuration.
- No special timing, flash loan, or oracle manipulation is required.

## Recommendation

The extension must recover the original user identity rather than trusting the pool-forwarded `sender`. The cleanest correct fix is for the router to encode `msg.sender` into `extensionData` (or a dedicated field within it) before calling `pool.swap`, and for `SwapAllowlistExtension.beforeSwap` to decode and verify the real caller when `sender` is a recognized router intermediary. Alternatively, add a `getRealSender()` interface to the router that the extension can call back, or adopt ERC-2771-style meta-transaction forwarding. Using `tx.origin` is a simpler fallback but introduces its own security risks and should be avoided in production.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, router, true)   // allowlist the router
  alice (not individually allowlisted) calls:
    router.exactInputSingle({pool: pool, ...})
      → pool.swap(msg.sender=router, ...)
        → _beforeSwap(sender=router, ...)
          → SwapAllowlistExtension.beforeSwap(sender=router)
            → allowedSwapper[pool][router] == true  → PASSES
  Alice's swap executes despite never being individually allowlisted.

Alternatively:
  admin calls setAllowedToSwap(pool, alice, true)    // allowlist alice directly
  alice calls router.exactInputSingle({pool: pool, ...})
    → pool.swap(msg.sender=router, ...)
      → SwapAllowlistExtension.beforeSwap(sender=router)
        → allowedSwapper[pool][router] == false → REVERTS NotAllowedToSwap
  Alice cannot use the router even though she is allowlisted.
```

Foundry test: deploy pool with `SwapAllowlistExtension`, call `setAllowedToSwap(pool, router, true)`, then call `router.exactInputSingle` from an unallowlisted EOA and assert the swap succeeds. Confirm the bypass by checking `allowedSwapper[pool][alice] == false` while the swap completes.
