### Title
Unchecked `transferFrom` Return Value Enables usdcE Drain Without USDC Payment — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` directly without checking the return value. If the USDC token's `transferFrom` returns `false` instead of reverting, the function silently continues, withdrawing usdcE from the DDA and sending it to the caller — without the caller having provided any USDC. The protocol's own `ERC20Helper.safeTransferFrom` exists precisely to prevent this, and is even used on the very next line for the outbound transfer, making the omission inconsistent and exploitable.

---

### Finding Description

`ContractOwner.replaceUsdcEWithUsdc` is an externally callable function (no access control modifier) that is intended to swap usdcE held in a `DirectDepositV1` (DDA) contract for native USDC provided by the caller. The swap logic is:

1. Read the usdcE balance of the DDA.
2. Pull that amount of USDC from `msg.sender` into the DDA via `transferFrom`.
3. Withdraw usdcE from the DDA to `ContractOwner` via `DirectDepositV1.withdraw`.
4. Forward the usdcE from `ContractOwner` to `msg.sender` via `safeTransfer`.

Step 2 uses a raw interface call:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
```

The return value is never checked. If USDC's `transferFrom` returns `false` (a valid ERC20 failure mode), execution continues uninterrupted into steps 3 and 4, which unconditionally drain usdcE out of the DDA and deliver it to the caller.

The codebase already contains `ERC20Helper.safeTransferFrom`, which handles both missing return values and `false` returns correctly. It is imported and used in `ContractOwner` via `using ERC20Helper for IERC20Base`. The outbound transfer on the very next line correctly uses `safeTransfer`, making the omission on the inbound leg a clear inconsistency. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

An attacker who calls `replaceUsdcEWithUsdc` for any subaccount whose DDA holds usdcE, while holding zero USDC (or having approved zero), can receive the full usdcE balance of that DDA for free if the USDC token's `transferFrom` returns `false` rather than reverting. The DDA's usdcE balance is permanently drained; the protocol receives no USDC in exchange. This is a direct, irreversible asset loss from user-owned DDA contracts.

---

### Likelihood Explanation

The function is callable by any unprivileged external address on chain ID 57073 with no further preconditions beyond a non-empty DDA existing for the target subaccount. The hardcoded USDC address (`0x2D270e6886d130D724215A266106e6832161EAEd`) is a specific deployment on the Ink chain whose exact return-value behavior under failure conditions must be verified; if it ever returns `false` on insufficient allowance rather than reverting, the exploit is immediately reachable. Even absent that specific behavior, the missing check violates the invariant that the inbound transfer must succeed before the outbound transfer proceeds. [3](#0-2) 

---

### Recommendation

Replace the raw `transferFrom` call with `ERC20Helper.safeTransferFrom`, which is already available via the `using ERC20Helper for IERC20Base` directive at the top of the contract:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [4](#0-3) 

---

### Proof of Concept

1. A DDA exists for some `subaccount` and holds `N` usdcE (e.g., deposited by a legitimate user).
2. Attacker calls `replaceUsdcEWithUsdc(subaccount)` with zero USDC allowance granted to `ContractOwner`.
3. `IERC20Base(usdc).transferFrom(attacker, dda, N)` is called. USDC returns `false` (insufficient allowance). Return value is ignored; no revert.
4. `DirectDepositV1(dda).withdraw(usdcE)` transfers all `N` usdcE from the DDA to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, N)` sends all `N` usdcE to the attacker.
6. Attacker receives `N` usdcE; DDA is drained; no USDC was ever transferred. [5](#0-4)

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
