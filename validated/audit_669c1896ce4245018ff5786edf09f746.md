### Title
Stale `digestToSubaccount` Entry After Isolated Subaccount Closure Allows Funds to Be Credited to an Orphaned Subaccount — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

`_tryCloseIsolatedSubaccount` clears the parent-link and slot-mask for a closed isolated subaccount but never clears the `digestToSubaccount` mapping for any pending orders that were registered at creation time. A subsequent `matchOrders` call silently redirects the order sender to the now-orphaned isolated subaccount, executes balance updates against it, and leaves the credited funds permanently inaccessible because `parentSubaccounts[closedSubaccount]` is already `bytes32(0)`.

---

### Finding Description

When an isolated subaccount is created via `CreateIsolatedSubaccount`, the protocol records the order digest → isolated subaccount mapping: [1](#0-0) 

When the isolated subaccount is later closed, `_tryCloseIsolatedSubaccount` clears three pieces of state: [2](#0-1) 

It does **not** clear `digestToSubaccount[digest]` for any order that was registered at creation time. The stale entry persists.

In `matchOrders`, the isolated-subaccount guard fires on the *original* `order.sender` (the parent), which passes. The redirect then overwrites `taker.order.sender` with the stale closed subaccount: [3](#0-2) 

`_validateOrder` performs no liveness check on the redirected sender. The signature check passes because `address(uint160(bytes20(isolatedSubaccount)))` encodes the parent's address, which is the expected signer: [4](#0-3) 

Balance updates then execute against the closed isolated subaccount. Because `parentSubaccounts[closedSubaccount]` is already `bytes32(0)`, any subsequent `_tryCloseIsolatedSubaccount` call would transfer the residual balance to the zero subaccount, permanently destroying the funds. [5](#0-4) 

The `CloseIsolatedSubaccount` sequencer path and the liquidation path both call the same internal function and share the same omission: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

Any quote balance or unrealized PnL credited to the closed isolated subaccount after the match is permanently inaccessible. `parentSubaccounts[closedSubaccount] == bytes32(0)` means there is no recovery path: the funds cannot be swept back to the parent, and the only reachable code path that would move them (`_tryCloseIsolatedSubaccount`) would send them to the zero subaccount. This is a direct, irreversible loss of user funds.

---

### Likelihood Explanation

The attacker controls both the creation and the closure of the isolated subaccount. The sequencer processes transactions in ordered batches and has no on-chain obligation to verify that a `digestToSubaccount` entry still refers to a live subaccount before submitting `MatchOrders`. A high-throughput sequencer that does not re-validate subaccount liveness at match time will naturally trigger this path. No sequencer key compromise is required — only the sequencer's normal, honest processing of a close followed by a match of a previously registered order.

---

### Recommendation

Inside `_tryCloseIsolatedSubaccount`, clear every `digestToSubaccount` entry that maps to the subaccount being closed. Alternatively, add an explicit liveness guard in `matchOrders` immediately after the `digestToSubaccount` redirect:

```solidity
if (digestToSubaccount[ordersInfo.taker.digest] != bytes32(0)) {
    taker.order.sender = digestToSubaccount[ordersInfo.taker.digest];
    // ADD: require parentSubaccounts[taker.order.sender] != bytes32(0), ERR_INVALID_TAKER
}
```

The same guard must be applied to the maker path.

---

### Proof of Concept

1. Attacker calls `CreateIsolatedSubaccount` with order `O` for product `P`. On-chain: `digestToSubaccount[digest(O)] = isoSubaccount`.
2. Attacker submits `CloseIsolatedSubaccount` for `isoSubaccount`. On-chain: `parentSubaccounts[isoSubaccount] = bytes32(0)`, `isolatedSubaccounts[parent][id] = bytes32(0)`, but `digestToSubaccount[digest(O)]` still equals `isoSubaccount`.
3. Sequencer submits `MatchOrders` pairing order `O` with a counterparty order. `matchOrders` passes the `!isIsolatedSubaccount(taker.order.sender)` guard (original sender is the parent), then redirects `taker.order.sender` to `isoSubaccount`.
4. `_validateOrder` passes (signature recovers to the parent address, which equals `address(uint160(bytes20(isoSubaccount)))`).
5. `_updateBalances` credits quote delta to `isoSubaccount`. `parentSubaccounts[isoSubaccount] == bytes32(0)` — no recovery path exists. Funds are permanently lost.

    function _tryCloseIsolatedSubaccount(bytes32 subaccount) internal {
        uint32 productId = RiskHelper.getIsolatedProductId(subaccount);
        if (productId == 0) {
            return;
        }
        IPerpEngine.Balance memory balance = perpEngine.getBalance(
            productId,
            subaccount
        );
        if (balance.amount == 0) {
            uint8 id = RiskHelper.getIsolatedId(subaccount);
            address addr = address(uint160(bytes20(subaccount)));
            bytes32 parent = parentSubaccounts[subaccount];
            if (balance.vQuoteBalance != 0) {
                perpEngine.updateBalance(
                    productId,
                    subaccount,
                    0,
                    -balance.vQuoteBalance
                );
                perpEngine.updateBalance(
                    productId,
                    parent,
                    0,
                    balance.vQuoteBalance
                );
            }
            int128 quoteBalance = spotEngine
                .getBalance(QUOTE_PRODUCT_ID, subaccount)
                .amount;
            if (quoteBalance != 0) {
                spotEngine.updateBalance(
                    QUOTE_PRODUCT_ID,
                    subaccount,
                    -quoteBalance
                );
                spotEngine.updateBalance(
                    QUOTE_PRODUCT_ID,
                    parent,
                    quoteBalance
                );
            }
            isolatedSubaccountsMask[addr] &= ~uint256(0) ^ (1 << id);
            isolatedSubaccounts[parent][id] = bytes32(0);
            parentSubaccounts[subaccount] = bytes32(0);

            emit CloseIsolatedSubaccount(subaccount, parent);
        }
    }
