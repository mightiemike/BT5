### Title
Linked Signer Can Overwrite Itself via `LinkSigner` Transaction, Enabling Permanent Account Takeover - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The sequencer path for `LinkSigner` transactions calls `validateSignedTx` with `allowLinkedSigner = true`, meaning the current linked signer's signature is accepted as sufficient authorization to replace the linked signer with any arbitrary address. No validation is performed to restrict this operation to the account owner. A compromised linked signer can therefore permanently escalate its own privileges by overwriting itself with an attacker-controlled address, locking the legitimate owner out of their subaccount.

---

### Finding Description

**Root cause — `allowLinkedSigner = true` on `LinkSigner` in the sequencer path:**

In `EndpointTx.sol`, `processTransactionImpl` handles the `LinkSigner` transaction type as follows:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` delegates to `validateSignature`, which passes the current linked signer address to `verifier.validateSignature`:

```solidity
// Verifier.sol lines 291–303
function validateSignature(...) public pure {
    address recovered = ECDSA.recover(digest, signature);
    require(
        (recovered != address(0)) &&
            ((recovered == address(uint160(bytes20(sender)))) ||
                (recovered == linkedSigner)),   // ← linked signer accepted
        ERR_INVALID_SIGNATURE
    );
}
``` [2](#0-1) 

This means the existing linked signer's ECDSA signature is sufficient to authorize a `LinkSigner` transaction that replaces the linked signer with any new address — including an attacker-controlled one.

**Contrast with the slow-mode path**, which correctly restricts `LinkSigner` to the account owner via `validateSender` (an `msg.sender` check), with no signature path for the linked signer:

```solidity
// EndpointTx.sol lines 232–239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);   // msg.sender must be account owner
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
}
``` [3](#0-2) 

The slow-mode path enforces that only the account owner (`msg.sender == address(uint160(bytes20(txn.sender)))`) can mutate `linkedSigners`. The sequencer path does not enforce this restriction, creating an inconsistency that is exploitable.

**The `signer` field itself carries no validation** — it is a raw `bytes32` accepted from the transaction payload and stored directly after truncating to 20 bytes:

```solidity
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [4](#0-3) 

The `LinkSigner` struct confirms `signer` is a plain `bytes32` with no constraints:

```solidity
struct LinkSigner {
    bytes32 sender;
    bytes32 signer;
    uint64 nonce;
}
``` [5](#0-4) 

---

### Impact Explanation

A compromised linked signer (e.g., a leaked API key or hot wallet) can:

1. Sign a `LinkSigner` transaction naming an attacker-controlled address as the new `signer`.
2. Submit it to the sequencer (the normal off-chain flow).
3. The sequencer submits it on-chain; `allowLinkedSigner = true` causes the signature to be accepted.
4. `linkedSigners[victim_subaccount]` is overwritten with the attacker's address.
5. The attacker now controls the subaccount: they can sign `WithdrawCollateral`, `TransferQuote`, `LiquidateSubaccount`, and all other `allowLinkedSigner = true` transactions.
6. If the victim attempts to revoke by submitting a slow-mode `LinkSigner`, the attacker can race to re-link their address via the faster sequencer path, creating a persistent takeover.

Concrete corrupted state: `linkedSigners[victim_subaccount]` is permanently set to an attacker-controlled address, granting unauthorized signing authority over all subaccount operations.

---

### Likelihood Explanation

- Any subaccount that has ever called `LinkSigner` is exposed.
- The attacker's only precondition is obtaining the linked signer's private key (a hot wallet or API key, which is a realistic compromise target).
- No admin access, governance capture, or sequencer compromise is required.
- The attack is executable through the normal sequencer submission flow available to any user.

---

### Recommendation

In `processTransactionImpl`, change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in the sequencer path:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only account owner may change the linked signer
);
```

This aligns the sequencer path with the slow-mode path's security model: only the account owner's key can authorize a change to the linked signer. The linked signer retains all trading permissions but loses the ability to escalate its own privileges.

### Proof of Concept

**Setup:**
- Victim subaccount: `0xAlice000000000000000000000000000000000000` + 12-byte name → `bytes32 aliceSubaccount`
- Linked signer currently set to `0xHotWallet` (Alice's API key)
- Attacker has compromised `0xHotWallet`

**Steps:**

1. Attacker constructs a `SignedLinkSigner` payload:
   ```
   tx.sender = aliceSubaccount
   tx.signer = bytes32(uint256(uint160(0xAttacker)))  // attacker's address, left-padded
   tx.nonce  = currentNonce(aliceSubaccount)
   ```
2. Attacker signs the EIP-712 digest (computed over `sender`, `signer`, `nonce`) using `0xHotWallet`'s private key.
3. Attacker submits the signed transaction to the sequencer's off-chain API.
4. Sequencer calls `processTransactionImpl` on-chain.
5. `validateSignedTx(..., true)` recovers `0xHotWallet` from the signature; since `0xHotWallet == linkedSigners[aliceSubaccount]`, the check passes.
6. `linkedSigners[aliceSubaccount]` is set to `0xAttacker`.
7. Attacker can now sign `WithdrawCollateral` transactions draining Alice's collateral to any address, or sign further `LinkSigner` transactions to prevent Alice from reclaiming control.
