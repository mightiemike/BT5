Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` gates the router address instead of the originating user, enabling full per-user allowlist bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks `sender`, which the pool populates with its own `msg.sender` — the `MetricOmmSimpleRouter` contract address for all router-mediated swaps, not the originating user. A pool admin who allowlists the router (the natural action to enable router-based swaps) inadvertently grants every on-chain user access to the restricted pool, completely defeating the per-user allowlist. Conversely, if only individual user addresses are allowlisted, those users cannot swap through the router at all, breaking the standard periphery path.

## Finding Description

In `MetricOmmPool.swap`, the pool passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-231
_beforeSwap(
  msg.sender,   // ← this is the router when called via router
  ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged as the first argument to each extension's `beforeSwap` hook.

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

Here `msg.sender` is the pool and `sender` is the router address when the call originates from `MetricOmmSimpleRouter.exactInputSingle`:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    ...
  );
```

The pool's `msg.sender` is the router contract, so the extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. The actual user's address is only available in the router's transient callback context and is never forwarded to the pool or extension.

This is structurally asymmetric with `DepositAllowlistExtension.beforeAddLiquidity`, which correctly checks `owner` (the position owner, not the adder contract):

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L38
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
```

The deposit hook works because `addLiquidity` accepts an explicit `owner` parameter distinct from `msg.sender`. The swap interface has no equivalent parameter — `swap` only takes `recipient`, not a separate `swapper` identity.

## Impact Explanation

**Complete allowlist bypass (fund-impacting):** When a pool admin allowlists the router address to enable router-based swaps for their intended users, `allowedSwapper[pool][router] == true`. Any non-allowlisted user can then call `router.exactInputSingle(...)` and the extension passes unconditionally. Unauthorized swaps execute at oracle-derived prices, draining LP assets. This is a direct loss of LP principal meeting Critical/High Sherlock thresholds.

**Broken core swap flow:** If the admin allowlists only specific user addresses (not the router), those users cannot swap through the router — the extension sees `sender = router` and reverts with `NotAllowedToSwap`. The standard periphery path is permanently broken for the intended users, constituting broken core pool functionality.

## Likelihood Explanation

The bypass requires the pool admin to allowlist the router. This is the natural and expected configuration step: a pool admin deploying a restricted pool who wants allowlisted users to use the standard periphery will allowlist the router. The extension provides no mechanism to gate individual users through the router, so the admin has no correct alternative. The trigger is a predictable admin action, not a malicious one. Once the router is allowlisted, the bypass requires no special access — any user calling `router.exactInputSingle` with the target pool succeeds.

## Recommendation

The `beforeSwap` hook must gate the actual initiating user, not the intermediary router:

1. **Pass the real user via `extensionData`:** The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. Requires a coordinated convention between router and extension.
2. **Add a `swapper` parameter to the swap interface:** Mirror the deposit pattern by adding a `swapper` parameter (analogous to `owner` in `addLiquidity`) that the pool populates with the economically relevant actor. This is a protocol-level change but closes the gap cleanly and symmetrically.

Until fixed, pool admins must not allowlist the router address in `SwapAllowlistExtension`; allowlisted users must call `pool.swap()` directly.

## Proof of Concept

```
Setup:
  pool deployed with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, alice, true)    // alice is the intended user
  admin calls setAllowedToSwap(pool, router, true)   // to let alice use the router

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, recipient: bob, ...})

  router calls:
    pool.swap(bob, zeroForOne, amount, priceLimit, "", extensionData)
    // pool's msg.sender = router

  pool calls (MetricOmmPool.sol L230-231):
    _beforeSwap(sender=router, ...)

  extension checks (SwapAllowlistExtension.sol L37):
    allowedSwapper[pool][router] == true  ← passes

  Result:
    bob's swap executes at oracle price
    per-user allowlist is completely bypassed
    LP assets drained by unauthorized user

Foundry test outline:
  1. Deploy pool with SwapAllowlistExtension
  2. setAllowedToSwap(pool, router, true)
  3. Call router.exactInputSingle from an address not in the allowlist
  4. Assert swap succeeds (demonstrating bypass)
  5. Assert allowedSwapper[pool][bob] == false (confirming bob was never authorized)
```
