### Title
Stale `lastLiquidationFees` Causes Insurance Understatement During Finalization — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`lastLiquidationFees` is a global storage variable that is written only in `_handleLiquidationPayment` and read (but never reset) in `_finalizeSubaccount`. When a second subaccount is finalized without any intervening liquidation step, the stale value from the previous liquidation is subtracted from the insurance passed to `socializeSubaccount`, causing depositors to absorb more socialization loss than necessary while the insurance fund incorrectly retains the withheld amount.

---

### Finding Description

`lastLiquidationFees` is declared as a protocol-wide storage slot: [1](#0-0) 

It is written at the end of every non-finalization liquidation step: [2](#0-1) 

In `_finalizeSubaccount`, the code subtracts `lastLiquidationFees` from the working insurance before passing it to `perpEngine.socializeSubaccount`, then adds it back afterward: [3](#0-2) 

The design intent (per the inline comment at lines 581–585) is to prevent the fees just collected in the *immediately preceding* liquidation step from being consumed by socialization, since those fees were just added to `insurance` and would otherwise block the socialization gate. This is correct when the finalization immediately follows a liquidation step for the **same** subaccount.

The bug: `_finalizeSubaccount` **never resets `lastLiquidationFees` to zero** after it completes. `liquidateSubaccountImpl` returns early when `_finalizeSubaccount` returns `true`, so `_handleLiquidationPayment` is never called and `lastLiquidationFees` is not refreshed: [4](#0-3) 

This means that if a second subaccount B is finalized without any preceding liquidation step (e.g., B holds only a negative USDC balance with all positions already at zero), `lastLiquidationFees` still holds the value `X` from subaccount A's last liquidation. The insurance passed to `perpEngine.socializeSubaccount` for B is `insurance - X` instead of `insurance`.

`perpEngine.socializeSubaccount` uses the insurance parameter to cover negative `vQuoteBalance` before socializing the remainder across open-interest holders: [5](#0-4) 

After `perpEngine.socializeSubaccount` returns, the spot-level insurance cover and spot socialization gate also use the reduced `v.insurance`: [6](#0-5) 

Then `lastLiquidationFees` is added back and stored: [7](#0-6) 

Net effect: the insurance fund ends up retaining `X` that should have been applied to B's losses. Depositors absorb `X` more in socialization loss than the protocol invariant requires.

---

### Impact Explanation

The invariant "insurance must be fully applied before socialization" is broken. Concretely:

- Without the bug: `insurance` covers B's full deficit; `spotEngine.socializeSubaccount` is called only for the residual (or not at all); `insurance` ends at `0` (or the true residual).
- With the bug: `insurance - X` is applied; `spotEngine.socializeSubaccount` is called for `X` more than necessary; `insurance` ends at `X` instead of `0`.

The `X` withheld is `lastLiquidationFees` from the prior liquidation — a fraction of the prior liquidation's notional value. This is a concrete, measurable transfer of loss from the insurance fund to depositors, qualifying as an accounting failure in insurance handling that transfers value incorrectly.

---

### Likelihood Explanation

The trigger condition — a subaccount with zero positions but a negative USDC balance being finalized directly — is a normal protocol state (e.g., a subaccount that borrowed USDC and had all non-quote positions closed through prior liquidation steps targeting specific products, followed immediately by a `productId == type(uint32).max` finalization call). No special privileges are required; the sequencer processes these as ordinary signed `LiquidateSubaccount` transactions. The condition is reachable in any multi-subaccount liquidation sequence.

---

### Recommendation

Reset `lastLiquidationFees` to zero at the start of `_finalizeSubaccount` (or immediately after the `v.insurance += lastLiquidationFees` / `insurance = v.insurance` write at lines 410–411). Since finalization does not collect new fees, the correct value to exclude is `0`, not the stale value from a prior subaccount's liquidation:

```solidity
// At the top of _finalizeSubaccount, after the productId guard:
lastLiquidationFees = 0;
```

Alternatively, scope `lastLiquidationFees` per-subaccount (mapping `bytes32 => int128`) so that each subaccount's finalization only excludes fees from its own last liquidation step.

---

### Proof of Concept

```
State: insurance = 1000, lastLiquidationFees = 0

Step 1 — Liquidate subaccount A (one product step):
  _handleLiquidationPayment: liquidationFees = 50
  insurance = 1050
  lastLiquidationFees = 50

Step 2 — Finalize subaccount A (productId = type(uint32).max):
  v.insurance = 1050 - 50 = 1000   ← correct, 50 just added
  perpEngine.socializeSubaccount(A, 1000) → returns 1000 (A has no perp vQuote debt)
  insuranceCover = min(1000, -quoteBalance_A) → covers A's spot debt
  v.insurance += 50 → insurance = 1000 - debtA + 50
  lastLiquidationFees still = 50   ← NOT reset

Step 3 — Finalize subaccount B (no prior liquidation step; B has quoteBalance = -200, no positions):
  v.insurance = insurance - 50     ← stale! should be insurance - 0
  perpEngine.socializeSubaccount(B, insurance-50) → no perp positions, returns insurance-50
  insuranceCover = min(insurance-50, 200)
  Suppose insurance = 180: insuranceCover = 130, v.insurance = 0
  v.insurance <= 0 → spotEngine.socializeSubaccount(B) called  ← 50 extra socialized to depositors
  v.insurance += 50 → insurance = 50   ← fund retains 50 it should have spent

Without bug: insuranceCover = 180, v.insurance = 0, no spot socialization, insurance = 0
```

    function socializeSubaccount(bytes32 subaccount, int128 insurance)
        external
        returns (int128)
    {
        require(msg.sender == address(_clearinghouse), ERR_UNAUTHORIZED);

        uint32[] memory _productIds = getProductIds();
        for (uint128 i = 0; i < _productIds.length; ++i) {
            uint32 productId = _productIds[i];
            (State memory state, Balance memory balance) = getStateAndBalance(
                productId,
                subaccount
            );
            if (balance.vQuoteBalance < 0) {
                int128 insuranceCover = MathHelper.min(
                    insurance,
                    -balance.vQuoteBalance
                );
                insurance -= insuranceCover;
                balance.vQuoteBalance += insuranceCover;
                state.availableSettle += insuranceCover;

                // actually socialize if still not enough
                if (balance.vQuoteBalance < 0) {
                    // socialize across all other participants
                    int128 fundingPerShare = -balance.vQuoteBalance.div(
                        state.openInterest
                    );
                    state.cumulativeFundingLongX18 += fundingPerShare;
                    state.cumulativeFundingShortX18 -= fundingPerShare;
                    balance.vQuoteBalance = 0;
                }
                _setState(productId, state);
                _setBalanceAndUpdateBitmap(productId, subaccount, balance);
            }
        }
        return insurance;
    }
