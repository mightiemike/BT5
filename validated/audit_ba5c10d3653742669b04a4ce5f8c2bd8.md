### Title
Missing Liquidator Health Check in `_finalizeSubaccount` Positive PnL Settlement — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`_finalizeSubaccount` settles all positive `vQuoteBalance` from the liquidatee's perp products against the liquidator's quote balance with no subsequent health check on the liquidator. The non-finalization path (`_handleLiquidationPayment`) explicitly enforces `!isUnderInitial(txn.sender)` after every balance mutation, but the finalization path omits this guard entirely, allowing the liquidator to be left insolvent.

---

### Finding Description

**Entry point:** `Clearinghouse.liquidateSubaccount` → `delegatecall` → `ClearinghouseLiq.liquidateSubaccountImpl` → `_finalizeSubaccount` (triggered when `txn.productId == type(uint32).max`).

**Positive PnL settlement loop** (`_finalizeSubaccount`, lines 322–338):

```solidity
for (uint32 i = 0; i < v.perpIds.length; ++i) {
    uint32 perpId = v.perpIds[i];
    IPerpEngine.Balance memory balance = perpEngine.getBalance(perpId, txn.liquidatee);
    if (balance.vQuoteBalance > 0) {
        _settlePnlAgainstLiquidator(txn, perpId, balance.vQuoteBalance, spotEngine, perpEngine);
    }
}
``` [1](#0-0) 

**`_settlePnlAgainstLiquidator`** unconditionally debits the liquidator's quote:

```solidity
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, -pnl);  // line 269
``` [2](#0-1) 

After `_finalizeSubaccount` returns `true`, `liquidateSubaccountImpl` returns immediately with **no health check on `txn.sender`**:

```solidity
if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
    ...
    return;   // ← no isUnderInitial(txn.sender) check
}
``` [3](#0-2) 

**Contrast with `_handleLiquidationPayment`**, which always enforces the liquidator health invariant:

```solidity
require(
    txn.sender == N_ACCOUNT || !isUnderInitial(txn.sender),
    ERR_SUBACCT_HEALTH
);
``` [4](#0-3) 

This guard is entirely absent from the finalization code path.

---

### Impact Explanation

A liquidatee can be constructed (or arise naturally) with:
- All perp `amount == 0` (required by line 319 for finalization to proceed)
- Large positive `vQuoteBalance` across multiple perp products (accumulated from profitable closed positions)
- Large negative spot liabilities making them under maintenance health (required by line 603)

When finalization is triggered, the total positive PnL across all perp products is transferred to the liquidatee's quote balance, paid entirely by the liquidator. If the sum of positive `vQuoteBalance` values exceeds the liquidator's quote balance, the liquidator's quote goes negative. Because no health check is performed, the liquidator is left insolvent — creating bad debt in the protocol. The liquidatee's remaining negative quote is then covered by insurance or socialized, meaning the protocol absorbs the liquidatee's liabilities while the liquidator bears the cost of the liquidatee's profits with no protection.

**Critical impact:** bad debt creation, liquidator insolvency, incorrect liquidation outcome.

---

### Likelihood Explanation

The finalization path is a standard, reachable production code path invoked by any non-isolated subaccount submitting `productId = type(uint32).max`. The state preconditions (closed perp positions with accumulated positive `vQuoteBalance` and negative spot liabilities) are achievable through normal trading activity. An attacker can deliberately engineer this state: open and profitably close perp positions to accumulate `vQuoteBalance`, then borrow spot assets to push maintenance health negative, then trigger finalization against a liquidator with limited quote. No privileged access is required.

---

### Recommendation

Add the same liquidator health guard that exists in `_handleLiquidationPayment` immediately before `_finalizeSubaccount` returns `true`:

```solidity
// After all PnL settlement in _finalizeSubaccount, before returning true:
require(
    txn.sender == N_ACCOUNT || !isUnderInitial(txn.sender),
    ERR_SUBACCT_HEALTH
);
```

Alternatively, cap the total positive PnL settled against the liquidator to an amount that keeps the liquidator above initial health, or require the liquidator to pre-commit sufficient quote collateral before finalization.

---

### Proof of Concept

1. Deploy protocol with two perp products (perpId=2, perpId=4).
2. Set up `liquidatee`:
   - `perpEngine.balance[2][liquidatee] = {amount: 0, vQuoteBalance: +500_000e18}`
   - `perpEngine.balance[4][liquidatee] = {amount: 0, vQuoteBalance: +500_000e18}`
   - `spotEngine.balance[0][liquidatee] = -1_200_000e18` (quote liability → under maintenance health)
   - All spot non-quote balances ≤ 0 (finalization precondition satisfied)
3. Set up `liquidator`:
   - `spotEngine.balance[0][liquidator] = +100_000e18` (quote balance, initial health > 0)
4. Submit `LiquidateSubaccount{sender: liquidator, liquidatee: liquidatee, productId: type(uint32).max, ...}`.
5. `_finalizeSubaccount` settles `+500_000 + +500_000 = +1_000_000` quote against the liquidator.
6. `spotEngine.balance[0][liquidator]` becomes `100_000 - 1_000_000 = -900_000e18`.
7. No `isUnderInitial(txn.sender)` check fires — transaction succeeds.
8. Assert: `getHealth(liquidator, INITIAL) < 0` → liquidator is insolvent. Bad debt created.
