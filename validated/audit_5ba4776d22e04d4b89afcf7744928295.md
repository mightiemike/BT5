### Title
`MINIMUM_LIQUIDITY` Defined But Never Enforced — Large NLP Holder Can Burn Below Minimum Floor via `burnNlp` - (File: core/contracts/Clearinghouse.sol)

---

### Summary

`MINIMUM_LIQUIDITY` (`10**3`) is declared in `Constants.sol` but is never referenced or enforced anywhere in the production codebase. The `burnNlp` function in `Clearinghouse.sol` contains no check that the remaining NLP supply stays at or above this floor after a burn. A large NLP holder can burn their entire unlocked position, reducing the total circulating NLP supply below `MINIMUM_LIQUIDITY` with no on-chain resistance.

---

### Finding Description

`MINIMUM_LIQUIDITY` is defined at: [1](#0-0) 

A `grep` across all production Solidity files confirms it has exactly **one occurrence** — its definition. It is imported nowhere and checked nowhere.

The `burnNlp` path in `Clearinghouse.sol` is: [2](#0-1) 

The only guard present is that the caller has enough **unlocked** NLP to cover the requested burn amount: [3](#0-2) 

After that check passes, the function unconditionally debits the caller's NLP balance and credits `N_ACCOUNT`: [4](#0-3) 

There is no post-burn assertion that the total NLP in circulation (i.e., the sum of all user NLP balances, equivalently the negative of `N_ACCOUNT`'s balance) remains `>= MINIMUM_LIQUIDITY`. The `mintNlp` path similarly never seeds a permanent minimum-liquidity lock: [5](#0-4) 

---

### Impact Explanation

`MINIMUM_LIQUIDITY` exists to protect the NLP pool from being fully drained to near-zero supply. When the circulating NLP supply approaches zero:

- The oracle-price-based NLP valuation (`nlpAmount = quoteAmount.div(oraclePriceX18)`) can produce extreme or undefined results for subsequent minters.
- The NLP rebalance accounting (`_validateNlpRebalance`, `_applyNlpRebalance`) operates over pool subaccounts whose balances are derived from the NLP supply; a near-zero supply corrupts these proportional calculations.
- The `N_ACCOUNT` balance, which tracks unissued NLP, can be manipulated to an extreme positive value, breaking the invariant that `N_ACCOUNT.balance + sum(user NLP balances) == total NLP supply`.

The corrupted state delta is: **total circulating NLP supply drops below `MINIMUM_LIQUIDITY`**, violating the protocol's own declared invariant.

---

### Likelihood Explanation

Any NLP holder whose unlocked balance constitutes the majority of circulating NLP supply can trigger this. The NLP lock period is only 4 days (`NLP_LOCK_PERIOD = 4 * 24 * 60 * 60`): [6](#0-5) 

After the lock expires, `getNlpUnlockedBalance` returns the full unlocked amount and the burn proceeds with no floor check. A whale who minted the majority of NLP supply can wait 4 days and drain the pool below `MINIMUM_LIQUIDITY` in a single `BurnNlp` transaction routed through the `Endpoint`. [7](#0-6) 

---

### Recommendation

Add a post-burn check in `burnNlp` that enforces the minimum liquidity floor. After debiting the caller's NLP and crediting `N_ACCOUNT`, assert:

```solidity
// total circulating NLP = negative of N_ACCOUNT's balance
int128 nAccountBalance = spotEngine.getBalance(NLP_PRODUCT_ID, N_ACCOUNT).amount;
require(
    -nAccountBalance >= int128(uint128(MINIMUM_LIQUIDITY)),
    "NLP supply below MINIMUM_LIQUIDITY"
);
```

Alternatively, mirror the Uniswap V2 pattern: on the **first** `mintNlp`, permanently lock `MINIMUM_LIQUIDITY` NLP tokens to `address(0)` so they can never be burned, guaranteeing the floor is always maintained.

---

### Proof of Concept

1. Alice (whale) calls `mintNlp` with a large `quoteAmount`, receiving the vast majority of circulating NLP supply. `N_ACCOUNT` is debited accordingly.
2. All other NLP holders hold only a negligible fraction of supply.
3. Alice waits 4 days for `NLP_LOCK_PERIOD` to expire.
4. Alice submits a `BurnNlp` transaction via the `Endpoint` for her entire unlocked NLP balance.
5. `burnNlp` passes the only guard (`getNlpUnlockedBalance(txn.sender).amount >= nlpAmount`) and executes the burn.
6. After execution, total circulating NLP = (other holders' tiny balances) which is `< MINIMUM_LIQUIDITY = 10**3`.
7. No revert occurs. The NLP pool now operates below its declared minimum liquidity floor, corrupting subsequent `mintNlp`/`burnNlp` accounting and NLP rebalance calculations. [2](#0-1) [1](#0-0)

```text
    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
```
