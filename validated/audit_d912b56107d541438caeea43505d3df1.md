### Title
`PriceVelocityGuardExtension` First-Swap Velocity Check Bypass Due to Uninitialized `lastMidPriceX64` — (File: `metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

The `PriceVelocityGuardExtension.beforeSwap` velocity enforcement block is entirely skipped on the first swap for every pool because `lastMidPriceX64` defaults to zero. The guard is gated behind `if (prevMid != 0)`, which is always false on the first call, allowing the first swap to execute at any oracle price regardless of how far it has moved from the pool's intended reference.

---

### Finding Description

In `PriceVelocityGuardExtension.beforeSwap`, the function reads the stored `lastMidPriceX64` into `prevMid`, immediately overwrites the storage with the current oracle mid-price, and then gates the entire velocity check behind `if (prevMid != 0)`:

```solidity
uint128 prevMid = s.lastMidPriceX64;
uint64 prevBlock = s.lastUpdateBlock;

s.lastMidPriceX64 = midPrice;
s.lastUpdateBlock = uint64(block.number);

if (prevMid != 0) {                          // ← always false on first swap
    uint64 maxChange = s.maxChangePerBlockE18;
    if (maxChange != 0) {
        // squared-envelope velocity check
    }
}
``` [1](#0-0) 

Because `lastMidPriceX64` is a Solidity storage mapping that zero-initialises, `prevMid` is `0` on the first `beforeSwap` call for any pool. The entire velocity enforcement block is skipped unconditionally, and the swap settles at whatever oracle price is currently reported, regardless of how extreme that price is.

The admin can call `setLastMidPrice` to pre-seed the reference price before the first swap: [2](#0-1) 

However, this is not enforced. Unlike `OracleValueStopLossExtension`, which has a factory-gated `initialize` function that atomically sets all config at pool creation, `PriceVelocityGuardExtension` has no `initialize` hook: [3](#0-2) 

The extension silently fails open on the first swap for every pool where the admin has not manually pre-seeded the reference price. This is the direct structural analog of the `claimTimeout` bug: just as `reward.claimedAt` defaulting to zero allows the first claim to bypass the timeout, `lastMidPriceX64` defaulting to zero allows the first swap to bypass the velocity cap.

The pool's `swap` function calls `_beforeSwap(msg.sender, ...)` before settlement, so the extension is on the critical path: [4](#0-3) 

The `ExtensionCalling._beforeSwap` dispatcher passes the oracle bid/ask prices that the extension uses to derive `midPrice`: [5](#0-4) 

---

### Impact Explanation

The velocity guard is specifically designed to protect LPs from rapid oracle price movements. If the oracle price has moved dramatically before the first swap (due to a market event or oracle feed anomaly), the first swap executes at the current extreme price. LPs lose principal because the pool settles a trade at a price the velocity guard was configured to block. The loss is bounded to the first swap per pool, but for pools with significant liquidity the LP loss can be material.

**Impact: Medium** — direct loss of LP principal on the first swap of any pool using this extension without admin pre-seeding.

---

### Likelihood Explanation

This affects every pool using `PriceVelocityGuardExtension` where the admin does not manually call `setLastMidPrice` before the first swap. The extension provides no `initialize` hook and no on-chain enforcement of pre-seeding. The bypass window is open from pool deployment until the first swap executes, which is the highest-risk period (pool launch, no prior price history).

**Likelihood: High** — will occur for every pool that does not have the admin manually pre-seed the reference price, which is the default state.

---

### Recommendation

Add a factory-gated `initialize` function to `PriceVelocityGuardExtension` (mirroring `OracleValueStopLossExtension.initialize`) that accepts an initial mid-price and sets `lastMidPriceX64` and `lastUpdateBlock` atomically at pool creation. This ensures the velocity guard is active from the very first swap. Alternatively, if the first-swap skip is intentional by design, remove `setLastMidPrice` to avoid the false impression that pre-seeding activates the guard from block zero, and document the open first-swap window explicitly.

---

### Proof of Concept

1. Deploy a pool with `PriceVelocityGuardExtension` configured with `maxChangePerBlockE18 = 1e16` (1% per block cap).
2. Do **not** call `setLastMidPrice` — `lastMidPriceX64` remains `0`.
3. Allow the oracle price to move 50% (well beyond the 1% cap) before any swap occurs.
4. Execute the first swap on the pool.
5. `beforeSwap` reads `prevMid = 0`, the `if (prevMid != 0)` block is skipped entirely, and the swap executes at the 50%-moved oracle price.
6. LPs receive far less than the velocity-guarded price would have permitted, losing principal on the settlement.
7. After this first swap, `lastMidPriceX64` is set to the 50%-moved price, and all subsequent swaps are correctly checked against it — the guard is permanently active only after the damage is done.
