### Title
`verify_transaction_inclusion` Rejects Legitimately Empty Merkle Proofs, Causing `verify_transaction_inclusion_v2` to Panic for Single-Transaction Blocks - (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` contains an overly strict guard `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` that unconditionally rejects empty merkle proofs. For a Bitcoin block containing exactly one transaction (the coinbase), the merkle proof is legitimately empty — the transaction hash IS the merkle root. `verify_transaction_inclusion_v2` (the recommended, non-deprecated entry point) correctly validates the coinbase sub-proof for this case, but then delegates to `verify_transaction_inclusion`, which panics on the empty-proof guard. A legitimate NEAR caller supplying a fully valid proof for a single-transaction block is denied a successful verification result.

### Finding Description

`verify_transaction_inclusion_v2` performs two sequential checks before returning a result:

1. **Coinbase proof check** — verifies that `compute_root_from_merkle_proof(coinbase_tx_id, 0, &coinbase_merkle_proof) == header.merkle_root`.
2. **Transaction proof check** — delegates to `verify_transaction_inclusion(args.into())`. [1](#0-0) 

For a block with exactly one transaction (the coinbase), the correct inputs are:

- `tx_id = coinbase_tx_id`
- `tx_index = 0`
- `merkle_proof = []` (empty — the tx hash is the root)
- `coinbase_merkle_proof = []` (same length, also empty)

Step 1 passes: `compute_root_from_merkle_proof` with an empty proof simply returns `transaction_hash` unchanged (the `for` loop over an empty slice is a no-op), so the result equals `merkle_root` for a single-tx block. [2](#0-1) 

Step 2 panics: `verify_transaction_inclusion` contains the guard:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [3](#0-2) 

This guard fires unconditionally on an empty slice, even though `compute_root_from_merkle_proof` would have produced the correct root without any proof elements. The guard is structurally analogous to the original report's `require(rewards > 0)` that checked only one reward sub-component while ignoring the other: here, the coinbase sub-proof validation succeeds, but the transaction-proof guard causes the entire call to abort.

### Impact Explanation

Any unprivileged NEAR caller — including a downstream smart contract that uses `verify_transaction_inclusion_v2` as a bridge gate — that supplies a valid SPV proof for a coinbase transaction in a single-transaction block will receive a panic instead of `true`. A contract that gates fund release or cross-chain state transitions on a `true` result from `verify_transaction_inclusion_v2` will be permanently blocked for this class of valid proofs, with no alternative non-deprecated path available. [4](#0-3) 

### Likelihood Explanation

Single-transaction blocks (coinbase only) occur on mainnet during periods of very low mempool activity and are common on testnets and regtest environments used by integrators. The `verify_transaction_inclusion_v2` function is explicitly the recommended replacement for the deprecated v1, so integrators are directed toward the path that contains the bug. The trigger requires no privileged role — any caller can supply the described arguments.

### Recommendation

Remove the `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard from `verify_transaction_inclusion`. The underlying `compute_root_from_merkle_proof` already handles the empty-proof case correctly (returning the transaction hash itself), so the guard provides no safety benefit and actively rejects valid inputs. If a non-empty proof is desired as a policy constraint, the check should be moved to `verify_transaction_inclusion_v2` only and conditioned on `tx_index != 0` or block transaction count, not applied unconditionally. [3](#0-2) 

### Proof of Concept

1. Submit a Bitcoin block header for a block whose `merkle_root` equals a single coinbase `tx_id` (single-transaction block).
2. Call `verify_transaction_inclusion_v2` with:
   - `tx_id = <coinbase_tx_id>`
   - `tx_block_blockhash = <block_hash>`
   - `tx_index = 0`
   - `merkle_proof = []`
   - `coinbase_tx_id = <coinbase_tx_id>`
   - `coinbase_merkle_proof = []`
   - `confirmations = 1`
3. The length-equality check passes (`0 == 0`).
4. The coinbase proof check passes: `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[])` returns `coinbase_tx_id`, which equals `header.block_header.merkle_root`.
5. `verify_transaction_inclusion(args.into())` is called; execution reaches `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` and panics — despite the proof being cryptographically valid. [1](#0-0) [2](#0-1)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
