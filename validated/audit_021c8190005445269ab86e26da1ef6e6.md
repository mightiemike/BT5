### Title
`DepositAllowlistExtension.beforeAddLiquidity` gates on `owner` instead of `sender`, allowing any unprivileged caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is documented as gating `addLiquidity` by **depositor address**. Its `beforeAddLiquidity` hook, however, validates the `owner` argument (the position beneficiary) rather than the `sender` argument (the actual `msg.sender` who called `addLiquidity` and who will pay the tokens). Because `MetricOmmPool.addLiquidity` explicitly supports an operator pattern where `msg.sender ≠ owner`, any address can deposit into a restricted pool by supplying an allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct addresses to the extension hook:

```
_beforeAddLiquidity(msg.sender /*sender*/, owner /*owner*/, salt, deltas, extensionData);
``` [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` receives both but silently discards `sender` (the first parameter is unnamed `address`) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [2](#0-1) 

The contract's own NatSpec and the admin setter both use the word **depositor**, not owner: [3](#0-2) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the first parameter):

```solidity
function beforeSwap(address sender, address, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
``` [4](#0-3) 

The pool's own interface documentation confirms the operator pattern is intentional and unsupervised:

> "msg.sender pays but need not equal owner (operator pattern)." [5](#0-4) 

`_validateOwner` in the periphery only rejects `address(0)` — it does not require `owner == msg.sender`: [6](#0-5) 

---

### Impact Explanation

A pool admin deploys a restricted pool with `DepositAllowlistExtension` to enforce KYC/whitelist controls on who may provide liquidity. Any unprivileged address can bypass this control by calling `pool.addLiquidity(allowlistedAddress, salt, deltas, callbackData, "")` directly. The extension sees `owner = allowlistedAddress` (which passes the check) and never inspects the actual caller. The unauthorized caller pays the tokens via the modify-liquidity callback and the position is minted under the allowlisted address. The pool admin's core access-control invariant — that only approved depositors may add liquidity — is broken for every pool using this extension.

---

### Likelihood Explanation

- The pool's `addLiquidity` is a public `external` function with no caller restriction of its own.
- The operator pattern (`sender ≠ owner`) is explicitly documented and tested.
- Any address that knows one allowlisted address can execute the bypass with zero special privilege.
- The `DepositAllowlistExtension` is a production periphery contract intended for real pool deployments.

---

### Recommendation

Change `beforeAddLiquidity` to validate `sender` (the actual depositor/caller) instead of `owner`, consistent with how `SwapAllowlistExtension` handles `beforeSwap`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is instead to gate by position owner (not caller), the NatSpec, the setter parameter name (`depositor`), and the event name (`AllowedToDepositSet`) must all be updated to reflect that, and the asymmetry with `SwapAllowlistExtension` must be explicitly documented.

---

### Proof of Concept

```solidity
// Pool configured with DepositAllowlistExtension.
// allowedDepositor[pool][alice] = true  (alice is whitelisted)
// bob is NOT whitelisted

// Bob calls the pool directly — no periphery router involved
pool.addLiquidity(
    alice,          // owner = allowlisted address → check passes
    salt,
    deltas,
    callbackData,   // bob's own callback that pays tokens
    ""
);

// Result: bob's tokens are in the pool, position credited to alice.
// The allowlist guard was never triggered for bob.
```

The `CallExtension` library correctly validates the returned selector, so the extension call itself succeeds and returns `beforeAddLiquidity.selector`, completing the bypass silently. [7](#0-6)
