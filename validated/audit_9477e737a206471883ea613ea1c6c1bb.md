### Title
Stale `_cached_sha256_treehash` Propagated by `Program.wrap()` Causes Tree-Hash Mismatch After Atom Mutation — (File: `wheel/python/clvm_rs/program.py`)

---

### Summary

`Program.wrap()` unconditionally copies `_cached_sha256_treehash` from any `CLVMStorage` object passed to it. Because `atom` is a plain, publicly writable Python attribute with no setter guard or cache-invalidation hook, a caller can supply a `CLVMStorage` object whose cached hash no longer matches its actual atom content. After wrapping, `Program.tree_hash()` returns the stale hash without recomputing, producing a puzzle-hash mismatch that propagates silently through equality checks, `curry_hash()`, and serialization.

---

### Finding Description

**Root cause — `Program.wrap()` blindly inherits the cached hash:** [1](#0-0) 

```python
o._cached_serialization = getattr(v, "_cached_serialization", None)
o._cached_sha256_treehash = getattr(v, "_cached_sha256_treehash", None)
```

There is no check that the copied hash actually corresponds to `v.atom`. If `v._cached_sha256_treehash` was set when `v.atom` held a different value, the `Program` wrapper silently inherits the wrong hash.

**Cache is written to the underlying object by `sha256_treehash`:** [2](#0-1) 

```python
r = getattr(obj, "_cached_sha256_treehash", None)
if r is not None:
    self.cache_hits += 1
    hash_stack.append(r)
    return
...
try:
    setattr(obj, "_cached_sha256_treehash", r)
except AttributeError:
    pass
```

Once `_cached_sha256_treehash` is set on a `CLVMStorage` object, every subsequent call to `sha256_treehash` on that object returns the cached value immediately, regardless of whether `atom` has changed.

**`Program.tree_hash()` trusts the cached value unconditionally:** [3](#0-2) 

```python
def tree_hash(self) -> bytes:
    if self._cached_sha256_treehash is None:
        self._cached_sha256_treehash = sha256_treehash(self._unwrapped)
    return self._cached_sha256_treehash
```

**`atom` is a plain mutable attribute — no setter guard exists:** [4](#0-3) 

The `CLVMStorage` protocol declares `atom: Optional[bytes]` as a plain instance attribute. Neither `CLVMStorage` nor `Program` defines a property setter that would clear `_cached_sha256_treehash` or `_cached_serialization` on mutation.

**Same staleness applies to `_cached_serialization`:** [5](#0-4) 

```python
r = getattr(sexp, "_cached_serialization", None)
if r is not None:
    yield r
    continue
```

`sexp_to_byte_iterator` (used by `__bytes__` and therefore by `run_with_cost`) also short-circuits on a stale `_cached_serialization`, meaning the wrong byte sequence is serialized and executed.

---

### Impact Explanation

| Affected API | Corrupted result |
|---|---|
| `Program.tree_hash()` | Returns hash of the *old* atom, not the current one |
| `Program.__eq__()` | Equality comparison uses wrong hash; two distinct programs may compare equal or unequal incorrectly |
| `Program.curry_hash()` | Puzzle hash computed from wrong tree hash |
| `Program.__bytes__()` / `run_with_cost()` | Stale `_cached_serialization` causes the wrong CLVM byte sequence to be executed |

The most severe downstream consequence is that `run_with_cost()` serializes the stale bytes and passes them to the Rust evaluator, so the program that actually runs differs from the program the caller believes it is running. In wallet code this can produce wrong puzzle hashes; in node code it can cause API-level divergence between what is validated and what is stored.
