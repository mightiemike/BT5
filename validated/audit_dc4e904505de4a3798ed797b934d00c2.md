Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of originating user, breaking allowlist enforcement for all router-mediated swaps — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to every configured extension. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. This produces two broken outcomes: if the router is allowlisted, every user bypasses the curated pool's access control; if the router is not allowlisted, every allowlisted user is blocked from using the protocol's primary swap interface.

## Finding Description
`MetricOmmPool.swap` captures `msg.sender` and forwards it as `sender` to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // direct caller of pool.swap()
    recipient, ...
);
```

`ExtensionCalling._beforeSwap` encodes that value and dispatches it to every extension in `BEFORE_SWAP_ORDER`:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks that forwarded value against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension's caller) and `sender` is whoever called `pool.swap()`.

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
```

The router stores the real user address only in transient callback context (for payment), but the pool never sees it. The effective call chain is:

```
user → router.exactInputSingle()
         → pool.swap()          [msg.sender = router]
             → extension.beforeSwap(sender = router, ...)
                 → allowedSwapper[pool][router]   ← wrong actor
```

The same misbinding applies to `exactInput` (line 104), `exactOutputSingle` (line 136), `exactOutput` (line 165), and the recursive `_exactOutputIterateCallback` (line 220), covering every public router entry point.

No existing guard corrects this: the pool has no mechanism to surface the original user address to extensions, and the extension has no fallback identity check.

## Impact Explanation
**Scenario A — router allowlisted (bypass):** A pool admin who allowlists the router as a trusted intermediary inadvertently opens the pool to every user. Any EOA not in the allowlist can call `router.exactInputSingle()` and the extension passes because `allowedSwapper[pool][router] == true`. The curated pool's access control is fully defeated — the exact wrong value evaluated is `allowedSwapper[pool][router]` where `allowedSwapper[pool][user]` is the invariant-correct check.

**Scenario B — router not allowlisted (broken functionality):** A legitimately allowlisted user who routes through the router receives `NotAllowedToSwap` because `allowedSwapper[pool][router] == false`. The only working path is a direct `pool.swap()` call, making the router — the protocol's documented primary swap interface — unusable for any curated pool. This constitutes broken core swap functionality.

Both outcomes break the core invariant that the allowlist enforces the same policy regardless of which supported public entrypoint reaches the pool.

## Likelihood Explanation
Every swap routed through `MetricOmmSimpleRouter` against a pool that has `SwapAllowlistExtension` in its `BEFORE_SWAP_ORDER` triggers this mismatch. The router is the documented primary swap path. Any curated pool that deploys this extension and expects users to use the router is immediately affected. No special attacker capability is required — a normal EOA calling any router function suffices.

## Recommendation
The pool must surface the original user's address to extensions. The preferred fix is a pool-level change: add an `originalSender` field populated from a transient-storage slot that the router writes before calling `pool.swap()` (the router already uses transient storage for callback context via `_setNextCallbackContext`). The extension then checks `originalSender` instead of `sender`. Alternatively, require callers to embed their address in `extensionData` with a verifiable signature, but this shifts burden to every integrator and is fragile.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` in `BEFORE_SWAP_ORDER`.
2. **Scenario A:** Call `setAllowedToSwap(pool, router, true)`. Call `router.exactInputSingle(...)` from any EOA not individually allowlisted. The swap succeeds because `allowedSwapper[pool][router] == true` — the allowlist is bypassed.
3. **Scenario B:** Leave the router un-allowlisted. Call `setAllowedToSwap(pool, user, true)`. Call `router.exactInputSingle(...)` from that user. The swap reverts with `NotAllowedToSwap` because `allowedSwapper[pool][router] == false`, even though the user is individually permitted. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
