### Title
`get_tip_resource_bounds_hash` Produces Divergent Hashes for `ValidResourceBounds::L1Gas` vs `AllResources` with Zero L2/L1DataGas, Causing Transaction Hash Mismatch Between RPC Ingestion and Protobuf State-Sync Deserialization Paths - (File: crates/starknet_api/src/transaction_hash.rs)

### Summary

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` produces **structurally different hash preimages** for `ValidResourceBounds::L1Gas(bounds)` versus `ValidResourceBounds::AllResources(AllResourceBounds { l1_gas: bounds, l2_gas: ZERO, l1_data_gas: ZERO })`. The protobuf deserializer `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` silently collapses any transaction whose `l2_gas` and `l1_data_gas` are both zero into `ValidResourceBounds::L1Gas`, while the RPC/gateway ingestion path always produces `ValidResourceBounds::AllResources` (via `InternalRpcInvokeTransactionV3`). The same on-wire transaction bytes therefore yield two different transaction hashes depending on which deserialization path is taken, breaking the canonical-hash invariant that every component must agree on.

### Finding Description

**Step 1 – The hash function branches on variant, not on values.**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` lines 188–211:

```
L1Gas  → poseidon( tip, concat(L1_GAS, amount, price), concat(L2_GAS, 0, 0) )
AllResources → poseidon( tip, concat(L1_GAS, amount, price), concat(L2_GAS, 0, 0), concat(L1_DATA_GAS, 0, 0) )
```

The `L1Gas` branch omits the `L1_DATA_GAS` element entirely; `AllResources` always appends it, even when it is zero. These two Poseidon calls produce **different field elements** for identical numeric inputs.

**Step 2 – The protobuf deserializer collapses `AllResources(zeros)` → `L1Gas`.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` lines 417–436:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

Any transaction whose `l2_gas` and `l1_data_gas` are zero is silently re-typed as `L1Gas` on deserialization. This converter is used in the state-sync path (`TryFrom<protobuf::InvokeV3> for InvokeTransactionV3`, lines 593–660 of `transaction.rs`).

**Step 3 – The RPC/gateway path always uses `AllResources`.**

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds` (never `ValidResourceBounds`). `InternalRpcInvokeTransactionV3` also carries `resource_bounds: AllResourceBounds`. Its `InvokeTransactionV3Trait` implementation returns `ValidResourceBounds::AllResources(self.resource_bounds)` unconditionally (`crates/starknet_api/src/rpc_transaction.rs` lines 636–638). The hash computed at the gateway therefore always uses the `AllResources` branch → **hash H₁** (4-element Poseidon).

**Step 4 – The state-sync path computes a different hash.**

When the same transaction is received via state-sync protobuf, `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3` produces `resource_bounds: ValidResourceBounds::L1Gas(...)`. `InvokeTransactionV3::calculate_transaction_hash` calls `get_invoke_transaction_v3_hash` which calls `get_tip_resource_bounds_hash` with `L1Gas` → **hash H₂** (3-element Poseidon). H₁ ≠ H₂.

**Step 5 – The `rpc_transaction.rs` protobuf converter is immune but the `transaction.rs` converter is not.**

The mempool/RPC protobuf path uses `TryFrom<protobuf::ResourceBounds> for AllResourceBounds` (`crates/apollo_protobuf/src/converters/rpc_transaction.rs` lines 212–223), which always produces `AllResourceBounds` and is therefore consistent. Only the state-sync `FullTransaction` path uses the diverging `ValidResourceBounds` converter.

### Impact Explanation

A transaction submitted via the RPC gateway with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is committed to a block with hash H₁. Any node that later receives this block via state-sync deserializes the transaction as `L1Gas` and recomputes hash H₂ ≠ H₁. Concretely:

- `validate_transaction_hash` (`transaction_hash.rs` lines 170–185) will return `false` for this transaction when called on the state-sync-deserialized form, causing the node to treat a valid committed transaction as having an invalid hash.
- The `starknet_getTransactionReceipt` / `starknet_getTransactionByHash` RPC endpoints serve H₁, but any client that independently recomputes the hash from the transaction fields (as the Starknet spec requires) will obtain H₂, making the receipt authoritative-looking but wrong.
- Block hash computation that folds transaction hashes (`concat_counts`, `PartialBlockHash`) will diverge between the sequencer and a re-executing verifier node.

This matches the allowed impact: **"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value"** and **"Wrong state, receipt, event … or revert result from blockifier/syscall/execution logic for accepted input."**

### Likelihood Explanation

The trigger is a valid Invoke V3 transaction with `l2_gas = 0` and `l1_data_gas = 0`. This is a legal pre-0.13.3-style transaction that the gateway accepts today (the `RpcInvokeTransactionV3` struct places no lower bound on these fields). Any user or automated tool submitting a V3 transaction without explicitly setting L2/L1DataGas bounds will produce this condition. No privileged access is required; the transaction is accepted through normal RPC admission.

### Recommendation

1. **Fix the protobuf deserializer** in `crates/apollo_protobuf/src/converters/transaction.rs` to never produce `ValidResourceBounds::L1Gas` for V3 transactions received via state-sync. V3 transactions must always be deserialized as `AllResources`, matching the RPC path. The `L1Gas` variant should only be produced for explicitly versioned pre-0.13.3 transactions.

2. **Alternatively, fix `get_tip_resource_bounds_hash`** to produce an identical hash for `L1Gas(bounds)` and `AllResources(bounds, ZERO, ZERO)` by always appending the `L1_DATA_GAS` element (even when zero) for V3 transactions, making the hash depend only on numeric values and not on the Rust enum variant.

3. **Add a round-trip hash test** that serializes an `InvokeTransactionV3` with zero L2/L1DataGas to protobuf, deserializes it, and asserts that `calculate_transaction_hash` returns the same value before and after the round-trip.
