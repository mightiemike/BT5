### Title
`WithdrawCollateralV2` Accepted by Slow-Mode Submission but Silently Fails on Execution, Causing Permanent Fee Loss — (`core/contracts/EndpointTx.sol`)

---

### Summary

`EndpointTx.submitSlowModeTransactionImpl()` accepts `WithdrawCollateralV2` transactions and charges a slow-mode fee, but `EndpointTx.processSlowModeTransactionImpl()` has no handler for this transaction type and unconditionally reverts. Because `_executeSlowModeTransaction()` silently swallows the revert, the user's slow-mode fee is permanently lost and the withdrawal never executes. This is a direct analog to the external report's class: a routing/dispatch layer accepts a user operation and takes custody of a fee, but the execution layer has no corresponding implementation, making the operation permanently inaccessible through that path.

---

### Finding Description

`IEndpoint.TransactionType` defines `WithdrawCollateralV2` (enum index 32), which extends `WithdrawCollateral` with a `sendTo` field for directing withdrawals to a custom address. [1](#0-0) 

`processTransactionImpl()` fully handles `WithdrawCollateralV2` in the sequencer path, including signature validation and fee charging. [2](#0-1) 

However, `processSlowModeTransactionImpl()` handles `WithdrawCollateral` but has **no case for `WithdrawCollateralV2`**. Any unrecognized type falls through to `else { revert(); }`. [3](#0-2) [4](#0-3) 

The desynchronization is in `submitSlowModeTransactionImpl()`. It has no guard for `WithdrawCollateralV2` — the type falls into the `else` branch, which charges `SLOW_MODE_FEE` via `chargeSlowModeFee()` (a real ERC20 `safeTransferFrom`) and stores the transaction in `slowModeTxs`. [5](#0-4) 

When the stored transaction is later executed, `_executeSlowModeTransaction()` calls `processSlowModeTransaction` inside a `try/catch`. The revert from the unhandled type is silently swallowed. The comment `// try return funds now removed` confirms that fee refunds on failure were intentionally removed, so the slow-mode fee is permanently lost. [6](#0-5) 

The `chargeSlowModeFee` function in `EndpointStorage` performs a real token transfer from the user: [7](#0-6) 

---

### Impact Explanation

Any user who submits a `WithdrawCollateralV2` slow-mode transaction suffers two concrete harms:

1. **Permanent fee loss**: The slow-mode fee (an ERC20 token amount scaled by `SLOW_MODE_FEE`) is transferred from the user and never returned.
2. **Withdrawal never executes**: The user's collateral remains locked in the protocol. This is especially severe when the sequencer is unresponsive and slow mode is the only available withdrawal path. Users who specifically need to withdraw to a custom `sendTo` address (e.g., a hardware wallet or a different contract) have no alternative path via slow mode — `WithdrawCollateral` in slow mode hardcodes `sendTo = address(0)`, sending only to the default subaccount address. [3](#0-2) 

---

### Likelihood Explanation

The slow-mode path is the user's censorship-resistance fallback when the sequencer is unresponsive. A user who has previously used `WithdrawCollateralV2` via the sequencer path (the normal path) would reasonably attempt the same transaction type via slow mode during a sequencer outage. The `submitSlowModeTransactionImpl` function gives no indication that the type is unsupported — it accepts the transaction and charges the fee without error. The likelihood is **medium**: it requires a sequencer outage scenario, but within that scenario the trigger is trivially reachable by any user.

---

### Recommendation

Add a `WithdrawCollateralV2` handler to `processSlowModeTransactionImpl()` mirroring the existing `WithdrawCollateral` handler, decoding the `sendTo` field and passing it to `clearinghouse.withdrawCollateral()`. Alternatively, add an explicit revert for `WithdrawCollateralV2` in `submitSlowModeTransactionImpl()` so users are not charged a fee for an operation that cannot succeed.

---

### Proof of Concept

```
1. User calls Endpoint.submitSlowModeTransaction(
       abi.encodePacked(uint8(TransactionType.WithdrawCollateralV2), abi.encode(signedTx))
   )
2. submitSlowModeTransactionImpl: WithdrawCollateralV2 is not DepositCollateral,
   not in the owner-only list → falls to else branch
   → chargeSlowModeFee transfers SLOW_MODE_FEE tokens from user
   → transaction stored in slowModeTxs[txCount]
3. After SLOW_MODE_TX_DELAY (3 days), anyone calls executeSlowModeTransaction()
4. _executeSlowModeTransaction calls this.processSlowModeTransaction(sender, tx)
   inside try/catch
5. processSlowModeTransactionImpl: no case for WithdrawCollateralV2
   → hits else { revert(); }
6. try/catch in _executeSlowModeTransaction swallows the revert silently
7. Result: user's slow-mode fee is permanently lost, withdrawal never executed,
   collateral remains locked
``` [8](#0-7) [9](#0-8)

    function submitSlowModeTransactionImpl(bytes calldata transaction) public {
        IEndpoint.TransactionType txType = IEndpoint.TransactionType(
            uint8(transaction[0])
        );

        // special case for DepositCollateral because upon
        // slow mode submission we must take custody of the
        // actual funds

        address sender = msg.sender;

        if (txType == IEndpoint.TransactionType.DepositCollateral) {
            revert();
        } else if (txType == IEndpoint.TransactionType.DepositInsurance) {
            IEndpoint.DepositInsurance memory txn = abi.decode(
                transaction[1:],
                (IEndpoint.DepositInsurance)
            );
            require(
                txn.amount >= uint128(SLOW_MODE_FEE),
                ERR_DEPOSIT_TOO_SMALL
            );
            handleDepositTransfer(_getQuote(), sender, uint256(txn.amount));
        } else if (
            txType == IEndpoint.TransactionType.WithdrawInsurance ||
            txType == IEndpoint.TransactionType.DelistProduct ||
            txType == IEndpoint.TransactionType.DumpFees ||
            txType == IEndpoint.TransactionType.RebalanceXWithdraw ||
            txType == IEndpoint.TransactionType.UpdateTierFeeRates ||
            txType == IEndpoint.TransactionType.AddNlpPool ||
            txType == IEndpoint.TransactionType.UpdateNlpPool ||
            txType == IEndpoint.TransactionType.DeleteNlpPool ||
            txType == IEndpoint.TransactionType.ForceRebalanceNlpPool ||
            txType == IEndpoint.TransactionType.NlpProfitShare ||
            txType == IEndpoint.TransactionType.UpdateBuilder
        ) {
            require(sender == owner());
        } else {
            chargeSlowModeFee(_getQuote(), sender);
            slowModeFees += SLOW_MODE_FEE;
        }

        IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
        requireUnsanctioned(sender);
        slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: transaction
        });
        // TODO: to save on costs we could potentially just emit something
        // for now, we can just create a separate loop in the engine that queries the remote
        // sequencer for slow mode transactions, and ignore the possibility of a reorgy attack
        slowModeConfig = _slowModeConfig;
    }

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
