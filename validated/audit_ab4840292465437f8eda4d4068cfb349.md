### Title
Missing Negative Modulus Validation in `op_modpow` Produces Incorrect Results and Consensus Divergence Between `num_bigint` and `malachite_bigint` Paths - (File: src/more_ops.rs)

### Summary
`op_modpow` explicitly rejects a negative exponent and a zero modulus, but does not reject a **negative modulus**. An attacker-controlled CLVM atom with its high bit set (e.g., `0xf9` = −7 in two's complement) passes the zero-check and is forwarded directly to `num_bigint::BigInt::modpow` or `malachite_bigint::BigInt::modpow`. Because the two libraries handle negative modulus differently, the same program produces different `NodePtr` values on nodes running with versus without the `MALACHITE` flag, constituting a consensus divergence.

### Finding Description

`op_modpow` performs three sign checks before calling the underlying bignum library: [1](#0-0) 

```
if exponent.sign() == Sign::Minus  → error (negative exponent rejected)
if modulus.sign() == Sign::NoSign  → error (zero modulus rejected)
// ← no check for modulus.sign() == Sign::Minus
let ret = base.modpow(&exponent, &modulus);   // called with negative modulus
```

The identical gap exists in `op_modpow_malachite`: [2](#0-1) 

`Number` is `num_bigint::BigInt` and `Malachite` is `malachite_bigint::BigInt`: [3](#0-2) 

The two crate versions in use are `num-bigint = "0.4.6"` and `malachite-bigint = "0.9.1"`: [4](#0-3) 

`num_bigint::BigInt::modpow` with a negative modulus computes the result modulo the absolute value of the modulus and returns a value whose sign matches the modulus (i.e., negative). `malachite_bigint::BigInt::modpow` follows a different convention for negative modulus (the malachite library's `Integer::mod_pow` is defined to return a value in `[0, |m|)` regardless of the sign of `m`). The two paths therefore return **different atoms** for the same input when the modulus is negative.

The `MALACHITE` flag is the runtime switch between the two paths: [5](#0-4) [6](#0-5) 

`op_modpow` is reachable in consensus mode (opcode 60); it is only disabled in mempool mode via `DISABLE_OP`: [7](#0-6) [8](#0-7) 

### Impact Explanation

1. **Incorrect computation result**: Any CLVM program that calls `(modpow base exp negative_modulus)` receives a negative atom as the result instead of an error. Downstream puzzle logic that assumes `modpow` always returns a value in `[0, modulus)` will silently compute the wrong answer.

2. **Consensus divergence**: Nodes running with `ClvmFlags::MALACHITE` and nodes running without it evaluate the same coin puzzle to different `NodePtr` values (one negative, one non-negative). This splits consensus: one set of nodes accepts a spend that the other rejects, or both accept but record different puzzle hashes.

The corrupted result is the atom produced by `a.new_number(ret)` at line 1282 / `a.new_malachite_number(ret)` at line 1320 — a negative big-integer atom instead of a non-negative one or an `EvalErr`. [9](#0-8) 

### Likelihood Explanation

The attack requires only crafting a CLVM atom whose high bit is set (any byte string starting with `0x80`–`0xff`) and passing it as the third argument to opcode 60. No special privileges, keys, or chain state are required. The program is valid CLVM and passes all cost checks. The `MALACHITE` flag is a named, documented `ClvmFlags` bit that can be toggled by any caller of `run_program`, making the divergence reachable through the public API and through any node that enables the flag during a softfork activation.

### Recommendation

Add a sign check for the modulus immediately after the zero check in both `op_modpow` and `op_modpow_malachite`:

```rust
if modulus.sign() == Sign::NoSign {
    return Err(EvalErr::DivisionByZero(input));
}
// ADD:
if modulus.sign() == Sign::Minus {
    return Err(EvalErr::InvalidOpArg(
        input,
        "ModPow with Negative Modulus".to_string(),
    ));
}
```

Apply the same fix using `malachite_bigint::Sign::Minus` in `op_modpow_malachite`. Add a test case analogous to the existing `test_modpow_basic` negative-exponent case.

### Proof of Concept

Construct a CLVM program that calls `(modpow 3 10 0xf9)` where the atom `0xf9` encodes −7 in CLVM's two's complement signed-integer representation (`0xf9` = 249 unsigned; high bit set → negative; two's complement value = −7).

- **Without `MALACHITE`**: `num_bigint::BigInt::modpow(3, 10, -7)` computes `3^10 mod 7 = 4`, then adjusts sign to match the negative modulus → returns the atom `0xfc` (−4). No error is raised.
- **With `MALACHITE`**: `malachite_bigint::BigInt::modpow(3, 10, -7)` returns `4` (non-negative, in `[0, 7)`).

The two nodes store different atoms as the result of the same opcode invocation. Neither raises `EvalErr::DivisionByZero` or `EvalErr::InvalidOpArg`. The missing guard is at: [10](#0-9)
