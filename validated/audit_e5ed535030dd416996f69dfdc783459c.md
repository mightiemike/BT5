### Title
Merkle Second-Preimage Attack: `verify_transaction_inclusion` Accepts Internal Node Hashes as Valid Transaction Proofs — (`merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is a publicly callable NEAR contract function that performs no validation of proof length against tree depth and no check that the supplied `tx_id` is a leaf-level transaction hash. Any unprivileged caller can supply an internal merkle node hash as `tx_id` with a shortened proof that still reconstructs the canonical merkle root, causing the function to return `true` for a hash that does not correspond to any real transaction in the block.

---

### Finding Description

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` is a pure hash-climbing function with no structural constraints: [1](#0-0) 

It accepts any `transaction_hash`, any `transaction_position`, and any-length `merkle_proof`, iterating unconditionally through the proof elements. There is no check that `merkle_proof.len()` equals `ceil(log2(n))` for the block's actual transaction count, and no check that the starting hash is a leaf.

`verify_transaction_inclusion` in `contract/src/lib.rs` imposes only one guard on the proof: [2](#0-1) 

The sole guard is `!args.merkle_proof.is_empty()`. There is no access control (`#[private]` is absent), no proof-depth validation, and no check that `args.tx_id` is a leaf transaction. The function is publicly callable by any NEAR account.

The codebase's own documentation confirms the flaw: [3](#0-2) 

> "This function is vulnerable to the standard Bitcoin merkle tree second-preimage attack — it may return `true` for an internal node hash rather than a real transaction hash."

The function is marked `#[deprecated]` since 0.5.0 in favor of `verify_transaction_inclusion_v2`, but it remains deployed and callable: [4](#0-3) 

---

### Impact Explanation

An attacker can fabricate a "proof of inclusion" for any internal merkle node of a canonical block. For a block with transactions `[tx0, tx1, tx2, tx3]`:

- `H01 = hash(tx0, tx1)` — an internal node, not a transaction
- `H23 = hash(tx2, tx3)` — an internal node, not a transaction
- `root = hash(H01, H23)`

The attacker calls `verify_transaction_inclusion` with:
- `tx_id = H01`
- `tx_index = 0`
- `merkle_proof = [H23]`

`compute_root_from_merkle_proof(H01, 0, [H23])` computes `hash(H01, H23) = root`, which equals `header.block_header.merkle_root`, so the function returns `true`.

`H01` is not a transaction. Any downstream system (bridge, oracle, cross-chain protocol) that calls `verify_transaction_inclusion` and trusts its return value can be deceived into accepting a fabricated event as proven.

---

### Likelihood Explanation

- The function is publicly callable with no access control.
- The attack requires only knowledge of any two sibling internal node hashes from a canonical block's merkle tree, which are derivable from publicly available Bitcoin block data.
- No privileged role, key compromise, or social engineering is required.
- The exploit is mechanically straightforward and locally reproducible.

---

### Recommendation

1. **Remove or gate `verify_transaction_inclusion` (v1).** Since `verify_transaction_inclusion_v2` exists as the replacement, the v1 function should either be removed entirely or protected with a `#[private]` annotation so it cannot be called externally. Leaving a deprecated but callable function with a known second-preimage vulnerability on-chain is an active risk.

2. **Add proof-depth validation in `verify_transaction_inclusion_v2`.** The coinbase-proof length equality check (`merkle_proof.len() == coinbase_merkle_proof.len()`) in v2 constrains depth indirectly, but does not prevent an attacker from supplying a proof whose length is shorter than the actual tree depth (using an internal node as the starting hash). A stricter fix would require the contract to store or receive the block's transaction count and validate `merkle_proof.len() == ceil(log2(tx_count))`. [5](#0-4)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
