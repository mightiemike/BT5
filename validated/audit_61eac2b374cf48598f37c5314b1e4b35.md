### Title
Difficulty Calculation for Fork Blocks Uses Mainchain Ancestor Instead of Fork Ancestor - (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

### Summary
When validating a fork block's proof-of-work, the difficulty retarget calculation fetches the boundary block's timestamp from the **mainchain** via `get_header_by_height`, rather than traversing the fork's actual ancestor chain. This is a direct analog to the Balancer report: a stale/cached parameter (mainchain block at a given height) is used instead of the current one (the fork's actual ancestor at that height), producing an incorrect difficulty requirement for fork blocks.

### Finding Description

In all three chain implementations, when a difficulty retarget is needed, the code calls `blocks_getter.get_header_by_height(first_block_height)` to obtain the timestamp of the block at the start of the difficulty interval. [1](#0-0) 

`get_header_by_height` is implemented to look up only the **mainchain** mapping: [2](#0-1) 

When a fork block is being validated, `prev_block_header` is correctly the fork's previous block (fetched via `get_prev_header` from `headers_pool`). However, the boundary block at `first_block_height` is fetched from `mainchain_height_to_header`, which maps to the **mainchain** block at that height — not the fork's ancestor at that height. [3](#0-2) [4](#0-3) 

The developers themselves flagged this concern with a TODO comment in the Dogecoin implementation: [5](#0-4) 

**Desynchronization scenario (Dogecoin, post-height 145,000):**

With `difficulty_adjustment_interval = 1`, every block is a retarget boundary. For a fork block at height `H+2` (whose `prev_block_header` is the fork's block at `H+1`), the code computes `height_first = H+1 - 1 = H` and calls `get_header_by_height(H)`. If the fork diverges at height `H`, the mainchain block at `H` is **different** from the fork's block at `H`. The contract uses the mainchain block's timestamp, not the fork's block's timestamp, to compute the expected difficulty. [6](#0-5) 

**Desynchronization scenario (Bitcoin/Litecoin, 2016-block intervals):**

If a fork diverges before a 2016-block difficulty boundary and a fork block crosses that boundary, `first_block_height` falls within the fork's diverged segment. The mainchain block at that height has a different timestamp than the fork's ancestor, producing a wrong expected difficulty. [7](#0-6) 

### Impact Explanation

The `expected_bits` returned by `get_next_work_required` is compared against `block_header.bits`: [8](#0-7) 

If the mainchain boundary block has a **later** timestamp than the fork's boundary block, `actual_time_taken` is inflated, producing a **lower** (easier) difficulty target. A fork block that does not meet the actual required difficulty passes the check. Conversely, a valid fork block can be incorrectly rejected.

Accepted fork blocks with under-required difficulty accumulate `chain_work` (computed from their own `bits` field) and can trigger a chain reorganization: [9](#0-8) 

A successful reorg causes `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` to confirm transactions in a chain that did not satisfy the protocol's actual PoW requirement. [10](#0-9) 

### Likelihood Explanation

For **Dogecoin** (post-height 145,000, `difficulty_adjustment_interval = 1`): any fork that is 2 or more blocks deep triggers the bug on every subsequent fork block. This is a routine scenario for any fork submission.

For **Bitcoin/Litecoin** (`difficulty_adjustment_interval = 2016`): the fork must span a 2016-block boundary, which is less frequent but entirely possible for a deep fork or reorg.

An unprivileged NEAR caller with the `trusted_relayer` role (or bypassing it via `Role::UnrestrictedSubmitBlocks`) can submit adversarial fork blocks via the public `submit_blocks` entrypoint. [11](#0-10) 

### Recommendation

Replace `get_header_by_height` (mainchain lookup) with an ancestor traversal that walks the fork's actual chain backward from `prev_block_header` to reach the block at `first_block_height`. This mirrors how Bitcoin Core's reference implementation walks `pindexFirst` backward through `pprev` pointers rather than looking up by height in a global index.

### Proof of Concept

1. Contract is initialized with Dogecoin mainchain at height 145,001 (post-new-difficulty-protocol).
2. Mainchain block at height 145,001 has timestamp `T_main`.
3. Attacker submits a fork block at height 145,001 with timestamp `T_fork < T_main` (valid per MTP check). This fork block is stored in `headers_pool`.
4. Attacker submits a fork block at height 145,002 (prev = fork block at 145,001). `get_next_work_required` computes `height_first = 145,001 - 1 = 145,000`. `get_header_by_height(145,000)` returns the mainchain block at 145,000 (shared ancestor — same in both chains). Difficulty is correct here.
5. Attacker submits a fork block at height 145,003 (prev = fork block at 145,002). `height_first = 145,002 - 1 = 145,001`. `get_header_by_height(145,001)` returns the **mainchain** block at 145,001 (timestamp `T_main`), not the fork's block at 145,001 (timestamp `T_fork`).
6. `actual_time_taken = T_main - T_mainchain_145000` instead of `T_fork - T_mainchain_145000`. Since `T_main > T_fork`, the computed target is easier than required.
7. The fork block at 145,003 passes `expected_bits == block_header.bits` with a lower-than-required difficulty, and is accepted into `headers_pool`.
8. If the fork accumulates sufficient `chain_work`, `reorg_chain` is triggered, and the under-difficulty fork becomes the canonical chain, corrupting `verify_transaction_inclusion` results.

    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }

        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        // compute merkle tree root and check if it matches block's original merkle tree root

            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
