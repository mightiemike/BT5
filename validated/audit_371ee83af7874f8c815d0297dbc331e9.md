### Title
NFT Owners Cannot Burn Their Own NFTs Unless Also Denom Creator — (`x/nft/keeper/keeper.go`)

### Summary
`keeper.BurnNFT` requires the caller to pass **both** an ownership check (`IsOwner`) and a denom-creator check (`IsDenomCreator`). Any NFT owner who is not the denom creator is permanently blocked from burning NFTs they legitimately own.

### Finding Description

`MsgBurnNFT.ValidateBasic` only validates address format and ID format: [1](#0-0) 

It passes the message to `msgServer.BurnNFT`, which calls `m.Keeper.BurnNFT`: [2](#0-1) 

Inside `keeper.BurnNFT`, two sequential authorization checks are applied:

```
1. IsOwner      — sender must own the NFT
2. IsDenomCreator — sender must also be the denom creator
``` [3](#0-2) 

`IsDenomCreator` returns `ErrUnauthorized` if the address does not match the denom's stored `Creator` field: [4](#0-3) 

The standard NFT lifecycle is: creator mints to a recipient (`MintNFT` accepts a separate `owner` parameter). Once the NFT is in the recipient's hands, the recipient is the owner but not the creator, so `IsDenomCreator` will always reject their burn attempt. [5](#0-4) 

Note that `BurnNFTUnverified` (used for IBC) skips the creator check, confirming the dual-check in `BurnNFT` is not a universal design requirement but an inconsistency: [6](#0-5) 

### Impact Explanation

Any NFT minted to a recipient who is not the denom creator becomes permanently non-burnable by that recipient. The NFT is stuck in their account forever. The only party who can burn it is the denom creator — but the creator can only burn it if they also own it (the `IsOwner` check runs first). So in practice, once an NFT is transferred away from the creator, **no one** can burn it via the normal `MsgBurnNFT` path.

Concrete state delta: `deleteNFT` and `decreaseSupply` are never called for non-creator owners, leaving the NFT record and supply count permanently inflated.

### Likelihood Explanation

This is triggered by the standard, intended usage of the NFT module: mint to a recipient who is a different address from the creator. This is the normal marketplace/airdrop pattern. Every such NFT is affected. No special privileges or governance are required — any user who receives an NFT via `MintNFT` with `recipient != sender` is affected.

### Recommendation

Remove the `IsDenomCreator` check from `BurnNFT`. Ownership (`IsOwner`) is the correct and sufficient authorization for burning. The creator check is appropriate for `MintNFT` and arguably for `EditNFT`, but not for burn, where the owner's right to destroy their own asset is the standard invariant.

```go
// BurnNFT deletes a specified NFT
func (k Keeper) BurnNFT(ctx sdk.Context, denomID, tokenID string, owner sdk.AccAddress) error {
    if !k.HasDenomID(ctx, denomID) {
        return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
    }

    nft, err := k.IsOwner(ctx, denomID, tokenID, owner)
    if err != nil {
        return err
    }
    // Remove the IsDenomCreator check here

    k.deleteNFT(ctx, denomID, nft)
    k.deleteOwner(ctx, denomID, tokenID, owner)
    k.decreaseSupply(ctx, denomID)
    return nil
}
```

func (k Keeper) IsDenomCreator(ctx sdk.Context, denomID string, address sdk.AccAddress) (types.Denom, error) {
	denom, err := k.GetDenom(ctx, denomID)
	if err != nil {
		return types.Denom{}, err
	}

	creator, err := sdk.AccAddressFromBech32(denom.Creator)
	if err != nil {
		panic(err)
	}

	if !creator.Equals(address) {
		return types.Denom{}, sdkerrors.Wrapf(types.ErrUnauthorized, "%s is not the creator of %s", address, denomID)
	}

	return denom, nil
