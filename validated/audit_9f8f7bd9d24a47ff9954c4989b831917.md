### Title
`promise_batch_action_add_gas_key_with_function_call` uses raw wire length instead of trie-ID length for ML-DSA-65 gas-key fee calculation ŌĆö (File: `runtime/near-vm-runner/src/logic/logic.rs`)

---

### Summary

The host function `promise_batch_action_add_gas_key_with_function_call` passes `public_key_len as usize` ŌĆö the raw WASM-memory wire length of the public key ŌĆö to `gas_key_add_key_exec_fee`. For ML-DSA-65 keys the wire length is **1953 bytes** (1 borsh tag + 1952 raw key bytes), but the actual trie-key identifier is only **33 bytes** (1 tag + 32-byte SHA3-256 hash). Every other storage-fee path in the codebase was updated to call `trie_id_len()` for exactly this reason; the host-function path was not.

---

### Finding Description

The protocol documentation explicitly states:

> `PublicKey::trie_id_len()` is a new contract that all storage-cost code must respect. Callers that still use `len()` for trie-storage costing will misprice ML-DSA-65 keys by ~1900 bytes.

The runtime-side path (`permission_exec_fees` in `runtime/runtime/src/config.rs`) correctly calls `public_key.trie_id_len()`:

```rust
let nonce_fee = gas_key_add_key_exec_fee(
    fees,
    account_id.len(),
    public_key.trie_id_len(),   // ŌåÉ correct
    gas_key_info.num_nonces,
);
```

The host-function path in `logic.rs` does not:

```rust
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receipt_receiver_id.len(),
    public_key_len as usize,    // ŌåÉ raw wire length, NOT trie_id_len()
    num_nonces,
);
```

`gas_key_add_key_exec_fee` feeds `public_key_len` directly into `access_key_key_len(account_id_len, public_key_len)`:

```rust
let nonce_key_len =
    access_key_key_len(account_id_len, public_key_len) + size_of::<NonceIndex>();
let per_byte = cfg
    .fee(ActionCosts::gas_key_byte)
    .exec_fee()
    .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64)
    .unwrap()
    .checked_mul(num_nonces)
    .unwrap();
```

For ML-DSA-65 the per-nonce key-byte count is inflated from 33 to 1953 ŌĆö a divergence of **+1920 bytes per nonce**. The `gas_key_byte` fee is multiplied by this inflated length for every nonce slot, so the total exec-fee overcharge scales with `num_nonces`.

The exact divergent values are:
- Correct trie-ID length: `1 + ML_DSA_65_HASH_LENGTH` = **33 bytes**
- Actual value used: `1 + ML_DSA_65_PUBLIC_KEY_LENGTH` = **1953 bytes**
- Overcharge per nonce: **1920 bytes ├Ś `gas_key_byte` exec fee**

---

### Impact Explanation

Any unprivileged user who calls a contract that invokes `promise_batch_action_add_gas_key_with_function_call` with an ML-DSA-65 public key will be charged ~59├Ś the correct per-byte exec fee for each nonce slot. The overcharge is burnt gas (not redirected to an attacker), but it constitutes a direct, deterministic loss of NEAR tokens for every affected user. With the default `gas_key_byte` fee and a typical `num_nonces` value the overcharge per `AddKey` action is on the order of tens of Tgas, translating to a meaningful fraction of a NEAR token per call.

**Severity: High** ŌĆö deterministic, unprivileged-user-triggered loss of funds proportional to `num_nonces`.

---

### Likelihood Explanation

The `PostQuantumSignatures` protocol feature gate must be active for ML-DSA-65 `AddKey` actions to be accepted. Once the feature activates on mainnet, any contract that programmatically adds gas keys (e.g., a key-management or multisig contract) and accepts ML-DSA-65 keys will trigger the overcharge for every caller. No special privilege is required; a normal user submitting a transaction to such a contract is sufficient.

---

### Recommendation

Replace `public_key_len as usize` with the trie-ID length of the decoded public key, mirroring the runtime-side fix:

```rust
// After: let public_key = self.get_public_key(public_key_ptr, public_key_len)?;
let decoded_pk = public_key.decode()?;
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receipt_receiver_id.len(),
    decoded_pk.trie_id_len(),   // ŌåÉ use trie-ID length, not wire length
    num_nonces,
);
// Pass decoded_pk to append_action_add_gas_key_with_function_call below.
```

The same audit should be applied to `promise_batch_action_add_gas_key_with_full_access` and any other host functions that forward a raw `public_key_len` to a fee helper.

---

### Proof of Concept

1. `PostQuantumSignatures` activates on mainnet.
2. A user calls a contract that invokes `promise_batch_action_add_gas_key_with_function_call` with an ML-DSA-65 public key (1953 borsh bytes) and `num_nonces = N`.
3. Inside the host function, `gas_key_add_key_exec_fee` receives `public_key_len = 1953`.
4. `access_key_key_len(account_id_len, 1953)` returns `account_id_len + 1955`.
5. The correct value is `access_key_key_len(account_id_len, 33)` = `account_id_len + 35`.
6. The per-nonce per-byte fee is multiplied by `(account_id_len + 1955 + NONCE_VALUE_LEN)` instead of `(account_id_len + 35 + NONCE_VALUE_LEN)`.
7. For a 10-character account ID and `N = 16` nonces, the overcharge is approximately `1920 ├Ś gas_key_byte_exec_fee ├Ś 16` gas units ŌĆö roughly 59├Ś the correct fee ŌĆö burnt from the user's balance.

    /// Length, in bytes, of the on-trie identifier for an access-key
    /// entry owned by this public key. For ed25519/secp256k1 this matches
    /// `len()`; for ML-DSA-65 the trie stores a SHA3-256 hash (33 bytes
    /// including the type tag), not the 1953-byte borsh-encoded pubkey.
    /// Used by storage-fee calculations on the runtime side; cheap to call
    /// (no hashing) - for ML-DSA-65 this returns the size of the digest
    /// form without actually hashing the pubkey.
    pub fn trie_id_len(&self) -> usize {
        match self {
            Self::ED25519(_) => 1 + ed25519_dalek::PUBLIC_KEY_LENGTH,
            Self::SECP256K1(_) => 1 + 64,
            Self::MLDSA65(_) => 1 + ML_DSA_65_HASH_LENGTH,
        }
    }
