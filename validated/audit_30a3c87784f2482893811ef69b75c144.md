### Title
Permanent `submit_blocks` Revert After Year 2106 Due to Unchecked `u32` Timestamp Truncation — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`, `contract/src/zcash.rs`)

---

### Summary

Every chain's `check_pow` implementation converts the NEAR block timestamp to `u32` via `.unwrap()`. After year 2106, when the NEAR block timestamp in seconds exceeds `u32::MAX`, the `.unwrap()` panics and permanently breaks `submit_blocks`. Additionally, the subsequent addition `current_timestamp + MAX_FUTURE_BLOCK_TIME_LOCAL` overflows in the narrow window just before that boundary. This is a direct structural analog to the reported `_getTwav()` bug: both involve `u32` timestamp arithmetic that silently relies on a type boundary that will eventually be crossed, with no wrapping or overflow guard in place.

---

### Finding Description

All four chain implementations share the identical pattern inside `check_pow`:

```rust
let current_timestamp = u32::try_from(env::block_timestamp_ms() / 1000).unwrap();
require!(
    block_header.time <= current_timestamp + MAX_FUTURE_BLOCK_TIME_LOCAL,
    "time-too-new: block timestamp too far in the future"
);
``` [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

`env::block_timestamp_ms()` returns a `u64` in milliseconds. Dividing by 1000 gives seconds. `u32::MAX = 4294967295`, which corresponds to ~year 2106. Two distinct failure modes exist:

**Failure 1 — `unwrap()` panic (permanent, post-2106):** Once the NEAR block timestamp in seconds exceeds `u32::MAX`, `u32::try_from(...)` returns `Err(...)` and `.unwrap()` panics. Every subsequent call to `submit_blocks` reverts unconditionally. The contract is permanently unable to ingest new Bitcoin headers.

**Failure 2 — addition overflow (narrow window, just before 2106):** `MAX_FUTURE_BLOCK_TIME_LOCAL = 2 * 60 * 60 = 7200` (u32). [5](#0-4) 

When `current_timestamp > u32::MAX - 7200`, the expression `current_timestamp + MAX_FUTURE_BLOCK_TIME_LOCAL` overflows. With Rust's `overflow-checks = true` (standard for NEAR WASM builds), this panics. Without overflow checks, it wraps to a small value, causing all valid blocks to be rejected as "time-too-new."

The inconsistency mirrors the external report exactly: `_updateTWAV` used `unchecked` arithmetic while `_getTwav` did not. Here, the `u32` type is used consistently for Bitcoin block timestamps (correct, since Bitcoin's own format is `u32`), but the NEAR wall-clock timestamp — a `u64` — is silently truncated to `u32` with no guard, creating the same class of latent overflow.

---

### Impact Explanation

After year 2106, `submit_blocks` permanently reverts on every call. The contract can no longer advance its view of the Bitcoin chain. `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` continue to execute against a frozen, stale chain state, returning results that do not reflect the actual Bitcoin chain. Any downstream NEAR contract relying on SPV proofs would receive incorrect or permanently stale verification results. [6](#0-5) 

---

### Likelihood Explanation

The trigger is the passage of time, not attacker-controlled input. Year 2106 is ~80 years away. However:
- The contract has no built-in upgrade path for this specific arithmetic path.
- The external report's judges accepted the identical timeline ("84 years") as Medium severity precisely because the contract is not guaranteed to be upgraded in time.
- The `u32::try_from(...).unwrap()` failure is a hard panic with no recovery path, unlike the Solidity case where an `unchecked` block was the fix.

---

### Recommendation

Replace the truncating conversion and bare addition with overflow-safe equivalents:

```rust
// Safe conversion: use u64 for the comparison, avoid truncation
let current_timestamp_secs: u64 = env::block_timestamp_ms() / 1000;
let max_allowed: u64 = current_timestamp_secs
    .saturating_add(u64::from(MAX_FUTURE_BLOCK_TIME_LOCAL));
require!(
    u64::from(block_header.time) <= max_allowed,
    "time-too-new: block timestamp too far in the future"
);
```

This eliminates both the `unwrap()` panic and the `u32` addition overflow, and remains correct for all timestamps representable by Bitcoin's `u32` field.

---

### Proof of Concept

1. Deploy the Bitcoin-feature contract on NEAR.
2. Fast-forward the NEAR block timestamp to any value where `timestamp_ms / 1000 > 4294967295` (i.e., after year 2106 in a test environment, or by setting `env::block_timestamp_ms()` to `4294967296000` in a unit test).
3. Call `submit_blocks` with any valid Bitcoin header batch.
4. Observe that `u32::try_from(4294967296u64)` returns `Err(...)`, `.unwrap()` panics, and the transaction reverts with a runtime trap.
5. Repeat indefinitely — every subsequent call reverts. The contract is permanently frozen.

The same test applies to all four chain builds (`bitcoin`, `litecoin`, `dogecoin`, `zcash`) since all share the identical conversion pattern. [1](#0-0) [5](#0-4)

    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
