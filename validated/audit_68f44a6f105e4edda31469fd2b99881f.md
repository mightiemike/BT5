Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of End User, Enabling Complete Allowlist Bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap()` gates swaps on the `sender` argument, which the pool populates with its own `msg.sender` — the direct caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the router is that direct caller. A pool admin who allowlists the router to enable standard-periphery access inadvertently grants every unprivileged address the ability to swap on the restricted pool, rendering the allowlist completely void.

## Finding Description

**Root cause — wrong actor bound in the extension:**

`SwapAllowlistExtension.beforeSwap()` evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the first argument forwarded by the pool. [1](#0-0) 

**How the pool populates `sender`:**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`, binding `sender` to whoever called `pool.swap()`. [2](#0-1) 

`ExtensionCalling._beforeSwap()` then encodes that value as the first argument to `IMetricOmmExtensions.beforeSwap`, passing it unchanged to the extension. [3](#0-2) 

**How the router becomes `msg.sender` of `pool.swap()`:**

`MetricOmmSimpleRouter.exactInputSingle()` calls `IMetricOmmPoolActions(params.pool).swap(...)` directly, making the router the `msg.sender` of that call. [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

**Full exploit call chain:**

```
User → router.exactInputSingle()
     → pool.swap()          [router is msg.sender]
     → _beforeSwap(sender = router)
     → extension.beforeSwap(sender = router)
     → allowedSwapper[pool][router] == true  ✓  (bypass)
```

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. No existing guard in the pool, the extension, or the router checks the originating EOA.

**The two broken states this creates:**

| Pool admin action | Consequence |
|---|---|
| Allowlists the router to support standard periphery | Every unprivileged user can swap by routing through the router — allowlist is completely void |
| Does not allowlist the router | Individually allowlisted users cannot use the router at all — standard periphery is broken for the curated pool |

## Impact Explanation

A curated pool (KYC-gated, institutional, or partner-only) that deploys `SwapAllowlistExtension` and allowlists `MetricOmmSimpleRouter` to support normal user flows inadvertently opens swap access to every on-chain address. Any actor — including sanctioned addresses, competitors, or bots — can call `exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` on the router and successfully execute swaps on the restricted pool. This is an admin-boundary break: an unprivileged path (`MetricOmmSimpleRouter`) bypasses the pool admin's configured access control, satisfying the contest impact gate.

## Likelihood Explanation

High. `MetricOmmSimpleRouter` is the protocol's own production periphery. Any pool admin who wants their curated pool to be usable by end users through the standard interface must allowlist the router. The wrong-actor binding is invisible from the admin's perspective — `setAllowedToSwap(pool, router, true)` reads as "allow router-mediated swaps," but it silently means "allow all swaps." No special attacker capability is required beyond calling the public router. [6](#0-5) 

## Recommendation

The extension must gate the economic actor (the end user), not the intermediary. Two viable approaches:

1. **Pass real user through `extensionData`**: Have the router encode `msg.sender` into `extensionData` and have the extension decode and verify it. This requires a documented encoding convention between router and extension.
2. **Dedicated allowlist-aware router**: Require that allowlisted pools be accessed only through a purpose-built router that appends a verified user identity to `extensionData`, rather than the generic `MetricOmmSimpleRouter`.

The cleanest fix is option 1 with a documented encoding standard so pool admins know which router versions are compatible with the allowlist extension.

## Proof of Concept

```
Setup:
  - Pool deployed with SwapAllowlistExtension as beforeSwap hook
  - Pool admin calls: setAllowedToSwap(pool, address(router), true)
    (intending to enable standard-periphery access for their allowlisted users)
  - Attacker address is NOT individually allowlisted

Attack:
  1. Attacker calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  2. Router calls pool.swap(recipient, ...) — router is msg.sender
  3. Pool calls _beforeSwap(msg.sender=router, ...)
  4. ExtensionCalling encodes sender=router and calls extension.beforeSwap(router, ...)
  5. Extension evaluates: allowedSwapper[pool][router] == true  ✓
  6. Swap executes — attacker trades on the curated pool with zero allowlist friction

Result: The allowlist is completely bypassed. Any user can swap on the
        restricted pool by routing through MetricOmmSimpleRouter.
```
