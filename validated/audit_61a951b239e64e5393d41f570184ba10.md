### Title
SwapAllowlistExtension gates the router address instead of the actual user, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument passed by the pool, which is `msg.sender` of the `pool.swap()` call. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router, not the end user. The extension therefore checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. This produces two fund-impacting failure modes: (1) if the pool admin allowlists the router to enable router-mediated swaps, every unpermissioned user can bypass the allowlist by routing through it; (2) if the pool admin does not allowlist the router, every legitimately allowlisted user is permanently locked out of the router.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router is the direct caller of `pool.swap()`: [4](#0-3) 

So the extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

The admin configures per-user allowances via `setAllowedToSwap`: [5](#0-4) 

There is no mechanism to recover the original end-user identity inside the extension; the pool does not forward `tx.origin` or any signed user credential.

---

### Impact Explanation

**Bypass path (High):** A pool admin deploys a restricted pool (e.g., KYC/institutional) and allowlists the router address so that allowlisted users can swap through the standard periphery. Because the extension sees `sender = router`, every non-allowlisted user can now call `exactInputSingle` / `exactInput` / `exactOutput` through the router and pass the guard. The allowlist is completely nullified for router-mediated swaps, allowing arbitrary users to drain or trade against a pool that was intended to be access-controlled.

**Lock path (Medium):** A pool admin allowlists specific user addresses but does not allowlist the router. Those users can call `pool.swap()` directly and pass the check (`sender = user`), but every router-mediated call fails because `allowedSwapper[pool][router]` is `false`. The router — the primary production swap interface — is permanently broken for all allowlisted pools that do not also allowlist the router contract.

Both outcomes are reachable by any unprivileged user with no special setup beyond using the public router.

---

### Likelihood Explanation

Any pool that deploys `SwapAllowlistExtension` and expects the router to work correctly hits one of the two failure modes deterministically. The router is the standard entry point documented in the periphery; pool admins are expected to use it. No exploit sophistication is required — a normal `exactInputSingle` call through the router is sufficient to trigger either the bypass or the lock.

---

### Recommendation

The extension must check the economically relevant actor, not the immediate pool caller. Two sound approaches:

1. **Check `sender` only for direct pool calls; require an explicit user credential for router calls.** The router could forward the original `msg.sender` inside `extensionData`, and the extension could verify a signed permit or a factory-registered router mapping before trusting it.

2. **Allowlist the router as a pass-through and require the router to attest the real user.** Add a `trustedRouter` registry to the extension; when `sender` is a trusted router, decode the real user from `extensionData` and check that address instead.

The simplest safe fix is to never allowlist the router address itself and instead have the router pass the original caller in `extensionData`, with the extension verifying a factory-registered router signature before substituting the identity.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured.
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (to allow router-mediated swaps for allowlisted users).
  - Pool admin calls setAllowedToSwap(pool, alice, true).
  - Pool admin does NOT call setAllowedToSwap(pool, bob, true).

Attack:
  - Bob (not allowlisted) calls MetricOmmSimpleRouter.exactInputSingle(
        pool=pool, recipient=bob, ...
    ).
  - Router calls pool.swap(bob, ...) with msg.sender = router.
  - Pool calls _beforeSwap(sender=router, ...).
  - Extension checks allowedSwapper[pool][router] == true → passes.
  - Bob's swap executes against the restricted pool.

Result:
  - Bob bypasses the allowlist entirely.
  - Alice's allowlist entry is irrelevant for router-mediated swaps.
  - Any user can trade on the "restricted" pool via the router.
```
