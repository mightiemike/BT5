### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Proof Verification — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to mitigate the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase Merkle proof. However, the deprecated `verify_transaction_inclusion` function remains a publicly callable NEAR contract method with no access restriction. Any unprivileged caller can invoke it directly, bypassing the coinbase proof check entirely, and obtain a `true` result for a forged transaction inclusion claim.

---

### Finding Description

The contract exposes two public SPV proof verification entry points:

**`verify_transaction_inclusion` (deprecated):** Verifies only that `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == header.merkle_root`. It performs no coinbase proof check. The function itself carries an explicit warning:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [1](#0-0) 

**`verify_transaction_inclusion_v2`:** Adds a mandatory coinbase proof step before delegating to the deprecated function: [2](#0-1) 

The `#[deprecated]` Rust attribute is a **compile-time warning only**. It imposes no runtime restriction. Because `verify_transaction_inclusion` is declared `pub` and annotated with `#[pause]` (not `#[private]`), it is a fully reachable NEAR contract method callable by any account. [3](#0-2) 

The coinbase proof requirement in v2 is therefore **conditionally applied** — it applies only when the caller chooses to call v2. An adversary who calls v1 directly skips it entirely. This is the direct analog to the reported bug class: a verification step that is supposed to be mandatory is instead gated by a condition the attacker controls.

The underlying Merkle proof computation in `compute_root_from_merkle_proof` is position-agnostic and will accept any 32-byte value as the leaf: [4](#0-3) 

---

### Impact Explanation

An attacker can exploit the well-documented 64-byte transaction Merkle proof forgery (CVE-class: [bitmex.com/blog/64-Byte-Transactions](https://www.bitmex.com/blog/64-Byte-Transactions)) against any downstream contract or application that calls `verify_transaction_inclusion`:

1. Craft a 64-byte blob equal to the concatenation of two adjacent Merkle tree node hashes at any internal level of a real block's transaction Merkle tree.
2. Compute `H = double_sha256(blob)` — this is a valid internal node hash that also appears in the tree.
3. Construct a Merkle proof from `H` up to the root (all sibling hashes are real and on-chain verifiable).
4. Call `verify_transaction_inclusion` with `tx_id = H`, `tx_index = <position of H's subtree>`, and the constructed proof.
5. `compute_root_from_merkle_proof` traverses the proof correctly and returns the real `merkle_root`.
6. The function returns `true` — asserting that a non-existent transaction was included in the block.

Any recipient contract that gates asset releases, bridge withdrawals, or state transitions on a `true` result from `verify_transaction_inclusion` is deceived into accepting a fabricated payment proof. [5](#0-4) 

---

### Likelihood Explanation

- The function is unconditionally reachable by any NEAR account with no role, stake, or permission requirement.
- The 64-byte forgery technique is publicly documented and requires only knowledge of a real block's Merkle tree, which is public data.
- The only prerequisite is that the target application calls `verify_transaction_inclusion` rather than `verify_transaction_inclusion_v2` — a realistic scenario given the function is still exported and the deprecation is invisible at runtime.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public contract interface entirely, or annotate it with `#[private]` so it is only callable by the contract itself (as it is already used internally by `verify_transaction_inclusion_v2`). The deprecation notice is insufficient because NEAR's ABI exposes all `pub` methods regardless of Rust-level deprecation.

---

### Proof of Concept

```
# Attacker knows block B with merkle_root R and at least 2 transactions.
# Let node_left and node_right be two sibling hashes at depth 1 of B's Merkle tree.

blob        = node_left || node_right          # 64 bytes
forged_txid = double_sha256(blob)              # = merkle_root R (for a 2-tx block)
                                               # or an internal node for deeper trees

# For a 2-tx block: forged_txid == R, proof = []
# But require!(!merkle_proof.is_empty()) blocks empty proof.

# For a 4-tx block: pick internal node at depth 1.
# forged_txid = hash(tx[0], tx[1])
# proof = [ hash(tx[2], tx[3]) ]   (the sibling subtree root)
# tx_index = 0 (even position)
# compute_root_from_merkle_proof(forged_txid, 0, [hash(tx[2],tx[3])])
#   = double_sha256(forged_txid || hash(tx[2],tx[3]))
#   = double_sha256(hash(tx[0],tx[1]) || hash(tx[2],tx[3]))
#   = merkle_root  ✓

near call <contract> verify_transaction_inclusion \
  '{"tx_id": "<forged_txid>", "tx_block_blockhash": "<B_hash>",
    "tx_index": 0, "merkle_proof": ["<hash(tx2,tx3)>"],
    "confirmations": 1}'
# → true   (no coinbase check performed)
``` [6](#0-5)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
