### Title
Unprotected `creditDeposit()` in `DirectDepositV1` Allows Anyone to Force Token Deposits Into Protocol — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` is an `external` function with no access control. Any unprivileged caller can invoke it at any time to sweep all ERC-20 token balances held by the DDA contract into the Endpoint's slow-mode deposit queue, immediately transferring custody of those tokens out of the DDA and into the protocol.

---

### Finding Description

`DirectDepositV1` is a per-user deposit address contract. It holds tokens temporarily before they are credited to a fixed `subaccount` (set immutably at construction). The contract exposes two privileged withdrawal paths (`withdraw()` and `withdrawNative()`, both `onlyOwner`) and one deposit path (`creditDeposit()`).

`creditDeposit()` carries no access control modifier: [1](#0-0) 

The function iterates over every spot product, reads the DDA's token balance, approves the Endpoint, and calls `depositCollateralWithReferral` for the hardcoded `subaccount`. Inside `Endpoint.depositCollateralWithReferral`, `handleDepositTransfer` immediately pulls the tokens from `msg.sender` (the DDA) into the Endpoint contract: [2](#0-1) 

A slow-mode transaction is then enqueued with `executableAt = block.timestamp + SLOW_MODE_TX_DELAY` (hardcoded three-day delay). The tokens are no longer in the DDA — they are held by the Endpoint pending sequencer processing.

The owner's only direct-withdrawal escape hatch is: [3](#0-2) 

Once `creditDeposit()` has been called by an attacker, `withdraw()` will transfer zero tokens because the DDA balance is already zero.

---

### Impact Explanation

An attacker can front-run any owner action on the DDA (e.g., a `withdraw()` call intended to recover tokens without going through the protocol) by calling `creditDeposit()` first. This:

1. **Immediately removes tokens from the DDA** — `handleDepositTransfer` pulls them into the Endpoint in the same transaction.
2. **Locks them in the protocol for at least three days** — the slow-mode delay prevents the owner from recovering them via the normal withdrawal path until the sequencer processes the queued `DepositCollateral` transaction and the subsequent `WithdrawCollateral` slow-mode transaction clears.
3. **Corrupts the owner's intended asset flow** — tokens the owner intended to use outside the protocol (e.g., emergency recovery, redeployment to a different subaccount) are forcibly committed to the fixed `subaccount` set at DDA construction.

The tokens are not permanently lost, but the owner's direct custody is broken and recovery requires navigating the protocol's slow-mode queue under a multi-day delay.

---

### Likelihood Explanation

The entry path requires no privilege, no capital, and no special knowledge beyond the DDA contract address. Any on-chain observer can monitor DDA token balances and call `creditDeposit()` at will. The attack is trivially repeatable: every time the owner sends tokens to the DDA, the attacker can immediately drain them into the protocol before the owner acts.

---

### Recommendation

Add an `onlyOwner` modifier to `creditDeposit()`, consistent with the access control applied to `withdraw()` and `withdrawNative()`:

```solidity
function creditDeposit() external onlyOwner {
    ...
}
```

This ensures only the DDA owner can decide when and whether to commit tokens to the protocol.

---

### Proof of Concept

```solidity
// Attacker has no privileges — only needs the DDA address.
function test_force_credit_deposit() external {
    // Owner sends 1000 USDC to the DDA intending to withdraw directly.
    usdc.transfer(address(dda), 1000e6);

    // Attacker front-runs the owner's withdraw() call.
    vm.prank(ATTACKER);
    dda.creditDeposit(); // no revert — no access control

    // Owner's withdraw() now transfers 0 tokens.
    vm.prank(OWNER);
    dda.withdraw(IIERC20Base(address(usdc)));
    assertEq(usdc.balanceOf(OWNER), 0); // tokens are gone from DDA

    // Tokens are locked in Endpoint slow-mode queue for 3 days.
}
```

    function creditDeposit() external {
        uint32[] memory productIds = spotEngine.getProductIds();
        for (uint256 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            address tokenAddr = spotEngine.getToken(productId);
            require(tokenAddr != address(0), "Invalid productId.");
            IIERC20Base token = IIERC20Base(tokenAddr);
            uint256 balance = token.balanceOf(address(this));
            if (balance != 0) {
                token.approve(address(endpoint), balance);
                endpoint.depositCollateralWithReferral(
                    subaccount,
                    productId,
                    uint128(balance),
                    "-1"
                );
            }
        }
    }
