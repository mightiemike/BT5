### Title
EIP-712 Domain Separator Uses Product ID as Verifying Contract Instead of `address(this)`, Enabling Cross-Deployment Order Replay — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.getDigest` constructs an EIP-712 domain separator using `address(uint160(productId))` as the `verifyingContract` field instead of `address(this)`. This is a hardcoded incorrect value — directly analogous to the Opera-Bridge chain ID misconfiguration — that causes order signatures to be bound to a product-ID-derived address rather than the actual contract instance. As a result, a valid order signature from any `OffchainExchange` deployment is equally valid on any other deployment of the same contract on the same chain that shares the same product ID.

---

### Finding Description

In `OffchainExchange.getDigest`, the domain separator is constructed as:

```solidity
bytes32 domainSeparator = keccak256(
    abi.encode(
        _TYPE_HASH,
        _EIP712NameHash(),
        _EIP712VersionHash(),
        block.chainid,
        address(uint160(productId))   // ← wrong: should be address(this)
    )
);
``` [1](#0-0) 

The EIP-712 standard mandates that `verifyingContract` be the address of the contract that will consume the signature, so that signatures cannot be replayed on a different contract instance. Here, `address(uint160(productId))` is a deterministic value derived solely from the product ID — for example, product ID `1` maps to `0x0000000000000000000000000000000000000001`. This value is identical across every deployment of `OffchainExchange` on the same chain that lists the same product.

The resulting digest is then used directly in `_checkSignature` to recover and validate the order signer:

```solidity
function _checkSignature(
    bytes32 subaccount,
    bytes32 digest,
    address linkedSigner,
    bytes memory signature
) internal view virtual returns (bool) {
    address signer = ECDSA.recover(digest, signature);
``` [2](#0-1) 

Note: `EndpointTx.validateSignedTx` correctly wraps `Verifier.computeDigest` output with `_hashTypedDataV4` for user-signed slow-mode transactions, providing chain and contract binding for those paths. However, `OffchainExchange.getDigest` constructs its own domain separator inline and does not use `address(this)`, leaving order signatures unbound to the contract instance. [3](#0-2) 

---

### Impact Explanation

If `OffchainExchange` is ever redeployed or upgraded on the same chain (chain ID 57073 prod or 763373 test) with overlapping product IDs, every order signature ever signed against the old deployment remains cryptographically valid against the new deployment. An attacker who has observed any historical signed order (e.g., from mempool, off-chain order books, or prior sequencer batches) can submit it to the new contract. This results in unauthorized order execution — fills at stale prices, re-execution of already-settled trades, or forced position changes — causing direct asset loss to the affected subaccounts.

---

### Likelihood Explanation

Protocol upgrades and redeployments are routine. The Nado codebase already shows two distinct network environments (`test` chainId 763373, `prod` chainId 57073) and uses upgradeable proxy patterns throughout. Any redeployment of `OffchainExchange` — whether for a bug fix, feature upgrade, or migration — on either chain with the same product IDs immediately activates the replay surface. No privileged access is required; any party who has ever seen a signed order can exploit this. [4](#0-3) 

---

### Recommendation

Replace `address(uint160(productId))` with `address(this)` in the domain separator construction inside `getDigest`:

```solidity
bytes32 domainSeparator = keccak256(
    abi.encode(
        _TYPE_HASH,
        _EIP712NameHash(),
        _EIP712VersionHash(),
        block.chainid,
        address(this)   // correct: bind to this contract instance
    )
);
```

Alternatively, use the inherited `_domainSeparatorV4()` from `EIP712Upgradeable`, which already computes the correct domain separator using `address(this)` and `block.chainid`. This is consistent with how `EndpointTx` handles its own digest path via `_hashTypedDataV4`.

---

### Proof of Concept

1. User Alice signs an order for product ID `1` on the current `OffchainExchange` at address `0xAAAA...`. The domain separator encodes `verifyingContract = address(uint160(1)) = 0x0000...0001`.
2. The protocol redeploys `OffchainExchange` at address `0xBBBB...` (same chain, same product ID `1`). The new contract's `getDigest` also encodes `verifyingContract = 0x0000...0001` — identical to the old deployment.
3. Attacker submits Alice's old signed order to the new `OffchainExchange` at `0xBBBB...`.
4. `_checkSignature` calls `ECDSA.recover(digest, signature)` where `digest` is computed by `getDigest` on the new contract. Because the domain separator is identical, the recovered signer matches Alice's address.
5. The order executes against Alice's subaccount on the new deployment without her consent. [5](#0-4)

        address linkedSigner,

    ) internal view virtual returns (bool) {
        address signer = ECDSA.recover(digest, signature);
