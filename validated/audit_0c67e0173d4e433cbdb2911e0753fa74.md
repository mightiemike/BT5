### Title
Off-by-One in `forceRebalanceNlpPool` Health Check Loop Skips Pool 0 Solvency Invariant — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`forceRebalanceNlpPool` applies balance changes to **all** NLP pools (indices `0..n-1`) but then only verifies initial health for pools at indices `1..n-1`. Pool 0 — the protocol's own NLP pool — is never health-checked after the rebalance, meaning it can be left in a negative-health (insolvent) state with no on-chain revert.

---

### Finding Description

`_applyNlpRebalance` iterates from `i = 0`, updating the QUOTE balance of every pool subaccount including `nlpPools[0]`: [1](#0-0) 

Immediately after, `forceRebalanceNlpPool` runs a post-condition health check — but starts the loop at `i = 1`, silently skipping `nlpPools[0]`: [2](#0-1) 

Pool 0 is the protocol-owned NLP pool (its `owner` is enforced to be `address(0)` and it is never deletable): [3](#0-2) 

Because pool 0 holds the protocol's own liquidity, leaving it in negative initial health directly corrupts the protocol's solvency invariant. The correct loop should start at `i = 0`, consistent with every other loop over `nlpPools` in the same file.

---

### Impact Explanation

**High.** `nlpPools[0]` is the protocol's core liquidity subaccount. A forced rebalance that drains it below zero initial health passes silently — no revert, no enforcement. This constitutes a solvency/accounting corruption: the protocol's own pool can be made insolvent on-chain, corrupting the global health accounting that all liquidation and settlement logic depends on.

---

### Likelihood Explanation

**Medium.** The `ForceRebalanceNlpPool` transaction is submitted by the sequencer via `Endpoint.submitTransactionsChecked`. A sequencer bug, miscalculation, or a crafted rebalance vector that shifts quote balance away from pool 0 while satisfying the zero-sum `_validateNlpRebalance` check (net delta = 0) would silently leave pool 0 unhealthy. No external attacker key or governance capture is required — only a sequencer submission with a rebalance array that assigns a sufficiently negative `nlpPoolRebalanceX18[0]`. [4](#0-3) 

---

### Recommendation

Change the loop start index from `1` to `0`:

```diff
- for (uint128 i = 1; i < nlpPools.length; i++) {
+ for (uint128 i = 0; i < nlpPools.length; i++) {
```

This aligns with the pattern used in `_applyNlpRebalance`, `_validateNlpRebalance`, `mintNlp`, and `burnNlp`, all of which iterate from `i = 0`.

---

### Proof of Concept

1. Protocol has two NLP pools: `nlpPools[0]` (protocol pool) and `nlpPools[1]` (external pool).
2. Sequencer submits `ForceRebalanceNlpPool` with `nlpPoolRebalanceX18 = [-X, +X]` for some large `X`. The net sum is 0, so `_validateNlpRebalance` passes.
3. `_applyNlpRebalance` subtracts `X` from `nlpPools[0].subaccount` and adds `X` to `nlpPools[1].subaccount`.
4. The health check loop starts at `i = 1` — only `nlpPools[1]` is checked (it gained balance, so it passes).
5. `nlpPools[0]` is now in negative initial health. No revert occurs.
6. The protocol's own pool is insolvent; all downstream health calculations and liquidation logic that depend on the global state are now operating on corrupted accounting. [5](#0-4)
