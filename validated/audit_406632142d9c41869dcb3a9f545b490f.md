### Title
Cost Undercharge via Unit Mismatch Between `int_atom` Byte Length and `limbs_for_int` Magnitude Bytes in `op_multiply` — (File: `src/more_ops.rs`)

---

### Summary

In `op_multiply`, the variable `l0` is initialized from `int_atom` (which returns the actual serialized byte length of the atom, including any leading zero byte required for positive numbers whose high bit is set), but after each multiplication it is overwritten with `limbs_for_int(&total)`, which returns only the magnitude byte count (`bits().div_ceil(8)`). For positive numbers whose most-significant byte has its high bit set, the serialized form requires one extra leading zero byte, so `limbs_for_int` underestimates the true byte length by exactly 1. Because `l0` is then fed into the quadratic cost formula for every subsequent operand, the cost of `op_multiply` is systematically undercharged whenever intermediate products land in that class of values.

---

### Finding Description

`op_multiply` in `src/more_ops.rs` maintains `l0` as the "size" of the running product for cost purposes: [1](#0-0) 

For the first operand, `l0` is set by `int_atom`, which returns the actual atom byte length — the number of bytes in the on-wire CLVM representation, including any leading `0x00` byte that prevents a positive value from being interpreted as negative. [2](#0-1) 

After each subsequent multiplication, `l0` is updated with: [3](#0-2) 

`limbs_for_int` is defined as: [4](#0-3) 

`v.bits()` (from `num-bigint::BigInt`) returns the number of bits in the absolute value. For a positive integer whose most-significant byte has its high bit set — e.g., 128 = `0x80` — `bits()` = 8, so `limbs_for_int` = 1. But the canonical CLVM signed big-endian serialization of 128 is `[0x00, 0x80]` = **2 bytes**, because without the leading zero the value would be parsed as negative. `int_atom` would return 2 for this atom; `limbs_for_int` returns 1.

The cost formula that uses `l0` is: [5](#0-4) 

When `l0` is 1 less than the true serialized byte length, both the linear term (`l0 * MUL_LINEAR_COST_PER_BYTE`) and the quadratic term (`l0 * l1 / MUL_SQUARE_COST_PER_BYTE_DIVIDER`) are undercharged. Because `l0` is re-used as the "left operand size" for every subsequent argument in the same `*` expression, the undercharge accumulates across all remaining operands.

The same mismatch appears in `op_ash` and `op_lsh`, where `l1 = limbs_for_int(&v)` is used for the output size: [6](#0-5) [7](#0-6) 

Those are single-shot operations so the undercharge is bounded to one byte per call, but the root cause is identical.

The inconsistency is confirmed by comparing with `op_unknown` cost function 2, which is documented as mimicking `op_multiply` but uses `l0 += l1` (a conservative overestimate) rather than `limbs_for_int`: [8](#0-7) 

---

### Impact Explanation

CLVM cost is the primary DoS-prevention mechanism for the Chia blockchain. Every transaction's CLVM program is executed under a `max_cost` budget; programs that exceed it are rejected. If `op_multiply` undercharges cost, an attacker can craft a program whose true computational work exceeds what the cost budget should permit, while still passing the `check_cost` gate. This is a consensus-critical arithmetic accounting error: the corrupted value is the accumulated `cost` returned in the `Reduction`, which determines whether a block's transactions are accepted or rejected.

---

### Likelihood Explanation

The trigger condition — a positive intermediate product whose most-significant byte has its high bit set — is common. Any multiplication of two large positive integers (e.g., numbers near 2^(8k−1) for any k) produces such a result. An attacker who controls the CLVM bytes of a spend bundle can deliberately choose operands to maximize the number of iterations where `limbs_for_int` underestimates, accumulating undercharge across all operands in a single `*` expression. No privileged access is required; crafting CLVM bytes is the normal attacker capability.

---

### Recommendation

Replace `limbs_for_int(&total)` with a function that returns the actual serialized byte length — i.e., the number of bytes that `a.new_number(total)` would allocate, which already accounts for the leading zero byte. One correct approach is to compute the length after allocation:

```rust
// after: total *= ...;
let tmp = a.new_number(total.clone())?;
l0 = a.atom_len(tmp);
// (then discard tmp and continue; the final allocation happens at the end)
```

Or, equivalently, adjust `limbs_for_int` to add 1 when the most-significant bit of the magnitude is set (i.e., when `v.bits() % 8 == 0` and `v > 0`). Apply the same fix to the `l1` computation in `op_ash` and `op_lsh`.

---

### Proof of Concept

Consider the CLVM program `(* 0x0080 0x0080)`:

- First operand atom `[0x00, 0x80]`: `int_atom` returns `l0 = 2` (correct serialized length).
- Second operand atom `[0x00, 0x80]`: `buf.len()` = 2, so `l1 = 2`.
- Cost charged: `(2 + 2) * MUL_LINEAR_COST_PER_BYTE + (2 * 2) / MUL_SQUARE_COST_PER_BYTE_DIVIDER`.
- Product = 128 × 128 = 16384 = `0x4000`. Serialized as `[0x40, 0x00]` = 2 bytes. `limbs_for_int(16384)` = `ceil(15/8)` = 2. Correct here.

Now consider `(* 0x0080 0x0080 0x0002)`:

- After first multiplication, `total` = 16384, `l0 = limbs_for_int(16384) = 2`. Still correct.
- After second multiplication, `total` = 32768 = `0x8000`. Serialized as `[0x00, 0x80, 0x00]` = **3 bytes**. But `limbs_for_int(32768)` = `ceil(16/8)` = 2. **`l0` is set to 2 instead of 3.**
- If there were a fourth operand, the cost formula would use `l0 = 2` instead of `l0 = 3`, undercharging by `MUL_LINEAR_COST_PER_BYTE + l1 / MUL_SQUARE_COST_PER_BYTE_DIVIDER`.

By chaining many operands that keep the intermediate product in the "high-bit-set" class (e.g., repeatedly multiplying by 2), an attacker maximizes the number of iterations where `l0` is underestimated, accumulating undercharged cost across the entire expression.
