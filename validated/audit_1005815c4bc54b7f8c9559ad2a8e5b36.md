Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks LP Share Recipient (`owner`) Instead of Actual Token Provider (`sender`), Enabling Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the first ABI parameter (`sender`, the actual token-providing caller) and instead checks the second parameter (`owner`, the LP-share recipient) against the allowlist. This allows any address not on the allowlist to deposit into a restricted pool by naming an allowlisted address as `owner`, while legitimately allowlisted depositors who route LP shares to a separate address are incorrectly blocked. Both outcomes directly contradict the contract's stated invariant: *"Gates `addLiquidity` by depositor address, per pool."*

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as the first argument to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` then ABI-encodes the call with `sender` in the first slot and `owner` in the second: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` discards the first slot (unnamed `address`) and names the second slot `owner`, then checks `allowedDepositor[msg.sender][owner]`: [3](#0-2) 

The guard therefore evaluates whether the LP-share *recipient* is allowlisted, not whether the actual token-providing *caller* is allowlisted. By contrast, `SwapAllowlistExtension.beforeSwap` correctly names and checks the first parameter (`sender`): [4](#0-3) 

**Exploit path:**
1. Pool is deployed with `DepositAllowlistExtension`; only `alice` is allowlisted (`allowedDepositor[pool][alice] = true`).
2. `bob` (not allowlisted) calls `pool.addLiquidity(owner = alice, salt, deltas, callbackData, extensionData)`.
3. The extension evaluates `allowedDepositor[pool][alice]` → `true` → hook returns the valid selector.
4. `LiquidityLib.addLiquidity` executes; the pool calls `bob.metricOmmSwapCallback(...)` and `bob` transfers the required tokens.
5. LP shares are minted to `alice`; `bob` has successfully deposited into a pool he is not authorized to access.

No existing guard prevents this: the allowlist check is the sole gate, and it is evaluated on the wrong address.

## Impact Explanation
**Allowlist bypass (unauthorized deposit):** Any unprivileged address can inject liquidity into a pool that the admin intended to restrict, by simply naming any allowlisted address as `owner`. The pool admin's access-control intent is completely defeated.

**False denial (authorized depositor blocked):** A legitimately allowlisted depositor who routes LP ownership to a separate vault or multisig (a standard DeFi pattern) is incorrectly reverted with `NotAllowedToDeposit`, breaking a legitimate and expected use-case.

Both outcomes break the core invariant of the extension and constitute broken core pool functionality (restricted liquidity provision is rendered non-functional). The bypass is a direct, permissionless path to unauthorized fund injection into restricted pools.

## Likelihood Explanation
The bypass requires a single call to `addLiquidity` with any allowlisted address as `owner`. The allowlist is publicly readable on-chain via `allowedDepositor`. No special privileges, flash loans, or oracle manipulation are needed. The trigger condition is exactly the intended production use-case of the extension (a pool with a non-empty, non-all-open allowlist), making exploitation trivially repeatable by any observer.

## Recommendation
Name and use the first parameter (`sender`) instead of the second (`owner`) in `beforeAddLiquidity`:

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
      external
      view
      override
      returns (bytes4)
  {
-     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
          revert IMetricOmmPoolActions.NotAllowedToDeposit();
      }
      return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
```

This mirrors the correct pattern in `SwapAllowlistExtension`, which checks `sender` (the actual swapper) rather than `recipient`.

## Proof of Concept
1. Deploy pool with `DepositAllowlistExtension`; call `setAllowedToDeposit(pool, alice, true)`.
2. From `bob` (not allowlisted), call `pool.addLiquidity(owner = alice, salt, deltas, callbackData, extensionData)`.
3. Extension evaluates `allowedDepositor[pool][alice]` → `true` → hook passes.
4. Pool executes `bob.metricOmmSwapCallback(...)`, `bob` transfers tokens.
5. LP shares minted to `alice`; `bob` has bypassed the allowlist.
6. Conversely, `alice` calling `pool.addLiquidity(owner = alice_vault, ...)` where `alice_vault` is not allowlisted is incorrectly reverted, confirming the false-denial path. [3](#0-2) [5](#0-4)
