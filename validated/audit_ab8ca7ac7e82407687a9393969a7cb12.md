### Title
`verify_transaction_inclusion` Accepts Internal Merkle Tree Nodes as Valid Transaction Proofs, Enabling SPV Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The still-callable `verify_transaction_inclusion` function validates an SPV proof by checking only that the supplied `tx_id` and `merkle_proof` reproduce the stored block `merkle_root`. It performs no check that `tx_id` is a leaf-level transaction hash rather than an internal Merkle tree node. An unprivileged NEAR caller can forge a proof of inclusion for a transaction that does not exist in the block by supplying an internal node hash as `tx_id` with a shortened proof path, causing any consuming DApp to accept a fabricated Bitcoin event.

---

### Finding Description

`verify_transaction_inclusion` (lines 288–323 of `contract/src/lib.rs`) performs three checks before returning a result:

1. The requested block is in the current main chain.
2. The block has enough confirmations.
3. The Merkle proof is non-empty.

It then delegates entirely to `merkle_tools::compute_root_from_merkle_proof`:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
```

`compute_root_from_merkle_proof` (lines 34–52 of `merkle-tools/src/lib.rs`) iterates over the proof hashes, combining them left-or-right based on `current_position % 2`, and returns the computed root. It does not validate that the starting hash is a leaf-level transaction hash; it accepts any 32-byte value as the starting point.

**The attack (64-byte transaction / CVE-2017-12842):**

In Bitcoin's Merkle tree every internal node `N` at depth `D-1` is:

```
N = double_sha256(left_txid || right_txid)
```

where `left_txid` and `right_txid` are 32-byte child hashes. An attacker can construct a 64-byte "transaction" `T = left_txid || right_txid`. Its `txid = double_sha256(T) = N`, which is exactly the internal node. The attacker then:

1. Picks any real block `B` whose Merkle tree has depth `D`.
2. Selects an internal node `N` at depth `D-1` with known children.
3. Submits `tx_id = N`, `tx_index = N's position at depth D-1`, and a Merkle proof of length `D-1` (the path from `N` to the root).
4. `compute_root_from_merkle_proof` starts at `N` and walks `D-1` steps to the root — producing the correct `merkle_root`.
5. `verify_transaction_inclusion` returns `true`.

The function's own documentation acknowledges this:

> **Warning**: This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash.

Despite this warning and the `#[deprecated]` annotation, the function remains a live, `#[pause]`-gated public entry point callable by any unprivileged NEAR account.

The analog to the external report is direct: just as `ResourceLockValidator` checked only the `ResourceLock` hash and not the full `userOp`, `verify_transaction_inclusion` checks only that the Merkle proof path reaches the stored root, without verifying that the starting value is a valid leaf. In both cases an attacker substitutes a structurally different input that satisfies the partial check.

`verify_transaction_inclusion_v2` (lines 347–369) was introduced to fix this by requiring a coinbase proof of equal depth, which pins the tree depth and forces the supplied `tx_id` to be a leaf. However, v1 was never removed and remains independently callable.

---

### Impact Explanation

Any external NEAR DApp or cross-contract call that invokes `verify_transaction_inclusion` (v1) can be fed a forged proof. The function returns `true` for a transaction that was never broadcast or mined. A consuming contract that gates fund releases, token mints, bridge unlocks, or state transitions on this result will act on a fabricated Bitcoin event. The corrupted proof result is the return value `true` for a non-existent transaction against a real, confirmed block.

---

### Likelihood Explanation

The function is callable by any unprivileged NEAR account with no staking or role requirement beyond the contract not being paused. The 64-byte attack requires only knowledge of a block's Merkle tree structure, which is fully public. The proof construction is deterministic and requires no brute-force search. Any DApp that has not migrated to v2 — or that calls v1 directly — is immediately exploitable.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the contract entirely. If backward compatibility must be preserved, replace the function body with an unconditional `env::panic_str("use verify_transaction_inclusion_v2")` so that no caller can obtain a `true` result from the vulnerable path. All callers should be directed to `verify_transaction_inclusion_v2`, which correctly pins tree depth via the coinbase proof length check.
