### Title
User-Controlled `confirmations` Parameter Allows Bypassing Confirmation Depth Requirement - (File: contract/src/lib.rs)

---

### Summary

The `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` functions accept a caller-supplied `confirmations` field with no minimum enforcement. Any unprivileged NEAR caller can pass `confirmations = 0`, causing the contract to return `true` for a transaction in the most recently submitted block — one that may still be reorganized. This is the direct analog of the reported fee bypass: just as `_proxyFeeInWei` is user-controlled and can be zeroed to avoid fees, `confirmations` is user-controlled and can be zeroed to bypass the confirmation-depth security requirement.

---

### Finding Description

`verify_transaction_inclusion` accepts `ProofArgs`, which includes a `confirmations: u64` field supplied entirely by the caller. [1](#0-0) 

Inside the function, the contract enforces only an **upper bound**: [2](#0-1) 

Then it checks that the block has at least `confirmations` blocks on top of it: [3](#0-2) 

When `confirmations = 0`, the expression `(heaviest - target + 1) >= 0` is trivially true for any `u64`, so the check always passes. There is no admin-set minimum, no floor, and no protocol-enforced lower bound. The same flaw is inherited by `verify_transaction_inclusion_v2`, which delegates to `verify_transaction_inclusion` after its coinbase proof check: [4](#0-3) 

The `confirmations` field in `ProofArgsV2` is equally user-controlled: [5](#0-4) 

---

### Impact Explanation

**Impact: High.**

A recipient NEAR contract (e.g., a cross-chain bridge or asset-release contract) that calls `verify_transaction_inclusion` to gate an action on Bitcoin finality receives a `true` result even when `confirmations = 0`. This means a transaction in the most recently submitted block — which is the block most susceptible to a chain reorganization — is treated as fully verified. If a reorg occurs after the recipient contract has acted on the result, the accepted Bitcoin transaction may no longer exist on the canonical chain, enabling double-spend or fraudulent cross-chain claims. The exact corrupted value is the **proof result**: the function returns `true` when the protocol's intended security invariant (a meaningful confirmation depth) has not been satisfied.

---

### Likelihood Explanation

**Likelihood: High.**

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are public, `#[pause]`-gated (not role-gated) view-like functions callable by any unprivileged NEAR account or contract. Crafting a call with `confirmations = 0` requires no special access, no leaked keys, and no social engineering. Any actor who wants to claim a Bitcoin transaction is "verified" before it has meaningful finality can do so permissionlessly. [6](#0-5) 

---

### Recommendation

Add an admin-configurable `min_confirmations: u64` field to the `BtcLightClient` contract state. In both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2`, enforce:

```rust
require!(
    args.confirmations >= self.min_confirmations,
    "Confirmations below protocol minimum"
);
```

This mirrors the recommended fix in the external report: replace the user-controlled value with a sensible admin-set value so the protocol can enforce the security parameter rather than relying on callers to self-impose it.

---

### Proof of Concept

1. Deploy the BTC light client contract with any valid genesis configuration.
2. Submit one or more block headers via `submit_blocks`.
3. Call `verify_transaction_inclusion` with a valid Merkle proof for a transaction in the most recently submitted block and set `confirmations = 0`.
4. The upper-bound check (`0 <= gc_threshold`) passes trivially.
5. The depth check (`(tip_height - target_height + 1) >= 0`) passes trivially for any `u64`.
6. The function returns `true` despite the transaction having zero confirmations.
7. A recipient contract acting on this result processes a transaction that may be invalidated by a subsequent chain reorganization. [7](#0-6)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
