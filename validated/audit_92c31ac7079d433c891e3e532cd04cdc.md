### Title
Fork Difficulty Validation Uses Main-Chain Timestamps Instead of Fork Ancestors — (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`)

---

### Summary

When validating the proof-of-work difficulty for a fork block, `get_next_work_required` calls `get_header_by_height` to retrieve the ancestor block at the start of the difficulty window. `get_header_by_height` always reads from `mainchain_height_to_header`, which only contains the **current main chain**. For a fork block whose ancestor chain diverges from the main chain, this returns the wrong block — the main chain block at that height instead of the fork's actual ancestor. The difficulty is therefore computed against the wrong timestamp, allowing fork blocks to be accepted with a lower (or higher) PoW target than the protocol requires.

---

### Finding Description

`get_header_by_height` is defined in `contract/src/lib.rs`:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // ← always the main chain
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

In `dogecoin.rs`, `get_next_work_required` calls this function to obtain the timestamp of the first block in the retarget window:

```rust
// TODO: check if it is correct to get block header by height from mainchain
//       without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [2](#0-1) 

The same pattern appears in `bitcoin.rs`:

```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
``` [3](#0-2) 

Fork blocks are stored only in `headers_pool` via `store_fork_header`, never in `mainchain_height_to_header`:

```rust
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
``` [4](#0-3) 

So when `get_header_by_height(height_first)` is called during fork-block validation, it returns the **main chain block** at `height_first`, not the fork's ancestor at that height. The two are different whenever the fork diverges before `height_first`.

The desynchronization is structurally identical to the external report: one parameter (the fork block being validated) is updated, while a coupled parameter (the timestamp used for difficulty) is silently taken from a different, unrelated state (the main chain). The developer's own `TODO` comment confirms awareness of the ambiguity.

---

### Impact Explanation

For **Dogecoin** (after block 145 000, where `difficulty_adjustment_interval = 1`), the retarget window is one block, so `height_first = prev_fork_block.height - 1`. Every fork block beyond the second uses the main chain's timestamp at that height instead of the fork's actual ancestor timestamp.

Concrete attack path:

1. Fork diverges from main-chain block A₀ (height 0). Attacker submits F₁ with a timestamp `T₁'` significantly later than the main chain's A₁ timestamp `T₁`.
2. When validating F₃ (height 3), `height_first = 1`. `get_header_by_height(1)` returns A₁ with timestamp `T₁`.
3. Digishield computes `modulated_timespan = F₂.time − T₁`. Because `T₁ < T₁'`, this timespan is **longer** than the correct `F₂.time − T₁'`, producing a **lower** (easier) difficulty target.
4. The attacker mines F₃ against this reduced target, satisfying the PoW check with less work than the protocol requires.

Repeating this across many fork blocks lets an attacker accumulate chain work faster than honest miners, potentially triggering a reorg that replaces the canonical chain. Once the fraudulent fork becomes the main chain, `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` will confirm transactions in attacker-controlled blocks.

For **Bitcoin**, the same bug applies at every 2016-block retarget boundary when the fork spans multiple difficulty epochs.

---

### Likelihood Explanation

The entry path is the public `submit_blocks` call, reachable by any NEAR account (including untrusted relayers). No privileged role is required. The attacker only needs to submit a sequence of fork headers with crafted timestamps — a capability explicitly supported by the contract's fork-handling logic. The `TODO` comment in the production code confirms the developers identified this as an open question, increasing the likelihood that it has not been mitigated.

---

### Recommendation

Replace `get_header_by_height` with an ancestor walk (`get_prev_header`) that follows the fork's own chain back to `height_first`, rather than reading from the main-chain index. This mirrors how `get_median_time_past` correctly traverses the fork chain via `get_prev_header` instead of `get_header_by_height`.

---

### Proof of Concept

**State before attack (Dogecoin, post-145 000):**

| Height | Main chain block | Fork block |
|--------|-----------------|------------|
| 0 | A₀ (time = 1000) | A₀ (shared) |
| 1 | A₁ (time = 1060) | F₁ (time = 9999) ← attacker sets far-future timestamp |
| 2 | A₂ (time = 1120) | F₂ (time = 10060) |
| 3 | A₃ (time = 1180) | F₃ (to be submitted) |

**Difficulty check for F₃:**

- `prev_block_header` = F₂ (height 2, time = 10060)
- `height_first = 2 − 1 = 1`
- `get_header_by_height(1)` → **A₁** (time = 1060) ← wrong; should be F₁ (time = 9999)
- `modulated_timespan = 10060 − 1060 = 9000` (correct would be `10060 − 9999 = 61`)
- Digishield clamps to `max_timespan`, but the resulting target is still far easier than the correct one
- F₃ is accepted with PoW satisfying the artificially easy target

The attacker repeats this for every subsequent fork block, accumulating chain work at a fraction of the honest cost. Once `fork.chain_work > mainchain.chain_work`, `reorg_chain` promotes the fork, and all subsequent SPV proofs are served from the attacker's chain. [5](#0-4) [6](#0-5)

```rust
fn get_next_work_required(
    config: &DogecoinConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    // Dogecoin: Special rules for minimum difficulty blocks with Digishield
    if allow_min_difficulty_for_block(config, block_header, prev_block_header) {
        // Special difficulty rule for testnet:
        // If the new block's timestamp is more than 2* nTargetSpacing minutes
        // then allow mining of a min-difficulty block.
        return config.proof_of_work_limit_bits;
    }

    // Only change once per difficulty adjustment interval
    let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
    let difficulty_adjustment_interval = if new_difficulty_protocol {
        1
    } else {
        config.difficulty_adjustment_interval
    };

    if (prev_block_header.block_height + 1) % difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            // Special difficulty rule for testnet:
            // If the new block's timestamp is more than 2* 10 minutes
            // then allow mining of a min-difficulty block.
            if block_header.time
                > prev_block_header.block_header.time + config.pow_target_spacing * 2
            {
                return config.proof_of_work_limit_bits;
            }

            // Return the last non-special-min-difficulty-rules-block
            let mut current_block_header = prev_block_header.clone();

            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            return current_block_header.block_header.bits;
        }

        return prev_block_header.block_header.bits;
    }

    // Litecoin: This fixes an issue where a 51% attack can change difficulty at will.
    // Go back the full period unless it's the first retarget after genesis. Code courtesy of Art Forz
    let mut blocks_to_go_back = difficulty_adjustment_interval - 1;
    if prev_block_header.block_height + 1 != difficulty_adjustment_interval {
        blocks_to_go_back = difficulty_adjustment_interval;
    }

    // Go back by what we want to be 14 days worth of blocks
    let height_first = prev_block_header
        .block_height
        .checked_sub(blocks_to_go_back)
        .unwrap_or_else(|| env::panic_str("Height underflow when calculating first block height"));

    // TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
    let first_block_time = blocks_getter
        .get_header_by_height(height_first)
        .block_header
        .time;

    calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))
```
