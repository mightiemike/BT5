### Title
`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` Misclassifies `AllResources` V3 Transactions as `L1Gas`, Producing a Wrong Transaction Hash During P2P Sync — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf-to-`ValidResourceBounds` conversion used in the P2P block-sync path applies a value-based heuristic: if both `l1_data_gas` and `l2_gas` are zero, it returns `ValidResourceBounds::L1Gas` instead of `ValidResourceBounds::AllResources`. This is the wrong discriminant for a V3 `AllResources` transaction whose gas bounds happen to be zero. Because `get_tip_resource_bounds_hash` produces a structurally different hash for `L1Gas` (2 resource elements) versus `AllResources` (3 resource elements, even when the third is zero), the transaction hash computed by a syncing node diverges from the hash committed by the sequencer, breaking hash canonicalization across the conversion boundary.

---

### Finding Description

**Root cause — wrong discriminant in protobuf deserialization**

In `crates/apollo_protobuf/src/converters/transaction.rs`, the conversion from `protobuf::ResourceBounds` to `ValidResourceBounds` is:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

The intent is to distinguish old pre-0.13.3 V3 transactions (which only signed L1 gas) from new `AllResources` transactions. But the discriminant is the *value* of the bounds, not a structural field. A new V3 `AllResources` transaction with `l1_data_gas = {0, 0}` and `l2_gas = {0, 0}` satisfies `l1_data_gas.is_zero() && l2_gas.is_zero()` and is therefore silently downgraded to `L1Gas`.

**Hash divergence — `get_tip_resource_bounds_hash` is variant-sensitive**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` produces structurally different output depending on the variant:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 elements total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 elements total
    }
});
``` [2](#0-1) 

Even when `l1_data_gas` is all-zero, `get_concat_resource` encodes the resource name bytes (`b"L1_DATA"`) into the Felt, producing a non-zero value: [3](#0-2) 

So `poseidon_hash(tip, L1_gas_felt, L2_gas_felt)` ≠ `poseidon_hash(tip, L1_gas_felt, L2_gas_felt, L1_data_gas_felt_with_name_bytes)`.

**Sequencer path uses `AllResources` — syncing path uses `L1Gas`**

The gateway and `TransactionConverter` always receive `RpcInvokeTransactionV3` with `resource_bounds: AllResourceBounds`, which is unconditionally wrapped as `ValidResourceBounds::AllResources`: [4](#0-3) 

The hash is computed and stored as `AllResources` (3-element hash). The mempool P2P propagation path uses a separate converter (`rpc_transaction.rs`) that also always produces `AllResources`: [5](#0-4) 

But the **block-sync P2P path** uses `transaction.rs`, which applies the value-based heuristic and can produce `L1Gas` for the same transaction. When the syncing node recomputes the hash from the deserialized transaction, it gets a 2-element hash that does not match the 3-element hash committed by the sequencer.

**`InternalRpcTransactionWithoutTxHash::calculate_transaction_hash` is the canonical hash entry point** [6](#0-5) 

Both the sequencer and the syncing node call this function. The sequencer calls it with `AllResources`; the syncing node calls it with `L1Gas`. The outputs differ.

---

### Impact Explanation

This is a **hash/domain canonicalization failure** at the protobuf conversion boundary. A V3 `AllResources` transaction with zero `l1_data_gas` and `l2_gas` is committed by the sequencer under hash H₁ (3-element resource hash). A syncing node deserializing the same transaction from protobuf computes hash H₂ ≠ H₁ (2-element resource hash). Any hash-verification step in the sync path will reject the block or record the transaction under the wrong hash, breaking state consistency. This matches the allowed impact: **"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."**

---

### Likelihood Explanation

A V3 transaction with `AllResources` where both `l1_data_gas` and `l2_gas` are fully zero (`max_amount = 0, max_price_per_unit = 0`) is an unusual but structurally valid submission. The gateway's fee-bounds check (`check_fee_bounds`) may reject such a transaction if `minimal_gas_amount.l2_gas > 0`, but for transactions where the minimal L2 gas is zero or fee enforcement is disabled, the transaction is accepted. The condition is also reachable via any path that constructs `AllResourceBounds` with zero fields before calling `calculate_transaction_hash`.

---

### Recommendation

In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` (`crates/apollo_protobuf/src/converters/transaction.rs`), use structural presence — not value — to discriminate between `L1Gas` and `AllResources`. Specifically, produce `L1Gas` only when `l1_data_gas` is absent (`None`) in the protobuf message, and produce `AllResources` whenever all three fields are present (even if zero):

```rust
Ok(match value.l1_data_gas {
    None => ValidResourceBounds::L1Gas(l1_gas),
    Some(_) => ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }),
})
```

This mirrors the intent of the TODO comment (`// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.`) and eliminates the value-based ambiguity. [7](#0-6) 

---

### Proof of Concept

1. Submit a V3 invoke transaction with `resource_bounds = AllResourceBounds { l1_gas: {X, Y}, l2_gas: {0, 0}, l1_data_gas: {0, 0} }` to the gateway.
2. `TransactionConverter::convert_rpc_tx_to_internal` wraps it as `ValidResourceBounds::AllResources(...)` and calls `get_invoke_transaction_v3_hash`, which calls `get_tip_resource_bounds_hash` with 3 elements → hash **H₁**.
3. The transaction is included in a block and committed.
4. A syncing node receives the block via P2P. It deserializes the transaction using `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `transaction.rs`. Since `l1_data_gas.is_zero() && l2_gas.is_zero()`, it produces `ValidResourceBounds::L1Gas(l1_gas)`.
5. The syncing node calls `get_invoke_transaction_v3_hash` with `L1Gas`, which calls `get_tip_resource_bounds_hash` with 2 elements → hash **H₂**.
6. H₁ ≠ H₂ because `poseidon_hash(..., L1_data_gas_felt_with_name_bytes)` ≠ `poseidon_hash(...)`. The syncing node's hash verification fails, causing block rejection or incorrect transaction storage. [8](#0-7) [9](#0-8)

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
}

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

fn get_concat_resource(
    resource_bounds: &ResourceBounds,
    resource_name: &ResourceName,
) -> Result<Felt, StarknetApiError> {
    let max_amount = resource_bounds.max_amount.0.to_be_bytes();
    let max_price = resource_bounds.max_price_per_unit.0.to_be_bytes();
    let concat_bytes =
        [[0_u8].as_slice(), resource_name.as_slice(), max_amount.as_slice(), max_price.as_slice()]
            .concat();
    Ok(Felt::from_bytes_be(&concat_bytes.try_into().expect("Expect 32 bytes")))
}
