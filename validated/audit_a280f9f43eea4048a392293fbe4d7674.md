### Title
`block_casm_hash_v1_declares` Version Gate Skips `DeclareTransactionV2`, Allowing V1 Compiled Class Hash to Be Committed to State — (`crates/blockifier/src/transaction/transactions.rs`)

---

### Summary

The `block_casm_hash_v1_declares` feature flag, enabled in `VersionedConstants` from Starknet 0.14.1 onward, is intended to reject any declare transaction whose `compiled_class_hash` field carries a V1 (Poseidon) hash instead of the required V2 (Blake2) hash. The enforcement call `check_compile_class_hash_v2_declaration()` is guarded by `self.version() >= TransactionVersion::THREE`, so it is silently skipped for every `DeclareTransactionV2`. A V2 declare carrying a V1 compiled class hash is therefore accepted and committed to state unchanged, producing a wrong `class_hash → compiled_class_hash` mapping that the OS/prover will later reject.

---

### Finding Description

In `crates/blockifier/src/transaction/transactions.rs`, the `Executable::run_execute` implementation for `DeclareTransaction` merges the V2 and V3 arms into a single `match` branch:

```rust
// lines 176-193
starknet_api::transaction::DeclareTransaction::V2(DeclareTransactionV2 {
    compiled_class_hash, ..
})
| starknet_api::transaction::DeclareTransaction::V3(DeclareTransactionV3 {
    compiled_class_hash, ..
}) => {
    if context.tx_context.block_context.versioned_constants.block_casm_hash_v1_declares
        && self.version() >= TransactionVersion::THREE   // ← V2 is excluded
    {
        self.check_compile_class_hash_v2_declaration()?
    }
    try_declare(self, state, class_hash, Some(*compiled_class_hash))?
}
```

`DeclareTransaction::V2` returns `TransactionVersion::TWO` from `version()`, which is strictly less than `TransactionVersion::THREE`. The guard therefore never fires for V2 transactions, regardless of the flag value.

`check_compile_class_hash_v2_declaration()` (in `crates/starknet_api/src/executable_transaction.rs`, lines 228–244) recomputes the compiled class hash using `HashVersion::V2` (Blake2) and returns `DeclareTransactionCasmHashMissMatch` if the transaction's `compiled_class_hash` field does not match. Because this call is bypassed for V2, a V2 declare whose `compiled_class_hash` is the Poseidon (V1) hash of the CASM passes through `try_declare` and is written to state.

The `block_casm_hash_v1_declares` flag was introduced precisely to prevent V1 hashes from entering state once the protocol migrated to V2 hashes (versioned constants diff shows it toggled `true` at 0.14.1). The analogous `enable_casm_hash_migration` flag was enabled at the same version boundary, meaning the OS is expected to encounter only V2 hashes in state for blocks at that version or later.

---

### Impact Explanation

**Wrong state from blockifier execution logic for accepted input (Critical).**

When `block_casm_hash_v1_declares = true` and a `DeclareTransactionV2` carrying a V1 compiled class hash is executed:

1. `try_declare` writes `class_hash → compiled_class_hash_v1` into the state diff.
2. The OS, operating at a block version ≥ 0.14.1, selects the Blake2 (`V2`) compiled-class-hash entry point to verify the class.
3. The OS-computed V2 hash does not equal the stored V1 hash; the OS assertion fails.
4. The resulting block cannot be proven, and the committed state diff contains an incorrect `class_hash_to_compiled_class_hash` entry.

This satisfies the "Wrong state … from blockifier/syscall/execution logic for accepted input" critical impact criterion.

---

### Likelihood Explanation

`DeclareTransactionV2` transactions reach the blockifier through two production paths:

- **State sync** (`apollo_central_sync`): the sync layer replays historical blocks from the feeder gateway, which may contain V2 declares with V1 hashes committed before the flag was introduced. Once the Apollo node's versioned constants are at 0.14.1+, those replayed V2 transactions bypass the check.
- **`native_blockifier` Python interface** (`py_declare`): accepts all four declare versions and passes them directly to the blockifier without the gateway's V3-only filter.

Neither path requires any special privilege beyond the ability to submit or replay a V2 declare transaction.

---

### Recommendation

Extend the version gate to cover `DeclareTransactionV2` as well:

```diff
- if context.tx_context.block_context.versioned_constants.block_casm_hash_v1_declares
-     && self.version() >= TransactionVersion::THREE
+ if context.tx_context.block_context.versioned_constants.block_casm_hash_v1_declares
+     && self.version() >= TransactionVersion::TWO
  {
      self.check_compile_class_hash_v2_declaration()?
  }
```

This mirrors the existing pattern for `disable_cairo0_redeclaration`, which applies to V0/V1 without a version lower-bound exclusion, and is consistent with the intent expressed in the TODO comment in `transaction_converter.rs`:

```rust
// TODO(Aviv): Ensure that we do not want to
// allow declare with compiled class hash v1.
```

### Proof of Concept

1. Run the sequencer with versioned constants ≥ 0.14.1 (`block_casm_hash_v1_declares = true`).
2. Construct a `DeclareTransactionV2` whose `compiled_class_hash` field is set to `casm.hash(&HashVersion::V1)` (Poseidon hash) instead of `casm.hash(&HashVersion::V2)` (Blake2 hash).
3. Deliver the transaction via the `native_blockifier` Python interface or inject it through the state-sync replay path.
4. Observe that `run_execute` does **not** call `check_compile_class_hash_v2_declaration()` (the `self.version() >= THREE` guard is false) and `try_declare` commits `class_hash → compiled_class_hash_v1` to state.
5. Confirm the wrong value: `state.get_compiled_class_hash(class_hash)` returns the V1 hash, not the V2 hash that the OS will compute during proving.

```rust
    /// Verifies that the compiled class hash field in the declare tx,
    /// is compiled_class_hash_v2 of the compiled contract.
    pub fn check_compile_class_hash_v2_declaration(&self) -> Result<(), StarknetApiError> {
        let compiled_class = &self.class_info.contract_class;
        let compiled_class_hash_v2 = match &compiled_class {
            ContractClass::V0(_) => return Ok(()),
            ContractClass::V1((casm, _)) => casm.hash(&HashVersion::V2),
        };
        let compiled_class_hash = self.compiled_class_hash();
        if compiled_class_hash_v2 != compiled_class_hash {
            let err_var = CasmHashMismatch {
                hash: self.class_hash(),
                actual: compiled_class_hash,
                expected: compiled_class_hash_v2,
            };
            return Err(StarknetApiError::DeclareTransactionCasmHashMissMatch(Box::new(err_var)));
        }
        Ok(())
    }
```
