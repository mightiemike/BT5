### Title
Duplicate-Last-Leaf Position Reuse in Odd-Width Merkle Tree Allows Same Transaction to Verify at Two Distinct Indices — (`merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

---

### Summary

`compute_root_from_merkle_proof` does not bind a proof to a unique leaf position. In a Bitcoin block with an odd transaction count, the last leaf is duplicated during Merkle tree construction. This means the same `tx_id` and the same sibling proof path produce an identical root when submitted with either the real last index **N** or the phantom index **N+1**. `verify_transaction_inclusion_v2` returns `true` for both. The coinbase-proof guard added in v2 does not close this gap because it only validates that the coinbase is at position 0; it is entirely independent of `tx_index`.

---

### Finding Description

**Merkle tree construction for an odd-count block (e.g., 3 txs: T0, T1, T2):**

```
Level 0 (leaves): T0  T1  T2  T2   ← T2 duplicated
Level 1:          H(T0,T1)  H(T2,T2)
Level 2 (root):   H(H(T0,T1), H(T2,T2))
```

**Proof for T2 at index 2 (real position):**

`compute_root_from_merkle_proof(T2, 2, [T2, H(T0,T1)])`
- Step 1: position=2 (even) → `H(T2, T2)` = `H(T2,T2)`; position → 1
- Step 2: position=1 (odd) → `H(H(T0,T1), H(T2,T2))` = root ✓

**Proof for T2 at index 3 (phantom position):**

`compute_root_from_merkle_proof(T2, 3, [T2, H(T0,T1)])`
- Step 1: position=3 (odd) → `H(T2, T2)` = `H(T2,T2)`; position → 1
- Step 2: position=1 (odd) → `H(H(T0,T1), H(T2,T2))` = root ✓

Both calls return the same root. The contract accepts both.

The core issue is in `compute_root_from_merkle_proof`: [1](#0-0) 

The function only uses `current_position % 2` to decide left/right placement at each level. It has no knowledge of the actual tree width at any level, so it cannot detect that index 3 is out of range for a 3-transaction block.

The v2 coinbase guard: [2](#0-1) 

fixes the 64-byte forgery attack by anchoring proof depth to the coinbase path, but it is completely orthogonal to `tx_index`. It verifies `coinbase_tx_id` at hardcoded position `0` and says nothing about whether the caller's `tx_index` is within the valid range of the block's transaction list.

After the coinbase check passes, `verify_transaction_inclusion` is called with the attacker-supplied `tx_index`: [3](#0-2) 

which delegates directly to `compute_root_from_merkle_proof` with no bounds check: [4](#0-3) 

---

### Impact Explanation

A downstream bridge, unlock, or mint contract that uses `(block_hash, tx_index)` as its replay-protection key can be triggered twice for the same underlying transaction:

1. Attacker submits a legitimate proof: `tx_index=2` → bridge processes the event (e.g., mints tokens).
2. Attacker submits the same proof with `tx_index=3` → `verify_transaction_inclusion_v2` returns `true` again → bridge processes the same event a second time.

The contract is asserting "there is a transaction T2 at index 3 in this block" when no transaction exists at index 3. This is a false positive from the light client, which is the exact invariant the verification function is supposed to enforce.

---

### Likelihood Explanation

- The attack requires only a block with an odd transaction count, which is common on Bitcoin mainnet.
- The attacker needs no privileged role; `verify_transaction_inclusion_v2` is a public view/call.
- The proof construction is straightforward: reuse the legitimate proof for the last transaction, change `tx_index` from N to N+1.
- The coinbase proof is unchanged and valid, so the v2 guard passes without modification.

---

### Recommendation

In `compute_root_from_merkle_proof`, enforce that the claimed position is consistent with the proof length. Specifically, after processing all proof levels, `current_position` must equal 0 (i.e., the path must have reached the root from a valid leaf). Additionally, the caller should pass the total transaction count so the function can reject `tx_index >= tx_count`. Alternatively, require that `tx_index < 2^(proof.len())` AND that the proof path does not produce a self-pairing at any level unless `tx_index` is the last leaf in an odd-width level — which requires tracking tree width per level.

The simplest safe fix: require `tx_index < 2^(merkle_proof.len())` as a necessary (though not sufficient) bound, and additionally verify that after all steps `current_position == 0`.

---

### Proof of Concept

For a 3-transaction block with transactions T0 (coinbase), T1, T2:

```rust
// Both of these calls return `true` against the same block:

// Call 1 — legitimate
verify_transaction_inclusion_v2(ProofArgsV2 {
    tx_id: T2,
    tx_block_blockhash: block_hash,
    tx_index: 2,                        // real position
    merkle_proof: vec![T2, H(T0,T1)],
    coinbase_tx_id: T0,
    coinbase_merkle_proof: vec![T1, H(T2,T2)],
    confirmations: 0,
});

// Call 2 — phantom position, identical proof
verify_transaction_inclusion_v2(ProofArgsV2 {
    tx_id: T2,
    tx_block_blockhash: block_hash,
    tx_index: 3,                        // does not exist in the block
    merkle_proof: vec![T2, H(T0,T1)],  // identical proof
    coinbase_tx_id: T0,
    coinbase_merkle_proof: vec![T1, H(T2,T2)],
    confirmations: 0,
});
```

Both return `true`. A bridge using `(block_hash, tx_index)` as a deduplication key will process the event twice.

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
```
