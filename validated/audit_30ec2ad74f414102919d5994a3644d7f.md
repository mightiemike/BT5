### Title
SwapAllowlistExtension Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router's address, not the actual user's address. If the pool admin allowlists the router (a natural operational requirement so users can access slippage protection and multi-hop routing), every unpermissioned user can bypass the swap allowlist by routing through the public router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs its identity check as:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension is called by the pool), and `sender` is the value the pool passes — which is `msg.sender` of `pool.swap()`. [1](#0-0) 

`MetricOmmPool.swap()` passes `msg.sender` as `sender` to `_beforeSwap`: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly, making itself `msg.sender` of that call: [3](#0-2) 

So when a user routes through the router, the extension sees `sender = router address`, not the actual user. The extension then checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`.

The same identity-loss applies to `exactInput`, `exactOutputSingle`, and `exactOutput` — all of them call `pool.swap()` directly from the router contract. [4](#0-3) 

The router does not forward the original user's identity to the pool in any form. There is no mechanism in the extension interface to receive the true initiator.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` in `BEFORE_SWAP_ORDER` is intended to restrict swaps to a specific set of addresses (e.g., KYC-verified users, protocol partners, or whitelisted market makers). If the pool admin allowlists the router address — a natural operational step so that users can benefit from slippage protection, deadline enforcement, and multi-hop routing — the allowlist is completely defeated: any address, regardless of allowlist status, can call `MetricOmmSimpleRouter.exactInputSingle()` and execute a swap in the restricted pool. The pool's access-control invariant is broken for every swap routed through the periphery.

Depending on the pool's purpose, this can result in:
- Unauthorized users draining liquidity from a permissioned pool
- Compliance/KYC controls being silently bypassed
- Swap-direction restrictions (e.g., one-sided allowlists) being circumvented

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap interface documented and deployed by the protocol. A pool admin who wants users to access slippage protection or multi-hop routing has no choice but to allowlist the router. The bypass is then reachable by any unpermissioned user with no special setup, no privileged access, and no non-standard tokens. The only precondition is that the pool admin has allowlisted the router — a natural and expected operational action.

---

### Recommendation

`SwapAllowlistExtension.beforeSwap` should gate on the **economically relevant actor**, not the immediate caller. Two complementary fixes:

1. **Check both `sender` and the original user**: Extend the extension interface or `extensionData` to carry the original initiator when the router is the caller. The router would encode `msg.sender` into `extensionData`, and the extension would verify it.

2. **Alternatively, check `sender` only when it is not a trusted router**: Maintain a registry of trusted periphery contracts in the extension; when `sender` is a trusted router, require the extension data to contain a signed or encoded user address that is then checked against the allowlist.

The simplest safe default is to **not allowlist the router address** and document that pools with `SwapAllowlistExtension` must be accessed directly, not through the router. However, this removes the router's usability benefits for permissioned pools and should be explicitly documented.

---

### Proof of Concept

```
Setup:
  1. Deploy pool with SwapAllowlistExtension in BEFORE_SWAP_ORDER.
  2. Pool admin calls setAllowedToSwap(pool, alice, true)       // alice is allowlisted
  3. Pool admin calls setAllowedToSwap(pool, router, true)      // router allowlisted for UX
  4. bob is NOT allowlisted.

Direct call (blocked correctly):
  bob → pool.swap(...)
    → _beforeSwap(sender=bob, ...)
    → allowedSwapper[pool][bob] == false → revert NotAllowedToSwap ✓

Router bypass (allowlist defeated):
  bob → MetricOmmSimpleRouter.exactInputSingle(pool=pool, ...)
    → pool.swap(...)  [msg.sender = router]
    → _beforeSwap(sender=router, ...)
    → allowedSwapper[pool][router] == true → passes ✓
    → swap executes for bob despite bob not being allowlisted ✗
```

Root cause: `sender` in `SwapAllowlistExtension.beforeSwap` is the router address, not `bob`. The allowlist check passes because the router is allowlisted, not because `bob` is. [1](#0-0) [5](#0-4) [6](#0-5)
