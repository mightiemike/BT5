### Title
Bonus Reward Checkpoints Advance Without Payment on Insufficient Pool During Redelegation Slash — (`x/tieredrewards/keeper/slash.go`)

---

### Summary

In `x/tieredrewards`, when a redelegation slash fires and the bonus pool is insufficient, `processEventsAndClaimBonus` advances the position's reward checkpoints and decrements validator event reference counts in the store **before** the pool sufficiency check. The `BeforeRedelegationSlashed` hook deliberately swallows `ErrInsufficientBonusPool` to avoid a chain halt, then persists the position with the advanced checkpoints. The result is that the accrual window is permanently closed without any payment — the user's bonus rewards for that period are irreversibly lost.

---

### Finding Description

`processEventsAndClaimBonus` in `x/tieredrewards/keeper/claim_rewards.go` performs two categories of state mutation before it checks whether the pool can cover the computed bonus:

1. **Store writes** — `decrementEventRefCount` is called inside the event loop, writing to the `ValidatorEvents` store for every processed event. If an event's reference count reaches zero, the event is garbage-collected.
2. **In-memory checkpoint advancement** — `applyBonusAccrualCheckpoint` sets `pos.LastBonusAccrual = blockTime` and `pos.UpdateLastKnownBonded(bonded)` updates `LastKnownBonded`. These are written to the in-memory `pos` struct.

Only after all of the above does the function call `sufficientBonusPoolBalance`:

```go
// claim_rewards.go lines 172-199: event loop — decrements ref counts in store
for _, entry := range events {
    ...
    pos.UpdateLastEventSeq(entry.Seq)
    if err := k.decrementEventRefCount(ctx, valAddr, entry.Seq); err != nil { // store write
        return nil, err
    }
}

// lines 215-217: checkpoint advancement — in-memory
applyBonusAccrualCheckpoint(&pos.Position, blockTime)
pos.UpdateLastKnownBonded(bonded)

// line 230: pool check — AFTER all mutations
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err  // returns ErrInsufficientBonusPool here
}
```

In `slashRedelegationPosition` (`slash.go` lines 54–64), `ErrInsufficientBonusPool` is explicitly swallowed:

```go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
        // error swallowed — execution continues with advanced pos
    } else {
        return err
    }
}
```

Execution then falls through to `setPositionWithState` (lines 71 or 77), which persists the position with `LastBonusAccrual = blockTime` and the decremented event ref counts already committed to the store. No bonus is paid. The accrual window `[old_LastBonusAccrual, blockTime]` is permanently consumed.

---

### Impact Explanation

The corrupted values are:

- `pos.LastBonusAccrual` — advanced to `blockTime` without payment; the next claim starts from `blockTime`, permanently skipping the unpaid window.
- `ValidatorEvents[valAddr][seq].ReferenceCount` — decremented in the store; events may be garbage-collected, making the accrual history unrecoverable.
- `pos.LastEventSeq` — advanced past events that were never paid out.

The bonus rewards accrued during `[old_LastBonusAccrual, blockTime]` are permanently lost to the position owner. The pool retains the funds (no theft), but the user's entitlement is silently forfeited. This matches the external report's **permanent freezing of unclaimed yield** impact.

---

### Likelihood Explanation

The trigger requires two concurrent conditions:

1. A tier position is in an active redelegation (i.e., `MsgTierRedelegate` was called and the redelegation entry has not yet matured).
2. The bonus rewards pool (`RewardsPoolName`) has insufficient balance to cover the accrued bonus at the time of the redelegation slash.

The pool can be drained naturally by the `BeginBlocker` top-up mechanism (which transfers pool funds to `x/distribution` every block when `TargetBaseRewardsRate > 0`) or by other users claiming large bonus rewards. A validator misbehavior event (double-sign or downtime) that triggers `SlashRedelegation` while the pool is low is a realistic scenario, not a theoretical one. An adversary who can observe pool balance and time a validator slash (e.g., by controlling a validator) can reliably trigger this.

---

### Recommendation

Move the `sufficientBonusPoolBalance` check and the `SendCoinsFromModuleToAccount` call **before** any store mutations (ref count decrements) and before advancing the in-memory checkpoints. Alternatively, restructure `processEventsAndClaimBonus` so that all store writes are deferred until after the pool check succeeds. In `slashRedelegationPosition`, if the pool is insufficient, the position's checkpoints should **not** be advanced — the position should be left at its current `LastBonusAccrual` so the user can claim the bonus once the pool is replenished.

---

### Proof of Concept

Sequence of events:

1. User calls `MsgLockTier` → position created, `LastBonusAccrual = T0`.
2. User calls `MsgTierRedelegate` → redelegation entry created, `RedelegationMappings[unbondingId] = positionId`.
3. Time advances to `T1`; bonus accrues for `[T0, T1]`.
4. Bonus pool is empty (drained by BeginBlocker top-up or other claims).
5. Validator misbehaves → staking calls `SlashRedelegation` → fires `BeforeRedelegationSlashed(ctx, unbondingId, sharesToUnbond)`.
6. `slashRedelegationPosition` calls `processEventsAndClaimBonus`:
   - Event ref counts decremented in store.
   - `pos.LastBonusAccrual` set to `T1` in memory.
   - `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool`.
7. Error swallowed in `slashRedelegationPosition`; `setPositionWithState` called with `pos.LastBonusAccrual = T1`.
8. Position persisted: `LastBonusAccrual = T1`, no bonus paid.
9. User later calls `MsgClaimTierRewards` → bonus computed for `[T1, now]` only; the `[T0, T1]` window is permanently gone.
