### Title
Protobuf `ValidResourceBounds` Deserialization Collapses `AllResources` to `L1Gas` When `l2_gas` and `l1_data_gas` Are Zero, Producing a Divergent Transaction Hash - (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion in `crates/apollo_protobuf/src/converters/transaction.rs` uses the runtime condition `l1_data_gas.is_zero() && l2_gas.is_zero()` to decide between `ValidResourceBounds::L1Gas` and `ValidResourceBounds::AllResources`. Because `get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` hashes a **different number of resource-bound felts** depending on that variant, a post-0.13.3 `AllResources` transaction whose `l2_gas` and `l1_data_gas` happen to be zero will produce a **different transaction hash** after a protobuf round-trip than the hash that was computed and signed at submission time.

---

### Finding Description

**Hash computation is variant-sensitive.**

`get_tip_resource_bounds_hash` builds the fee-fields hash that is committed into every V3 transaction hash:

```
// L1Gas  → hashes [tip, L1_GAS_concat, L2_GAS_zero_concat]          (3 elements)
// AllResources → hashes [tip, L1_GAS_concat, L2_GAS_concat, L1_DATA_GAS_concat]  (4 elements)
```

Even when `l2_gas` and `l1_data_gas` are both zero, the `AllResources` path appends a fourth element (`L1_DATA_GAS_zero_concat`) that the `L1Gas` path omits entirely. The two poseidon digests are therefore **structurally different** for the same numeric field values.

**Protobuf deserialization silently changes the variant.**

```rust
// crates/apollo_protobuf/src/converters/transaction.rs  line 431
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← drops AllResources semantics
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

A transaction that was originally submitted as `AllResources` (the only variant the gateway produces for post-0.13.3 transactions) with `l2_gas = 0` and `l1_data_gas = 0` is silently re-typed to `L1Gas` after protobuf deserialization. The comment on line 426 even acknowledges the ambiguity: `// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.`

**The hash is recomputed after deserialization.**

In the consensus/P2P path, `convert_consensus_tx_to_internal_consensus_tx` calls `convert_rpc_tx_to_internal`, which ends with:

```rust
let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
```

`calculate_transaction_hash` for an invoke V3 transaction calls `get_invoke_transaction_v3_hash`, which calls `get_tip_resource_bounds_hash`. If the deserialized `ValidResourceBounds` is now `L1Gas` instead of `AllResources`, the recomputed hash diverges from the hash the sender signed and stored.

**Concrete divergence.**

| Step | Variant used | Elements hashed | Resulting hash |
|------|-------------|-----------------|----------------|
| Sender submits `AllResources(l1=X, l2=0, l1d=0)` | `AllResources` | tip, L1_GAS(X), L2_GAS(0), L1_DATA_GAS(0) | H_A |
| Receiver deserializes protobuf | `L1Gas(l1=X)` | tip, L1_GAS(X), L2_GAS(0) | H_B ≠ H_A |

---

### Impact Explanation

The recomputed hash `H_B` does not match the sender's signed hash `H_A`. Depending on where the recomputed hash is used:

- **Mempool/gateway admission**: a valid transaction is rejected because the hash the receiver derives does not match the hash the sender signed and propagated.
- **Signature binding**: the signature covers `H_A`; the sequencer stores or executes the transaction under `H_B`, binding the wrong hash to the signer's intent.
- **State/receipt**: if the wrong hash is committed to the block, the on-chain receipt and event logs reference a hash that cannot be reproduced from the transaction fields, breaking block verification.

This matches the allowed impact: *"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

---

### Likelihood Explanation

The trigger requires a post-0.13.3 `AllResources` transaction where both `l2_gas.max_amount` and `l1_data_gas.max_amount` are zero (only `l1_gas` is non-zero). The gateway's stateless validator accepts such transactions (it only requires at least one resource bound to be non-zero). A user who sets only `l1_gas` and leaves the other two at zero — a plausible configuration for a legacy-style V3 transaction — will hit this path. The trigger is fully unprivileged.

---

### Recommendation

The `ValidResourceBounds` variant must be preserved across serialization boundaries. The protobuf schema should carry an explicit discriminant (e.g., a boolean `is_all_resources` field, or a `oneof` wrapper) rather than inferring the variant from whether numeric fields happen to be zero. Until the schema is updated, the conversion should default to `AllResources` for any transaction that arrives on a post-0.13.3 code path, since `L1Gas` is only valid for pre-0.13.3 transactions:

```rust
// Proposed fix: always produce AllResources; L1Gas is only for legacy pre-0.13.3 txs
Ok(ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }))
```

Alternatively, add a version tag to the protobuf `ResourceBounds` message and gate the `L1Gas` branch on an explicit version check.

---

### Proof of Concept

1. Craft an `AllResources` invoke V3 transaction with `l1_gas = {max_amount: 1000, max_price_per_unit: 1}`, `l2_gas = {0, 0}`, `l1_data_gas = {0, 0}`.
2. Submit to the gateway. The gateway calls `InternalRpcTransactionWithoutTxHash::calculate_transaction_hash`, which calls `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash` with `AllResources`. The hash `H_A` is computed over **4 elements**: `[tip, L1_GAS_concat, L2_GAS_zero_concat, L1_DATA_GAS_zero_concat]`.
3. The transaction is propagated to a peer via protobuf. The peer deserializes `protobuf::ResourceBounds` via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Because `l2_gas.is_zero() && l1_data_gas.is_zero()`, the peer produces `ValidResourceBounds::L1Gas(l1_gas)`.
4. The peer calls `calculate_transaction_hash` on the deserialized transaction. `get_tip_resource_bounds_hash` now hashes only **3 elements**: `[tip, L1_GAS_concat, L2_GAS_zero_concat]`, producing `H_B ≠ H_A`.
5. The peer rejects the transaction (hash mismatch) or, if it accepts it, stores it under `H_B` — a hash the account never signed.

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

pub(crate) fn get_invoke_transaction_v3_hash<T: InvokeTransactionV3Trait>(
    transaction: &T,
    chain_id: &ChainId,
    transaction_version: &TransactionVersion,
) -> Result<TransactionHash, StarknetApiError> {
    let tip_resource_bounds_hash =
        get_tip_resource_bounds_hash(&transaction.resource_bounds(), transaction.tip())?;
    let paymaster_data_hash =
        HashChain::new().chain_iter(transaction.paymaster_data().0.iter()).get_poseidon_hash();
    let data_availability_mode = concat_data_availability_mode(
        transaction.nonce_data_availability_mode(),
        transaction.fee_data_availability_mode(),
    );
    let account_deployment_data_hash = HashChain::new()
        .chain_iter(transaction.account_deployment_data().0.iter())
        .get_poseidon_hash();
    let calldata_hash =
        HashChain::new().chain_iter(transaction.calldata().0.iter()).get_poseidon_hash();
    let mut hash_chain = HashChain::new()
        .chain(&INVOKE)
        .chain(&transaction_version.0)
        .chain(transaction.sender_address().0.key())
        .chain(&tip_resource_bounds_hash)
        .chain(&paymaster_data_hash)
        .chain(&Felt::try_from(chain_id)?)
        .chain(&transaction.nonce().0)
        .chain(&data_availability_mode)
        .chain(&account_deployment_data_hash)
        .chain(&calldata_hash);
    if !transaction.proof_facts().0.is_empty() {
        let proof_facts_hash =
            HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
        hash_chain = hash_chain.chain(&proof_facts_hash);
    }
    Ok(TransactionHash(hash_chain.get_poseidon_hash()))
