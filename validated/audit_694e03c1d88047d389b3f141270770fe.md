### Title
CEI Violation in `withdrawCollateral` — Token Transfer Before Balance Update With No Reentrancy Guard — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` transfers tokens to `sendTo` before updating the `spotEngine` balance or running the health check. No reentrancy guard exists anywhere in the codebase. The publicly callable `submitFastWithdrawal` in `BaseWithdrawPool` provides a concrete re-entry surface: a malicious recipient contract can re-enter `submitFastWithdrawal` with a valid signed withdrawal transaction during the token transfer, draining pool liquidity before the `spotEngine` balance is decremented.

---

### Finding Description

`Clearinghouse.withdrawCollateral` follows the Interactions-Effects-Checks anti-pattern: [1](#0-0) 

Line 408 calls `handleWithdrawTransfer`, which chains into `BaseWithdrawPool.submitWithdrawal`, which calls `token.safeTransfer(sendTo, amount)`. Only after that external call returns does execution reach line 412 (`spotEngine.updateBalance`) and line 419 (health check). The `spotEngine` balance for the withdrawing subaccount is therefore stale — still reflecting the full pre-withdrawal amount — during the entire token transfer.

The internal transfer chain is: [2](#0-1) [3](#0-2) 

The `token.safeTransfer(sendTo, amount)` at line 131 of `BaseWithdrawPool` is the exact point where control passes to an attacker-controlled contract. At that moment, `spotEngine` still records the full balance.

`submitFastWithdrawal` is the publicly callable re-entry surface — it has no access control and no reentrancy guard: [4](#0-3) 

Critically, `submitFastWithdrawal` does **not** update the `spotEngine` balance. It only marks `markedIdxs[idx] = true` and transfers tokens from the pool. The `spotEngine` balance decrement for a fast withdrawal only occurs later, when the sequencer processes the same transaction via `processTransactionImpl` → `withdrawCollateral` → `submitWithdrawal`. At that point `submitWithdrawal` returns early (line 124–126) because `markedIdxs[idx]` is already set — but `withdrawCollateral` still calls `spotEngine.updateBalance` after `handleWithdrawTransfer` returns.

A grep across the entire codebase confirms there is no `nonReentrant` modifier or `ReentrancyGuard` import anywhere.

---

### Impact Explanation

An attacker with a malicious contract as their subaccount wallet and a valid verifier-signed fast-withdrawal transaction (legitimately obtained by having deposited funds) can:

1. Trigger a slow-mode `WithdrawCollateral` for their full balance X.
2. When `submitWithdrawal` calls `token.safeTransfer(attacker_contract, X)`, re-enter `submitFastWithdrawal` with a valid signed transaction for amount Y and a higher `idx`.
3. `submitFastWithdrawal` marks `markedIdxs[fast_idx] = true` and transfers Y tokens from the pool to the attacker — while the `spotEngine` balance is still X (not yet decremented).
4. Execution returns to `withdrawCollateral`, which decrements the balance by X and runs the health check against the now-correct balance.
5. When the sequencer later processes the fast-withdrawal transaction, `submitWithdrawal` returns early (markedIdxs already set), so no second token transfer occurs — but `withdrawCollateral` still decrements the balance by Y. If the attacker's balance is already 0 or insufficient, this pushes the balance negative, corrupting the accounting.

The pool's liquidity (provided by fast-withdrawal liquidity providers) is drained by Y without a corresponding on-chain balance debit at the time of the attack. The `spotEngine` accounting is corrupted: the attacker's balance is decremented twice (once for the slow withdrawal, once when the sequencer processes the fast withdrawal), but the attacker received tokens for both.

---

### Likelihood Explanation

Any user who:
- Has deposited collateral and obtained a valid verifier-signed fast-withdrawal transaction (normal protocol usage), and
- Controls a contract address as their subaccount wallet (straightforward to arrange),

can execute this attack. The slow-mode path (`processSlowModeTransactionImpl`, line 217–229) resolves `sendTo` to `address(uint160(bytes20(sender)))`, so if the user's registered address is a contract, the re-entry fires automatically. No privileged access, no sequencer compromise, and no leaked keys are required. [5](#0-4) 

---

### Recommendation

1. **Apply CEI order in `withdrawCollateral`**: call `spotEngine.updateBalance` and run the health check *before* calling `handleWithdrawTransfer`.
2. **Add a `ReentrancyGuard`** (OpenZeppelin `nonReentrant`) to `withdrawCollateral` in `Clearinghouse` and to `submitFastWithdrawal` in `BaseWithdrawPool`.
3. **Add a `nonReentrant` modifier to `submitWithdrawal`** in `BaseWithdrawPool` to prevent cross-function reentrancy from the token transfer callback.

---

### Proof of Concept

```
Attacker setup:
  - AttackerContract deployed; its address is the subaccount wallet.
  - AttackerContract holds balance X in spotEngine (deposited legitimately).
  - Attacker obtains a valid verifier-signed WithdrawCollateral tx for amount Y, idx=B.

Step 1: AttackerContract submits slow-mode WithdrawCollateral for amount X.
Step 2: After 3-day delay, sequencer calls executeSlowModeTransaction.
        → processSlowModeTransactionImpl → clearinghouse.withdrawCollateral(sender, productId, X, address(0), idx=A)
Step 3: withdrawCollateral calls handleWithdrawTransfer(token, AttackerContract, X, A)
        → token.safeTransfer(withdrawPool, X)
        → submitWithdrawal(token, AttackerContract, X, A)
          → markedIdxs[A] = true
          → token.safeTransfer(AttackerContract, X)   ← RE-ENTRY POINT
Step 4: AttackerContract.receive() fires.
        → calls submitFastWithdrawal(idx=B, signedTx_Y, sigs)
          → markedIdxs[B] = true
          → token.safeTransfer(AttackerContract, Y)   ← Y tokens drained from pool
Step 5: Re-entry returns. withdrawCollateral resumes:
        → spotEngine.updateBalance(productId, sender, -X)   ← only X decremented
        → health check passes (balance = original - X, which may be ≥ 0)
Step 6: Sequencer later processes the fast-withdrawal tx (idx=B):
        → withdrawCollateral → handleWithdrawTransfer → submitWithdrawal
          → markedIdxs[B] already true → returns early (no token transfer)
        → spotEngine.updateBalance(productId, sender, -Y)   ← balance goes to (original - X - Y)
        
Result: AttackerContract received X + Y tokens.
        spotEngine balance decremented by X + Y (correct in total),
        but Y tokens were taken from the pool's liquidity (not from the Clearinghouse's
        collateral backing), leaving pool LPs short by Y with no recourse.
```

        bytes[] calldata signatures

        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
