Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of End-User, Allowing Any User to Bypass the Swap Allowlist via `MetricOmmSimpleRouter` â€” (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is the immediate caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router, not the originating user. If the pool admin allowlists the router to restore router-mediated access for legitimate users, the check passes for every caller regardless of individual allowlist status, completely defeating the access control.

## Finding Description
In `MetricOmmPool.swap()`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` forwards this value unchanged to every configured extension, including `SwapAllowlistExtension.beforeSwap`, which checks `allowedSwapper[msg.sender][sender]` â€” where `msg.sender` is the pool and `sender` is the immediate caller of `pool.swap()`: [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly with no mechanism to forward the original user's address â€” `extensionData` is passed through as-is from the caller's params, not automatically populated with the originating `msg.sender`: [3](#0-2) 

The pool therefore passes the router address as `sender` to the extension. The extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. This creates an irresolvable dilemma: if the router is not allowlisted, all allowlisted users are blocked from using the standard periphery path; if the router is allowlisted to restore that access, the check becomes `allowedSwapper[pool][router] == true` for every caller, and any unprivileged user bypasses the restriction by calling any of `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` on the router. The same pattern applies to multi-hop paths in `exactInput` and the recursive `exactOutput` flow. [4](#0-3) 

## Impact Explanation
A pool deployer using `SwapAllowlistExtension` to restrict trading to specific addresses (e.g., KYC-verified counterparties or protocol-controlled addresses) loses that protection entirely once the router is allowlisted. Any user can execute swaps against the restricted pool. This breaks the documented core invariant of the extension ("Gates `swap` by swapper address, per pool") and constitutes broken core pool functionality causing potential loss of LP principal â€” a direct allowed impact under the contest scope.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard documented periphery swap entry point. Any user who discovers the bypass needs only to call the router with a valid swap path â€” no privileged access, special tokens, or unusual setup is required. The bypass is reachable on every pool that has `SwapAllowlistExtension` configured and the router allowlisted, and is repeatable on every swap.

## Recommendation
The `sender` forwarded to extension hooks must represent the economic actor (the original EOA or contract initiating the trade), not the immediate caller of `pool.swap()`. Two complementary fixes:

1. **Router-level**: Have the router encode the original `msg.sender` into `extensionData` for each hop, so extensions can decode and verify the true initiator.
2. **Extension-level**: Update `SwapAllowlistExtension.beforeSwap` to decode and verify the original user from `extensionData` when the immediate `sender` is a known router, or require that the pool is only accessed directly when the allowlist is active.

The cleanest long-term fix is for the pool to accept an explicit `originator` field in the swap call that the router populates with `msg.sender`, and for the extension interface to expose this field rather than relying on the immediate `sender`.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, alice, true)
  - Pool admin calls setAllowedToSwap(pool, router, true)  // required for alice to use router

Attack:
  - bob (not allowlisted) calls:
      router.exactInputSingle({
          pool: restrictedPool,
          tokenIn: token0,
          ...
      })
  - Router calls pool.swap(recipient, ...) â€” pool's msg.sender = router
  - Pool calls _beforeSwap(sender=router, ...)
  - Extension checks: allowedSwapper[pool][router] == true  âś“
  - Bob's swap executes successfully despite not being allowlisted

Result:
  - Bob trades in a pool restricted to alice only
  - The allowlist invariant is completely bypassed
```

    _validatePath(params.tokens, params.pools, params.extensionDatas);

    uint256 last = params.pools.length - 1;
    int128 amount = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn);

    for (uint256 i = 0; i <= last; i++) {
      address pool = params.pools[i];
      bool zeroForOne = MetricOmmSwapPath.resolveZeroForOneBitmap(params.zeroForOneBitMap, i);

      _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, i == 0 ? msg.sender : address(this), params.tokens[i]);
      (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(pool)
        .swap(
          i == last ? params.recipient : address(this),
          zeroForOne,
          amount,
          MetricOmmSwapPath.openLimit(zeroForOne),
          "",
          params.extensionDatas[i]
        );

      int128 amountInActual = MetricOmmSwapResults.extractAmountIn(zeroForOne, amount0Delta, amount1Delta);

    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
