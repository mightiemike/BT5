### Title
V3 Transaction with Zero L2/L1DataGas Bounds Silently Downgrades to `L1Gas` Variant, Producing Wrong Transaction Hash and Gas Mode in `estimateFee`/`simulateTransactions` - (File: crates/apollo_rpc/src/v0_8/transaction.rs)

### Summary
The `From<ResourceBoundsMapping> for ValidResourceBounds` conversion in the RPC layer silently reclassifies a V3 `AllResources` transaction as a `L1Gas` (pre-0.13.3) transaction whenever `l2_gas` and `l1_data_gas` are both zero. This causes `estimateFee` and `simulateTransactions` to compute the transaction hash with only 2 resource bounds in the preimage instead of 3, and to execute under `GasVectorComputationMode::NoL2Gas` instead of `All`. The hash and fee estimate returned are authoritative-looking but diverge from what the actual on-chain transaction produces, because the gateway path never applies this downgrade.

### Finding Description

In `crates/apollo_rpc/src/v0_8/transaction.rs`, the conversion from the RPC wire type `ResourceBoundsMapping` to the internal `ValidResourceBounds` is:

```rust
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
            Self::L1Gas(value.l1_gas)
        } else {
            Self::AllResources(AllResourceBounds {
                l1_gas: value.l1_gas,
                l1_data_gas: value.l1_data_gas,
                l2_gas: value.l2_gas,
            })
        }
    }
}
``` [1](#0-0) 

When a V3 transaction is submitted to `estimateFee` or `simulateTransactions` with `l1_data_gas = 0` and `l2_gas = 0`, it is classified as `ValidResourceBounds::L1Gas`. This propagates into two downstream invariant breaks:

**1. Transaction hash preimage divergence.**
`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` branches on the variant:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 elements
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 elements
    }
});
``` [2](#0-1) 

The `L1Gas` path hashes `[tip, L1_GAS_concat, L2_GAS_concat]`; the `AllResources` path hashes `[tip, L1_GAS_concat, L2_GAS_concat, L1_DATA_GAS_concat]`. Even when all three bounds are numerically identical (L2=0, L1DataGas=0), the Poseidon hash over 2 elements differs from the hash over 3 elements. The transaction hash embedded in the simulation trace (visible to the contract via `get_execution_info`) is therefore wrong.

**2. Gas computation mode mismatch.**
`ValidResourceBounds::L1Gas` returns `GasVectorComputationMode::NoL2Gas`:

```rust
pub fn get_gas_vector_computation_mode(&self) -> GasVectorComputationMode {
    match self {
        Self::AllResources(_) => GasVectorComputationMode::All,
        Self::L1Gas(_) => GasVectorComputationMode::NoL2Gas,
    }
}
``` [3](#0-2) 

Under `NoL2Gas`, L2 (Sierra) gas is not tracked and the fee is computed from L1 gas only. Under `All`, all three resources are tracked. The fee estimate returned by `estimateFee` is therefore computed under a different accounting mode than the actual on-chain execution.

**The gateway path never applies this downgrade.**
Transactions submitted via the gateway use `RpcInvokeTransactionV3` which carries `resource_bounds: AllResourceBounds` directly:

```rust
pub struct InternalRpcInvokeTransactionV3 {
    pub resource_bounds: AllResourceBounds,
    ...
}
``` [4](#0-3) 

The `InternalRpcInvokeTransactionV3 → InvokeTransactionV3` conversion wraps it as `ValidResourceBounds::AllResources` unconditionally:

```rust
fn resource_bounds(&self) -> ValidResourceBounds {
    ValidResourceBounds::AllResources(self.resource_bounds)
}
``` [5](#0-4) 

So the same V3 transaction with zero L2/L1DataGas bounds produces hash H₁ (2-element preimage) in `estimateFee`/`simulateTransactions` and hash H₂ (3-element preimage) when actually sequenced. H₁ ≠ H₂.

### Impact Explanation

`simulateTransactions` returns a transaction trace in which the transaction hash seen by the contract (via `get_execution_info`) is H₁, while the actual on-chain hash will be H₂. Any contract logic that stores or checks `get_execution_info().transaction_hash` will behave differently in simulation than on-chain. `estimateFee` computes the fee under `NoL2Gas` mode while the on-chain execution uses `All` mode, producing a fee estimate that does not match actual charges. Both endpoints return authoritative-looking wrong values. This matches: **High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

### Likelihood Explanation

The trigger is unprivileged: any caller of `starknet_estimateFee` or `starknet_simulateTransactions` who submits a V3 transaction with `l2_gas.max_amount = 0`, `l2_gas.max_price_per_unit = 0`, `l1_data_gas.max_amount = 0`, `l1_data_gas.max_price_per_unit = 0` (a common pattern for users who only care about L1 gas costs, or for client-side proving transactions where all prices are zero) will silently receive wrong results. No special privileges, malformed bytes, or peer access are required.

### Recommendation

V3 transactions should always be treated as `AllResources` regardless of whether the bounds are zero. The conversion should not use zero-value heuristics to select the variant:

```rust
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        Self::AllResources(AllResourceBounds {
            l1_gas: value.l1_gas,
            l1_data_gas: value.l1_data_gas,
            l2_gas: value.l2_gas,
        })
    }
}
```

Alternatively, `BroadcastedDeclareV3Transaction` and the V3 `InvokeTransaction` RPC structs should use `AllResourceBounds` directly instead of `ResourceBoundsMapping`, matching the gateway's `RpcInvokeTransactionV3`.

### Proof of Concept

1. Construct a V3 invoke transaction with `resource_bounds = { l1_gas: { max_amount: 0x1000, max_price_per_unit: 0x1 }, l2_gas: { max_amount: 0x0, max_price_per_unit: 0x0 }, l1_data_gas: { max_amount: 0x0, max_price_per_unit: 0x0 } }`.

2. Submit it to `starknet_simulateTransactions`. The `ResourceBoundsMapping → ValidResourceBounds` conversion at `crates/apollo_rpc/src/v0_8/transaction.rs:188-199` produces `ValidResourceBounds::L1Gas(ResourceBounds { max_amount: 0x1000, max_price_per_unit: 0x1 })`.

3. `get_tip_resource_bounds_hash` at `crates/starknet_api/src/transaction_hash.rs:187-211` hashes `[tip, L1_GAS_concat, L2_GAS_concat]` — 2 elements. Call this hash H₁.

4. Submit the same transaction via the gateway. `InternalRpcInvokeTransactionV3::resource_bounds()` returns `ValidResourceBounds::AllResources(AllResourceBounds { l1_gas: 0x1000/0x1, l2_gas: 0/0, l1_data_gas: 0/0 })`. `get_tip_resource_bounds_hash` hashes `[tip, L1_GAS_concat, L2_GAS_concat, L1_DATA_GAS_concat]` — 3 elements. Call this hash H₂.

5. Observe H₁ ≠ H₂. The simulation trace reports H₁ as the transaction hash; the actual sequenced transaction has hash H₂. The simulation result is wrong.
