### Title
`paid_fee_on_l1` Hardcoded to `Fee(1)` in Consensus-to-Internal L1Handler Conversion, Bypassing Blockifier Fee Validation — (`crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

The `convert_consensus_l1_handler_to_internal_l1_handler` function always injects a hardcoded `Fee(1)` as `paid_fee_on_l1` when converting a consensus-layer `L1HandlerTransaction` into an executable `L1HandlerTransaction`. The `ConsensusTransaction::L1Handler` variant carries only `transaction::L1HandlerTransaction`, which has no `paid_fee_on_l1` field, so the real fee paid on L1 is structurally absent from the consensus wire message and can never be propagated to validators. Every validator therefore executes every L1 handler with `paid_fee_on_l1 = 1`, making the blockifier's fee guard permanently trivially satisfied and producing an incorrect fee value in the resulting executable object and in the cende blob.

---

### Finding Description

`executable_transaction::L1HandlerTransaction` carries a `paid_fee_on_l1: Fee` field and exposes a `create()` constructor that accepts it as an explicit parameter:

```rust
// crates/starknet_api/src/executable_transaction.rs:390-396
pub fn create(
    raw_tx: crate::transaction::L1HandlerTransaction,
    chain_id: &ChainId,
    paid_fee_on_l1: Fee,
) -> Result<L1HandlerTransaction, StarknetApiError> {
    let tx_hash = raw_tx.calculate_transaction_hash(chain_id, &raw_tx.version)?;
    Ok(Self { tx: raw_tx, tx_hash, paid_fee_on_l1 })
}
```

The consensus-layer type `ConsensusTransaction::L1Handler` wraps only `transaction::L1HandlerTransaction`, which has no `paid_fee_on_l1` field:

```rust
// crates/starknet_api/src/consensus_transaction.rs:9-12
pub enum ConsensusTransaction {
    RpcTransaction(RpcTransaction),
    L1Handler(transaction::L1HandlerTransaction),   // no paid_fee_on_l1
}
```

When a validator receives an L1 handler through consensus and calls `convert_consensus_tx_to_internal_consensus_tx`, the conversion path reaches:

```rust
// crates/apollo_transaction_converter/src/transaction_converter.rs:473-483
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
```

The real fee paid on L1 is never read; `Fee(1)` is always substituted. The blockifier then checks:

```rust
// crates/blockifier/src/transaction/l1_handler_transaction.rs:103-113
let paid_fee = self.paid_fee_on_l1;
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee { paid_fee, actual_fee: receipt.fee },
    )));
}
```

Because `Fee(1) != Fee(0)` is always true, this guard is permanently bypassed for every L1 handler transaction processed through the consensus validation path.

Additionally, `CentralL1HandlerTransaction` (the cende blob representation) is populated directly from the executable transaction's `paid_fee_on_l1`:

```rust
// crates/apollo_consensus_orchestrator/src/cende/central_objects.rs:383-393
impl From<L1HandlerTransaction> for CentralL1HandlerTransaction {
    fn from(tx: L1HandlerTransaction) -> CentralL1HandlerTransaction {
        CentralL1HandlerTransaction {
            ...
            paid_fee_on_l1: tx.paid_fee_on_l1,   // always Fee(1) from consensus path
        }
    }
}
```

So the cende blob emitted for every consensus-path L1 handler always carries `paid_fee_on_l1 = 1` regardless of the actual ETH/STRK value locked on L1.

---

### Impact Explanation

**Incorrect fee value in blockifier execution (Critical — Incorrect fee/resource accounting):** The `paid_fee_on_l1` field carried into `execute_raw` is always `Fee(1)`, not the actual fee. The fee guard that is supposed to reject zero-fee L1 handlers is permanently trivially satisfied. Any L1 handler transaction that reaches a validator through consensus — including one whose actual on-chain fee is zero due to a bug or manipulation in the L1 events provider — will be executed and committed rather than rejected, producing a wrong execution result for accepted input.

**Wrong value in cende blob (Critical — Wrong receipt/state value):** The `paid_fee_on_l1` field in `CentralL1HandlerTransaction` is always `1` for every L1 handler committed through the consensus path, diverging from the actual fee recorded on L1. Any downstream system (central sequencer, prover, or verifier) that reads this field from the blob receives an authoritative-looking but incorrect value.

---

### Likelihood Explanation

The trigger is automatic and unprivileged: every L1 handler transaction that passes through the consensus validation path (i.e., every block that includes an L1 handler on a validator node) exercises this code path. No special attacker capability is required. The practical severity of the fee-guard bypass is currently mitigated by the Starknet L1 contract enforcing non-zero fees before emitting the event, but the structural absence of `paid_fee_on_l1` from the consensus wire format means the sequencer has no independent enforcement and the wrong value is unconditionally written into the cende blob.

---

### Recommendation

Add `paid_fee_on_l1` to `ConsensusTransaction::L1Handler` (or introduce a wrapper type) so the real fee is carried through the consensus wire message:

```rust
pub enum ConsensusTransaction {
    RpcTransaction(RpcTransaction),
    L1Handler(transaction::L1HandlerTransaction, Fee),  // carry paid_fee_on_l1
}
```

Then propagate it in `convert_consensus_l1_handler_to_internal_l1_handler`:

```rust
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
    paid_fee_on_l1: Fee,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        paid_fee_on_l1,  // use real value
    )?)
}
```

---

### Proof of Concept

1. A proposer fetches an L1 handler from the L1 events provider; the provider records `paid_fee_on_l1 = X` (e.g., `Fee(1_000_000)`).
2. The proposer wraps it as `ConsensusTransaction::L1Handler(tx)` — `paid_fee_on_l1` is structurally dropped because `transaction::L1HandlerTransaction` has no such field.
3. The consensus message is broadcast; every validator receives only the raw `transaction::L1HandlerTransaction`.
4. Each validator calls `convert_consensus_tx_to_internal_consensus_tx` → `convert_consensus_l1_handler_to_internal_l1_handler` → `L1HandlerTransaction::create(tx, chain_id, Fee(1))`.
5. The blockifier executes with `paid_fee_on_l1 = Fee(1)` instead of `Fee(1_000_000)`. The fee guard passes trivially.
6. The cende blob records `paid_fee_on_l1 = 1` for this transaction, diverging from the actual on-chain value `X`.
7. If the L1 events provider ever delivers a transaction with `paid_fee_on_l1 = 0` (e.g., due to a bug or a future protocol change), the blockifier fee guard — which is the sequencer's only independent enforcement — is bypassed and the transaction is executed and committed.
