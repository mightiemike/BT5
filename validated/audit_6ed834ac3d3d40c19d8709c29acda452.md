### Title
Fork Retarget Uses Mainchain Boundary Block Instead of Fork Ancestor — (`contract/src/bitcoin.rs`, `contract/src/lib.rs`)

---

### Summary

`get_header_by_height` is implemented to read exclusively from `mainchain_height_to_header`. When `get_next_work_required` calls it to fetch the interval-start block for a difficulty retarget, it always returns the **mainchain** block at that height — even when the block being validated belongs to a fork that diverged before that height. This means the retarget timestamp used for a fork block is taken from the wrong chain, allowing a fork block whose `bits` field encodes an incorrect difficulty to pass the `check_pow` guard.

---

### Finding Description

**Root cause — `get_header_by_height` is mainchain-only:**

`BlocksGetter::get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

It reads from `mainchain_height_to_header`, which is only populated by `store_block_header` (mainchain submissions). Fork blocks go through `store_fork_header`, which writes only to `headers_pool`:

```rust
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
``` [2](#0-1) 

**Root cause — retarget reads the wrong block:**

At a retarget boundary, `get_next_work_required` computes:

```rust
let first_block_height =
    prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);

let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
``` [3](#0-2) 

`blocks_getter` here is `self` (a `BtcLightClient`), so `get_header_by_height(first_block_height)` returns the **mainchain** block at `first_block_height`, not the fork's actual ancestor at that height.

**Developer acknowledgment in `dogecoin.rs`:**

The same pattern in `dogecoin.rs` carries an explicit TODO:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [4](#0-3) 

This confirms the developers identified the issue but left it unresolved in both `bitcoin.rs` and `dogecoin.rs`/`litecoin.rs`.

**Entry point — `submit_blocks` with `UnrestrictedSubmitBlocks`:**

`submit_blocks` is gated by `#[trusted_relayer]` with `bypass_roles(Role::DAO, Role::UnrestrictedSubmitBlocks)`:

```rust
#[trusted_relayer(
    bypass_roles(Role::DAO, Role::UnrestrictedSubmitBlocks),
    ...
)]
``` [5](#0-4) 

Any account granted `UnrestrictedSubmitBlocks` (the standard role for relayers) can call `submit_blocks` and submit arbitrary fork blocks, including ones at retarget boundaries.

---

### Impact Explanation

A fork block at a retarget boundary passes `check_pow` with a `bits` value computed from the **mainchain** boundary block's timestamp rather than the fork's actual ancestor's timestamp. If the timestamps differ, the accepted `bits` is incorrect for the fork chain.

`chain_work` is accumulated directly from `bits`:

```rust
let (current_block_computed_chain_work, overflow) = prev_block_header
    .chain_work
    .overflowing_add(work_from_bits(header.bits));
``` [6](#0-5) 

If the wrong `bits` encodes higher difficulty than the fork deserves, each subsequent fork block accumulates inflated `chain_work`. Once `current_header.chain_work > total_main_chain_chainwork`, `reorg_chain` is triggered:

```rust
if current_header.chain_work > total_main_chain_chainwork {
    log!("Chain reorg");
    self.reorg_chain(current_header, last_main_chain_block_height);
}
``` [7](#0-6) 

This corrupts `mainchain_tip_blockhash` and the entire `mainchain_height_to_header` index.

---

### Likelihood Explanation

- The attacker only needs `UnrestrictedSubmitBlocks`, the standard role granted to relayers.
- The fork must diverge before the first retarget boundary (height 2016 for Bitcoin). The bug manifests at the **second** retarget (height 4032), where `first_block_height = 2016` — a height that exists on the mainchain but has a different block on the fork.
- The attacker controls the fork block timestamps, so they can engineer the timestamp difference to produce a favorable (higher-difficulty) `bits` value.
- No privileged key compromise, DAO access, or social engineering is required.

---

### Recommendation

`get_header_by_height` must walk the fork's ancestor chain rather than reading from the mainchain index. The correct approach is to traverse backwards from `prev_block_header` using `get_prev_header` until reaching `first_block_height`, mirroring how `get_prev_header` is already used for the `pow_allow_min_difficulty_blocks` path. Alternatively, maintain a separate height-to-header index per fork tip, or pass the fork's ancestor at `first_block_height` explicitly into `get_next_work_required`.

---

### Proof of Concept

```
State setup (Bitcoin mainnet, difficulty_adjustment_interval = 2016):
  - Init contract at height 0 (genesis)
  - Submit mainchain blocks 1..=4031 with timestamps T_main[h]
    (mainchain_height_to_header[2016] = mainchain block with time T_main_2016)

Fork construction:
  - Fork diverges at height 1000: submit fork blocks 1001..=4031
    with fork ancestor at height 2016 having time T_fork_2016 ≠ T_main_2016

Attack at height 4032 (retarget):
  - get_next_work_required is called with prev = fork block at height 4031
  - first_block_height = 4031 - 2015 = 2016
  - get_header_by_height(2016) returns mainchain block (time = T_main_2016)
    instead of fork ancestor (time = T_fork_2016)
  - expected_bits = calculate_next_work_required(..., T_main_2016)
    ≠ correct value = calculate_next_work_required(..., T_fork_2016)

  - Attacker submits fork block at height 4032 with bits = expected_bits
    (the wrong value) — check_pow passes
  - Block is stored in headers_pool with chain_work from wrong bits
  - If wrong bits > correct bits: inflated chain_work per block
  - Attacker extends fork until chain_work > mainchain → reorg triggered
  - mainchain_tip_blockhash is overwritten with attacker's fork tip
```
