### Title
Linked Signer Can Overwrite Its Own Authorization — Broken Access Control on `LinkSigner` Fast-Path (`allowLinkedSigner = true`) - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`. This means the currently registered linked signer can sign a new `LinkSigner` transaction and replace itself with any arbitrary address — including an attacker-controlled one. Only the subaccount owner should be able to mutate the linked signer slot. The slow-mode path correctly enforces owner-only access via `validateSender`, but the fast sequencer path does not.

---

### Finding Description

**Root cause — fast-path `LinkSigner` with `allowLinkedSigner = true`:**

In `processTransactionImpl`, the `LinkSigner` branch decodes a `SignedLinkSigner` and calls `validateSignedTx` with the fifth argument set to `true`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true          // ← allowLinkedSigner
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` resolves the permitted signer as either the subaccount owner address OR `getLinkedSigner(sender)`:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

This means the existing linked signer can produce a valid signature for a `LinkSigner` transaction that sets `signedTx.tx.signer` to any address, including an attacker-controlled one.

**Contrast — slow-mode `LinkSigner` is owner-only:**

The slow-mode path for the same transaction type uses `validateSender`, which requires `address(uint160(bytes20(txn.sender))) == msg.sender` — i.e., only the actual subaccount owner can change the linked signer via slow mode:

```solidity
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [3](#0-2) 

The two paths are inconsistent. The fast path grants the linked signer a privilege — mutating its own slot — that the slow path reserves exclusively for the owner.

**Broken invariant:**

`linkedSigners[subaccount]` should only be writable by `address(uint160(bytes20(subaccount)))` (the subaccount owner). The fast-path `LinkSigner` violates this by accepting a signature from the current value of `linkedSigners[subaccount]`. [4](#0-3) 

---

### Impact Explanation

A malicious or compromised linked signer can:

1. Sign a `LinkSigner` transaction pointing `signedTx.tx.signer` to an attacker-controlled address.
2. The sequencer includes this in a `submitTransactionsChecked` batch; `linkedSigners[victim_subaccount]` is overwritten.
3. The attacker's new linked signer now has full linked-signer authority: it can sign `WithdrawCollateral` (V1, funds go to owner address), `MintNlp`, `BurnNlp`, `TransferQuote` (same-owner subaccounts), `LiquidateSubaccount`, and — critically — order submissions via `MatchOrders` / `MatchOrdersWithAmount`. [5](#0-4) 

4. The attacker can submit orders at adversarial prices (e.g., sell at far-below-market, buy at far-above-market), draining the subaccount's collateral value through the matching engine.
5. The legitimate owner's only recovery path is the slow-mode `LinkSigner`, which carries a hardcoded 3-day delay (`SLOW_MODE_TX_DELAY`). During this window the attacker retains full linked-signer authority. [6](#0-5) 

Additionally, the attacker can set the linked signer to `address(0)` to permanently lock out the legitimate linked signer before the owner can react, or chain the replacement repeatedly to extend the attack window.

---

### Likelihood Explanation

Linked signers are a standard feature used by traders for automated bots and API keys. Any scenario where a linked signer key is leaked, phished, or operated by a semi-trusted third party (e.g., a trading service) creates a realistic trigger. The attacker needs only a single valid linked-signer signature — no privileged protocol access, no admin keys, no governance capture. The sequencer will include the transaction in the normal batch flow.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
``` [7](#0-6) 

This aligns the fast path with the slow-mode path, which already enforces owner-only access via `validateSender`. The linked signer role should be able to trade and operate on behalf of the owner, but it must never be able to mutate its own authorization slot.

---

### Proof of Concept

**Preconditions:**
- Victim subaccount `V` has set linked signer `LS_legit` (e.g., a trading bot key).
- Attacker controls `LS_legit` (compromised key, malicious service, etc.).

**Steps:**

1. Attacker constructs a `SignedLinkSigner` transaction:
   - `sender = V`
   - `signer = attacker_address` (attacker-controlled EOA)
   - `nonce = current nonce of V`
2. Attacker signs the EIP-712 digest using `LS_legit`.
3. Attacker submits the transaction to the sequencer's mempool.
4. Sequencer includes it in a `submitTransactionsChecked` batch.
5. `processTransactionImpl` reaches the `LinkSigner` branch, calls `validateSignedTx(..., true)`, which accepts the `LS_legit` signature.
6. `linkedSigners[V]` is overwritten with `attacker_address`.
7. Attacker now signs orders on behalf of `V` using `attacker_address`, submitting adversarial trades through the sequencer.
8. Victim attempts recovery via slow-mode `LinkSigner` — must wait 3 days before the reset takes effect. [1](#0-0) [8](#0-7)
