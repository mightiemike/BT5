Audit Report

## Title
Missing Minimum Timelock Validation in `initialize()` Allows Pool Admin to Bypass All Timelocked Stop-Loss Setters Atomically — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`OracleValueStopLossExtension.initialize()` validates `drawdownE6` and `decayPerSecondE8` but accepts any value — including zero — for `timelock`. With `timelock = 0`, the `_afterTimelock` helper returns `block.timestamp`, causing `_requireElapsed`'s strict-less-than guard to pass immediately. Every timelocked setter (drawdown, decay, watermarks) becomes executable in the same transaction as its proposal, allowing the pool admin to atomically disable the stop-loss and drain LP principal with no circuit-breaker.

## Finding Description

In `initialize()` at lines 56–62, `_validateDrawdown` and `_validateDecay` are called but there is no corresponding `_validateTimelock`:

```solidity
// lines 56-62
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// ← no timelock validation; timelock = 0 accepted silently
oracleStopLossConfig[pool] = PoolStopLossConfig({
    drawdownE6: drawdownE6, decayPerSecondE8: decayPerSecondE8, timelock: timelock, initialized: true
});
``` [1](#0-0) 

With `timelock = 0`, `_afterTimelock` returns exactly `block.timestamp`:

```solidity
// lines 297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
``` [2](#0-1) 

And `_requireElapsed` uses a strict `<` comparison:

```solidity
// lines 301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
``` [3](#0-2) 

`block.timestamp < block.timestamp` is always `false`, so the revert never fires. Every timelocked setter — `proposeOracleStopLossDrawdown`/`executeOracleStopLossDrawdown`, `proposeOracleStopLossDecay`/`executeOracleStopLossDecay`, and `proposeOracleStopLossHighWatermarks`/`executeOracleStopLossHighWatermarks` — routes through these two helpers and becomes instantly executable. [4](#0-3) 

The most destructive path: setting `drawdownE6 = 0` causes `afterSwap` to short-circuit at line 217, silently disabling all stop-loss checks for every subsequent swap:

```solidity
// lines 215-217
PoolStopLossConfig memory cfg = oracleStopLossConfig[pool_];
uint256 drawdown = cfg.drawdownE6;
if (drawdown == 0) return;
``` [5](#0-4) 

`proposeOracleStopLossTimelock` (lines 78–84) also lacks a minimum-value check on `newTimelock`, but exploiting that path requires waiting for the existing timelock to elapse first, making the initialization path the primary attack vector. [6](#0-5) 

## Impact Explanation

The contract's own NatSpec states: *"Drawdown and decay changes are timelocked so LPs can react."* With `timelock = 0` that guarantee is void. The pool admin — semi-trusted only within caps and timelocks — can:

1. Set `drawdownE6 = 0` atomically, disabling the stop-loss entirely, then execute a large swap that drains LP value with no circuit-breaker.
2. Set `decayPerSecondE8 = 1e8` (100%/s) instantly, collapsing all watermarks to zero within one second.
3. Zero out per-bin watermarks instantly, resetting the high-water reference so the next swap sees no breach regardless of actual value loss.

All three paths result in direct loss of LP principal. This is an admin-boundary break: the pool admin exceeds the timelock cap that bounds their trust level, which is an explicitly allowed impact under the contest rules.

## Likelihood Explanation

Pool creation is permissionless; any pool creator can supply `timelockSeconds = 0` in `extensionInitData`. No special privilege beyond pool creation is required. The missing check is a single omitted bounds validation analogous to the validated fields immediately above it. A deployer who misreads the field, uses a default zero value, or deliberately omits it produces the same outcome. Once deployed, the pool admin can exploit the condition at any time in a single transaction.

## Recommendation

Add a minimum timelock constant and enforce it in `initialize` alongside the existing validators, and also in `proposeOracleStopLossTimelock` before storing `newTimelock`:

```solidity
uint32 private constant MIN_TIMELOCK = 1 days; // 86400 seconds

function _validateTimelock(uint256 t) private pure {
    if (t < MIN_TIMELOCK) revert OracleStopLossTimelockTooShort(t);
}
```

Call `_validateTimelock(timelock)` at line 58 (after `_validateDecay`) in `initialize`, and call it inside `proposeOracleStopLossTimelock` before writing `sched.pendingTimelock`.

## Proof of Concept

```solidity
// 1. Pool created with timelock = 0
bytes memory initData = abi.encode(
    uint32(50_000),  // drawdownE6 = 5%
    uint32(58),      // decayPerSecondE8
    uint32(0)        // timelock = 0 — no validation, accepted silently
);
// factory calls ext.initialize(pool, initData)

// 2. Pool admin atomically disables stop-loss in one tx:
ext.proposeOracleStopLossDrawdown(pool, 0);
// executeAfter = block.timestamp + 0 = block.timestamp

ext.executeOracleStopLossDrawdown(pool);
// _requireElapsed: block.timestamp < block.timestamp == false → no revert
// oracleStopLossConfig[pool].drawdownE6 = 0

// 3. afterSwap now returns immediately at "if (drawdown == 0) return;"
// All subsequent swaps bypass the stop-loss; LP value can be drained freely.
```

Foundry test: deploy the extension with a mock factory, call `initialize` with `timelock = 0`, then call `proposeOracleStopLossDrawdown(pool, 0)` and `executeOracleStopLossDrawdown(pool)` in the same block. Assert `oracleStopLossConfig[pool].drawdownE6 == 0` and that `afterSwap` returns without reverting on a value-draining swap.
