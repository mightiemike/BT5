### Title
Unclaimed Weekly Airdrop Tokens Are Permanently Locked With No Admin Recovery Path - (File: `core/contracts/Airdrop.sol`)

### Summary
The `Airdrop.sol` contract holds ERC-20 tokens pre-funded for weekly Merkle-based distributions. Once a week's Merkle root is registered and tokens are deposited, any portion that users fail to claim is permanently locked in the contract. There is no owner or admin function to recover unclaimed token balances from past weeks, mirroring the M-03 dust-accumulation finding from the PoolTogether report.

### Finding Description
The `Airdrop` contract distributes tokens week-by-week via Merkle proofs. The owner registers a root per week via `registerMerkleRoot()`, and users call `claim()` to receive their allocation. The `_verifyProof()` function enforces a one-time claim per `(week, sender)` pair by requiring `claimed[week][sender] == 0` and then setting it to `totalAmount`. [1](#0-0) [2](#0-1) 

The only `onlyOwner` function in the contract is `registerMerkleRoot()`. There is no `recoverUnclaimed()`, `sweep()`, or equivalent function that would allow the owner to retrieve tokens allocated to a week that have not been claimed. [3](#0-2) 

The contract holds the cumulative token balance for all registered weeks. Any user who:
- Is sanctioned at claim time (`isSanctioned` check at line 53–56),
- Has lost their private key,
- Finds gas costs exceed the value of a small allocation, or
- Simply never claims,

leaves their week's allocation permanently stranded in the contract. Because the `claimed` mapping only records what was claimed (not what remains), and there is no accounting of per-week total deposits vs. total claimed, the contract has no on-chain mechanism to compute or recover the residual. [4](#0-3) 

### Impact Explanation
ERC-20 tokens deposited into `Airdrop.sol` for weekly distributions that go unclaimed are permanently locked. Over many weeks, this compounds: each new `registerMerkleRoot()` call adds a new week's allocation, and any unclaimed portion from prior weeks remains irrecoverable. The protocol loses the ability to redeploy or redistribute those tokens.

### Likelihood Explanation
Sanctioned addresses are explicitly blocked from claiming (line 53–56), meaning their allocated tokens are guaranteed to be locked. Additionally, users with small allocations may rationally skip claiming if gas costs exceed the reward value — a realistic scenario on any chain where gas is non-trivial. Both paths are reachable without any privileged action. [5](#0-4) 

### Recommendation
Add an owner-callable recovery function that allows sweeping unclaimed tokens after a configurable expiry period (e.g., N weeks after a given week's root was registered). For example:

```solidity
function recoverUnclaimed(address recipient) external onlyOwner {
    // Only callable after all claimable weeks have expired
    uint256 balance = IERC20(token).balanceOf(address(this));
    SafeERC20.safeTransfer(IERC20(token), recipient, balance);
}
```

A time-gated variant (e.g., only callable after `pastWeeks` has advanced by a threshold) would preserve user claim windows while preventing permanent lock-up.

### Proof of Concept
1. Owner calls `registerMerkleRoot(1, root1)` and funds the contract with 1,000,000 tokens for week 1.
2. 900,000 tokens are claimed by users over the following weeks.
3. 100,000 tokens remain unclaimed (e.g., sanctioned addresses, lost keys, dust allocations).
4. Owner calls `registerMerkleRoot(2, root2)` and funds another 1,000,000 tokens for week 2.
5. The 100,000 unclaimed tokens from week 1 are now permanently mixed into the contract balance with no recovery path.
6. Calling any function on the contract with the owner key cannot retrieve those tokens — `registerMerkleRoot` is the only `onlyOwner` entry point, and it does not transfer tokens out. [6](#0-5)
