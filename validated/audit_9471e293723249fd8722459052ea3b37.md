### Title
`get_split_parent_shard_uids()` Stamps Parent Shard UID with the *New* Layout's Frozen Version, Breaking `ShardUId` Uniqueness at the V1→V2 Layout Boundary — (`File: core/primitives/src/shard_layout/mod.rs`)

---

### Summary

`ShardUId` packs `(ShardVersion, shard_id)` and is the sole mechanism for uniquely identifying shards across layout changes in the database. The module-level comment guarantees that the version increments with every layout change so that `ShardUId` values are globally unique. Starting from `ShardLayoutV2`, the version is frozen at `3` and never increments again. `get_split_parent_shard_uids()` constructs the parent shard's `ShardUId` using `self.version()` — the *new* layout's version — rather than the *previous* layout's version. For V2→V3 transitions this is harmless because both layouts share version `3`. For the first dynamic-resharding step that promotes a `ShardLayoutV1` (version `< 3`) to a `ShardLayoutV2` (version `= 3`), the returned parent `ShardUId` carries the wrong version, so every downstream DB lookup and cold-store mapping that uses it silently targets a key that does not exist, causing state-access failures and incorrect `StateShardUIdMapping` entries for child shards.

---

### Finding Description

The `ShardUId` struct packs two fields:

```
pub struct ShardUId {
    pub version: ShardVersion,   // 4 bytes LE
    pub shard_id: u32,           // 4 bytes LE
}
```

The module-level documentation states the invariant explicitly:

> "The version number should increment as when sharding changes. This guarantees the version number is unique across different shard layouts, which in turn guarantees `ShardUId` is different across shards from different shard layouts." [1](#0-0) 

`ShardLayoutV2::new()` hard-codes `VERSION = 3` and explicitly notes the invariant is abandoned:

```rust
// In the v2 layout the version is not updated with every shard layout.
const VERSION: ShardVersion = 3;
``` [2](#0-1) 

`ShardLayoutV3` also hard-codes the same constant:

```rust
const VERSION: ShardVersion = 3;
``` [3](#0-2) 

`get_split_parent_shard_uids()` constructs the parent's `ShardUId` using `self.version()` — the version of the **new** layout, not the **previous** layout that actually owns the parent shard:

```rust
pub fn get_split_parent_shard_uids(&self) -> BTreeSet<ShardUId> {
    let parent_shard_ids = self.get_split_parent_shard_ids();
    parent_shard_ids
        .into_iter()
        .map(|shard_id| ShardUId::new(self.version(), shard_id))  // ← new layout's version
        .collect()
}
``` [4](#0-3) 

The same mistake is replicated in `ReshardingEventType::from_shard_layout()`, where the comment explicitly acknowledges the error but incorrectly dismisses it:

```rust
// Please note the use of the next shard layout version.
// Technically speaking the current shard layout version
// should be used for the parent. However since
// ShardLayoutV2 the version is frozen so it is ok.
let parent_shard = ShardUId::new(next_shard_layout.version(), parent_id);
``` [5](#0-4) 

The "it is ok" reasoning only holds for V2→V3 and V3→V3 transitions (both sides version `3`). It **fails** for the first dynamic resharding that promotes a `ShardLayoutV1` (mainnet simple-nightshade-v2 uses version `2`) to a `ShardLayoutV2` (version `3`). In that case:

- Parent shard state is stored in the DB under `ShardUId { version: 2, shard_id: P }`.
- `get_split_parent_shard_uids()` returns `ShardUId { version: 3, shard_id: P }`.
- Every consumer of that UID looks up a key that does not exist.

`ancestor_uids()` in `ShardLayoutV3` has the same defect — it hard-codes `VERSION` for all ancestor UIDs regardless of which layout generation they belong to:

```rust
ancestor_ids.iter().map(|id| ShardUId::new(VERSION, *id)).collect()
``` [6](#0-5) 

---

### Impact Explanation

**`get_resharding_parent_shard_uid()`** (called during block production and validation) returns the wrong parent `ShardUId`:

```rust
let split_parent_shard_uids = next_layout.get_split_parent_shard_uids();
Ok(split_parent_shard_uids.into_iter().next())
``` [7](#0-6) 

**`update_state_shard_uid_mapping()`** (cold-store resharding boundary) maps child shards to the wrong parent UID:

```rust
let split_parents = shard_layout.get_split_parent_shard_uids();
for parent_shard_uid in split_parents {
    let mapped_shard_uid = get_shard_uid_mapping(&cold_store, parent_shard_uid);
    for child_shard_uid in children {
        update.trie_store_update().set_shard_uid_mapping(child_shard_uid, mapped_shard_uid);
    }
}
``` [8](#0-7) 

`get_shard_uid_mapping` will not find any entry for `ShardUId { version: 3, shard_id: P }` (the parent was stored under version `2`), so it falls back to returning the UID itself. Child shards are then mapped to a non-existent parent key. All subsequent trie state reads for those child shards via `TrieStoreAdapter` will follow the mapping to a key with no data, producing `MissingTrieValue` storage errors. This breaks chunk application and state witness validation for every shard involved in the first dynamic resharding event, constituting a network-halting condition for nodes that track those shards.

---

### Likelihood Explanation

`ProtocolFeature::DynamicResharding` is assigned protocol version `85`, which equals `STABLE_PROTOCOL_VERSION`. The feature is therefore active on mainnet. The bug fires the first time the runtime's trie-memory threshold triggers an actual shard split while the current layout is still a `ShardLayoutV1` (version `< 3`). No privileged action is required; the split is proposed automatically by chunk producers and confirmed by the epoch manager. Any node tracking the split shard will encounter the wrong parent UID. [9](#0-8) [10](#0-9) 

---

### Recommendation

`get_split_parent_shard_uids()` must use the **previous** layout's version when constructing the parent `ShardUId`. The caller already has access to both layouts (e.g., `get_resharding_parent_shard_uid()` holds `current_layout` and `next_layout`). The fix is to pass the previous layout into `get_split_parent_shard_uids()` or to add a sibling method `get_split_parent_shard_uids_from_prev_layout(prev_layout: &ShardLayout)` that calls `ShardUId::new(prev_layout.version(), shard_id)`. The same correction is needed in `ReshardingEventType::from_shard_layout()` and `ancestor_uids()`.

---

### Proof of Concept

Concrete divergent values for the first mainnet dynamic resharding (V1 simple-nightshade-v2 → V2):

| Field | Correct value | Actual value produced |
|---|---|---|
| Parent layout version | `2` (ShardLayoutV1 simple-nightshade-v2) | `3` (ShardLayoutV2 frozen constant) |
| Parent `ShardUId` bytes | `[02 00 00 00 | shard_id_le]` | `[03 00 00 00 | shard_id_le]` |
| DB key prefix for parent state | `02 00 00 00 …` | `03 00 00 00 …` (no matching rows) |
| `StateShardUIdMapping` target for children | `ShardUId { version: 2, shard_id: P }` | `ShardUId { version: 3, shard_id: P }` |

The mismatch is structurally identical to the Chainlink analog: just as `roundId - 1` crosses an aggregator-version boundary and points to a non-existent round, `ShardUId::new(self.version(), parent_id)` crosses a layout-version boundary and points to a non-existent DB key. [11](#0-10) [12](#0-11)
