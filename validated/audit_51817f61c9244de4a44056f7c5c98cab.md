### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downcasts `AllResources` to `L1Gas` When `l2_gas` and `l1_data_gas` Are Zero, Producing a Different Transaction Hash Preimage ŌĆö (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf converter for `ValidResourceBounds` uses a value-based heuristic to reconstruct the variant on deserialization. When an `AllResources` transaction carries zero `l2_gas` and zero `l1_data_gas`, the deserialized variant becomes `L1Gas`. Because `get_tip_resource_bounds_hash` includes `L1_DATA_GAS` in the hash preimage only for `AllResources`, the hash computed after a protobuf round-trip differs from the hash computed at submission time. Any code path that recomputes the transaction hash from a protobuf-deserialized `InvokeTransactionV3` (e.g., P2P sync hash validation) will produce a mismatched hash, causing valid transactions to be rejected.

---

### Finding Description

**Serialization side** ŌĆö `From<ValidResourceBounds> for protobuf::ResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs`:

```rust
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),      // zero
    l1_data_gas: Some(ResourceBounds::default().into()), // zero
},
ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) =>
    protobuf::ResourceBounds {
        l1_gas: Some(l1_gas.into()),
        l2_gas: Some(l2_gas.into()),
        l1_data_gas: Some(l1_data_gas.into()),
    },
```

When `AllResources` carries `l2_gas = 0` and `l1_data_gas = 0`, both arms emit **identical protobuf bytes**. There is no type tag in the wire format. [1](#0-0) 

**Deserialization side** ŌĆö `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`:

```rust
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
// ...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ŌåÉ wrong variant for AllResources txs
} else {
    ValidResourceBounds::AllResources(...)
})
```

The condition `l1_data_gas.is_zero() && l2_gas.is_zero()` is true for both a legacy `L1Gas` transaction **and** a modern `AllResources` transaction whose `l2_gas` and `l1_data_gas` happen to be zero. The original variant is irrecoverably lost. [2](#0-1) 

**Hash preimage divergence** ŌĆö `get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs`:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // L1_DATA_GAS absent
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // L1_DATA_GAS present
    }
});
```

`L1Gas` hashes `[tip, L1_GAS, L2_GAS]` (2 resource elements).  
`AllResources` hashes `[tip, L1_GAS, L2_GAS, L1_DATA_GAS]` (3 resource elements).  
Even when `l1_data_gas = 0`, the extra element changes the Poseidon output. [3](#0-2) 

**Submission path always uses `AllResources`** ŌĆö `From<RpcInvokeTransactionV3> for InvokeTransactionV3`:

```rust
resource_bounds: ValidResourceBounds::AllResources(tx.resource_bounds),
```

Every V3 invoke submitted via the RPC gateway is stored as `AllResources`, regardless of whether `l2_gas` and `l1_data_gas` are zero. [4](#0-3) 

The round-trip is therefore:

```
AllResources({l1_gas: X, l2_gas: 0, l1_data_gas: 0})
  ŌåÆ serialize ŌåÆ protobuf bytes (identical to L1Gas(X))
  ŌåÆ deserialize ŌåÆ L1Gas(X)
  ŌåÆ hash() ŌåÆ poseidon(tip, L1_GAS_X, L2_GAS_0)   ŌåÉ missing L1_DATA_GAS element
```

whereas the original hash was:

```
AllResources({l1_gas: X, l2_gas: 0, l1_data_gas: 0})
  ŌåÆ hash() ŌåÆ poseidon(tip, L1_GAS_X, L2_GAS_0, L1_DATA_GAS_0)
```

---

### Impact Explanation

Any node that receives an `InvokeTransactionV3` via P2P protobuf and then calls `validate_transaction_hash` will recompute the hash using the deserialized `L1Gas` variant, obtaining a value that does not match the hash computed at submission time (which used `AllResources`). This causes the transaction ŌĆö and any block containing it ŌĆö to fail hash validation on syncing nodes, while the proposer accepted it correctly. This matches the allowed impact: **"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."** [5](#0-4) 

---

### Likelihood Explanation

Any user can submit a V3 invoke transaction with `AllResourceBounds` where `l2_gas.max_amount = 0`, `l2_gas.max_price_per_unit = 0`, `l1_data_gas.max_amount = 0`, `l1_data_gas.max_price_per_unit = 0`. The stateless gateway validator imposes no lower bound on these fields. [6](#0-5) 

The trigger is unprivileged, requires no special access, and is reachable through the normal RPC submission path followed by P2P propagation.

---

### Recommendation

The protobuf `ResourceBounds` message must carry an explicit type discriminator (e.g., a boolean `is_all_resources` field, or a separate oneof), so that `AllResources` with zero `l2_gas`/`l1_data_gas` is not silently collapsed to `L1Gas` on deserialization. Until the wire format is extended, the deserializer should default to `AllResources` whenever `l1_data_gas` is explicitly present (even if zero), reserving `L1Gas` only for messages where `l1_data_gas` is absent (`None`):

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas_opt = value.l1_data_gas;
let l1_data_gas = l1_data_gas_opt.unwrap_or_default().try_into()?;
Ok(if l1_data_gas_opt.is_none() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)   // only for legacy 0.13.2 messages
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [7](#0-6) 

---

### Proof of Concept

1. Construct a V3 invoke transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`.
2. Compute `hash_A = get_tip_resource_bounds_hash(AllResources(...), tip)` ŌĆö this includes `L1_DATA_GAS` in the Poseidon chain.
3. Serialize the transaction's `ValidResourceBounds::AllResources(...)` to `protobuf::ResourceBounds` via `From<ValidResourceBounds>`.
4. Deserialize back via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` ŌĆö the result is `ValidResourceBounds::L1Gas(X)`.
5. Compute `hash_B = get_tip_resource_bounds_hash(L1Gas(X), tip)` ŌĆö this omits `L1_DATA_GAS`.
6. Assert `hash_A != hash_B` ŌĆö the two Poseidon hashes differ because the input vectors have different lengths.
7. Submit the transaction to the gateway (which stores `hash_A`). When a syncing node receives the transaction via P2P, deserializes it, and calls `validate_transaction_hash` with `expected_hash = hash_A`, the recomputed hash is `hash_B`, causing validation to return `false` and the transaction to be rejected. [1](#0-0) [8](#0-7)

// An implementation of the SNIP: https://github.com/EvyatarO/SNIPs/blob/snip-8/SNIPS/snip-8.md
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
