### Title
Hardcoded `paid_fee_on_l1 = Fee(1)` in Consensus-to-Internal L1Handler Conversion Bypasses Fee Sufficiency Check and Corrupts Committed Receipt Data - (File: `crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary
When a validator receives an `L1HandlerTransaction` from the proposer via consensus, the conversion from `ConsensusTransaction::L1Handler` to `InternalConsensusTransaction::L1Handler` hardcodes `paid_fee_on_l1 = Fee(1)` instead of the actual fee paid on L1. This is the direct sequencer analog of the Scroll `tx.origin`-as-default-refund-address bug: a wrong default is silently substituted at a public-to-internal conversion boundary, causing the wrong value to flow into both the fee-sufficiency gate and the committed receipt/blob data.

### Finding Description

`ConsensusTransaction::L1Handler` wraps `transaction::L1HandlerTransaction`, which carries no `paid_fee_on_l1` field:

```rust
// crates/starknet_api/src/consensus_transaction.rs
pub enum ConsensusTransaction {
    RpcTransaction(RpcTransaction),
    L1Handler(transaction::L1HandlerTransaction),   // no paid_fee_on_l1
}
```

When a validator converts this to an executable transaction, `convert_consensus_l1_handler_to_internal_l1_handler` fills the missing field with a hardcoded sentinel:

```rust
// crates/apollo_transaction_converter/src/transaction_converter.rs  lines 473-483
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
```

The resulting `executable_transaction::L1HandlerTransaction` carries `paid_fee_on_l1 = Fee(1)` regardless of what was actually paid on L1. This value then flows into two downstream consumers:

**1. Fee-sufficiency gate in the blockifier** (`crates/blockifier/src/transaction/l1_handler_transaction.rs` lines 103–113):

```rust
let paid_fee = self.paid_fee_on_l1;
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee { paid_fee, actual_fee: receipt.fee },
    )));
}
```

Because `Fee(1) ≠ Fee(0)`, the gate always passes on every validator node, independent of the actual L1 fee.

**2. Committed receipt serialization** (`crates/apollo_consensus_orchestrator/src/cende/central_objects.rs` lines 383–393):

```rust
impl From<L1HandlerTransaction> for CentralL1HandlerTransaction {
    fn from(tx: L1HandlerTransaction) -> CentralL1HandlerTransaction {
        CentralL1HandlerTransaction {
            hash_value: tx.tx_hash,
            ...
            paid_fee_on_l1: tx.paid_fee_on_l1,   // always Fee(1) on validators
        }
    }
}
```

Every validator node writes `paid_fee_on_l1 = 1` into the central blob for every L1 handler transaction, regardless of the true L1 fee.

### Impact Explanation

**Fee-gate bypass**: The blockifier's only fee-sufficiency check for L1 handler transactions is `paid_fee == Fee(0)`. With the hardcoded `Fee(1)`, this check is permanently satisfied on all validator nodes. If a proposer were to include an L1 handler transaction whose actual on-chain fee was zero (e.g., due to a bug or future change in the L1 contract), every validator would execute it without rejection, producing wrong state and wrong receipts. This matches the "Incorrect fee… with economic impact" impact category.

**Wrong committed receipt data**: All validator nodes commit `paid_fee_on_l1 = 1` into the `CentralL1HandlerTransaction` blob, diverging from the proposer's true value. Any downstream system (prover, explorer, L1 settlement contract) that reads `paid_fee_on_l1` from the committed blob receives a fabricated value. This matches the "Wrong state, receipt… from blockifier/syscall/execution logic" impact category.

### Likelihood Explanation

This affects every validator node for every L1 handler transaction included in any proposed block. The conversion path is unconditional — there is no code path that recovers the real `paid_fee_on_l1` on the validator side. The trigger requires only that an L1 message be sent on L1 and included in a block proposal, which is the normal operating condition.

### Recommendation

Include `paid_fee_on_l1` in `ConsensusTransaction::L1Handler` (i.e., use `executable_transaction::L1HandlerTransaction` or a dedicated consensus wrapper that carries the fee field) so that the actual fee is propagated through the consensus wire format and is available to validators at conversion time. Until then, the hardcoded `Fee(1)` sentinel should at minimum be documented as producing an authoritative-looking but fabricated value in committed receipts.

### Proof of Concept

1. L1 user sends a message to Starknet on L1, paying `paid_fee_on_l1 = 500_000` wei.
2. The proposer's L1 provider reads the event and creates `executable_transaction::L1HandlerTransaction { paid_fee_on_l1: Fee(500_000), … }`.
3. The proposer broadcasts `ConsensusTransaction::L1Handler(transaction::L1HandlerTransaction { … })` — **`paid_fee_on_l1` is absent** from this type.
4. Each validator calls `convert_consensus_l1_handler_to_internal_l1_handler`, which calls `L1HandlerTransaction::create(tx, &self.chain_id, Fee(1))`.
5. The blockifier on each validator evaluates `if paid_fee == Fee(0)` → `Fee(1) ≠ Fee(0)` → passes.
6. `CentralL1HandlerTransaction { paid_fee_on_l1: Fee(1), … }` is written into the committed blob by every validator, instead of `Fee(500_000)`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
