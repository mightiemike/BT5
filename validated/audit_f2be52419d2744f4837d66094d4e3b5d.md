### Title
`op_bls_verify` Discards Accumulated Cost on Signature Failure, Enabling Undercharged Execution DoS - (File: src/bls_ops.rs)

---

### Summary

`op_bls_verify` in `src/bls_ops.rs` accumulates cost proportional to the number of `(pk, msg)` pairs and message byte lengths, then calls `aggregate_verify`. When `aggregate_verify` returns `false` (invalid signature), the function returns `Err(EvalErr::BLSVerifyFailed(input))` — carrying **no cost**. The interpreter propagates this error without returning any cost to the caller. An attacker can submit attacker-controlled CLVM bytes containing a `bls_verify` call with many large messages and a deliberately invalid signature, causing full nodes to perform expensive BLS pairing work at zero cost, enabling a chain-wide DoS vector.

---

### Finding Description

In `op_bls_verify` (`src/bls_ops.rs`, lines 364–401):

```rust
let mut cost = BLS_PAIRING_BASE_COST;          // 3,000,000
// ...
while !nilp(a, args) {
    // parse pk and msg ...
    cost += BLS_PAIRING_COST_PER_ARG;           // +1,200,000 per pair
    cost += msg.as_ref().len() as Cost * BLS_MAP_TO_G2_COST_PER_BYTE;  // +4 per byte
    cost += DST_G2.len() as Cost * BLS_MAP_TO_G2_COST_PER_DST_BYTE;   // +172 fixed
    check_cost(cost, max_cost)?;
    items.push((pk, msg));
}

if !aggregate_verify(&signature, items) {
    Err(EvalErr::BLSVerifyFailed(input))?   // ← cost is discarded here
} else {
    Ok(Reduction(cost, a.nil()))            // ← cost only returned on success
}
``` [1](#0-0) 

The cost accumulation loop correctly charges per pair and per message byte, and `check_cost` enforces the `max_cost` ceiling during argument parsing. However, once all arguments are parsed and `aggregate_verify` is invoked, if it returns `false`, the function returns `Err(EvalErr::BLSVerifyFailed(input))` — an error variant that carries **no cost field**.

In `apply_op` (`src/run_program.rs`, lines 441–447), the operator result is propagated with `?`:

```rust
let r = self.dialect.op(
    self.allocator, operator, operand_list, max_cost, current_extensions,
)?;
self.push(r.1)?;
Ok(r.0)
``` [2](#0-1) 

The `?` propagates the `Err` upward through `run_program`, which also returns `Err` to the caller. The caller receives zero cost information. The `EvalErr` enum has no cost-carrying variant for `BLSVerifyFailed`: [3](#0-2) 

The same structural defect exists in `op_bls_pairing_identity` (lines 353–357), which also discards cost on `aggregate_pairing` failure, though without per-byte message cost it is less severe. [4](#0-3) 

---

### Impact Explanation

The broken invariant is: **cost returned must be proportional to work done**. When `aggregate_verify` returns `false`, the cost returned is `0`, but the work done is `O(N × pairing_cost + N × msg_len × hash_cost)`. BLS pairing is among the most expensive operations in the system — `BLS_PAIRING_COST_PER_ARG = 1,200,000` per pair, calibrated against real pairing time (~1.2 ms per pair at benchmark). With N pairs and large messages, the discarded cost can approach `max_cost` (up to 11,000,000,000 for the Chia blockchain).

The caller — the Chia mempool or any downstream consumer of `run_program` — receives `Err` with cost = 0. Failed transactions are not included in blocks and pay no fee. The attacker can therefore submit an unbounded stream of expensive-to-validate CLVM programs at zero on-chain cost, causing full nodes to perform real BLS pairing work for free. [5](#0-4) 

---

### Likelihood Explanation

The attack requires only the ability to submit CLVM programs to the mempool — the standard public interface of any Chia full node. The attacker-controlled bytes are the CLVM program itself: a `bls_verify` opcode with N `(pk, msg)` pairs (valid G1 points, arbitrary-length messages) and a single invalid G2 signature. No special privileges, keys, or insider access are required. The `bls_verify` operator is a standard, always-available opcode in the default `ChiaDialect`. [6](#0-5) 

---

### Recommendation

Return the accumulated cost even on `BLSVerifyFailed`, so that callers can account for the work done. The fix mirrors how `check_cost` already enforces the ceiling during argument parsing — the cost should also be surfaced on failure:

```rust
if !aggregate_verify(&signature, items) {
    // Option A: return cost alongside the error (requires EvalErr to carry cost)
    // Option B: consume all remaining budget
    Err(EvalErr::BLSVerifyFailed(input))?
}
```

A concrete approach is to change `EvalErr::BLSVerifyFailed` to carry the accumulated cost, and have `apply_op` extract and add it to the running total before propagating the error. Alternatively, the operator can return `Ok(Reduction(cost, <error_sentinel>))` and let the interpreter raise the error after charging cost — consistent with how the EVM recommendation suggests consuming all gas on restoration failure.

The same fix should be applied to `op_bls_pairing_identity`.

---

### Proof of Concept

Attacker constructs the following CLVM program (pseudocode):

```
(bls_verify
  <invalid_G2_signature_96_bytes>   ; deliberately wrong signature
  <valid_G1_pubkey_1>               ; 48 bytes
  <large_message_1_N_bytes>         ; e.g. 1 MB
  <valid_G1_pubkey_2>
  <large_message_2_N_bytes>
  ...                               ; repeat K times
)
```

1. Attacker submits this serialized CLVM program to the Chia mempool.
2. The mempool calls `run_program` → `apply_op` → `op_bls_verify`.
3. `op_bls_verify` parses K pairs, accumulates cost up to `K × (1,200,000 + N × 4 + 172)`, calls `aggregate_verify`.
4. `aggregate_verify` performs K full BLS pairings (~1.2 ms each) and returns `false`.
5. `op_bls_verify` returns `Err(BLSVerifyFailed)` — cost = 0.
6. The mempool rejects the transaction. No fee is charged.
7. Attacker repeats at high frequency. Each iteration costs the attacker nothing but forces the full node to perform K × 1.2 ms of BLS pairing work.

With K = 9 pairs and N = 0 (empty messages), the discarded cost is `3,000,000 + 9 × 1,200,000 = 13,800,000` — well within a single transaction's budget — while the node performs ~10.8 ms of pairing work per rejected submission. [7](#0-6)

pub fn op_bls_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let mut cost = BLS_PAIRING_BASE_COST;
    check_cost(cost, max_cost)?;

    let mut args = input;

    // the first argument is the signature
    let signature = a.g2(first(a, args)?)?;

    // followed by a variable number of (G1, msg)-pairs (as a flat list)
    args = rest(a, args)?;

    let mut items = Vec::<(PublicKey, Atom)>::new();
    while !nilp(a, args) {
        let pk = a.g1(first(a, args)?)?;
        args = rest(a, args)?;
        let msg = atom(a, first(a, args)?, "bls_verify message")?;
        args = rest(a, args)?;

        cost += BLS_PAIRING_COST_PER_ARG;
        cost += msg.as_ref().len() as Cost * BLS_MAP_TO_G2_COST_PER_BYTE;
        cost += DST_G2.len() as Cost * BLS_MAP_TO_G2_COST_PER_DST_BYTE;
        check_cost(cost, max_cost)?;

        items.push((pk, msg));
    }

    if !aggregate_verify(&signature, items) {
        Err(EvalErr::BLSVerifyFailed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
}
