### Title
Proof-Verification Forgery via Publicly Callable Deprecated `verify_transaction_inclusion` — (File: `contract/src/lib.rs`)

### Summary
The deprecated `verify_transaction_inclusion` function remains a live, publicly callable NEAR entry point. It lacks the coinbase Merkle proof depth-anchor check introduced in `verify_transaction_inclusion_v2`. Any unprivileged NEAR caller can supply a fake `tx_id` that equals an internal Merkle tree node and receive a `true` verification result, deceiving any recipient contract that consumes it.

### Finding Description
`verify_transaction_inclusion` (v1) is still exposed as a `#[pause]`-gated public method callable by any NEAR account. Its only Merkle integrity check is:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0) 

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` simply walks the proof path from the supplied hash to the root, with no constraint on proof length or on whether the starting hash is a leaf versus an internal node:

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;
    for proof_hash in merkle_proof {
        ...
    }
    current_hash
}
``` [2](#0-1) 

An internal Merkle node `N` at tree depth `d` has a valid proof of length `d` that leads to the block's `merkle_root`. Supplying `tx_id = N` with that proof causes the function to return `true` even though `N` is not a real transaction.

`verify_transaction_inclusion_v2` closes this gap by requiring `coinbase_merkle_proof.len() == merkle_proof.len()` and verifying the coinbase proof against the stored `merkle_root`, anchoring the tree depth so a shorter proof for an internal node cannot match: [3](#0-2) 

But v1 enforces none of this and remains callable.

### Impact Explanation
A recipient NEAR contract that calls `verify_transaction_inclusion` (v1) to gate a cross-chain action (e.g., token unlock, bridge settlement) can be made to execute that action for a transaction that never existed. The attacker needs only a real mainchain block and knowledge of its Merkle tree structure — both are public Bitcoin data. The corrupted state is the verification result returned to the caller: `true` for a forged `tx_id`.

### Likelihood Explanation
The attack requires no privileged role, no leaked key, and no social engineering. Bitcoin Merkle trees are fully public. Any block with more than one transaction has internal nodes that can serve as forged `tx_id` values. The only barrier is constructing the correct `tx_index` and proof path, which is straightforward from the block's transaction list. Likelihood is **medium-high** for any deployment where downstream contracts consume v1 results.

### Recommendation
Remove `verify_transaction_inclusion` (v1) from the public ABI entirely, or gate it with an access-control role so it cannot be called by unprivileged accounts. All callers should be migrated to `verify_transaction_inclusion_v2`. If backward compatibility is required, add a runtime `require!(false, "use verify_transaction_inclusion_v2")` to make the deprecated path unconditionally revert.

### Proof of Concept

1. Identify any mainchain block `B` with ≥ 2 transactions. Its Merkle tree has at least one internal node `N = hash(tx0, tx1)`.
2. Compute the Merkle proof `P` of length `d` from `N` to `B.merkle_root` (standard Merkle path, skipping the leaf level).
3. Call on-chain:
   ```
   verify_transaction_inclusion({
     tx_id: N,
     tx_block_blockhash: B.hash,
     tx_index: <position of N at depth d>,
     merkle_proof: P,
     confirmations: 1,
   })
   ```
4. `compute_root_from_merkle_proof(N, position, P)` reconstructs `B.merkle_root` correctly because `N` is a genuine internal node with a valid path.
5. The function returns `true`. `N` is not a real transaction. [4](#0-3) [2](#0-1)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
