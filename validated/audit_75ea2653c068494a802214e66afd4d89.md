### Title
Incomplete v8 Migration Silently Skips Undelegated Vesting-Owned Positions, Enabling Permanent Vesting Lock Bypass — (File: `x/tieredrewards/migrations/v2/migrate.go`)

---

### Summary

The v8 migration's `exitVestedAccountsPositions` function calls `ForceFullExitWithDelegation` for each vesting-owned position. `ForceFullExitWithDelegation` silently returns `nil` when the position is not delegated, leaving undelegated vesting-owned positions alive. For a `CommitDelegationToTier`-origin position, the vesting account's `DelegatedVesting` (DV) was set stale-high when the owner originally delegated and was never corrected when the delegation moved into the tier. After the upgrade, the owner calls `MsgWithdrawFromTier`, which sends tokens as a plain bank transfer without calling `alignVestingDelegationTracking`. The stale-high DV makes `SpendableCoins = Balance`, bypassing the permanent vesting lock.

---

### Finding Description

**Step 1 — Migration skips undelegated positions.**

`exitVestedAccountsPositions` collects all vesting-owned position IDs and calls `ForceFullExitWithDelegation` on each: [1](#0-0) 

`ForceFullExitWithDelegation` guards on `IsDelegated()` and returns `nil` (no error, no state change) for undelegated positions: [2](#0-1) 

The undelegated position is never deleted and the vesting account's DV/DF is never corrected.

**Step 2 — DV is stale-high for CommitDelegation-origin positions.**

When a vesting account originally delegates via `x/staking`, the bank-side `TrackDelegation` hook sets `DelegatedVesting = commitAmount`. When `CommitDelegationToTier` moves that delegation to the position's delegator account via `transferDelegationToPosition`, DV is not decremented — it remains stale-high. This is the known pre-migration invariant violation that `ForceFullExitWithDelegation` + `alignVestingDelegationTracking` is designed to fix for *delegated* positions: [3](#0-2) 

For an undelegated position, neither function is called.

**Step 3 — `MsgWithdrawFromTier` does not correct DV/DF.**

`validateAddToPosition` and the broader `msg_validate.go` show that only `validateNewPosition` (called by `LockTier` and `CommitDelegationToTier`) contains the vesting account check: [4](#0-3) 

`validateAddToPosition` — and by the same pattern `WithdrawFromTier`'s validator — has no vesting check and no call to `alignVestingDelegationTracking`: [5](#0-4) 

`deletePosition`, called at the end of `WithdrawFromTier`, also does not call `alignVestingDelegationTracking`: [6](#0-5) 

**Step 4 — Vesting lock arithmetic breaks.**

For a `PermanentLockedAccount` with `OriginalVesting = X` and `DelegatedVesting = X` (stale-high):

```
LockedCoins   = OriginalVesting = X
SpendableCoins = Balance − max(LockedCoins − DV − DF, 0)
               = Balance − max(X − X − 0, 0)
               = Balance
```

All tokens in the account become spendable, fully bypassing the permanent vesting lock.

**Step 5 — The existing integration test does not cover this path.**

`assert_v8_no_vesting_owned_positions` queries all surviving positions and asserts none are vesting-owned. However, `setup_pre_v8_upgrade` only seeds *delegated* vesting positions; it never creates an undelegated one. The undelegated case is untested: [7](#0-6) 

---

### Impact Explanation

The corrupted value is `DelegatedVesting` in the vesting account's auth store entry. After `MsgWithdrawFromTier` completes, `DelegatedVesting` remains equal to `OriginalVesting` while the actual on-chain delegation is zero. The Cosmos SDK vesting spendability formula then yields `SpendableCoins = Balance`, allowing the owner of a `PermanentLockedAccount` to spend tokens that are contractually and permanently locked. This is a complete bypass of the vesting lock for the committed amount.

func (k Keeper) validateAddToPosition(ctx context.Context, pos types.PositionState, owner string) error {
	if !pos.IsOwner(owner) {
		return types.ErrNotPositionOwner
	}

	if !pos.IsDelegated() {
		return types.ErrPositionNotDelegated
	}

	if pos.HasTriggeredExit() {
		return types.ErrPositionTriggeredExit
	}

	tier, err := k.getTier(ctx, pos.TierId)
	if err != nil {
		return err
	}

	if tier.IsCloseOnly() {
		return types.ErrTierIsCloseOnly
	}

	return nil
}

func (k Keeper) deletePosition(ctx context.Context, pos types.Position, update *ValidatorTransition) error {

		return errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid owner address")

	delAddr, err := sdk.AccAddressFromBech32(pos.DelegatorAddress)
	if err != nil {
		return errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid delegator address")

	if err != nil {
		return err
	}

	}

	if err := k.removeBaseRewardsRouting(ctx, delAddr, owner); err != nil {
		return err
	}

	// Defensive
	if err := k.deletePositionRedelegationMappings(ctx, pos.Id); err != nil {
		return err
	}

	if err := k.Positions.Remove(ctx, pos.Id); err != nil {
		return err
	}
	if err := k.PositionsByOwner.Remove(ctx, collections.Join(owner, pos.Id)); err != nil {
		return err

		return err
