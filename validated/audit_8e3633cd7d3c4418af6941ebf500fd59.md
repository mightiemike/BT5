### Title
`op_bls_verify` Silently Succeeds With Zero (pubkey, message) Pairs — (`File: src/bls_ops.rs`)

### Summary

`op_bls_verify` (opcode 59) requires a G2 signature as its first argument, then expects a flat list of `(G1_pubkey, message)` pairs. When called with only a signature and no pairs, the inner loop never executes, `items` remains empty, `aggregate_verify` is called with an empty set (which returns `true` vacuously), and the operator returns `Ok(Reduction(cost, nil))` — success — without verifying anything. There is no guard that rejects a zero-pair invocation.

### Finding Description

In `src/bls_ops.rs`, `op_bls_verify` is structured as follows:

1. Extract the G2 signature from the first argument.
2. Iterate over the remaining arguments as `(G1_pubkey, message)` pairs, accumulating them into `items`.
3. Call `aggregate_verify(&signature, items)` and fail only if it returns `false`. [1](#0-0) 

The loop at lines 382–394 is guarded by `while !nilp(a, args)`. If the caller provides only the signature with no subsequent arguments, `args` is already nil after `rest(a, args)?` on line 379, the loop body never runs, and `items` is an empty `Vec`. [2](#0-1) 

`aggregate_verify` from `chia_bls` follows the BLS aggregate-verify specification: an empty set of `(pubkey, message)` pairs is vacuously valid (the identity element in Gt is the expected pairing result), so it returns `true`. Consequently the operator returns `Ok(Reduction(cost, a.nil()))` — the CLVM success value — without having verified any signature at all. [3](#0-2) 

There is no post-loop check equivalent to `if items.is_empty() { return Err(...) }`. This is the exact structural analog to the Liquity `redeemCollateral` bug: a loop that finds nothing to process, followed by unconditional success.

### Impact Explanation

Any CLVM coin puzzle that uses `bls_verify` as a spend guard can be bypassed by an attacker who supplies a CLVM program that invokes `(bls_verify <valid_g2_point>)` with no `(pubkey, message)` pairs. The operator returns nil (success), so the guard is satisfied without any cryptographic check. Because CLVM execution is deterministic and consensus-critical, all full nodes would accept such a spend. The corrupted result is the `Reduction(cost, nil)` returned by `op_bls_verify` — a false "verified" signal — which propagates up through `run_program` as a successful execution.

### Likelihood Explanation

The attacker-controlled entry path is direct: craft CLVM bytes for opcode 59 (`bls_verify`) with a single G2 argument and no further arguments. This is reachable from any context that executes attacker-supplied CLVM — mempool submission, coin spend validation, or the Python `run_program` API exposed by the wheel. No special privileges or social engineering are required. The identity G2 point (`0xc0` followed by 95 zero bytes) is a valid compressed infinity point and is accepted by `a.g2()`.

### Recommendation

Add an explicit check after the loop that rejects a zero-pair invocation:

```rust
if items.is_empty() {
    return Err(EvalErr::InvalidOpArg(
        input,
        "bls_verify requires at least one (pubkey, message) pair".to_string(),
    ));
}
```

This mirrors the fix recommended for the Liquity bug: a descriptive error that fails the operation when the precondition for meaningful work is not met.

pub fn op_bls_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let mut cost = BLS_PAIRING_BASE_COST;
    check_cost(cost, max_cost)?;

    let mut args = input;

    // the first argument is the signature
    let signature = a.g2(first(a, args)?)?;

    // followed by a variable number of (G1, msg)-pairs (as a flat list)
    args = rest(a, args)?;

    let mut items = Vec::<(PublicKey, Atom)>::new();
    while !nilp(a, args) {
        let pk = a.g1(first(a, args)?)?;
        args = rest(a, args)?;
        let msg = atom(a, first(a, args)?, "bls_verify message")?;
        args = rest(a, args)?;

        cost += BLS_PAIRING_COST_PER_ARG;
        cost += msg.as_ref().len() as Cost * BLS_MAP_TO_G2_COST_PER_BYTE;
        cost += DST_G2.len() as Cost * BLS_MAP_TO_G2_COST_PER_DST_BYTE;
        check_cost(cost, max_cost)?;

        items.push((pk, msg));
    }

    if !aggregate_verify(&signature, items) {
        Err(EvalErr::BLSVerifyFailed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
}
