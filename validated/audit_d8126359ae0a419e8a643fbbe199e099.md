### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the Coinbase Merkle Proof Forgery Guard — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase merkle proof. However, the original `verify_transaction_inclusion` (v1) remains a fully public, externally callable NEAR contract method. Any unprivileged caller can invoke v1 directly, bypassing the coinbase proof check entirely and obtaining a `true` result for a forged transaction inclusion proof.

---

### Finding Description

The contract exposes two public SPV verification endpoints:

**v1** — `verify_transaction_inclusion` — marked `#[deprecated]` but still fully public and callable: [1](#0-0) 

Its own docstring explicitly warns: [2](#0-1) 

It performs no coinbase proof validation. It only checks that the supplied `merkle_proof` is non-empty and that `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == header.merkle_root`: [3](#0-2) 

**v2** — `verify_transaction_inclusion_v2` — the secure replacement, which first validates a coinbase merkle proof before delegating to v1: [4](#0-3) 

The coinbase guard is present only in v2: [5](#0-4) 

The `#[deprecated]` Rust attribute is a **compile-time lint only**. It generates warnings for Rust callers in the same codebase but has **zero effect on NEAR cross-contract or direct RPC calls**. Any external NEAR account can call `verify_transaction_inclusion` at runtime without restriction.

The `ProofArgsV2 → ProofArgs` conversion used by v2 simply drops `coinbase_tx_id` and `coinbase_merkle_proof`: [6](#0-5) 

This means the entire coinbase guard is skipped when v1 is called directly.

---

### Impact Explanation

The 64-byte transaction Merkle proof forgery attack (documented at https://www.bitmex.com/blog/64-Byte-Transactions) allows an attacker to craft a `tx_id` that is actually the hash of an internal Merkle tree node (which is 64 bytes, the same size as a concatenated hash pair). By supplying this internal node hash as `tx_id` along with a valid sibling path, `compute_root_from_merkle_proof` will compute the correct `merkle_root` and the function returns `true` — falsely asserting that a non-existent transaction was included in a Bitcoin block.

Any downstream NEAR contract that calls `verify_transaction_inclusion` and acts on a `true` result (e.g., releasing funds, minting tokens, updating state) can be deceived into processing a fraudulent Bitcoin transaction proof. This directly corrupts the core security guarantee of the light client.

---

### Likelihood Explanation

The attack requires no privileged role, no leaked keys, and no social engineering. The attacker only needs to:
1. Identify a real Bitcoin block stored in the contract's `headers_pool`.
2. Construct a valid internal Merkle node hash that maps to the block's `merkle_root` via a chosen sibling path.
3. Call `verify_transaction_inclusion` directly via NEAR RPC with the crafted `ProofArgs`.

This is a realistic, low-barrier attack for anyone familiar with Bitcoin Merkle tree structure.

---

### Recommendation

Remove the `#[pause]` public exposure of `verify_transaction_inclusion` by making it `pub(crate)` or `private`, so it can only be called internally by `verify_transaction_inclusion_v2`. Alternatively, add the coinbase proof check directly inside `verify_transaction_inclusion` and remove the v2 delegation pattern. The `#[deprecated]` attribute alone provides no runtime protection on NEAR.

---

### Proof of Concept

1. Deploy the contract with a known Bitcoin block at height `H` whose `merkle_root` is `R`.
2. Find two leaf hashes `A` and `B` in that block such that `double_sha256(A || B) == R` (i.e., a two-transaction block, or use any internal node).
3. Craft `ProofArgs { tx_id: A, tx_block_blockhash: <hash of block H>, tx_index: 0, merkle_proof: [B], confirmations: 1 }`.
4. Call `verify_transaction_inclusion` via NEAR RPC.
5. The function returns `true` even though `A` may not be a real transaction — it is an internal Merkle node.
6. Calling `verify_transaction_inclusion_v2` with the same `tx_id` but without a valid `coinbase_merkle_proof` would panic, demonstrating the bypass.

The root cause is identical in structure to H-02: one entry point (`safeTransferFrom` / `verify_transaction_inclusion_v2`) carries the security check; the sibling entry point (`safeBatchTransferFrom` / `verify_transaction_inclusion`) does not, and remains reachable by any unprivileged caller.     #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
