### Title
Wrong-Channel MsgTransfer Bypasses Burn, Escrows NFT, and Mints Unbacked Voucher on Destination Chain — (`x/nft-transfer/keeper/packet.go`, `x/nft-transfer/types/trace.go`)

---

### Summary

`createOutgoingPacket` resolves the `fullClassPath` from the IBC hash and calls `IsAwayFromOrigin(sourcePort, sourceChannel, fullClassPath)` to decide whether to escrow or burn. There is no guard that the `sourceChannel` supplied by the user matches the **outermost** prefix of `fullClassPath`. An attacker holding a multi-hop IBC NFT can deliberately supply an inner-prefix channel, causing `IsAwayFromOrigin` to return `true` (away from origin) when the correct answer is `false` (back to origin). The NFT is escrowed instead of burned, and the destination chain mints a new unbacked voucher.

---

### Finding Description

**`IsAwayFromOrigin` logic** (correct in isolation):

```
IsAwayFromOrigin(sourcePort, sourceChannel, fullClassPath) bool
  prefixClassID = sourcePort + "/" + sourceChannel + "/"
  return !strings.HasPrefix(fullClassPath, prefixClassID)
``` [1](#0-0) 

For a 2-hop NFT with `fullClassPath = "nft/c1/nft/c2/myclass"`, the correct return channel is `c1` (outermost prefix). If the attacker instead supplies `sourceChannel = c2`:

- `prefixClassID = "nft/c2/"`
- `"nft/c1/nft/c2/myclass"` does **not** start with `"nft/c2/"` → returns `true` (wrong: away from origin)

**`createOutgoingPacket` has no channel-vs-prefix guard:**

```go
if strings.HasPrefix(classID, nfttypes.IBCPrefix) {
    fullClassPath, err = k.ClassPathFromHash(ctx, classID)
    ...
}
isAwayFromOrigin := types.IsAwayFromOrigin(sourcePort, sourceChannel, fullClassPath)
// no check: does sourceChannel == outermost prefix of fullClassPath?
if isAwayFromOrigin {
    // ESCROW — wrong for a return transfer
    k.nftKeeper.TransferOwner(ctx, classID, tokenID, sender, escrowAddress)
} else {
    // BURN — correct for a return transfer
    k.nftKeeper.BurnNFTUnverified(ctx, classID, tokenID, sender)
}
packetData := types.NewNonFungibleTokenPacketData(fullClassPath, ...)
``` [2](#0-1) 

`SendTransfer` only validates that the channel exists on-chain; it does not validate that the channel matches the class path prefix: [3](#0-2) 

**On the destination chain**, `processReceivedPacket` receives `classId = "nft/c1/nft/c2/myclass"` from source `(nft, c2)`, evaluates `IsAwayFromOrigin("nft", "c2", "nft/c1/nft/c2/myclass") = true`, and mints a new voucher with `classTrace = "nft/dest_c/nft/c1/nft/c2/myclass"` — an unbacked voucher with no corresponding escrow on the origin chain: [4](#0-3) 

---

### Impact Explanation

| State change | Effect |
|---|---|
| NFT escrowed on source chain (not burned) | NFT is locked in escrow; the escrow is not backed by any corresponding escrow on the origin chain |
| New voucher minted on destination chain | Unauthorized voucher minting — unbacked NFT supply inflation |
| Class trace on destination chain gains an extra hop | Permanently corrupted trace path; the voucher can never be correctly redeemed back to the original NFT |

The attacker retains the ability to trigger a timeout/ack-error refund on the source chain (since the NFT is escrowed, not burned), recovering their original NFT while the destination chain retains the unbacked voucher — a net duplication of the NFT.

---

### Likelihood Explanation

**Preconditions:**
1. Attacker holds an IBC-prefixed NFT with a multi-hop class path (normal result of any A→B→C transfer).
2. The attacker's chain has a valid IBC channel whose ID matches an inner segment of the class path.

IBC channel IDs are sequential integers (`channel-0`, `channel-1`, …). Any chain with more than one IBC connection will have channels whose IDs overlap with inner-prefix segments of multi-hop class paths. This is a realistic and common network topology. No privileged access, governance, or operator compromise is required — only a standard `MsgTransfer` signed by the NFT owner.

---

### Recommendation

In `createOutgoingPacket` (or `SendTransfer`), after resolving `fullClassPath` for an IBC-prefixed classID, validate that the `sourceChannel` matches the outermost prefix of `fullClassPath`. Concretely:

```go
if strings.HasPrefix(classID, nfttypes.IBCPrefix) {
    fullClassPath, err = k.ClassPathFromHash(ctx, classID)
    ...
    // Guard: for a return transfer the channel must match the outermost prefix
    expectedPrefix := types.GetClassPrefix(sourcePort, sourceChannel)
    if strings.HasPrefix(fullClassPath, expectedPrefix) {
        // sink: burn — OK
    } else {
        // source: escrow — but verify no inner prefix matches sourceChannel
        // to prevent deliberate mis-routing
    }
}
```

More precisely: reject the transfer if `sourceChannel` appears as an inner (non-outermost) prefix of `fullClassPath`, since that indicates the user is deliberately mis-routing a return transfer.

---

### Proof of Concept

**Setup:** A→B→C multi-hop transfer.

1. Chain A: native NFT `myclass`. Send A→B via `(nft, channel-1)` on A / `(nft, channel-2)` on B.
   - B stores `ClassTrace{Path:"nft/channel-2", BaseClassId:"myclass"}`, classID = `ibc/HASH1`.

2. B sends to C via `(nft, channel-3)` on B / `(nft, channel-4)` on C.
   - `IsAwayFromOrigin("nft","channel-3","nft/channel-2/myclass")` = `true` → B escrows. Correct.
   - C stores `ClassTrace{Path:"nft/channel-4/nft/channel-2", BaseClassId:"myclass"}`, classID = `ibc/HASH2`.
   - `fullClassPath` on C = `"nft/channel-4/nft/channel-2/myclass"`.

3. **Exploit:** Attacker on C submits `MsgTransfer` with `sourceChannel = channel-2` (C has `channel-2` connecting to some chain D, not B).
   - `fullClassPath = "nft/channel-4/nft/channel-2/myclass"`
   - `IsAwayFromOrigin("nft","channel-2","nft/channel-4/nft/channel-2/myclass")`:
     - prefix = `"nft/channel-2/"`, path starts with `"nft/channel-4/"` → **returns `true`** (wrong)
   - `ibc/HASH2` is **escrowed** on C (should have been burned).
   - Packet `classId = "nft/channel-4/nft/channel-2/myclass"` sent to D.

4. Chain D receives packet from `(nft, channel-2)`:
   - `IsAwayFromOrigin("nft","channel-2","nft/channel-4/nft/channel-2/myclass")` = `true`
   - D mints voucher with `classTrace = "nft/channel-D/nft/channel-4/nft/channel-2/myclass"` — **unbacked**.

5. Attacker triggers timeout on C → `refundPacketToken` unescrows `ibc/HASH2` back to attacker.
   - Attacker recovers original NFT on C; D retains the unbacked voucher. [5](#0-4) [1](#0-0)

func (k Keeper) refundPacketToken(ctx sdk.Context, packet channeltypes.Packet, data types.NonFungibleTokenPacketData) error {
	sender, err := sdk.AccAddressFromBech32(data.Sender)
	if err != nil {
		return err
	}

	classTrace := types.ParseClassTrace(data.ClassId)
	voucherClassID := classTrace.IBCClassID()

	isAwayFromOrigin := types.IsAwayFromOrigin(packet.GetSourcePort(),
		packet.GetSourceChannel(), data.ClassId)

	escrowAddress := types.GetEscrowAddress(packet.GetSourcePort(), packet.GetSourceChannel())

	if isAwayFromOrigin {
		// unescrow tokens back to the sender
		for _, tokenID := range data.TokenIds {
			if err := k.nftKeeper.TransferOwner(ctx, voucherClassID, tokenID, escrowAddress, sender); err != nil {
				return err
			}
		}

		// we are sink chain, mint voucher back to sender

			if err := k.nftKeeper.MintNFT(ctx, voucherClassID, tokenID, "", data.TokenUris[i], "", escrowAddress, sender); err != nil {
				return err
			}
		}
	}

	return nil
