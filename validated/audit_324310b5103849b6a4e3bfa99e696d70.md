### Title
SwapAllowlistExtension Allowlist Bypassed via Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

The `SwapAllowlistExtension.beforeSwap` hook checks the `sender` argument forwarded by the pool, which is the direct caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `sender` resolves to the router's address, not the end user's address. If the pool admin allowlists the router (required for any user to reach the pool through the router), every non-allowlisted user can bypass the per-user restriction by calling the router instead of the pool directly.

### Finding Description

`SwapAllowlistExtension.beforeSwap` gates swaps with:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension's caller) and `sender` is the value forwarded by the pool from its own `msg.sender`. The pool sets `sender = msg.sender` of `pool.swap()`:

```solidity
_beforeSwap(
    msg.sender,   // ← direct caller of pool.swap(), not the end user
    recipient, ...
);
``` [2](#0-1) 

When a user calls `router.exactInputSingle()`, the router calls `pool.swap()` with `msg.sender = router`. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. [3](#0-2) 

The pool admin faces an inescapable dilemma:

| Configuration | Effect |
|---|---|
| Router **not** allowlisted | Even allowlisted users cannot use the router |
| Router **allowlisted** | Every non-allowlisted user can bypass the allowlist via the router |

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users from doing the same. The invariant stated in the protocol's own audit targets — *"A curated pool must enforce the same allowlist policy regardless of which supported public entrypoint reaches it"* — is structurally broken. [4](#0-3) 

### Impact Explanation

A non-allowlisted user can trade against LP positions in a pool that was configured to restrict counterparties. LPs in curated pools (e.g., institutional or KYC-gated pools) suffer unauthorized trades against their positions. Because the pool is oracle-driven and the attacker can choose timing and direction, this is equivalent to the external report's "unlimited slippage" scenario: the guard that was supposed to prevent unwanted execution is silently bypassed, and the LP bears the resulting adverse selection.

### Likelihood Explanation

The bypass requires only that the router be allowlisted for the target pool — a configuration any pool admin would apply when they want their allowlisted users to access the router. The attacker needs no special privilege: any EOA can call `MetricOmmSimpleRouter.exactInputSingle()`. The router is a public, factory-validated contract. [5](#0-4) 

### Recommendation

The extension must check the ultimate end user, not the intermediate router. Two approaches:

1. **Pass end-user identity through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a coordinated change to the router and extension.
2. **Check both `sender` and a decoded inner caller**: The extension accepts a structured `extensionData` payload that carries the original user address when the router is the direct caller, and falls back to `sender` for direct pool calls.

### Proof of Concept

```
1. Pool admin deploys pool with SwapAllowlistExtension.
2. Pool admin calls setAllowedToSwap(pool, alice, true)   // Alice is allowed
3. Pool admin calls setAllowedToSwap(pool, router, true)  // router allowlisted so Alice can use it
4. Charlie (not allowlisted) calls:
       router.exactInputSingle({pool: pool, ...})
5. Router calls pool.swap(...) with msg.sender = router
6. Pool calls extension.beforeSwap(sender=router, ...)
7. Extension checks allowedSwapper[pool][router] == true  → passes
8. Charlie's swap executes against LP positions.
```

Charlie never appears in the allowlist. The check at line 37 of `SwapAllowlistExtension.sol` evaluates the router's address, which is allowlisted, so the guard silently passes for any caller who routes through the router. [4](#0-3) [5](#0-4) [2](#0-1) [6](#0-5)

  function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
      );
    int128 out = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    amountOut = MetricOmmSwapInputs.int128ToUint128(out);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
