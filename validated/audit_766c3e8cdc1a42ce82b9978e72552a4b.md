### Title
Unprotected `initialize()` in `BaseProxyManager` Enables Front-Running to Seize Full Protocol Upgrade Control — (`core/contracts/BaseProxyManager.sol`)

---

### Summary

`BaseProxyManager.initialize()` carries no caller restriction. Any unprivileged actor who observes the proxy deployment transaction in the mempool can call `initialize()` before the deployer, claiming ownership of the `ProxyManager` and the `submitter` role. Because `ProxyManager` is the sole authority that can upgrade every core Nado proxy, this single front-run yields complete, permanent control over the protocol's upgrade path.

---

### Finding Description

`BaseProxyManager.initialize()` is declared `external initializer` with no access guard:

```solidity
function initialize() external initializer {
    __Ownable_init();
    submitter = msg.sender;
    proxyManagerHelper = new ProxyManagerHelper();
}
``` [1](#0-0) 

The constructor correctly calls `_disableInitializers()` to protect the *implementation* contract: [2](#0-1) 

However, `_disableInitializers()` only locks the implementation's own storage. The proxy's initialization state is independent and starts at zero. Between the block that deploys the proxy and the block that calls `initialize()` on it, any caller can race in.

The team already recognised this risk for `ContractOwner` and added an explicit deployer check there:

```solidity
function initialize(...) external initializer {
    require(_deployer == msg.sender, "expected deployed to initialize");
    ...
}
``` [3](#0-2) 

No equivalent guard exists in `BaseProxyManager.initialize()`.

Once `initialize()` is front-run, the attacker becomes:
- **`owner`** — the role checked by `onlyOwner` on `migrateAll`, `forceMigrateSelf`, `registerRegularProxy`, `bootstrapEndpointTx`, `updateSubmitter`, and `refreshProxyManagerHelper`.
- **`submitter`** — the role checked by `onlySubmitter` on `submitImpl` and `refreshCodeHash`.
- **controller of `proxyManagerHelper`** — the helper that calls `upgradeClearinghouseLiq` and `upgradeEndpointTx` on behalf of the ProxyManager. [4](#0-3) 

With owner access the attacker can immediately call `migrateAll` to point any registered proxy — `Clearinghouse`, `Endpoint`, `SpotEngine`, `PerpEngine`, `OffchainExchange` — at an arbitrary malicious implementation: [5](#0-4) 

They can also call `forceMigrateSelf` to replace the ProxyManager implementation itself, making the takeover permanent and irrecoverable: [6](#0-5) 

---

### Impact Explanation

An attacker who wins the initialization race owns the entire upgrade system. They can:

1. Upgrade `Clearinghouse` to a contract that drains all deposited collateral.
2. Upgrade `Endpoint` to a contract that accepts arbitrary sequencer-signed transactions, bypassing all signature verification.
3. Upgrade `SpotEngine` / `PerpEngine` to manipulate balances for any subaccount.
4. Replace `ProxyManager` itself to prevent recovery.

The corrupted state delta is: `owner` and `submitter` of `ProxyManager` set to the attacker's address, giving them unilateral control over every upgradeable contract in the protocol. All user funds held in `Clearinghouse` and routed through `Endpoint` are at risk.

---

### Likelihood Explanation

The attack requires only a single mempool observation and a competing transaction. No special privilege, leaked key, or social engineering is needed. The window exists whenever proxy deployment and `initialize()` are submitted as separate transactions — a common pattern in multi-step deployment scripts. The fact that `ContractOwner` has an explicit deployer guard while `BaseProxyManager` does not suggests the gap was overlooked rather than intentional.

---

### Recommendation

Add a deployer-binding check identical to the pattern already used in `ContractOwner`:

```solidity
address private immutable _expectedDeployer;

constructor(address expectedDeployer) {
    _expectedDeployer = expectedDeployer;
    _disableInitializers();
}

function initialize() external initializer {
    require(msg.sender == _expectedDeployer, "only deployer");
    __Ownable_init();
    submitter = msg.sender;
    proxyManagerHelper = new ProxyManagerHelper();
}
```

Alternatively, ensure all deployments use a factory or `CREATE2`-based script that atomically deploys the proxy and calls `initialize()` within the same transaction, and document this as a hard deployment invariant.

---

### Proof of Concept

1. Deployer broadcasts `deployProxy(BaseProxyManager_impl)` — transaction is pending in the mempool.
2. Attacker observes the pending transaction, extracts the predicted proxy address (deterministic via `CREATE2` or nonce), and broadcasts `proxy.initialize()` with a higher gas price.
3. Attacker's `initialize()` is mined first: `owner = attacker`, `submitter = attacker`.
4. Deployer's `initialize()` reverts (`InvalidInitialization`).
5. Attacker calls `registerRegularProxy("Clearinghouse", clearinghouseProxy)` followed by `submitImpl("Clearinghouse", maliciousImpl)` (as submitter), then `migrateAll([{name:"Clearinghouse", impl:maliciousImpl}])` (as owner).
6. `Clearinghouse` proxy now delegates to the attacker's implementation; all deposited collateral is accessible. [1](#0-0) [7](#0-6) [8](#0-7)
