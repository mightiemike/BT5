### Title
NLP Pool Owner Cannot Self-Claim Profit Share; Only Protocol Owner Can Initiate Transfer - (File: `core/contracts/EndpointTx.sol`, `core/contracts/ContractOwner.sol`)

---

### Summary

The `NlpProfitShare` transaction type is gated behind `require(sender == owner())` in the slow-mode submission path. The NLP pool owner — whose address is already verified as the sole valid recipient — has no on-chain mechanism to initiate their own profit share claim. Only the protocol owner can submit this transaction, creating a structural single-point-of-failure identical in class to the reported bug.

---

### Finding Description

In `submitSlowModeTransactionImpl` (`EndpointTx.sol`), `NlpProfitShare` is grouped with a set of admin-only transaction types that all require `sender == owner()`:

```solidity
} else if (
    txType == IEndpoint.TransactionType.WithdrawInsurance ||
    ...
    txType == IEndpoint.TransactionType.NlpProfitShare ||
    ...
) {
    require(sender == owner());
}
``` [1](#0-0) 

The only on-chain entry point for an NLP pool owner to trigger a profit share is `ContractOwner.nlpProfitShare()`, which is itself `onlyOwner`:

```solidity
function nlpProfitShare(
    uint64 poolId,
    bytes32 recipient,
    uint128 amount
) external onlyOwner {
``` [2](#0-1) 

When the slow-mode transaction is eventually processed, `processSlowModeTransactionImpl` enforces that the recipient must equal the pool owner's address:

```solidity
require(
    address(uint160(bytes20(txn.recipient))) ==
        nlpPools[txn.poolId].owner,
    ERR_UNAUTHORIZED
);
``` [3](#0-2) 

This means the protocol already knows and enforces who the rightful recipient is. Despite this, the pool owner cannot self-initiate the claim. The actual balance transfer in `Clearinghouse.nlpProfitShare` moves quote balance from the pool subaccount to the recipient: [4](#0-3) 

`NlpProfitShare` is absent from `processTransactionImpl` (the sequencer fast path), so there is no alternative route — it is exclusively a slow-mode, owner-submitted operation. [5](#0-4) 

---

### Impact Explanation

An NLP pool owner's earned profit share accumulates in the pool subaccount (`nlpPools[poolId].subaccount`) and can only be released by the protocol owner submitting a `NlpProfitShare` slow-mode transaction. If the protocol owner is unresponsive, goes offline, or is compromised, the NLP pool owner's profit share is permanently locked with no self-rescue path. The corrupted state is the quote balance in the pool subaccount that should be transferable to the pool owner but is not. This is a direct asset-locking impact on a non-privileged party (the NLP pool owner).

---

### Likelihood Explanation

Medium. The NLP pool owner is an unprivileged external party who has provided liquidity and earned profit share. The dependency on the protocol owner to release those funds is a structural design flaw that activates whenever the owner is unavailable — whether due to key loss, operational failure, or deliberate withholding. The pool owner has no fallback.

---

### Recommendation

Allow the NLP pool owner to self-submit a `NlpProfitShare` slow-mode transaction. In `submitSlowModeTransactionImpl`, remove `NlpProfitShare` from the `require(sender == owner())` block and instead validate that `msg.sender` is the pool owner for the specified `poolId`. The recipient check already enforced in `processSlowModeTransactionImpl` ensures the funds can only go to the correct address, so there is no security regression in permitting self-submission.

---

### Proof of Concept

1. Protocol owner calls `ContractOwner.nlpProfitShare(poolId, recipient, amount)` — this is the **only** way to initiate a profit share claim. [2](#0-1) 

2. Internally, `_submitSlowModeTransaction` is called, which delegates to `submitSlowModeTransactionImpl`. Because `NlpProfitShare` is in the owner-only branch, any call from the NLP pool owner directly to `Endpoint.submitSlowModeTransaction` with a `NlpProfitShare` payload reverts at:
   ```solidity
   require(sender == owner());
   ``` [6](#0-5) 

3. The NLP pool owner has no alternative path — `NlpProfitShare` does not appear in `processTransactionImpl` (the sequencer fast path), so the sequencer cannot process it either without the owner first submitting it via slow mode. [7](#0-6) 

4. Profit share balance sits in `nlpPools[poolId].subaccount` indefinitely until the protocol owner acts. The pool owner cannot unilaterally recover their funds. [8](#0-7)

    function nlpProfitShare(
        bytes32 poolSubaccount,
        bytes32 recipient,
        uint128 amount
    ) external onlyEndpoint {
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 toTransfer = int128(amount);
        ISpotEngine spotEngine = _spotEngine();

        spotEngine.updateBalance(QUOTE_PRODUCT_ID, poolSubaccount, -toTransfer);
        spotEngine.updateBalance(QUOTE_PRODUCT_ID, recipient, toTransfer);

        require(
            getHealth(poolSubaccount, IProductEngine.HealthType.INITIAL) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
