### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Merkle Proof Protection — (File: `contract/src/lib.rs`)

---

### Summary

The contract exposes two transaction-inclusion verification entry points. The protection against 64-byte Merkle-proof forgery (coinbase Merkle proof check) is only enforced in `verify_transaction_inclusion_v2`. The original `verify_transaction_inclusion` (v1) is marked `#[deprecated]` but remains a fully reachable public NEAR contract method. Any unprivileged NEAR caller can invoke v1 directly, skipping the coinbase check entirely — an exact structural analog to the external report's "second leg only executes under a condition" pattern.

---

### Finding Description

The contract defines two public verification functions:

**v1 — no coinbase check:** [1](#0-0) 

```rust
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
    // ... confirmations check, mainchain membership check ...
    merkle_tools::compute_root_from_merkle_proof(
        args.tx_id,
        usize::try_from(args.tx_index).unwrap(),
        &args.merkle_proof,
    ) == header.block_header.merkle_root   // ← no coinbase guard
}
```

**v2 — coinbase check added, then delegates to v1:** [2](#0-1) 

```rust
#[pause]
pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
    require!(
        args.merkle_proof.len() == args.coinbase_merkle_proof.len(), ...
    );
    require!(
        merkle_tools::compute_root_from_merkle_proof(
            args.coinbase_tx_id.clone(), 0usize, &args.coinbase_merkle_proof,
        ) == header.block_header.merkle_root,
        "Incorrect coinbase merkle proof"
    );
    #[allow(deprecated)]
    self.verify_transaction_inclusion(args.into())   // ← protection only here
}
```

Rust's `#[deprecated]` attribute is a **compiler-only warning**; it imposes no runtime restriction. `verify_transaction_inclusion` is compiled into the WASM binary as a normal public method and is callable by any NEAR account via a standard function call.

The structural parallel to the external report is exact:

| External report | This repository |
|---|---|
| Two-leg trade; slippage guard only in leg 2 | Two verification functions; coinbase guard only in v2 |
| Leg 2 skipped when `borrowToken == DAI` | Coinbase guard skipped when caller invokes v1 directly |
| No slippage protection on the executed path | No 64-byte forgery protection on the v1 path |

The 64-byte transaction Merkle-proof forgery works as follows: every internal Merkle-tree node is the double-SHA256 of two concatenated 32-byte child hashes (64 bytes). An attacker can present any such internal node as a `tx_id`, supply the siblings above it as the `merkle_proof`, and `compute_root_from_merkle_proof` will reconstruct the correct Merkle root — returning `true` for a transaction that does not exist. [3](#0-2) 

The coinbase-length equality check in v2 (`merkle_proof.len() == coinbase_merkle_proof.len()`) defeats this because an internal node at tree level L produces a proof of length `(depth − L)`, which differs from the leaf-level coinbase proof length `depth`. v1 has no such guard. [4](#0-3) 

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` (v1) to gate fund releases — e.g., a Bitcoin bridge releasing wrapped tokens upon "proof" of a deposit — will accept a forged proof and release funds for a Bitcoin transaction that never occurred. The corrupted value is the boolean return of the verification function, which downstream contracts treat as authoritative.

---

### Likelihood Explanation

High. The entry point is a standard public NEAR contract method requiring no special role, no staking, and no privileged key. The attacker needs only:
1. Any block already accepted into the contract's mainchain (publicly observable).
2. The block's transaction list (publicly available from any Bitcoin node).
3. The ability to compute the Merkle tree and pick an internal node.

All three are trivially available to any external actor.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or add an unconditional `env::panic_str` to it so it cannot be called at runtime:

```rust
#[deprecated(...)]
#[pause]
pub fn verify_transaction_inclusion(&self, ...) -> bool {
    env::panic_str("verify_transaction_inclusion is removed; use verify_transaction_inclusion_v2");
}
```

Alternatively, inline the coinbase check directly into v1 and remove the separate v2 entry point.

---

### Proof of Concept

1. Let block `B` be any mainchain block with ≥ 2 transactions `[T0, T1, ...]`.
2. Compute the Merkle tree. The root's left child is `N = SHA256d(T0 ‖ T1)` — an internal node.
3. The Merkle proof for `N` at position 0 in level 1 is just `[right_sibling_of_N_at_level_1, ...]` — one element shorter than a leaf proof.
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node, not a real transaction)
   - `tx_block_blockhash = hash(B)`
   - `tx_index = 0`
   - `merkle_proof = [siblings above N]`
   - `confirmations = 1`
5. `compute_root_from_merkle_proof(N, 0, proof)` reconstructs the correct Merkle root of `B`.
6. The function returns `true` — a forged inclusion proof accepted.

`verify_transaction_inclusion_v2` would reject this because `merkle_proof.len()` (depth − 1) ≠ `coinbase_merkle_proof.len()` (depth). v1 has no such check. [5](#0-4)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

            usize::try_from(args.tx_index).unwrap(),

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
