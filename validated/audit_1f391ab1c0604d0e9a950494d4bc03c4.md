### Title
Slow Mode `WithdrawCollateral` Bypasses Per-Product `withdrawFeeX18` Fee — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `processSlowModeTransactionImpl` function handles `WithdrawCollateral` without charging the per-product `withdrawFeeX18` fee, while the fast mode path (`processTransactionImpl`) always charges it. Any user can exploit this by routing a `WithdrawCollateral` through `submitSlowModeTransaction`, paying only the flat `SLOW_MODE_FEE` ($1) instead of the proportional `withdrawFeeX18` fee, breaking the protocol's fee-accounting invariant.

---

### Finding Description

**Fast mode path** — `processTransactionImpl` (lines 413–436):

```solidity
validateSignedTx(signedTx.tx.sender, signedTx.tx.nonce, transaction, signedTx.signature, true);
chargeFee(
    signedTx.tx.sender,
    spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18,
    signedTx.tx.productId
);
clearinghouse.withdrawCollateral(...);
```

`chargeFee` deducts `withdrawFeeX18` from the sender's balance and credits `sequencerFee[productId]`:

```solidity
function chargeFee(bytes32 sender, int128 fee, uint32 productId) internal {
    spotEngine.updateBalance(productId, sender, -fee);
    sequencerFee[productId] += fee;
}
```

**Slow mode path** — `processSlowModeTransactionImpl` (lines 217–229):

```solidity
validateSender(txn.sender, sender);
clearinghouse.withdrawCollateral(
    txn.sender, txn.productId, txn.amount, address(0), nSubmissions
);
```

No `chargeFee` call. No `withdrawFeeX18` deducted. The only fee charged for slow mode is `SLOW_MODE_FEE` ($1 flat) at submission time in `submitSlowModeTransactionImpl`, which flows into `slowModeFees` — a completely separate accounting bucket from `sequencerFee[productId]`.

The two paths share the same terminal action (`clearinghouse.withdrawCollateral`) but only one enforces the fee invariant. This is structurally identical to the reference bug: two privileged execution paths reach the same state-mutating function, but only one enforces the protocol's accounting constraint.

---

### Impact Explanation

- `sequencerFee[productId]` is under-credited for every slow mode withdrawal where `withdrawFeeX18 > 0`.
- The `claimSequencerFees` flow (called via `DumpFees`) sweeps `sequencerFee[productId]` into the X account. Under-crediting this mapping directly reduces sequencer revenue.
- For a $10,000 withdrawal at `withdrawFeeX18 = 0.1%` (consistent with `FAST_WITHDRAWAL_FEE_RATE = 1_000_000_000_000_000`), the fast mode fee is $10; the slow mode cost is $1. The user saves $9 per withdrawal; the sequencer fee pool loses $9.
- The broken invariant is concrete: `sequencerFee[productId]` must reflect all withdrawal fees collected, but slow mode withdrawals contribute zero to it.

---

### Likelihood Explanation

- The entry point is `submitSlowModeTransaction`, callable by any non-sanctioned user with a valid subaccount.
- No privileged role is required. The attacker only needs to hold a balance and wait 3 days.
- The attack is economically rational whenever `withdrawFeeX18 * withdrawalAmount > SLOW_MODE_FEE ($1)`. For any withdrawal above ~$1,000 at 0.1% fee, the user profits from the bypass.
- The slow mode path for `WithdrawCollateral` is explicitly supported and documented; it is not an edge case.

---

### Recommendation

Add the `withdrawFeeX18` charge to the slow mode `WithdrawCollateral` handler in `processSlowModeTransactionImpl`, mirroring the fast mode path:

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.WithdrawCollateral memory txn = abi.decode(
        transaction[1:], (IEndpoint.WithdrawCollateral)
    );
    validateSender(txn.sender, sender);
+   chargeFee(
+       txn.sender,
+       spotEngine.getConfig(txn.productId).withdrawFeeX18,
+       txn.productId
+   );
    clearinghouse.withdrawCollateral(
        txn.sender, txn.productId, txn.amount, address(0), nSubmissions
    );
```

---

### Proof of Concept

1. User holds $10,000 USDC in subaccount `S`.
2. `withdrawFeeX18` for USDC product is `1e15` (0.1%).
3. User calls `Endpoint.submitSlowModeTransaction` with `WithdrawCollateral{sender: S, productId: USDC_ID, amount: 10000e6, nonce: 0}`.
4. `submitSlowModeTransactionImpl` charges `SLOW_MODE_FEE` = $1 → `slowModeFees += 1e6`.
5. After 3 days, anyone calls `executeSlowModeTransaction`.
6. `processSlowModeTransactionImpl` dispatches to the `WithdrawCollateral` branch: calls `validateSender`, then directly calls `clearinghouse.withdrawCollateral` — **no `chargeFee` call**.
7. User receives $10,000. `sequencerFee[USDC_ID]` is $0 instead of $10.
8. Net saving for user: $9. Net loss to sequencer fee pool: $9 per such withdrawal.
