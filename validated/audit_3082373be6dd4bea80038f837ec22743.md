### Title
Arithmetic Overflow in `PriceVelocityGuardExtension.beforeSwap()` Causes Swap DoS When Oracle Price Moves Extremely - (File: metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol)

### Summary

The velocity guard's squared-comparison arithmetic in `beforeSwap` overflows `uint256` when the oracle mid-price moves dramatically relative to the stored `lastMidPriceX64`. Instead of reverting with the expected `PriceVelocityExceeded` error, the guard panics with an arithmetic overflow, permanently blocking all swaps through the pool until admin intervention.

### Finding Description

In `PriceVelocityGuardExtension.beforeSwap()`, the guard computes the squared actual change to avoid an on-chain `sqrt`:

```solidity
uint256 changeE18 = (delta * 1e18) / uint256(prevMid);
uint256 actualSq  = changeE18 * changeE18;           // ÔåÉ unchecked product
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);
if (actualSq > allowedSq) revert PriceVelocityExceeded(actualSq, allowedSq);
```

`delta` is the absolute difference of two `uint128` mid-prices, so `delta Ôëñ type(uint128).max Ôëê 3.4 ├ù 10┬│Ôü©`. `prevMid` is also `uint128` and can be as small as 1 (the pool only requires `bidPriceX64 > 0`). Therefore:

```
changeE18 = (delta ├ù 1e18) / prevMid
          Ôëñ (3.4e38 ├ù 1e18) / 1
           = 3.4e56
```

`type(uint256).max Ôëê 1.15 ├ù 10ÔüÀÔüÀ`, so `changeE18` fits in `uint256`. However, the **square** does not:

```
actualSq = changeE18┬▓ Ôëñ (3.4e56)┬▓ = 1.16e113  >>  type(uint256).max
```

Solidity 0.8+ checked arithmetic causes this multiplication to revert with a panic (`0x11`). Because `beforeSwap` is called inside `MetricOmmPool.swap()` before any state is committed, every swap attempt reverts with a panic rather than `PriceVelocityExceeded`. The stored `lastMidPriceX64` is also not updated (the state write at line 57 is rolled back with the transaction), so the condition persists on every subsequent swap.

The intermediate step `delta * 1e18` does **not** overflow (`3.4e38 ├ù 1e18 = 3.4e56 < 1.15e77`), so the overflow is invisible until the squaring step.

### Impact Explanation

Any pool that has `PriceVelocityGuardExtension` wired into its `beforeSwap` order and has `maxChangePerBlockE18 Ôëá 0` becomes completely unable to execute swaps once the overflow condition is met. The pool's swap path is:

```
MetricOmmPool.swap()
  ÔåÆ _beforeSwap()
    ÔåÆ _callExtensionsInOrder(BEFORE_SWAP_ORDER, ...)
      ÔåÆ PriceVelocityGuardExtension.beforeSwap()  ÔåÉ panics here
```

All swap transactions revert. LPs can still call `removeLiquidity` (which does not invoke `beforeSwap`), so principal is not permanently locked. However, the pool is rendered entirely unusable for trading ÔÇö an "unusable swap flow" per the impact gate ÔÇö until the pool admin calls `setLastMidPrice` to reset the stored reference price to a value close to the current oracle price.

### Likelihood Explanation

The overflow requires `changeE18 > sqrt(type(uint256).max) Ôëê 1.16 ├ù 10┬│Ôü©`, i.e.:

```
delta / prevMid > 1.16e20
```

Concrete triggering scenarios:

1. **Admin sets a very small seed price:** `setLastMidPrice(pool, 1)` followed by the oracle quoting any mid-price above `1.16e20` (in Q64.64 units, that is `Ôëê 6.3 ├ù 10┬│` in real terms ÔÇö a modest price for many token pairs).
2. **Extreme oracle move between two swaps:** If the oracle price jumps by a factor of `> 1.16e20` between consecutive swaps (e.g., a token price crash from a large value to near-zero, or vice versa), the overflow triggers without any admin action.
3. **Long dormancy + price reset:** If no swap occurs for a long time and the oracle price drifts far from the stored `lastMidPriceX64`, the first swap after dormancy can overflow.

### Recommendation

Replace the bare squaring with overflow-safe arithmetic. The simplest correct approach is to compare `changeE18` directly against `maxChange * sqrt(1 + blockDiff)` using integer square-root, or to cap `changeE18` before squaring:

```solidity
// Option A: cap changeE18 at a safe ceiling before squaring
uint256 SAFE_MAX = 1e38; // sqrt(type(uint256).max) Ôëê 1.16e38
uint256 cappedChange = changeE18 > SAFE_MAX ? SAFE_MAX : changeE18;
uint256 actualSq = cappedChange * cappedChange;
// Any value above SAFE_MAX already exceeds any realistic allowedSq.

// Option B: use Math.mulDiv for the comparison
// actualSq > allowedSq  Ôƒ║  changeE18 > sqrt(allowedSq)
// Compute without squaring by comparing changeE18 against
// an integer sqrt of allowedSq (OZ Math.sqrt is available).
```

### Proof of Concept

```
1. Deploy PriceVelocityGuardExtension and wire it into a pool's beforeSwap order.
2. Admin calls setMaxChangePerBlock(pool, 1e18)  // 100% per block ÔÇö non-zero guard active.
3. Admin calls setLastMidPrice(pool, 1)           // seed prevMid = 1 (minimum non-zero).
4. Oracle quotes bidPriceX64 = type(uint128).max - 1,
                  askPriceX64 = type(uint128).max.
   ÔåÆ midPrice Ôëê type(uint128).max Ôëê 3.4e38
5. Any caller invokes pool.swap(...).
   ÔåÆ delta = midPrice - prevMid Ôëê 3.4e38
   ÔåÆ changeE18 = (3.4e38 ├ù 1e18) / 1 = 3.4e56
   ÔåÆ actualSq = 3.4e56 ├ù 3.4e56 = 1.16e113  ÔåÆ PANIC (arithmetic overflow)
6. Every subsequent swap also panics because lastMidPriceX64 is never updated
   (the state write is rolled back with the reverting transaction).
7. Pool swaps are blocked until admin calls setLastMidPrice(pool, currentOracleMid).
