### Title
Hardcoded Protocol Treasury Reward Rate Silently Overrides Configurable `protocol_reward_rate` for Mainnet/Testnet — (`File: chain/epoch-manager/src/reward_calculator.rs`)

### Summary

`RewardCalculator::calculate_reward` contains a compile-time branch that, whenever `genesis_protocol_version == PROD_GENESIS_PROTOCOL_VERSION` (i.e., the production mainnet/testnet genesis version 29), discards the `protocol_reward_rate` field read from `GenesisConfig` and substitutes a hardcoded literal `Rational32::new_raw(1, 10)` (10%). The `_protocol_version` parameter accepted by the function is unused (prefixed `_`), so no protocol-version gate exists to ever change this value through the normal versioned-parameter upgrade path.

### Finding Description

In `chain/epoch-manager/src/reward_calculator.rs`, `calculate_reward` contains:

```rust
let use_hardcoded_value = self.genesis_protocol_version == PROD_GENESIS_PROTOCOL_VERSION;
let protocol_reward_rate = if use_hardcoded_value {
    Rational32::new_raw(1, 10)   // ← hardcoded 10%, ignores self.protocol_reward_rate
} else {
    self.protocol_reward_rate
};
``` [1](#0-0) 

`PROD_GENESIS_PROTOCOL_VERSION` is the constant `29`, the genesis version of both mainnet and testnet. [2](#0-1) 

The function signature accepts `_protocol_version: ProtocolVersion` but the leading underscore signals it is never read; there is no version-gated branch that could alter the treasury rate at a future protocol upgrade. [3](#0-2) 

`RewardCalculator` is constructed from `GenesisConfig.protocol_reward_rate` in `RewardCalculator::new`: [4](#0-3) 

Yet for every mainnet/testnet epoch reward calculation, `self.protocol_reward_rate` is silently replaced by the literal `1/10`. The `ProtocolConfigView` RPC type exposes `protocol_reward_rate` from `GenesisConfig`: [5](#0-4) 

This creates a divergence: the value advertised to external observers via the RPC endpoint is the genesis-config field, while the value actually applied in every epoch reward computation is the hardcoded literal.

### Impact Explanation

Every epoch, `epoch_protocol_treasury` — the share of newly minted tokens sent to the protocol treasury account — is computed as:

```
epoch_total_reward * protocol_reward_rate
``` [6](#0-5) 

Because `protocol_reward_rate` is hardcoded at `1/10` for mainnet/testnet, any governance or protocol-upgrade decision to change the treasury share (e.g., reduce it to 5% or raise it to 15%) would be silently ignored: the `GenesisConfig` field would be updated, the RPC would report the new value, but the actual minting computation would continue using 10%. Validators, token holders, and treasury-dependent contracts would observe a permanent divergence between the advertised and effective treasury rate with no on-chain signal of the discrepancy.

### Likelihood Explanation

The branch is unconditionally taken for every mainnet/testnet node on every epoch boundary. Any future governance proposal to adjust `protocol_reward_rate` through the normal parameter-update path (versioned `GenesisConfig` or `EpochConfig`) would fail silently because the hardcoded override is not gated by any protocol version check. The `_protocol_version` parameter being unused confirms no upgrade path exists within the current code.

### Recommendation

Replace the compile-time identity check with a protocol-version–gated parameter read. Specifically:

1. Remove the `use_hardcoded_value` branch and the hardcoded `Rational32::new_raw(1, 10)` literal.
2. Honour `self.protocol_reward_rate` unconditionally (the genesis config for mainnet already stores `1/10`, so behaviour is preserved at current versions).
3. Wire the unused `_protocol_version` parameter into a `RuntimeConfigStore` lookup so future protocol upgrades can change the treasury rate through the standard versioned-parameter mechanism, consistent with how `burnt_gas_reward` and other economic ratios are managed.
4. Add a snapshot test asserting the treasury rate for the current stable protocol version to prevent accidental regression.

### Proof of Concept

The divergence is directly observable in the existing test `test_adjust_max_inflation`, which explicitly constructs a `RewardCalculator` with `protocol_reward_rate: Ratio::new(0, 1)` (zero) but `genesis_protocol_version: PROD_GENESIS_PROTOCOL_VERSION`, then asserts that the treasury receives exactly `expected_total / 10` — confirming the configured zero rate is discarded and the hardcoded 10% is applied: [7](#0-6) [8](#0-7)
