### Title
Fork Headers Written to `headers_pool` Are Never Garbage-Collected, Permanently Locking Storage Deposits — (`contract/src/lib.rs`)

---

### Summary

The contract stores fork (sidechain) block headers in `headers_pool` during normal block submission but provides no mechanism to ever remove them. The garbage-collection path exclusively targets mainchain blocks. Storage deposits paid by relayers for fork blocks are permanently consumed with no refund path, directly mirroring the `riskPoolBalance` pattern: state is accumulated during normal operations, never used for its intended purpose after the reorg window closes, and is only incidentally cleared through an unintended path (winning a reorg).

---

### Finding Description

When `submit_block_header_inner` determines a submitted header extends a fork rather than the mainchain tip, it calls `store_fork_header`:

```rust
// contract/src/lib.rs:560
self.store_fork_header(&current_header);
```

`store_fork_header` writes exclusively to `headers_pool` and creates no mainchain index entries:

```rust
// contract/src/lib.rs:664-667
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
```

The garbage-collection function `run_mainchain_gc` iterates only over `mainchain_height_to_header`:

```rust
// contract/src/lib.rs:401-408
for height in start_removal_height..end_removal_height {
    let blockhash = &self
        .mainchain_height_to_header
        .get(&height)
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
    self.remove_block_header(blockhash);
    self.mainchain_height_to_header.remove(&height);
}
```

Because fork blocks are never inserted into `mainchain_height_to_header`, GC can never reach them. The project's own documentation confirms this explicitly:

> "Only mainchain blocks are deleted; fork/sidechain blocks are not affected"
> — `contract/CLAUDE.md:39`

`remove_block_header` is called from exactly two sites: `run_mainchain_gc` (mainchain path only) and `reorg_chain` (which removes the *old mainchain* blocks being displaced, not the fork blocks being promoted). Fork blocks that never accumulate enough chainwork to trigger a reorg are permanently stranded in `headers_pool`.

The storage deposit mechanism in `submit_blocks` charges the caller for the net storage increase at submission time:

```rust
// contract/src/lib.rs:182-188
let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());
require!(amount >= required_deposit, ...);
```

Since fork blocks are never freed, the NEAR tokens paid as storage deposits for them are permanently locked in the contract with no reclaim path.

---

### Impact Explanation

Every fork block submitted by a relayer costs a non-refundable NEAR storage deposit. The fork block's only intended use is to support a potential chain reorg; once the mainchain advances far enough that the fork can no longer win, the fork block is useless but remains in storage indefinitely. There is no `withdraw_storage_deposit`, no fork GC function, and no administrative escape hatch to reclaim these tokens. On a live Bitcoin or Dogecoin deployment, natural short forks occur continuously (orphan blocks), meaning this is not a theoretical edge case — fork entries accumulate as a routine consequence of normal relayer operation.

---

### Likelihood Explanation

Natural Bitcoin orphan blocks occur at a rate of roughly 1–2 per day on mainnet. Every orphan block submitted by a relayer (which is the correct behavior — the relayer must submit all observed headers to maintain fork-choice correctness) produces a permanently stranded `headers_pool` entry. On Dogecoin, with its 1-minute block time and AuxPoW merged mining, orphan rates are higher. This is triggered by normal, correct relayer behavior, not by an adversary.

---

### Recommendation

Implement a fork GC mechanism analogous to `run_mainchain_gc`. Track the height at which each fork block was submitted (or use the block's own `block_height`). During GC, remove fork blocks whose height falls below `mainchain_initial_blockhash.block_height` — i.e., blocks that are older than the oldest mainchain block the contract still tracks and therefore can never participate in a reorg. Alternatively, remove fork blocks during `run_mainchain_gc` by scanning `headers_pool` entries that have no corresponding entry in `mainchain_header_to_height` and whose height is below the GC cutoff.

---

### Proof of Concept

1. Deploy the contract with `gc_threshold = 100`.
2. Submit 101 mainchain headers (GC activates, oldest mainchain block is pruned, storage deposit partially freed).
3. Submit one fork header branching from any stored mainchain block. The relayer pays a storage deposit for this fork entry.
4. Advance the mainchain by 200 more blocks. GC runs repeatedly, pruning mainchain blocks below the threshold.
5. Call `get_mainchain_size()` — it reflects only mainchain blocks within the threshold.
6. Observe that the fork block from step 3 remains in `headers_pool` indefinitely. The storage deposit paid in step 3 is permanently locked. No callable function exists to remove the fork entry or reclaim the deposit. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    pub fn run_mainchain_gc(&mut self, batch_size: u64) {
        let initial_blockheader = self
            .headers_pool
            .get(&self.mainchain_initial_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let tip_blockheader = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let amount_of_headers_we_store =
            tip_blockheader.block_height - initial_blockheader.block_height + 1;

        if amount_of_headers_we_store > self.gc_threshold {
            let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
            let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);

            let start_removal_height = initial_blockheader.block_height;
            let end_removal_height = initial_blockheader.block_height + selected_amount_to_remove;
            env::log_str(&format!(
                "Num of blocks to remove {selected_amount_to_remove}"
            ));

            for height in start_removal_height..end_removal_height {
                let blockhash = &self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

                self.remove_block_header(blockhash);
                self.mainchain_height_to_header.remove(&height);
            }

            self.mainchain_initial_blockhash = self
                .mainchain_height_to_header
                .get(&end_removal_height)
                .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        }
    }
