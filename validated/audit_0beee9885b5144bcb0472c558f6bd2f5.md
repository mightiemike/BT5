### Title
One Paused or Reverting Token in `creditDeposit()` Blocks All Multi-Asset Deposits via `DirectDepositV1` — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` iterates over every product registered in the `SpotEngine` and attempts to approve and deposit each token balance in a single atomic loop. If any single token's `approve()` or `depositCollateralWithReferral()` call reverts — for example, because a supported token (e.g., USDC) is paused — the entire transaction reverts. All other tokens already held in the DDA contract become stuck, and no deposits for any product can proceed until the owner manually recovers each token individually.

---

### Finding Description

`DirectDepositV1.creditDeposit()` is the sole mechanism for crediting multi-asset balances held in the contract to the bound subaccount on the `Endpoint`. The function iterates over all product IDs returned by `spotEngine.getProductIds()`:

```solidity
// core/contracts/DirectDepositV1.sol lines 83–101
function creditDeposit() external {
    uint32[] memory productIds = spotEngine.getProductIds();
    for (uint256 i = 0; i < productIds.length; i++) {
        uint32 productId = productIds[i];
        address tokenAddr = spotEngine.getToken(productId);
        require(tokenAddr != address(0), "Invalid productId.");   // hard revert
        IIERC20Base token = IIERC20Base(tokenAddr);
        uint256 balance = token.balanceOf(address(this));
        if (balance != 0) {
            token.approve(address(endpoint), balance);            // can revert
            endpoint.depositCollateralWithReferral(               // can revert
                subaccount,
                productId,
                uint128(balance),
                "-1"
            );
        }
    }
}
```

There are two independent revert paths that block the entire loop:

1. **Hard `require` on zero-address token** (line 88): If `spotEngine.getToken(productId)` returns `address(0)` for any product ID in the list (e.g., a newly registered product whose config is not yet fully set, or a special internal product like `NLP_PRODUCT_ID`), the call reverts unconditionally, preventing all subsequent products from being processed.

2. **Reverting `approve()` or `depositCollateralWithReferral()`** (lines 92–98): Many widely-used ERC-20 tokens (USDC, USDT) implement a `pause()` mechanism that causes all state-changing calls — including `approve()` — to revert. If any token in the product list is paused at the time `creditDeposit()` is called, the entire transaction reverts. Tokens for all other products that were already held in the DDA remain stuck.

The `Endpoint.depositCollateralWithReferral()` itself contains an explicit comment acknowledging this exact failure mode for the minimum-deposit check:

```solidity
// core/contracts/Endpoint.sol lines 137–142
if (!isValidDepositAmount(subaccount, productId, amount)) {
    // we cannot revert here, otherwise direct deposit could be blocked when there are
    // multiple assets awaiting credit but one of them is below the minimum deposit amount.
    // we can just skip the deposit and continue with the next asset.
    return;
}
```

The developers correctly handled the minimum-deposit edge case by returning early instead of reverting. However, the same defensive pattern was not applied in `DirectDepositV1.creditDeposit()` for token-level failures (paused token, zero-address token), leaving the loop vulnerable to a single broken product blocking all others.

---

### Impact Explanation

When `creditDeposit()` reverts due to one broken product:

- All ERC-20 token balances already transferred into the DDA contract are frozen. They cannot be deposited to the subaccount until the owner calls `withdraw(token)` for each token individually.
- The DDA is rendered completely non-functional for all products, not just the broken one.
- The `withdraw()` recovery path is `onlyOwner`, so if the owner key is unavailable or the DDA is operated by an automated system without a responsive owner, funds remain locked for an indefinite period.
- Users relying on the DDA for automated multi-asset deposits (e.g., native-token wrapping flows via `receive()`) will silently accumulate stuck balances with no on-chain indication of failure beyond a reverted transaction.

---

### Likelihood Explanation

USDC (Circle) and USDT (Tether) both implement a guardian-controlled `pause()` that causes `approve()` and `transfer()` to revert. Both are realistic collateral assets in a DEX. A regulatory freeze, exploit response, or routine maintenance pause on either token is a plausible real-world event. Because `creditDeposit()` is the only deposit path for the DDA, any such event immediately and completely disables the contract for all products simultaneously.

---

### Recommendation

Replace the hard `require` and bare external calls in the loop with a `try/catch` (or a continue-on-failure pattern) so that a single broken product is skipped rather than aborting the entire batch:

```solidity
function creditDeposit() external {
    uint32[] memory productIds = spotEngine.getProductIds();
    for (uint256 i = 0; i < productIds.length; i++) {
        uint32 productId = productIds[i];
        address tokenAddr = spotEngine.getToken(productId);
        if (tokenAddr == address(0)) continue;          // skip, do not revert
        IIERC20Base token = IIERC20Base(tokenAddr);
        uint256 balance = token.balanceOf(address(this));
        if (balance != 0) {
            try token.approve(address(endpoint), balance) {
                try endpoint.depositCollateralWithReferral(
                    subaccount, productId, uint128(balance), "-1"
                ) {} catch {}
            } catch {}
        }
    }
}
```

This mirrors the pattern already used in `Endpoint.depositCollateralWithReferral()` for the minimum-deposit case.

### Proof of Concept

1. Deploy `DirectDepositV1` bound to a subaccount, with USDC (product 0) and WETH (product 1) registered in the `SpotEngine`.
2. Transfer 1000 USDC and 1 WETH to the DDA contract.
3. Circle pauses USDC (or a test harness calls `pause()` on a mock USDC).
4. Any caller invokes `creditDeposit()`.
5. The loop reaches product 0 (USDC), calls `token.approve(...)`, which reverts because USDC is paused.
6. The entire `creditDeposit()` transaction reverts.
7. Both the 1000 USDC and the 1 WETH remain stuck in the DDA contract.
8. WETH cannot be deposited despite being fully functional, because the loop never reaches product 1.
9. Recovery requires the owner to call `withdraw(WETH)` and `withdraw(USDC)` separately — the automated deposit path is completely broken. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
