### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the 64-Byte Transaction Forgery Protection Introduced in v2 — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) was superseded by `verify_transaction_inclusion_v2` specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability. The v2 function enforces a coinbase proof check before delegating to v1. However, v1 is still `pub` and protected only by `#[pause]`, meaning any unprivileged NEAR caller can invoke it directly, bypassing the coinbase proof check entirely. The `#[deprecated]` attribute is a Rust compiler lint only — it has no effect on on-chain callability.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced to close the 64-byte transaction forgery attack vector. Its mitigation is the coinbase proof check: [1](#0-0) 

After that check passes, it delegates to v1: [2](#0-1) 

The old v1 function is still `pub` and only gated by `#[pause]`: [3](#0-2) 

The `#[deprecated]` macro emits a Rust compiler warning; it does not restrict who can call the method on-chain. Any NEAR account can call `verify_transaction_inclusion` directly when the contract is not paused, skipping the coinbase proof entirely. The function's own documentation acknowledges the risk: [4](#0-3) 

The core verification in v1 is a raw Merkle root comparison with no guard against internal-node forgery: [5](#0-4) 

This is structurally identical to the M-10 pattern: the protocol "upgraded" its verification security (v1 → v2), but the old verification path remains reachable, creating a permanent bypass route.

---

### Impact Explanation

The 64-byte transaction forgery attack (documented at https://www.bitmex.com/blog/64-Byte-Transactions) allows an attacker to supply an internal Merkle tree node hash as `tx_id`. Because v1 only checks that `compute_root_from_merkle_proof(tx_id, tx_index, proof) == merkle_root`, a crafted internal-node hash with a matching proof path will return `true` for a transaction that does not exist on Bitcoin. Any downstream NEAR contract or bridge protocol that calls `verify_transaction_inclusion` to gate fund releases or state transitions will accept a fraudulent proof, enabling theft or unauthorized state changes.

---

### Likelihood Explanation

The attack requires no privileged role, no leaked key, and no social engineering. The 64-byte forgery technique is publicly documented and the construction of a valid internal-node proof is straightforward for anyone with access to a Bitcoin block's Merkle tree. The function is callable by any NEAR account whenever the contract is not paused.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or restrict it to a privileged role (e.g., `Role::DAO`) so it cannot be called by unprivileged accounts. The preferred fix is removal, since `verify_transaction_inclusion_v2` already covers all legitimate use cases. If backward compatibility is required, at minimum add the same coinbase proof check to v1 so both paths are equally safe.

---

### Proof of Concept

1. Attacker identifies a confirmed Bitcoin block already stored in the light client (block hash `B`, merkle root `R`).
2. Attacker picks any internal Merkle tree node `N` at depth `d` in that block's transaction tree. `N` is 32 bytes and is a valid SHA-256 hash of two child hashes — it is **not** a real transaction.
3. Attacker constructs a Merkle proof path `P` such that `compute_root_from_merkle_proof(N, index, P) == R`. This is trivially possible because `N` is itself a node in the tree.
4. Attacker calls `verify_transaction_inclusion` with `tx_id = N`, `tx_block_blockhash = B`, `tx_index = index`, `merkle_proof = P`, `confirmations = 1`.
5. v1 checks `mainchain_header_to_height` (block is on mainchain ✓), checks confirmations (✓), then returns `compute_root_from_merkle_proof(N, index, P) == R` → **`true`**.
6. No coinbase proof was ever required. The attacker has obtained a `true` verification result for a non-existent transaction. [6](#0-5)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
