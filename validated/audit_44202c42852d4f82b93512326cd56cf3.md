### Title
SwapAllowlistExtension Router Pass-Through: Allowlisting `MetricOmmSimpleRouter` Grants Blanket Swap Access to All Users — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap()` checks the `sender` argument passed by the pool, which is always `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, that `msg.sender` is the router contract, not the end-user. If the pool admin allowlists the router address, every non-allowlisted EOA can execute swaps through the router, completely defeating the per-user gate.

---

### Finding Description

The call chain is:

```
EOA → MetricOmmSimpleRouter.exactInputSingle()
        → pool.swap(msg.sender = router, ...)          [MetricOmmPool.sol:231]
            → _beforeSwap(sender = router, ...)         [ExtensionCalling.sol:160-176]
                → SwapAllowlistExtension.beforeSwap(sender = router)
                    → allowedSwapper[msg.sender][sender]
                    → allowedSwapper[pool][router]      ← EOA address never appears
```

In `MetricOmmPool.swap()`, `_beforeSwap` is called with `msg.sender` as the `sender` argument: [1](#0-0) 

`ExtensionCalling._beforeSwap()` forwards that `sender` value unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap()` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router: [3](#0-2) 

Meanwhile, `MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly with no forwarding of the original caller: [4](#0-3) 

The end-user's address (`msg.sender` of `exactInputSingle`) is stored only in transient storage as the `payer` for the callback — it is never passed to the pool or the extension. The extension has no mechanism to recover it.

---

### Impact Explanation

A pool admin who wants to allow router-mediated swaps will naturally call `setAllowedToSwap(pool, router, true)`. This is a reasonable and expected configuration. The consequence is that `allowedSwapper[pool][router] = true` becomes a blanket pass-through: every EOA that calls `exactInputSingle`, `exactInput`, or `exactOutputSingle` on the router will pass the `beforeSwap` check, regardless of whether they are individually allowlisted. The invariant that "only explicitly allowlisted end-users may swap" is broken. Any non-allowlisted address can drain tokens from a pool that was intended to be access-controlled.

---

### Likelihood Explanation

The `SwapAllowlistExtension` is explicitly documented as gating swaps by swapper address, and the router is the standard public entry point for swaps. A pool admin who deploys a restricted pool and wants it to be usable via the router has no correct way to configure this — allowlisting the router opens it to everyone, and not allowlisting the router blocks all router users including allowlisted ones. The misconfiguration is the only natural path.

---

### Recommendation

The extension must check the ultimate end-user, not the immediate caller. Options:

1. **Pass original caller through extension data**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a convention between router and extension.
2. **Check `tx.origin` as a fallback**: Only viable if the pool is never called from another contract.
3. **Redesign the extension to accept a "real sender" field**: The pool interface could carry an explicit `originator` separate from `sender`.

The current design makes `SwapAllowlistExtension` incompatible with any router-mediated flow.
