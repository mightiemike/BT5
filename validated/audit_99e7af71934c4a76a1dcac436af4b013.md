Audit Report

## Title
`SwapAllowlistExtension` checks router address as swapper instead of actual user, enabling full allowlist bypass — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the router contract is `msg.sender` of `pool.swap()`, so the extension checks `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][actualUser]`. Any pool admin who allowlists the router to enable router-based swaps for their approved users inadvertently grants swap access to every user, completely collapsing per-user access control.

## Finding Description

**Step 1 — Pool passes `msg.sender` of `pool.swap()` as `sender` to extensions.**

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, recipient, ...)` at line 230–240. The `msg.sender` here is whoever directly called `pool.swap()`.

**Step 2 — `SwapAllowlistExtension` checks only `sender`.**

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` in the extension is the pool (used as the mapping key); `sender` is whoever called `pool.swap()`.

**Step 3 — `MetricOmmSimpleRouter` is `msg.sender` of every `pool.swap()` call.**

`exactInputSingle` at line 72–80 calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` directly — the router contract is `msg.sender`. The same applies to `exactOutputSingle` (L136–137), `exactInput` (L104–112), and `exactOutput` (L165–181).

**Step 4 — The allowlist check resolves to `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.**

When `blockedUser` calls `router.exactInputSingle(...)`:
- `sender` in the extension = router address
- Extension checks `allowedSwapper[pool][router]` — passes if router is allowlisted
- The actual user identity is never visible to the extension; `extensionData` is passed through but `SwapAllowlistExtension` ignores it entirely

**Step 5 — The bypass path.**

A pool admin who wants their allowlisted users to swap via the router must call `setAllowedToSwap(pool, address(router), true)`. Once set, every user — including those never individually allowlisted — can call `router.exactInputSingle(...)` and the extension returns `beforeSwap.selector` without reverting. There is no alternative: if the admin does not allowlist the router, their approved users cannot use the standard periphery at all.

## Impact Explanation

A curated pool protected by `SwapAllowlistExtension` is fully open to any user the moment the pool admin allowlists the router. The allowlist invariant — "only approved addresses may swap" — is silently broken. Unauthorized users can execute swaps, extract value from LP positions, and interact with the pool in ways the admin explicitly intended to prevent. This constitutes a broken core pool functionality causing direct loss of curation policy with potential LP principal loss in pools designed for restricted counterparties.

## Likelihood Explanation

Likelihood is high. Allowlisting the router is the natural and expected action for any pool admin who wants their allowlisted users to use the standard periphery (`MetricOmmSimpleRouter` is the primary user-facing swap interface). There is no documentation warning that allowlisting the router collapses per-user access control. The admin has no alternative path: not allowlisting the router means approved users cannot swap through it; allowlisting it means all users can.

## Recommendation

The `sender` argument passed to `beforeSwap` must represent the economic actor (the end user), not the intermediary contract. Two complementary fixes:

1. **Router-level:** Have `MetricOmmSimpleRouter` forward the actual caller's address through `extensionData` (e.g., ABI-encode `msg.sender` as a prefix) so extensions can recover it.
2. **Extension-level:** `SwapAllowlistExtension` should decode the actual user from `extensionData` when `sender` is a known router, or the pool interface should expose a dedicated `originator` field separate from `sender`.

The cleanest fix is to add an `originator` parameter to the `beforeSwap` hook that the pool sets to the transaction origin or to a router-supplied value, and have `SwapAllowlistExtension` check `originator` instead of `sender`.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedUser = 0xAAAA  (the only intended swapper)
  blockedUser = 0xBBBB  (should never be able to swap)

Admin actions:
  swapExtension.setAllowedToSwap(pool, allowedUser, true)
  swapExtension.setAllowedToSwap(pool, address(router), true)
    ↑ admin does this so allowedUser can swap via the router

Attack:
  blockedUser calls router.exactInputSingle({pool: pool, ...})
    → router calls pool.swap(recipient, ...)   [msg.sender = router]
    → pool calls _beforeSwap(sender=router, ...)
    → extension checks allowedSwapper[pool][router] == true  ✓
    → swap executes — blockedUser receives tokens

Result:
  blockedUser successfully swaps in a pool they were never authorized to access.
  The allowlist is completely bypassed.
```
