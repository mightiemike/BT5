### Title
`nonDefaultFeeTierMask` Desynchronization Causes Zero Fees for Non-Default Tier Users on Newly Listed Products — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`nonDefaultFeeTierMask` is a per-tier bitmask that gates whether `getTierFeeRateX18` reads from the `feeRates` mapping or returns a hardcoded default. When `updateTierFeeRates` is called, it sets the tier's bit in `nonDefaultFeeTierMask` and populates `feeRates[tier][productId]` only for products registered at that moment. Any product added afterward has `feeRates[tier][newProductId] == 0` (uninitialized), but the tier bit in `nonDefaultFeeTierMask` is already set. As a result, `getTierFeeRateX18` returns `{makerRateX18: 0, takerRateX18: 0}` for non-default tier users trading the new product, instead of the intended configured rate.

---

### Finding Description

`getTierFeeRateX18` uses `nonDefaultFeeTierMask` to decide which fee source to use:

```solidity
// OffchainExchange.sol L933-L946
function getTierFeeRateX18(uint32 tier, uint32 productId)
    public view returns (FeeRates memory)
{
    if (nonDefaultFeeTierMask & (1 << tier) != 0) {
        return feeRates[tier][productId];   // reads from mapping
    }
    return FeeRates({ makerRateX18: 0, takerRateX18: 200_000_000_000_000 }); // 2 bps default
}
```

`updateTierFeeRates` sets the tier bit and populates `feeRates` only for currently registered products:

```solidity
// OffchainExchange.sol L962-L991
function updateTierFeeRates(IEndpoint.UpdateTierFeeRates memory txn) external onlyEndpoint {
    if (txn.productId == QUOTE_PRODUCT_ID) {
        uint32[] memory spotProductIds = spotEngine.getProductIds();  // snapshot at call time
        uint32[] memory perpProductIds = perpEngine.getProductIds();
        for (uint32 i = 0; i < spotProductIds.length; i++) { ... feeRates[txn.tier][spotProductIds[i]] = ...; }
        for (uint32 i = 0; i < perpProductIds.length; i++) { ... feeRates[txn.tier][perpProductIds[i]] = ...; }
    } else {
        feeRates[txn.tier][txn.productId] = FeeRates(txn.makerRateX18, txn.takerRateX18);
    }
    nonDefaultFeeTierMask |= uint128(1) << txn.tier;  // tier bit set globally, not per-product
}
```

The mask is per-tier, but the fee data is per-(tier, product). When a new product is added via `SpotEngine.addOrUpdateProduct` or `PerpEngine.addOrUpdateProduct` after `updateTierFeeRates` has already been called for a tier, `feeRates[tier][newProductId]` remains zero. Yet `nonDefaultFeeTierMask & (1 << tier) != 0` is true, so `getTierFeeRateX18` returns `{0, 0}` for that (tier, newProduct) pair.

This zero rate flows directly into `applyFee` via `getUserFeeRateWithBuilder`:

```solidity
// OffchainExchange.sol L498-L506
uint32 feeTier = feeTiers[address(uint160(bytes20(sender)))];
FeeRates memory userFeeRates = getTierFeeRateX18(feeTier, productId);
int128 feeRate = taker ? userFeeRates.takerRateX18 : userFeeRates.makerRateX18;
```

And in `applyFee`:

```solidity
// OffchainExchange.sol L556-L560
int128 keepRateX18 = ONE - feeInfo.feeRate;  // feeRate == 0 → keepRate == ONE
int128 newMeteredQuote = (meteredQuote > 0)
    ? meteredQuote.mul(keepRateX18)
    : meteredQuote.div(keepRateX18);
orderInfo.fee = meteredQuote - newMeteredQuote;  // == 0
```

With `feeRate == 0`, `keepRateX18 == ONE`, so `newMeteredQuote == meteredQuote` and `orderInfo.fee == 0`. No fee is charged.

---

### Impact Explanation

Any trader assigned to a non-default fee tier (tier ≥ 1) pays **zero taker and maker fees** when trading a product listed after the tier's fee rates were configured. The protocol collects no fee revenue from these trades. The discrepancy is silent — `getTierFeeRateX18` returns `{0, 0}` without reverting, and `collectedFees` in `marketInfo` is never incremented for these trades.

**Impact: Medium** — systematic protocol fee revenue loss on newly listed products for all non-default tier users. No direct asset theft from user balances, but the protocol's fee accounting is corrupted for the affected (tier, product) pairs.

---

### Likelihood Explanation

**Likelihood: Medium** — the preconditions are routine protocol operations: (1) at least one non-default fee tier must have been configured via `updateTierFeeRates`, and (2) a new product must be listed afterward. Both are expected over the protocol's lifetime. No attacker action is required beyond simply trading the new product while holding a non-default fee tier assignment.

---

### Recommendation

When `addOrUpdateProduct` is called in `SpotEngine` or `PerpEngine`, the `OffchainExchange` should initialize `feeRates[tier][newProductId]` for every tier that already has its bit set in `nonDefaultFeeTierMask`. Alternatively, `getTierFeeRateX18` should treat a zero `feeRates` entry for a non-default tier as "not yet configured" and fall back to the default rate, rather than returning `{0, 0}`.

---

### Proof of Concept

1. Sequencer calls `updateTierFeeRates` with `tier=1`, `productId=QUOTE_PRODUCT_ID`, `takerRateX18=500_000_000_000_000` (5 bps) for all currently listed products. `nonDefaultFeeTierMask` bit 1 is now set.
2. Sequencer calls `updateFeeTier` to assign Alice to tier 1.
3. Admin calls `SpotEngine.addOrUpdateProduct` to list a new token (e.g., `productId=5`). `feeRates[1][5]` is never written; it remains `{0, 0}`.
4. Alice submits a taker order on product 5. `getUserFeeRateWithBuilder` calls `getTierFeeRateX18(1, 5)`. Since `nonDefaultFeeTierMask & (1 << 1) != 0`, it returns `feeRates[1][5]` = `{0, 0}`.
5. `applyFee` computes `orderInfo.fee = 0`. Alice pays no taker fee.
6. A default-tier (tier 0) user trading the same product pays 2 bps taker fee via the hardcoded fallback. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
