### Title
Front-Running `registerMerkleRoot` Enables Double-Claiming of Rolled-Over Airdrop Rewards ‚Äî (`File: core/contracts/Airdrop.sol`)

---

### Summary

`Airdrop.sol` tracks claimed amounts per-week via `claimed[week][sender]`. Because the off-chain system is designed to roll unclaimed rewards from week N into week N+1's Merkle tree, a user can front-run `registerMerkleRoot()` to claim week N, then claim week N+1 (which already includes week N's amount), receiving the same rewards twice.

---

### Finding Description

`Airdrop.sol` maintains a per-week claimed mapping: [1](#0-0) 

The only double-claim guard is a per-week zero-check: [2](#0-1) 

`registerMerkleRoot` advances `pastWeeks` and stores a new root with no pause on claiming: [3](#0-2) 

`claim()` is fully open to any caller at any time: [4](#0-3) 

Because the off-chain system includes unclaimed week-N rewards inside week-N+1's `totalAmount` leaf (the stated rollover design), and because `claimed[N][user]` and `claimed[N+1][user]` are independent slots, claiming week N does not reduce what the user is entitled to claim in week N+1. There is no `claimingPaused` flag, no deadline, and no cross-week deduction to prevent this.

---

### Impact Explanation

A user with unclaimed rewards of amount **X** in week N and new rewards **Y** in week N+1 (where the week-N+1 leaf encodes `totalAmount = X + Y`) can extract **2X + Y** tokens instead of the correct **X + Y**. The excess **X** is drained from the contract's token balance, directly reducing the pool available for other claimants. The impact is concrete token theft proportional to the rolled-over unclaimed balance.

---

### Likelihood Explanation

The attack requires:
1. The off-chain system to include unclaimed rewards in the next week's Merkle tree ‚Äî this is the documented design.
2. The attacker to observe a pending `registerMerkleRoot` transaction in the mempool and submit a `claim` for week N before it confirms.

On an EVM-compatible L2 (Ink Chain), mempool visibility varies, but the attack window also exists without strict front-running: a user can simply claim week N immediately upon noticing that week N+1 is about to be registered (e.g., by monitoring the owner's wallet or a public announcement). Likelihood is **medium-high** given the design dependency.

---

### Recommendation

Add a `claimingPaused` boolean that is set to `true` before `registerMerkleRoot` is called and reset to `false` after the new root is committed. Gate `claim()` on `!claimingPaused`. This mirrors the fix applied in the referenced report (`eaf15c`, `7a5149`) and ensures no claims can be processed against a stale Merkle root while a new distribution is being registered.

Alternatively, enforce a per-week claiming deadline: after the deadline for week N passes, `claim()` for week N is rejected. The deadline is set when `registerMerkleRoot` is called for week N+1, ensuring the snapshot is taken before any new claims against week N are accepted.

---

### Proof of Concept

1. Off-chain system computes week-1 Merkle tree: Alice has `totalAmount = 100`.
2. Alice does **not** claim week 1.
3. Off-chain system computes week-2 Merkle tree: Alice's leaf is `totalAmount = 150` (100 rolled over + 50 new).
4. Owner submits `registerMerkleRoot(2, root2)` ‚Äî visible in the mempool.
5. Alice front-runs with `claim([{week: 1, totalAmount: 100, proof: proof1}])`.
   - `claimed[1][Alice] = 100`; Alice receives **100 tokens**.
6. `registerMerkleRoot(2, root2)` confirms.
7. Alice calls `claim([{week: 2, totalAmount: 150, proof: proof2}])`.
   - `claimed[2][Alice] == 0` ‚Üí passes; Alice receives **150 tokens**.
8. Alice has received **250 tokens** instead of the correct **150 tokens**, stealing **100 tokens** from the contract. [5](#0-4)

```text
    function _verifyProof(
        uint32 week,
        address sender,
        uint256 totalAmount,
        bytes32[] calldata proof
    ) internal {
        require(claimed[week][sender] == 0, "Already claimed.");
        require(
            merkleRoots[week] != bytes32(0),
            "Week hasn't been registered."
        );
        require(
            !ISanctionsList(sanctions).isSanctioned(sender),
            "address is sanctioned."
        );
        bytes32 leaf = keccak256(
            bytes.concat(keccak256(abi.encode(sender, totalAmount)))
        );
        bool isValidLeaf = MerkleProof.verify(proof, merkleRoots[week], leaf);
        require(isValidLeaf, "Invalid proof.");
        claimed[week][sender] = totalAmount;
    }
```
