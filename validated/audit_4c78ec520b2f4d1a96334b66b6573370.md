### Title
`WithdrawCollateralV2` Accepted by Slow-Mode Submission but Silently Dropped at Execution — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

`submitSlowModeTransactionImpl` accepts `WithdrawCollateralV2` transactions without error — charging the slow-mode fee and enqueuing them — but `processSlowModeTransactionImpl` contains no handler for this type and falls through to `revert()`. The revert is silently swallowed by the `try/catch` in `_executeSlowModeTransaction`. The user loses their slow-mode fee, waits three days, and their collateral withdrawal never executes. The censorship-resistance path for `WithdrawCollateralV2` is permanently broken.

---

### Finding Description

`submitSlowModeTransactionImpl` explicitly blocks only `DepositCollateral` (hard `revert()`). Every other unrecognised transaction type falls through to the fee-charging `else` branch:

```
} else {
    chargeSlowModeFee(_getQuote(), sender);
    slowModeFees += SLOW_MODE_FEE;
}
```

`WithdrawCollateralV2` is not in the explicit block-list, so it silently passes through, pays the fee, and is enqueued. [1](#0-0) 

When the enqueued transaction is later executed, `processSlowModeTransactionImpl` is called. It handles `WithdrawCollateral` (V1) at line 217 but has **no case for `WithdrawCollateralV2`**. The function terminates at the final `else { revert(); }`: [2](#0-1) [3](#0-2) 

That revert is caught silently by the `try/catch` wrapper in `_executeSlowModeTransaction`:

```solidity
try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
    // "try return funds now removed"
}
``` [4](#0-3) 

The structural mismatch is the root cause: the submission gate and the execution gate are two separate dispatch tables, and `WithdrawCollateralV2` (enum value 32, added later than `WithdrawCollateral`) was added to `processTransactionImpl` but never back-ported to `processSlowModeTransactionImpl`. [5](#0-4) [6](#0-5) 

The same gap exists for `MintNlp`, `BurnNlp`, `TransferQuote`, `CreateIsolatedSubaccount`, and `CloseIsolatedSubaccount` — all user-facing types present in `processTransactionImpl` but absent from `processSlowModeTransactionImpl`. `WithdrawCollateralV2` is the highest-impact instance because it directly controls collateral egress.

---

### Impact Explanation

A user who is being censored by the sequencer (the exact scenario slow mode is designed for) submits `WithdrawCollateralV2` to withdraw collateral to a custom `sendTo` address. The transaction:

1. Charges and permanently consumes the slow-mode fee.
2. Is enqueued with a three-day `executableAt` delay.
3. Silently reverts on execution — no event, no error, no refund.
4. Leaves the user's on-chain collateral balance unchanged.

The user cannot use `WithdrawCollateral` (V1) as a substitute if they need `sendTo != address(0)` (e.g., withdrawing directly to a cold wallet or a different address). The censorship-resistance guarantee of the slow-mode path is broken for this withdrawal variant. The corrupted state is: **the user's collateral balance is not decremented, the withdrawal pool never receives the funds, and the slow-mode fee is permanently lost**. [7](#0-6) 

---

### Likelihood Explanation

`WithdrawCollateralV2` is the current, actively-used withdrawal path in `processTransactionImpl` (it supports `sendTo` and `feeX18` fields absent from V1). Any user who copies the V2 transaction format into a slow-mode submission — a natural action when the sequencer is censoring them — will trigger the silent failure. No special privileges or unusual conditions are required; the attacker-controlled entry path is the public `submitSlowModeTransaction` function callable by any EOA. [8](#0-7) 

---

### Recommendation

Either:

1. **Add a `WithdrawCollateralV2` handler in `processSlowModeTransactionImpl`** — decode the `SignedWithdrawCollateralV2` struct, apply `validateSender`, and call `clearinghouse.withdrawCollateral` with the decoded `sendTo` (mirroring the V1 slow-mode pattern).

2. **Or explicitly revert in `submitSlowModeTransactionImpl`** for `WithdrawCollateralV2` (as is done for `DepositCollateral`) so users receive an immediate error rather than a silent three-day failure.

Apply the same audit to `MintNlp`, `BurnNlp`, `TransferQuote`, `CreateIsolatedSubaccount`, and `CloseIsolatedSubaccount` — each is accepted by the submission gate but unhandled by the execution gate.

---

### Proof of Concept

```
1. Alice is being censored by the sequencer and cannot execute fast-mode transactions.
2. Alice calls Endpoint.submitSlowModeTransaction(
       abi.encodePacked(
           uint8(TransactionType.WithdrawCollateralV2),
           abi.encode(SignedWithdrawCollateralV2{
               tx: WithdrawCollateralV2{
                   sender: aliceSubaccount,
                   productId: QUOTE_PRODUCT_ID,
                   amount: 1000e6,
                   nonce: currentNonce,
                   sendTo: aliceColdWallet,
                   appendix: 0
               },
               signature: ...,
               feeX18: 0
           })
       )
   );
3. submitSlowModeTransactionImpl: txType is WithdrawCollateralV2, not in any explicit branch
   → falls to else { chargeSlowModeFee(...); slowModeFees += SLOW_MODE_FEE; }
   → transaction enqueued with executableAt = block.timestamp + 3 days.
4. Three days later, sequencer (or Alice) calls executeSlowModeTransaction().
5. _executeSlowModeTransaction calls:
       try this.processSlowModeTransaction(alice, txBytes) {} catch { /* silent */ }
6. processSlowModeTransactionImpl: no case for WithdrawCollateralV2
   → else { revert(); }
7. Revert is caught. No withdrawal. No refund. Slow-mode fee permanently lost.
8. Alice's collateral balance is unchanged; she cannot withdraw to her cold wallet via slow mode.
``` [9](#0-8) [10](#0-9)

    function submitSlowModeTransactionImpl(bytes calldata transaction) public {
        IEndpoint.TransactionType txType = IEndpoint.TransactionType(
            uint8(transaction[0])
        );

        // special case for DepositCollateral because upon
        // slow mode submission we must take custody of the
        // actual funds

        address sender = msg.sender;

        if (txType == IEndpoint.TransactionType.DepositCollateral) {
            revert();
        } else if (txType == IEndpoint.TransactionType.DepositInsurance) {
            IEndpoint.DepositInsurance memory txn = abi.decode(
                transaction[1:],
                (IEndpoint.DepositInsurance)
            );
            require(
                txn.amount >= uint128(SLOW_MODE_FEE),
                ERR_DEPOSIT_TOO_SMALL
            );
            handleDepositTransfer(_getQuote(), sender, uint256(txn.amount));
        } else if (
            txType == IEndpoint.TransactionType.WithdrawInsurance ||
            txType == IEndpoint.TransactionType.DelistProduct ||
            txType == IEndpoint.TransactionType.DumpFees ||
            txType == IEndpoint.TransactionType.RebalanceXWithdraw ||
            txType == IEndpoint.TransactionType.UpdateTierFeeRates ||
            txType == IEndpoint.TransactionType.AddNlpPool ||
            txType == IEndpoint.TransactionType.UpdateNlpPool ||
            txType == IEndpoint.TransactionType.DeleteNlpPool ||
            txType == IEndpoint.TransactionType.ForceRebalanceNlpPool ||
            txType == IEndpoint.TransactionType.NlpProfitShare ||
            txType == IEndpoint.TransactionType.UpdateBuilder
        ) {
            require(sender == owner());
        } else {
            chargeSlowModeFee(_getQuote(), sender);
            slowModeFees += SLOW_MODE_FEE;
        }

        IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
        requireUnsanctioned(sender);
        slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: transaction
        });
        // TODO: to save on costs we could potentially just emit something
        // for now, we can just create a separate loop in the engine that queries the remote
        // sequencer for slow mode transactions, and ignore the possibility of a reorgy attack
        slowModeConfig = _slowModeConfig;
    }

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
