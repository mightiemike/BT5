### Title
Stale `digestToSubaccount` After Close Allows Margin-Free Re-Entry With Reduced Fillable Amount — (`core/contracts/OffchainExchange.sol`)

### Summary
`_tryCloseIsolatedSubaccount` never clears `digestToSubaccount[digest]` or `filledAmounts[digest]`. When a trader resubmits the same order after close, `createIsolatedSubaccount` hits the early-return guard and transfers **no margin** to the isolated subaccount. `matchOrders` then routes fills to that zero-margin subaccount while `_validateOrder` deducts the stale `filledAmounts` from the order's remaining size. The base `isHealthy` implementation is a no-op (`return true`), so the fill is accepted unconditionally, leaving an undercollateralized perp position.

### Finding Description

**Root cause — `_tryCloseIsolatedSubaccount` clears slot-tracking but not digest state:** [1](#0-0) 

The function zeroes `isolatedSubaccountsMask`, `isolatedSubaccounts[parent][id]`, and `parentSubaccounts[subaccount]`, but **never touches** `digestToSubaccount[digest]` or `filledAmounts[digest]`.

**Early-return in `createIsolatedSubaccount` skips margin transfer:** [2](#0-1) 

Because `digestToSubaccount[digest]` is still non-zero after close, the function returns the stale (now orphaned) isolated subaccount immediately — the margin transfer block at lines 1074–1087 is never reached. [3](#0-2) 

**`matchOrders` routes to the zero-margin subaccount:** [4](#0-3) 

**`_validateOrder` deducts stale `filledAmounts` from remaining size:** [5](#0-4) 

If 50 % was filled before close, only 50 % is fillable after re-creation, even though the trader intended a fresh full-size position.

**Health check is a no-op in the base contract:** [6](#0-5) 

No override of `isHealthy` exists anywhere in the scoped codebase (grep confirms all 6 occurrences are in `OffchainExchange.sol`). The `require(isHealthy(...))` guards at lines 826–827 therefore never revert, allowing the fill against a zero-margin subaccount to succeed. [7](#0-6) 

**`filledAmounts` is only ever incremented, never reset:** [8](#0-7) 

### Impact Explanation
After the re-creation path, the isolated subaccount holds a perp position with **zero quote collateral**. Its `parentSubaccounts` entry was cleared during close (line 204), making it orphaned — it cannot be properly settled or closed through normal protocol flows. Any adverse price movement creates an unrecoverable deficit: the protocol has extended a leveraged position with no margin backing it. Additionally, the trader can only fill the residual `order.amount - filledAmounts[digest]` fraction, not the full intended size, causing a permanent reduction in the order's effective capacity for that digest.

### Likelihood Explanation
The path is reachable by any trader who:
1. Creates an isolated subaccount with a non-expired order.
2. Gets partially filled.
3. Has the subaccount closed (via `CloseIsolatedSubaccount` slow-mode tx, liquidation, or sequencer-initiated close).
4. Resubmits the same signed order to `CreateIsolatedSubaccount`.

Steps 1–4 are all supported production transaction types processed by the sequencer. No privileged access is required.

### Recommendation
In `_tryCloseIsolatedSubaccount`, after confirming `balance.amount == 0`, also clear the digest mapping and reset filled amounts for all digests that map to this subaccount. Alternatively, introduce a reverse mapping from `isolatedSubaccount → digest[]` so that on close, `digestToSubaccount[digest]` is deleted and `filledAmounts[digest]` is zeroed. At minimum, `createIsolatedSubaccount` must not return early when `digestToSubaccount[digest]` points to a subaccount whose `parentSubaccounts` entry has been cleared (i.e., a closed subaccount); it should treat that state as equivalent to `bytes32(0)` and proceed with fresh subaccount creation and margin transfer.

### Proof of Concept

```
State-machine test (Hardhat):

1. Alice calls CreateIsolatedSubaccount(order, productId=2, margin=100e18)
   → digestToSubaccount[D] = ISO_1
   → spotBalance(ISO_1) = 100e18, spotBalance(Alice) -= 100e18

2. Sequencer calls matchOrders (50 % fill)
   → filledAmounts[D] = 50e18
   → ISO_1 holds perp position

3. Sequencer calls CloseIsolatedSubaccount(ISO_1)
   → _tryCloseIsolatedSubaccount: balance.amount == 0 → clears mask/slots/parent
   → spotBalance(Alice) += 100e18 (margin returned)
   → digestToSubaccount[D] still = ISO_1   ← BUG
   → filledAmounts[D] still = 50e18        ← BUG

4. Alice calls CreateIsolatedSubaccount(same order)
   → digestToSubaccount[D] != 0 → early return ISO_1
   → NO margin transferred
   → spotBalance(ISO_1) = 0

5. Sequencer calls matchOrders (remaining 50 %)
   → taker.order.sender = digestToSubaccount[D] = ISO_1
   → _validateOrder: order.amount(100) - filledAmounts[D](50) = 50 → valid
   → fill executes; ISO_1 receives perp position
   → isHealthy(ISO_1) → true (no-op)
   → filledAmounts[D] = 100e18

Assert: ISO_1.spotBalance == 0, ISO_1.perpBalance != 0, parentSubaccounts[ISO_1] == 0
→ Undercollateralized orphaned position confirmed.

        uint32 productId = RiskHelper.getIsolatedProductId(subaccount);
        if (productId == 0) {
            return;
        }
        IPerpEngine.Balance memory balance = perpEngine.getBalance(
            productId,
            subaccount
        );
        if (balance.amount == 0) {
            uint8 id = RiskHelper.getIsolatedId(subaccount);
            address addr = address(uint160(bytes20(subaccount)));
            bytes32 parent = parentSubaccounts[subaccount];
            if (balance.vQuoteBalance != 0) {
                perpEngine.updateBalance(
                    productId,
                    subaccount,
                    0,
                    -balance.vQuoteBalance
                );
                perpEngine.updateBalance(
                    productId,
                    parent,
                    0,
                    balance.vQuoteBalance
                );
            }
            int128 quoteBalance = spotEngine
                .getBalance(QUOTE_PRODUCT_ID, subaccount)
                .amount;
            if (quoteBalance != 0) {
                spotEngine.updateBalance(

                    subaccount,

                    parent,
                    quoteBalance

            isolatedSubaccounts[parent][id] = bytes32(0);

        }
    }
