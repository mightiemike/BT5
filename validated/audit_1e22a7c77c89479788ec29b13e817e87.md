### Title
Fee-Less Fast Withdrawals via Integer Truncation in `fastWithdrawalFeeAmount` — (`File: core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`fastWithdrawalFeeAmount` in `BaseWithdrawPool` computes the fast-withdrawal fee using fixed-point multiplication followed by a final integer division to convert back to native token decimals. For tokens with fewer than 18 decimals (e.g., USDC with 6 decimals), the final division `feeX18 / int128(multiplier)` truncates toward zero in Solidity. When `feeX18 < multiplier`, the function returns `0`, and `submitFastWithdrawal` proceeds without collecting any fee — an exact structural analog to the AAVE `calculateLoanOriginationFee` / `wadMul` truncation bug.

---

### Finding Description

`fastWithdrawalFeeAmount` is defined at `BaseWithdrawPool.sol:134–149`:

```solidity
function fastWithdrawalFeeAmount(
    IERC20Base token,
    uint32 productId,
    uint128 amount
) public view returns (int128) {
    uint8 decimals = token.decimals();
    require(decimals <= MAX_DECIMALS);
    int256 multiplier = int256(10**(MAX_DECIMALS - uint8(decimals)));
    int128 amountX18 = int128(amount) * int128(multiplier);

    int128 proportionalFeeX18 = FAST_WITHDRAWAL_FEE_RATE.mul(amountX18);
    int128 minFeeX18 = 5 * spotEngine().getConfig(productId).withdrawFeeX18;

    int128 feeX18 = MathHelper.max(proportionalFeeX18, minFeeX18);
    return feeX18 / int128(multiplier);   // ← truncating integer division
}
```

**Constants involved:**
- `FAST_WITHDRAWAL_FEE_RATE = 1_000_000_000_000_000` (1e15, i.e. 0.1%) — `Constants.sol:25`
- `MAX_DECIMALS = 18` — `Constants.sol:19`
- `MathSD21x18.mul` computes `(int256(x) * y) / ONE_X18` with truncation toward zero — `MathSD21x18.sol:54–59`

**Concrete arithmetic for USDC (6 decimals):**

1. `multiplier = 10^(18−6) = 10^12`
2. `amountX18 = amount × 10^12`
3. `proportionalFeeX18 = (1e15 × amount × 10^12) / 1e18 = amount × 10^9`
4. Final return: `feeX18 / 10^12`

For `proportionalFeeX18 < 10^12` → `amount × 10^9 < 10^12` → **`amount < 1000` USDC base units (< 0.001 USDC)**, the proportional fee is zero after division.

The `minFeeX18` floor (`5 × withdrawFeeX18`) is also subject to the same final division. If `withdrawFeeX18` is configured below `2×10^11` (i.e., below $0.0000002 in X18 representation), then `minFeeX18 / 10^12 = 0` as well, making the entire returned fee zero regardless of the minimum fee configuration.

**No zero-fee guard exists.** `submitFastWithdrawal` at `BaseWithdrawPool.sol:81–114` uses the returned fee directly:

```solidity
int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

if (sendTo == msg.sender) {
    require(transferAmount > uint128(fee), "Fee larger than balance");
    transferAmount -= uint128(fee);   // subtracts 0
} else {
    safeTransferFrom(token, msg.sender, uint128(fee));  // transfers 0
}
fees[productId] += fee;   // accumulates 0
```

When `fee == 0`, the user receives the full fast-withdrawal amount with no fee deducted and no fee credited to the protocol.

---

### Impact Explanation

A user who submits a fast withdrawal for a qualifying small amount (or for any amount if `withdrawFeeX18` is configured small enough) receives the fast-withdrawal service — immediate on-chain settlement bypassing the sequencer queue — without paying the intended 0.1% fee. The protocol's `fees[productId]` accumulator is not incremented, permanently losing that revenue. The broken invariant is: **every fast withdrawal must pay a non-zero fee**.

---

### Likelihood Explanation

`submitFastWithdrawal` is a `public` function with no access control beyond signature verification of the withdrawal transaction itself. Any user who has signed a `WithdrawCollateral` or `WithdrawCollateralV2` transaction can call it directly. For USDC (the primary quote asset on Nado), the zero-fee threshold is amounts below 1000 base units (< 0.001 USDC). While individually small, the path is fully permissionless and the condition is deterministic — any caller can verify the threshold off-chain and craft qualifying withdrawals. If `withdrawFeeX18` is small, the threshold scales up and larger amounts become fee-less.

---

### Recommendation

Add a zero-fee guard in `fastWithdrawalFeeAmount` or in `submitFastWithdrawal` that reverts when the computed fee is zero but the withdrawal amount is non-zero:

```solidity
int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);
require(fee > 0, "Fee rounds to zero");
```

Alternatively, perform the fee computation entirely in the X18 domain and only convert back to native decimals for the actual transfer, ensuring the minimum representable fee is always at least 1 native unit.

---

### Proof of Concept

**Setup:** USDC token (6 decimals), `FAST_WITHDRAWAL_FEE_RATE = 1e15`, `MAX_DECIMALS = 18`.

**Call:** `fastWithdrawalFeeAmount(usdc, productId, 999)` (999 USDC base units = $0.000999)

**Trace:**
```
multiplier      = 10^12
amountX18       = 999 * 10^12 = 999_000_000_000_000
proportionalFeeX18 = (1e15 * 999_000_000_000_000) / 1e18
                   = 999_000_000_000_000_000_000_000_000_000 / 1e18
                   = 999_000_000_000   (< 10^12)
minFeeX18       = 5 * withdrawFeeX18  (assume small, e.g. 1e10 → minFeeX18 = 5e10)
feeX18          = max(999_000_000_000, 5e10) = 999_000_000_000
return          = 999_000_000_000 / 10^12 = 0  ← integer truncation
```

**Result:** `submitFastWithdrawal` proceeds with `fee = 0`. The user receives 999 USDC base units with no fee deducted. `fees[productId]` is unchanged. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
