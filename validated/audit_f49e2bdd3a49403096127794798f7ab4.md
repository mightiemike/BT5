### Title
NEAR Storage Deposit Tokens Permanently Locked After GC Frees Storage — (`contract/src/lib.rs`)

---

### Summary

`submit_blocks` accepts NEAR tokens to cover on-chain storage costs. When the built-in garbage collector (`run_mainchain_gc`) later frees that storage, the NEAR tokens originally paid for it remain in the contract's account balance with no withdrawal mechanism. The contract exposes no admin or owner function to recover these accumulated funds.

---

### Finding Description

`submit_blocks` is `#[payable]` and measures net storage growth to determine how much deposit to keep:

```rust
let initial_storage = env::storage_usage();
// ... insert headers ...
self.run_mainchain_gc(num_of_headers);
let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());
``` [1](#0-0) 

`saturating_sub` means that when GC removes more bytes than the new headers added, `diff_storage_usage` collapses to `0`, `required_deposit` is `0`, and the **current caller's** entire attached deposit is refunded:

```rust
let refund = amount.saturating_sub(required_deposit);
if refund > NearToken::from_near(0) {
    Promise::new(env::predecessor_account_id()).transfer(refund).into()
``` [2](#0-1) 

The NEAR tokens paid by **previous callers** for the storage that GC just freed are never returned to anyone. They remain in the contract's account balance. The contract defines no `withdraw`, `transfer`, or admin-recovery function anywhere in its public API. [3](#0-2) 

The GC is called unconditionally on every `submit_blocks` invocation:

```rust
self.run_mainchain_gc(num_of_headers);
``` [4](#0-3) 

`run_mainchain_gc` removes entries from `mainchain_height_to_header`, `mainchain_header_to_height`, and `headers_pool`, freeing the storage bytes that were paid for by earlier relayer calls: [5](#0-4) 

The contract's role system (`DAO`, `PauseManager`, `CodeDeployer`, etc.) contains no role or function that can transfer the contract's native NEAR balance out. [6](#0-5) 

---

### Impact Explanation

Every time GC frees storage that was paid for by a previous relayer, those NEAR tokens are permanently locked in the contract. With the recommended `gc_threshold = 52704` (one year of Bitcoin blocks), the contract continuously cycles through paying-and-freeing storage. The cumulative locked balance grows monotonically over the contract's lifetime with no recovery path. At NEAR's current storage cost (~1 NEAR per 10 KB) and Bitcoin's ~80-byte headers, the locked amount is material over months of operation.

---

### Likelihood Explanation

This occurs during every normal `submit_blocks` call once the chain has grown past `gc_threshold`. The test suite explicitly documents this steady-state behavior: *"After 3 batches with deposit (~12+85+85+85=267 total), GC kicks in and subsequent batches can be submitted for free."* [7](#0-6) 

Any unprivileged relayer (the primary production entry point) triggers this path on every call. Likelihood is high.

---

### Recommendation

Add an owner/DAO-gated withdrawal function that transfers the contract's available balance (total balance minus current storage staking requirement) to a designated recipient:

```rust
pub fn withdraw_available_balance(&mut self, recipient: AccountId) -> Promise {
    // Only DAO role may call this
    let storage_cost = env::storage_byte_cost()
        .saturating_mul(env::storage_usage().into());
    let available = env::account_balance().saturating_sub(storage_cost);
    require!(available > NearToken::from_near(0), "No available balance");
    Promise::new(recipient).transfer(available)
}
```

Alternatively, track each relayer's cumulative storage deposit and refund them proportionally when GC frees their storage.

---

### Proof of Concept

1. Relayer A calls `submit_blocks` with 100 headers, attaching 1 NEAR. The contract stores the headers; 1 NEAR is retained as storage deposit.
2. Relayer B calls `submit_blocks` with 100 new headers (no deposit attached or zero deposit).
   - GC removes the 100 headers from step 1, freeing their storage.
   - `diff_storage_usage = saturating_sub(new_usage, old_usage) = 0` (net zero or negative).
   - `required_deposit = 0`; Relayer B's deposit (if any) is fully refunded.
3. Relayer A's 1 NEAR is now in the contract's available balance with no function to retrieve it.
4. Repeat indefinitely. The contract's locked NEAR balance grows with each GC cycle.

The existing integration test `test_payment_on_block_submission` already demonstrates step 2 (zero-deposit submission succeeds after GC), confirming the freed-storage path is reachable in production. [8](#0-7)

    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
