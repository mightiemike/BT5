### Title
Division-by-Zero in `PerpEngine.socializeSubaccount` Permanently Blocks Finalization of Underwater Subaccounts — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` unconditionally divides by `state.openInterest` when insurance is insufficient to cover a negative `vQuoteBalance`. No guard exists for the case where `state.openInterest == 0`. `MathSD21x18.div` explicitly reverts on a zero denominator, causing the entire finalization path to revert and permanently trapping the subaccount in an un-finalizable state.

---

### Finding Description

In `PerpEngine.socializeSubaccount`, after insurance is exhausted, the remaining negative `vQuoteBalance` is socialized across open-interest holders:

```solidity
if (balance.vQuoteBalance < 0) {
    int128 fundingPerShare = -balance.vQuoteBalance.div(
        state.openInterest   // ← no zero-check
    );
``` [1](#0-0) 

`MathSD21x18.div` enforces a hard revert when the denominator is zero:

```solidity
function div(int128 x, int128 y) internal pure returns (int128) {
    unchecked {
        require(y != 0, ERR_DIV_BY_ZERO);
``` [2](#0-1) 

`openInterest` is the sum of absolute position sizes across all participants, updated in `_updateBalance`:

```solidity
state.openInterest -= balance.amount.abs();   // pre-update
...
state.openInterest += balance.amount;          // post-update (long)
state.openInterest -= balance.amount;          // post-update (short)
``` [3](#0-2) 

`openInterest` reaches zero legitimately when every participant has closed their position in a given product. A subaccount can simultaneously hold `balance.amount == 0` (required by `_finalizeSubaccount`) yet retain a negative `vQuoteBalance` — the accumulated unrealized loss from trades that were already closed. These two conditions are fully independent.

The call chain that triggers the revert is:

```
liquidateSubaccountImpl
  └─ _finalizeSubaccount          (txn.productId == type(uint32).max)
       └─ perpEngine.socializeSubaccount
            └─ MathSD21x18.div(vQuoteBalance, 0)  ← REVERT "DBZ"
``` [4](#0-3) [5](#0-4) 

---

### Impact Explanation

Once the revert is triggered:

- `_finalizeSubaccount` always reverts for that subaccount — there is no alternative code path.
- The negative `vQuoteBalance` (bad debt) can never be cleared.
- The insurance fund balance (`insurance` storage variable) is never decremented or returned, leaving it in a state that cannot be reconciled.
- Any liquidator attempting to finalize the subaccount will always receive a revert, making the subaccount permanently un-finalizable.

This satisfies the Critical scope: locked funds and bad debt that cannot be cleared.

---

### Likelihood Explanation

The preconditions are reachable through normal protocol operation:

1. A product experiences a period of low activity; all participants close their positions → `openInterest` drops to 0.
2. One participant closed their position at a loss, leaving `balance.amount == 0` and `vQuoteBalance < 0`.
3. That participant's account becomes undercollateralized (e.g., due to funding payments or spot losses).
4. A liquidator calls `liquidateSubaccountImpl` with `productId == type(uint32).max` to finalize.
5. Insurance is insufficient to fully cover `vQuoteBalance` → socialization branch is entered → division by zero → permanent revert.

No privileged access, oracle manipulation, or external dependency failure is required.

---

### Recommendation

Add a zero-check for `state.openInterest` before the division. When `openInterest == 0` there are no other participants to socialize the loss to; the remaining bad debt should be absorbed by the insurance fund or written off directly:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No open interest to socialize against; write off the bad debt
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(
            state.openInterest
        );
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

The `updateStates` function already applies this same guard pattern (`if (state.openInterest == 0) { continue; }`) and should serve as the reference fix. [6](#0-5)

### Proof of Concept

```solidity
// 1. Deploy protocol, add perpProduct (productId = 2)

// 3. Price drops; Alice closes position at a loss:
//      balance.amount = 0, balance.vQuoteBalance = -500e18
// 4. Bob closes position → openInterest = 0
// 5. Alice's spot USDC balance goes to 0 (e.g., via funding payments)
//    → Alice is under maintenance margin
// 6. Liquidator calls liquidateSubaccountImpl with productId = type(uint32).max
//    → _finalizeSubaccount is entered
//    → insurance (e.g., 100e18) < 500e18 needed
//    → socializeSubaccount called
//    → balance.vQuoteBalance still < 0 after insurance cover
//    → fundingPerShare = -(-400e18).div(0)  ← REVERT "DBZ"
// 7. assert: call always reverts; Alice's subaccount is permanently un-finalizable
```
