### Title
Unvalidated User-Supplied `oraclePriceX18` in NLP Mint/Burn Enables Incorrect NLP Token Calculations - (File: `core/contracts/EndpointTx.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

`MintNlp` and `BurnNlp` transactions carry a user-signed `oraclePriceX18` field that is accepted and applied on-chain without any comparison to the current stored price (`priceX18[NLP_PRODUCT_ID]`). Because the NLP amount minted or the quote amount returned on burn is computed directly from this field, a user who signs with a stale or manipulated price receives an incorrect token amount, extracting value from the NLP pool at the expense of other NLP holders.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, both the `MintNlp` and `BurnNlp` branches decode a `SignedMintNlp` / `SignedBurnNlp` struct that embeds an `oraclePriceX18` field supplied by the signer. The code validates the user's EIP-712 signature over the full struct (including the price), then immediately writes the price into `priceX18[NLP_PRODUCT_ID]` and forwards it to `Clearinghouse.mintNlp` / `burnNlp`:

```
priceX18[NLP_PRODUCT_ID] = signedTx.oraclePriceX18;   // EndpointTx.sol ~547 / ~567
clearinghouse.mintNlp(signedTx.tx, signedTx.oraclePriceX18, ...);
``` [1](#0-0) [2](#0-1) 

Inside `Clearinghouse.mintNlp`, the NLP amount is computed as:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
``` [3](#0-2) 

Inside `Clearinghouse.burnNlp`, the quote returned is:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
``` [4](#0-3) 

At no point is `oraclePriceX18` compared against the previously stored `priceX18[NLP_PRODUCT_ID]`, bounded by a maximum deviation, or checked for staleness. The contract unconditionally overwrites the stored price and uses the caller-supplied value for all arithmetic. [5](#0-4) 

The on-chain price for all other products is updated exclusively through sequencer-submitted `UpdatePrice` transactions and stored in `priceX18` / `_risk().value[productId].priceX18`. [6](#0-5) [7](#0-6) 

For NLP specifically, the price is instead taken from the user's signed transaction, creating a divergence: every other product's price is sequencer-controlled and stored, while the NLP price is user-controlled per-transaction with no on-chain guard.

---

### Impact Explanation

**For `BurnNlp`**: a user who signs with an inflated `oraclePriceX18` (e.g., 2× the true NLP NAV) receives twice the quote amount they are entitled to. The excess quote is drawn from the NLP pool's collective balance, directly diluting all other NLP holders.

**For `MintNlp`**: a user who signs with a deflated `oraclePriceX18` (e.g., 0.5× the true NAV) receives twice as many NLP tokens for the same quote deposit, diluting existing NLP holders' proportional claim on pool assets.

The corrupted state delta is: `spotEngine.balance[QUOTE_PRODUCT_ID][sender]` and `spotEngine.balance[NLP_PRODUCT_ID][sender]` are set to incorrect values, with the corresponding NLP pool subaccounts (`nlpPools[i].subaccount`) receiving incorrect rebalance amounts. [8](#0-7) [9](#0-8) 

---

### Likelihood Explanation

The user signs the full `SignedMintNlp` / `SignedBurnNlp` struct, including `oraclePriceX18`. The sequencer is the off-chain gatekeeper that is expected to reject transactions whose embedded price deviates from the true NLP NAV. However:

1. There is no on-chain enforcement of this expectation — the contract applies whatever price is in the signed struct.
2. The sequencer may process a user's transaction that was signed when the NLP price was different (e.g., signed during a price spike, submitted after the price reverted), without any on-chain staleness window to catch it.
3. This is structurally identical to the original report's root cause: the system relies on an off-chain actor (sequencer here, oracle multisig in the original) to keep prices fresh, with no on-chain rejection of stale values.

The likelihood is conditional on sequencer behavior, but the missing on-chain guard means the protocol has no defense-in-depth if the sequencer fails to validate prices in user-signed NLP transactions.

---

### Recommendation

Add an on-chain bound check in `processTransactionImpl` before forwarding `oraclePriceX18` to the clearinghouse. Specifically, require that the user-supplied price does not deviate from the current stored `priceX18[NLP_PRODUCT_ID]` by more than a configurable tolerance (e.g., ±5%):

```solidity
int128 storedPrice = priceX18[NLP_PRODUCT_ID];
require(
    signedTx.oraclePriceX18 >= storedPrice.mul(MAX_PRICE_DEVIATION_LOWER) &&
    signedTx.oraclePriceX18 <= storedPrice.mul(MAX_PRICE_DEVIATION_UPPER),
    ERR_INVALID_PRICE
);
```

This mirrors the recommendation in the original report: store the latest oracle update and reject operations if the price has not been updated within a reasonable bound.

---

### Proof of Concept

1. The current on-chain `priceX18[NLP_PRODUCT_ID]` is `1e18` (1 USDC per NLP token).
2. Attacker holds 1000 NLP tokens and constructs a `SignedBurnNlp` with `oraclePriceX18 = 2e18` (2 USDC per NLP).
3. Attacker signs the struct and submits to the sequencer.
4. Sequencer (without price validation) submits the transaction on-chain.
5. `EndpointTx` writes `priceX18[NLP_PRODUCT_ID] = 2e18` and calls `clearinghouse.burnNlp` with `oraclePriceX18 = 2e18`. [10](#0-9) 
6. `burnNlp` computes `quoteAmount = 1000 * 2e18 / 1e18 = 2000 USDC` and credits the attacker. [11](#0-10) 
7. Attacker receives 2000 USDC instead of the correct 1000 USDC, extracting 1000 USDC from the NLP pool at the expense of other NLP holders.
