### Title
Cost Undercharge via Truncating Integer Division in Multiply Quadratic Cost Term — (File: `src/more_ops.rs`)

---

### Summary

The `op_multiply` operator and the `op_unknown` cost-function-2 path both compute the quadratic component of their execution cost using integer floor division, systematically truncating the charged cost downward in the attacker's favor. This is the direct analog of the HomoraBank rounding bug: a division that should round up (ceiling) instead rounds down (floor), causing the protocol to undercharge for computation. An attacker supplying crafted CLVM bytes can exploit this to execute more computation than the declared cost budget permits.

---

### Finding Description

In `src/more_ops.rs`, the cost model for big-integer multiplication is:

```
cost_per_step = MUL_COST_PER_OP
              + (l0 + l1) * MUL_LINEAR_COST_PER_BYTE
              + (l0 * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER   ← truncates
```

where `MUL_SQUARE_COST_PER_BYTE_DIVIDER = 128`.

The quadratic term `(l0 * l1) / 128` uses Rust's integer `/` operator, which is floor division (truncation toward zero). Whenever `(l0 * l1) % 128 != 0`, the charged cost is strictly less than the true cost. The correct expression to charge in the protocol's favor is ceiling division: `(l0 * l1 + 127) / 128`.

This truncation appears in three code paths inside `op_multiply`: [1](#0-0) [2](#0-1) [3](#0-2) 

And identically in the `op_unknown` cost-function-2 branch: [4](#0-3) 

The constant and the two sites that define the divider: [5](#0-4) 

In `op_unknown`, the truncated cost is subsequently multiplied by `cost_multiplier + 1` (attacker-controlled, up to ~46 million before the `u32::MAX` cap triggers): [6](#0-5) 

This amplifies the per-call undercharge: a 1-unit truncation error before multiplication becomes up to ~46.7 million cost units of undercharge after multiplication.

---

### Impact Explanation

**Corrupted value**: the `Cost` field of the `Reduction` returned by `op_multiply` and `op_unknown`. It is strictly less than the true computational cost whenever `(l0 * l1) % 128 != 0`.

**For `op_multiply`**: the undercharge is at most 1 cost unit per pairwise multiplication step. With `max_cost = 11 000 000 000` and a minimum per-step cost of ~897, an attacker can drive ~12 million steps, accumulating up to ~12 million cost units of undercharge (~0.11 % of the budget). This allows the attacker to execute slightly more computation than the declared budget permits.

**For `op_unknown` (cost_function 2)**: the truncation error is amplified by `cost_multiplier + 1`. The maximum `cost_multiplier` that keeps the final cost below `u32::MAX` (the hard cap at line 262) is approximately 46 663 716 (when base cost = 92). A single call can therefore undercharge by up to ~46.7 million cost units. With two such calls fitting inside `max_cost`, the total undercharge reaches ~93 million cost units (~0.85 % of the budget).

The net effect is that an attacker can execute more big-integer multiplications than the cost budget should allow, weakening the DoS protection that the cost model is designed to enforce. Because every full node enforces the same `max_cost` limit, systematic undercharging shifts the true resource consumption above the declared limit, degrading the consensus-level guarantee that no program exceeds its declared cost.

---

### Likelihood Explanation

The condition `(l0 * l1) % 128 != 0` is satisfied for the overwhelming majority of operand-size pairs. For example, any pair where both operands are 1–11 bytes gives `l0 * l1 ≤ 121 < 128`, so the quadratic term is charged as 0 instead of 1. An attacker needs only to submit a CLVM program containing the `*` opcode (opcode byte `0x12`) with operands of chosen byte lengths; no special privilege or key material is required.

---

### Recommendation

Replace every instance of the truncating division with ceiling division:

```rust
// Before (truncates in attacker's favor):
cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;

// After (rounds up in protocol's favor):
cost += (l0 as Cost * l1 + MUL_SQUARE_COST_PER_BYTE_DIVIDER - 1)
        / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
```

Apply this fix to all three paths in `op_multiply` (lines 616, 624, 644) and to the `op_unknown` cost-function-2 branch (line 238).

---

### Proof of Concept

**Trigger for `op_multiply`** — operands of 11 bytes each:

- `l0 = 11`, `l1 = 11`
- Quadratic term charged: `(11 × 11) / 128 = 121 / 128 = 0`
- Correct (ceiling) value: `⌈121/128⌉ = 1`
- Undercharge: **1 cost unit per step**

A CLVM program that chains `(* A B C D …)` with 11-byte atoms repeats this undercharge at every pairwise step. Over ~12 million steps the cumulative undercharge reaches ~12 million cost units.

**Trigger for `op_unknown` amplification** — opcode bytes encoding `cost_function = 2`, `cost_multiplier = 46 663 716`, two 1-byte arguments:

- Base cost before multiplication: 92 (MUL_BASE_COST, no args beyond first)
- Quadratic term: `(1 × 1) / 128 = 0` (should be 1)
- Charged cost before multiplication: 92; true cost: 93
- After multiplication: charged = `92 × 46 663 717 ≈ 4 293 M`; true = `93 × 46 663 717 ≈ 4 339 M`
- Undercharge per call: **~46.7 million cost units**
- Two such calls fit within `max_cost = 11 B`; total undercharge: **~93 million cost units**
