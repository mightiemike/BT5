### Title
Transaction Inclusion Verification Permanently Fails for Single-Transaction Bitcoin Blocks Due to Unconditional Empty-Proof Rejection — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (and `verify_transaction_inclusion_v2`, which delegates to it) unconditionally panics when the supplied Merkle proof is empty. For a Bitcoin block containing exactly one transaction, the Merkle root *is* the transaction hash and the correct proof is the empty vector. The underlying `compute_root_from_merkle_proof` already handles this correctly, but the guard fires first, making it permanently impossible for any downstream consumer to verify a transaction that was mined in a single-transaction block.

This is the direct analog of the Beanstalk finding: just as EIP-712 `ECDSA.recover` silently rejects every valid signature produced by a contract wallet (a legitimate but unsupported signer class), the empty-proof guard silently rejects every valid inclusion proof for a legitimate but unsupported block class.

---

### Finding Description

`verify_transaction_inclusion` contains:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0) 

`compute_root_from_merkle_proof` iterates over the proof slice; with an empty slice the loop body never executes and `current_hash` (the transaction hash) is returned unchanged:

```rust
for proof_hash in merkle_proof {   // never entered when proof is empty
    ...
}
current_hash   // returns transaction_hash as-is
``` [2](#0-1) 

For a block whose transaction list has exactly one entry, Bitcoin's Merkle construction sets `merkle_root = txid`. Therefore `compute_root_from_merkle_proof(txid, 0, &[]) == merkle_root` is mathematically true, but the `require!` fires before that comparison is ever reached.

`verify_transaction_inclusion_v2` (the recommended replacement) calls the deprecated function after its own coinbase-proof check:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

For a single-transaction block the coinbase proof is also empty, but `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[])` returns `coinbase_tx_id`, which equals `merkle_root`, so the v2 coinbase check passes. The call then falls through to `verify_transaction_inclusion`, which panics on the empty `merkle_proof`. Neither public entry point can succeed. [4](#0-3) 

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` to gate an action (e.g., releasing bridged funds, crediting a deposit, settling a bet) on a transaction mined in a single-transaction block will receive a permanent panic. Because the block is already finalized on Bitcoin and the light-client state is correct, there is no alternative path: the proof is valid, the block is stored, but the verification call will never return `true`. Assets gated behind that call are permanently inaccessible — the exact impact class of the reference report.

---

### Likelihood Explanation

Single-transaction blocks (coinbase only) are rare on mainnet today but are protocol-valid and have occurred historically. They are common in regtest/signet environments used for integration testing. Any bridge or application that supports the full Bitcoin block space — including early-chain or test-chain blocks — is exposed. The entry path requires no privilege: any unprivileged NEAR caller can invoke `verify_transaction_inclusion_v2` with a valid `ProofArgsV2` whose `merkle_proof` and `coinbase_merkle_proof` are both empty.

---

### Recommendation

Remove the blanket empty-proof guard and instead handle the single-transaction case explicitly:

```rust
// Single-transaction block: merkle_root == tx_id, proof is empty by definition.
if args.merkle_proof.is_empty() {
    return args.tx_id == header.block_header.merkle_root;
}
```

Alternatively, remove the guard entirely — `compute_root_from_merkle_proof` already returns the correct result for an empty proof, so the subsequent equality check is sufficient.

---

### Proof of Concept

1. A Bitcoin block is mined containing only the coinbase transaction `T` with hash `H`. Its `merkle_root` field equals `H`.
2. The relayer submits the block header; it is accepted and stored in `headers_pool`.
3. A downstream NEAR contract calls `verify_transaction_inclusion_v2` with:
   - `tx_id = H`, `tx_block_blockhash = <block hash>`, `tx_index = 0`
   - `merkle_proof = []`, `coinbase_tx_id = H`, `coinbase_merkle_proof = []`
4. The coinbase check inside v2 computes `compute_root_from_merkle_proof(H, 0, &[]) == H == merkle_root` → passes.
5. The call delegates to `verify_transaction_inclusion`; `require!(!args.merkle_proof.is_empty(), …)` fires → **panic**.
6. The downstream contract's action (fund release, credit, etc.) is permanently blocked despite the transaction being genuinely included and the light-client state being correct. [5](#0-4) [6](#0-5)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
