### Title
Permanently Locked NEAR Storage Deposits After GC — No Withdrawal Mechanism - (`contract/src/lib.rs`)

### Summary

`submit_blocks` requires callers to attach NEAR tokens to cover on-chain storage costs for new block headers. When `run_mainchain_gc` later removes those headers (freeing storage), the NEAR tokens originally deposited to cover that storage remain permanently locked inside the contract. No withdrawal or reclaim function exists anywhere in the contract, so every GC cycle silently converts prior depositors' NEAR into irrecoverable contract balance.

### Finding Description

**Root cause — storage deposit accounting only covers the current caller, never past depositors.**

`submit_blocks` measures storage before and after its own execution and refunds the current caller's excess deposit: [1](#0-0) 

The critical sequence inside `submit_blocks` is:

1. `initial_storage = env::storage_usage()` — snapshot taken before any mutation.
2. New headers are inserted (storage grows).
3. `run_mainchain_gc(num_of_headers)` is called — old headers are deleted (storage shrinks).
4. `diff_storage_usage = env::storage_usage().saturating_sub(initial_storage)` — only the **net** change is charged to the current caller.

This means the current caller is correctly charged (or refunded) for the net storage delta. However, the NEAR tokens deposited by **previous** callers to cover the storage of the blocks that GC just deleted are never returned to anyone. They remain in the contract's account balance as unspendable excess.

`run_mainchain_gc` is also a standalone public entry point: [2](#0-1) 

It is not `#[payable]`, carries no refund logic, and is callable by any unprivileged account (when the contract is not paused). Calling it standalone frees storage and locks the corresponding prior deposits with no recourse.

The contract exposes no `withdraw`, `claim_storage_refund`, or equivalent function anywhere in `lib.rs`. Once NEAR enters the contract as a storage deposit and the corresponding storage is later freed by GC, those tokens are permanently trapped.

### Impact Explanation

Every GC cycle — a routine, expected operation — converts prior relayers' NEAR deposits into permanently irrecoverable contract balance. With the recommended `gc_threshold = 52704` blocks and the test-observed deposit rate of `STORAGE_DEPOSIT_PER_BLOCK = 500 milliNEAR`: [3](#0-2) 

a single full GC cycle can lock on the order of **~26,000 NEAR** with no path to recovery. This is a direct, permanent loss of relayer funds — an exact analog to the surplus collateral permanently stuck in `CollSurplusPool` in H-01.

### Likelihood Explanation

GC is not an edge case; it is the contract's designed steady-state behaviour. Every `submit_blocks` call internally invokes `run_mainchain_gc`, and any unprivileged account can also call it directly. The loss is therefore triggered automatically and continuously during normal protocol operation, requiring no special attacker capability.

### Recommendation

Add a privileged withdrawal function (callable by the DAO or contract owner) that transfers the contract's excess balance — i.e., `env::account_balance() - env::storage_usage() * env::storage_byte_cost()` — to a designated treasury or back to a tracked depositor registry. Alternatively, track per-depositor storage contributions and issue pro-rata refunds when GC frees the corresponding entries.

### Proof of Concept

1. Relayer A calls `submit_blocks` with 100 headers, attaching 50 NEAR. Storage grows; 50 NEAR is consumed.
2. Relayer B calls `submit_blocks` with 100 more headers, attaching 50 NEAR. GC threshold is exceeded; `run_mainchain_gc` deletes Relayer A's 100 headers inside the same call. Net storage delta for Relayer B is near zero, so Relayer B is refunded most of their 50 NEAR.
3. Relayer A's 50 NEAR remains in the contract. The storage it covered is gone. No function exists to reclaim it.
4. Alternatively, any unprivileged account calls `run_mainchain_gc(52704)` directly. Storage is freed; all prior depositors' NEAR is locked with zero recourse.

The absence of any withdrawal path is confirmed by exhaustive inspection of `contract/src/lib.rs`: the only outbound `Promise::transfer` is the per-call excess refund inside `submit_blocks` itself. [4](#0-3)

    #[pause(except(roles(Role::UnrestrictedRunGC)))]
    pub fn run_mainchain_gc(&mut self, batch_size: u64) {
        let initial_blockheader = self
            .headers_pool
            .get(&self.mainchain_initial_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let tip_blockheader = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let amount_of_headers_we_store =
            tip_blockheader.block_height - initial_blockheader.block_height + 1;

        if amount_of_headers_we_store > self.gc_threshold {
            let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
            let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);

            let start_removal_height = initial_blockheader.block_height;
            let end_removal_height = initial_blockheader.block_height + selected_amount_to_remove;
            env::log_str(&format!(
                "Num of blocks to remove {selected_amount_to_remove}"
            ));

            for height in start_removal_height..end_removal_height {
                let blockhash = &self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

                self.remove_block_header(blockhash);
                self.mainchain_height_to_header.remove(&height);
            }

            self.mainchain_initial_blockhash = self
                .mainchain_height_to_header
                .get(&end_removal_height)
                .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        }
    }
