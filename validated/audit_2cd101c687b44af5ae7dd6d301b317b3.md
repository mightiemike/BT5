### Title
`env::block_timestamp_ms()` Used as Sole "Local Time" Reference in `check_pow` Future-Timestamp Guard — (`contract/src/bitcoin.rs`, `contract/src/dogecoin.rs`, `contract/src/litecoin.rs`, `contract/src/zcash.rs`)

---

### Summary

All four chain-specific `check_pow` implementations use `env::block_timestamp_ms()` — the NEAR block timestamp, set by the current NEAR block producer — as the sole "local time" reference to enforce the `time-too-new` guard on submitted Bitcoin/altcoin headers. A NEAR block producer who inflates this value can expand the acceptance window for future-dated source-chain headers beyond the intended `MAX_FUTURE_BLOCK_TIME_LOCAL` (7200 seconds), allowing a colluding relayer to commit headers the real source chain would reject, corrupting the light client's canonical chain state.

---

### Finding Description

Every chain module enforces the same pattern inside `check_pow`:

```rust
// bitcoin.rs, litecoin.rs, zcash.rs (identical structure)
let current_timestamp = u32::try_from(env::block_timestamp_ms() / 1000).unwrap();
require!(
    block_header.time <= current_timestamp + MAX_FUTURE_BLOCK_TIME_LOCAL,
    "time-too-new: block timestamp too far in the future"
);
``` [1](#0-0) 

```rust
// dogecoin.rs

    block_header.time <= current_timestamp + MAX_FUTURE_BLOCK_TIME_LOCAL,

`MAX_FUTURE_BLOCK_TIME_LOCAL` is a fixed constant of 7200 seconds (2 hours): [5](#0-4) 

`env::block_timestamp_ms()` is the NEAR block timestamp. It is not derived from any consensus-verified external source — it is set by the NEAR block producer who includes the `submit_blocks` transaction. NEAR protocol requires only that each block's timestamp be strictly greater than the previous block's timestamp; there is no hard upper bound enforced by the protocol that prevents a producer from setting a timestamp significantly ahead of wall-clock time.

The `check_pow` function is called unconditionally from `submit_block_header` whenever `skip_pow_verification` is `false` (the production setting): [6](#0-5) 

The public entry point `submit_blocks` calls `submit_block_header` for every header in the batch: [7](#0-6) 

---

### Impact Explanation

If a NEAR block producer sets `block_timestamp_ms()` to a value `Δ` seconds ahead of real wall-clock time, the effective acceptance window for source-chain header timestamps becomes `real_time + Δ + 7200` seconds instead of `real_time + 7200` seconds. A colluding relayer can then submit source-chain headers whose `time` field exceeds what the real Bitcoin/Dogecoin/Litecoin/Zcash network would accept. Such headers pass the `time-too-new` guard, pass PoW verification (the PoW hash check is independent of the timestamp), and are stored into `headers_pool` and potentially promoted to `mainchain_tip_blockhash`. This corrupts the canonical chain mapping (`mainchain_height_to_header`, `mainchain_header_to_height`) and the `mainchain_tip_blockhash` state variable. Downstream consumers calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` then operate against a corrupted chain view, potentially returning `true` for transactions in headers that the real source chain would never finalize. [8](#0-7) 

---

### Likelihood Explanation

The attack requires a NEAR block producer to be either the relayer itself or colluding with the relayer. NEAR uses a rotating validator set; any validator scheduled to produce the block that includes the `submit_blocks` call can set an inflated timestamp. The `trusted_relayer` macro restricts who may call `submit_blocks`, but it does not restrict which NEAR block producer processes the transaction. The combination of a motivated relayer and a single cooperating validator is a realistic threat for a production bridge carrying value. [9](#0-8) 

---

### Recommendation

Replace the sole reliance on `env::block_timestamp_ms()` with a more manipulation-resistant time reference. Concrete options:

1. **Accept a caller-supplied `max_timestamp` deadline** in `submit_blocks` and require `block_header.time <= max_timestamp`, then additionally verify `max_timestamp <= env::block_timestamp_ms() / 1000 + MAX_FUTURE_BLOCK_TIME_LOCAL`. This mirrors the external report's recommended fix of accepting a user-supplied deadline.
2. **Use the MTP of the previous block** (already computed in `get_median_time_past`) as the upper-bound reference instead of `env::block_timestamp_ms()`, since MTP is derived entirely from committed on-chain header data and cannot be inflated by a single NEAR block producer.
3. **Enforce a strict NEAR timestamp drift bound** at the contract level: reject any `submit_blocks` call where `env::block_timestamp_ms() / 1000` exceeds the last accepted header's `time` by more than a configurable maximum, preventing a producer from silently inflating the reference point.

---

### Proof of Concept

1. Attacker controls or colludes with a NEAR block producer scheduled to process a `submit_blocks` transaction.
2. Attacker (as relayer) crafts a Bitcoin header with `time = real_now + 10_000` (≈ 2.8 hours in the future). This header has valid PoW for its `bits` value.
3. The NEAR block producer sets the NEAR block timestamp to `(real_now + 3_000) * 1_000` ms (3000 seconds ahead of wall clock).
4. Inside `check_pow` (bitcoin.rs line 35): `current_timestamp = real_now + 3000`.
5. The guard evaluates: `real_now + 10_000 <= (real_now + 3000) + 7200` → `10_000 <= 10_200` → **passes**.
6. The MTP check (`time > get_median_time_past(...)`) also passes because the header's `time` exceeds the median of the previous 11 headers.
7. The header is stored into `headers_pool` and, if its `chain_work` exceeds the current tip, promoted to `mainchain_tip_blockhash`.
8. The light client's canonical chain now contains a header with a timestamp the real Bitcoin network would reject, and all subsequent `verify_transaction_inclusion` calls operate against this corrupted state. [10](#0-9) [11](#0-10)
