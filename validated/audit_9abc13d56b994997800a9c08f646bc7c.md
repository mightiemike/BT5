Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks LP Share Recipient (`owner`) Instead of Token Provider (`sender`), Enabling Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the first ABI parameter (`sender`, the actual token-providing caller) and instead checks the second parameter (`owner`, the LP-share recipient) against the allowlist. Any address not on the allowlist can bypass the restriction by naming any allowlisted address as `owner`. Conversely, a legitimately allowlisted depositor who routes LP shares to a non-allowlisted address is incorrectly blocked, breaking the contract's stated invariant of gating `addLiquidity` by depositor address.

## Finding Description

`MetricOmmPool.addLiquidity` invokes the hook as `_beforeAddLiquidity(msg.sender, owner, ...)`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes the call with `sender` as the first argument and `owner` as the second: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` discards the first slot (unnamed `address`) and binds the second slot to `owner`, then checks `allowedDepositor[msg.sender][owner]`: [3](#0-2) 

This means the guard evaluates whether the LP-share *recipient* is allowlisted, not whether the *token provider* (`msg.sender` of `addLiquidity`) is allowlisted. The correct pattern is demonstrated by `SwapAllowlistExtension.beforeSwap`, which names the first parameter `sender` and checks it: [4](#0-3) 

## Impact Explanation

**Allowlist bypass (unauthorized deposit):** An unprivileged caller `bob` calls `pool.addLiquidity(owner = alice, ...)` where `alice` is allowlisted. The extension evaluates `allowedDepositor[pool][alice]` → `true` and permits the call. `bob` provides the tokens via the liquidity callback; `alice` receives LP shares. The pool admin's intent to restrict who injects liquidity is completely defeated — an unauthorized party can freely deposit into a restricted pool, diluting existing LPs and corrupting pool state the admin intended to control.

**False denial (authorized depositor blocked):** A legitimately allowlisted depositor `alice` calling `addLiquidity(owner = alice_vault, ...)` where `alice_vault` is not on the allowlist is incorrectly reverted with `NotAllowedToDeposit`, blocking a normal DeFi pattern of separating token provision from LP share custody.

Both outcomes break the NatSpec invariant: *"Gates `addLiquidity` by depositor address, per pool."* [5](#0-4) 

## Likelihood Explanation

The bypass requires a single call to `addLiquidity` with any allowlisted address as `owner`. The allowlist is publicly readable on-chain via `allowedDepositor`. No special privileges, flash loans, or oracle manipulation are needed. Pools deploying this extension specifically to restrict depositors are the exact production target, making the trigger condition identical to the intended use case.

## Recommendation

Name and use the first parameter (`sender`) instead of the second (`owner`):

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
      external view override returns (bytes4)
  {
-     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
          revert IMetricOmmPoolActions.NotAllowedToDeposit();
      }
      return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
```

This mirrors the correct pattern in `SwapAllowlistExtension`. [4](#0-3) 

## Proof of Concept

1. Deploy pool with `DepositAllowlistExtension`; set `allowedDepositor[pool][alice] = true`. `bob` is not allowlisted.
2. `bob` calls `pool.addLiquidity(owner = alice, salt, deltas, callbackData, extensionData)`.
3. `_beforeAddLiquidity(msg.sender=bob, owner=alice, ...)` is called; the extension receives `(bob, alice, ...)`.
4. The hook discards `bob` (first slot) and checks `allowedDepositor[pool][alice]` → `true` → returns valid selector.
5. `LiquidityLib.addLiquidity` executes; pool calls `bob.metricOmmSwapCallback(...)` and `bob` transfers tokens.
6. LP shares are minted to `alice`; `bob` has successfully deposited into a pool he is not authorized to access.
7. Conversely, `alice` calling `addLiquidity(owner = alice_vault, ...)` where `alice_vault` is not allowlisted is incorrectly reverted. [3](#0-2)
