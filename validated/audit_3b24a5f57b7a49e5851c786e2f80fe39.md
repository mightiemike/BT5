### Title
`op_unknown` Cost Guard Applied Before Multiplier Allows Returned Cost to Exceed `max_cost` Budget — (File: `src/more_ops.rs`)

---

### Summary

In `op_unknown`, the `check_cost` guard is applied to the **pre-multiplied** base cost. After passing the guard, the cost is multiplied by `(cost_multiplier + 1)`, where the multiplier is fully attacker-controlled via the opcode bytes. The final returned `Reduction` cost can be orders of magnitude larger than `max_cost`, causing the VM's remaining budget to underflow when the caller subtracts the returned cost from its `u64` budget counter.

---

### Finding Description

`op_unknown` in `src/more_ops.rs` handles unknown opcodes in lenient mode. The opcode atom encodes two fields: a `cost_function` (2 bits) and a `cost_multiplier` (up to 4 bytes, i.e., up to `u32::MAX ≈ 4.29 × 10⁹`).

The function computes a base cost from the arguments according to the cost function, then calls `check_cost(cost, max_cost)?` — and only **then** multiplies the cost by `(cost_multiplier + 1)`:

```rust
// src/more_ops.rs lines 209–266
let mut cost = match cost_function {
    0 => 1,
    1 => { /* ARITH_BASE_COST + per-arg + per-byte */ }
    2 => { /* MUL_BASE_COST + per-op + linear + square */ }
    3 => { /* CONCAT_BASE_COST + per-arg + per-byte */ }
    _ => 1,
};

assert!(cost > 0);

check_cost(cost, max_cost)?;      // ← guard on PRE-multiplied cost
cost *= cost_multiplier + 1;      // ← multiplier applied AFTER guard
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
``` [1](#0-0) 

The `cost_multiplier` is decoded directly from the attacker-supplied opcode bytes:

```rust
let cost_multiplier: u64 = match u32_from_u8(&op[0..op.len() - 1]) {
    Some(v) => v as u64,
    None => { return Err(EvalErr::Invalid(o))?; }
};
``` [2](#0-1) 

The broken invariant: `check_cost` is supposed to enforce that the operator's cost does not exceed the remaining budget (`max_cost`). But because the guard fires before the multiplication, the returned cost can be `(cost_multiplier + 1)` times larger than `max_cost`. For example, with `cost_function = 0` (base cost = 1), a `cost_multiplier` of `0x00FFFFFF` passes `check_cost(1, max_cost)` for any `max_cost ≥ 1`, yet returns `Reduction(16_777_216, nil)`.

The analogous flaw to the NodeSale report: in NodeSale, the total price is computed from the current tier without accounting for tier crossings during a bulk purchase. Here, the total cost is checked at the base level without accounting for the multiplier that is applied afterward — the "tier crossing" is the multiplication step that is invisible to the guard.

---

### Impact Explanation

The CLVM execution loop passes `max_cost` (the remaining budget) to each operator and subtracts the returned cost from the running budget. `Cost` is `u64`: [3](#0-2) 

If the VM performs `remaining_budget -= returned_cost` using unsigned 64-bit arithmetic and `returned_cost > remaining_budget`, the subtraction wraps around to a value near `u64::MAX`. Any subsequent `check_cost` call against this wrapped value will pass trivially, giving all remaining operators an effectively unlimited budget. This allows a CLVM program to execute far beyond its declared cost limit, breaking the consensus-critical property that every node enforces the same cost ceiling. Nodes that enforce cost limits differently (e.g., mempool vs. consensus mode) would diverge on whether the program is valid.

---

### Likelihood Explanation

The attacker-controlled entry path is direct: unknown opcode bytes are part of the CLVM program serialization, which is externally supplied. An attacker submitting a transaction to the Chia mempool or constructing a coin puzzle can embed an unknown opcode with an arbitrary `cost_multiplier`. No special privileges, social engineering, or compromised infrastructure are required. The opcode format is documented and the multiplier field is up to 4 bytes, giving full control over the amplification factor. [4](#0-3) 

---

### Recommendation

Move `check_cost` to after the multiplication, so the guard is applied to the value that will actually be returned:

```rust
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
}
check_cost(cost, max_cost)?;   // ← guard on POST-multiplied cost
Ok(Reduction(cost as Cost, allocator.nil()))
```

This mirrors the correct pattern used in other operators (e.g., `op_bls_g1_multiply`) where `check_cost` is called on the final accumulated cost before returning. [5](#0-4) 

---

### Proof of Concept

Craft a 5-byte unknown opcode atom:
- Bytes `[0x00, 0xFF, 0xFF, 0xFF, 0x00]`
- `op[op.len()-1] = 0x00` → `cost_function = 0` (constant cost = 1)
- `op[0..4] = [0x00, 0xFF, 0xFF, 0xFF]` → `cost_multiplier = 0x00FFFFFF = 16,777,215`

Execution with `max_cost = 10`:
1. `cost = 1` (cost_function 0)
2. `check_cost(1, 10)` → passes (1 ≤ 10)
3. `cost *= 16_777_215 + 1 = 16_777_216`
4. `16_777_216 ≤ u32::MAX` → no error
5. Returns `Reduction(16_777_216, nil)`

The VM subtracts `16_777_216` from its remaining budget of `10` (u64): result wraps to `u64::MAX - 16_777_205`, giving all subsequent operators an effectively unlimited budget. [1](#0-0)

    // 1: computed like operator add, multiplied by (multiplier + 1)
    // 2: computed like operator mul, multiplied by (multiplier + 1)
    // 3: computed like operator concat, multiplied by (multiplier + 1)
