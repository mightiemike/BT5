Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks Caller-Controlled `owner` Instead of `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` is intended to restrict which addresses may deposit liquidity into a permissioned pool. However, the guard evaluates `allowedDepositor[msg.sender][owner]` where `owner` is a free parameter supplied by the caller, not the actual depositor (`msg.sender` of `addLiquidity`, forwarded as `sender`). Any unprivileged address can bypass the allowlist by passing any already-allowlisted address as `owner`, completely defeating the access control the extension is meant to enforce.

## Finding Description
`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address from the caller and passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner` to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes and forwards both `sender` and `owner` to every configured extension: [2](#0-1) 

Inside `DepositAllowlistExtension.beforeAddLiquidity`, the first parameter (`sender`, the actual `msg.sender`) is unnamed and silently discarded. The guard is evaluated only against `owner`: [3](#0-2) 

Because `owner` is freely chosen by the caller with no on-chain validation, any address can pass the guard by setting `owner` to any address already present in `allowedDepositor[pool]`. The token transfer obligation falls on `msg.sender` via the liquidity callback, so the unauthorized caller funds the position while LP shares are attributed to the allowlisted `owner` address.

The asymmetry with `SwapAllowlistExtension.beforeSwap` confirms this is a logic error: the swap extension correctly checks `sender` (the actual caller) and discards `recipient`: [4](#0-3) 

## Impact Explanation
A pool configured with `DepositAllowlistExtension` (e.g., KYC-gated, institutional-only, or regulatory-compliance pools) provides no actual access control on deposits. Any unprivileged address can acquire LP positions and earn fees in pools intended to be permissioned. This constitutes broken core pool functionality — the allowlist invariant that only approved depositors may add liquidity is completely violated, enabling unauthorized actors to participate in restricted pools and earn LP fees.

## Likelihood Explanation
Exploitation requires no special privileges, no flash loan, and no oracle manipulation. Any EOA or contract can call `addLiquidity` with a crafted `owner` argument. The only prerequisite is knowing one allowlisted address, which is publicly readable from the `allowedDepositor` mapping. The attack is repeatable and permissionless.

## Recommendation
Replace the `owner` check with a `sender` check, mirroring the pattern used in `SwapAllowlistExtension`:

```solidity
// After (correct):
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```
Setup:
  - Pool P is deployed with DepositAllowlistExtension E.
  - Pool admin calls E.setAllowedToDeposit(P, alice, true).
  - bob is NOT in the allowlist.

Attack:
  1. bob calls P.addLiquidity(alice, salt, deltas, callbackData, extensionData).
  2. E.beforeAddLiquidity(bob /*sender, discarded*/, alice /*owner, checked*/, ...) is called.
  3. allowedDepositor[P][alice] == true → guard passes.
  4. bob's callback funds the position; LP shares are minted to alice.

Result:
  - bob has successfully deposited into a pool he is not authorized to access.
  - The deposit allowlist is completely bypassed.
```
