### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables usdcE Drain Without USDC Compensation ‚Äî (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the return value. If the USDC token returns `false` on failure rather than reverting, the caller receives usdcE from the DDA without ever providing USDC, draining the DDA's usdcE balance.

---

### Finding Description

`replaceUsdcEWithUsdc` is a migration helper callable by **any unprivileged address** on Ink Mainnet (chain ID 57073). Its intended flow is:

1. Pull USDC from `msg.sender` into the target DDA via `transferFrom`.
2. Withdraw usdcE from the DDA to `ContractOwner` via `DirectDepositV1.withdraw`.
3. Forward that usdcE to `msg.sender` via `safeTransfer`.

The critical flaw is at step 1:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
```

This is a raw interface call ‚Äî the `bool` return value is silently discarded. [1](#0-0) 

Contrast this with every other outbound transfer in the codebase, which routes through `ERC20Helper.safeTransferFrom` ‚Äî a low-level call that explicitly asserts `success && (data.length == 0 || abi.decode(data, (bool)))`. [2](#0-1) 

Steps 2 and 3 execute unconditionally after step 1, regardless of whether the USDC pull succeeded:

```solidity
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE)); // usdcE ‚Üí ContractOwner
IERC20Base(usdcE).safeTransfer(msg.sender, balance);           // usdcE ‚Üí attacker
``` [3](#0-2) 

`DirectDepositV1.withdraw` is `onlyOwner`, and `ContractOwner` is the owner of every DDA it deploys, so the withdrawal always succeeds from `ContractOwner`'s perspective. [4](#0-3) 

---

### Impact Explanation

If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` returns `false` on a failed `transferFrom` (e.g., caller has no approval or insufficient balance) rather than reverting, an attacker can:

- Call `replaceUsdcEWithUsdc(subaccount)` for any DDA that holds usdcE.
- Receive the full usdcE balance of that DDA without depositing any USDC.
- The DDA's usdcE balance is reduced to zero; the attacker's wallet gains usdcE at zero cost.

The exact corrupted asset delta: `directDepositV1.usdcE balance ‚Üí 0`, `attacker.usdcE balance += balance`.

---

### Likelihood Explanation

Circle's native USDC on EVM chains typically reverts on failure, which would cause the whole transaction to revert and prevent exploitation. However:

- The function contains no access control ‚Äî any address can call it on Ink Mainnet.
- The return value is structurally unchecked, meaning any token at that address that returns `false` on failure (including future upgrades or redeployments) would be immediately exploitable.
- The pattern diverges from every other transfer in the codebase, which uses `ERC20Helper.safeTransferFrom`.

Likelihood is **low** given current USDC behavior, but the structural defect is real and the entry path is fully unprivileged.

---

### Recommendation

Replace the raw `transferFrom` call with the protocol's own safe wrapper:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

This routes through `ERC20Helper.safeTransferFrom`, which checks both the low-level call success and the decoded boolean return value, consistent with all other token pulls in the codebase. [2](#0-1) 

---

### Proof of Concept

1. Identify a DDA (`directDepositV1Address[subaccount]`) that holds a non-zero usdcE balance.
2. As an attacker with **zero USDC approval** to `ContractOwner`, call:
   ```solidity
   ContractOwner(contractOwner).replaceUsdcEWithUsdc(subaccount);
   ```
3. If USDC returns `false` on the failed `transferFrom` (rather than reverting):
   - `transferFrom` silently fails ‚Äî no USDC leaves the attacker.
   - `DirectDepositV1.withdraw(usdcE)` executes ‚Äî usdcE moves from DDA to `ContractOwner`.
   - `safeTransfer(msg.sender, balance)` executes ‚Äî usdcE moves from `ContractOwner` to attacker.
4. Attacker receives `balance` usdcE at zero cost; DDA usdcE balance is zeroed. [5](#0-4)

    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
