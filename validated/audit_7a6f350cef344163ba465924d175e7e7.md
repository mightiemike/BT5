### Title
`SwapAllowlistExtension` gates the router address instead of the actual user when swaps route through `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `swap` function is called with `msg.sender = router`, so the extension receives the **router's address** as `sender` instead of the **actual user's address**. The allowlist therefore gates the router, not the individual user, breaking the intended curation policy.

---

### Finding Description

`SwapAllowlistExtension` is a production extension that pool admins deploy to restrict which addresses may trade on a curated pool: [1](#0-0) 

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

Here `msg.sender` is the pool (correct pool-namespacing) and `sender` is the first argument the pool passes when it calls the hook. The pool's `swap` entry point does not accept an explicit `sender` parameter — it uses `msg.sender` of the `swap` call as the actor it forwards to the extension.

`MetricOmmSimpleRouter.exactInputSingle` calls the pool directly: [2](#0-1) 

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

`msg.sender` to the pool is the **router contract**, not the end user. The pool therefore passes `router` as `sender` to `beforeSwap`. The extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

The same mismatch occurs in `exactInput` (intermediate hops use `address(this)` = router) and `exactOutput` / `exactOutputSingle`. [3](#0-2) 

Contrast this with `DepositAllowlistExtension`, which checks the `owner` parameter — an explicit argument that `addLiquidity` accepts separately from `msg.sender`, allowing the operator pattern to preserve the real depositor identity: [4](#0-3) 

No equivalent explicit-owner parameter exists on `swap`, so the actor identity is irrecoverably lost at the pool boundary when an intermediary router is used.

---

### Impact Explanation

Two concrete fund-impacting outcomes:

**Path A — Allowlist bypass (higher severity):** A pool admin allowlists the router address (a natural operational step to enable router-based trading on a curated pool). Any non-allowlisted user can then call `exactInputSingle` / `exactInput` / `exactOutput` through the router and the extension passes, because `allowedSwapper[pool][router] == true`. The entire swap curation policy is nullified for all router-routed trades, allowing unrestricted access to a pool that was intended to be restricted (e.g., KYC-gated, institutional-only, or rate-limited).

**Path B — Broken core functionality (medium severity):** If the router is not allowlisted, every individually-allowlisted user who routes through `MetricOmmSimpleRouter` receives `NotAllowedToSwap`, making the router unusable on any allowlist-protected pool. Users must call the pool directly, bypassing slippage protection, multi-hop routing, and deadline enforcement provided by the periphery.

Both paths represent a broken invariant: *"A curated pool must enforce the same allowlist policy regardless of which supported public entrypoint reaches it."*

---

### Likelihood Explanation

- The `MetricOmmSimpleRouter` is the primary documented user-facing swap entrypoint.
- Pool admins who want to allow router-based swaps on a curated pool have no documented alternative to allowlisting the router address, making Path A a natural operational mistake.
- No special privileges, malicious tokens, or off-chain manipulation are required — a standard `exactInputSingle` call suffices.
- The `DepositAllowlistExtension` works correctly (checks `owner`), so admins have no reason to suspect the swap extension behaves differently.

---

### Recommendation

The pool's `swap` function should accept an explicit `swapper` / `onBehalfOf` parameter (analogous to `owner` in `addLiquidity`) that the router populates with `msg.sender` (the real user) before calling the pool. The extension then checks that explicit parameter instead of the forwarded `msg.sender`.

Alternatively, `SwapAllowlistExtension` can read the real user from `extensionData` (a caller-supplied bytes field already threaded through the hook), but this requires the router to encode and sign the user identity, adding complexity and a new trust assumption.

The cleanest fix mirrors the liquidity operator pattern: add `address swapper` to `IMetricOmmPoolActions.swap` so the pool can forward the true initiator to all extension hooks.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured on `beforeSwap`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-based swaps (or `setAllowAllSwappers(pool, true)` is not set, but router is individually listed).
3. Non-allowlisted `attacker` calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(recipient, ...)` — pool's `msg.sender` is the router.
5. Pool calls `extension.beforeSwap(router, recipient, ...)`.
6. Extension evaluates `allowedSwapper[pool][router] == true` → passes.
7. Attacker's swap executes on a pool that was supposed to reject them.

Conversely, if the router is not allowlisted:

3. Allowlisted `user` calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
4–5. Same as above; extension evaluates `allowedSwapper[pool][router] == false` → reverts `NotAllowedToSwap`.
6. Allowlisted user cannot use the router at all on this pool. [1](#0-0) [5](#0-4) [6](#0-5)
