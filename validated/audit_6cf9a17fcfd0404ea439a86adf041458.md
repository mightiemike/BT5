Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Unprivileged Bypass of the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension` is documented as gating `addLiquidity` by depositor address, but its `beforeAddLiquidity` hook silently discards the `sender` parameter and checks only `owner` (the position-recipient) against the allowlist. Because `addLiquidity` imposes no `msg.sender == owner` constraint, any unprivileged address can bypass the allowlist by supplying any allowlisted address as `owner`, paying the tokens themselves while the position is minted to that address.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner` to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` correctly encodes both values and forwards them to every configured extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` names the first parameter (the actual caller) as unnamed/discarded and checks only `owner`: [3](#0-2) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender`: [4](#0-3) 

`removeLiquidity` enforces `msg.sender == owner`, closing the analogous path there, but `addLiquidity` has no such guard: [5](#0-4) 

The root cause is the asymmetry: `beforeAddLiquidity` evaluates `allowedDepositor[pool][owner]` instead of `allowedDepositor[pool][sender]`, so the check is trivially satisfied by any caller who names an allowlisted address as the position recipient.

## Impact Explanation

This is an admin-boundary break. The deposit allowlist is a pool-admin-configured access control intended to restrict which addresses may provide liquidity (e.g., KYC-gated or regulatory-compliance pools). An unprivileged address bypasses this control entirely: it calls `addLiquidity` with `owner` set to any allowlisted address, the extension approves the call, the unprivileged caller's tokens are collected via callback, and shares are minted to the allowlisted `owner`. The pool receives liquidity from a source the admin explicitly intended to exclude, violating the permissioned-deposit invariant and potentially altering bin state, liquidity distribution, and swap pricing in pools where the allowlist was the primary admission control.

## Likelihood Explanation

Exploitation requires only a standard `addLiquidity` call with `owner` set to any address already in the allowlist. No special privileges, flash loans, or oracle manipulation are needed. The `allowedDepositor` mapping is public, so an attacker can trivially identify a valid `owner` on-chain. The attack is repeatable at any time while the misconfigured extension is active.

## Recommendation

Replace the `owner` check with a `sender` check, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate on both the caller and the position recipient, both should be checked explicitly.

## Proof of Concept

1. Pool `P` is deployed with `DepositAllowlistExtension` as a `beforeAddLiquidity` hook.
2. Pool admin calls `setAllowedToDeposit(P, alice, true)`. Bob is **not** allowlisted.
3. Bob calls `P.addLiquidity(owner = alice, salt = 0, deltas = ..., callbackData = ..., extensionData = ...)`.
4. The pool calls `DepositAllowlistExtension.beforeAddLiquidity(sender=bob, owner=alice, ...)`.
5. The hook evaluates `allowedDepositor[P][alice]` → `true`; no revert.
6. `LiquidityLib.addLiquidity` mints shares to `alice`; Bob's tokens are collected via callback.
7. Bob has successfully deposited into a pool he was explicitly excluded from, violating the pool admin's access-control invariant.
