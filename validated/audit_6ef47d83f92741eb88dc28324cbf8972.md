### Title
Unbounded `NlpLockedBalanceQueue` Growth Enables Permanent NLP Token Lock via Block Gas Limit Exhaustion — (`core/contracts/SpotEngineState.sol`)

---

### Summary

The `NlpLockedBalanceQueue` for a subaccount grows without bound as a user mints NLP tokens across different oracle timestamps. The `tryUnlockNlpBalance` function processes all unlocked entries in a single unbounded `while` loop with no batch limit or cap. A user who accumulates a sufficiently large queue will find that every subsequent NLP balance operation — including `burnNlp` — permanently reverts due to block gas limit exhaustion, locking their NLP tokens indefinitely.

---

### Finding Description

In `SpotEngineState.sol`, `tryUnlockNlpBalance` iterates over all matured entries in the `NlpLockedBalanceQueue` in one unbounded loop: [1](#0-0) 

There is no cap on `queue.balanceCount`. A new entry is appended each time a user mints NLP at an oracle timestamp that differs from the last entry's `unlockedAt - NLP_LOCK_PERIOD`: [2](#0-1) 

Because the oracle time advances with every sequencer `SpotTick`, a user can create one new queue entry per tick by submitting a `MintNlp` transaction at each tick. Over the NLP lock period (days to weeks), this accumulates thousands of entries.

`tryUnlockNlpBalance` is invoked on every NLP balance mutation through two paths:

1. **`getNlpUnlockedBalance`** — called directly by `burnNlp` in `Clearinghouse.sol` before checking the unlocked amount: [3](#0-2) 

2. **`handleNlpLockedBalance`** — called by `updateBalance` for `NLP_PRODUCT_ID`, which is also invoked inside `burnNlp`: [4](#0-3) 

Both paths are triggered unconditionally by `burnNlp`. If the queue is large enough, the loop in `tryUnlockNlpBalance` exhausts the block gas limit, causing `burnNlp` to revert permanently.

The `burnNlp` call chain in `Clearinghouse.sol` confirms both invocations: [5](#0-4) 

---

### Impact Explanation

A user who has minted NLP across N distinct oracle timestamps accumulates N entries in their `NlpLockedBalanceQueue`. After the lock period expires, the first call to `burnNlp` (or any subsequent `MintNlp`) triggers `tryUnlockNlpBalance`, which must process all N entries atomically. Each iteration performs multiple `SLOAD`/`SSTORE` operations (reading `queue.balances[i]`, writing `unlockedBalanceSum`, deleting the entry, incrementing `unlockedUpTo`). At ~5,000–20,000 gas per iteration and a 30M gas block limit, roughly 1,500–6,000 entries suffice to permanently block execution.

**Exact corrupted state**: `nlpLockedBalanceQueues[subaccount].balanceCount` grows beyond the processable threshold, making `burnNlp` permanently unreachable. The user's NLP balance is frozen — they cannot convert it back to USDC collateral. Since `tryUnlockNlpBalance` is `public` but also unbounded, even a direct external call cannot rescue the queue once it exceeds the gas limit.

---

### Likelihood Explanation

The entry path is fully user-controlled: the attacker submits `MintNlp` transactions once per sequencer oracle tick. No privileged role is required. The cost scales with the number of mints (transaction fees plus minimum NLP mint amounts). Over a multi-week lock period with frequent ticks, accumulating thousands of entries is operationally feasible. The attack is self-inflicted but results in permanent capital loss, not merely temporary inconvenience.

---

### Recommendation

1. **Cap `balanceCount`**: Enforce a maximum number of entries per `NlpLockedBalanceQueue` (e.g., 100–200) and revert or consolidate on overflow.
2. **Batch processing**: Allow `tryUnlockNlpBalance` to process at most `K` entries per call and track progress, so callers can drain the queue across multiple transactions before attempting `burnNlp`.
3. **Aggressive consolidation**: Merge entries within the same calendar day or epoch rather than per oracle tick, bounding queue growth to `lock_period / consolidation_window`.

---

### Proof of Concept

```
1. Attacker calls MintNlp once per sequencer SpotTick for T ticks
   → nlpLockedBalanceQueues[attacker].balanceCount == T

2. Attacker waits NLP_LOCK_PERIOD seconds
   → all T entries now satisfy unlockedAt <= getOracleTime()

3. Attacker calls burnNlp(...)
   → Clearinghouse.burnNlp calls spotEngine.getNlpUnlockedBalance(attacker)
   → getNlpUnlockedBalance calls tryUnlockNlpBalance(attacker)
   → while loop iterates T times, each iteration: SLOAD + SSTORE + SSTORE + SSTORE
   → if T > ~1500, gas consumed > 30,000,000 (block gas limit)
   → transaction reverts with out-of-gas

4. Every subsequent burnNlp or MintNlp call also reverts (same loop)
   → attacker's NLP balance is permanently frozen
   → USDC collateral is permanently inaccessible
```

    function tryUnlockNlpBalance(bytes32 subaccount)
        public
        returns (Balance memory)
    {
        NlpLockedBalanceQueue storage queue = nlpLockedBalanceQueues[
            subaccount
        ];
        while (
            queue.unlockedUpTo < queue.balanceCount &&
            queue.balances[queue.unlockedUpTo].unlockedAt <= getOracleTime()
        ) {
            // we can unlock this balance
            queue.unlockedBalanceSum.amount += queue
                .balances[queue.unlockedUpTo]
                .balance
                .amount;
            delete queue.balances[queue.unlockedUpTo];
            queue.unlockedUpTo++;
        }

        return queue.unlockedBalanceSum;
    }

    function handleNlpLockedBalance(bytes32 subaccount, int128 amountDelta)
        internal
    {
        _assertInternal();

        // N_ACCOUNT is not limited by lock period
        if (subaccount == N_ACCOUNT) return;

        tryUnlockNlpBalance(subaccount);
        if (amountDelta > 0) {
            NlpLockedBalanceQueue storage queue = nlpLockedBalanceQueues[
                subaccount
            ];
            if (
                queue.balanceCount > 0 &&
                queue.balances[queue.balanceCount - 1].unlockedAt ==
                getOracleTime() + NLP_LOCK_PERIOD
            ) {
                queue
                    .balances[queue.balanceCount - 1]
                    .balance
                    .amount += amountDelta;
            } else {
                queue.balances[queue.balanceCount] = NlpLockedBalance({
                    balance: Balance({amount: amountDelta}),
                    unlockedAt: getOracleTime() + NLP_LOCK_PERIOD
                });
                queue.balanceCount++;
            }
        } else if (amountDelta < 0) {
            Balance memory balanceSum = nlpLockedBalanceQueues[subaccount]
                .unlockedBalanceSum;

            nlpLockedBalanceQueues[subaccount].unlockedBalanceSum = balanceSum;
