### Title
`gas_prices_to_hash` version boundary frozen at `BlockHashVersion::V0_13_4` silently omits `l2_gas_consumed` and `next_l2_gas_price` for all Starknet 0.14.x blocks ŌĆö (`File: crates/starknet_api/src/block_hash/block_hash_calculator.rs`)

### Summary

The `BlockHashVersion` enum has exactly two variants (`V0_13_2`, `V0_13_4`). Every Starknet version Ōēź 0.13.4 ŌĆö including the entire 0.14.x series up to the current `LATEST` of 0.14.4 ŌĆö is silently collapsed to `BlockHashVersion::V0_13_4` by the `TryFrom<StarknetVersion>` conversion. The `gas_prices_to_hash` function therefore never executes any code path that would include `l2_gas_consumed` or `next_l2_gas_price` in the block hash preimage. Both fields are stored in `BlockHeaderWithoutHash` and are explicitly forwarded to the external block-hash CLI by the production `echonet` monitoring tool. A developer-acknowledged TODO in the same function states these fields must be added "after 0.14.0", but no new `BlockHashVersion` variant exists and `PartialBlockHashComponents` has no slots for them, so the gate can never fire.

### Finding Description

**`PartialBlockHashComponents` has no slots for the missing fields:** [3](#0-2) 

`PartialBlockHashComponents` carries `l1_gas_price`, `l1_data_gas_price`, `l2_gas_price` but has no `l2_gas_consumed` or `next_l2_gas_price` fields. Even if a new `BlockHashVersion` variant were added, `calculate_block_hash` cannot include these values because they are not threaded through the struct.

**The fields exist in the block header and are used by the production echonet CLI:** [4](#0-3) 

`BlockHeaderWithoutHash` carries both `l2_gas_consumed: GasAmount` and `next_l2_gas_price: GasPrice`. [5](#0-4) 

The production `echonet` monitoring tool's `_compute_block_hash` method explicitly passes both fields to the external block-hash CLI binary and then compares the result against the mainnet block hash. If the CLI's output matches mainnet, mainnet already commits these fields; the Rust sequencer path does not.

**Starknet version 0.14.x is current production:** [6](#0-5) 

`StarknetVersion::LATEST` is `V0_14_4`. All blocks produced since 0.14.1 are affected.

### Impact Explanation

`calculate_block_hash` is the canonical function used by the sequencer to commit a block hash to storage and to supply the value returned by the `get_block_hash` syscall. If `l2_gas_consumed` and `next_l2_gas_price` belong in the preimage for 0.14.x blocks (as the echonet CLI and the TODO both indicate), then every block hash produced by the Rust path since 0.14.1 is wrong. Consequences:

- **Wrong storage value**: the `block_hashes` MDBX table stores an incorrect hash for every 0.14.x block.
- **Wrong syscall result**: contracts calling `get_block_hash` receive the wrong value, silently corrupting any contract logic that depends on it (e.g., attestation, randomness, cross-block commitments).
- **Wrong L1 message / state commitment**: if the incorrect hash is submitted to L1 contracts or included in a proof, on-chain verification fails or accepts a wrong state root.

This matches the **Critical** impact category: *Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.*

### Likelihood Explanation

The trigger is unconditional: every block with `starknet_version >= V0_14_1` hits the frozen `V0_13_4` branch. No special transaction, no privileged role, and no adversarial input is required. The current `LATEST` version is `V0_14_4`, so the condition has been continuously true for all recently produced blocks.

### Recommendation

1. Add a new `BlockHashVersion::V0_14_1` (or `V0_14_0`) variant to the enum and extend `TryFrom<StarknetVersion>` to map 0.14.x versions to it.
2. Add `l2_gas_consumed: GasAmount` and `next_l2_gas_price: GasPrice` fields to `PartialBlockHashComponents` and update `PartialBlockHashComponents::new` to populate them from the block header.
3. Update `gas_prices_to_hash` to include these two fields in the Poseidon preimage when `block_hash_version >= V0_14_1`.
4. Add a regression test analogous to `test_block_hash_regression` that covers a `V0_14_x` version and verifies the hash changes when `l2_gas_consumed` or `next_l2_gas_price` is modified.
5. Verify the new Rust output matches the external block-hash CLI used by echonet.

#[derive(Clone, Debug, PartialEq, Eq, PartialOrd)]
pub enum BlockHashVersion {
    V0_13_2,
    V0_13_4,
}

impl From<BlockHashVersion> for StarknetVersion {
    fn from(value: BlockHashVersion) -> Self {
        match value {
            BlockHashVersion::V0_13_2 => StarknetVersion::V0_13_2,
            BlockHashVersion::V0_13_4 => StarknetVersion::V0_13_4,
        }
    }
}

impl TryFrom<StarknetVersion> for BlockHashVersion {
    type Error = StarknetApiError;

    fn try_from(value: StarknetVersion) -> StarknetApiResult<Self> {
        if value < Self::V0_13_2.into() {
            Err(StarknetApiError::BlockHashVersion { version: value.to_string() })
        } else if value < Self::V0_13_4.into() {
            // Starknet versions 0.13.2 and 0.13.3 both have the same block hash mechanism.
            Ok(Self::V0_13_2)
        } else {
            Ok(Self::V0_13_4)
        }
    }
