### Title
Silent Success on Missing Execution Outcome Desynchronizes Relayer Chain Pointer — (`File: relayer/src/near_client.rs`)

### Summary

`parse_submit_blocks_response` returns `Ok(SubmitResult { gas_burnt: 0 })` — a success signal — when `response.final_execution_outcome` is `None`. The `Synchronizer` in `main.rs` interprets this as a confirmed submission and unconditionally advances `first_block_height_to_submit` past the unconfirmed batch. The on-chain `BtcLightClient` state is never updated for those blocks, but the relayer never re-submits them, permanently desynchronizing the off-chain pointer from the on-chain canonical chain.

### Finding Description

`parse_submit_blocks_response` has two code paths:

1. When `final_execution_outcome` is `Some(...)`: correctly handles `Failure` variants and returns `Ok(SubmitResult { gas_burnt })` on success.
2. When `final_execution_outcome` is `None`: falls through to the default return at line 340. [1](#0-0) 

The fallback at line 340 is:

```rust
Ok(SubmitResult { gas_burnt: 0 })
```

This is structurally identical to a successful submission with zero gas — indistinguishable from a real success to the caller. [2](#0-1) 

In `prepare_and_submit_batches`, the `Ok(Ok(result))` arm unconditionally stores the advanced height:

```rust
Ok(Ok(result)) => {
    first_block_height_to_submit.store(tx.last_block_height + 1, ...);
    if result.gas_burnt > 0 { ... }  // skipped when gas_burnt == 0
}
``` [3](#0-2) 

The `gas_burnt == 0` branch skips `on_success`, so the adaptive batcher is not updated either — but critically, `first_block_height_to_submit` is already advanced past the unsubmitted batch.

The `submit_blocks` loop only retries on `TimeoutError` or `UnknownTransaction`. Any other `Ok(response)` — including one with `final_execution_outcome: None` — is passed directly to `parse_submit_blocks_response` and returned as success. [4](#0-3) 

### Impact Explanation

The relayer's `first_block_height_to_submit` pointer advances past a batch of Bitcoin block headers that were never confirmed on-chain. The `BtcLightClient` contract is missing those headers. Any subsequent call to `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for a transaction in one of those missing blocks will panic with `"block does not belong to the current main chain"` or `"cannot find requested transaction block"`, returning a false negative to consumers. The gap is permanent: the relayer has no logic to detect or backfill skipped heights after advancing the pointer. [5](#0-4) 

### Likelihood Explanation

The NEAR JSON-RPC defines `final_execution_outcome` as `Option<FinalExecutionOutcomeWithReceiptView>`. Even with `wait_until: TxExecutionStatus::Executed`, the field can be `None` under transient network conditions, RPC node inconsistencies, or when a transaction is included in a block but receipts are not yet propagated. This is not a theoretical edge case — it is a documented optional field that the SDK explicitly allows to be absent. Any relayer operator running against a degraded or adversarially configured RPC endpoint will trigger this path. [6](#0-5) 

### Recommendation

Replace the silent fallback with an explicit error:

```rust
// Instead of:
Ok(SubmitResult { gas_burnt: 0 })

// Use:
Err(CustomError::TxExecutionError(
    "No final execution outcome in response".to_string()
))
```

This ensures the `Synchronizer` does not advance `first_block_height_to_submit` when the submission outcome is unknown, and the batch will be retried on the next sync iteration. [1](#0-0) 

### Proof of Concept

1. Relayer calls `submit_blocks` for headers at heights `[N, N+k]`.
2. NEAR RPC returns `Ok(response)` where `response.final_execution_outcome` is `None` (e.g., due to a slow node or adversarial RPC).
3. `parse_submit_blocks_response` returns `Ok(SubmitResult { gas_burnt: 0 })`.
4. `prepare_and_submit_batches` matches `Ok(Ok(result))` and executes `first_block_height_to_submit.store(N+k+1, ...)`.
5. The `BtcLightClient` contract has no headers for heights `[N, N+k]`.
6. A consumer calls `verify_transaction_inclusion` for a transaction in block `N+j`. The contract panics: `"block does not belong to the current main chain"`.
7. The relayer never re-submits the missing range because its pointer is already past it. [7](#0-6)

    fn parse_submit_blocks_response(
        response: &RpcTransactionResponse,
    ) -> Result<SubmitResult, CustomError> {
        if let Some(ref final_execution_outcome) = response.final_execution_outcome {
            let outcome = final_execution_outcome.clone().into_outcome();

            if let near_primitives::views::FinalExecutionStatus::Failure(ref err) = outcome.status {
                let err_str = format!("{err:?}");
                if err_str.contains("PrevBlockNotFound") {
                    return Err(CustomError::PrevBlockNotFound);
                } else if err_str.contains("Exceeded the maximum amount of gas") {
                    return Err(CustomError::GasExceeded);
                }
                return Err(CustomError::TxExecutionError(err_str));
            }

            let gas_burnt = outcome.transaction_outcome.outcome.gas_burnt
                + outcome
                    .receipts_outcome
                    .iter()
                    .map(|r| r.outcome.gas_burnt)
                    .sum::<u64>();

            return Ok(SubmitResult { gas_burnt });
        }

        Ok(SubmitResult { gas_burnt: 0 })
    }
