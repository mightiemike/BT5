### Title
Accumulated NEAR Storage Deposits Are Permanently Locked After GC — (`contract/src/lib.rs`)

### Summary

`submit_blocks` collects NEAR token deposits from callers to cover on-chain storage costs. `run_mainchain_gc` later frees that storage, releasing the storage-staking obligation, but the NEAR tokens that were paid for those now-freed blocks remain in the contract's balance permanently. No withdrawal function exists anywhere in the contract, so every GC cycle silently locks an additional tranche of NEAR tokens with no recovery path.

### Finding Description

`submit_blocks` is `#[payable]` and enforces that the caller covers the net storage increase of the call: [1](#0-0) 

The deposit accounting measures `env::storage_usage()` before and after the full call body, which includes the `run_mainchain_gc` invocation: [2](#0-1) 

When GC removes old mainchain blocks it calls `remove_block_header` and `mainchain_height_to_header.remove`, which frees storage bytes: [3](#0-2) [4](#0-3) 

Because GC runs **inside** the same call that measures storage delta, the freed bytes from old blocks offset the newly consumed bytes from new blocks. The net `diff_storage_usage` is therefore smaller (possibly zero) than the raw cost of the new headers alone. The caller of the current call is correctly charged only the net delta and receives a refund of any excess.

However, the NEAR tokens that were paid in **previous** `submit_blocks` calls for the blocks now being GC'd are already sitting in the contract's balance. When GC frees those blocks' storage, the contract's minimum required balance (storage stake) decreases, making those tokens "free" balance inside the contract account. There is no `withdraw`, `claim_storage_refund`, or any other function that can move this freed balance back to the original depositors or to any authorized party. The entire public API of the contract contains only view functions, `submit_blocks`, `run_mainchain_gc`, and `migrate`: [5](#0-4) 

The `Upgradable` and `Pausable` plugins add no withdrawal capability. The `trusted_relayer` macro manages staking metadata, not token balances.

### Impact Explanation

Every block that is submitted, paid for, and later garbage-collected contributes a permanent, irrecoverable NEAR token balance to the contract. Over the contract's lifetime the locked amount grows monotonically with the number of GC cycles. For a production deployment with `gc_threshold = 52704` (≈ one year of Bitcoin blocks) and a non-trivial storage byte cost, the locked amount becomes material. The tokens are not burned; they are simply inaccessible to any party forever.

### Likelihood Explanation

This condition is triggered by normal, expected operation. `run_mainchain_gc` is called automatically at the end of every `submit_blocks` invocation: [6](#0-5) 

No adversarial input is required. Any relayer submitting blocks causes GC to run once the chain exceeds `gc_threshold`, which is the intended steady-state behavior. The vulnerability is therefore certain to manifest on any live deployment.

### Recommendation

Add a privileged withdrawal function (callable by `Role::DAO` or a designated treasury role) that transfers the contract's free balance (total balance minus current storage stake) to a specified recipient. Alternatively, track per-depositor storage contributions and issue pro-rata refunds when GC frees their blocks, mirroring the pattern used for the per-call refund already present in `submit_blocks`.

### Proof of Concept

1. Relayer calls `submit_blocks` with 100 headers and attaches deposit `D` covering 100 × `block_storage_size × byte_cost`. Contract retains exactly `D` NEAR for those 100 blocks.
2. Chain grows past `gc_threshold`. Next `submit_blocks` call GCs the original 100 blocks. `diff_storage_usage` for the new call is reduced by the freed bytes; the new caller pays only the net delta and is refunded correctly.
3. The original `D` NEAR tokens from step 1 now sit in the contract's free balance. No function in the contract can move them out.
4. Repeat for every GC cycle. The locked balance grows without bound and is permanently unrecoverable.

    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
