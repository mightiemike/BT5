### Title
`SwapAllowlistExtension` gates on router address instead of original user, enabling allowlist bypass through `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is `msg.sender` of `MetricOmmPool.swap`. When a user routes through `MetricOmmSimpleRouter`, the pool sees the **router** as `msg.sender`, not the original user. If the pool admin allowlists the router address (a natural configuration to support router-mediated swaps for allowlisted users), every user — including those not on the allowlist — can bypass the per-user swap gate by routing through the router.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` is on the allowlist, keyed by `msg.sender` (the pool): [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exact*`, the router calls `pool.swap(...)`, making `msg.sender` of `pool.swap` the **router contract**. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][originalUser]`.

A pool admin who wants allowlisted users to be able to use the router must add the router address to the allowlist. Once the router is allowlisted, the check `allowedSwapper[pool][router]` passes for **every** caller of the router — including users who are explicitly not on the per-user allowlist. The extension has no mechanism to recover the original user's identity from the router call.

The `DepositAllowlistExtension` does not share this flaw because it gates on `owner` (the position recipient), which the pool passes through correctly regardless of who the payer/sender is: [4](#0-3) 

---

### Impact Explanation

A curated pool using `SwapAllowlistExtension` to restrict trading to a known set of counterparties loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter` once the router is allowlisted. Unauthorized users can execute live swaps against oracle-priced liquidity, draining LP value at oracle-fair prices or front-running allowlisted users in a pool that was designed to be closed. This is a direct broken-core-functionality / unauthorized-swap impact.

---

### Likelihood Explanation

The pool admin is semi-trusted and has a legitimate reason to allowlist the router: without it, allowlisted users cannot use the router at all, which degrades UX (no slippage protection, no multi-hop). The admin cannot simultaneously support router-mediated swaps for allowlisted users and block non-allowlisted users, because the router is a single shared address. Any pool that enables router support for its allowlisted users is immediately exposed. The trigger requires no privileged action by the attacker — only a normal `MetricOmmSimpleRouter` call.

---

### Recommendation

The extension must identify the **original user**, not the intermediary. Two approaches:

1. **Pass original caller through router**: Have `MetricOmmSimpleRouter` supply the original `msg.sender` in `extensionData`, and update `SwapAllowlistExtension.beforeSwap` to decode and check that value when `sender` is a known trusted router.
2. **Transient-storage originator pattern**: The router writes the original caller into transient storage before calling `pool.swap`; the extension reads it back. This avoids trusting `extensionData` supplied by the user.

Either way, the extension must verify the identity of the economic actor, not the intermediary contract.

---

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured on `beforeSwap`.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — Alice is the intended sole swapper.
3. Pool admin calls `setAllowedToSwap(pool, router, true)` — to let Alice use the router.
4. Charlie (not on the allowlist) calls `MetricOmmSimpleRouter.exactInput(...)` targeting the pool.
5. The router calls `pool.swap(recipient, ...)` — `msg.sender` of `pool.swap` = router.
6. Pool calls `_beforeSwap(sender=router, ...)` → extension evaluates `allowedSwapper[pool][router]` → `true`.
7. Charlie's swap executes successfully against the curated pool, bypassing the per-user allowlist entirely. [3](#0-2) [5](#0-4)
