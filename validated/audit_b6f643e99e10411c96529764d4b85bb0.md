### Title
Child `ChunkExtra.proposed_split` not cleared after shard split, causing `InvalidChunkHeaderShardSplit` for first child chunk - (File: `chain/chain/src/resharding/manager.rs`)

---

### Summary

During dynamic resharding, `split_shard` in `chain/chain/src/resharding/manager.rs` creates each child shard's `ChunkExtra` by cloning the parent's `ChunkExtra` and updating only `state_root` and `congestion_info`. The `proposed_split` field (`ChunkExtraV5.proposed_split: Option<TrieSplit>`) is **not** reset to `None`. The child shards therefore inherit the parent's non-`None` split proposal. When the first chunk header for a child shard is produced in epoch N+1, `compute_proposed_split` returns `None` (the resharding cooldown blocks a new proposal). The chunk-header validator then compares `prev_chunk_extra.proposed_split = Some(parent_split)` against `chunk_header.proposed_split = None`, finds a mismatch, and rejects the chunk with `InvalidChunkHeaderShardSplit`. This is the exact analog of the DODO bug: a state-changing operation (shard split) updates some fields of a tracked record but leaves one critical field (`proposed_split`) stale, and a subsequent operation (chunk validation) reads that stale field and produces an incorrect result.

---

### Finding Description

**Operation that mutates state without updating the tracked field — `split_shard` in `manager.rs`:**

```rust
// TODO(resharding): set all fields of `ChunkExtra`. Consider stronger
// typing. Clarify where it should happen when `State` and
// `FlatState` update is implemented.
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
// ← proposed_split is NOT cleared; it retains parent's Some(TrieSplit)
chain_store_update.save_chunk_extra(block_hash, &new_shard_uid, child_chunk_extra.into());
``` [1](#0-0) 

The parent's final chunk (the last chunk of epoch N) is the one that proposed the split, so `parent_chunk_extra.proposed_split = Some(TrieSplit { boundary_account, left_memory, right_memory })`. After the clone, both child `ChunkExtra` records carry this same `Some(TrieSplit)`.

**Subsequent operation that reads the stale field — `validate_chunk_with_chunk_extra_and_receipts_root` in `validate.rs`:**

```rust
if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
    DYNAMIC_RESHARDING_VALIDATION_FAILURES.with_label_values(&["chunk_header"]).inc();
    return Err(Error::InvalidChunkHeaderShardSplit(...));
}
``` [2](#0-1) 

When the first chunk of a child shard is produced in epoch N+1, `compute_proposed_split` returns `None` because the resharding cooldown (`can_reshard`) blocks a new proposal. The chunk header therefore carries `proposed_split = None`. The validator reads `prev_chunk_extra.proposed_split = Some(parent_split)` from the stale child `ChunkExtra` and rejects the chunk.

The codebase's own documentation explicitly acknowledges the root cause:

> **`chain/chain/src/resharding/manager.rs:249`** — The resharding manager doesn't set all `ChunkExtra` fields (notably the new `proposed_split` field). [5](#0-4) 

And the comment in `get_upcoming_shard_split` confirms the divergent value path:

---

### Impact Explanation

Every dynamic resharding event produces two child shards whose initial `ChunkExtra` records carry a stale `proposed_split = Some(parent_split)`. The first chunk header produced for each child shard in epoch N+1 carries `proposed_split = None` (cooldown-enforced). `validate_chunk_with_chunk_extra_and_receipts_root` rejects every such chunk with `InvalidChunkHeaderShardSplit`. Because all honest chunk producers for the child shards compute the same `None`, no valid chunk can be accepted for either child shard at epoch N+1 height 1. This halts chunk production for both child shards immediately after every dynamic resharding event — a **Critical** protocol-level liveness failure.

---

### Likelihood Explanation

`ProtocolFeature::DynamicResharding` is assigned protocol version 85, which is below the current `STABLE_PROTOCOL_VERSION = 86`. [7](#0-6) 

The feature activates automatically whenever an `EpochConfig` with `ShardLayoutConfig::Dynamic` is deployed. No privileged action beyond normal epoch progression is required. The bug fires deterministically on the first block of epoch N+1 after any shard split — zero attacker involvement needed.

---

### Recommendation

In `split_shard` (`chain/chain/src/resharding/manager.rs`), after cloning the parent `ChunkExtra` for each child, explicitly clear the `proposed_split` field before saving:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
// ADD: clear the inherited proposed_split — child shards start with no pending split
if let ChunkExtra::V5(ref mut v5) = child_chunk_extra {
    v5.proposed_split = None;
}
chain_store_update.save_chunk_extra(block_hash, &new_shard_uid, child_chunk_extra.into());
```

This resolves the stale-field invariant: `prev_chunk_extra.proposed_split = None` for the child shard's first block, matching the `None` that `compute_proposed_split` returns under the resharding cooldown.

---

### Proof of Concept

1. Configure a network with `ShardLayoutConfig::Dynamic` and `min_epochs_between_resharding = 1`.
2. Drive a shard's trie memory above `memory_usage_threshold` so that `compute_proposed_split` returns `Some(split)` for the parent shard's last chunk of epoch N.
3. Observe that `parent_chunk_extra.proposed_split = Some(split)` is stored.
4. At the epoch N boundary, `split_shard` clones `parent_chunk_extra` into both child `ChunkExtra` records without clearing `proposed_split`.
5. In epoch N+1, the first chunk producer for either child shard calls `compute_proposed_split`; `can_reshard` returns `false` (cooldown), so `proposed_split = None` is embedded in the chunk header.
6. `validate_chunk_with_chunk_extra_and_receipts_root` compares `prev_chunk_extra.proposed_split = Some(split)` against `chunk_header.proposed_split = None`, finds a mismatch, and returns `Err(Error::InvalidChunkHeaderShardSplit(...))`.
7. Both child shards are unable to produce an accepted chunk at height 1 of epoch N+1; the chain stalls on those shards. [1](#0-0) [2](#0-1)
