Audit Report

## Title
Router-mediated swaps bind `sender` to the router address, enabling allowlist bypass on curated pools - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to every extension hook. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the original user. `SwapAllowlistExtension.beforeSwap()` checks `allowedSwapper[pool][sender]`, so it evaluates the router's address rather than the actual trader's address. A pool admin who allowlists the router address to enable router-mediated swaps for their permitted users inadvertently grants every unprivileged user the ability to bypass the allowlist.

## Finding Description

**Call path:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle()` (or `exactInput` / `exactOutputSingle` / `exactOutput`).
2. Router calls `IMetricOmmPoolActions(pool).swap(recipient, ...)` — at this point `msg.sender` inside the pool is the router address.
3. `MetricOmmPool.swap()` (`MetricOmmPool.sol:230`) calls `_beforeSwap(msg.sender, ...)`, forwarding the router address as `sender`.
4. `ExtensionCalling._beforeSwap()` (`ExtensionCalling.sol:149-176`) encodes `sender` (= router) and dispatches to every configured extension.
5. `SwapAllowlistExtension.beforeSwap()` (`SwapAllowlistExtension.sol:37`) evaluates:
   ```solidity
   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender])
   ```
   Here `msg.sender` = pool (correct), but `sender` = router (wrong actor).

**Root cause:** The pool has no mechanism to propagate the original EOA/contract through the router. The router never embeds the caller's identity in the swap call; it simply becomes the pool's `msg.sender`.

**Why existing guards fail:** The `onlyPool` modifier on the extension only verifies the pool is the caller of the extension — it does not verify that the `sender` argument reflects the true economic actor. The router's `_requireExpectedCallbackCaller` guard only validates the callback direction (pool → router), not the identity forwarded to extensions.

**Exploit flow:**
- Pool admin configures `SwapAllowlistExtension` to gate a curated pool.
- Admin allowlists specific users for direct pool calls.
- Admin also allowlists the router address (`setAllowedToSwap(pool, router, true)`) so that permitted users can trade via the router.
- Any unprivileged user calls `router.exactInputSingle({pool: curatedPool, ...})`.
- The extension sees `sender = router`, which is allowlisted → swap succeeds.
- The unprivileged user trades on a pool they were never permitted to access.

**Exact wrong value:** `sender` passed to `SwapAllowlistExtension.beforeSwap()` is `address(router)` instead of the originating user's address.

## Impact Explanation
An unprivileged trader can bypass a curated pool's swap allowlist by routing through `MetricOmmSimpleRouter`. This directly undermines the pool's access-control policy, allowing disallowed parties to execute swaps and extract value from LP positions that were only intended to be exposed to vetted counterparties. This is a direct allowlist bypass by an unprivileged public caller, matching the "Admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" and "Broken core pool functionality causing loss of funds" impact categories. Severity: High.

## Likelihood Explanation
The router is a standard, publicly deployed periphery contract. Any user can call it. The bypass requires only that the pool admin has allowlisted the router address — a natural and expected administrative action for any pool that intends to support router-mediated trading. No special privileges, flash loans, or oracle manipulation are required. The attack is repeatable on every transaction.

## Recommendation
The router must forward the originating caller's identity to the pool so extensions can gate on the true economic actor. One approach: add an optional `payer` field to the swap call signature, or use a dedicated transient storage slot (already used for callback context) that the pool reads and passes as `sender` to extensions when the direct caller is a known router. Alternatively, `SwapAllowlistExtension.beforeSwap()` should check both `sender` (the direct pool caller) and an additional `origin` argument representing the transaction originator, and the pool/extension interface should be extended to carry that value. At minimum, document that allowlisting the router grants access to all users and that per-user gating is only enforceable via direct pool calls.
