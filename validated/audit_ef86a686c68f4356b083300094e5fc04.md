### Title
`Treehasher._cached_sha256_treehash` Cache Is Prefix-Agnostic, Enabling Cross-Domain Hash Replay — (`wheel/python/clvm_rs/tree_hash.py`)

### Summary

The `Treehasher` class in `tree_hash.py` is designed to support configurable `atom_prefix` and `pair_prefix` values for domain-separated tree hashing. However, the per-object cache attribute `_cached_sha256_treehash` carries no domain identifier. Any hash computed and cached by one `Treehasher` instance is silently reused by any other `Treehasher` instance — including one with entirely different prefixes — producing a wrong hash without error. Additionally, the inner closures `handle_obj` and `handle_pair` inside `sha256_treehash` call the module-level `shatree_atom`/`shatree_pair` (bound to `CHIA_TREEHASHER` at import time) rather than `self.shatree_atom`/`self.shatree_pair`, so even on a cache miss a custom-prefix hasher silently computes and caches the CHIA hash.

### Finding Description

**Root cause — two compounding defects in `sha256_treehash`:**

**Defect 1 — inner closures ignore `self`'s prefix:**

Inside `Treehasher.sha256_treehash`, the nested functions `handle_obj` and `handle_pair` call the module-level names `shatree_atom` and `shatree_pair`:

```python
# tree_hash.py line 63
r = shatree_atom(obj.atom)          # always CHIA_TREEHASHER.shatree_atom
# tree_hash.py line 86
r = shatree_pair(p0, p1)            # always CHIA_TREEHASHER.shatree_pair
```

These names are bound at module load time to `CHIA_TREEHASHER`'s methods (lines 108–109). A `Treehasher` constructed with different prefixes therefore always computes the CHIA hash, regardless of its own `atom_prefix`/`pair_prefix`.

**Defect 2 — prefix-agnostic cache replay:**

`handle_obj` reads `_cached_sha256_treehash` from the object and returns it immediately if present, with no check of which prefix domain produced it:

```python
# tree_hash.py lines 57-61
r = getattr(obj, "_cached_sha256_treehash", None)
if r is not None:
    self.cache_hits += 1
    hash_stack.append(r)
    return
```

`handle_obj` and `handle_pair` then write the (CHIA-prefix) result back to the same attribute:

```python
setattr(obj, "_cached_sha256_treehash", r)   # lines 66, 90
```

So once any `CLVMStorage` object has been hashed by `CHIA_TREEHASHER`, every subsequent call to any `Treehasher.sha256_treehash` on that object — regardless of prefix — returns the cached CHIA hash. The custom hasher's `atom_prefix`/`pair_prefix` are never consulted.

**Exploit flow:**

1. Attacker-controlled CLVM bytes are deserialized into a `CLVMStorage` / `Program` object (e.g., via `Program.from_bytes`).
2. The object's tree hash is computed by `CHIA_TREEHASHER` (e.g., via `Program.tree_hash()`), setting `_cached_sha256_treehash = H_chia`.
3. A protocol layer creates a custom `Treehasher(atom_prefix=b"\x03", pair_prefix=b"\x04")` for domain separation and calls `custom_hasher.sha256_treehash(obj)`.
4. `handle_obj` finds `_cached_sha256_treehash = H_chia` and returns it immediately — the custom-prefix hash `H_custom ≠ H_chia` is never computed.
5. The protocol layer uses `H_chia` as if it were `H_custom`, collapsing two distinct hash domains into one.

### Impact Explanation

Any caller that instantiates a `Treehasher` with non-CHIA prefixes for domain separation receives silently wrong hashes. Because the cache is written back with the CHIA value, the corruption is permanent for the lifetime of the object. If the resulting hash is used for puzzle-hash verification, coin-ID derivation, or any authentication check, an attacker who can supply a CLVM object that was previously hashed in the CHIA domain can cause it to pass verification in a different domain — a direct hash-domain confusion / replay.

### Likelihood Explanation

`Treehasher` is a public, documented class with configurable prefix parameters, explicitly designed for reuse with different prefixes. `CurryTreehasher` already imports and uses the module-level `shatree_atom`/`shatree_pair` directly. Any downstream library or wallet code that constructs a `Treehasher` with custom prefixes (a natural use of the API) is silently affected. The attacker's only requirement is to supply a CLVM program that is hashed in the standard CHIA domain before being passed to the custom hasher — a trivially achievable condition via any normal deserialization path.

### Recommendation

1. **Fix the closure capture**: Replace the module-level calls with `self`-bound calls inside `sha256_treehash`:

```python
# use self.shatree_atom / self.shatree_pair, not the module-level names
r = self.shatree_atom(obj.atom)   # line 63
r = self.shatree_pair(p0, p1)     # line 86
```

2. **Key the cache by prefix domain**: Store the cached hash under a prefix-qualified attribute name, e.g. `_cached_treehash_{atom_prefix.hex()}_{pair_prefix.hex()}`, so that different `Treehasher` instances do not share a single cache slot.

### Proof of Concept

```python
from clvm_rs.tree_hash import Treehasher, CHIA_TREEHASHER
from clvm_rs.program import Program

# Build a simple atom object
obj = Program.to(b"hello")

# Hash with CHIA_TREEHASHER first — sets _cached_sha256_treehash
chia_hash = CHIA_TREEHASHER.sha256_treehash(obj)

# Create a custom hasher with different prefixes
custom_hasher = Treehasher(atom_prefix=b"\x03", pair_prefix=b"\x04")

# Expected: custom_hasher computes sha256(b"\x03" + b"hello")
# Actual:   returns the cached CHIA hash sha256(b"\x01" + b"hello")
custom_hash = custom_hasher.sha256_treehash(obj)

assert chia_hash == custom_hash, "Cache replay confirmed: custom hasher returned CHIA hash"
# This assertion passes — the two hashes are identical, proving the replay
```
