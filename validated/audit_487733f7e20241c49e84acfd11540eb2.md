Audit Report

## Title
`SwapAllowlistExtension.beforeSwap()` Checks Router Address as Swapper, Enabling Full Per-User Allowlist Bypass via Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap()` gates swaps by checking the `sender` parameter, which the pool sets to `msg.sender` of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, `sender` is the router's address, not the actual end user. If the pool admin allowlists the router — the necessary step to enable router-based swaps for permitted users — every unprivileged user can bypass the per-user restriction by calling through the router, completely defeating the allowlist.

## Finding Description

**The guard in `SwapAllowlistExtension.beforeSwap()`** checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (the caller of the extension) and `sender` is the first argument forwarded by the pool: [1](#0-0) 

**`MetricOmmPool.swap()` sets `sender` to its own `msg.sender`**, i.e., whoever called `pool.swap()`: [2](#0-1) 

**`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly**, making the router the `msg.sender` inside the pool, so `sender` forwarded to the extension is the router's address, not the end user: [3](#0-2) 

**`ExtensionCalling._beforeSwap()` passes `sender` unchanged** to every configured extension: [4](#0-3) 

The result is a binary trap for the pool admin:

| Admin configuration | What extension sees | Result |
|---|---|---|
| Allowlist `alice`, `bob` only | `sender = router` | Router not listed → all router swaps revert for everyone |
| Allowlist `alice`, `bob`, **and router** | `sender = router` | Router listed → **any user** can swap via router |

Neither configuration achieves "only `alice` and `bob` can swap through the router." The only configuration that makes the router usable silently removes the per-user gate entirely.

**Contrast with `DepositAllowlistExtension`**, which correctly checks `owner` — the position owner explicitly passed by the caller and forwarded unchanged — so it is not affected by intermediary routing: [5](#0-4) 

The swap extension has no equivalent "real actor" field to check; `sender` is structurally the immediate pool caller, not the end user.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` (KYC-gated, institutional-only, or whitelist-only) and with the router allowlisted loses its access control entirely for router-based swaps. Any unprivileged address can call `router.exactInputSingle()` and trade in the restricted pool. This is an admin-boundary break via an unprivileged path: the pool admin's intent to restrict swappers is fully circumvented without any special privileges, flash loans, or unusual token behavior. The corrupted value is the `allowedSwapper[pool][sender]` extension decision — it evaluates `true` for the router address on behalf of all users, not just the intended allowlisted ones.

## Likelihood Explanation
The scenario is realistic and likely. A pool admin deploying `SwapAllowlistExtension` who wants permitted users to access the standard router will naturally allowlist the router address — this is the only way to make the router functional for any user. The bypass requires no special privileges: any EOA or contract can call `router.exactInputSingle()` with the restricted pool as target. The attack is repeatable on every swap and requires no setup beyond a standard router call.

## Recommendation
The extension must identify the actual end user, not the intermediary. The most practical fix is to require routers wishing to be allowlisted to encode the actual swapper in `extensionData`. The extension would decode and verify that address instead of `sender`:

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata extensionData)
    external view override returns (bytes4)
{
    if (allowAllSwappers[msg.sender]) return IMetricOmmExtensions.beforeSwap.selector;
    // If extensionData is non-empty, treat it as the real swapper (router must encode msg.sender)
    address swapper = extensionData.length >= 20
        ? abi.decode(extensionData, (address))
        : sender;
    if (!allowedSwapper[msg.sender][swapper]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

This requires the router to pass `abi.encode(msg.sender)` as `extensionData` for allowlist-gated pools, which `MetricOmmSimpleRouter.exactInputSingle()` already forwards via `params.extensionData`. A structural fix (dedicated "on-behalf-of" field in the swap call) would be more robust but requires a breaking interface change.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` as `extension1` in `beforeSwap` order.
2. Admin calls `extension.setAllowedToSwap(pool, alice, true)` — only Alice is permitted.
3. Admin calls `extension.setAllowedToSwap(pool, router, true)` — router is allowlisted so Alice can use it.
4. Bob (not allowlisted) calls:
   ```solidity
   router.exactInputSingle(ExactInputSingleParams({
       pool: restrictedPool,
       recipient: bob,
       tokenIn: token0,
       amountIn: 1e18,
       amountOutMinimum: 0,
       zeroForOne: true,
       priceLimitX64: 0,
       deadline: block.timestamp,
       extensionData: ""
   }));
   ```
5. Router calls `pool.swap(bob, true, ...)` — pool records `msg.sender = router`.
6. Pool calls `_beforeSwap(router, bob, ...)` — `sender = router`.
7. Extension evaluates `allowedSwapper[pool][router] == true` → passes.
8. Bob's swap executes in the restricted pool despite not being on the allowlist.
