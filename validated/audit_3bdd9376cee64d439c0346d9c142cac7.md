### Title
Division-by-Zero in `PerpEngine.socializeSubaccount` When `openInterest == 0` Permanently Blocks Bankrupt Subaccount Finalization — (`core/contracts/PerpEngine.sol`)

### Summary

`PerpEngine.socializeSubaccount` unconditionally divides `balance.vQuoteBalance` by `state.openInterest` when insurance is insufficient to cover a negative vQuote balance. No guard exists for `state.openInterest == 0`. When all open positions in a product are closed (a valid market state), this division reverts with `ERR_DIV_BY_ZERO`, permanently blocking `ClearinghouseLiq._finalizeSubaccount` and leaving the bankrupt subaccount's bad debt unresolvable.

### Finding Description

In `PerpEngine.socializeSubaccount`:

```solidity
if (balance.vQuoteBalance < 0) {
    int128 insuranceCover = MathHelper.min(insurance, -balance.vQuoteBalance);
    insurance -= insuranceCover;
    balance.vQuoteBalance += insuranceCover;
    state.availableSettle += insuranceCover;

    if (balance.vQuoteBalance < 0) {
        // socialize across all other participants
        int128 fundingPerShare = -balance.vQuoteBalance.div(
            state.openInterest          // <-- reverts if openInterest == 0
        );
        ...
    }
}
```

`state.openInterest` is the sum of `|balance.amount|` across all subaccounts for a given product. It is a valid protocol state for `openInterest` to be zero — this happens when every participant has closed their position. `updateStates` already handles this case with an explicit `if (state.openInterest == 0) { continue; }` guard, confirming the developers know this value can be zero. No equivalent guard exists in `socializeSubaccount`.

A subaccount can have `balance.amount == 0` (position closed) but `balance.vQuoteBalance < 0` (realized loss from the trade). `_finalizeSubaccount` explicitly requires `balance.amount == 0` for all perp products before proceeding, so the liquidatee's contribution to `openInterest` is always zero at finalization time. If all other market participants have also closed their positions, `state.openInterest == 0` for that product.

### Impact Explanation

When this revert occurs:
1. `_finalizeSubaccount` reverts entirely (no partial state is committed).
2. The bankrupt subaccount can never be finalized — `productId == type(uint32).max` is the only finalization path and it always calls `socializeSubaccount` unconditionally.
3. The bad debt is permanently stuck: it cannot be socialized, insurance cannot be applied, and the subaccount cannot be closed.
4. The protocol carries unresolvable undercollateralization with no recovery path.

This is a Critical accounting/solvency failure: the socialization mechanism — the last-resort mechanism for distributing bad debt — is broken under a reachable market condition, leaving the protocol with irrecoverable bad debt.

### Likelihood Explanation

The condition is reachable without any privileged access:
- A subaccount closes a perp position at a loss (`amount → 0`, `vQuoteBalance < 0`).
- All other participants independently close their positions (normal market activity, or a low-liquidity/newly-listed product).
- `openInterest` reaches 0 for that product.
- The subaccount's quote balance is also insufficient to cover the loss (it is bankrupt).
- Any liquidator calls `liquidateSubaccountImpl` with `productId == type(uint32).max`.

This is more likely on low-liquidity or newly-listed perp products where open interest can naturally reach zero.

### Recommendation

Add a zero-check before the division, consistent with the existing guard in `updateStates`:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No participants to socialize to; absorb via insurance or write off
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

The protocol must also decide the policy when `openInterest == 0`: the loss has no participants to be distributed to, so it should either be absorbed by the insurance fund (already partially handled before this branch) or written off.
