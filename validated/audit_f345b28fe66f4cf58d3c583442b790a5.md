### Title
Withdrawal fee (`withdrawFeeX18`) is not charged in the slow-mode `WithdrawCollateral` path, enabling fee bypass — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The fast-path (`processTransactionImpl`) charges the per-product `withdrawFeeX18` fee on every `WithdrawCollateral` transaction. The slow-mode path (`processSlowModeTransactionImpl`) processes the identical withdrawal without charging this fee. Any unprivileged user can exploit this asymmetry to bypass the withdrawal fee entirely by routing their withdrawal through the slow-mode queue.

---

### Finding Description

**Fast-path** (`processTransactionImpl`, `EndpointTx.sol` lines 413–436):

```solidity
chargeFee(
    signedTx.tx.sender,
    spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18,
    signedTx.tx.productId
);
clearinghouse.withdrawCollateral(
    signedTx.tx.sender,
    signedTx.tx.productId,
    signedTx.tx.amount,
    address(0),
    nSubmissions
);
``` [1](#0-0) 

**Slow-mode path** (`processSlowModeTransactionImpl`, `EndpointTx.sol` lines 217–229):

```solidity
validateSender(txn.sender, sender);
clearinghouse.withdrawCollateral(
    txn.sender,
    txn.productId,
    txn.amount,
    address(0),
    nSubmissions
);
``` [2](#0-1) 

The slow-mode path calls `clearinghouse.withdrawCollateral` directly with no preceding `chargeFee` call. The `chargeFee` helper both deducts the fee from the user's spot balance and credits `sequencerFee[productId]`: [3](#0-2) 

Neither deduction occurs in the slow-mode path. The user pays only the flat `SLOW_MODE_FEE` ($1) at submission time: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

For any withdrawal where `withdrawFeeX18 * amount > SLOW_MODE_FEE`, the user saves money by using slow-mode. For example, at a 0.1% withdrawal fee rate (consistent with `FAST_WITHDRAWAL_FEE_RATE`), a $10,000 withdrawal incurs a $10 fee via the fast-path but only a $1 flat fee via slow-mode — a $9 saving per transaction. At scale or for large withdrawals, this drains protocol fee revenue that would otherwise accumulate in `sequencerFee[productId]` and be claimed by the protocol.

The broken invariant is: **all `WithdrawCollateral` executions must deduct `withdrawFeeX18` from the withdrawing subaccount's balance**. The slow-mode path violates this invariant unconditionally. [6](#0-5) 

---

### Likelihood Explanation

**High.** The entry point `submitSlowModeTransaction` is a public, permissionless function callable by any user: [7](#0-6) 

The only cost to the attacker is the flat $1 slow-mode fee and a 3-day delay (`SLOW_MODE_TX_DELAY`). No special privileges, linked signers, or sequencer cooperation are required. Any user who is aware of the asymmetry will rationally prefer slow-mode for large withdrawals. [8](#0-7) 

---

### Recommendation

In `processSlowModeTransactionImpl`, add the same `chargeFee` call that exists in the fast-path before invoking `clearinghouse.withdrawCollateral`:

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.WithdrawCollateral memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.WithdrawCollateral)
    );
    validateSender(txn.sender, sender);
+   chargeFee(
+       txn.sender,
+       spotEngine.getConfig(txn.productId).withdrawFeeX18,
+       txn.productId
+   );
    clearinghouse.withdrawCollateral(
        txn.sender,
        txn.productId,
        txn.amount,
        address(0),
        nSubmissions
    );
```

This enforces the same fee constraint in both execution paths, closing the asymmetry.

---

### Proof of Concept

1. User constructs `transaction = abi.encodePacked(uint8(TransactionType.WithdrawCollateral), abi.encode(WithdrawCollateral({sender: subaccount, productId: 1, amount: 100_000e6, nonce: 0})))`.
2. User calls `Endpoint.submitSlowModeTransaction(transaction)`, paying the flat $1 `SLOW_MODE_FEE` in quote tokens.
3. After 3 days, anyone calls `Endpoint.executeSlowModeTransaction()`.
4. `_executeSlowModeTransaction` calls `this.processSlowModeTransaction(sender, txn.tx)`.
5. `processSlowModeTransactionImpl` decodes the `WithdrawCollateral` struct, calls `validateSender`, then calls `clearinghouse.withdrawCollateral` — **no `withdrawFeeX18` is charged**.
6. The user receives the full $100,000 withdrawal. At a 0.1% fee rate, the protocol loses $100 in fee revenue per such transaction versus the fast-path. [9](#0-8) [10](#0-9)

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
