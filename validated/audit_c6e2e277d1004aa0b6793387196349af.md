### Title
Gateway `validate_resource_bounds` Unconditionally Checks `l2_gas.max_price_per_unit` Against `min_gas_price` Even When L2 Gas Amount Is Zero, Incorrectly Rejecting Valid Transactions - (File: crates/apollo_gateway/src/stateless_transaction_validator.rs)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` applies the `min_gas_price` floor check unconditionally to `l2_gas.max_price_per_unit` for every transaction, including those whose L2 gas `max_amount` is zero. A transaction that carries only L1 gas bounds (non-zero `l1_gas.max_amount` and `l1_gas.max_price_per_unit`, zero L2 gas) has a total fee well above zero and is economically valid, yet it is rejected at the gateway with `MaxGasPriceTooLow` because its `l2_gas.max_price_per_unit` is `0`, which is below the production `min_gas_price` of `8_000_000_000`. This is the sequencer-native analog of the external report's incorrect range check: a condition that should be guarded by "is this resource actually being used?" is instead applied unconditionally, causing valid transactions to be refused before sequencing.

### Finding Description

`validate_resource_bounds` performs two checks after the zero-fee guard:

```rust
// Line 66 – zero-fee guard (passes for L1-only txs because l1_gas fee > 0)
if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0) { … }

// Line 71 – min-price check applied to l2_gas unconditionally
if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
    return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow { … });
}
```

`AllResourceBounds` carries three independent resource slots (`l1_gas`, `l2_gas`, `l1_data_gas`). When a sender sets `l2_gas.max_amount = 0` and `l2_gas.max_price_per_unit = 0` (the `Default` value), the L2 gas price is semantically irrelevant — no L2 gas will be consumed and no L2 fee will be charged. The zero-fee guard at line 66 correctly passes such a transaction when `l1_gas` is non-zero. However, the subsequent check at line 71 reads `l2_gas.max_price_per_unit.0` (which is `0`) and compares it against `min_gas_price = 8_000_000_000`, producing a spurious `MaxGasPriceTooLow` rejection.

The test suite avoids this by using `DEFAULT_VALIDATOR_CONFIG_FOR_TESTING` which sets `min_gas_price: 0`, masking the production-only failure. The production default is `8_000_000_000`.

### Impact Explanation

Any user submitting a V3 transaction with only L1 gas bounds (`l1_gas` non-zero, `l2_gas` and `l1_data_gas` at default zero) will have their transaction rejected at the gateway with `MaxGasPriceTooLow`, even though the transaction is structurally and economically valid. This is a gateway admission defect: **valid transactions are rejected before sequencing**, matching the "High – Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing" impact category.

### Likelihood Explanation

`AllResourceBounds` with only L1 gas set is a well-defined and supported transaction shape (it is the natural form for pre-0.13.3 style V3 transactions). The `valid_l1_gas` test case in the test suite explicitly constructs such a transaction and expects it to pass — it only passes in tests because `min_gas_price` is set to `0` there. In production, every such transaction is rejected. Any wallet or SDK that constructs a V3 transaction with only L1 gas bounds will be silently blocked.

### Recommendation

Guard the `min_gas_price` check with a condition that the L2 gas resource is actually being used:

```rust
if resource_bounds.l2_gas.max_amount.0 > 0
    && resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price
{
    return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
        gas_price: resource_bounds.l2_gas.max_price_per_unit,
        min_gas_price: self.config.min_gas_price,
    });
}
```

Similarly, consider whether `min_gas_price` should also be enforced for `l1_gas.max_price_per_unit` and `l1_data_gas.max_price_per_unit` when those resources are non-zero.

### Proof of Concept

1. Construct a V3 Invoke transaction with:
   - `l1_gas: ResourceBounds { max_amount: GasAmount(1000), max_price_per_unit: GasPrice(1_000_000_000) }`
   - `l2_gas: ResourceBounds::default()` → `max_amount = 0, max_price_per_unit = 0`
   - `l1_data_gas: ResourceBounds::default()`
2. Submit to the gateway with production config (`min_gas_price = 8_000_000_000`, `validate_resource_bounds = true`).
3. `max_possible_fee(Tip::ZERO)` = `1000 * 1_000_000_000 = 1_000_000_000 > 0` → zero-fee guard passes.
4. `l2_gas.max_price_per_unit.0 = 0 < 8_000_000_000` → `MaxGasPriceTooLow` is returned.
5. The transaction is rejected despite being economically valid.

The root cause is at line 71 of `validate_resource_bounds`, which applies the price floor to `l2_gas.max_price_per_unit` without first checking whether `l2_gas.max_amount > 0`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        {
            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
        }

        if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
            return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
                gas_price: resource_bounds.l2_gas.max_price_per_unit,
                min_gas_price: self.config.min_gas_price,
            });
        }

        // TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
        if let RpcTransaction::Declare(_) = tx {
        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
            return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
                gas_amount: resource_bounds.l2_gas.max_amount,
                max_gas_amount: self.config.max_l2_gas_amount,
            });
        }

        Ok(())
    }
