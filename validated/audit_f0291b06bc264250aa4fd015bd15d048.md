### Title
`burnNlp` Uses Weaker MAINTENANCE Health Check, Allowing Voluntary Undercollateralization Below Initial Margin — (File: `core/contracts/Clearinghouse.sol`)

### Summary
`Clearinghouse.burnNlp` enforces only a `MAINTENANCE` health check after reducing a subaccount's collateral, while every other voluntary collateral-reducing operation (`withdrawCollateral`, `transferQuote`, `mintNlp`, `nlpProfitShare`, `forceRebalanceNlpPool`) enforces an `INITIAL` health check. A user with open borrows can exploit this asymmetry to extract collateral that the initial margin requirement would otherwise lock, leaving their subaccount in a state that is below initial margin but above maintenance margin — a form of undercollateralization that the protocol's own invariants prohibit for all other operations.

### Finding Description
In `Clearinghouse.sol`, the `burnNlp` function converts a user's NLP tokens back into quote tokens. After updating balances, it performs a health check:

```solidity
// Clearinghouse.sol:526-529
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
```

Every other function that voluntarily reduces a subaccount's collateral uses `INITIAL` health:

| Function | Health Type Used |
|---|---|
| `withdrawCollateral` | `INITIAL` |
| `transferQuote` | `INITIAL` (via `_isAboveInitial`) |
| `mintNlp` | `INITIAL` |
| `nlpProfitShare` | `INITIAL` |
| `forceRebalanceNlpPool` | `INITIAL` |
| **`burnNlp`** | **`MAINTENANCE`** ← weaker |

The `INITIAL` and `MAINTENANCE` weights are distinct per-product values stored in `RiskHelper.RiskStore` (`longWeightInitial` vs `longWeightMaintenance`). Because `MAINTENANCE` weights are always more permissive than `INITIAL` weights, a user can burn NLP and end up with:

- `getHealth(sender, MAINTENANCE) >= 0` — passes the check
- `getHealth(sender, INITIAL) < 0` — would fail any other collateral-reducing check

This gap is the exploitable zone. The user has extracted collateral that the initial margin requirement was designed to keep locked.

### Impact Explanation
A user with borrowed funds (negative quote balance) and NLP tokens as collateral can use `burnNlp` to receive quote tokens back while reducing their health to just above MAINTENANCE. The extracted quote tokens represent collateral that should have been locked by the INITIAL margin requirement. The subaccount is now below initial margin — undercollateralized from the protocol's perspective — but not yet liquidatable (liquidation requires MAINTENANCE health < 0). This creates a window of undercollateralization that cannot be resolved by liquidation, violating the protocol's core solvency invariant for voluntary operations.

### Likelihood Explanation
The entry path is fully unprivileged: any user who holds NLP tokens and has open borrows can sign a `BurnNlp` transaction and submit it through the normal sequencer path (`EndpointTx.processTransactionImpl` → `clearinghouse.burnNlp`). No special access, governance capture, or external dependency is required. The condition (holding NLP while having borrows) is a normal user state in the NLP system.

### Recommendation
Replace the `MAINTENANCE` health check in `burnNlp` with an `INITIAL` health check, consistent with all other voluntary collateral-reducing operations:

```solidity
// Replace:
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);

// With:
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
```

If the burn fee can legitimately cause a small health decrease (as the comment suggests), the correct fix is still to use `INITIAL` and disallow burns that would push the user below initial margin, rather than silently permitting undercollateralization.

### Proof of Concept

1. Alice deposits 100 USDC as collateral and mints NLP tokens worth 100 USDC. Her INITIAL health is positive.
2. Alice borrows 60 USDC against her collateral. Her INITIAL health is now at the margin (e.g., just above 0).
3. Alice attempts `withdrawCollateral` for any amount — this fails because her INITIAL health would drop below 0.
4. Alice instead signs a `BurnNlp` transaction to burn her NLP tokens and receive quote tokens back.
5. `burnNlp` executes: NLP balance decreases, quote balance increases, but the net effect on health is negative because the burn fee and the NLP weight reduction outpace the quote gain.
6. The MAINTENANCE check at line 527 passes (health is still above MAINTENANCE threshold).
7. Alice's INITIAL health is now negative — she is undercollateralized — but she is not liquidatable.
8. Alice has extracted collateral that `withdrawCollateral` would have blocked.
