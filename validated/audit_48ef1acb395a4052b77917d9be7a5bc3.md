### Title
Unguarded `creditDeposit()` Allows Sanctioned Caller to Bypass Depositor Sanctions Check — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` is an `external` function with no access control. When invoked, it calls `endpoint.depositCollateralWithReferral()` with `msg.sender` equal to the DDA contract address. The depositor sanctions check inside `depositCollateralWithReferral` evaluates `requireUnsanctioned(msg.sender)` — which resolves to the DDA contract, not the actual human caller — allowing a sanctioned user to deposit tokens into the protocol while bypassing the depositor-side sanctions enforcement.

---

### Finding Description

`DirectDepositV1.creditDeposit()` iterates over all registered product IDs, approves each token balance held by the DDA contract, and calls `endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1")` for each non-zero balance. [1](#0-0) 

The function carries no `onlyOwner` or any other access modifier, making it callable by any external address.

Inside `Endpoint.depositCollateralWithReferral`, the protocol enforces two sanctions checks:

```solidity
requireUnsanctioned(msg.sender);   // checks the DDA contract address
requireUnsanctioned(sender);       // checks the subaccount owner address
``` [2](#0-1) 

Because `creditDeposit()` is the direct caller of `depositCollateralWithReferral`, `msg.sender` inside that function is the DDA contract — not the human who triggered `creditDeposit()`. The DDA contract itself is never sanctioned, so `requireUnsanctioned(msg.sender)` always passes regardless of who called `creditDeposit()`.

The second check, `requireUnsanctioned(sender)`, evaluates the address embedded in the `subaccount` bytes32 (the subaccount owner). A sanctioned attacker who deploys a DDA pointing to a fresh, non-sanctioned subaccount owner address they control passes both checks.

---

### Impact Explanation

A sanctioned user (Alice) can:

1. Deploy a `DirectDepositV1` contract with a fresh, non-sanctioned address as the `subaccount` owner (an address Alice controls but that is not yet on the sanctions list).
2. Transfer tokens to the DDA.
3. Call `creditDeposit()` — `msg.sender` inside `depositCollateralWithReferral` is the DDA contract (not sanctioned); `sender` is the fresh subaccount owner (not sanctioned). Both checks pass.
4. Alice's tokens are credited to the protocol subaccount she controls, fully bypassing the depositor sanctions enforcement.

This is a direct analog to the external report's finding: just as `addLiquidity` allowed sanctioned users to bypass the AccessManager geolocation check by calling the function directly, `creditDeposit()` allows sanctioned users to bypass `requireUnsanctioned(msg.sender)` by routing the deposit through the DDA contract. [1](#0-0) [3](#0-2) 

---

### Likelihood Explanation

**Medium.** The attacker must deploy a `DirectDepositV1` contract (permissionless, no admin required) and control a non-sanctioned subaccount owner address. Both steps are achievable by any on-chain actor with no privileged access. The only friction is the cost of deploying the DDA contract and the need to use a fresh address as the subaccount owner.

---

### Recommendation

Add an `onlyOwner` modifier to `creditDeposit()` so that only the DDA owner can trigger deposits:

```solidity
function creditDeposit() external onlyOwner {
    ...
}
```

Additionally, consider passing the actual `msg.sender` of `creditDeposit()` through to `depositCollateralWithReferral` so the sanctions check evaluates the human caller rather than the intermediary contract.

---

### Proof of Concept

1. Alice is a sanctioned address.
2. Alice deploys `DirectDepositV1(endpoint, spotEngine, freshSubaccount, wrappedNative)` where `freshSubaccount` encodes a fresh address Alice controls that is not on the sanctions list.
3. Alice transfers 1000 USDC to the DDA contract.
4. Alice calls `dda.creditDeposit()`.
5. Inside `depositCollateralWithReferral`:
   - `requireUnsanctioned(msg.sender)` → `msg.sender` is the DDA contract → **passes**.
   - `requireUnsanctioned(sender)` → `sender` is the fresh subaccount owner → **passes**.
6. 1000 USDC is deposited into Alice's controlled subaccount. Alice has successfully deposited into the protocol despite being sanctioned, bypassing the depositor-side sanctions check. [1](#0-0) [4](#0-3)

    function creditDeposit() external {
        uint32[] memory productIds = spotEngine.getProductIds();
        for (uint256 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            address tokenAddr = spotEngine.getToken(productId);
            require(tokenAddr != address(0), "Invalid productId.");
            IIERC20Base token = IIERC20Base(tokenAddr);
            uint256 balance = token.balanceOf(address(this));
            if (balance != 0) {
                token.approve(address(endpoint), balance);
                endpoint.depositCollateralWithReferral(
                    subaccount,
                    productId,
                    uint128(balance),
                    "-1"
                );
            }
        }
    }

    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned
        requireUnsanctioned(msg.sender);
        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
            return;
        }

        handleDepositTransfer(
            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)
        );
