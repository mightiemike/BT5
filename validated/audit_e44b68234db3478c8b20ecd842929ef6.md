### Title
Unvalidated `_cached_sha256_treehash` Trusted Blindly in `sha256_treehash`, Enabling Puzzle Hash Forgery - (File: `wheel/python/clvm_rs/tree_hash.py`)

---

### Summary

`sha256_treehash` in the Python wheel unconditionally trusts the `_cached_sha256_treehash` attribute on any `CLVMStorage` object without verifying it matches the object's actual content. An attacker-controlled Python object with a pre-set forged hash causes `Program.tree_hash()`, `Program.__eq__()`, and `Program.curry_hash()` to operate on a fabricated value, corrupting puzzle hash derivation.

---

### Finding Description

`Treehasher.sha256_treehash` in `wheel/python/clvm_rs/tree_hash.py` implements a non-recursive tree hash traversal. At the top of `handle_obj`, it reads the cached hash from any node:

```python
r = getattr(obj, "_cached_sha256_treehash", None)
if r is not None:
    self.cache_hits += 1
    hash_stack.append(r)
    return
``` [1](#0-0) 

There is no check that `r` is 32 bytes, that it is `bytes` at all, or that it equals `sha256(prefix || content)` for the node's actual atom or pair children. Any non-`None` value is accepted and propagated up the hash stack.

`Program.wrap()` copies this attribute verbatim from any wrapped `CLVMStorage`:

```python
o._cached_sha256_treehash = getattr(v, "_cached_sha256_treehash", None)
``` [2](#0-1) 

`Program.tree_hash()` then short-circuits on the copied value:

```python
def tree_hash(self) -> bytes:
    if self._cached_sha256_treehash is None:
        self._cached_sha256_treehash = sha256_treehash(self._unwrapped)
    return self._cached_sha256_treehash
``` [3](#0-2) 

The `CLVMStorage` protocol explicitly documents `_cached_sha256_treehash` as an optional speed hint, making it a well-known attribute that any conforming object may supply: [4](#0-3) 

---

### Impact Explanation

Three downstream consumers are directly corrupted:

1. **`Program.__eq__`** compares programs by `tree_hash()`:
   ```python
   return self.tree_hash() == other_obj.tree_hash()
   ``` [5](#0-4) 
   A forged hash makes two programs with different content compare as equal (or unequal when they should match), bypassing puzzle identity checks in wallet code.

2. **`Program.curry_hash`** derives puzzle hashes from `self.tree_hash()`:
   ```python
   quoted_mod_hash = curry_treehasher.calculate_hash_of_quoted_mod_hash(self.tree_hash())
   return curry_treehasher.curry_and_treehash(quoted_mod_hash, *args)
   ``` [6](#0-5) 
   A forged mod hash produces an incorrect curried puzzle hash, corrupting coin address derivation.

3. **Sub-node hash propagation**: because `handle_obj` trusts the cached hash of any sub-node, a single forged leaf causes every ancestor pair's hash to be computed from the forged value, silently corrupting the entire tree hash.

---

### Likelihood Explanation

The `CLVMStorage` protocol is duck-typed; any Python object with `atom` and `pair` attributes qualifies. `to_clvm_object` passes `CLVMStorage` objects through unchanged:

```python
if is_clvm_storage(v):
    did_convert.append(v)
    continue
``` [7](#0-6) 

Any caller that accepts user-supplied `CLVMStorage` objects (e.g., a wallet parsing third-party puzzle descriptions, a library accepting plugin-provided CLVM trees, or a test harness) and then calls `.tree_hash()`, `==`, or `.curry_hash()` is vulnerable. No privileged access is required; only the ability to pass a Python object with `_cached_sha256_treehash` set to an arbitrary value.

---

### Recommendation

Validate the cached hash before trusting it. The simplest fix is to remove the cache shortcut from `handle_obj` for externally-supplied objects, or to verify the cached value matches the computed hash on first use. At minimum, add a type and length guard:

```python
r = getattr(obj, "_cached_sha256_treehash", None)
if isinstance(r, bytes) and len(r) == 32:
    ...
```

A stronger fix is to only populate `_cached_sha256_treehash` from within the trusted `Treehasher` itself (i.e., never read it from external objects, only write it after computing the correct value), mirroring how the Rust `ObjectCache` computes and stores values internally without trusting external input. [8](#0-7)

### Proof of Concept

```python
from clvm_rs.program import Program
from clvm_rs.clvm_storage import CLVMStorage

class MaliciousStorage(CLVMStorage):
    atom = b"real_content"
    pair = None
    # Pre-set a completely fabricated 32-byte hash
    _cached_sha256_treehash = b"\xde\xad" * 16

evil = MaliciousStorage()
p = Program.to(evil)

# Returns the forged hash, not sha256(b"\x01" + b"real_content")
print(p.tree_hash().hex())  # deaddeaddeaddeaddeaddeaddeaddeaddeaddeaddeaddeaddeaddeaddeaddead

# Program equality is now based on the forged hash
p2 = Program.new_atom(b"different_content")
p2._cached_sha256_treehash = b"\xde\xad" * 16
print(p == p2)  # True — different content, same forged hash

# curry_hash derives an incorrect puzzle hash
print(p.curry_hash())  # computed from forged mod hash
```

The forged hash propagates through `Program.wrap()` at line 142 and is returned unchecked by `Program.tree_hash()` at line 284, with the root cause being the unconditional trust of `_cached_sha256_treehash` in `handle_obj` at line 57–61 of `tree_hash.py`. [9](#0-8) [10](#0-9)
