### Title
`burnNlp` Uses Weaker MAINTENANCE Health Check Instead of INITIAL, Enabling Collateral Extraction from Borderline Accounts — (`File: core/contracts/Clearinghouse.sol`)

### Summary

The `burnNlp` function enforces only a `MAINTENANCE` health check after burning NLP tokens, while every other collateral-reducing operation (`withdrawCollateral`) enforces the stricter `INITIAL` health check. This asymmetry allows a user whose account sits between the MAINTENANCE and INITIAL health thresholds to extract quote value through `burnNlp` when all standard withdrawal paths would be blocked.

### Finding Description

`withdrawCollateral` enforces `HealthType.INITIAL` for all normal subaccounts:

```solidity
IProductEngine.HealthType healthType = sender == X_ACCOUNT
    ? IProductEngine.HealthType.PNL
    : IProductEngine.HealthType.INITIAL;
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
``` [1](#0-0) 

`burnNlp`, which is also a value-extracting operation (it converts NLP into quote tokens), enforces only `HealthType.MAINTENANCE`:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [2](#0-1) 

The asymmetry is reinforced by `mintNlp`, which correctly uses `INITIAL` health on the deposit side:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [3](#0-2) 

So the stricter check (`INITIAL`) is applied when adding NLP exposure, but the weaker check (`MAINTENANCE`) is applied when removing it — a direct analog to the external report's pattern of a validation check being placed in the wrong function.

### Impact Explanation

A user whose account health is between the MAINTENANCE and INITIAL thresholds (i.e., not yet liquidatable but blocked from standard withdrawals) can call `burnNlp` to convert locked NLP into quote tokens. This bypasses the INITIAL health gate that `withdrawCollateral` enforces, allowing value extraction from an account that the protocol considers too risky for normal withdrawals. The extracted quote tokens can then be withdrawn via a separate `withdrawCollateral` call (which would now pass INITIAL health because the quote balance increased), or the user can simply leave the account in a maintenance-only-healthy state, increasing liquidation risk for the protocol.

### Likelihood Explanation

Medium. The exploit requires the user's account to be in the MAINTENANCE-to-INITIAL health band, which is a specific but realistic state (e.g., after adverse price movement). The `BurnNlp` transaction type is a standard sequencer-processed path reachable by any user who holds NLP. No privileged access is required. [4](#0-3) 

### Recommendation

Replace `HealthType.MAINTENANCE` with `HealthType.INITIAL` in the post-burn health check inside `burnNlp`, consistent with how `withdrawCollateral` and `mintNlp` treat collateral-affecting operations. If a weaker check is intentionally desired for NLP burns (e.g., to handle burn-fee edge cases), the rationale should be explicitly documented and the threshold should be no weaker than INITIAL.

### Proof of Concept

1. User mints NLP: account passes INITIAL health check. NLP is locked for `NLP_LOCK_PERIOD` (4 days). [5](#0-4) 

2. Price moves adversely. Account health drops: MAINTENANCE health = +5, INITIAL health = -3.

3. User attempts `withdrawCollateral` → reverts because INITIAL health < 0. [1](#0-0) 

4. After 4 days, user calls `burnNlp` for their full NLP balance. The MAINTENANCE health check passes (+5 ≥ 0). Quote tokens are credited to the user's subaccount. [6](#0-5) 

5. User now holds quote tokens extracted from an account that was blocked from standard withdrawal, having bypassed the INITIAL health gate that governs all other collateral-reducing operations.
