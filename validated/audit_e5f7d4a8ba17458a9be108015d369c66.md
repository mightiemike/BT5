Audit Report

## Title
SwapAllowlistExtension Bypassed via MetricOmmSimpleRouter: Router Address Checked Instead of Originating User — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the router becomes `msg.sender` of `pool.swap()`, so the extension checks the router's allowlist status rather than the originating user's. If the pool admin allowlists the router (the only way to permit router-based swaps for legitimate users), every unpermissioned user can bypass the swap allowlist by routing through the router.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes `sender` verbatim into the extension call: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router `msg.sender` of that call: [4](#0-3) 

The same pattern holds for `exactInput`, `exactOutputSingle`, and `exactOutput` — in every case the router is the immediate caller of `pool.swap()`: [5](#0-4) [6](#0-5) [7](#0-6) 

The extension sees `sender = router_address`, not the originating EOA. The pool admin faces an inescapable dilemma: not allowlisting the router blocks all legitimate users from using it, while allowlisting the router lets every non-allowlisted user bypass the guard via the router. There is no configuration that simultaneously permits allowlisted users to use the router and blocks non-allowlisted users from doing so. The exact wrong value is `allowedSwapper[pool][router]` being evaluated instead of `allowedSwapper[pool][originating_user]`.

## Impact Explanation
`SwapAllowlistExtension` is the production mechanism for restricting pool access to specific counterparties (e.g., KYC'd addresses, designated market makers, or institutional LPs). Once the router is allowlisted — the only way to give legitimate users router access — the guard is completely neutralised. Any unpermissioned address can execute swaps against a pool explicitly configured to deny them. This constitutes a broken core pool functionality / admin-boundary break: an unprivileged path (the public router) lets any user bypass a configured access guard with direct fund-flow consequences, as the pool's liquidity is accessible at oracle prices to addresses the pool admin intended to exclude.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap interface; pool admins who deploy a swap-allowlisted pool and want to support router users will naturally allowlist the router. No special privilege, token balance, or setup is required — any EOA can call `exactInputSingle` on the router. The bypass is unconditional once the router is allowlisted; it does not depend on timing, oracle state, or any other condition.

## Recommendation
The extension must gate on the originating user, not the immediate caller of `pool.swap()`. Two complementary fixes:

1. **Pass the originating sender through the router.** The router already stores the originating `msg.sender` in transient storage as the payer (via `_setNextCallbackContext`). Expose it as a dedicated `extensionData` field or a separate transient slot that the extension can read, and have `SwapAllowlistExtension` verify that address instead of (or in addition to) the `sender` argument.

2. **Alternatively, check both `sender` and the originating user.** If `sender` is a known router/aggregator, the extension should look up the actual payer from transient context rather than treating the router as the swapper.

A simpler short-term mitigation: document that `SwapAllowlistExtension` is incompatible with `MetricOmmSimpleRouter` and revert in `beforeSwap` if `sender` is the router address, forcing direct-pool-only access for allowlisted pools.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true          // alice is the only permitted swapper
  allowedSwapper[pool][router] = true         // admin allowlists router so alice can use it

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, ...})

  router calls:
    pool.swap(recipient, ...)                 // msg.sender = router

  pool calls _beforeSwap(sender=router, ...)
  _beforeSwap encodes sender=router and calls extension

  extension checks:
    allowedSwapper[pool][router] == true      // PASSES — bob's swap executes

Result:
  bob swaps on a pool he is not permitted to access.
  The swap allowlist is completely bypassed.
```

Foundry test outline:
1. Deploy pool with `SwapAllowlistExtension` configured in `beforeSwapOrder`.
2. Call `setAllowedToSwap(pool, alice, true)` and `setAllowedToSwap(pool, router, true)`.
3. From `bob` (not allowlisted), call `router.exactInputSingle(...)` targeting the pool.
4. Assert the swap succeeds (no `NotAllowedToSwap` revert), confirming the bypass.
