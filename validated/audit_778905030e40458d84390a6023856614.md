Audit Report

## Title
`SwapAllowlistExtension` checks router address as swapper identity, enabling allowlist bypass for any user via the router - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `sender`, which the pool sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the end-user. If the pool admin allowlists the router to enable router-based swaps for permitted users, every unprivileged user can bypass the allowlist by calling the router, because the extension only sees the router address.

## Finding Description

**Call chain:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle()` (or any other swap entry point).
2. The router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — at this point `msg.sender` inside the pool is the **router address**.
3. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` — passing the router address as `sender`.
4. `ExtensionCalling._beforeSwap` encodes and dispatches `IMetricOmmExtensions.beforeSwap(sender=router, ...)` to the extension.
5. `SwapAllowlistExtension.beforeSwap` evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the **router**, not the end-user.

**Root cause:** `MetricOmmPool.swap` passes `msg.sender` (the direct caller) as `sender` to the extension hook. The router is always the direct caller, so the extension can never distinguish individual end-users when they go through the router.

**Exploit path:**
- Pool admin creates a restricted pool with `SwapAllowlistExtension`, allowlisting specific KYC'd users.
- Pool admin also allowlists the router address so those users can use the router.
- Any non-allowlisted user calls `MetricOmmSimpleRouter.exactInputSingle()` targeting the restricted pool.
- The extension sees `sender = router`, which is allowlisted, and permits the swap.
- The non-allowlisted user successfully swaps in a pool they should be barred from.

**Existing guards reviewed and found insufficient:**
- `BaseMetricExtension.onlyPool` only verifies `msg.sender` is a factory pool — it does not validate the `sender` argument.
- The router's `_requireFactoryPool` check only validates the pool address, not the end-user identity.
- There is no mechanism for the router to forward the original `msg.sender` to the extension in a trustworthy way; `extensionData` is user-controlled and not used by the extension for identity.

## Impact Explanation
The `SwapAllowlistExtension` is a core access-control primitive. Its bypass allows unprivileged users to execute swaps in pools explicitly configured to restrict access (e.g., permissioned or KYC-gated pools). The pool admin cannot simultaneously allow allowlisted users to use the router and block non-allowlisted users from doing the same — the two goals are mutually exclusive given the current design. This constitutes a broken core pool access-control invariant reachable by any unprivileged router caller.

## Likelihood Explanation
The bypass requires the pool admin to have allowlisted the router address, which is a natural and expected configuration step for any pool that intends to support router-based trading for its permitted users. The attacker needs only to call the public router with the restricted pool address — no special privileges, no flash loans, no complex setup. The condition is repeatable on every block.

## Recommendation
Pass the original end-user address through the hook rather than the direct caller. Options:
1. Have the router encode the original `msg.sender` into `extensionData` and have the extension decode and verify it (requires a trusted encoding convention).
2. Add a separate `originSender` field to the extension hook interface that the pool populates from a transient-storage slot set by the router before calling `pool.swap`.
3. Alternatively, document that the allowlist gates the direct caller only, and require pool admins to allowlist individual user addresses rather than the router — and never allowlist the router when per-user gating is intended.

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Setup:
// 1. Deploy pool with SwapAllowlistExtension.
// 2. Pool admin calls setAllowedToSwap(pool, allowedUser, true).
// 3. Pool admin calls setAllowedToSwap(pool, address(router), true)
//    (to let allowedUser use the router).
// 4. Non-allowlisted attacker calls:

router.exactInputSingle(ExactInputSingleParams({
    pool:           restrictedPool,
    recipient:      attacker,
    zeroForOne:     true,
    amountIn:       1e18,
    amountOutMinimum: 0,
    priceLimitX64:  0,
    tokenIn:        token0,
    deadline:       block.timestamp,
    extensionData:  ""
}));
// Extension sees sender = address(router), which IS allowlisted → swap succeeds.
// Attacker bypasses the per-user allowlist.
```
