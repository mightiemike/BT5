### Title
Insurance Fund Accounting Inflated Without Actual Token Transfer — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.depositInsurance` increments the `insurance` accounting variable without ever pulling tokens into the contract. When insurance is later withdrawn or used to cover bad debt during liquidation, the clearinghouse either reverts (if it holds no tokens) or silently drains user collateral tokens that happen to be held in the same contract.

---

### Finding Description

In `Clearinghouse.sol`, `depositInsurance` is:

```solidity
function depositInsurance(bytes calldata transaction)
    external
    virtual
    onlyEndpoint
{
    IEndpoint.DepositInsurance memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.DepositInsurance)
    );
    require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
    int256 multiplier = int256(
        10**(MAX_DECIMALS - _decimals(QUOTE_PRODUCT_ID))
    );
    int128 amount = int128(txn.amount) * int128(multiplier);
    insurance += amount;   // ← accounting updated, no transferFrom
}
``` [1](#0-0) 

The caller-side entry point in `ContractOwner.depositInsurance` approves the endpoint to spend tokens, then submits a slow-mode transaction:

```solidity
function depositInsurance(uint128 amount) external onlyOwner {
    IERC20Base quoteToken = IERC20Base(spotEngine.getToken(QUOTE_PRODUCT_ID));
    quoteToken.approve(address(endpoint), uint256(amount));   // approval only
    IEndpoint.DepositInsurance memory _txn = IEndpoint.DepositInsurance(amount);
    _submitSlowModeTransaction(
        IEndpoint.TransactionType.DepositInsurance,
        abi.encode(_txn)
    );
}
``` [2](#0-1) 

The approval is granted to the endpoint, but the endpoint's processing of `DepositInsurance` (in `EndpointTx.sol`) only calls `clearinghouse.depositInsurance(transaction)` — no `transferFrom` is ever executed: [3](#0-2) 

Contrast this with `depositCollateral`, where the Endpoint transfers tokens before updating accounting. For insurance, that transfer step is entirely absent.

When `withdrawInsurance` is later called, it checks `amount <= insurance` (the inflated accounting value), decrements it, and then calls `handleWithdrawTransfer`:

```solidity
require(amount <= insurance, ERR_NO_INSURANCE);
insurance -= amount;
...
handleWithdrawTransfer(token, txn.sendTo, txn.amount, idx);
``` [4](#0-3) 

`handleWithdrawTransfer` does `token.safeTransfer(withdrawPool, amount)` — pulling from the clearinghouse's actual token balance: [5](#0-4) 

Because the clearinghouse holds user collateral for all products, this transfer either reverts (if the clearinghouse is empty) or silently drains user collateral tokens.

The same `insurance` variable is consumed during liquidation via `PerpEngine.socializeSubaccount`, where it is used to cover bad debt: [6](#0-5) 

An inflated `insurance` value causes the protocol to believe it can absorb more bad debt than it actually can, corrupting the solvency invariant.

---

### Impact Explanation

**High.** The insurance fund is the last line of defense against bad debt during liquidations. If `insurance` is inflated without token backing:

1. `withdrawInsurance` will drain user collateral tokens held in the clearinghouse, directly stealing deposited funds.
2. During socialization of bankrupt subaccounts, the protocol believes it has insurance coverage it does not have, leading to incorrect bad-debt accounting and potential under-coverage of losses for all counterparties.

---

### Likelihood Explanation

**High.** Every single call to `depositInsurance` inflates the accounting without transferring tokens. This is not an edge case — it is the normal operational path for funding the insurance pool. The protocol is structurally broken for this function from the first use.

---

### Recommendation

In `Clearinghouse.depositInsurance`, add a `safeTransferFrom` call to pull the quote token from the caller (or from the pre-approved `ContractOwner`) into the clearinghouse before incrementing `insurance`. Mirror the pattern used in `depositCollateral`, where the Endpoint transfers tokens before updating accounting. Alternatively, enforce an invariant check (analogous to `assertUtilization`) that the clearinghouse's actual quote token balance is always ≥ the sum of all user quote balances plus `insurance`.

---

### Proof of Concept

1. Owner calls `ContractOwner.depositInsurance(1_000_000e6)`.
2. This approves the endpoint for `1_000_000e6` USDC and submits a slow-mode `DepositInsurance` transaction.
3. Sequencer processes the transaction; `EndpointTx` calls `clearinghouse.depositInsurance(...)`.
4. `insurance` is incremented by `1_000_000 * 1e18` (normalized). No `transferFrom` is called. The USDC remains in the owner's wallet.
5. Owner calls `ContractOwner.withdrawInsurance(1_000_000e6, recipient)`.
6. `withdrawInsurance` passes the `amount <= insurance` check, decrements `insurance`, and calls `token.safeTransfer(withdrawPool, 1_000_000e6)`.
7. The clearinghouse transfers `1_000_000e6` USDC from its own balance — which is entirely composed of user collateral — to the withdraw pool, effectively stealing user funds. If the clearinghouse balance is insufficient, the call reverts, making insurance withdrawals permanently broken.

        external
        returns (int128)
    {
        require(msg.sender == address(_clearinghouse), ERR_UNAUTHORIZED);

        uint32[] memory _productIds = getProductIds();
        for (uint128 i = 0; i < _productIds.length; ++i) {
            uint32 productId = _productIds[i];
            (State memory state, Balance memory balance) = getStateAndBalance(
                productId,
                subaccount
            );
            if (balance.vQuoteBalance < 0) {
                int128 insuranceCover = MathHelper.min(
                    insurance,
                    -balance.vQuoteBalance
                );
                insurance -= insuranceCover;
                balance.vQuoteBalance += insuranceCover;
                state.availableSettle += insuranceCover;

                // actually socialize if still not enough
                if (balance.vQuoteBalance < 0) {
                    // socialize across all other participants
                    int128 fundingPerShare = -balance.vQuoteBalance.div(
                        state.openInterest
                    );
                    state.cumulativeFundingLongX18 += fundingPerShare;
                    state.cumulativeFundingShortX18 -= fundingPerShare;
                    balance.vQuoteBalance = 0;
                }
                _setState(productId, state);
                _setBalanceAndUpdateBitmap(productId, subaccount, balance);
            }
        }
        return insurance;
    }
