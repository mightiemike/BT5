## Title
Attacker-Controlled Block-Sync Peer Can Inject Wrong `class_hash` into Stored `DeclareTransactionV3` via Unvalidated `DeclareV3WithoutClass` Protobuf Field — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

---

## Summary

The p2p block-sync client accepts `TransactionWithReceipt` protobuf messages from untrusted peers and stores the deserialized `DeclareTransactionV3` (including its `class_hash`) directly into the block body **without recomputing or validating the transaction hash**. An attacker-controlled peer can send a `DeclareV3WithoutClass` message with a tampered `class_hash` field and a matching (but wrong) `transaction_hash` field. The node stores both values verbatim. A TODO comment in the production code explicitly acknowledges this gap.

---

## Finding Description

**Step 1 — Deserialization: `class_hash` is taken verbatim from the protobuf field.**

`TryFrom<protobuf::transaction_in_block::DeclareV3WithoutClass> for DeclareTransactionV3` reads `class_hash` directly from the outer protobuf field: [1](#0-0) 

`compiled_class_hash` comes from `DeclareV3Common.compiled_class_hash`, also fully attacker-controlled: [2](#0-1) 

**Step 2 — `transaction_hash` is taken verbatim from the protobuf field.**

`TryFrom<protobuf::TransactionInBlock> for (Transaction, TransactionHash)` reads the hash from the wire without recomputing it: [3](#0-2) 

**Step 3 — No hash validation in the transaction sync client.**

`parse_data_for_block` in the transaction stream factory pushes the attacker-supplied `transaction_hash` directly into the block body. The TODO comment is the production acknowledgement of the missing guard: [4](#0-3) 

**Step 4 — `validate_transaction_hash` is never called on this path.**

A grep across the entire repo shows `validate_transaction_hash` is only referenced in `transaction_hash_test.rs` (tests) and its own definition in `transaction_hash.rs`. It is never invoked in the p2p sync client pipeline. [5](#0-4) 

**Step 5 — The canonical hash computation uses `class_hash`.**

`get_declare_transaction_v3_hash` chains `class_hash().0` then `compiled_class_hash().0` into the Poseidon hash state: [6](#0-5) 

An attacker who sets `class_hash = X_fake` can compute `H(... || X_fake || compiled_class_hash)` and supply that as `transaction_hash` in the protobuf. The stored `FullTransaction` is then internally self-consistent but carries the wrong `class_hash`.

**Step 6 — No validation against the block header's `transaction_commitment`.**

The header sync stores the block header (including `transaction_commitment`) separately. The transaction sync code never checks that the received transaction hashes match the stored commitment. The header sync code itself also does not enforce this cross-check: [7](#0-6) 

---

## Impact Explanation

The stored `DeclareTransactionV3` in the block body has a wrong `class_hash`. This affects:

1. **RPC `get_transaction_by_hash` / `get_block_with_txs`** — returns the wrong `class_hash` to clients as an authoritative value.
2. **RPC `trace_transaction` / `simulate_transaction`** — re-executes the block using the stored transaction; the blockifier looks up the compiled class by the wrong `class_hash`, selecting the wrong CASM artifact or failing to find the class entirely.
3. **State consistency** — the block body's `class_hash` diverges from the state diff's `class_hash_to_compiled_class_hash` mapping (synced separately), creating an inconsistent node view.

---

## Likelihood Explanation

Any peer that the syncing node connects to for the `Transaction` SQMR protocol can send tampered `DeclareV3WithoutClass` messages. No operator privilege is required. The attack is silent (no error is returned; the tampered data is stored as valid). The TODO comment confirms the gap is known but unaddressed.

---

## Recommendation

In `parse_data_for_block` (`crates/apollo_p2p_sync/src/client/transaction.rs`), before pushing `transaction_hash` into the block body:

1. Recompute the transaction hash from the deserialized transaction fields using `get_transaction_hash` (or `validate_transaction_hash`) and reject the peer if it does not match the wire hash.
2. After collecting all transaction hashes for a block, verify the Merkle `transaction_commitment` against the value stored in the block header.

---

## Proof of Concept

```rust
// Construct a canonical DeclareTransactionV3
let canonical_tx = DeclareTransactionV3 {
    class_hash: ClassHash(Felt::from(0xABCDu64)),
    compiled_class_hash: CompiledClassHash(Felt::from(0x1234u64)),
    // ... other fields
};

// Serialize to protobuf DeclareV3WithoutClass, then tamper class_hash
let mut proto: protobuf::transaction_in_block::DeclareV3WithoutClass =
    canonical_tx.clone().into();
proto.class_hash = Some(Felt::from(0xDEADBEEFu64).into()); // tampered

// Deserialize
let tampered_tx = DeclareTransactionV3::try_from(proto).unwrap();
assert_ne!(tampered_tx.class_hash, canonical_tx.class_hash); // diverges

// Compute hash of tampered tx — attacker supplies this as transaction_hash on wire
let tampered_hash = get_declare_transaction_v3_hash(
    &tampered_tx, &chain_id, &TransactionVersion::THREE
).unwrap();
let canonical_hash = get_declare_transaction_v3_hash(
    &canonical_tx, &chain_id, &TransactionVersion::THREE
).unwrap();
assert_ne!(tampered_hash, canonical_hash); // different hashes, both "valid"

// The sync client stores tampered_tx with tampered_hash — no check rejects this.
```

pub(crate) fn get_declare_transaction_v3_hash<T: DeclareTransactionV3Trait>(
    transaction: &T,
    chain_id: &ChainId,
    transaction_version: &TransactionVersion,
) -> Result<TransactionHash, StarknetApiError> {
    let tip_resource_bounds_hash =
        get_tip_resource_bounds_hash(&transaction.resource_bounds(), transaction.tip())?;
    let paymaster_data_hash =
        HashChain::new().chain_iter(transaction.paymaster_data().0.iter()).get_poseidon_hash();
    let data_availability_mode = concat_data_availability_mode(
        transaction.nonce_data_availability_mode(),
        transaction.fee_data_availability_mode(),
    );
    let account_deployment_data_hash = HashChain::new()
        .chain_iter(transaction.account_deployment_data().0.iter())
        .get_poseidon_hash();

    Ok(TransactionHash(
        HashChain::new()
            .chain(&DECLARE)
            .chain(&transaction_version.0)
            .chain(transaction.sender_address().0.key())
            .chain(&tip_resource_bounds_hash)
            .chain(&paymaster_data_hash)
            .chain(&Felt::try_from(chain_id)?)
            .chain(&transaction.nonce().0)
            .chain(&data_availability_mode)
            .chain(&account_deployment_data_hash)
            .chain(&transaction.class_hash().0)
            .chain(&transaction.compiled_class_hash().0)
            .get_poseidon_hash(),
    ))
