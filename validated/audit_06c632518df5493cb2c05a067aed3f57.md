### Title
Perp Builder Fees Credited as Unbacked Spot USDC in `claimBuilderFee`, Creating Bad Debt — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`claimBuilderFee` iterates only `spotEngine.getProductIds()` and unconditionally calls `spotEngine.updateBalance(productId, sender, collectedFee)`. Because all perp products register with `quoteId = QUOTE_PRODUCT_ID = 0`, their builder fees accumulate in `collectedBuilderFee[0][builderId]` — the same slot as spot USDC fees. When a builder claims, the combined amount is credited as real spot USDC, even though the perp portion was deducted from traders' virtual perp vQuote. This inflates the spot engine's USDC supply without a backing deposit, creating bad debt.

---

### Finding Description

**Fee accrual — perp quoteId is always 0:**

`PerpEngine.addOrUpdateProduct` hardcodes `QUOTE_PRODUCT_ID` as the quoteId for every perp market: [1](#0-0) 

`updateMarket` then stores this in `quoteIds[productId] = 0`: [2](#0-1) 

During order matching, builder fees are stored under `collectedBuilderFee[market.quoteId][builderId]`: [3](#0-2) 

For every perp product, `market.quoteId == 0`, so perp builder fees land in `collectedBuilderFee[0][builderId]` — the same slot as spot USDC fees.

**Fee deduction is against perp vQuote:**

For perp trades, `_updateBalances` routes the `quoteDelta` (which includes the builder fee deduction) to `perpEngine.updateBalance(productId, subaccount, baseDelta, quoteDelta)`: [4](#0-3) 

The trader's perp vQuote is reduced by `builderFee`. No one in the perp engine receives this vQuote — it is held externally in `collectedBuilderFee`.

**Claim credits everything as spot USDC:**

`claimBuilderFee` only iterates `spotEngine.getProductIds()` and calls the single-argument `spotEngine.updateBalance(productId, sender, collectedFee)`: [5](#0-4) 

`QUOTE_PRODUCT_ID = 0` is always present in `spotEngine.getProductIds()` because `SpotEngine.initialize` pushes it unconditionally: [6](#0-5) 

So `collectedBuilderFee[0][builderId]` — containing both spot and perp fees — is found, zeroed, and credited as spot USDC via `spotEngine.updateBalance(0, sender, amount)`.

**Contrast with `dumpFees`, which handles this correctly:**

`dumpFees` explicitly separates spot and perp fee routing — perp fees go to `perpEngine.updateBalance(productId, X_ACCOUNT, 0, fees)` as vQuote: [7](#0-6) 

`claimBuilderFee` has no equivalent perp branch.

---

### Impact Explanation

The accounting mismatch is:

- **Perp engine**: trader's vQuote was reduced by `builderFee`; no one in the perp engine ever receives it. The perp engine carries a permanent net-negative vQuote hole equal to all perp builder fees ever claimed.
- **Spot engine**: builder receives spot USDC (`spotEngine.updateBalance(0, builder, fee)`) with no corresponding USDC deposit. `totalDepositsNormalized` is not increased, but the builder's balance is. This is unbacked USDC — bad debt in the spot engine.

The builder extracts real, withdrawable USDC that was never deposited, at the expense of all spot USDC holders whose claims are diluted. The magnitude scales with perp trading volume and builder fee rates over time.

The second sub-claim in the question — that fees are permanently locked if `QUOTE_PRODUCT_ID` is not in spot product IDs — is **false**: product 0 is always present. The fees are not locked; they are misrouted.

---

### Likelihood Explanation

Any registered builder who has facilitated perp trades can trigger this by submitting a `ClaimBuilderFee` transaction through the normal endpoint flow. No special privileges are required beyond being the owner of a `builderId`. The path is fully reachable via `EndpointTx.processTransactionImpl` → `IOffchainExchange.claimBuilderFee`. [8](#0-7) 

---

### Recommendation

`claimBuilderFee` must mirror the engine-routing logic of `dumpFees`. Since `collectedBuilderFee` is keyed by `quoteId` and all perp products share `quoteId = 0`, the function cannot distinguish spot-USDC fees from perp-vQuote fees using the current storage layout.

Two options:
1. Change the storage key to `collectedBuilderFee[productId][builderId]` (not `quoteId`) so the engine can be looked up via `clearinghouse.getEngineByProduct(productId)` at claim time, routing spot fees to `spotEngine` and perp fees to `perpEngine.updateBalance(productId, sender, 0, fee)`.
2. Maintain separate mappings: `collectedSpotBuilderFee` and `collectedPerpBuilderFee`, and route each to the correct engine at claim time.

---

### Proof of Concept

```solidity
// 1. Builder accumulates perp fees from BTC-PERP trades (productId = 2, quoteId = 0)
//    collectedBuilderFee[0][builderId] = X (perp vQuote deducted from traders)

// 2. Builder calls ClaimBuilderFee via endpoint
//    EndpointTx routes to OffchainExchange.claimBuilderFee(sender, builderId)

// 3. claimBuilderFee iterates spotEngine.getProductIds() = [0, 1, 3, ...]
//    At productId=0: collectedBuilderFee[0][builderId] = X (non-zero)
//    => spotEngine.updateBalance(0, sender, X)   // builder gets real USDC
//    => collectedBuilderFee[0][builderId] = 0

// 4. Assert:
//    spotEngine.getBalance(0, sender).amount increased by X  ✓ (real USDC, withdrawable)
//    perpEngine total vQuote has a hole of X                 ✓ (never settled)
//    No USDC was deposited to back this credit               ✓ (bad debt)

// 5. Builder withdraws USDC — real funds leave the protocol unbacked
```
