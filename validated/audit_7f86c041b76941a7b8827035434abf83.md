### Title
`BaseEngine._initialize()` Uses `initializer` Instead of `onlyInitializing`, Stripping `initializer` Guard from `SpotEngine` and `PerpEngine` — (File: `core/contracts/BaseEngine.sol`)

---

### Summary

`BaseEngine._initialize()` is decorated with the `initializer` modifier instead of `onlyInitializing`. Because this internal parent function consumes the `initializer` slot, the child contracts `SpotEngine` and `PerpEngine` cannot safely add `initializer` to their own public `initialize()` functions — doing so would cause initialization to revert. The result is that `SpotEngine.initialize()` and `PerpEngine.initialize()` carry **no `initializer` guard at all**, leaving them callable by any external actor before legitimate deployment.

---

### Finding Description

`BaseEngine._initialize()` is the shared initialization routine for both engine contracts:

```solidity
// core/contracts/BaseEngine.sol
function _initialize(
    address _clearinghouseAddr,
    address _offchainExchange,
    address _endpointAddr,
    address _admin
) internal initializer {          // ← should be onlyInitializing
    __Ownable_init();
    setEndpoint(_endpointAddr);
    transferOwnership(_admin);
    _clearinghouse = IClearinghouse(_clearinghouseAddr);
    canApplyDeltas[_endpointAddr] = true;
    canApplyDeltas[_clearinghouseAddr] = true;
    canApplyDeltas[_offchainExchangeAddr] = true;
}
``` [1](#0-0) 

Both child contracts call this function from their own `initialize()`, but neither child carries the `initializer` modifier:

```solidity
// SpotEngine.sol
function initialize(
    address _clearinghouse, address _offchainExchange,
    address _quote, address _endpoint, address _admin
) external {                      // ← no initializer modifier
    _initialize(_clearinghouse, _offchainExchange, _endpoint, _admin);
    ...
}
``` [2](#0-1) 

```solidity
// PerpEngine.sol
function initialize(
    address _clearinghouse, address _offchainExchange,
    address, address _endpoint, address _admin
) external {                      // ← no initializer modifier
    _initialize(_clearinghouse, _offchainExchange, _endpoint, _admin);
}
``` [3](#0-2) 

Per OpenZeppelin's specification, `initializer` is for the **outermost** initialization function; parent init functions in an inheritance chain must use `onlyInitializing`. When `_initialize` carries `initializer`, calling it from a child function that also carries `initializer` would cause the child's call to revert (because `_initializing` is already `true` and `_initialized` is already `1` when the parent's `initializer` check runs). The only way the current code avoids this revert is by **omitting `initializer` from the child functions entirely** — which removes all access control from the public entry points.

The same pattern exists in `BaseWithdrawPool._initialize()`:

```solidity
// core/contracts/BaseWithdrawPool.sol
function _initialize(address _clearinghouse, address _verifier)
    internal
    initializer          // ← should be onlyInitializing
{
    __Ownable_init();
    clearinghouse = _clearinghouse;
    verifier = _verifier;
}
``` [4](#0-3) 

```solidity
// WithdrawPool.sol
function initialize(address _clearinghouse, address _verifier) external {
    _initialize(_clearinghouse, _verifier);   // ← no initializer modifier
}
``` [5](#0-4) 

---

### Impact Explanation

`SpotEngine.initialize()` and `PerpEngine.initialize()` are `external` with no `initializer` guard and no other access control. Any caller can invoke them before the legitimate deployer does. The first successful call to `_initialize` sets `_initialized = 1`, permanently locking out subsequent calls. An attacker who wins the race:

1. Passes attacker-controlled addresses for `_clearinghouseAddr`, `_offchainExchangeAddr`, `_endpointAddr`, and `_admin`.
2. `__Ownable_init()` sets `owner = msg.sender` (attacker), then `transferOwnership(_admin)` transfers to the attacker-supplied admin.
3. The attacker becomes the owner of `SpotEngine` / `PerpEngine`.
4. The legitimate `Clearinghouse.addEngine()` call subsequently fails because `_initialize` reverts on the second invocation.

As owner of `SpotEngine`, the attacker can call `updateRisk()` to manipulate weight and price parameters for any product, corrupting health calculations and enabling illegitimate liquidations or blocking legitimate ones — directly affecting user collateral and solvency accounting. [6](#0-5) 

For `WithdrawPool`, the attacker-owned contract can call `removeLiquidity()` to drain token balances held for pending withdrawals. [7](#0-6) 

---

### Likelihood Explanation

The attack window is the deployment sequence: `SpotEngine` / `PerpEngine` / `WithdrawPool` are deployed as proxies, and their `initialize()` functions must be called in a subsequent transaction. On any public mempool chain, an attacker can observe the pending initialization transaction and front-run it. No privileged access, leaked keys, or social engineering is required — only the ability to submit a transaction with a higher gas price.

---

### Recommendation

Change `initializer` to `onlyInitializing` in both parent init functions, and add `initializer` to the child's public entry points:

```diff
// BaseEngine.sol
- function _initialize(...) internal initializer {
+ function _initialize(...) internal onlyInitializing {

// SpotEngine.sol
- function initialize(...) external {
+ function initialize(...) external initializer {

// PerpEngine.sol
- function initialize(...) external {
+ function initialize(...) external initializer {

// BaseWithdrawPool.sol
- function _initialize(...) internal initializer {
+ function _initialize(...) internal onlyInitializing {

// WithdrawPool.sol
- function initialize(...) external {
+ function initialize(...) external initializer {
```

---

### Proof of Concept

1. Deploy `SpotEngine` behind a `TransparentUpgradeableProxy`.
2. Before the legitimate deployer calls `Clearinghouse.addEngine()`, attacker calls:
   ```solidity
   SpotEngine(proxy).initialize(
       attackerClearinghouse,
       attackerExchange,
       attackerQuote,
       attackerEndpoint,
       attackerAdmin
   );
   ```
3. `_initialize` runs: `__Ownable_init()` sets owner to attacker, `transferOwnership(attackerAdmin)` finalizes attacker ownership.
4. Legitimate `Clearinghouse.addEngine()` calls `productEngine.initialize(...)`, which calls `_initialize(...)`, which reverts with `"Initializable: contract is already initialized"`.
5. Protocol deployment is permanently bricked for `SpotEngine`; attacker retains ownership of the initialized instance. [8](#0-7)
