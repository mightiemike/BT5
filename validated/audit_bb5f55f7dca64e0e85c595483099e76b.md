### Title
Duplicate-Last-Leaf Position Ambiguity in `compute_root_from_merkle_proof` Allows Same Proof to Verify Two Distinct `tx_index` Values — (`merkle-tools/src/lib.rs`)

---

### Summary

`compute_root_from_merkle_proof` contains no bounds check on `transaction_position`. In Bitcoin's Merkle tree, when the leaf count is odd, the last leaf is duplicated. This means the same `(tx_id, merkle_proof)` pair produces the same root for both `tx_index = n` (the real last leaf) and `tx_index = n+1` (the phantom duplicate slot). `verify_transaction_inclusion_v2` returns `true` for both calls against the same live block. The reorg framing in the question is a red herring — the attack requires no reorg at all.

---

### Finding Description

**`compute_root_from_merkle_proof`** (merkle-tools/src/lib.rs, lines 34–52) iterates over proof hashes and uses only `current_position % 2` to decide left/right placement, then divides by 2: [1](#0-0) 

There is no check that `transaction_position < actual_leaf_count`. The function accepts any `usize` position.

**Bitcoin's odd-width duplication rule:** for a tree with 3 leaves `[T0, T1, T2]`, the protocol pads level-0 to `[T0, T1, T2, T2]`. The canonical proof for `T2` at position 2 is `[T2, H(T0,T1)]`.

**Ambiguity proof (concrete arithmetic):**

| Call | tx\_id | tx\_index | proof | Step 1 (pos%2) | Step 2 (pos%2) | Result |
|------|--------|-----------|-------|----------------|----------------|--------|
| A | T2 | 2 (even) | [T2, H(T0,T1)] | H(T2,T2) | H(H(T0,T1),H(T2,T2)) = root | ✓ |
| B | T2 | 3 (odd) | [T2, H(T0,T1)] | H(T2,T2) | H(H(T0,T1),H(T2,T2)) = root | ✓ |

Both calls produce the identical intermediate hash `H(T2,T2)` at level 1 — one because position 2 is even (hash goes left), the other because position 3 is odd (hash goes right) — and both then combine with `H(T0,T1)` in the same orientation (position 1 is odd in both cases). The root matches in both cases.

**`verify_transaction_inclusion_v2`** passes `args.tx_index` directly to `compute_root_from_merkle_proof` with no upper-bound validation: [2](#0-1) 

The coinbase proof check (lines 358–365) validates tree depth parity but does nothing to prevent the duplicate-leaf position collision for the transaction being proven. [3](#0-2) 

**Why the reorg framing does not apply:** `verify_transaction_inclusion` (called by v2) panics if `tx_block_blockhash` is absent from `mainchain_header_to_height`. After a reorg, `remove_block_header` removes the old block from that map, so the old block hash becomes unusable. The duplicate-leaf attack requires no reorg — both calls A and B above target the same live mainchain block. [4](#0-3)

### Impact Explanation

`verify_transaction_inclusion_v2` is a public, unpermissioned view function. Any downstream bridge or token-release contract that calls it and gates value release on its return value is exposed. If the downstream contract tracks processed proofs by `(tx_id, tx_index)` — a natural design — an attacker can:

1. Submit call A (`tx_index = n`) → `true` → claim value.
2. Submit call B (`tx_index = n+1`, same proof) → `true` → claim value again.

The light client itself has no replay ledger; that is the caller's responsibility. But the caller cannot defend against this because the light client certifies two distinct `(tx_id, tx_index)` pairs as valid for the same on-chain event.

---

### Likelihood Explanation

- Requires only a Bitcoin block with an odd transaction count (extremely common — the majority of Bitcoin blocks have an odd number of transactions).
- Requires no privileged role, no relayer compromise, no reorg, no special timing.
- The attacker constructs both calls offline from public block data and submits them sequentially.

---

### Recommendation

In `compute_root_from_merkle_proof`, enforce that `transaction_position < (1 << merkle_proof.len())` **and** that the position is consistent with the actual leaf count. Since the light client does not store the transaction count, the cleanest fix is to reject any proof where `transaction_position` is odd **and** the supplied sibling equals `transaction_hash` itself (i.e., detect the duplicate-leaf slot explicitly):

```rust
// At each level, if position is odd and proof_hash == current_hash,
// this is a phantom duplicate slot — reject.
if current_position % 2 == 1 && proof_hash == &current_hash {
    return Err(MerkleError::DuplicateLeafPosition);
}
```

Alternatively, require callers to supply the leaf count and validate `transaction_position < leaf_count` before entering the loop.

### Proof of Concept

```
Tree: [coinbase, T1, T2]   (3 leaves, odd)
Block merkle root = H(H(coinbase, T1), H(T2, T2))

coinbase_merkle_proof = [T1, H(T2,T2)]   // proves coinbase at index 0
tx_merkle_proof       = [T2, H(coinbase,T1)]

Call 1: verify_transaction_inclusion_v2(
    tx_id=T2, tx_block_blockhash=B, tx_index=2,
    merkle_proof=[T2, H(coinbase,T1)],
    coinbase_tx_id=coinbase, coinbase_merkle_proof=[T1, H(T2,T2)],
    confirmations=1
) → true   ← legitimate

Call 2: verify_transaction_inclusion_v2(
    tx_id=T2, tx_block_blockhash=B, tx_index=3,   // ← phantom slot
    merkle_proof=[T2, H(coinbase,T1)],             // ← identical proof
    coinbase_tx_id=coinbase, coinbase_merkle_proof=[T1, H(T2,T2)],
    confirmations=1
) → true   ← should be false; position 3 does not exist
```

Both calls return `true` against the same unmodified mainchain block. A downstream contract releasing funds on each unique `(tx_id, tx_index)` pair pays out twice for one Bitcoin transaction.
