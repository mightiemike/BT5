### Title
Difficulty Adjustment Uses Stale Mainchain Timestamp Instead of Fork Ancestor During Fork Validation - (File: `contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

---

### Summary

`get_header_by_height` always resolves blocks from `mainchain_height_to_header`, which only contains mainchain blocks. Fork blocks are stored exclusively in `headers_pool` via `store_fork_header` and are never inserted into `mainchain_height_to_header`. When `get_next_work_required` calls `blocks_getter.get_header_by_height(first_block_height)` to obtain the anchor timestamp for a difficulty adjustment, it silently reads the **mainchain** block at that height rather than the true fork-chain ancestor. This is the direct analog of the BondNFT `totalShares` desynchronization: a shared state variable (`mainchain_height_to_header`) is not updated when a fork block is stored, so the difficulty calculation uses stale data from the wrong chain.

### Finding Description

Fork blocks are stored by `store_fork_header`, which writes only to `headers_pool`: [1](#0-0) 

Mainchain blocks are stored by `store_block_header`, which writes to all three maps including `mainchain_height_to_header`: [2](#0-1) 

`get_header_by_height` resolves exclusively through `mainchain_height_to_header`: [3](#0-2) 

In `dogecoin.rs`, `get_next_work_required` calls this function to obtain the anchor timestamp for the difficulty window. The developers themselves flagged this with a TODO: [4](#0-3) 

The same pattern appears in `bitcoin.rs`: [5](#0-4) 

And in `litecoin.rs`: [6](#0-5) 

When a fork diverges at height K and the difficulty window anchor (`first_block_height`) falls within the fork (i.e., `first_block_height > K`), the contract reads the **mainchain** block's timestamp at `first_block_height` instead of the fork ancestor's timestamp. The fork ancestor at that height is a different block with a potentially different timestamp, stored only in `headers_pool` and unreachable via `get_header_by_height`.

---

### Impact Explanation

The difficulty check enforces `expected_bits == block_header.bits`. `expected_bits` is computed from the anchor timestamp returned by `get_header_by_height`. Because that timestamp belongs to the mainchain block rather than the fork ancestor, the computed `expected_bits` diverges from what the fork chain's actual timing would require.

An attacker who controls fork block timestamps can craft fork blocks at heights K+1, K+2, … such that the mainchain anchor timestamp produces a **lower** `expected_bits` (easier target) than the fork chain's real timing would dictate. The fork block at the adjustment boundary is then accepted with lower difficulty than it should carry. Conversely, the attacker can engineer a **higher** `expected_bits`, giving fork blocks inflated chainwork relative to the actual mining effort expended.

For **Dogecoin** (1-block adjustment interval after height 145,000), `height_first = prev_block_height - 1`, so the wrong block is used for every fork block that is ≥ 2 blocks deep from the divergence point — a condition met by virtually every non-trivial fork submission. [7](#0-6) 

For **Bitcoin** and **Litecoin** (2016-block interval), the bug activates whenever the fork diverges within the current difficulty epoch and the fork reaches the next retarget boundary. [8](#0-7) 

The broken invariant: the difficulty adjustment mechanism is supposed to enforce that each block's `bits` field reflects the actual elapsed time on **that chain**. After this bug, the enforcement is based on a different chain's elapsed time, decoupling the fork chain's difficulty from its real block production rate.

---

### Likelihood Explanation

The entry path is fully unprivileged: any caller of `submit_blocks` can supply adversarial fork headers. For Dogecoin, the condition is triggered by any fork ≥ 3 blocks long submitted after height 145,000 — a routine occurrence. For Bitcoin/Litecoin, it requires a fork that spans a retarget boundary, which is less frequent but still reachable by a persistent relayer or attacker. No privileged role, leaked key, or social engineering is required. [9](#0-8) 

---

### Recommendation

Replace the `get_header_by_height` call in `get_next_work_required` with a backward walk from `prev_block_header` using `get_prev_header`, traversing exactly `blocks_to_go_back` steps. This follows the fork chain's own ancestry rather than the mainchain index, matching the reference Bitcoin Core implementation which walks back through `pindexFirst = pindexLast->GetAncestor(nHeightFirst)` along the actual chain. [10](#0-9) 

---

### Proof of Concept

**Dogecoin scenario (height > 145,000, 1-block adjustment interval):**

1. Mainchain has blocks at heights 0…N (N > 145,002), each with timestamp `T_i`.
2. Attacker submits a fork diverging at height K (K > 145,000, K < N−2).
3. Fork block at K+1 is submitted with timestamp `T_{K+1}' ≠ T_{K+1}` (e.g., much earlier, compressing the apparent elapsed time).
4. Fork block at K+2 is submitted with timestamp `T_{K+2}'`.
5. When fork block at K+3 is submitted, `get_next_work_required` computes `height_first = K+1` and calls `get_header_by_height(K+1)`.
6. `get_header_by_height` returns the **mainchain** block at K+1 (timestamp `T_{K+1}`), not the fork block (timestamp `T_{K+1}'`).
7. `calculate_next_work_required` uses `T_{K+1}` as `first_block_time`, producing `expected_bits` based on the mainchain's elapsed time, not the fork's.
8. If `T_{K+1} < T_{K+1}'` (mainchain block is earlier), `modulated_timespan` is larger, difficulty decreases, and the fork block at K+3 is accepted with a lower difficulty than the fork chain's actual timing requires.
9. The fork block passes `expected_bits == block_header.bits` and the PoW check, despite carrying incorrect difficulty for its chain. [11](#0-10) [3](#0-2)

```rust
    pub fn submit_blocks(
        &mut self,

    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
```

```rust
fn get_next_work_required(
    config: &DogecoinConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    // Dogecoin: Special rules for minimum difficulty blocks with Digishield
    if allow_min_difficulty_for_block(config, block_header, prev_block_header) {
        // Special difficulty rule for testnet:
        // If the new block's timestamp is more than 2* nTargetSpacing minutes
        // then allow mining of a min-difficulty block.
        return config.proof_of_work_limit_bits;
    }

    // Only change once per difficulty adjustment interval
    let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
    let difficulty_adjustment_interval = if new_difficulty_protocol {
        1
    } else {
        config.difficulty_adjustment_interval
    };

    if (prev_block_header.block_height + 1) % difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            // Special difficulty rule for testnet:
            // If the new block's timestamp is more than 2* 10 minutes
            // then allow mining of a min-difficulty block.
            if block_header.time
                > prev_block_header.block_header.time + config.pow_target_spacing * 2
            {
                return config.proof_of_work_limit_bits;
            }

            // Return the last non-special-min-difficulty-rules-block
            let mut current_block_header = prev_block_header.clone();

            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            return current_block_header.block_header.bits;
        }

        return prev_block_header.block_header.bits;
    }

    // Litecoin: This fixes an issue where a 51% attack can change difficulty at will.
    // Go back the full period unless it's the first retarget after genesis. Code courtesy of Art Forz
    let mut blocks_to_go_back = difficulty_adjustment_interval - 1;
    if prev_block_header.block_height + 1 != difficulty_adjustment_interval {
        blocks_to_go_back = difficulty_adjustment_interval;
    }

    // Go back by what we want to be 14 days worth of blocks
    let height_first = prev_block_header
        .block_height
        .checked_sub(blocks_to_go_back)
        .unwrap_or_else(|| env::panic_str("Height underflow when calculating first block height"));

    // TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
    let first_block_time = blocks_getter
        .get_header_by_height(height_first)
        .block_header
        .time;

    calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))
```
