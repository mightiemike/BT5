### Title
Fork Difficulty Retarget Uses Main-Chain Ancestor Instead of Fork Ancestor — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

When a fork block at a difficulty-retarget height is submitted, the contract computes the expected difficulty using `get_header_by_height`, which always reads from `mainchain_height_to_header` (the canonical chain index). If the fork diverges from the main chain at a height *before* the start of the current difficulty-adjustment interval, the block at the interval boundary on the main chain is a *different block* from the fork's actual ancestor at that height. The difficulty calculation therefore uses the wrong timestamp, producing an incorrect target. An adversarial relayer can exploit this to get fork blocks accepted at a lower difficulty than the protocol requires, and if those blocks accumulate enough chainwork, they become the new canonical chain.

The developers themselves flagged this in a `TODO` comment:

> `// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor`

---

### Finding Description

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // ← always the canonical chain
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0)

By contrast, `get_prev_header` — used everywhere else to walk the ancestor chain — correctly reads from `headers_pool`, which contains *all* submitted headers (main chain and forks):

```rust
fn get_prev_header(&self, current_header: &LightHeader) -> ExtendedHeader {
    self.headers_pool
        .get(&current_header.prev_block_hash)   // ← follows the actual fork chain
        .unwrap_or_else(|| env::panic_str("PrevBlockNotFound"))
}
``` [5](#0-4) 

The desynchronization: `get_prev_header` follows the fork's true ancestor chain, but `get_header_by_height` silently substitutes the main-chain block at the same height. These two blocks can have different timestamps whenever the fork diverges before the start of the difficulty-adjustment interval.

---

### Impact Explanation

The difficulty retarget formula (Bitcoin/Litecoin) is:

```
actual_time_taken = prev_block_time − first_block_time
new_target        = prev_target × actual_time_taken / pow_target_timespan
``` [6](#0-5) 

If the main-chain block at `first_block_height` has an *earlier* timestamp than the fork's true ancestor at that height, `actual_time_taken` is inflated, the computed target is raised (difficulty lowered), and the contract accepts the fork's retarget block at a difficulty lower than the protocol requires. A fork built on this weakened target can accumulate chainwork faster than it should, eventually triggering a reorg and corrupting the canonical chain stored in `mainchain_tip_blockhash`, `mainchain_height_to_header`, and `mainchain_header_to_height`. Downstream consumers calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` then operate against a fraudulent chain.

---

### Likelihood Explanation

The attack requires:
1. Submitting a fork that diverges from the main chain at a height *before* the start of the current 2016-block (Bitcoin/Litecoin) difficulty-adjustment interval — a normal occurrence during any reorg that spans an interval boundary.
2. Crafting the fork's block at the interval-start height with a timestamp later than the main chain's block at that height (within the MTP and future-time constraints), so that when the contract reads the main-chain block's earlier timestamp, it computes a lower difficulty.

No privileged role is required. Any relayer can call `submit_blocks` with adversarial headers. [7](#0-6) 

---

### Recommendation

Replace `get_header_by_height` calls inside `get_next_work_required` with a backward walk using `get_prev_header`, starting from `prev_block_header` and stepping back `blocks_to_go_back` times. This mirrors how Bitcoin Core actually traverses the ancestor chain and eliminates the dependency on the canonical-chain index for fork validation:

```rust
let mut cursor = prev_block_header.clone();
for _ in 0..blocks_to_go_back {
    cursor = blocks_getter.get_prev_header(&cursor.block_header);
}
let first_block_time = cursor.block_header.time;
```

This is exactly what `get_prev_header` is designed for and what Zcash's implementation already does (it walks back `pow_averaging_window` steps via `get_prev_header` without ever calling `get_header_by_height`). [8](#0-7) 

---

### Proof of Concept

**Setup (Bitcoin, `difficulty_adjustment_interval = 2016`):**

1. Main chain has blocks at heights 0–4031. The block at height 2016 (interval boundary) has timestamp `T_main`.
2. An attacker prepares a fork that diverges at height 2010 (before the boundary). The fork's block at height 2016 has timestamp `T_fork > T_main` (e.g., 10 minutes later, within MTP/future-time limits).
3. The attacker submits fork blocks 2011–4031 via `submit_blocks`.

**During submission of fork block 4032 (next retarget):**

- `check_pow` → `check_target` → `get_next_work_required`
- `first_block_height = 4031 − (2016 − 1) = 2016`
- `get_header_by_height(2016)` returns the **main-chain** block with timestamp `T_main`
- `actual_time_taken = T_fork_tip − T_main` (inflated by `T_fork − T_main`)
- Computed target is higher (difficulty lower) than the protocol requires for the fork

**Result:** Fork block 4032 is accepted with insufficient PoW. If the fork's cumulative chainwork exceeds the main chain's, `reorg_chain` promotes it to canonical, corrupting the light client's view of the Bitcoin chain. [9](#0-8)
