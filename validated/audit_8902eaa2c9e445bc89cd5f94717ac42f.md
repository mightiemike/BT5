### Title
Linked Signer Can Arbitrarily Overwrite `linkedSigners` Mapping via Fast-Mode `LinkSigner` Transaction — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The fast-mode `LinkSigner` transaction handler in `EndpointTx.sol` invokes `validateSignedTx` with `allowLinkedSigner = true`, permitting the **existing linked signer** — not just the subaccount owner — to sign and submit a `LinkSigner` transaction that overwrites `linkedSigners[sender]` with any arbitrary address. This is a direct analog to the `suifren_update_last_epoch_mixed` public-visibility bug: a party with limited delegated authority can mutate the exact state variable that gates authorization for all subsequent subaccount operations, bypassing the intended ownership invariant.

---

### Finding Description

In `EndpointTx.sol`, the fast-mode `LinkSigner` branch calls `validateSignedTx` with the fifth argument hardcoded to `true`: [1](#0-0) 

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true          // ← allowLinkedSigner
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
```

`validateSignedTx` forwards `allowLinkedSigner` into `validateSignature`: [2](#0-1) 

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
```

`Verifier.validateSignature` then accepts a signature from **either** the subaccount owner address **or** the linked signer: [3](#0-2) 

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
```

Because `linkedSigners` is the exact mapping consulted by every subsequent `validateSignature` / `validateCompactSignature` call to authorize subaccount operations: [4](#0-3) 

…a linked signer can sign a `LinkSigner` transaction to replace itself with any address it controls, permanently re-establishing signing authority even after the account owner attempts revocation.

The slow-mode `LinkSigner` path (the owner's revocation mechanism) does **not** require a signature — it only checks `validateSender` — but it is queued and processed after fast-mode batches: [5](#0-4) 

This ordering asymmetry means a malicious linked signer can always race ahead of the owner's slow-mode revocation by continuously submitting fast-mode `LinkSigner` transactions.

---

### Impact Explanation

A compromised or malicious linked signer can:

1. Change `linkedSigners[subaccount]` to a new address they control.
2. Use that new address to sign `WithdrawCollateral`, `TransferQuote`, `LiquidateSubaccount`, `MintNlp`, or `BurnNlp` transactions on behalf of the subaccount.
3. Drain all collateral from the subaccount.
4. Prevent the account owner from reliably revoking access, because every slow-mode revocation can be immediately overwritten by a fast-mode `LinkSigner` from the attacker's current linked-signer address.

The corrupted state variable is `linkedSigners[subaccount]` in `EndpointStorage`. The corrupted asset delta is the full collateral balance of the targeted subaccount across all product engines.

---

### Likelihood Explanation

Linked signers are a standard feature used by traders who operate with hot-wallet signing keys. Any user who has ever called `LinkSigner` is exposed. The attacker's only precondition is that their address was at some point set as a linked signer — either legitimately (key compromise scenario) or by social engineering the owner into setting it. No admin privilege, sequencer compromise, or cryptographic break is required.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the fast-mode `LinkSigner` handler so that only the subaccount owner's key can authorize a linked-signer change:

```solidity
// EndpointTx.sol — fast-mode LinkSigner branch
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This mirrors the fix applied in the reference report: restricting the mutating function to the intended privileged caller only.

---

### Proof of Concept

1. Owner of subaccount `S` sets `linkedSigners[S] = addrA` (legitimate hot-wallet setup, slow-mode or fast-mode with owner key).
2. `addrA` is compromised by an attacker.
3. Attacker signs a fast-mode `LinkSigner` transaction: `sender = S`, `signer = addrB` (attacker-controlled), using `addrA`'s key. Nonce is the current `nonces[owner(S)]`.
4. Sequencer submits the batch; `linkedSigners[S]` is overwritten to `addrB`.
5. Owner detects the compromise and submits a slow-mode `LinkSigner` to set `linkedSigners[S] = address(0)`.
6. Before the slow-mode queue is drained, attacker (now holding `addrB`) signs another fast-mode `LinkSigner`: `sender = S`, `signer = addrB`. Sequencer processes it; `linkedSigners[S]` is reset to `addrB`.
7. Attacker signs a `WithdrawCollateral` transaction from `addrB`, draining `S`'s collateral.

The root cause — `allowLinkedSigner = true` on the `LinkSigner` fast-mode branch — is the necessary and sufficient vulnerable step. No privileged access beyond the already-set linked signer is required.
