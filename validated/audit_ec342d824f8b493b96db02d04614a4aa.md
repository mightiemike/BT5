### Title
Contract with function body > 196,608 bytes deployed at protocol v83 becomes permanently unexecutable after upgrade to v84 ‚Äî (`runtime/near-vm-runner/src/prepare/prepare_v2.rs`)

### Summary

`prepare_v2` (used at protocol v83, where `reftypes_bulk_memory=false` and `vm_kind=NearVm`) has no `function_body_size_limit` field and performs no per-function-body size check. `prepare_v3` (used at protocol v84+, where `reftypes_bulk_memory=true` and `vm_kind=Wasmtime`) enforces `max_function_body_size=196,608` bytes per function body. A contract with a single function body of, e.g., 200,000 bytes can be deployed at v83 (within the 4 MiB `max_contract_size` limit), but every subsequent call at v84+ will fail with `PrepareError::FunctionBodyTooLarge`, permanently bricking the contract.

### Finding Description

The `prepare_contract` router in `runtime/near-vm-runner/src/prepare.rs` dispatches to `prepare_v2` when `!config.reftypes_bulk_memory && config.vm_kind != VMKind::Wasmtime`. At protocol v83 both conditions hold (`reftypes_bulk_memory=false`, `vm_kind=NearVm`). `prepare_v2::PrepareContext` has no `function_body_size_limit` field and its `CodeSectionEntry` arm only counts locals ‚Äî it never checks function body byte size. At protocol v84, `84.yaml` simultaneously flips `reftypes_bulk_memory: false‚Üítrue`, `vm_kind: NearVm‚ÜíWasmtime`, and introduces `max_function_body_size: 196_608` (new, no prior value). The router now dispatches to `prepare_v3`, which reads `limits.max_function_body_size.unwrap_or(u64::MAX)` and rejects any function body whose `func.range().len() as u64 > 196_608` with `PrepareError::FunctionBodyTooLarge`. Because NEAR stores raw wasm bytes and re-prepares on every call, a contract stored at v83 with a 200,000-byte function body will fail every call at v84+.

The question's claim of `PrepareError::Deserialization` is slightly wrong ‚Äî the actual error is `PrepareError::FunctionBodyTooLarge` ‚Äî but the divergence is otherwise exactly as described.

### Impact Explanation

Any unprivileged user who deployed a contract at v83 with a function body between 196,609 and ~4,194,304 bytes (the `max_contract_size` ceiling) loses the ability to call that contract after the v84 upgrade. The contract's wasm bytes remain in state but can never be prepared again. This is a permanent, irreversible loss of contract executability for all affected accounts, with no recovery path short of redeploying (which requires the account to still be accessible and the owner to be aware of the issue).

### Likelihood Explanation

The window is the v83‚Üív84 upgrade epoch. Any contract deployed before that epoch boundary with a function body exceeding 196,608 bytes is affected. Legitimate compilers (e.g., Rust/wasm-pack targeting NEAR) can produce function bodies larger than 196 KiB for non-trivial contracts. The attacker does not need any special privilege ‚Äî a standard `DeployContract` action suffices.

### Recommendation

Before activating `max_function_body_size` at v84, scan all deployed contracts on-chain for function bodies exceeding the new limit and either:
1. Grandfather existing contracts (skip the check for contracts deployed before v84), or
2. Enforce the limit at deploy time starting from v84 (already done via the increased compute cost in `84.yaml`), but do not apply it retroactively to contracts already in state.

Alternatively, add a migration step that identifies and flags affected contracts so their owners can redeploy before the limit takes effect.
