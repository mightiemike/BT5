### Title
Isolated Subaccount `digestToSubaccount` Mapping Not Cleared on Closure Enables Ghost Position Creation — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

When an isolated subaccount is closed via `_tryCloseIsolatedSubaccount`, the `digestToSubaccount` mapping is not cleared. This allows subsequent `matchOrders` calls to redirect order senders to the now-closed isolated subaccount, opening new positions in an untracked subaccount with no parent association — corrupting protocol accounting.

---

### Finding Description

`_tryCloseIsolatedSubaccount` (lines 160–208) clears three tracking structures when closing an isolated subaccount:

```
isolatedSubaccountsMask[addr] &= ~uint256(0) ^ (1 << id);   // bit cleared
isolatedSubaccounts[parent][id] = bytes32(0);                 // slot freed
parentSubaccounts[subaccount] = bytes32(0);                   // parent link severed
``` [1](#0-0) 

However, the function does **not** clear `digestToSubaccount[digest]`, which was set when the isolated order was originally created and maps the order's EIP-712 digest to the isolated subaccount address. [2](#0-1) 

In `matchOrders`, after the isolated-subaccount guard at lines 644–651 (which checks the *original* `order.sender`), the sender is silently redirected:

```solidity
if (digestToSubaccount[ordersInfo.taker.digest] != bytes32(0)) {
    taker.order.sender = digestToSubaccount[ordersInfo.taker.digest];
}
``` [3](#0-2) 

Because the redirect happens *after* the guard, the isolated-subaccount check is bypassed. The post-match health gate is also inert — `isHealthy` unconditionally returns `true` in `OffchainExchange`: [4](#0-3) 

The result: a new position is opened inside the closed isolated subaccount. Because `parentSubaccounts[subaccount]` is `bytes32(0)` and the mask bit is cleared, this position is completely untracked — a **ghost position** with no parent responsible for its losses.

---

### Impact Explanation

Ghost positions in closed isolated subaccounts are invisible to the parent's health calculation. If the ghost position accrues losses, those losses cannot be attributed to any parent subaccount and will instead be socialized against the insurance fund during finalization. This directly corrupts the solvency accounting of the protocol: the insurance fund absorbs losses that should have been covered by the position owner, and the parent subaccount's health is understated relative to its true risk exposure.

---

### Likelihood Explanation

The scenario requires a partially-filled isolated order to remain open at the time the isolated subaccount is liquidated and closed. In volatile markets this is a routine occurrence: a user places a large isolated order, partial fills execute, the market moves sharply, the isolated subaccount is liquidated (position zeroed), `_tryCloseIsolatedSubaccount` fires, and the remaining unfilled portion of the order is still live in the sequencer's order book. The sequencer then legitimately attempts to fill the remainder — no malicious actor is required. Likelihood is **medium**.

---

### Recommendation

Inside `_tryCloseIsolatedSubaccount`, also clear `digestToSubaccount[digest]` (and `digestToMargin[digest]`) for every order digest associated with the closing isolated subaccount. Alternatively, add an active-subaccount guard inside `matchOrders` before the sender redirect:

```solidity
if (digestToSubaccount[ordersInfo.taker.digest] != bytes32(0)) {
    bytes32 iso = digestToSubaccount[ordersInfo.taker.digest];
    require(isIsolatedSubaccountActive(taker.order.sender, iso), ERR_INVALID_TAKER);
    taker.order.sender = iso;
}
```

---

### Proof of Concept

1. User signs isolated order (digest `D`, amount 10) → sequencer calls `createIsolatedSubaccount` → `digestToSubaccount[D] = ISO_1`.
2. Sequencer matches 5 units: `filledAmounts[D] = 5`. ISO_1 holds a position of 5.
3. Market moves adversely; ISO_1's maintenance health < 0. Liquidator calls `liquidateSubaccountImpl` → `_finalizeSubaccount` → `tryCloseIsolatedSubaccount(ISO_1)`.
   - ISO_1's position is zeroed during finalization.
   - `parentSubaccounts[ISO_1] = bytes32(0)`, mask bit cleared, slot freed.
   - **`digestToSubaccount[D]` is NOT cleared — still equals `ISO_1`.**
4. Sequencer processes the remaining 5 units of order `D` via `matchOrders`.
5. Line 673: `digestToSubaccount[D] != bytes32(0)` → `taker.order.sender = ISO_1`.
6. Isolated-subaccount guard (lines 644–651) already passed against the original sender.
7. `_updateBalances` opens a new 5-unit position inside ISO_1.
8. ISO_1 now holds a live position with `parentSubaccounts[ISO_1] == bytes32(0)` and no mask bit — a ghost position outside all health accounting. [5](#0-4) [6](#0-5)

    function _tryCloseIsolatedSubaccount(bytes32 subaccount) internal {
        uint32 productId = RiskHelper.getIsolatedProductId(subaccount);
        if (productId == 0) {
            return;
        }
        IPerpEngine.Balance memory balance = perpEngine.getBalance(
            productId,
            subaccount
        );
        if (balance.amount == 0) {
            uint8 id = RiskHelper.getIsolatedId(subaccount);
            address addr = address(uint160(bytes20(subaccount)));
            bytes32 parent = parentSubaccounts[subaccount];
            if (balance.vQuoteBalance != 0) {
                perpEngine.updateBalance(
                    productId,
                    subaccount,
                    0,
                    -balance.vQuoteBalance
                );
                perpEngine.updateBalance(
                    productId,
                    parent,
                    0,
                    balance.vQuoteBalance
                );
            }
            int128 quoteBalance = spotEngine
                .getBalance(QUOTE_PRODUCT_ID, subaccount)
                .amount;
            if (quoteBalance != 0) {
                spotEngine.updateBalance(
                    QUOTE_PRODUCT_ID,
                    subaccount,
                    -quoteBalance
                );
                spotEngine.updateBalance(
                    QUOTE_PRODUCT_ID,
                    parent,
                    quoteBalance
                );
            }
            isolatedSubaccountsMask[addr] &= ~uint256(0) ^ (1 << id);
            isolatedSubaccounts[parent][id] = bytes32(0);
            parentSubaccounts[subaccount] = bytes32(0);

            emit CloseIsolatedSubaccount(subaccount, parent);
        }
    }
