### Title
SwapAllowlistExtension Bypass via Router: Allowlist Gates Router Address Instead of Original User — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument passed by the pool, which is `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router contract**, not the original user. If the pool admin allowlists the router address (to permit allowlisted users to use the router), any unprivileged user can bypass the per-user swap restriction by calling the router, because the extension sees `sender = router` and the router is allowlisted.

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this value unchanged as the first argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the immediate caller of `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` (and all other router entry points) calls `pool.swap()` directly with no access control, making the pool see `msg.sender = router`: [4](#0-3) 

The router itself has no per-user access control — any EOA can call it: [5](#0-4) 

**Broken invariant**: A pool admin who wants to allow allowlisted users to swap via the router must add the router address to `allowedSwapper[pool]`. Once the router is allowlisted, the extension check `allowedSwapper[pool][router]` passes for every caller of the router — including users who are not individually allowlisted — because the extension never sees the original user's address.

### Impact Explanation

Any user can execute swaps on a pool whose admin intended to restrict trading to a specific set of addresses. The pool's swap allowlist — the only on-chain mechanism preventing unauthorized trading — is fully bypassed. Unauthorized swaps can:

- Drain LP-owned token reserves at oracle-quoted prices, causing direct loss of LP principal.
- Disrupt pool economics (fee accrual, bin cursor position) in ways the admin explicitly tried to prevent.
- Allow adversarial actors to trade against a restricted pool during sensitive periods (e.g., oracle updates, low-liquidity windows).

This matches the allowed impact gate: **admin-boundary break — pool admin's allowlist is bypassed by an unprivileged path**.

### Likelihood Explanation

The scenario is directly reachable whenever a pool admin:
1. Deploys a pool with `SwapAllowlistExtension` to restrict swaps to specific addresses.
2. Also allowlists the router so that those specific addresses can use the standard periphery UX.

Step 2 is the natural operational choice — requiring allowlisted users to call the pool directly (bypassing the router) is an unusual and undocumented restriction. Any attacker who observes that the router is allowlisted can immediately exploit this with a single `exactInputSingle` call. No privileged access, no special setup, and no non-standard tokens are required.

### Recommendation

The extension must gate the **original user**, not the immediate caller of `pool.swap()`. Two sound approaches:

1. **Check `sender` only for direct pool calls; reject router-mediated calls unless the original user is also allowlisted.** This requires the router to forward the original user's address in `extensionData`, and the extension to decode and verify it — but this is trust-dependent on the router.

2. **Preferred — check `sender` and reject any `sender` that is itself a contract (i.e., not an EOA).** This prevents router bypass entirely, at the cost of requiring allowlisted users to call the pool directly.

3. **Alternatively**, document explicitly that allowlisting the router opens the pool to all users, and provide a separate `SwapAllowlistExtension` variant that decodes a signed user identity from `extensionData` for router-mediated flows.

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true          // alice is the intended user
  allowedSwapper[pool][router] = true         // admin adds router so alice can use it

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, ...})

  Execution trace:
    router.exactInputSingle()
      → pool.swap(recipient, ...) [msg.sender = router]
        → _beforeSwap(sender=router, ...)
          → SwapAllowlistExtension.beforeSwap(sender=router, ...)
            → allowedSwapper[pool][router] == true  ✓  (no revert)
        → swap executes, bob receives output tokens

Result:
  bob swaps successfully on a pool he is not allowlisted for.
  The allowlist invariant is broken; LP funds are exposed to unauthorized trading.
```
