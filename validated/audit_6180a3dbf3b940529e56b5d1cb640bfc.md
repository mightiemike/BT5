Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` gates `owner` instead of `sender`, allowing any caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `addLiquidity` caller who pays tokens via callback) and instead checks `owner`, a parameter freely supplied by the caller. Because any caller can set `owner` to any allowlisted address, the allowlist gate is trivially bypassed: an unpermissioned address can deposit into a restricted pool by naming an allowlisted address as the position recipient.

## Finding Description
`MetricOmmPool.addLiquidity` dispatches the before-hook as:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
//                  ^^^^^^^^^^  ^^^^^
//                  actual payer  caller-supplied recipient
```

The extension receives both identities but drops `sender` (unnamed first argument) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

**Direct pool path:** Bob (unpermissioned) deploys a contract implementing `metricOmmModifyLiquidityCallback` and calls `pool.addLiquidity(alice, salt, deltas, callbackData, "")`. The extension evaluates `allowedDepositor[pool][alice] → true` and passes. The pool calls `metricOmmModifyLiquidityCallback` on Bob's contract; Bob pays the tokens and Alice receives the LP position.

**Liquidity adder path:** Bob calls `adder.addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, "")`. The adder calls `pool.addLiquidity(alice, ...)` with itself as `msg.sender`. The extension again checks `allowedDepositor[pool][alice] → true`. The adder's callback pulls tokens from `msg.sender` of the adder call (Bob, stored as `payer` in transient context). `_validateOwner` only rejects `address(0)` and does not enforce `owner == msg.sender`, so this path is equally exploitable.

The `allowedDepositor` mapping is `public`, making all allowlisted addresses trivially discoverable on-chain.

## Impact Explanation
The `DepositAllowlistExtension` is the sole on-chain mechanism for pool admins to restrict liquidity providers. Checking `owner` instead of `sender` makes the gate completely ineffective: any address can deposit into a restricted pool by naming any allowlisted address as `owner`. This is a confirmed admin-boundary break — an unprivileged path bypasses an admin-configured access control. Secondary impact includes griefing of allowlisted LPs by forcing unwanted positions onto their addresses without their consent.

## Likelihood Explanation
Exploitation requires only: (1) reading the public `allowedDepositor` mapping to find one allowlisted address, and (2) calling `addLiquidity` directly on the pool or through the liquidity adder with that address as `owner`. No special privileges, flash loans, oracle manipulation, or privileged setup are needed. Every pool that deploys `DepositAllowlistExtension` with a non-empty allowlist is affected on every `addLiquidity` call.

## Recommendation
Replace the unnamed first argument with `sender` and gate on it instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`sender` is `msg.sender` of the `addLiquidity` call — the entity that pays tokens via the callback — which is the economically relevant identity the allowlist is meant to gate.

## Proof of Concept
1. Deploy a pool with `DepositAllowlistExtension` configured as the `beforeAddLiquidity` hook.
2. Call `extension.setAllowedToDeposit(pool, alice, true)` from the pool admin. Bob is **not** allowlisted.
3. From Bob's contract, call `pool.addLiquidity(alice, 0, deltas, callbackData, "")` directly (Bob's contract implements `metricOmmModifyLiquidityCallback` to pay tokens).
4. Observe: the extension evaluates `allowedDepositor[pool][alice] → true` and does **not** revert.
5. The pool calls `metricOmmModifyLiquidityCallback` on Bob's contract; Bob's tokens are pulled and Alice receives the LP shares.
6. Alternatively, call `adder.addLiquidityExactShares(pool, alice, 0, deltas, max0, max1, "")` from Bob's EOA — same result via the adder path.
7. Bob has successfully deposited into a pool he was not permitted to access.
