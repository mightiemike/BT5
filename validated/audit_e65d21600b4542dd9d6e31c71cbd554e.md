### Title
Bypassable Coinbase-Proof Security Control Allows Merkle-Proof Forgery via Deprecated `verify_transaction_inclusion` — (File: `contract/src/lib.rs`)

### Summary
The contract exposes two SPV verification entry points: the secure `verify_transaction_inclusion_v2` (which requires a coinbase Merkle proof to defeat the 64-byte-transaction forgery attack) and the deprecated `verify_transaction_inclusion` (which omits that check). The deprecated function carries no access control and remains fully callable by any unprivileged NEAR account. Any recipient contract that calls it can be deceived into accepting a forged proof where an internal Merkle-tree node is presented as a confirmed transaction hash.

### Finding Description
`verify_transaction_inclusion` is a `pub` function gated only by `#[pause]`; it has no `#[private]` or `#[trusted_relayer]` restriction. Its sole Merkle check is:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
```

`compute_root_from_merkle_proof` treats `args.tx_id` as a leaf and hashes it upward through the supplied siblings. It does not verify that `tx_id` is a real leaf (transaction) rather than an internal node. An attacker who knows the Merkle tree of any already-submitted Bitcoin block can supply an internal node `H_internal` at the appropriate tree position together with the sibling nodes above it; the computation will reproduce the correct Merkle root and the function will return `true`.

`verify_transaction_inclusion_v2` closes this gap by first requiring that the coinbase transaction (always at position 0, always a real leaf) produces the same Merkle root, making it impossible to substitute an internal node. That check is entirely absent from the deprecated path.

The only guard in the deprecated function against the trivial degenerate case is:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
```

This prevents passing the root itself as `tx_id` with an empty proof, but it does not prevent passing any internal node one or more levels below the root.

### Impact Explanation
Any NEAR contract (bridge, DeFi protocol, escrow) that calls `verify_transaction_inclusion` to gate asset release or state transitions can be tricked into believing a non-existent Bitcoin transaction was confirmed. The attacker does not need to mine a block or control any privileged role; they only need a real Bitcoin block already tracked by the light client and knowledge of its public Merkle tree (available from any Bitcoin block explorer). The forged proof causes the verification function to return `true`, directly corrupting the proof result consumed by the recipient contract.

### Likelihood Explanation
The deprecated function is reachable by any NEAR account with no staking, role, or deposit requirement beyond the standard NEAR gas fee. The Merkle tree of every Bitcoin block is public. Constructing the forged proof is straightforward: pick any internal node at height `h` in the tree, set `tx_index` to the leaf-range index that maps to that node, and supply the `h` sibling hashes above it. The only prerequisite is that the target block has been submitted to the contract, which is the normal operating state of the relay. Likelihood is **high** for any downstream contract that has not migrated to `verify_transaction_inclusion_v2`.

### Recommendation
- **Short term:** Remove or hard-revert `verify_transaction_inclusion` so it always panics (or delegates unconditionally to `verify_transaction_inclusion_v2`), eliminating the bypassable path entirely.
- **Long term:** Enforce coinbase-proof validation as a non-optional invariant inside the single shared verification routine so that no future entry point can omit it.

### Proof of Concept

1. Identify any Bitcoin block already tracked by the contract, e.g. block at height `H` with Merkle root `R` and four transactions `[T0, T1, T2, T3]`.
2. Compute `H01 = SHA256d(T0 ‖ T1)` (an internal node at tree level 1, position 0).
3. Call `verify_transaction_inclusion` with:
   - `tx_id = H01`
   - `tx_block_blockhash` = hash of the block at height `H`
   - `tx_index = 0` (position of `H01` at level 1, treated as a leaf)
   - `merkle_proof = [H23]` where `H23 = SHA256d(T2 ‖ T3)`
   - `confirmations = 1`
4. `compute_root_from_merkle_proof(H01, 0, [H23])` = `SHA256d(H01 ‖ H23)` = `R`.
5. The function returns `true`, certifying that the non-existent "transaction" `H01` is confirmed in the block.

            args.confirmations <= self.gc_threshold,

        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,

#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
