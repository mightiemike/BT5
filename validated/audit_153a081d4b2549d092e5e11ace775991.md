Audit Report

## Title
Per-Bin Additional Fee Has No Cap, Allowing Pool Admin to Exceed Factory Fee Limits - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

## Summary
`setPoolBinAdditionalFees` in `MetricOmmPoolFactory.sol` forwards `addFeeBuyE6` and `addFeeSellE6` directly to the pool with no upper-bound validation, while the analogous `setPoolAdminFees` enforces `maxAdminSpreadFeeE6` and `maxAdminNotionalFeeE8` caps. A pool admin can set per-bin additional fees up to `uint16` max (65535 in E6 = 6.5535%) on any bin, bypassing the factory's cap system and causing direct loss of funds to swappers routing through that bin.

## Finding Description
`setPoolAdminFees` enforces caps before writing:

```solidity
// MetricOmmPoolFactory.sol L414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

`setPoolBinAdditionalFees` has no equivalent check and forwards raw caller-supplied values straight to the pool:

```solidity
// MetricOmmPoolFactory.sol L450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The pool-side handler `setBinAdditionalFees` (L464-474) only validates the bin index range and writes the fee values unconditionally. These values are then used directly in swap math:

```solidity
// MetricOmmPool.sol L910
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
// MetricOmmPool.sol L1177
params.baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6)
```

The additional fee is added on top of the base spread fee for every swap routed through the targeted bin. No existing guard in either the factory or the pool constrains the magnitude of `addFeeBuyE6`/`addFeeSellE6`.

## Impact Explanation
Swappers whose trades route through the targeted bin pay the base spread fee **plus** the uncapped per-bin additional fee. At `uint16` max (65535 in E6 = 6.5535%), the additional fee constitutes a direct loss of user principal that flows to `adminFeeDestination`. The factory's published cap system (`maxAdminSpreadFeeE6`) is designed to bound pool admin fee extraction but is entirely bypassed for this code path. This is a Medium-severity admin-boundary break with direct loss of user funds.

## Likelihood Explanation
Low — requires a malicious or compromised pool admin. However, the inconsistency is structural: `setPoolAdminFees` is capped while `setPoolBinAdditionalFees` is not. The pool admin can apply the maximum fee to any bin at any time, including immediately before a pending swap lands, making the attack repeatable and front-runnable.

## Recommendation
Add a cap check inside `setPoolBinAdditionalFees` before forwarding to the pool, mirroring the pattern used in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxAdminBinAdditionalFeeE6` cap that can be tuned independently.

## Proof of Concept
1. Pool is deployed with `spreadFeeE6 = 10_000` (1%) and `maxAdminSpreadFeeE6 = 200_000` (20%).
2. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
3. Factory forwards the call; pool writes `addFeeBuyE6 = 65535`, `addFeeSellE6 = 65535` for bin 0 with no revert — confirmed by `setBinAdditionalFees` at `MetricOmmPool.sol` L469-472 which only checks bin index range.
4. A victim's swap routing through bin 0 computes effective fee as `baseFeeX64 + Math.mulDiv(65535, ONE_X64, 1e6)` (L910), yielding 1% base + 6.5535% additional = 7.5535% total spread fee.
5. The excess 6.5535% flows to `adminFeeDestination`, constituting a direct loss of principal to the swapper that the factory's cap system was supposed to prevent. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
