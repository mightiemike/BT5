### Title
Authz Grantee Can Redirect Granter's NFT to Attacker-Controlled Receiver via Unconstrained `MsgTransfer` Grant — (`x/nft-transfer/keeper/msg_server.go`, `x/nft-transfer/keeper/packet.go`)

---

### Summary

The `x/nft-transfer` module has no `TransferAuthorization` type. Any `MsgTransfer` grant is therefore a `GenericAuthorization`. A grantee can construct a `MsgTransfer` with `Sender = granter` and `Receiver = attacker_counterparty_address`, wrap it in `MsgExec`, and cause the granter's NFT to be escrowed and delivered to the attacker on the counterparty chain. The granter's NFT is permanently lost.

---

### Finding Description

**Step 1 — `GetSigners()` returns `msg.Sender` verbatim.**

`MsgTransfer.GetSigners()` parses and returns `msg.Sender` with no cross-check against the actual transaction signer: [1](#0-0) 

**Step 2 — `Keeper.Transfer` uses `msg.Sender` as the authoritative sender.**

The msg server parses `msg.Sender` and passes it directly to `SendTransfer` without any additional signer-equality check: [2](#0-1) 

**Step 3 — `createOutgoingPacket` owner check passes because `sender` == granter == NFT owner.**

The only ownership guard is `sender.Equals(owner)`. When the attacker sets `msg.Sender` to the granter's address, `sender` is the granter, who is the legitimate owner, so the check succeeds: [3](#0-2) 

**Step 4 — Packet is constructed with `receiver` = attacker's counterparty address.**

The packet data is built from `sender.String()` (granter) and `receiver` (attacker-supplied), so the counterparty chain delivers the NFT to the attacker: [4](#0-3) 

**Step 5 — No `TransferAuthorization` type exists to restrict the `Receiver` field.**

A search across the entire repository finds zero occurrences of `TransferAuthorization`. The IBC fungible-token transfer module in ibc-go ships a `TransferAuthorization` that constrains allowed channels and spend limits; this NFT-transfer module has no equivalent. The only available grant type is therefore `GenericAuthorization`, which imposes no restriction on any field of the inner message.



---

### Impact Explanation

An authz grantee holding a `GenericAuthorization` for `MsgTransfer` can:

1. Set `msg.Sender` = granter (victim) address.
2. Set `msg.Receiver` = attacker's address on the counterparty chain.
3. Submit `MsgExec` wrapping that `MsgTransfer`.

The authz ante handler verifies the grantee signed the outer tx; the inner message's `GetSigners()` returns the granter, the grant is found, and the message executes. The granter's NFT is moved to the escrow address on the source chain and the IBC packet instructs the counterparty to deliver it to the attacker. The granter permanently loses the NFT with no recourse.

---

### Likelihood Explanation

Any scenario where a user grants `MsgTransfer` authorization to a third party (e.g., a marketplace contract, a relayer helper, a delegated operator) is exploitable. The attacker needs only one valid grant; no privileged role, governance action, or key compromise is required. The grant itself is a normal on-chain transaction.

---

### Recommendation

Implement a `TransferAuthorization` type (analogous to ibc-go's `TransferAuthorization` for ICS-20) that records the allowed `(sourcePort, sourceChannel, receiver)` tuples. Register it in the codec and use it in the authz grant acceptance check so that a grantee cannot redirect the granter's NFT to an arbitrary receiver. Until this type exists, `GenericAuthorization` for `MsgTransfer` must be treated as granting full, unrestricted transfer capability over all of the granter's NFTs.

---

### Proof of Concept

```go
// Keeper-level test (no IBC relayer needed; assert escrow + packet receiver)

func TestAuthzGranteeRedirectsNFT(t *testing.T) {
    // Setup: victim mints NFT, attacker obtains GenericAuthorization for MsgTransfer
    // victim  = granter
    // attacker = grantee

    // 1. Victim mints NFT
    nftKeeper.MintNFT(ctx, "classA", "token1", "", "", "", victim, victim)

    // 2. Victim grants GenericAuthorization for MsgTransfer to attacker
    grant := authz.NewGrant(time.Now().Add(time.Hour),
        authz.NewGenericAuthorization(sdk.MsgTypeURL(&nfttransfertypes.MsgTransfer{})), nil)
    authzKeeper.SaveGrant(ctx, attacker, victim, grant)

    // 3. Attacker constructs MsgTransfer with Sender=victim, Receiver=attacker_counterparty
    innerMsg := nfttransfertypes.NewMsgTransfer(
        "nft-transfer", "channel-0",
        "classA", []string{"token1"},
        victim.String(),          // Sender = granter
        "cosmos1attacker...",     // Receiver = attacker on counterparty
        clienttypes.ZeroHeight(), uint64(time.Now().Add(time.Hour).UnixNano()),
    )

    // 4. Wrap in MsgExec and execute
    execMsg := authz.NewMsgExec(attacker, []sdk.Msg{innerMsg})
    _, err := authzMsgServer.Exec(ctx, &execMsg)
    require.NoError(t, err)

    // 5. Assert: NFT is now owned by escrow address (not victim)
    escrow := nfttransfertypes.GetEscrowAddress("nft-transfer", "channel-0")
    nft, _ := nftKeeper.GetNFT(ctx, "classA", "token1")
    require.Equal(t, escrow.String(), nft.GetOwner().String())

    // 6. Assert: outgoing packet receiver is attacker, not victim
    // (inspect the packet committed to the IBC store)
    // packet.Data.Receiver == "cosmos1attacker..."
}
```

The NFT leaves the victim's ownership and the IBC packet carries the attacker's address as receiver — the invariant that only the NFT owner controls the destination of an IBC transfer is broken.
