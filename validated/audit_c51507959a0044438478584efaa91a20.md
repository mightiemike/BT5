### Title
Double-Spend via Uninitialized `minIdx` on Pool Replacement — (`core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool` initializes `minIdx` to `0` and provides no admin setter for it. When the pool is replaced via `Clearinghouse.setWithdrawPool`, the new pool has no record of withdrawals already processed on the old pool. Any user who holds a valid signed withdrawal transaction for an old idx can call `submitFastWithdrawal` on the new pool — passing both guards — and receive a second payout. Because the sequencer will never call `submitWithdrawal` for that old idx on the new pool, the pool is never replenished, violating the core accounting invariant.

---

### Finding Description

**`BaseWithdrawPool._initialize`** sets only `clearinghouse` and `verifier`; `minIdx` is left at its default of `0`: [1](#0-0) 

There is no setter for `minIdx` anywhere in the contract. The only code that advances it is inside `submitWithdrawal`: [2](#0-1) 

`submitFastWithdrawal` gates on exactly two conditions: [3](#0-2) 

When a new pool is deployed and `setWithdrawPool` is called: [4](#0-3) 

…the new pool has `minIdx = 0` and an empty `markedIdxs` mapping. Every idx that was processed on the *old* pool satisfies both guards on the new pool (`markedIdxs[idx] == false` and `idx > 0`).

`Clearinghouse.handleWithdrawTransfer` only replenishes the pool when it processes a *new* withdrawal: [5](#0-4) 

It will never call `submitWithdrawal` for an idx it already settled on the previous pool. So the new pool pays out via `handleWithdrawTransfer → token.safeTransfer(to, amount)` but is never topped up. [6](#0-5) 

---

### Impact Explanation

Every user who holds a valid signed withdrawal transaction for an idx processed on the old pool can call `submitFastWithdrawal` on the new pool and receive a second payout. The new pool's liquidity is drained proportionally to the number of such replayed withdrawals, leaving it unable to service legitimate fast withdrawals. This is a direct, unrecoverable loss of protocol-held collateral — a solvency failure for the pool.

---

### Likelihood Explanation

The precondition is a pool replacement via `setWithdrawPool`, which is a supported owner operation. The attack window opens at deployment of the new pool and closes only after the sequencer processes the *first* new withdrawal on it (advancing `minIdx` past all old idxs). During that window — which could span many blocks if no new withdrawals are pending — any holder of an old signed transaction can replay it. The owner action is legitimate (not malicious), and the signed transactions are in users' possession by design.

---

### Recommendation

1. Add an `initMinIdx(uint64 _minIdx) external onlyOwner` setter (callable only before the first `submitWithdrawal`) so the deployer can set `minIdx` to the current sequencer head when deploying or replacing the pool.
2. Alternatively, pass the initial `minIdx` as a constructor/initializer argument and enforce it in `_initialize`.
3. Document that `setWithdrawPool` must be followed immediately by an `initMinIdx` call before any user interaction is possible.

---

### Proof of Concept

```
1. Deploy OldPool; Clearinghouse.setWithdrawPool(OldPool).
2. Sequencer processes withdrawals idx=1..100 on OldPool.
   - For each, handleWithdrawTransfer sends tokens to OldPool and calls submitWithdrawal.
   - OldPool.markedIdxs[1..100] = true; OldPool.minIdx = 100.
   - Users receive funds from OldPool.
3. Owner deploys NewPool; Clearinghouse.setWithdrawPool(NewPool).
   - NewPool.minIdx = 0; NewPool.markedIdxs[*] = false.
4. Attacker (original user for idx=50) calls:
     NewPool.submitFastWithdrawal(50, signedTx50, sigs)
   - require(!markedIdxs[50])  → passes (false)
   - require(50 > minIdx=0)    → passes
   - markedIdxs[50] = true
   - NewPool transfers `amount` to attacker  ← second payout
5. Sequencer never calls submitWithdrawal(50) on NewPool.
   - NewPool balance decreases by `amount` with no replenishment.
6. Assert: NewPool token balance decreased; attacker received funds twice.
``` [7](#0-6)

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
```

    function submitWithdrawal(
        IERC20Base token,
        address sendTo,
        uint128 amount,
        uint64 idx
    ) public {
        require(msg.sender == clearinghouse);

        if (markedIdxs[idx]) {
            return;
        }
        markedIdxs[idx] = true;
        // set minIdx to most recent withdrawal submitted by sequencer
        minIdx = idx;

        handleWithdrawTransfer(token, sendTo, amount);
    }
