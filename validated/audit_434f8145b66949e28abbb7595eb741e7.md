### Title
Unchecked `approve()` Return Value Enables Silent Deposit Failure or Accounting Corruption ‚Äî (`File: core/contracts/DirectDepositV1.sol`, `core/contracts/ContractOwner.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` and `ContractOwner.wrapVaultAsset()` call `approve()` on ERC20 tokens without checking the boolean return value. The codebase explicitly implements `safeTransfer` and `safeTransferFrom` wrappers in `ERC20Helper` and inside `DirectDepositV1` itself, but provides no equivalent `safeApprove`. For non-standard ERC20 tokens that signal failure via `return false` rather than `revert`, the approval silently fails while execution continues, causing the subsequent deposit call to proceed without a valid allowance.

---

### Finding Description

**Instance 1 ‚Äî `DirectDepositV1.creditDeposit()` (primary)**

`creditDeposit()` is an unrestricted `external` function with no access-control modifier. It iterates over all spot product IDs, and for each token held by the DDA contract, calls:

```solidity
token.approve(address(endpoint), balance);   // line 92 ‚Äî return value discarded
endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1");
```

The `IIERC20Base` interface (defined locally in `DirectDepositV1.sol`) declares `approve` as returning `bool`, but the return value is never read. The same contract carefully wraps `transfer` in a low-level call that checks both `success` and the decoded boolean:

```solidity
// DirectDepositV1.sol lines 69-81
function safeTransfer(IIERC20Base self, address to, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(...);
    require(success && (data.length == 0 || abi.decode(data, (bool))), "Transfer failed");
}
```

No analogous `safeApprove` exists anywhere in the codebase ‚Äî `ERC20Helper` only provides `safeTransfer` and `safeTransferFrom`.

**Instance 2 ‚Äî `ContractOwner.wrapVaultAsset()` (secondary)**

`wrapVaultAsset()` is also an unrestricted `external` function. After withdrawing asset tokens from the DDA into `ContractOwner`, it calls:

```solidity
assetToken.approve(tokenAddr, 0);           // line 530 ‚Äî return value discarded
assetToken.approve(tokenAddr, assetBalance); // line 531 ‚Äî return value discarded
IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1);
```

`ContractOwner` imports `ERC20Helper` and uses it for transfers (`using ERC20Helper for IERC20Base`), but calls `approve` directly without the safe wrapper.

---

### Impact Explanation

For non-standard ERC20 tokens (e.g., USDT on some chains, BNB, ZRX) that return `false` on approval failure instead of reverting:

- **`creditDeposit()`**: The `approve()` silently fails, leaving the endpoint's allowance at zero. `depositCollateralWithReferral` is then called. If the endpoint's internal `transferFrom` also does not revert on failure (a pattern seen in non-compliant tokens), the protocol records a deposit credit for the subaccount without actually receiving the tokens. This corrupts the subaccount's collateral balance and the protocol's solvency accounting. Even in the case where the endpoint does revert, the failure is opaque ‚Äî the caller receives no indication that the approval step was the root cause, and user funds remain stranded in the DDA contract with no automatic retry.

- **`wrapVaultAsset()`**: If the first `approve(tokenAddr, 0)` fails silently (as required by USDT-style tokens before re-approving), the second `approve(tokenAddr, assetBalance)` will also fail silently (USDT reverts if current allowance is non-zero). The subsequent `deposit()` call proceeds without allowance and reverts, but the tokens have already been pulled from the DDA into `ContractOwner`. The full transaction reverts, but the pattern is fragile and the silent failure masks the true error.

---

### Likelihood Explanation

`creditDeposit()` and `wrapVaultAsset()` are both callable by any unprivileged external address. The Nado protocol is designed to support multiple spot collateral tokens via `spotEngine.getProductIds()`. If any listed token is a non-standard ERC20 that returns `false` on `approve()` failure (rather than reverting), the silent failure path is reachable on every call to `creditDeposit()` for that token. The inconsistency with the carefully implemented `safeTransfer` pattern in the same contracts indicates this was an oversight rather than an intentional design choice.

---

### Recommendation

1. Add a `safeApprove` function to `ERC20Helper` using the same low-level call pattern as `safeTransfer`/`safeTransferFrom`, and use it in `ContractOwner.wrapVaultAsset()`.
2. Add an equivalent `safeApprove` helper inside `DirectDepositV1` (mirroring its existing `safeTransfer`) and replace the bare `token.approve(...)` call at line 92.
3. For the USDT-style double-approve pattern in `wrapVaultAsset()`, verify the reset-to-zero approve also uses the safe wrapper.

pragma solidity ^0.8.0;

import "../interfaces/IERC20Base.sol";
import "../common/Errors.sol";

// @dev Adapted from https://github.com/Uniswap/v3-core/blob/main/contracts/libraries/TransferHelper.sol
library ERC20Helper {
    function safeTransfer(
        IERC20Base self,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
        );
        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
}
