### Title
Stale `digestToSubaccount` Mapping After Isolated Subaccount Close Causes Incorrect Order Routing and Accounting Corruption — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

When an isolated subaccount is closed via `_tryCloseIsolatedSubaccount`, the `digestToSubaccount` and `digestToMargin` mappings are never cleared. This mirrors the original bug class exactly: a status-clearing action (closing the isolated subaccount) zeroes out some state fields but leaves a stale amount/routing field intact. A subsequent execution path (`matchOrders`) reads the stale mapping and routes the order to the now-closed, zero-balance isolated subaccount, corrupting accounting.

---

### Finding Description

In `OffchainExchange.sol`, when an isolated order is created, two mappings are populated:

```solidity
// which isolated subaccount does an isolated order create
mapping(bytes32 => bytes32) internal digestToSubaccount;

// how much margin does an isolated order require
mapping(bytes32 => int128) internal digestToMargin;
``` [1](#0-0) 

When `_tryCloseIsolatedSubaccount` is called (triggered by liquidation), it clears three fields:

```solidity
isolatedSubaccountsMask[addr] &= ~uint256(0) ^ (1 << id);
isolatedSubaccounts[parent][id] = bytes32(0);
parentSubaccounts[subaccount] = bytes32(0);
``` [2](#0-1) 

But `digestToSubaccount[digest]` and `digestToMargin[digest]` are **never cleared**. They persist pointing to the now-closed isolated subaccount.

In `matchOrders`, the stale `digestToSubaccount` is unconditionally used to redirect the taker's sender:

```solidity
if (digestToSubaccount[ordersInfo.taker.digest] != bytes32(0)) {
    taker.order.sender = digestToSubaccount[ordersInfo.taker.digest];
}
``` [3](#0-2) 

Because `digestToSubaccount[digest]` is non-zero (stale), the sender is redirected to the closed isolated subaccount. The closed isolated subaccount has zero balance (all funds were transferred to the parent during close). The trade then executes against it, creating a negative balance in a subaccount whose `parentSubaccounts` entry has already been zeroed out.

The `isHealthy` guard in `matchOrders` uses the base implementation which always returns `true`:

```solidity
function isHealthy(
    bytes32 /* subaccount */
) internal view virtual returns (bool) {
    return true;
}
``` [4](#0-3) 

So the trade is not blocked.

---

### Impact Explanation

After the trade executes against the closed isolated subaccount:
- The isolated subaccount holds a non-zero perp position with no margin.
- `parentSubaccounts[subaccount]` is `bytes32(0)` (cleared during close).
- Any subsequent `_tryCloseIsolatedSubaccount` call will attempt to transfer the residual `vQuoteBalance` to `bytes32(0)` (the zero-address subaccount), permanently losing those funds from the protocol's accounting.

This is a concrete accounting corruption: funds are misdirected to an unrecoverable subaccount address. [5](#0-4) 

---

### Likelihood Explanation

The trigger path is realistic:
1. A user creates an isolated order (normal user action), setting `digestToSubaccount[D]`.
2. The order is partially filled; `filledAmounts[D]` is non-zero but the order is not exhausted.
3. The isolated subaccount becomes undercollateralized (e.g., adverse price move) and a liquidator liquidates it — an unprivileged, incentivized action.
4. `liquidateSubaccountImpl` → `_finalizeSubaccount` → `tryCloseIsolatedSubaccount` clears the subaccount state but leaves `digestToSubaccount[D]` stale.
5. The sequencer, acting normally, attempts to match the remaining portion of the order and routes it to the closed isolated subaccount. [6](#0-5) 

---

### Recommendation

In `_tryCloseIsolatedSubaccount`, after clearing the subaccount mappings, also clear the associated digest mappings. This requires storing the digest on the isolated subaccount at creation time, or iterating over known digests. At minimum, add:

```solidity
// pseudocode — requires digest to be stored on the isolated subaccount
bytes32 digest = isolatedSubaccountToDigest[subaccount];
if (digest != bytes32(0)) {
    digestToSubaccount[digest] = bytes32(0);
    digestToMargin[digest] = 0;
    isolatedSubaccountToDigest[subaccount] = bytes32(0);
}
``` [7](#0-6) 

---

### Proof of Concept

1. User submits isolated order for 100 units → `digestToSubaccount[D] = isoSubaccount`, `digestToMargin[D] = margin`.
2. Sequencer partially fills 50 units → `filledAmounts[D] = 50`.
3. Price moves adversely; isolated subaccount is undercollateralized.
4. Liquidator calls `liquidateSubaccountImpl` → `_tryCloseIsolatedSubaccount(isoSubaccount)`:
   - `parentSubaccounts[isoSubaccount] = bytes32(0)` ✓ cleared
   - `isolatedSubaccounts[parent][id] = bytes32(0)` ✓ cleared
   - `digestToSubaccount[D]` ✗ **still = isoSubaccount**
5. Sequencer submits `matchOrders` for remaining 50 units of digest `D`.
6. `digestToSubaccount[D] != bytes32(0)` → `taker.order.sender = isoSubaccount` (closed, zero-balance).
7. `isHealthy(isoSubaccount)` returns `true` (base implementation).
8. Trade executes: `isoSubaccount` receives −50 base, +quote → negative base balance, no parent to absorb it.
9. `parentSubaccounts[isoSubaccount] == bytes32(0)` → any future close attempt transfers residual balance to `bytes32(0)`, permanently lost. [8](#0-7) [9](#0-8)

        uint32 productId = RiskHelper.getIsolatedProductId(subaccount);
        if (productId == 0) {
            return;
        }
        IPerpEngine.Balance memory balance = perpEngine.getBalance(
            productId,
            subaccount
        );
        if (balance.amount == 0) {
            uint8 id = RiskHelper.getIsolatedId(subaccount);
            address addr = address(uint160(bytes20(subaccount)));
            bytes32 parent = parentSubaccounts[subaccount];
            if (balance.vQuoteBalance != 0) {
                perpEngine.updateBalance(
                    productId,
                    subaccount,
                    0,
                    -balance.vQuoteBalance
                );
                perpEngine.updateBalance(
                    productId,
                    parent,
                    0,
                    balance.vQuoteBalance
                );

                .getBalance(QUOTE_PRODUCT_ID, subaccount)
                .amount;
            if (quoteBalance != 0) {

        }
