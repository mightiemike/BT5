### Title
Missing Post-Multiplication Cost Invariant Check in `op_unknown` Allows Cost Limit Bypass - (File: `src/more_ops.rs`)

---

### Summary

`op_unknown` in `src/more_ops.rs` checks the computed base cost against `max_cost` before applying the `cost_multiplier`, but never re-checks the invariant `cost ≤ max_cost` after the multiplication. An attacker-controlled unknown opcode can cause `op_unknown` to return a `Reduction` whose cost exceeds the caller-supplied `max_cost`, violating the contract that every operator must enforce the cost budget.

---

### Finding Description

`op_unknown` computes a base cost from the opcode's `cost_function` field (0–3), then scales it by `(cost_multiplier + 1)` where `cost_multiplier` is derived from the leading bytes of the opcode atom. The guard sequence is: [1](#0-0) 

```
assert!(cost > 0);

check_cost(cost, max_cost)?;      // ← guards base cost only
cost *= cost_multiplier + 1;      // ← scales cost; may now exceed max_cost
if cost > u32::MAX as u64 {       // ← only guards against u32 overflow
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))   // ← returned cost may exceed max_cost
}
```

`check_cost` at line 260 verifies the pre-multiplication cost. After line 261 multiplies by `(cost_multiplier + 1)`, the only guard is the `u32::MAX` overflow check at line 262. There is **no** second `check_cost(cost, max_cost)` call after the multiplication. Any value in the range `(max_cost, u32::MAX]` is silently returned as a valid `Reduction`.

The `cost_multiplier` is decoded from the opcode atom prefix via `u32_from_u8`, giving an attacker full control over a value up to `u32::MAX`. [2](#0-1) 

The base cost for `cost_function = 0` is exactly `1`: [3](#0-2) 

So with `cost_function = 0`, the pre-multiplication cost is always `1`, which passes `check_cost(1, max_cost)` for any positive `max_cost`. The attacker then freely chooses `cost_multiplier` to make the final cost land anywhere in `(max_cost, u32::MAX]`.

---

### Impact Explanation

`op_unknown` is the authoritative cost enforcer for unknown opcodes; `max_cost` is passed precisely so the function can reject programs that exceed the budget. By returning a `Reduction` with `cost > max_cost`, `op_unknown` violates this contract. The returned cost is added to the running total in `run_program`. Even if `run_program` performs a secondary check, the broken invariant creates a **consensus divergence** risk: any alternative CLVM implementation that trusts `op_unknown`'s return value without re-checking will accept a different set of programs than one that does re-check. This is the same class of impact as the reference report — a post-computation invariant that is assumed to hold but is not enforced.

Unknown opcodes are reachable in non-mempool (consensus) mode because `NO_UNKNOWN_OPS` is not set by default: [4](#0-3) 

---

### Likelihood Explanation

The trigger requires only a crafted CLVM atom used as an opcode — fully attacker-controlled bytes passed through the normal `run_program` evaluation path. No special privileges, configuration, or social engineering are needed. The opcode format is documented in the source comments, making construction trivial.

---

### Recommendation

Add a `check_cost` call **after** the multiplication, before returning:

```rust
check_cost(cost, max_cost)?;
cost *= cost_multiplier + 1;
check_cost(cost, max_cost)?;   // ← add this
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

This mirrors the pattern used in every other operator in the file, where `check_cost` is called after each incremental cost update. [1](#0-0) 

---

### Proof of Concept

Construct a 5-byte unknown opcode atom with:
- Bytes 0–3: `cost_multiplier` encoded as big-endian `u32`. Choose `cost_multiplier = max_cost` (e.g., `0x000003E8` = 1000 if `max_cost = 1000`).
- Byte 4: `0x00` → `cost_function = 0`, base cost = 1.

Execution trace:
1. `cost_function = 0` → `cost = 1`
2. `check_cost(1, 1000)` → passes (1 ≤ 1000)
3. `cost *= 1000 + 1` → `cost = 1001`
4. `1001 ≤ u32::MAX` → passes
5. Returns `Ok(Reduction(1001, nil))` — cost **1001 > max_cost 1000**

The opcode bytes `[0x00, 0x00, 0x03, 0xE8, 0x00]` do not start with `0xff 0xff` and are not empty, so neither reserved-opcode guard fires. [5](#0-4) [1](#0-0)
