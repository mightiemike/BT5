Audit Report

## Title
SwapAllowlistExtension Gates on Router Address Instead of Originating Swapper, Enabling Full Allowlist Bypass - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which `MetricOmmPool.swap` sets to its own `msg.sender`. When any user routes through `MetricOmmSimpleRouter`, `msg.sender` inside `pool.swap` is the router contract, so the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. A pool admin who allowlists the router to restore usability simultaneously grants every non-allowlisted address unrestricted swap access, fully defeating the allowlist invariant.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension hook:**

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` at line 231. [1](#0-0) 

**Step 2 — `ExtensionCalling._beforeSwap` forwards `sender` unchanged:**

`ExtensionCalling._beforeSwap` encodes `sender` as the first positional argument to `IMetricOmmExtensions.beforeSwap`. [2](#0-1) 

**Step 3 — Extension checks `sender` against the per-pool allowlist:**

`SwapAllowlistExtension.beforeSwap` uses `sender` (the first argument) as the identity to gate on. `msg.sender` here is the pool (the caller of the extension), and `sender` is whatever address called `pool.swap`. [3](#0-2) 

**Step 4 — Router calls `pool.swap` directly, making itself `msg.sender`:**

`MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(...)` directly. At that call site, `msg.sender` inside `pool.swap` is the router contract, not the originating user. [4](#0-3) 

The same applies to `exactInput` (line 104), `exactOutputSingle` (line 136), and `exactOutput` (line 165). [5](#0-4) 

**Why existing guards fail:**

There is no mechanism in the pool, the extension calling path, or the router to substitute the originating user's address for the router's address before the extension check. The `extensionData` field is user-controlled and not authenticated. The `allowedSwapper` mapping has no concept of "trusted intermediary vs. direct caller." [6](#0-5) 

**Exact corrupted value:** `allowedSwapper[pool][router]` — a single storage entry intended to represent the router as a trusted intermediary that instead acts as a universal pass for every caller of that router on that pool.

## Impact Explanation

Any pool configured with `SwapAllowlistExtension` is subject to one of two mutually exclusive failure modes:

- **Mode A (allowlist blocks legitimate users):** Admin allowlists individual users (Alice). Alice calls `exactInputSingle`; extension sees `sender = router`, which is not allowlisted, and reverts. The protocol's primary swap interface is unusable for every curated pool.
- **Mode B (allowlist is fully bypassed):** Admin adds the router to `allowedSwapper[pool][router] = true` to fix Mode A. Now any non-allowlisted address (Bob) calls `exactInputSingle` → router calls `pool.swap()` → extension sees `sender = router` → `allowedSwapper[pool][router] == true` → swap executes. The allowlist provides zero protection.

This constitutes broken core pool functionality: the `SwapAllowlistExtension` is the sole on-chain mechanism for curating swap access to a pool, and it is rendered completely ineffective when the canonical router is used. Non-allowlisted users can freely swap on pools intended to be restricted, which may result in unauthorized access to curated liquidity and violation of pool admin access-control invariants.

## Likelihood Explanation

No privileged attacker capability is required. The trigger is a routine admin action (allowlisting the router) taken in direct response to an observed UX failure (Mode A). The router is the canonical, documented user-facing entry point. Any non-allowlisted address can exploit Mode B by simply calling a public router function. The condition is deterministic and repeatable with no special setup beyond the admin's natural remediation step.

## Recommendation

The extension must check the economic actor, not the intermediary. Two sound approaches:

1. **Pass the originating user via authenticated `extensionData`.** The router populates `extensionData` with `abi.encode(msg.sender)` before calling the pool. The extension reads and verifies that field. This requires the extension to also verify `msg.sender` (the pool) is a known factory-deployed pool to prevent spoofing.

2. **Require direct pool calls for curated pools.** Document and enforce (e.g., via a registry check in the extension) that pools with `SwapAllowlistExtension` must not be used with the shared router. Deploy a pool-specific router that enforces identity before calling the pool.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  admin calls extension.setAllowedToSwap(pool, alice, true)
  // Mode A observed: alice cannot use router → admin "fixes" it:
  admin calls extension.setAllowedToSwap(pool, router, true)

Attack (Bob, not allowlisted):
  bob calls router.exactInputSingle({pool: pool, recipient: bob, ...})
  → router calls pool.swap(bob, ...)          // msg.sender = router
  → pool calls _beforeSwap(router, ...)
  → extension checks allowedSwapper[pool][router] == true  ✓
  → swap executes; bob receives tokens from a curated pool
  → allowlist invariant broken: allowedSwapper[pool][router] acts as universal pass
```

Foundry test outline:
1. Deploy pool with `SwapAllowlistExtension`.
2. `setAllowedToSwap(pool, alice, true)` and `setAllowedToSwap(pool, router, true)`.
3. `vm.prank(bob)` → `router.exactInputSingle(...)` → assert swap succeeds despite Bob not being allowlisted.
4. `vm.prank(bob)` → `pool.swap(...)` directly → assert revert with `NotAllowedToSwap` (confirming direct-call protection still works, only router path is bypassed).
