### Title
v8 Migration Fails When Bonus Pool Is Empty for Vesting-Owned Positions — (`x/tieredrewards/keeper/force_exit.go`, `x/tieredrewards/migrations/v2/migrate.go`)

### Summary

The v8 upgrade migration (`Migrate1to2`) calls `ForceFullExitWithDelegation` for every vesting-account-owned tier position. That function unconditionally calls `claimRewards`, which internally calls `processEventsAndClaimBonus`. If the rewards pool has insufficient balance to cover accrued bonus, `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool` — an error that `ForceFullExitWithDelegation` propagates without any special handling. The error bubbles all the way up through `exitVestedAccountsPositions` → `Migrate` → `Migrate1to2` → `RunMigrations` → the v8 upgrade handler, causing the entire upgrade to fail and the chain to halt at the upgrade height.

### Finding Description

The v8 migration has two steps:

1. **`backfillDelegatorAddress`** — sets `DelegatorAddress` on all existing positions.
2. **`exitVestedAccountsPositions`** — walks all positions, identifies those owned by vesting accounts, and calls `ForceFullExitWithDelegation` on each. [1](#0-0) 

`ForceFullExitWithDelegation` unconditionally calls `claimRewards` before transferring the delegation back to the owner: [2](#0-1) 

`claimRewards` calls `processEventsAndClaimBonus`, which at line 230 calls `sufficientBonusPoolBalance`. If the pool balance is less than the accrued bonus, it returns `ErrInsufficientBonusPool`: [3](#0-2) [4](#0-3) 

This error is **not caught** in `ForceFullExitWithDelegation` — it is wrapped and returned: [2](#0-1) 

Compare this to the slash hook, which **explicitly** swallows `ErrInsufficientBonusPool` to prevent a chain halt: [5](#0-4) 

The upgrade handler propagates any error from `RunMigrations` directly: [6](#0-5) 

The ADR explicitly documents this design intent for user-driven paths ("fail atomically, user retries after pool is replenished") but the migration has no retry mechanism — it either succeeds atomically or the upgrade halts: [7](#0-6) 

### Impact Explanation

If any vesting-account-owned tier position has accrued non-zero bonus rewards and the `RewardsPoolName` module account has insufficient balance at upgrade time, the entire v8 upgrade fails. The chain halts at the upgrade height. Recovery requires an emergency governance proposal with a patched binary. All positions — not just vesting-owned ones — are affected by the halt.

The corrupted invariant is: the upgrade handler's `VersionMap` is never committed, leaving the chain in a state where the upgrade height has been reached but the new module version has not been recorded, requiring manual intervention.

### Likelihood Explanation

The `BeginBlocker` continuously drains the rewards pool to top up base staking rewards. If the pool is not actively replenished by governance, it can reach zero. A vesting-account-owned position that has been delegated for any meaningful duration (days to weeks) will have accrued non-zero bonus. Both conditions are independently reachable in normal chain operation, and their conjunction at upgrade time is realistic — especially since the upgrade is scheduled in advance and the pool balance is not a migration pre-condition that is checked or enforced.

### Recommendation

In `ForceFullExitWithDelegation`, handle `ErrInsufficientBonusPool` the same way the slash hook does — forfeit the bonus silently and continue with the delegation transfer and position deletion:

```go
posState, baseRewards, bonusRewards, err := k.claimRewards(ctx, posState)
if err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        // Forfeit bonus to avoid blocking the upgrade, matching the
        // slash-hook pattern in slash.go.
        logger.Error("force-exit: insufficient bonus pool, forfeiting bonus",
            "position_id", posID, "error", err)
        // Re-fetch posState without bonus claim
        posState, err = k.getPositionState(ctx, posID)
        if err != nil {
            return fmt.Errorf("re-get position %d after pool forfeit: %w", posID, err)
        }
    } else {
        return fmt.Errorf("claim rewards for position %d: %w", posID, err)
    }
}
```

Alternatively, pre-fund the rewards pool as part of the upgrade handler before calling `RunMigrations`, or add a pre-upgrade check that verifies the pool has sufficient balance to cover all vesting-owned positions' accrued bonus.

### Proof of Concept

1. Before the v8 upgrade, a user with a `PermanentLockedAccount` creates a tier position via `MsgLockTier` (permitted on v7).
2. The position accrues bonus rewards over time (e.g., 30+ days at any non-zero `BonusApy`).
3. The `BeginBlocker` drains the rewards pool to zero (or governance never replenishes it).
4. The v8 upgrade height is reached.
5. `registerV8UpgradeHandler` fires → `RunMigrations` → `Migrate1to2` → `exitVestedAccountsPositions` → `ForceFullExitWithDelegation(posID)` → `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool`.
6. The error propagates to the upgrade handler, which returns it to `BaseApp`.
7. The chain panics / halts at the upgrade height. [8](#0-7) [9](#0-8) [4](#0-3)

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
