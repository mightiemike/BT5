### Title
SwapAllowlistExtension Bypass via Router — Allowlisted Router Grants Unrestricted Swap Access to All Users - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument forwarded by the pool, which equals `msg.sender` of the `pool.swap()` call. When swaps are routed through `MetricOmmSimpleRouter`, `sender` is the router address, not the actual end-user. If the pool admin allowlists the router (required for any router-mediated swap to succeed), every user on the network can bypass the per-user allowlist by routing through the router.

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs the following check:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension caller) and `sender` is the first argument forwarded by the pool. The pool always sets `sender = msg.sender` of the `pool.swap()` call:

```solidity
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    ...
);
``` [2](#0-1) 

`ExtensionCalling._beforeSwap` then ABI-encodes that `sender` and dispatches it verbatim to every extension in `BEFORE_SWAP_ORDER`:

```solidity
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, zeroForOne, amountSpecified, priceLimitX64,
     packedSlot0Initial, bidPriceX64, askPriceX64, extensionData)
)
``` [3](#0-2) 

**Direct swap path** (`user → pool.swap()`):
- Extension receives `sender = user`
- Check: `allowedSwapper[pool][user]` — correctly gates the actual user.

**Router-mediated swap path** (`user → router.exactX() → pool.swap()`):
- `msg.sender` of `pool.swap()` is the **router**, not the user
- Extension receives `sender = router`
- Check: `allowedSwapper[pool][router]`

For router-mediated swaps to work at all on an allowlisted pool, the pool admin must allowlist the router. Once `allowedSwapper[pool][router] = true`, the check passes for **every** user who routes through the router, regardless of whether that user is individually allowlisted or explicitly blocked.

The analog to the external report is exact: just as the Arbitrum validator's cumulative timer starts at the level-zero block edge and omits the top-level assertion's unrivaled time (the "root" contribution is never accumulated), the allowlist guard starts its identity check at the immediate caller of `pool.swap()` and omits the actual end-user's identity (the "root" actor is never checked). In both cases, the missing top-level term causes the guard to undercount the relevant input, allowing an adversary to exploit the gap.

### Impact Explanation

Any user can bypass the swap allowlist on any pool that has the router allowlisted. The allowlist is a core access-control feature; its failure allows unauthorized addresses to trade against the pool. LPs who deployed the allowlist to protect against specific counterparties (e.g., known toxic-flow addresses, non-KYC'd users in a permissioned pool) receive no protection. Unauthorized swaps drain LP value through spread and notional fees paid to the wrong counterparties and through adverse selection that the allowlist was designed to prevent.

### Likelihood Explanation

High. The bypass requires only that the pool admin has allowlisted the router — a necessary step for any router-mediated swap to function on an allowlisted pool. The attacker needs no special privilege: calling `router.exactX(...)` is a public, permissionless action available to any address. No admin cooperation, no malicious setup, and no non-standard token behavior is required.

### Recommendation

The extension must verify the **original end-user** identity, not the immediate caller of `pool.swap()`. Two viable approaches:

1. **Extension-data forwarding**: Require the router to encode the original `msg.sender` into `extensionData`; the extension decodes and checks that address. The pool admin must configure the router to always include this field.
2. **Separate originator field**: Add an `originator` parameter to the `beforeSwap` interface that the pool populates from a transient-storage context set by the router before calling `pool.swap()`, so the extension can check the true initiator.

Until fixed, pools that need per-user swap gating should not allowlist the router; they must require users to call `pool.swap()` directly.

### Proof of Concept

```
1. Pool admin deploys pool with SwapAllowlistExtension as BEFORE_SWAP extension.
2. Pool admin calls:
       extension.setAllowedToSwap(pool, router, true)   // router allowlisted
       // attacker is NOT allowlisted
3. attacker calls router.exactX*(pool, ...) — router calls pool.swap(msg.sender=router).
4. pool calls extension.beforeSwap(sender=router, ...).
5. Extension evaluates: allowedSwapper[pool][router] == true → passes.
6. Swap executes. attacker has bypassed the allowlist with zero privilege.

Direct call check (for comparison):
   attacker calls pool.swap() directly.
   Extension evaluates: allowedSwapper[pool][attacker] == false → reverts NotAllowedToSwap.
``` [4](#0-3) [5](#0-4) [6](#0-5)
