### Title
Governance Can Retroactively Extend `exit_duration` for Existing Tier Positions, Locking User Funds Indefinitely - (File: `x/tieredrewards/keeper/msg_server_auth.go`)

---

### Summary

`MsgUpdateTier` allows governance to change a tier's `exit_duration` at any time with no protection for existing position holders. Unlike `BonusApy` changes — which trigger a full reward settlement before the new rate takes effect — `exit_duration` changes are applied immediately and silently. Any user who has not yet called `MsgTriggerExitFromTier` will have their exit lock computed against the new, potentially much longer, duration. Governance can extend `exit_duration` from 1 year to an arbitrarily large value (e.g., 100 years), effectively freezing all unlocked positions in the tier indefinitely.

---

### Finding Description

In `UpdateTier`, the code explicitly guards against `BonusApy` changes by settling all tier positions at the old rate before storing the new value:

```go
if !oldTier.BonusApy.Equal(msg.Tier.BonusApy) {
    if err := ms.claimRewardsAndUpdateTierPositions(ctx, msg.Tier.Id); err != nil {
        return nil, err
    }
}
if err := ms.SetTier(ctx, msg.Tier); err != nil { ... }
```

There is **no equivalent guard for `exit_duration`**. The new duration is stored immediately and unconditionally. [1](#0-0) 

When a user later calls `MsgTriggerExitFromTier`, the handler reads the **current** tier state to compute `ExitUnlockAt`:

```go
tier, err := ms.getTier(ctx, pos.TierId)
...
pos.TriggerExit(sdkCtx.BlockTime(), tier.ExitDuration)
``` [2](#0-1) 

`TriggerExit` stores `ExitUnlockAt = blockTime + tier.ExitDuration` directly on the position:

```go
func (p *Position) TriggerExit(blockTime time.Time, duration time.Duration) {
    p.ExitTriggeredAt = blockTime
    p.ExitUnlockAt = blockTime.Add(duration)
}
``` [3](#0-2) 

The `exit_duration` is **not snapshotted at position creation time**. The `Position` struct stores only `ExitTriggeredAt` and `ExitUnlockAt` (set at trigger time), not the duration that was in effect when the user entered the tier. [4](#0-3) 

The `Tier` struct stores `ExitDuration` as a mutable field managed entirely by governance: [5](#0-4) 

---

### Impact Explanation

A user who locks tokens in a tier advertised with `exit_duration = 1 year` has no guarantee that this duration will still apply when they call `TriggerExitFromTier`. If governance passes `MsgUpdateTier` changing `exit_duration` to 100 years before the user triggers exit, the user's `ExitUnlockAt` will be set to `now + 100 years`. The user cannot undelegate (`MsgTierUndelegate` requires `exit elapsed`) or exit with delegation (`MsgExitTierWithDelegation` requires `exit elapsed`) until that timestamp passes.

The user's only recourse is `MsgClearPosition` (cancel exit), but this merely resets the exit timestamps — the user would need to trigger exit again, which would again use the new 100-year duration. The user's staked CRO is effectively frozen for an arbitrarily long period.

The corrupted invariant is: **`position.ExitUnlockAt` no longer reflects the `exit_duration` that was in effect when the user entered the tier**, breaking the user's reasonable expectation about when their funds become accessible. [6](#0-5) 

---

### Likelihood Explanation

The entry path is a standard governance proposal (`MsgUpdateTier`) submitted and passed through `x/gov`. This is a supported production entrypoint reachable by any token holder who can accumulate sufficient voting power or persuade validators. The `exit_duration` field is explicitly part of the `Tier` struct that `MsgUpdateTier` replaces wholesale, and there is no on-chain validation preventing a very large value. The integration tests themselves demonstrate that `exit_duration` changes take effect immediately for future `TriggerExitFromTier` calls. [7](#0-6) 

---

### Recommendation

Snapshot `exit_duration` at position creation time and store it on the `Position` record (e.g., a new `exit_duration` field). `TriggerExitFromTier` should use the position's stored duration rather than reading the current tier state. Alternatively, `UpdateTier` should iterate all existing positions in the tier and, for any position that has not yet triggered exit, record the original duration so it is honoured at trigger time. The `BonusApy` guard pattern already present in `UpdateTier` is the correct model to follow. [8](#0-7) 

---

### Proof of Concept

1. Governance creates Tier 1 with `exit_duration = 1 year`, `bonus_apy = 4%`.
2. Alice calls `MsgLockTier(tier_id=1, amount=1_000_000, validator=V)`. Her position is created; she expects to wait 1 year after triggering exit.
3. Governance passes `MsgUpdateTier` setting `exit_duration = 100 years` (BonusApy unchanged, so no reward settlement is triggered).
4. Alice calls `MsgTriggerExitFromTier(position_id=Alice's)`. The handler reads `tier.ExitDuration = 100 years` and sets `ExitUnlockAt = now + 100 years`.
5. Alice attempts `MsgTierUndelegate` — rejected: `exit elapsed` check fails (`block_time < ExitUnlockAt`).
6. Alice attempts `MsgExitTierWithDelegation` — rejected for the same reason.
7. Alice's 1,000,000 basecro delegation is locked for 100 years with no recourse.

The root cause is the absence of any guard in `UpdateTier` for `exit_duration` changes, in direct contrast to the existing `BonusApy` guard. [9](#0-8)

func (ms msgServer) UpdateTier(ctx context.Context, msg *types.MsgUpdateTier) (*types.MsgUpdateTierResponse, error) {
	if err := ms.requireAuthority(msg.Authority); err != nil {
		return nil, err
	}

	oldTier, err := ms.getTier(ctx, msg.Tier.Id)
	if err != nil {
		return nil, err
	}

	if !oldTier.BonusApy.Equal(msg.Tier.BonusApy) {
		if err := ms.claimRewardsAndUpdateTierPositions(ctx, msg.Tier.Id); err != nil {
			return nil, err
		}
	}

	if err := ms.SetTier(ctx, msg.Tier); err != nil {
		return nil, err
	}

	if err := ms.emitTierChangedEvent(ctx, types.TierChangeAction_TIER_CHANGE_ACTION_UPDATE, msg.Tier); err != nil {
		return nil, err
	}

	return &types.MsgUpdateTierResponse{}, nil
}

```markdown
### Exit (two paths after exit commitment)

1. **Trigger exit** -- `MsgTriggerExitFromTier`. Sets `ExitTriggeredAt` and `ExitUnlockAt`. Bonus continues accruing until `ExitUnlockAt`.

After exit commitment elapses (`block_time >= ExitUnlockAt`), the user chooses one of two paths:

2a. **Exit with delegation (instant)** -- `MsgExitTierWithDelegation(position_id, amount)`. Transfers the module's delegation back to the owner on the same validator (Unbond + Delegate, no unbonding queue). Claims rewards first. Supports partial exits: only the specified amount is transferred; position is reduced accordingly but position size needs to still be at least min lock amount. If the full amount is transferred, the position is deleted. Requires the validator to be bonded.

2b. **Undelegate + Withdraw (21-day wait)** -- `MsgTierUndelegate` claims rewards and starts the staking unbonding period. After unbonding completes, `MsgWithdrawFromTier` transfers `Amount` to the owner and deletes the position.

**Cancel exit** -- `MsgClearPosition` can cancel exit at any point during or after the exit commitment, as long as the position is delegated. Settles rewards first, then resets exit timestamps. This allows the user to re-lock the position and resume earning bonus rewards.

### CloseOnly Tiers

When a tier is marked `CloseOnly` by governance, these messages are **blocked**: LockTier, CommitDelegation, AddToPosition, TierRedelegate, ClearPosition. **Allowed**: TriggerExit, TierUndelegate, WithdrawFromTier, ClaimTierRewards, ExitTierWithDelegation. This lets governance sunset a tier while existing positions close out.

def test_update_tier_non_apy_no_claim(cluster):
    """Changing exit_duration without changing APY does not trigger rewards claiming.

    Verified by checking that the owner balance is unchanged.
    """
    owner = cluster.address("signer1")

    # Confirm the signer1 position on Tier 3 exists
    result = query_command(cluster, MODULE, "positions-by-owner", owner)
    tier3_pos = next(
        p for p in result.get("positions", []) if int(p["tier_id"]) == TIER_3_ID
    )
    assert tier3_pos is not None
    balance_before = cluster.balance(owner, DENOM)

    rsp = submit_gov_proposal(
        cluster,
        "community",
        MSG_UPDATE_TIER,
        {
            "tier": {
                "id": TIER_3_ID,
                "exit_duration": "10s",
                "bonus_apy": "0.030000000000000000",
                "min_lock_amount": "2000000",
                "close_only": False,
            }
        },
        title="Update Tier 3 exit_duration",
        summary="Only exit_duration changes, no APY change",
    )
    approve_proposal(cluster, rsp, msg=f",{MSG_UPDATE_TIER}")

    # No rewards should have been paid out to the owner.
    assert (
        cluster.balance(owner, DENOM) == balance_before
    ), "owner balance should not change when APY is unchanged"

    # Verify exit_duration is updated
    result = query_tiers(cluster)
    tier3 = next(
        (t for t in result.get("tiers", []) if int(t["id"]) == TIER_3_ID), None
    )
    assert tier3["exit_duration"] == "10s"

    # Verify the new exit_duration applies to new positions
    owner = cluster.address("signer2")
    validator = get_validator_addr(cluster, 0)

    before = before_ids(cluster, owner)
    rsp = lock_tier(cluster, owner, TIER_3_ID, 2_000_000, validator=validator)
    assert rsp["code"] == 0, rsp["raw_log"]
    pos_id = new_pos_id(cluster, owner, before)

    rsp = trigger_exit(cluster, owner, pos_id)
    assert rsp["code"] == 0, rsp["raw_log"]

    pos = query_position(cluster, pos_id)["position"]
    triggered_at = isoparse(pos["exit_triggered_at"])
    unlock_at = isoparse(pos["exit_unlock_at"])
    actual_duration = (unlock_at - triggered_at).total_seconds()

    assert actual_duration == 10, (
        f"new position should use updated exit_duration (10s), "
        f"got {actual_duration}s"
    )
