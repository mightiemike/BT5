### Title
`ValidResourceBounds` Protobuf Deserialization Silently Collapses `AllResources` to `L1Gas`, Producing a Different Transaction Hash on Synced Nodes — (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` silently converts an `AllResources` variant with zero `l2_gas` and zero `l1_data_gas` into the `L1Gas` variant. Because `get_tip_resource_bounds_hash` hashes a different number of resource elements for each variant (2 for `L1Gas`, 3 for `AllResources`), the transaction hash computed on a syncing node diverges from the hash computed at the originating gateway. Any V3 transaction submitted with only `l1_gas` bounds set (a valid and common pattern) is affected.

### Finding Description

**Step 1 – Originating node (gateway/batcher)**

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds`. When the transaction converter calls `InternalRpcInvokeTransactionV3::resource_bounds()`, it always returns `ValidResourceBounds::AllResources(self.resource_bounds)`. [1](#0-0) 

`get_invoke_transaction_v3_hash` then calls `get_tip_resource_bounds_hash` with this `AllResources` value. [2](#0-1) 

For `AllResources`, `get_tip_resource_bounds_hash` builds a 3-element resource vector: `[L1_GAS_concat, L2_GAS_concat, L1_DATA_GAS_concat]` and hashes it with the tip. [3](#0-2) 

For a transaction with `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`, the hash is:
```
H1 = poseidon(tip, concat(L1_GAS, X), concat(L2_GAS, 0), concat(L1_DATA, 0))
```

**Step 2 – Protobuf serialization**

`From<ValidResourceBounds> for protobuf::ResourceBounds` faithfully serializes all three fields (l1_gas=X, l2_gas=0, l1_data_gas=0). [4](#0-3) 

**Step 3 – Protobuf deserialization on syncing node (the bug)**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` checks: if `l1_data_gas.is_zero() && l2_gas.is_zero()`, it returns `ValidResourceBounds::L1Gas(l1_gas)` instead of `AllResources`. [5](#0-4) 

The `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is silently collapsed to `L1Gas(X)`.

**Step 4 – Hash recomputation on syncing node**

For `L1Gas`, `get_tip_resource_bounds_hash` builds only a 2-element resource vector: `[L1_GAS_concat, L2_GAS_concat(0)]` — the `L1_DATA_GAS_concat(0)` element is absent. [6](#0-5) 

The recomputed hash is:
```
H2 = poseidon(tip, concat(L1_GAS, X), concat(L2_GAS, 0))
```

`H1 ≠ H2`. The `l1_data_gas` zero-element is "unused" in the `L1Gas` path — exactly analogous to V3Proxy's unused `amountInMax - amounts[0]` being silently dropped.

**Step 5 – Secondary divergence: gas vector computation mode**

Beyond the hash, the deserialized `L1Gas` variant returns `GasVectorComputationMode::NoL2Gas` from `get_gas_vector_computation_mode()`, while the original `AllResources` returns `GasVectorComputationMode::All`. This changes fee estimation, pre-validation bounds checks, and execution resource accounting on the syncing node. [7](#0-6) 

**Trigger condition**

A V3 transaction with `AllResourceBounds { l1_gas: X (non-zero), l2_gas: 0, l1_data_gas: 0 }` is explicitly accepted by the stateless validator (test cases confirm `l1_gas`-only bounds are valid). [8](#0-7) 

The `DeprecatedResourceBoundsMapping` JSON deserializer has the same collapse logic, affecting any node that reads such a transaction from storage or JSON. [9](#0-8) 

### Impact Explanation

A syncing node stores `InvokeTransactionV3` with `resource_bounds = L1Gas(X)` instead of `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`. Consequently:

1. **Wrong transaction hash**: Any RPC call that returns or verifies the transaction hash (`starknet_getTransactionByHash`, `starknet_getTransactionReceipt`, trace APIs) returns H2 instead of H1. If the node also calls `validate_transaction_hash`, valid transactions are rejected.
2. **Wrong fee/gas accounting**: The `NoL2Gas` computation mode is used instead of `All`, causing incorrect fee estimation and resource bound checks on the syncing node.
3. **Wrong state/receipt**: Execution replayed on the syncing node uses different resource bounds semantics, potentially producing divergent receipts.

This matches: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value"* (High) and potentially *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions"* (High).

### Likelihood Explanation

Any user submitting a V3 transaction with only `l1_gas` bounds set (the most common pattern for pre-0.13.3 style transactions submitted via the V3 API) triggers this. No special privileges are required. The condition `l2_gas.is_zero() && l1_data_gas.is_zero()` is the default for many wallets and SDKs that have not yet adopted full three-resource bounds.

### Recommendation

Remove the `L1Gas`/`AllResources` discriminant logic from the protobuf deserializer. Since the protobuf wire format already carries all three fields, always deserialize to `AllResources` when all three are present. The `L1Gas` variant should only be constructed from pre-0.13.3 wire formats that genuinely lack the `l1_data_gas` field (i.e., when `l1_data_gas` is absent/`None`, not when it is present but zero):

```rust
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let l1_gas = value.l1_gas.ok_or(missing("ResourceBounds::l1_gas"))?.try_into()?;
        let l2_gas = value.l2_gas.ok_or(missing("ResourceBounds::l2_gas"))?.try_into()?;
        // Only use L1Gas variant when l1_data_gas is genuinely absent (old wire format).
        Ok(match value.l1_data_gas {
            None if l2_gas.is_zero() => ValidResourceBounds::L1Gas(l1_gas),
            l1_data_gas_opt => {
                let l1_data_gas = l1_data_gas_opt.unwrap_or_default().try_into()?;
                ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
            }
        })
    }
}
```

The same fix applies to `TryFrom<DeprecatedResourceBoundsMapping> for ValidResourceBounds` and `From<ResourceBoundsMapping> for ValidResourceBounds` in `crates/apollo_rpc/src/v0_8/transaction.rs`.

### Proof of Concept

```rust
#[test]
fn test_allresources_zero_l2_l1data_hash_mismatch_across_protobuf() {
    use starknet_api::transaction::fields::{
        AllResourceBounds, ResourceBounds, ValidResourceBounds,
    };
    use starknet_api::execution_resources::GasAmount;
    use starknet_api::block::GasPrice;
    use apollo_protobuf::protobuf;
    use std::convert::TryInto;

    // A V3 transaction with only l1_gas set (l2_gas=0, l1_data_gas=0).
    let original = ValidResourceBounds::AllResources(AllResourceBounds {
        l1_gas: ResourceBounds {
            max_amount: GasAmount(1000),
            max_price_per_unit: GasPrice(100),
        },
        l2_gas: ResourceBounds::default(),    // zero
        l1_data_gas: ResourceBounds::default(), // zero
    });

    // Serialize to protobuf (as done during block sync).
    let proto: protobuf::ResourceBounds = original.into();

    // Deserialize on the receiving node.
    let deserialized: ValidResourceBounds = proto.try_into().unwrap();

    // BUG: deserialized is L1Gas, not AllResources.
    assert!(
        matches!(deserialized, ValidResourceBounds::L1Gas(_)),
        "Deserialized as L1Gas instead of AllResources"
    );

    // Hash divergence: AllResources hashes 3 resource elements, L1Gas hashes 2.
    use starknet_api::transaction_hash::get_tip_resource_bounds_hash;
    use starknet_api::transaction::fields::Tip;
    let tip = &Tip(0);
    let h1 = get_tip_resource_bounds_hash(&original, tip).unwrap();
    let h2 = get_tip_resource_bounds_hash(&deserialized, tip).unwrap();
    assert_ne!(h1, h2, "Hash must differ: AllResources(3 elements) vs L1Gas(2 elements)");
}
```
