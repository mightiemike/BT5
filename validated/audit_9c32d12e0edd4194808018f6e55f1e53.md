### Title
Equihash Solution Validity Return Value Silently Discarded — Block Accepted Despite Invalid PoW - (File: `contract/src/zcash.rs`)

### Summary

In the Zcash build of the BTC light client, `equihash::is_valid_solution` returns `Result<bool, Error>`. The contract handles only the `Err` variant (by panicking) but silently discards the inner `bool`. An `Ok(false)` result — meaning the solution is structurally parseable but cryptographically invalid — causes no panic and no rejection. The block is accepted and written into the canonical mainchain state.

### Finding Description

In `contract/src/zcash.rs`, inside `check_pow`, the Equihash solution is verified as follows:

```rust
equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
``` [1](#0-0) 

`equihash::is_valid_solution` has the signature `fn is_valid_solution(...) -> Result<bool, Error>`. The `.unwrap_or_else(|e| ...)` call converts the `Result` into its inner `bool` by panicking on `Err`. However, the resulting `bool` is **never bound, tested, or asserted**. It is immediately dropped.

- `Ok(true)` → solution is valid → dropped silently (correct outcome, but by accident)
- `Ok(false)` → solution is **invalid** → dropped silently, **no panic, no rejection**
- `Err(e)` → parse/structural error → panics (the only case that is handled)

The `check_pow` function is called unconditionally when `skip_pow_verification` is `false` (the production setting), from `check_target` in `lib.rs`:

```rust
fn check_target(&self, block_header: &Header, prev_block_header: &ExtendedHeader) {
    self.check_pow(block_header, prev_block_header);
}
``` [2](#0-1) 

After `check_pow` returns without panicking, `submit_block_header_inner` is called, which writes the header into `headers_pool`, updates `mainchain_tip_blockhash`, and updates both lookup maps:

```rust
self.store_block_header(&current_header);
self.mainchain_tip_blockhash = current_header.block_hash;
``` [3](#0-2) 

### Impact Explanation

A Zcash block header with a structurally valid but cryptographically invalid Equihash solution (one that parses without error but does not satisfy the Equihash constraints) will pass all checks and be permanently written into the contract's canonical chain state. This corrupts:

- `mainchain_tip_blockhash` — the contract's view of the chain tip
- `mainchain_height_to_header` / `mainchain_header_to_height` — the height-to-hash index
- `headers_pool` — the full header store

Any downstream consumer calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` against a block accepted via this path will receive a forged proof result, since the block's presence in the mainchain is the sole authorization check. [4](#0-3) 

### Likelihood Explanation

The entry point is `submit_blocks`, which is callable by any trusted relayer (an unprivileged NEAR account that has staked). The `trusted_relayer` macro restricts callers to registered relayers, but registration is open and permissionless by design (relayers stake to join). A malicious relayer can craft a Zcash header with a valid structure but an invalid Equihash solution and submit it directly. No privileged access, leaked keys, or social engineering is required. [5](#0-4) 

### Recommendation

Assert the boolean return value of `equihash::is_valid_solution`:

```rust
let is_valid = equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
require!(is_valid, "Invalid Equihash solution: solution does not satisfy constraints");
```

This mirrors the pattern used for every other PoW check in the codebase, where the result of the validation is explicitly tested with `require!` before proceeding.

### Proof of Concept

1. Register as a trusted relayer on the Zcash deployment of the contract.
2. Construct a Zcash block header where:
   - All fields (`version`, `prev_block_hash`, `bits`, `time`) are valid and pass all other checks.
   - `nonce` and `solution` are set to values that are structurally valid (correct byte lengths for Equihash 200/9) but do not satisfy the Equihash proof-of-work constraints — causing `equihash::is_valid_solution` to return `Ok(false)`.
3. Call `submit_blocks` with this header.
4. Observe that the call succeeds and the header is stored as the new `mainchain_tip_blockhash`.
5. Call `verify_transaction_inclusion` with a fabricated Merkle proof against this block; the contract will return `true` for a transaction that was never included in any real Zcash block. [6](#0-5)

    #[pause]
    #[trusted_relayer]
    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
```

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
