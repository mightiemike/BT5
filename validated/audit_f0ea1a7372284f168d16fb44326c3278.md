Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of End User, Enabling Allowlist Bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the `pool.swap()` call. When `MetricOmmSimpleRouter` is used, `sender` resolves to the router's address rather than the actual end user. Any pool admin who allowlists the router — a natural and necessary step to support router-mediated swaps — inadvertently grants every unprivileged address the ability to bypass the per-user allowlist entirely.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension hook.**

In `MetricOmmPool.swap`, `msg.sender` is forwarded verbatim as the `sender` argument to `_beforeSwap`: [1](#0-0) 

**Step 2 — Extension checks `sender`, which equals the router when routing through `MetricOmmSimpleRouter`.**

`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [2](#0-1) 

**Step 3 — The router calls `pool.swap()` as itself, not as the end user.**

`MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(...)` directly. The pool's `msg.sender` is therefore the router contract address: [3](#0-2) 

**Result:** The extension evaluates `allowedSwapper[pool][router_address]`. If the router is allowlisted, the check passes for every caller regardless of whether they are individually allowlisted. The extension has no mechanism to inspect the original `msg.sender` stored in the router's transient callback context (`_getPayer()` / `_setNextCallbackContext`), so the actual end user identity is permanently invisible to the hook.

## Impact Explanation
This is an admin-boundary break. A pool admin deploying a curated pool with `SwapAllowlistExtension` to restrict trading to a known set of addresses will also need to allowlist the router to support standard periphery usage. Once the router is allowlisted, the per-user allowlist is completely inoperative for all router-mediated paths: any unprivileged address can call `router.exactInputSingle(...)` and execute swaps on the curated pool. The core access-control invariant the extension is designed to enforce — that only explicitly approved addresses may swap — is broken for the most common swap path.

## Likelihood Explanation
The scenario is highly realistic. Any pool admin who (a) deploys a pool with `SwapAllowlistExtension` and (b) wants to support the standard `MetricOmmSimpleRouter` must allowlist the router. This is not a misconfiguration — it is the only way to enable router-mediated swaps. The bypass requires no special privileges: any EOA or contract can call `router.exactInputSingle(...)`. The attack is repeatable and requires no setup beyond the router being allowlisted.

## Recommendation
The extension must verify the actual end user, not the intermediary. Two viable approaches:

1. **Pass the original caller through `extensionData`**: Require the router to encode `msg.sender` into `extensionData` and have the extension decode and verify it. This requires a convention between router and extension.
2. **Expose a `recipient`-aware check**: Use the `recipient` parameter (already passed to `beforeSwap`) as a proxy, or add a dedicated `originalSender` field to the extension call interface.
3. **Disallow router allowlisting explicitly**: Document that the router address must never be added to `allowedSwapper` and enforce this in `setAllowedToSwap` by checking against a registry of known routers.

The cleanest fix is option 1: the router encodes `msg.sender` into `extensionData`, and `SwapAllowlistExtension` decodes and checks it when the caller is a known router.

## Proof of Concept

```solidity
// 1. Deploy pool with SwapAllowlistExtension; allowlist only `alice`.
// 2. Pool admin also calls setAllowedToSwap(pool, router, true) to enable router usage.
// 3. `bob` (not allowlisted) calls:
//    router.exactInputSingle(ExactInputSingleParams({
//        pool: pool, recipient: bob, zeroForOne: true,
//        amountIn: 1e18, amountOutMinimum: 0, ...
//    }));
// 4. pool.swap() is called with msg.sender == router.
// 5. SwapAllowlistExtension checks allowedSwapper[pool][router] == true → passes.
// 6. bob successfully swaps on the curated pool despite not being allowlisted.
```
