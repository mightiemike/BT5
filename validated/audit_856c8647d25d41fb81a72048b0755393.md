**Step 4: Invariant broken**

`availableSettle` represents the pool's accumulated realized gains available for distribution to profitable subaccounts. It must only decrease when PnL is paid out. After this call with negative `positionPnl`, it increases — phantom settle capacity is created from nothing.

### Title
`availableSettle` inflated when settling negative-PnL subaccount — (`core/contracts/PerpEngine.sol`)

### Summary
`PerpEngine.settlePnl` uses `MathHelper.min(positionPnl, state.availableSettle)` without clamping to zero. When `positionPnl < 0` and `state.availableSettle > 0`, `canSettle` is negative, and `state.availableSettle -= canSettle` increases `availableSettle`, creating phantom settlement capacity.

### Finding Description
In `PerpEngine.settlePnl` (lines 77–104), `canSettle` is computed as `min(positionPnl, state.availableSettle)`. When a subaccount has negative unrealized PnL and the pool has positive `availableSettle`, `canSettle` is negative. The subsequent `state.availableSettle -= canSettle` subtracts a negative value, increasing `availableSettle`. Simultaneously, `balance.vQuoteBalance -= canSettle` improves the subaccount's perp balance, while `Clearinghouse._settlePnl` debits the same amount from the subaccount's spot quote — a net-zero wash for the subaccount but a net-positive inflation of the pool's settlement capacity.

### Impact Explanation
`availableSettle` is inflated beyond the pool's actual earned quote. Subsequent legitimate settlers with positive PnL can extract more quote than the pool actually accumulated, draining the perp engine's quote reserves and breaking solvency.

### Likelihood Explanation
The sequencer calls `settlePnl` for subaccounts routinely (e.g., before or during liquidation). Any subaccount with negative unrealized PnL and a product with positive `availableSettle` triggers the bug. No special attacker privileges are required beyond submitting a `SettlePnl` slow-mode transaction (if that path is open) or waiting for the sequencer to do so naturally.

### Recommendation
Clamp `canSettle` to non-negative before applying mutations:

```solidity
if (canSettle <= 0) {
    productIds >>= 32;
    continue;
}
```

Or change `getSettlementState` to return `max(0, min(positionPnl, state.availableSettle))`.

### Proof of Concept
State-machine test:
1. Set `state.availableSettle = 50e18`, subaccount `positionPnl = -100e18`.
2. Call `settlePnl` for that subaccount.
3. Assert `state.availableSettle_after == 150e18` (inflated by 100e18).
4. Assert a second subaccount with `positionPnl = +150e18` can now settle the full 150e18, extracting 100e18 more than the pool earned. [6](#0-5) [7](#0-6)

        while (productIds != 0) {
            uint32 productId = uint32(productIds & ((1 << 32) - 1));
            // otherwise it means the product is a spot.

                    State memory state,
                    Balance memory balance

                balance.vQuoteBalance -= canSettle;

                totalSettled += canSettle;

                _setState(productId, state);
                _setBalanceAndUpdateBitmap(productId, subaccount, balance);
            }
