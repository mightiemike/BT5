### Title
Deprecated `verify_transaction_inclusion` Accepts Internal Merkle-Tree Nodes as Valid Transaction IDs, Enabling Proof-Verification Forgery — (`File: contract/src/lib.rs`)

### Summary

The deprecated `verify_transaction_inclusion` function remains callable by any unprivileged NEAR account. It performs no validation that the supplied `tx_id` is a leaf node (a real transaction hash) rather than an internal node of the Merkle tree. An attacker can craft a `(tx_id, tx_index, merkle_proof)` triple where `tx_id` is an internal Merkle node, causing the function to return `true` for a transaction that does not exist in the block. Any downstream contract or off-chain system that consumes this result as a trusted proof of Bitcoin transaction inclusion is deceived.

### Finding Description

`verify_transaction_inclusion` is still a live, publicly callable NEAR contract method. Its only guard is `#[pause]`, which is inactive under normal operation. The function's own inline documentation acknowledges the flaw:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [1](#0-0) 

The verification logic in `compute_root_from_merkle_proof` is purely positional: it hashes the supplied `tx_id` with each sibling in `merkle_proof` according to `tx_index` parity and checks whether the result equals `header.block_header.merkle_root`. [2](#0-1) 

Because Bitcoin's Merkle tree is a binary tree of 32-byte nodes, any internal node `N` at depth `d` already satisfies `compute_root_from_merkle_proof(N, idx, proof) == merkle_root` when `proof` contains the `d` siblings on the path from `N` to the root. The function cannot distinguish this from a genuine leaf-level proof. [3](#0-2) 

The `#[deprecated]` Rust attribute emits only a compile-time warning; it does not remove the method from the deployed WASM binary or restrict callers at runtime. [4](#0-3) 

The v2 counterpart fixes this by requiring a coinbase Merkle proof (index 0), which anchors the tree and makes internal-node substitution infeasible. The v1 path has no such anchor. [5](#0-4) 

### Impact Explanation

Any NEAR contract or off-chain bridge that calls `verify_transaction_inclusion` and acts on a `true` return value — e.g., to release funds, mint tokens, or record a Bitcoin payment — can be deceived into accepting a fabricated proof. The attacker does not need to mine a block or control any privileged role; they only need a publicly observable Bitcoin block and the ability to send a NEAR transaction.

### Likelihood Explanation

The attack requires only:
1. A publicly available Bitcoin block (Merkle tree structure is fully public).
2. The ability to call a NEAR contract method (no staking, no role, no deposit beyond gas).
3. Knowledge of the 64-byte transaction Merkle proof forgery technique (publicly documented since 2017 at the Bitmex link cited in the contract's own comments).

The function is deployed and unpaused in production. The barrier is low.

### Recommendation

- **Remove** `verify_transaction_inclusion` from the deployed contract or gate it with an unconditional `env::panic_str("deprecated: use verify_transaction_inclusion_v2")` to prevent any runtime invocation.
- **Do not** rely on `#[deprecated]` as a security boundary; it is a compiler hint only.
- Ensure all consuming contracts and off-chain integrations have migrated to `verify_transaction_inclusion_v2`, which enforces the coinbase-proof anchor. [6](#0-5) 

### Proof of Concept

```
1. Attacker selects any confirmed Bitcoin block B whose header is stored
   in the light client (mainchain_header_to_height contains its hash).

2. From the public Bitcoin blockchain, attacker reads the full Merkle tree
   of block B and picks any internal node N at depth d with siblings
   [s_1, s_2, ..., s_d] on the path to the root.

3. Attacker computes tx_index such that the parity sequence of
   (tx_index >> 0) % 2, (tx_index >> 1) % 2, ...
   matches the left/right position of N at each level.

4. Attacker calls:
     verify_transaction_inclusion({
       tx_id:             N,          // internal node, not a real tx
       tx_block_blockhash: hash(B),
       tx_index:          computed_index,
       merkle_proof:      [s_1, ..., s_d],
       confirmations:     1,
     })

5. compute_root_from_merkle_proof(N, tx_index, proof) == merkle_root(B)
   → function returns true.

6. Any downstream contract that trusted this result now believes a
   non-existent transaction was confirmed in block B.
``` [7](#0-6) [3](#0-2)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
