### Title
Linked Signer Can Forcibly Override Subaccount's Linked Signer Setting - (`core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction type is processed with `allowLinkedSigner = true`, meaning the currently-set linked signer of a subaccount can sign a new `LinkSigner` transaction to replace itself with any arbitrary address. This bypasses the subaccount owner's explicit intent and mirrors the exact vulnerability class from the external report: a party with limited delegated trust can override a user-controlled preference flag through a side-channel operation.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction branch validates the signature with `allowLinkedSigner = true`:

```solidity
// EndpointTx.sol lines 576-590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true   // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which passes the current linked signer to `Verifier.validateSignature`:

```solidity
// EndpointTx.sol lines 172-184
function validateSignature(..., bool allowLinkedSigner) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

`Verifier.validateSignature` accepts the signature if it recovers to either the subaccount owner address **or** the linked signer:

```solidity
// Verifier.sol lines 291-304
function validateSignature(...) public pure {
    address recovered = ECDSA.recover(digest, signature);
    require(
        (recovered != address(0)) &&
            ((recovered == address(uint160(bytes20(sender)))) ||
                (recovered == linkedSigner)),
        ERR_INVALID_SIGNATURE
    );
}
``` [3](#0-2) 

Therefore, the current linked signer can sign a `LinkSigner` transaction to replace itself with any address — including an attacker-controlled one — without the subaccount owner's knowledge or consent.

Contrast this with the slow-mode path for `LinkSigner`, which correctly restricts the operation to the subaccount owner only via `validateSender`:

```solidity
// EndpointTx.sol lines 232-239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    ...
    validateSender(txn.sender, sender);  // msg.sender must be the owner address
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

The fast (sequencer-submitted) path lacks this restriction, creating the asymmetry.

---

### Impact Explanation

Once the linked signer replaces itself with an attacker-controlled address, the attacker can sign any transaction that accepts `allowLinkedSigner = true`, including:

- `WithdrawCollateral` (line 418-424, `allowLinkedSigner = true`) — drains collateral to the subaccount's own address
- `LiquidateSubaccount` (line 397-403, `allowLinkedSigner = true`) — can be used to manipulate positions
- `MintNlp` / `BurnNlp` (lines 539-544, 559-564, `allowLinkedSigner = true`) — manipulates NLP positions [5](#0-4) 

The corrupted state is `linkedSigners[victim_subaccount]` in `EndpointStorage`. The asset delta is the full collateral balance of the victim subaccount, withdrawable via a subsequent `WithdrawCollateral` signed by the attacker's address.

---

### Likelihood Explanation

Any user who has set a linked signer (e.g., a trading bot, an automated strategy, or a hot wallet) is at risk. The attack requires:

1. The linked signer key to be accessible to the attacker (e.g., compromised trading bot, insider, or the linked signer itself acting maliciously).
2. The sequencer to process the malicious `LinkSigner` transaction — which it will, since it is a structurally valid signed transaction.

No special protocol state, governance action, or admin access is required. The nonce is per-address and known to the linked signer since it is actively trading.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only the subaccount owner may change the linked signer
);
``` [6](#0-5) 

This mirrors the correct behavior already present in the slow-mode path, where `validateSender` enforces that only the subaccount owner can submit a `LinkSigner` transaction.

---

### Proof of Concept

1. Alice owns subaccount `alice_subaccount` and sets linked signer to address `B` (a trading bot) via a `LinkSigner` transaction signed by Alice.
2. `linkedSigners[alice_subaccount] = B` is stored on-chain.
3. Attacker (controlling `B`, or `B` acting maliciously) constructs a new `LinkSigner` transaction: `{sender: alice_subaccount, signer: attacker_address, nonce: current_nonce}` and signs it with key `B`.
4. The sequencer receives this transaction. `validateSignedTx` is called with `allowLinkedSigner = true`. `getLinkedSigner(alice_subaccount)` returns `B`. The signature from `B` is valid. The transaction passes.
5. `linkedSigners[alice_subaccount] = attacker_address` is written.
6. Attacker signs a `WithdrawCollateral` transaction for `alice_subaccount` using `attacker_address`. The sequencer processes it. Alice's collateral is withdrawn.

The root cause is in `EndpointTx.sol` at the `LinkSigner` branch of `processTransactionImpl` (lines 576–590), where `allowLinkedSigner = true` permits the existing linked signer to mutate the very state that governs its own authority. [1](#0-0)
