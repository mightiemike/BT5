### Title
Builder Fees from Perp Products Permanently Locked Due to Incomplete Product Iteration in `claimBuilderFee` — (`File: core/contracts/OffchainExchange.sol`)

---

### Summary

`claimBuilderFee` only iterates `spotEngine.getProductIds()` when sweeping accumulated builder fees, omitting `perpEngine.getProductIds()`. Because builder fees are collected for both spot and perp trades via the shared `applyFee` path, any builder fees keyed to a `quoteId` that is not present in the spot engine's product list are permanently locked in the contract and irrecoverable.

---

### Finding Description

Builder fees are accumulated in the mapping `collectedBuilderFee[quoteId][builderId]` inside `applyFee`, which is called for every matched order regardless of whether the product is spot or perp. [1](#0-0) [2](#0-1) 

The `_getCallState` function confirms that both spot and perp products route through the same `matchOrders` → `applyFee` execution path: [3](#0-2) 

When a builder calls `claimBuilderFee`, the function iterates **only** `spotEngine.getProductIds()`: [4](#0-3) 

In contrast, `dumpFees` — which performs the analogous sweep for protocol-collected trading fees — explicitly comments *"loop over all spot and perp product ids"* and iterates both engines: [5](#0-4) 

The `market.quoteId` used as the key in `collectedBuilderFee` is read from `quoteIds[productId]`: [6](#0-5) 

If any perp product has a `quoteId` that is not present in `spotEngine.getProductIds()`, the fees stored at `collectedBuilderFee[perpQuoteId][builderId]` are never iterated, never paid out, and never zeroed — they are permanently locked.

---

### Impact Explanation

Builder fees earned from perp product trades are permanently locked in the `collectedBuilderFee` mapping. Builders cannot recover these funds through any supported contract entrypoint. The asset delta is a direct, unrecoverable loss of earned fee revenue for every builder who routes perp order flow. There is no admin escape hatch or alternative claim path in the contract.

---

### Likelihood Explanation

The trigger is structural and automatic: every perp trade that includes a builder fee silently accumulates locked value. Any deployment where perp products are configured with a `quoteId` not present in the spot engine's product list (e.g., a perp-specific synthetic quote token, or a future product addition) activates the bug. The `dumpFees` function's explicit dual-engine iteration demonstrates that the protocol design intends both engines to be covered; the omission in `claimBuilderFee` is an oversight, not a deliberate choice.

---

### Recommendation

Mirror the `dumpFees` pattern in `claimBuilderFee` by also iterating `perpEngine.getProductIds()` after the spot loop. For perp product IDs, the payout should credit the builder's spot balance in the corresponding quote token (consistent with how perp fees are denominated):

```diff
  function claimBuilderFee(bytes32 sender, uint32 builderId) external onlyEndpoint {
      ...
      uint32[] memory productIds = spotEngine.getProductIds();
      for (uint32 i = 0; i < productIds.length; i++) {
          uint32 productId = productIds[i];
          int128 collectedFee = collectedBuilderFee[productId][builderId];
          if (collectedFee == 0) continue;
          emit ClaimBuilderFee(builderId, productId, sender, collectedFee);
          spotEngine.updateBalance(productId, sender, collectedFee);
          collectedBuilderFee[productId][builderId] = 0;
      }
+
+     productIds = perpEngine.getProductIds();
+     for (uint32 i = 0; i < productIds.length; i++) {
+         uint32 productId = productIds[i];
+         uint32 quoteId = quoteIds[productId];
+         int128 collectedFee = collectedBuilderFee[quoteId][builderId];
+         if (collectedFee == 0) continue;
+         emit ClaimBuilderFee(builderId, quoteId, sender, collectedFee);
+         spotEngine.updateBalance(quoteId, sender, collectedFee);
+         collectedBuilderFee[quoteId][builderId] = 0;
+     }
  }
```

---

### Proof of Concept

1. A builder registers with `builderId = 1`.
2. A perp product (e.g., `productId = 5`) is configured with `quoteIds[5] = 3`, where product ID `3` is **not** in `spotEngine.getProductIds()`.
3. Traders match perp orders on product `5` with a non-zero `builderFeeRate`. Each match calls `applyFee`, which executes `collectedBuilderFee[3][1] += builderFee`.
4. The builder calls `claimBuilderFee`. The function iterates `spotEngine.getProductIds()` — product `3` is absent from this list.
5. `collectedBuilderFee[3][1]` is never read, never paid out, and never zeroed. The accumulated fees are permanently locked.
6. `dumpFees` (the reference-correct function) would have iterated `perpEngine.getProductIds()` and handled product `5`, confirming the omission is unintentional.
