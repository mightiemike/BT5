### Title
`LastBonusAccrual` Checkpoint Advanced Unconditionally When Truncated Bonus Is Zero, Causing Permanent Staker Reward Loss - (File: `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`processEventsAndClaimBonus` in `x/tieredrewards/keeper/claim_rewards.go` unconditionally advances the `LastBonusAccrual` timestamp checkpoint before checking whether the computed bonus truncated to zero. When `computeSegmentBonus` returns zero due to integer truncation (small position size or short time delta), the accrual window is permanently discarded and the staker receives nothing for that period.

---

### Finding Description

`computeSegmentBonus` computes the bonus reward for a time segment using the formula:

```
shares * tokensPerShare * tier.BonusApy * durationSeconds / SecondsPerYear
```

The final step is `.TruncateInt()`, which performs integer truncation on the `math.LegacyDec` result. [1](#0-0) 

For small positions or short time deltas, this product is less than 1 `basecro` and truncates to zero. For example, with a 1 CRO (10^8 basecro) position, 4% `BonusApy`, and a 5-second block interval:

```
10^8 * 0.04 * 5 / 31,536,000 ≈ 0.634 basecro → TruncateInt() = 0
```

The critical flaw is in `processEventsAndClaimBonus`: `applyBonusAccrualCheckpoint` is called **unconditionally at line 215**, before the zero-check at line 219: [2](#0-1) 

`applyBonusAccrualCheckpoint` advances `pos.LastBonusAccrual` to `blockTime`: [3](#0-2) 

Because the checkpoint is advanced before the zero-check, the time window `[old LastBonusAccrual, blockTime]` is permanently consumed. The next call to `processEventsAndClaimBonus` starts accrual from `blockTime`, not from the previous checkpoint. The fractional bonus that was earned but truncated is irrecoverably lost.

This same code path is triggered not only by `MsgClaimTierRewards` but also by every lifecycle settlement call (`claimRewardsAndUpdateTierPositions` → `processEventsAndClaimBonus`), which is invoked during `AddToPosition`, `TierRedelegate`, `ClearPosition`, and `TierUndelegate`: [4](#0-3) 

---

### Impact Explanation

A staker with a position small enough that each block's bonus truncates to zero will **never receive any bonus rewards**, even though they are legitimately entitled to them. The `LastBonusAccrual` checkpoint is advanced on every claim or lifecycle event, permanently discarding the fractional accrual. Over time, the cumulative loss equals the full bonus APY on the position. The bonus pool retains funds that should have been paid to stakers.

For positions where the per-block bonus is non-zero but fractional (e.g., 63.4 basecro truncated to 63), the 0.4 basecro per block is also permanently lost on every claim, compounding over the position's lifetime.

---

### Likelihood Explanation

The `MinLockAmount` for a tier is governance-controlled and can be set to any value. If it is set at or below the threshold where `MinLockAmount * BonusApy * blockTime / SecondsPerYear < 1 basecro`, all minimum-size positions will earn zero bonus on every block-level claim. Additionally, any user who calls `MsgClaimTierRewards` more frequently than once per `SecondsPerYear / (position_tokens * BonusApy)` seconds will trigger the truncation-to-zero path. This is a standard, unprivileged Cosmos SDK transaction requiring no special permissions.

---

### Recommendation

Move `applyBonusAccrualCheckpoint` to execute **only when `totalBonus > 0`**, or alternatively accumulate fractional bonus in a persistent `Dec` field on the position and only truncate when transferring. This mirrors the fix recommended in the original report: do not advance the state checkpoint when the computed reward is zero.

---

### Proof of Concept

1. Governance creates a tier with `BonusApy = 0.04` (4%) and `MinLockAmount = 1_0000_0000` basecro (1 CRO).
2. A staker calls `MsgLockTier` with `Amount = 1_0000_0000` basecro.
3. After 5 seconds (one block), the staker calls `MsgClaimTierRewards`.
4. `computeSegmentBonus` computes: `1e8 * 0.04 * 5 / 31_536_000 ≈ 0.634` → `TruncateInt()` = **0**.
5. `applyBonusAccrualCheckpoint` advances `LastBonusAccrual` to `blockTime` (line 215).
6. `totalBonus.IsZero()` is true (line 219); zero coins returned.
7. The 0.634 basecro of earned bonus is permanently lost. Repeating every block, the staker earns **0 bonus** for the entire duration of their position despite being entitled to 4% APY. [5](#0-4) [6](#0-5)

		return err
	}
	if len(ids) == 0 {
		return nil
	}

		if err != nil {
			return err
		}
		if !pos.IsDelegated() {
			continue
		}

			return err
		}

			return err
		}
		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return err
		}
	}

	return nil
