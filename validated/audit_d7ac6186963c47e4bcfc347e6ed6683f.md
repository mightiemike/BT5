Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of original user, enabling allowlist bypass via `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` parameter, which `MetricOmmPool.swap` always sets to its own `msg.sender` — the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter` is that direct caller, the extension evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][original_user]`, producing two fund-impacting outcomes: allowlisted users are silently blocked from using the standard periphery, and if the admin adds the router to the allowlist as a workaround, every unprivileged user can bypass the per-user allowlist entirely.

## Finding Description

**Root cause — wrong actor bound in `beforeSwap`**

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the first parameter — the direct caller of `pool.swap()`: [3](#0-2) 

**Router path — sender is always the router, never the user**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly; `msg.sender` seen by the pool is the router: [4](#0-3) 

For multi-hop `exactInput`, every hop is called from the router, so `sender = router` for all pools in the path: [5](#0-4) 

For `exactOutput`, intermediate hops are triggered inside `_exactOutputIterateCallback`, which executes on the router contract, so `msg.sender` of each subsequent `pool.swap()` call is again the router: [6](#0-5) 

In every router-mediated path the extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][original_user]`. There is no existing guard that recovers the original user's address before the allowlist check.

## Impact Explanation

**Path A — false block (broken core swap flow):** A pool admin allowlists Alice. Alice calls `router.exactInputSingle(pool, ...)`. The extension checks `allowedSwapper[pool][router]` → `false` → `NotAllowedToSwap`. Alice is blocked from using the standard periphery even though she is explicitly allowlisted. The only workaround is to call `pool.swap()` directly, bypassing the router's slippage protection, deadline enforcement, and multi-hop composition — a broken core swap flow causing loss of usability and potential fund loss from unprotected slippage.

**Path B — allowlist bypass (curation failure):** The pool admin, observing that allowlisted users cannot use the router, adds the router address to the allowlist (`setAllowedToSwap(pool, router, true)`). Now `allowedSwapper[pool][router] = true`. Any unprivileged user — including addresses the admin explicitly never allowlisted — can call `router.exactInputSingle(pool, ...)` and the check passes. The per-user allowlist is completely defeated. On a curated pool where the allowlist is the primary access-control mechanism, this constitutes a complete admin-boundary break reachable by any unprivileged trader.

Both paths meet Sherlock Medium/High thresholds: Path A breaks core swap functionality for legitimate users; Path B allows unauthorized access to a restricted pool.

## Likelihood Explanation

Path A is triggered by any allowlisted user who uses the standard router — the normal, expected interaction pattern. No special attacker capability is required; it is the default failure mode. Path B requires the admin to take the natural workaround action (adding the router to the allowlist), after which any unprivileged address can exploit it with a single `exactInputSingle` call. Both conditions are highly likely in production deployments that use `SwapAllowlistExtension` with `MetricOmmSimpleRouter`.

## Recommendation

Pass the original user's address through the call chain rather than the direct `msg.sender`. Two concrete approaches:

1. **Preferred — thread the originator through `extensionData`:** Have the router encode the original `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check that address instead of `sender`.

2. **Alternative — add an originator parameter to the pool interface:** Extend `pool.swap()` with an explicit `originator` address parameter that the router sets to `msg.sender` before calling the pool, and forward that to extensions instead of the pool-level `msg.sender`.

Either fix must ensure the extension checks the address that initiated the router call, not the router contract itself.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Setup:
// 1. Deploy pool with SwapAllowlistExtension configured.
// 2. Pool admin calls setAllowedToSwap(pool, alice, true).
//    Router is NOT on the allowlist.

// Path A — false block:
// Alice calls router.exactInputSingle({pool: pool, ...}).
// Pool.swap() is called with msg.sender = router.
// Extension checks allowedSwapper[pool][router] → false → revert NotAllowedToSwap.
// Alice cannot swap despite being allowlisted.

// Path B — bypass:
// Admin observes Path A and calls setAllowedToSwap(pool, router, true).
// Mallory (never allowlisted) calls router.exactInputSingle({pool: pool, ...}).
// Pool.swap() is called with msg.sender = router.
// Extension checks allowedSwapper[pool][router] → true → swap succeeds.
// Mallory bypasses the per-user allowlist entirely.

function testAllowlistBypass() public {
    // alice is allowlisted, router is not
    swapAllowlist.setAllowedToSwap(address(pool), alice, true);

    // Path A: alice blocked via router
    vm.prank(alice);
    vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
    router.exactInputSingle(ExactInputSingleParams({pool: address(pool), ...}));

    // Admin adds router as workaround
    swapAllowlist.setAllowedToSwap(address(pool), address(router), true);

    // Path B: mallory (never allowlisted) bypasses via router
    vm.prank(mallory);
    router.exactInputSingle(ExactInputSingleParams({pool: address(pool), ...}));
    // succeeds — allowlist fully bypassed
}
```
