### Title
Silent Error Swallowing in `_executeSlowModeTransaction` Causes Permanent Fund Loss on Deposit Failure - (`File: core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint._executeSlowModeTransaction` wraps `processSlowModeTransaction` in a `try/catch` that silently discards all non-OOG failures. The slow mode queue entry is deleted **before** the `try` block executes. For deposit-type slow mode transactions, the user's ERC-20 tokens were already transferred to the clearinghouse in a prior transaction; if the execution step fails and is silently swallowed, the tokens are permanently stranded and the user's on-chain balance is never credited.

---

### Finding Description

`_executeSlowModeTransaction` in `Endpoint.sol` deletes the queued transaction from `slowModeTxs` unconditionally, then attempts execution inside a `try/catch`:

```solidity
SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
delete slowModeTxs[_slowModeConfig.txUpTo++];   // consumed before execution

// ...
try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
    if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
        assembly { invalid() }
    }
    // try return funds now removed          // <-- refund logic explicitly removed
}
``` [1](#0-0) 

The two-transaction lifecycle for deposits is:

**Transaction 1** — `depositCollateralWithReferral`:
1. `handleDepositTransfer(token, msg.sender, amount)` — ERC-20 tokens leave the user's wallet and enter the clearinghouse.
2. A `SlowModeTx` of type `DepositCollateral` is appended to `slowModeTxs`. [2](#0-1) 

**Transaction 2** — `executeSlowModeTransaction` (or sequencer-driven):
1. The queue entry is deleted.
2. `processSlowModeTransaction` → `Clearinghouse.depositCollateral` is called to credit the user's spot balance.
3. If this call reverts for **any** reason, the `catch` block is entered, the error is discarded, and no balance is credited. [3](#0-2) 

Conditions that can cause `depositCollateral` to revert after the deposit was queued:
- `_decimals(txn.productId)` reverts because the product's token address was zeroed out or the product was removed (`ERR_INVALID_PRODUCT`).
- `spotEngine.updateBalance` reverts due to an internal invariant violation.
- Any future upgrade to `Clearinghouse` or `SpotEngine` that tightens validation.

Because the token transfer in Transaction 1 is final and irreversible, and the queue entry is already deleted, the user has no recourse: the tokens are permanently locked in the clearinghouse with no corresponding balance entry.

The comment `// try return funds now removed` is an explicit acknowledgment that a refund path was intentionally deleted, leaving the catch block empty.

---

### Impact Explanation

**Accounting corruption / fund loss**: A user's ERC-20 tokens are transferred to the clearinghouse in Transaction 1. If Transaction 2 fails silently, the tokens are stranded — the user's spot balance is never incremented, the queue entry is gone, and there is no on-chain mechanism to recover the funds. This is a direct, permanent loss of user assets.

The corrupted state delta: `spotEngine.balances[productId][sender]` is never updated despite the clearinghouse holding the corresponding tokens.

---

### Likelihood Explanation

The `executeSlowModeTransaction` function is callable by any unprivileged address after the `SLOW_MODE_TX_DELAY` timeout. A product configuration change (e.g., token address update, product removal) between deposit queuing and execution is a realistic trigger. The empty `catch` block and the removed refund comment indicate this path has been exercised in development and the safety net was deliberately stripped out.

---

### Recommendation

Mirror the fix recommended for H-7: perform all state-mutating steps atomically and only consume the queue entry on success. Concretely:

1. Do **not** delete the queue entry before attempting execution.
2. On success, delete the entry.
3. On failure, either retain the entry for retry or execute a refund path that returns the deposited tokens to the original sender.

```solidity
// pseudocode
try this.processSlowModeTransaction(txn.sender, txn.tx) {
    delete slowModeTxs[_slowModeConfig.txUpTo]; // only consume on success
} catch {
    if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
        assembly { invalid() }
    }
    // refund or retain for retry
    _refundSlowModeTx(txn);
}
```

---

### Proof of Concept

1. User calls `depositCollateralWithReferral(subaccount, productId, 1000e6)`.
   - 1000 USDC is transferred from user to clearinghouse.
   - A `DepositCollateral` slow mode tx is appended at index `N`.

2. Between queuing and execution, the product's token address is updated (or the product is delisted), causing `_decimals(productId)` to revert.

3. After `SLOW_MODE_TX_DELAY`, anyone calls `executeSlowModeTransaction()`.
   - `delete slowModeTxs[N]` executes — entry is gone.
   - `processSlowModeTransaction` → `depositCollateral` → `_decimals` reverts.
   - `catch` block is entered; error is discarded.
   - `nSubmissions` is incremented; `slowModeConfig` is updated.

4. User's spot balance for `productId` remains 0. The 1000 USDC is permanently locked in the clearinghouse. The queue entry no longer exists. There is no recovery path. [4](#0-3) [5](#0-4)

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }

    function executeSlowModeTransaction() external {
        SlowModeConfig memory _slowModeConfig = slowModeConfig;
        _executeSlowModeTransaction(_slowModeConfig, false);
        nSubmissions += 1;
        slowModeConfig = _slowModeConfig;
    }
