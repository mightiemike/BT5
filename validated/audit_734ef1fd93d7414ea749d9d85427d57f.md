### Title
Missing Zero Address Check in `upgradeClearinghouseLiq()` Enables Silent Liquidation Failure — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`upgradeClearinghouseLiq()` in `Clearinghouse.sol` sets the `clearinghouseLiq` address without a zero address guard. If accidentally set to `address(0)`, every subsequent call to `liquidateSubaccount()` silently no-ops via `delegatecall` to `address(0)`, permanently disabling the liquidation system and enabling unchecked bad-debt accumulation leading to protocol insolvency.

---

### Finding Description

`upgradeClearinghouseLiq()` assigns `clearinghouseLiq` directly from the caller-supplied argument with no zero address validation: [1](#0-0) 

`liquidateSubaccount()` then uses `clearinghouseLiq` as the target of a `delegatecall`, checking only whether the call succeeded: [2](#0-1) 

In the EVM, a `delegatecall` to `address(0)` (which has no code) returns `(true, "")` — success with empty returndata. The `require(success, string(result))` check at line 661 passes unconditionally. The function returns without error, without executing any liquidation logic, and without emitting any failure signal.

By contrast, `setWithdrawPool()` in the same contract already applies the correct pattern: [3](#0-2) 

The analogous `upgradeEndpointTx()` in `Endpoint.sol` also lacks a zero address check, but is partially mitigated by a guard in `_delegatecallEndpointTx`: [4](#0-3) [5](#0-4) 

`upgradeClearinghouseLiq()` has no equivalent guard, making it the higher-severity instance.

---

### Impact Explanation

With `clearinghouseLiq == address(0)`:

- Every `liquidateSubaccount()` call silently succeeds without executing liquidation logic.
- Underwater subaccounts are never closed; their bad debt is never absorbed.
- The protocol's insurance fund is not replenished from liquidation proceeds.
- Bad debt accumulates unchecked across all products, leading to protocol-wide insolvency and loss of depositor funds.

The corrupted state variable is `clearinghouseLiq`, and the broken invariant is that `liquidateSubaccount()` must always execute real liquidation logic when called.

---

### Likelihood Explanation

`proxyManagerHelper` is an operationally active role that performs routine upgrades. A misconfigured upgrade transaction — e.g., passing `address(0)` as a placeholder or due to a scripting error — would silently corrupt the liquidation path with no on-chain revert to signal the mistake. The absence of a zero address check (unlike `setWithdrawPool`) means there is no safety net.

---

### Recommendation

Add a zero address check in `upgradeClearinghouseLiq()`:

```solidity
function upgradeClearinghouseLiq(address _clearinghouseLiq) external {
    require(
        msg.sender == IProxyManager(_getProxyManager()).getProxyManagerHelper(),
        ERR_UNAUTHORIZED
    );
    require(_clearinghouseLiq != address(0), "zero clearinghouseLiq");
    clearinghouseLiq = _clearinghouseLiq;
}
```

Apply the same fix to `upgradeEndpointTx()` in `Endpoint.sol` for defense-in-depth.

---

### Proof of Concept

1. `proxyManagerHelper` calls `Clearinghouse.upgradeClearinghouseLiq(address(0))`.
2. `clearinghouseLiq` is written as `address(0)` — no revert occurs.
3. A liquidator submits a signed `LiquidateSubaccount` transaction through the sequencer.
4. `Clearinghouse.liquidateSubaccount()` executes `address(0).delegatecall(liquidateSubaccountCall)`.
5. EVM returns `(true, "")` — the `require(success, string(result))` at line 661 passes.
6. The function returns normally; no liquidation state is modified.
7. The underwater subaccount remains open; bad debt accumulates silently across the protocol. [6](#0-5)
