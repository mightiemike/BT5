### Title
CEI Violation in `withdrawCollateral` Enables ERC777 Reentrancy to Drain WithdrawPool Funds — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` performs the token transfer to the recipient **before** updating the subaccount's balance in `SpotEngine` and before executing the health check. No reentrancy guard exists anywhere in the protocol. A recipient contract registered as an ERC777 token hook receiver can re-enter the publicly callable `BaseWithdrawPool.submitFastWithdrawal` during the transfer callback, obtaining pool tokens for a second withdrawal (`idx2`) before the first withdrawal's (`idx1`) balance decrement has occurred. When the sequencer later processes `idx2` via slow mode (which silently swallows reverts), the health check fails and the balance decrement is rolled back — leaving the attacker with double the tokens but only a single balance decrement.

---

### Finding Description

`Clearinghouse.withdrawCollateral` executes in this order:

```
1. handleWithdrawTransfer(token, sendTo, amount, idx)   ← token leaves Clearinghouse → WithdrawPool → sendTo
2. spotEngine.updateBalance(productId, sender, -amount) ← balance decremented
3. require(getHealth(sender, healthType) >= 0)          ← health check
``` [1](#0-0) 

The token transfer at step 1 routes through `handleWithdrawTransfer`, which calls `token.safeTransfer(withdrawPool, amount)` and then `BaseWithdrawPool.submitWithdrawal`, which in turn calls `token.safeTransfer(sendTo, amount)`. [2](#0-1) [3](#0-2) [4](#0-3) 

If `sendTo` is a contract registered as an ERC777 `tokensReceived` hook, the hook fires at this inner `safeTransfer` call — **before** `spotEngine.updateBalance` has been called. At this moment the subaccount's recorded balance in `SpotEngine` is still at its pre-withdrawal level.

`BaseWithdrawPool.submitFastWithdrawal` is a **public, permissionless** function. Any caller who holds a valid sequencer-signed withdrawal transaction can invoke it to receive tokens from the pool immediately. [5](#0-4) 

A `grep` across the entire codebase confirms **no `nonReentrant` modifier or `ReentrancyGuard` is used anywhere** in the protocol.

---

### Impact Explanation

**Concrete asset delta:** The attacker drains `amount A` tokens from the `WithdrawPool` beyond their legitimate entitlement.

**Step-by-step:**

1. Attacker holds subaccount balance `B` where `B − A ≥ 0` but `B − 2A < 0`.
2. Attacker legitimately obtains two sequencer-signed withdrawal transactions: `idx1` (amount `A`) and `idx2` (amount `A`).
3. Attacker deploys a contract that implements the ERC777 `tokensReceived` hook and registers it as the recipient (`sendTo`).
4. Sequencer processes `idx1` via `submitTransactionsChecked` → `processTransaction` → `withdrawCollateral`.
   - `handleWithdrawTransfer` → `submitWithdrawal` → `token.safeTransfer(attackerContract, A)` → **ERC777 hook fires**.
   - Inside the hook: attacker calls `submitFastWithdrawal(idx2, transaction2, signatures2)`.
     - `markedIdxs[idx2] = true`; pool transfers `A` tokens to attacker.
   - Hook returns.
   - `spotEngine.updateBalance` decrements balance: `B → B − A`.
   - Health check: `B − A ≥ 0` ✓ — transaction succeeds.
5. Sequencer later processes `idx2` (via slow mode `_executeSlowModeTransaction` with `try/catch`):
   - `submitWithdrawal` sees `markedIdxs[idx2] == true` → returns early, **no token transfer**.
   - `spotEngine.updateBalance` decrements balance: `B − A → B − 2A`.
   - Health check: `B − 2A < 0` → **REVERTS**.
   - The `try/catch` in `_executeSlowModeTransaction` silently swallows the revert. [6](#0-5) 

6. **Final state:** Attacker holds `2A` tokens. Balance was decremented only by `A` (the `idx2` decrement was rolled back). The `WithdrawPool` is short `A` tokens — stolen from liquidity providers.

---

### Likelihood Explanation

- The attack requires the collateral token to support ERC777 or any transfer-callback mechanism. The protocol is designed to support arbitrary ERC20-compatible tokens, making this a realistic token configuration.
- The attacker needs two valid sequencer-signed withdrawal transactions, both of which are legitimately obtainable by any user.
- No privileged access, governance capture, or key compromise is required.
- The slow-mode `try/catch` silent-revert path is a standard protocol execution path, not an edge case.

---

### Recommendation

1. **Apply the Checks-Effects-Interactions pattern**: In `Clearinghouse.withdrawCollateral`, call `spotEngine.updateBalance` and perform the health check **before** calling `handleWithdrawTransfer`.
2. **Add a reentrancy guard**: Apply OpenZeppelin's `ReentrancyGuardUpgradeable` to `withdrawCollateral` and `submitFastWithdrawal` (or add a protocol-wide guard at the `Endpoint` entry point).
