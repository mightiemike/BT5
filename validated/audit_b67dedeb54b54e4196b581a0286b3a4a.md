### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Merkle Proof Forgery Protection — (`contract/src/lib.rs`)

---

### Summary

The contract exposes two public SPV verification methods: the deprecated `verify_transaction_inclusion` and the current `verify_transaction_inclusion_v2`. The v2 method was introduced specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability. However, the deprecated v1 method remains fully callable by any unprivileged NEAR account. An attacker can deliberately invoke the deprecated method to bypass the coinbase Merkle proof validation and have a forged transaction inclusion proof accepted as `true`.

---

### Finding Description

`verify_transaction_inclusion` is marked `#[deprecated]` in Rust, but this attribute only produces compiler warnings — it does not restrict on-chain callability. The method remains a live, public NEAR contract entry point. [1](#0-0) 

`verify_transaction_inclusion_v2` was introduced to close the 64-byte transaction Merkle proof forgery attack (referenced explicitly in the code comments at the bitmex blog link). It does so by requiring the caller to also supply a coinbase Merkle proof and validating it against the block's Merkle root before delegating to the v1 logic: [2](#0-1) 

The v1 method performs no such coinbase validation — it computes only the transaction's Merkle path and compares it to the stored Merkle root: [3](#0-2) 

The contract's own warning comment acknowledges the risk:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [4](#0-3) 

Both methods accept caller-supplied `ProofArgs` / `ProofArgsV2` via Borsh, with no access control, no role check, and no pause guard difference that would prevent the v1 path from being chosen deliberately. [5](#0-4) 

---

### Impact Explanation

An attacker who wants to falsely prove a Bitcoin transaction was included in a block can:

1. Craft a 64-byte payload that is simultaneously a valid-looking internal Merkle tree node and a plausible `tx_id`.
2. Construct a Merkle path that leads from this crafted node to the real Merkle root stored on-chain.
3. Call `verify_transaction_inclusion` (v1) directly, bypassing the coinbase proof check entirely.
4. Receive `true` from the contract.

Any NEAR consumer contract that calls `verify_transaction_inclusion` (v1) — whether by mistake or because an attacker controls the call — will accept a forged SPV proof. This corrupts the fundamental trust guarantee of the light client: that a `true` result means a real Bitcoin transaction was confirmed on-chain.

---

### Likelihood Explanation

The 64-byte transaction forgery technique is publicly documented (the bitmex blog post is cited in the contract's own comments). The existence of the deprecated but still-callable v1 method is visible to anyone reading the contract ABI or source. No privileged role, leaked key, or social engineering is required — any NEAR account can call `verify_transaction_inclusion` directly. Consumer contracts that integrated before v2 was introduced may still be calling v1, and a malicious proof submitter can target those consumers.

---

### Recommendation

Remove `verify_transaction_inclusion` as a callable public entry point. In NEAR/Rust, `#[deprecated]` does not restrict on-chain invocation. The method should either be deleted entirely or converted to a private helper. If backward compatibility is required for a transition period, add an explicit `env::panic_str` at the top of the v1 method body to make it unconditionally revert when called externally.

---

### Proof of Concept

1. Deploy the contract (Bitcoin feature flag) with a known set of mainchain headers.
2. Identify a block whose Merkle root is known on-chain.
3. Construct a 64-byte crafted `tx_id` and a Merkle path that hashes to the stored Merkle root (following the technique at https://www.bitmex.com/blog/64-Byte-Transactions).
4. Call `verify_transaction_inclusion` with `ProofArgs { tx_id: <crafted>, tx_block_blockhash: <real block hash>, tx_index: <chosen index>, merkle_proof: <crafted path>, confirmations: 1 }`.
5. Observe the method returns `true` for a transaction that does not exist on Bitcoin.
6. Call `verify_transaction_inclusion_v2` with the same `tx_id` and any coinbase proof — the coinbase validation step will reject it, demonstrating that v2 closes the gap that v1 leaves open. [6](#0-5)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
