### Title
Difficulty Adjustment for Fork Blocks Uses Mainchain Height Index Instead of Fork Ancestor, Enabling Incorrect PoW Acceptance — (File: `contract/src/dogecoin.rs`)

---

### Summary

In `get_next_work_required`, the reference block for difficulty computation is fetched via `blocks_getter.get_header_by_height(height_first)`. This call always resolves to the **mainchain** block at `height_first` (via `mainchain_height_to_header`), not the fork's actual ancestor at that height. When a fork diverges before `height_first`, the two blocks differ in timestamp, causing the difficulty computed for the fork block to be wrong. An unprivileged relayer can exploit this to submit fork blocks that satisfy an artificially lowered difficulty target, potentially triggering a chain reorganization with less cumulative proof-of-work than the protocol requires.

---

### Finding Description

`get_next_work_required` in `contract/src/dogecoin.rs` computes the expected `bits` for an incoming block by comparing the timestamp of the block at `height_first` against the previous block's timestamp:

```rust
// TODO: check if it is correct to get block header by height from mainchain
// without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [1](#0-0) 

`get_header_by_height` is implemented in `lib.rs` as a direct lookup into `mainchain_height_to_header`:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [2](#0-1) 

This lookup is unconditional: it always returns the **mainchain** block at `height_first`, regardless of whether the block being validated is a mainchain extension or a fork block.

The difficulty check is invoked for every submitted block, including fork blocks, inside `submit_block_header`:

```rust
if !skip_pow_verification {
    self.check_target(&block_header, &prev_block_header);
    ...
}
``` [3](#0-2) 

`check_target` → `check_pow` → `get_next_work_required` is the full call chain. For a fork block at height `H+1` whose chain diverged at height `D < height_first`:

- `prev_block_header` is the fork's block at height `H` (correctly fetched from `headers_pool`)
- `height_first = H - blocks_to_go_back`
- `get_header_by_height(height_first)` returns the **mainchain** block at `height_first`, not the fork's ancestor at that height

These two blocks have different timestamps whenever the fork diverged before `height_first`. The timestamp difference feeds directly into `calculate_next_work_required`:

```rust
let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;
let mut modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;
``` [4](#0-3) 

If the mainchain's block at `height_first` has an **earlier** timestamp than the fork's actual ancestor at that height, `modulated_timespan` is inflated, the computed target is easier (lower `bits` value accepted), and the fork block passes a difficulty check it should have failed.

The TODO comment in the source code explicitly flags this concern, confirming the developers identified but did not resolve it: [5](#0-4) 

---

### Impact Explanation

An attacker who controls fork block timestamps (within MTP and `MAX_FUTURE_BLOCK_TIME_LOCAL` bounds) can craft a fork chain where the difficulty check for each successive fork block uses the mainchain's earlier-timestamped reference block. This inflates the apparent elapsed time, causing `calculate_next_work_required` to return a lower difficulty (easier target) than the protocol mandates. Fork blocks mined at this reduced difficulty are accepted by the contract. If the fork's cumulative `chain_work` (computed from the accepted, artificially low `bits`) exceeds the mainchain's, `submit_block_header_inner` triggers `reorg_chain`, replacing the canonical chain with a fork that was mined with less actual proof-of-work than required. Downstream consumers of `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` then operate against a corrupted canonical chain. [6](#0-5) 

---

### Likelihood Explanation

`submit_blocks` is callable by any unprivileged NEAR account (subject only to the `trusted_relayer` gate, which has an `UnrestrictedSubmitBlocks` bypass role and is the normal relayer path). Any fork submission that diverges more than `blocks_to_go_back` blocks before the tip exercises this code path. The attacker does not need privileged access; they only need to submit a sequence of valid-PoW fork headers with controlled timestamps, which is the standard relayer operation. The TODO comment confirms the issue was known and unresolved. [7](#0-6) 

---

### Recommendation

Replace the height-indexed mainchain lookup with an ancestor walk that follows `prev_block_hash` links from `prev_block_header` backward through `headers_pool` until the block at `height_first` is reached. This mirrors the approach used in `reorg_chain`, which correctly traverses fork chains via `prev_block_hash` rather than the height index. Alternatively, pass the fork's ancestor at `height_first` as an explicit parameter to `get_next_work_required` so the caller (which already holds `prev_block_header`) can supply the correct reference block. [8](#0-7) 

---

### Proof of Concept

**Setup:** Dogecoin mainchain at heights `D`, `D+1`, …, `H` (all post-145000). Mainchain block at height `H-1` has timestamp `T_main`.

**Attack:**

1. Attacker submits a fork block at height `D+1` that shares the parent of the mainchain block at `D+1` but carries timestamp `T_fork_D+1 > T_main_D+1`. This block passes its own difficulty check because `get_header_by_height(D)` returns the shared ancestor (correct at this depth).

2. Attacker continues building the fork up to height `H`. The fork block at height `H` has timestamp `T_fork_H`.

3. Attacker submits a fork block at height `H+1`. `get_next_work_required` computes `height_first = H - 1` and calls `get_header_by_height(H-1)`, returning the **mainchain** block at `H-1` with timestamp `T_main`. The fork's actual ancestor at `H-1` has timestamp `T_fork_H-1 > T_main`.

4. `modulated_timespan = T_fork_H - T_main` (inflated vs. correct `T_fork_H - T_fork_H-1`). `calculate_next_work_required` returns a lower `bits` value (easier target).

5. The fork block at `H+1` is accepted at this reduced difficulty. Its `chain_work` contribution is `work_from_bits(bits_easy)`.

6. Repeating across multiple fork blocks, the fork accumulates `chain_work > mainchain chain_work`. `submit_block_header_inner` calls `reorg_chain`, replacing the canonical chain. [9](#0-8) [10](#0-9)
