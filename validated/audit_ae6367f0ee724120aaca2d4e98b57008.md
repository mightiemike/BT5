### Title
Storage-freed NEAR Tokens Permanently Locked in `BtcLightClient` Contract — (`contract/src/lib.rs`)

---

### Summary

`submit_blocks` collects NEAR deposits from relayers to pay for on-chain storage. The embedded call to `run_mainchain_gc` deletes old block headers, freeing that storage. In NEAR Protocol, freed storage converts locked NEAR back into the contract's available balance. However, `BtcLightClient` contains no withdrawal function and no approved spender, so every NEAR token released by GC is permanently trapped in the contract.

---

### Finding Description

`submit_blocks` is marked `#[payable]` and measures storage growth to charge the caller:

```rust
// contract/src/lib.rs:173-197
let amount = env::attached_deposit();
let initial_storage = env::storage_usage();          // snapshot before

for header in headers { self.submit_block_header(...); }

self.run_mainchain_gc(num_of_headers);               // may DELETE headers

let diff_storage_usage = env::storage_usage()
    .saturating_sub(initial_storage);               // net delta (floor 0)
let required_deposit = env::storage_byte_cost()
    .saturating_mul(diff_storage_usage.into());

let refund = amount.saturating_sub(required_deposit);
if refund > NearToken::from_near(0) {
    Promise::new(env::predecessor_account_id()).transfer(refund)...
}
``` [1](#0-0) 

`run_mainchain_gc` removes entries from `headers_pool` and `mainchain_height_to_header` for every block older than `gc_threshold`:

```rust
// contract/src/lib.rs:401-408
for height in start_removal_height..end_removal_height {
    let blockhash = &self.mainchain_height_to_header.get(&height)...;
    self.remove_block_header(blockhash);
    self.mainchain_height_to_header.remove(&height);
}
``` [2](#0-1) 

`remove_block_header` deletes from both `mainchain_header_to_height` and `headers_pool`:

```rust
// contract/src/lib.rs:659-662
fn remove_block_header(&mut self, header_block_hash: &H256) {
    self.mainchain_header_to_height.remove(header_block_hash);
    self.headers_pool.remove(header_block_hash);
}
``` [3](#0-2) 

In NEAR Protocol, deleting storage entries reduces `env::storage_usage()`, which converts the previously locked NEAR (storage staking) back into the contract account's **available** balance. The `saturating_sub` in `submit_blocks` means the *current* caller's deposit is fully refunded when GC removes more bytes than were added. But the NEAR deposited by **previous** callers for the headers that GC just deleted is now sitting in the contract's available balance.

A grep across all contract source files confirms the only two occurrences of `Promise::new` / `transfer` are the single refund path inside `submit_blocks` — there is no `withdraw`, no treasury transfer, and no approved spender anywhere in the contract. [4](#0-3) 

---

### Impact Explanation

Every time GC fires and removes headers, the NEAR tokens that were deposited for those headers accumulate in the contract's available balance with no recovery path. With the recommended `gc_threshold = 52704` (≈ one year of Bitcoin blocks) and each `ExtendedHeader` serialized to several hundred bytes, the steady-state locked amount is on the order of hundreds of NEAR tokens. Once GC begins cycling, this balance grows monotonically and is irrecoverable without a contract upgrade.

---

### Likelihood Explanation

GC is triggered automatically on every `submit_blocks` call once the chain exceeds `gc_threshold` blocks. The relayer is designed to run continuously, so GC fires on every block submission in steady-state operation. This is not an edge case — it is the normal operating mode of a live deployment. Any trusted relayer (an unprivileged NEAR account holding the `UnrestrictedSubmitBlocks` role) is the entry point. [5](#0-4) 

---

### Recommendation

Add a privileged `withdraw` function (callable only by the DAO or a designated treasury role) that transfers the contract's available balance to a configured treasury account:

```rust
pub fn withdraw(&mut self, amount: NearToken, receiver: AccountId) {
    // require DAO role
    Promise::new(receiver).transfer(amount)
}
```

Alternatively, track the cumulative freed storage per original depositor and allow them to reclaim it, mirroring the storage-management pattern used by NEAR's standard fungible token contracts.

---

### Proof of Concept

1. Relayer submits 52,705 block headers across multiple `submit_blocks` calls, each time attaching the required NEAR deposit. The contract's storage grows and NEAR is locked.
2. On the 52,705th header submission, `run_mainchain_gc` fires and removes the oldest header. The NEAR deposited for that header is released into the contract's available balance.
3. From this point forward, every `submit_blocks` call removes one header (GC keeps the chain at `gc_threshold`). The freed NEAR accumulates in the contract's available balance.
4. Inspect `env::account_balance()` on the contract account — it grows with each GC cycle.
5. Attempt to call any function to withdraw this balance — none exists. The NEAR is permanently locked. [6](#0-5)

    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }

    pub fn run_mainchain_gc(&mut self, batch_size: u64) {
        let initial_blockheader = self
            .headers_pool
            .get(&self.mainchain_initial_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let tip_blockheader = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let amount_of_headers_we_store =
            tip_blockheader.block_height - initial_blockheader.block_height + 1;

        if amount_of_headers_we_store > self.gc_threshold {
            let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
            let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);

            let start_removal_height = initial_blockheader.block_height;
            let end_removal_height = initial_blockheader.block_height + selected_amount_to_remove;
            env::log_str(&format!(
                "Num of blocks to remove {selected_amount_to_remove}"
            ));

            for height in start_removal_height..end_removal_height {
                let blockhash = &self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

                self.remove_block_header(blockhash);
                self.mainchain_height_to_header.remove(&height);
            }

            self.mainchain_initial_blockhash = self
                .mainchain_height_to_header
                .get(&end_removal_height)
                .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        }
    }
