### Title
Slow Mode Fee Permanently Lost on Silent Transaction Failure — (`core/contracts/Endpoint.sol`)

---

### Summary

When a user submits a slow mode transaction, they pay a `SLOW_MODE_FEE` upfront via ERC20 transfer. If the queued transaction later fails during execution, the failure is swallowed silently and the fee is **never refunded**. A removed-but-commented refund path (`// try return funds now removed`) confirms this was a deliberate deletion of the recovery mechanism.

---

### Finding Description

`submitSlowModeTransactionImpl` in `EndpointTx.sol` charges the caller a `SLOW_MODE_FEE` before enqueuing the transaction: [1](#0-0) 

The fee is pulled from the user's wallet via `chargeSlowModeFee`: [2](#0-1) 

The slow mode transaction is stored with a three-day execution delay: [3](#0-2) 

When `_executeSlowModeTransaction` runs (callable by anyone after the delay), it wraps execution in a `try/catch`. On failure, the catch block does nothing — and the comment explicitly marks where a refund was removed: [4](#0-3) 

The `SLOW_MODE_FEE` accumulates in `slowModeFees` and is never returned to the originating user on failure: [5](#0-4) 

---

### Impact Explanation

A user who submits a slow mode `WithdrawCollateral` (or `LinkSigner`, etc.) pays the fee from their wallet at submission time. If the transaction fails at execution time — for any reason — the fee is silently consumed. The user receives no service and no refund. This is a direct, concrete asset loss from the user's wallet balance (quote token).

---

### Likelihood Explanation

The three-day window between submission and execution creates multiple realistic failure paths:

1. **Liquidation path (unprivileged trigger):** A user submits a slow mode `WithdrawCollateral` for their full balance and pays the fee. An unprivileged liquidator, observing the pending queue, liquidates the user's subaccount (legitimately, if the account becomes undercollateralised). When the slow mode transaction executes, the balance is zero, the withdrawal fails silently, and the fee is lost. The liquidator is a reachable, unprivileged caller who directly causes the failure.

2. **Self-inflicted path:** The user's own trading losses reduce their balance below the withdrawal amount before the three-day delay expires. Same silent failure, same fee loss.

Both paths are reachable without any privileged access.

---

### Recommendation

Restore a refund mechanism inside the `catch` block of `_executeSlowModeTransaction`. When a slow mode transaction fails, the `SLOW_MODE_FEE` should be returned to `txn.sender` (pull-payment pattern preferred to avoid the sender blocking the refund push). The removed code path (`// try return funds now removed`) should be reinstated with a pull-payment design.

---

### Proof of Concept

1. User calls `submitSlowModeTransaction` with a `WithdrawCollateral` payload for their full balance. `chargeSlowModeFee` pulls `SLOW_MODE_FEE` from the user's wallet. `slowModeFees += SLOW_MODE_FEE` is recorded.
2. The transaction is stored: `executableAt = block.timestamp + SLOW_MODE_TX_DELAY` (3 days).
3. Within the 3-day window, an unprivileged liquidator calls the liquidation path, draining the user's subaccount balance to zero.
4. After 3 days, anyone calls `executeSlowModeTransaction()`. Inside `_executeSlowModeTransaction`, `processSlowModeTransaction` is called via `try`. The `WithdrawCollateral` reverts (insufficient balance). The `catch` block fires, logs nothing, refunds nothing.
5. The user has lost `SLOW_MODE_FEE` from their wallet with zero service delivered. The `// try return funds now removed` comment at line 226 of `Endpoint.sol` confirms the refund path was deliberately excised. [6](#0-5)

```text
    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
```
