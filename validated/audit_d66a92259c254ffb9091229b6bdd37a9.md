### Title
Leaf-Node / Internal-Node Domain Confusion in Merkle Proof Verification Allows Proof Forgery — (`merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (still publicly callable despite its `#[deprecated]` marker) passes any caller-supplied `tx_id` directly into `compute_root_from_merkle_proof` without distinguishing leaf nodes from internal Merkle-tree nodes. Both are represented as the same untagged `H256` type. An unprivileged NEAR caller can supply an internal node hash as `tx_id` together with a crafted sibling-path, causing the function to return `true` for a Bitcoin transaction that was never included in the block.

---

### Finding Description

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` treats its `transaction_hash` argument as an opaque 32-byte value and iterates upward through the tree unconditionally: [1](#0-0) 

There is no domain tag, length check, or structural constraint that distinguishes a real transaction hash (a leaf) from an internal node (the double-SHA256 of two concatenated child hashes). Both are `H256` — the same 32-byte format.

`verify_transaction_inclusion` in `contract/src/lib.rs` feeds the caller-controlled `args.tx_id` directly into this function: [2](#0-1) 

The function is still exported as a public NEAR contract method. The Rust `#[deprecated]` attribute only emits a compiler warning; it does not restrict on-chain invocation: [3](#0-2) 

The function's own documentation acknowledges the gap: [4](#0-3) 

`verify_transaction_inclusion_v2` was introduced to close this gap via a coinbase-proof requirement, but it does not remove or gate the v1 path: [5](#0-4) 

Any caller that invokes `verify_transaction_inclusion` directly bypasses the coinbase-proof mitigation entirely.

---

### Impact Explanation

A downstream contract or bridge that calls `verify_transaction_inclusion` to gate fund releases, token mints, or state transitions will receive a forged `true` result. The attacker does not need to break any cryptographic primitive — they only need to identify an internal node in a real, mainchain-accepted block and construct the corresponding sibling path. This enables theft or double-spend in any cross-chain application that relies on the v1 verification endpoint.

---

### Likelihood Explanation

The entry path requires no privilege: any NEAR account can call `verify_transaction_inclusion`. The required inputs (a real block hash accepted by the light client, an internal node hash, and its sibling path) are all derivable from public Bitcoin chain data. The attack is deterministic and requires no brute force.

---

### Recommendation

- **Short term**: Remove `verify_transaction_inclusion` from the public contract interface entirely, or add an explicit domain check that rejects any `tx_id` whose position and proof length are inconsistent with a leaf node (i.e., enforce `tx_index < 2^(merkle_proof.len())`). At minimum, gate the v1 method behind the same coinbase-proof requirement as v2.
- **Long term**: Document the distinction between leaf-node hashes and internal-node hashes in the Merkle proof API. Add a decoding/validation routine that asserts the supplied `tx_id` cannot be an internal node of the tree described by the proof, and cover this invariant with unit tests.

---

### Proof of Concept

1. Identify any Bitcoin block `B` whose hash is recorded in the light client's mainchain (e.g., a block with 4 transactions: `tx0, tx1, tx2, tx3`).
2. Compute the level-1 internal node: `I = double_sha256(tx0 || tx1)`.
3. Compute the sibling of `I` at level 1: `S = double_sha256(tx2 || tx3)`.
4. Note that `double_sha256(I || S) == merkle_root` of block `B`.
5. Call `verify_transaction_inclusion` with:
   - `tx_id = I` (the internal node, not a real txid)
   - `tx_block_blockhash = B`
   - `tx_index = 0` (position of `I` at level 1)
   - `merkle_proof = [S]` (one sibling, reaching the root in one step)
   - `confirmations = 1`
6. `compute_root_from_merkle_proof(I, 0, [S])` computes `double_sha256(I || S) == merkle_root` → returns `true`.
7. The contract confirms inclusion of a transaction that does not exist in block `B`. [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
