### Title
`withdrawFromDirectDepositV1()` Token Recovery Blocked by Front-Running `creditDeposit()` — (`core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.withdrawFromDirectDepositV1()` uses a `postBalance - preBalance` delta pattern to measure how many tokens were pulled from a `DirectDepositV1` (DDA) contract and forward them to the caller. Because `DirectDepositV1.creditDeposit()` carries no access control and can be called by anyone, a malicious actor can front-run the owner's recovery transaction, drain the DDA into the protocol, and cause `withdrawFromDirectDepositV1` to revert — permanently preventing the intended token recovery path.

---

### Finding Description

`ContractOwner.withdrawFromDirectDepositV1()` records a pre-call balance, triggers `DirectDepositV1.withdraw()` to pull tokens into `ContractOwner`, then computes and forwards the delta:

```solidity
uint256 preBalance = IERC20Base(token).balanceOf(address(this));
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(token));
uint256 postBalance = IERC20Base(token).balanceOf(address(this));
require(postBalance > preBalance, "empty");
IERC20Base(token).safeTransfer(msg.sender, postBalance - preBalance);
``` [1](#0-0) 

`DirectDepositV1.creditDeposit()` is a public, permissionless function that sweeps every token balance held by the DDA and deposits it into the Endpoint as collateral for the DDA's bound subaccount:

```solidity
function creditDeposit() external {
    ...
    uint256 balance = token.balanceOf(address(this));
    if (balance != 0) {
        token.approve(address(endpoint), balance);
        endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1");
    }
}
``` [2](#0-1) 

`ContractOwner.creditDepositV1()` is also permissionless, providing a second public entry point to the same drain:

```solidity
function creditDepositV1(bytes32 subaccount) external {
    ...
    DirectDepositV1(directDepositV1).creditDeposit();
}
``` [3](#0-2) 

**Attack sequence:**

1. Owner submits `withdrawFromDirectDepositV1(subaccount, token)` to recover tokens sitting in the DDA.
2. Attacker observes the pending transaction in the mempool and front-runs it by calling `DirectDepositV1(dda).creditDeposit()` (or `ContractOwner.creditDepositV1(subaccount)`).
3. `creditDeposit()` sweeps the DDA's entire token balance into the Endpoint as collateral for the subaccount — the DDA is now empty.
4. The owner's `withdrawFromDirectDepositV1` executes: `DirectDepositV1.withdraw()` transfers 0 tokens to `ContractOwner`, so `postBalance == preBalance`.
5. `require(postBalance > preBalance, "empty")` reverts the transaction.
6. The tokens are now locked inside the protocol as collateral for the subaccount rather than returned to the owner.

---

### Impact Explanation

The owner's only direct recovery path for tokens held in the DDA is permanently blocked. The tokens are forcibly deposited as collateral into the protocol's subaccount. Depending on the subaccount's health at the time of the attack, those tokens may be consumed by liquidation or settlement, making them unrecoverable. Even in the best case, the owner must navigate the slow-mode withdrawal queue (with its built-in delay) to retrieve funds that were supposed to be immediately recoverable via `withdrawFromDirectDepositV1`.

---

### Likelihood Explanation

`creditDeposit()` requires no privileges and costs only gas. An attacker monitoring the mempool for `withdrawFromDirectDepositV1` calls can reliably front-run them on any chain without a private mempool. The attack is repeatable: every time the owner attempts recovery, the attacker can drain the DDA again. There is no economic cost beyond gas, and the attack can be automated.

---

### Recommendation

Replace the `postBalance - preBalance` delta pattern in `withdrawFromDirectDepositV1` with a direct full-balance transfer. After calling `DirectDepositV1.withdraw()`, transfer the entire current balance of `ContractOwner` for that token (or, better, have `DirectDepositV1.withdraw()` transfer directly to `msg.sender` of `ContractOwner` rather than routing through `ContractOwner`'s own balance). This eliminates the dependency on a delta that can be zeroed by a concurrent `creditDeposit()` call.

Additionally, consider adding access control to `DirectDepositV1.creditDeposit()` so that only the owner (`ContractOwner`) can trigger it, preventing any external party from draining the DDA at will.

---

### Proof of Concept

```
1. DDA holds 1000 USDC for subaccount S.
2. Owner calls: ContractOwner.withdrawFromDirectDepositV1(S, USDC)
   - preBalance = ContractOwner.balanceOf(USDC) = X
3. Attacker front-runs: DirectDepositV1(dda).creditDeposit()
   - DDA's 1000 USDC deposited into Endpoint as collateral for S
   - DDA balance = 0
4. Owner's tx resumes: DirectDepositV1.withdraw(USDC) → transfers 0 to ContractOwner
   - postBalance = X (unchanged)
   - require(X > X, "empty") → REVERT
5. Owner receives 0 USDC. 1000 USDC is now collateral in the protocol under subaccount S.
``` [4](#0-3) [2](#0-1)
