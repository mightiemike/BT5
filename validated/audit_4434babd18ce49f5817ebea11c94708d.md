### Title
Linked Signer Can Self-Replace to Attacker-Controlled Address via `LinkSigner` with `allowLinkedSigner=true` — (`core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction handler in `processTransactionImpl` calls `validateSignedTx` with `allowLinkedSigner=true`, meaning the **current linked signer's signature is sufficient** to authorize overwriting `linkedSigners[subaccount]` with any new address. There is no check that the authorizing signature must come from the subaccount owner. A compromised or malicious linked signer can therefore permanently transfer signing authority to an attacker-controlled address.

---

### Finding Description

In `EndpointTx.sol`, the sequencer-path handler for `LinkSigner` is:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

`validateSignedTx` delegates to `validateSignature`, which when `allowLinkedSigner=true` passes the current linked signer address to the verifier:

```solidity
// EndpointTx.sol lines 172–184
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

The verifier accepts the signature if the recovered address is **either** the subaccount owner **or** the linked signer:

```solidity
// Verifier.sol lines 297–303
address recovered = ECDSA.recover(digest, signature);
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

After validation, `linkedSigners[signedTx.tx.sender]` is unconditionally overwritten with the new signer value — no ownership check is performed. [4](#0-3)

### Impact Explanation

A linked signer (granted limited trading authority by the subaccount owner) can:
1. Sign a `LinkSigner{sender: victimSubaccount, signer: attacker2}` transaction.
2. Have the sequencer process it — `validateSignedTx` passes because the current linked signer's signature is accepted.
3. `linkedSigners[victimSubaccount]` is overwritten with `attacker2`.
4. `attacker2` can now sign `WithdrawCollateral`, `TransferQuote`, and all other `allowLinkedSigner=true` transactions on behalf of the victim.
5. The victim retains the ability to recover only by submitting a new owner-signed `LinkSigner` transaction — but by then, assets may already be drained.

This directly enables **unauthorized withdrawals and trades** on behalf of the victim, matching the Critical scope target.

---

### Likelihood Explanation

The precondition — that the attacker already holds linked signer status — is a realistic scenario. Linked signers are commonly granted to trading bots, hot wallets, or third-party services. Any such party that turns malicious (or is compromised) can execute this escalation. The sequencer path is a standard production entrypoint and requires no special privileges beyond submitting a validly signed transaction.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This ensures only the address embedded in the first 20 bytes of the subaccount `bytes32` (the owner) can authorize a linked signer change, consistent with the invariant. The slow-mode path already enforces this correctly via `validateSender`. [6](#0-5)

### Proof of Concept

```
Setup:
  victimSubaccount = bytes32(abi.encodePacked(victimOwner, bytes12(0)))
  linkedSigners[victimSubaccount] = attacker1   // victim previously granted attacker1 linked signer

Attack:
  1. attacker1 constructs LinkSigner{ sender: victimSubaccount, signer: attacker2, nonce: N }
  2. attacker1 signs the EIP-712 digest of this transaction
  3. Sequencer receives and calls processTransactionImpl with this signed tx
  4. validateSignedTx(..., allowLinkedSigner=true):
       - validateNonce passes (nonce N is correct)
       - validateSignature: recovered == attacker1 == getLinkedSigner(victimSubaccount) → passes
       - requireSubaccount: victimSubaccount exists → passes
  5. linkedSigners[victimSubaccount] = attacker2

Assert:
  linkedSigners[victimSubaccount] == attacker2   // attacker2 now controls the subaccount
  attacker2 can sign WithdrawCollateral for victimSubaccount → funds drained
  victimOwner must race to submit a new owner-signed LinkSigner to recover control
```
