Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks caller-controlled `owner` instead of actual depositor `sender`, fully bypassing the deposit allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of `addLiquidity`) and instead checks `owner`, which is a free argument supplied by the caller. Any unprivileged address can bypass the allowlist by passing any already-allowlisted address as `owner`, paying tokens themselves while the allowlisted address receives the LP position. The deposit allowlist access-control boundary is completely defeated.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as the first argument to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both `sender` and `owner` faithfully to the extension hook: [2](#0-1) 

However, `DepositAllowlistExtension.beforeAddLiquidity` names the first parameter with an unnamed placeholder (discarding `sender`) and checks `owner` instead: [3](#0-2) 

Because `owner` is freely chosen by the caller of `addLiquidity`, any address can pass the guard by supplying an allowlisted address as `owner`. The sibling `SwapAllowlistExtension.beforeSwap` correctly checks `sender`: [4](#0-3) 

The asymmetry between the two extensions confirms the check in `DepositAllowlistExtension` is on the wrong identity. Note also that `removeLiquidity` enforces `msg.sender == owner` at line 206, so the `owner`-based check there would be equivalent — but `addLiquidity` has no such constraint, making the wrong check exploitable. [5](#0-4) 

## Impact Explanation
The deposit allowlist is an admin-configured access-control boundary intended to gate `addLiquidity` by depositor identity (e.g., for KYC/compliance). Because the check operates on `owner` (attacker-controlled) rather than `sender` (actual depositor), the boundary is bypassed by any unprivileged address with a single standard call. This is an **admin-boundary break**: a security control set by the pool admin is circumvented through a normal, publicly accessible call path. Unauthorized parties can inject liquidity into restricted pools, defeating compliance restrictions and altering LP distribution in ways the pool admin explicitly sought to prevent.

## Likelihood Explanation
Exploitation requires a single call to `pool.addLiquidity` with `owner` set to any allowlisted address. No privileges, flash loans, or multi-step setup are needed. Allowlisted addresses are discoverable on-chain via the `allowedDepositor` mapping. The bypass is unconditional and repeatable.

## Recommendation
Check `sender` (the actual depositor) instead of `owner`, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
1. Pool is deployed with `DepositAllowlistExtension` as a `beforeAddLiquidity` extension.
2. Pool admin allowlists address **A**: `setAllowedToDeposit(pool, A, true)`.
3. Unauthorized address **B** (not on the allowlist) calls:
   ```solidity
   pool.addLiquidity(
       /* owner = */ A,   // allowlisted — passes the guard
       salt,
       deltas,
       callbackData,
       extensionData
   );
   ```
4. Pool calls `_beforeAddLiquidity(msg.sender=B, owner=A, …)`.
5. Extension evaluates `allowedDepositor[pool][A]` → `true` → no revert.
6. `LiquidityLib.addLiquidity` creates the position keyed to `(A, salt)` and calls back to **B** to collect tokens.
7. **B** pays the tokens; **A** receives the LP shares.
8. **B** has successfully deposited into a restricted pool without being allowlisted. The deposit allowlist is fully bypassed.
