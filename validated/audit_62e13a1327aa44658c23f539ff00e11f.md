### Title
SwapAllowlistExtension Checks Router Address Instead of Actual User, Allowing Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed by the pool, which is `msg.sender` of the `pool.swap()` call. When swaps are routed through the public `MetricOmmSimpleRouter`, `sender` is the router's address, not the actual user's address. A pool admin who allowlists the router (the natural step to let allowlisted users reach the pool via the standard periphery) inadvertently opens the gate to every unprivileged user, because the extension cannot distinguish individual callers behind the shared router address.

### Finding Description
`SwapAllowlistExtension.beforeSwap` performs the identity check as:

```solidity
// SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (the extension caller), and `sender` is the value the pool forwarded from its own `msg.sender`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← this becomes `sender` in the extension
    recipient,
    ...
);
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The pool's `msg.sender` is the router, so the extension receives `sender = router_address`. The extension then checks `allowedSwapper[pool][router_address]`, not `allowedSwapper[pool][actual_user]`.

The same pattern applies to `exactInput` (multi-hop), `exactOutputSingle`, and `exactOutput`. In the multi-hop `exactOutput` recursive callback path, intermediate hops are also called with `msg.sender = router`:

```solidity
// MetricOmmSimpleRouter.sol L220-228
(int128 amount0DeltaReturned, int128 amount1DeltaReturned) = IMetricOmmPoolActions(pool)
    .swap(
        msg.sender,   // ← router is still the caller
        zeroForOne,
        ...
    );
```

**The invariant break**: The pool admin intends to restrict swaps to a named set of addresses. The extension enforces this restriction only when users call `pool.swap()` directly. Any user who routes through the public, permissionless `MetricOmmSimpleRouter` is checked as the router address. If the router is allowlisted (which is necessary for allowlisted users to use the standard periphery), the allowlist is completely bypassed for all users.

This is the direct analog to the DYAD H-05 bug: just as DYAD's liquidation threshold used `totalDeposits` as a denominator that any user could manipulate to trigger liquidations on others, the Metric OMM allowlist uses the immediate `msg.sender` as the checked identity — a value that any user can substitute with the router address by routing through the shared periphery contract.

### Impact Explanation
Any unprivileged user can swap on a pool that the admin intended to restrict to a specific allowlist. The swap allowlist guard — the only mechanism preventing unauthorized access to the pool's swap flow — is rendered ineffective for all router-mediated swaps. Depending on the pool's purpose (e.g., institutional-only pools, KYC-gated pools, pools restricted to specific market makers), this allows unauthorized parties to extract value from the pool at oracle-anchored prices, bypassing the intended access boundary. The impact is a broken core pool invariant: the configured guard does not gate the actual economic actor.

### Likelihood Explanation
The likelihood is high. The `MetricOmmSimpleRouter` is the standard, documented periphery entry point for swaps. Any pool admin who wants allowlisted users to be able to use the router must allowlist the router address. The admin has no other option: the extension provides no mechanism to pass or verify the original user's identity through the router. The moment the router is allowlisted, the allowlist is open to all users. This is not a misconfiguration that requires unusual behavior — it is the natural and expected setup for any pool that uses both the allowlist extension and the standard router.

### Recommendation
The `SwapAllowlistExtension` should verify the identity of the actual economic actor, not the immediate caller of `pool.swap()`. Two viable approaches:

1. **Extension-data attestation**: Require the router to embed the original `msg.sender` in `extensionData` and have the extension verify it (e.g., via a trusted forwarder signature or a registry of trusted routers that attest the caller).
2. **Separate router-aware allowlist**: Add a second mapping `allowedSwapperViaRouter` that the extension checks when `sender` is a known router, falling back to verifying the user identity from `extensionData`.

At minimum, document that allowlisting the router address is equivalent to `setAllowAllSwappers(pool, true)` and that the extension cannot enforce per-user restrictions for router-mediated swaps.

### Proof of Concept
```
Setup:
  - Pool deployed with SwapAllowlistExtension
  - Admin calls setAllowedToSwap(pool, alice, true)       // alice is allowlisted
  - Admin calls setAllowedToSwap(pool, router, true)      // router allowlisted so alice can use it

Attack (executed by bob, who is NOT allowlisted):
  1. bob calls router.exactInputSingle({pool: pool, ...})
  2. Router calls pool.swap(recipient=bob, ...)
     → pool.msg.sender = router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. Extension checks: allowedSwapper[pool][router] == true  ✓
  5. Swap executes; bob receives output tokens

Result:
  - bob, a non-allowlisted address, successfully swaps on the restricted pool
  - The allowlist guard is bypassed entirely
  - Direct pool call by bob (without router) would correctly revert:
      allowedSwapper[pool][bob] == false → NotAllowedToSwap ✓
  - But router-mediated call succeeds because the router is allowlisted
