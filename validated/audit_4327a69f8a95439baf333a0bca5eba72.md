### Title
`FeeEstimation.overall_fee` Includes L2 Gas Cost But `l2_gas_consumed` Is Absent, Making the Returned Fee Breakdown Inconsistent with Its Own Description - (`File: crates/apollo_rpc_execution/src/objects.rs`)

### Summary

The `FeeEstimation` struct returned by `starknet_estimateFee` and `starknet_simulateTransactions` exposes `l2_gas_price` but omits `l2_gas_consumed`. The `overall_fee` field is populated from `receipt.fee`, which for V3 (`AllResources`) transactions includes L2 gas costs. However, the struct's own doc-comment and the OpenRPC spec both declare that `overall_fee` equals `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price` — a formula that excludes L2 gas entirely. Any client that reconstructs or verifies the fee using the documented formula will compute a value that is systematically lower than the actual fee returned, and because `l2_gas_consumed` is absent, the client has no way to close the gap or set accurate L2 gas resource bounds.

### Finding Description

`tx_execution_output_to_fee_estimation` in `crates/apollo_rpc_execution/src/objects.rs` builds the `FeeEstimation` response as follows:

```rust
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),       // L1 gas only
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(), // L1 data gas only
    l1_data_gas_price,
    l2_gas_price,                                   // price present …
    overall_fee: tx_execution_output.execution_info.receipt.fee, // … but includes L2 cost
    unit: tx_execution_output.price_unit,
})
```

`receipt.fee` is the actual fee charged by the blockifier, which for a V3 transaction with `AllResources` bounds equals:

```
l1_gas * l1_gas_price + l1_data_gas * l1_data_gas_price + l2_gas * l2_gas_price
```

The struct's own doc-comment states:

```
/// The total amount of fee. This is equal to:
/// gas_consumed * gas_price + data_gas_consumed * data_gas_price.
```

The OpenRPC spec (`crates/apollo_rpc/resources/V0_8/starknet_api_openrpc.json`) repeats the same claim:

> "equals to gas_consumed\*gas_price + data_gas_consumed\*data_gas_price"

Neither the struct nor the spec mentions L2 gas. The `l2_gas_consumed` field is explicitly deferred by a TODO:

```rust
// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is
// close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.
```

This creates a three-way inconsistency:

| What the spec says | What the struct exposes | What `overall_fee` actually contains |
|---|---|---|
| `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price` | `gas_consumed`, `data_gas_consumed`, `l2_gas_price` (no `l2_gas_consumed`) | L1 gas + L1 data gas + **L2 gas** |

A client that applies the documented formula to verify the returned `overall_fee` will always compute a value that is lower than the actual fee by exactly `l2_gas * l2_gas_price`. Because `l2_gas_consumed` is absent from the response, the client cannot determine the L2 gas amount, cannot verify the fee, and cannot construct accurate `l2_gas` resource bounds for a follow-up V3 transaction.

### Impact Explanation

This matches the allowed impact: **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

The `overall_fee` field carries an authoritative-looking description that is factually wrong for any V3 transaction that consumes L2 gas. A client that trusts the documented formula and uses it to set `l2_gas.max_amount` in a subsequent transaction will either:

- Set the bound to zero (formula gives no L2 component) → transaction reverts with `MaxGasAmountExceeded` for L2 gas, or
- Overshoot the bound arbitrarily to compensate for the opaque discrepancy → economic overpayment analogous to the external bug.

The `l2_gas_price` field is present in the response, making the omission of `l2_gas_consumed` especially misleading: the price is exposed but the quantity is hidden, so the fee cannot be reconstructed.

### Likelihood Explanation

Every V3 (`AllResources`) transaction that executes Cairo code consumes L2 gas. Since Starknet 0.13.3, all new transactions use `AllResources` bounds. Any client calling `starknet_estimateFee` or `starknet_simulateTransactions` for a V3 transaction and relying on the documented formula will encounter this discrepancy on every call.

### Recommendation

1. Add `l2_gas_consumed: Felt` to `FeeEstimation` and populate it from `gas_vector.l2_gas.0` in `tx_execution_output_to_fee_estimation`.
2. Update the doc-comment and the OpenRPC spec description of `overall_fee` to include the L2 gas term: `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price + l2_gas_consumed * l2_gas_price`.
3. Remove or resolve the TODO at line 104–105 of `crates/apollo_rpc_execution/src/objects.rs`.

```rust
pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}
```
