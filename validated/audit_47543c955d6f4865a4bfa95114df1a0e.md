### Title
`promise_batch_action_transfer_to_gas_key` Host Function Uses Borsh Key Length Instead of Trie Key Length for Exec-Fee Calculation, Causing ~21× Overcharge for ML-DSA-65 Gas Keys — (`runtime/near-vm-runner/src/logic/logic.rs`, `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`)

---

### Summary

The host functions `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, and `promise_batch_action_add_gas_key_with_function_call` compute the **exec fee** using the raw guest-supplied `public_key_len` (the borsh-encoded wire length) instead of `PublicKey::trie_id_len()` (the on-trie identifier length). For ED25519 and SECP256K1 keys the two values are identical, so no divergence exists. For ML-DSA-65 keys they diverge dramatically: borsh length = 1954 bytes, trie-id length = 34 bytes. The runtime (transaction) path already calls `trie_id_len()` correctly; the host-function path does not, breaking the fee-parity invariant that the protocol documentation explicitly requires.

---

### Finding Description

`PublicKey::len()` returns the borsh-encoded wire size (33 / 65 / **1954** for ED25519 / SECP256K1 / ML-DSA-65). `PublicKey::trie_id_len()` returns the on-trie identifier size (33 / 65 / **34**), because ML-DSA-65 access keys are stored in the trie under a SHA3-256 hash, not the full 1953-byte key. [1](#0-0) 

The exec-fee helper `gas_key_transfer_exec_fee` is documented as "Based on the access key trie key length + estimated value length (what the receiver needs to read/write in the trie)" and uses its `public_key_len` argument to compute `trie_key_len = access_key_key_len(account_id_len, public_key_len)`: [2](#0-1) 

In the **runtime (transaction) path**, `exec_fee` is called correctly with `action.public_key.trie_id_len()`: [3](#0-2) 

In the **host-function path** (`VMLogic`), the same helper is called with the raw `public_key_len as usize` — the borsh wire length supplied by the guest: [4](#0-3) 

The Wasmtime runner duplicates the same mistake: [5](#0-4) 

The same pattern repeats for `gas_key_add_key_exec_fee` inside `promise_batch_action_add_gas_key_with_full_access` and `promise_batch_action_add_gas_key_with_function_call`: [6](#0-5) [7](#0-6) 

For a typical account ID of 10 bytes and `AccessKey::min_gas_key_borsh_len()` ≈ 50 bytes:

| Path | `public_key_len` used | `trie_key_len` | per-byte multiplier |
|---|---|---|---|
| Runtime (tx) | 34 (trie hash) | 10 + 34 = 44 | 44 + 50 = **94** |
| Host function | 1954 (borsh) | 10 + 1954 = 1964 | 1964 + 50 = **2014** |

The exec fee is **≈ 21× larger** in the host-function path than in the transaction path for the same ML-DSA-65 key.

---

### Impact Explanation

Any smart contract that calls `promise_batch_action_transfer_to_gas_key` (or the add-gas-key variants) with an ML-DSA-65 public key will have its `used_gas` inflated by ~21× for the exec-fee component. Because `used_gas` is deducted from the contract's gas budget at the time of the host-function call, this can:

1. **Exhaust the gas budget** of the calling function, causing `GasExceeded` and reverting the entire call. Gas already burned before the failure is permanently lost.
2. **Break fee parity**: the same logical action costs ~21× more gas when initiated from a contract than from a signed transaction, violating the protocol invariant documented in `docs/architecture/how/post_quantum_signatures.md` ("Every storage-stake and trie-byte-priced fee path was updated to call `trie_id_len()`"). [8](#0-7) 

---

### Likelihood Explanation

The ML-DSA-65 key type is gated behind the `GasKeys` and ML-DSA-65 protocol features (visible in version-85/86 snapshots). Once both features are active, any unprivileged contract can trigger the overcharge by passing a borsh-encoded ML-DSA-65 public key to the host function. No special privilege is required; the guest controls `public_key_len` directly.

---

### Recommendation

Replace `public_key_len as usize` with `public_key.trie_id_len()` in all three host-function call sites. Because `public_key` is a `PublicKeyHandle` at that point, decode it before the fee calculation (or add a `trie_id_len()` method to `PublicKeyHandle`):

```rust
// Before fee calculation, decode the key:
let decoded_key = public_key.decode()?;
let send = gas_key_transfer_send_fee(&self.fees_config, sir, public_key_len as usize);
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    decoded_key.trie_id_len(),   // ← fix
);
// Use decoded_key for append_action_transfer_to_gas_key below.
```

Apply the same fix to `gas_key_add_key_exec_fee` calls in `promise_batch_action_add_gas_key_with_full_access` and `promise_batch_action_add_gas_key_with_function_call`.

---

### Proof of Concept

1. Enable ML-DSA-65 and GasKeys protocol features on a test network.
2. Deploy a contract that calls `promise_batch_action_transfer_to_gas_key` with an ML-DSA-65 public key (borsh length 1954) and attaches, say, 5 Tgas.
3. Observe that the call fails with `GasExceeded` even though the equivalent signed transaction (which uses `trie_id_len()` = 34) succeeds with the same gas budget.
4. Confirm the divergence by comparing `gas_burnt` from the host-function receipt against `gas_burnt` from the transaction receipt for the same action — the host-function path burns ~21× more gas for the exec-fee component.

    /// entry owned by this public key. For ed25519/secp256k1 this matches

    /// form without actually hashing the pubkey.

            Self::SECP256K1(_) => 1 + 64,
            Self::MLDSA65(_) => 1 + ML_DSA_65_HASH_LENGTH,

/// Exec fee for TransferToGasKey / WithdrawFromGasKey actions.
/// Based on the access key trie key length + estimated value length (what the
/// receiver needs to read/write in the trie).
pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}

### 5. Storage usage and fee plumbing

The storage-stake calculation
(`runtime/runtime/src/access_keys.rs::access_key_storage_usage`) and the
gas-key fee helpers (`gas_key_*_fee` in `runtime/runtime/src/config.rs`) use
`PublicKey::trie_id_len()` rather than `PublicKey::len()`:

- `len()` reports the borsh-encoded length (33 / 65 / 1953 across the
  three `PublicKey` variants).
- `trie_id_len()` reports the on-trie length (33 / 65 / **33**).

The two diverge only for `PublicKey::MLDSA65`. Every storage-stake and
trie-byte-priced fee path was updated to call `trie_id_len()`.
