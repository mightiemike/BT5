### Title
No User-Controlled Slippage Protection in NLP Mint/Burn — (`core/contracts/Clearinghouse.sol`, `core/contracts/interfaces/IEndpoint.sol`)

---

### Summary

Users minting or burning NLP tokens have no on-chain mechanism to specify a minimum acceptable output amount. The `oraclePriceX18` used to compute the NLP/quote conversion is appended by the sequencer and is **not** part of the user's EIP-712 signed message. There is no minimum-output guard anywhere in the mint or burn execution path, leaving users with zero price protection.

---

### Finding Description

**User-signed structs contain no price bound:**

`MintNlp` (what the user signs):
```solidity
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;   // user commits this quote
    uint64 nonce;
    // ← no minNlpAmount, no priceLimit
}
``` [1](#0-0) 

`BurnNlp` (what the user signs):
```solidity
struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;     // user commits this NLP
    uint64 nonce;
    // ← no minQuoteAmount, no priceLimit
}
``` [2](#0-1) 

**The sequencer appends the price outside the signed payload:**

```solidity
struct SignedMintNlp {
    MintNlp tx;
    bytes signature;
    int128 oraclePriceX18;        // ← sequencer-controlled, NOT signed by user
    int128[] nlpPoolRebalanceX18;
}
``` [3](#0-2) 

```solidity
struct SignedBurnNlp {
    BurnNlp tx;
    bytes signature;
    int128 oraclePriceX18;        // ← sequencer-controlled, NOT signed by user
    int128[] nlpPoolRebalanceX18;
}
``` [4](#0-3) 

**Execution uses the sequencer-supplied price with no floor/ceiling check:**

In `mintNlp`:
```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
``` [5](#0-4) 

In `burnNlp`:
```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
``` [6](#0-5) 

Neither function checks that the computed output meets any user-specified minimum. The `oraclePriceX18` is set by the sequencer in `EndpointTx.processTransactionImpl` and stored directly:
```solidity
priceX18[NLP_PRODUCT_ID] = signedTx.oraclePriceX18;
clearinghouse.mintNlp(signedTx.tx, signedTx.oraclePriceX18, ...);
``` [7](#0-6) 

---

### Impact Explanation

A user who signs `MintNlp { quoteAmount: 1000e18 }` irrevocably commits 1000 USDC. The NLP tokens received are `1000e18 / oraclePriceX18`. If the sequencer supplies an inflated `oraclePriceX18`, the user receives far fewer NLP tokens than the market rate would imply, with no on-chain recourse.

Symmetrically, a user who signs `BurnNlp { nlpAmount: X }` irrevocably commits X NLP. The quote received is `X * oraclePriceX18 - burnFee`. A deflated `oraclePriceX18` causes the user to receive far less quote than expected.

The corrupted state delta is the user's `NLP_PRODUCT_ID` balance (mint) or `QUOTE_PRODUCT_ID` balance (burn) in the `SpotEngine`, which is set to an amount the user never agreed to.

---

### Likelihood Explanation

Exploitation in its worst form requires a malicious or compromised sequencer. However, the design flaw is unconditional: **no user, under any circumstances, can specify a minimum output**. Even an honest sequencer using a stale oracle price during high NLP volatility can cause material loss. The NLP lock period (`NLP_LOCK_PERIOD = 4 days`) means a user cannot immediately exit a bad mint, compounding the exposure. [8](#0-7) 

---

### Recommendation

1. Add a `minNlpAmount` field to `MintNlp` and a `minQuoteAmount` field to `BurnNlp` so users can express their slippage tolerance as part of the signed payload.
2. In `Clearinghouse.mintNlp` and `Clearinghouse.burnNlp`, assert the computed output meets the user-specified minimum before applying balance updates.
3. Consider bounding `oraclePriceX18` against the last on-chain `priceX18[NLP_PRODUCT_ID]` by a maximum deviation percentage to limit sequencer price discretion.

---

### Proof of Concept

1. User signs `MintNlp { sender: alice, quoteAmount: 10_000e18, nonce: 0 }` when NLP oracle price is `$1.00`, expecting ~10,000 NLP.
2. Sequencer constructs `SignedMintNlp { tx: ..., signature: aliceSig, oraclePriceX18: 2e18, nlpPoolRebalanceX18: [...] }` (price doubled).
3. `Clearinghouse.mintNlp` executes: `nlpAmount = 10_000e18 / 2e18 = 5_000e18`.
4. Alice's `QUOTE_PRODUCT_ID` balance decreases by 10,000 USDC; her `NLP_PRODUCT_ID` balance increases by only 5,000 NLP — a 50% loss with no revert path.
5. The same logic applies in reverse for `burnNlp` with a deflated price.
