### Title
`SwapAllowlistExtension` gates on the immediate caller (router) instead of the economic actor, allowing any user to bypass the swap allowlist via `MetricOmmSimpleRouter` ŌĆö (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the first argument forwarded by the pool ŌĆö which equals `msg.sender` of `pool.swap()`. When a user routes through the public `MetricOmmSimpleRouter`, `msg.sender` of `pool.swap()` is the router contract, not the end user. If the pool admin allowlists the router address (the only way to permit router-mediated swaps), every user on the network can bypass the curated allowlist by routing through the router.

---

### Finding Description

**Call chain:**

```
User ŌåÆ MetricOmmSimpleRouter.exactInputSingle()
         ŌåÆ pool.swap(recipient, ..., extensionData)   // msg.sender = router
              ŌåÆ _beforeSwap(msg.sender=router, recipient, ...)
                   ŌåÆ SwapAllowlistExtension.beforeSwap(sender=router, ...)
                        ŌåÆ allowedSwapper[pool][router]  ŌåÉ checked, NOT the user
```

In `MetricOmmPool.swap`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the first argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, i.e., `allowedSwapper[pool][router]`: [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap`, the router is `msg.sender` of the pool call: [4](#0-3) 

The router is a public, permissionless contract ŌĆö any user can call it. The pool admin has two losing options:

1. **Allowlist the router** ŌåÆ every user on the network can bypass the curated allowlist by routing through the router.
2. **Do not allowlist the router** ŌåÆ individually allowlisted users cannot use the router at all, breaking the supported periphery path.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly gates on the `owner` argument (the position owner), not on `sender` (the liquidity adder), so the deposit allowlist is not affected: [5](#0-4) 

The existing test suite only exercises the direct-pool-call path (where `sender` == the allowlisted `TestCaller`), never the router-mediated path, so the bypass is untested: [6](#0-5) 

---

### Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of addresses (e.g., KYC'd counterparties, whitelisted market makers) loses that guarantee entirely once the router is allowlisted. Any unprivileged user can call `MetricOmmSimpleRouter.exactInputSingle` or `exactInput` and trade against the pool, draining LP value at oracle-derived prices that the pool admin intended to expose only to trusted parties. This is a direct loss of LP principal and a complete curation failure.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap entrypoint documented in the periphery. Any pool admin who deploys a swap-allowlisted pool and also wants to support router-mediated swaps (the normal UX path) will allowlist the router, immediately opening the bypass to all users. The trigger requires no special privilege ŌĆö any EOA can call the router.

---

### Recommendation

Gate the allowlist on the economic actor, not the immediate caller. The cleanest fix is to check the `recipient` argument (the address that receives the output tokens) or to require the extension to read a user-supplied identity from `extensionData`. Alternatively, mirror the deposit extension's design: pass a dedicated "swapper identity" field that the router populates from `msg.sender` before calling the pool, and check that field in the extension.

A minimal fix in `SwapAllowlistExtension.beforeSwap`:

```solidity
// Instead of checking `sender` (the router), decode the real user from extensionData:
function beforeSwap(
    address sender,
    address recipient,
    bool, int128, uint128, uint256, uint128, uint128,
    bytes calldata extensionData
) external view override returns (bytes4) {
    address swapper = extensionData.length >= 20
        ? abi.decode(extensionData, (address))
        : sender;
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][swapper]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

The router would then encode `msg.sender` into `extensionData` before forwarding to the pool.

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin allowlists only `alice` (allowedSwapper[pool][alice] = true)
  - Pool admin also allowlists the router (allowedSwapper[pool][router] = true)
    to permit alice to swap via the router

Attack:
  - bob (not allowlisted) calls MetricOmmSimpleRouter.exactInputSingle(
        pool, tokenIn, tokenOut, amountIn, recipient=bob, ...
    )
  - Router calls pool.swap(bob, ...) with msg.sender = router
  - Pool calls _beforeSwap(sender=router, ...)
  - SwapAllowlistExtension checks allowedSwapper[pool][router] ŌåÆ true
  - Swap executes for bob despite bob not being allowlisted

Result:
  - bob successfully trades on a curated pool
  - The allowlist policy is completely bypassed
  - LP funds are exposed to an unauthorized counterparty
```
