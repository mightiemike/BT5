### Title
Stale `digestToSubaccount` Mapping After Isolated Subaccount Closure Routes Trade Proceeds to Orphaned Subaccount — (`core/contracts/OffchainExchange.sol`)

---

### Summary

When `_tryCloseIsolatedSubaccount` closes an isolated subaccount, it clears `parentSubaccounts`, `isolatedSubaccounts`, and `isolatedSubaccountsMask`, but never clears the `digestToSubaccount[digest]` entry. This stale mapping causes `matchOrders` to silently redirect balance updates for any remaining partial fills of the associated order to the now-orphaned isolated subaccount, permanently stranding the user's trade proceeds.

---

### Finding Description

`_tryCloseIsolatedSubaccount` performs the following cleanup when `balance.amount == 0`:

```solidity
isolatedSubaccountsMask[addr] &= ~uint256(0) ^ (1 << id);
isolatedSubaccounts[parent][id] = bytes32(0);
parentSubaccounts[subaccount] = bytes32(0);
``` [1](#0-0) 

It does **not** clear `digestToSubaccount[digest]`, which was set during `createIsolatedSubaccount` to associate the order digest with the isolated subaccount. [2](#0-1) 

In `matchOrders`, before any validation or balance update, the sender is unconditionally overridden if a `digestToSubaccount` entry exists:

```solidity
if (digestToSubaccount[ordersInfo.taker.digest] != bytes32(0)) {
    taker.order.sender = digestToSubaccount[ordersInfo.taker.digest];
}
if (digestToSubaccount[ordersInfo.maker.digest] != bytes32(0)) {
    maker.order.sender = digestToSubaccount[ordersInfo.maker.digest];
}
``` [3](#0-2) 

After the isolated subaccount is closed, `parentSubaccounts[isolatedSubaccount]` is `bytes32(0)`. Any subsequent match against the stale digest routes `_updateBalances` to the orphaned isolated subaccount. Since `parentSubaccounts` is zeroed, any later call to `_tryCloseIsolatedSubaccount` on this subaccount would attempt to transfer its quote balance to `bytes32(0)` — the zero subaccount — permanently destroying those funds. [4](#0-3) 

The same stale entry also affects `createIsolatedSubaccount`: if the same order digest is submitted again after closure, the function returns the old orphaned subaccount immediately without creating a new one, so any margin transferred in the subsequent match goes to the orphaned subaccount. [5](#0-4) 

---

### Impact Explanation

Trade proceeds from the remaining fill of a partially-filled isolated order are credited to the orphaned isolated subaccount rather than the parent subaccount. Because `parentSubaccounts[orphanedSubaccount] == bytes32(0)`, the only recovery path (`_tryCloseIsolatedSubaccount`) would transfer the balance to the zero subaccount, destroying the funds. The parent subaccount receives nothing. [6](#0-5) 

---

### Likelihood Explanation

The scenario requires: (1) an isolated order to be partially filled, (2) the isolated subaccount's position to be closed to zero (triggering `_tryCloseIsolatedSubaccount`), and (3) the sequencer to subsequently match the remaining portion of the same order. Steps 1 and 2 are routine in normal perpetual trading. Step 3 follows naturally if the sequencer does not track on-chain isolated subaccount closure state before submitting `MatchOrders`. The sequencer is the only party that submits `MatchOrders`, but it is an unprivileged path from the perspective of the on-chain invariant — the contract itself provides no guard against this. [7](#0-6) 

---

### Recommendation

In `_tryCloseIsolatedSubaccount`, also delete the `digestToSubaccount` entry for every digest that maps to the subaccount being closed. Alternatively, add a guard in `matchOrders` that checks whether the isolated subaccount returned by `digestToSubaccount` is still active (i.e., `parentSubaccounts[subaccount] != bytes32(0)`) before overriding `order.sender`. [8](#0-7) 

---

### Proof of Concept

**State setup:**
- User submits `CreateIsolatedSubaccount` with order digest `D` for product `P`.
- `digestToSubaccount[D] = isolatedSubaccount` is written.
- `parentSubaccounts[isolatedSubaccount] = parentSubaccount`.

**Partial fill:**
- Sequencer submits `MatchOrders` for 50% of order `D`.
- `matchOrders` overrides `taker.order.sender = isolatedSubaccount` via `digestToSubaccount[D]`.
- Balance updates go to `isolatedSubaccount`. ✓

**Isolated subaccount closure:**
- User closes the position; `balance.amount == 0`.
- `_tryCloseIsolatedSubaccount` is called:
  - `parentSubaccounts[isolatedSubaccount] = bytes32(0)` ✓
  - `isolatedSubaccounts[parent][id] = bytes32(0)` ✓
  - `digestToSubaccount[D]` **NOT cleared** ✗

**Stale mapping exploited:**
- Sequencer submits `MatchOrders` for the remaining 50% of order `D`.
- `matchOrders` reads `digestToSubaccount[D] = isolatedSubaccount` (stale) and overrides `taker.order.sender`.
- `_updateBalances` credits the orphaned `isolatedSubaccount`.
- `parentSubaccount` receives nothing.
- If `_tryCloseIsolatedSubaccount` is later called on the orphaned subaccount, `parent = bytes32(0)` and the quote balance is transferred to the zero subaccount — funds are permanently lost. [9](#0-8)

    function _tryCloseIsolatedSubaccount(bytes32 subaccount) internal {
        uint32 productId = RiskHelper.getIsolatedProductId(subaccount);
        if (productId == 0) {
            return;
        }
        IPerpEngine.Balance memory balance = perpEngine.getBalance(
            productId,
            subaccount
        );
        if (balance.amount == 0) {
            uint8 id = RiskHelper.getIsolatedId(subaccount);
            address addr = address(uint160(bytes20(subaccount)));
            bytes32 parent = parentSubaccounts[subaccount];
            if (balance.vQuoteBalance != 0) {
                perpEngine.updateBalance(
                    productId,
                    subaccount,
                    0,
                    -balance.vQuoteBalance
                );
                perpEngine.updateBalance(
                    productId,
                    parent,
                    0,
                    balance.vQuoteBalance
                );
            }
            int128 quoteBalance = spotEngine
                .getBalance(QUOTE_PRODUCT_ID, subaccount)
                .amount;
            if (quoteBalance != 0) {
                spotEngine.updateBalance(
                    QUOTE_PRODUCT_ID,
                    subaccount,
                    -quoteBalance
                );
                spotEngine.updateBalance(
                    QUOTE_PRODUCT_ID,
                    parent,
                    quoteBalance
                );
            }
            isolatedSubaccountsMask[addr] &= ~uint256(0) ^ (1 << id);
            isolatedSubaccounts[parent][id] = bytes32(0);
            parentSubaccounts[subaccount] = bytes32(0);

            emit CloseIsolatedSubaccount(subaccount, parent);
        }
    }
