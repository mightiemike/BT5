### Title
Duplicate Bonus Reward Drain via Repeated Position ID in Batch Claim — (`x/tieredrewards/keeper/msg_server.go`)

### Summary

`MsgClaimTierRewards` accepts a slice of position IDs and processes them in a loop. All positions are loaded from the store **before** any claim is executed. If the same position ID appears more than once in `msg.PositionIds`, the stale in-memory copy is used for every duplicate iteration, causing the bonus reward computation and `SendCoinsFromModuleToAccount` transfer to execute multiple times for the same accrual window, draining the `RewardsPoolName` module account.

### Finding Description

`ClaimTierRewards` in `x/tieredrewards/keeper/msg_server.go` (lines 429–468) first loads every requested position into a `[]types.PositionState` slice, then passes the entire slice to `claimRewardsAndUpdatesPositions`. [1](#0-0) 

There is no deduplication of `msg.PositionIds`. If the same ID appears at index 0 and index 1, both slice entries hold the identical original state (`LastBonusAccrual = T0`, `LastEventSeq = S0`).

Inside `claimRewardsAndUpdatesPositions`, the loop processes each slice entry independently: [2](#0-1) 

For **iteration 0** (first copy of position P):
- `processEventsAndClaimBonus` computes the bonus for the segment `[T0, blockTime]`, calls `bankKeeper.SendCoinsFromModuleToAccount` to transfer bonus `B` to the owner, and updates the in-memory `pos.LastBonusAccrual` to `blockTime`.
- `setPosition` persists the updated checkpoint to the store. [3](#0-2) 

For **iteration 1** (second copy of position P, still holding `LastBonusAccrual = T0`):
- `processEventsAndClaimBonus` is called again with the stale in-memory copy. `segmentStart` is still `T0`, so the tail-segment computation `computeSegmentBonus(pos, tier, T0, blockTime, currentRate)` produces the same bonus `B`.
- `bankKeeper.SendCoinsFromModuleToAccount` fires again, transferring another `B` to the owner.
- `setPosition` overwrites the store with the (now-identical) updated checkpoint.

The net result: the owner receives `N × B` bonus coins for a single accrual window by passing the same position ID `N` times. Base rewards are not doubled because `WithdrawDelegationRewards` zeroes out the distribution record on the first call, but the bonus pool transfer has no such guard.

### Impact Explanation

The `RewardsPoolName` module account balance is drained by `(N-1) × B` extra tokens per transaction. An attacker with any valid delegated position can repeat this indefinitely (each transaction correctly advances `LastBonusAccrual`, so the next transaction's `B` is smaller, but the attacker can wait for more time to pass and repeat). The corrupted invariant is: **bonus coins paid out ≤ bonus accrued for the position's accrual window**.

### Likelihood Explanation

The entry path is a standard, unprivileged `MsgClaimTierRewards` transaction. No special role, key, or social engineering is required. The attacker only needs to own one delegated tiered-rewards position. The message format accepts a repeated `position_ids` field, and no deduplication or uniqueness check is enforced before processing.

### Recommendation

Before calling `claimRewardsAndUpdatesPositions`, deduplicate `msg.PositionIds` and return an error (or silently deduplicate) if any ID appears more than once. The simplest fix is to add a uniqueness check inside `MsgClaimTierRewards.Validate()` or at the top of the `ClaimTierRewards` handler:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, id := range msg.PositionIds {
    if _, ok := seen[id]; ok {
        return nil, sdkerrors.ErrInvalidRequest.Wrapf("duplicate position id %d", id)
    }
    seen[id] = struct{}{}
}
```

### Proof of Concept

1. User creates a delegated tiered-rewards position with ID `42`. Bonus accrues for period `[T0, T1]` = bonus `B`.
2. User submits `MsgClaimTierRewards{Owner: user, PositionIds: [42, 42, 42]}`.
3. `ClaimTierRewards` loads position `42` three times into `positions[0..2]`, all with `LastBonusAccrual = T0`.
4. `claimRewardsAndUpdatesPositions` iterates:
   - i=0: computes bonus `B`, sends `B` to user, persists `LastBonusAccrual = T1`.
   - i=1: stale copy still has `LastBonusAccrual = T0`; computes bonus `B` again, sends `B` to user again.
   - i=2: same; sends `B` a third time.
5. User receives `3B` instead of `B`. The `RewardsPoolName` account is short by `2B`.
6. The position's `LastBonusAccrual` is correctly set to `T1` after the transaction, so the next honest claim yields 0 for the same window — but the attacker already extracted the surplus. [4](#0-3) [2](#0-1) [5](#0-4)

func (ms msgServer) ClaimTierRewards(ctx context.Context, msg *types.MsgClaimTierRewards) (*types.MsgClaimTierRewardsResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	positions := make([]types.PositionState, 0, len(msg.PositionIds))
	for _, posId := range msg.PositionIds {
		pos, err := ms.getPositionState(ctx, posId)
		if err != nil {
			return nil, err
		}

		if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
			return nil, err
		}

		positions = append(positions, pos)
	}

	totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventTierRewardsClaimed{
		Owner:        msg.Owner,
		PositionIds:  msg.PositionIds,
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
	}); err != nil {
		return nil, err
	}

	return &types.MsgClaimTierRewardsResponse{
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
		PositionIds:  msg.PositionIds,
	}, nil
}

func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
}
