### Title
Linked Signer Can Unilaterally Replace Itself With an Attacker-Controlled Address via `LinkSigner` Transaction ‚Äî (File: `core/contracts/EndpointTx.sol`)

---

### Summary
The sequencer-path handler for `LinkSigner` transactions in `EndpointTx.processTransactionImpl()` passes `allowLinkedSigner = true` to `validateSignedTx`. This means the **current linked signer** ‚Äî not just the subaccount owner ‚Äî can sign a `LinkSigner` transaction that overwrites `linkedSigners[sender]` with an arbitrary attacker-controlled address. Once replaced, the attacker's address inherits full signing authority over the victim's subaccount and can sign withdrawals, order cancellations, and other privileged operations.

---

### Finding Description

In `EndpointTx.processTransactionImpl()`, the `LinkSigner` branch is:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which passes the current linked signer to the verifier as an accepted signer:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

This means the verifier accepts a signature from **either** the subaccount owner **or** the current linked signer. Because the handler then unconditionally writes `linkedSigners[signedTx.tx.sender] = <new address>`, the current linked signer can sign a `LinkSigner` transaction that replaces itself with any address.

By contrast, the slow-mode path for the same transaction type correctly enforces that only the subaccount owner (the address embedded in the `bytes32` sender) can submit it:

```solidity
validateSender(txn.sender, sender);   // msg.sender must equal address(subaccount)
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [3](#0-2) 

The sequencer path has no equivalent ownership check, creating an asymmetry that allows the linked signer to mutate the `linkedSigners` mapping without the subaccount owner's consent.

The `linkedSigners` mapping is stored in `EndpointStorage`:

```solidity
mapping(bytes32 => address) internal linkedSigners;
``` [4](#0-3) 

---

### Impact Explanation

Once the attacker's address is installed as the linked signer, it can sign any transaction that accepts `allowLinkedSigner = true`, including:

- `WithdrawCollateral` / `WithdrawCollateralV2` ‚Äî drain all collateral from the subaccount
- `TransferQuote` ‚Äî transfer quote balance to an attacker-controlled subaccount
- Further `LinkSigner` transactions ‚Äî permanently lock the owner out

The corrupted state delta is `linkedSigners[victimSubaccount]`, which transitions from a trusted hot-wallet address to an attacker-controlled address. All subsequent signed transactions from the attacker are accepted as legitimate by `Verifier.validateSignature`. The victim's on-chain balances in `SpotEngine` and `PerpEngine` are fully at risk.

---

### Likelihood Explanation

Any user who has ever set a linked signer is exposed. The attack requires only that the linked signer (Bob) is malicious or compromised. Bob submits a `LinkSigner` transaction signed by himself, naming Charlie as the new signer. The sequencer processes it because the signature is valid and the nonce is correct. No admin cooperation, no governance capture, and no leaked private keys beyond the already-trusted linked signer are required. The attack is a single transaction.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in the sequencer path, matching the invariant enforced by the slow-mode path:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
-   true
+   false   // only the subaccount owner may change the linked signer
);
```

This ensures that only the private key corresponding to the address embedded in `signedTx.tx.sender` can authorize a linked-signer change, consistent with the slow-mode path's `validateSender` check.

---

### Proof of Concept

1. Alice owns subaccount `aliceSub` (address `0xAlice` embedded in bytes32). She sets Bob (`0xBob`) as her linked signer via a slow-mode `LinkSigner` transaction. State: `linkedSigners[aliceSub] = 0xBob`.

2. Bob constructs a `SignedLinkSigner` transaction:
   - `sender = aliceSub`
   - `signer = bytes32(uint256(uint160(0xAttacker)))`
   - `nonce = currentNonce(0xAlice)`
   - Signs with Bob's private key (`0xBob`)

3. Bob submits this transaction to the sequencer. The sequencer calls `processTransactionImpl`. `validateSignedTx(..., true)` passes because `getLinkedSigner(aliceSub) == 0xBob` and the signature is valid.

4. `linkedSigners[aliceSub]` is overwritten with `0xAttacker`. State: `linkedSigners[aliceSub] = 0xAttacker`.

5. The attacker signs a `WithdrawCollateralV2` transaction for `aliceSub`. `validateSignedTx(..., true)` passes because `getLinkedSigner(aliceSub) == 0xAttacker`. All of Alice's collateral is transferred to the attacker's address.

Alice's funds are lost. The broken invariant is: **`linkedSigners[subaccount]` must only be writable by the address whose private key corresponds to the address embedded in the subaccount `bytes32`.** [1](#0-0) [5](#0-4)
