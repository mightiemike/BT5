### Title
Governance-Updatable Tier Parameters Break Existing Position Invariants Without Migrating State - (File: `x/tieredrewards/keeper/tier.go`)

---

### Summary
The `SetTier` function in the tieredrewards keeper allows governance to overwrite tier parameters (e.g., `MinLockAmount`, `ExitDuration`) for a tier that already has active positions, without validating whether those existing positions remain consistent with the new parameters. This is the direct analog of the Gearbox `creditManager` parameter-upgrade bug: mutable protocol parameters retroactively invalidate existing user positions, trapping funds or extending lock periods without user consent.

---

### Finding Description

`SetTier` is the write path for tier configuration and is explicitly documented as being used by governance messages, genesis, and chain upgrades:

```go
// SetTier writes a tier after validation. Used by governance messages, genesis, and chain upgrades.
func (k Keeper) SetTier(ctx context.Context, tier types.Tier) error {
    if err := tier.Validate(); err != nil {
        return err
    }
    if err := k.Tiers.Set(ctx, tier.Id, tier); err != nil { ... }
    return nil
}
``` [1](#0-0) 

The validation performed here (`tier.Validate()`) checks only the internal consistency of the new tier struct — it does **not** check whether any of the currently active positions for that tier would still satisfy the new parameters.

By contrast, `deleteTier` correctly guards against active positions:

```go
func (k Keeper) deleteTier(ctx context.Context, tierId uint32) error {
    hasPositions, err := k.hasPositionsForTier(ctx, tierId)
    ...
    if hasPositions {
        return types.ErrTierHasActivePositions
    }
``` [2](#0-1) 

This asymmetry is the root cause: deletion is guarded, but **update is not**.

The broken invariant surfaces concretely in `ExitTierWithDelegation`. When a user attempts a partial exit, the handler fetches the **current** tier (post-governance-update) and enforces the **new** `MinLockAmount` against the remaining position balance:

```go
tier, err := ms.getTier(ctx, pos.TierId)
...
if !tier.MeetsMinLockRequirement(remainingPositionAmount) {
    return nil, errorsmod.Wrapf(types.ErrMinLockAmountNotMet,
        "remaining amount %s is below tier minimum %s", remainingPositionAmount, tier.MinLockAmount)
}
``` [3](#0-2) 

A position that was created when `MinLockAmount` was, say, 1,000 CRO, and holds 1,500 CRO, would be unable to partially exit after governance raises `MinLockAmount` to 2,000 CRO — the remaining amount after any partial withdrawal would always be below the new minimum, blocking the partial-exit path entirely.

Similarly, `TriggerExitFromTier` reads `tier.ExitDuration` at trigger time:

```go
tier, err := ms.getTier(ctx, pos.TierId)
...
pos.TriggerExit(sdkCtx.BlockTime(), tier.ExitDuration)
``` [4](#0-3) 

If governance increases `ExitDuration` after a user entered the tier but before they trigger exit, the user faces a longer unbonding period than the one they agreed to at position creation.

---

### Impact Explanation

Two concrete fund-level impacts:

1. **Partial-exit path permanently blocked**: If governance raises `MinLockAmount` above a position's current balance, `ExitTierWithDelegation` will always revert with `ErrMinLockAmountNotMet` for any partial amount. The user's only remaining path is a full exit (`ExitWithFullDelegation`), which may not be what they want and forces them to lose their tier position entirely. The delegation (and thus the staked tokens) is locked inside the position's sub-delegator account until a full exit is performed.

2. **Lock period extended without consent**: If governance raises `ExitDuration`, users who have not yet triggered exit will be subject to the new, longer duration. Their tokens remain locked for longer than the terms under which they entered the tier.

Both impacts directly affect user-controlled staked funds (`basecro`/`carson`) held in position delegator accounts.

---

### Likelihood Explanation

The entry path is a standard Cosmos SDK governance proposal, submittable by any token holder with sufficient deposit. The governance module is the `authority` for parameter-update messages throughout this codebase (as confirmed in `x/inflation/keeper/msg_server.go`): [5](#0-4) 

A governance proposal to tighten tier parameters (e.g., raise `MinLockAmount` to increase protocol security or adjust economic incentives) is a plausible, non-adversarial action that would inadvertently trigger this bug for all existing positions in the affected tier. The likelihood is moderate: it requires a governance vote to pass, but the impact is automatic and affects all existing positions in the tier once the proposal executes.

---

### Recommendation

`SetTier` must validate that the new parameters do not break existing positions before committing the update. Specifically:

1. If `MinLockAmount` is being increased, iterate all active positions for the tier and verify each position's current delegation amount meets the new minimum. Reject the update (or require a migration) if any position would be invalidated.
2. If `ExitDuration` is being changed, consider whether the new duration should apply only to positions created after the update, not retroactively to existing ones (store `ExitDuration` on the position at creation time, not read it from the tier at trigger time).
3. Mirror the guard already present in `deleteTier`: before applying a parameter change that tightens constraints, check `hasPositionsForTier` and either reject or require explicit migration.

---

### Proof of Concept

**Setup**: Tier 1 is created with `MinLockAmount = 1000 CRO`. Alice creates a position with 1500 CRO via `MsgLockTier`.

**Governance action**: A governance proposal is submitted and passes, calling `SetTier` with Tier 1's `MinLockAmount` raised to 2000 CRO. `SetTier` succeeds because it only validates the new tier struct — it does not check Alice's existing 1500 CRO position. [6](#0-5) 

**Alice attempts partial exit**: Alice submits `MsgExitTierWithDelegation` with `Amount = 200 CRO`. After the transfer, her remaining position would be 1300 CRO. The handler fetches the current tier (now with `MinLockAmount = 2000 CRO`) and evaluates:

```
tier.MeetsMinLockRequirement(1300 CRO)  →  false  →  ErrMinLockAmountNotMet
``` [7](#0-6) 

Alice cannot partially exit. Any partial amount she tries to withdraw leaves a remainder below 2000 CRO. She is forced into a full exit or permanently locked in the tier, with her 1500 CRO staked tokens inaccessible via the partial-exit path she relied upon when entering the tier.
