### Title
Linked Signer Can Overwrite Its Own Authorization — (`EndpointTx.sol`)

### Summary
In the fast-mode `LinkSigner` transaction handler, `validateSignedTx` is called with `allowLinkedSigner = true`. This permits the **current linked signer** of a subaccount to sign and submit a new `LinkSigner` transaction, replacing the linked signer mapping with any attacker-controlled address — without the subaccount owner's knowledge or consent.

### Finding Description
In `EndpointTx.sol`, the fast-mode dispatch for `TransactionType.LinkSigner` is:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` accepts a valid signature from **either** the subaccount owner **or** the current linked signer:

```solidity
verifier.validateCompactSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

The `linkedSigners` mapping is the sole authorization record for which external key may act on behalf of a subaccount:

```solidity
return
    RiskHelper.isIsolatedSubaccount(subaccount)
        ? linkedSigners[IOffchainExchange(offchainExchange).getParentSubaccount(subaccount)]
        : linkedSigners[subaccount];
``` [3](#0-2) 

Because the current linked signer is permitted to sign a `LinkSigner` transaction, it can overwrite `linkedSigners[subaccount]` with any address it chooses — including an attacker-controlled key — without the subaccount owner ever signing anything.

The slow-mode path for `LinkSigner` correctly validates only the subaccount owner via `validateSender(txn.sender, sender)` and does **not** pass `allowLinkedSigner`:

```solidity
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

This inconsistency between fast-mode and slow-mode is the root cause.

### Impact Explanation
A compromised or malicious linked signer (e.g., a trading bot key, a third-party service key, or a key leaked from a hot wallet) can:

1. Craft a `SignedLinkSigner` transaction setting `signer` to an attacker-controlled address.
2. Submit it through the sequencer's fast-mode path.
3. The `linkedSigners[victim_subaccount]` mapping is now permanently overwritten.
4. The attacker's new key can sign `WithdrawCollateral`, `TransferQuote`, or any other `allowLinkedSigner = true` transaction, draining the subaccount's collateral.

The corrupted state is `linkedSigners[subaccount]` — the exact authorization record that gates all fast-mode transaction signing for the victim subaccount.

**Impact: High** — full subaccount takeover and collateral theft.

### Likelihood Explanation
Any entity that has ever been granted linked-signer status (e.g., a market-making bot, a frontend session key, a third-party integration) can exploit this. Key compromise of a linked signer is a realistic threat model. The attack requires only a single valid fast-mode transaction submission — no special privileges, no governance capture, no sequencer compromise.

**Likelihood: High** — reachable by any current or past linked signer with a valid nonce.

### Recommendation
Change the `allowLinkedSigner` flag to `false` for the fast-mode `LinkSigner` handler, so that only the subaccount owner's key can authorize a change to the linked signer:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only subaccount owner may change linked signer
);
```

This aligns the fast-mode path with the slow-mode path, which already enforces owner-only authorization for this operation.

### Proof of Concept
1. Alice owns subaccount `0xAlice000...` and has set `linkedSigners[0xAlice000...] = botKey`.
2. `botKey` is compromised by an attacker.
3. Attacker uses `botKey` to sign a `SignedLinkSigner` transaction: `{ sender: 0xAlice000..., signer: attackerKey, nonce: N }`.
4. Attacker submits this via the sequencer fast-mode path.
5. `validateSignedTx(..., true)` recovers `botKey` from the signature, matches it against `getLinkedSigner(0xAlice000...)` → passes.
6. `linkedSigners[0xAlice000...] = attackerKey` is written.
7. Attacker now signs a `SignedWithdrawCollateral` transaction with `attackerKey`, draining Alice's collateral.
