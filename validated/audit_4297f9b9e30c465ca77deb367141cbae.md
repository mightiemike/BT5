### Title
`revert_state_diff` unconditionally deletes `stateless_compiled_class_hash_v2` entries for migrated (not declared) classes, causing `MissingCompiledClassHashV2` errors on subsequent execution - (File: `crates/apollo_storage/src/state/mod.rs`)

### Summary

`revert_state_diff()` calls `delete_compiled_class_hashes_v2()` for every class hash present in the reverted block's `class_hash_to_compiled_class_hash` state diff. That set includes both newly-declared classes **and** classes that were only migrated (V1 Poseidon → V2 Blake CASM hash) in the reverted block. The `stateless_compiled_class_hash_v2` table is a stateless, non-block-versioned store whose entries are written once at declaration time. Deleting those entries during a migration-block revert leaves the class permanently declared but without a V2 hash, causing every subsequent call to `get_compiled_class_hash_v2()` for that class to return `StateError::MissingCompiledClassHashV2`, which propagates through `should_migrate()` → `CasmHashMigrationData::from_state()` → `get_tx_weights()` and aborts transaction execution.

### Finding Description

**The stale-deletion site** is `delete_compiled_class_hashes_v2`:

```rust
fn delete_compiled_class_hashes_v2<'env>(
    txn: &'env DbTransaction<'env, RW>,
    class_hashes: impl Iterator<Item = &'env ClassHash>,
    compiled_class_hash_v2_table: &'env StatelessCompiledClassHashV2Table<'env>,
) -> StorageResult<()> {
    for class_hash in class_hashes {
        compiled_class_hash_v2_table.delete(txn, class_hash)?;   // unconditional
    }
    Ok(())
}
``` [1](#0-0) 

It is called from `revert_state_diff()` with the full key-set of `class_hash_to_compiled_class_hash`:

```rust
delete_compiled_class_hashes_v2(
    inner_txn,
    thin_state_diff.class_hash_to_compiled_class_hash.keys(),
    &compiled_class_hash_v2_table,
)?;
``` [2](#0-1) 

That key-set contains two disjoint populations:

1. **Newly declared classes** – their V2 hash was written to `stateless_compiled_class_hash_v2` in the same block being reverted. Deleting it here is correct.
2. **Migrated classes** – declared in an *earlier* block (their V2 hash was written then), but their `compiled_class_hash` entry was updated from V1 to V2 in the block being reverted. Deleting the V2 hash here is **wrong**: the declaration still exists in an earlier block.

The migration path that populates `class_hash_to_compiled_class_hash` for migrated classes (without re-writing `stateless_compiled_class_hash_v2`) is visible in `native_blockifier`:

```rust
// Add the migrated class hash to the state diff.
for (class_hash, compiled_class_hash) in migrated_class_hash_to_compiled_class_hash {
    thin_state_diff
        .class_hash_to_compiled_class_hash
        .insert(class_hash, compiled_class_hash);
}
``` [3](#0-2) 

Notice that `set_executable_class_hash_v2()` is called only for *newly declared* classes, not for migrated ones:

```rust
for (class_hash, casm_contract_class, compiled_class_hash_v2) in undeclared_casm_contracts {
    append_txn = append_txn.append_casm(&class_hash, &casm_contract_class)?;
    append_txn =
        append_txn.set_executable_class_hash_v2(&class_hash, compiled_class_hash_v2)?;
}
``` [4](#0-3) 

The correct pattern already exists for `declared_classes_block_table` – it guards deletion with a block-number check:

```rust
if class_block_entry == block_number {
    declared_classes_block_table.delete(txn, class_hash)?;
    deleted_data.push(*class_hash);
}
``` [5](#0-4) 

`delete_compiled_class_hashes_v2` has no such guard.

**Execution failure chain after the bad revert:**

`should_migrate()` calls `get_compiled_class_hash_v2()`:

```rust
let compiled_class_hash_v2 = state_reader.get_compiled_class_hash_v2(
    class_hash,
    &state_reader.get_compiled_class(class_hash)?,
)?;
``` [6](#0-5) 

`ApolloReader::get_compiled_class_hash_v2()` reads from `stateless_compiled_class_hash_v2` and returns `MissingCompiledClassHashV2` when the entry is absent:

```rust

    &self,
    class_hash: ClassHash,
    _compiled_class: &RunnableCompiledClass,
) -> StateResult<CompiledClassHash> {
    self.read_compiled_class_hash_v2(class_hash)?
        .ok_or(StateError::MissingCompiledClassHashV2(class_hash))
}
``` [7](#0-6) 

That error propagates through `CasmHashMigrationData::from_state()`:

And through `get_tx_weights()` into the block-execution pipeline, aborting every transaction that touches the affected class. [9](#0-8)

### Impact Explanation

After reverting any block that contains CASM-hash migrations, the `stateless_compiled_class_hash_v2` table permanently loses V2 entries for the migrated classes. Any subsequent transaction that executes one of those classes will fail with `MissingCompiledClassHashV2` inside `get_tx_weights()`. This produces a wrong execution result (failed transaction) for valid, accepted input — matching the critical impact: **"Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input."**

The in-memory `compiled_class_hash_v2_cache` inside `ContractClassManager` may mask the issue until the node restarts or the LRU cache evicts the entry, at which point the storage read returns `None` and the error manifests.

### Likelihood Explanation

- `enable_casm_hash_migration = true` is the default for all Starknet versions ≥ 0.14.1.
- Block reverts are a normal, unprivileged operation during state synchronisation (e.g., after a micro-fork or reorg).
- No special transaction crafting is required; any ordinary invoke that executes a previously migrated class triggers the failure.
- The window between the revert and the next node restart is the only period where the in-memory cache provides a false safety net.

### Recommendation

Mirror the block-number guard already used in `delete_declared_classes_block`. In `delete_compiled_class_hashes_v2`, only delete the V2 hash when the class was declared in the block being reverted (i.e., `declared_classes_block_table.get(txn, class_hash)? == Some(block_number)`). Alternatively, separate the "declared in this block" and "migrated in this block" key-sets before calling the two deletion helpers, and pass only the declared-in-this-block set to `delete_compiled_class_hashes_v2`.

### Proof of Concept

1. Block N: declare class C → `stateless_compiled_class_hash_v2[C] = H_v2`.
2. Block M > N: execute a transaction touching C with `enable_casm_hash_migration = true` → migration fires, `compiled_class_hash[C, M] = H_v2`; state diff records `{C: H_v2}`.
3. `revert_state_diff(M)` is called (e.g., during state sync after a reorg).
4. `delete_compiled_class_hashes_v2` iterates `{C}` and deletes `stateless_compiled_class_hash_v2[C]`.
5. Block M′: submit any invoke that calls into class C.
6. `should_migrate(state, C)` calls `get_compiled_class_hash_v2(C)` → `get_executable_class_hash_v2(C)` returns `None` → `MissingCompiledClassHashV2(C)`.
7. `CasmHashMigrationData::from_state()` returns `Err(MissingCompiledClassHashV2(C))`.
8. `get_tx_weights()` returns the same error; the transaction is aborted with a wrong execution result despite being a valid, accepted transaction.
