### Title
`SwapAllowlistExtension` gates the router address instead of the actual end-user, allowing any user to bypass per-user swap restrictions via `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` — the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` inside the pool is the router contract, not the end user. If the router is allowlisted for a pool, every user on the network can bypass the per-user swap restriction by routing through the router.

---

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the hook.**

`MetricOmmPool.swap()` calls `_beforeSwap` with `msg.sender` as the first argument: [1](#0-0) 

`_beforeSwap` encodes that value as the `sender` field forwarded to every configured extension: [2](#0-1) 

**Step 2 — Router is the `msg.sender` the pool sees.**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly. Inside the pool, `msg.sender` is therefore the router address, not the originating user: [3](#0-2) 

**Step 3 — The allowlist checks the router, not the user.**

`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (enforced by `onlyPool`) and `sender` is the router: [4](#0-3) 

The allowlist mapping is keyed `allowedSwapper[pool][swapper]`: [5](#0-4) 

**Resulting invariant break — two symmetric failure modes:**

| Configuration | Effect |
|---|---|
| Router is allowlisted, specific users are not | Every user bypasses per-user restrictions by routing through the router |
| Specific users are allowlisted, router is not | Allowlisted users cannot use the router at all — core swap path broken |

In the first mode, a pool admin who intends to restrict swaps to KYC'd or institutional addresses (a natural use of `SwapAllowlistExtension`) and also allowlists the router for convenience inadvertently opens the pool to all users. The guard is completely defeated.

---

### Impact Explanation

Any unprivileged user can swap in a pool whose admin configured `SwapAllowlistExtension` to restrict access, provided the router is allowlisted. This breaks the admin-boundary invariant: a pool admin-configured guard is bypassed by an unprivileged path (`MetricOmmSimpleRouter`). Unauthorized swaps in a restricted pool cause LP-side harm through unintended price impact and liquidity consumption, and violate the access-control invariant the pool admin explicitly set.

---

### Likelihood Explanation

Allowlisting the router is the natural configuration for any restricted pool that still wants to support the standard periphery swap path. A pool admin who adds the router to the allowlist (expecting it to act as a trusted intermediary) does not realize this grants every user on the network unrestricted access. The trigger requires no privileged action from the attacker — a standard `exactInputSingle` call suffices.

---

### Recommendation

The `beforeSwap` hook must gate the actual end user, not the immediate caller of `pool.swap()`. Two viable approaches:

1. **Pass the originating user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires the extension to trust the router's encoding, so the extension should additionally verify that `sender` (the pool-level caller) is a known trusted router.

2. **Check `recipient` instead of `sender`**: For single-hop swaps the recipient is often the end user, but this breaks for multi-hop paths where intermediate recipients are the router itself.

The cleanest fix is approach 1, combined with an explicit trusted-router registry in the extension so that only verified routers can supply the originating-user claim.

---

### Proof of Concept

```
1. Pool admin deploys a pool with SwapAllowlistExtension configured.
2. Admin calls setAllowedToSwap(pool, router, true)   // allowlist the router
3. Admin calls setAllowedToSwap(pool, alice, true)    // allowlist alice
4. Bob (not allowlisted) calls:
       router.exactInputSingle({pool: pool, ...})
5. Inside pool.swap(), msg.sender = router.
6. _beforeSwap(sender=router, ...) is called.
7. SwapAllowlistExtension checks allowedSwapper[pool][router] == true → passes.
8. Bob's swap executes successfully despite not being allowlisted.
```

Bob, an unprivileged and non-allowlisted user, successfully swaps in a pool the admin intended to restrict, by routing through `MetricOmmSimpleRouter`.
