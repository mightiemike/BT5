### Title
Unchecked `tx_index` Crosses Merkle Tree Depth Boundary, Enabling False Position Claims in Proof Verification - (File: `merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

---

### Summary

`compute_root_from_merkle_proof` silently discards high-order bits of `transaction_position` that exceed the tree depth. Because neither `verify_transaction_inclusion` nor `verify_transaction_inclusion_v2` validates that `tx_index < 2^merkle_proof.len()`, an unprivileged caller can supply an out-of-bounds `tx_index` (e.g., `p + 2^d`) for a transaction that genuinely sits at position `p` in a depth-`d` tree, and the proof still verifies as `true`. This is the direct analog of the Solana memory-gap bug: just as `memset` silently crosses a stack-frame gap by not checking that the entire range falls within one region, `compute_root_from_merkle_proof` silently crosses the tree-depth boundary by not checking that the position falls within the valid leaf range.

---

### Finding Description

`compute_root_from_merkle_proof` iterates over the proof elements and at each level uses only the lowest bit of `current_position` to decide left/right placement, then right-shifts by 1:

```rust
for proof_hash in merkle_proof {
    if current_position % 2 == 0 {
        current_hash = compute_hash(&current_hash, proof_hash);
    } else {
        current_hash = compute_hash(proof_hash, &current_hash);
    }
    current_position /= 2;
}
``` [1](#0-0) 

After `d` iterations (where `d = merkle_proof.len()`), only bits `0..d-1` of the original `transaction_position` have been consumed. Any bits at position `d` or higher are silently ignored. Consequently, positions `p`, `p + 2^d`, `p + 2·2^d`, … all produce identical traversal paths and identical computed roots.

`verify_transaction_inclusion` converts the caller-supplied `tx_index: u64` to `usize` and passes it directly, with no upper-bound check:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [2](#0-1) 

`verify_transaction_inclusion_v2` adds a coinbase-proof length parity check but then delegates to the same unchecked v1 path:

```rust
require!(
    args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
    ...
);
...
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

`tx_index` is a plain `u64` field in both `ProofArgs` and `ProofArgsV2` with no declared invariant: [4](#0-3) 

---

### Impact Explanation

Any downstream NEAR contract that consumes the `bool` result of `verify_transaction_inclusion[_v2]` and uses the caller-supplied `(tx_block_blockhash, tx_index)` tuple as a unique transaction identifier for replay protection can be bypassed. An attacker who has already processed transaction `T` at position `p` in a block with tree depth `d` can re-submit the identical `tx_id` and `merkle_proof` with `tx_index = p + k·2^d` for any `k ≥ 1`. The contract returns `true`, the recipient contract sees a novel `(blockhash, tx_index)` key, and the same Bitcoin event is processed a second (or Nth) time. Position-gated logic (e.g., "only accept transactions at index 0") is equally bypassable by choosing `k` such that `p + k·2^d` satisfies the gate while the actual position `p` does not.

---

### Likelihood Explanation

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are public, unpermissioned NEAR contract methods callable by any account. The attacker needs only a valid Bitcoin transaction, its real merkle proof (public on-chain data), and the ability to craft a NEAR call with an inflated `tx_index`. No privileged role, private key, or social engineering is required.

---

### Recommendation

Add a depth-boundary guard before invoking `compute_root_from_merkle_proof`. In `verify_transaction_inclusion`:

```rust
let depth = args.merkle_proof.len();
require!(
    depth < 64 && args.tx_index < (1u64 << depth),
    "tx_index exceeds the number of leaves implied by the merkle proof depth"
);
```

Alternatively, enforce the invariant inside `compute_root_from_merkle_proof` itself so every call site is protected:

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &[H256],
) -> H256 {
    let depth = merkle_proof.len();
    assert!(
        depth >= 64 || transaction_position < (1usize << depth),
        "transaction_position out of range for proof depth"
    );
    ...
}
```

### Proof of Concept

Suppose a Bitcoin block contains 4 transactions (depth `d = 2`). Transaction `T` is at position `p = 1`. A valid proof `[sibling_0, parent_sibling]` exists.

**Legitimate call** — passes:
```
tx_id    = hash(T)
tx_index = 1          // real position
merkle_proof = [sibling_0, parent_sibling]
```

**Replay call with out-of-bounds index** — also passes:
```
tx_id    = hash(T)
tx_index = 5          // 1 + 2^2 = 5, does not exist in a 4-leaf tree
merkle_proof = [sibling_0, parent_sibling]   // identical proof
```

Inside `compute_root_from_merkle_proof` with `transaction_position = 5` (`0b101`):
- Level 0: `5 % 2 = 1` → right → same as position 1
- Level 1: `5 / 2 = 2`, `2 % 2 = 0` → left → same as position 1

The computed root is identical to the legitimate call. The function returns `true`. A recipient contract that tracks `(blockhash, tx_index)` to prevent replay now sees `(blockhash, 5)` as a fresh, unprocessed event and executes the action a second time. [5](#0-4) [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
