### Title
NEAR Storage-Deposit Refunds Accumulate in `BtcLightClient` With No Withdrawal Path - (File: contract/src/lib.rs)

### Summary
`BtcLightClient` accepts NEAR token deposits via the `#[payable]` `submit_blocks` method to cover on-chain storage costs. The public `run_mainchain_gc` method removes old block headers, which causes NEAR Protocol to return the freed storage cost to the contract's account balance. Because the contract has no withdrawal function, this accumulated NEAR is permanently locked.

### Finding Description

`submit_blocks` is marked `#[payable]` and charges callers for net storage growth: [1](#0-0) 

The refund logic at line 190–197 only returns the excess of the *current call's* deposit over the *net* storage delta measured after GC runs. It does not return NEAR that was deposited in *previous* calls for blocks that GC has now freed.

`run_mainchain_gc` is also exposed as a standalone public method callable by any account: [2](#0-1) 

When it removes headers from `headers_pool`, `mainchain_height_to_header`, and `mainchain_header_to_height`, NEAR Protocol automatically credits the freed storage cost back to the contract's account balance. This NEAR was originally deposited by relayers in prior `submit_blocks` calls.

A grep across the entire contract confirms there is no `withdraw`, `drain`, `rescue`, or equivalent function — the only outbound `Promise::transfer` is the per-call refund inside `submit_blocks`: [3](#0-2) 

Two concrete accumulation paths exist:

1. **Standalone `run_mainchain_gc` call**: Any caller invokes `run_mainchain_gc(large_batch_size)`. Storage is freed, NEAR is credited to the contract's balance, and nothing transfers it out.

2. **GC frees more than new headers add inside `submit_blocks`**: `diff_storage_usage = env::storage_usage().saturating_sub(initial_storage)` uses saturating subtraction. When GC removes more bytes than the new batch adds, `diff_storage_usage = 0`, `required_deposit = 0`, and the current caller's full deposit is refunded — but the freed NEAR from prior depositors' storage remains in the contract. [4](#0-3) 

### Impact Explanation

**Medium.** NEAR tokens deposited by relayers for storage costs accumulate permanently in the contract as GC progressively frees old block headers. The contract's balance grows monotonically over the operational lifetime of the light client (which is designed to run indefinitely with a `gc_threshold` of ~52,704 blocks per year). These funds are irrecoverable without a contract upgrade.

### Likelihood Explanation

**Medium.** GC is invoked on every `submit_blocks` call and is also directly callable by any unprivileged account via `run_mainchain_gc`. The accumulation is a normal consequence of steady-state operation, not an edge case. The longer the contract runs, the larger the locked balance grows.

### Recommendation

Add a privileged withdrawal function (callable only by `Role::DAO` or an equivalent admin role) that transfers the contract's excess balance to a designated recipient:

```rust
#[access_control_any(roles(Role::DAO))]
pub fn withdraw_near(&mut self, amount: NearToken, recipient: AccountId) -> Promise {
    Promise::new(recipient).transfer(amount)
}
```

Alternatively, track the total storage deposit per relayer and refund each relayer's proportional share when their blocks are GC'd.

### Proof of Concept

1. Deploy `BtcLightClient` and initialize with a `gc_threshold` of, say, 100.
2. Relayer A calls `submit_blocks` with 100 headers, attaching a deposit of 50 NEAR. The contract stores the headers and keeps the required storage deposit.
3. Relayer B calls `submit_blocks` with 100 more headers. GC triggers and removes Relayer A's 100 headers. The freed storage NEAR is credited to the contract's balance. `diff_storage_usage` reflects only the net change (≈ 0 if equal sizes), so Relayer B's deposit is fully refunded.
4. The NEAR originally deposited by Relayer A for the now-deleted headers remains in the contract's balance.
5. Call `run_mainchain_gc(1000)` directly — more storage is freed, more NEAR accumulates.
6. Confirm there is no callable method to extract this balance. The contract's `env::account_balance()` grows with no egress path.

    #[payable]
    #[pause]
    #[trusted_relayer]
    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }

    #[pause(except(roles(Role::UnrestrictedRunGC)))]
    pub fn run_mainchain_gc(&mut self, batch_size: u64) {
        let initial_blockheader = self
            .headers_pool
            .get(&self.mainchain_initial_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let tip_blockheader = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let amount_of_headers_we_store =
            tip_blockheader.block_height - initial_blockheader.block_height + 1;

        if amount_of_headers_we_store > self.gc_threshold {
            let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
            let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);

            let start_removal_height = initial_blockheader.block_height;
            let end_removal_height = initial_blockheader.block_height + selected_amount_to_remove;
            env::log_str(&format!(
                "Num of blocks to remove {selected_amount_to_remove}"
            ));

            for height in start_removal_height..end_removal_height {
                let blockhash = &self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

                self.remove_block_header(blockhash);
                self.mainchain_height_to_header.remove(&height);
            }

            self.mainchain_initial_blockhash = self
                .mainchain_height_to_header
                .get(&end_removal_height)
                .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        }
    }
