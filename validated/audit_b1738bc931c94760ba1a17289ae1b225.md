### Title
NLP Token Health Contribution Does Not Account for Burn Fee, Overstating Collateral Value — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

The health contribution of NLP tokens uses the full `oraclePriceX18` without deducting the burn fee charged in `burnNlp()`. This is the direct Nado analog to the Sandclock `investedAssets()` bug: an asset-valuation path reports gross value while a separate execution path silently deducts a fee, creating a persistent overstatement of collateral worth.

---

### Finding Description

In `Clearinghouse.burnNlp()`, when a user redeems NLP tokens, a burn fee is deducted before the quote is returned:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
```

The actual redeemable value of `nlpAmount` NLP tokens is therefore `nlpAmount × oraclePriceX18 − burnFee`, not `nlpAmount × oraclePriceX18`.

However, the NLP token price stored in the spot engine is set to the full `oraclePriceX18` by both `mintNlp` and `burnNlp`:

```solidity
spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);
```

`BaseEngine._calculateProductHealth()` then computes the health contribution of every spot product — including NLP — as:

```solidity
health += amount.mul(weight).mul(risk.priceX18);
```

where `risk.priceX18` is the full `oraclePriceX18`. No burn fee is subtracted. The health contribution of NLP collateral is therefore overstated by `burnFee × weight` for every subaccount holding NLP tokens.

The post-burn health check in `burnNlp()` uses `HealthType.MAINTENANCE` (the more lenient threshold), not `HealthType.INITIAL`:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
```

Because maintenance weights are higher than initial weights, a user who has borrowed up to their INITIAL health limit (which was inflated by the burn fee) can still pass the post-burn MAINTENANCE check, leaving the account in a state where it has borrowed more than its actual collateral value.

---

### Impact Explanation

A user holding NLP tokens can borrow up to their INITIAL health limit, which is inflated by `burnFee × initialWeight`. After burning NLP tokens, the post-burn MAINTENANCE health check passes because the gap between initial and maintenance weights absorbs the burn fee discrepancy. The account is left with borrowed funds exceeding the true redeemable value of its NLP collateral by the burn fee amount. For large positions the burn fee is 0.1% of position value (`quoteAmount / 1000`); for small positions the minimum is `ONE` ($1). At scale (e.g., $10M NLP position), the overstatement is $10,000 per subaccount. If such a subaccount is subsequently liquidated, the protocol recovers less from the NLP collateral than the risk model assumed, creating a bad-debt shortfall equal to the unaccounted burn fee.

---

### Likelihood Explanation

The discrepancy is structural and passive — it applies to every subaccount that holds NLP tokens and borrows against them. No special timing or sequencer cooperation is required. Any unprivileged user can mint NLP, borrow up to the inflated INITIAL health limit, and then burn NLP. The entry path is fully user-controlled via the standard `MintNlp` → `BurnNlp` transaction flow through `Endpoint`.

---

### Recommendation

The NLP price used in health calculations should reflect the net redeemable value after the burn fee. One approach is to register an effective price for the NLP product that discounts the oracle price by the burn fee rate:

```
effectivePriceX18 = oraclePriceX18.mul(ONE - BURN_FEE_RATE)
```

where `BURN_FEE_RATE = ONE / 1000` (0.1%). This effective price should be passed to `spotEngine.updatePrice(NLP_PRODUCT_ID, effectivePriceX18)` in both `mintNlp` and `burnNlp`, so that health calculations reflect the actual worst-case redeemable value of NLP collateral.

---

### Proof of Concept

1. User holds `nlpAmount` NLP tokens. `oraclePriceX18 = P`. Spot engine stores `priceX18 = P` for `NLP_PRODUCT_ID`.
2. Health contribution: `nlpAmount × initialWeight × P`. User borrows `B = nlpAmount × initialWeight × P` in quote. INITIAL health = 0 (at limit). This borrow is permitted.
3. User submits `BurnNlp`. `burnFee = max(1e18, nlpAmount × P / 1000)`. Quote returned = `nlpAmount × P − burnFee`.
4. Post-burn MAINTENANCE health = `(nlpAmount × P − burnFee) × maintenanceWeight − B`.
   = `nlpAmount × P × (maintenanceWeight − initialWeight) − burnFee × maintenanceWeight`.
   Since `maintenanceWeight − initialWeight ≫ burnFee / (nlpAmount × P)` for typical risk parameters, this is positive — the check passes.
5. The subaccount has borrowed `B = nlpAmount × initialWeight × P` but its NLP collateral was only worth `nlpAmount × P − burnFee`. The protocol's risk model assumed full `nlpAmount × P` as collateral. The shortfall is `burnFee × initialWeight`, which is unrecoverable if the subaccount is later liquidated.

```text
    function _calculateProductHealth(
        uint32 productId,
        bytes32 subaccount,
        IProductEngine.HealthType healthType
    ) internal returns (int128 health) {
        RiskHelper.Risk memory risk = _risk(productId);
        (int128 amount, int128 quoteAmount) = _getBalance(
            productId,
            subaccount
        );
        int128 weight = RiskHelper._getWeightX18(risk, amount, healthType);
        health += quoteAmount;

        if (amount != 0) {
            if (weight == 2 * ONE) {
                return -INF;
            }
            health += amount.mul(weight).mul(risk.priceX18);
            emit PriceQuery(productId);
        }
    }
```
