### Title
Deprecated `verify_transaction_inclusion` Accepts Forged SPV Proofs via Caller-Supplied Internal Merkle Node as `tx_id` — (`contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function remains publicly callable by any unprivileged NEAR account. It relies entirely on caller-supplied `tx_id` without validating that it represents a real leaf transaction rather than an internal Merkle tree node. An attacker can supply an internal Merkle node hash as `tx_id` together with a valid proof path from that node to the block's Merkle root, causing the function to return `true` for a transaction that does not exist in the block.

---

### Finding Description

`verify_transaction_inclusion` at `contract/src/lib.rs:288–323` delegates proof verification to `merkle_tools::compute_root_from_merkle_proof` (`merkle-tools/src/lib.rs:34–52`). That function accepts any 32-byte hash as its starting point and iterates over the caller-supplied proof elements to reconstruct a root:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
```

The only guard on the proof inputs is:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
```

There is **no validation** that `args.tx_id` is a leaf-level transaction hash rather than an internal Merkle tree node. Bitcoin's Merkle tree is built by double-SHA256-hashing pairs of 32-byte hashes (64 bytes total). Every internal node at depth D is itself a valid starting point for a proof of length D that terminates at the Merkle root. An attacker who picks any internal node N from a real block's Merkle tree can supply:

- `tx_id` = N (the internal node hash)
- `tx_index` = the position of N within its level
- `merkle_proof` = the sibling hashes from N's level up to the root

`compute_root_from_merkle_proof` will faithfully reconstruct the correct Merkle root from these inputs, and the function returns `true` — asserting that a non-existent transaction is included in the block.

The v2 function (`verify_transaction_inclusion_v2`, lines 347–369) mitigates this by requiring a separate coinbase Merkle proof, which forces the proof depth to be consistent with the real tree structure. The v1 function has no such guard and remains callable.

The analog to the external report is direct: just as `swapBack` relied on `balanceOf` — an externally manipulable value — instead of internal accounting, `verify_transaction_inclusion` relies on a caller-supplied `tx_id` — an externally manipulable value — without validating it against the block's actual transaction set. The missing validation (coinbase proof) is the analog to the missing slippage check: both omissions allow the function to produce an incorrect result that downstream consumers trust.

---

### Impact Explanation

Any NEAR contract that calls `verify_transaction_inclusion` (v1) to gate a privileged action (e.g., a bridge contract releasing funds upon SPV proof) can be deceived into accepting a fraudulent proof. The attacker does not need to forge a hash preimage; they only need to read the Merkle tree of any real block already stored in the contract's `headers_pool` and extract an internal node. The function returns `true`, and the consuming contract proceeds as if the fake transaction is confirmed.

---

### Likelihood Explanation

The function is `pub` with no access-control annotation beyond `#[pause]`. Any unprivileged NEAR account can call it. The 64-byte transaction Merkle forgery technique is publicly documented (referenced in the contract's own comments at line 268) and requires no special computational resources — the attacker only needs to read existing block data from the contract's state.

---

### Recommendation

1. **Remove or gate the deprecated function.** Add an access-control role requirement to `verify_transaction_inclusion` (v1) so it cannot be called by unprivileged accounts, or remove it entirely.
2. **Direct all callers to `verify_transaction_inclusion_v2`**, which enforces coinbase proof validation and prevents this class of forgery.

---

### Proof of Concept

1. Submit a real Bitcoin block with ≥ 2 transactions to the contract via `submit_blocks`. The block is now in `headers_pool` and `mainchain_header_to_height`.
2. Compute the block's Merkle tree offline. Pick any internal node N at depth D ≥ 1 (D ≥ 1 satisfies the non-empty proof requirement).
3. Record the sibling hashes at each level from D up to the root — this is the forged `merkle_proof` of length D.
4. Determine the position of N within its level — this is `tx_index`.
5. Call `verify_transaction_inclusion` with:
   - `tx_block_blockhash` = the real block hash
   - `tx_id` = N (the internal node hash, not a real transaction)
   - `tx_index` = position of N
   - `merkle_proof` = sibling hashes from N to root
   - `confirmations` = 1
6. `compute_root_from_merkle_proof(N, position, siblings)` reconstructs the correct Merkle root; the function returns `true` for a transaction that does not exist. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
