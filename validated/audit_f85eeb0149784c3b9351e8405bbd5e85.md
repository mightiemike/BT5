### Title
4-byte secp opcode dispatch bypasses `ENABLE_SECP_OPS` gate — (`src/chia_dialect.rs`)

### Summary

`ChiaDialect::op` contains two separate dispatch paths for secp operations. The 1-byte opcode path (opcodes 64/65) is correctly gated behind `ENABLE_SECP_OPS`, but the 4-byte opcode path (`0x13d61f00` / `0x1c3a8f00`) dispatches directly to `op_secp256k1_verify` / `op_secp256r1_verify` with **no flag check at all**. Any `ChiaDialect` instance — including one constructed with `ClvmFlags::empty()` — will execute secp signature verification when presented with the 4-byte encoding.

### Finding Description

In `ChiaDialect::op`, the 4-byte branch (lines 157–183) reads the opcode and matches it: [1](#0-0) 

The match arms for `0x13d61f00` and `0x1c3a8f00` call `op_secp256k1_verify` / `op_secp256r1_verify` unconditionally — no `flags.contains(ClvmFlags::ENABLE_SECP_OPS)` guard is present.

Contrast with the 1-byte path, which is correctly gated: [2](#0-1) 

The `ENABLE_SECP_OPS` flag is documented as the enabler for secp opcodes 64 and 65: [3](#0-2) 

The flag description says nothing about the 4-byte aliases being separately ungated, so the omission is inconsistent with the stated semantics.

Additionally, the 4-byte secp opcodes also bypass the `NO_UNKNOWN_OPS` check: unknown 4-byte opcodes fall through to `unknown_operator` (which enforces `NO_UNKNOWN_OPS`), but the two recognized secp 4-byte values skip that path entirely and call the op function directly. [4](#0-3) 

### Impact Explanation

A caller that constructs `ChiaDialect::new(ClvmFlags::empty())` (or any flags set without `ENABLE_SECP_OPS`) and then inspects `flags.contains(ENABLE_SECP_OPS)` to decide whether secp verification is reachable will conclude it is disabled. In reality, any CLVM program that encodes the operator as the 4-byte atom `[0x13, 0xd6, 0x1f, 0x00]` will successfully invoke `op_secp256k1_verify` and return `Ok(Reduction(1300000, nil))` on a valid signature. Consensus-adjacent or mempool-adjacent code that uses the flag as a capability query — e.g., to decide whether to pre-validate or reject secp-bearing spends — will reach the wrong conclusion.

### Likelihood Explanation

The 4-byte cost-encoded opcodes are a documented part of the CLVM operator encoding scheme and are reachable via normal `run_program` calls with attacker-controlled CLVM. No special privileges or configuration are required beyond constructing a `ChiaDialect` without `ENABLE_SECP_OPS`.

### Recommendation

Add an `ENABLE_SECP_OPS` guard to the 4-byte dispatch branch, mirroring the 1-byte path:

```rust
0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

If the 4-byte aliases are intentionally always-on (i.e., they predate the softfork flag), that design decision should be explicitly documented and the `ENABLE_SECP_OPS` docstring updated to clarify it only gates the 1-byte opcodes.

pub fn op_secp256k1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256K1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256k1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256k1_verify pubkey")?;
    let verifier = K1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256k1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is message
    let msg = atom(a, msg, "secp256k1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256k1_verify sig")?;
    let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
