### Title
Unprotected `initialize` in `WithdrawPool` Allows Any Caller to Seize Ownership and Drain Pooled Funds — (`File: core/contracts/WithdrawPool.sol`)

---

### Summary

`WithdrawPool.initialize` is an `external` function with no caller restriction. Any address can call it before the deployer does, becoming the contract owner and gaining full control over pooled withdrawal funds and the verifier used for fast-withdrawal signature checks.

---

### Finding Description

`WithdrawPool` inherits from `BaseWithdrawPool`, which uses OpenZeppelin's `initializer` modifier to gate `_initialize`. The `initializer` modifier prevents re-initialization, but it does **not** restrict who can perform the first call. The public-facing `initialize` function in `WithdrawPool` adds no additional access control:

```solidity
// core/contracts/WithdrawPool.sol
contract WithdrawPool is BaseWithdrawPool {
    function initialize(address _clearinghouse, address _verifier) external {
        _initialize(_clearinghouse, _verifier);
    }
}
``` [1](#0-0) 

The internal `_initialize` sets `clearinghouse`, `verifier`, and calls `__Ownable_init()`, making `msg.sender` the owner:

```solidity
function _initialize(address _clearinghouse, address _verifier)
    internal
    initializer
{
    __Ownable_init();
    clearinghouse = _clearinghouse;
    verifier = _verifier;
}
``` [2](#0-1) 

The `BaseWithdrawPool` constructor calls `_disableInitializers()` to protect the implementation contract, but the proxy's storage is unprotected until `initialize` is called on the proxy. If any block elapses between proxy deployment and the deployer's initialization call, an attacker can front-run it.

The same pattern exists in `SpotEngine.initialize` and `PerpEngine.initialize`, both of which are `external` with no access control: [3](#0-2) [4](#0-3) 

Both delegate to `BaseEngine._initialize`, which also calls `__Ownable_init()` and `transferOwnership(_admin)` — but `_admin` is attacker-controlled if the attacker calls first. [5](#0-4) 

---

### Impact Explanation

**`WithdrawPool` — direct asset theft:**

`WithdrawPool` receives ERC20 tokens from `Clearinghouse.handleWithdrawTransfer` before routing them to users:

```solidity
token.safeTransfer(withdrawPool, uint256(amount));
BaseWithdrawPool(withdrawPool).submitWithdrawal(token, to, amount, idx);
``` [6](#0-5) 

An attacker who seizes ownership via `initialize` can immediately call `removeLiquidity` (guarded only by `onlyOwner`) to drain any ERC20 balance held in the pool: [7](#0-6) 

Additionally, the attacker controls `verifier`. `submitFastWithdrawal` calls `v.requireValidTxSignatures` using the stored `verifier` address: [8](#0-7) 

A malicious verifier can approve any withdrawal transaction, bypassing the multi-signer threshold entirely.

**`SpotEngine` / `PerpEngine` — protocol accounting corruption:**

An attacker who seizes ownership of `SpotEngine` or `PerpEngine` can call `addOrUpdateProduct` and `updateRisk` to set arbitrary risk weights and prices, corrupting health calculations for all subaccounts and enabling under-collateralized positions or blocking legitimate liquidations. [9](#0-8) [10](#0-9) 

---

### Likelihood Explanation

Upgradeable proxy deployments routinely separate the `deploy` transaction from the `initialize` transaction, especially in multi-step deployment scripts. Any block gap between these two transactions is an exploitable window. A mempool-watching attacker can front-run the initialization call with a higher gas price. This is a well-known and actively exploited attack vector on mainnet.

---

### Recommendation

Add an access control check to `WithdrawPool.initialize` (and analogously to `SpotEngine.initialize` and `PerpEngine.initialize`) so that only a trusted deployer or factory address can call it:

```solidity
function initialize(address _clearinghouse, address _verifier) external {
    require(msg.sender == EXPECTED_DEPLOYER, "unauthorized initializer");
    _initialize(_clearinghouse, _verifier);
}
```

Alternatively, use OpenZeppelin's `initializer` modifier in combination with a factory pattern that deploys and initializes in a single atomic transaction, eliminating the front-running window entirely.

---

### Proof of Concept

1. Attacker monitors the mempool for a `WithdrawPool` proxy deployment transaction.
2. Before the deployer's `initialize` call is mined, attacker submits:
   ```solidity
   withdrawPool.initialize(attackerAddress, maliciousVerifier);
   ```
   with higher gas, front-running the deployer.
3. Attacker is now the owner of `WithdrawPool` with `clearinghouse = attackerAddress` and `verifier = maliciousVerifier`.
4. As soon as any ERC20 tokens are transferred into `WithdrawPool` (via `Clearinghouse.handleWithdrawTransfer`), attacker calls:
   ```solidity
   withdrawPool.removeLiquidity(productId, balance, attackerAddress);
   ```
   draining the full token balance.
5. Alternatively, attacker deploys `maliciousVerifier` that returns `true` for any signature check, then calls `submitFastWithdrawal` with a crafted transaction to withdraw arbitrary amounts without valid user signatures. [1](#0-0) [2](#0-1) [11](#0-10) [7](#0-6)
