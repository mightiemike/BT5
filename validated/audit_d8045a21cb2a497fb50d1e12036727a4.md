### Title
Silent Bonus Checkpoint Advancement During Redelegation Slash Permanently Forfeits Accrued Rewards - (File: `x/tieredrewards/keeper/slash.go`)

---

### Summary

In `slashRedelegationPosition`, when `processEventsAndClaimBonus` fails with `ErrInsufficientBonusPool`, the error is silently swallowed. However, the position's bonus checkpoints (`LastEventSeq`, `LastBonusAccrual`, `LastKnownBonded`) have already been advanced in memory before the pool balance check fires. These advanced checkpoints are then persisted via `setPositionWithState`, permanently forfeiting the accrued bonus rewards for the position owner with no recovery path.

---

### Finding Description

In `processEventsAndClaimBonus`, the function advances position checkpoints **before** checking whether the pool can cover the computed bonus: [1](#0-0) 

```go
pos.UpdateLastEventSeq(entry.Seq)   // checkpoint advanced inside event loop
``` [2](#0-1) 

```go
applyBonusAccrualCheckpoint(&pos.Position, blockTime)  // LastBonusAccrual advanced
pos.UpdateLastKnownBonded(bonded)                       // LastKnownBonded advanced
```

Only **after** all three checkpoints are advanced does the function check pool balance: [3](#0-2) 

```go
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err   // returns error — but pos already has advanced checkpoints
}
```

The caller `slashRedelegationPosition` catches this error and deliberately swallows it: [4](#0-3) 

```go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
        // error swallowed — pos still carries advanced checkpoints, no bonus paid
    } else {
        return err
    }
}
```

Execution then falls through to `setPositionWithState`, which persists the modified `pos` with its advanced checkpoints: [5](#0-4) 

```go
if fullSlash {
    ...
    return k.setPositionWithState(ctx, pos, ...)   // persists advanced checkpoints
}
pos.Delegation.Shares = pos.Delegation.Shares.Sub(sharesToUnbond)
return k.setPositionWithState(ctx, pos, nil)       // persists advanced checkpoints
```

Because `LastBonusAccrual` and `LastEventSeq` now point past the accrual period, the user can never re-claim the forfeited bonus — the accounting window is permanently closed.

---

### Impact Explanation

A position owner who has an active redelegation entry slashed while the bonus pool is empty permanently loses all bonus rewards that accrued up to the slash block. The corrupted invariant is:

> **bonus accrued (computed from checkpoints) == bonus paid to owner**

After this path executes, the checkpoints advance as if payment occurred, but no `SendCoinsFromModuleToAccount` transfer was made. The position's `LastBonusAccrual` and `LastEventSeq` are permanently ahead of the last paid period, making the gap unclaimable.

The `RewardsPoolName` module account balance is unaffected (no coins moved), but the position owner's expected reward balance is silently reduced. This is the direct analog of the Aave bug: a try-catch-style silent swallow causes a permanent accounting mismatch between what was earned and what was received.

---

### Likelihood Explanation

Three conditions must coincide:

1. A position owner has executed `MsgTierRedelegate`, creating a live redelegation entry tracked in `RedelegationMappings`.
2. The source validator is slashed (downtime or double-sign) during the redelegation unbonding window (up to 21 days).
3. The `tieredrewards` bonus pool (`RewardsPoolName`) is empty or below the computed bonus at the slash block.

The pool can be drained by normal `MsgClaimTierRewards` activity across many positions. Validator slashes are routine protocol events. The combination is low-probability per individual position but becomes increasingly likely as the number of positions and validators grows. The ADR itself acknowledges this path exists: [6](#0-5) 

> "Bonus forfeits silently if the pool is insufficient (chain-halt avoidance)."

The design choice trades user fund loss for chain liveness, but the root cause — advancing checkpoints before the payment check — is avoidable.

---

### Recommendation

Reorder `processEventsAndClaimBonus` so that the pool balance check occurs **before** any checkpoint is advanced. If the pool is insufficient, return early without modifying `pos`, preserving the user's ability to retry once the pool is replenished (consistent with the behavior documented for all other user-driven paths): [7](#0-6) 

> "User-driven paths (ClaimTierRewards, AddToPosition, Undelegate, Redelegate, ClearPosition): fail atomically. User retries after pool is replenished."

Alternatively, in `slashRedelegationPosition`, if `ErrInsufficientBonusPool` is caught, explicitly revert the in-memory checkpoint changes on `pos` before calling `setPositionWithState`, so the position is saved with its original checkpoints and the user can claim later.

---

### Proof of Concept

1. User calls `MsgTierRedelegate` — a `RedelegationMappings[unbondingId → positionId]` entry is created.
2. The bonus pool is drained to zero by concurrent `MsgClaimTierRewards` calls from other positions.
3. The source validator is slashed; the staking module fires `BeforeRedelegationSlashed(unbondingId, sharesToUnbond)`.
4. `slashRedelegationPosition` is called → `processEventsAndClaimBonus` runs, advances `LastEventSeq` (line 193), `LastBonusAccrual` (line 215), `LastKnownBonded` (line 217) in memory, then returns `ErrInsufficientBonusPool` at line 231.
5. The error is caught and swallowed at `slash.go:56-63`.
6. `setPositionWithState` persists the position with advanced checkpoints and zero bonus paid.
7. User later calls `MsgClaimTierRewards` — `processEventsAndClaimBonus` finds no events since the now-advanced `LastEventSeq` and computes zero bonus for the already-advanced `LastBonusAccrual` window. The accrued rewards are permanently unclaimable. [8](#0-7) [9](#0-8)
