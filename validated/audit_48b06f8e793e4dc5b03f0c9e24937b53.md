### Title
Stale `LastKnownBonded` Initialization Causes Bonus Overpayment for Unbonded Validator Periods — (`x/tieredrewards/keeper/position.go`)

---

### Summary

`createDelegatedPosition` unconditionally initializes `LastKnownBonded = true` for every new position, regardless of the validator's actual bonded state at creation time. When a position is created while a validator is unbonded (i.e., after an `UNBOND` event has been recorded but before the subsequent `BOND` event), the lazy event-replay logic in `processEventsAndClaimBonus` treats the entire gap between position creation and the next `BOND` event as a bonded segment and pays out bonus rewards for it — even though the validator was unbonded throughout that window.

---

### Finding Description

**Root cause — `position.go` line 56:** [1](#0-0) 

```go
pos := types.NewPosition(id, owner, tier.Id, delAddr.String(), blockHeight, lastEventSeq, blockTime, true, blockTime)
```

The eighth argument (`true`) is `LastKnownBonded`. It is hardcoded to `true` for every new position, independent of whether the target validator is currently bonded or unbonded.

**How the lazy replay uses this value:** [2](#0-1) 

```go
bonded := pos.LastKnownBonded   // always true at creation
segmentStart := pos.LastBonusAccrual
```

The replay loop then computes bonus for every segment where `bonded == true`: [3](#0-2) 

```go
if bonded {
    bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
    totalBonus = totalBonus.Add(bonus)
}
```

**Concrete trigger sequence:**

1. Validator V is bonded. Three existing positions are on V.
2. V unbonds → `AfterValidatorBeginUnbonding` records an `UNBOND` event at seq=N with `ReferenceCount=3`.
3. Attacker calls `MsgLockTier` (or `MsgCommitDelegationToTier`) targeting V while V is unbonded.
   - `createDelegatedPosition` reads `lastEventSeq = N` and sets `LastKnownBonded = true`.
   - `PositionCountByValidator[V]` is incremented to 4.
4. V rebonds → `AfterValidatorBonded` records a `BOND` event at seq=N+1 with `ReferenceCount=4` (includes the new position).
5. Attacker calls `MsgClaimTierRewards`.

In `processEventsAndClaimBonus`:
- `events = [seq=N+1 (BOND at T_bond)]`
- `bonded = pos.LastKnownBonded = true`
- `segmentStart = pos.LastBonusAccrual = T_create`

Processing the BOND event:
```
if bonded {   // true — incorrectly
    bonus = computeSegmentBonus(pos, tier, T_create, T_bond, tokensPerShare)
    // pays bonus for [T_create, T_bond] — the entire unbonded window
}
```

The validator was unbonded during `[T_create, T_bond]`, so no bonus should accrue. The correct initial value of `LastKnownBonded` should be `false` (matching the validator's actual state at position creation).

**Why the existing `val.IsBonded()` guard does not help:** [4](#0-3) 

```go
if bonded && val.IsBonded() {
    ...
    bonus := k.computeSegmentBonus(*pos, tier, segmentStart, blockTime, currentRate)
```

This guard only protects the *final open segment* (from the last event to `blockTime`). It does not protect the segment computed *inside the event loop* before the BOND event is processed.

---

### Impact Explanation

The attacker receives bonus rewards from the `RewardsPoolName` module account for a period during which the validator was unbonded. The overpayment is:

```
overpayment ≈ position_tokens × tier.BonusApy × unbonded_duration / SecondsPerYear
```

For a 4 % APY tier, a 1 000 000 basecro position, and a 30-day unbonded window, the overpayment is approximately 3 288 basecro per position. Multiple positions can be opened in the same window, multiplying the drain. The rewards pool is a finite module account; repeated exploitation depletes it, denying legitimate bonus rewards to honest stakers.

---

### Likelihood Explanation

Validators are jailed for downtime (signed-blocks-window = 10 in the test config) and must be unjailed, producing exactly the UNBOND → BOND event pair required. An attacker monitoring the chain can observe the UNBOND event and submit `MsgLockTier` in the same or next block before the validator is unjailed. No privileged access is required; any account with sufficient funds can open a position.

---

### Recommendation

In `createDelegatedPosition`, query the validator's actual bonded state and use it to initialize `LastKnownBonded`:

```go
val, err := k.stakingKeeper.GetValidator(ctx, valAddr)
if err != nil {
    return types.Position{}, err
}
initialBonded := val.IsBonded()

pos := types.NewPosition(id, owner, tier.Id, delAddr.String(), blockHeight, lastEventSeq, blockTime, initialBonded, blockTime)
```

This mirrors the fix described in the external report: the cached value must be anchored to the actual state at the time the snapshot is taken, not assumed to be a fixed default.

### Proof of Concept

```
Block 100: Validator V bonded. Positions P1, P2, P3 exist on V.
Block 200: V jailed → AfterValidatorBeginUnbonding fires.
           UNBOND event seq=5, ReferenceCount=3, TokensPerShare=X, Timestamp=T200.
Block 201: Attacker submits MsgLockTier(tier=1, amount=1_000_000, validator=V).
           createDelegatedPosition:
             lastEventSeq = 5
             LastKnownBonded = true   ← BUG: V is unbonded
             LastBonusAccrual = T201
           PositionCountByValidator[V] = 4.
Block 300: V unjailed → AfterValidatorBonded fires.
           BOND event seq=6, ReferenceCount=4, TokensPerShare=Y, Timestamp=T300.
Block 301: Attacker submits MsgClaimTierRewards(positionId=attacker_pos).

processEventsAndClaimBonus:
  bonded = true  (LastKnownBonded)
  segmentStart = T201

  event seq=6 (BOND, T300):
    bonded == true → bonus = shares * Y * BonusApy * (T300-T201) / SecondsPerYear
    ← PAID for unbonded window T201–T300

  final segment (T300–T301):
    bonded && val.IsBonded() → bonus = shares * currentRate * BonusApy * 1s / SecondsPerYear
    ← correctly paid

Result: attacker receives bonus for ~99 seconds of unbonded time they should not have earned.
```
