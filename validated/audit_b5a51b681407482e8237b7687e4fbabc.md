### Title
SwapAllowlistExtension Checks the Router (Direct Pool Caller) Instead of the End User, Allowing Any Caller to Bypass the Swap Allowlist — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps on `sender`, which is the **direct caller of `pool.swap()`** (e.g., `MetricOmmSimpleRouter`), not the end user who initiated the transaction. If the router is added to the allowlist, any unprivileged user can route through it and bypass the restriction entirely.

---

### Finding Description

In `MetricOmmPool.swap`, the pool calls the extension with `msg.sender` (the direct pool caller) as the `sender` argument: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` is on the allowlist for the pool (`msg.sender` inside the extension is the pool): [2](#0-1) 

The allowlist mapping is keyed `allowedSwapper[pool][swapper]`. When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router becomes `msg.sender` to the pool: [3](#0-2) 

So the extension sees `sender = router address`. If the pool admin has allowlisted the router (a natural operational step so that users can swap through the standard periphery), the check `allowedSwapper[pool][router]` returns `true` for **every caller of the router**, regardless of whether the actual end user is authorized.

The `recipient` (the second, unnamed parameter in `beforeSwap`) — the address that actually receives the output tokens — is never checked at all. [4](#0-3) 

---

### Impact Explanation

The swap allowlist is completely nullified for any pool that allowlists the router. Any unprivileged user can swap in a restricted pool by calling `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`). The pool receives and sends real tokens; the attacker receives the output. This is a direct loss of the access-control invariant the pool admin configured, with fund-impacting consequences (unauthorized parties interact with the pool's liquidity).

---

### Likelihood Explanation

High. The `MetricOmmSimpleRouter` is the standard swap periphery. Pool admins who want their users to swap through the router must allowlist it. Once the router is allowlisted, the restriction is unconditionally bypassed for all callers. No special privilege or setup is required from the attacker beyond calling the public router.

---

### Recommendation

Check the **end user** rather than the direct pool caller. Two complementary fixes:

1. **In `SwapAllowlistExtension.beforeSwap`**: check the `recipient` (the second parameter, currently unnamed/ignored) instead of — or in addition to — `sender`. The recipient is the address that benefits from the swap output and is the actor the allowlist should gate.

2. **Alternatively**, follow the Ion Protocol resolution pattern: do not allowlist router contracts at all; instead, require that every end user be individually allowlisted, and have the router pass the end user's address as `sender` (which would require a protocol-level change to how `sender` is forwarded through the pool).

---

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][router] = true   ← admin allowlists the router
  allowedSwapper[pool][attacker] = false ← attacker is NOT allowlisted

Attack:
  attacker calls:
    MetricOmmSimpleRouter.exactInputSingle({
      pool:      pool,
      recipient: attacker,
      zeroForOne: true,
      amountIn:  X,
      ...
    })

  Router calls:
    pool.swap(recipient=attacker, zeroForOne=true, amountSpecified=X, ...)
    → msg.sender to pool = router

  Pool calls extension:
    SwapAllowlistExtension.beforeSwap(sender=router, recipient=attacker, ...)
    → allowedSwapper[pool][router] == true  ✓ passes

  Swap executes; attacker receives output tokens.
  Allowlist is bypassed with zero privilege.
``` [2](#0-1) [5](#0-4) [6](#0-5)
