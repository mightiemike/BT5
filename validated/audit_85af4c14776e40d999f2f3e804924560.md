### Title
Missing Lower-Bound Validation on `confirmations` Allows Zero-Confirmation SPV Proof Acceptance — (File: `contract/src/lib.rs`)

### Summary

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations: u64` field with no lower-bound check. When a caller passes `confirmations = 0`, the confirmation-depth guard reduces to `actual_depth >= 0`, which is trivially true for any `u64` value, meaning the function returns `true` for a transaction in the chain-tip block — a block with zero finality that can be reorganized away. Any downstream NEAR contract that forwards a caller-supplied `confirmations` value to this function is exposed to a double-spend path.

### Finding Description

In `verify_transaction_inclusion` the only bounds checks on `confirmations` are:

```rust
require!(
    args.confirmations <= self.gc_threshold,   // upper bound only
    "..."
);
// ...
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [1](#0-0) 

There is no `require!(args.confirmations >= 1, ...)` or any equivalent minimum. When `confirmations = 0` is supplied:

- The upper-bound check `0 <= gc_threshold` always passes.
- The depth check `(tip_height - target_height + 1) >= 0` is always true for `u64` arithmetic.

The function then proceeds to the Merkle-root comparison and returns `true` for any transaction whose block is in the main chain, including the chain-tip block itself (zero blocks of confirmation depth).

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after the coinbase-proof check, so it inherits the same flaw. [2](#0-1) 

The `confirmations` field is defined as a plain `u64` in both `ProofArgs` and `ProofArgsV2` with no validation at the type level. [3](#0-2) 

The existing test suite explicitly uses `confirmations: 0` as a valid call, confirming the contract accepts it without error. [4](#0-3) 

### Impact Explanation

The SPV verification functions are the primary security output of this light client. Their purpose is to give downstream NEAR contracts a trustless guarantee that a Bitcoin transaction is buried under at least N blocks of proof-of-work. With `confirmations = 0` accepted silently, that guarantee is voided: the returned `true` carries no finality information. A downstream contract that forwards a caller-supplied `confirmations` value to `verify_transaction_inclusion_v2` can be tricked into releasing funds or executing irreversible actions for a Bitcoin transaction that is still in the chain tip and subject to reorganization, enabling a double-spend.

**Impact: 3 / 10**

### Likelihood Explanation

Exploitation requires a downstream contract to pass a caller-controlled `confirmations` value directly to the light client rather than hardcoding a safe minimum. Contracts that hardcode the confirmation count are unaffected. However, the API design actively invites this pattern — the parameter is part of the public `ProofArgs` struct and the contract imposes no floor, so integrators may reasonably assume any non-negative value is safe.

**Likelihood: 2 / 10**

### Recommendation

Add an explicit lower-bound guard at the start of `verify_transaction_inclusion`:

```rust
require!(args.confirmations >= 1, "confirmations must be at least 1");
```

Alternatively, document clearly in the ABI that `confirmations = 0` bypasses all finality guarantees and that downstream contracts must enforce their own minimum before calling this function.

### Proof of Concept

1. Deploy the contract and submit one block header so the chain tip is at height H.
2. As any unprivileged NEAR account, call `verify_transaction_inclusion_v2` with:
   - `tx_block_blockhash` = the chain-tip block hash (height H)
   - a valid Merkle proof for a transaction in that block
   - `confirmations = 0`
3. The function returns `true` despite the transaction having zero blocks of confirmation depth beyond its own block.
4. A downstream contract that trusts this result and releases funds is now exposed to a Bitcoin reorg that removes the tip block, invalidating the transaction. [5](#0-4) [6](#0-5)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
```
