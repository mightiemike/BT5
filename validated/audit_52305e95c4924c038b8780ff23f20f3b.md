### Title
Merkle Position Aliasing in `compute_root_from_merkle_proof` Allows False `tx_index` Claims — (`merkle-tools/src/lib.rs`)

### Summary
`compute_root_from_merkle_proof` uses only the parity sequence of `transaction_position` (via repeated `% 2` and `/ 2`) to determine left/right placement at each tree level. It never validates that the supplied index is within the actual leaf count of the tree. Any two indices that share the same sequence of even/odd parities at each halving step produce an identical root computation. Concretely, index `0` and index `4` (= 1 × 2²) are indistinguishable for a 2-level proof, so a valid proof for leaf[0] at index 0 also passes verification when submitted with `tx_index = 4`, violating the invariant that a `true` return means `tx_id` is at position `tx_index`.

### Finding Description

`compute_root_from_merkle_proof` iterates over the proof hashes, using `current_position % 2` to decide left/right and `current_position /= 2` to advance: [1](#0-0) 

There is no check that `transaction_position < 2^(proof.len())` (the maximum valid leaf index for a tree of that depth). For a proof of length `n`, any index of the form `k × 2^n` (k ≥ 0) produces the exact same sequence of `% 2` results as index `0`, because every intermediate value is even.

`verify_transaction_inclusion` passes the caller-supplied `args.tx_index` directly to this function with no bounds check: [2](#0-1) 

`tx_index` is a plain `u64` field in `ProofArgs` with no validation: [3](#0-2) 

`verify_transaction_inclusion_v2` ultimately delegates to `verify_transaction_inclusion` with the same attacker-controlled `tx_index`: [4](#0-3) 

### Proof of Concept

For a 4-leaf tree with leaves `[L0, L1, L2, L3]`, the valid proof for `L0` at index `0` is:
```
proof = [L1, hash(L2, L3)]
```

**Verification with index = 0 (legitimate):**
- Step 1: `0 % 2 = 0` → `hash(L0, L1)` = `N01`; position → `0`
- Step 2: `0 % 2 = 0` → `hash(N01, N23)` = `root`; position → `0`
- ✅ Matches stored `merkle_root`

**Verification with index = 4 (exploit):**
- Step 1: `4 % 2 = 0` → `hash(L0, L1)` = `N01`; position → `2`
- Step 2: `2 % 2 = 0` → `hash(N01, N23)` = `root`; position → `1`
- ✅ Also matches stored `merkle_root`

The call `verify_transaction_inclusion({ tx_id: L0, tx_index: 4, merkle_proof: [L1, N23], ... })` returns `true` even though position 4 does not exist in a 4-transaction block. The invariant stated in the contract's own doc comment — *"True if `tx_id` is at the claimed position"* — is broken. [5](#0-4) 

### Impact Explanation

Any caller (including cross-chain bridge contracts) that relies on `tx_index` to identify *which* transaction at a specific position is confirmed (e.g., to distinguish coinbase from non-coinbase, or to enforce ordering invariants) will accept a proof for a position that does not exist. The transaction `tx_id` IS in the block, but the position claim is false. This violates the stated contract invariant and can enable false cross-chain proof acceptance in systems that use `tx_index` as a semantic identifier.

### Likelihood Explanation

`verify_transaction_inclusion` is a public, unpermissioned NEAR view call. Any external account can call it with arbitrary `tx_index` values. No privileged role, key leak, or social engineering is required. The aliasing is deterministic and requires only knowledge of a valid proof for any transaction in the target block.

### Recommendation

Add a bounds check in `compute_root_from_merkle_proof` (or at the call site in `verify_transaction_inclusion`) to reject any `transaction_position >= 2^(merkle_proof.len())`:

```rust
// In compute_root_from_merkle_proof or verify_transaction_inclusion:
require!(
    transaction_position < (1usize << merkle_proof.len()),
    "tx_index out of range for the given proof depth"
);
```

This ensures that only indices within the actual leaf range of the tree are accepted, eliminating the aliasing class entirely. [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
