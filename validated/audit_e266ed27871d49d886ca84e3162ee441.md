### Title
Coinbase Proof Bypass via Internal-Node Substitution Defeats 64-Byte Forgery Mitigation — (`contract/src/lib.rs`, `merkle-tools/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` is intended to prevent the 64-byte Merkle proof forgery attack by requiring a valid coinbase proof of equal depth. However, the coinbase check itself accepts any value — including an internal Merkle tree node — as `coinbase_tx_id`, as long as it hashes to the stored `merkle_root` with the supplied proof. An attacker can supply an internal node as both the "coinbase" and the "transaction", satisfying every guard and causing the function to return `true` for a transaction that was never in the block.

---

### Finding Description

The guard in `verify_transaction_inclusion_v2` is:

```rust
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,                          // position hardcoded to 0
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);
``` [1](#0-0) 

`compute_root_from_merkle_proof` is a pure hash-chain traversal with no awareness of tree depth or leaf vs. internal-node distinction:

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;
    for proof_hash in merkle_proof {
        ...
        current_position /= 2;
    }
    current_hash
}
``` [2](#0-1) 

The only structural guard between the two proofs is a length-equality check:

```rust
require!(
    args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
    ...
);
``` [3](#0-2) 

Because `compute_root_from_merkle_proof` accepts any starting hash and any proof length, an attacker can substitute an **internal node** for `coinbase_tx_id` and supply a proof that is **shorter than the real tree depth**. The coinbase check passes because the internal node genuinely hashes to the root via the shorter path. The equal-length requirement is then trivially satisfied by making the transaction proof the same short length. The downstream `verify_transaction_inclusion` call has no additional leaf-vs-node check — it only re-runs the same `compute_root_from_merkle_proof` comparison. [4](#0-3) 

---

### Impact Explanation

Any caller of the public `verify_transaction_inclusion_v2` endpoint can obtain a `true` return value for a transaction ID that corresponds to an internal Merkle tree node rather than a real Bitcoin transaction. Smart contracts or bridge applications that gate asset releases, cross-chain messages, or state transitions on this return value can be triggered with a completely fabricated transaction.

---

### Likelihood Explanation

The attack requires only public on-chain data (the block's Merkle tree, which is derivable from any Bitcoin block explorer). No privileged role, leaked key, or social engineering is needed. Any NEAR account can call `verify_transaction_inclusion_v2` directly.

---

### Recommendation

The coinbase check must enforce that `coinbase_tx_id` is a **leaf at the correct tree depth**, not merely that it hashes to the root. The standard approach is to require that the coinbase proof length equals `ceil(log2(tx_count))` — i.e., the actual depth of the Merkle tree — and to derive `tx_count` from the stored block header or a separately validated field. Alternatively, require the caller to supply the raw coinbase transaction bytes and verify the txid by hashing them, ensuring the "coinbase" cannot be an internal node.

---

### Proof of Concept

Consider a real Bitcoin block containing exactly 4 transactions: `[T0, T1, T2, T3]`.

```
Merkle tree:
  root = H(I01, I23)
  I01  = H(T0, T1)      ← internal node, depth 1
  I23  = H(T2, T3)      ← internal node, depth 1
```

Attacker submits `ProofArgsV2`:

```
coinbase_tx_id        = I01          // internal node, NOT the real coinbase
coinbase_merkle_proof = [I23]        // depth-1 proof
tx_id                 = I01          // same internal node as the "transaction"
tx_index              = 0
merkle_proof          = [I23]        // same depth-1 proof
confirmations         = 0
```

**Step 1 — length check:**
`len([I23]) == len([I23])` → passes.

**Step 2 — coinbase check:**
`compute_root_from_merkle_proof(I01, 0, [I23])`
= `H(I01, I23)` = `H(H(T0,T1), H(T2,T3))` = `root` → passes.

**Step 3 — transaction check (inside `verify_transaction_inclusion`):**
`compute_root_from_merkle_proof(I01, 0, [I23])` = `root` → passes.

`verify_transaction_inclusion_v2` returns **`true`** for the internal node `I01`, which is not a real transaction. The 64-byte forgery mitigation is completely bypassed. [5](#0-4) [2](#0-1) [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
