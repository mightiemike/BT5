### Title
Permanent Loss of Accrued Bonus Rewards on Full Redelegation Slash with Empty Bonus Pool — (`x/tieredrewards/keeper/slash.go`)

### Summary
When `BeforeRedelegationSlashed` fires for a full slash (`sharesToUnbond >= pos.Delegation.Shares`) and the bonus pool is empty, `slashRedelegationPosition` swallows `ErrInsufficientBonusPool` from `processEventsAndClaimBonus`, then calls `ClearBonusCheckpoints()` and saves the position as undelegated. The position owner permanently loses all accrued bonus rewards with no path to recovery.

### Finding Description
`processEventsAndClaimBonus` mutates the `pos` pointer (advancing `LastEventSeq`, `LastBonusAccrual`, `LastKnownBonded`) and decrements event reference counts in the store **before** checking pool sufficiency at line 230. When `ErrInsufficientBonusPool` is returned, these side-effects are already committed.

Back in `slashRedelegationPosition`, the error is intentionally swallowed (to prevent chain halt). For a full slash, the code then sets `pos.Delegation = nil` and calls `pos.ClearBonusCheckpoints()`, resetting `LastEventSeq` to 0 and `LastBonusAccrual` to the zero time. The position is saved in this state via `setPositionWithState`.

The result is a position that:
- Has `Delegation = nil` (so `IsDelegated()` returns false)
- Has all bonus checkpoints zeroed
- Has had its event ref counts decremented in the store (permanent)
- Has paid zero bonus to the owner

Any subsequent call to `processEventsAndClaimBonus` on this position returns immediately with zero coins because of the `!pos.IsDelegated()` guard. The user cannot re-delegate the same position (it is fully slashed). The accrued bonus is permanently unrecoverable.

### Impact Explanation
Position owners who hold a redelegating position and are subject to a full redelegation slash at a time when the bonus pool is empty permanently lose all bonus rewards that accrued since their last checkpoint. This is a direct, irreversible financial loss. The invariant that a position owner must either receive their bonus or retain the ability to claim it is broken.

A secondary impact is that event reference counts are decremented without a corresponding position checkpoint update, potentially causing premature garbage-collection of validator events that other positions on the same validator still reference.

### Likelihood Explanation
The bonus pool can be empty through natural depletion (no governance action required). Full redelegation slashes occur when a validator that is the source of a redelegation commits a double-sign infraction. Both conditions are independently reachable on a live chain. Their co-occurrence is uncommon but requires no privileged access, no operator compromise, and no social engineering.

### Recommendation
Separate the checkpoint-advancement and ref-count-decrement side-effects from the payment step in `processEventsAndClaimBonus`. Specifically:

1. Do not call `applyBonusAccrualCheckpoint`, `UpdateLastEventSeq`, or `decrementEventRefCount` until after `sufficientBonusPoolBalance` succeeds.
2. Alternatively, in `slashRedelegationPosition`, when `ErrInsufficientBonusPool` is swallowed for a full slash, do **not** call `ClearBonusCheckpoints()`. Instead, record the owed bonus amount in a separate claimable-debt store keyed by position ID, so the owner can claim it once the pool is refilled.
3. At minimum, ensure that ref count decrements are not committed to the store when the bonus payment fails.

	event, err := k.ValidatorEvents.Get(ctx, key)

		return nil // already cleaned up

	}

		return k.ValidatorEvents.Remove(ctx, key)
	}

	event.ReferenceCount--
