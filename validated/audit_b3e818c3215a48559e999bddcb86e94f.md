Audit Report

## Title
`SwapAllowlistExtension` gates on the router address instead of the end-user, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `sender` is the pool's `msg.sender` at the time of the `swap` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end-user. The extension therefore checks whether the router is allowlisted, not whether the individual user is allowlisted. Any non-allowlisted user can bypass the curated-pool gate by calling any router entry-point instead of calling `pool.swap` directly.

## Finding Description

**Root cause:** `MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // <-- pool's msg.sender, i.e. the router when routed
  recipient, ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to every configured extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L149-177
abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

Here `msg.sender` is the pool and `sender` is the router — not the end-user.

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making itself the pool's `msg.sender`:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(params.recipient, params.zeroForOne, ..., "", params.extensionData);
```

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

**Exploit flow:**
1. Pool admin deploys pool with `SwapAllowlistExtension` as a `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` — required for any router-mediated swap to succeed.
3. Pool admin does **not** call `setAllowedToSwap(pool, attacker, true)`.
4. Attacker calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
5. Pool calls `_beforeSwap(msg.sender=router, ...)`.
6. Extension checks `allowedSwapper[pool][router]` → `true`.
7. Swap executes. The attacker, who is not on the allowlist, has successfully traded on the curated pool.

**Contrast with `DepositAllowlistExtension`:** The deposit extension correctly gates on `owner` (the position owner, the economic actor), not on `sender` (the caller of `addLiquidity`):

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L38
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
```

The swap allowlist should follow the same pattern but has no equivalent `recipient` or payer field to gate on.

**Existing guards are insufficient:** There is no mechanism in the current extension interface to recover the true end-user. The `extensionData` field is caller-supplied and unauthenticated, so it cannot be trusted as a source of the real user identity without a convention enforced by the router.

## Impact Explanation

The pool admin configures the allowlist to gate individual swapper addresses. The invariant — only allowlisted addresses may trade on the curated pool — is broken in two symmetric ways:

1. **Allowlist bypass (direct loss path):** If the pool admin allowlists the router so that allowlisted users can trade through it, every non-allowlisted user can also trade through the router. The gate is fully open to the public. A disallowed user can execute swaps on a curated pool, draining LP value or extracting oracle-anchored profit that the pool admin intended to restrict to a specific counterparty set. This is a direct loss of LP principal and protocol fees.

2. **Broken core functionality:** If the pool admin does not allowlist the router, allowlisted users cannot use the canonical periphery path at all. The allowlist becomes incompatible with the router.

Both outcomes constitute broken core pool functionality causing loss of funds or unusable swap flows, meeting Sherlock contest thresholds.

## Likelihood Explanation

`SwapAllowlistExtension` is a production extension explicitly listed as a supported guard. `MetricOmmSimpleRouter` is the canonical user-facing swap interface. Any pool that deploys both — the expected production configuration for a curated pool — is immediately vulnerable. No special conditions, privileged access, flash loans, or unusual token behavior is required. Any public user can call the router. The bypass is deterministic and repeatable.

## Recommendation

The `beforeSwap` hook must gate on the **original end-user**, not on the immediate caller of the pool.

**Short term:** Have the router populate `extensionData` with `abi.encode(msg.sender)` before calling the pool, and have the extension decode and verify this value. This requires a convention between the router and the extension and does not protect against users who call `pool.swap` directly with a spoofed `extensionData`.

**Long term (preferred):** Change the `sender` argument forwarded to extensions to carry the **economic actor** — the address that will pay for the swap — rather than the pool's `msg.sender`. The router already tracks the true payer in transient storage via `_getPayer()`; this value should be forwarded as `sender` to the pool's `swap` call, or the extension interface should include a separate `payer` field. The `DepositAllowlistExtension` correctly gates on `owner` rather than `sender`; the swap allowlist should follow the same pattern.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured as a `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, router, true)`.
3. Pool admin does **not** call `setAllowedToSwap(pool, attacker, true)`.
4. Attacker calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. Pool calls `_beforeSwap(msg.sender=router, ...)`.
6. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][router]` → `true`.
7. Swap executes successfully for the non-allowlisted attacker.

The `FullMetricExtensionTest` integration test at `metric-periphery/test/extensions/FullMetricExtension.t.sol` (lines 55–74) demonstrates the allowlist working correctly for direct pool calls (`_swap` uses `callers[0]` directly) but does not test the router-mediated path, leaving this bypass untested.
