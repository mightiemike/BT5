### Title
Static `ShardedRpcNode.tracked_shards` Never Updated After Resharding Causes Stale Shard-Routing in the Sharded RPC Pool - (File: `chain/jsonrpc/src/sharded_rpc.rs`)

### Summary

`ShardedRpcPool` stores each remote node's shard coverage as a static `Vec<ShardId>` set once at pool initialization from the JSON config. After a dynamic resharding event, parent shards are replaced by child shards with new `ShardId`s. The routing functions `nodes_for_account_in_epochs` and `nodes_for_shard_in_epochs` check `node.tracked_shards.contains(&shard_id)` against the post-resharding child `ShardId`, which was never present in the static config. The pool therefore concludes no remote node covers the child shard and either falls back to the local node (which may also not track it) or returns `UNAVAILABLE_SHARD`, silently dropping queries that a remote node could have served.

### Finding Description

`ShardedRpcNode` holds a `tracked_shards: Vec<ShardId>` field populated once at construction from `ShardedRpcNodeConfig.tracked_shards`:

```rust
// chain/jsonrpc/src/sharded_rpc.rs
pub struct ShardedRpcNode {
    pub client: Arc<JsonRpcClient>,
    pub tracked_shards: Vec<ShardId>,   // ← set once, never updated
}
``` [1](#0-0) 

At pool construction, `tracked_shards` is copied verbatim from the static JSON config field `ShardedRpcNodeConfig.tracked_shards`:

```rust
.map(|node_config| ShardedRpcNode {
    client: ...,
    tracked_shards: node_config.tracked_shards.clone(),  // ← static snapshot
})
``` [2](#0-1) 

The routing function `nodes_for_shard_in_epochs` explicitly acknowledges the static nature and performs a plain membership check:

```rust
// Check remote nodes. Their `tracked_shards` is a static config and
// not epoch-dependent, so a single membership check is enough.
for node in &self.nodes {
    if node.tracked_shards.contains(&shard_id) {
        result.push(RpcNodeHandle::RemoteNode(node.client.clone()));
    }
}
``` [3](#0-2) 

`nodes_for_account_in_epochs` has the same pattern:

```rust
for node in &self.nodes {
    if epoch_shard_ids
        .iter()
        .any(|(_epoch_id, shard_id)| node.tracked_shards.contains(shard_id))
    {
        result.push(RpcNodeHandle::RemoteNode(node.client.clone()));
    }
}
``` [4](#0-3) 

After a resharding event, a parent shard (e.g., `ShardId(1)`) is replaced by two child shards (e.g., `ShardId(2)`, `ShardId(3)`). The remote node's `ShardTracker` correctly handles descendant tracking via `check_if_descendant_of_tracked_shard`, but the pool's static `tracked_shards` list still contains only `ShardId(1)`. The membership check `node.tracked_shards.contains(&ShardId(2))` returns `false`, so the pool never routes to that remote node for child-shard queries.

The local node's tracking is correctly computed via `shard_tracker.rpc_tracks_shard_at_epoch()` which is epoch-aware:

```rust
match self.shard_tracker.rpc_tracks_shard_at_epoch(shard_id, epoch_id) {
    Ok(true) => { result.push(RpcNodeHandle::LocalNode); break; }
    ...
}
``` [5](#0-4) 

The asymmetry is the root cause: the local node's coverage is re-evaluated per epoch; remote nodes' coverage is frozen at startup.

The codebase itself acknowledges this is broken. The test `test_rpc_query_after_resharding` is marked `#[ignore]` with the explicit comment:

```
// TODO(sharded-rpc): currently ignored because the RPC pool's tracked_shards
// is static and becomes stale after resharding — child shards are unknown to
// the pool, causing UNAVAILABLE_SHARD errors.
``` [6](#0-5) 

The `ShardedRpcNodeConfig` type in the public JSON config also documents this as static:

```rust
pub struct ShardedRpcNodeConfig {
    pub address: String,
    /// Shards that this node tracks (static configuration for now)
    pub tracked_shards: Vec<ShardId>,
}
``` [7](#0-6) 

### Impact Explanation

After any dynamic resharding event, the sharded RPC pool's routing table for remote nodes becomes permanently incorrect. Queries for accounts whose shard was split are never forwarded to the remote node that actually holds the data. The pool falls back to the local node; if the local node also does not track the child shard, the query returns `UNAVAILABLE_SHARD`. This silently breaks the availability guarantee of the sharded RPC service for all post-resharding child shards, for the entire lifetime of the running process (until restart with an updated config).

### Likelihood Explanation

Dynamic resharding is a planned, protocol-level event that will occur on mainnet. The `ShardedRpcPool` is a production feature configured via `config.json` (`rpc.sharded_rpc`) and deployed in the sharded benchmark environment. Any RPC query for an account in a child shard after resharding triggers the bug. No special privileges are required; any user submitting a standard JSON-RPC query (e.g., `query`, `block_effects`) after a resharding epoch boundary hits this path.

### Recommendation

Replace the static `Vec<ShardId>` in `ShardedRpcNode` with an epoch-aware lookup. Two concrete approaches:

1. **Query the remote node's `tracked_shards` dynamically**: Add a new JSON-RPC endpoint (e.g., `tracked_shards_at_epoch`) that each pool node exposes, and call it during routing to get the current tracked set for the relevant epoch.

2. **Propagate `ShardTracker` semantics to remote nodes**: Store `Vec<ShardUId>` (which encodes the shard layout version) instead of `Vec<ShardId>`, and apply the same `check_if_descendant_of_tracked_shard` logic used for the local node when evaluating remote nodes.

The `ShardTracker::check_if_descendant_of_tracked_shard` already implements the correct ancestor-walk logic: [8](#0-7) 

The same logic should be applied to remote nodes' shard coverage, keyed by `ShardUId` rather than raw `ShardId`.

### Proof of Concept

1. Configure a sharded RPC pool with two nodes: `rpc0` tracking `ShardId(0)` and `rpc1` tracking `ShardId(1)` (matching the `ReshardingRpcHarness` setup).
2. Trigger a dynamic resharding that splits `ShardId(1)` into `ShardId(2)` and `ShardId(3)`.
3. After resharding, query an account that lands in `ShardId(2)` or `ShardId(3)`.
4. `nodes_for_account_in_epochs` resolves the account to `ShardId(2)` in the post-resharding epoch.
5. `node.tracked_shards.contains(&ShardId(2))` returns `false` for both `rpc0` (`[0]`) and `rpc1` (`[1]`).
6. No remote node is selected; the pool falls back to `LocalNode`.
7. If the local node does not track `ShardId(2)`, the query returns `UNAVAILABLE_SHARD`.

This is exactly the scenario the `#[ignore]`d test `test_rpc_query_after_resharding` in `test-loop-tests/src/tests/sharded_rpc_resharding.rs` was written to cover but cannot pass due to this bug. [9](#0-8)
