### Title
NLP Burners Cannot Specify Minimum Quote Amount Received, Exposing Them to Unfavorable Oracle Price Execution ‚Äî (`core/contracts/Clearinghouse.sol`)

---

### Summary

The `BurnNlp` struct that a user signs contains only `sender`, `nlpAmount`, and `nonce`. The `oraclePriceX18` that determines the quote amount returned to the user is supplied by the sequencer in the outer `SignedBurnNlp` wrapper and is not constrained by the user's EIP-712 signature. There is no `minQuoteAmount` field anywhere in the user-controlled signed data. As a result, an NLP burner has no on-chain mechanism to enforce a minimum acceptable quote amount, and can receive significantly less than expected if the oracle price moves adversely between signing and execution.

---

### Finding Description

The `BurnNlp` struct that the user signs is defined as:

```solidity
struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint64 nonce;
}
``` [1](#0-0) 

The outer `SignedBurnNlp` struct, which the sequencer constructs and submits, carries the oracle price:

```solidity
struct SignedBurnNlp {
    BurnNlp tx;
    bytes signature;
    int128 oraclePriceX18;
    int128[] nlpPoolRebalanceX18;
}
``` [2](#0-1) 

In `Clearinghouse.burnNlp`, the quote amount returned to the user is computed entirely from the sequencer-supplied `oraclePriceX18`:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
``` [3](#0-2) 

The user's signed `BurnNlp` struct contains no field that bounds the minimum acceptable `quoteAmount`. The sequencer is free to use any `oraclePriceX18` value when it batches and submits the transaction. There is no on-chain check of the form `require(quoteAmount >= txn.minQuoteAmount)`.

Compare this to `MintNlp`, where the user explicitly specifies `quoteAmount` (the exact amount they spend), giving them full control over their input cost:

```solidity
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;
    uint64 nonce;
}
``` [4](#0-3) 

For burning, the asymmetry is clear: the user controls the NLP input but has no control over the quote output.

---

### Impact Explanation

An NLP holder who signs a `BurnNlp` transaction expecting to receive `Q` quote tokens (based on the oracle price at signing time) may receive materially less if:

1. The oracle price drops between signing and sequencer execution (normal market movement).
2. The sequencer delays execution while the price falls.

The received quote is `nlpAmount √ó oraclePriceX18 ‚àí burnFee`. A 10% drop in `oraclePriceX18` translates directly to a ~10% reduction in the quote received, with no floor enforced by the contract. For large NLP positions this represents a significant, unprotected financial loss.

---

### Likelihood Explanation

NLP burn transactions are submitted off-chain and batched by the sequencer via `submitTransactionsChecked`. There is inherent latency between a user signing a `BurnNlp` and the sequencer including it in a batch. During volatile market conditions, the oracle price used at execution time can differ substantially from the price at signing time. This is a realistic, non-adversarial scenario that requires no special conditions beyond normal market price movement. [5](#0-4) 

---

### Recommendation

Add a `minQuoteAmount` field to the `BurnNlp` struct so that the user's EIP-712 signature covers their minimum acceptable output:

```solidity
struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint128 minQuoteAmount; // slippage protection
    uint64 nonce;
}
```

In `Clearinghouse.burnNlp`, enforce it after computing `quoteAmount`:

```solidity
require(quoteAmount >= int128(txn.minQuoteAmount), ERR_SLIPPAGE_TOO_HIGH);
``` [6](#0-5) 

---

### Proof of Concept

1. Alice holds 1,000 NLP tokens. At signing time, `oraclePriceX18 = 10e18` ($10/NLP), so she expects to receive ~$9,990 in quote (after the 0.1% burn fee).
2. Alice signs and submits a `BurnNlp` with `nlpAmount = 1000e18`. Her signed struct contains no minimum quote constraint.
3. Before the sequencer includes her transaction, the oracle price drops to `oraclePriceX18 = 9e18` ($9/NLP).
4. The sequencer submits `SignedBurnNlp` with the new lower price.
5. `Clearinghouse.burnNlp` executes: `quoteAmount = 1000e18 √ó 9e18 / 1e18 ‚àí fee ‚âà $8,991`.
6. Alice receives ~$8,991 instead of ~$9,990 ‚Äî a ~$999 (~10%) loss she had no ability to prevent. [7](#0-6)

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);

    function submitTransactionsChecked(
        uint64 idx,
        bytes[] calldata transactions,
        bytes32 e,
        bytes32 s,
        uint8 signerBitmask
    ) external {
        validateSubmissionIdx(idx);
        require(msg.sender == sequencer);
        // TODO: if one of these transactions fails this means the sequencer is in an error state
        // we should probably record this, and engage some sort of recovery mode

        bytes32 digest = keccak256(abi.encode(idx));
        for (uint256 i = 0; i < transactions.length; ++i) {
            digest = keccak256(abi.encodePacked(digest, transactions[i]));
        }
        verifier.requireValidSignature(digest, e, s, signerBitmask);

        for (uint256 i = 0; i < transactions.length; i++) {
            bytes calldata transaction = transactions[i];
            processTransaction(transaction);
            nSubmissions += 1;
        }
    }
