Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Original Caller, Allowing Allowlist Bypass via Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension::beforeSwap` receives `sender = router_address` (the pool's immediate `msg.sender`) when a swap is routed through `MetricOmmSimpleRouter`. The hook checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][original_user]`. If the router is allowlisted to enable router-mediated swaps for legitimate users, every address on the network can bypass the allowlist by calling through the router, completely nullifying the extension's access control.

## Finding Description
When a user calls `MetricOmmSimpleRouter::exactInputSingle` (or any `exact*` variant), the router stores the original `msg.sender` only in transient callback context for payment purposes (`_setNextCallbackContext`, line 71), then calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` directly (lines 72–80). The pool's `msg.sender` is therefore the router. Inside `MetricOmmPool::swap`, `_beforeSwap(msg.sender, ...)` is called at line 231, passing the router address as `sender`. `ExtensionCalling::_beforeSwap` encodes that router address and dispatches to the extension (lines 149–177). `SwapAllowlistExtension::beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]` (line 37) where `msg.sender` is the pool and `sender` is the router — not the original user. The original user's identity is permanently lost at the pool boundary. This creates a binary failure: if the router is not allowlisted, all allowlisted users are blocked from using the router; if the router is allowlisted (the only way to enable router-mediated swaps), every user bypasses the allowlist.

## Impact Explanation
The `SwapAllowlistExtension` is the protocol's mechanism for restricting pool access to specific addresses (e.g., institutional traders, whitelisted market makers). For any pool using this extension where the router is allowlisted, the access control is completely nullified: any unprivileged user can execute swaps in the restricted pool by routing through `MetricOmmSimpleRouter`. This constitutes broken core pool functionality — the extension's sole purpose is defeated — and enables unauthorized actors to interact with pools whose LPs and admins explicitly intended to restrict access, potentially draining liquidity through unrestricted trading.

## Likelihood Explanation
The router is the standard, documented entrypoint for swaps. Pool admins who want their allowlisted users to use the router must call `setAllowedToSwap(pool, router, true)`, which simultaneously opens the pool to all users. There is no in-protocol warning, guard, or alternative mechanism. This is a near-certain misconfiguration path for any pool that uses both `SwapAllowlistExtension` and the router.

## Recommendation
The extension must verify the original initiator, not the immediate caller. The preferred fix is to require the router to forward the original caller's address in `extensionData`, and have the extension decode and verify it — with the pool or factory enforcing that only trusted routers can set this field. Alternatively, document explicitly that pools using `SwapAllowlistExtension` must allowlist individual users directly and must NOT allowlist the router, and that router-mediated swaps are architecturally incompatible with this extension.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router usage for allowlisted users.
3. Unprivileged attacker (not in the allowlist) calls `MetricOmmSimpleRouter::exactInputSingle` targeting the restricted pool.
4. The router calls `pool.swap(recipient, ...)` — pool's `msg.sender` is the router.
5. Pool calls `_beforeSwap(router, ...)`. Extension checks `allowedSwapper[pool][router] == true`. Check passes.
6. Attacker's swap executes successfully despite not being allowlisted.
