### Title
Unauthenticated Invoke Injection via Zero-Nonce Validation Skip — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` in the Apollo gateway uses `account_nonce == Nonce(Felt::ZERO)` as a sentinel for "account not yet deployed" and skips the `__validate__` entry-point (signature check) for any invoke with `nonce == 1` when the mempool reports *any* transaction from that address. Because the mempool check is not restricted to deploy-account transactions, an unprivileged attacker who observes a victim's pending `deploy_account` in the mempool can inject an invoke with an arbitrary (invalid) signature for the victim's address, bypassing signature verification entirely and landing the transaction in the mempool.

---

### Finding Description

**Invariant broken:** Every transaction admitted to the mempool must have passed `__validate__` (signature verification) or have a cryptographically sound reason to skip it.

**Root cause — `skip_stateful_validations`:**

```
crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 429-461
```

```rust

    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // ...

                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                // ...
        }
    }
    Ok(false)
}
```

The function returns `true` (skip `__validate__`) when:
1. The transaction is an `Invoke`.
2. `tx.nonce() == 1`.
3. `account_nonce == 0` (on-chain state — account not yet deployed).
4. `account_tx_in_pool_or_recent_block(sender)` returns `true`.

**The flaw in condition 4:** `account_tx_in_pool_or_recent_block` is implemented as:

```
crates/apollo_mempool/src/mempool.rs  lines 697-700
```

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

This returns `true` if *any* transaction from the address is present — not specifically a `deploy_account`. The comment in `skip_stateful_validations` acknowledges this:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

This reasoning is circular: the "future nonces that passed validations" could themselves have been admitted via this same skip path.

**Contrast with the Python/native path** (`crates/native_blockifier/src/py_validator.rs` lines 109-110):

```rust
let deploy_account_not_processed =
    deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
```

The Python wrapper requires the caller to supply an explicit `deploy_account_tx_hash`. The Rust gateway path has no such requirement.

**Attack path:**

1. Victim submits `deploy_account` for address `A`. It enters the mempool. On-chain nonce for `A` is still `0`.
2. Attacker submits `Invoke { sender_address: A, nonce: 1, signature: [0x0, 0x0] }` (arbitrary invalid signature).
3. Stateless validator passes (no signature check).
4. `validate_nonce`: `0 ≤ 1 ≤ max_allowed_nonce_gap` — passes.
5. `validate_by_mempool`: nonce range check passes.
6. `skip_stateful_validations`: `nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(A)==true` (victim's deploy_account is there) → returns `true`.
7. `run_validate_entry_point` is called with `validate: false` → `__validate__` is **not executed**.
8. Invalid-signature invoke is admitted to the mempool.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker can inject signature-invalid invoke transactions for any account that has a pending `deploy_account` in the mempool. Consequences:
- The attacker's nonce-1 invoke occupies the slot, preventing the victim's legitimate nonce-1 invoke from being accepted (duplicate-nonce rejection).
- The victim's `deploy_account + invoke` UX flow is broken: the deploy_account commits, but the paired invoke was replaced by the attacker's invalid one, which will revert or be rejected at execution time.
- The mempool can be flooded with invalid-signature transactions for all accounts currently in the "pending deploy" state, degrading sequencer throughput.

---

### Likelihood Explanation

**Medium.** The attacker only needs to monitor the public mempool for `deploy_account` transactions (observable via RPC or P2P gossip), then race to submit a crafted invoke before the victim's own nonce-1 invoke arrives. No privileged access, no special keys, and no on-chain funds are required. The window is the time between the victim's `deploy_account` entering the mempool and the victim's nonce-1 invoke being submitted.

---

### Recommendation

Replace the weak `account_tx_in_pool_or_recent_block` check with a type-specific check that verifies a `deploy_account` transaction (not just any transaction) is pending for the address. Concretely, expose a `deploy_account_tx_in_pool(address)` query from the mempool, or pass the expected `deploy_account` tx hash through the gateway flow (mirroring the Python path in `PyValidator::should_run_stateful_validations`).

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
