### Title
`op_multiply` Cost Undercharge Due to Mixed Byte-Length Scales in Running Product Tracking — (`File: src/more_ops.rs`)

### Summary

`op_multiply` tracks the byte-size of its running product using `limbs_for_int(&total)`, which computes the **magnitude byte count** (stripping the CLVM sign byte). All subsequent argument sizes (`l1`) are measured using the **actual CLVM atom byte length** (including the sign byte for positive numbers whose high bit is set). These two scales are mixed directly in the cost formula, causing systematic cost undercharging whenever the running product is a positive number whose most-significant bit is set.

### Finding Description

In `op_multiply`, after each multiplication step, `l0` is updated as:

```rust
l0 = limbs_for_int(&total);
``` [1](#0-0) 

`limbs_for_int` is defined as:

```rust
fn limbs_for_int(v: &Number) -> usize {
    v.bits().div_ceil(8) as usize
}
``` [2](#0-1) 

This computes the number of bytes needed to represent the **magnitude** of the integer. However, CLVM uses signed big-endian encoding, which requires an extra leading `0x00` byte whenever the high bit of the magnitude is set (to distinguish positive from negative). For example, the value `128` is encoded as `0x0080` (2 bytes), but `limbs_for_int(128)` returns `1` because `128.bits() = 8`, and `8.div_ceil(8) = 1`.

The internal test confirms this intentional stripping:

```rust
// redundant leading zeros don't count, since they aren't stored internally
let expected = if !bytes.is_empty() && bytes[0] == 0 {
    bytes.len() - 1
} else {
    bytes.len()
};
assert_eq!(limbs_for_int(&bigint), expected);
``` [3](#0-2) 

Meanwhile, `l1` for each subsequent argument is measured using the **actual atom byte length**:

- For `Buffer` atoms: `buf.len()` (actual bytes on the wire, including sign byte)
- For `U32` atoms: `len_for_value(val)` (actual CLVM encoding byte length) [4](#0-3) 

The cost formula then mixes these two different scales:

```rust
cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;
cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
``` [5](#0-4) 

Whenever `total` is a positive number with its high bit set (e.g., 128, 32768, 2^(8k−1) for any k), `l0 = limbs_for_int(&total)` is exactly **1 byte less** than the actual CLVM atom size of `total`. The cost for the next multiplication step is therefore undercharged by:

```
Δcost = l1 * MUL_LINEAR_COST_PER_BYTE + l1 / MUL_SQUARE_COST_PER_BYTE_DIVIDER
      = l1 * 6 + l1 / 128
```

For `l1 = 256` (the maximum allowed), this is `≈ 1538` cost units per multiplication step.

Note also that the **first iteration** uses `int_atom` to set `l0`, which returns the actual atom byte length — so the first `l0` is on the correct scale. Only subsequent updates via `limbs_for_int` introduce the scale mismatch. [6](#0-5) 

### Impact Explanation

The cost charged for `op_multiply` is systematically lower than the intended model whenever the running product has its high bit set. This has two concrete consequences:

1. **Undercharged execution**: Programs using `op_multiply` with operands that produce intermediate products with high bits set will consume less cost than intended, potentially allowing programs that should exceed `max_cost` to pass validation.

2. **Consensus divergence**: If the Python reference CLVM implementation computes `l0` using the actual atom byte length (the correct scale), then for any program where the product has its high bit set, the two implementations compute different costs. A program could be accepted by one implementation and rejected by the other, causing a chain split.

### Likelihood Explanation

This is triggered by any `op_multiply` call where the running product is a positive number with its high bit set. This is a common arithmetic outcome — for example, multiplying `64 * 2 = 128` (0x80, high bit set), or `16384 * 2 = 32768` (0x8000, high bit set). An attacker can deliberately craft CLVM programs that repeatedly land the product in this state to maximize the accumulated undercharge. The entry path is fully attacker-controlled: the CLVM program bytes and arguments are supplied externally.

### Recommendation

Replace `limbs_for_int` with a function that computes the actual CLVM atom byte length of the running product, consistent with how `l1` is measured. Specifically, the byte length of a positive `Number` in CLVM signed big-endian encoding is:

```
actual_len = magnitude_bytes + (1 if high bit of magnitude is set else 0)
```

A correct replacement for line 649 would be:

```rust
let bits = total.bits() as usize;
l0 = bits.div_ceil(8) + if bits % 8 == 0 && bits > 0 { 1 } else { 0 };
```

This ensures `l0` and `l1` are always measured on the same scale (actual CLVM atom byte length), eliminating the arithmetic semantic mismatch.

### Proof of Concept

A CLVM program that multiplies `128` (encoded as `0x0080`, 2 bytes) by a 256-byte atom:

- First iteration: `l0 = int_atom(...).1 = 2` (correct, actual byte length)
- After multiplication: `total = 128 * large_number`, `l0 = limbs_for_int(&total)` — if `total` has its high bit set, `l0` is underestimated by 1
- Cost charged: `(l0 + l1) * 6 + (l0 * l1) / 128` with `l0` = actual − 1
- Cost undercharge per step: `l1 * 6 + l1/128 ≈ 1538` for `l1 = 256`

Repeating this pattern across multiple multiplication steps accumulates the undercharge, allowing a program to consume more computation than its reported cost reflects.
