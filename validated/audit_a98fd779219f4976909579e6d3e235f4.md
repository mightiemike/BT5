### Title
Chain Reorg Silently Promotes Fork Blocks to Canonical Chain Without Per-Block Log Events — (File: `contract/src/lib.rs`)

---

### Summary

The `submit_block_header_inner` function emits a `log!("Block {}: saving to mainchain", ...)` event for every block added to the canonical chain via the normal submission path. However, when a chain reorganization is triggered via `reorg_chain`, every fork block promoted into the canonical chain updates the same two lookup maps (`mainchain_height_to_header`, `mainchain_header_to_height`) with no equivalent per-block log event. Off-chain systems that rely on NEAR transaction logs to track canonical chain state will silently diverge from on-chain reality after any reorg.

---

### Finding Description

**Normal main-chain submission path** (`submit_block_header_inner`, lines 537–548):

When the incoming block's parent is the current tip, the contract emits a per-block log and then calls `store_block_header`, which inserts the block into both canonical-chain maps:

```rust
// contract/src/lib.rs  lines 540–548
log!("Block {}: saving to mainchain", current_header.block_hash);
// ...
self.store_block_header(&current_header);
self.mainchain_tip_blockhash = current_header.block_hash;
```

`store_block_header` (lines 650–656) inserts into `mainchain_height_to_header` and `mainchain_header_to_height`.

**Reorg path** (`reorg_chain`, lines 575–647):

When a fork's accumulated chainwork exceeds the current main chain, `reorg_chain` is called. Inside it, every fork block is promoted into the canonical chain maps:

```rust
// contract/src/lib.rs  lines 628–630
let main_chain_block = self
    .mainchain_height_to_header
    .insert(&current_height, &current_block_hash);
self.mainchain_header_to_height
    .insert(&current_block_hash, &current_height);
```

No per-block log event is emitted anywhere inside `reorg_chain`. The only log that exists is the single `log!("Chain reorg")` emitted at line 564 *before* `reorg_chain` is called — it carries no information about which block hashes are being installed at which heights.

**The desynchronization**: both code paths write to the same canonical-chain maps, but only the normal path emits per-block log events. Any off-chain system that reconstructs canonical chain state from NEAR logs will have a correct view during normal operation and a silently stale/incorrect view after every reorg.

---

### Impact Explanation

NEAR smart-contract logs are the standard mechanism for off-chain indexers, bridge consumers, and monitoring infrastructure to track on-chain state changes without polling every getter. A relayer, bridge verifier, or downstream contract that watches for `"saving to mainchain"` log lines to maintain a local mapping of `height → block_hash` will:

1. Never learn which fork blocks were promoted during a reorg.
2. Continue to believe the pre-reorg block hashes are canonical at those heights.
3. Potentially submit or accept Merkle inclusion proofs anchored to stale (now non-canonical) block hashes.

The on-chain proof-verification functions (`verify_transaction_inclusion`, `verify_transaction_inclusion_v2`) use on-chain state and are not themselves affected. The impact falls entirely on off-chain consumers that use logs as their source of truth for canonical chain state — a common and documented pattern in NEAR bridge infrastructure.

---

### Likelihood Explanation

Chain reorganizations are a normal, expected event in Bitcoin operation. Any unprivileged relayer can submit a sequence of fork headers with sufficient accumulated chainwork to trigger `reorg_chain`. No privileged role, leaked key, or social engineering is required. The trigger is fully attacker-controlled via the public `submit_blocks` entry point.

---

### Recommendation

Emit a per-block log event inside `reorg_chain` for each fork block promoted to the canonical chain, mirroring the `"saving to mainchain"` log in `submit_block_header_inner`. For example:

```rust
// inside the while loop in reorg_chain, after the inserts:
log!("Block {}: promoted to mainchain via reorg at height {}", current_block_hash, current_height);
```

This ensures that any off-chain observer watching NEAR logs receives a complete and consistent record of every canonical chain state change, regardless of whether it arrived via normal submission or reorg.

---

### Proof of Concept

1. Initialize the contract with a genesis block and a short main chain (e.g., heights 0–5).
2. Submit a fork chain branching from height 2 with higher cumulative chainwork than the main chain tip (heights 2–6 on the fork).
3. The contract calls `reorg_chain`; fork blocks at heights 2–6 are inserted into `mainchain_height_to_header` and `mainchain_header_to_height` with no per-block log events.
4. An off-chain observer replaying NEAR transaction logs sees `"saving to mainchain"` only for heights 0–5 (the original chain). It has no log evidence that heights 2–6 now map to different block hashes.
5. The observer's local `height → hash` table is now incorrect for heights 2–6, while `get_block_hash_by_height` on-chain returns the correct (reorged) hashes.
