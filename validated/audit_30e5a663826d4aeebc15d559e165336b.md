### Title
Difficulty Calculation for Fork Blocks Uses Stale Mainchain Ancestor State, Enabling Incorrect PoW Validation — (`contract/src/bitcoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

When validating a fork block at a difficulty-adjustment boundary, the contract computes the expected difficulty using `get_header_by_height`, which always resolves against the **mainchain** height-to-hash index. For fork blocks whose chain diverged before the difficulty-adjustment window's first block, this returns the wrong ancestor's timestamp, producing an incorrect difficulty target. The developers themselves flagged this with a `TODO` comment. This is a direct analog of the reported cross-module desynchronization class: a critical calculation uses cached state from one module (the mainchain index) when it must use live state from another module (the fork's actual ancestor chain).

---

### Finding Description

`get_header_by_height` in `lib.rs` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

It exclusively consults `mainchain_height_to_header`. Fork blocks are stored only in `headers_pool` via `store_fork_header`; they are **never** inserted into `mainchain_height_to_header`. [2](#0-1) 

In `bitcoin.rs`, at every 2016-block difficulty adjustment, the expected bits are computed by fetching the first block of the interval:

```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(config, prev_block_header, interval_tail_extend_header.block_header.time.into())
``` [3](#0-2) 

In `dogecoin.rs`, the same call is made, and the developers explicitly flagged the problem with a `TODO`:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [4](#0-3) 

**Concrete desynchronization scenario (Bitcoin):**

- Mainchain has blocks 0 … N. Block at height `B = 2016` (start of second difficulty period) has timestamp `T_main`.
- Attacker submits fork blocks starting at height `B = 2016`. The fork's block at height `B` carries timestamp `T_fork > T_main` (attacker-controlled, within the 2-hour future-timestamp allowance).
- Fork blocks `B … B+2015` are stored only in `headers_pool`, not in `mainchain_height_to_header`.
- When the attacker submits fork block at height `B + 2016` (next adjustment boundary), `get_next_work_required` calls `get_header_by_height(B)`, which returns the **mainchain's** block at `B` with timestamp `T_main`, not the fork's block with `T_fork`.
- `actual_time_taken = prev_fork_block.time − T_main`. Because `T_main < T_fork`, the computed timespan is **larger** than the fork's true timespan, yielding a **lower** required difficulty target.
- Fork blocks at and after `B + 2016` are accepted with less PoW than the protocol requires.

The same logic applies to Dogecoin's pre-145,000 difficulty periods via the identical `get_header_by_height` call. [5](#0-4) 

---

### Impact Explanation

An attacker who builds a fork diverging at or before a difficulty-adjustment boundary can cause the contract to compute a lower-than-correct difficulty target for subsequent fork blocks. Blocks with insufficient PoW pass `check_pow` and are stored. If the fork's accumulated `chain_work` eventually exceeds the mainchain's, `reorg_chain` is triggered, replacing the canonical chain with the attacker's fraudulent fork. All subsequent `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` calls operate against the corrupted canonical chain, returning incorrect proof results to consuming contracts. [6](#0-5) 

---

### Likelihood Explanation

The attacker must submit a fork of at least `difficulty_adjustment_interval` blocks (2016 for Bitcoin, smaller for Dogecoin pre-145k) before the manipulation takes effect. This requires valid PoW at the current difficulty for all preceding fork blocks, making the attack resource-intensive but not infeasible for a well-resourced adversary. The `TODO` comment in the production source confirms the developers identified this as an open correctness question, indicating the code path is reachable and the issue is unresolved. [7](#0-6) 

---

### Recommendation

Replace `get_header_by_height` in difficulty calculations with an ancestor-walk that follows `prev_block_hash` links from `prev_block_header` backward by `difficulty_adjustment_interval − 1` steps, using `get_prev_header`. This mirrors how `get_median_time_past` correctly traverses the chain:

```rust
// Correct: walk the fork's actual ancestor chain
let mut cursor = prev_block_header.clone();
for _ in 0..(difficulty_adjustment_interval - 1) {
    cursor = blocks_getter.get_prev_header(&cursor.block_header);
}
let first_block_time = cursor.block_header.time;
``` [8](#0-7) 

This is consistent with how Bitcoin Core resolves the interval-start block via `GetAncestor`, which walks the chain rather than using a height index.

---

### Proof of Concept

1. Deploy the contract with Bitcoin feature, genesis at height 0, `skip_pow_verification = false`.
2. Submit mainchain blocks 0 … 4031 (two full difficulty periods). Record the mainchain's block at height 2016 with timestamp `T_main`.
3. Construct a fork starting at height 2016. Set the fork's block at height 2016 with timestamp `T_fork = T_main + 7000` (within the 2-hour future-timestamp window at submission time).
4. Submit fork blocks 2016 … 4031 with valid PoW at the current difficulty.
5. Submit fork block 4032. The contract calls `get_header_by_height(2016)`, returns the mainchain block with `T_main`. The computed `actual_time_taken` is inflated by 7000 seconds relative to the fork's true timespan, producing a lower `expected_bits`.
6. Observe that fork block 4032 passes `check_pow` with a PoW hash that would fail against the correct difficulty target derived from the fork's own block at height 2016. [9](#0-8) [10](#0-9)

pub fn get_median_time_past(
    block_header: ExtendedHeader,
    prev_block_getter: &impl BlocksGetter,
) -> u32 {
    use btc_types::network::MEDIAN_TIME_SPAN;

    let mut median_time = [0u32; MEDIAN_TIME_SPAN];
    let mut current_header = block_header;

    for slot in &mut median_time {
        *slot = current_header.block_header.time;
        current_header = prev_block_getter.get_prev_header(&current_header.block_header);
    }

    median_time.sort_unstable();
    median_time[median_time.len() / 2]
}
