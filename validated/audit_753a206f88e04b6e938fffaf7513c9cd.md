### Title
Unchecked u64 Multiplication Overflow in `op_unknown` Cost Calculation Enables Undercharged Execution - (File: src/more_ops.rs)

### Summary
The `op_unknown` function in `src/more_ops.rs` multiplies a pre-validated `cost: u64` by `cost_multiplier + 1` without overflow protection. In Rust release builds, this multiplication silently wraps on overflow, producing a tiny cost value that passes the subsequent `u32::MAX` guard and is returned to the caller. An attacker can craft CLVM bytes with a large-multiplier unknown opcode and large arguments to trigger this wrap, causing the VM to accept the operation at a near-zero cost, undercharging execution and breaking cost-based consensus.

### Finding Description

In `op_unknown`, the cost multiplier is extracted from the opcode bytes as a `u32` (max `u32::MAX = 4_294_967_295`) and widened to `u64`: [1](#0-0) 

The per-argument cost is accumulated and checked against `max_cost` before the multiplication: [2](#0-1) 

The critical sequence is:
1. `check_cost(cost, max_cost)?` — bounds `cost` to at most `max_cost` (e.g., 11 billion on Chia mainnet).
2. `cost *= cost_multiplier + 1` — **unchecked u64 multiplication**. With `cost ≈ 11_000_000_000` and `cost_multiplier = u32::MAX`, the product is `≈ 4.7 × 10¹⁹`, which exceeds `u64::MAX ≈ 1.8 × 10¹⁹`. In a release build, Rust wraps this silently.
3. `if cost > u32::MAX as u64` — the post-wrap value is small and passes this guard.
4. `Ok(Reduction(cost as Cost, allocator.nil()))` — the wrapped, tiny cost is returned.

`Cost` is defined as `u64`: [3](#0-2) 

The cost_multiplier is bounded to `u32::MAX` by `u32_from_u8`: [4](#0-3) 

### Impact Explanation

The returned `Reduction(wrapped_cost, nil)` is accumulated by the `run_program` loop into the total execution cost. A wrapped cost of, say, `3` instead of the correct `47_000_000_000` means the program consumes almost no budget. An attacker can therefore execute an unknown opcode that should cost billions of gas units for effectively free, bypassing the cost limit. This is a **consensus-critical undercharged execution** bug: nodes compiled in release mode silently accept the program with a tiny cost, while debug builds panic on overflow, causing a consensus split between node configurations.

### Likelihood Explanation

The trigger requires only attacker-controlled CLVM bytes: an unknown opcode whose leading bytes encode a large `cost_multiplier` (up to `u32::MAX`) and whose cost-function bits select mode 1, 2, or 3 with enough argument bytes to push the pre-multiplication cost above `u64::MAX / (u32::MAX + 1) ≈ 4.3 billion`. This threshold is reachable within the standard 11-billion cost limit. The opcode format is fully documented and the byte layout is public, making crafting such a payload straightforward.

### Recommendation

Replace the bare multiplication with a checked or saturating variant and treat overflow as an error:

```rust
// Before (line 261):
cost *= cost_multiplier + 1;

// After:
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::CostExceeded)?;
```

This ensures that any overflow is treated as exceeding the cost limit rather than silently wrapping to a small value.

### Proof of Concept

Craft an unknown opcode atom with:
- Bytes `[0x00, 0xFF, 0xFF, 0xFF, 0xFF, 0x40]`: the first 5 bytes encode `cost_multiplier = 0xFFFFFFFF = 4_294_967_295`; the last byte has bits `01` in positions 7–6, selecting cost_function = 1 (arith-like).
- Pass enough large atom arguments so that the accumulated `cost` (before multiplication) reaches approximately `4_294_967_297` (just above `u64::MAX / 4_294_967_296`).
- `cost * (cost_multiplier + 1) = 4_294_967_297 * 4_294_967_296` overflows `u64`, wrapping to a value near `4_294_967_296` (≈ 4 billion), which is less than `u32::MAX * 2` and passes the guard.
- The function returns `Ok(Reduction(~4_294_967_296, nil))` instead of an error, accepting the operation at a fraction of its true cost. [2](#0-1)
