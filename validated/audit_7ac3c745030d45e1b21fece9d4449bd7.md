### Title
`delistProduct()` Gated by Deployer EOA Instead of Multisig Owner, Bypassing Governance Control — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.delistProduct()` is protected by `onlyDeployer` rather than `onlyOwner`. The deployer is a plain EOA set at initialization, while the owner is the multisig. This means the deployer EOA can unilaterally force-settle all perp positions for any set of subaccounts without multisig approval, bypassing the governance control that every other sensitive function in the same contract requires.

---

### Finding Description

`ContractOwner` maintains two distinct privileged roles:

- `deployer` — the EOA that called `initialize()`, stored as a plain address.
- `owner` — the multisig, set via `transferOwnership(multisig)` in the same `initialize()` call. [1](#0-0) 

Every high-impact administrative function in the contract is gated by `onlyOwner` (the multisig): `withdrawInsurance`, `depositInsurance`, `rebalanceXWithdraw`, `updateTierFeeRates`, `addEngine`, `assignPubKey`, `spotUpdateRisk`, `perpUpdateRisk`, `setWithdrawPool`, `setSpreads`, etc. [2](#0-1) 

However, `delistProduct()` is the sole function that submits a slow-mode transaction directly to the endpoint while being gated only by `onlyDeployer`: [3](#0-2) 

In `Clearinghouse.delistProduct()`, this transaction forcibly zeroes out every listed subaccount's perp position and credits/debits the corresponding quote amount at the supplied price: [4](#0-3) 

The price is constrained to the current oracle price (`require(txn.priceX18 == _getPriceX18(txn.productId))`), so direct price manipulation is blocked. However, the deployer EOA can still:

1. Unilaterally force-close all perp positions for any set of subaccounts on any perp product.
2. Trigger `tryCloseIsolatedSubaccount()` for isolated subaccounts as a side effect.
3. Do all of this without any multisig co-signature or governance vote.

The other `onlyDeployer` functions (`submitSpotAddOrUpdateProductCall`, `submitPerpAddOrUpdateProductCall`, `clearSpotAddOrUpdateProductCalls`, `clearPerpAddOrUpdateProductCalls`) only queue data that still requires a separate `onlyOwner` call to execute, so they carry a two-step check. `delistProduct` has no such second gate — it submits the slow-mode transaction directly. [5](#0-4) 

---

### Impact Explanation

**Severity: Low**

The deployer EOA has unilateral, single-key control over `delistProduct()`, a function that forcibly settles user perp positions. If the deployer key is compromised or acts maliciously, it can force-close positions for any set of subaccounts at the current oracle price without multisig approval. This is a governance bypass for a critical protocol operation. Direct financial theft via price manipulation is blocked by the oracle price check, but forced position closure at an inopportune oracle price can still cause user losses (e.g., forced closure during a price spike).

---

### Likelihood Explanation

**Likelihood: High**

`delistProduct()` is always controlled by the deployer EOA in the current implementation. The deployer key is a single EOA with no multisig protection, and there is no mechanism to rotate or revoke it. The asymmetry is permanent unless the contract is upgraded.

---

### Recommendation

Replace `onlyDeployer` with `onlyOwner` on `delistProduct()` in `ContractOwner.sol`, consistent with every other sensitive administrative function in the same contract. This ensures that force-delisting a perp product requires multisig approval.

---

### Proof of Concept

1. Deployer EOA calls `ContractOwner.delistProduct([productId], [currentOraclePrice], [victim_subaccounts])`.
2. `onlyDeployer` passes — no multisig required.
3. `_submitSlowModeTransaction(DelistProduct, ...)` enqueues the transaction.
4. When the sequencer processes it, `Clearinghouse.delistProduct()` is called.
5. For each subaccount in the list, `perpEngine.updateBalance()` zeroes the perp position and credits/debits quote at the oracle price — forcibly closing positions without user consent and without multisig governance approval. [3](#0-2) [6](#0-5)

    function delistProduct(bytes calldata transaction) external onlyEndpoint {
        IEndpoint.DelistProduct memory txn = abi.decode(
            transaction[1:],
            (IEndpoint.DelistProduct)
        );
        // only perp can be delisted
        require(
            productToEngine[txn.productId] == _perpEngine(),
            ERR_INVALID_PRODUCT
        );
        require(txn.priceX18 == _getPriceX18(txn.productId), ERR_INVALID_PRICE);
        IPerpEngine perpEngine = _perpEngine();
        for (uint256 i = 0; i < txn.subaccounts.length; i++) {
            IPerpEngine.Balance memory balance = perpEngine.getBalance(
                txn.productId,
                txn.subaccounts[i]
            );
            int128 baseDelta = -balance.amount;
            int128 quoteDelta = -baseDelta.mul(txn.priceX18);
            perpEngine.updateBalance(
                txn.productId,
                txn.subaccounts[i],
                baseDelta,
                quoteDelta
            );
            if (RiskHelper.isIsolatedSubaccount(txn.subaccounts[i])) {
                IOffchainExchange(
                    IEndpoint(getEndpoint()).getOffchainExchange()
                ).tryCloseIsolatedSubaccount(txn.subaccounts[i]);
            }
        }
    }
