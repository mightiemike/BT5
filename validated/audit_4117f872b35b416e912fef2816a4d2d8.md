### Title
Lack of Slippage Protection in `burnNlp` and `mintNlp` Allows Users to Receive Worse-Than-Expected Exchange Rates — (`core/contracts/Clearinghouse.sol`)

---

### Summary

The `burnNlp` and `mintNlp` functions in `Clearinghouse.sol` execute NLP token conversions at a sequencer-supplied `oraclePriceX18` that is **not committed to by the user's signature**. Because the NLP price fluctuates continuously as traders profit and lose against the pool, a user who signs a `BurnNlp` or `MintNlp` transaction has no on-chain guarantee about the exchange rate that will be applied when the sequencer eventually includes the transaction in a batch. There is no `minQuoteAmountOut` or `minNlpAmountOut` parameter in either transaction struct.

---

### Finding Description

The `BurnNlp` and `MintNlp` transaction structs contain only the user's intent (amount to burn/mint) and a nonce:

```solidity
struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;   // user commits to burning this many NLP
    uint64 nonce;
    // NO minQuoteAmountOut
}

struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount; // user commits to spending this much quote
    uint64 nonce;
    // NO minNlpAmountOut
}
``` [1](#0-0) 

The `oraclePriceX18` used for the actual conversion is carried in the outer `SignedBurnNlp` / `SignedMintNlp` wrapper and is provided by the sequencer at execution time — it is structurally impossible for it to be covered by the user's inner signature (the `signature` field is itself inside the outer struct, so the Verifier must hash only the inner `BurnNlp`/`MintNlp` struct).

In `Clearinghouse.burnNlp`, the quote amount returned to the user is computed entirely from the sequencer-supplied price:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
...
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
``` [2](#0-1) 

Similarly, in `mintNlp`, the NLP amount minted is:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
``` [3](#0-2) 

The NLP price is not static — it changes as the protocol's trading PnL is socialized across NLP holders. Between the moment a user signs a `BurnNlp` transaction and the moment the sequencer includes it in a batch, the NLP price can decrease (due to trader profits) or increase (due to trader losses). The user has no on-chain mechanism to reject execution at an unfavorable price.

---

### Impact Explanation

A user who signs `BurnNlp{nlpAmount: X}` expecting to receive `X * P_expected` quote assets will receive `X * P_actual` where `P_actual < P_expected` if the NLP price has dropped. The difference is a direct, unrecoverable financial loss to the user with no recourse. The same applies in reverse for `mintNlp` — a user spending a fixed `quoteAmount` receives fewer NLP tokens if the price has risen.

---

### Likelihood Explanation

NLP price changes continuously as the protocol's perpetual and spot engines accrue PnL. The sequencer processes transactions in batches with inherent latency. Any user submitting a `BurnNlp` or `MintNlp` transaction during a period of NLP price volatility (e.g., a large trader position being liquidated, a sudden market move) is exposed to this slippage with no protection. This is a realistic, recurring condition — not a rare edge case.

---

### Recommendation

Add a `minQuoteAmountOut` field to `BurnNlp` and a `minNlpAmountOut` field to `MintNlp`. These fields should be part of the inner signed struct so they are covered by the user's EIP-712 signature. In `Clearinghouse.burnNlp` and `mintNlp`, add a `require` check after computing the output amount:

```solidity
// In burnNlp:
require(quoteAmount >= int128(txn.minQuoteAmountOut), ERR_SLIPPAGE);

// In mintNlp:
require(nlpAmount >= int128(txn.minNlpAmountOut), ERR_SLIPPAGE);
``` [4](#0-3) [5](#0-4) 

---

### Proof of Concept

1. NLP oracle price is currently `1.00 USDC` per NLP.
2. Alice signs `BurnNlp{sender: alice, nlpAmount: 1000e18, nonce: 5}` expecting to receive ~`1000 USDC`.
3. Before the sequencer includes Alice's transaction in a batch, a large trader closes a profitable position, causing the NLP price to drop to `0.90 USDC`.
4. The sequencer submits the batch with `SignedBurnNlp{tx: alice_tx, oraclePriceX18: 0.90e18, ...}`.
5. `Clearinghouse.burnNlp` executes: `quoteAmount = 1000e18 * 0.90e18 / 1e18 = 900e18`.
6. Alice receives `~900 USDC` instead of the expected `~1000 USDC` — a `~100 USDC` loss with no on-chain protection. [6](#0-5) [7](#0-6)

```text
    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
