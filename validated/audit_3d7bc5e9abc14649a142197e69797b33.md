### Title
Insufficient Bonus Pool Blocks All Exit Paths, Freezing User Staked Funds - (`x/tieredrewards/keeper/claim_rewards.go`)

### Summary

The `x/tieredrewards` module mandates reward settlement (`claimRewards`) as a prerequisite for every exit path from a delegated position. When the bonus rewards pool holds insufficient funds to cover accrued bonus, `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool` and the entire transaction reverts. Because both `MsgTierUndelegate` and `MsgExitTierWithDelegation` unconditionally call `claimRewards` before performing any delegation transfer, a user with accrued bonus and an empty pool has no path to recover their staked tokens.

### Finding Description

`claimRewards` is called unconditionally before every position-mutating exit operation:

- `TierUndelegate` → `claimRewards` → `undelegate` [1](#0-0) 
- `ExitTierWithDelegation` → `claimRewards` → `transferDelegationFromPosition` [2](#0-1) 

Inside `claimRewards`, `processEventsAndClaimBonus` computes the accrued bonus and then calls `sufficientBonusPoolBalance`. If the pool balance is less than the accrued bonus, it returns a hard error:

```go
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err
}
``` [3](#0-2) 

`sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool` whenever the pool cannot cover the bonus:

```go
if !poolBalance.IsAllGTE(bonus) {
    return errorsmod.Wrapf(types.ErrInsufficientBonusPool, ...)
}
``` [4](#0-3) 

The only remaining exit path, `WithdrawFromTier`, does not call `claimRewards`, but it requires the position to already be undelegated — a state that can only be reached via `TierUndelegate`, which itself requires `claimRewards` to succeed first. [5](#0-4) 

There is no "forfeit bonus and exit anyway" path. The user has no mechanism to bypass the reward settlement requirement.

The bonus pool can be drained by the `BeginBlocker`, which transfers from the rewards pool to the distribution module whenever the fee collector balance falls short of the target base rewards rate: [6](#0-5) 

This means normal chain operation can deplete the pool without any malicious action.

### Impact Explanation

A user who has accrued non-zero bonus rewards and whose validator's pool is empty cannot:
- Call `TierUndelegate` (fails with `ErrInsufficientBonusPool`)
- Call `ExitTierWithDelegation` (fails with `ErrInsufficientBonusPool`)
- Call `WithdrawFromTier` (requires undelegated state, unreachable)

Their staked tokens remain locked inside the tier module's per-position delegator address with no exit until the pool is externally replenished. The corrupted invariant is: **a user's staked tokens are inaccessible despite the exit commitment having elapsed**.

### Likelihood Explanation

The scenario is reachable by any user who:
1. Holds a delegated position with `BonusApy > 0` and time has elapsed (bonus accrued)
2. Attempts to exit when the rewards pool is empty or underfunded

The pool can reach zero through normal BeginBlocker operation (base rewards top-up draining the pool) or simply through lack of replenishment. No privileged action or malicious actor is required. The ADR itself acknowledges this: *"Pool empty (user-driven): Message fails atomically. No state change. Retry after pool replenished."* — but provides no on-chain mechanism for the user to exit without waiting for an external party to fund the pool. [7](#0-6) 

### Recommendation

Decouple the bonus claim from the exit precondition. Specifically:

1. In `TierUndelegate` and `ExitTierWithDelegation`, if `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, allow the operation to proceed while recording the unclaimed bonus as a debt (or simply forfeiting it), rather than reverting the entire transaction.
2. Alternatively, add a `MsgForceExitSkipBonus` message that allows a user to exit their position while forfeiting any unclaimed bonus, callable only when the pool is insufficient.
3. At minimum, ensure `WithdrawFromTier` is reachable without requiring `claimRewards` to succeed — for example, by allowing `TierUndelegate` to proceed with a zero-bonus settlement when the pool is empty.

### Proof of Concept

1. User calls `MsgLockTier` with `BonusApy > 0` tier. Position is created and delegated.
2. Time passes; bonus accrues (`totalBonus > 0` in `processEventsAndClaimBonus`).
3. Rewards pool is drained to zero (via BeginBlocker top-up or lack of funding).
4. User triggers exit: `MsgTriggerExitFromTier` succeeds (no reward claim here). [8](#0-7) 
5. Exit commitment elapses.
6. User calls `MsgTierUndelegate` → `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance` → returns `ErrInsufficientBonusPool` → entire tx reverts. [9](#0-8) 
7. User calls `MsgExitTierWithDelegation` → same failure path. [2](#0-1) 
8. User calls `MsgWithdrawFromTier` → fails validation because position is still delegated (undelegation never completed). [10](#0-9) 
9. User's staked tokens are frozen until an external party funds the pool.

func (ms msgServer) TriggerExitFromTier(ctx context.Context, msg *types.MsgTriggerExitFromTier) (*types.MsgTriggerExitFromTierResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	pos, err := ms.getPositionState(ctx, msg.PositionId)
	if err != nil {
		return nil, err
	}

	if err := ms.validateTriggerExit(pos.Position, msg.Owner); err != nil {
		return nil, err
	}

	tier, err := ms.getTier(ctx, pos.TierId)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	pos.TriggerExit(sdkCtx.BlockTime(), tier.ExitDuration)

	if err := ms.setPosition(ctx, pos.Position, nil); err != nil {
		return nil, err
	}

	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventExitTriggered{
		PositionId:   pos.Id,
		TierId:       pos.TierId,
		Owner:        pos.Owner,
		ExitUnlockAt: pos.ExitUnlockAt,
	}); err != nil {
		return nil, err
	}

	return &types.MsgTriggerExitFromTierResponse{
		ExitUnlockAt: pos.ExitUnlockAt,
		PositionId:   pos.Id,
	}, nil
}
