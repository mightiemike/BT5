### Title
Pending `depositCollateralWithReferral` Tokens Transferred to Clearinghouse Are Not Reflected in SpotEngine Balance, Causing Permanent Loss on Migration or Silent Slow-Mode Failure — (`core/contracts/Endpoint.sol`, `core/contracts/EndpointStorage.sol`)

---

### Summary

`depositCollateralWithReferral` in `Endpoint.sol` immediately transfers user tokens to the `Clearinghouse` contract but defers the corresponding `SpotEngine` balance credit to a slow-mode transaction that executes up to 3 days later. During this window, the tokens are physically held by `Clearinghouse` but are not reflected in any user balance. If the slow-mode transaction fails silently — or if the protocol is migrated/upgraded while slow-mode deposit transactions are pending — the deposited tokens are permanently unrecoverable. The refund path for failed slow-mode transactions was explicitly removed, as confirmed by the inline comment `// try return funds now removed`.

---

### Finding Description

`depositCollateralWithReferral` performs a two-phase deposit:

**Phase 1 — immediate token transfer:** [1](#0-0) 

`handleDepositTransfer` pulls tokens from `msg.sender` into `Endpoint`, then forwards them to `Clearinghouse`: [2](#0-1) 

**Phase 2 — deferred balance credit (up to 3 days later):** [3](#0-2) 

The actual `spotEngine.updateBalance` call that credits the user's internal balance only happens when the queued `SlowModeTx` is eventually executed: [4](#0-3) 

Between Phase 1 and Phase 2, the tokens are in `Clearinghouse` but the user's `SpotEngine` balance is zero. This is the exact desynchronization class described in the external report.

**The refund path was removed.** In `_executeSlowModeTransaction`, if the slow-mode call reverts for any non-OOG reason, execution silently continues with no refund: [5](#0-4) 

The comment `// try return funds now removed` at line 226 explicitly confirms that the recovery mechanism was deleted. There is no fallback path to return tokens to a user whose `DepositCollateral` slow-mode tx fails.

---

### Impact Explanation

Any user who calls `depositCollateralWithReferral` and whose corresponding slow-mode `DepositCollateral` transaction has not yet been executed faces permanent token loss if:

1. **Protocol migration or upgrade:** If the `Clearinghouse` or `SpotEngine` is upgraded while slow-mode deposit transactions are pending, the new implementation may not process the queued transactions, or may process them in a way that reverts. The tokens are already in `Clearinghouse` but the user's `SpotEngine` balance was never credited. This is the direct analog to the Beanstalk migration bug: tokens transferred to the protocol are not part of any tracked internal balance and will not be returned.

2. **Silent slow-mode failure:** If `clearinghouse.depositCollateral` reverts for any reason during slow-mode execution (e.g., a product's token address becomes `address(0)` due to reconfiguration, causing `_decimals` to revert with `ERR_INVALID_PRODUCT`), the slow-mode tx is silently dropped. The tokens remain in `Clearinghouse` with no balance credit and no refund. [6](#0-5) 

The corrupted state delta is: `Clearinghouse` ERC-20 balance increases by `amount`, but `SpotEngine.balances[productId][subaccount].amountNormalized` is never updated. The user's collateral is permanently stranded.

---

### Likelihood Explanation

The 3-day `SLOW_MODE_TX_DELAY` creates a persistent window during which any number of users can have tokens in `Clearinghouse` with no corresponding `SpotEngine` balance. The protocol uses upgradeable proxies (`ProxyManager`, `BaseProxyManager`) and the `Clearinghouse.depositCollateral` function is marked `virtual`, meaning upgrades are an expected operational event. Any upgrade or migration that does not first drain all pending slow-mode deposit transactions will silently strand user funds. This is not a theoretical edge case — it is a structural property of the two-phase deposit design combined with the removed refund mechanism.

---

### Recommendation

1. **Process all pending `DepositCollateral` slow-mode transactions before any migration or upgrade** of `Clearinghouse` or `SpotEngine`.
2. **Restore a refund path** for failed `DepositCollateral` slow-mode transactions. If `clearinghouse.depositCollateral` reverts, the tokens should be returned to the original depositor rather than silently abandoned.
3. **Alternatively**, credit the `SpotEngine` balance atomically within `depositCollateralWithReferral` at deposit time (Phase 1), eliminating the desynchronization window entirely, consistent with how the sequencer-path `DepositCollateral` transaction works.

---

### Proof of Concept

1. User calls `Endpoint.depositCollateralWithReferral(subaccount, productId, amount, referral)`.
2. `handleDepositTransfer` executes: `amount` tokens are pulled from user and forwarded to `Clearinghouse`. [7](#0-6) 
3. A `SlowModeTx` is queued with `executableAt = block.timestamp + 3 days`. [8](#0-7) 
4. `SpotEngine.balances[productId][subaccount]` remains at its pre-deposit value — the user has no credited balance.
5. The protocol is upgraded (or `clearinghouse.depositCollateral` reverts for any reason during slow-mode execution).
6. The slow-mode tx is silently dropped. No refund is issued. [9](#0-8) 
7. User's `amount` tokens are permanently stranded in `Clearinghouse` with no corresponding `SpotEngine` balance entry and no recovery path.

    function depositCollateral(IEndpoint.DepositCollateral calldata txn)
        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        uint8 decimals = _decimals(txn.productId);

        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
        int128 amountRealized = int128(txn.amount) * int128(multiplier);

        spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
        emit ModifyCollateral(amountRealized, txn.sender, txn.productId);
    }
