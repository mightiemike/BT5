### Title
`DirectDepositV1.creditDeposit()` Reverts Entirely if Any Token in the Product List Is Paused, Blocking All Pending Deposits — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` iterates over all spot product IDs and makes unguarded external calls (`token.approve()` + `endpoint.depositCollateralWithReferral()`) for each. If any single token in the product list is paused (e.g., USDC), the entire call reverts, permanently blocking all other tokens sitting in the DDA from being credited to the subaccount until the paused token is unpaused.

---

### Finding Description

`creditDeposit()` fetches all product IDs from the spot engine and loops over them: [1](#0-0) 

For each product with a non-zero balance, it calls `token.approve(address(endpoint), balance)` followed by `endpoint.depositCollateralWithReferral(...)`. Neither call is wrapped in a `try/catch` or guarded by a skip condition. If `token.approve()` reverts — which USDC does when its contract is paused — the entire `creditDeposit()` transaction reverts, and no token in the loop gets credited.

The developers were explicitly aware of this exact pattern. In `Endpoint.depositCollateralWithReferral()`, a comment reads: [2](#0-1) 

> *"we cannot revert here, otherwise direct deposit could be blocked when there are multiple assets awaiting credit but one of them is below the minimum deposit amount. we can just skip the deposit and continue with the next asset."*

That guard only covers the `isValidDepositAmount` check. It does **not** protect against a revert inside `token.approve()` or inside `handleDepositTransfer` (which calls `token.safeTransferFrom`). A paused token bypasses the `isValidDepositAmount` guard entirely and reverts at the ERC-20 call level, propagating up through the loop.

---

### Impact Explanation

All tokens sitting in the DDA contract — including tokens whose own contracts are fully operational — cannot be credited to the subaccount. They are locked in the DDA until the paused token is unpaused. Users who sent multiple assets to the DDA expecting automatic crediting suffer a complete deposit freeze across all assets, not just the paused one.

---

### Likelihood Explanation

USDC (Circle's stablecoin, a primary collateral asset in DeFi protocols) implements a `pause` mechanism that causes `approve()` and `transfer()` to revert when activated. Circle has historically paused USDC in response to security incidents. The likelihood of a pause is low but non-zero, and the impact when it occurs is high — identical in structure to the Ion Protocol finding rated Medium.

---

### Recommendation

Wrap each iteration's external calls in a `try/catch` block so that a revert on one token does not abort the entire loop. This is consistent with the existing intent expressed in `depositCollateralWithReferral`:

```solidity
function creditDeposit() external {
    uint32[] memory productIds = spotEngine.getProductIds();
    for (uint256 i = 0; i < productIds.length; i++) {
        uint32 productId = productIds[i];
        address tokenAddr = spotEngine.getToken(productId);
        if (tokenAddr == address(0)) continue;
        IIERC20Base token = IIERC20Base(tokenAddr);
        uint256 balance = token.balanceOf(address(this));
        if (balance != 0) {
            try token.approve(address(endpoint), balance) {
                try endpoint.depositCollateralWithReferral(
                    subaccount, productId, uint128(balance), "-1"
                ) {} catch {}
            } catch {}
        }
    }
}
```

---

### Proof of Concept

1. A DDA contract holds 500 USDC and 1 WETH (both are registered spot products).
2. Circle pauses USDC (e.g., in response to a bridge exploit).
3. A user (or keeper) calls `creditDeposit()`.
4. The loop reaches the USDC product; `token.balanceOf()` returns 500 (does not revert when paused).
5. `token.approve(address(endpoint), 500)` **reverts** — USDC's `approve` enforces the `whenNotPaused` modifier.
6. The entire `creditDeposit()` transaction reverts.
7. Neither USDC nor WETH is credited to the subaccount.
8. Both assets remain stranded in the DDA until USDC is unpaused.
