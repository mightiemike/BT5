### Title
No User-Controlled Maximum Withdrawal Fee in `WithdrawCollateral` (V1) Allows Admin Fee Increase to Silently Charge More Than Expected — (`EndpointTx.sol`)

---

### Summary

The `WithdrawCollateral` (V1) transaction path in `EndpointTx.sol` reads the withdrawal fee at execution time from the admin-controlled `spotEngine.getConfig(productId).withdrawFeeX18`. The fee is not committed to in the user's EIP-712 signature. If the admin increases `withdrawFeeX18` between when the user signs and when the sequencer processes the transaction, the user is silently charged a higher fee than anticipated with no on-chain protection mechanism.

---

### Finding Description

In `EndpointTx.sol`, the `WithdrawCollateral` (V1) handler charges the fee by reading the live config at execution time:

```solidity
chargeFee(
    signedTx.tx.sender,
    spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18,  // live admin value
    signedTx.tx.productId
);
``` [1](#0-0) 

The `WithdrawCollateral` struct that the user signs contains only `sender`, `productId`, `amount`, and `nonce` — no fee field:

```solidity
struct WithdrawCollateral {
    bytes32 sender;
    uint32 productId;
    uint128 amount;
    uint64 nonce;
}
``` [2](#0-1) 

The EIP-712 digest computed in `Verifier.sol` for this transaction type covers only those four fields — no fee is committed to cryptographically:

```solidity
digest = keccak256(
    abi.encode(
        keccak256(bytes(WITHDRAW_COLLATERAL_SIGNATURE)),
        signedTx.tx.sender,
        signedTx.tx.productId,
        signedTx.tx.amount,
        signedTx.tx.nonce
    )
);
``` [3](#0-2) 

The admin can update `withdrawFeeX18` at any time via `SpotEngine.addOrUpdateProduct()`, which is `onlyOwner` and applies the new config immediately with no bounds check and no timelock:

```solidity
function addOrUpdateProduct(..., Config calldata config, ...) public onlyOwner {
    ...
    configs[productId] = config;
    ...
}
``` [4](#0-3) 

The initial `withdrawFeeX18` for the quote product is set to `ONE` (1e18), confirming the field is a meaningful, non-trivial value: [5](#0-4) 

---

### Impact Explanation

A user signs a `WithdrawCollateral` (V1) transaction expecting to pay the fee visible at signing time. If the admin increases `withdrawFeeX18` before the sequencer processes the transaction, the user's subaccount is debited the new higher fee. The user receives less collateral than they anticipated, with no on-chain mechanism to reject or cap the fee. The asset delta is the difference between the fee at signing time and the fee at execution time, taken directly from the user's collateral balance.

---

### Likelihood Explanation

Medium. The admin does not need to act maliciously — a routine fee adjustment (e.g., to reflect gas cost changes or protocol economics) is sufficient to trigger this. The sequencer-based model means there is always a latency window between user signing and on-chain execution, during which a fee update can land. The `WithdrawCollateral` (V1) path remains active alongside V2 in the `TransactionType` enum. [6](#0-5) 

---

### Recommendation

1. **Short-term**: Add a `maxFeeX18` field to the `WithdrawCollateral` struct and include it in the EIP-712 digest. In the handler, revert if `spotEngine.getConfig(productId).withdrawFeeX18 > signedTx.tx.maxFeeX18`. This mirrors the partial protection already present in the V2 path (though note that V2's `feeX18` is itself not included in the V2 digest — a separate issue).
2. **Long-term**: Implement a timelock or notice period for `withdrawFeeX18` changes in `SpotEngine.addOrUpdateProduct()`, so users have time to cancel pending transactions before a fee increase takes effect.

---

### Proof of Concept

1. `withdrawFeeX18` for product `1` is currently `1e15` (0.1% of 1e18 scale).
2. User signs a `WithdrawCollateral` transaction: `productId=1`, `amount=1000e6 USDC`, `nonce=42`. They expect to pay ~0.1 USDC in fees.
3. Admin calls `SpotEngine.addOrUpdateProduct(1, ..., Config{..., withdrawFeeX18: 5e16, ...}, ...)` — raising the fee to 5%.
4. Sequencer submits the user's signed transaction to `Endpoint`.
5. `EndpointTx` reads `spotEngine.getConfig(1).withdrawFeeX18` → `5e16` and calls `chargeFee(sender, 5e16, 1)`.
6. User receives `950 USDC` instead of `999 USDC`. The 49 USDC difference is taken without the user's consent and without any on-chain revert path. [7](#0-6)
