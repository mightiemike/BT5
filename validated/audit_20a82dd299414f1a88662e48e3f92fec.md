Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual Swapper, Allowing Any User to Bypass Per-User Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is the pool's `msg.sender` ‚Äî the router contract address when a swap is routed through `MetricOmmSimpleRouter`. The extension evaluates `allowedSwapper[pool][router_address]` instead of `allowedSwapper[pool][actual_user]`. Any unprivileged user can bypass a pool's per-user allowlist by routing through the standard periphery router, completely defeating the pool admin's curation policy.

## Finding Description

The full call chain confirms the wrong-actor binding:

1. User calls `MetricOmmSimpleRouter.exactInputSingle(params)` ‚Äî `msg.sender` = user.
2. Router calls `IMetricOmmPoolActions(params.pool).swap(recipient, zeroForOne, amount, priceLimit, "", params.extensionData)` ‚Äî pool's `msg.sender` = **router address** (`MetricOmmSimpleRouter.sol` L72‚Äì80).
3. Pool's `swap()` calls `_beforeSwap(msg.sender, ...)` passing the **router address** as `sender` (`MetricOmmPool.sol` L230‚Äì231).
4. `ExtensionCalling._beforeSwap` encodes and forwards `sender` (router) to the extension (`ExtensionCalling.sol` L160‚Äì176).
5. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]` = `allowedSwapper[pool][router_address]` (`SwapAllowlistExtension.sol` L37).

The actual end-user address is never consulted. The pool admin has no on-chain mechanism to distinguish "router called by allowlisted user" from "router called by anyone."

This creates an irreconcilable dilemma:
- **If the router IS allowlisted** (required for the pool to be usable via the standard interface): every user who routes through the router passes the check, regardless of individual allowlist status.
- **If the router is NOT allowlisted**: even individually allowlisted users cannot swap through the standard periphery path ‚Äî the extension reverts on every router-mediated call.

For multihop paths, `exactInput` passes `address(this)` (the router itself) as the payer context for hops after the first (`MetricOmmSimpleRouter.sol` L103), so the same router address appears as `sender` on every hop, compounding the issue.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of counterparties (e.g., KYC'd addresses, whitelisted market makers, or protocol-controlled addresses) is fully bypassable by any unprivileged user who calls the pool through `MetricOmmSimpleRouter`. The attacker pays no extra cost beyond normal swap fees. This constitutes an admin-boundary break: an unprivileged path (the standard public router) bypasses the pool admin's allowlist restriction, violating the pool's curation policy and potentially causing direct financial loss (LP value leakage to arbitrageurs who should have been excluded, or regulatory/compliance breach on restricted pools).

## Likelihood Explanation

- `MetricOmmSimpleRouter` is the standard, documented periphery entry point for all swaps. Any user who reads the protocol docs will use it.
- No special privileges, flash loans, or unusual conditions are required ‚Äî a normal `exactInputSingle` call suffices.
- The bypass is reachable on every router-mediated swap to any pool using `SwapAllowlistExtension`.
- The only mitigation available to the admin (not allowlisting the router) makes the pool unusable via the standard interface, which is not a realistic operational choice.

## Recommendation

The `sender` argument passed to `beforeSwap` must represent the economic initiator, not the intermediary. Two complementary fixes:

1. **In the router:** pass the actual user address as an explicit field in `extensionData`, and have the extension decode it when `sender` is a known router address.
2. **Architectural fix:** the pool's `_beforeSwap` should pass both `msg.sender` (the immediate caller) and an optional `initiator` field that the router populates with the real user, so extensions can choose which identity to gate on.
3. **In the extension (self-contained):** require pools using this extension to be called directly (not via router), or gate on `tx.origin` as a last resort with documented caveats.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension.
  - Pool admin calls setAllowedToSwap(pool, alice, true)  // alice is allowlisted
  - Pool admin calls setAllowedToSwap(pool, router, true) // REQUIRED for router to work

Attack:
  - Bob (not allowlisted) calls router.exactInputSingle({pool: pool, ...})
  - Router calls pool.swap(...) ‚Äî pool's msg.sender = router
  - Pool calls _beforeSwap(msg.sender=router, ...) ‚Üí extension.beforeSwap(sender=router, ...)
  - Extension checks allowedSwapper[pool][router] == true ‚Üí PASSES
  - Bob's swap executes despite not being individually allowlisted.

Alternatively (if admin does NOT allowlist the router):
  - Alice (allowlisted) calls router.exactInputSingle({pool: pool, ...})
  - Extension checks allowedSwapper[pool][router] == false ‚Üí REVERTS
  - Alice cannot use the standard periphery path at all.
```
