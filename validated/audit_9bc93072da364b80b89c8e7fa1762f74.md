### Title
Gateway Admits Unsigned Invoke Transactions for Undeployed Accounts via Unchecked `skip_stateful_validations` — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the Apollo gateway bypasses the `__validate__` entry-point call (the only on-chain signature check) for any invoke transaction with `nonce == 1` submitted against an account whose on-chain nonce is `0`, provided that *any* transaction for that sender address already exists in the mempool. Because the check is keyed only on address presence—not on the submitter's identity or signature—an unprivileged attacker can inject a signature-less invoke transaction for a victim's undeployed account, causing the gateway to admit it without cryptographic verification.

### Finding Description

`skip_stateful_validations` (lines 429–461) returns `true` when three conditions hold simultaneously:

1. The incoming transaction is an `Invoke` with `nonce == Nonce(Felt::ONE)`.
2. The on-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed).
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` constructs `ExecutionFlags { validate: false, … }` and passes it to the blockifier's `StatefulValidator::perform_validations`. [2](#0-1) 

Inside `StatefulValidator::perform_validations`, when `execution_flags.validate == false` the `__validate__` call is skipped entirely and the function returns `Ok(())`: [3](#0-2) 

`AccountTransaction::validate_tx` also short-circuits immediately when `validate == false`: [4](#0-3) 

The `account_tx_in_pool_or_recent_block` predicate checks only whether *any* transaction for the address is known to the mempool state or pool—it does not restrict to `DeployAccount` transactions and does not verify the caller's identity: [5](#0-4) 

The `validate_by_mempool` call that precedes `skip_stateful_validations` only validates nonce ordering, fee escalation, and duplicate hashes—it does not verify the cryptographic signature: [6](#0-5) 

### Impact Explanation

An attacker who observes that victim address `V` has any pending transaction in the mempool (e.g., a `DeployAccount`) and whose on-chain nonce is `0` can:

1. Craft an `InvokeV3` transaction for `V` with `nonce = 1` and an arbitrary (invalid) signature.
2. Submit it to the gateway. The gateway skips `__validate__`, so no signature check occurs.
3. The transaction is admitted to the mempool.
4. If the attacker pays a higher fee, the mempool's fee-escalation logic replaces the victim's legitimate `nonce = 1` invoke with the attacker's invalid one.
5. During block execution the attacker's transaction fails (signature invalid), consuming the sequencing slot and evicting the victim's transaction.

This matches the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The preconditions are observable from the public mempool: any address with a pending `DeployAccount` (or any future-nonce invoke that passed the nonce-gap check) and on-chain nonce `0` is a valid target. No privileged access is required. The attacker only needs to submit a well-formed `InvokeV3` with `nonce = 1` and a garbage signature.

### Recommendation

1. **Restrict the skip condition to `DeployAccount` transactions only.** The `account_tx_in_pool_or_recent_block` query should be replaced with a dedicated `deploy_account_in_pool(sender_address)` query that returns `true` only when a `DeployAccount` transaction for that address is present in the mempool.

2. **Verify the signature against the class hash declared in the pending `DeployAccount`.** When a `DeployAccount` is in the mempool, its `class_hash` is known. The gateway can instantiate a temporary state with the declared class and run `__validate__` against it, eliminating the need to skip signature verification entirely.

3. **Bind the skip to the submitter's identity.** At minimum, require that the invoke transaction's sender address matches the address of the pending `DeployAccount` and that the `DeployAccount` was submitted in the same gateway session or carries a verifiable linkage.

#   validate_nonce(nonce=1, account_nonce=0) → OK (within max_allowed_nonce_gap)

#           victim's legitimate invoke is gone from the mempool.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
