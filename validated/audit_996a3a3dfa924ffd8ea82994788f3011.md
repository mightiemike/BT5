### Title
Linked Signer Can Self-Escalate by Authorizing Its Own Replacement ŌĆö (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The sequencer-path handler for `LinkSigner` transactions passes `allowLinkedSigner = true` to `validateSignedTx`. This means the **current linked signer** ŌĆö a delegated hot-wallet key ŌĆö can sign a valid `LinkSigner` transaction that replaces itself with any attacker-controlled address. Once the replacement is installed, the attacker's address becomes the new linked signer and can authorize withdrawals, transfers, and other high-impact operations on the subaccount without the subaccount owner's involvement.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch decodes a `SignedLinkSigner` and calls:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true          // allowLinkedSigner = true
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` resolves the permitted signer as:

```solidity
allowLinkedSigner ? getLinkedSigner(sender) : address(0)
``` [2](#0-1) 

`Verifier.validateSignature` then accepts the signature if it recovers to **either** the subaccount owner address **or** the current linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

There is no restriction on what `signedTx.tx.signer` (the new linked signer) may be set to. The linked signer can therefore sign a `LinkSigner` transaction naming any attacker-controlled address as the new signer, and the contract will accept it and overwrite `linkedSigners[subaccount]`.

**Contrast with the slow-mode path**, which correctly restricts `LinkSigner` to the subaccount owner via `validateSender` (which checks `msg.sender == address(uint160(bytes20(txn.sender)))`):

```solidity
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

The sequencer path applies no equivalent owner-only restriction.

---

### Impact Explanation

Once the attacker installs their own address as the new linked signer, they can sign any transaction that accepts `allowLinkedSigner = true`. This includes:

- **`WithdrawCollateral`** (sequencer path, line 418ŌĆō436): drains the subaccount's collateral to any address.
- **`WithdrawCollateralV2`** (line 437ŌĆō465): when `sendTo == address(0)`, also allows linked signer.
- **`TransferQuote`** (line 593ŌĆō614): moves quote balance to an arbitrary recipient subaccount. [5](#0-4) 

The result is **complete subaccount takeover and full collateral theft** without any further action from the original owner.

---

### Likelihood Explanation

Linked signers are explicitly designed as hot-wallet keys for trading automation (API keys, bots). They are materially more likely to be exposed than the subaccount owner's cold key. A single leaked linked-signer private key ŌĆö from a compromised server, misconfigured environment variable, or phishing ŌĆö is sufficient to trigger this escalation. The attacker needs only to craft and submit one `LinkSigner` transaction through the sequencer.

---

### Recommendation

Change the `LinkSigner` branch in `processTransactionImpl` to pass `allowLinkedSigner = false`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This mirrors the invariant already enforced in the slow-mode path and ensures that only the subaccount owner (the address embedded in the `bytes32` sender) can install or replace a linked signer.

---

### Proof of Concept

1. Victim subaccount `S` has owner address `O` and linked signer `L` (a hot-wallet key).
2. Attacker compromises `L` (e.g., leaked API key).
3. Attacker uses `L` to sign a `SignedLinkSigner` transaction:
   - `tx.sender = S`
   - `tx.signer = bytes32(uint256(uint160(attacker_address)))`
   - `tx.nonce = current nonce for O`
   - `signature = sign(digest, L_private_key)`
4. Attacker submits this transaction to the sequencer.
5. `processTransactionImpl` reaches the `LinkSigner` branch; `validateSignedTx(..., true)` passes because `L` is the current linked signer.
6. `linkedSigners[S]` is overwritten with `attacker_address`.
7. Attacker now signs a `SignedWithdrawCollateral` transaction with `attacker_address`'s key, draining all collateral from `S`. [6](#0-5) [5](#0-4)
