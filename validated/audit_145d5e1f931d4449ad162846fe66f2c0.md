Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of End User, Enabling Full Allowlist Bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps on the `sender` argument, which is `msg.sender` inside `MetricOmmPool.swap()` — the direct pool caller, not the end user. When swaps are routed through `MetricOmmSimpleRouter`, `sender` resolves to the router contract address. A pool admin who allowlists the router (the only way to let permitted users swap via the router) inadvertently opens the pool to every unpermitted user who calls the router, fully defeating the allowlist.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // direct caller of pool.swap()
    recipient,
    ...
);
``` [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks that identity against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [2](#0-1) 

Here `msg.sender` is the pool (the extension's caller) and `sender` is the address the pool received as `msg.sender` — i.e., the router when the user goes through `MetricOmmSimpleRouter`.

`MetricOmmSimpleRouter.exactInputSingle` calls the pool directly without forwarding the real user's identity:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
``` [3](#0-2) 

The pool receives `msg.sender = router`. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. The real user's address is stored only in transient callback context (used for payment), never forwarded to the pool or extension.

The same flaw applies to `exactOutputSingle` (L135-137) and the multi-hop `exactInput` (L103-112) paths. [4](#0-3) 

`DepositAllowlistExtension` does not share this flaw: it checks the `owner` argument, which is a caller-supplied parameter that correctly identifies the actual depositor, not the intermediary. [5](#0-4) 

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter` once the router is allowlisted. Unauthorized users can execute swaps against the pool at oracle-derived prices, exposing LP assets to uninvited counterparties. This breaks the core access-control invariant the pool admin configured and constitutes an admin-boundary break with direct LP-asset exposure — matching the "Admin-boundary break" allowed impact category.

## Likelihood Explanation

The trigger is realistic: a pool admin who wants their allowlisted users to be able to use the standard router must allowlist the router address, because the router always appears as `msg.sender` to the pool and there is no other mechanism to pass the real user's identity to the extension. Once the router is allowlisted, the bypass is available to any unpermitted address with no further preconditions — a single `router.exactInputSingle(...)` call suffices.

## Recommendation

Gate the actual end user, not the intermediary. Two complementary fixes:

1. **In `SwapAllowlistExtension`**: Ignore `sender` (the direct pool caller) and instead require the actual user identity to be passed in `extensionData`. The router already forwards `extensionData` unchanged, so the user can encode their address there, and the extension can verify it (e.g., via a signature or a trusted forwarder pattern).

2. **In `MetricOmmSimpleRouter`**: Encode `msg.sender` (the real user) into the `extensionData` forwarded to the pool so that allowlist extensions can read it without requiring the user to supply it manually.

Alternatively, document clearly that `SwapAllowlistExtension` gates the **direct pool caller**, not the end user, and that allowlisting the router opens the pool to all router users.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice]  = true   (alice is the only permitted trader)
  allowedSwapper[pool][router] = true   (admin adds this so alice can use the router)

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, ..., recipient: bob, ...})

  pool.swap() receives msg.sender = router
  _beforeSwap(router, ...)
  extension checks: allowedSwapper[pool][router] == true  ✓
  swap executes — bob receives output tokens

Result:
  bob, who is not on the allowlist, successfully swaps against the pool.
  The allowlist guard is fully bypassed.
```

A Foundry integration test can confirm this by deploying a pool with `SwapAllowlistExtension`, allowlisting only `alice` and the router, then calling `router.exactInputSingle` from `bob`'s address and asserting the swap succeeds without revert.
