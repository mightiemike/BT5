### Title
Checkpoint Advanced Before Pool Check in `slashRedelegationPosition` Causes Permanent Bonus Reward Loss — (`x/tieredrewards/keeper/claim_rewards.go`, `x/tieredrewards/keeper/slash.go`)

---

### Summary

In `processEventsAndClaimBonus`, `applyBonusAccrualCheckpoint` advances `pos.LastBonusAccrual` to `blockTime` **before** `sufficientBonusPoolBalance` is checked. When `slashRedelegationPosition` calls this function and the pool is empty, `ErrInsufficientBonusPool` is deliberately swallowed, but the already-mutated `pos` (with advanced `LastBonusAccrual` and `LastEventSeq`, and already-decremented event ref counts in the store) is then persisted by `setPositionWithState`. The bonus for the processed segment is permanently unrecoverable.

---

### Finding Description

**Step 1 — Entry point.**
`BeforeRedelegationSlashed` (a staking hook) calls `slashRedelegationPosition`: [1](#0-0) 

**Step 2 — Error swallowing.**
`slashRedelegationPosition` calls `processEventsAndClaimBonus(ctx, &pos)` (pointer receiver, so all mutations are visible to the caller), then explicitly swallows `ErrInsufficientBonusPool`: [2](#0-1) 

**Step 3 — Checkpoint advanced before pool check.**
Inside `processEventsAndClaimBonus`, the ordering is:

1. **Line 193** — `pos.UpdateLastEventSeq(entry.Seq)` advances `LastEventSeq` in-memory.
2. **Lines 196–198** — `decrementEventRefCount` is written to the **persistent store** (not rolled back on error).
3. **Line 215** — `applyBonusAccrualCheckpoint` advances `pos.LastBonusAccrual` to `blockTime` in-memory.
4. **Line 217** — `pos.UpdateLastKnownBonded` updates bonded state in-memory.
5. **Line 230** — `sufficientBonusPoolBalance` is checked — **only here** does the function discover the pool is empty and return `ErrInsufficientBonusPool`. [3](#0-2) 

**Step 4 — Mutated state persisted.**
After the error is swallowed, `slashRedelegationPosition` calls `setPositionWithState` with the already-mutated `pos`: [4](#0-3) 

This persists:
- `LastBonusAccrual` = `blockTime` (advanced past the unpaid segment)
- `LastEventSeq` = last processed event (advanced past processed events)
- `LastKnownBonded` = updated

The event ref counts were already decremented in the store (step 3.2), so those events may be garbage-collected. Any subsequent call to `processEventsAndClaimBonus` for this position will start from the advanced `LastBonusAccrual` and `LastEventSeq`, skipping the entire unpaid segment permanently. [5](#0-4) 

---

### Impact Explanation

The position owner permanently loses all bonus rewards accrued up to `blockTime` at the moment of the redelegation slash. The invariant that `LastBonusAccrual` must not advance past the last successfully paid bonus segment is violated. Refilling the pool afterward does not help — the checkpoint and event sequence have already moved forward, and the accrued-but-unpaid segment is gone.

---

### Likelihood Explanation

The `RewardsPoolName` module account can reach zero balance through normal operation (e.g., high claim volume, delayed pool replenishment, or a governance/admin delay in funding). A redelegation slash is a standard Cosmos SDK staking event that fires whenever a validator with an active redelegation entry is slashed. No special privileges are required; the combination of an empty pool and a redelegation slash is a realistic production scenario.

---

### Recommendation

Move `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` to **after** the `sufficientBonusPoolBalance` check and the `SendCoinsFromModuleToAccount` call succeeds. Similarly, `decrementEventRefCount` should only be committed after the payment succeeds, or the function should be restructured to compute the bonus amount first, verify pool sufficiency, pay, and only then advance all checkpoints and decrement ref counts.

---

func applyBonusAccrualCheckpoint(pos *types.Position, blockTime time.Time) {
	accrualEnd := blockTime
	if pos.CompletedExitLockDuration(blockTime) {
		accrualEnd = pos.ExitUnlockAt
	}
	pos.UpdateLastBonusAccrual(accrualEnd)
}
