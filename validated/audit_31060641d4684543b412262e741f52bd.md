### Title
Deposited Collateral Not Immediately Credited Allows Unfair Liquidation — (File: `core/contracts/Endpoint.sol`)

---

### Summary

When a user calls `depositCollateral()` or `depositCollateralWithReferral()` in `Endpoint.sol`, the ERC20 transfer is executed immediately, but the balance credit is only queued as a slow mode transaction with a hardcoded 3-day delay (`SLOW_MODE_TX_DELAY`). The off-chain sequencer controls when `ExecuteSlowMode` is processed relative to `UpdatePrice` and `LiquidateSubaccount` transactions. This creates a window in which a user has already paid collateral into the contract but their on-chain balance has not been updated, leaving them exposed to liquidation despite having taken protective action.

---

### Finding Description

In `Endpoint.depositCollateralWithReferral()`, the ERC20 transfer is performed immediately:

```solidity
handleDepositTransfer(
    IERC20Base(spotEngine.getToken(productId)),
    msg.sender,
    uint256(amount)
);
```

But the balance credit is only enqueued as a slow mode transaction:

```solidity
slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
    ...
    tx: abi.encodePacked(uint8(TransactionType.DepositCollateral), ...)
});
``` [1](#0-0) 

`SLOW_MODE_TX_DELAY` is defined as 3 days: [2](#0-1) 

The slow mode transaction is only applied to the user's balance when the sequencer submits an `ExecuteSlowMode` transaction (via `processTransaction` → `_executeSlowModeTransaction` with `fromSequencer = true`, which bypasses the 3-day wall-clock check): [3](#0-2) 

The sequencer submits ordered transaction batches via `submitTransactionsChecked()`. There is no on-chain enforcement that pending `DepositCollateral` slow mode transactions must be processed before `UpdatePrice` or `LiquidateSubaccount` transactions in the same batch: [4](#0-3) 

Liquidation health checks read the current on-chain balance state via `isUnderMaintenance()`: [5](#0-4) 

Because the deposit has not yet been credited, the health check does not account for the user's deposited funds, and `liquidateSubaccountImpl` proceeds: [6](#0-5) 

---

### Impact Explanation

A user who calls `depositCollateral()` to protect a position from liquidation has their funds immediately taken from their wallet and held in the contract, but their subaccount balance is not updated. If the sequencer processes a `LiquidateSubaccount` transaction before the corresponding `ExecuteSlowMode` for that deposit, the user is liquidated at a penalty (liquidation fee + unfavorable liquidation price) despite having already committed collateral. The user suffers real asset loss — liquidation penalties and position closure at adverse prices — that would not have occurred had the deposit been credited immediately.

---

### Likelihood Explanation

This is reachable in two concrete scenarios:

1. **Sequencer downtime / recovery**: During sequencer downtime, users deposit on-chain to protect positions. When the sequencer recovers, it processes accumulated price updates and liquidations before flushing the slow mode deposit queue. This is the direct analog of the reference report.

2. **Normal operation ordering**: Even without downtime, the sequencer controls the ordering of transactions within each `submitTransactionsChecked` batch. A batch containing `[UpdatePrice, LiquidateSubaccount]` submitted before a batch containing `[ExecuteSlowMode]` produces the same outcome. There is no protocol-level guarantee of ordering.

---

### Recommendation

Credit the deposit balance immediately within `depositCollateralWithReferral()` rather than queuing it as a slow mode transaction. If the slow mode queue is required for sequencer bookkeeping, the balance update should be applied atomically at deposit time and the slow mode entry used only for sequencer state synchronization, not as the gating mechanism for balance credit.

---

### Proof of Concept

1. User holds a position near maintenance margin threshold.
2. Price moves adversely; user calls `Endpoint.depositCollateral()` to add collateral. ERC20 tokens are transferred to the contract. A slow mode tx is queued with `executableAt = block.timestamp + 3 days`.
3. Sequencer (online or recovering from downtime) submits a batch: `[UpdatePrice(newLowerPrice), LiquidateSubaccount(user)]`. No `ExecuteSlowMode` is included.
4. `processTransactionImpl` processes `UpdatePrice`, updating `priceX18[productId]` to the new lower price.
5. `processTransactionImpl` processes `LiquidateSubaccount`. `isUnderMaintenance(user)` returns `true` because the deposited collateral has not been credited. Liquidation proceeds, charging the user a liquidation fee and closing their position at a penalty.
6. The slow mode `DepositCollateral` transaction eventually executes (either via sequencer or after 3 days via `executeSlowModeTransaction()`), crediting the deposit to the now-liquidated account — too late to prevent the liquidation. [7](#0-6) [8](#0-7)
