### Title
Silent Bonus Checkpoint Advance Without Payment During Insufficient-Pool Redelegation Slash — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

### Summary

`slashRedelegationPosition` calls `processEventsAndClaimBonus` via a pointer to `pos`. That function mutates the position's checkpoints in-memory **and** decrements event reference counts in the store **before** it checks pool balance. When the pool is insufficient it returns `ErrInsufficientBonusPool`. `slashRedelegationPosition` deliberately swallows that error, then calls `setPositionWithState` with the already-mutated `pos`, persisting the advanced checkpoints. The victim's accrued bonus is permanently unrecoverable.

### Finding Description

**Step 1 — Checkpoint and ref-count mutations happen before the pool check.**

Inside `processEventsAndClaimBonus`: [1](#0-0) 

`pos.UpdateLastEventSeq(entry.Seq)` mutates the caller's `pos` struct through the pointer, and `k.decrementEventRefCount(...)` is a direct KV-store write — both happen inside the event loop, before the pool check. [2](#0-1) 

`applyBonusAccrualCheckpoint` and `UpdateLastKnownBonded` also mutate `pos` through the pointer before the pool check. [3](#0-2) 

Only here does the pool check occur. When it fails, the function returns `(nil, ErrInsufficientBonusPool)` — but `pos.LastBonusAccrual`, `pos.LastEventSeq`, `pos.LastKnownBonded` are already advanced in the caller's struct, and the event ref-count decrements are already written to the store.

**Step 2 — The caller swallows the error and persists the mutated position.** [4](#0-3) 

`ErrInsufficientBonusPool` is caught, logged, and execution falls through. The `pos` variable now carries the advanced checkpoints. [5](#0-4) 

For a partial slash, `setPositionWithState(ctx, pos, nil)` is called with the already-mutated `pos`, persisting `LastBonusAccrual = blockTime`, the advanced `LastEventSeq`, and the updated `LastKnownBonded` — all without having transferred a single coin to the owner.

For a full slash, `ClearBonusCheckpoints()` overwrites the accrual timestamp, but the event ref-count decrements are already committed and the bonus is still never paid.

**Step 3 — The bonus is permanently unrecoverable.**

On the next claim, `processEventsAndClaimBonus` starts from the now-advanced `LastBonusAccrual` and `LastEventSeq`. The segment that accrued before the slash is gone. Because the ref-counts were also decremented, those events may have been garbage-collected, making recovery impossible even if the pool is later replenished.

### Impact Explanation

A position owner whose redelegation is slashed while the rewards pool is empty permanently loses all bonus rewards that accrued since their last checkpoint. The loss is silent — no error is surfaced on-chain, no event is emitted for the forfeiture, and no retry path exists. This is a direct, permanent loss of economically valuable user funds from the `RewardsPoolName` module account that should have been transferred to the owner.

### Likelihood Explanation

The rewards pool can reach zero through normal operation: the BeginBlocker top-up drains it to cover base-reward shortfalls, and any user can legitimately claim their own bonus rewards. A targeted attacker who holds a large position can claim their own accrued bonus to drain the pool to zero, then wait for any redelegation slash event (double-sign or downtime) against a validator that has a victim's redelegating position. The attacker does not need validator-operator access — they only need to time their own legitimate `MsgClaimTierRewards` call before a slash that is already in progress or predictable (e.g., a jailed validator's pending slash).

### Recommendation

Move the `sufficientBonusPoolBalance` check **before** any mutation of `pos` or any store write inside `processEventsAndClaimBonus`. Alternatively, if the pool-insufficient path must be a silent forfeit in the hook context, reset the in-memory `pos` checkpoints back to their pre-call values before returning the error, so that `slashRedelegationPosition` persists the original (unpaid, unadvanced) state. The event ref-count decrements must also be deferred until after the pool check succeeds.

### Proof of Concept

```
1. Fund the rewards pool with exactly enough to cover one claim.
2. Create position A (attacker) and position B (victim), both delegated to validator V.
3. Victim redelegates from V to V2 — a RedelegationMapping entry is created.
4. Advance time so both positions accrue bonus.
5. Attacker calls MsgClaimTierRewards for position A, draining the pool to zero.
6. Slash validator V2 (redelegation slash path) — BeforeRedelegationSlashed fires.
7. slashRedelegationPosition → processEventsAndClaimBonus:
   - LastBonusAccrual, LastEventSeq, LastKnownBonded mutated on pos (pointer).
   - Event ref-counts decremented in store.
   - sufficientBonusPoolBalance returns ErrInsufficientBonusPool.
   - Error returned to slashRedelegationPosition.
8. slashRedelegationPosition swallows the error, calls setPositionWithState(pos).
9. Assert: victim's bank balance is unchanged (no bonus paid).
10. Assert: victim's position.LastBonusAccrual == blockTime (checkpoint advanced).
11. Replenish pool. Victim calls MsgClaimTierRewards.
12. Assert: victim receives zero bonus for the pre-slash period (permanently lost).
```
