### Title
Linked Signer Can Replace Itself via `LinkSigner`, Enabling Full Subaccount Takeover and Collateral Theft — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction handler in `EndpointTx.sol` validates the signature with `allowLinkedSigner = true`. Because `Verifier.validateSignature` treats the linked signer as fully equivalent to the subaccount owner, the current linked signer can sign a `LinkSigner` transaction to replace itself with any attacker-controlled address. The attacker's new linked signer then signs a `WithdrawCollateral` transaction — also validated with `allowLinkedSigner = true` — to drain the subaccount's collateral.

---

### Finding Description

In `processTransactionImpl`, the `LinkSigner` branch decodes the transaction and calls `validateSignedTx` with the fifth argument hardcoded to `true`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true          // ← allowLinkedSigner
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [1](#0-0) 

`validateSignedTx` delegates to `validateSignature`, which passes the current linked signer to `Verifier.validateSignature`:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` accepts the recovered address if it equals **either** the subaccount owner **or** the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

This means the linked signer has identical authority to the owner for the `LinkSigner` operation itself — it can overwrite `linkedSigners[subaccount]` to any address.

Once the linked signer is replaced, the attacker-controlled address signs a `WithdrawCollateral` transaction, which is also validated with `allowLinkedSigner = true`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true          // ← allowLinkedSigner
);
``` [4](#0-3) 

This drains the subaccount's collateral.

The protocol designers were aware that linked signers should not have unrestricted withdrawal authority: `WithdrawCollateralV2` conditionally restricts the linked signer based on whether `sendTo` is non-zero:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    signedTx.tx.sendTo == address(0)   // ← linked signer blocked when sendTo != 0
);
``` [5](#0-4) 

No equivalent restriction was applied to `LinkSigner`, leaving the privilege-escalation path open.

---

### Impact Explanation

**Impact: High.**

Any subaccount that has a linked signer set is fully compromised if that linked signer key is malicious or becomes compromised. The attacker can:

1. Permanently replace the linked signer with an address they control.
2. Use the new linked signer to sign `WithdrawCollateral` and drain all collateral from the subaccount.

The corrupted state is `linkedSigners[subaccount]` (overwritten to attacker address) and the subaccount's spot balances (drained to zero via `clearinghouse.withdrawCollateral`). [6](#0-5) 

---

### Likelihood Explanation

**Likelihood: High.**

Linked signers are a standard feature used by trading bots, sequencer integrations, and third-party services. Any of these scenarios creates a realistic threat:

- A compromised trading bot key.
- A malicious third-party service that was granted linked signer status.
- A user who set a linked signer and later lost control of that key.

The attack requires no special on-chain conditions, no admin access, and no governance capture — only possession of the current linked signer's private key. [7](#0-6) 

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction handler so that only the subaccount owner (the address embedded in `sender`) can change the linked signer:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
``` [8](#0-7) 

This mirrors the design intent already visible in `WithdrawCollateralV2`, where the linked signer is blocked from redirecting funds to an arbitrary `sendTo` address.

### Proof of Concept

```
Precondition:
  - Victim subaccount: sender = bytes32(victimAddress << 96 | subaccountId)
  - linkedSigners[sender] = botAddress  (set legitimately by victim)

Step 1 — Attacker (controlling botAddress) crafts a LinkSigner tx:
  LinkSigner {
      sender: victimSubaccount,
      signer: bytes32(attackerAddress),   // attacker-controlled key
      nonce:  currentNonce
  }
  Signs with botAddress private key.
  Submits to sequencer (or via slow-mode queue).

Step 2 — Sequencer processes the tx:
  validateSignedTx(..., true)
    → getLinkedSigner(victimSubaccount) == botAddress
    → ECDSA.recover(digest, sig) == botAddress  ✓ passes
  linkedSigners[victimSubaccount] = attackerAddress   ← state corrupted

Step 3 — Attacker crafts a WithdrawCollateral tx:
  WithdrawCollateral {
      sender:    victimSubaccount,
      productId: QUOTE_PRODUCT_ID,
      amount:    fullBalance,
      nonce:     currentNonce
  }
  Signs with attackerAddress private key.

Step 4 — Sequencer processes the tx:
  validateSignedTx(..., true)
    → getLinkedSigner(victimSubaccount) == attackerAddress
    → ECDSA.recover(digest, sig) == attackerAddress  ✓ passes
  clearinghouse.withdrawCollateral(victimSubaccount, ..., address(0), ...)
    → funds sent to victimAddress (the owner's wallet)
    → but attacker already controls victimAddress via the linked signer chain,
      or repeats for all products until balance is zero.

Result: All collateral drained from victimSubaccount.
``` [9](#0-8)
