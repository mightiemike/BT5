### Title
Silent Discard of Failed Slow-Mode `DepositCollateral` Transactions Permanently Locks User Funds After `EndpointTx` Upgrade — (File: `core/contracts/Endpoint.sol`)

---

### Summary

`depositCollateralWithReferral` transfers user tokens into the `Endpoint` contract **before** queuing a slow-mode `DepositCollateral` transaction. If that queued transaction subsequently fails during execution — including after an `EndpointTx` upgrade that introduces any breaking change in `processSlowModeTransactionImpl` — `_executeSlowModeTransaction` silently discards the failure. The refund path was explicitly removed (comment: `"try return funds now removed"`). The user's tokens are permanently locked with no recovery mechanism.

---

### Finding Description

**Step 1 — Tokens transferred before slow-mode tx is queued.**

`depositCollateralWithReferral` calls `handleDepositTransfer` to pull tokens from the caller into the `Endpoint` contract, then enqueues a `DepositCollateral` slow-mode transaction: [1](#0-0) 

The token transfer and the slow-mode queue entry are **not atomic with the eventual on-chain credit**. Tokens are in the contract; the user's balance has not yet been updated.

**Step 2 — Slow-mode execution deletes the tx before attempting it, then swallows failures.**

`_executeSlowModeTransaction` removes the entry from `slowModeTxs` unconditionally before calling `processSlowModeTransaction`: [2](#0-1) 

It then wraps execution in a `try/catch` that silently discards any non-OOG revert: [3](#0-2) 

The comment on line 226 — `"try return funds now removed"` — explicitly confirms that a refund path existed and was deliberately deleted, leaving no recovery for the deposited tokens.

**Step 3 — `EndpointTx` is upgradeable, creating the desynchronization window.**

`upgradeEndpointTx` replaces the `endpointTx` address used by every `_delegatecallEndpointTx` call: [4](#0-3) 

`processSlowModeTransaction` delegates to `EndpointTx.processSlowModeTransactionImpl` via this address: [5](#0-4) 

If the new `EndpointTx` implementation introduces any breaking change in `processSlowModeTransactionImpl` — a new validation, a renamed function, a changed struct layout, or a new revert condition for a product that was valid at submission time — every pending `DepositCollateral` slow-mode tx queued against the old implementation will revert. Because the tx was already deleted from the queue and the catch block is empty, the tokens are gone.

**Step 4 — `ClearinghouseLiq` upgrade compounds the surface.**

`upgradeClearinghouseLiq` similarly replaces the `clearinghouseLiq` delegatecall target: [6](#0-5) 

Any breaking change there can also cause `clearinghouse.depositCollateral` to revert inside the silent catch block.

---

### Impact Explanation

User ERC20 tokens transferred to the `Endpoint` contract via `depositCollateralWithReferral` are permanently locked. The slow-mode queue entry is deleted before execution; the catch block has no refund logic; there is no admin rescue function for stranded deposits. The corrupted asset delta is the full `amount` of the deposit, denominated in the deposited token.

---

### Likelihood Explanation

The trigger requires an `EndpointTx` or `ClearinghouseLiq` upgrade that is not backward-compatible with the encoding or validation logic of already-queued slow-mode transactions. The protocol has an active upgrade mechanism (`migrateAll`, `upgradeEndpointTx`, `upgradeClearinghouseLiq`) and the slow-mode queue has a 3-day delay window (`SLOW_MODE_TX_DELAY = 3 days`), meaning any upgrade during that window creates the exposure. The comment `"try return funds now removed"` confirms the developers are aware of the failure path and chose not to handle it, matching the RocketPool team's "we control upgrades" posture. Likelihood is **low-medium**: it requires an upgrade with a breaking change, but the 3-day window and the absence of any pre-upgrade queue-drain check make it a realistic operational risk.

---

### Recommendation

1. **Restore a refund path** in the `catch` block of `_executeSlowModeTransaction` for `DepositCollateral` transactions: if execution fails, transfer the deposited tokens back to `txn.sender`.
2. **Drain or migrate the slow-mode queue** before any `EndpointTx` or `ClearinghouseLiq` upgrade, analogous to the RocketPool recommendation of checking for pending operations before upgrading.
3. **Decouple token custody from queue submission**: do not transfer tokens until the slow-mode tx is actually processed, or use a two-phase commit that holds tokens in escrow with an explicit cancellation path.

---

### Proof of Concept

```
1. Alice calls Endpoint.depositCollateralWithReferral(aliceSubaccount, productId, 1000e6, "")
   → handleDepositTransfer pulls 1000 USDC from Alice into Endpoint
   → slowModeTxs[N] = SlowModeTx{ executableAt: now+3days, sender: Alice, tx: DepositCollateral{...} }

2. Within the 3-day window, admin calls upgradeEndpointTx(newEndpointTx)
   → newEndpointTx.processSlowModeTransactionImpl now reverts for DepositCollateral
     (e.g., new validation rejects the old struct encoding, or product was delisted in new logic)

3. After 3 days, Alice (or anyone) calls Endpoint.executeSlowModeTransaction()
   → _executeSlowModeTransaction:
       delete slowModeTxs[N]          // tx removed from queue
       try this.processSlowModeTransaction(Alice, tx) {}
       catch { /* gasleft check passes, silent return */ }
   → No revert, no refund, no balance credit

4. Alice's 1000 USDC is permanently locked in Endpoint.
   slowModeTxs[N] is gone. No admin rescue function exists.
``` [7](#0-6) [8](#0-7) [4](#0-3)

    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned
        requireUnsanctioned(msg.sender);
        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
            return;
        }

        handleDepositTransfer(
            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)
        );
        // copy from submitSlowModeTransaction
        SlowModeConfig memory _slowModeConfig = slowModeConfig;

        slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: abi.encodePacked(
                uint8(TransactionType.DepositCollateral),
                abi.encode(
                    DepositCollateral({
                        sender: subaccount,
                        productId: productId,
                        amount: amount
                    })
                )
            )
        });
        slowModeConfig = _slowModeConfig;
    }

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
