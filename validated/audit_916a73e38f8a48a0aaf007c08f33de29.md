### Title
`GatewayDynamicConfig` updates stored in `ConfigManager` but never propagated to the gateway component â€” (`crates/apollo_config_manager/src/config_manager.rs`)

### Summary

`NodeDynamicConfig` carries a `gateway_dynamic_config` field that is accepted and stored by `ConfigManager` via `set_node_dynamic_config`, but the `ConfigManagerClient` trait exposes no `get_gateway_dynamic_config` method, and the gateway component is never passed a `config_manager_client` at construction. As a result, any live update to `gateway_dynamic_config.native_classes_whitelist` is silently discarded: the gateway continues executing stateful validation with the stale whitelist it received at startup, while the batcher's contract class manager may operate under the updated whitelist. This is a direct sequencer-native analog of the "parameter updated at the top level but not propagated to dependent components" pattern from the original report.

### Finding Description

`NodeDynamicConfig` declares `gateway_dynamic_config: Option<GatewayDynamicConfig>` as a first-class dynamic field alongside `batcher_dynamic_config`, `mempool_dynamic_config`, etc. [1](#0-0) 

`ConfigManagerRunner::update_config` reads the config file, constructs a fresh `NodeDynamicConfig`, and calls `set_node_dynamic_config` on the `ConfigManager`, which atomically replaces the stored config including `gateway_dynamic_config`. [2](#0-1) 

`ConfigManager::set_node_dynamic_config` stores the entire `NodeDynamicConfig` (including the new `gateway_dynamic_config`) under an `RwLock`. [3](#0-2) 

However, the `ConfigManagerClient` trait lists getters for every other component's dynamic config but has **no `get_gateway_dynamic_config` method**: [4](#0-3) 

Correspondingly, the `ConfigManagerRequest` enum has no `GetGatewayDynamicConfig` variant, and the `handle_config_request!` macro in `ConfigManager` does not dispatch it: [5](#0-4) 

At construction, the gateway receives a cloned `GatewayConfig` but is **not passed a `config_manager_client`**, so it has no channel through which to poll for updates: [6](#0-5) 

Compare this to the batcher, which receives `config_manager_client` and uses `get_batcher_dynamic_config` to refresh its config on each storage-reader cycle: [7](#0-6) 

`GatewayDynamicConfig` contains `native_classes_whitelist: NativeClassesWhitelist`, which controls whether the gateway's own `ContractClassManager` selects a native-compiled artifact or a CASM artifact for a given class hash during stateful validation: [8](#0-7) 

### Impact Explanation

The gateway's `ContractClassManager` uses `native_classes_whitelist` to decide whether to run a class through the Cairo Native path or the CASM path during stateful transaction validation. If the whitelist is updated at runtime (e.g., to remove a class that was found to produce divergent results under native compilation, or to add a newly audited class), the gateway continues using the stale whitelist indefinitely. This means:

1. **Wrong compiled artifact selected for execution during admission**: the gateway may execute stateful validation with a native artifact for a class that has been removed from the whitelist (or vice versa), while the batcher executes the same class under the updated whitelist. This is a direct instance of "Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution."

2. **Admission divergence**: if native and CASM produce different execution outcomes for a specific class (e.g., due to a native-compiler bug or a class that exercises a divergent code path), the gateway may accept a transaction that the batcher will revert, or reject a transaction that the batcher would accept. This maps to "Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."

### Likelihood Explanation

The `ConfigManagerRunner` is enabled by default in production deployments (`enable_config_updates: true` in `config_manager_config.json`) and polls every 60 seconds. Any operator-triggered update to `gateway_dynamic_config.native_classes_whitelist` in the config file will be silently dropped. The divergence is permanent until the gateway process is restarted. No special attacker capability is required beyond the ability to observe that the gateway's behavior does not match the updated config. [9](#0-8) 

### Recommendation

1. Add `GetGatewayDynamicConfig` to `ConfigManagerRequest`, `ConfigManagerResponse`, and `ConfigManagerClient`, mirroring the pattern used for `GetBatcherDynamicConfig`.
2. Pass a `config_manager_client` to `create_gateway` and have the gateway poll `get_gateway_dynamic_config` on each request cycle (or on a timer), applying the updated `native_classes_whitelist` to its `ContractClassManager`.
3. Alternatively, move `native_classes_whitelist` into `GatewayStaticConfig` if it is not intended to be hot-reloadable, and remove `GatewayDynamicConfig` from `NodeDynamicConfig` to eliminate the false expectation of live propagation.

### Proof of Concept

1. Start the sequencer with `enable_config_updates: true`.
2. Update the config file to change `gateway_dynamic_config.native_classes_whitelist` (e.g., remove a class hash that was previously whitelisted for native execution).
3. Wait for `ConfigManagerRunner::update_config` to fire (â‰¤60 s). Observe via logs that `set_node_dynamic_config` succeeds and the new `NodeDynamicConfig` is stored.
4. Submit a transaction that invokes the class hash that was removed from the whitelist.
5. Observe that the gateway's `ContractClassManager` still selects the native artifact for that class (stale whitelist), while the batcher's `ContractClassManager` (which does poll `get_batcher_dynamic_config`) uses the updated whitelist and selects CASM.
6. If native and CASM produce divergent results for that class, the gateway accepts the transaction while the batcher reverts it â€” or vice versa.

The root cause is the missing `GetGatewayDynamicConfig` dispatch path in `ConfigManager::handle_request` and the absence of a `config_manager_client` in the gateway's construction call. [10](#0-9) [11](#0-10)

    pub(crate) async fn update_config(
        &mut self,
    ) -> Result<NodeDynamicConfig, Box<dyn std::error::Error + Send + Sync>> {
        let config = load_and_validate_config(self.cli_args.clone(), false).map_err(|e| {
            CONFIG_MANAGER_UPDATE_ERRORS.increment(1);
            error!("ConfigManagerRunner: failed to update config: {e}");
            e
        })?;
        let node_dynamic_config = NodeDynamicConfig::from(&config);

        // Compare the previous and the newly read node dynamic config.
        if self.latest_node_dynamic_config == node_dynamic_config {
            // No change, so no action is needed.
            Ok(node_dynamic_config)
        } else {
            // Log the diff between the latest and the new node dynamic config.
            self.log_config_diff(&self.latest_node_dynamic_config, &node_dynamic_config);
            // Update the latest node dynamic config.
            self.latest_node_dynamic_config = node_dynamic_config.clone();
            match self
                .config_manager_client
                .set_node_dynamic_config(node_dynamic_config.clone())
                .await
            {
                Ok(()) => {
                    info!("Successfully updated dynamic config");
                    Ok(node_dynamic_config)
                }
                Err(e) => Err(format!("Failed to update dynamic config: {:?}", e).into()),
            }
        }
    }

    pub(crate) async fn set_node_dynamic_config(
        &self,
        node_dynamic_config: NodeDynamicConfig,
    ) -> ConfigManagerResult<()> {
        info!("ConfigManager: updating node dynamic config");
        if let Err(errors) = node_dynamic_config.validate() {
            error!(
                "ConfigManager: dynamic config update rejected: {errors}. Keeping previous config."
            );
            return Err(ConfigManagerError::InvalidConfig(errors.to_string()));
        }
        let mut config = self.latest_node_dynamic_config.write().await;
        *config = node_dynamic_config;
        Ok(())
    }
