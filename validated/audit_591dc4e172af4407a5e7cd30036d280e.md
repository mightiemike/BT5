### Title
Linked signer can replace itself via `LinkSigner` transaction due to `allowLinkedSigner = true` — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` fast-mode transaction is validated with `allowLinkedSigner = true`, meaning the current linked signer — not just the account owner — can sign a `LinkSigner` transaction to overwrite the `linkedSigners` mapping to any address. This allows a compromised linked signer to maintain or transfer persistent signing authority over the account, and to race against the account owner's revocation attempts.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch reads:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true          // allowLinkedSigner = true
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [1](#0-0) 

The fifth argument `true` is passed to `validateSignedTx`, which forwards it to `validateSignature`:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` accepts a signature from either the account owner address or the linked signer address:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

This means the current linked signer can sign a `LinkSigner` transaction to overwrite `linkedSigners[subaccount]` to any address — including an attacker-controlled one.

**The slow-mode `LinkSigner` path does not have this flaw.** It uses `validateSender`, which enforces that `msg.sender` is the address embedded in the subaccount bytes (i.e., the account owner):

```solidity
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

`validateSender` enforces:

```solidity
require(
    address(uint160(bytes20(txSender))) == sender ||
        sender == address(this),
    ERR_SLOW_MODE_WRONG_SENDER
);
``` [5](#0-4) 

The fast-mode path has no equivalent owner-only restriction.

**Nonce sharing compounds the issue.** The nonce counter is keyed on `address(uint160(bytes20(sender)))` — the owner's address — and is shared between the account owner and the linked signer for the same subaccount:

```solidity
function validateNonce(bytes32 sender, uint64 nonce) internal virtual {
    require(
        nonce == nonces[address(uint160(bytes20(sender)))]++,
        ERR_WRONG_NONCE
    );
}
``` [6](#0-5) 

If the account owner and the attacker (holding the compromised linked signer key) both submit `LinkSigner` transactions with the same nonce, whichever is sequenced first wins and invalidates the other. The attacker can race the owner's revocation.

---

### Impact Explanation

A compromised linked signer key can:

1. Sign a `LinkSigner` transaction to change `linkedSigners[subaccount]` to an attacker-controlled address.
2. Submit it to the sequencer for fast-mode processing.
3. If the account owner simultaneously submits a revocation `LinkSigner` transaction, the attacker can race it — both share the same nonce counter, so the first sequenced wins.
4. Once the attacker controls the linked signer slot, they can sign `WithdrawCollateral` transactions (also `allowLinkedSigner = true`) to drain the account's collateral.

The corrupted state is `linkedSigners[subaccount]` — the account's signing authority delegation — and the downstream asset delta is full collateral withdrawal from the account.

---

### Likelihood Explanation

Linked signers are typically hot wallets or API keys used for automated trading, which are materially more likely to be compromised than cold-wallet account owners. The attacker needs only the linked signer's private key — no admin access, no governance capture, no sequencer compromise. This is a realistic threat for any account that has set a linked signer.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`, so only the account owner's key is accepted:

```diff
 validateSignedTx(
     signedTx.tx.sender,
     signedTx.tx.nonce,
     transaction,
     signedTx.signature,
-    true
+    false
 );
```

This aligns the fast-mode path with the slow-mode path, which already correctly restricts `LinkSigner` to the account owner via `validateSender`.

---

### Proof of Concept

1. Alice (owner of subaccount `S`) sets Bob's address as her linked signer via a `LinkSigner` transaction.
2. An attacker compromises Bob's private key.
3. The attacker constructs a `SignedLinkSigner` transaction: `sender = S`, `signer = attacker_address`, `nonce = current_nonce`.
4. The attacker signs it with Bob's key and submits it to the sequencer.
5. The sequencer calls `processTransactionImpl` → `validateSignedTx(..., true)` → `validateSignature` accepts Bob's signature because `recovered == linkedSigner`.
6. `linkedSigners[S]` is overwritten to `attacker_address`.
7. Alice notices and signs a revocation `LinkSigner` transaction with `nonce = current_nonce + 1`.
8. The attacker, now holding `attacker_address`'s key (which is the current linked signer), signs another `LinkSigner` transaction with `nonce = current_nonce + 1` to re-set the linked signer back to themselves.
9. Whichever transaction the sequencer processes first wins; the attacker can attempt to front-run Alice's revocation.
10. Once the attacker controls the linked signer slot, they sign a `WithdrawCollateral` transaction to drain Alice's collateral.
