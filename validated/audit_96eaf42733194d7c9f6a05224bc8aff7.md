### Title
Linked Signer Can Unilaterally Rotate Itself via `LinkSigner` Fast Path, Enabling Persistent Subaccount Hijacking — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The fast-path `LinkSigner` transaction is validated with `allowLinkedSigner = true`, meaning the **current linked signer** — not just the subaccount owner — can authorize a new `LinkSigner` transaction to replace itself with any arbitrary address. This is the on-chain analog of the overlay attack: a malicious or compromised linked signer silently "overlays" an attacker-controlled signer on top of the legitimate one, hijacking the subaccount without the owner's knowledge or consent.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch calls `validateSignedTx` with `allowLinkedSigner = true`: [1](#0-0) 

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true                          // ← allowLinkedSigner = true
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
```

`validateSignedTx` with `allowLinkedSigner = true` passes the current linked signer to `validateSignature`: [2](#0-1) 

`validateSignature` then accepts a signature from **either** the subaccount owner address **or** the linked signer: [3](#0-2) 

Because the `LinkSigner` transaction itself mutates `linkedSigners`, the current linked signer can sign a `LinkSigner` payload that replaces itself with any attacker-controlled address — no subaccount owner key required.

**Contrast with the slow-mode path**, which correctly restricts `LinkSigner` to the subaccount owner via `validateSender` (no signature from the linked signer is accepted): [4](#0-3) 

The asymmetry between the two paths is the root cause: the fast path grants the linked signer a privilege — self-rotation — that the slow-mode path explicitly withholds.

---

### Impact Explanation

Once the attacker's address is installed as the linked signer, it can authorize any transaction that accepts `allowLinkedSigner = true`. The most direct theft vector is `TransferQuote`, which is also validated with `allowLinkedSigner = true`: [5](#0-4) 

`TransferQuote.recipient` is an arbitrary `bytes32` subaccount — including one controlled by the attacker. The attacker can therefore drain the victim's entire quote balance to their own subaccount in a single sequencer batch.

The hijack is **persistent**: even if the victim notices and submits a revocation `LinkSigner` via slow mode (3-day delay), the attacker's installed linked signer can immediately re-issue another fast-path `LinkSigner` to re-install itself, racing ahead of the victim's slow-mode revocation.

Corrupted state: `linkedSigners[victim_subaccount]` → attacker address; victim quote balance → zero.

---

### Likelihood Explanation

Linked signers are by design hot wallets or API-key-backed addresses used for automated trading. They are materially more likely to be compromised than the cold-wallet subaccount owner key. A single private-key leak, phishing event, or malicious SDK integration is sufficient to trigger the attack. No admin access, sequencer compromise, or governance capture is required — only a valid ECDSA signature from the currently registered linked signer.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` branch in `processTransactionImpl`:

```solidity
// EndpointTx.sol — processTransactionImpl, LinkSigner branch
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // ← only the subaccount owner key may rotate the linked signer
);
```

This aligns the fast path with the slow-mode path, which already enforces owner-only authorization for `LinkSigner`.

---

### Proof of Concept

1. Alice owns `alice_subaccount` and has registered `alice_hot_wallet` as her linked signer via a prior `LinkSigner` transaction.
2. Attacker obtains the private key of `alice_hot_wallet` (e.g., via a compromised trading bot).
3. Attacker constructs a `SignedLinkSigner` payload:
   - `tx.sender = alice_subaccount`
   - `tx.signer = attacker_address` (packed as `bytes32`)
   - `tx.nonce = current_nonce_of_alice`
   - `signature` = ECDSA signature over the EIP-712 digest, signed by `alice_hot_wallet`
4. Attacker submits the payload to the sequencer. The sequencer calls `processTransactionImpl`; `validateSignedTx(..., true)` accepts the `alice_hot_wallet` signature; `linkedSigners[alice_subaccount]` is set to `attacker_address`.
5. Attacker constructs a `SignedTransferQuote` payload:
   - `tx.sender = alice_subaccount`
   - `tx.recipient = attacker_subaccount`
   - `tx.amount = alice_full_quote_balance`
   - `signature` = ECDSA signature signed by `attacker_address` (now the registered linked signer)
6. Sequencer processes `TransferQuote`; `allowLinkedSigner = true` accepts the attacker's signature; Alice's entire quote balance is transferred to `attacker_subaccount`.
7. If Alice attempts slow-mode revocation (3-day delay), the attacker re-issues step 3 with the next nonce before the slow-mode tx executes, maintaining persistent control.

        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
