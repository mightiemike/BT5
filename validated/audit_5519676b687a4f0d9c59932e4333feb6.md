### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Merkle Proof Guard — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is still a live, unprivileged public entry point on the contract. `verify_transaction_inclusion_v2` added a mandatory coinbase Merkle proof check specifically to close the 64-byte transaction forgery vulnerability. Because v1 is never removed or access-gated, any NEAR caller can invoke it directly, bypassing the only guard that prevents Merkle proof forgery. This is a direct analog to M-02: one operation path (v1) lacks a safety check that a parallel path (v2) enforces, and the less-guarded path is fully reachable by an unprivileged caller.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced to mitigate the well-known 64-byte transaction Merkle proof forgery attack (https://www.bitmex.com/blog/64-Byte-Transactions). Its protection is a coinbase Merkle proof check: it requires the caller to prove that the coinbase transaction (index 0) hashes to the block's `merkle_root`, which anchors the tree depth and prevents an internal node from being mistaken for a leaf transaction. [1](#0-0) 

The deprecated v1 function performs no such check. It fetches the block header from `headers_pool` and directly compares the computed Merkle root against the caller-supplied `tx_id` and `merkle_proof`, with no coinbase anchor. [2](#0-1) 

The contract's own documentation acknowledges this gap explicitly: [3](#0-2) 

Despite this, v1 is declared `pub` and carries only a `#[deprecated]` Rust attribute and a `#[pause]` gate. `#[deprecated]` is a compiler hint — it does not restrict runtime call dispatch. The function remains a fully reachable NEAR contract method callable by any account. [4](#0-3) 

The two paths side-by-side:

| Path | Coinbase proof check | Reachable by unprivileged caller |
|---|---|---|
| `verify_transaction_inclusion` (v1) | **No** | **Yes** |
| `verify_transaction_inclusion_v2` | Yes | Yes |

This is structurally identical to M-02: v2 enforces the invariant; v1 does not; v1 is still reachable.

---

### Impact Explanation

An attacker who constructs a 64-byte value that is simultaneously a valid-looking `tx_id` and an internal Merkle tree node can supply a crafted `merkle_proof` to v1 and receive a `true` return value for a transaction that does not exist in the block. Any downstream NEAR contract or off-chain bridge that calls `verify_transaction_inclusion` directly — or that an integrator routes to v1 — will accept a forged proof as valid. This corrupts the SPV guarantee that is the contract's sole security property.

---

### Likelihood Explanation

The 64-byte attack is a documented, public technique with known construction methods. The deprecated v1 method is part of the contract's public ABI and is callable by any NEAR account without any role or stake requirement. An attacker needs only to: (1) identify a target block on the canonical chain, (2) construct a 64-byte fake `tx_id` matching an internal node, (3) build the corresponding Merkle sibling path, and (4) call `verify_transaction_inclusion` directly. No privileged access, leaked keys, or social engineering is required.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public ABI entirely, or gate it behind a role (e.g., `Role::DAO`) so it cannot be called by unprivileged accounts. The `#[deprecated]` attribute provides no runtime protection. If backward compatibility is required, the function body should be replaced with a delegation to v2 that requires the caller to supply a coinbase proof.

---

### Proof of Concept

1. Identify any block hash `B` present in `mainchain_header_to_height` (i.e., on the canonical chain).
2. Construct a 64-byte value `fake_tx_id` that equals an internal Merkle tree node at depth `d` in block `B`'s transaction tree.
3. Build `merkle_proof` of length `d` consisting of the sibling hashes that, when combined with `fake_tx_id` at `tx_index`, reproduce `B`'s `merkle_root`.
4. Call `verify_transaction_inclusion` with `tx_id = fake_tx_id`, `tx_block_blockhash = B`, `tx_index` matching the constructed path, `merkle_proof` as above, `confirmations = 1`.
5. The function returns `true`.

The same call to `verify_transaction_inclusion_v2` would fail at the coinbase proof check (lines 358–365) because the coinbase transaction at index 0 would not hash to `merkle_root` via the supplied `coinbase_merkle_proof` of the same length, exposing the depth mismatch. v1 never performs this check. [5](#0-4)

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
