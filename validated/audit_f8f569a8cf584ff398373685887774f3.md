Looking at the actual code:

**`SwapAllowlistExtension.beforeSwap`** checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool (the caller of the extension) and `sender` is the argument passed by the pool — which is `msg.sender` of `pool.swap()`.

Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the originating user, breaking router-mediated swaps and enabling allowlist bypass — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is the pool's own `msg.sender` — the router contract when users swap via `MetricOmmSimpleRouter`. The extension therefore checks whether the router is allowlisted, not the actual economic actor. This produces two concrete broken-functionality paths: (1) individually allowlisted users are blocked from using the router because the router address is not in the allowlist; (2) if the pool admin allowlists the router to restore router access, every non-allowlisted user bypasses the per-user gate entirely.

## Finding Description

**Root cause — pool passes its own `msg.sender` as `sender`:**

`MetricOmmPool.swap()` calls `_beforeSwap` with `msg.sender` as the first argument:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // whoever called pool.swap() — the router, not the user
    recipient,
    zeroForOne,
    amountSpecified,
    priceLimitX64,
    packedSlot0Initial,
    bidPriceX64,
    askPriceX64,
    extensionData
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged as `sender` to every configured extension.

**Extension checks `sender` (the router) against the allowlist:**

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension's caller) and `sender` is the router address. The lookup `allowedSwapper[pool][router]` is evaluated, not `allowedSwapper[pool][user]`.

**Router call path — user address is never forwarded:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The original `msg.sender` (the user) is stored only in transient callback context for payment purposes; it is never passed to `pool.swap()` as a parameter that the pool could forward to extensions.

**Exploit path A — allowlisted user blocked from router:**
1. Pool admin deploys pool with `SwapAllowlistExtension` and calls `setAllowedToSwap(pool, alice, true)`.
2. Alice calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
3. Router calls `pool.swap(...)` — pool's `msg.sender` = router address.
4. Extension checks `allowedSwapper[pool][router]` → `false` → reverts `NotAllowedToSwap`.
5. Alice cannot use the router despite being individually allowlisted.

**Exploit path B — allowlist bypass via router:**
1. Pool admin, to fix path A, calls `setAllowedToSwap(pool, router, true)`.
2. Any unprivileged user (not in the allowlist) calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
3. Router calls `pool.swap(...)` — pool's `msg.sender` = router.
4. Extension checks `allowedSwapper[pool][router]` → `true` → passes.
5. Every user bypasses the per-user allowlist by routing through the router.

**Existing guards are insufficient:** There is no mechanism in the router, pool, or extension to thread the originating user's address through to the extension check. The `extensionData` bytes field is user-controlled and not authenticated, so it cannot serve as a trusted identity carrier.

## Impact Explanation
Path A renders the primary swap interface (`MetricOmmSimpleRouter`) completely unusable for any pool that uses `SwapAllowlistExtension` with individual allowlist entries — a broken core swap flow. Path B allows any unprivileged user to bypass a pool-admin-configured access control by routing through the public router, defeating the entire purpose of the allowlist. Both paths are triggered by normal, unprivileged user actions with no special preconditions beyond the pool having the extension configured.

## Likelihood Explanation
Any pool that deploys `SwapAllowlistExtension` with per-address entries (the primary use case) is immediately affected. No attacker capability beyond calling the public router is required. The condition is deterministic and repeatable on every router-mediated swap attempt.

## Recommendation
The extension must check the true originating user, not the immediate caller of `pool.swap()`. Options:
1. **Pass originating user via authenticated `extensionData`:** The router encodes `msg.sender` into `extensionData` and the extension reads it — but this requires a trust model where the extension only accepts this encoding from known routers.
2. **Add a `tx.origin` fallback (not recommended for general use):** Only viable in restricted, non-contract-caller contexts.
3. **Preferred — pool-level originator field:** Extend the `swap()` interface with an explicit `originator` parameter that the pool passes to extensions, allowing routers to supply the true user address in a standardized, auditable way.
