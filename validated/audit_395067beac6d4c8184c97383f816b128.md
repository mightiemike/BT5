### Title
Deflationary / Fee-on-Transfer Token Deposit Overcredits Subaccount Balance, Causing Clearinghouse Insolvency — (File: `core/contracts/EndpointStorage.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

`Endpoint.depositCollateralWithReferral()` transfers the caller-supplied `amount` in two sequential hops (user → Endpoint → Clearinghouse) without measuring the actual tokens received at each step. `Clearinghouse.depositCollateral()` then credits the original `amount` to the subaccount's on-chain balance. When a fee-on-transfer (deflationary) token is configured as a product's collateral, the Clearinghouse holds fewer tokens than the sum of all credited balances, making the protocol insolvent for that product.

---

### Finding Description

The deposit entry point is `Endpoint.depositCollateralWithReferral()`: [1](#0-0) 

It delegates to `handleDepositTransfer()` in `EndpointStorage`: [2](#0-1) 

`handleDepositTransfer` performs **two** sequential ERC-20 transfers, both for the same nominal `amount`:

1. `safeTransferFrom(token, from, amount)` — user → Endpoint
2. `safeTransferTo(token, address(clearinghouse), amount)` — Endpoint → Clearinghouse

Neither call measures the actual balance delta. With a fee-on-transfer token, each hop deducts a fee from the recipient, so Clearinghouse receives `amount − fee₁ − fee₂` (or the second transfer reverts if Endpoint's balance is insufficient, causing a DoS; if Endpoint holds a pre-existing balance of that token the transfer succeeds and the discrepancy is silently introduced).

The slow-mode transaction queued during this call encodes the original `amount`. When the sequencer processes it, `Clearinghouse.depositCollateral()` credits the full nominal value: [3](#0-2) 

`amountRealized` is derived directly from `txn.amount` — the original caller-supplied figure — with no reconciliation against actual token holdings.

On the withdrawal side, `withdrawCollateral` transfers the full credited `amount` out of the Clearinghouse: [4](#0-3) 

Because the Clearinghouse holds fewer tokens than the sum of credited balances, later withdrawers cannot be made whole.

---

### Impact Explanation

**Corrupted state delta:** `spotEngine` balance for the depositing subaccount is inflated by `fee₁ + fee₂` per deposit. Across many depositors the Clearinghouse's actual token reserve diverges from the aggregate of all credited balances, eventually making `withdrawCollateral` revert for some users (insolvency).

**Scope:** Any product whose collateral token implements a transfer fee. The protocol imposes no whitelist or fee-on-transfer guard on `spotEngine.getConfig(productId).token`.

---

### Likelihood Explanation

The Nado protocol is designed to support arbitrary ERC-20 tokens as product collateral. Fee-on-transfer tokens (e.g., tokens with auto-staking mechanics, reflection tokens, or tokens with protocol fees) are a well-known token class. Any product configured with such a token — whether by governance or by a future product listing — immediately exposes all depositors to this accounting gap. No privileged attacker is required; ordinary user deposits are the trigger.

---

### Recommendation

Replace the two-hop transfer in `handleDepositTransfer` with a balance-before / balance-after check so that only the tokens actually received are credited:

```solidity
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
    uint256 actualReceived = token.balanceOf(address(clearinghouse)) - before;
    // pass actualReceived (not amount) into the slow-mode tx payload
}
```

Alternatively, enforce that only non-rebasing, non-fee-on-transfer tokens may be listed as product collateral by adding an on-chain invariant check at product registration time.

---

### Proof of Concept

1. A product is configured with a fee-on-transfer token `T` (1% fee per transfer).
2. Alice calls `depositCollateralWithReferral(subaccount, productId, 1000e18, "")`.
3. `handleDepositTransfer` executes:
   - `safeTransferFrom(T, Alice, 1000e18)` → Endpoint receives `990e18`.
   - `safeTransferTo(T, clearinghouse, 1000e18)` → if Endpoint has ≥ `1000e18` of `T` (from prior activity), Clearinghouse receives `990e18`; Endpoint's balance decreases by `1000e18`.
4. The slow-mode tx encodes `amount = 1000e18`.
5. Sequencer processes it; `Clearinghouse.depositCollateral` credits Alice's subaccount with `1000e18 * multiplier`.
6. Alice's on-chain balance = `1000e18`; Clearinghouse actual holding = `990e18`.
7. Alice withdraws `1000e18`; `withdrawCollateral` calls `token.safeTransfer(withdrawPool, 1000e18)` — Clearinghouse is short `10e18`, causing a revert or draining another user's collateral. [2](#0-1) [3](#0-2) [5](#0-4)
