### Title
Insurance Fund Drained Beyond Single-Call Cost via Sequential Spread/Spot Liability Liquidations — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

In `_handleLiquidationPayment`, when `txn.amount < 0` (liability buyback), the execution order is:

1. `insurance = spotEngine.updateQuoteFromInsurance(liquidatee, insurance)` — insurance is reduced to cover the negative quote balance
2. `insurance += v.liquidationFees` — fees are credited back to insurance

Because step 2 happens **before** the next call's step 1, a liquidator who splits a single liability liquidation into N sequential calls causes each call's fees to be immediately consumed by the next call's `updateQuoteFromInsurance`. The insurance fund ends up drained by more than a single combined call of the same total amount would cost.

---

### Finding Description

The storage variable `insurance` in `ClearinghouseStorage` is mutated in this exact sequence inside `_handleLiquidationPayment`:

```
// Step A: insurance reduced by covering negative quote balance
insurance = spotEngine.updateQuoteFromInsurance(txn.liquidatee, insurance);  // lines 501-506 / 538-543

// Step B: fees credited back
insurance += v.liquidationFees;   // line 579
lastLiquidationFees = v.liquidationFees;  // line 586
``` [1](#0-0) [2](#0-1) 

`updateQuoteFromInsurance` in `SpotEngine` tops up the liquidatee's negative quote balance up to `min(insurance, -balanceAmount)` and returns the reduced insurance: [3](#0-2) 

The developer comment at lines 581–585 explicitly acknowledges that fees being credited back to `insurance` before finalization can "refill" insurance and block socialization — but the same refill effect applies across sequential `_handleLiquidationPayment` calls, not just at finalization. [4](#0-3) 

---

### Impact Explanation

**Concrete accounting proof (two calls vs. one):**

Let:
- `I` = initial insurance
- `Q` = liquidatee's initial quote balance (negative)
- `lp` = `liquidationPrice * A` (negative, since `A < 0`)
- `f` = `liquidationFees` per call (positive, since `LIQUIDATION_FEE_FRACTION = 50%` and both factors are negative)

**Combined (one call, amount = 2A):**
- `updateQuoteFromInsurance` covers `min(I, -(Q + 2·lp))`
- If `I < -(Q + 2·lp)`: covers `I`, insurance → `0 + 2f = 2f`

**Sequential (two calls, each amount = A), assuming `I < -(Q + lp)`:**

| Step | Action | Insurance |
|------|--------|-----------|
| Call 1 step A | `updateQuoteFromInsurance` covers `I` | `0` |
| Call 1 step B | `+= f` | `f` |
| Call 2 step A | `updateQuoteFromInsurance` covers `min(f, -(Q+2·lp+I))` = `C₂` | `f - C₂` |
| Call 2 step B | `+= f` | `2f - C₂` |

Final insurance: `2f - C₂` vs. `2f` in the combined case.

**The insurance fund is drained by an extra `C₂ = min(f, -(Q + 2·lp + I)) > 0`.**

With `LIQUIDATION_FEE_FRACTION = 50%` and a spread penalty of at least `MIN_SPREAD_LIQ_PENALTY_X18 = 0.25%`, fees per call are `0.5 × 0.25% × |notional|`. For a $1M position split into 1000 calls of $1000 each, the cumulative extra drain is `999 × f_per_call`, which can be a meaningful fraction of the insurance fund. [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

- The liquidator controls `txn.amount` and can freely split any liability liquidation into N smaller calls.
- `liquidateSubaccountImpl` has no guard preventing multiple sequential liquidations of the same liquidatee in the same batch.
- The only constraint on minimum call size is `perpSizeIncrement`, which can be small.
- The condition `I < -(Q + lp)` (insurance insufficient to fully cover a partial liquidation) is exactly the stressed scenario where the insurance fund is already under pressure — the worst time for extra drain.
- No admin or sequencer compromise is required; any liquidator can do this. [7](#0-6) 

---

### Recommendation

Move `insurance += v.liquidationFees` to **before** the `updateQuoteFromInsurance` call, or accumulate fees separately and only credit them after all sequential liquidations for a given liquidatee are complete. Alternatively, snapshot the insurance value at the start of each `liquidateSubaccountImpl` call and only allow `updateQuoteFromInsurance` to draw from the pre-fee snapshot, preventing fees from one call from being consumed by the next.

The `lastLiquidationFees` mechanism already partially addresses this for `_finalizeSubaccount` (by withholding the last call's fees from socialization), but no equivalent protection exists for the sequential `_handleLiquidationPayment` path. [8](#0-7) 

---

### Proof of Concept

Setup (Hardhat, unmodified contracts):

1. Deploy with `insurance = 500 USDC`, liquidatee has `quote = -1100 USDC`, spread position of size `2A`.
2. **Combined:** Submit one `LiquidateSubaccount` with `isEncodedSpread=true`, `amount = 2A < 0`.
   - `updateQuoteFromInsurance` draws `500` from insurance (fully depleted).
   - `insurance += 2f` → final insurance = `2f`.
3. **Sequential:** Reset state. Submit two calls each with `amount = A < 0`.
   - Call 1: `updateQuoteFromInsurance` draws `500` (fully depleted). `insurance += f` → `f`.
   - Call 2: `updateQuoteFromInsurance` draws `min(f, remaining_negative_balance)` = `C₂ > 0`. `insurance += f` → `2f - C₂`.
4. Assert `insurance_sequential < insurance_combined` by exactly `C₂`.

The exact storage delta is `insurance` (slot in `ClearinghouseStorage`) ending lower in the sequential case, with the difference being `C₂ = min(f, -(Q + 2·lp + I))` — a concrete, measurable protocol asset loss. [9](#0-8)
