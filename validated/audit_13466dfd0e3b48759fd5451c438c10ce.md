### Title
Insurance Fund Has No Autonomous Replenishment Mechanism, Enabling Depositor-Loss Socialization During Cascading Liquidations — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

The Nado protocol's `insurance` fund is the sole backstop for bad debt during liquidations. When it is exhausted, `SpotEngine.socializeSubaccount()` is triggered, which permanently reduces `cumulativeDepositsMultiplierX18` for every depositor — a direct, irreversible loss of deposited value. There is no autonomous on-chain mechanism to replenish the insurance fund; the only replenishment path is a manual `depositInsurance()` transaction routed through the sequencer. During a market crash with cascading liquidations, bad debt can outpace both the insurance balance and the small per-liquidation fee accruals, leaving the protocol in a state of permanent depositor-loss socialization with no self-correcting mechanism.

---

### Finding Description

**Step 1 — Insurance is the only bad-debt backstop.**

`insurance` is declared as `int128 internal insurance` in `ClearinghouseStorage.sol`. [1](#0-0) 

It is consumed in two places during liquidation finalization:

- `perpEngine.socializeSubaccount(txn.liquidatee, v.insurance)` — insurance covers negative `vQuoteBalance` in perp positions.
- `insuranceCover = MathHelper.min(v.insurance, -quoteBalance.amount)` — insurance covers residual negative quote balance. [2](#0-1) 

**Step 2 — When insurance is exhausted, socialization is triggered.**

```solidity
if (v.insurance <= 0) {
    spotEngine.socializeSubaccount(txn.liquidatee);
}
``` [3](#0-2) 

**Step 3 — Socialization permanently reduces all depositors' value.**

`SpotEngine.socializeSubaccount()` iterates every product with a negative balance and reduces `cumulativeDepositsMultiplierX18`:

```solidity
state.cumulativeDepositsMultiplierX18 = (totalDeposited + balance.amount)
    .div(state.totalDepositsNormalized);
``` [4](#0-3) 

Because all subaccount balances are stored as `amountNormalized` and denormalized via `cumulativeDepositsMultiplierX18`, reducing this multiplier below its previous value means every depositor's real balance shrinks — they receive less than they deposited.

**Step 4 — The only replenishment path is manual and sequencer-gated.**

`depositInsurance()` is the sole mechanism to add funds to `insurance`. It is `onlyEndpoint`, meaning it requires the sequencer to submit a transaction:

```solidity
function depositInsurance(bytes calldata transaction)
    external
    virtual
    onlyEndpoint
{
    ...
    insurance += amount;
}
``` [5](#0-4) 

The only autonomous source of insurance growth is the per-liquidation fee fraction:

```solidity
insurance += v.liquidationFees;
``` [6](#0-5) 

`LIQUIDATION_FEE_FRACTION` is 50% of the penalty spread, which is itself only 0.5%–20% of the oracle price delta. In a rapid market crash, accumulated fees are negligible relative to the bad debt generated by simultaneous undercollateralized subaccounts. [7](#0-6) 

**Step 5 — The `lastLiquidationFees` reservation makes the threshold even tighter.**

Before checking whether insurance can cover remaining bad debt, the protocol subtracts `lastLiquidationFees` from the working insurance balance:

```solidity
v.insurance = insurance;
v.insurance -= lastLiquidationFees;
v.canLiquidateMore = (quoteBalance.amount + v.insurance) > 0;
``` [8](#0-7) 

This means socialization can be triggered even when `insurance > 0`, as long as `insurance ≤ lastLiquidationFees`. The effective usable insurance is always less than the stored value.

---

### Impact Explanation

When the insurance fund is exhausted during cascading liquidations:

1. `spotEngine.socializeSubaccount()` is called, permanently reducing `cumulativeDepositsMultiplierX18`.
2. Every depositor in the SpotEngine loses value proportionally — their normalized balance now maps to fewer real tokens.
3. This loss is irreversible; there is no mechanism to restore the multiplier.
4. The protocol continues to accept new deposits at the reduced multiplier, meaning new depositors are unaffected, but existing depositors have already suffered a permanent haircut.

**Severity: Medium.** The impact is direct, irreversible financial loss to all depositors. The trigger requires a market crash with simultaneous undercollateralized subaccounts, which is a realistic scenario for a perpetuals DEX.

---

### Likelihood Explanation

- Perpetuals DEXes are specifically exposed to correlated liquidation cascades during sharp market downturns.
- The insurance fund starts at zero and grows only through small liquidation fee fractions.
- A single large undercollateralized subaccount with bad debt exceeding the insurance balance is sufficient to trigger socialization.
- The sequencer cannot guarantee real-time `depositInsurance()` transactions during a market crash, especially if the sequencer itself is under load.

---

### Recommendation

1. **Autonomous fee routing:** Redirect a configurable fraction of all trading fees (currently accumulated in `FEES_ACCOUNT`) directly to `insurance` on each `claimSequencerFees()` call, without requiring a separate sequencer transaction.
2. **Insurance floor check:** Add a circuit-breaker that pauses new position openings when `insurance` falls below a configurable minimum threshold, reducing the rate of new bad-debt exposure.
3. **Transparent insurance accounting:** Emit an event whenever `socializeSubaccount()` is triggered so depositors can observe socialization events on-chain.

---

### Proof of Concept

1. Protocol launches; `insurance = 0`.
2. Over time, liquidation fees accumulate: `insurance = X` (small).
3. Market crash: 10 subaccounts simultaneously fall below maintenance health.
4. Sequencer submits `liquidateSubaccount` with `productId = type(uint32).max` (finalize) for each.
5. First few finalizations drain `insurance` to cover bad debt via `perpEngine.socializeSubaccount()` and `insuranceCover`.
6. For the N-th subaccount, `v.insurance = insurance - lastLiquidationFees ≤ 0`.
7. `spotEngine.socializeSubaccount(txn.liquidatee)` is called.
8. Inside `SpotEngine.socializeSubaccount()`, for each product with `balance.amount < 0`:
   ```solidity
   state.cumulativeDepositsMultiplierX18 =
       (totalDeposited + balance.amount).div(state.totalDepositsNormalized);
   ```
   This permanently reduces the multiplier.
9. All existing depositors now hold less value than they deposited. No on-chain mechanism exists to restore the multiplier or autonomously replenish `insurance`. [9](#0-8) [10](#0-9)

    function socializeSubaccount(bytes32 subaccount) external {
        require(msg.sender == address(_clearinghouse), ERR_UNAUTHORIZED);

        uint32[] memory _productIds = getProductIds();
        for (uint128 i = 0; i < _productIds.length; ++i) {
            uint32 productId = _productIds[i];

            State memory state = states[productId];
            Balance memory balance = balanceNormalizedToBalance(
                state,
                balances[productId][subaccount]
            );
            if (balance.amount < 0) {
                int128 totalDeposited = state.totalDepositsNormalized.mul(
                    state.cumulativeDepositsMultiplierX18
                );

                state.cumulativeDepositsMultiplierX18 = (totalDeposited +
                    balance.amount).div(state.totalDepositsNormalized);

                require(state.cumulativeDepositsMultiplierX18 > 0);

                state.totalBorrowsNormalized += balance.amount.div(
                    state.cumulativeBorrowsMultiplierX18
                );

                _setBalanceAndUpdateBitmap(
                    productId,
                    subaccount,
                    BalanceNormalized({amountNormalized: 0})
                );
                _setState(productId, state);
            }
        }
    }
