Audit Report

## Title
`SwapAllowlistExtension` gates the router address instead of the originating swapper, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` which is always `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` of `pool.swap()` is the router contract, not the end user. To permit any router-mediated swap for allowlisted users, the pool admin must allowlist the router address itself. Once the router is allowlisted, every unprivileged user can bypass the allowlist by routing through the router, rendering the extension's access control ineffective.

## Finding Description

**Root cause — `sender` is the direct caller of `pool.swap()`, not the originating user:**

In `MetricOmmPool.swap()`, `msg.sender` (the direct caller) is forwarded as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` re-encodes this value and dispatches it to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then gates on that `sender` value, keyed by `msg.sender` (the pool): [3](#0-2) 

**Exploit flow:**

1. Pool admin deploys a pool with `SwapAllowlistExtension` and allowlists only Alice (`allowedSwapper[pool][alice] = true`).
2. Alice wants to use `MetricOmmSimpleRouter`. The router calls `pool.swap()` directly: [4](#0-3) 
3. When the router calls `pool.swap()`, `msg.sender` inside the pool is the router address. The extension checks `allowedSwapper[pool][router]` — this reverts because the router is not allowlisted.
4. To allow Alice (or any allowlisted user) to use the router, the admin must allowlist the router address: `allowedSwapper[pool][router] = true`.
5. Once the router is allowlisted, Bob (not on the allowlist) calls `MetricOmmSimpleRouter.exactInputSingle()`. The router calls `pool.swap()` with `msg.sender = router`. The extension checks `allowedSwapper[pool][router]` — this passes. Bob successfully swaps on a pool he was supposed to be excluded from.

**Why existing guards fail:**

The `onlyPool` modifier in `BaseMetricExtension` only ensures the extension is called by a legitimate pool — it does not authenticate the originating user: [5](#0-4) 

There is no mechanism in the call chain to propagate the original `tx.origin` or the user who called the router. The extension has no way to distinguish Alice routing through the router from Bob routing through the same router.

## Impact Explanation
The `SwapAllowlistExtension` is entirely defeated for any pool that permits router-mediated swaps. Any unprivileged user can execute swaps on a pool that the admin intended to restrict to a specific set of addresses. This constitutes broken core pool functionality (the allowlist access control) and an admin-boundary break where an unprivileged path bypasses admin-set restrictions.

## Likelihood Explanation
The condition requires the pool admin to have allowlisted the router address — a necessary and expected admin action for any pool that wants to support router-mediated swaps for its allowlisted users. Once that condition is met (which is the normal operational state for any router-compatible restricted pool), the bypass is trivially repeatable by any unprivileged user with zero additional preconditions. No special privileges, flash loans, or oracle manipulation are required.

## Recommendation
The extension must authenticate the originating user rather than the direct caller. Two viable approaches:

1. **Pass `tx.origin` as an additional parameter** in the `beforeSwap` hook so the extension can check the end user. This requires a protocol-level change to the hook interface.
2. **Require swappers to call the pool directly** (document that the allowlist extension is incompatible with router intermediaries), and enforce this by checking that `sender == tx.origin` inside `beforeSwap`.
3. **Router-level allowlist forwarding**: have the router pass the originating user address in `extensionData`, and update `SwapAllowlistExtension` to decode and verify it — combined with a signature or trusted-forwarder pattern to prevent spoofing.

## Proof of Concept

```solidity
// Setup
pool.setExtension(swapAllowlistExtension);
swapAllowlistExtension.setAllowedToSwap(pool, alice, true);
// Admin must allowlist router to let alice use it:
swapAllowlistExtension.setAllowedToSwap(pool, address(router), true);

// Attack: Bob (not allowlisted) bypasses the allowlist via router
vm.prank(bob);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: bob,
    tokenIn: token0,
    zeroForOne: true,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// Bob's swap succeeds — allowlist bypassed
```
