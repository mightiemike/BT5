### Title
Attacker-Controlled `_cached_sha256_treehash` Trusted Without Integrity Verification in `Program.wrap()` and `sha256_treehash()` - (File: `wheel/python/clvm_rs/program.py`, `wheel/python/clvm_rs/tree_hash.py`)

---

### Summary

`Program.wrap()` blindly copies `_cached_sha256_treehash` from any caller-supplied `CLVMStorage`-compatible Python object without verifying that the cached value matches the object's actual content. `Program.tree_hash()` and `sha256_treehash()` both short-circuit on a non-`None` cached value, returning it directly. An attacker who can pass a Python object to `Program.to()` or `Program.wrap()` can inject an arbitrary 32-byte tree hash that will be returned as the authoritative puzzle hash for that program, corrupting every downstream operation that depends on it.

---

### Finding Description

**Root cause 1 — `Program.wrap()` propagates the cache blindly** [1](#0-0) 

`wrap()` calls `getattr(v, "_cached_sha256_treehash", None)` on the caller-supplied object and stores whatever it finds directly into `o._cached_sha256_treehash`. No check is made that the value is a valid 32-byte digest, let alone that it is the correct SHA-256 tree hash of `v`'s actual content.

**Root cause 2 — `Program.tree_hash()` returns the cached value unconditionally** [2](#0-1) 

If `_cached_sha256_treehash` is not `None`, it is returned immediately. The real hash is never computed.

**Root cause 3 — `sha256_treehash()` (the `Treehasher` method) also trusts the cache** [3](#0-2) 

`handle_obj` reads `_cached_sha256_treehash` via `getattr` and, if present, pushes it onto the hash stack without recomputing. This means even the "fresh" computation path is bypassed for any sub-tree that carries a pre-set cache attribute.

**Entry path through `Program.to()`** [4](#0-3) 

`to_clvm_object()` checks `is_clvm_storage(v)` (which only requires `atom` and `pair` attributes) and, if true, appends the object directly to `did_convert` without stripping or validating any cached fields. The object then flows into `Program.wrap()` with its attacker-controlled `_cached_sha256_treehash` intact.

**Downstream impact — `__eq__` uses `tree_hash()` for equality** [5](#0-4) 

`Program.__eq__` compares programs solely by their `tree_hash()` return values. A poisoned cache makes two structurally different programs compare as equal, or two identical programs compare as different.

---

### Impact Explanation

The tree hash of a CLVM program is its canonical identifier on the Chia blockchain — it is the puzzle hash of a coin. Any code that calls `Program.tree_hash()` to identify a coin, verify a puzzle, or check a spend condition receives the attacker-supplied value instead of the real hash. Concretely:

- **Coin identification is corrupted**: a wallet or full node that wraps an attacker-supplied `CLVMStorage` object and then calls `.tree_hash()` will derive the wrong puzzle hash, causing it to misidentify which coin is being spent.
- **Equality checks are broken**: `Program.__eq__` returns wrong results, so puzzle-matching logic (e.g., "is this the expected puzzle?") can be bypassed or falsely triggered.
- **`curry_hash()` is corrupted**: `curry_hash()` calls `self.tree_hash()` as its starting point; a poisoned hash propagates into all curried puzzle hash derivations. [6](#0-5) 

---

### Likelihood Explanation

The `CLVMStorage` protocol is a structural (duck-typed) protocol requiring only `atom` and `pair` attributes. [7](#0-6) 

Any Python object satisfying this minimal interface is accepted by `Program.to()` and `Program.wrap()`. The `_cached_sha256_treehash` attribute is documented as an optional optimization field, so any library or application that builds `CLVMStorage`-compatible objects (e.g., custom puzzle builders, wallet libraries, test harnesses) can inadvertently or deliberately set it. The attack requires no special privileges — only the ability to pass a Python object to `Program.to()` or `Program.wrap()`, which is the normal public API.

---

### Recommendation

1. **Strip the cache in `wrap()`**: do not propagate `_cached_sha256_treehash` from an external object. Only set it after computing the hash from the object's actual content, or leave it `None` and let `tree_hash()` compute it lazily.

```python
# In Program.wrap():
# Remove this line entirely:
# o._cached_sha256_treehash = getattr(v, "_cached_sha256_treehash", None)
o._cached_sha256_treehash = None  # always recompute from content
```

2. **Validate the cached value in `sha256_treehash()`**: if the cache is to be trusted for performance, at minimum assert it is a `bytes` object of length 32 before using it. Ideally, add a debug/assertion mode that recomputes and compares.

3. **Document the invariant**: make explicit in `CLVMStorage` that `_cached_sha256_treehash`, if set, must equal the SHA-256 tree hash of the object's content, and that `Program.wrap()` does not verify this.

### Proof of Concept

```python
from clvm_rs.program import Program

class MaliciousCLVMStorage:
    """Minimal CLVMStorage with a poisoned tree hash cache."""
    atom = b"real content"
    pair = None
    _cached_sha256_treehash = b"\xde\xad" * 16  # 32 bytes of attacker-chosen garbage

evil = MaliciousCLVMStorage()

# Program.to() calls to_clvm_object(), which sees is_clvm_storage(evil)==True
# and passes it directly to Program.wrap(), which copies _cached_sha256_treehash.
p = Program.to(evil)

# tree_hash() returns the attacker-controlled value, not the real hash
assert p.tree_hash() == b"\xde\xad" * 16  # poisoned hash returned

# The real hash of b"real content" as an atom is different:
real = Program.new_atom(b"real content")
assert real.tree_hash() != p.tree_hash()  # same content, different reported hash

# __eq__ is now broken: same bytes, different reported identity
assert p != real  # falsely reports inequality
```
