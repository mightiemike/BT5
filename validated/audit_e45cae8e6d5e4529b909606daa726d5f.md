### Title
Chain-Reorg Front-Running Allows Stale SPV Proof Acceptance Before `submit_blocks` Settles - (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` return a point-in-time boolean that reflects the current `mainchain_header_to_height` map. When `submit_blocks` is called with a fork of higher chainwork, `reorg_chain` atomically evicts previously-canonical blocks from that map. An attacker who observes a pending reorg-triggering `submit_blocks` call can front-run it with a `verify_transaction_inclusion` call, obtaining a `true` result for a transaction that will be orphaned the moment the reorg lands.

---

### Finding Description

`verify_transaction_inclusion` validates main-chain membership by reading `mainchain_header_to_height` at call time:

```rust
let target_block_height = self
    .mainchain_header_to_height
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));
``` [1](#0-0) 

This map is the sole source of truth for "is this block canonical?" and it is mutated atomically inside `reorg_chain` when a fork overtakes the main chain:

```rust
if current_header.chain_work > total_main_chain_chainwork {
    log!("Chain reorg");
    self.reorg_chain(current_header, last_main_chain_block_height);
}
``` [2](#0-1) 

Inside `reorg_chain`, every previously-canonical block at the reorged heights is removed from `mainchain_header_to_height` and replaced by the fork block:

```rust
let main_chain_block = self
    .mainchain_height_to_header
    .insert(&current_height, &current_block_hash);
self.mainchain_header_to_height
    .insert(&current_block_hash, &current_height);

if let Some(current_main_chain_blockhash) = main_chain_block {
    self.remove_block_header(&current_main_chain_blockhash);
}
``` [3](#0-2) 

`remove_block_header` deletes the entry from `mainchain_header_to_height` and `headers_pool`:

```rust
fn remove_block_header(&mut self, header_block_hash: &H256) {
    self.mainchain_header_to_height.remove(header_block_hash);
    self.headers_pool.remove(header_block_hash);
}
``` [4](#0-3) 

Because `verify_transaction_inclusion` is a read-only `&self` function and `submit_blocks` is a separate `&mut self` call, there is no atomicity guarantee between a consumer's verification call and the next relayer submission. The state change is sudden — one `submit_blocks` call flips the entire reorged segment — and it is predictable because the competing fork is visible on the Bitcoin P2P network before the relayer picks it up.

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after the coinbase check, so it carries the identical exposure:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [5](#0-4) 

---

### Impact Explanation

A consumer contract (e.g., a cross-chain bridge) that calls `verify_transaction_inclusion` and acts on the `true` result in the same logical flow can be made to release assets for a Bitcoin transaction that is subsequently orphaned. The attacker receives the bridged asset on NEAR while the corresponding Bitcoin UTXO is double-spent on the orphaned chain. The `confirmations` parameter does not eliminate the risk; it only raises the required reorg depth. With `confirmations = 1` (or `0`), even a 1-block natural reorg suffices.

**Impact: Medium** — funds locked in any consumer bridge contract that relies on a single `verify_transaction_inclusion` call are at risk whenever a reorg occurs.

---

### Likelihood Explanation

The relayer is an off-chain service with a polling interval. During that interval, a competing fork visible on the Bitcoin P2P network is not yet reflected in the NEAR contract state. An attacker who monitors Bitcoin nodes can detect the fork and submit the verification call before the relayer catches up. 1–2 block reorgs on Bitcoin mainnet occur multiple times per year; on testnets they are far more frequent. No privileged role is required — `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are both public, unpermissioned calls.

**Likelihood: Medium**

---

### Recommendation

1. **Minimum confirmation floor**: Enforce a protocol-level minimum `confirmations` value (e.g., 6 for Bitcoin mainnet) inside `verify_transaction_inclusion` so that a single-block or shallow reorg cannot invalidate a recently-accepted proof.
2. **Finality window parameter**: Expose a per-network `finality_depth` in the contract state and reject proof calls where `confirmations < finality_depth`.
3. **Consumer-side re-verification**: Document clearly that the returned boolean is valid only at the block height at which the call executes. Consumer contracts must not cache or act on the result across separate NEAR transactions without re-calling the verification function.

---

### Proof of Concept

```
State before attack:
  mainchain: ... → [B-2] → [B-1] → [B] (tip, height H)
  fork (not yet submitted to NEAR): ... → [F-1] → [F] (height H, higher chainwork)
  TX is in block [B]

Step 1 – Attacker monitors Bitcoin P2P network, sees fork [F-1]→[F] forming.

Step 2 – Attacker calls (NEAR tx A):
  verify_transaction_inclusion({ tx_id: TX, tx_block_blockhash: B_hash, confirmations: 1, ... })
  → mainchain_header_to_height.get(B_hash) = H  ✓
  → heaviest height H − target height H + 1 = 1 ≥ 1  ✓
  → Merkle proof valid  ✓
  → returns true

Step 3 – Consumer bridge contract releases bridged asset to attacker on NEAR.

Step 4 – Relayer submits fork headers [F-1], [F] via submit_blocks (NEAR tx B).
  → submit_block_header_inner detects chain_work([F]) > chain_work([B])
  → reorg_chain called
  → mainchain_header_to_height.remove(B_hash)
  → mainchain_header_to_height.insert(F_hash, H)

Step 5 – Block [B] is now orphaned. TX is no longer on the canonical chain.
  A second call to verify_transaction_inclusion for TX would now panic:
  "block does not belong to the current main chain"

Attacker has received bridged assets for an orphaned Bitcoin transaction.
```
