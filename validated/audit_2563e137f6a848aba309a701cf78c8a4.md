Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks LP recipient `owner` instead of actual depositor `sender`, allowing full allowlist bypass — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension` is documented as gating `addLiquidity` by depositor address, but its `beforeAddLiquidity` hook silently discards the `sender` parameter and validates `owner` (the LP position recipient) instead. Because `owner` is a free caller-controlled parameter, any address can bypass the deposit allowlist by supplying any already-allowlisted address as `owner`, depositing tokens into a restricted pool without being on the allowlist themselves.

## Finding Description
`MetricOmmPool.addLiquidity` calls the extension hook with both `msg.sender` (the actual token provider) and `owner` (the LP recipient, freely chosen by the caller):

```solidity
// metric-core/contracts/MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both parameters faithfully via `abi.encodeCall`. However, `DepositAllowlistExtension.beforeAddLiquidity` leaves the first parameter (`sender`) unnamed and checks only `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol lines 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

Since `owner` is a parameter the caller passes directly to `pool.addLiquidity(owner, ...)`, any unprivileged address can pass the guard by supplying any address that the pool admin has already allowlisted. The actual token depositor (`msg.sender`) is never validated.

For contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual swapper) and ignores `recipient`: [3](#0-2) 

The asymmetry confirms the deposit extension has the wrong parameter bound.

Additionally, `removeLiquidity` enforces `msg.sender == owner`, meaning the named `owner` (who did not initiate the deposit) can later call `removeLiquidity` to extract the tokens provided by the unauthorized depositor: [4](#0-3) 

## Impact Explanation
This is an admin-boundary break: an unprivileged path completely bypasses a configured pool guard. A pool admin who configures `DepositAllowlistExtension` to restrict which addresses may deposit liquidity receives no protection — any EOA or contract can deposit into the restricted pool. Unauthorized liquidity enters the pool, corrupting the pool admin's intended liquidity composition. The unauthorized depositor loses their tokens into the pool; the named `owner` receives LP shares they did not request and can extract them via `removeLiquidity`. This constitutes a direct loss of depositor funds and a broken core pool access-control invariant.

## Likelihood Explanation
The bypass requires no special privilege. Any EOA or contract can call `pool.addLiquidity` with a known allowed owner address. Allowed owner addresses are discoverable on-chain via `AllowedToDepositSet` events or by reading `allowedDepositor` storage. The attacker only needs to implement `IMetricOmmAddLiquidityCallback` to provide tokens in the callback. The attack is repeatable and requires no elevated role.

## Recommendation
Change `beforeAddLiquidity` to check `sender` (the actual depositor) instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Also update `setAllowedToDeposit`, `isAllowedToDeposit`, and the `allowedDepositor` mapping semantics documentation to confirm they refer to the `msg.sender` of `addLiquidity`, not the LP position recipient.

## Proof of Concept
```solidity
// Setup: pool has DepositAllowlistExtension configured.
// allowedDepositor[pool][alice] = true; bob is NOT on the allowlist.

// Bob (unauthorized) calls addLiquidity with owner = alice (an allowed address).
// The extension checks allowedDepositor[pool][alice] → true → passes.
// Bob provides tokens via the addLiquidity callback.
// LP shares are credited to alice.

pool.addLiquidity(
    alice,       // owner = allowed address (bypass key)
    0,           // salt
    deltas,
    callbackData,
    extensionData
);
// Result: bob (unauthorized) deposited tokens; alice received LP shares.
// alice (or anyone who controls alice) calls removeLiquidity to extract bob's tokens.
// The deposit allowlist provided zero protection.
```
