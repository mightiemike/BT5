Audit Report

## Title
`DepositAllowlistExtension` gates on `owner` instead of `sender`, allowing any unprivileged caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently drops the `sender` argument and checks the caller-supplied `owner` address against the allowlist. Because `owner` is a free parameter of `addLiquidity`, any non-allowlisted party can pass an allowlisted address as `owner` and the guard passes, while the actual depositor (who pays tokens via the callback) is never checked.

## Finding Description
`MetricOmmPool.addLiquidity` calls `_beforeAddLiquidity(msg.sender, owner, ...)` where `msg.sender` is the real depositor and `owner` is an arbitrary address supplied by the caller that receives the LP position. [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both as positional arguments to the extension via `abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))`. [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first `address` argument but leaves it unnamed and unused. It checks only `owner`: [3](#0-2) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly names and checks `sender`: [4](#0-3) 

The exploit path: attacker (Bob, not allowlisted) calls `pool.addLiquidity(owner = alice, ...)` where Alice is allowlisted. The guard evaluates `allowedDepositor[pool][alice]` → `true` → passes. Bob pays tokens via the modify-liquidity callback; Alice receives the LP shares. The allowlist is completely ineffective.

## Impact Explanation
The pool admin's access-control intent is entirely defeated. Any non-allowlisted address can deposit by nominating any allowlisted address as `owner`. This constitutes an admin-boundary break where an unprivileged path bypasses a pool admin-configured access control. Additionally, the allowlisted `owner` receives an unsolicited LP position they did not authorize, exposing them to pool-price risk on capital they never intended to commit.

## Likelihood Explanation
The bypass requires only a single direct call to `addLiquidity` with a known allowlisted address as `owner`. No special permissions, flash loans, or privileged access are needed. The `allowedDepositor` mapping is public, so any actor can observe the allowlist state and execute this immediately and repeatedly.

## Recommendation
Replace the unnamed first parameter with `sender` and check it instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```solidity
// Alice is allowlisted; Bob is not.
allowlistExtension.setAllowedToDeposit(pool, alice, true);

// Bob calls addLiquidity with owner = alice.
// The guard checks allowedDepositor[pool][alice] == true → passes.
// Bob pays tokens via callback; Alice receives LP shares she never requested.
vm.prank(bob);
pool.addLiquidity(
    alice,          // owner — allowlisted, guard passes
    0,              // salt
    deltas,
    callbackData,   // Bob's contract pays tokens here
    ""
);

// Bob has bypassed the allowlist entirely.
// Alice now holds an unsolicited LP position.
assertGt(positionShares(pool, alice, 0, binIdx), 0);
```
