### Title
Slippage Threshold Defined But Never Enforced in NLP Swap/Rebalance Operations — (`core/contracts/common/Errors.sol`, `core/contracts/EndpointTx.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

`ERR_SLIPPAGE_TOO_HIGH` is declared in the shared errors library but is never referenced in any production contract. The NLP pool rebalancing and swap operations in `EndpointTx.sol` and `Clearinghouse.sol` execute DEX-facing trades without enforcing any slippage bound, leaving those operations fully exposed to sandwich attacks and extreme price execution.

---

### Finding Description

`core/contracts/common/Errors.sol` defines the error constant:

```solidity
string constant ERR_SLIPPAGE_TOO_HIGH = "STH";
``` [1](#0-0) 

A repository-wide search confirms this constant is **only present in `Errors.sol`** and is never imported or referenced in any `require()` or revert path across `EndpointTx.sol`, `Clearinghouse.sol`, `OffchainExchange.sol`, or any other production contract. [2](#0-1) 

`EndpointTx.sol` and `Clearinghouse.sol` contain the NLP pool rebalancing and swap logic (144 and 114 matches respectively for swap/rebalance/NLP-related terms). These functions execute trades against external DEX integrations. Because no `require(slippage <= MAX_SLIPPAGE, ERR_SLIPPAGE_TOO_HIGH)` guard is ever placed before or during those calls, any caller-supplied or sequencer-supplied slippage parameter passes through unchecked. [3](#0-2) 

The existence of the error string strongly indicates the developers intended to add this check but never completed the implementation.

---

### Impact Explanation

Without a slippage ceiling, NLP rebalance swaps and LP mint/burn operations that touch external DEX liquidity can be executed at arbitrarily unfavorable prices. An attacker who can observe the pending transaction (e.g., via the public mempool or by front-running the sequencer submission) can sandwich the swap: push the price before the transaction, let the protocol trade at the manipulated price, and extract the difference. The direct asset delta is a loss of collateral value from the NLP pool, which socializes losses across all NLP depositors. [1](#0-0) 

---

### Likelihood Explanation

The entry path is the sequencer's `submitTransactions` call, which is processed by `EndpointTx` via `delegatecall` from `Endpoint`. NLP rebalance transactions are a normal, recurring protocol operation. Any party who can observe the transaction before finalization (public mempool on Ink Chain, or a compromised/colluding sequencer) can execute the sandwich. The likelihood is **medium**: it requires mempool visibility or sequencer-level access, but the operation itself is routine and the missing guard is a straightforward omission. [4](#0-3) 

---

### Recommendation

Enforce a maximum slippage threshold at the entry point of every NLP swap and rebalance function in `EndpointTx.sol` and `Clearinghouse.sol`. The already-defined `ERR_SLIPPAGE_TOO_HIGH` constant should be used in a `require` guard:

```solidity
require(slippage <= MAX_SLIPPAGE_BPS, ERR_SLIPPAGE_TOO_HIGH);
```

Place this check at the top of each function that accepts a caller-supplied or sequencer-supplied slippage value before any DEX interaction is initiated. Define `MAX_SLIPPAGE_BPS` as a protocol-level constant (e.g., 100 = 1%) with an owner-adjustable setter bounded by an absolute ceiling. [1](#0-0) 

---

### Proof of Concept

1. `ERR_SLIPPAGE_TOO_HIGH` is declared at `Errors.sol:90` but produces zero hits when searched across all `.sol` files outside of its own declaration — confirming it is never enforced. [1](#0-0) 
2. `EndpointTx.sol` processes `MatchOrders` and NLP rebalance transactions routed from `Endpoint.submitTransactions` without any slippage gate. [5](#0-4) 
3. A sequencer or front-runner submits an NLP rebalance transaction with an extreme slippage value (e.g., 0 or `type(uint128).max`). The transaction executes at the manipulated price. The NLP pool absorbs the loss, which is distributed across all depositors.
