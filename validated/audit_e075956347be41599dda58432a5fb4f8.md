Audit Report

## Title
Stale high-watermark after bin drain causes false `OracleStopLossTriggered` on re-populated bins — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`_afterSwapOracleStopLoss` skips `_checkAndUpdateWatermarks` for any bin with `totalShares == 0` via a `continue` on line 238. This leaves `highWatermarks[pool][binIdx].token0`, `.token1`, and `.lastDecayTs` frozen at their pre-drain values. When the bin is later re-populated at a different oracle price, the first swap computes a fresh metric against the stale watermark and can revert with `OracleStopLossTriggered` even though no value was leaked. With `decayPerSecondE8 == 0` (a valid configuration), the pool remains permanently stuck in that swap direction until the pool admin executes a timelocked watermark reset.

## Finding Description
In `_afterSwapOracleStopLoss`, the loop over traversed bins skips `_checkAndUpdateWatermarks` entirely for empty bins:

```solidity
for (uint256 i = 0; i < count; i++) {
    uint256 totalShares = PoolStateLibrary._decodeBinTotalShares(shares[i]);
    if (totalShares == 0) continue;   // watermark AND lastDecayTs frozen
    ...
    _checkAndUpdateWatermarks(...);
}
``` [1](#0-0) 

For a bin that previously had a watermark set (e.g., W₀ = 2000 units at oracle price P1=1), draining it to `totalShares=0` leaves `highWatermarks[pool][binIdx]` untouched. Any subsequent swap through the empty bin fires `continue`, keeping the stale watermark and frozen `lastDecayTs`.

When a new LP re-adds liquidity at oracle price P2=2 and a swap traverses the bin, `_metrics` computes:

- `metricT0 = t0·SCALE/shares + (t1·Q64/midP2)·SCALE/shares`

At P2=2 with equal token amounts, `metricT0` is ~1500 units, while the stale watermark W₀=2000 produces a floor of `2000 × 0.9 = 1800`. Since `1500 < 1800`, `_applyWatermark` sets `breached=true`: [2](#0-1) 

And `_checkAndUpdateWatermarks` reverts: [3](#0-2) 

The skip is not required to prevent division by zero: `_metrics` floors `shares` at `minShares` (≥1), so with `t0=t1=0` it safely returns `(0, 0)`: [4](#0-3) 

The existing `test_skipsEmptyBins` only verifies that a bin that was *always* empty keeps watermark=0; it never tests the drain-then-refill case: [5](#0-4) 

## Impact Explanation
Swaps through a re-populated bin are permanently reverted with `OracleStopLossTriggered` even though no value was leaked. With `decayPerSecondE8 == 0` (a valid configuration per `_validateDecay` which only rejects values `> E8`), the stale watermark never decays and the pool remains stuck in that swap direction until the pool admin executes a timelocked watermark reset. This constitutes broken core swap functionality — an unusable swap flow — meeting the contest's allowed impact gate. [6](#0-5) 

## Likelihood Explanation
The sequence — LP removes all liquidity from a bin, oracle price moves, new LP re-adds liquidity, swap traverses the bin — is a routine pool lifecycle event requiring no malicious actor and no privileged access. Any pool with a non-zero drawdown threshold and `decayPerSecondE8 == 0` is permanently exposed whenever a bin is fully drained and re-populated at a materially different price. The condition is repeatable and requires no special setup beyond normal LP and swap operations.

## Recommendation
Remove the `continue` and instead call `_checkAndUpdateWatermarks` with `(metricT0, metricT1) = (0, 0)` for empty bins. This resets the watermark to 0 and advances `lastDecayTs`, so the next re-population ratchets the watermark up from zero with no false breach. The revert guard inside `_checkAndUpdateWatermarks` must be conditioned on `totalShares > 0` as well (or the empty-bin path must write `hwmS.token0 = 0; hwmS.token1 = 0; hwmS.lastDecayTs = block.timestamp` directly without checking for breach), since `_applyWatermark(0, hwm>0, floor)` would set `breached=true`.

## Proof of Concept
```solidity
function test_staleWatermark_falseStopLoss_afterDrainAndRefill() public {
    uint128 priceP1 = uint128(Q64);     // mid = 1
    uint128 priceP2 = uint128(2 * Q64); // mid = 2

    // 1. Populate bin, set watermark at P1
    _storeBin(0, 1000, 1000, BIN_SHARES);
    _configure(100_000, 0);             // 10% drawdown, no decay
    _exposeStopLoss(0, 0, priceP1, false);

    // 2. Drain bin (LP removes all liquidity)
    _storeBin(0, 0, 0, 0);

    // 3. Swap traverses empty bin — watermark frozen at P1 value
    _exposeStopLoss(0, 0, priceP2, false);

    // 4. New LP re-adds liquidity at P2
    _storeBin(0, 1000, 1000, BIN_SHARES);

    // 5. Next swap — false stop-loss: metricT0 at P2 < floor of stale watermark
    vm.expectRevert();
    _exposeStopLoss(0, 0, priceP2, true);
}
```
