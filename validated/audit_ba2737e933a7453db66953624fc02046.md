Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Enabling Allowlist Bypass via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the immediate caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, `sender` resolves to the router's address rather than the actual end-user. A pool admin who allowlists the router to enable router-based trading inadvertently grants every non-allowlisted user the ability to bypass the per-user allowlist by routing through the router.

## Finding Description
`MetricOmmPool.swap()` invokes `_beforeSwap(msg.sender, recipient, ...)` at line 230, passing its own `msg.sender` as `sender`. When `MetricOmmSimpleRouter.exactInputSingle()` (line 72–80) calls `pool.swap(params.recipient, ...)`, the pool's `msg.sender` is the router contract. `ExtensionCalling._beforeSwap()` (line 149–176) forwards this `sender` verbatim to `IMetricOmmExtensions.beforeSwap`. `SwapAllowlistExtension.beforeSwap()` (line 37) then evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`. For a curated pool to support any router-based trading, the admin must call `setAllowedToSwap(pool, router, true)`. Once that entry is set, every user — regardless of individual allowlist status — can call the router and pass the check, because the extension only sees the router address. There is no configuration that simultaneously permits router-based trading and enforces per-user allowlist policy.

## Impact Explanation
Any non-allowlisted address can call `MetricOmmSimpleRouter` and execute swaps on a pool explicitly configured to exclude them. Unauthorized counterparties trade against LP capital in a KYC-gated or compliance-restricted pool, constituting a direct policy bypass with fund-impacting consequences: LP capital is exposed to counterparties the pool admin explicitly intended to exclude.

## Likelihood Explanation
No special privileges are required; `MetricOmmSimpleRouter` is a public, permissionless contract. The bypass requires a single direct call to the router. Pool admins have a natural incentive to allowlist the router (otherwise the router is entirely unusable for their pool), making the vulnerable configuration the expected production state.

## Recommendation
1. **Trusted-forwarder pattern**: Have `MetricOmmSimpleRouter` encode the original `msg.sender` into `extensionData`. `SwapAllowlistExtension` should decode and verify it when `sender` is a known trusted router, then apply the allowlist check against the decoded end-user.
2. **Explicit router registry**: `BaseMetricExtension` or the factory could maintain a registry of trusted routers; when `sender` is a trusted router, the extension reads the actual user from `extensionData`.
3. **Incompatibility guard**: At minimum, document and enforce that `SwapAllowlistExtension` is incompatible with `MetricOmmSimpleRouter` by reverting in `beforeSwap` if `sender` is a known router address.
