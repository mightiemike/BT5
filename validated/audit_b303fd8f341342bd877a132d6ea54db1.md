### Title
Unprotected `initialize()` on `PerpEngine` and `SpotEngine` Allows Arbitrary Caller to Seize Ownership — (File: `core/contracts/PerpEngine.sol`, `core/contracts/SpotEngine.sol`)

---

### Summary
`PerpEngine.initialize()` and `SpotEngine.initialize()` are declared `external` with no access control and no `initializer` modifier on the outer function. Neither contract defines a constructor calling `_disableInitializers()`. Any unprivileged caller who reaches the function before the legitimate deployer can supply attacker-controlled addresses for `_clearinghouse`, `_offchainExchange`, `_endpoint`, and `_admin`, seizing ownership of the engine and full control over which addresses are permitted to apply balance deltas.

---

### Finding Description

Both engine contracts expose a public initialization entry point with no caller restriction:

`PerpEngine.initialize()`: [1](#0-0) 

`SpotEngine.initialize()`: [2](#0-1) 

Both delegate to `BaseEngine._initialize()`, which carries the `initializer` modifier: [3](#0-2) 

The `initializer` modifier prevents a *second* call, but it places no restriction on *who* makes the first call. The outer `initialize()` functions are `external` with no `onlyOwner`, no deployer check, and no `initializer` modifier of their own.

Contrast this with `ContractOwner`, which explicitly guards its initializer: [4](#0-3) 

Neither `PerpEngine` nor `SpotEngine` (nor their state base contracts `PerpEngineState`, `SpotEngineState`) define a constructor calling `_disableInitializers()`. The inheritance chain is:

```
PerpEngine → PerpEngineState → BaseEngine → EndpointGated → OwnableUpgradeable
SpotEngine → SpotEngineState → BaseEngine → EndpointGated → OwnableUpgradeable
```

None of these define `constructor() { _disableInitializers(); }`, leaving the implementation contract itself initializable by anyone. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

Inside `_initialize`, the execution sequence is:

```
__Ownable_init()          → owner = msg.sender (attacker)
setEndpoint(_endpointAddr) → succeeds because owner is now attacker
transferOwnership(_admin)  → owner = attacker-supplied _admin
canApplyDeltas[_endpointAddr]       = true
canApplyDeltas[_clearinghouseAddr]  = true
canApplyDeltas[_offchainExchangeAddr] = true
``` [7](#0-6) 

An attacker who wins the initialization:

1. **Becomes owner** of `PerpEngine`/`SpotEngine`, gaining access to `onlyOwner` functions such as `addOrUpdateProduct` and `updateRisk`.
2. **Controls `canApplyDeltas`** — the mapping that gates every call to `updateBalance`, `settlePnl`, and `socializeSubaccount`. The attacker can whitelist their own contracts to freely write arbitrary balance deltas for any subaccount.
3. **Sets a malicious `_clearinghouse`** — downstream calls from `_addOrUpdateProduct` that invoke `_clearinghouse.registerProduct()` will hit attacker-controlled code.
4. **Poisons the implementation contract** — even if the proxy is later re-initialized correctly, a compromised implementation can be exploited via `delegatecall` from the proxy.

The `canApplyDeltas` corruption is the most severe consequence: it breaks the invariant that only the legitimate endpoint, clearinghouse, and offchain exchange may mutate engine balances, enabling direct theft of any subaccount's spot or perp balance. [8](#0-7) 

---

### Likelihood Explanation

The attack requires only a standard front-run of the deployment transaction on any EVM chain. No special privileges, leaked keys, or governance capture are needed. The window exists from the moment the proxy (or implementation) is deployed until the legitimate `initialize()` transaction is confirmed. On chains with a public mempool this is a straightforward MEV opportunity. Likelihood is **High** if deployment is not done atomically (deploy + initialize in a single transaction or via a factory).

---

### Recommendation

1. Add `constructor() { _disableInitializers(); }` to both `PerpEngine` and `SpotEngine` (or to `BaseEngine`) to prevent initialization of the implementation contract.
2. Add the `initializer` modifier directly to the external `initialize()` functions in `PerpEngine` and `SpotEngine`, not only to the internal `_initialize()`.
3. Consider adding a deployer/factory address check analogous to `ContractOwner`'s `require(_deployer == msg.sender, ...)` guard, or deploy and initialize atomically via a factory contract.

---

### Proof of Concept

```solidity
// Attacker deploys or observes the PerpEngine proxy address before initialization.
// Attacker calls:
PerpEngine(perpEngineProxy).initialize(
    address(attackerClearinghouse),   // _clearinghouse
    address(attackerOffchainExchange),// _offchainExchange
    address(0),                       // ignored third param
    address(attackerEndpoint),        // _endpoint
    address(attacker)                 // _admin → becomes owner
);

// Now attacker is owner. They whitelist themselves:
// canApplyDeltas[attacker] = true  (set during _initialize via _offchainExchangeAddr)

// Attacker calls updateBalance directly on any subaccount:
PerpEngine(perpEngineProxy).updateBalance(
    productId,
    victimSubaccount,
    int128(1_000_000e18),  // arbitrary credit
    0
);
// Victim subaccount now has an inflated perp balance with no corresponding collateral.
```

The same attack path applies identically to `SpotEngine.initialize()`. [9](#0-8) [10](#0-9)
