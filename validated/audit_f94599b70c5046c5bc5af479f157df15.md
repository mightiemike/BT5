### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Proof Validation — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is marked `#[deprecated]` but is still a live, publicly callable on-chain method. Any NEAR account can invoke it directly, skipping the coinbase Merkle proof validation that `verify_transaction_inclusion_v2` was introduced to enforce. This re-exposes the 64-byte transaction Merkle proof forgery vulnerability that v2 was designed to close.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to mitigate the 64-byte transaction Merkle proof forgery attack. It does so by requiring the caller to supply a coinbase Merkle proof that must match the block's Merkle root before delegating to the inner check: [1](#0-0) 

However, `verify_transaction_inclusion` is still a public `#[near]` method. Rust's `#[deprecated]` attribute is a **compile-time lint only** — it imposes zero runtime restriction. Any NEAR account (including a malicious downstream contract) can call `verify_transaction_inclusion` directly, skipping the coinbase proof step entirely: [2](#0-1) 

The function itself documents the risk in its own `# Warning` block:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [3](#0-2) 

With the coinbase check absent, an attacker can supply a crafted 64-byte payload whose `double_sha256` hash equals an internal Merkle tree node, obtain a `true` return value, and present that result to any downstream contract that trusts the light client's verification output.

The `verify_transaction_inclusion_v2` wrapper that enforces the coinbase check is: [4](#0-3) 

The coinbase proof requirement is the only guard against the forgery. Calling the deprecated function bypasses it entirely.

---

### Impact Explanation

A downstream NEAR contract that calls `verify_transaction_inclusion` — or that an attacker routes through it — can be made to accept a forged Bitcoin transaction inclusion proof. If that contract releases funds, mints tokens, or takes any irreversible action conditioned on a `true` result, the attacker achieves unauthorized execution with no valid on-chain Bitcoin transaction. The corrupted value is the proof result returned by the light client, which downstream consumers treat as authoritative.

---

### Likelihood Explanation

The entry path requires no privileged role: any NEAR account can call `verify_transaction_inclusion` directly. The 64-byte transaction forgery technique is publicly documented (https://www.bitmex.com/blog/64-Byte-Transactions) and well-understood. The only prerequisite is constructing a 64-byte payload whose hash collides with an internal Merkle node in a block already accepted by the light client — a feasible offline computation. Likelihood is **medium-high** given the low barrier to entry and the existence of public tooling for this attack class.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or gate it with an unconditional `env::panic_str` that makes it unreachable at runtime. Alternatively, inline the coinbase proof validation directly into `verify_transaction_inclusion` so both entry points enforce the same security invariant, eliminating the bypass path.

---

### Proof of Concept

1. Attacker identifies a block `B` accepted by the light client whose Merkle tree has at least one internal node `N`.
2. Attacker constructs a 64-byte payload `P` such that `double_sha256(P) == N` (the standard 64-byte forgery technique).
3. Attacker calls `verify_transaction_inclusion` with `tx_id = N`, `tx_block_blockhash = B`, a valid `tx_index` pointing to `N`'s position in the tree, and a Merkle proof path that reconstructs the Merkle root from `N`.
4. Inside `compute_root_from_merkle_proof`, the computation `compute_root_from_merkle_proof(N, index, proof) == header.merkle_root` evaluates to `true`. [5](#0-4) 

5. The function returns `true`. Any downstream contract that trusts this result treats the forged transaction as confirmed.

This is the exact attack that `verify_transaction_inclusion_v2` was designed to prevent, but the deprecated entry point leaves it fully reachable by any unprivileged NEAR caller.

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
