### Title
Host-Function Gas-Key Fee Calculation Uses Wire-Format Key Length Instead of Trie-Key Length for ML-DSA-65 Keys, Causing ~59× Fee Overcharge vs. Transaction Path — (`runtime/near-vm-runner/src/logic/logic.rs`, `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`)

---

### Summary

The `PostQuantumSignatures` feature (protocol version 85) stores ML-DSA-65 access keys in the trie as a 32-byte SHA3-256 hash (33 bytes including the type tag) rather than the full 1952-byte public key. The design intent, explicitly documented, is that every fee path uses `PublicKey::trie_id_len()` (33 for ML-DSA-65) rather than `PublicKey::len()` (1953 for ML-DSA-65). The transaction-level fee path correctly uses `trie_id_len()`. However, the host-function implementations of `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, and `promise_batch_action_add_gas_key_with_function_call` pass the raw WASM-supplied `public_key_len` (1953 bytes for ML-DSA-65) to the fee helpers instead of the decoded key's `trie_id_len()`. This produces a ~59× overcharge on the per-byte fee component for any contract that uses ML-DSA-65 keys via host functions, breaking the fee-parity invariant between the transaction path and the host-function path.

---

### Finding Description

**Design intent (documented):**

> The storage-stake calculation and the gas-key fee helpers use `PublicKey::trie_id_len()` rather than `PublicKey::len()`: `len()` reports the borsh-encoded length (33 / 65 / **1953** across the three `PublicKey` variants). `trie_id_len()` reports the on-trie length (33 / 65 / **33**). The two diverge only for `PublicKey::MLDSA65`. Every storage-stake and trie-byte-priced fee path was updated to call `trie_id_len()`. [1](#0-0) 

**Transaction path — correct:**

`total_send_fees` for `TransferToGasKey` uses `action.public_key.trie_id_len()`: [2](#0-1) 

`exec_fee` for `TransferToGasKey` and `WithdrawFromGasKey` also uses `action.public_key.trie_id_len()`: [3](#0-2) 

`permission_exec_fees` for gas-key `AddKey` uses `public_key.trie_id_len()`: [4](#0-3) 

**Host-function path — wrong:**

`promise_batch_action_transfer_to_gas_key` in `logic.rs` passes the raw `public_key_len as usize` to both `gas_key_transfer_send_fee` and `gas_key_transfer_exec_fee`: [5](#0-4) 

The identical mistake is present in the Wasmtime runner: [6](#0-5) 

`promise_batch_action_add_gas_key_with_full_access` passes `public_key_len as usize` to `gas_key_add_key_exec_fee`: [7](#0-6) 

`promise_batch_action_add_gas_key_with_function_call` does the same: [8](#0-7) 

**The two representations for ML-DSA-65:**

```
PublicKey::len()        = 1 + 1952 = 1953  (borsh wire format)
PublicKey::trie_id_len()= 1 + 32   =   33  (on-trie SHA3-256 hash)
``` [9](#0-8) 

**Fee helper semantics:**

`gas_key_transfer_exec_fee` is explicitly documented as "Based on the access key trie key length + estimated value length (what the receiver needs to read/write in the trie)." Passing 1953 instead of 33 inflates the trie-key-length argument by 1920 bytes. [10](#0-9) 

**Fee parity test gap:**

The existing parity test `test_gas_key_fee_parity` only exercises `KeyType::ED25519` keys (for which `len()` == `trie_id_len()` == 33), so the mismatch is invisible to the test suite: [11](#0-10) 

---

### Impact Explanation

For every host-function call that involves an ML-DSA-65 gas key:

| Fee component | Transaction path | Host-function path | Ratio |
|---|---|---|---|
| Per-byte send fee | 33 × `gas_key_byte` | 1953 × `gas_key_byte` | ~59× |
| Per-byte exec fee | 33 × `gas_key_byte` | 1953 × `gas_key_byte` | ~59× |

The overcharge is 1920 × `gas_key_byte` gas units per action. Contracts that budget gas based on the documented trie-key-length semantics will run out of gas and fail. The fee-parity invariant — that the same action costs the same gas whether submitted as a transaction or dispatched via a host function — is broken for ML-DSA-65 keys across all three affected host functions.

---

### Likelihood Explanation

ML-DSA-65 keys are gated behind `ProtocolFeature::PostQuantumSignatures` (stabilized at protocol version 85). Any contract that calls `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, or `promise_batch_action_add_gas_key_with_function_call` with an ML-DSA-65 public key will trigger the overcharge. The feature is live on mainnet (stable version 86 ≥ 85), so the affected code paths are reachable by any unprivileged user who deploys or calls such a contract.

---

### Recommendation

In each of the three host-function implementations, decode the public key before computing fees and use `decoded_key.trie_id_len()` in place of `public_key_len as usize`:

```rust
// Before fee calculation, decode the key:
let decoded_key = public_key.decode()?;
let key_trie_len = decoded_key.trie_id_len();

// Then pass key_trie_len to both fee helpers:
let send = gas_key_transfer_send_fee(&self.fees_config, sir, key_trie_len);
let exec = gas_key_transfer_exec_fee(&self.fees_config, receiver_id.len(), key_trie_len);
```

Apply the same fix to `promise_batch_action_add_gas_key_with_full_access` and `promise_batch_action_add_gas_key_with_function_call`, and add ML-DSA-65 coverage to `test_gas_key_fee_parity`.

---

### Proof of Concept

1. Enable `PostQuantumSignatures` (protocol version ≥ 85, already stable on mainnet).
2. Generate an ML-DSA-65 key pair; `public_key.len()` = 1953, `public_key.trie_id_len()` = 33.
3. Deploy a contract that calls `promise_batch_action_transfer_to_gas_key` with the ML-DSA-65 public key.
4. Submit the identical `TransferToGasKey` action as a signed transaction using the same ML-DSA-65 key.
5. Compare `gas_burnt` on the two execution receipts. The host-function receipt will show a per-byte fee component ~59× larger than the transaction receipt, confirming the divergence between `public_key_len` (1953) and `trie_id_len()` (33).

/// Exec fee for TransferToGasKey / WithdrawFromGasKey actions.
/// Based on the access key trie key length + estimated value length (what the
/// receiver needs to read/write in the trie).
pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}
