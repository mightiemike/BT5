### Title
Dead `DISABLE_OP` Size Guard in `op_div` / `op_divmod` / `op_mod` Due to Inverted Limit Bounds — (File: `src/more_ops.rs`)

---

### Summary

In `op_div`, `op_divmod`, and `op_mod` (and their malachite variants), the `DISABLE_OP`-gated size check uses the bound `a0_len > 2048`, while the unconditional check uses `a0_len > 256`. Because `2048 > 256`, the `DISABLE_OP` branch is permanently dead code. The flag that is supposed to impose a stricter mempool-mode limit on the dividend size has no effect, and the unconditional check enforces a limit that is too strict for consensus mode.

---

### Finding Description

In `op_div` at `src/more_ops.rs` lines 665–669:

```rust
if flags.contains(ClvmFlags::DISABLE_OP) && a0_len > 2048 {
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
if a0_len > 256 || a1_len > 1024 {
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
```

The identical pattern appears in `op_divmod` (lines 713–717), `op_mod` (lines 769–773), and all three malachite variants (`op_div_malachite` lines 690–694, `op_divmod_malachite` lines 742–746, `op_mod_malachite` lines 794–798).

`DISABLE_OP` is part of `MEMPOOL_MODE` and is used elsewhere to impose stricter restrictions — for opcode 60 (`modpow`) it completely disables the operator:

```rust
60 => {
    if flags.contains(ClvmFlags::DISABLE_OP) {
        return Err(EvalErr::Unimplemented(o))?;
    }
    op_modpow
}
```

For `div`/`divmod`/`mod`, the intent is clearly to restrict the dividend size to 256 bytes in mempool mode while allowing up to 2048 bytes in consensus mode. The correct structure should be:

```rust
// mempool-mode stricter limit
if flags.contains(ClvmFlags::DISABLE_OP) && a0_len > 256 {
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
// consensus-mode limit
if a0_len > 2048 || a1_len > 1024 {
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
```

Instead, the bounds `256` and `2048` are swapped between the two checks. Because the unconditional check (`> 256`) is always stricter than the `DISABLE_OP` check (`> 2048`), the `DISABLE_OP` branch never executes.

---

### Impact Explanation

Two concrete broken invariants result from the swapped bounds:

1. **Consensus-mode over-rejection**: Any attacker-supplied CLVM program invoking `div`, `divmod`, or `mod` with a dividend atom of length 257–2048 bytes is rejected by this node in consensus mode, even though the correct consensus limit is 2048. A node running this code will reject blocks containing such programs that every correctly-implemented node accepts, causing a **chain split / consensus divergence**.

2. **Mempool-mode under-rejection**: Programs with dividend atoms of 257–256 bytes (i.e., the range that should be rejected in mempool mode but accepted in consensus mode) are not additionally restricted in mempool mode. The `DISABLE_OP` flag has zero effect on these three operators, so the mempool/consensus boundary is erased for `div`/`divmod`/`mod`.

The corrupted result is the `EvalErr::InvalidOpArg` error (or its absence) returned for attacker-controlled atom sizes in the range 257–2048 bytes.

---

### Likelihood Explanation

The entry path is fully attacker-controlled: any CLVM program submitted to the network can include a `div`, `divmod`, or `mod` call with a dividend atom of 257–2048 bytes. No special privileges, social engineering, or local configuration are required. The bug is triggered deterministically by the atom size alone, making it trivially reproducible.

---

### Recommendation

Swap the size bounds between the two checks in all six affected functions (`op_div`, `op_div_malachite`, `op_divmod`, `op_divmod_malachite`, `op_mod`, `op_mod_malachite`):

```diff
- if flags.contains(ClvmFlags::DISABLE_OP) && a0_len > 2048 {
+ if flags.contains(ClvmFlags::DISABLE_OP) && a0_len > 256 {
      return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
  }
- if a0_len > 256 || a1_len > 1024 {
+ if a0_len > 2048 || a1_len > 1024 {
      return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
  }
```

Apply the same fix to `op_divmod`, `op_mod`, and all three malachite variants.

---

### Proof of Concept

The dead-code nature of the `DISABLE_OP` branch is provable by inspection:

- `DISABLE_OP` branch fires only when `a0_len > 2048`.
- The unconditional branch fires when `a0_len > 256`.
- For any `a0_len > 2048`, the unconditional branch fires first (or simultaneously), so the `DISABLE_OP` branch is unreachable.
- For `a0_len` in `[257, 2048]`, the unconditional branch fires and the `DISABLE_OP` branch does not — regardless of whether `DISABLE_OP` is set.

Concrete trigger (consensus mode, no `DISABLE_OP`):

```
(/ <257-byte-atom> <1-byte-atom>)
```

This is rejected by the current code (`a0_len = 257 > 256`) but should succeed under the correct consensus limit of 2048. A node running this code will diverge from the canonical chain on any block containing such a program. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

pub fn op_div(a: &mut Allocator, input: NodePtr, max_cost: Cost, flags: ClvmFlags) -> Response {
    if flags.contains(ClvmFlags::MALACHITE) {
        return op_div_malachite(a, input, max_cost, flags);
    }
    let [v0, v1] = get_args::<2>(a, input, "/")?;
    let (a0, a0_len) = int_atom(a, v0, "/")?;
    let (a1, a1_len) = int_atom(a, v1, "/")?;
    if flags.contains(ClvmFlags::DISABLE_OP) && a0_len > 2048 {
        return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
    }
    if a0_len > 256 || a1_len > 1024 {
        return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
    }
    let cost = DIV_BASE_COST + ((a0_len + a1_len) as Cost) * DIV_COST_PER_BYTE;
    check_cost(cost, max_cost)?;
    if a1.sign() == Sign::NoSign {
        return Err(EvalErr::DivisionByZero(input));
    }
    let q = a0.div_floor(&a1);
    let q = a.new_number(q)?;
    Ok(malloc_cost(a, cost, q))
}
