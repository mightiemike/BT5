### Title
Interest Dilution via `executeSlowModeTransaction()` Changing `totalDepositsNormalized` Without Prior `updateStates` Call — (File: `core/contracts/SpotEngineState.sol`)

---

### Summary

Any unprivileged user can call `executeSlowModeTransaction()` on `Endpoint.sol` after the 3-day slow mode timeout to process a queued `DepositCollateral` transaction. This increases `totalDepositsNormalized` in `SpotEngine` without first calling `updateStates()` to snapshot accrued interest. When the sequencer later calls `updateStates()`, it uses the inflated `totalDepositsNormalized`, computes a lower utilization ratio, and applies a lower deposit rate multiplier — causing all existing depositors to receive less interest than they earned for the period before the new deposit arrived.

---

### Finding Description

**Vulnerability class:** Cross-contract desynchronization — a state-update function (`updateStates`) must be called before any balance change that alters `totalDepositsNormalized`, but the public `executeSlowModeTransaction()` path bypasses this ordering requirement.

**Root cause chain:**

`SpotEngineState._updateState()` computes the deposit interest rate multiplier using the current `totalDepositsNormalized` at the moment it is called:

```
utilizationRatioX18 = totalBorrows / (totalDepositsNormalized * cumulativeDepositsMultiplierX18)
depositRateMultiplierX18 = utilizationRatio * (borrowRateMultiplier - 1) + 1
cumulativeDepositsMultiplierX18 *= depositRateMultiplierX18
``` [1](#0-0) [2](#0-1) 

This is called via `updateStates(uint128 dt)`, which is `onlyEndpoint` and invoked by the sequencer: [3](#0-2) 

The sequencer normally ensures `updateStates` is called before any deposit is processed in its ordered transaction batches. However, `Endpoint.executeSlowModeTransaction()` is a **public, permissionless** function callable by anyone after the 3-day timeout: [4](#0-3) 

It calls `_executeSlowModeTransaction` → `processSlowModeTransaction` → `Clearinghouse.depositCollateral` → `spotEngine.updateBalance` → `_updateBalanceNormalized`, which directly mutates `totalDepositsNormalized`: [5](#0-4) [6](#0-5) 

**No call to `updateStates` occurs anywhere in this path.** The interest accrued since the last `updateStates` call is never snapshotted before `totalDepositsNormalized` grows.

**Attack scenario:**

1. Time `T0`: `updateStates` is last called. `totalDepositsNormalized = D`, `totalBorrowsNormalized = B`. Utilization = `B/D`.
2. Time `T0 + 3 days`: A user's queued `DepositCollateral` slow mode transaction becomes executable.
3. The user (or anyone) calls `executeSlowModeTransaction()`. `totalDepositsNormalized` becomes `D + d` (new deposit `d`). No interest is snapshotted.
4. Time `T0 + 3 days + ε`: Sequencer calls `updateStates(dt = 3 days + ε)`. It now computes utilization as `B / (D + d)` — lower than the correct `B / D` — and applies a lower `depositRateMultiplierX18` to the entire `dt` period.
5. All depositors who were present since `T0` receive interest calculated as if the new deposit `d` had been present for the entire 3-day period, not just from the moment it was processed.

---

### Impact Explanation

Existing depositors lose a portion of their accrued interest proportional to `d / (D + d)` and the length of the unsnapshotted period. For large deposits relative to the pool, or long periods between `updateStates` calls (e.g., during sequencer downtime), the loss can be material. The corrupted state variable is `cumulativeDepositsMultiplierX18`, which permanently under-accrues for all existing normalized balances. [7](#0-6) 

---

### Likelihood Explanation

The slow mode timeout is hardcoded at 3 days (`SLOW_MODE_TX_DELAY`): [8](#0-7) 

After this window, any address can call `executeSlowModeTransaction()`. This is a realistic scenario during sequencer downtime, censorship resistance usage, or simply a user self-executing their own deposit. No special privileges are required.

---

### Recommendation

Before processing a `DepositCollateral` (or `WithdrawCollateral`) slow mode transaction in `_executeSlowModeTransaction`, the protocol should ensure `updateStates` has been called with the current time delta. One approach is to call `spotEngine.updateStates(dt)` inside `executeSlowModeTransaction()` before dispatching the queued transaction, analogous to the Beedle recommendation of calling `Staking.update()` inside `Fees.sellProfits()`:

```solidity
function executeSlowModeTransaction() external {
    // Accrue interest before any balance change
    uint128 dt = uint128(block.timestamp) - times.spotTime;
    spotEngine.updateStates(dt);
    // ... then process the slow mode tx
    SlowModeConfig memory _slowModeConfig = slowModeConfig;
    _executeSlowModeTransaction(_slowModeConfig, false);
    nSubmissions += 1;
    slowModeConfig = _slowModeConfig;
}
```

Alternatively, enforce that `executeSlowModeTransaction` can only process non-balance-mutating transaction types, and require the sequencer to always precede deposits with an `updateStates` call.

---

### Proof of Concept

1. Deploy Nado with `totalDepositsNormalized = 1000e18`, `totalBorrowsNormalized = 800e18` (80% utilization). `updateStates` was last called at `T0`.
2. At `T0`, Alice has `balances[QUOTE][alice].amountNormalized = 500e18 / cumulativeDepositsMultiplierX18`.
3. Bob queues a `DepositCollateral` of `1000e18` at `T0 + 1 day` via `Endpoint.depositCollateralWithReferral`.
4. At `T0 + 3 days + 1`, Bob calls `executeSlowModeTransaction()`. `totalDepositsNormalized` becomes `2000e18`. No interest snapshotted.
5. Sequencer calls `updateStates(dt = 3 days + 1s)`. Utilization is now `800/2000 = 40%` instead of `800/1000 = 80%`. The deposit rate multiplier applied to the full 3-day period is computed at 40% utilization.
6. Alice's `cumulativeDepositsMultiplierX18` grows far less than it should have for the 3 days she was the only depositor at 80% utilization. Alice permanently loses the interest delta between 80% and 40% utilization applied over 3 days. [9](#0-8) [10](#0-9) [11](#0-10)

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }

    function executeSlowModeTransaction() external {
        SlowModeConfig memory _slowModeConfig = slowModeConfig;
        _executeSlowModeTransaction(_slowModeConfig, false);
        nSubmissions += 1;
        slowModeConfig = _slowModeConfig;
    }
