### Title
`u64→i64` Truncation in `get_last_blocks_info` Produces Near-Zero `min_height_to_fetch`, Causing a ~2⁶⁴-Iteration Loop That Permanently Blocks the `ClientActor` — (File: `chain/client/src/debug.rs`)

---

### Summary

When `enable_debug_rpc = true`, any unprivileged HTTP client can send a single GET request to `/debug/api/block_status?starting_height=18446744073709551615` (i.e., `u64::MAX`). Inside `get_last_blocks_info`, the expression `height_to_fetch as i64` silently wraps `u64::MAX` to `-1`. The subsequent `max(-1 - 1000, genesis_height_as_i64) as u64` evaluates to `0`, so `min_height_to_fetch = 0` instead of a value near `u64::MAX`. The decrement loop then runs from `u64::MAX` down to `0` — approximately 1.8 × 10¹⁹ iterations — permanently blocking the `ClientActor` from processing any further messages (block production, consensus, peer gossip).

---

### Finding Description

**Root cause — `u64 as i64` truncation in `get_last_blocks_info`:** [1](#0-0) 

```rust
let min_height_to_fetch = max(
    height_to_fetch as i64 - num_blocks as i64,   // ← lossy cast
    chain_store.get_genesis_height() as i64,
) as u64;
```

When `starting_height = u64::MAX = 18446744073709551615`:

| Step | Value |
|---|---|
| `height_to_fetch` | `18446744073709551615` (`u64::MAX`) |
| `height_to_fetch as i64` | `-1` (wraps) |
| `-1 - 1000` | `-1001` (no overflow) |
| `max(-1001, 0)` | `0` |
| `0 as u64` | `0` |
| **`min_height_to_fetch`** | **`0`** (should be ≈ `u64::MAX`) |

The loop that follows: [2](#0-1) 

```rust
while height_to_fetch > min_height_to_fetch || !block_hashes_to_force_fetch.is_empty() {
    let block_hashes = if height_to_fetch > min_height_to_fetch {
        ...
        height_to_fetch -= 1;   // decrements from u64::MAX → 0
        block_hashes
    } else { ... }
}
```

With `height_to_fetch = u64::MAX` and `min_height_to_fetch = 0`, the loop runs `u64::MAX` ≈ 1.8 × 10¹⁹ times. Each iteration calls `get_block_hashes_to_fetch` (a DB lookup). At even 1 µs per iteration, this takes ~585,000 years.

**The same cast defect exists in `find_first_height_to_fetch`** for non-`All` modes: [3](#0-2) 

```rust
let min_height_to_search = max(
    height_to_fetch as i64 - DEBUG_MAX_BLOCKS_TO_SEARCH as i64,
    chain_store.get_genesis_height() as i64,
) as u64;
while height_to_fetch > min_height_to_search { ... }
```

**The HTTP entry point** is registered unconditionally when `enable_debug_rpc = true`: [4](#0-3) 

The `DebugBlockStatusQuery` struct accepts `starting_height: Option<u64>` with no upper-bound validation: [5](#0-4) 

**`enable_debug_rpc` defaults to `false`** but is explicitly enabled in production-like deployments (benchmarks, mocknet, RPC nodes): [6](#0-5) [7](#0-6) 

---

### Impact Explanation

`get_last_blocks_info` is called synchronously inside the `ClientActor` message handler: [8](#0-7) 

The `ClientActor` is a single-threaded actor. While it is blocked in the near-infinite loop, it cannot process any other messages: block production, Doomslug approvals, peer gossip, sync, or any other consensus activity. The node becomes permanently unresponsive. Other nodes time out their requests to it and eventually mark it as lost, triggering removal from the active validator set.

---

### Likelihood Explanation

Any HTTP client that can reach the node's RPC port (default `0.0.0.0:3030`) can trigger this with a single unauthenticated GET request. No credentials, no stake, no validator key required. The only precondition is `enable_debug_rpc = true`, which is the standard configuration for RPC nodes, archive nodes, and any node running the debug UI. The nearcore localnet setup enables it by default for all nodes.

---

### Recommendation

Replace the lossy `as i64` / `as u64` round-trip with arithmetic that stays in `u64`:

```rust
// In get_last_blocks_info:
let min_height_to_fetch = height_to_fetch
    .saturating_sub(num_blocks)
    .max(chain_store.get_genesis_height());

// In find_first_height_to_fetch:
let min_height_to_search = height_to_fetch
    .saturating_sub(DEBUG_MAX_BLOCKS_TO_SEARCH)
    .max(chain_store.get_genesis_height());
```

`saturating_sub` correctly handles the case where `height_to_fetch < num_blocks` (returns 0) and the case where `height_to_fetch` is near `u64::MAX` (returns a value near `u64::MAX - num_blocks`), eliminating the wrap-around entirely.

Additionally, add an explicit upper-bound check on `starting_height` before entering the computation, rejecting values above the current `header_head.height`.

### Proof of Concept

**Prerequisites:** A nearcore node with `enable_debug_rpc: true` in `config.json`.

**Step 1 — Send the malicious request:**
```
GET http://<NODE_IP>:3030/debug/api/block_status?starting_height=18446744073709551615&mode=all
```

**Step 2 — Observe the node freeze.** The `ClientActor` enters the loop. All subsequent RPC calls (`/status`, `/health`, block production) hang indefinitely.

**Step 3 — Verify.** From another node's logs, observe timeout errors contacting the victim:
```
[ERROR] main - Network timeout on binary/compare_cert: request timed out
```

**Exact divergent value:**
- Expected `min_height_to_fetch`: ≈ `18446744073709550615` (`u64::MAX - 1000`)
- Actual `min_height_to_fetch`: `0`
- Divergence: `18446744073709550615` — the loop runs this many extra iterations [9](#0-8)
