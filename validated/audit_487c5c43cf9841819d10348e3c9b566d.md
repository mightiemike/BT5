### Title
Fork-Block Difficulty Validated Against Mainchain Ancestor Instead of Fork Ancestor, Enabling Under-Difficulty Fork Acceptance - (File: `contract/src/dogecoin.rs`, `contract/src/litecoin.rs`, `contract/src/bitcoin.rs`)

---

### Summary

The difficulty-adjustment calculation for fork blocks uses `get_header_by_height`, which exclusively reads from the **mainchain** height index. When a fork diverges before the retarget boundary block, the contract computes the expected `bits` for the fork block using the mainchain ancestor's timestamp rather than the fork's own ancestor's timestamp. This desynchronization between the mainchain height index and the fork's actual ancestor chain allows an attacker to submit fork blocks whose difficulty is validated against an incorrect (potentially easier) target, enabling chain-work accumulation with less mining effort than the protocol requires.

---

### Finding Description

`get_header_by_height` in `contract/src/lib.rs` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // ← mainchain-only index
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

This function is called during difficulty validation for every chain variant. In `dogecoin.rs`:

```rust

// without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [2](#0-1) 

The same pattern appears in `litecoin.rs`: [3](#0-2) 

And in `bitcoin.rs`: [4](#0-3) 

The desynchronization: when a fork diverges at or before `height_first`, the block stored in `mainchain_height_to_header[height_first]` is the **mainchain** block at that height, not the fork's ancestor. The two blocks can have different `time` fields. The difficulty formula uses `prev_block_time - first_block_time` as `actual_time_taken`: [5](#0-4) 

If the mainchain block at `height_first` has an **earlier** timestamp than the fork's true ancestor at that height, `actual_time_taken` is inflated, the computed target is easier (higher), and the `expected_bits` returned to `check_pow` corresponds to a lower difficulty than the protocol actually requires for that fork. [6](#0-5) 

The check `expected_bits == block_header.bits` then passes for a fork block whose `bits` encodes a lower difficulty than the true retarget value.

The codebase itself flags this with a `TODO` comment at the exact line of the bug, confirming the developers recognized the ambiguity but left it unresolved.

---

### Impact Explanation

An attacker who submits a fork chain diverging before the retarget boundary can have each retarget block's difficulty validated against the mainchain's boundary-block timestamp rather than the fork's own boundary-block timestamp. If the mainchain boundary block has an earlier timestamp (a common scenario when the fork was crafted to have later timestamps at that height), the contract accepts fork blocks with a lower `bits` value (easier PoW target) than the protocol requires. Each such block contributes `work_from_bits(bits)` to `chain_work`. With an artificially easy target, the attacker accumulates chain work faster than honest miners, eventually triggering an automatic reorg (`chain_work > total_main_chain_chainwork`) and displacing the legitimate mainchain. Downstream consumers of `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` then operate against a fraudulent canonical chain.

The impact is most severe on **Dogecoin** (post-block 145,000), where `difficulty_adjustment_interval = 1` means **every** fork block's difficulty is computed using `get_header_by_height`, so the desynchronization affects every single block in a fork rather than only retarget boundaries. [7](#0-6) 

---

### Likelihood Explanation

The entry point is the public `submit_blocks()` function, callable by any NEAR account (subject to the trusted-relayer stake check, but the stake requirement does not prevent a funded attacker). The attacker needs to:

1. Submit a fork that diverges before `height_first` — a normal fork-submission flow already supported by the contract.
2. Craft fork ancestor blocks with timestamps later than the corresponding mainchain blocks at `height_first` — timestamps in block headers are attacker-controlled inputs.
3. Mine the retarget block against the easier target — requires real PoW but at reduced cost proportional to the difficulty reduction.

No privileged role, leaked key, or social engineering is required. The attack is fully executable by an unprivileged proof submitter with sufficient mining resources.

---

### Recommendation

Replace `get_header_by_height` calls inside difficulty-adjustment functions with an ancestor-walking loop that follows `prev_block_hash` links through `headers_pool` starting from the fork's `prev_block_header`. This ensures the timestamp used for retargeting belongs to the fork's actual ancestor at `height_first`, not the mainchain block at that height. The existing `get_prev_header` traversal pattern used in `zcash.rs` (which walks `prev_block_hash` links) is the correct model to follow. [8](#0-7)

### Proof of Concept

**Setup (Dogecoin, post-block 145,000):**

- Mainchain tip at height `H` (block `M_H`). Mainchain block at height `H-2` (`M_{H-2}`) has `time = T_main`.
- Attacker submits a fork starting at height `H-3`. The fork's block at height `H-2` (`F_{H-2}`) has `time = T_main + Δ` where `Δ > 0` (a later timestamp, valid under MTP rules).

**Difficulty calculation for fork block at height `H` (retarget, `difficulty_adjustment_interval=1`):**

- `prev_block_header` = fork block at `H-1`, `height_first = H-2`
- `get_header_by_height(H-2)` returns **mainchain** block `M_{H-2}` with `time = T_main`
- `modulated_timespan = time(F_{H-1}) - T_main` — inflated by `Δ` compared to the correct value `time(F_{H-1}) - (T_main + Δ)`
- Inflated `modulated_timespan` → easier `new_target` → lower `expected_bits`

**Attacker sets `bits` in their fork block at `H` to match this easier `expected_bits`, mines the block against the easier target, and submits it. `check_pow` passes. The fork block is stored with `chain_work` computed from the easier `bits`. Repeated across many blocks, the fork's cumulative chain work exceeds the mainchain's, triggering `reorg_chain` and replacing the legitimate mainchain.** [9](#0-8)
