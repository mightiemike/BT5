### Title
Negative `positionPnl` in `settlePnl` inflates `availableSettle` and drains user spot quote — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.settlePnl` applies settlement unconditionally regardless of the sign of `canSettle`. When a subaccount has a negative `positionPnl`, `getSettlementState` returns a negative `canSettle` value. The subsequent arithmetic then **increases** `state.availableSettle` (subtracting a negative), **increases** `balance.vQuoteBalance`, and returns a negative `totalSettled` to `Clearinghouse._settlePnl`, which calls `spotEngine.updateBalance` with a negative amount — draining the user's spot quote balance. The net result is fabricated settlement capacity and a broken accounting invariant.

---

### Finding Description

**`getSettlementState`** computes `canSettle` as:

```solidity
// PerpEngine.sol lines 135-138
availableSettle = MathHelper.min(
    calculatePositionPnl(balance, productId),
    state.availableSettle
);
``` [1](#0-0) 

`MathHelper.min` is a plain signed comparison with no floor at zero:

```solidity
// MathHelper.sol lines 15-17
function min(int128 a, int128 b) internal pure returns (int128) {
    return a < b ? a : b;
}
``` [2](#0-1) 

When `positionPnl = -100e18` and `state.availableSettle = 50e18`, `min(-100e18, 50e18) = -100e18`, so `canSettle = -100e18`.

**`settlePnl`** then applies this without any positivity guard:

```solidity
// PerpEngine.sol lines 94-97
state.availableSettle -= canSettle;   // 50e18 - (-100e18) = 150e18  ← INFLATED
balance.vQuoteBalance -= canSettle;   // vQuoteBalance += 100e18     ← IMPROVED
totalSettled += canSettle;            // totalSettled = -100e18
``` [3](#0-2) 

**`Clearinghouse._settlePnl`** then passes the negative `amountSettled` directly to the spot engine:

```solidity
// Clearinghouse.sol lines 617-627
int128 amountSettled = perpEngine.settlePnl(subaccount, productIds);
_spotEngine().updateBalance(QUOTE_PRODUCT_ID, subaccount, amountSettled);
``` [4](#0-3) 

This reduces the user's spot quote balance by `|positionPnl|`.

---

### Impact Explanation

Three simultaneous invariant violations occur per call:

| State variable | Expected direction | Actual direction |
|---|---|---|
| `state.availableSettle` | Decreases (capacity consumed) | **Increases** (capacity fabricated) |
| `balance.vQuoteBalance` | Decreases (PnL realized out) | **Increases** (loss erased) |
| User spot quote | Increases (PnL received) | **Decreases** (value drained) |

The `availableSettle` inflation is the critical protocol-level impact: every call with a negative-PnL subaccount permanently inflates the pool of settlement capacity. Other subaccounts with genuine positive PnL can then settle against this inflated capacity, draining the protocol of real quote collateral beyond what is backed. This is a solvency/undercollateralization failure.

---

### Likelihood Explanation

`Clearinghouse.settlePnl` is `onlyEndpoint`, but the sequencer is expected to process settlement requests for any subaccount, including those with negative PnL. Additionally, `_settlePnl` is called internally during liquidation flows (referenced in `ClearinghouseLiq.sol`), meaning the bug can be triggered automatically on underwater accounts without any user action. Any subaccount that has ever had a losing perp position and a non-zero `availableSettle` in the product state is a trigger. [5](#0-4) 

---

### Recommendation

Add a positivity guard in `PerpEngine.settlePnl` before applying the settlement:

```solidity
if (canSettle <= 0) {
    productIds >>= 32;
    continue;
}
```

This mirrors the semantic intent: settlement only transfers realized **positive** PnL from `availableSettle` to the user's spot quote. Negative PnL should not be settled. [6](#0-5) 

---

### Proof of Concept

```
Setup:
  productId = 2 (even, perp)
  state.availableSettle = 50e18
  balance.amount = -10e18 (short position)
  priceX18 = 15e18
  balance.vQuoteBalance = 50e18
  positionPnl = 15e18 * (-10e18) + 50e18 = -100e18

Call: Clearinghouse.settlePnl({ subaccounts: [victim], productIds: [productId] })

Trace:
  getSettlementState → canSettle = min(-100e18, 50e18) = -100e18
  state.availableSettle -= -100e18  → 150e18  (was 50e18)
  balance.vQuoteBalance -= -100e18  → 150e18  (was 50e18)
  totalSettled = -100e18
  spotEngine.updateBalance(QUOTE_PRODUCT_ID, subaccount, -100e18)

Assert:
  state.availableSettle == 150e18   // increased by 100e18 (WRONG)
  user spot quote decreased by 100e18 (WRONG)
  user perp vQuoteBalance increased by 100e18 (WRONG)
```
