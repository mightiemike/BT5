### Title
Deprecated `verify_transaction_inclusion` Accepts Caller-Supplied Internal Merkle Node as Valid Transaction, Enabling Proof-Verification Forgery — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is a publicly callable NEAR contract method that accepts a caller-supplied `tx_id` and a Merkle proof, then returns `true` if the computed Merkle root matches the stored block header's `merkle_root`. The function performs **no check** that `tx_id` is a leaf-level transaction hash rather than an internal Merkle tree node. The contract's own inline warning acknowledges this: it explicitly delegates that validation to "a higher level of verification" — an off-chain assumption that is never enforced on-chain. Any unprivileged NEAR caller can supply an internal node hash as `tx_id` with a shortened proof path and receive a `true` return value for a transaction that never existed.

---

### Finding Description

`verify_transaction_inclusion` is marked `#[deprecated]` but remains a live, unguarded public method — only `#[pause]` is applied, meaning it is callable by any account when the contract is not paused. [1](#0-0) 

The function's sole on-chain check is that the Merkle proof is non-empty and that `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof)` equals the stored `header.block_header.merkle_root`: [2](#0-1) 

`compute_root_from_merkle_proof` is a pure path-traversal function — it hashes `tx_id` upward through the proof nodes using `tx_index` to determine left/right ordering at each level: [3](#0-2) 

Because the function does not validate that `tx_id` is a leaf (i.e., that the proof length equals `log2(tree_size)`), an attacker can supply any **internal node** `N` at depth `d` as `tx_id`, provide the `(tree_depth - d)`-length proof path from `N` to the root, and the function will compute the correct root and return `true`. The contract's own documentation admits this: [4](#0-3) 

This is the well-documented 64-byte transaction Merkle proof forgery (https://www.bitmex.com/blog/64-Byte-Transactions). `verify_transaction_inclusion_v2` was introduced to fix it via a coinbase proof length check: [5](#0-4) 

However, `verify_transaction_inclusion` was never removed or access-controlled, so the unfixed path remains fully reachable.

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate a privileged action — token minting, bridge release, cross-chain settlement — can be made to execute that action without any real Bitcoin transaction having occurred. The attacker supplies a real block hash (already stored in the contract's `headers_pool`) and a valid internal-node proof; the contract returns `true`; the downstream contract proceeds as if the Bitcoin event happened.

The corrupted proof result is: `verify_transaction_inclusion` returns `true` for a caller-chosen internal Merkle node `N` that is not a transaction, breaking the invariant that the function only attests to leaf-level transaction inclusion.

---

### Likelihood Explanation

The attack requires no privileges, no key material, and no social engineering. The attacker needs only:
1. A Bitcoin block already accepted by the light client (publicly observable on-chain).
2. Knowledge of any internal Merkle node in that block (derivable from public block data).
3. The ability to call a NEAR contract method.

The 64-byte forgery technique is publicly documented and well understood. Any developer integrating this contract via `verify_transaction_inclusion` (rather than `verify_transaction_inclusion_v2`) is exposed.

---

### Recommendation

1. **Remove or hard-restrict `verify_transaction_inclusion`**: Add an access-control role guard (e.g., `DAO` only) or remove the method entirely. A `#[deprecated]` attribute is a compile-time hint, not a runtime restriction.
2. **Enforce proof-depth validation inside `verify_transaction_inclusion`**: Require that `merkle_proof.len()` equals the expected tree depth derived from the block's known transaction count, or require a coinbase co-proof (as `verify_transaction_inclusion_v2` does) before delegating to the inner check.
3. **Do not rely on callers to validate `tx_id` externally**: The comment "We assume that validation … is performed at a higher level" is the exact off-chain trust assumption that the LiFi class of bugs exploits. That assumption must be enforced on-chain.

---

### Proof of Concept

**Setup**: Block `B` is already stored in the contract (submitted by the relayer). `B` has Merkle root `M` and contains 8 transactions, so the Merkle tree has depth 3.

**Attack**:
1. Attacker reads the full block from Bitcoin RPC and computes the internal node `N` at depth 1, position 0 (i.e., `hash(tx0 || tx1)`).
2. Attacker derives the 2-step proof path from `N` to `M`: `[hash(tx2||tx3), hash(hash(tx4||tx5)||hash(tx6||tx7))]`.
3. Attacker calls:
   ```
   verify_transaction_inclusion({
     tx_id: N,                  // internal node, not a real transaction
     tx_block_blockhash: B,
     tx_index: 0,               // position of N at depth 1
     merkle_proof: [sibling_at_depth1, sibling_at_depth0],
     confirmations: 1
   })
   ```
4. `compute_root_from_merkle_proof(N, 0, proof)` traverses upward and correctly reconstructs `M`.
5. The function returns `true`.
6. Any downstream contract gating on this result treats the call as proof that a Bitcoin transaction with id `N` was confirmed in block `B`. [6](#0-5) [3](#0-2)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
