### Title
`DepositAllowlistExtension` Guards LP Owner Instead of Token Depositor, Allowing Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` checks the LP position `owner` parameter against the per-pool allowlist, but ignores the `sender` parameter (the actual caller of `addLiquidity` who provides tokens via callback). Because `addLiquidity` imposes no restriction that `msg.sender == owner`, any unprivileged address can deposit tokens into an allowlist-gated pool by supplying an allowlisted address as `owner`.

---

### Finding Description

`BaseMetricExtension` declares `beforeAddLiquidity` with the `onlyPool` modifier and two leading address parameters: `sender` (the `msg.sender` of `addLiquidity`) and `owner` (the LP position recipient). [1](#0-0) 

`DepositAllowlistExtension` overrides this hook, drops the `onlyPool` modifier, and silently discards the first parameter (`sender`) by leaving it unnamed. The allowlist check is performed exclusively against `owner`: [2](#0-1) 

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the position recipient, with no requirement that they be equal: [3](#0-2) 

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both addresses to the extension: [4](#0-3) 

Because the extension ignores `sender` and only validates `owner`, an unprivileged caller (Alice, not on the allowlist) can call `addLiquidity(owner = Bob, ...)` where Bob is allowlisted. The extension sees `allowedDepositor[pool][Bob] == true` and permits the call. Alice's address is never evaluated. Alice provides tokens through the liquidity callback (called on `msg.sender` of `addLiquidity`, i.e., Alice), and Bob receives the LP position.

By contrast, `removeLiquidity` enforces `msg.sender == owner`, so only Bob can later withdraw — but Alice has already deposited unrestricted tokens into the pool. [5](#0-4) 

The `SwapAllowlistExtension` does not share this flaw — it correctly checks `sender` (the first parameter, the actual swapper): [6](#0-5) 

---

### Impact Explanation

The deposit allowlist is the pool admin's primary mechanism for restricting who can provide liquidity (e.g., KYC/AML compliance, permissioned pools, curated LP sets). The bypass allows any unprivileged address to inject tokens into the pool by routing through an allowlisted `owner`. This breaks the core access-control invariant the extension is designed to enforce, constitutes an admin-boundary break via an unprivileged path, and can be used to manipulate pool liquidity distribution or circumvent regulatory controls without the pool admin's consent.

---

### Likelihood Explanation

The trigger requires no special privilege: any external address can call `addLiquidity` with an arbitrary `owner`. The only prerequisite is knowing at least one allowlisted address (which is observable on-chain via `AllowedToDepositSet` events or direct `allowedDepositor` reads) and being willing to supply tokens. The path is always reachable on any pool that has deployed `DepositAllowlistExtension` with a non-empty allowlist.

---

### Recommendation

Change the allowlist check in `beforeAddLiquidity` to validate `sender` (the actual token depositor) rather than `owner` (the LP position recipient), matching the intent of the extension name and the `setAllowedToDeposit` API:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    onlyPool
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Restore the `onlyPool` modifier (dropped from the base) to prevent direct calls to the extension from non-pool addresses.

---

### Proof of Concept

1. Pool P is deployed with `DepositAllowlistExtension` E. Pool admin calls `E.setAllowedToDeposit(P, Bob, true)`. Alice is not allowlisted.
2. Alice calls `P.addLiquidity(owner = Bob, salt, deltas, callbackData, extensionData)`.
3. Pool calls `E.beforeAddLiquidity(sender=Alice, owner=Bob, ...)`.
4. Extension evaluates `allowedDepositor[P][Bob] == true` → passes. Alice's address is never checked.
5. Alice's liquidity callback fires; Alice transfers tokens to the pool.
6. Bob receives the LP shares. Alice has deposited into the allowlist-gated pool without being allowlisted. [2](#0-1) [7](#0-6)
