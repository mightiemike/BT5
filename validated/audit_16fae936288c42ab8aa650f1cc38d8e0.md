Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension` is documented as gating `addLiquidity` by depositor address, but its `beforeAddLiquidity` hook discards the `sender` argument (the actual `msg.sender` of the pool call) and instead checks the caller-supplied `owner` argument. Because `owner` is a free parameter chosen by the caller, any disallowed address can bypass the allowlist by passing an already-allowed address as `owner`, completely nullifying the pool admin's curation policy.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner` to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` correctly encodes both arguments and forwards them to the extension: [2](#0-1) 

However, `DepositAllowlistExtension.beforeAddLiquidity` silently discards the first parameter (`sender`) by leaving it unnamed, and gates only on `owner`: [3](#0-2) 

The storage layout and setter confirm the intent is to gate the **depositor** (the actual caller), not the position owner: [4](#0-3) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender`: [5](#0-4) 

The wrong value is the extension decision: `allowedDepositor[pool][owner]` is evaluated when `allowedDepositor[pool][sender]` should be. No existing guard compensates for this — the reentrancy guard and pool-level checks are unrelated to allowlist enforcement.

## Impact Explanation
Any disallowed address can call `pool.addLiquidity(allowedUser, salt, deltas, callbackData, "")`. The extension checks `allowedDepositor[pool][allowedUser]` → passes. The disallowed address pays tokens via the liquidity callback and the LP position is minted to `allowedUser`. If the disallowed address and `allowedUser` collude, `allowedUser` calls `removeLiquidity` and returns the proceeds, giving the disallowed address full economic participation in a curated pool. This is an admin-boundary break: an unprivileged path bypasses a pool-admin-configured access control with direct fund-flow consequences on curated pools.

## Likelihood Explanation
Exploitation requires only a single `addLiquidity` call with `owner` set to any address already in the allowlist (e.g., any existing LP, which is publicly observable on-chain). No special permissions, flash loans, or multi-step setup are needed. Any disallowed address can exploit this immediately and repeatably.

## Recommendation
Replace the `owner` check with a `sender` check in `beforeAddLiquidity`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
1. Pool admin deploys a pool with `DepositAllowlistExtension` and calls `setAllowedToDeposit(pool, alice, true)`. Bob is not allowed.
2. Bob calls `pool.addLiquidity(alice, salt, deltas, callbackData, "")`.
3. Pool calls `extension.beforeAddLiquidity(bob, alice, salt, deltas, "")`.
4. Extension checks `allowedDepositor[pool][alice]` → `true` → no revert.
5. Bob's callback pays the tokens; the LP position is minted to `alice`.
6. Alice calls `pool.removeLiquidity(alice, salt, deltas, "")` and transfers proceeds to Bob.
7. Bob has fully participated in a curated pool despite being explicitly excluded by the pool admin.
