### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas`, Causing Transaction Hash Preimage Divergence — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion silently produces `ValidResourceBounds::L1Gas` whenever `l1_data_gas` and `l2_gas` are both zero, even when the original transaction was submitted as `AllResources`. Because `get_tip_resource_bounds_hash` hashes a **different number of elements** depending on the variant (2 for `L1Gas`, 3 for `AllResources`), any component that recomputes the transaction hash after a protobuf round-trip will produce a hash that diverges from the one originally committed at the gateway. This breaks `validate_transaction_hash` for any V3 transaction whose `l2_gas` and `l1_data_gas` happen to be zero — a common case for simple transfers.

---

### Finding Description

**Step 1 — Hash preimage is variant-dependent.**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` builds the fee-fields hash differently depending on the `ValidResourceBounds` variant:

```
L1Gas      → poseidon_hash(tip, L1_GAS_packed, L2_GAS_packed)          // 3 elements
AllResources → poseidon_hash(tip, L1_GAS_packed, L2_GAS_packed, L1_DATA_GAS_packed) // 4 elements
``` [1](#0-0) 

**Step 2 — Protobuf deserialization silently changes the variant.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` applies the following rule:

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)   // ← silently drops AllResources
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [2](#0-1) 

A V3 transaction submitted with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is serialized to protobuf and deserialized back as `ValidResourceBounds::L1Gas(X)`. The `l1_data_gas` field is absent in the wire message (or present but zero), and the TODO comment confirms this is a known backward-compatibility concession for 0.13.2.

**Step 3 — Hash recomputation diverges.**

`validate_transaction_hash` recomputes the hash from the deserialized `Transaction` object and compares it to the stored hash:

```
Original hash H1 = poseidon(…, AllResources_hash_4_elements, …)
Recomputed hash H2 = poseidon(…, L1Gas_hash_3_elements, …)
H1 ≠ H2  →  validate_transaction_hash returns false
``` [3](#0-2) 

**Step 4 — The `InternalRpcTransactionWithoutTxHash::calculate_transaction_hash` path is also affected.**

When a `ConsensusTransaction` received over P2P is converted back to `InternalConsensusTransaction`, the converter calls `tx_without_hash.calculate_transaction_hash(&self.chain_id)` to recompute the hash. If the resource bounds variant was silently changed during protobuf deserialization, the recomputed hash will differ from the one the proposer stored, and the transaction will be treated as having a wrong hash. [4](#0-3) [5](#0-4) 

---

### Impact Explanation

Any V3 transaction with `l2_gas = 0` and `l1_data_gas = 0` — a common configuration for simple STRK transfers that only consume L1 gas — will have its hash silently recomputed to a different value after a protobuf round-trip. This causes:

- **P2P sync**: `validate_transaction_hash` returns `false` for a valid transaction, causing the syncing node to reject it. The node cannot make progress on blocks containing such transactions.
- **Consensus**: A validator that receives a block proposal over P2P and recomputes the transaction hash will disagree with the proposer, potentially causing consensus failures or the validator to mark the block as invalid.

This matches the allowed impact: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing**, and **Critical — Wrong state, receipt, event, or revert result from blockifier/syscall/execution logic for accepted input** (if the wrong hash is stored and later used for receipt lookup or event indexing).

---

### Likelihood Explanation

The trigger is unprivileged and requires no special access. Any user submitting a standard V3 invoke transaction with zero `l2_gas` and `l1_data_gas` (e.g., a simple token transfer that only uses L1 gas) will produce a transaction that hits this path. The TODO comment in the conversion code confirms the condition is intentionally tolerated for backward compatibility, making it a permanent fixture of the current codebase rather than a transient edge case. [6](#0-5) 

---

### Recommendation

1. **Remove the silent downgrade.** The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion should not produce `L1Gas` from a protobuf message that was serialized from an `AllResources` transaction. The variant should be encoded explicitly in the protobuf schema (e.g., a `oneof` or a boolean flag), not inferred from whether fields are zero.

2. **Enforce the TODO.** The comment `// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.` should be resolved. Once 0.13.2 support is dropped, assert that `l1_data_gas` is always present and remove the `unwrap_or_default()` fallback.

3. **Add a round-trip hash invariant test.** A test that serializes a V3 transaction with zero `l2_gas`/`l1_data_gas` to protobuf, deserializes it, recomputes the hash, and asserts it equals the original hash would have caught this.

### Proof of Concept

```

   l1_data_gas field is present but zero (or absent, defaulting to zero on the wire).

   TryFrom<protobuf::ResourceBounds> for ValidResourceBounds:
     l1_data_gas.is_zero() && l2_gas.is_zero()  →  ValidResourceBounds::L1Gas(1000)

6. H1 ≠ H2.
   validate_transaction_hash(tx, block_number, chain_id, H1, options) → false.
   Valid transaction rejected by the peer.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
}
