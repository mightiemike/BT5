### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables usdcE Drain Without USDC Payment — (`File: core/contracts/ContractOwner.sol`)

---

### Summary
`ContractOwner.replaceUsdcEWithUsdc()` calls the raw `IERC20Base.transferFrom()` without checking its return value. If the USDC token returns `false` on failure instead of reverting, the function proceeds to withdraw usdcE from the victim's Direct Deposit Address (DDA) and deliver it to the caller — without the caller ever providing USDC. The contract already imports and uses `ERC20Helper` with `safeTransferFrom`, making this omission inconsistent and avoidable.

---

### Finding Description

`ContractOwner` declares `using ERC20Helper for IERC20Base` at line 24, giving every `IERC20Base` instance access to the checked `safeTransferFrom` wrapper defined in `ERC20Helper.safeTransferFrom`. [1](#0-0) 

`ERC20Helper.safeTransferFrom` low-level-calls `transferFrom` and requires `success && (data.length == 0 || abi.decode(data, (bool)))`, reverting on any failure. [2](#0-1) 

Inside `replaceUsdcEWithUsdc`, the USDC pull from the caller uses the raw interface call instead:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // return value ignored
``` [3](#0-2) 

Immediately after, the function unconditionally:
1. Calls `DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE))`, which `safeTransfer`s the DDA's entire usdcE balance to `ContractOwner` (the caller of `withdraw` is `ContractOwner`). [4](#0-3) 
2. Calls `IERC20Base(usdcE).safeTransfer(msg.sender, balance)`, forwarding that usdcE to the original caller. [5](#0-4) 

`DirectDepositV1.withdraw` uses its own internal `safeTransfer` to move the full token balance to `msg.sender` (i.e., `ContractOwner`). [6](#0-5) 

The inconsistency is stark: the outbound `safeTransfer` on line 618 is safe, but the inbound `transferFrom` on line 616 is not. There is no access control on `replaceUsdcEWithUsdc` beyond the chain-ID check. [7](#0-6) 

---

### Impact Explanation

If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink chain (chainid 57073) returns `false` on a failed transfer (e.g., insufficient balance or allowance) rather than reverting, an attacker can:

- Call `replaceUsdcEWithUsdc(victimSubaccount)` with zero USDC balance or allowance.
- The `transferFrom` silently fails; no USDC moves to the DDA.
- The function still drains the DDA's entire usdcE balance to `ContractOwner`, then forwards it to the attacker.

**Corrupted asset delta**: Attacker gains `balance` usdcE; the victim subaccount's DDA loses `balance` usdcE; no USDC is ever deposited in exchange. This is direct collateral theft from any subaccount whose DDA holds usdcE.

---

### Likelihood Explanation

The USDC token address is hardcoded and chain-specific (Ink, chainid 57073). Standard Circle USDC implementations revert on failure rather than returning `false`, which limits immediate exploitability. However:

- The function is permissionless (any caller on the correct chain).
- If the token is upgraded, replaced, or is a non-standard bridged variant that returns `false`, the full theft path is immediately open.
- The rest of the codebase consistently uses `safeTransfer`/`safeTransferFrom`; this is an isolated deviation that creates a latent vulnerability.

Likelihood: **Low-Medium** (conditional on token behavior, but the root cause is entirely in Nado's code).

---

### Recommendation

Replace the raw `transferFrom` call with the `safeTransferFrom` wrapper already available via `ERC20Helper`:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

This is consistent with every other token transfer in the contract and eliminates the unchecked-return-value risk entirely. [8](#0-7) 

---

### Proof of Concept

1. A subaccount `S` has a deployed DDA (`directDepositV1Address[S] != address(0)`) holding `N` usdcE.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(S)` on Ink chain with zero USDC allowance.
3. `IERC20Base(usdc).transferFrom(attacker, dda, N)` returns `false` (no revert).
4. `DirectDepositV1(dda).withdraw(usdcE)` transfers `N` usdcE from DDA → `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, N)` transfers `N` usdcE from `ContractOwner` → attacker.
6. Attacker holds `N` usdcE; DDA holds 0 usdcE; attacker paid 0 USDC. [9](#0-8)

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
