### Title
AuxPoW Version Flag Not Checked When `aux_data` Is Absent, Allowing AuxPoW-Flagged Blocks to Bypass Merged-Mining Validation - (File: `contract/src/dogecoin.rs`)

### Summary
The Dogecoin `submit_block_header` function contains a field (`BLOCK_VERSION_AUXPOW`) in the block header's `version` that signals whether a block uses AuxPoW (merged mining). The `check_aux` function verifies this flag is set *when* AuxPoW data is provided, but `submit_block_header` never checks the inverse: when `aux_data` is `None`, it does not verify that the AuxPoW flag is absent. As a result, a block with the AuxPoW version flag set can be submitted without AuxPoW data and will be validated using only a direct Scrypt hash check — the wrong validation for a merged-mined block — allowing the contract to accept a block the real Dogecoin network would reject.

### Finding Description

In `contract/src/dogecoin.rs`, `BLOCK_VERSION_AUXPOW` is defined as `0x100` and is the protocol-level flag indicating a block was merged-mined: [1](#0-0) 

The `check_aux` function enforces that this flag **is** set when AuxPoW data is present: [2](#0-1) 

However, `submit_block_header` branches on whether `aux_data` is `Some` or `None`, and in the `None` branch it only checks the direct Scrypt hash against the target — with no check that the AuxPoW flag is absent from the version: [3](#0-2) 

The missing guard is the inverse of what `check_aux` enforces. The contract checks: "if AuxPoW data is present, the flag must be set." It never checks: "if the flag is set, AuxPoW data must be present."

In the real Dogecoin protocol, a block with `version & BLOCK_VERSION_AUXPOW != 0` is a merged-mined block. Its own Scrypt hash is **not** the proof of work — the PoW lives in the parent Bitcoin block's coinbase. Validating such a block by checking its own Scrypt hash is semantically incorrect and accepts a block the Dogecoin network would reject outright.

### Impact Explanation

An attacker submits a `(Header, None)` pair where `header.version & 0x100 != 0`. The contract skips `check_aux` entirely and falls into the direct Scrypt hash branch. If the attacker finds a nonce such that the Scrypt hash of the crafted header meets the `bits` target, the block is accepted and stored in `headers_pool`, potentially advancing `mainchain_tip_blockhash` to a hash that does not correspond to any block on the real Dogecoin chain. Downstream consumers calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` against this corrupted tip will receive incorrect proof results. [4](#0-3) 

### Likelihood Explanation

The attacker must find a Scrypt hash meeting the current Dogecoin difficulty target, which requires meaningful hashrate. However, the vulnerability is structurally reachable by any unprivileged NEAR caller via the public `submit_blocks` entry point with no role or stake requirement. The `trusted_relayer` macro can be bypassed if the relayer stake requirement is met or if the `UnrestrictedSubmitBlocks` role is granted. Additionally, a legitimate relayer with a bug that omits `aux_data` for AuxPoW-flagged blocks would trigger this silently. Since virtually all modern Dogecoin mainnet blocks (post block ~371337) carry the AuxPoW flag, the window for accidental or deliberate exploitation is wide. [5](#0-4) 

### Recommendation

Add an explicit check in the `None` branch of `submit_block_header` to reject any block whose version has the AuxPoW flag set but no AuxPoW data was provided:

```rust
} else {
    require!(
        block_header.version & BLOCK_VERSION_AUXPOW == 0,
        "AuxPoW flag set in version but no AuxPoW data provided"
    );
    let pow_hash = block_header.block_hash_pow();
    require!(
        U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
        "block should have correct pow"
    );
}
```

### Proof of Concept

1. Construct a `Header` with `version = 0x00620100` (chain ID `0x0062` for Dogecoin mainnet, AuxPoW flag `0x100` set), valid `prev_block_hash` pointing to a known tip, and any `merkle_root`, `time`, and `bits` matching the expected target.
2. Iterate `nonce` until `scrypt(header_bytes)` produces a hash whose little-endian `U256` value is ≤ `target_from_bits(bits)`.
3. Call `submit_blocks` on the NEAR contract with `[(header, None)]`.
4. The contract enters the `else` branch at `dogecoin.rs:182`, checks only the Scrypt hash, passes, and stores the block — despite the AuxPoW flag being set and no AuxPoW validation having occurred.
5. `mainchain_tip_blockhash` now points to a block hash that does not exist on the real Dogecoin network. [6](#0-5) [7](#0-6)
