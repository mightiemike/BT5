### Title
Compromised Linked Signer Can Front-Run Revocation to Drain Funds and Persistently Re-Link Itself — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` sequencer transaction is validated with `allowLinkedSigner = true`, meaning the **current linked signer** can sign a `LinkSigner` transaction to mutate `linkedSigners[subaccount]`. When a subaccount owner attempts to revoke a compromised linked signer, the attacker can front-run the revocation by submitting a re-linking transaction consuming the same nonce, creating a persistent race condition. Simultaneously, the compromised signer can drain collateral via `WithdrawCollateral`, which also accepts linked-signer signatures.

---

### Finding Description

`EndpointStorage.sol` stores a per-subaccount delegated signer:

```solidity
mapping(bytes32 => address) internal linkedSigners;
``` [1](#0-0) 

The nonce is keyed on the address portion of the subaccount bytes32 and is **shared** between the subaccount owner and any linked signer:

```solidity
require(
    nonce == nonces[address(uint160(bytes20(sender)))]++,
    ERR_WRONG_NONCE
);
``` [2](#0-1) 

In `processTransactionImpl`, the `LinkSigner` transaction is validated with `allowLinkedSigner = true`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true          // ← linked signer may sign this
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [3](#0-2) 

`validateSignature` resolves the permitted signer as either the subaccount owner **or** the current linked signer:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [4](#0-3) 

`WithdrawCollateral` (V1) is also validated with `allowLinkedSigner = true`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true
);
``` [5](#0-4) 

Because the nonce is shared and `LinkSigner` accepts the linked signer's signature, the compromised linked signer can race the owner's revocation on two axes:

1. **Fund drainage axis** — submit `WithdrawCollateral` signed by the compromised key at nonce N, consuming the nonce before the owner's `LinkSigner(signer=0, nonce=N)` is processed.
2. **Persistent re-link axis** — submit `LinkSigner(signer=attacker, nonce=N)` signed by the compromised key, consuming nonce N and causing the owner's revocation to fail with `ERR_WRONG_NONCE`. The owner must retry at N+1, and the race repeats indefinitely.

The slow-mode `LinkSigner` path (lines 232–239) does not require a signature but carries a hardcoded `SLOW_MODE_TX_DELAY` of three days, during which the compromised signer remains active and can continue submitting signed transactions through the sequencer. [6](#0-5) 

---

### Impact Explanation

**Corrupted state:** `linkedSigners[subaccount]` remains set to the attacker-controlled address, or collateral balances are reduced to zero via `WithdrawCollateral`. The attacker can drain the entire spot collateral of the targeted subaccount one nonce at a time, and can prevent the owner from ever successfully revoking the linked signer through the sequencer path. The only escape is the slow-mode path, which exposes a three-day window of continued attacker access.

---

### Likelihood Explanation

Requires a compromised linked signer key. Once that precondition is met, the exploit requires no privileged access — the attacker submits standard signed transactions to the sequencer before the owner's revocation is ordered. The sequencer processes transactions in submission order; an attacker monitoring the mempool or the owner's intent can reliably win the race. Likelihood is **Low** (key compromise is the gating condition) but the impact once triggered is **High** (full collateral loss).

---

### Recommendation

**Short term:** Document explicitly that a linked signer holds the same signing authority as the subaccount owner for all sequencer transaction types, including `LinkSigner` itself. Users must treat linked signer keys with the same security posture as their primary key.

**Long term:** Change `allowLinkedSigner` to `false` for `LinkSigner` transactions in `processTransactionImpl`. Revocation of a linked signer should require the subaccount owner's direct signature, not the linked signer's, eliminating the re-link race entirely. Separately, consider whether `WithdrawCollateral` should require owner-only signatures when a linked signer is set, or introduce a withdrawal time-lock that can be cancelled by the owner.

---

### Proof of Concept

1. Alice calls `LinkSigner(sender=Alice_sub, signer=Bob, nonce=0)` signed by Alice → `linkedSigners[Alice_sub] = Bob`.
2. Bob's key is compromised.
3. Alice constructs `LinkSigner(sender=Alice_sub, signer=address(0), nonce=1)` signed by Alice and submits it to the sequencer to revoke Bob.
4. The attacker, observing Alice's pending revocation, immediately submits `WithdrawCollateral(sender=Alice_sub, amount=MAX, nonce=1)` signed by Bob (valid because `allowLinkedSigner = true` and `linkedSigners[Alice_sub] == Bob`).
5. The sequencer processes the attacker's withdrawal first (nonce 1 consumed); Alice's `LinkSigner` reverts with `ERR_WRONG_NONCE`.
6. Alice retries with nonce 2; the attacker submits another `WithdrawCollateral` at nonce 2. The loop continues until Alice's collateral is exhausted.
7. Alternatively, at step 4 the attacker submits `LinkSigner(sender=Alice_sub, signer=Bob, nonce=1)` signed by Bob; if processed first, Bob re-links himself and Alice's revocation fails permanently through the sequencer path.
8. Alice's only remaining option is the slow-mode `LinkSigner`, which takes three days to execute — during which Bob retains full signing authority.
