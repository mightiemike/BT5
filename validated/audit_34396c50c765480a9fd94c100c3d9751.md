### Title
Bonus Reward Claim Coupled with Undelegate/Redelegate/Exit Blocks User Fund Withdrawal When Bonus Pool Is Depleted — (File: x/tieredrewards/keeper/msg_server.go)

---

### Summary

In the `x/tieredrewards` module, `TierUndelegate`, `TierRedelegate`, `AddToTierPosition`, `ExitTierWithDelegation`, and `ClearPosition` all call `claimRewards` as a mandatory prerequisite before executing their core logic. Inside `claimRewards`, `processEventsAndClaimBonus` checks `sufficientBonusPoolBalance` before transferring bonus tokens from the module pool. If the pool has insufficient balance, an error is returned that propagates up and causes the entire calling operation to revert. Users with accrued non-zero bonus rewards are permanently blocked from undelegating or exiting their positions when the bonus pool is depleted.

---

### Finding Description

In `msg_server.go`, every critical position-exit path calls `claimRewards` before performing its core state change:

- `TierUndelegate` (line 166): calls `claimRewards` before `undelegate`
- `TierRedelegate` (line 229): calls `claimRewards` before `redelegate`
- `AddToTierPosition` (line 314): calls `claimRewards` before `delegate`
- `ExitTierWithDelegation` (line 532): calls `claimRewards` before `transferDelegationFromPosition`
- `ClearPosition` (line 406): calls `claimRewards` before `ClearExit` [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

Inside `claimRewards`, `processEventsAndClaimBonus` is invoked. That function computes accrued bonus, then — before paying — calls `sufficientBonusPoolBalance`. If the check fails, it returns an error immediately: [6](#0-5) 

This error propagates through `claimRewards`: [7](#0-6) 

…and causes the entire calling message handler to return an error and revert. There is no code path in any of the five handlers that skips `claimRewards` or catches its error gracefully.

The bonus pool (`types.RewardsPoolName`) is a module account funded with a finite token supply. As bonus rewards are paid out via `bankKeeper.SendCoinsFromModuleToAccount`, the pool balance decreases monotonically. When the pool is empty or holds less than a position's accrued bonus, every operation that calls `claimRewards` for that position will fail. [8](#0-7) 

The position's delegator account is a derived account created by `createPositionDelegatorAccount` and controlled exclusively by the tier module — the position owner has no direct key for it and cannot call the staking module's `Undelegate` directly to bypass the tier module. [9](#0-8) 

---

### Impact Explanation

When the bonus pool is depleted, any position owner with accrued non-zero bonus rewards cannot:

- Undelegate their staked tokens (`TierUndelegate`)
- Redelegate to a different validator (`TierRedelegate`)
- Exit with their delegation (`ExitTierWithDelegation`)
- Clear their position (`ClearPosition`)

Because the position's delegator address is a module-derived account, the owner has no alternative path to recover their staked tokens. The staked principal is permanently locked until the bonus pool is refilled (which requires a governance action or privileged operation). The corrupted invariant is: **the position owner's staked token balance in the derived delegator account becomes unreachable**.

---

### Likelihood Explanation

The bonus pool has a finite balance. Normal protocol operation continuously drains it as users claim rewards. This is not a theoretical edge case — it is a natural lifecycle event. Additionally, any user with tier positions can accelerate depletion by repeatedly calling `ClaimTierRewards` (which does not gate on pool sufficiency before the check, but does drain the pool for each successful claim). Once the pool reaches zero, all positions with accrued bonus are simultaneously blocked from exiting.

---

### Recommendation

Decouple bonus reward payment from core position operations. In `TierUndelegate`, `TierRedelegate`, `ExitTierWithDelegation`, and `ClearPosition`, if `processEventsAndClaimBonus` fails due to insufficient pool balance, the operation should proceed with zero bonus paid (or defer the bonus claim) rather than reverting entirely. Bonus rewards can be claimed separately via `ClaimTierRewards` once the pool is refilled. This mirrors the fix described in the external report: separating dividend/reward payments from withdrawal/deposit operations.

---

### Proof of Concept

1. A user calls `LockTier` or `CommitDelegationToTier`, creating a position delegated to validator V. The position's delegator address is a module-derived account.
2. Over time, the position accrues non-zero bonus rewards (validator V remains bonded, time passes, `totalBonus > 0`).
3. The bonus pool (`RewardsPoolName`) is depleted — either naturally through other users claiming rewards, or by an attacker with many positions calling `ClaimTierRewards` repeatedly.
4. The user calls `TierUndelegate` to exit their position.
5. `msg_server.TierUndelegate` calls `ms.claimRewards(ctx, pos)` at line 166.
6. `claimRewards` calls `processEventsAndClaimBonus`, which computes `totalBonus > 0` and reaches `sufficientBonusPoolBalance` at line 230.
7. The pool is empty; `sufficientBonusPoolBalance` returns an error.
8. The error propagates: `processEventsAndClaimBonus` → `claimRewards` → `TierUndelegate` → transaction reverts.
9. The user's staked tokens remain locked in the derived delegator account with no alternative exit path. [10](#0-9) [11](#0-10)
