### Title
Unbounded `productIds` Array Iteration in Liquidation Finalization Causes Permanent Fund Lockup — (`core/contracts/ClearinghouseLiq.sol`, `core/contracts/BaseEngine.sol`)

---

### Summary

The `productIds` storage array in `BaseEngine` grows without any upper-bound check each time a new spot or perp product is registered. Multiple critical liquidation and settlement functions iterate over the entire array in a single transaction. As the number of listed products grows through normal protocol operation, these functions will eventually exceed the block gas limit, making it permanently impossible to finalize insolvent subaccounts, socialize losses, or clear NLP pool positions — locking user funds and threatening protocol solvency.

---

### Finding Description

`BaseEngine._addOrUpdateProduct()` unconditionally appends to the `productIds` storage array with no cap:

```solidity
// BaseEngine.sol
productIds.push(productId);   // line 249 — no length check
``` [1](#0-0) 

`ClearinghouseLiq._finalizeSubaccount()` fetches both `v.spotIds` and `v.perpIds` and then iterates over them in **five separate unbounded loops** within a single call frame:

- Loop 1 (lines 301–311): verify all spot assets are closed  
- Loop 2 (lines 313–320): verify all perp positions are zero  
- Loop 3 (lines 323–338): settle all positive perp PnL  
- Loop 4 (lines 346–366): settle all negative perp PnL  
- Loop 5 (lines 373–384): re-verify spot assets if insurance is sufficient [2](#0-1)

---

### Impact Explanation

When the cumulative number of listed spot and perp products is large enough, a single call to `liquidateSubaccountImpl` with `productId = type(uint32).max` (the finalization path) will consume more gas than the block gas limit across its chained loops. The concrete consequences are:

1. **Insolvent subaccounts cannot be finalized.** The insurance fund cannot cover their negative quote balance, and `socializeSubaccount` cannot distribute losses across depositors.
2. **Protocol solvency is corrupted.** Uncleared insolvent accounts accumulate, and the insurance fund is never properly debited, corrupting the `insurance` accounting variable.
3. **NLP pool positions cannot be cleared** via `clearNlpPoolPosition`, permanently locking NLP collateral.
4. **User funds are locked.** Depositors whose funds are subject to socialization cannot recover their assets because the socialization step itself reverts.

The corrupted state is the `insurance` balance and the `vQuoteBalance` / `amountNormalized` fields of all affected subaccounts — concrete on-chain asset values.

---

### Likelihood Explanation

The Nado protocol is designed to list an expanding set of spot and perp markets. Each call to `SpotEngine.addOrUpdateProduct` or `PerpEngine.addOrUpdateProduct` permanently appends to `productIds`. There is no removal mechanism and no cap. With a realistic deployment of 50–100 products (common for a DEX), the seven-plus sequential full-array loops in the liquidation finalization path will approach or exceed the 30M gas block limit on EVM-compatible chains. This is a near-certainty as the protocol matures, not a theoretical edge case.

---

### Recommendation

1. **Enforce a hard cap** on `productIds.length` inside `_addOrUpdateProduct`, e.g.:
   ```solidity
   require(productIds.length < MAX_PRODUCTS, ERR_TOO_MANY_PRODUCTS);
   ```
2. **Refactor multi-pass loops** in `_finalizeSubaccount` to combine checks into a single pass over each array.
3. **Use the existing product bitmap** (already maintained via `_setProductBit`) to iterate only products a subaccount actually holds, rather than the global `productIds` array, in health and liquidation calculations.

---

### Proof of Concept

**Trigger path (unprivileged liquidator):**

1. Protocol lists N spot products and M perp products over time via `addOrUpdateProduct` (normal operation; no cap enforced at line 249 of `BaseEngine.sol`).
2. A subaccount becomes insolvent (maintenance health < 0).
3. Liquidator calls `Endpoint → liquidateSubaccount({ productId: type(uint32).max, ... })`.
4. `liquidateSubaccountImpl` calls `_finalizeSubaccount` → 5 loops over `spotIds[N]` + `perpIds[M]`, then `socializeSubaccount` → 1 loop over `spotIds[N]`, then `socializeSubaccount` (perp) → 1 loop over `perpIds[M]`.
5. Total iterations ≈ 7·(N+M). At ~50 products each, this is ~700 SLOAD-heavy iterations, exceeding the block gas limit.
6. Transaction reverts with OOG. The insolvent subaccount is never cleared. Insurance is never debited. Losses are never socialized. Depositor funds are permanently locked. [8](#0-7) [9](#0-8)

        external
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.sender != txn.liquidatee, ERR_UNAUTHORIZED);
        require(isUnderMaintenance(txn.liquidatee), ERR_NOT_LIQUIDATABLE);
        require(
            txn.liquidatee != X_ACCOUNT && txn.liquidatee != N_ACCOUNT,
            ERR_NOT_LIQUIDATABLE
        );
        require(
            txn.productId != QUOTE_PRODUCT_ID,
            ERR_INVALID_LIQUIDATION_PARAMS
        );

        ISpotEngine spotEngine = ISpotEngine(
            address(engineByType[IProductEngine.EngineType.SPOT])
        );
        IPerpEngine perpEngine = IPerpEngine(
            address(engineByType[IProductEngine.EngineType.PERP])
        );

        if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
            if (RiskHelper.isIsolatedSubaccount(txn.liquidatee)) {
                IOffchainExchange(
                    IEndpoint(getEndpoint()).getOffchainExchange()
                ).tryCloseIsolatedSubaccount(txn.liquidatee);
            }
            return;
        }

        if (
            (txn.amount < 0) &&
            (txn.isEncodedSpread ||
                address(productToEngine[txn.productId]) == address(spotEngine))
        ) {
            // when it's spread or spot liquidation, we need to make sure the liquidatee has

            _assertCanLiquidateLiability(txn, spotEngine, perpEngine);
            _settlePositivePerpPnl(txn, spotEngine, perpEngine);
        }

        _assertLiquidationAmount(txn, spotEngine, perpEngine);

        // beyond this point, we can be sure that we can liquidate the entire
        // liquidation amount knowing that the insurance fund will remain solvent
        // subsequently we can just blast the remainder of the liquidation and
        // cover the quote balance from the insurance fund at the end
        _handleLiquidationPayment(txn, spotEngine, perpEngine);

    function socializeSubaccount(bytes32 subaccount, int128 insurance)
        external
        returns (int128)
    {
        require(msg.sender == address(_clearinghouse), ERR_UNAUTHORIZED);

        uint32[] memory _productIds = getProductIds();
        for (uint128 i = 0; i < _productIds.length; ++i) {
            uint32 productId = _productIds[i];
            (State memory state, Balance memory balance) = getStateAndBalance(
                productId,
                subaccount
            );
            if (balance.vQuoteBalance < 0) {
                int128 insuranceCover = MathHelper.min(
                    insurance,
                    -balance.vQuoteBalance
                );
                insurance -= insuranceCover;
                balance.vQuoteBalance += insuranceCover;
                state.availableSettle += insuranceCover;

                // actually socialize if still not enough
                if (balance.vQuoteBalance < 0) {
                    // socialize across all other participants
                    int128 fundingPerShare = -balance.vQuoteBalance.div(
                        state.openInterest
                    );
                    state.cumulativeFundingLongX18 += fundingPerShare;
                    state.cumulativeFundingShortX18 -= fundingPerShare;
                    balance.vQuoteBalance = 0;
                }
                _setState(productId, state);
                _setBalanceAndUpdateBitmap(productId, subaccount, balance);
            }
        }
        return insurance;
    }
