### Title
`ValidResourceBounds` Variant Downgrade in P2P State-Sync Protobuf Deserialization Produces Wrong Transaction Hash — (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary
The protobuf deserializer for `ValidResourceBounds` in the state-sync path silently downgrades an `AllResources` V3 transaction whose `l2_gas` and `l1_data_gas` are both zero to `ValidResourceBounds::L1Gas`. Because `get_tip_resource_bounds_hash` produces a structurally different Poseidon preimage for the two variants — `AllResources` appends a non-zero `l1_data_gas` felt even when the bound is zero — any component that recomputes the transaction hash from the deserialized object (echonet re-execution, proof generation, hash validation) obtains a value that diverges from the hash committed by the proposing node. The same downgrade also silently switches the fee-computation mode from `GasVectorComputationMode::All` to `NoL2Gas`, so a transaction that the proposer reverted (L2 gas exceeded the zero bound) may execute successfully on the syncing node, producing divergent state.

### Finding Description

**Conversion boundary — `crates/apollo_protobuf/src/converters/transaction.rs` lines 417–436**

```rust
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        // ...
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        // ...
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)          // ← downgrade
        } else {
            ValidResourceBounds::AllResources(...)
        })
    }
}
```

The heuristic was introduced to handle pre-0.13.3 transactions that omit `l1_data_gas` from the wire message. However, it also fires for any post-0.13.3 `AllResources` transaction where both `l2_gas` and `l1_data_gas` happen to be zero — a configuration the gateway explicitly accepts (test case `valid_l1_gas` in `apollo_gateway`).

**Hash domain boundary — `crates/starknet_api/src/transaction_hash.rs` lines 188–211**

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],                          // 2-element preimage
        ValidResourceBounds::AllResources(all) => {
            vec![get_concat_resource(&all.l1_data_gas, L1_DATA_GAS)?]     // 3-element preimage
        }
    });
    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```

`get_concat_resource` packs the 7-byte ASCII resource name into the upper bits of a 32-byte felt:

```rust
let concat_bytes =
    [[0_u8].as_slice(), resource_name.as_slice(), max_amount.as_slice(), max_price.as_slice()]
    .concat();
```

For `L1_DATA_GAS` with zero amounts the result is `0x004c315f44415441_0000000000000000_00000000000000000000000000000000`, a non-zero felt. Therefore the Poseidon hash over `[tip, l1_felt, l2_felt, l1_data_felt]` is always different from the hash over `[tip, l1_felt, l2_felt]`, even when all bounds are zero.

**Fee-mode divergence — `crates/blockifier/src/fee/fee_checks.rs` lines 197–236**

```rust
match valid_resource_bounds {
    ValidResourceBounds::AllResources(all_resource_bounds) => {
        FeeCheckReport::check_all_gas_amounts_within_bounds(
            &all_resource_bounds.to_max_amounts(), gas_vector)   // checks l2_gas ≤ 0 → revert
    }
    ValidResourceBounds::L1Gas(l1_bounds) => {
        FeeCheckReport::check_l1_gas_amount_within_bounds(...)   // converts l2_gas to l1_gas equiv
    }
}
```

A transaction submitted with `AllResources { l2_gas.max_amount = 0 }` is reverted by the proposer the moment any L2 gas is consumed. After the downgrade the syncing node applies the `L1Gas` path, which folds L2 gas into an L1-equivalent and checks it against `l1_gas.max_amount`. If that bound is non-zero the transaction succeeds, producing a different receipt and state root.

### Impact Explanation

Two concrete impacts, both reachable from an unprivileged user-submitted transaction:

1. **Wrong transaction hash (High — RPC/admission rejects valid blocks).** Any node that recomputes the transaction hash from the deserialized `L1Gas` object (echonet replay, `validate_transaction_hash`, proof verification) obtains a hash that does not match the one committed by the proposer. The block is rejected as invalid.

2. **Wrong execution result (Critical — wrong state/receipt from blockifier).** A transaction reverted on the proposer (L2 gas exceeded zero bound) executes successfully on the syncing node, writing storage and emitting events that the canonical state does not contain. This is a consensus-breaking state divergence.

### Likelihood Explanation

The trigger is a standard V3 invoke/declare/deploy-account transaction with `l1_gas > 0`, `l2_gas = 0`, `l1_data_gas = 0`. The gateway's stateless validator explicitly accepts this configuration (the `valid_l1_gas` test case in `apollo_gateway/src/stateless_transaction_validator_test.rs` passes with only `l1_gas` non-zero). No privileged role is required; any user can submit such a transaction.

### Recommendation

Remove the value-based heuristic. The protobuf schema should carry an explicit discriminant for the `ValidResourceBounds` variant, or the deserializer should unconditionally produce `AllResources` whenever all three resource-bound fields are present in the wire message (even if zero), reserving `L1Gas` only for messages where `l1_data_gas` is absent (the genuine 0.13.2 backward-compatibility case). The TODO comment at line 426 acknowledges this debt; it should be resolved before the 0.13.2 compatibility window closes.
