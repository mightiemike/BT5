### Title
Raw `approve()` Call in `DirectDepositV1.creditDeposit()` Reverts for Non-Standard ERC20 Tokens (USDT) â€” (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` using the raw `IIERC20Base.approve()` interface. In Solidity `^0.8.0`, the ABI decoder expects a 32-byte `bool` return value from any function declared to return `bool`. USDT and similar non-standard ERC20 tokens return zero bytes from `approve()`, causing the ABI decoder to revert. This permanently blocks collateral deposits for any subaccount whose DDA holds a non-standard token.

---

### Finding Description

`DirectDepositV1.sol` defines its own `IIERC20Base` interface with `approve` declared as returning `bool`: [1](#0-0) 

Inside `creditDeposit()`, this raw interface method is called directly: [2](#0-1) 

Solidity `^0.8.0` strict ABI decoding requires the callee to return exactly 32 bytes when the return type is `bool`. USDT returns nothing (0 bytes), so the ABI decoder reverts unconditionally. The entire `creditDeposit()` loop aborts, and no collateral is forwarded to the endpoint for any product in that iteration.

The contract already recognizes this class of problem for `transfer`: it implements a custom `safeTransfer` using low-level `call` with `data.length == 0` tolerance: [3](#0-2) 

The `ERC20Helper` library similarly provides `safeTransfer` and `safeTransferFrom` with the same low-level pattern: [4](#0-3) 

Neither `DirectDepositV1.sol` nor `ERC20Helper.sol` provides a `safeApprove` equivalent. The `approve` call at line 92 is the only unguarded token interaction remaining.

A secondary instance exists in `ContractOwner.wrapVaultAsset()`, which calls raw `assetToken.approve()` twice on an arbitrary vault asset token: [5](#0-4) 

---

### Impact Explanation

Any subaccount whose DDA (Direct Deposit Address) receives a USDT-like token will have `creditDeposit()` permanently revert. The token balance sits locked in the DDA contract â€” it cannot be forwarded to the endpoint as collateral. The subaccount never receives the credited position. Because `creditDeposit()` iterates all spot products and reverts on the first non-standard `approve`, even other standard-token balances in the same DDA are blocked in the same call.

---

### Likelihood Explanation

`creditDeposit()` is an `external` function with no access control: [6](#0-5) 

Any caller â€” including the depositing user â€” can trigger it. If USDT is ever listed as a spot product (a realistic scenario for a trading protocol), every DDA holding USDT will be bricked on the first `creditDeposit()` call. No privileged access or special conditions are required beyond the token being registered.

---

### Recommendation

Replace the raw `token.approve()` call in `creditDeposit()` with a low-level safe wrapper analogous to the existing `safeTransfer`:

```solidity
function safeApprove(
    IIERC20Base self,
    address spender,
    uint256 amount
) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IIERC20Base.approve.selector, spender, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        "Approve failed"
    );
}
```

Then replace line 92:
```solidity
// Before
token.approve(address(endpoint), balance);

// After
safeApprove(token, address(endpoint), balance);
```

Apply the same fix to the two `assetToken.approve()` calls in `ContractOwner.wrapVaultAsset()` at lines 530â€“531, using `ERC20Helper` extended with a `safeApprove` method.

---

### Proof of Concept

1. A USDT-like token (returns 0 bytes from `approve`) is registered as a spot product in `SpotEngine`.
2. A user sends USDT to their DDA address (created via `ContractOwner.createDirectDepositV1`).
3. Anyone calls `DirectDepositV1.creditDeposit()`.
4. The loop reaches the USDT product; `token.approve(address(endpoint), balance)` is executed.
5. Solidity `^0.8.0` ABI decoder expects 32 bytes for the declared `bool` return; USDT returns 0 bytes â†’ revert.
6. The entire transaction reverts. The USDT balance remains in the DDA. The subaccount receives no collateral credit.
7. Every subsequent call to `creditDeposit()` for this DDA reverts identically â€” the funds are permanently stuck unless the owner calls `withdraw()`. [7](#0-6)
