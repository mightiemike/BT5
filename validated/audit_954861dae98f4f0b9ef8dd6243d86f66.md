### Title
Stale `linkedSigners` Entry After NLP Pool Deletion Enables Unauthorized Signing Authority — (File: `core/contracts/EndpointTx.sol`)

---

### Summary
`deleteNlpPool` clears the NLP pool's entry in the `nlpSigners` registry but does not clear the corresponding `linkedSigners` entry for the same subaccount. This creates a cross-contract desynchronization: the NLP pool registry considers the pool deleted and its subaccount unauthorized, but the signer authorization layer (`linkedSigners`) still grants a previously-linked address full signing authority over the deleted pool's subaccount.

---

### Finding Description

When an NLP pool is deleted via `deleteNlpPool`, the function calls `updateNlpPool(poolId, address(0), uint128(0))`, which sets `nlpSigners[nlpPools[poolId].subaccount] = address(0)`. [1](#0-0) [2](#0-1) 

However, `deleteNlpPool` never touches `linkedSigners[poolSubaccount]`. If the pool owner (or the address embedded in the pool subaccount bytes32) had previously submitted a `LinkSigner` transaction for the pool's subaccount, that entry persists in `EndpointStorage.linkedSigners` indefinitely after deletion. [3](#0-2) 

The signer resolution function `getLinkedSignerOrNlpSigner` checks `linkedSigners` first, before falling back to `nlpSigners`: [4](#0-3) 

After pool deletion, `nlpSigners[poolSubaccount]` is `address(0)`, but `getLinkedSignerOrNlpSigner(poolSubaccount)` still returns the stale `linkedSigners` value. This stale address is passed as `takerLinkedSigner` / `makerLinkedSigner` in `MatchOrders` processing: [5](#0-4) 

Additionally, the stale linked signer can authorize fast-mode `WithdrawCollateral`, `TransferQuote`, `MintNlp`, and `BurnNlp` transactions for the deleted pool's subaccount, since all of these call `validateSignedTx(..., allowLinkedSigner: true)` which resolves through `getLinkedSigner`: [6](#0-5) 

The stale linked signer can also re-link itself to a new address via a `LinkSigner` transaction (since `allowLinkedSigner: true` is set there too), perpetuating the unauthorized authorization indefinitely: [7](#0-6) 

The deleted pool's subaccount remains registered in `subaccountIds` (it was recorded via `_recordSubaccount` at pool creation and is never deregistered), so `requireSubaccount` checks pass for it: [8](#0-7) 

---

### Impact Explanation

After `deleteNlpPool`, the stale linked signer retains the ability to:

1. **Sign `MatchOrders`** for the deleted pool's subaccount — the sequencer reads `getLinkedSignerOrNlpSigner` and passes the stale address as the valid linked signer to `OffchainExchange.matchOrders`, where `_validateOrder` accepts it.
2. **Sign `WithdrawCollateral` / `TransferQuote`** — if the deleted pool's subaccount accumulates any balance post-deletion (e.g., via `MatchOrders` trading profits or `SettlePnl`), the stale linked signer can drain it.
3. **Perpetuate the stale authorization** — by signing a new `LinkSigner` transaction to re-delegate to any address.

The broken invariant is: after `deleteNlpPool`, no address should retain signing authority over the pool's subaccount. The `nlpSigners` registry reflects this, but `linkedSigners` does not.

---

### Likelihood Explanation

**Medium.** The precondition is that a `LinkSigner` transaction was submitted for the NLP pool's subaccount before deletion. NLP pool owners have a natural incentive to do this (to delegate signing to a hot wallet). Fast-mode exploitation requires the sequencer to process transactions for the deleted pool's subaccount, which is a trusted role; however, the sequencer has no on-chain mechanism to detect that a pool has been deleted and its linked signer is stale — the `linkedSigners` mapping appears valid. The stale entry is permanent and self-perpetuating.

---

### Recommendation

In `deleteNlpPool` (or `updateNlpPool` when `owner == address(0)`), explicitly clear the `linkedSigners` entry for the pool's subaccount:

```solidity
function deleteNlpPool(uint64 poolId) private {
    require(poolId > 0 && poolId < nlpPools.length);
    bytes32 subaccount = nlpPools[poolId].subaccount;
    clearinghouse.clearNlpPoolPosition(subaccount);
    linkedSigners[subaccount] = address(0);   // <-- add this
    updateNlpPool(poolId, address(0), uint128(0));
}
```

---

### Proof of Concept

1. Owner calls `AddNlpPool` (slow mode, owner-only) → `addNlpPool(Alice, weight)` → `nlpPools[1].subaccount = poolSubaccount`, `nlpSigners[poolSubaccount] = Alice`.
2. Alice (or the address embedded in `poolSubaccount`) submits a fast-mode `LinkSigner` transaction: `sender = poolSubaccount`, `signer = Mallory`. Sequencer processes it → `linkedSigners[poolSubaccount] = Mallory`.
3. Owner calls `DeleteNlpPool` (slow mode) → `deleteNlpPool(1)` → `nlpSigners[poolSubaccount] = address(0)`, `linkedSigners[poolSubaccount]` **remains `Mallory`**.
4. Sequencer processes a `MatchOrders` transaction where `poolSubaccount` is the maker. `getLinkedSignerOrNlpSigner(poolSubaccount)` returns `Mallory` (from stale `linkedSigners`). Mallory's signature on the order is accepted by `_validateOrder`.
5. The deleted pool's subaccount accumulates quote balance from the trade. Mallory signs a `WithdrawCollateral` transaction; sequencer processes it; funds are withdrawn to Mallory's address. [1](#0-0) [9](#0-8) [3](#0-2)
