### Title
`burnNlp()` Enforces Only Maintenance Health Check While `mintNlp()` Enforces Initial Health Check, Allowing Users to Bypass the Initial Health Invariant — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

The `burnNlp()` function in `Clearinghouse.sol` only checks **MAINTENANCE** health after burning NLP tokens, while the symmetric `mintNlp()` function checks **INITIAL** health. This asymmetric invariant enforcement allows a user to burn NLP and leave their subaccount with `INITIAL health < 0` (but `MAINTENANCE health >= 0`), bypassing the protocol's initial margin requirement.

---

### Finding Description

`mintNlp()` enforces the INITIAL health invariant after state mutation: [1](#0-0) 

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
```

`burnNlp()` only enforces the weaker MAINTENANCE health invariant: [2](#0-1) 

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
```

The code comment at line 523–525 explicitly acknowledges that burning NLP **can decrease health**: [3](#0-2) 

```
// Burning NLP can decrease health if the burn fee exceeds the health improvement
// from the withdrawal. This check prevents malicious actors from deliberately
// creating unhealthy subaccounts through NLP burns.
```

The comment says the check prevents "unhealthy subaccounts" — but "unhealthy" here means only **under MAINTENANCE** (liquidatable). The check does **not** prevent the subaccount from going under **INITIAL** health. The burn fee is `max(1, quoteAmount / 1000)` (0.1% of quote value), which is deducted from the quote returned to the user: [4](#0-3) 

```solidity
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
```

If NLP carries a higher initial weight than the quote token (i.e., `longWeightInitialX18 > 1e18` for NLP, or if the user's health is already near the INITIAL threshold), the net health change from burning NLP is:

```
Δhealth ≈ nlpAmount × nlpPrice × (1 − nlpWeight) − burnFee
```

When `burnFee > nlpAmount × nlpPrice × (1 − nlpWeight)`, health decreases. The MAINTENANCE check passes, but INITIAL health goes negative.

The asymmetry is structurally identical to H-06: `mintNlp()` (the "minting" path) enforces the stricter invariant; `burnNlp()` (the "redeeming" path) does not.

---

### Impact Explanation

A user can call `burnNlp()` to deliberately push their subaccount below INITIAL health while remaining above MAINTENANCE health. Once under INITIAL health, the subaccount:

- **Cannot** withdraw collateral (`withdrawCollateral` checks INITIAL health at line 419)
- **Cannot** transfer quote (`transferQuote` checks INITIAL health at line 249)
- **Cannot** mint more NLP (`mintNlp` checks INITIAL health at line 480)
- **Can** continue burning NLP (only MAINTENANCE health is checked)

This allows a user to maintain open perp positions in a state that violates the initial margin invariant — a state the protocol explicitly prohibits for all other user-initiated operations. If the market moves adversely, the subaccount may cross the MAINTENANCE threshold and be liquidated with insufficient buffer, increasing insurance fund exposure. The initial health requirement exists precisely to provide this buffer. [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

Any user holding NLP tokens can trigger this. The entry path is:

1. User signs a `BurnNlp` transaction.
2. Sequencer submits it via `processTransactionImpl` → `clearinghouse.burnNlp()`.
3. `burnNlp()` applies the balance delta and checks only MAINTENANCE health.
4. If the burn fee exceeds the health improvement from converting NLP to quote, INITIAL health goes negative.

No privileged access is required. The user only needs unlocked NLP balance: [7](#0-6) 

The condition is reachable whenever a user's INITIAL health is near zero and they burn a non-trivial NLP amount.

---

### Recommendation

Change the health check in `burnNlp()` from `MAINTENANCE` to `INITIAL`, consistent with `mintNlp()`:

```solidity
// Before (line 526-529):
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);

// After:
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
```

If the intent is to allow NLP exits even when under initial health (e.g., to let users deleverage), a dedicated modifier or flag should make this explicit and bounded, rather than silently using the weaker MAINTENANCE threshold.

---

### Proof of Concept

```solidity
// 1. User has NLP tokens and open perp positions.
//    INITIAL health = +ε (just above zero).
//    MAINTENANCE health = +δ (comfortably above zero).

// 2. User submits BurnNlp transaction for a large nlpAmount.
//    burnFee = max(1, quoteAmount / 1000)
//    quoteAmount returned = nlpAmount * nlpPrice - burnFee

// 3. Inside burnNlp():
//    spotEngine.updateBalance(NLP_PRODUCT_ID, sender, -nlpAmount);  // health decreases
//    spotEngine.updateBalance(QUOTE_PRODUCT_ID, sender, quoteAmount); // health increases (less than decrease due to fee)
//    _applyNlpRebalance(...);

//    getHealth(sender, MAINTENANCE) >= 0  ← passes
//    getHealth(sender, INITIAL)     < 0   ← NOT checked, invariant broken

// 4. After the call:
//    - Subaccount is under INITIAL health (invariant violated)
//    - Subaccount is above MAINTENANCE health (not liquidatable)
//    - User retains open perp positions in a state the protocol prohibits
//      for all other user-initiated operations.

    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
