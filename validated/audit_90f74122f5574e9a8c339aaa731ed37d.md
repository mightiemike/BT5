### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the Coinbase Merkle Proof Check — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced specifically to add a coinbase Merkle proof check that mitigates the 64-byte transaction Merkle proof forgery vulnerability. However, the original `verify_transaction_inclusion` (v1) is still a `pub` NEAR-callable function. Any unprivileged NEAR caller can invoke v1 directly, completely skipping the coinbase proof validation, and receive a `true` result for a forged transaction inclusion proof.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces a coinbase Merkle proof check before delegating to v1: [1](#0-0) 

The check at lines 358–365 ensures the coinbase transaction is at index 0 of the Merkle tree, which prevents an attacker from supplying a 64-byte internal Merkle node as `tx_id` (the known forgery vector documented at https://www.bitmex.com/blog/64-Byte-Transactions).

The v1 function, however, is still `pub` and directly callable on-chain: [2](#0-1) 

Rust's `#[deprecated]` attribute only emits a compiler warning; it does **not** remove the function from the contract's public ABI or prevent on-chain invocation. The function's own doc comment acknowledges the risk explicitly:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [3](#0-2) 

The only validation v1 performs is:

```
compute_root_from_merkle_proof(args.tx_id, tx_index, &args.merkle_proof) == header.block_header.merkle_root
``` [4](#0-3) 

There is no check that `tx_id` corresponds to a leaf (real transaction) rather than an internal node. The coinbase guard added in v2 is the only mechanism that closes this gap, and it is entirely absent from the v1 path.

---

### Impact Explanation

A downstream NEAR contract that calls `verify_transaction_inclusion` (v1) — or that an attacker tricks into calling v1 — can be made to accept a forged Bitcoin transaction inclusion proof. The attacker supplies a crafted 64-byte internal Merkle node as `tx_id` along with a valid Merkle path, causing the function to return `true` for a transaction that was never included in the Bitcoin block. This corrupts the proof-verification result that the light client is designed to provide, enabling fraud in any protocol that relies on it (e.g., cross-chain bridges, peg-in/peg-out systems, SPV-based settlement).

---

### Likelihood Explanation

The entry path requires no privileges: any NEAR account can call `verify_transaction_inclusion` directly. The 64-byte forgery technique is well-documented and the inputs needed (a valid block in the contract's mainchain, a crafted 64-byte node, and a matching Merkle path) are all attacker-constructible. The only barrier is knowledge of the attack, which is publicly available.

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` (or gate it with an access-control role that no external caller holds), so that it can only be reached through `verify_transaction_inclusion_v2`. Alternatively, apply the coinbase Merkle proof check directly inside v1 and remove the separate v2 entry point. The deprecated marker alone provides no on-chain protection.

---

### Proof of Concept

1. Identify any block `B` in the contract's mainchain with a known Merkle root `R` and at least two transactions.
2. Construct a 64-byte value `fake_tx_id` that is the hash of two adjacent leaf nodes in `B`'s Merkle tree (i.e., an internal node whose subtree root equals `R` after applying the remaining proof path).
3. Build a `merkle_proof` path from `fake_tx_id` up to `R`.
4. Call `verify_transaction_inclusion` (v1) with `tx_id = fake_tx_id`, `tx_block_blockhash = B`, `tx_index = <matching index>`, `merkle_proof = <crafted path>`, `confirmations = 1`.
5. The function returns `true` because `compute_root_from_merkle_proof(fake_tx_id, ...) == R`, with no coinbase check performed. [5](#0-4) 

The v2 path would have rejected this at the coinbase proof step: [6](#0-5)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

                args.coinbase_tx_id.clone(),
                0usize,

            "Incorrect coinbase merkle proof"
