### Title
Linked Signer Can Unilaterally Reassign Itself to an Attacker-Controlled Address — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner: true`. This means the **current linked signer** — not just the subaccount owner — can sign a `LinkSigner` transaction that replaces itself with any arbitrary address. A malicious or compromised linked signer can permanently redirect signing authority over a subaccount to an attacker-controlled address, enabling unauthorized order submission and asset manipulation.

---

### Finding Description

In `EndpointTx.processTransactionImpl` (the sequencer-submitted fast path), the `LinkSigner` case is:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true   // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner: true` resolves the valid signer as either the subaccount owner **or** the current `linkedSigners[sender]`. This means the current linked signer can produce a valid signature for a `LinkSigner` transaction that writes a new, attacker-controlled address into `linkedSigners[signedTx.tx.sender]`. [2](#0-1) 

By contrast, the **slow mode path** for `LinkSigner` uses `validateSender`, which enforces that only the subaccount owner (`msg.sender == address(uint160(bytes20(txn.sender)))`) can change the linked signer:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    ...
    validateSender(txn.sender, sender);   // owner-only
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
}
``` [3](#0-2) 

The two paths are inconsistent. The fast path breaks the invariant that only the subaccount owner controls who the linked signer is.

---

### Impact Explanation

Once the linked signer is replaced with an attacker-controlled address (`Mallory`), Mallory can sign any transaction that accepts linked signers, including:

- **`WithdrawCollateral` (V1)** — linked signer allowed (`allowLinkedSigner: true`). Funds are sent to the subaccount owner's address, so direct off-protocol theft is blocked here. [4](#0-3) 

- **`MatchOrders` / `MatchOrdersWithAmount`** — the linked signer is passed as `takerLinkedSigner` / `makerLinkedSigner` and used to validate order signatures. Mallory can sign orders that trade the victim's assets at manipulated prices against a colluding counterparty, draining the subaccount through unfavorable fills. [5](#0-4) 

- **`MintNlp` / `BurnNlp`** — linked signer allowed (`allowLinkedSigner: true`). The signer supplies `oraclePriceX18`, which directly controls how many NLP tokens are minted or how much quote is returned on burn, enabling value extraction. [6](#0-5) 

- **`TransferQuote`** — linked signer allowed. Funds can be moved between the victim's own subaccounts, but combined with order manipulation this enables full account drain. [7](#0-6) 

The corrupted state is `linkedSigners[victim_subaccount]` — a persistent mapping that survives until the owner explicitly revokes it. Because the owner may not monitor this mapping, the attacker retains access indefinitely.

---

### Likelihood Explanation

Linked signers are the standard mechanism for automated trading bots and API keys in Nado. Any user who has set a linked signer is exposed. The attack requires only that the linked signer (or an attacker who has compromised it) submits a `LinkSigner` transaction to the sequencer — a normal, unprivileged operation. No admin access, governance capture, or key leakage beyond the linked signer itself is required.

---

### Recommendation

Change `allowLinkedSigner` to `false` in the `LinkSigner` case of `processTransactionImpl`, making it consistent with the slow mode path:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
``` [8](#0-7) 

This ensures that the linked signer — like the oracle in the AbraNFT report — cannot be mutated by any party other than the account owner after it has been established.

---

### Proof of Concept

1. Alice sets linked signer to `Bob` (a trading bot) via `LinkSigner` slow mode or fast mode signed by Alice.
2. `linkedSigners[Alice_subaccount] = Bob`.
3. Bob (malicious or compromised) constructs a `SignedLinkSigner` transaction: `sender = Alice_subaccount`, `signer = Mallory`, signed by Bob.
4. Bob submits this to the sequencer. The sequencer includes it in a batch.
5. `processTransactionImpl` calls `validateSignedTx(..., allowLinkedSigner: true)`. `getLinkedSigner(Alice_subaccount)` returns `Bob`. Bob's signature is valid. The transaction passes.
6. `linkedSigners[Alice_subaccount] = Mallory`.
7. Mallory now signs `MatchOrders` transactions selling Alice's assets at near-zero prices to a colluding account, draining Alice's subaccount. Alice's `linkedSigners` entry now points to Mallory; Alice may not detect this until significant damage is done. [9](#0-8) [1](#0-0)
