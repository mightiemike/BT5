Audit Report

## Title
SwapAllowlistExtension checks router address instead of originating EOA, allowing any user to bypass per-user swap allowlist via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` equal to `msg.sender` as seen by the pool, which is the router contract address when the swap originates from `MetricOmmSimpleRouter`. If the pool admin allowlists the router to support router-based swaps, every user of that router passes the check regardless of whether their own address is in `allowedSwapper[pool]`, defeating the per-user curation the extension is designed to enforce.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap` (L230–231). When the call originates from `MetricOmmSimpleRouter.exactInputSingle`, `msg.sender` at the pool is the router contract address (L72–80 of the router). `ExtensionCalling._beforeSwap` forwards this `sender` value unchanged to the extension (L160–176). `SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]` — i.e., `allowedSwapper[pool][router]` (L37–39). The router does not inject the originating EOA into `extensionData`; it passes `params.extensionData` verbatim from the caller. There is no mechanism within the extension to recover the original EOA: `tx.origin` is insecure, and the payer stored in transient storage via `_setNextCallbackContext` is inaccessible to the extension contract. Any non-allowlisted EOA can therefore execute a swap against a restricted pool by routing through `MetricOmmSimpleRouter`.

## Impact Explanation
A pool admin who deploys `SwapAllowlistExtension` to restrict swaps to a curated set of counterparties (e.g., KYC'd addresses, whitelisted market makers) and also allowlists the router to support router-based swaps inadvertently grants unrestricted swap access to every user of that router. Any non-allowlisted EOA can drain pool reserves by routing through `MetricOmmSimpleRouter`, causing unauthorized token outflow from a pool whose admin explicitly intended to restrict access. This is broken core pool functionality with direct fund impact, matching the "Broken core pool functionality causing loss of funds" allowed impact category.

## Likelihood Explanation
The scenario is straightforward: any pool using `SwapAllowlistExtension` that also needs to support router-based swaps must allowlist the router, at which point the per-user gate is fully open. The router is a public, permissionless contract. No privileged access is required by the attacker. The precondition (router allowlisted) is the natural operational state for any pool that intends to support router-mediated swaps alongside the allowlist extension.

## Recommendation
Thread the originating caller through the swap path so the extension can check the true economic actor. One approach: have the router encode the originating EOA (`msg.sender`) into `extensionData` before calling `pool.swap`, and have `SwapAllowlistExtension.beforeSwap` decode and check that address when present. Alternatively, introduce a router-aware allowlist that validates the payer stored in transient storage (`_getPayer()`) rather than the pool-visible `sender`, or gate on `recipient` for router flows where recipient equals the originating user.
