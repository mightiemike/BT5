### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable and Vulnerable to 64-Byte Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` (v1) function is still a live, public NEAR method callable by any unprivileged account. It is explicitly documented as vulnerable to the 64-byte Merkle proof forgery attack, yet the test suite contains zero tests for either `verify_transaction_inclusion` or `verify_transaction_inclusion_v2`. An attacker can supply an internal Merkle tree node as `tx_id` with a valid sibling proof and receive a `true` return value, falsely asserting transaction inclusion in a confirmed Bitcoin block.

---

### Finding Description

`verify_transaction_inclusion` is marked `#[deprecated]` at line 283, but in Rust the `#[deprecated]` attribute only emits a compiler warning — it does not restrict callability. The function remains `pub` and is a valid NEAR contract method reachable by any account. [1](#0-0) 

The function's own inline warning states:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [2](#0-1) 

The root cause is in `compute_root_from_merkle_proof`, which accepts any 32-byte hash as `transaction_hash` and walks the proof path without distinguishing leaf nodes from internal nodes: [3](#0-2) 

The verification logic in v1 reduces entirely to:

```
compute_root_from_merkle_proof(tx_id, tx_index, &merkle_proof) == header.merkle_root
``` [4](#0-3) 

There is no check that `tx_id` is a leaf-level hash. An internal node at depth `d` with a valid sibling path of length `d` will satisfy this equality for the real block's `merkle_root`.

`verify_transaction_inclusion_v2` was introduced to fix this by requiring a coinbase proof of equal length, anchoring the tree depth: [5](#0-4) 

But v1 remains callable in parallel, and the test suite exercises neither function. The entire test module is gated `#[cfg(feature = "bitcoin")]` and covers only block submission, fork promotion, and PoW rejection — no SPV proof paths at all. [6](#0-5) 

---

### Impact Explanation

Any NEAR contract or user that calls `verify_transaction_inclusion` (v1) can be deceived into accepting a forged transaction inclusion proof. Concretely:

- For a Bitcoin block with transactions `[T0, T1, T2, T3]`, the internal node `N = H(T0 ∥ T1)` is a valid 32-byte value.
- An attacker calls v1 with `tx_id = N`, `tx_index = 0`, `merkle_proof = [H(T2 ∥ T3)]`.
- `compute_root_from_merkle_proof(N, 0, [H(T2∥T3)])` = `H(N ∥ H(T2∥T3))` = real `merkle_root`.
- The function returns `true` for a transaction that does not exist.

Any downstream protocol (bridge, payment verifier, oracle consumer) that trusts this result is exposed to theft or unauthorized state transitions.

---

### Likelihood Explanation

**Medium.** The attack requires only public on-chain data (the block's Merkle tree, available from any Bitcoin node). No hash preimage computation or mining is needed. The attacker selects an existing internal node and constructs the sibling path mechanically. The only constraint is that the recipient contract must not independently validate the transaction content — a common assumption for SPV-based bridges.

---

### Recommendation

1. **Remove `verify_transaction_inclusion` (v1) from the public ABI** or gate it with an access-control role so external callers cannot invoke it. A `#[deprecated]` attribute alone is insufficient.
2. **Add integration tests** covering:
   - Happy-path SPV proof via `verify_transaction_inclusion_v2`.
   - Rejection of an internal-node forgery attempt against v2 (should fail at the coinbase proof check).
   - Confirmation that v1 is no longer directly callable by unprivileged accounts (if removed from the ABI).

---

### Proof of Concept

Given a real mainchain Bitcoin block `B` with `merkle_root = R` and at least four transactions `[T0, T1, T2, T3]`:

1. Compute `N01 = double_sha256(T0 ∥ T1)` and `N23 = double_sha256(T2 ∥ T3)` (both are public internal nodes).
2. Verify: `double_sha256(N01 ∥ N23) == R`. ✓
3. Call `verify_transaction_inclusion` with:
   - `tx_id = N01` (an internal node, not a real transaction)
   - `tx_block_blockhash = B`
   - `tx_index = 0`
   - `merkle_proof = [N23]`
   - `confirmations = 1`
4. Inside the contract, `compute_root_from_merkle_proof(N01, 0, [N23])` computes `double_sha256(N01 ∥ N23) = R`, which equals `header.block_header.merkle_root`.
5. The function returns **`true`** — falsely asserting that the non-existent transaction `N01` is confirmed in block `B`. [7](#0-6) [8](#0-7)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
