### Title
Missing Minimum Output Amount in `MintNlp`/`BurnNlp` — No User-Controlled Slippage Protection - (File: `core/contracts/interfaces/IEndpoint.sol`, `core/contracts/EndpointTx.sol`)

---

### Summary

The `MintNlp` and `BurnNlp` transaction structs allow users to specify only an input amount. The output amount is computed at execution time using `oraclePriceX18`, which is injected by the sequencer into `SignedMintNlp`/`SignedBurnNlp` and is structurally outside the user-signed payload. Neither struct contains a `minOutputAmount` field, so users have no on-chain guarantee that the NLP tokens received (on mint) or quote tokens received (on burn) will meet any minimum threshold.

---

### Finding Description

`MintNlp` and `BurnNlp` are the NLP liquidity-provision operations in the Nado protocol. Their signed structs are:

```solidity
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;   // input: quote tokens to spend
    uint64 nonce;
    // no minNlpAmount
}

struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;     // input: NLP tokens to burn
    uint64 nonce;
    // no minQuoteAmount
}
``` [1](#0-0) 

The sequencer-submitted wrappers carry the execution price as a separate field:

```solidity
struct SignedMintNlp {
    MintNlp tx;
    bytes signature;
    int128 oraclePriceX18;        // sequencer-provided, not user-signed
    int128[] nlpPoolRebalanceX18;
}

struct SignedBurnNlp {
    BurnNlp tx;
    bytes signature;
    int128 oraclePriceX18;        // sequencer-provided, not user-signed
    int128[] nlpPoolRebalanceX18;
}
``` [2](#0-1) 

In `EndpointTx.processTransactionImpl`, `validateSignedTx` is called with the user's signature, and then `oraclePriceX18` is forwarded directly to `clearinghouse.mintNlp`/`burnNlp` without any user-controlled bound check:

```solidity
validateSignedTx(signedTx.tx.sender, signedTx.tx.nonce, transaction, signedTx.signature, true);
chargeFee(signedTx.tx.sender, HEALTHCHECK_FEE);
priceX18[NLP_PRODUCT_ID] = signedTx.oraclePriceX18;
clearinghouse.mintNlp(signedTx.tx, signedTx.oraclePriceX18, nlpPools, signedTx.nlpPoolRebalanceX18);
``` [3](#0-2) 

Compare this to `WithdrawCollateralV2`, where the sequencer-provided `feeX18` is explicitly bounded:

```solidity
require(signedTx.feeX18 >= 0);
require(signedTx.feeX18 <= currentFeeX18);
``` [4](#0-3) 

No equivalent bound exists for `oraclePriceX18` in the NLP mint/burn path.

---

### Impact Explanation

- **MintNlp**: A user commits `quoteAmount` quote tokens. The NLP tokens received = `quoteAmount / oraclePriceX18`. If the sequencer submits a high `oraclePriceX18`, the user receives significantly fewer NLP tokens than expected with no recourse.
- **BurnNlp**: A user commits `nlpAmount` NLP tokens. The quote received = `nlpAmount × oraclePriceX18`. If the sequencer submits a low `oraclePriceX18`, the user receives significantly less quote than expected.

In both cases the user has already committed their input tokens (the transaction is signed and submitted). There is no on-chain mechanism to revert based on an unacceptable output amount. The user suffers a direct asset loss with no ability to set a floor.

---

### Likelihood Explanation

The sequencer is a trusted party, but the structural absence of a user-controlled minimum output means:

1. Even an honest sequencer operating with a stale oracle price (e.g., during high volatility between signing and execution) can cause the user to receive an unexpectedly low output.
2. A compromised or malicious sequencer can exploit this with zero on-chain resistance — there is no `require(outputAmount >= minOutputAmount)` guard anywhere in the path.
3. The protocol already demonstrates awareness of this pattern for withdrawals (`feeX18 <= currentFeeX18`), making the omission in the NLP path a concrete gap rather than an intentional design choice.

---

### Recommendation

Add `minOutputAmount` fields to both structs and enforce them in the clearinghouse execution path:

```solidity
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;
    uint128 minNlpAmount;   // user-specified minimum NLP to receive
    uint64 nonce;
}

struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint128 minQuoteAmount; // user-specified minimum quote to receive
    uint64 nonce;
}
```

In `Clearinghouse.mintNlp` / `burnNlp`, add:

```solidity
require(computedNlpAmount >= txn.minNlpAmount, ERR_SLIPPAGE_TOO_HIGH);
// and
require(computedQuoteAmount >= txn.minQuoteAmount, ERR_SLIPPAGE_TOO_HIGH);
```

The error constant `ERR_SLIPPAGE_TOO_HIGH` already exists in the codebase. [5](#0-4) 

---

### Proof of Concept

1. Alice signs a `BurnNlp` transaction: `{ sender: alice, nlpAmount: 1000e18, nonce: 5 }`, expecting to receive ~$1000 USDC at the current NLP price of $1.00.
2. The sequencer delays execution. NLP oracle price drops to $0.50 (or the sequencer provides `oraclePriceX18 = 0.5e18`).
3. The sequencer submits `SignedBurnNlp { tx: alice's signed tx, oraclePriceX18: 0.5e18, ... }`.
4. `clearinghouse.burnNlp` executes: Alice receives $500 USDC instead of $1000.
5. Alice's NLP tokens are already burned. There is no minimum-output guard to revert the transaction.
6. Alice loses $500 of value with no on-chain protection. [6](#0-5)
