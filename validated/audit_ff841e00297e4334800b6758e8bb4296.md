Audit Report

## Title
Missing `owner != address(0)` Validation in `addLiquidity` Permanently Locks Deposited Funds - (File: metric-core/contracts/MetricOmmPool.sol)

## Summary

`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` address with no zero-address guard. Because `removeLiquidity` enforces `msg.sender == owner` and the EVM guarantees `msg.sender != address(0)` for every external call, any position minted to `address(0)` is permanently irremovable. The depositor's tokens are locked in the pool forever, and `binTotals` is permanently inflated, distorting all subsequent swap and LP operations.

## Finding Description

`addLiquidity` in `MetricOmmPool.sol` passes `owner` directly into `LiquidityLib.addLiquidity` without any zero-address check:

```solidity
// MetricOmmPool.sol L182-196
function addLiquidity(
    address owner,          // ← never validated against address(0)
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
    if (deltas.binIdxs.length == 0) return (0, 0);
    if (deltas.binIdxs.length != deltas.shares.length) revert LiquidityDeltaLengthMismatch();
    _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
    (amount0Added, amount1Added) = LiquidityLib.addLiquidity(
      _liquidityContext(), owner, salt, deltas, callbackData, binTotals, _binStates, _binTotalShares, _positionBinShares
    );
    ...
}
```

Inside `LiquidityLib.addLiquidity`, the position key is computed as `keccak256(abi.encode(owner, salt, bin))` and shares are credited to `_positionBinShares[posKey]`. When `owner = address(0)`, shares are credited to the zero-address key. The token transfer occurs via the callback (`IMetricOmmModifyLiquidityCallback(msg.sender).metricOmmModifyLiquidityCallback(...)`) — the actual depositor supplies the tokens, but the position is recorded under `address(0)`.

The sole ownership gate in `removeLiquidity` is:

```solidity
// MetricOmmPool.sol L206
if (msg.sender != owner) revert NotPositionOwner();
```

Since `msg.sender` is never `address(0)` in any external call, no address can ever satisfy this check for a position owned by `address(0)`. The position — and the underlying token balances it represents — is permanently frozen.

The periphery contract `MetricOmmPoolLiquidityAdder` does include `_validateOwner` which rejects `address(0)`, but this guard exists only in the periphery and is trivially bypassed by calling the core pool directly. Any contract implementing `IMetricOmmModifyLiquidityCallback` can call `pool.addLiquidity(address(0), ...)` without going through the periphery.

## Impact Explanation

- **Direct loss of user principal**: every token deposited into an `owner = address(0)` position is permanently locked; the depositor cannot recover them.
- **Pool accounting distortion**: `binTotals.scaledToken0` / `scaledToken1` grow by the locked amount, inflating apparent liquidity and skewing `curPosInBin` for all subsequent swaps and LP operations on the pool.
- **Broken core pool functionality**: the remove-liquidity flow is permanently broken for the affected shares, violating the invariant that every LP can exit their position.

This constitutes a direct loss of user principal and broken core pool functionality, meeting the Critical/High threshold under the allowed impact gate.

## Likelihood Explanation

No privileged role is required. Any EOA or contract that implements `IMetricOmmModifyLiquidityCallback` can call `pool.addLiquidity(address(0), salt, deltas, callbackData, "")` directly, bypassing the periphery's `_validateOwner` guard entirely. A malicious actor can deliberately grief a pool by locking a meaningful amount of liquidity, distorting pricing for all other LPs. A user who zero-initialises a struct or a buggy router can trigger this accidentally.

## Recommendation

Add an explicit zero-address guard at the top of `addLiquidity` in `MetricOmmPool.sol`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
    require(owner != address(0), "invalid owner");   // ← add this
    if (deltas.binIdxs.length == 0) return (0, 0);
    ...
}
```

This mirrors the guard already present in the periphery's `_validateOwner` and closes the bypass path through direct pool calls.

## Proof of Concept

1. Deploy a `MetricOmmPool` with no extensions (or extensions that do not validate `owner`).
2. Deploy an attacker contract implementing `IMetricOmmModifyLiquidityCallback` that transfers tokens in the callback.
3. Call `pool.addLiquidity(address(0), salt, deltas, callbackData, "")` from the attacker contract.
4. The callback transfers `amount0Added` / `amount1Added` tokens from the attacker into the pool; `_positionBinShares[keccak256(abi.encode(address(0), salt, binIdx))]` is incremented and `binTotals.scaledToken0/scaledToken1` are inflated.
5. Attempt `pool.removeLiquidity(address(0), salt, deltas, "")` from any address → reverts with `NotPositionOwner` because `msg.sender != address(0)`.
6. The deposited tokens are permanently locked; `binTotals` reflects the inflated balance, distorting all future swap prices for the pool.
