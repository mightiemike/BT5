### Title
Unchecked `transferFrom()` Return Value in `replaceUsdcEWithUsdc` Allows Caller to Drain usdcE Without Providing USDC - (File: core/contracts/ContractOwner.sol)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the return value. If the USDC token silently returns `false` instead of reverting on a failed transfer, the function continues execution, withdrawing usdcE from the DirectDepositV1 address (DDA) and sending it to the caller — without the caller ever providing USDC.

---

### Finding Description

`replaceUsdcEWithUsdc()` is a public function callable by any address on chain 57073 (Ink mainnet). Its intended purpose is a 1:1 swap: the caller provides USDC to the DDA, and receives usdcE back. The sequence is:

1. Read `balance = IERC20Base(usdcE).balanceOf(directDepositV1)`.
2. Pull USDC from caller into DDA via `transferFrom` — **return value unchecked**.
3. Withdraw usdcE from DDA to `ContractOwner` via `DirectDepositV1.withdraw()`.
4. Forward usdcE from `ContractOwner` to caller via `safeTransfer`. [1](#0-0) 

Step 2 uses a raw `transferFrom()` call with no `require()` and no `safeTransferFrom()` wrapper:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
``` [2](#0-1) 

Immediately after, step 4 uses the safe wrapper correctly:

```solidity
IERC20Base(usdcE).safeTransfer(msg.sender, balance);
``` [3](#0-2) 

This inconsistency means that if the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` silently returns `false` on a failed transfer (e.g., insufficient allowance or balance), the function does not revert. Execution continues: the DDA's usdcE is drained to `ContractOwner` and forwarded to the caller, while no USDC was ever received.

The project's own `ERC20Helper.safeTransferFrom()` exists precisely to handle this case: [4](#0-3) 

It is used consistently everywhere else in the codebase (e.g., `BaseWithdrawPool.safeTransferFrom`, `Clearinghouse.handleWithdrawTransfer`), but was omitted here. [5](#0-4) 

---

### Impact Explanation

If the USDC token on Ink chain silently returns `false` on a failed `transferFrom` (non-compliant ERC20 behavior), any caller can:

1. Call `replaceUsdcEWithUsdc(subaccount)` for any DDA that holds usdcE.
2. Provide zero USDC allowance (causing `transferFrom` to silently fail).
3. Receive the full usdcE balance of the DDA for free.

This constitutes direct token theft from the DDA. The corrupted asset delta is the full usdcE balance of the targeted DDA, transferred to the attacker with no USDC consideration.

---

### Likelihood Explanation

The function has no access control beyond `block.chainid == 57073`. Any unprivileged caller can invoke it for any subaccount whose DDA holds usdcE. The likelihood depends on whether the specific USDC deployment at the hardcoded address is a non-reverting ERC20. Bridged or wrapped USDC variants on newer chains sometimes deviate from Circle's canonical implementation. The code pattern is unconditionally wrong regardless of current token behavior, and the risk materializes immediately if the token is ever upgraded or replaced.

---

### Recommendation

Replace the raw `transferFrom` call with the project's existing `ERC20Helper.safeTransferFrom`:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

This mirrors the pattern already used in `BaseWithdrawPool` and `Clearinghouse` throughout the codebase. [4](#0-3) 

---

### Proof of Concept

1. A DDA for some `subaccount` holds `N` usdcE tokens.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` with zero USDC allowance granted to `ContractOwner`.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, N)` returns `false` silently (non-reverting token).
4. `DirectDepositV1(directDepositV1).withdraw(usdcE)` executes: `N` usdcE moves from DDA → `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, N)` executes: `N` usdcE moves from `ContractOwner` → attacker.
6. Attacker receives `N` usdcE; no USDC was transferred. [6](#0-5)

    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
