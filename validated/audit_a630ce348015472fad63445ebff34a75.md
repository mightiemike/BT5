### Title
Host-function gas-key fee path uses borsh wire length instead of trie-id length for ML-DSA-65 keys, causing ~59× fee overcharge — (File: `runtime/near-vm-runner/src/logic/logic.rs`)

---

### Summary

The host-function implementations of `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, and `promise_batch_action_add_gas_key_with_function_call` pass the raw `public_key_len` (the borsh wire length, **1953 bytes** for ML-DSA-65) to `gas_key_transfer_send_fee`, `gas_key_transfer_exec_fee`, and `gas_key_add_key_exec_fee`. The runtime action-dispatch path (`total_send_fees` / `exec_fee` in `runtime/runtime/src/config.rs`) correctly calls `action.public_key.trie_id_len()` (**33 bytes** for ML-DSA-65) for the same fee helpers. The two values are identical for ED25519 (33) and SECP256K1 (65), so the divergence is invisible until an ML-DSA-65 key is used with a gas-key host function.

---

### Finding Description

`PublicKey` exposes two length methods whose values diverge only for the ML-DSA-65 variant:

| method | ED25519 | SECP256K1 | ML-DSA-65 |
|---|---|---|---|
| `len()` (borsh wire) | 33 | 65 | **1953** |
| `trie_id_len()` (on-trie hash) | 33 | 65 | **33** |

The documentation in `core/crypto/src/signature.rs` explicitly warns: "For storage-fee accounting use `PublicKey::trie_id_len` instead; for ML-DSA-65 those two diverge (1953 wire vs 33 on-trie)." The architecture doc (`docs/architecture/how/post_quantum_signatures.md`) states: "Every storage-stake and trie-byte-priced fee path was updated to call `trie_id_len()`."

The caller is responsible for passing the correct length, and the host-function callers pass the wrong one.

---

### Impact Explanation

For an ML-DSA-65 gas key:

- **Send fee** (`gas_key_transfer_send_fee`): charged with `public_key_len = 1953` instead of `trie_id_len = 33`. The send fee is **immediately burnt** and is not refunded. The overcharge factor is `1953 / 33 ≈ 59×` on the per-byte component.
- **Exec fee** (`gas_key_transfer_exec_fee`, `gas_key_add_key_exec_fee`): charged with `public_key_len = 1953` instead of `33`. The exec fee is reserved as "used gas" and attached to the receipt; the runtime re-charges the correct amount on execution, so the excess is partially refunded — but the send-fee overcharge is permanent.
- For `gas_key_add_key_exec_fee` with `num_nonces` nonces, the per-nonce overcharge compounds: each nonce entry's trie key is priced at `access_key_key_len(account_id_len, 1953)` instead of `access_key_key_len(account_id_len, 33)`.

The result is a **protocol-level fee inconsistency**: the same `TransferToGasKey` or `AddKey(GasKeyFullAccess)` action costs ~59× more gas when dispatched from a contract via host function than when submitted as a regular signed transaction. Any contract that budgets gas based on the documented trie-id length will run out of gas at the host-function call site.

---

### Likelihood Explanation

Requires both `ProtocolFeature::PostQuantumSignatures` and `ProtTPFeature::GasKeys` to be active simultaneously, and a contract to call the affected host functions with an ML-DSA-65 public key. Both features are currently nightly. Once either stabilises alongside the other, any on-chain contract that manages ML-DSA-65 gas keys programmatically hits the overcharge unconditionally — no special attacker action is needed; the wrong length is always passed.

---

### Recommendation

In `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, and `promise_batch_action_add_gas_key_with_function_call` (both in `logic.rs` and `wasmtime_runner/logic.rs`), decode the public key before computing fees and use `trie_id_len()`:

```rust
// After: let public_key = self.get_public_key(public_key_ptr, public_key_len)?;
let decoded_key = public_key.decode()?;
let key_trie_len = decoded_key.trie_id_len();

let send = gas_key_transfer_send_fee(&self.fees_config, sir, key_trie_len);
let exec = gas_key_transfer_exec_fee(&self.fees_config, receiver_id.len(), key_trie_len);
// ... then append action using decoded_key directly (no second decode needed)
```

The same fix applies to `gas_key_add_key_exec_fee` calls in the add-gas-key host functions.

### Proof of Concept

Exact divergent values (from `core/crypto/src/signature.rs`): [12](#0-11) [13](#0-12) 

The test `test_ml_dsa_65_trie_id_len_is_hash_size` in `runtime/runtime/src/access_keys.rs` confirms the divergence: [14](#0-13) 

A contract calling `promise_batch_action_transfer_to_gas_key` with a borsh-serialised ML-DSA-65 public key passes `public_key_len = 1953`. The host function then calls `gas_key_transfer_send_fee(..., 1953)`, burning `1953 × gas_key_byte_send_fee` instead of the correct `33 × gas_key_byte_send_fee`. The runtime, processing the same action from a signed transaction, calls `gas_key_transfer_send_fee(..., action.public_key.trie_id_len())` = `gas_key_transfer_send_fee(..., 33)`. The two code paths produce divergent gas costs for identical on-chain state transitions.

pub fn gas_key_transfer_send_fee(

pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}
```

pub fn gas_key_add_key_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
    num_nonces: NonceIndex,
) -> GasKeyAddFee {
    let num_nonces = num_nonces as u64;
    let base =
        cfg.fee(ActionCosts::gas_key_nonce_write_base).exec_fee().checked_mul(num_nonces).unwrap();
    let nonce_key_len =
        access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64)
        .unwrap()
        .checked_mul(num_nonces)
        .unwrap();
    GasKeyAddFee { base, per_byte }
```
