### Title
Coinbase Merkle Proof Does Not Validate That `coinbase_tx_id` Is a Leaf Node — Internal Node Hash Bypasses `verify_transaction_inclusion_v2` — (`contract/src/lib.rs`, `merkle-tools/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` is intended to prevent the 64-byte Merkle proof forgery attack by requiring a valid coinbase proof. However, it never validates that `coinbase_tx_id` is an actual leaf node (real transaction) in the Merkle tree. An attacker can supply an internal Merkle tree node hash as `coinbase_tx_id`, construct a valid proof path from that internal node to the root, and simultaneously use the same internal node hash as `tx_id` with the same proof. All guards pass and the function returns `true` for a `tx_id` that does not correspond to any real transaction.

---

### Finding Description

The function `verify_transaction_inclusion_v2` enforces three guards before returning:

1. **Length check** — `merkle_proof.len() == coinbase_merkle_proof.len()`
2. **Coinbase check** — `compute_root_from_merkle_proof(coinbase_tx_id, 0, &coinbase_merkle_proof) == header.block_header.merkle_root`
3. **Transaction inclusion check** (delegated to `verify_transaction_inclusion`) — `compute_root_from_merkle_proof(tx_id, tx_index, &merkle_proof) == header.block_header.merkle_root`, with the additional guard `require!(!args.merkle_proof.is_empty())` [1](#0-0) 

The coinbase check at lines 358–365 only verifies that the supplied `coinbase_tx_id`, when treated as a leaf at position 0, produces the correct Merkle root via `compute_root_from_merkle_proof`. It does **not** verify that `coinbase_tx_id` is actually a leaf node (i.e., a real transaction hash). Any internal node of the Merkle tree satisfies this check if the attacker provides the correct sibling path. [2](#0-1) 

`compute_root_from_merkle_proof` is a pure hash-chain computation with no leaf/internal-node distinction: [3](#0-2) 

The `From<ProofArgsV2> for ProofArgs` conversion passes `tx_id` and `merkle_proof` directly to `verify_transaction_inclusion` without any additional validation: [4](#0-3)

### Impact Explanation

Any downstream contract or application that calls `verify_transaction_inclusion_v2` to confirm a Bitcoin payment will accept a proof for a `tx_id` that is an internal Merkle tree node — a hash that does not correspond to any real Bitcoin transaction. An attacker can fabricate "proof of payment" for a transaction that never existed, causing the light client to return `true` for a non-existent transaction.

---

### Likelihood Explanation

The attack requires only:
- Knowledge of a real Bitcoin block already stored in the contract's `headers_pool` (publicly observable on-chain)
- The ability to compute the Merkle tree of that block (all transaction IDs are public)
- A single public NEAR contract call to `verify_transaction_inclusion_v2`

No privileged role, private key, or social engineering is required. The call is fully permissionless.

---

### Recommendation

After the coinbase check passes, enforce that `coinbase_tx_id` is at a true leaf depth. The proof depth (i.e., `coinbase_merkle_proof.len()`) must equal `ceil(log2(N))` where `N` is the number of transactions in the block. Since the contract stores only block headers (not full transaction lists), the simplest mitigation is to require that `coinbase_merkle_proof.len()` equals `merkle_proof.len()` **and** that `tx_index < 2^(proof_depth)`, and additionally require `coinbase_tx_id != tx_id` when `tx_index != 0`. More robustly, the contract should require that the caller also supply the total transaction count and validate the proof depth against it, or require that `coinbase_tx_id` is explicitly distinct from any internal node by enforcing a minimum proof depth relative to the block's known transaction count.

### Proof of Concept

Consider a block with 4 transactions `[tx0, tx1, tx2, tx3]` already stored in the contract:

```
node01     = dsha256(tx0 || tx1)
node23     = dsha256(tx2 || tx3)
merkle_root = dsha256(node01 || node23)
```

The attacker calls `verify_transaction_inclusion_v2` with:

```
tx_id                = node01          // internal node, NOT a real transaction
tx_block_blockhash   = <valid block>
tx_index             = 0
merkle_proof         = [node23]        // length 1
coinbase_tx_id       = node01          // same internal node
coinbase_merkle_proof = [node23]       // length 1
confirmations        = 0
```

**Guard-by-guard trace:**

| Guard | Evaluation | Result |
|---|---|---|
| Length check | `1 == 1` | PASS |
| Coinbase check | `dsha256(node01 \|\| node23) == merkle_root` | PASS |
| Proof non-empty | `[node23]` is not empty | PASS |
| Inclusion check | `dsha256(node01 \|\| node23) == merkle_root` | PASS → returns `true` |

`verify_transaction_inclusion_v2` returns `true` for `tx_id = node01`, which is an internal Merkle tree node and not a real Bitcoin transaction.

Note: The specific PoC in the question (empty proofs on a 2-transaction tree) is blocked by the `require!(!args.merkle_proof.is_empty())` guard at line 315. [5](#0-4) 
However, the general attack using a 4-transaction (or any ≥4-transaction) block with non-empty proofs is not blocked by any existing guard.

#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
