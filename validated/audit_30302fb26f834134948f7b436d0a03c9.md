Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any unpermissioned payer to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument and only checks `owner` against `allowedDepositor`. Because `MetricOmmPoolLiquidityAdder` allows any caller to freely specify an arbitrary `owner`, an address not on the allowlist can deposit tokens into a curated pool by naming any allowlisted address as `owner`. The allowlist invariant is broken with zero privileged access.

## Finding Description
`DepositAllowlistExtension.beforeAddLiquidity` is declared with an unnamed first positional argument (`sender`) that is never read:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner` to `_beforeAddLiquidity`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` forwards both to the hook:

```solidity
// metric-core/contracts/ExtensionCalling.sol L95-98
_callExtensionsInOrder(
    BEFORE_ADD_LIQUIDITY_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
);
```

`MetricOmmPoolLiquidityAdder.addLiquidityWeighted(pool, owner, ...)` accepts an arbitrary `owner` from any caller and calls `pool.addLiquidity(owner, ...)`. When the hook fires, `msg.sender` is the pool, `sender` is the `MetricOmmPoolLiquidityAdder` contract address, and `owner` is the caller-supplied bob. The hook checks `allowedDepositor[pool][bob]` — if bob is allowlisted, the check passes and alice's tokens (pulled via callback from `payer = msg.sender = alice`) enter the pool unchecked.

The `allowedDepositor` mapping name and `isAllowedToDeposit` function semantics both indicate the intent is to gate the actual depositor (payer), not the LP share recipient. The NatSpec on `addLiquidity` documents "owner must pass allowlist" but the extension's own naming (`allowedDepositor`, `DepositAllowlistExtension`) contradicts this as the intended gate identity.

The exploit path:
1. Pool admin deploys pool with `DepositAllowlistExtension` and calls `setAllowedToDeposit(pool, bob, true)`.
2. Alice (not allowlisted) calls `MetricOmmPoolLiquidityAdder.addLiquidityWeighted(pool, bob, salt, delta, ...)`.
3. Adder calls `pool.addLiquidity(bob, salt, delta, KIND_PAY_encoded, extensionData)`.
4. Pool calls `_beforeAddLiquidity(sender=liquidityAdder, owner=bob, ...)`.
5. Hook checks `allowedDepositor[pool][bob]` → `true` → no revert.
6. Pool calls back `metricOmmModifyLiquidityCallback`; adder pulls tokens from `payer = alice`.
7. Alice's tokens enter the pool; bob receives LP shares. Alice has bypassed the allowlist.

## Impact Explanation
The deposit allowlist is the primary curation mechanism for pools restricting who may provide liquidity (KYC-gated, regulatory compliance, curated LP sets). Bypassing it means unpermissioned funds enter the pool, the pool admin's token-flow policy is violated, and if the allowlist is used to exclude destabilizing depositors (e.g., whale limits), that protection is entirely removed. This constitutes a direct policy bypass with fund-level consequences on curated pools.

## Likelihood Explanation
No privileged access is required. Any address can call `MetricOmmPoolLiquidityAdder.addLiquidityWeighted` with an arbitrary `owner`. The attacker only needs to know one allowlisted address, which is publicly readable from the `allowedDepositor` mapping. The bypass is unconditional and works on every curated pool using `DepositAllowlistExtension`.

## Recommendation
Check `sender` (the actual caller of `addLiquidity`, i.e., the router/adder contract) in addition to or instead of `owner`. For the adder pattern where the true payer is `msg.sender` of the adder (not the adder itself), the pool admin should also allowlist the adder contract or the extension should be redesigned to receive the true payer identity via `extensionData`. At minimum, the `owner`-only check must be replaced:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender]
        && !allowedDepositor[msg.sender][sender]
        && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
1. Deploy pool with `DepositAllowlistExtension` configured as a `beforeAddLiquidity` hook.
2. Pool admin calls `setAllowedToDeposit(pool, bob, true)` — bob is allowlisted; alice is not.
3. Alice approves `MetricOmmPoolLiquidityAdder` for her tokens.
4. Alice calls `MetricOmmPoolLiquidityAdder.addLiquidityWeighted(pool, bob, salt, delta, maxCap, maxCap, ...)`.
5. Adder calls `pool.addLiquidity(bob, salt, delta, KIND_PROBE_encoded, extensionData)` (probe phase), then `pool.addLiquidity(bob, salt, scaledDelta, KIND_PAY_encoded, extensionData)` (pay phase).
6. On the pay call, pool invokes `_beforeAddLiquidity(sender=adder, owner=bob, ...)` → hook checks `allowedDepositor[pool][bob]` → `true` → passes.
7. Pool calls `metricOmmModifyLiquidityCallback`; adder pulls tokens from alice.
8. Assert `positionBinShares(pool, bob, salt, binIdx) > 0` and alice's token balance decreased — confirming alice deposited into an allowlisted pool without being allowlisted. This is directly demonstrated by the existing test `test_weighted_canAddOnBehalfOfAnotherOwner` in `metric-periphery/test/MetricOmmPoolLiquidityAdder.t.sol` L303-314, which shows alice paying while bob receives shares on a pool with no allowlist; the same flow bypasses the allowlist when bob is allowlisted and alice is not.
