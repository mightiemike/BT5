Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end-user. A pool admin who allowlists the router to support normal UX inadvertently grants every user — including those not individually allowlisted — the ability to bypass the swap allowlist entirely.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs its identity check at line 37:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` is the pool (correct — it namespaces per pool). `sender` is the value passed by the pool, which is `msg.sender` of the pool's own `swap()` call (`MetricOmmPool.sol` L231: `_beforeSwap(msg.sender, recipient, ...)`).

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(params.recipient, ...)` at line 72–80. The pool's `msg.sender` is the **router address**, so the extension evaluates `allowedSwapper[pool][router]`. The actual end-user's address is never seen by the extension.

This creates two mutually exclusive broken states:
- **Router allowlisted**: every user bypasses the allowlist by routing through the router.
- **Router not allowlisted**: every individually-allowlisted user is blocked from using the router.

`DepositAllowlistExtension.beforeAddLiquidity` does not share this flaw: it checks `owner` (the position owner, always the user-specified address), not `sender` (the caller), correctly gating the economically relevant party even through `MetricOmmPoolLiquidityAdder`.

## Impact Explanation
A curated pool (KYC-only, institutional, partner-restricted) using `SwapAllowlistExtension` that allowlists the router to support normal UX loses its curation guarantee entirely. Any unpermissioned user can trade at oracle prices against LP positions whose owners expected only authorized counterparties. This constitutes broken core pool functionality with direct LP exposure: unauthorized swap flow against a restricted pool can cause LP losses through adverse selection or policy violation.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap interface. The bypass requires only a single call to `exactInputSingle` — no special privileges, no flash loans, no complex setup. The precondition (router is allowlisted) is the natural operational state for any pool that wants to support router-mediated swaps, making this the default broken configuration for any curated pool.

## Recommendation
The extension must check the identity of the actual end-user. The cleanest fix is to add a `trustedRouter` mapping to `SwapAllowlistExtension`: when `sender` is a registered trusted router, redirect the identity check to `recipient` (the second argument of `beforeSwap`, currently unnamed/ignored). This requires no changes to the pool or router. Alternatively, document that pools using `SwapAllowlistExtension` must not allowlist the router and that allowlisted users must call `pool.swap()` directly — operationally fragile but requires no code change.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  pool admin calls setAllowedToSwap(pool, router, true)   // allowlist the router
  pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  attacker calls MetricOmmSimpleRouter.exactInputSingle({
      pool:      pool,
      recipient: attacker,
      ...
  })

  → router calls pool.swap(attacker, ...)           // MetricOmmSimpleRouter.sol L72-80
  → pool calls extension.beforeSwap(sender=router, recipient=attacker, ...)  // MetricOmmPool.sol L230-231
  → extension checks allowedSwapper[pool][router] → true  // SwapAllowlistExtension.sol L37
  → swap executes; attacker trades in the curated pool
  → allowlist policy is completely bypassed
```

The attacker pays no fees beyond normal swap fees and requires no collateral. The attack is repeatable on every block.
