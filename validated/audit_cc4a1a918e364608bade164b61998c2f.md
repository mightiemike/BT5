Audit Report

## Title
Unguarded `refundETH()` allows any caller to drain excess ETH left in `MetricOmmSimpleRouter` after a WETH swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`refundETH()` is a public, payable function with no access control that sends the router's entire ETH balance to `msg.sender`. When a user calls `exactInputSingle{value: X}(...)` with `tokenIn == WETH` and sends more ETH than `amountIn`, the `pay()` function deposits only the exact `amountIn` as WETH, leaving the surplus ETH stranded in the router. Any attacker can then call `refundETH()` to claim that surplus.

## Finding Description

`refundETH()` unconditionally transfers `address(this).balance` to `msg.sender` with no restriction on who the caller is: [1](#0-0) 

ETH enters the router legitimately via `exactInputSingle{value: X}(...)` (and other payable swap functions). The `receive()` guard only blocks plain ETH transfers from non-WETH addresses; it does not prevent ETH from arriving via `msg.value` on a payable call: [2](#0-1) 

Inside the swap callback, `_justPayCallback` calls `pay()` with the exact `amountIn` as `value`. In the WETH branch of `pay()`, when `nativeBalance >= value`, only `value` is deposited and transferred — the remainder stays in the contract: [3](#0-2) 

After `exactInputSingle` returns, the surplus ETH (`msg.value - amountIn`) is stranded in the router: [4](#0-3) 

The intended safe pattern is `multicall([exactInputSingle(...), refundETH()])` in one transaction, where `multicall` uses `delegatecall`: [5](#0-4) 

If the user calls `exactInputSingle` directly (not via multicall) or forgets to append `refundETH()` to the multicall, the excess ETH is stranded and claimable by any address in any subsequent transaction.

## Impact Explanation

Direct loss of user principal in native ETH. The attacker receives ETH that belongs to the user. No privileged role is required — the attacker only needs to observe the mempool or simply call `refundETH()` in a subsequent block. The loss equals the full excess ETH the user sent above `amountIn`. This meets the Critical/High threshold for direct loss of user principal.

## Likelihood Explanation

Users interacting with WETH swap paths commonly send a round-number ETH value slightly above the required `amountIn` to avoid slippage-induced reverts. Any such transaction that does not bundle `refundETH()` in the same multicall is immediately exploitable. Front-running or back-running the victim's transaction is trivial and requires no special capability. The vulnerability is repeatable against every such transaction.

## Recommendation

Record the initiating `msg.sender` in transient storage at the top of each payable swap entry point and enforce that only that address can call `refundETH()`. Alternatively, add a guard that restricts `refundETH()` to only be callable via `delegatecall` from the router itself (i.e., through `multicall`). The simplest correct fix:

```solidity
// In each payable entry point (exactInputSingle, exactInput, etc.):
// Store msg.sender in transient storage as the authorized refund recipient.

function refundETH() external payable override {
    address authorized = _getStoredPayer(); // read from transient storage
    require(msg.sender == authorized, "Not authorized");
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

## Proof of Concept

```solidity
function test_refundETH_drainByAttacker() public {
    // User sends 2 ETH for a 1 ETH WETH exactInputSingle swap (no multicall, no refundETH)
    router.exactInputSingle{value: 2 ether}(ExactInputSingleParams({
        pool: wethPool,
        tokenIn: WETH,
        recipient: user,
        amountIn: 1 ether,
        amountOutMinimum: 0,
        zeroForOne: true,
        priceLimitX64: 0,
        deadline: block.timestamp,
        extensionData: ""
    }));

    // pay() deposited exactly 1 ETH as WETH; 1 ETH surplus remains in router
    assertEq(address(router).balance, 1 ether);

    // Attacker (unrelated address) calls refundETH — no guard prevents this
    vm.prank(attacker);
    router.refundETH();

    // Attacker received the user's excess ETH
    assertEq(attacker.balance, 1 ether);
    assertEq(address(router).balance, 0);
}
```
