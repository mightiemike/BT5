### Title
Deprecated `verify_transaction_inclusion` Remains Directly Callable, Bypassing Coinbase Merkle Proof Guard — (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` is a public NEAR entry point that was superseded by `verify_transaction_inclusion_v2`. The v2 function adds a mandatory coinbase Merkle proof check specifically to block the 64-byte transaction forgery attack. Because `verify_transaction_inclusion` carries no `#[private]` attribute and no access-control guard, any unprivileged NEAR caller can invoke it directly, skipping the coinbase validation entirely and obtaining a `true` proof result for a fabricated transaction.

### Finding Description

`verify_transaction_inclusion_v2` is the secure code path. It first verifies a coinbase Merkle proof against the block's `merkle_root`, then delegates to `verify_transaction_inclusion` for the actual tx proof: [1](#0-0) 

The coinbase check is the only mechanism that prevents an internal Merkle tree node from being accepted as a valid `tx_id`. Without it, `verify_transaction_inclusion` reduces to a raw Merkle path computation: [2](#0-1) 

`verify_transaction_inclusion` is declared `pub` with only `#[pause]` — no `#[private]`, no role guard: [3](#0-2) 

The function's own doc-comment acknowledges the risk: *"This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash."* [4](#0-3) 

The structural parallel to M-06 is exact:

| M-06 (MIMO) | This repository |
|---|---|
| `MIMOProxyFactory.deployFor()` callable directly | `verify_transaction_inclusion` callable directly |
| Bypasses `MIMOProxyRegistry` registration | Bypasses `verify_transaction_inclusion_v2` coinbase check |
| Proxies lack full functionality | Proof result is forgeable |
| Fix: restrict factory to registry-only calls | Fix: restrict v1 to internal-only calls |

### Impact Explanation

An attacker can obtain a `true` return value from `verify_transaction_inclusion` for a transaction that does not exist in the Bitcoin blockchain. Any NEAR contract that consumes this result — e.g., a bridge, a payment gateway, or a cross-chain settlement contract — will treat the forged proof as valid. This enables theft of funds or assets gated behind Bitcoin transaction confirmation.

### Likelihood Explanation

The entry point is fully open to any NEAR account with no deposit, no role, and no staking requirement. The 64-byte transaction forgery technique is well-documented (https://www.bitmex.com/blog/64-Byte-Transactions) and requires only knowledge of a real Bitcoin block's Merkle tree structure, which is public. The attack is straightforward to execute by any caller who knows the contract ABI.

### Recommendation

Add `#[private]` to `verify_transaction_inclusion` so the NEAR runtime rejects any cross-contract or external call that is not a self-call:

```rust
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
#[private]   // ← add this
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
```

Alternatively, inline the v1 logic into `verify_transaction_inclusion_v2` and remove the v1 entry point entirely, eliminating the bypass path in the same way the MIMO mitigation eliminated the direct factory path.

### Proof of Concept

1. Identify any confirmed Bitcoin block `B` whose `merkle_root` is stored in the contract.
2. Obtain the full Merkle tree of `B`. Pick any internal node `N` at depth `d` (not a leaf). Its hash `H(N)` is 32 bytes and is a valid SHA256d digest.
3. Construct a Merkle proof path from `N` to the root — this is a standard sibling-hash list of length `d`.
4. Call `verify_transaction_inclusion` directly on the NEAR contract:
   - `tx_id` = `H(N)` (hash of the internal node, not a real txid)
   - `tx_block_blockhash` = hash of block `B`
   - `tx_index` = index corresponding to `N`'s position
   - `merkle_proof` = sibling hashes from `N` to root
   - `confirmations` = any value ≤ current depth
5. `compute_root_from_merkle_proof(H(N), index, proof)` reconstructs the correct `merkle_root` because `N` is genuinely part of the tree.
6. The function returns `true` — a confirmed inclusion proof for a transaction that does not exist. [5](#0-4)

```rust
    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
```
