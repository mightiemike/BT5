### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables usdcE Theft from Direct Deposit Accounts — (`core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` is a permissionless `external` function that uses a raw, unchecked `IERC20Base.transferFrom` call to pull USDC from the caller before releasing usdcE. If the USDC token at the hardcoded address on Ink chain (57073) behaves as a type-2 ERC20 — returning `false` on failure rather than reverting — the transfer silently fails and the function continues, allowing any caller to drain usdcE from any Direct Deposit Account (DDA) without paying.

---

### Finding Description

The entire Nado codebase routes ERC20 transfers through `ERC20Helper.safeTransfer` / `safeTransferFrom`, which use low-level `call` and explicitly check `success && (data.length == 0 || abi.decode(data, (bool)))` to handle all three ERC20 token variants correctly. [1](#0-0) 

`replaceUsdcEWithUsdc` is the sole exception. At line 616 it issues a raw high-level `transferFrom` call whose boolean return value is silently discarded: [2](#0-1) 

The function is `external` with no `onlyOwner` or `onlyDeployer` guard — only a `block.chainid == 57073` check. Any EOA or contract on Ink chain can call it for any `subaccount` that has a deployed DDA with a non-zero usdcE balance.

The three-step asset flow inside the `if (balance > 0)` block is:

1. **Line 616** — `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` — raw call, return value ignored.
2. **Line 617** — `DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE))` — moves all usdcE from the DDA to `ContractOwner` (`msg.sender` of `withdraw` is `ContractOwner`).
3. **Line 618** — `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` — sends usdcE from `ContractOwner` to the original caller. [3](#0-2) 

Steps 2 and 3 execute unconditionally after step 1. If step 1 returns `false` (type-2 token failure), the caller receives usdcE without having transferred any USDC.

The `withdraw` helper in `DirectDepositV1` uses the internal `safeTransfer` wrapper, so it will not revert on a type-3 token — it will successfully move usdcE to `ContractOwner` regardless of the failed step 1. [4](#0-3) 

---

### Impact Explanation

An attacker who calls `replaceUsdcEWithUsdc(subaccount)` with zero USDC allowance (or zero USDC balance) — when the hardcoded USDC token returns `false` instead of reverting — receives the full usdcE balance of the target DDA at no cost. This is a direct, complete theft of usdcE collateral from user-owned DDAs. The corrupted asset delta is: attacker gains `balance` usdcE; the DDA loses `balance` usdcE; no USDC is ever transferred.

---

### Likelihood Explanation

The function is permissionless and callable by any address on Ink chain (chainid 57073). The only precondition is that at least one DDA holds a non-zero usdcE balance, which is the exact scenario the function is designed to handle. Whether the specific USDC deployment at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink is type-2 determines exploitability; bridged or wrapped USDC variants on newer chains frequently omit revert-on-failure behavior. The inconsistency with the rest of the codebase — which uniformly uses `safeTransferFrom` — indicates this was an oversight rather than an intentional design choice.

---

### Recommendation

Replace the raw `transferFrom` call at line 616 with `ERC20Helper.safeTransferFrom`, consistent with every other transfer site in the codebase:

```solidity
// Before (line 616)
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`ERC20Helper` is already imported and `using ERC20Helper for IERC20Base` is already declared in `ContractOwner`. [5](#0-4) 

---

### Proof of Concept

1. Identify any `subaccount` whose DDA (`directDepositV1Address[subaccount]`) holds a non-zero usdcE balance.
2. On Ink chain (chainid 57073), call `ContractOwner.replaceUsdcEWithUsdc(subaccount)` from an EOA with **zero** USDC allowance to `ContractOwner`.
3. If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` returns `false` on the failed `transferFrom` (type-2 behavior):
   - Line 616 executes, returns `false`, no revert.
   - Line 617 executes: DDA's entire usdcE balance is transferred to `ContractOwner`.
   - Line 618 executes: `ContractOwner` sends the full usdcE balance to the attacker's address.
4. Attacker holds `balance` usdcE; DDA holds 0 usdcE; no USDC was ever moved. [2](#0-1)

```text
    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```
