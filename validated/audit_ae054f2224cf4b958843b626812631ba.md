### Title
Reentrancy via ERC777 Callback in `withdrawCollateral` Corrupts `nSubmissions` and Desynchronizes `markedIdxs` Double-Spend Guard - (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` performs the token transfer **before** decrementing the subaccount balance and running the health check. No `nonReentrant` guard exists anywhere in the codebase. If the collateral token implements a transfer callback (ERC777 `_callTokensReceived`, or any hook-bearing ERC20), the recipient can reenter `Endpoint.executeSlowModeTransaction` during the transfer. Because `executeSlowModeTransaction` caches `slowModeConfig` in memory and writes it back only after the call returns, the reentrant invocation reads a stale `txUpTo` from storage, silently fails on the already-deleted slow-mode entry, and then writes `txUpTo + 1` back to storage. The outer frame does the same write. The net result is that `nSubmissions` is incremented **twice** instead of once, permanently skipping one `idx` value in `BaseWithdrawPool.markedIdxs` and breaking the double-spend guard between slow and fast withdrawals.

---

### Finding Description

**Root cause — checks-effects-interactions violation in `withdrawCollateral`:** [1](#0-0) 

The token transfer at line 408 (`handleWithdrawTransfer`) is issued before `spotEngine.updateBalance` (line 412) and `getHealth` (line 419). No `nonReentrant` modifier is present on any contract in the codebase. [2](#0-1) 

`handleWithdrawTransfer` first moves tokens to `withdrawPool`, then calls `BaseWithdrawPool.submitWithdrawal`, which in turn calls `handleWithdrawTransfer(token, sendTo, amount)` — the final `token.safeTransfer(sendTo, amount)` that triggers the callback. [3](#0-2) 

**Reentrancy window — stale `slowModeConfig` in `executeSlowModeTransaction`:** [4](#0-3) 

`slowModeConfig` is loaded into a **memory** variable at the start and written back only after `_executeSlowModeTransaction` returns. During the token transfer deep inside that call, the storage copy of `slowModeConfig.txUpTo` is still the old value `N`.

**Reentrant call behaviour:**

Inside `_executeSlowModeTransaction`, the slow-mode entry at index `N` is deleted from storage before execution: [5](#0-4) 

When the callback reenters `executeSlowModeTransaction`, it reads `txUpTo = N` from storage (stale), loads the already-deleted `slowModeTxs[N]` (empty struct), and the inner `processSlowModeTransaction` call fails silently inside the `try/catch`. The reentrant frame then executes:

```
nSubmissions += 1;          // N → N+1
slowModeConfig = _slowModeConfig;  // writes txUpTo = N+1 to storage
```

The outer frame resumes and executes the same two lines again:

```
nSubmissions += 1;          // N+1 → N+2   ← extra increment
slowModeConfig = _slowModeConfig;  // writes txUpTo = N+1 again (idempotent)
```

**`nSubmissions` is now `N+2` instead of `N+1`.**

**How `nSubmissions` is used as the withdrawal `idx`:** [6](#0-5) 

The `idx` passed to `withdrawCollateral` is `nSubmissions` at call time. `BaseWithdrawPool.submitWithdrawal` marks `markedIdxs[N] = true` and sets `minIdx = N`. [7](#0-6) 

After the attack, `markedIdxs[N+1]` is **never set**. The next sequencer submission uses `idx = N+2`, leaving index `N+1` permanently unmarked.

**Fast-withdrawal double-spend gate is now bypassable:** [8](#0-7) 

`submitFastWithdrawal` only requires `!markedIdxs[idx]` and `idx > minIdx`. With `markedIdxs[N+1] == false` and `minIdx == N`, any caller holding a valid sequencer-signed withdrawal transaction for `idx = N+1` can submit it as a fast withdrawal, even though the corresponding slow withdrawal was already executed.

---

### Impact Explanation

An attacker who controls the `sendTo` address of a slow-mode withdrawal denominated in a callback-capable token (ERC777 or any hook-bearing ERC20) can corrupt `nSubmissions`, permanently skipping one `markedIdxs` slot. If the attacker also possesses a valid sequencer-signed withdrawal transaction for the skipped `idx` (obtainable from the sequencer's public submission history before the attack), they can submit it as a fast withdrawal via `BaseWithdrawPool.submitFastWithdrawal`, effectively double-spending the withdrawal: once via the slow path and once via the fast path. The corrupted `nSubmissions` also permanently desynchronizes the sequencer's expected `idx` from the on-chain counter, which can cause all future `submitTransactionsChecked` calls to revert until the sequencer detects and compensates for the drift.

---

### Likelihood Explanation

The attack requires: (1) a collateral token with a transfer callback (ERC777 is a deployed standard; custom hook tokens are also plausible), (2) the attacker controls the `sendTo` address of a slow-mode withdrawal, and (3) the attacker holds a valid sequencer-signed withdrawal transaction for the skipped `idx`. Condition (2) is trivially satisfied — the user submits their own withdrawal. Condition (3) is satisfiable because sequencer signatures are broadcast on-chain and can be captured before the sequencer processes the corresponding batch. No admin or privileged access is required. No `nonReentrant` guard exists anywhere in the codebase.

---

### Recommendation

1. **Apply checks-effects-interactions**: move `handleWithdrawTransfer` to after `spotEngine.updateBalance` and `getHealth` in `Clearinghouse.withdrawCollateral`.
2. **Add `nonReentrant` to `executeSlowModeTransaction`** (and `submitFastWithdrawal`) using OpenZeppelin `ReentrancyGuardUpgradeable`.
3. **Snapshot `nSubmissions` before the call** and assert it is unchanged after, or increment it before the external call.

### Proof of Concept

```
Setup:
  - Token T is ERC777 (or any hook-bearing ERC20).
  - Attacker A has a subaccount with balance B in token T.
  - A submits a slow-mode WithdrawCollateral for amount B, sendTo = AttackerContract.
  - Sequencer signs (but has not yet submitted) a second withdrawal W2 with idx = N+1.
    A captures the Schnorr signatures for W2 from the mempool/event log.

Attack (nSubmissions = N at start):
  1. Anyone calls Endpoint.executeSlowModeTransaction().
     → _executeSlowModeTransaction loads slowModeConfig (txUpTo=N) into memory.
     → slowModeTxs[N] is deleted from storage; txUpTo incremented to N+1 in memory.
     → processSlowModeTransaction → processSlowModeTransactionImpl
       → clearinghouse.withdrawCollateral(A, T, B, AttackerContract, N)
         → handleWithdrawTransfer: token.safeTransfer(withdrawPool, B)
           → BaseWithdrawPool.submitWithdrawal: markedIdxs[N]=true, minIdx=N
             → token.safeTransfer(AttackerContract, B)
               *** ERC777 tokensReceived callback fires ***

  2. Inside callback: AttackerContract calls Endpoint.executeSlowModeTransaction() again.
     → Reads slowModeConfig from storage: txUpTo = N (stale).
     → slowModeTxs[N] is empty (already deleted).
     → processSlowModeTransaction(address(0), "") reverts, caught silently.
     → nSubmissions += 1  (N → N+1)
     → slowModeConfig = _slowModeConfig  (writes txUpTo=N+1 to storage)

  3. Outer frame resumes (balance still not decremented, health not checked yet).
     → spotEngine.updateBalance(T, A, -B)   [balance decremented now]
     → getHealth check passes.
     → nSubmissions += 1  (N+1 → N+2)   ← EXTRA INCREMENT
     → slowModeConfig = _slowModeConfig  (writes txUpTo=N+1 again, idempotent)

Result:
  - nSubmissions = N+2 (should be N+1).
  - markedIdxs[N+1] is never set; minIdx = N.
  - A calls BaseWithdrawPool.submitFastWithdrawal(N+1, W2_tx, captured_signatures).
    → !markedIdxs[N+1] ✓, N+1 > minIdx=N ✓ → fast withdrawal executes.
  - A receives the withdrawal amount a second time.
```
