Audit Report

## Title
`addLiquidity` missing `whenNotPaused` guard allows LP deposits into paused pool, exposing principal to loss on unpause — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary
`MetricOmmPool` defines a `pauseLevel` state variable and a `whenNotPaused` modifier that reverts when `pauseLevel != 0`, but applies this modifier only to `swap`. `addLiquidity` carries only `nonReentrant`, so any caller can deposit tokens into a pool the protocol has declared unsafe. When the pool is unpaused, the deposited liquidity sits in bins priced at the oracle state that triggered the pause, and an attacker can immediately drain it via `swap` at the stale/manipulated price.

## Finding Description
`pauseLevel` is declared at L71–72 and transitions are enforced by `setPause` (L455–461). The `whenNotPaused` modifier (L174–177) calls `_checkNotPaused` (L643–645), which reverts with `PoolPaused` when `pauseLevel != 0`.

`swap` (L217–224) carries `whenNotPaused nonReentrant(PoolActions.SWAP)` — correctly blocked during a pause.

`addLiquidity` (L182–196) carries only `nonReentrant(PoolActions.ADD_LIQUIDITY)`. No pause check is present. The function proceeds to call `_beforeAddLiquidity`, `LiquidityLib.addLiquidity` (which writes to `binTotals`, `_binStates`, `_binTotalShares`, `_positionBinShares`), and `_afterAddLiquidity` — all without verifying `pauseLevel == 0`.

`removeLiquidity` (L199–212) is identically unguarded.

Exploit path:
1. Oracle reports a manipulated price (e.g., bid/ask shifted 10% below market).
2. Protocol detects the anomaly and calls `setPause(1)` → `pauseLevel = 1`.
3. `swap` now reverts with `PoolPaused`. `addLiquidity` does not.
4. An unsuspecting LP (or attacker front-running the unpause) calls `addLiquidity` with a large deposit. Tokens enter bins whose price ranges are anchored to the manipulated oracle mid-price. `binTotals` and `_binStates` are updated to reflect the deposit at the compromised price.
5. Protocol calls `setPause(0)` → `pauseLevel = 0`.
6. Attacker immediately calls `swap` (zeroForOne), buying token0 from the newly deposited bins at the manipulated (below-market) ask price. The LP's deposited token0 is drained; the attacker captures the price-manipulation spread.
7. LP has no recourse: the pool's accounting correctly reflects the swap against their deposit.

Existing guards are insufficient: `_beforeAddLiquidity` / `_afterAddLiquidity` extension hooks (e.g., `DepositAllowlistExtension`) run normally but enforce allowlist membership, not pool pause state. They cannot substitute for the missing `whenNotPaused` check.

## Impact Explanation
Direct loss of LP principal. The LP's tokens enter the pool in a state the protocol itself declared unsafe (oracle-manipulated or otherwise compromised). The pool's own accounting (`binTotals`, `_binStates`) records the deposit at the compromised price, and the subsequent swap correctly drains those bins. This is a fund-impacting broken core pool invariant: the pause mechanism is supposed to prevent all fund-impacting interactions during an unsafe state, but `addLiquidity` bypasses it entirely, resulting in a direct, irrecoverable loss of deposited principal above Sherlock thresholds for any non-trivial deposit.

## Likelihood Explanation
Pausing is a real operational event triggered by `setPause` (callable by factory, which is controlled by admin or protocol). LPs interacting through a router or UI will not inspect `pauseLevel` before calling `addLiquidity`. An attacker watching the mempool for a `setPause` transaction can front-run the unpause with a large `addLiquidity` call (or back-run the pause with one), then immediately swap on unpause. No special privilege is required — only a standard `addLiquidity` call from any address. The attack is repeatable across every pause/unpause cycle.

## Recommendation
Add `whenNotPaused` to `addLiquidity`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
```

`removeLiquidity` should remain unguarded (or guarded only at `pauseLevel == 2`) so existing LPs can always exit their positions — consistent with the principle that token returns to depositors must remain possible even in a restricted state.

## Proof of Concept

```
1. Deploy pool; oracle reports fair bid/ask.
2. Manipulate oracle: bid/ask shifted 10% below market.
3. Call factory → pool.setPause(1).
   - swap() reverts with PoolPaused. ✓
   - addLiquidity() has no pause check — succeeds. ✗
4. Call addLiquidity() with 1000 token0 + 1000 token1.
   - Tokens deposited into bins priced at manipulated oracle mid.
   - binTotals and _binStates updated at compromised price.
5. Call pool.setPause(0) (unpause).
6. Call swap(zeroForOne=true, large amountSpecified).
   - Pool uses current oracle price (still manipulated or now recovered).
   - Buys token0 from LP's bins at below-market ask.
   - LP's token0 drained; attacker profits the manipulation spread.
7. LP's position now holds only token1 at the manipulated exchange rate.
   Net loss = manipulation spread × deposited amount.
```

Foundry test outline:
- Deploy `MetricOmmPool` with a mock price provider returning a manipulated price.
- Call `setPause(1)` via factory.
- Assert `swap` reverts with `PoolPaused`.
- Assert `addLiquidity` succeeds and updates `binTotals`.
- Call `setPause(0)`.
- Call `swap` as attacker; assert LP's bin is drained and attacker receives token0 at below-market price.
- Assert LP's remaining claim is less than deposited principal.
