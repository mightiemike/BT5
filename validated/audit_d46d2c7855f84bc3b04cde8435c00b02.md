Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap` passes `msg.sender` (the router contract) as `sender` to `_beforeSwap`, which forwards it unchanged to `SwapAllowlistExtension.beforeSwap`. The extension gates on `allowedSwapper[pool][router]` rather than the actual end user, creating a two-sided failure: allowlisted users are blocked from using the router, and if the admin allowlists the router to restore access, every user — including non-allowlisted ones — bypasses the gate entirely. The allowlist, which is the primary access-control mechanism for curated pools, is completely nullified for all router-mediated swaps.

## Finding Description
**Root cause — pool passes `msg.sender` (router) as `sender`:**

`MetricOmmPool.swap` at L230–240 calls `_beforeSwap(msg.sender, ...)`. When invoked via `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the end user.

`ExtensionCalling._beforeSwap` at L149–177 forwards this value unchanged via `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))`.

`SwapAllowlistExtension.beforeSwap` at L37 then evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router — never the actual user.

`MetricOmmSimpleRouter.exactInputSingle` at L72–80 calls `pool.swap(params.recipient, ...)` with no mechanism to pass the original caller's identity as a distinct `swapper` argument. The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

**Contrast with deposit path:** `DepositAllowlistExtension.beforeAddLiquidity` at L32–42 checks `owner` (the second parameter), which is the explicit `owner` argument passed to `addLiquidity` — decoupled from `msg.sender`. The swap path has no equivalent explicit `swapper` field.

**Exploit flow:**
1. Admin deploys a curated pool with `SwapAllowlistExtension` and allowlists Alice: `allowedSwapper[pool][alice] = true`.
2. Alice calls `router.exactInputSingle(...)`. The router calls `pool.swap(alice_as_recipient, ...)`. Pool passes `msg.sender = router` to `_beforeSwap`. Extension checks `allowedSwapper[pool][router]` = `false` → reverts `NotAllowedToSwap`. Alice cannot trade despite being explicitly permitted.
3. Admin observes the breakage and adds the router: `allowedSwapper[pool][router] = true`.
4. Non-allowlisted Bob calls `router.exactInputSingle(...)`. Extension checks `allowedSwapper[pool][router]` = `true` → passes. Bob trades on a pool restricted to vetted counterparties. The curation policy is completely nullified.

**Existing guards are insufficient:** The `allowAllSwappers[pool]` escape hatch does not help — it is a separate flag. There is no on-chain mechanism within the current interface for the router to supply the original caller's address as the checked identity.

## Impact Explanation
This breaks the core access-control invariant of curated pools: only allowlisted addresses may swap. LPs who deposited under the assumption that only vetted counterparties trade against them are exposed to unrestricted counterparty flow once the admin performs the natural remediation step (allowlisting the router). This constitutes broken core pool functionality causing loss of funds (LP positions exposed to unintended counterparties) and an admin-boundary break (the pool admin's access-control policy is bypassed by any unprivileged user routing through the public router). The impact is High severity: the allowlist extension is the sole gating mechanism for curated pools, and its bypass is complete and unconditional for all router-mediated swaps.

## Likelihood Explanation
The path from Phase 1 to Phase 2 is a natural administrative response requiring no attacker sophistication. The router is a public, documented periphery contract whose address is known. Any user who discovers the router is allowlisted can exploit it immediately and repeatedly. No special permissions, front-running, or oracle manipulation are required.

## Recommendation
1. **In `MetricOmmPool.swap`**: Accept an explicit `swapper` parameter representing the economically relevant actor. The router would supply `msg.sender` (the end user) as `swapper`, and the pool would pass it to `_beforeSwap` instead of `msg.sender`. This mirrors the `addLiquidity` / `owner` pattern already used by `DepositAllowlistExtension`.
2. **In `SwapAllowlistExtension.beforeSwap`**: Until the pool interface is updated, document that the extension is incompatible with router-mediated swaps and that allowlisting the router address opens the gate to all users.

## Proof of Concept
```solidity
// Setup: curated pool with SwapAllowlistExtension
swapExtension.setAllowedToSwap(address(pool), alice, true);

// Phase 1: Alice is blocked despite being allowlisted
vm.prank(alice);
// Reverts NotAllowedToSwap — extension sees sender = router
router.exactInputSingle(ExactInputSingleParams({ pool: address(pool), recipient: alice, ... }));

// Admin "fixes" by allowlisting the router
swapExtension.setAllowedToSwap(address(pool), address(router), true);

// Phase 2: Non-allowlisted Bob bypasses the gate
vm.prank(bob); // bob is NOT in the allowlist
router.exactInputSingle(ExactInputSingleParams({ pool: address(pool), recipient: bob, ... }));
// Succeeds — allowlist completely bypassed
```

Foundry integration test: deploy pool with `SwapAllowlistExtension`, allowlist only `alice`, confirm `router.exactInputSingle` reverts for `alice` (Phase 1), then `setAllowedToSwap(pool, router, true)` and confirm `router.exactInputSingle` succeeds for `bob` (Phase 2).
