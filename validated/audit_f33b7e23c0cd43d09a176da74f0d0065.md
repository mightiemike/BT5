### Title
Proposer unconditionally binds `fee_proposal_fri` into `ProposalCommitment` while validator omits it for pre-V0_14_3 blocks, producing divergent commitment hashes - (`crates/apollo_consensus_orchestrator/src/build_proposal.rs`)

### Summary

`proposal_commitment_from` is a version-gated hash function: for pre-V0_14_3 blocks it must return `ProposalCommitment(partial.0)`, and for V0_14_3+ blocks it must return `Poseidon(partial.0, fee_proposal_fri)`. The validator path correctly gates on `init.fee_proposal_fri` (which is `None` for pre-V0_14_3 blocks). The proposer path in `build_proposal.rs` unconditionally passes `Some(args.fee_proposal)`, so it always computes the Poseidon form regardless of block version. The two sides therefore produce different `ProposalCommitment` values for any block with `starknet_version < V0_14_3`, breaking the canonicalization invariant that proposer and validator must agree on the same commitment hash.

### Finding Description

`proposal_commitment_from` is documented and implemented with an explicit version boundary: [1](#0-0) 

The validator path in `handle_proposal_part` passes `fee_proposal = args.init.fee_proposal_fri`, which is `None` for pre-V0_14_3 blocks (enforced by `is_proposal_init_valid`): [2](#0-1) [3](#0-2) 

The proposer path in `build_proposal.rs` unconditionally wraps `args.fee_proposal` (type `GasPrice`, never `Option`) in `Some(...)`: [4](#0-3) 

`args.fee_proposal` is always a concrete `GasPrice` value computed by `compute_proposer_fee_proposal`, which never returns `None`: [5](#0-4) 

For any block with `starknet_version < V0_14_3` (i.e., V0_14_0, V0_14_1, V0_14_2 — all supported by the orchestrator's versioned constants):

- **Proposer** computes: `Poseidon(partial.0, fee_proposal_fri)` — always, unconditionally.
- **Validator** computes: `partial.0` — because `init.fee_proposal_fri == None`.

These are two distinct field elements. Consensus requires both sides to agree on the same `ProposalCommitment`.

The orchestrator's versioned constants confirm V0_14_0 through V0_14_4 are all production versions: [6](#0-5) 

### Impact Explanation

For any block proposed with `starknet_version ∈ {V0_14_0, V0_14_1, V0_14_2}`, the proposer emits a `ProposalFin` carrying `Poseidon(partial.0, fee_proposal_fri)` while every validator independently computes `partial.0`. The two values are cryptographically distinct. Consensus cannot reach agreement: the proposer's commitment and the validator's commitment diverge, so no quorum can form on the correct block. This is a broken hash/commitment canonicalization invariant — the same block produces two different authoritative-looking commitment values depending on which code path computes it, matching the **High** impact: *"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

### Likelihood Explanation

The bug is triggered by any block proposal whose `starknet_version` is below V0_14_3. This is reachable during any protocol version transition from V0_14_2 → V0_14_3 (or earlier). No privileged access is required; the proposer role rotates among validators. The orchestrator explicitly supports V0_14_0–V0_14_2 via versioned constants, so the affected code path is production-reachable.

### Recommendation

In `build_proposal.rs`, gate the `fee_proposal` argument on the block's Starknet version, mirroring the validator's logic in `is_proposal_init_valid`:

```rust
// Derive fee_proposal only for V0_14_3+ blocks, matching the validator's gate.
let fee_proposal = if block_starknet_version >= StarknetVersion::V0_14_3 {
    Some(args.fee_proposal)
} else {
    None
};
let proposal_commitment = proposal_commitment_from(
    info.proposal_commitment.partial_block_hash,
    fee_proposal,
);
```

`ProposalBuildArguments` should carry the block's `starknet_version` (or the `fee_proposal` field should be changed to `Option<GasPrice>` set to `None` for pre-V0_14_3 blocks at the call site in `sequencer_consensus_context.rs`).

### Proof of Concept

1. Network is running on Starknet version V0_14_2 (pre-V0_14_3).
2. Proposer calls `build_proposal`. `compute_proposer_fee_proposal` returns `GasPrice(X)` for some `X > 0`.
3. Proposer computes: `proposal_commitment_from(partial, Some(GasPrice(X)))` → `Poseidon(partial.0, X)`. This value is sent in `ProposalFin`.
4. Validator receives `ProposalInit` with `fee_proposal_fri = None` (correct for V0_14_2). `is_proposal_init_valid` accepts it.
5. Validator computes: `proposal_commitment_from(partial, None)` → `partial.0`.
6. `Poseidon(partial.0, X) ≠ partial.0` for any `X`. The validator's commitment diverges from the proposer's.
7. Consensus cannot form a quorum; the block is never finalized.

The divergence is exact and deterministic: `Poseidon(partial.0, fee_proposal_fri)` vs `partial.0`, with `fee_proposal_fri` being the non-zero gas price computed by `compute_proposer_fee_proposal`. [7](#0-6) [4](#0-3) [3](#0-2)

/// Bind `fee_proposal_fri` to the proposal commitment hash.
///
/// Pre-V0_14_3 blocks have `fee_proposal = None` and the commitment is just `partial.0`,
/// preserving on-chain behavior. From V0_14_3 onward, the commitment is
/// `Poseidon(partial.0, fee_proposal_fri)`, so a proposer cannot equivocate on
/// `fee_proposal_fri` without changing the commitment that consensus signs over.
///
/// This computation lives in the orchestrator (not in `starknet_api`) because the
/// fee_proposal is a consensus-time signal — `starknet_api`'s block-hash primitives
/// remain unaware of it, and `BlockHash` is unaffected.
pub(crate) fn proposal_commitment_from(
    partial: PartialBlockHash,
    fee_proposal: Option<GasPrice>,
) -> ProposalCommitment {
    let Some(fee_proposal) = fee_proposal else {
        return ProposalCommitment(partial.0);
    };
    ProposalCommitment(Poseidon::hash_array(&[partial.0, Felt::from(fee_proposal.0)]))
}

        SNIP35_FEE_TARGET_ATTO_USD.set_lossy(target_atto_usd_per_l2_gas);

            fee_actual,

        );
        SNIP35_FEE_PROPOSAL_FRI.set_lossy(proposal.0);

    }
