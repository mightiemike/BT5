### Title
Sequencer-Controlled `minIdx` Permanently Invalidates Fast Withdrawal Claims Without On-Chain Verification — (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

The `submitFastWithdrawal` function in `BaseWithdrawPool.sol` requires a sequencer-assigned `idx` and sequencer-produced multi-signatures that are not independently derivable or verifiable on-chain by the user. The `minIdx` state variable is unconditionally advanced by the sequencer via `submitWithdrawal`, and any fast withdrawal attempt with `idx ≤ minIdx` is permanently and irrevocably rejected. Users are therefore entirely dependent on the sequencer to (1) provide the correct `idx` and pre-signed data, and (2) not advance `minIdx` past their assigned `idx` before they can act. This is a direct structural analog to the Linea `l2MessagingBlocksOffsets` censorship issue: unverified operator-supplied data gates a user's ability to claim funds, and operator non-cooperation permanently forecloses the claim path.

---

### Finding Description

**Entry point — `submitFastWithdrawal`:**

```solidity
function submitFastWithdrawal(
    uint64 idx,
    bytes calldata transaction,
    bytes[] calldata signatures
) public {
    require(!markedIdxs[idx], "Withdrawal already submitted");
    require(idx > minIdx, "idx too small");          // ← permanent gate
    markedIdxs[idx] = true;
    Verifier v = Verifier(verifier);
    v.requireValidTxSignatures(transaction, idx, signatures);
    ...
}
``` [1](#0-0) 

The `idx` parameter is the value of `nSubmissions` that the sequencer intends to use when it eventually processes the withdrawal on-chain. This value is:

- **Not deterministic from the user's perspective** — `nSubmissions` is a monotonically increasing counter incremented by the sequencer for every transaction it submits.
- **Not recorded on-chain at request time** — there is no on-chain mapping from a user's withdrawal request to a future `nSubmissions` slot.
- **Signed by ALL sequencer signers** — `requireValidTxSignatures` requires `nSignatures == nSigner` (every registered signer, not a quorum), making the signed data exclusively producible by the sequencer. [2](#0-1) 

**`minIdx` advancement — `submitWithdrawal`:**

```solidity
function submitWithdrawal(
    IERC20Base token,
    address sendTo,
    uint128 amount,
    uint64 idx
) public {
    require(msg.sender == clearinghouse);
    if (markedIdxs[idx]) { return; }
    markedIdxs[idx] = true;
    minIdx = idx;                    // ← unconditionally advanced
    handleWithdrawTransfer(token, sendTo, amount);
}
``` [3](#0-2) 

Every time the sequencer processes **any** withdrawal through the Clearinghouse, `minIdx` is advanced to that withdrawal's `nSubmissions` value. The call chain is:

`Endpoint.submitTransactionsChecked` → `EndpointTx.processTransactionImpl` (WithdrawCollateral, line 430–436) → `Clearinghouse.withdrawCollateral` → `Clearinghouse.handleWithdrawTransfer` → `BaseWithdrawPool.submitWithdrawal`. [4](#0-3) [5](#0-4) 

**Concrete censorship path:**

1. User submits a withdrawal request off-chain to the sequencer.
2. Sequencer assigns `idx = 100` (the planned `nSubmissions` slot) and produces `(idx=100, txn, sigs)`.
3. Sequencer processes other users' withdrawals at `nSubmissions = 150, 200, 250`, advancing `minIdx` to `250`.
4. Sequencer never processes the target user's withdrawal at `idx = 100` (selective omission).
5. User calls `submitFastWithdrawal(100, txn, sigs)` → `require(100 > 250)` → **permanent revert**.
6. The user's fast withdrawal claim is irrecoverably blocked; the signed data is now worthless.

The user's only remaining path is the slow-mode queue (`submitSlowModeTransaction` → `WithdrawCollateral`), which imposes a mandatory 3-day delay (`SLOW_MODE_TX_DELAY`). [6](#0-5) 

---

### Impact Explanation

A user who has obtained valid sequencer-signed fast withdrawal data for `idx = N` loses the ability to use that data permanently once the sequencer advances `minIdx` past `N` by processing any other withdrawal. The user:

- Cannot recover the fast withdrawal path for that `idx` — `minIdx` is never decremented.
- Must fall back to the 3-day slow-mode queue, losing the core value proposition of the fast withdrawal system.
- Has no on-chain mechanism to independently verify or reconstruct the correct `idx` for their withdrawal, making them structurally dependent on sequencer data provision.

The corrupted state is `minIdx` — once advanced, it permanently gates all prior fast withdrawal signatures. The asset delta is the loss of immediate liquidity access for the full withdrawal amount.

---

### Likelihood Explanation

The sequencer is a single privileged actor that controls `nSubmissions` sequencing and the production of fast withdrawal signatures. Selective omission of a specific user's withdrawal from the sequencer's processing queue — while continuing to process others — is operationally straightforward and leaves no on-chain evidence distinguishing censorship from a processing delay. The `minIdx` advancement is a side effect of normal operation (processing any other withdrawal), so the censorship window closes automatically without any special action. This makes targeted fast withdrawal censorship low-effort and deniable.

---

### Recommendation

1. **Bind `idx` to the withdrawal request on-chain**: Record the assigned `idx` in a mapping at withdrawal request time so users can independently verify and reconstruct it without relying on sequencer-provided off-chain data.
2. **Replace `minIdx` with per-withdrawal deduplication**: Instead of a global advancing cursor, use only `markedIdxs` for deduplication. The `minIdx` guard adds no security benefit that `markedIdxs` does not already provide, but it creates a permanent censorship window.
3. **Provide an SDK or public specification** documenting how `idx` is assigned so users can independently compute the expected `idx` for their withdrawal from on-chain state (`nSubmissions`).

    function requireValidTxSignatures(
        bytes calldata txn,
        uint64 idx,
        bytes[] calldata signatures
    ) public view {
        require(signatures.length <= 256, "too many signatures");
        bytes32 data = keccak256(
            abi.encodePacked(uint256(block.chainid), uint256(idx), txn)
        );
        bytes32 hashedMsg = keccak256(
            abi.encodePacked("\x19Ethereum Signed Message:\n32", data)
        );

        uint256 nSignatures = 0;
        for (uint256 i = 0; i < signatures.length; i++) {
            if (signatures[i].length > 0) {
                nSignatures += 1;
                require(
                    checkIndividualSignature(
                        hashedMsg,
                        signatures[i],
                        uint8(i)
                    ),
                    "invalid signature"
                );
            }
        }
        require(nSignatures == nSigner, "not enough signatures");
    }
