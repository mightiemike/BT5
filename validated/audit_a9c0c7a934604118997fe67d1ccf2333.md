Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the end user, allowing complete allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is `msg.sender` at the pool level — the router contract, not the end user. When a pool admin allowlists `MetricOmmSimpleRouter` for usability, every user who routes through it passes the allowlist check regardless of their own allowlist status, completely defeating the access-control invariant the extension is designed to enforce.

## Finding Description
The call chain is fully confirmed in production code:

1. `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly with no `sender` forwarding: [1](#0-0) 

2. Inside `MetricOmmPool.swap`, `msg.sender` is the router. The pool passes it verbatim as `sender` to `_beforeSwap`: [2](#0-1) 

3. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router — never the end user: [3](#0-2) 

The check `allowedSwapper[pool][router]` substitutes for `allowedSwapper[pool][user]`. No existing guard in the extension or pool recovers the real user identity. The `extensionData` field is passed through but the extension never reads it to recover the originating EOA. [4](#0-3) 

## Impact Explanation
A pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a curated set (e.g., KYC'd counterparties). Allowlisting `MetricOmmSimpleRouter` — the natural operational choice to preserve usability — causes `allowedSwapper[pool][router] = true`. Any non-allowlisted EOA can then call `exactInputSingle` on the router and execute swaps against the pool's liquidity without authorization. This is a broken core access-control invariant with direct fund impact: unauthorized parties can drain or trade against restricted pool liquidity. The inverse also holds: if the router is not allowlisted, legitimately allowlisted users cannot use the router, breaking the intended UX.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary public swap entry point. Pool admins who deploy `SwapAllowlistExtension` will almost certainly allowlist the router to allow their permitted users to swap conveniently. No special permissions, flash loans, or unusual token behavior are required — any EOA can call `exactInputSingle`. The bypass is triggered by normal, expected operational configuration.

## Recommendation
The extension must verify the identity of the actual end user, not the intermediary. Two concrete approaches:

1. **Encode the real user in `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling the pool; the extension decodes and verifies it. This requires a coordinated convention between router and extension, and the extension must also verify `msg.sender` (the pool) is a known trusted pool to prevent spoofed `extensionData`.

2. **ERC-2771-style forwarding**: The pool accepts an explicit `swapper` parameter (separate from `msg.sender`) that the router populates with its own `msg.sender` before the pool call, and the pool forwards that value as `sender` to extensions. This mirrors the `_msgSender()` pattern and cleanly separates the intermediary from the originating user.

Until fixed, `SwapAllowlistExtension` must not be used with any pool accessible through `MetricOmmSimpleRouter`.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured
  - allowAllSwappers[pool] = false
  - allowedSwapper[pool][router] = true   ← admin allowlists router for usability
  - allowedSwapper[pool][attacker] = false ← attacker is NOT allowlisted

Attack:
  1. attacker calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  2. Router calls pool.swap(recipient, ...) — msg.sender inside pool = router address
  3. Pool calls _beforeSwap(sender=router, ...)
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] → true ✓
  5. Swap executes — attacker receives output tokens

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds; allowlist completely bypassed

Foundry test outline:
  - Deploy MockPool + SwapAllowlistExtension + MetricOmmSimpleRouter
  - setAllowedToSwap(pool, router, true)
  - setAllowedToSwap(pool, attacker, false)
  - vm.prank(attacker); router.exactInputSingle(...)
  - Assert swap succeeds (demonstrating bypass)
  - Assert isAllowedToSwap(pool, attacker) == false (confirming attacker was never authorized)
```
