### Title
Slow-Mode Fee Tokens Permanently Locked in Endpoint Contract — (File: `core/contracts/EndpointStorage.sol`, `core/contracts/EndpointTx.sol`)

---

### Summary

Every non-privileged slow-mode transaction charges a quote-token fee that is physically transferred into the `Endpoint` contract. The accounting variable `slowModeFees` is incremented to track these tokens, but no function in the protocol ever decrements it or transfers those tokens out. The `DumpFees` mechanism only handles per-product `sequencerFee` balances (which are off-chain accounting entries, not real token transfers). The slow-mode fee tokens are permanently locked.

---

### Finding Description

When a user submits a slow-mode transaction via `submitSlowModeTransaction`, the call is delegated to `EndpointTx.submitSlowModeTransactionImpl`. For all non-privileged transaction types (e.g., `WithdrawCollateral`, `LinkSigner`, `ClaimBuilderFee`), the following two lines execute:

```solidity
// EndpointTx.sol lines 370-371
chargeSlowModeFee(_getQuote(), sender);
slowModeFees += SLOW_MODE_FEE;
```

`chargeSlowModeFee` is defined in `EndpointStorage.sol`:

```solidity
// EndpointStorage.sol lines 83-93
function chargeSlowModeFee(IERC20Base token, address from) internal virtual {
    require(address(token) != address(0));
    token.safeTransferFrom(
        from,
        address(this),   // <-- tokens land in the Endpoint contract
        clearinghouse.getSlowModeFee()
    );
}
```

The quote tokens are transferred to `address(this)` — the `Endpoint` contract itself. The variable `slowModeFees` (declared `internal` at `EndpointStorage.sol` line 55) is incremented but:

1. It is **never decremented** anywhere in the codebase.
2. There is **no function** in `Endpoint.sol` or `EndpointTx.sol` that transfers these tokens out of the Endpoint.
3. The `DumpFees` slow-mode transaction type (`EndpointTx.sol` lines 244–253) only processes `sequencerFee[spotIds[i]]` — per-product off-chain accounting balances charged via `chargeFee` → `spotEngine.updateBalance`. It does not touch `slowModeFees` and does not move any tokens out of the Endpoint.

The `handleDepositTransfer` path (`EndpointStorage.sol` lines 111–119) routes deposit tokens directly to the `Clearinghouse`, not to the Endpoint itself, so it is unrelated. There is no `withdraw`, `recoverFees`, or equivalent function on the Endpoint contract.

---

### Impact Explanation

Every slow-mode transaction submitted by any user permanently locks `SLOW_MODE_FEE` worth of quote tokens (USDC) inside the `Endpoint` contract. Over the protocol's lifetime, the total locked amount grows proportionally to the number of slow-mode transactions processed. These tokens are irrecoverable: no privileged or unprivileged path exists to extract them. The `slowModeFees` state variable is a dead accumulator — it is written but never read or acted upon.

---

### Likelihood Explanation

High. The trigger is any call to `submitSlowModeTransaction` with a non-privileged transaction type. This is a standard user-facing entrypoint (e.g., submitting a `WithdrawCollateral` or `LinkSigner` via slow mode). No special conditions, permissions, or attacker-controlled state are required. Every such call contributes to the locked balance.

---

### Recommendation

Add a fee-withdrawal mechanism for `slowModeFees` analogous to the existing `DumpFees` path for `sequencerFee`. Concretely:

- Extend the `DumpFees` slow-mode transaction handler in `EndpointTx.processSlowModeTransactionImpl` to also transfer the accumulated `slowModeFees` tokens out of the Endpoint (e.g., to the `X_ACCOUNT` or directly to the clearinghouse), and reset `slowModeFees` to zero.
- Alternatively, route slow-mode fee collection through `handleDepositTransfer` so the tokens land in the `Clearinghouse` from the start, consistent with how deposit collateral is handled.

---

### Proof of Concept

1. User calls `Endpoint.submitSlowModeTransaction(withdrawCollateralTx)`.
2. Endpoint delegatecalls `EndpointTx.submitSlowModeTransactionImpl`.
3. The `else` branch at line 369–372 of `EndpointTx.sol` executes:
   - `chargeSlowModeFee(_getQuote(), sender)` → `IERC20(quoteToken).safeTransferFrom(user, address(endpoint), fee)` — tokens now in Endpoint.
   - `slowModeFees += SLOW_MODE_FEE` — accounting incremented.
4. Sequencer later submits `DumpFees` transaction. `EndpointTx.processSlowModeTransactionImpl` lines 244–253 execute, clearing `sequencerFee[*]` and calling `clearinghouse.claimSequencerFees`. `slowModeFees` is untouched; no token transfer out of Endpoint occurs.
5. Repeat for every slow-mode submission. `slowModeFees` grows; the corresponding quote tokens remain locked in the Endpoint contract forever.
