### Title
Missing Zero-Address Check for `_wrappedNative` in `DirectDepositV1` Constructor Causes Permanent ETH Loss - (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.sol` is a non-upgradeable deposit adapter contract. Its constructor accepts `_wrappedNative` (the wrapped-native token contract address) without any zero-address validation. If deployed with `_wrappedNative = address(0)` — e.g., due to a front-end or deployment-script bug — the `receive()` function silently forwards all incoming ETH to `address(0)`, permanently burning it. Because the contract is non-upgradeable, there is no recovery path.

---

### Finding Description

The `DirectDepositV1` constructor stores all four address parameters directly without zero-address guards: [1](#0-0) 

The `_wrappedNative` parameter is stored as `wrappedNative` and is used in two places:

**1. Constructor body** — if the contract has an ETH balance at deploy time, it forwards it to `wrappedNative`: [2](#0-1) 

**2. `receive()` fallback** — every ETH transfer to the DDA is forwarded to `wrappedNative`: [3](#0-2) 

If `wrappedNative` is `address(0)`, the EVM call `address(0).call{value: X}("")` **succeeds** (returns `true`) because `address(0)` has no code and accepts ETH. The `require(success, ...)` check passes silently, and the ETH is permanently burned. No error is raised, no event is emitted.

The same class of missing zero check applies to `_endpoint` and `_spotEngine`: [4](#0-3) 

If either is `address(0)`, `creditDeposit()` will revert on every call, trapping all ERC-20 token balances in the DDA with no recovery path (the contract is non-upgradeable and has no `setEndpoint`/`setSpotEngine` setter).

---

### Impact Explanation

- **ETH loss**: Any ETH sent to a misconfigured DDA (with `_wrappedNative = address(0)`) is permanently burned. The `receive()` function is callable by any user and gives no indication of failure.
- **ERC-20 lock**: If `_endpoint` or `_spotEngine` is `address(0)`, `creditDeposit()` always reverts. ERC-20 tokens accumulate in the DDA and cannot be deposited or recovered (no owner-rescue path for the specific token flow).
- **Non-upgradeable**: `DirectDepositV1` inherits from `Ownable` (not `OwnableUpgradeable`) and has no proxy. A misconfigured deployment is permanent. [5](#0-4) 

---

### Likelihood Explanation

`DirectDepositV1` is deployed programmatically (via `ContractOwner` or deployment tasks). A bug in the deployment front-end or script — exactly the scenario described in the reference report — could pass an uninitialized variable as `_wrappedNative`. Because the constructor emits `DirectDepositV1Created` without validating parameters, the misconfiguration is not detectable on-chain until a user sends ETH and loses it. [6](#0-5) 

---

### Recommendation

Add zero-address guards at the top of the constructor for all four address parameters:

```solidity
constructor(
    address _endpoint,
    address _spotEngine,
    bytes32 _subaccount,
    address payable _wrappedNative
) {
    require(_endpoint      != address(0), "endpoint is zero");
    require(_spotEngine    != address(0), "spotEngine is zero");
    require(_subaccount    != bytes32(0), "subaccount is zero");
    require(_wrappedNative != address(0), "wrappedNative is zero");
    // ... rest of constructor
}
```

---

### Proof of Concept

1. Deploy `DirectDepositV1` with `_wrappedNative = address(0)` (simulating a front-end bug where an uninitialized variable is used).
2. Call `receive()` by sending 1 ETH to the deployed DDA address.
3. Observe: `address(0).call{value: 1 ether}("")` returns `(true, "")`. The `require(success, ...)` passes. The ETH is gone. No revert, no event.
4. Confirm: `address(dda).balance == 0`, `address(0).balance` increased by 1 ETH (on a local fork). The DDA is permanently broken with no upgrade path. [3](#0-2)

    constructor(
        address _endpoint,
        address _spotEngine,
        bytes32 _subaccount,
        address payable _wrappedNative
    ) {
        endpoint = IIEndpoint(_endpoint);
        spotEngine = IISpotEngine(_spotEngine);
        subaccount = _subaccount;
        wrappedNative = _wrappedNative;
        uint256 balance = address(this).balance;
        if (balance != 0) {
            // shouldn't revert even if the transfer fails, otherwise the funds
            // will be stuck in the DDA forever.
            (bool success, ) = wrappedNative.call{value: balance}("");
            if (!success) {
                emit NativeTokenTransferFailed(balance);
            }
        }
        emit DirectDepositV1Created(version(), subaccount, address(this));
    }
