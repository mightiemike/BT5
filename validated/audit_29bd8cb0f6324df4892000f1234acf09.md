### Title
Linked Signer Can Overwrite Itself via `LinkSigner` Transaction, Permanently Escalating Privileges Over a Subaccount — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction handler inside `processTransactionImpl` passes `allowLinkedSigner = true` to `validateSignedTx`. This permits the **current linked signer** — a limited delegate — to sign a `LinkSigner` transaction and replace itself with any arbitrary address. The invariant that only the subaccount owner can change the linked signer is broken, enabling a malicious linked signer to permanently escalate its authority over the subaccount.

---

### Finding Description

In `EndpointTx.sol`, the fast-mode `LinkSigner` handler at lines 576–590 calls `validateSignedTx` with the fifth argument hardcoded to `true`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true          // allowLinkedSigner = true
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [1](#0-0) 

`validateSignedTx` forwards this flag to `validateSignature`, which passes the current linked signer address to `verifier.validateSignature`:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` then accepts a signature from **either** the account owner **or** the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

Because the linked signer's signature is accepted for a `LinkSigner` transaction, the linked signer can craft and submit a `LinkSigner` transaction that sets `signedTx.tx.signer` to any attacker-controlled address, overwriting `linkedSigners[subaccount]`.

**The inconsistency is confirmed by two other code paths in the same file:**

1. The **slow-mode** `LinkSigner` handler enforces that only the account owner can change the linked signer via `validateSender`, which checks `msg.sender == address(uint160(bytes20(txn.sender)))`:

```solidity
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

2. The **`WithdrawCollateralV2`** handler deliberately restricts the linked signer by only setting `allowLinkedSigner = true` when `sendTo == address(0)` (i.e., the linked signer cannot redirect funds to an arbitrary address):

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    signedTx.tx.sendTo == address(0)
);
``` [5](#0-4) 

The `WithdrawCollateralV2` design shows the protocol authors are aware that linked signers must be restricted from performing high-privilege actions. The `LinkSigner` handler fails to apply the same restriction, creating an asymmetry: the linked signer cannot withdraw to an arbitrary address, but it **can** replace itself with an arbitrary address — which is a strictly more powerful capability.

---

### Impact Explanation

Once the linked signer overwrites itself with an attacker-controlled address, the attacker's address becomes the new linked signer. The new linked signer can sign any transaction on behalf of the subaccount that accepts `allowLinkedSigner = true`, including `WithdrawCollateral`, `WithdrawCollateralV2` (to the default address), `MintNlp`, `BurnNlp`, `TransferQuote`, and further `LinkSigner` transactions. This constitutes full, persistent takeover of the subaccount's signing authority and enables complete collateral drainage.

The corrupted state is: `linkedSigners[victim_subaccount]` is set to an attacker-controlled address without the account owner's consent.

---

### Likelihood Explanation

The linked signer is a user-set delegate address, commonly used for automated trading bots or hot wallets. The account owner expects this address to have limited, scoped authority. A malicious or independently-acting linked signer (e.g., a compromised bot key, a third-party integration, or a rogue delegate) can exploit this without any additional privilege — it only needs to submit a valid `LinkSigner` transaction signed by itself to the sequencer. The sequencer processes any transaction that passes on-chain validation; the on-chain validation explicitly permits this. No owner key, admin key, or sequencer compromise is required.

---

### Recommendation

Change `allowLinkedSigner` to `false` for `LinkSigner` transactions in `processTransactionImpl`. Only the subaccount owner (the address embedded in the `bytes32 sender`) should be permitted to change the linked signer, consistent with the slow-mode path's enforcement via `validateSender`.

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the account owner may change the linked signer
);
``` [6](#0-5) 

---

### Proof of Concept

1. Alice owns subaccount `S` and sets Bob's address as her linked signer via a `LinkSigner` transaction signed by Alice.
2. Bob (the linked signer) constructs a `SignedLinkSigner` struct with `tx.sender = S`, `tx.signer = Charlie` (attacker-controlled), and a valid nonce.
3. Bob signs the EIP-712 digest of this transaction with his own key.
4. Bob submits the signed transaction to the sequencer.
5. The sequencer includes it in a batch and calls `submitTransactionsChecked`.
6. `processTransactionImpl` reaches the `LinkSigner` branch and calls `validateSignedTx(..., true)`.
7. `validateSignature` passes Bob's address as `linkedSigner` to `verifier.validateSignature`; Bob's signature is accepted.
8. `linkedSigners[S]` is overwritten with Charlie's address.
9. Charlie can now sign `WithdrawCollateral`, `TransferQuote`, and further `LinkSigner` transactions on behalf of `S`, draining all collateral. [7](#0-6) [8](#0-7)
