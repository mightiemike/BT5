### Title
Unprivileged Caller Can Sweep Accidentally Sent Tokens via `creditDeposit()` — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` has no access control and sweeps the **entire** ERC-20 balance of the DDA contract into the hardcoded `subaccount`. This mirrors the UNCX bug class exactly: a privileged rescue path (`ContractOwner.withdrawFromDirectDepositV1`) exists to recover accidentally sent tokens, but any unprivileged caller can front-run it by calling `creditDeposit()`, permanently depositing those tokens into the DDA's subaccount and making them unrecoverable by the admin.

---

### Finding Description

`DirectDepositV1` is a per-user deposit adapter. Its `creditDeposit()` function is callable by anyone with no modifier:

```solidity
function creditDeposit() external {                          // no access control
    uint32[] memory productIds = spotEngine.getProductIds();
    for (uint256 i = 0; i < productIds.length; i++) {
        ...
        uint256 balance = token.balanceOf(address(this));   // entire contract balance
        if (balance != 0) {
            token.approve(address(endpoint), balance);
            endpoint.depositCollateralWithReferral(
                subaccount, productId, uint128(balance), "-1"
            );
        }
    }
}
``` [1](#0-0) 

The function uses `token.balanceOf(address(this))` — the **total** balance — rather than tracking only tokens that were intentionally forwarded for deposit. It then deposits that entire balance into the DDA's hardcoded `subaccount`.

The protocol owner has a dedicated rescue function:

```solidity
function withdrawFromDirectDepositV1(bytes32 subaccount, address token)
    external onlyOwner
{
    ...
    uint256 preBalance = IERC20Base(token).balanceOf(address(this));
    DirectDepositV1(directDepositV1).withdraw(IIERC20Base(token));
    uint256 postBalance = IERC20Base(token).balanceOf(address(this));
    ...
    IERC20Base(token).safeTransfer(msg.sender, postBalance - preBalance);
}
``` [2](#0-1) 

This rescue path relies on the DDA still holding the tokens. Because `creditDeposit()` is permissionless and sweeps the full balance, any caller can drain the DDA into the subaccount before the admin acts.

---

### Impact Explanation

If a supported ERC-20 token (any token registered in `SpotEngine`) is accidentally sent to a DDA — for example, a user sends to the wrong DDA address, or a protocol integration misdirects funds — the following occurs:

1. The tokens sit in the DDA, recoverable by the admin via `withdrawFromDirectDepositV1`.
2. Any unprivileged caller (including the DDA's subaccount owner) calls `creditDeposit()`.
3. `creditDeposit()` sweeps the full balance — including the accidentally sent tokens — into the DDA's `subaccount` via `endpoint.depositCollateralWithReferral`.
4. The DDA balance is now zero; `withdrawFromDirectDepositV1` finds nothing to rescue.
5. The tokens are now credited to the subaccount owner, who can withdraw them through normal protocol flows.

The accidental sender permanently loses their tokens. The admin rescue mechanism is fully bypassed. The corrupted state is: `SpotEngine` balance for the DDA's `subaccount` is inflated by the accidentally sent amount, and the admin's pre/post balance delta in `withdrawFromDirectDepositV1` is zero. [3](#0-2) 

---

### Likelihood Explanation

- DDAs are deployed per-user and their addresses are deterministic/public. Accidental token sends to contract addresses are a well-documented real-world occurrence.
- `creditDeposit()` requires no special role, no signature, no stake — any EOA can call it at any time.
- The subaccount owner is directly incentivized to call `creditDeposit()` immediately upon receiving any windfall tokens, before the admin can intervene.
- The attack requires no capital at risk for the caller (they are not the sender of the accidentally deposited tokens).

Likelihood: **Medium** — depends on accidental sends occurring, but the permissionless sweep is trivially exploitable once they do.

---

### Recommendation

Add access control to `creditDeposit()` so only the DDA's owner (i.e., `ContractOwner`) or the subaccount's controlling address can trigger it. Alternatively, track the "intended deposit amount" separately (e.g., via a mapping updated when tokens are transferred in through a controlled path) and deposit only that tracked amount rather than `balanceOf(address(this))`.

```solidity
// Option A: restrict caller
function creditDeposit() external onlyOwner { ... }

// Option B: snapshot-based approach
// record balance before transfer, deposit only the delta
``` [1](#0-0)

### Proof of Concept

1. User A accidentally sends 1000 USDC to User B's `DirectDepositV1` DDA address.
2. Admin prepares to call `ContractOwner.withdrawFromDirectDepositV1(subaccountB, USDC)` to rescue the funds.
3. User B (or any observer) front-runs the admin by calling `DirectDepositV1(ddaB).creditDeposit()`.
4. `creditDeposit()` reads `USDC.balanceOf(ddaB) = 1000`, approves the endpoint, and calls `endpoint.depositCollateralWithReferral(subaccountB, usdcProductId, 1000, "-1")`.
5. The endpoint pulls 1000 USDC from the DDA into the Clearinghouse and credits User B's subaccount.
6. Admin's `withdrawFromDirectDepositV1` call now finds `preBalance == postBalance` (DDA is empty) and reverts on `require(postBalance > preBalance, "empty")`.
7. User B withdraws 1000 USDC from their subaccount through normal protocol flows. [1](#0-0) [2](#0-1)
