### Title
`ENABLE_SECP_OPS` Flag Not Checked on 4-Byte Secp Opcode Path — (`File: src/chia_dialect.rs`)

### Summary
`ChiaDialect::op()` dispatches secp256k1/r1 verification through two independent opcode paths. The 1-byte alias path (opcodes 64, 65) correctly gates execution behind `ClvmFlags::ENABLE_SECP_OPS`. The 4-byte opcode path (`0x13d61f00`, `0x1c3a8f00`) dispatches the same functions unconditionally, without checking `ENABLE_SECP_OPS`. Any caller that relies on the absence of `ENABLE_SECP_OPS` to prevent secp verification is bypassed by attacker-crafted CLVM bytes that use the 4-byte opcode form.

### Finding Description
In `src/chia_dialect.rs`, `ChiaDialect::op()` handles 4-byte opcodes first:

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,
        0x1c3a8f00 => op_secp256r1_verify,
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
``` [1](#0-0) 

No flag check precedes the dispatch. The 1-byte alias path, reached only when `op_len == 1`, does enforce the flag:

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

`ENABLE_SECP_OPS` is defined as a distinct, exported flag:

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
``` [3](#0-2) 

It is exported to Python callers via the wheel API:

```rust
m.add("ENABLE_SECP_OPS", ClvmFlags::ENABLE_SECP_OPS.bits())?;
``` [4](#0-3) 

Critically, `ENABLE_SECP_OPS` is **not** included in `MEMPOOL_MODE`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [5](#0-4) 

This means in mempool mode, 1-byte opcodes 64/65 fall through to `unknown_operator` and are rejected (because `NO_UNKNOWN_OPS` is set), while 4-byte opcodes `0x13d61f00`/`0x1c3a8f00` execute `op_secp256k1_verify`/`op_secp256r1_verify` unconditionally.

### Impact Explanation
The broken invariant is: `ENABLE_SECP_OPS` is supposed to be the gate controlling whether secp verification is available. Because the 4-byte opcode path skips this check, any CLVM program using the 4-byte encoding of the secp opcodes will execute secp verification regardless of whether `ENABLE_SECP_OPS` is set. A downstream caller (e.g., `chia_rs` or a custom validator) that constructs a `ChiaDialect` without `ENABLE_SECP_OPS` expecting secp ops to be unavailable will have that expectation silently violated. In mempool mode specifically, the 4-byte secp opcodes execute while the 1-byte aliases are rejected, creating an asymmetric acceptance surface that can be exploited to craft transactions that pass mempool validation via the 4-byte path while behaving differently than the caller's flag configuration intends.

### Likelihood Explanation
The attacker-controlled entry path is direct: `run_serialized_chia_program` accepts raw CLVM bytes from the caller, which are deserialized and executed. An attacker who controls the CLVM program bytes can trivially encode the 4-byte opcode `0x13d61f00` or `0x1c3a8f00` as the operator atom. The `Allocator` will store it as a 4-byte atom, `op_len == 4` will be true, and the dispatch will proceed without any flag check. No special privileges or configuration are required beyond the ability to submit a CLVM program.

### Recommendation
Add an `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch path in `ChiaDialect::op()`, mirroring the check already present for the 1-byte aliases:

```rust
let f = match opcode {
    0x13d61f00 => {
        if !flags.contains(ClvmFlags::ENABLE_SECP_OPS) {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
        op_secp256k1_verify
    }
    0x1c3a8f00 => {
        if !flags.contains(ClvmFlags::ENABLE_SECP_OPS) {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
        op_secp256r1_verify
    }
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

Alternatively, if the 4-byte form is intentionally always available, the `ENABLE_SECP_OPS` flag documentation and name should be corrected to reflect that it only gates the 1-byte aliases, and downstream callers should be warned that the flag does not fully disable secp verification.

### Proof of Concept
1. Construct a CLVM program whose operator atom is the 4-byte encoding `[0x13, 0xd6, 0x1f, 0x00]` (secp256k1_verify) with valid pubkey/msg/sig arguments.
2. Call `run_serialized_chia_program` with `flags = MEMPOOL_MODE` (which does not include `ENABLE_SECP_OPS`).
3. Observe that `op_secp256k1_verify` executes and returns a result — secp verification ran despite `ENABLE_SECP_OPS` being absent from the flags.
4. Repeat with the 1-byte opcode 64 as the operator atom; observe that it is rejected as an unknown op (because `NO_UNKNOWN_OPS` is set in `MEMPOOL_MODE` and `ENABLE_SECP_OPS` is absent), confirming the asymmetry between the two dispatch paths.
