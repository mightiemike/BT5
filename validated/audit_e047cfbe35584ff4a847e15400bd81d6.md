### Title
Sanctioned Recipient Bypasses Sanctions Check in Fast Withdrawal Path - (`core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`submitFastWithdrawal` in `BaseWithdrawPool` transfers funds to a `sendTo` address decoded from the transaction payload without ever checking whether that address is sanctioned. The protocol enforces sanctions on both the depositor and the subaccount owner during deposits, but the symmetric check is entirely absent on the withdrawal recipient in the fast-withdrawal path.

---

### Finding Description

`depositCollateralWithReferral` in `Endpoint.sol` explicitly enforces sanctions on both the caller and the subaccount owner before crediting funds:

```solidity
// depositor / depositee need to be unsanctioned
requireUnsanctioned(msg.sender);
requireUnsanctioned(sender);
``` [1](#0-0) 

However, `submitFastWithdrawal` in `BaseWithdrawPool` performs no such check on the actual recipient. The `sendTo` address is decoded from the caller-supplied `transaction` bytes via `resolveFastWithdrawal`:

```solidity
(
    uint32 productId,
    address sendTo,
    uint128 transferAmount
) = resolveFastWithdrawal(transaction);
...
handleWithdrawTransfer(token, sendTo, transferAmount);
``` [2](#0-1) 

`resolveFastWithdrawal` resolves `sendTo` from the signed transaction data — either as the subaccount owner's address (V1) or as an explicit `sendTo` field (V2):

```solidity
address resolvedSendTo = signedTx.tx.sendTo == address(0)
    ? address(uint160(bytes20(signedTx.tx.sender)))
    : signedTx.tx.sendTo;
return (signedTx.tx.productId, resolvedSendTo, signedTx.tx.amount);
``` [3](#0-2) 

Neither `sendTo` nor `msg.sender` is passed through `requireUnsanctioned` at any point in `submitFastWithdrawal`. The `WithdrawPool` concrete contract adds no overrides: [4](#0-3) 

The `requireUnsanctioned` helper exists and is used elsewhere:

```solidity
function requireUnsanctioned(address sender) internal view virtual {
    require(!sanctions.isSanctioned(sender), ERR_WALLET_SANCTIONED);
}
``` [5](#0-4) 

---

### Impact Explanation

A sanctioned address can receive collateral tokens via the fast-withdrawal path. The protocol's sanctions enforcement — which blocks sanctioned addresses from depositing — is completely absent on the withdrawal side. This allows a sanctioned user (or any party directing funds to a sanctioned address via the V2 `sendTo` field) to extract funds from the protocol, violating the protocol's compliance invariant and potentially exposing the protocol to regulatory liability.

---

### Likelihood Explanation

`submitFastWithdrawal` is a `public` function with no access-control modifier. Any caller can invoke it with a validly-signed transaction. A sanctioned user who had funds in the protocol before being sanctioned (or whose subaccount was funded by a third party) can trigger this path directly. The `WithdrawCollateralV2` path additionally allows an arbitrary `sendTo` to be specified in the signed transaction, meaning even a third party can route funds to a sanctioned address. The only prerequisite is a valid `requireValidTxSignatures` check, which is satisfied by the normal sequencer-signed withdrawal flow.

---

### Recommendation

Add a `requireUnsanctioned` check on `sendTo` inside `submitFastWithdrawal`, after `resolveFastWithdrawal` resolves the recipient:

```diff
 (
     uint32 productId,
     address sendTo,
     uint128 transferAmount
 ) = resolveFastWithdrawal(transaction);
+requireUnsanctioned(sendTo);
 IERC20Base token = getToken(productId);
``` [6](#0-5) 

Since `BaseWithdrawPool` does not inherit `EndpointStorage`, the `requireUnsanctioned` logic should either be duplicated or the sanctions list address should be stored and exposed in `BaseWithdrawPool` similarly to how it is in `EndpointStorage`.

---

### Proof of Concept

1. User `A` is added to the sanctions list after having deposited collateral.
2. The sequencer produces a signed `WithdrawCollateral` (or `WithdrawCollateralV2`) transaction for user `A`'s subaccount.
3. Any caller (including `A` themselves) calls `WithdrawPool.submitFastWithdrawal(idx, transaction, signatures)`.
4. `resolveFastWithdrawal` decodes `sendTo = address(A)` from the transaction.
5. No sanction check is performed on `sendTo`.
6. `handleWithdrawTransfer` transfers tokens directly to `A`, bypassing the protocol's sanctions enforcement. [7](#0-6)

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
```
