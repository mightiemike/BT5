### Title
Fork Difficulty Calculation Uses Mainchain Ancestor Instead of Fork Ancestor, Enabling Difficulty Manipulation - (File: `contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

### Summary
When validating a fork block's difficulty target, `get_header_by_height` always returns the **mainchain** block at the requested height rather than the fork's actual ancestor. This causes the difficulty retarget formula to use the wrong timestamp, producing an incorrect expected difficulty for fork blocks. An attacker can exploit this to submit fork blocks whose difficulty does not match the protocol-correct target, corrupting the difficulty-validation invariant.

### Finding Description

During `submit_block_header`, when a block extends a fork (not the mainchain tip), `check_pow` → `get_next_work_required` is called to compute the expected difficulty. At a retarget boundary, the code fetches the boundary ancestor via `get_header_by_height`:

**Bitcoin** (`contract/src/bitcoin.rs`, lines 78–86):
```rust
let first_block_height =
    prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
```

**Dogecoin** (`contract/src/dogecoin.rs`, lines 286–297):
```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

The `get_header_by_height` implementation in `contract/src/lib.rs` (lines 677–682) exclusively consults `mainchain_height_to_header`:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

When the fork diverges **before** `height_first`, the fork's actual ancestor at that height is a different block than the mainchain block stored in `mainchain_height_to_header`. The timestamps differ, so the retarget formula receives the wrong `first_block_time`, producing an incorrect expected `bits` value.

The codebase itself acknowledges this with the TODO comment at `contract/src/dogecoin.rs` line 291.

### Impact Explanation

The difficulty retarget formulas in all three chains compute a new target as:

```
new_target = prev_target * actual_timespan / target_timespan
```

where `actual_timespan = prev_block_time - first_block_time`.

If the mainchain block at `height_first` has an **earlier** timestamp (`T_main`) than the fork's actual ancestor at that height (`T_fork`, where `T_fork > T_main`), then:

- Code computes: `actual_timespan = T_{prev} - T_main` (too large)
- Correct value: `actual_timespan = T_{prev} - T_fork` (smaller)
- Result: expected target is **higher** (difficulty is **lower**) than protocol-correct

An attacker can craft a fork where the fork's ancestor at `height_first` carries a later timestamp than the mainchain block at that height (within MTP constraints). The contract then accepts fork blocks whose `bits` field encodes a lower difficulty than the protocol requires. Those blocks pass both the `expected_bits == block_header.bits` check and the PoW hash check (since the target is inflated), but they do not represent the correct amount of work.

Because `chain_work` is accumulated from the block's actual `bits` field (`work_from_bits(block_header.bits)`), blocks with an artificially lowered difficulty carry less chainwork per block. However, the attacker can mine them faster with less hashpower, potentially accumulating enough total chainwork to trigger a reorg at a lower real-world cost than the protocol intends.

The corrupted state is: **the canonical chain can be reorganized using fork blocks that do not satisfy the correct difficulty target**, breaking the core security invariant that chainwork reflects real accumulated PoW.

### Likelihood Explanation

**Dogecoin** (highest likelihood): After block 145,000, `difficulty_adjustment_interval = 1`, meaning every block is a retarget boundary and `height_first = prev_block_height - 1`. Any fork that diverges two or more blocks back immediately triggers the wrong-ancestor lookup. A relayer or any unprivileged NEAR caller invoking `submit_blocks` with a crafted fork sequence can reach this path with minimal setup.

**Bitcoin / Litecoin** (lower likelihood): The retarget interval is 2016 blocks. The wrong-ancestor lookup only matters when the fork diverges before the start of the current 2016-block period, requiring the attacker to supply a fork spanning a full difficulty epoch. This is a much higher bar but remains reachable in principle.

### Recommendation

Replace `get_header_by_height` calls inside difficulty calculation with a proper ancestor traversal that walks the fork's own chain of `prev_block_hash` links back to `height_first`. This is the approach used by Bitcoin Core (`GetAncestor`). The traversal must use `headers_pool` (which stores all submitted headers, including fork blocks) rather than `mainchain_height_to_header` (which only stores mainchain blocks).

Alternatively, store the fork's ancestor hash at the retarget boundary inside `ExtendedHeader` at submission time, so it can be retrieved in O(1) without traversal.

### Proof of Concept

1. Initialize the contract with Dogecoin mainnet at height 145,001 (per-block retarget active).
2. Submit mainchain blocks 145,001 and 145,002 with timestamps `T_main_1` and `T_main_2`.
3. Submit a fork block at height 145,002 (diverging from 145,001) with timestamp `T_fork_2 > T_main_2`.
4. Submit a fork block at height 145,003 whose `bits` field encodes the difficulty computed using `T_main_2` (the mainchain block's timestamp) rather than `T_fork_2` (the fork's actual ancestor's timestamp). Because `T_main_2 < T_fork_2`, the contract computes a lower expected difficulty and accepts the block, even though the block's `bits` do not satisfy the protocol-correct retarget.
5. Repeat to accumulate chainwork and trigger `reorg_chain`, promoting the fork to the canonical chain.
