Audit Report

## Title
SwapAllowlistExtension Allowlist Bypassed via Router — Any EOA Can Swap Through `MetricOmmSimpleRouter` When Router Is Allowlisted - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` from `MetricOmmPool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is always the router address, not the originating EOA. Any pool admin who allowlists the router so their intended users can trade via the router inadvertently opens the gate to every EOA on the network, making the allowlist completely unenforceable for router-mediated swaps.

## Finding Description

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool (the caller of the extension) and `sender` is whoever called `pool.swap()`. [1](#0-0) 

`MetricOmmPool.swap` passes `msg.sender` (the immediate caller) as `sender` to `_beforeSwap`. [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` makes a plain external call to `pool.swap()`, so the pool sees `msg.sender = router`, not the originating EOA. [3](#0-2) 

`multicall` uses `delegatecall` which preserves `msg.sender` as the EOA inside `exactInputSingle`, but the subsequent external call to `pool.swap()` still presents `msg.sender = router` to the pool — so multicall does not change the fundamental issue. [4](#0-3) 

The two broken states are:
- **Allowlists specific EOAs only (not router):** No EOA can use the router with this pool — intended users are locked out.
- **Allowlists the router (to enable router usage):** Every EOA on the network can bypass the allowlist.

There is no configuration that simultaneously (a) allows intended EOAs to use the router and (b) blocks non-allowlisted EOAs from using the router. [5](#0-4) 

## Impact Explanation

Pool curation via `SwapAllowlistExtension` is completely broken for any pool that uses `MetricOmmSimpleRouter`. A non-allowlisted EOA can always trade in a curated pool by routing through the router, as long as the pool admin has allowlisted the router address (the only way to enable router usage for intended users). This constitutes broken core pool functionality with direct fund impact: the pool's curated trading restriction is unenforceable, allowing arbitrary actors to trade against LP positions that were intended to be restricted. [6](#0-5) 

## Likelihood Explanation

The router is the primary user-facing entry point for swaps. Any pool admin who deploys a `SwapAllowlistExtension`-gated pool and wants their allowlisted users to be able to use the router will allowlist the router address — this is the natural and expected configuration. The bypass is therefore reachable in any realistic production deployment of a curated pool. No special privileges or unusual conditions are required; any EOA can exploit this by simply calling `router.exactInputSingle()` or `router.multicall([exactInputSingle(...)])`. [7](#0-6) 

## Recommendation

The extension must receive the originating EOA, not the immediate `msg.sender` of `pool.swap()`. Two viable approaches:

1. **Pass the original payer through `extensionData`**: The router encodes `msg.sender` (the EOA) into `extensionData`; `SwapAllowlistExtension.beforeSwap` decodes and checks it. This requires a convention between router and extension, and the extension should only trust the originator value when `sender` is a known, trusted router registered with the factory.

2. **Add an `originator` field to the swap hook signature**: The pool passes both `sender` (immediate caller) and an `originator` (set by the router via a separate transient-storage slot or an explicit parameter), and the extension checks `originator`. Either approach must be authenticated. [6](#0-5) 

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Foundry integration test sketch
// Setup: pool with SwapAllowlistExtension; only `allowedEOA` is allowlisted (not the router).
// The pool admin then allowlists the router so that `allowedEOA` can use it.
// Attack: `attackerEOA` (not allowlisted) calls router.exactInputSingle(...).

function test_allowlistBypass_viaRouter() public {
    // 1. Pool admin allowlists the router (required for any user to use the router)
    swapExtension.setAllowedToSwap(address(pool), address(router), true);

    // 2. Confirm direct swap by attackerEOA reverts
    vm.prank(attackerEOA);
    vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
    pool.swap(attackerEOA, true, 1000, 0, "", "");

    // 3. attackerEOA bypasses allowlist via router
    vm.prank(attackerEOA);
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token0),
        tokenOut: address(token1),
        zeroForOne: true,
        amountIn: 1000,
        amountOutMinimum: 0,
        recipient: attackerEOA,
        deadline: block.timestamp + 1,
        priceLimitX64: 0,
        extensionData: ""
    }));
    // succeeds — pool sees msg.sender = router, allowedSwapper[pool][router] = true
}
```

The swap succeeds because `pool.swap` sees `msg.sender = router`, and `allowedSwapper[pool][router] = true`, while `attackerEOA` is never checked. [8](#0-7) [1](#0-0)
