### Title
`ConsensusTransaction::L1Handler` Structurally Drops `paid_fee_on_l1`; Hardcoded `Fee(1)` Bypasses Fee Enforcement and Produces Wrong Cende Blob Values - (File: crates/apollo_transaction_converter/src/transaction_converter.rs)

### Summary

`ConsensusTransaction::L1Handler` wraps `transaction::L1HandlerTransaction`, a type that has no `paid_fee_on_l1` field. When a validator converts a received consensus L1Handler transaction into its internal executable form, the converter unconditionally substitutes `Fee(1)` for the missing field. The blockifier fee check only tests `paid_fee != Fee(0)`, so `Fee(1)` always passes. The actual fee paid on L1 is permanently lost, and the cende blob records the wrong value for every L1Handler transaction sequenced through consensus.

### Finding Description

`ConsensusTransaction` is defined as:

```rust
pub enum ConsensusTransaction {
    RpcTransaction(RpcTransaction),
    L1Handler(transaction::L1HandlerTransaction),  // no paid_fee_on_l1
}
``` [1](#0-0) 

`transaction::L1HandlerTransaction` carries no `paid_fee_on_l1` field. The executable form, `executable_transaction::L1HandlerTransaction`, does carry it:

```rust
pub struct L1HandlerTransaction {
    pub tx: crate::transaction::L1HandlerTransaction,
    pub tx_hash: TransactionHash,
    pub paid_fee_on_l1: Fee,
}
``` [2](#0-1) 

The conversion from `ConsensusTransaction::L1Handler` to `InternalConsensusTransaction::L1Handler` is:

```rust
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
``` [3](#0-2) 

The protobuf serialization of `ConsensusTransaction::L1Handler` also omits `paid_fee_on_l1`:

```rust
impl From<L1HandlerTransaction> for protobuf::L1HandlerV0 {
    fn from(value: L1HandlerTransaction) -> Self {
        Self {
            nonce: Some(value.nonce.0.into()),
            address: Some(value.contract_address.into()),
            entry_point_selector: Some(value.entry_point_selector.0.into()),
            calldata: value.calldata.0.iter().map(|calldata| (*calldata).into()).collect(),
        }
    }
}
``` [4](#0-3) 

The field is structurally absent from the wire format, so no amount of patching the converter can recover the real value without a protocol change.

The blockifier fee enforcement check is:

```rust
let paid_fee = self.paid_fee_on_l1;
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(...));
}
``` [5](#0-4) 

`Fee(1)` is always non-zero, so the check unconditionally passes for every L1Handler transaction processed through consensus.

The cende blob serializes `paid_fee_on_l1` directly from the executable transaction:

```rust
impl From<L1HandlerTransaction> for CentralL1HandlerTransaction {
    fn from(tx: L1HandlerTransaction) -> CentralL1HandlerTransaction {
        CentralL1HandlerTransaction {
            hash_value: tx.tx_hash,
            ...
            paid_fee_on_l1: tx.paid_fee_on_l1,
        }
    }
}
``` [6](#0-5) 

Every L1Handler transaction committed through consensus will have `paid_fee_on_l1 = 1` in the cende blob, regardless of the actual fee paid on L1.

### Impact Explanation

Two concrete impacts:

1. **Wrong cende blob value**: Every L1Handler transaction sequenced through consensus produces a `CentralL1HandlerTransaction` with `paid_fee_on_l1 = 1` instead of the actual fee. This is a wrong receipt/L1 message value committed to the prover blob.

2. **Fee enforcement bypass**: The only fee check (`paid_fee != Fee(0)`) is trivially satisfied by the hardcoded `Fee(1)`. If an L1Handler with a zero actual fee were ever admitted (e.g., through a bug in the L1 contract or a future protocol change), it would execute without error.

### Likelihood Explanation

Every L1Handler transaction processed through the consensus path (i.e., on a validator node receiving a proposal) triggers this. It is not attacker-controlled — it is a structural invariant violation that fires unconditionally for all L1Handler transactions in consensus.

### Recommendation

Add `paid_fee_on_l1: Fee` to `ConsensusTransaction::L1Handler` (or to a wrapper type), propagate it through the protobuf `L1HandlerV0` message, and use the real value in `convert_consensus_l1_handler_to_internal_l1_handler` instead of the hardcoded `Fee(1)`. Until the protocol change is made, the cende blob will always record wrong `paid_fee_on_l1` values for L1Handler transactions.

### Proof of Concept

1. A proposer node receives an L1Handler transaction from the L1 scraper with `paid_fee_on_l1 = 500_000`.
2. The proposer includes it in a `ConsensusTransaction::L1Handler(transaction::L1HandlerTransaction { ... })` — `paid_fee_on_l1` is dropped here because the type has no such field.
3. A validator receives the protobuf `ConsensusTransaction`, deserializes it, and calls `convert_consensus_tx_to_internal_consensus_tx`.
4. The `L1Handler` arm calls `convert_consensus_l1_handler_to_internal_l1_handler(tx)`, which calls `executable_transaction::L1HandlerTransaction::create(tx, &self.chain_id, Fee(1))`.
5. The resulting `InternalConsensusTransaction::L1Handler` has `paid_fee_on_l1 = 1`.
6. The blockifier executes it; `paid_fee == Fee(1) != Fee(0)` → fee check passes.
7. The cende blob records `paid_fee_on_l1 = 1` instead of `500_000`.
