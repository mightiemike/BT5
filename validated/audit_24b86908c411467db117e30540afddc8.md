### Title
`setWithdrawPool` Unconditionally Replaces Pool Reference Without Migrating `markedIdxs` State, Enabling Double-Payment of Fast Withdrawals — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.setWithdrawPool` replaces the active `withdrawPool` address unconditionally without migrating the `markedIdxs` anti-replay state from the old pool to the new one. Any fast withdrawal already processed on the old pool will be processed a second time by the sequencer on the new pool, resulting in double-payment of user funds.

---

### Finding Description

The `Clearinghouse` contract holds a single `withdrawPool` address. All withdrawal token flows pass through it:

```solidity
// Clearinghouse.sol L377-385
function handleWithdrawTransfer(
    IERC20Base token,
    address to,
    uint128 amount,
    uint64 idx
) internal virtual {
    token.safeTransfer(withdrawPool, uint256(amount));
    BaseWithdrawPool(withdrawPool).submitWithdrawal(token, to, amount, idx);
}
``` [1](#0-0) 

The `BaseWithdrawPool` uses a `markedIdxs` mapping to prevent double-payment: when the sequencer processes a withdrawal, `submitWithdrawal` marks the idx so it cannot be paid out again. The fast-withdrawal path (`submitFastWithdrawal`) also marks the same idx, so when the sequencer later processes the same transaction, `submitWithdrawal` sees the idx already marked and returns early:

```solidity
// BaseWithdrawPool.sol L116-132
function submitWithdrawal(..., uint64 idx) public {
    require(msg.sender == clearinghouse);
    if (markedIdxs[idx]) {
        return;          // <-- anti-replay guard
    }
    markedIdxs[idx] = true;
    minIdx = idx;
    handleWithdrawTransfer(token, sendTo, amount);
}
``` [2](#0-1) 

The pool-switching function is:

```solidity
// Clearinghouse.sol L750-753
function setWithdrawPool(address _withdrawPool) external onlyOwner {
    require(_withdrawPool != address(0));
    withdrawPool = _withdrawPool;
}
``` [3](#0-2) 

The new pool is a fresh contract with an empty `markedIdxs` mapping and `minIdx = 0`. It has no knowledge of which indices were already processed on the old pool. The anti-replay invariant is therefore broken across the pool boundary.

Additionally, `ContractOwner.removeWithdrawPoolLiquidity` always queries `clearinghouse.getWithdrawPool()` for the current pool address, meaning it can only operate on the new pool after the switch — any ERC-20 tokens already transferred to the old pool but not yet distributed to users are stranded there and require a direct call to the old pool's `removeLiquidity`. [4](#0-3) 

---

### Impact Explanation

**Double-payment of fast withdrawals.** A user who submitted a fast withdrawal on the old pool (receiving tokens early) will receive the same tokens a second time when the sequencer processes the same withdrawal transaction against the new pool. Each such double-payment drains the new pool's token balance, ultimately at the expense of other users whose withdrawals cannot be fulfilled.

**Stranded tokens.** Any tokens already transferred to the old pool (from in-flight sequencer-processed withdrawals) are no longer reachable through the standard `removeWithdrawPoolLiquidity` admin path, requiring direct interaction with the old pool contract.

---

### Likelihood Explanation

Pool upgrades are a routine operational event (bug fixes, feature upgrades). The `submitFastWithdrawal` function is publicly callable by any address that holds valid sequencer-signed transaction data. Fast withdrawals are a core protocol feature with active usage. The window between a fast withdrawal being processed on the old pool and the sequencer settling the same transaction on-chain is non-zero by design (the sequencer batches transactions). Any pool switch during this window triggers the double-payment.

---

### Recommendation

1. **Migrate `minIdx`**: Before activating the new pool, set its `minIdx` to at least the old pool's current `minIdx`. This prevents replay of any idx that the old pool has already seen.
2. **Drain before switch**: Require that all pending fast withdrawals have been settled by the sequencer (i.e., the sequencer's `nSubmissions` has caught up to the old pool's `minIdx`) before allowing `setWithdrawPool` to proceed.
3. **Migrate `markedIdxs`**: Provide a migration function that copies the relevant portion of `markedIdxs` from the old pool to the new pool before the switch is finalized.

---

### Proof of Concept

1. Old pool is active. User calls `submitFastWithdrawal(idx=500, ...)` on the old pool. Old pool marks `markedIdxs[500] = true` and transfers tokens to the user.
2. Admin calls `setWithdrawPool(newPool)`. `Clearinghouse.withdrawPool` now points to `newPool`, which has `markedIdxs[500] = false` and `minIdx = 0`.
3. Sequencer processes the batch containing the withdrawal at `nSubmissions = 500`. It calls `clearinghouse.withdrawCollateral(...)`, which calls `handleWithdrawTransfer(token, user, amount, 500)`.
4. `handleWithdrawTransfer` transfers tokens from `Clearinghouse` to `newPool` and calls `newPool.submitWithdrawal(token, user, amount, 500)`.
5. `newPool` checks `markedIdxs[500]` → `false`. It marks it `true` and calls `handleWithdrawTransfer(token, user, amount)`, sending tokens to the user a second time.
6. The user has received the withdrawal amount twice. The new pool's token balance is depleted by one withdrawal amount that was never backed by a corresponding deposit. [5](#0-4) [1](#0-0) [3](#0-2)
