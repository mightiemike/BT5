### Title
Wrong `ClassHash::default()` in BroadcastedDeclareV1 Conversion Produces Divergent Transaction Hash in RPC Simulation/Fee Estimation — (`crates/apollo_rpc/src/v0_8/api/mod.rs`)

---

### Summary

The `TryFrom<BroadcastedDeclareTransaction> for ExecutableTransactionInput` conversion for the V1 arm deliberately sets `class_hash: ClassHash::default()` (i.e., `0x0`) instead of computing the real class hash from the supplied `contract_class` program. Because `class_hash` is a direct input to `get_declare_transaction_v1_hash`, the `tx_hash` computed by `calc_tx_hash` diverges from the hash the account actually signed. Any call to `starknet_estimateFee` or `starknet_simulateTransactions` with a DeclareV1 transaction is affected.

---

### Finding Description

**Conversion sets `class_hash` to zero:** [1](#0-0) 

The comment acknowledges this is a workaround: "The blockifier doesn't need the class hash, but it uses the SN_API `DeclareTransactionV0V1` which requires it." No attempt is made to derive the real class hash from `sn_api_contract_class`.

**`class_hash` is a direct preimage input to the V1 hash:** [2](#0-1) 

The hash is `H(declare, version, sender, 0, H(class_hash), max_fee, chain_id, nonce)`. With `class_hash = 0x0`, the inner hash becomes `H(0x0)` instead of `H(real_class_hash)`, producing a completely different `TransactionHash`.

**`calc_tx_hash` uses this zeroed struct:** [3](#0-2) 

`apply_on_transaction` wraps the `DeclareTransactionV0V1` (with `class_hash=0x0`) into `Transaction::Declare(DeclareTransaction::V1(tx))` and calls `get_transaction_hash`, which dispatches to `get_declare_transaction_v1_hash`. [4](#0-3) 

**The wrong hash is passed to the blockifier:** [5](#0-4) 

`tx_hash` (computed with `class_hash=0x0`) is passed directly to `BlockifierTransaction::from_api`. This is the hash the blockifier places in the execution context and exposes to `__validate__` via `get_tx_info().transaction_hash`.

**The state diff also uses the zeroed class hash:** [6](#0-5) 

`deprecated_declared_class_hash` is extracted from `DeclareTransactionV0V1.class_hash`, which is `0x0`, so the induced state diff records the class as declared under hash `0x0` rather than the real class hash.

---

### Impact Explanation

Two concrete wrong-value scenarios for any unprivileged caller:

1. **`validate=true` (default for `starknet_estimateFee`):** The account's `__validate__` receives a `tx_hash` computed with `class_hash=0x0`. Standard accounts (OpenZeppelin, Argent, etc.) verify the ECDSA signature against `tx_hash`. Since the user signed with the real class hash in the preimage, signature verification fails. The RPC returns an authoritative-looking "reverted" or "invalid signature" result for a perfectly valid transaction — a false negative that misleads the caller about transaction validity.

2. **`validate=false` (`starknet_simulateTransactions` with skip-validate):** Execution proceeds with the wrong `tx_hash` in the execution context and a wrong `class_hash=0x0` in the state diff. Any contract logic that reads `get_tx_info().transaction_hash` (e.g., for replay protection or event emission) observes the wrong value. The simulation trace and state diff are authoritative-looking but incorrect.

---

### Likelihood Explanation

- Reachable by any unprivileged user via standard JSON-RPC calls (`starknet_estimateFee`, `starknet_simulateTransactions`).
- DeclareV1 (Cairo 0) is deprecated but still accepted by RPC v0.8 per the spec.
- No authentication, rate-limiting, or guard prevents reaching this path.
- The divergence is deterministic and reproducible for any non-trivial `contract_class`.

---

### Recommendation

Compute the real class hash from `sn_api_contract_class` before constructing `DeclareTransactionV0V1`. The deprecated class hash for Cairo 0 contracts is defined as the Pedersen hash of the contract definition. Replace:

```rust
class_hash: ClassHash::default(),
```

with:

```rust
class_hash: compute_deprecated_class_hash(&sn_api_contract_class)
    .map_err(internal_server_error)?,
```

where `compute_deprecated_class_hash` is the canonical Cairo 0 class hash function already used elsewhere in the codebase (e.g., in the gateway path).

---

### Proof of Concept

```
1. Construct BroadcastedDeclareV1Transaction with a non-trivial contract_class
   (any Cairo 0 program with at least one entry point).

2. Call TryFrom<BroadcastedDeclareTransaction> for ExecutableTransactionInput (V1 arm).
   → Observe: declare_tx.class_hash == ClassHash::default() == 0x0

3. Call calc_tx_hash on the resulting ExecutableTransactionInput::DeclareV1.
   → hash_with_zero = get_declare_transaction_v1_hash(tx_with_zero_class_hash, chain_id, v1)

4. Independently compute the real class hash from the contract_class program,
   set it on a DeclareTransactionV0V1, and call get_declare_transaction_v1_hash.
   → hash_with_real = get_declare_transaction_v1_hash(tx_with_real_class_hash, chain_id, v1)

5. Assert hash_with_zero != hash_with_real  ← divergence confirmed.

6. Submit via starknet_estimateFee with validate=true:
   → __validate__ receives hash_with_zero; user signed hash_with_real → signature fails.
   → RPC returns "invalid signature" for a valid transaction.
```

pub(crate) fn get_declare_transaction_v1_hash(
    transaction: &DeclareTransactionV0V1,
    chain_id: &ChainId,
    transaction_version: &TransactionVersion,
) -> Result<TransactionHash, StarknetApiError> {
    Ok(TransactionHash(
        HashChain::new()
        .chain(&DECLARE)
        .chain(&transaction_version.0)
        .chain(transaction.sender_address.0.key())
        .chain(&Felt::ZERO) // No entry point selector in declare transaction.
        .chain(&HashChain::new().chain(&transaction.class_hash.0).get_pedersen_hash())
        .chain(&transaction.max_fee.0.into())
        .chain(&Felt::try_from(chain_id)?)
        .chain(&transaction.nonce.0)
        .get_pedersen_hash(),
    ))
}
