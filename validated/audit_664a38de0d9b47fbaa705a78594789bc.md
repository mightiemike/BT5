### Title
`ValidResourceBounds` Protobuf Deserialization Silently Downgrades `AllResources` to `L1Gas`, Causing P2P Transaction Rejection ŌĆö (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion uses a value-based heuristic (`l1_data_gas.is_zero() && l2_gas.is_zero()`) to distinguish legacy `L1Gas` (pre-0.13.3) from new `AllResources` (0.13.3+) transactions. A valid `AllResources` invoke V3 transaction with zero `l2_gas` and `l1_data_gas` bounds is silently downgraded to the `L1Gas` variant during protobuf deserialization. A downstream conversion step then hard-rejects any non-`AllResources` variant, causing every receiving peer to drop the transaction during P2P propagation even though the originating gateway accepted it.

### Finding Description

**Root cause ŌĆö value-based variant discrimination in protobuf deserialization:**

In `crates/apollo_protobuf/src/converters/transaction.rs`, the conversion from a protobuf `ResourceBounds` message to the internal `ValidResourceBounds` enum uses the *values* of the optional fields to decide which variant to produce:

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
// ...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ŌåÉ legacy variant
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

The intent is to map old 0.13.2 wire messages (where `l1_data_gas` is absent/zero and `l2_gas` is zero) to `L1Gas`. However, the condition is purely value-based: any `AllResources` transaction whose user-supplied `l2_gas` and `l1_data_gas` are both zero is indistinguishably mapped to `L1Gas`.

**Downstream hard-rejection of the downgraded variant:**

The mempool P2P path deserializes an incoming `protobuf::InvokeV3WithProof` through two chained conversions:

1. `protobuf::InvokeV3` ŌåÆ `InvokeTransactionV3` (calls the problematic `ValidResourceBounds::try_from` above)
2. `InvokeTransactionV3` ŌåÆ `RpcInvokeTransactionV3` ŌĆö this step explicitly rejects any non-`AllResources` variant:

```rust
resource_bounds: match value.resource_bounds {
    ValidResourceBounds::AllResources(bounds) => bounds,
    _ => {
        return Err(StarknetApiError::OutOfRange {
            string: "resource_bounds".to_string(),
        });
    }
},
``` [2](#0-1) 

The error is caught and re-wrapped as `DEPRECATED_RESOURCE_BOUNDS_ERROR`:

```rust
Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
``` [3](#0-2) 

where `DEPRECATED_RESOURCE_BOUNDS_ERROR = ProtobufConversionError::MissingField { field_description: "ResourceBounds::l1_data_gas" }`. [4](#0-3) 

The same two-step chain is traversed for the consensus P2P path:

```rust
protobuf::consensus_transaction::Txn::InvokeV3(txn) => {
    ConsensusTransaction::RpcTransaction(RpcTransaction::Invoke(
        RpcInvokeTransaction::V3(txn.try_into()?),
    ))
}
``` [5](#0-4) 

**The serialization side preserves the zero values faithfully**, so the problem is entirely in deserialization:

```rust
ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) =>
    protobuf::ResourceBounds {
        l1_gas: Some(l1_gas.into()),
        l2_gas: Some(l2_gas.into()),
        l1_data_gas: Some(l1_data_gas.into()),   // zero, but present
    },
``` [6](#0-5) 

The receiving side then calls `unwrap_or_default()` on the present-but-zero `l1_data_gas` field and applies the value-based heuristic, losing the `AllResources` identity.

**Hash domain divergence (secondary impact):**

Even if the rejection were bypassed, the hash preimage would differ. `get_tip_resource_bounds_hash` conditionally includes the `L1_DATA_GAS` resource element only for `AllResources`:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
``` [7](#0-6) 

Because `L1_DATA_GAS = b"L1_DATA"` is a non-zero resource name, `get_concat_resource` with zero amounts still produces a non-zero felt. The originating node hashes the transaction as `AllResources` (including this element); the receiving node, after downgrade, would hash it as `L1Gas` (omitting it), producing a different `TransactionHash`. [8](#0-7) 

### Impact Explanation

A user submits a valid `AllResources` invoke V3 transaction with `l2_gas = 0` and `l1_data_gas = 0`. The originating gateway accepts it (fee enforcement passes on `l1_gas` alone). When the gateway propagates the transaction to peers via mempool P2P or consensus P2P, every receiving node's protobuf deserializer silently downgrades the variant to `L1Gas` and then hard-rejects it. The transaction is never sequenced. This matches the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The `RpcInvokeTransactionV3` struct accepts any `AllResourceBounds` value, including zero `l2_gas` and `l1_data_gas`. No gateway config entry enforces a minimum `l2_gas` amount. A user who sets only `l1_gas` (e.g., to avoid L2 gas costs or by mistake) triggers the bug. The condition is narrow but reachable without any privileged access.

### Recommendation

Replace the value-based heuristic with a presence-based check. The `l1_data_gas` field is `Option` in the protobuf message; use its presence (not its value) to select the variant:

```rust
// In TryFrom<protobuf::ResourceBounds> for ValidResourceBounds:
match value.l1_data_gas {
    None => {
        // Legacy 0.13.2 wire format: no l1_data_gas field present.
        Ok(ValidResourceBounds::L1Gas(l1_gas))
    }
    Some(raw_l1_data_gas) => {
        let l1_data_gas: ResourceBounds = raw_l1_data_gas.try_into()?;
        Ok(ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }))
    }
}
```

This preserves backward compatibility with 0.13.2 peers (which omit `l1_data_gas`) while correctly round-tripping any `AllResources` transaction regardless of whether its bounds are zero.

### Proof of Concept

1. User constructs `RpcInvokeTransactionV3` with `resource_bounds: AllResourceBounds { l1_gas: <non-zero>, l2_gas: 0, l1_data_gas: 0 }`.
2. Gateway accepts the transaction; fee enforcement passes because `l1_gas` is non-zero.
3. Gateway serializes the transaction to `protobuf::MempoolTransaction` ŌåÆ `protobuf::InvokeV3WithProof` ŌåÆ `protobuf::InvokeV3`. The wire message carries `l2_gas = 0` and `l1_data_gas = 0` (both present, both zero).
4. Receiving peer deserializes `protobuf::InvokeV3` ŌåÆ `InvokeTransactionV3` via `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3`. Inside, `ValidResourceBounds::try_from(protobuf::ResourceBounds)` fires: `l1_data_gas.unwrap_or_default()` yields zero; `l2_gas` is zero; condition `l1_data_gas.is_zero() && l2_gas.is_zero()` is true ŌåÆ result is `ValidResourceBounds::L1Gas(l1_gas)`.
5. `InvokeTransactionV3` ŌåÆ `RpcInvokeTransactionV3` via `TryFrom<InvokeTransactionV3>`: `resource_bounds` is `L1Gas`, not `AllResources` ŌåÆ returns `Err(StarknetApiError::OutOfRange)`.
6. Error is mapped to `DEPRECATED_RESOURCE_BOUNDS_ERROR` in `TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3`.
7. Receiving peer rejects the transaction. The same rejection occurs on every peer and on the consensus P2P path.
8. The transaction is never included in a block despite being accepted by the originating gateway.

            return Err(missing("ResourceBounds::l1_gas"));

pub(crate) fn get_invoke_transaction_v3_hash<T: InvokeTransactionV3Trait>(
    transaction: &T,
    chain_id: &ChainId,
    transaction_version: &TransactionVersion,
) -> Result<TransactionHash, StarknetApiError> {
    let tip_resource_bounds_hash =
        get_tip_resource_bounds_hash(&transaction.resource_bounds(), transaction.tip())?;
    let paymaster_data_hash =
        HashChain::new().chain_iter(transaction.paymaster_data().0.iter()).get_poseidon_hash();
    let data_availability_mode = concat_data_availability_mode(
        transaction.nonce_data_availability_mode(),
        transaction.fee_data_availability_mode(),
    );
    let account_deployment_data_hash = HashChain::new()
        .chain_iter(transaction.account_deployment_data().0.iter())
        .get_poseidon_hash();
    let calldata_hash =
        HashChain::new().chain_iter(transaction.calldata().0.iter()).get_poseidon_hash();
    let mut hash_chain = HashChain::new()
        .chain(&INVOKE)
        .chain(&transaction_version.0)
        .chain(transaction.sender_address().0.key())
        .chain(&tip_resource_bounds_hash)
        .chain(&paymaster_data_hash)
        .chain(&Felt::try_from(chain_id)?)
        .chain(&transaction.nonce().0)
        .chain(&data_availability_mode)
        .chain(&account_deployment_data_hash)
        .chain(&calldata_hash);
    if !transaction.proof_facts().0.is_empty() {
        let proof_facts_hash =
            HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
        hash_chain = hash_chain.chain(&proof_facts_hash);
    }
    Ok(TransactionHash(hash_chain.get_poseidon_hash()))
}
