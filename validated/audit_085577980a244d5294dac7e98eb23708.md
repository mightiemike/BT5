### Title
Deprecated `verify_transaction_inclusion` Remains a Live Public Endpoint, Allowing 64-Byte Merkle Forgery to Bypass Coinbase-Proof Mitigation — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to mitigate the 64-byte transaction Merkle forgery vulnerability (CVE-2012-2459). However, the original `verify_transaction_inclusion` (v1) was never removed or access-restricted — it remains a fully callable public NEAR method. Any unprivileged caller can invoke v1 directly, bypassing the coinbase-proof validation that v2 enforces, and obtain a `true` SPV result for a transaction that was never included in a Bitcoin block.

---

### Finding Description

The contract exposes two public SPV verification endpoints:

**v1 — `verify_transaction_inclusion`** (lines 288–323): Computes a single Merkle path from `tx_id` to the stored `merkle_root`. It has no protection against a 64-byte transaction whose raw bytes, when double-SHA256'd, collide with an internal Merkle tree node. [1](#0-0) 

**v2 — `verify_transaction_inclusion_v2`** (lines 347–369): Adds the mitigation — it requires a coinbase Merkle proof of equal length, verifies the coinbase path independently against the stored `merkle_root`, and only then delegates to v1. This enforces tree-depth consistency and prevents the forgery. [2](#0-1) 

The `#[deprecated]` attribute on v1 is a **Rust compiler hint only** — it emits a warning at compile time for callers within the same codebase, but it imposes zero runtime restriction. The function remains a live, publicly callable NEAR method with no access-control guard, no `#[private]`, and no role check. [3](#0-2) 

The `compute_hash` function in `merkle-tools` concatenates two 32-byte hashes into a 64-byte buffer and double-SHA256s them. A 64-byte transaction undergoes the identical operation. There is no domain separation between leaf nodes and internal nodes. [4](#0-3) 

The `ProofArgs` struct accepted by v1 requires only `tx_id`, `tx_block_blockhash`, `tx_index`, `merkle_proof`, and `confirmations` — no coinbase data. [5](#0-4) 

---

### Impact Explanation

An attacker who controls a 64-byte transaction (or can craft one whose hash equals an internal Merkle node of a real Bitcoin block) can call `verify_transaction_inclusion` with:
- `tx_id` = the hash of the internal node (which equals the double-SHA256 of two real child hashes)
- A valid Merkle path from that internal node up to the real `merkle_root`
- A valid `tx_block_blockhash` for any real block stored in the contract

The function will return `true`. Any downstream NEAR DApp that calls `verify_transaction_inclusion` to gate a financial action (e.g., releasing bridged funds, minting wrapped BTC, confirming a payment) will be deceived into believing a fabricated transaction was confirmed on Bitcoin.

---

### Likelihood Explanation

The entry path requires no privileged role, no leaked key, and no social engineering. Any NEAR account can call `verify_transaction_inclusion` directly. The 64-byte transaction forgery technique is publicly documented (CVE-2012-2459, BitMEX research). The contract's own documentation acknowledges the attack vector explicitly in the v1 docstring warning. The only barrier is constructing a valid 64-byte transaction whose hash matches an internal node — a known, feasible operation.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or gate it with a role check (e.g., `#[private]`) so it cannot be called by external accounts. The `#[deprecated]` attribute provides no runtime protection. Since `verify_transaction_inclusion_v2` already delegates to v1 internally via `#[allow(deprecated)]`, the internal logic is preserved; only the external surface needs to be closed.

---

### Proof of Concept

1. Identify a real Bitcoin block stored in the contract (any `tx_block_blockhash` present in `headers_pool`).
2. Obtain the block's Merkle tree. Select any internal node `N` at depth `d` from the root.
3. Craft a 64-byte raw transaction whose `double_sha256` equals `N`. This is the known CVE-2012-2459 construction.
4. Build a Merkle path of length `(tree_depth - d)` from `N` up to `merkle_root`.
5. Call `verify_transaction_inclusion` with `tx_id = hash(N)`, the crafted path, and `confirmations = 1`.
6. The function returns `true` — the forged transaction is accepted as confirmed.
7. Call `verify_transaction_inclusion_v2` with the same `tx_id` and any coinbase proof of a **different length** — it panics. Call it with a coinbase proof of the **same length** — it correctly rejects the forgery because the coinbase path does not reconstruct the same root.

Step 6 vs. step 7 demonstrates the desynchronization: v1 accepts the forgery; v2 rejects it. Both endpoints are live and reachable by any unprivileged NEAR caller. [6](#0-5) [2](#0-1)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }

fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
