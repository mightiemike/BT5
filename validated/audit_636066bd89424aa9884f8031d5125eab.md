### Title
Missing `assertUtilization` Check in `matchOrders` Allows Spot Borrows to Exceed Pool Deposits — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.matchOrders` updates spot balances without enforcing the pool utilization invariant (`totalDeposits >= totalBorrows`). Trades that drive a user's spot balance negative (i.e., create a borrow) can push `totalBorrows > totalDeposits`, permanently breaking the invariant that is only enforced during `withdrawCollateral`. This is the direct Nado analog of Morpho's missing `borrowCaps[cToken]` check: a borrow-creating action bypasses the cap that the protocol enforces on a different code path.

---

### Finding Description

`SpotEngine.assertUtilization` enforces the solvency invariant:

```solidity
require(totalDeposits >= totalBorrows, ERR_MAX_UTILIZATION);
``` [1](#0-0) 

This check is called in exactly one place: `Clearinghouse.withdrawCollateral`, after reducing a user's spot balance. [2](#0-1) 

However, `OffchainExchange.matchOrders` also mutates spot balances via `_updateBalances` → `spotEngine.updateBalance` → `_updateBalanceNormalized`. When a trade drives a user's spot balance negative (e.g., selling a spot asset the user does not hold), `_updateBalanceNormalized` increases `totalBorrowsNormalized`: [3](#0-2) 

After `_updateBalances` completes in `matchOrders`, the only post-condition check is `isHealthy`, which unconditionally returns `true` in the base contract and provides zero protection: [4](#0-3) 

There is no call to `assertUtilization` anywhere in `matchOrders`. [5](#0-4) 

The `_updateBalanceNormalized` function correctly tracks the sign transition: if the resulting `newAmount` is negative, it uses `cumulativeBorrowsMultiplierX18` and adds to `totalBorrowsNormalized`. This accounting is correct, but the cap is never enforced at this call site. [6](#0-5) 

---

### Impact Explanation

Once `totalBorrows > totalDeposits` for a given spot product, every subsequent call to `Clearinghouse.withdrawCollateral` for that product reverts with `ERR_MAX_UTILIZATION`. Depositors cannot recover their funds. The protocol's core solvency invariant is permanently violated until the over-borrowed position is closed, which requires the attacker's cooperation or a forced liquidation path that may not exist for spot positions. This is a concrete asset-locking impact on all depositors of the affected spot product.

---

### Likelihood Explanation

The sequencer may filter orders that would create over-utilization off-chain, but there is no on-chain enforcement. Any unprivileged user who can get a matching order pair submitted — including via the slow-mode path in `Endpoint` — can trigger this. The slow-mode path is specifically designed for censorship resistance and is reachable without sequencer cooperation. Additionally, near-100% utilization is a realistic market condition under which a small incremental borrow crosses the threshold.

---

### Recommendation

Add an `assertUtilization` call in `matchOrders` after `_updateBalances` for spot products, conditioned on whether the product is a spot product:

```solidity
if (!callState.isPerp) {
    callState.spot.assertUtilization(callState.productId);
    if (market.quoteId != QUOTE_PRODUCT_ID) {
        callState.spot.assertUtilization(market.quoteId);
    }
}
```

Alternatively, enforce the cap inside `SpotEngine.updateBalance` itself so that no call site can bypass it.

---

### Proof of Concept

1. Pool state for spot asset X: `totalDeposits = 100`, `totalBorrows = 99` (99% utilization).
2. Attacker holds zero units of asset X. Attacker submits a signed sell order for 2 units of X.
3. A counterparty (or the attacker themselves via a second subaccount) submits a matching buy order.
4. Sequencer (or slow-mode) submits `MatchOrders` to `Endpoint`, which calls `OffchainExchange.matchOrders`.
5. `_updateBalances` → `spotEngine.updateBalance(X, attacker, -2)` → `_updateBalanceNormalized` transitions attacker's balance from 0 to -2; `totalBorrowsNormalized` increases so that `totalBorrows` becomes 101.
6. `isHealthy` returns `true`; no `assertUtilization` is called. Transaction succeeds.
7. `totalBorrows (101) > totalDeposits (100)`.
8. Every subsequent `withdrawCollateral` call for asset X reverts at `assertUtilization` with `ERR_MAX_UTILIZATION`.
9. All depositors of asset X are locked out of their funds. [1](#0-0) [7](#0-6) [8](#0-7)

        external
        onlyEndpoint
    {
        CallState memory callState = _getCallState(txn.matchOrders.productId);

        OrdersInfo memory ordersInfo;

        MarketInfo memory market = getMarketInfo(callState.productId);
        IEndpoint.SignedOrder memory taker = txn.matchOrders.taker;
        IEndpoint.SignedOrder memory maker = txn.matchOrders.maker;

        // isolated subaccounts cannot be used as sender
        require(
            !RiskHelper.isIsolatedSubaccount(taker.order.sender),
            ERR_INVALID_TAKER
        );
        require(
            !RiskHelper.isIsolatedSubaccount(maker.order.sender),
            ERR_INVALID_MAKER
        );

        ordersInfo = OrdersInfo(
            OrderInfo({
                digest: getDigest(callState.productId, taker.order),
                sender: taker.order.sender,
                amount: taker.order.amount,
                fee: 0,
                builderFee: 0,
                quoteDelta: 0,
                amountDelta: 0
            }),
            OrderInfo({
                digest: getDigest(callState.productId, maker.order),
                sender: maker.order.sender,
                amount: maker.order.amount,
                fee: 0,
                builderFee: 0,
                quoteDelta: 0,
                amountDelta: 0
            })
        );
        if (digestToSubaccount[ordersInfo.taker.digest] != bytes32(0)) {
            taker.order.sender = digestToSubaccount[ordersInfo.taker.digest];
        }
        if (digestToSubaccount[ordersInfo.maker.digest] != bytes32(0)) {
            maker.order.sender = digestToSubaccount[ordersInfo.maker.digest];
        }

        require(
            _validateOrder(
                callState,
                market,
                taker,
                ordersInfo.taker.digest,
                true,
                txn.takerLinkedSigner
            ),
            ERR_INVALID_TAKER
        );
        require(
            _validateOrder(
                callState,
                market,
                maker,
                ordersInfo.maker.digest,
                false,
                txn.makerLinkedSigner
            ),
            ERR_INVALID_MAKER
        );

        if (txn.takerAmountDelta != 0) {
            require(_isTWAP(taker.order.appendix), ERR_INVALID_TAKER);
            require(
                (txn.takerAmountDelta > 0) == (taker.order.amount > 0),
                ERR_INVALID_TAKER
            );
            if (taker.order.amount > 0) {
                require(
                    taker.order.amount >= txn.takerAmountDelta &&
                        maker.order.amount <= -txn.takerAmountDelta,
                    ERR_INVALID_TAKER
                );
            } else {
                require(
                    taker.order.amount <= txn.takerAmountDelta &&
                        maker.order.amount >= -txn.takerAmountDelta,
                    ERR_INVALID_TAKER
                );
            }

            taker.order.amount = txn.takerAmountDelta;
            maker.order.amount = -txn.takerAmountDelta;
        }

        // ensure orders are crossing
        require(
            (maker.order.amount > 0) != (taker.order.amount > 0),
            ERR_ORDERS_CANNOT_BE_MATCHED
        );
        if (maker.order.amount > 0) {
            require(
                maker.order.priceX18 >= taker.order.priceX18,
                ERR_ORDERS_CANNOT_BE_MATCHED
            );
        } else {

                maker.order.priceX18 <= taker.order.priceX18,
                ERR_ORDERS_CANNOT_BE_MATCHED
            );

        if (taker.order.amount < 0) {

                taker.order.amount,

            );
        } else if (taker.order.amount > 0) {
            ordersInfo.taker.amountDelta = MathHelper.min(
                taker.order.amount,
                -maker.order.amount
            );
        }

        ordersInfo.taker.amountDelta -=

            market.sizeIncrement;
        ordersInfo.maker.quoteDelta = ordersInfo.taker.amountDelta.mul(

        taker.order.amount -= ordersInfo.taker.amountDelta;

            true
        );

            ordersInfo.maker.fee = -ordersInfo.taker.fee;
