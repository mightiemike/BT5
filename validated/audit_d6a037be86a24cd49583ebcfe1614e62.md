### Title
Fee-on-Transfer Token Deposit Inflates Subaccount Balance Beyond Actual Clearinghouse Holdings — (File: `core/contracts/EndpointStorage.sol`)

---

### Summary

`EndpointStorage.handleDepositTransfer()` performs two sequential ERC20 transfers — user → Endpoint, then Endpoint → Clearinghouse — but enqueues a slow-mode `DepositCollateral` transaction recording the original caller-supplied `amount`. When the sequencer processes that transaction, `Clearinghouse.depositCollateral()` credits the full `amount` to the subaccount regardless of how much the Clearinghouse actually received. With a fee-on-transfer token, the Clearinghouse receives strictly less than `amount` (two fees are deducted across the two hops), yet the subaccount is credited the full `amount`, creating an unbacked balance that can be withdrawn at the expense of other depositors.

---

### Finding Description

The deposit path is:

1. **`Endpoint.depositCollateralWithReferral()`** (`Endpoint.sol:144-165`) calls `handleDepositTransfer(token, msg.sender, uint256(amount))` and then enqueues a `SlowModeTx` with `DepositCollateral{ sender: subaccount, productId: productId, amount: amount }` — the original caller-supplied value.

2. **`EndpointStorage.handleDepositTransfer()`** (`EndpointStorage.sol:111-119`) executes:
   ```solidity
   safeTransferFrom(token, from, amount);           // user → Endpoint
   safeTransferTo(token, address(clearinghouse), amount); // Endpoint → Clearinghouse
   ```
   With a fee-on-transfer token, the Endpoint receives `amount - fee₁` from the user, then attempts to forward `amount` to the Clearinghouse. If the Endpoint holds a pre-existing balance of that token (e.g., from slow-mode fees or prior deposits), the second transfer succeeds but the Clearinghouse receives only `amount - fee₂`. Net: Clearinghouse holds `amount - fee₁ - fee₂` but the slow-mode record says `amount`.

3. **`Clearinghouse.depositCollateral()`** (`Clearinghouse.sol:193-209`) is called by the sequencer with `txn.amount = amount` and computes:
   ```solidity
   int128 amountRealized = int128(txn.amount) * int128(multiplier);
   spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
   ```
   It credits the full `amount * multiplier` to the subaccount — no check against actual token balance received.

The slow-mode transaction is the permanent record. There is no reconciliation step between the token balance actually received by the Clearinghouse and the amount credited to the subaccount.

---

### Impact Explanation

The Clearinghouse's real token balance for the product is `amount - fee₁ - fee₂`, but the subaccount's credited balance is `amount`. The attacker can withdraw the full `amount`, forcing the Clearinghouse to pay out more than it received. The shortfall is covered by other depositors of the same token, directly draining their withdrawable collateral. `SpotEngine.assertUtilization()` (`SpotEngine.sol:232-241`) only checks `totalDeposits >= totalBorrows` in normalized accounting units — it does not compare the on-chain token balance against the sum of credited balances, so it does not catch this discrepancy.

---

### Likelihood Explanation

Any unprivileged user can call `Endpoint.depositCollateralWithReferral()` with a fee-on-transfer ERC20 token that has been listed as a spot product. The attacker needs no special role, no governance access, and no sequencer compromise. The only prerequisite is that the Endpoint holds a small pre-existing balance of the fee token (to allow the second hop to succeed), which is a realistic condition given slow-mode fee accumulation or prior deposits. The attack is repeatable and scales linearly with deposit size.

---

### Recommendation

Record the actual received amount rather than the caller-supplied amount. In `EndpointStorage.handleDepositTransfer()`, snapshot the Clearinghouse's token balance before and after the transfer and use the delta as the credited amount:

```solidity
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
    uint256 actualReceived = token.balanceOf(address(clearinghouse)) - before;
    require(actualReceived == amount, "Fee-on-transfer token not supported");
}
```

Alternatively, explicitly disallow fee-on-transfer tokens at product listing time.

---

### Proof of Concept

1. A fee-on-transfer token `FEE` (e.g., 1% fee per transfer) is listed as a spot product with `productId = P`.
2. The Endpoint holds a small pre-existing balance of `FEE` (e.g., from slow-mode fee accumulation).
3. Attacker calls `Endpoint.depositCollateralWithReferral(subaccount, P, 1000e6)`.
4. `handleDepositTransfer` pulls `1000e6` from attacker; Endpoint receives `990e6` (1% fee). Endpoint then forwards `1000e6` to Clearinghouse; Clearinghouse receives `990e6` (another 1% fee). Net: Clearinghouse holds `990e6` of `FEE`.
5. Slow-mode tx is enqueued with `amount = 1000e6`.
6. Sequencer processes the tx; `Clearinghouse.depositCollateral()` credits `1000e6 * multiplier` to attacker's subaccount.
7. Attacker calls `withdrawCollateral` for `1000e6`. Clearinghouse sends `1000e6` to attacker but only received `990e6` — the `10e6` shortfall is taken from other depositors' holdings.
8. Repeated deposits and withdrawals drain the Clearinghouse's `FEE` balance at the expense of honest depositors. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

        virtual

        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
