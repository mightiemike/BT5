### Title
`MsgBurnNFT.ValidateBasic` Uses `ValidateDenomID` Instead of `ValidateDenomIDWithIBC`, Permanently Blocking Burns of IBC-Originated NFTs — (`x/nft/types/msgs.go`, `x/nft/types/validation.go`)

### Summary

`ValidateDenomIDWithIBC` exists precisely to handle IBC denom IDs of the form `ibc/<64-hex-hash>`. However, `MsgBurnNFT.ValidateBasic` calls the narrower `ValidateDenomID` instead, which unconditionally rejects any denom ID containing `/` or exceeding 64 characters. Since IBC-received NFTs are stored under denom IDs in exactly that format, the burn message path is permanently blocked for them.

### Finding Description

**Two validators exist side-by-side:**

`ValidateDenomID` accepts only lowercase alphanumeric strings of length 3–64 starting with a letter: [1](#0-0) 

`ValidateDenomIDWithIBC` explicitly handles the `ibc/<64-hex-hash>` format (total length 68) and falls back to `ValidateDenomID` for non-IBC denoms: [2](#0-1) 

`MsgBurnNFT.ValidateBasic` calls only `ValidateDenomID`: [3](#0-2) 

When an NFT arrives via IBC, `processReceivedPacket` mints it under `voucherClassID = classTrace.IBCClassID()`, which produces `ibc/<sha256-hash>`: [4](#0-3) 

The `ibc/` prefix contains `/`, which is not alphanumeric, and the total length is 68, exceeding `MaxDenomLen = 64`. Both checks in `ValidateDenomID` fail, so `MsgBurnNFT{DenomId: "ibc/<64-hex>"}` is rejected at `ValidateBasic` before it ever reaches the keeper.

The same bug affects `MsgTransferNFT` and `MsgEditNFT` in the nft module, which also call `ValidateDenomID`: [5](#0-4) [6](#0-5) 

### Impact Explanation

The only remaining path for an IBC NFT holder is `x/nft-transfer`'s `MsgTransfer`, whose `ValidateBasic` does not call `ValidateDenomID` at all and only checks that `ClassId` is non-blank: [7](#0-6) 

If the originating IBC channel is live, the user can send the NFT back. However:
- Direct burn via `MsgBurnNFT` is **permanently blocked** regardless of channel state.
- If the IBC channel is closed or the counterparty chain is unavailable, the NFT is **permanently locked** in the owner's account with no on-chain exit path.
- The nft module's own `MsgTransferNFT` is equally blocked, so peer-to-peer transfer within the chain is also impossible for IBC-originated NFTs.

### Likelihood Explanation

This is reachable by any user who receives an NFT via `x/nft-transfer`. No special privileges are required. The failure is deterministic and reproducible with a unit test calling `MsgBurnNFT{DenomId: "ibc/" + strings.Repeat("ab", 32)}.ValidateBasic()` — it will return `ErrInvalidDenom`. The `ValidateDenomIDWithIBC` function already exists in the same file, confirming the developer was aware of the IBC format but omitted it from the message validators.

### Recommendation

Replace `ValidateDenomID` with `ValidateDenomIDWithIBC` in the `ValidateBasic` methods of `MsgBurnNFT`, `MsgTransferNFT`, and `MsgEditNFT` in `x/nft/types/msgs.go`: [8](#0-7) [9](#0-8) [10](#0-9)

```go
// Unit test — no chain setup required
func TestMsgBurnNFT_IBCDenomRejected(t *testing.T) {
    ibcDenomID := "ibc/" + strings.Repeat("ab", 32) // 4 + 64 = 68 chars, valid IBC hash
    msg := types.NewMsgBurnNFT(
        "cosmos1qnk2n4nlkpw9xfqntladh74er2xa62wgas3ek2",
        "tokenid1",
        ibcDenomID,
    )
    err := msg.ValidateBasic()
    // This WILL return ErrInvalidDenom — burn is blocked
    require.ErrorContains(t, err, "ErrInvalidDenom")
}
```

The `ibc/` prefix fails `IsAlphaNumeric` (contains `/`) and the length 68 exceeds `MaxDenomLen = 64`, so `ValidateDenomID` returns `ErrInvalidDenom` and the transaction is rejected by the ante-handler before execution. [11](#0-10)

```go
func ValidateDenomIDWithIBC(denomID string) error {

		}

		return nil
	}

}
```
