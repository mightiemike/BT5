### Title
Governance can set `ExitDuration` to an arbitrarily large value, permanently preventing users from exiting locked positions - (`x/tieredrewards/types/tier.go`)

### Summary

The `x/tieredrewards` module allows governance to update tier parameters via `MsgUpdateTier`. The `Tier.Validate()` function enforces that `ExitDuration > 0` but places **no upper bound** on the value. Governance can set `ExitDuration` to `math.MaxInt64` nanoseconds (~292 years), causing any subsequent `MsgTriggerExitFromTier` call to set `ExitUnlockAt` to a timestamp centuries in the future. Users who have not yet triggered their exit are permanently locked out of withdrawing their staked funds.

### Finding Description

The `Tier.Validate()` function in `x/tieredrewards/types/tier.go` validates tier parameters before they are written to state: [1](#0-0) 

The only constraint on `ExitDuration` is that it must be strictly positive. There is no maximum. Governance submits `MsgUpdateTier` through the `UpdateTier` handler: [2](#0-1) 

The handler only triggers a reward-claim sweep when `BonusApy` changes. A change to `ExitDuration` alone passes through with no protective action on existing positions.

When a user later calls `MsgTriggerExitFromTier`, the handler reads the **current** tier's `ExitDuration` and stamps `ExitUnlockAt` into the position at that moment: [3](#0-2) 

If governance has set `ExitDuration = math.MaxInt64` (9,223,372,036,854,775,807 ns ≈ 292 years), then `ExitUnlockAt` is set to `now + 292 years`. The two-step exit lifecycle requires this timestamp to elapse before `MsgExitTierWithDelegation` or `MsgTierUndelegate` can be executed, so the position is permanently frozen. [4](#0-3) 

### Impact Explanation

Users who have locked CRO in a tier and have **not yet triggered exit** before the governance change will have their funds permanently locked. They cannot call `ExitTierWithDelegation`, `TierUndelegate`, or `WithdrawFromTier` because all of these paths require the exit timer to have elapsed. The locked delegation remains in the position's sub-account indefinitely. This is a direct loss of access to user funds caused by a governance parameter change.

### Likelihood Explanation

The entry path is a standard on-chain governance proposal (`MsgUpdateTier`) that any governance participant can submit. The validation in `Tier.Validate()` accepts any positive duration, so a proposal setting `ExitDuration = math.MaxInt64` is syntactically and semantically valid and will be accepted by the chain. No special access beyond governance authority is required.

### Recommendation

Add an upper-bound check in `Tier.Validate()` for `ExitDuration`. For example, cap it at a protocol-defined maximum (e.g., 5 years in nanoseconds):

```go
const MaxExitDuration = 5 * 365 * 24 * int64(time.Hour) // 5 years in nanoseconds

if t.ExitDuration > MaxExitDuration {
    return fmt.Errorf("exit duration must not exceed %d ns (5 years): got %d", MaxExitDuration, t.ExitDuration)
}
```

This mirrors the existing `BonusApy` cap at `1.0` (100%), which was explicitly added to "prevent governance from draining the rewards pool." [5](#0-4) 

### Proof of Concept

1. Users lock CRO into Tier 1 (e.g., `ExitDuration = 1 year`).
2. Governance submits and passes `MsgUpdateTier` with `Tier{Id: 1, ExitDuration: math.MaxInt64, BonusApy: ..., MinLockAmount: ...}`.
3. `Tier.Validate()` passes because `math.MaxInt64 > 0`.
4. `UpdateTier` writes the new tier to state. No reward sweep is triggered (only `BonusApy` changes trigger that).
5. A user calls `MsgTriggerExitFromTier` for their position.
6. `TriggerExitFromTier` reads the current tier: `tier.ExitDuration = math.MaxInt64`.
7. `pos.TriggerExit(blockTime, math.MaxInt64)` sets `pos.ExitUnlockAt = blockTime + 292 years`.
8. The user calls `MsgExitTierWithDelegation` — `validateExitTierWithDelegation` rejects it because `ExitUnlockAt` has not passed.
9. The user's funds are permanently inaccessible.

func (ms msgServer) UpdateTier(ctx context.Context, msg *types.MsgUpdateTier) (*types.MsgUpdateTierResponse, error) {
	if err := ms.requireAuthority(msg.Authority); err != nil {
		return nil, err
	}

	oldTier, err := ms.getTier(ctx, msg.Tier.Id)
	if err != nil {
		return nil, err
	}

	if !oldTier.BonusApy.Equal(msg.Tier.BonusApy) {
		if err := ms.claimRewardsAndUpdateTierPositions(ctx, msg.Tier.Id); err != nil {
			return nil, err
		}
	}

	if err := ms.SetTier(ctx, msg.Tier); err != nil {
		return nil, err
	}

	if err := ms.emitTierChangedEvent(ctx, types.TierChangeAction_TIER_CHANGE_ACTION_UPDATE, msg.Tier); err != nil {
		return nil, err
	}

	return &types.MsgUpdateTierResponse{}, nil
}
