### Title
Snapshot-Based Interest Accrual Grants Full-Period Yield to Late Depositors — (`core/contracts/SpotEngineState.sol`)

---

### Summary

`_updateState` computes interest using a point-in-time snapshot of `totalDepositsNormalized` and applies it uniformly for the entire elapsed `dt`. Any deposit credited between two SpotTicks earns interest for the **full** `dt` window regardless of when within that window the deposit was made. An attacker can exploit this by executing a slow-mode deposit late in a long `dt` window, then withdrawing after the SpotTick inflates `cumulativeDepositsMultiplierX18`, capturing interest for time they never bore exposure.

---

### Finding Description

**Interest model mechanics**

`_updateState` computes a `depositRateMultiplierX18` from the current snapshot of deposits and borrows, then applies it to `cumulativeDepositsMultiplierX18` for the full `dt`: [1](#0-0) [2](#0-1) 

`dt` is the raw wall-clock delta since the last SpotTick, bounded only by `dt < 7 * SECONDS_PER_DAY`: [3](#0-2) 

`dt` is computed in `EndpointTx` as `txn.time - t.spotTime` and passed directly to `updateStates`: [4](#0-3) 

**Deposit normalization**

When `_updateBalanceNormalized` credits a deposit of amount `D` at the current CDM:

```
amountNormalized = (old_amountNormalized * CDM + D) / CDM
                 = old_amountNormalized + D / CDM
```

After `updateStates` multiplies CDM by `depositRateMultiplierX18 = R`:

```
balance = amountNormalized * CDM_new
        = (old_amountNormalized + D/CDM_old) * CDM_old * R
        = old_balance * R  +  D * R
```

The deposit `D` earns `D * (R - 1)` — the **full** `dt` interest — regardless of when within the `dt` window it was credited. [5](#0-4) 

**Attacker-controlled slow-mode execution**

`depositCollateralWithReferral` transfers tokens immediately and queues a slow-mode tx with a 3-day delay: [6](#0-5) 

After 3 days, `executeSlowModeTransaction()` is a **public** function callable by anyone — no sequencer required: [7](#0-6) 

The `fromSequencer = false` path only requires `txn.executableAt <= block.timestamp`: [8](#0-7) 

---

### Impact Explanation

The attacker earns `D * (depositRateMultiplier - 1)` in interest for the full `dt` period without having had funds at risk for that period. This interest is funded by the pool's borrowers and dilutes the yield of honest depositors who were in the pool for the entire `dt`. At high utilization and a near-maximum `dt` of ~7 days, the yield extracted per $1M deposited is on the order of $3,000–$10,000 per cycle, repeatable indefinitely.

---

### Likelihood Explanation

The exploit requires no sequencer compromise. The attacker only needs:
1. The slow-mode queue to reach their deposit (FIFO, observable on-chain).
2. A non-trivial `dt` between SpotTicks (any gap longer than a few hours makes the attack profitable at scale).

The sequencer is expected to tick frequently, but network congestion, operational delays, or deliberate observation of a growing `dt` all create windows. The attacker can monitor `times.spotTime` on-chain and call `executeSlowModeTransaction()` when the gap is large enough to be profitable, without any privileged access.

---

### Recommendation

Replace the global snapshot model with a time-weighted deposit accounting approach. Two concrete options:

1. **Checkpoint on deposit/withdrawal**: Record the current CDM at the time of each balance change and compute accrued interest only from that checkpoint forward, similar to how Aave tracks `liquidityIndex` per user.
2. **Reduce maximum `dt`**: Lower `7 * SECONDS_PER_DAY` to a much smaller bound (e.g., 1 hour) so the maximum extractable yield per cycle is negligible. This does not fix the root cause but limits the damage.

---

### Proof of Concept

```solidity
// Hardhat test (unmodified setup)
// Preconditions: product has 80% utilization, last SpotTick was 6 days ago (dt = 6 days)

// Step 1: Attacker calls depositCollateralWithReferral (tokens transferred immediately)
endpoint.depositCollateralWithReferral(attackerSubaccount, productId, 1_000_000e18, "");

// Step 2: Wait 3 days (slow-mode delay passes)
await time.increase(3 * 86400);

// Step 3: Attacker calls public executeSlowModeTransaction — no sequencer needed
endpoint.executeSlowModeTransaction();
// balance credited at current CDM (pre-tick)

// Step 4: Sequencer submits SpotTick with dt = 6 days (normal operation)
// updateStates(dt=518400) inflates cumulativeDepositsMultiplierX18

// Step 5: Attacker withdraws
// balance = amountNormalized * CDM_new = deposit * depositRateMultiplier
// profit = deposit * (depositRateMultiplier - 1) ≈ 0.3% of 1M = ~$3,000
// earned for 6 days of "free" exposure (days 0–3 before deposit was even queued,
// days 3–6 before slow-mode was executed)

// Assert: attacker balance after withdraw > 1_000_000e18
// Assert: honest depositor who held for full 6 days earned less per unit than attacker
```

The invariant `balanceNormalizedToBalance(state, balance).amount` must reflect only interest earned during the period the deposit was held — this is demonstrably broken: a deposit credited one second before `updateStates` earns identical interest to one held for the full `dt`.

    function _updateBalanceNormalized(
        State memory state,
        BalanceNormalized memory balance,
        int128 balanceDelta
    ) internal pure {
        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized -= balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized += balance.amountNormalized;
        }

        int128 cumulativeMultiplierX18;
        if (balance.amountNormalized > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        int128 newAmount = balance.amountNormalized.mul(
            cumulativeMultiplierX18
        ) + balanceDelta;

        if (newAmount > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        balance.amountNormalized = newAmount.div(cumulativeMultiplierX18);

        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized += balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized -= balance.amountNormalized;
        }
    }
