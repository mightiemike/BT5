Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks LP Recipient (`owner`) Instead of Actual Depositor (`sender`) — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` enforces the curated-pool deposit gate by checking `allowedDepositor[pool][owner]` — the LP-position recipient — while silently discarding the first unnamed parameter, which is `msg.sender` from the pool (the address that actually called `addLiquidity` and supplies the tokens via callback). Any non-allowlisted address can bypass the gate by nominating any known allowlisted address as `owner`, injecting tokens into the pool while the LP position is minted to that third party.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as the first argument and the caller-supplied `owner` as the second argument to `_beforeAddLiquidity`:

```solidity
// MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` encodes both and forwards them to the extension:

```solidity
// ExtensionCalling.sol L97
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

The extension receives `(sender, owner, ...)` but discards `sender` entirely and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The contract's own NatSpec states "Gates `addLiquidity` by depositor address, per pool" and the admin setter uses the name `depositor` throughout (`setAllowedToDeposit`, `allowedDepositor` mapping), making clear the intent is to gate the token-supplying actor, not the LP-position recipient. The implementation contradicts this intent.

**Attack path:**
1. Non-allowlisted attacker `A` identifies any allowlisted address `B` (readable from on-chain `AllowedToDepositSet` events).
2. `A` deploys a contract implementing `IMetricOmmModifyLiquidityCallback` and calls `pool.addLiquidity(owner=B, salt, deltas, callbackData, extensionData)`.
3. Pool calls `_beforeAddLiquidity(sender=A, owner=B, ...)` → extension evaluates `allowedDepositor[pool][B]` → `true` → no revert.
4. Pool calls `metricOmmModifyLiquidityCallback` on `A`; `A` pays the tokens.
5. LP position is minted to `B`; `A`'s tokens are now inside the curated pool, bypassing the allowlist entirely.

No existing guard prevents this: `removeLiquidity` enforces `msg.sender == owner` but `addLiquidity` has no such constraint, and the extension is the sole deposit gate.

## Impact Explanation

The deposit allowlist is the primary access-control boundary for curated pools. Bypassing it constitutes an admin-boundary break (an unprivileged path circumvents a pool-admin-configured restriction). Concrete consequences include:

- **Compliance/KYC bypass**: non-whitelisted counterparties inject tokens into regulated pools.
- **Pool-state manipulation**: the attacker shifts bin liquidity distribution, affecting oracle-derived bid/ask spreads and swap pricing for all subsequent traders.
- **LP dilution**: existing allowlisted LPs' share of fees and principal is diluted by unauthorized deposits.

This meets the "admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" and "broken core pool functionality" impact criteria.

## Likelihood Explanation

- `addLiquidity` is permissionless; any EOA or contract can call it with an arbitrary `owner`.
- No special privilege, flash loan, or oracle manipulation is required.
- Allowlisted addresses are publicly discoverable from `AllowedToDepositSet` events.
- The attacker only needs tokens and a callback-implementing contract.
- Likelihood: **High**.

## Recommendation

Replace the `owner` check with the first (currently unnamed) parameter — the actual `sender` — to match the stated intent of gating by depositor:

```solidity
function beforeAddLiquidity(
    address sender,   // the address that called addLiquidity — the real depositor
    address owner,
    uint80,
    LiquidityDelta calldata,
    bytes calldata
) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```
