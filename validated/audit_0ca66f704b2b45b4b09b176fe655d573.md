Audit Report

## Title
`addLiquidity` and `removeLiquidity` Bypass `whenNotPaused` Guard While `swap` Is Correctly Gated — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary

`MetricOmmPool` exposes a `whenNotPaused` modifier that reverts when `pauseLevel != 0`, and correctly applies it to `swap`. However, `addLiquidity` and `removeLiquidity` carry no such guard, allowing LPs to deposit into or withdraw from a paused pool. When the pool is unpaused after a bad-oracle or accounting-anomaly event, any liquidity deposited during the pause is immediately exposed to arbitrage at the stale price, causing direct LP principal loss.

## Finding Description

The `whenNotPaused` modifier is defined at [1](#0-0)  and delegates to `_checkNotPaused`, which reverts when `pauseLevel != 0`. [2](#0-1) 

`swap` correctly applies the modifier: [3](#0-2) 

`addLiquidity` carries only `nonReentrant` — no `whenNotPaused`: [4](#0-3) 

`removeLiquidity` is identically unguarded: [5](#0-4) 

`setPause` is gated to the factory and sets `pauseLevel` to a non-zero value to freeze the pool: [6](#0-5) 

The exploit path is:
1. Factory pauses the pool (`pauseLevel = 1`) due to a bad oracle or detected anomaly.
2. An LP calls `addLiquidity(owner, salt, deltas, callbackData, extensionData)` — the call succeeds because no `whenNotPaused` check exists; tokens are transferred into the pool via `LiquidityLib.addLiquidity`.
3. Factory unpauses the pool (believing the issue is resolved, or under time pressure).
4. An arbitrageur calls `swap` against the LP's newly deposited liquidity at the stale/bad oracle price, draining the LP's principal.

For `removeLiquidity`, the absence of the guard allows a subset of LPs to exit during a pause triggered by an accounting anomaly or insolvency event, leaving remaining LPs with undercollateralised claims.

Note: the `simulateSwapAndRevert` extension-hook corruption sub-claim is not independently impactful, as the terminal `revert SimulateSwap(...)` rolls back all state changes including any extension storage mutations made by `_beforeSwap`/`_afterSwap`.

## Impact Explanation

Direct loss of LP principal: an LP depositing during a pause has their funds immediately exposed to arbitrage at a bad oracle price upon unpause. This satisfies the "direct loss of user principal" criterion. For `removeLiquidity`, the open exit path during an insolvency-triggered pause enables a first-mover drain, leaving remaining LPs with undercollateralised claims — pool insolvency. Both impacts are above Sherlock medium thresholds.

## Likelihood Explanation

Any pause event (factory-triggered via `setPause`) immediately opens the window. No special privilege is required for an LP to call `addLiquidity` or `removeLiquidity` during the pause — these are standard unprivileged pool actions. The factory unpausing after believing an issue is resolved is a normal operational step, making the full exploit path realistic and repeatable.

## Recommendation

Apply `whenNotPaused` to `addLiquidity` and `removeLiquidity`. If emergency LP exits are an intentional design choice, document it explicitly and apply the guard only to `addLiquidity`:

```solidity
function addLiquidity(...) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) ...

function removeLiquidity(...) external whenNotPaused nonReentrant(PoolActions.REMOVE_LIQUIDITY) ...
```

## Proof of Concept

1. Deploy pool; factory calls `setPause(poolAddress, 1)` → `pauseLevel = 1`.
2. Verify `swap(...)` reverts with `PoolPaused()`. ✓
3. LP calls `addLiquidity(owner, salt, deltas, callbackData, extensionData)` — **succeeds**, tokens transferred into pool.
4. Factory calls `setPause(poolAddress, 0)` to unpause.
5. Arbitrageur calls `swap(...)` against LP's deposited liquidity at the stale oracle price — LP principal is drained.
