### Title
Malicious Proposer Can Bloat Block Store with Invalid Transactions via NoOpMempool - (File: app/app.go)

### Summary
Cronos POS Chain configures its proposal handlers with `mempool.NoOpMempool{}`, identical to the SEDA Chain vulnerability. Because `NoOpMempool` performs no transaction validation, the `ProcessProposalHandler` accepts any transactions a proposer includes — including invalid ones — up to `MaxBlockSize`. CometBFT stores raw transaction bytes for every included transaction in its BlockStore permanently.

### Finding Description
In `app/app.go`, the mempool and proposal handlers are configured as follows:

```go
// app/app.go lines 276-282
baseAppOptions = append(baseAppOptions, func(app *baseapp.BaseApp) {
    mempool := mempool.NoOpMempool{}
    app.SetMempool(mempool)
    handler := baseapp.NewDefaultProposalHandler(mempool, app)
    app.SetPrepareProposal(handler.PrepareProposalHandler())
    app.SetProcessProposal(handler.ProcessProposalHandler())
})
``` [1](#0-0) 

`mempool.NoOpMempool{}` is a no-op implementation that skips all transaction validation. When `NewDefaultProposalHandler` is initialized with this mempool, its `ProcessProposalHandler` performs no additional transaction verification — it accepts all transactions in the proposed block without checking their validity. A malicious block proposer can therefore include arbitrarily crafted, invalid transactions filling the block up to the CometBFT `MaxBlockSize` consensus parameter, and every validator will accept the block.

### Impact Explanation
CometBFT permanently stores raw transaction bytes for every transaction in every accepted block in its BlockStore. There is no BlockStore pruning configured in chain-main. A malicious proposer (any validator whose turn it is to propose) can permanently write up to `MaxBlockSize` bytes of garbage per block they propose, causing:
- Unbounded disk space consumption on all full nodes and validators
- Increased block propagation time (network bandwidth)
- Increased block processing overhead (CPU/memory)
- Significantly slower future node synchronization for new joiners

### Likelihood Explanation
Any validator in the active set will periodically become the block proposer via CometBFT's round-robin proposer selection. No special privileges beyond being a validator are required. The attack requires only that the proposer craft a block with invalid transactions, which is trivially achievable.

### Recommendation
Replace `mempool.NoOpMempool{}` with a proper mempool implementation (e.g., `mempool.NewSenderNonceMempool()` or a priority mempool) when configuring the proposal handlers. The `ProcessProposalHandler` should validate transactions before accepting them into a block.

### Proof of Concept
1. A validator waits for their turn as block proposer.
2. They craft a block containing one valid transaction followed by many invalid/garbage transactions, filling the block to `MaxBlockSize`.
3. They submit this block via `PrepareProposal`.
4. All validators call `ProcessProposal` — because `NoOpMempool` is used, `ProcessProposalHandler` skips validation and returns `ACCEPT`.
5. The block is committed; CometBFT writes all raw transaction bytes (including the invalid ones) to the BlockStore permanently.
6. Repeated over many proposal rounds, this permanently bloats the chain's storage on every node. [2](#0-1)
