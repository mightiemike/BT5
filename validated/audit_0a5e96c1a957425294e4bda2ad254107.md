### Title
Protobuf Deserialization Misclassifies `AllResources` Declare Transactions as `L1Gas` Variant, Causing Consensus Proposal Rejection — (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` converter applies a zero-threshold check to decide between the `L1Gas` and `AllResources` variants. A valid declare V3 transaction with non-zero `l1_gas` but zero `l2_gas` and `l1_data_gas` bounds passes gateway admission as `AllResourceBounds`, but is silently downgraded to `ValidResourceBounds::L1Gas` during consensus protobuf deserialization. The downstream conversion to `RpcDeclareTransactionV3` then hard-rejects any non-`AllResources` variant, causing the entire consensus proposal to be marked `Failed`.

### Finding Description

**Conversion boundary (root cause):**

In `crates/apollo_protobuf/src/converters/transaction.rs` lines 431–435, the deserialization of `protobuf::ResourceBounds` into `ValidResourceBounds` uses a value-threshold gate identical in structure to the lending-pool `DUST_THRESHOLD` removal:

```rust
// transaction.rs:431
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← variant silently changed
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

A transaction originally signed and admitted as `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is re-classified as `L1Gas(X)` on the receiving validator.

**Admission path (gateway accepts the transaction):**

`RpcDeclareTransactionV3` carries `AllResourceBounds` directly. The gateway's `validate_resource_bounds` in `crates/apollo_gateway/src/stateless_transaction_validator.rs` line 66 only rejects when `max_possible_fee == 0`. With non-zero `l1_gas`, the fee is positive and the transaction is admitted.

**Consensus path (validator rejects the proposal):**

When the proposer serializes the transaction, `From<RpcDeclareTransactionV3> for protobuf::DeclareV3WithClass` (in `rpc_transaction.rs` line 197) wraps the bounds as `ValidResourceBounds::AllResources(value.resource_bounds)`, then `From<ValidResourceBounds> for protobuf::ResourceBounds` (in `transaction.rs` lines 479–487) emits all three fields with `l2_gas = 0` and `l1_data_gas = 0`.

On the validator, `TryFrom<protobuf::DeclareV3WithClass> for RpcDeclareTransactionV3` (in `rpc_transaction.rs` lines 174–178) enforces:

```rust
// rpc_transaction.rs:174
resource_bounds: match common.resource_bounds {
    ValidResourceBounds::AllResources(resource_bounds) => resource_bounds,
    _ => {
        return Err(DEPRECATED_RESOURCE_BOUNDS_ERROR);  // ← hard rejection
    }
},
```

Because the intermediate `DeclareTransactionV3Common` was built with the misclassified `L1Gas` variant, this arm is hit and the conversion returns an error.

In `crates/apollo_consensus_orchestrator/src/validate_proposal.rs` lines 609–616, any conversion error causes the entire proposal to be rejected:

```rust
let conversion_results = match conversion_results {
    Ok(results) => results,
    Err(e) => {
        return HandledProposalPart::Failed(format!(
            "Failed to convert transactions. Stopping the build of the current proposal. {e:?}"
        ));
    }
};
```

**Hash domain divergence (secondary effect):**

Even if the hard rejection were removed, `get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` lines 203–208 produces a structurally different preimage for the two variants:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2-element hash
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3-element hash
    }
});
```

The proposer hashes with 3 elements; the validator would hash with 2, producing a different `tx_hash` and breaking the commitment.

### Impact Explanation

**High — Transaction conversion binds the wrong type, causing valid proposals to be rejected.**

A single declare transaction with `l1_gas > 0`, `l2_gas = 0`, `l1_data_gas = 0` — admitted by the gateway — is sufficient to make every validator reject the proposal that contains it. This is a liveness-breaking denial-of-service: any user (or the proposer itself) can trigger it without any privileged access.

### Likelihood Explanation

**Medium.** The gateway explicitly accepts declare transactions where only `l1_gas` is non-zero (the fee check passes). The zero-threshold condition in the protobuf converter is not documented as a rejection criterion, so legitimate wallets or tooling targeting pre-0.13.3 fee semantics can produce such transactions. An adversary who knows the condition can craft one deliberately.

### Recommendation

Remove the value-threshold variant selection from `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. The variant should be determined by a version field or an explicit presence flag, not by whether the numeric values happen to be zero. The simplest safe fix is to always produce `AllResources` when all three fields are present in the protobuf message (matching the behavior of `TryFrom<protobuf::ResourceBounds> for AllResourceBounds` in `rpc_transaction.rs` lines 212–223, which never applies a threshold).

### Proof of Concept

1. Craft a declare V3 transaction: `resource_bounds = AllResourceBounds { l1_gas: { max_amount: 1000, max_price_per_unit: 1 }, l2_gas: { max_amount: 0, max_price_per_unit: 0 }, l1_data_gas: { max_amount: 0, max_price_per_unit: 0 } }`.
2. Submit via RPC. Gateway's `validate_resource_bounds` computes `max_possible_fee = 1000 > 0` → admitted.
3. Proposer picks it up, serializes to protobuf: all three `ResourceLimits` fields are emitted, `l2_gas` and `l1_data_gas` both carry `max_amount = 0, max_price_per_unit = 0`.
4. Validator deserializes: `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` evaluates `l1_data_gas.is_zero() && l2_gas.is_zero()` → `true` → returns `ValidResourceBounds::L1Gas(l1_gas)`.
5. `TryFrom<protobuf::DeclareV3WithClass> for RpcDeclareTransactionV3` matches `common.resource_bounds` against `AllResources` → falls through to `_ => return Err(DEPRECATED_RESOURCE_BOUNDS_ERROR)`.
6. `validate_proposal` receives the error → `HandledProposalPart::Failed(...)` → proposal rejected, consensus stalls on any block containing this transaction. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));

            ValidResourceBounds::L1Gas(l1_gas)

// An implementation of the SNIP: https://github.com/EvyatarO/SNIPs/blob/snip-8/SNIPS/snip-8.md
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
