### Title
Unbounded Growth of `validated_g1_points` / `validated_g2_points` Caches Outside Heap Accounting — (`File: src/allocator.rs`)

### Summary

The `Allocator` struct maintains two `HashSet` caches — `validated_g1_points` and `validated_g2_points` — that accumulate every unique BLS G1/G2 point validated during program execution. These sets grow monotonically with no eviction, no size cap, and — critically — are **not subject to the CLVM heap limit** that governs all other allocator memory. They also intentionally survive `restore_checkpoint()` calls made inside softfork guards, meaning failed softfork guard executions leave stale entries in the caches permanently. An attacker who controls the CLVM bytes can force unbounded growth of these caches, consuming memory that is invisible to the consensus-enforced heap limit.

### Finding Description

`Allocator` declares two unbounded sets:

```rust
// src/allocator.rs lines 300-301
validated_g1_points: HashSet<[u8; 48]>,
validated_g2_points: HashSet<[u8; 96]>,
```

Every call to `validate_g1()` or `validate_g2()` inserts a new entry if the point has not been seen before:

```rust
// src/allocator.rs lines 1264-1280
pub fn validate_g1(&mut self, node: NodePtr, bytes: [u8; 48]) -> Result<()> {
    if !self.validated_g1_points.contains(&bytes) {
        G1Element::from_bytes(&bytes)...
        self.validated_g1_points.insert(bytes);
    }
    Ok(())
}
```

The CLVM heap limit (`heap_limit`) is enforced only against `atom_vec` / `u8_vec` growth. The `validated_g1_points` and `validated_g2_points` sets are allocated on the Rust heap entirely outside this accounting. There is a `clear_validation_caches()` method, but it is a manual call — it is **not** invoked by `restore_checkpoint()`. The comment at line 297-299 explicitly states this is intentional:

> "Using raw bytes instead of NodePtr since NodePtrs can be invalidated by restore_checkpoint() inside softfork guards."

This means that every softfork guard execution — including failed ones that roll back the allocator state — permanently adds its validated BLS points to the caches. An attacker can chain multiple softfork guard invocations, each validating a fresh batch of distinct G1/G2 points, accumulating cache entries that are never reclaimed.

### Impact Explanation

Each unique G1 point consumes 48 bytes in `validated_g1_points`; each G2 point consumes 96 bytes in `validated_g2_points`. These bytes are not counted against the CLVM heap limit. A program that validates N distinct G1 points uses `48 * N` bytes of unaccounted memory. The CLVM cost model charges for BLS operations, so N is bounded by `cost_limit / bls_op_cost`, but the resulting memory is invisible to the heap limit. A validator node processing a crafted block can therefore consume significantly more memory than the heap limit implies, breaking the memory-safety guarantee that the heap limit is supposed to provide. In a block-validation context where many programs are executed sequentially (each with its own `Allocator`), this is per-execution, but the discrepancy between enforced limit and actual usage is a concrete broken invariant.

### Likelihood Explanation

The entry path is direct: an attacker submits a Chia transaction whose puzzle or solution contains a CLVM program that calls `point_add` or `pubkey_for_exp` with many distinct valid G1/G2 points, optionally wrapped in softfork guards to multiply the effect across checkpoint boundaries. No special privileges are required — any coin spend is sufficient. The BLS operation cost provides a natural bound, but the memory accounting gap is real and reachable.

### Recommendation

1. Account for `validated_g1_points` and `validated_g2_points` memory within the heap limit, or impose a separate size cap on these sets (e.g., evict oldest entries when a threshold is reached).
2. Alternatively, clear the validation caches inside `restore_checkpoint()` (accepting the cost of re-validating points after a softfork guard rollback), or at minimum document the maximum possible cache size relative to the cost limit so node operators can size memory accordingly.
3. Consider whether the caches need to survive `restore_checkpoint()` at all — if the softfork guard re-validates points anyway, the cache benefit inside a failed guard is zero.

### Proof of Concept

```
; CLVM pseudocode
; Craft a program that calls point_add with N distinct valid G1 points
; inside a softfork guard that is designed to fail after validation.
; Each guard failure calls restore_checkpoint(), rolling back atoms/pairs
; but leaving all N points in validated_g1_points.
; Repeat with a new batch of N points in the next guard.
; After K guards: K*N entries in validated_g1_points, using K*N*48 bytes
; of heap memory not counted against heap_limit.

(softfork cost_limit 0
  (point_add G1_point_1 G1_point_2 ... G1_point_N)
  ; force failure here
  (x))
```

Each iteration of this pattern leaves N entries in `validated_g1_points` permanently. The outer program's cost limit bounds K*N, but the memory consumed (`K * N * 48` bytes for G1, `K * N * 96` bytes for G2) is entirely outside the heap limit enforced by `Allocator::new_limited()`.
