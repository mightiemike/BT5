### Title
`proof_facts` Conditionally Omitted from Invoke V3 Transaction Hash Preimage Allows Signature Reuse Across Proof-Bearing and Non-Proof-Bearing Transactions - (File: `crates/starknet_api/src/transaction_hash.rs`)

---

### Summary

`get_invoke_transaction_v3_hash` only appends `proof_facts` to the Poseidon hash chain when `proof_facts` is non-empty. This means a transaction signed with empty `proof_facts` produces the **same hash** as a transaction with the same fields but non-empty `proof_facts` — if the `proof_facts` hash happens to collide, or more critically, that the same signature over the base hash is valid for a transaction that carries proof facts that were never signed by the account. An attacker who observes a valid signature over a base invoke transaction can replay it with arbitrary `proof_facts` appended, because the account's signature was computed over a hash that did not commit to the `proof_facts` field.

---

### Finding Description

In `get_invoke_transaction_v3_hash`, the `proof_facts` field is conditionally included in the transaction hash preimage:

```rust
// crates/starknet_api/src/transaction_hash.rs, lines 399-403
if !transaction.proof_facts().0.is_empty() {
    let proof_facts_hash =
        HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
    hash_chain = hash_chain.chain(&proof_facts_hash);
}
``` [1](#0-0) 

When `proof_facts` is empty, the hash chain terminates after `calldata_hash`. When `proof_facts` is non-empty, the hash chain is extended with the proof facts hash. This means:

- A transaction `T_base` with `proof_facts = []` produces hash `H`.
- A transaction `T_proof` with the same base fields but `proof_facts = [v1, v2, ...]` produces hash `H' = Poseidon(H_base_chain || proof_facts_hash)`.

These are **different hashes**, so the same signature does not directly validate both. However, the invariant that is broken is the **analog of the external bug**: the `proof_facts` field is a parameter that is **not always committed to in the signed preimage**. Specifically, for a transaction with empty `proof_facts`, the account signs a hash that contains **no commitment to the proof_facts field at all**. The field is structurally absent from the preimage, not represented as a zero/empty hash.

This is confirmed by the Cairo OS implementation, which mirrors the same conditional:

```cairo
// For backward compatibility, we don't hash proof facts if they are empty.
if (proof_facts_size != 0) {
    poseidon_hash_update_with_nested_hash(
        data_ptr=proof_facts, data_length=proof_facts_size
    );
}
``` [2](#0-1) 

The concrete attack surface is:

1. A user submits a valid invoke V3 transaction with `proof_facts = []`. The account signs hash `H = Poseidon(INVOKE || version || sender || fee_hash || paymaster_hash || chain_id || nonce || da_mode || account_deployment_data_hash || calldata_hash)`.
2. An attacker takes the same transaction fields and the same signature, but submits it with a non-empty `proof_facts` payload (e.g., a valid SNOS proof facts blob referencing a historical block).
3. The gateway computes a **different** `tx_hash` for this new transaction (because `proof_facts` is now included in the hash), so the transaction is treated as a distinct transaction with a different hash.
4. The account's `__validate__` entrypoint is called with the new `tx_hash` (which includes `proof_facts`). The account verifies the user's ECDSA signature against this new hash — and the signature **fails**, because the user signed the old hash.

Wait — this means the account's own signature check would reject it. Let me re-examine the actual impact more carefully.

The real impact is the **inverse direction**: a user who submits a transaction **with** `proof_facts` signs hash `H'` which commits to the proof facts. But if the gateway or sequencer strips or ignores `proof_facts` during some conversion step, the account validation runs against a hash that does not include `proof_facts`, allowing the proof facts to be swapped.

Looking at the storage deserialization shim:

```rust
// Backward compatibility: proof_facts may not exist in old transactions.
// If no data remains, default to empty; otherwise, deserialize normally.
let proof_facts = if data.is_empty() {
    ProofFacts::default()
} else {
    ProofFacts::deserialize_from(data)?
};
``` [3](#0-2) 

A transaction stored in the old on-disk format (before `proof_facts` was added) is deserialized with `proof_facts = []`. When this transaction is later re-executed or re-hashed (e.g., during re-execution or RPC serving), `get_invoke_transaction_v3_hash` is called on the deserialized struct with `proof_facts = []`, producing the original hash `H`. This is correct for old transactions.

However, the **critical asymmetry** is: a new transaction submitted with non-empty `proof_facts` has its hash computed as `H'` (including proof facts). If this transaction is stored and then deserialized from an old-format node (one that hasn't yet re-synced), the `proof_facts` field is silently dropped and the deserialized transaction has `proof_facts = []`. Re-hashing this deserialized transaction produces `H ≠ H'`. The stored `tx_hash` in `InternalRpcTransaction` is `H'`, but the re-derived hash from the deserialized body is `H`. This is a **hash/body mismatch** that can cause wrong execution results, wrong receipt values, or wrong state when the transaction is re-executed from storage. [4](#0-3) 

---

### Impact Explanation

**Impact: High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value; and potentially Critical — Wrong state, receipt, event, or revert result from blockifier/syscall/execution logic.**

When a node running the old storage schema deserializes a new-format `InvokeTransactionV3` that carries `proof_facts`, the `proof_facts` field is silently dropped (defaulted to empty). Any subsequent operation that re-derives the transaction hash from the deserialized body — such as RPC `starknet_getTransactionByHash`, re-execution, fee estimation, or tracing — will compute a hash `H` that does not match the canonical hash `H'` under which the transaction was originally accepted and stored. This produces an authoritative-looking wrong transaction hash in RPC responses and potentially wrong execution context (the `tx_info.transaction_hash` seen by the contract during `__validate__` and `__execute__` would be wrong).

---

### Likelihood Explanation

**Medium.** The backward-compatibility shim is explicitly marked as temporary (`TODO(AvivG): Remove once all nodes have re-synced`), meaning it is present in production during the migration window. Any node that has not yet re-synced and receives a `proof_facts`-bearing transaction via P2P sync will silently drop the `proof_facts` on deserialization. The `allow_client_side_proving` gateway flag gates submission, but P2P-synced transactions bypass the gateway entirely. [5](#0-4) 

---

### Recommendation

1. **Add a version discriminant to the on-disk serialization format** for `InvokeTransactionV3` so that the presence or absence of `proof_facts` is explicit, not inferred from trailing bytes. This prevents silent field loss during deserialization.

2. **Always include `proof_facts` in the transaction hash preimage**, even when empty, by hashing an empty array rather than omitting the field entirely. This makes the hash a total commitment to all fields:
   ```rust
   // Always chain proof_facts (even if empty) to make hash a total commitment.
   let proof_facts_hash =
       HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
   hash_chain = hash_chain.chain(&proof_facts_hash);
   ```
   Note: this is a protocol-breaking change and requires a version gate.

3. **Validate hash consistency after deserialization**: after deserializing a stored `InternalRpcTransaction`, assert that `tx_without_hash.calculate_transaction_hash(chain_id) == stored_tx_hash`. Reject or flag any mismatch.

---

### Proof of Concept

**Setup**: Node A is running the old storage schema (pre-`proof_facts`). Node B submits a valid `InvokeTransactionV3` with non-empty `proof_facts` (e.g., a SNOS proof facts blob). Node B's gateway computes `tx_hash = H'` (including proof facts in the hash) and stores `InternalRpcTransaction { tx: ..., tx_hash: H' }`.

**Step 1**: Node A receives this transaction via P2P sync and stores it. During storage serialization, `proof_facts` is written. During deserialization on a node that hasn't re-synced, the backward-compat shim fires:

```rust
let proof_facts = if data.is_empty() {
    ProofFacts::default()  // proof_facts silently dropped
} else {
    ProofFacts::deserialize_from(data)?
};
``` [6](#0-5) 

**Step 2**: Node A serves an RPC `starknet_getTransactionByHash(H')`. It retrieves the stored transaction body (with `proof_facts = []` after deserialization) and re-derives the hash:

```rust
let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
``` [7](#0-6) 

This produces `H ≠ H'` because `proof_facts` is empty in the deserialized body but was non-empty in the original. The RPC response contains a transaction body whose hash does not match the requested hash — an authoritative-looking wrong value.

**Step 3**: If this transaction is re-executed (e.g., during `starknet_simulateTransactions` or blockifier re-execution), the `tx_info.transaction_hash` seen by the contract's `__validate__` entrypoint is `H`, not `H'`. The account's signature was computed over `H'`. The account's signature verification fails or, if the account uses a non-standard validator that reads `tx_info.transaction_hash` for other purposes, produces wrong behavior. [8](#0-7) [9](#0-8)

        self.account_deployment_data.serialize_into(&mut to_compress)?;
        self.proof_facts.serialize_into(&mut to_compress)?;
        if to_compress.len() > COMPRESSION_THRESHOLD_BYTES {
            IsCompressed::Yes.serialize_into(res)?;
            if to_compress.len() > crate::compression_utils::MAX_DECOMPRESSED_SIZE {

            let compressed = compress(to_compress.as_slice())?;
            compressed.serialize_into(res)?;
        } else {
            IsCompressed::No.serialize_into(res)?;
            to_compress.serialize_into(res)?;
        }
        Ok(())

        let data = &mut data.as_slice();

        let tip = Tip::deserialize_from(data)?;
