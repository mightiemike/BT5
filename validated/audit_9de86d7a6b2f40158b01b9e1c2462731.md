### Title
`proof_facts` silently dropped to empty in central-sync deserialization causes hash-preimage mismatch for Invoke V3 transactions ŌĆö (`crates/apollo_starknet_client/src/reader/objects/transaction.rs`)

---

### Summary

The `IntermediateInvokeTransaction` ŌåÆ `InvokeTransactionV3` conversion in the central-sync (feeder-gateway) reader silently defaults `proof_facts` to empty via `unwrap_or_default()` when the feeder gateway omits the field. Because `proof_facts` is only returned when the caller explicitly passes `response_flags: ["INCLUDE_PROOF_FACTS"]` (Starknet RPC v0.10), the central-sync path stores Invoke V3 transactions with:

- `tx_hash` = the canonical hash **computed WITH `proof_facts`** (as returned by the feeder gateway)
- `proof_facts` = **empty** (field absent from the feeder-gateway response)

The Rust hash function `get_invoke_transaction_v3_hash` conditionally appends `proof_facts` to the Poseidon chain only when non-empty. This means the stored hash and the hash recomputable from the stored body are **different values** ŌĆö a broken canonicalization invariant. Any RPC endpoint that re-executes or traces the stored transaction will either silently skip proof-facts validation (blockifier path) or produce a hash-mismatch abort (OS path), returning an authoritative-looking wrong value.

---

### Finding Description

**Hash computation ŌĆö conditional `proof_facts` inclusion**

`get_invoke_transaction_v3_hash` in `crates/starknet_api/src/transaction_hash.rs` appends `proof_facts` only when non-empty:

```rust
if !transaction.proof_facts().0.is_empty() {
    let proof_facts_hash =
        HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
    hash_chain = hash_chain.chain(&proof_facts_hash);
}
```

The same conditional gate exists in the Cairo OS (`transaction_hash.cairo`):

```cairo
// For backward compatibility, we don't hash proof facts if they are empty.
if (proof_facts_size != 0) {
    poseidon_hash_update_with_nested_hash(
        data_ptr=proof_facts, data_length=proof_facts_size
    );
}
```

So `hash(tx_body_with_proof_facts) ŌēĀ hash(tx_body_without_proof_facts)`.

**Central-sync deserialization ŌĆö silent default to empty**

`TryFrom<IntermediateInvokeTransaction> for InvokeTransactionV3` in `crates/apollo_starknet_client/src/reader/objects/transaction.rs` uses:

```rust
// proof_facts is optional for backward compatibility with V3 transactions created
// before client-side proving was added. Old V3 txs don't have this field.
proof_facts: invoke_tx.proof_facts.unwrap_or_default(),
```

`IntermediateInvokeTransaction.proof_facts` is `Option<ProofFacts>` and is absent from the feeder-gateway JSON unless `INCLUDE_PROOF_FACTS` is requested. The central-sync code does not request that flag, so for any Invoke V3 transaction that carries `proof_facts`, the deserialized object has `proof_facts = []`.

**Divergent stored state**

After central-sync ingestion of a block containing an Invoke V3 with non-empty `proof_facts`:

| Field | Stored value |
|---|---|
| `tx_hash` | `H = poseidon(ŌĆ”, proof_facts_hash)` ŌĆö correct, from feeder gateway |
| `tx.proof_facts` | `[]` ŌĆö silently dropped |

Recomputing the hash from the stored body yields `H' = poseidon(ŌĆ”)` (no `proof_facts` term), so `H' ŌēĀ H`.

**RPC re-execution path**

`stored_txn_to_executable_txn` in `crates/apollo_rpc/src/v0_8/api/mod.rs` converts the stored `InvokeTransactionV3` (with empty `proof_facts`) into an `ExecutableTransactionInput`. The blockifier's `validate_proof_facts` in `crates/blockifier/src/transaction/account_transaction.rs` returns `Ok(())` immediately for `ProofFactsVariant::Empty`, so the entire proof-facts validation is skipped. The RPC trace/simulation result is therefore wrong: it reflects execution of a transaction body that differs from the one the signer committed to.

For the OS re-execution path (`starknet_traceTransaction` via the Cairo OS), `compute_invoke_transaction_hash` recomputes `H'` and the OS asserts `H' == stored_tx_hash` (`AssertTransactionHash` hint). Because `H' ŌēĀ H`, the OS aborts, and the RPC returns an internal error instead of the correct trace.

---

### Impact Explanation

Any node that syncs via the central-sync path (feeder gateway) and serves RPC will:

1. Return `proof_facts = []` from `starknet_getTransactionByHash` for transactions that originally carried proof facts ŌĆö a wrong authoritative value.
2. Return a wrong execution trace from `starknet_traceTransaction` (OS abort due to hash mismatch, or blockifier execution with proof-facts validation silently bypassed).
3. Return a wrong fee estimate from `starknet_estimateFee` when the fee depends on proof-facts-gated execution paths.

This matches the **High** impact criterion: *RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.*

---

### Likelihood Explanation

The trigger is any Invoke V3 transaction with non-empty `proof_facts` that is committed to a block and then ingested by a node running central sync. Client-side proving is a production feature (the gateway already accepts such transactions, the blockifier validates them, and the protobuf/P2P path preserves them). The central-sync path is the only path that silently drops the field. No special privilege is required; any user who submits a client-side-proving transaction can trigger the divergence on any central-sync node.

---

### Recommendation

1. **Request `INCLUDE_PROOF_FACTS`** in the feeder-gateway query inside the central-sync reader so that `proof_facts` is always present in the response for V3 transactions.
2. **Assert hash consistency** after deserialization: recompute the transaction hash from the stored body and compare it with the stored `transaction_hash`; reject or log a fatal error on mismatch.
3. **Remove the `unwrap_or_default()` fallback** for `proof_facts` once all nodes have re-synced, replacing it with a hard error for V3 transactions that lack the field (mirroring the treatment of `resource_bounds`, `tip`, etc.).
