Audit Report

## Title
`SwapAllowlistExtension::beforeSwap` Checks Router Address Instead of End-User, Enabling Allowlist Bypass or Blocking Legitimate Users — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` parameter, which the pool sets to its own `msg.sender` — the router contract when users swap via `MetricOmmSimpleRouter`. Because the allowlist is configured per end-user address, the extension evaluates `allowedSwapper[pool][routerAddress]` instead of `allowedSwapper[pool][actualUser]`, completely breaking the per-user access control. This produces two fund-impacting outcomes: a full bypass when the router is allowlisted, or permanent blocking of legitimately allowlisted users who route through the standard swap path.

## Finding Description

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // <-- whoever called pool.swap()
    recipient,
    ...
);
``` [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this value unchanged as the first argument to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (correct) and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly — making the **router** the `msg.sender` of the pool, not the end-user: [4](#0-3) 

The same applies to `exactOutputSingle`: [5](#0-4) 

The asymmetry with `DepositAllowlistExtension.beforeAddLiquidity` confirms the design intent: the deposit extension receives both `sender` (direct caller) and `owner` (actual LP) as separate parameters and correctly gates on `owner`, not `sender`: [6](#0-5) 

The `beforeSwap` interface has a `recipient` parameter available as the second argument, but `SwapAllowlistExtension` ignores it (unnamed `address` in the signature). Even if used, `recipient` is attacker-controlled and cannot serve as a reliable identity check.

## Impact Explanation

**Bypass (Critical):** A pool admin who allowlists the router address (`setAllowedToSwap(pool, routerAddress, true)`) to enable router-based swaps inadvertently grants every address — including those explicitly never allowlisted — the ability to swap. The per-user restriction is nullified entirely, allowing unauthorized traders to execute swaps against restricted LP positions.

**Broken functionality (High):** A pool admin who allowlists individual user addresses finds those users permanently blocked from the standard swap path (the router is not allowlisted), making the pool's primary swap interface unusable for legitimate participants. LP assets become effectively locked in a pool with no usable swap path.

Both outcomes directly impact user funds and constitute broken core pool functionality.

## Likelihood Explanation

`SwapAllowlistExtension` is a production periphery contract. Any pool that configures it in `beforeSwap` order and expects per-user gating is affected. The router is the standard entry point for end-users, so the mismatch is triggered on every normal `exactInputSingle` or `exactOutputSingle` call. No special attacker capability is required beyond a standard router call.

## Recommendation

The `beforeSwap` hook interface does not carry a separate "actual initiator" parameter analogous to `owner` in `beforeAddLiquidity`. The minimal fix options are:

1. **Router encodes real user in `extensionData`:** Have `MetricOmmSimpleRouter` ABI-encode `msg.sender` into `extensionData` before calling `pool.swap`, and have `SwapAllowlistExtension.beforeSwap` decode and check it. The pool's existing transient callback context (which already stores the real payer) can be used to verify the binding.

2. **Pool-level fix:** Add an explicit `initiator` parameter to the `swap` function and `beforeSwap` hook (analogous to `owner` in `addLiquidity`), populated by the pool from a verified source rather than raw `msg.sender`.

The `recipient` parameter must **not** be used as a substitute — it is caller-controlled and can be set to any allowlisted address.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured in `beforeSwap` order.
2. Pool admin calls `setAllowedToSwap(pool, routerAddress, true)` to enable router-based swaps.
3. Attacker (address never allowlisted by the admin) calls `router.exactInputSingle(...)`.
4. Router calls `pool.swap(recipient=attacker, ...)` with `msg.sender = router`.
5. Pool calls `_beforeSwap(msg.sender=router, ...)` → extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
6. Attacker successfully swaps in a pool intended to be restricted.

Alternatively (broken-functionality path):
1. Pool admin calls `setAllowedToSwap(pool, userAddress, true)` for a legitimate user.
2. Legitimate user calls `router.exactInputSingle(...)`.
3. Extension checks `allowedSwapper[pool][router]` → `false` → `NotAllowedToSwap` revert.
4. Legitimate user is permanently blocked from the standard swap path despite being allowlisted.
