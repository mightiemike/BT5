### Title
Unprivileged Caller Can Aggressively Prune Mainchain Headers via `run_mainchain_gc`, Breaking Chain Reorganization and SPV Proof Verification - (File: `contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` in `contract/src/lib.rs` carries no caller-identity restriction when the contract is unpaused. Any NEAR account can invoke it with an arbitrarily large `batch_size`, causing the contract to delete all mainchain block headers that exceed `gc_threshold` in a single transaction. This irreversibly removes headers that are required for chain reorganization and SPV proof verification, permanently corrupting the light client's canonical chain state.

---

### Finding Description

`run_mainchain_gc` is decorated only with `#[pause(except(roles(Role::UnrestrictedRunGC)))]`:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
```

The `#[pause]` macro restricts access **only when the contract is paused**. When the contract is running normally (unpaused), the function is fully open to any caller. There is no role check, no `#[private]`, and no `require!(env::predecessor_account_id() == ...)` guard.

The `batch_size` parameter is accepted without an upper-bound check. The only cap applied internally is:

```rust
let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);
```

Passing `u64::MAX` as `batch_size` causes `selected_amount_to_remove` to equal `total_amount_to_remove`, meaning **all headers above the gc_threshold are deleted in one call**.

Under normal operation, GC is called from `submit_blocks` with `batch_size = num_of_headers` (the count of headers in that submission), keeping pruning gradual and proportional. An attacker bypasses this design by calling `run_mainchain_gc` directly.

The deleted headers are removed from both `mainchain_height_to_header` and `headers_pool`, and `mainchain_initial_blockhash` is advanced to the new oldest block. These deletions are **permanent** — there is no recovery path.

---

### Impact Explanation

Two core contract functions are broken by aggressive GC:

**1. Chain Reorganization (`submit_block_header` reorg path)**

The CLAUDE.md explicitly documents this invariant:

> If mainchain blocks near the fork point have been garbage collected, reorg will fail — the contract panics with `PrevBlockNotFound` when it cannot walk the chain back to the common ancestor.

If a fork's common ancestor block has been pruned by the attacker, any subsequent `submit_blocks` call that would trigger a reorg panics with `PrevBlockNotFound`. The fork can never be promoted to the mainchain, and the light client's fork-choice rule is permanently broken for that fork.

**2. SPV Proof Verification (`verify_transaction_inclusion` / `verify_transaction_inclusion_v2`)**

`verify_transaction_inclusion` looks up the target block in `mainchain_header_to_height`:

```rust
let target_block_height = self
    .mainchain_header_to_height
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));
```

`verify_transaction_inclusion_v2` looks up the block in `headers_pool`:

```rust
let header = self
    .headers_pool
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));
```

Both maps are cleared for pruned blocks by `remove_block_header`. Any SPV proof for a pruned block will panic, permanently denying verification for those transactions.

---

### Likelihood Explanation

The attack requires no privileged access, no staked tokens, and no special role. Any NEAR account can call `run_mainchain_gc({"batch_size": 18446744073709551615})` at any time the contract is unpaused. The mainchain grows beyond `gc_threshold` continuously during normal operation (the recommended `gc_threshold` is 52,704 blocks, approximately one year of Bitcoin blocks). Once the chain has grown even slightly beyond the threshold, the attack window is open. An attacker monitoring the chain can trigger the attack at any time, including precisely when a fork is being built or when a user is about to submit an SPV proof.

---

### Recommendation

Add a caller-identity restriction to `run_mainchain_gc` so that only trusted roles (e.g., `Role::DAO`, `Role::RelayerManager`, or a dedicated `GCManager` role) can invoke it directly. The internal call from `submit_blocks` can bypass this check since it is already gated by `#[trusted_relayer]`.

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
    require!(
        self.acl_has_role(Role::DAO, &env::predecessor_account_id())
            || self.acl_has_role(Role::RelayerManager, &env::predecessor_account_id()),
        "Unauthorized: only privileged roles may call run_mainchain_gc directly"
    );
    // ... existing logic
}
```

Alternatively, make the public entry point private and expose a separate privileged wrapper, keeping the internal call from `submit_blocks` unchanged.

### Proof of Concept

**Setup**: Deploy the contract with `gc_threshold = 10`. Submit 25 mainchain headers (chain grows to 25, which is 15 above threshold).

**Attack**:
```
near call <contract> run_mainchain_gc '{"batch_size": 18446744073709551615}' --accountId attacker.near
```

This single call removes all 15 excess headers in one transaction, advancing `mainchain_initial_blockhash` by 15 blocks.

**Consequence 1 — Reorg failure**: If a fork was submitted whose common ancestor is among the 15 pruned blocks, any subsequent `submit_blocks` call for that fork panics with `PrevBlockNotFound` when walking back to the common ancestor.

**Consequence 2 — SPV proof failure**: Calling `verify_transaction_inclusion_v2` for any of the 15 pruned blocks panics with `"cannot find requested transaction block"` because `headers_pool` no longer contains those entries.

Both effects are permanent — the deleted entries cannot be re-inserted without re-submitting the entire chain from the pruned point, which the contract does not support. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
