### Title
Unchecked u64 Multiplication Overflow in `op_unknown` Cost Computation Enables Cost Undercharging — (`File: src/more_ops.rs`)

### Summary
In `op_unknown`, the final cost multiplication `cost *= cost_multiplier + 1` is performed on a `u64` without overflow protection. In Rust release builds, integer overflow wraps silently (two's complement). An attacker-controlled unknown opcode can be crafted so that the pre-multiplication cost, bounded by `max_cost`, when multiplied by `cost_multiplier + 1`, wraps to a value ≤ `u32::MAX`, causing the function to return `Ok` with a drastically undercharged cost instead of the correct large value.

### Finding Description

In `src/more_ops.rs`, `op_unknown` computes a base cost from the opcode's arguments (bounded by `check_cost`), then multiplies by `cost_multiplier + 1`:

```rust
// line 260-265
check_cost(cost, max_cost)?;
cost *= cost_multiplier + 1;          // ← unchecked u64 multiplication
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

`cost_multiplier` is derived from `u32_from_u8(&op[0..op.len() - 1])`, so it is at most `u32::MAX = 4_294_967_295`, making `cost_multiplier + 1` at most `2^32 = 4_294_967_296`. [1](#0-0) 

`check_cost` at line 260 only guarantees `cost ≤ max_cost` before the multiplication. [2](#0-1) 

In the Chia blockchain, `max_cost` is ~11 billion (`11 × 10^9`). With `cost_multiplier + 1 = 2^32`:

```
11_000_000_000 × 4_294_967_296 = 47_244_640_256_000_000_000
u64::MAX                        = 18_446_744_073_709_551_615
```

The product overflows `u64`. In release mode, Rust wraps silently. If the attacker arranges `cost` to be an exact multiple of `2^32` (e.g., `cost = 2^32 = 4_294_967_296`), then:

```
4_294_967_296 × 4_294_967_296 = 2^64 ≡ 0  (mod 2^64)
```

The post-multiplication `cost` is `0`, which satisfies `0 ≤ u32::MAX`, so the function returns `Ok(Reduction(0, nil))` — zero cost for an operation that should have cost billions.

The `cost_function = 2` (MUL-like) path in `op_unknown` grows quadratically with atom size, making it feasible to reach `cost = 2^32` with two atoms of approximately 724 KB each (well within Chia's block limits), without triggering the `0xffff` reserved-opcode guard. [3](#0-2) 

### Impact Explanation

The cost returned by `op_unknown` is added directly to the running `cost` accumulator in `run_program`. [4](#0-3) 

A wrapped-to-zero (or near-zero) cost means the attacker's program consumes almost none of the cost budget for an operation that should have consumed billions of units. This allows:

1. **Cost undercharging / DoS bypass**: The attacker executes computationally expensive unknown opcodes for free, exhausting node resources while staying within the declared cost limit.
2. **Consensus divergence**: Nodes running debug builds (which panic on overflow) or future hardened builds will compute a different cost than release-mode nodes, causing a consensus split on the same block.

### Likelihood Explanation

- Unknown opcodes in lenient/consensus mode are a standard, documented feature of the Chia dialect. [5](#0-4) 
- The attacker fully controls the opcode bytes (selecting `cost_multiplier` and `cost_function`) and the argument atoms (selecting their sizes to hit the target pre-multiplication cost).
- The `0xffff` reserved-opcode guard does not prevent using `cost_multiplier` values up to `0xfffeffff = 4_294_901_759`, which is sufficient to trigger overflow.
- No special privileges are required; any CLVM program submitted to the network can invoke this path.

### Recommendation

Replace the bare multiplication with a checked or saturating variant, and reject the result if it overflows or exceeds `u32::MAX`:

```rust
// Replace:
cost *= cost_multiplier + 1;

// With:
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::Invalid(o))?;
```

This mirrors the intent of the existing `check_cost` guard and is the Rust equivalent of using a safe-math library. [2](#0-1)

pub fn op_unknown(
    allocator: &mut Allocator,
    o: NodePtr,
    mut args: NodePtr,
    max_cost: Cost,
) -> Response {
    // unknown opcode in lenient mode
    // unknown ops are reserved if they start with 0xffff
    // otherwise, unknown ops are no-ops, but they have costs. The cost is computed
    // like this:

    // byte index (reverse):
    // | 4 | 3 | 2 | 1 | 0          |
    // +---+---+---+---+------------+
    // | multiplier    |XX | XXXXXX |
    // +---+---+---+---+---+--------+
    //  ^               ^    ^
    //  |               |    + 6 bits ignored when computing cost
    // cost_multiplier  |
    // (up to 4 bytes)  + 2 bits
    //                    cost_function

    // 1 is always added to the multiplier before using it to multiply the cost, this
    // is since cost may not be 0.

    // cost_function is 2 bits and defines how cost is computed based on arguments:
    // 0: constant, cost is 1 * (multiplier + 1)
    // 1: computed like operator add, multiplied by (multiplier + 1)
    // 2: computed like operator mul, multiplied by (multiplier + 1)
    // 3: computed like operator concat, multiplied by (multiplier + 1)

    // this means that unknown ops where cost_function is 1, 2, or 3, may still be
    // fatal errors if the arguments passed are not atoms.

    let op_atom = allocator.atom(o);
    let op = op_atom.as_ref();

    if op.is_empty() || (op.len() >= 2 && op[0] == 0xff && op[1] == 0xff) {
        Err(EvalErr::Reserved(o))?;
    }
