### Title
Asymmetric Fee Formula in `applyFee` Overcharges Taker Buyers Relative to Taker Sellers — (`File: core/contracts/OffchainExchange.sol`)

---

### Summary

The `applyFee` function in `OffchainExchange.sol` uses two structurally different formulas to compute the taker fee depending on the sign of `meteredQuote`. When a taker is buying base (paying quote, `meteredQuote < 0`), the fee is computed via a division by `keepRateX18`, yielding `fee = |meteredQuote| * feeRate / (1 - feeRate)`. When a taker is selling base (receiving quote, `meteredQuote > 0`), the fee is computed via multiplication, yielding `fee = meteredQuote * feeRate`. These are not equivalent: the division formula always produces a strictly larger fee for the same absolute quote amount. This is the direct analog of the external report's inconsistent `amountWithFee` formulas.

---

### Finding Description

In `OffchainExchange.sol`, `applyFee` computes the net quote after fee as follows:

```solidity
int128 keepRateX18 = ONE - feeInfo.feeRate;
int128 newMeteredQuote = (meteredQuote > 0)
    ? meteredQuote.mul(keepRateX18)
    : meteredQuote.div(keepRateX18);
orderInfo.fee = meteredQuote - newMeteredQuote;
``` [1](#0-0) 

The sign of `meteredQuote` is determined by the taker's trade direction. From the matching logic:

```solidity
ordersInfo.maker.quoteDelta = ordersInfo.taker.amountDelta.mul(maker.order.priceX18);
ordersInfo.taker.quoteDelta = -ordersInfo.maker.quoteDelta;
``` [2](#0-1) 

- **Taker buying base** (`taker.order.amount > 0`): `taker.quoteDelta < 0` → `matchQuote < 0` → `meteredQuote < 0` → uses `.div(keepRateX18)`.
- **Taker selling base** (`taker.order.amount < 0`): `taker.quoteDelta > 0` → `matchQuote > 0` → `meteredQuote > 0` → uses `.mul(keepRateX18)`.

The two resulting fee formulas are:

| Direction | Formula | Fee on 100 quote at 1% rate |
|---|---|---|
| Taker selling (meteredQuote > 0) | `fee = Q * feeRate` | 1.0000 |
| Taker buying (meteredQuote < 0) | `fee = Q * feeRate / (1 - feeRate)` | 1.0101 |

`MathSD21x18.mul` computes `(x * y) / ONE_X18` and `MathSD21x18.div` computes `(x * ONE_X18) / y`, confirming the two paths are algebraically distinct. [3](#0-2) 

The `quoteDelta` is then updated as:

```solidity
orderInfo.quoteDelta = orderInfo.quoteDelta - orderInfo.fee - orderInfo.builderFee;
``` [4](#0-3) 

This means taker buyers have their `quoteDelta` reduced by a systematically larger fee than taker sellers for the same absolute matched quote, and the excess flows into `market.collectedFees` via `updateCollectedFees`. [5](#0-4) 

---

### Impact Explanation

Every taker buy order pays a fee of `Q * feeRate / (1 - feeRate)` while every taker sell order pays `Q * feeRate` for the same absolute quote amount `Q`. The excess fee charged to buyers is `Q * feeRate^2 / (1 - feeRate)`. At the default 2bps rate this is negligible per trade, but at higher configured fee tiers (e.g. 1%) the overcharge is ~1% of the fee itself per trade, accumulating systematically across all taker buy volume. The corrupted state is `orderInfo.quoteDelta` (the taker's net quote balance delta) and `market.collectedFees` (the protocol's fee accumulator), both of which diverge from the symmetric fee invariant on every taker buy.

---

### Likelihood Explanation

High. Every taker order processed through `matchOrders` or `matchOrdersWithAmount` triggers `applyFee` with `taker = true`. [6](#0-5) 

No special conditions are required. Any unprivileged user submitting a taker buy order through the supported endpoint entrypoints is affected on every fill.

---

### Recommendation

Use a single consistent formula for both directions. The symmetric approach is to always multiply by `keepRateX18`:

```solidity
// Consistent: fee = |meteredQuote| * feeRate regardless of direction
int128 newMeteredQuote = meteredQuote.mul(keepRateX18);
orderInfo.fee = meteredQuote - newMeteredQuote;
```

This matches the external report's recommendation: pick one formula and apply it uniformly. The `mul` path (`fee = Q * feeRate`) is the natural percentage-of-gross interpretation and should be used for both positive and negative `meteredQuote`.

---

### Proof of Concept

Given `feeRate = 0.01e18` (1%), `keepRateX18 = 0.99e18`, and a matched quote of 100 units:

**Taker selling** (`meteredQuote = +100e18`):
- `newMeteredQuote = 100e18 * 0.99e18 / 1e18 = 99e18`
- `fee = 100e18 - 99e18 = 1e18` → **1.0 units**

**Taker buying** (`meteredQuote = -100e18`):
- `newMeteredQuote = (-100e18 * 1e18) / 0.99e18 = -101.0101...e18`
- `fee = -100e18 - (-101.0101e18) = 1.0101e18` → **1.0101 units**

The buyer pays ~1.01% more fee than the seller for an identical absolute quote amount. The excess `0.0101` units per 100 quote is credited to `market.collectedFees` beyond what the symmetric formula would produce, representing a systematic overcharge on all taker buy volume. [7](#0-6)
