Looking at the `DepositAllowlistExtension`, I need to compare it against `SwapAllowlistExtension` and trace how `sender` vs `owner` flows through the hook plumbing.

**Key observation in `DepositAllowlistExtension.beforeAddLiquidity`:**

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
```

The first `address` parameter (unnamed, silently dropped) is `sender` — the actual `msg.sender` of `addLiquidity`, the party who pays tokens via callback. The extension checks `owner` instead.

Compare with `SwapAllowlistExtension`, which correctly checks `sender`:

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

The pool passes `msg.sender` as `sender` to both hooks:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

And `owner` is a free caller-controlled parameter in `addLiquidity`:

```solidity
function addLiquidity(address owner, uint80 salt, ...) external nonReentrant ...
```

### Title
`DepositAllowlistExtension` gates on `owner` instead of `sender`, allowing any unprivileged caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently drops the `sender` argument and checks the caller-supplied `owner` address against the allowlist. Because `owner` is a free parameter of `addLiquidity`, any non-allowlisted party can pass an allowlisted address as `owner` and the guard passes, while the actual depositor (who pays tokens via the callback) is never checked.

### Finding Description
`MetricOmmPool.addLiquidity` calls `_beforeAddLiquidity(msg.sender, owner, ...)` where `msg.sender` is the real depositor and `owner` is an arbitrary address supplied by the caller that receives the LP position. [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both as positional arguments to the extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first `address` argument but leaves it unnamed and unused. It checks only `owner`: [3](#0-2) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly names and checks `sender`: [4](#0-3) 

### Impact Explanation
A non-allowlisted caller (Bob) calls `pool.addLiquidity(owner = alice, ...)` where Alice is allowlisted. The guard checks `allowedDepositor[pool][alice]` → passes. Bob pays tokens via the modify-liquidity callback; Alice receives the LP shares. The allowlist is completely ineffective: every non-allowlisted address can deposit by nominating any allowlisted address as `owner`. Alice receives an unsolicited LP position she did not authorise; if she does not monitor the pool and remove it, she bears pool-price risk on capital she never intended to commit. The pool admin's access-control intent is entirely defeated.

### Likelihood Explanation
The bypass requires only a single direct call to `addLiquidity` with a known allowlisted address as `owner`. No special permissions, flash loans, or privileged access are needed. Any actor who can observe the allowlist state (public mapping) can execute this immediately.

### Recommendation
Replace the unnamed first parameter with `sender` and check it instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

### Proof of Concept
```solidity
// Alice is allowlisted; Bob is not.
allowlistExtension.setAllowedToDeposit(pool, alice, true);

// Bob calls addLiquidity with owner = alice.
// The guard checks allowedDepositor[pool][alice] == true → passes.
// Bob pays tokens via callback; Alice receives LP shares she never requested.
vm.prank(bob);
pool.addLiquidity(
    alice,          // owner — allowlisted, guard passes
    0,              // salt
    deltas,
    callbackData,   // Bob's contract pays tokens here
    ""
);

// Bob has bypassed the allowlist entirely.
// Alice now holds an unsolicited LP position.
assertGt(positionShares(pool, alice, 0, binIdx), 0);
```
