### Title
Missing EIP-712 Domain Separator in User Transaction Digests Enables Cross-Chain Signature Replay - (File: `core/contracts/Verifier.sol`)

---

### Summary

The `computeDigest` function in `Verifier.sol` computes transaction struct hashes for user-signed operations (withdrawals, liquidations, etc.) without including an EIP-712 domain separator — specifically, no `chainId` and no `verifyingContract` address. Because the sequencer signature path explicitly binds to `block.chainid` while the user signature path does not, a valid user signature from one chain deployment (e.g., Ink Sepolia testnet) can be replayed on another deployment (e.g., Ink mainnet) whenever the target subaccount's nonce matches.

---

### Finding Description

**Root cause — `Verifier.sol`, `computeDigest` (lines 321–360+):**

```solidity
function computeDigest(
    IEndpoint.TransactionType txType,
    bytes calldata transactionBody
) public pure returns (bytes32) {
    ...
    // LiquidateSubaccount path
    digest = keccak256(
        abi.encode(
            keccak256(bytes(LIQUIDATE_SUBACCOUNT_SIGNATURE)),
            signedTx.tx.sender,
            signedTx.tx.liquidatee,
            signedTx.tx.productId,
            signedTx.tx.isEncodedSpread,
            signedTx.tx.amount,
            signedTx.tx.nonce
        )
    );
    // WithdrawCollateral path
    digest = keccak256(
        abi.encode(
            keccak256(bytes(WITHDRAW_COLLATERAL_SIGNATURE)),
            signedTx.tx.sender,
            signedTx.tx.productId,
            signedTx.tx.amount,
            signedTx.tx.nonce
        )
    );
```

Neither path includes `chainId` or `address(this)`. The function is `public pure`, meaning it is the canonical digest clients sign — there is no domain separator added downstream before `validateSignature` is called.

**Contrast with the sequencer path — `Verifier.sol`, `requireValidTxSignatures` (lines 261–289):**

```solidity
bytes32 data = keccak256(
    abi.encodePacked(uint256(block.chainid), uint256(idx), txn)
);
```

Sequencer signatures are explicitly chain-bound. User signatures are not. This asymmetry is the concrete missing security configuration.

**Validation path — `EndpointTx.sol`, `validateSignature` (lines 172–184):**

```solidity
function validateSignature(
    bytes32 sender,
    bytes32 digest,
    bytes memory signature,
    bool allowLinkedSigner
) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
```

The `digest` is passed through directly to `Verifier.validateSignature` (lines 291–304), which only checks that `ECDSA.recover(digest, signature)` equals the sender address or linked signer. No domain binding is enforced at this layer either.

**Exploit path:**

1. User Alice signs a `WithdrawCollateral` transaction on Ink Sepolia testnet with `nonce = N`, `amount = X`, `productId = P`.
2. The Nado protocol is also deployed on Ink mainnet (same or deterministic addresses).
3. Alice's mainnet subaccount has `nonce = N` (common for new mainnet users who have not yet transacted).
4. Attacker submits Alice's testnet signature to the mainnet `Endpoint` as a slow-mode or sequencer-submitted transaction.
5. `computeDigest` produces the identical `bytes32` on mainnet (no chain binding), `validateSignature` recovers Alice's address, and the withdrawal executes — draining Alice's mainnet collateral.

The same path applies to `LiquidateSubaccount`: an attacker who obtained a user's testnet liquidation authorization signature can replay it on mainnet to trigger an unauthorized liquidation.

---

### Impact Explanation

- **Asset impact:** Unauthorized `WithdrawCollateral` drains a victim's collateral balance on the replay-target chain. The attacker receives the withdrawn tokens.
- **State impact:** Unauthorized `LiquidateSubaccount` corrupts the victim's position and transfers liquidation proceeds to the attacker's subaccount.
- **Scope:** Every user who has signed any transaction on one chain deployment is at risk on any other deployment where their nonce matches — which is the default state for all users at mainnet launch.

Severity: **High** — direct, unprivileged theft of user collateral with no precondition beyond possessing a previously broadcast signature.

---

### Likelihood Explanation

- Nado explicitly targets both Ink Sepolia testnet and Ink mainnet (per the wiki). Both deployments exist or will exist simultaneously.
- Testnet signatures are routinely broadcast on-chain and observable by any attacker monitoring the testnet `Endpoint`.
- New mainnet users start at `nonce = 0`, matching any testnet signature signed at `nonce = 0`. This is the default state for every user at mainnet launch.
- No privileged access is required. The attacker only needs to observe testnet transactions and submit them to mainnet.

Likelihood: **High** at mainnet launch; decreasing as users accumulate mainnet nonces that diverge from testnet nonces.

---

### Recommendation

Replace the raw struct hash in `computeDigest` with a full EIP-712 hash that includes a domain separator binding the digest to `chainId` and `verifyingContract`:

```solidity
bytes32 private constant DOMAIN_TYPEHASH = keccak256(
    "EIP712Domain(string name,uint256 chainId,address verifyingContract)"
);

function domainSeparator() public view returns (bytes32) {
    return keccak256(abi.encode(
        DOMAIN_TYPEHASH,
        keccak256(bytes("Nado")),
        block.chainid,
        address(this)
    ));
}

function computeDigest(...) public view returns (bytes32) {
    bytes32 structHash = keccak256(abi.encode(...));
    return keccak256(abi.encodePacked("\x19\x01", domainSeparator(), structHash));
}
```

This is the direct analog to the Android fix: just as `cleartextTrafficPermitted="false"` enforces that all connections use HTTPS, the domain separator enforces that all signatures are bound to a specific chain and contract — preventing replay across deployments.

### Proof of Concept

**Preconditions:**
- Nado deployed on Ink Sepolia (testnet) and Ink mainnet.
- Victim Alice has `nonce = 0` on mainnet (no prior mainnet transactions).

**Steps:**

1. Alice signs a testnet `WithdrawCollateral` with `nonce = 0`, `amount = 1000e6`, `productId = 0` (USDC).
2. The testnet `Endpoint` broadcasts the transaction; the signature is publicly visible on-chain.
3. Attacker calls `Verifier.computeDigest(WithdrawCollateral, encodedTx)` on mainnet — receives the identical `bytes32` as on testnet (no chain binding).
4. Attacker submits the same `SignedWithdrawCollateral` struct (with Alice's testnet signature) to the mainnet `Endpoint` via `submitSlowModeTransaction` or the sequencer submission path.
5. `validateSignature` recovers Alice's address from the testnet signature against the mainnet digest (identical), passes the check, and the withdrawal executes.
6. Alice's mainnet USDC collateral is transferred to the attacker-controlled recipient address.
