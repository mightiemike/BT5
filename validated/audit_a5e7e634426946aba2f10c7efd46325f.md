### Title
Missing `max_l2_gas_amount` Validation for Declare Transactions Allows Gateway Admission Policy Bypass - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The stateless gateway validator enforces a configured `max_l2_gas_amount` ceiling on L2 gas resource bounds for Invoke and DeployAccount transactions, but explicitly skips this check for Declare transactions. An unprivileged user can submit a Declare transaction with `l2_gas.max_amount` set to `u64::MAX`, bypassing the gateway's own configured admission limit and causing the gateway to accept a transaction it is configured to reject.

### Finding Description

In `StatelessTransactionValidator::validate_resource_bounds`, the check for `max_l2_gas_amount` is gated behind a type-match that short-circuits for Declare transactions:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { ... });
}
``` [1](#0-0) 

The `StatelessTransactionValidatorConfig` defines `max_l2_gas_amount` (default: `1_210_000_000`) as a hard admission ceiling: [2](#0-1) [3](#0-2) 

The production deployment config confirms this limit is active: [4](#0-3) 

The `validate_resource_bounds` flag is a shared pointer target that simultaneously controls the stateless validator, stateful validator, and mempool: [5](#0-4) 

When `validate_resource_bounds = true` (the production default), the stateless validator enforces the `max_l2_gas_amount` ceiling for Invoke and DeployAccount, but the Declare branch is an empty arm that falls through without any check. The stateful validator only checks the L2 gas *price* against the previous block's price, not the gas *amount*: [6](#0-5) 

The test suite explicitly confirms that a Declare transaction with `l2_gas.max_amount` exceeding the configured limit is accepted as valid: [7](#0-6) 

### Impact Explanation

**High. Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

A Declare transaction with `l2_gas.max_amount = u64::MAX` passes the stateless validator, passes the stateful validator (which only checks price), and is admitted into the mempool. The gateway's own configured `max_l2_gas_amount` admission policy is not enforced for this transaction type. The admitted transaction carries a resource-bound field that is inconsistent with the node's configured policy, which can affect bouncer weight accounting during block building — the batcher uses the declared `max_amount` as an upper bound on block gas consumption when deciding whether to include a transaction.

### Likelihood Explanation

**High.** Any unprivileged user with access to the public gateway endpoint can submit a well-formed Declare V3 transaction with an arbitrarily large `l2_gas.max_amount`. No special account state, privileged key, or malformed bytes are required. The bypass is unconditional for all Declare transactions regardless of the configured limit value.

### Recommendation

Remove the empty `Declare` arm and apply the same `max_l2_gas_amount` ceiling to Declare transactions as is applied to Invoke and DeployAccount transactions:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If Declare transactions legitimately require a higher L2 gas ceiling (e.g., due to compilation overhead), introduce a separate `max_l2_gas_amount_declare` config field rather than removing the check entirely.

### Proof of Concept

1. Construct a valid `RpcDeclareTransaction::V3` with:
   ```
   resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)
   resource_bounds.l2_gas.max_price_per_unit = GasPrice(min_gas_price)  // satisfies price check
   ```
2. Submit to the gateway's `add_declare_transaction` endpoint.
3. The stateless validator reaches line 79, matches `RpcTransaction::Declare(_)`, enters the empty arm, and returns `Ok(())` without checking `max_l2_gas_amount`.
4. The stateful validator checks only the L2 gas price (not amount) and passes.
5. The transaction is admitted to the mempool with `l2_gas.max_amount = u64::MAX`, violating the node's configured `max_l2_gas_amount = 1_210_000_000` policy.
