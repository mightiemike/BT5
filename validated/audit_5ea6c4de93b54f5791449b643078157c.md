### Title
Stale `LastKnownBonded=true` After Redelegate to Unbonding Validator Causes Bonus Overclaim on Re-bond — (`x/tieredrewards/keeper/msg_server.go`, `claim_rewards.go`)

### Summary

`TierRedelegate` unconditionally calls `pos.UpdateBonusCheckpoints(latestSeq, blockTime, true)` with a hardcoded `true` for `LastKnownBonded`. If `dstVal` appended an UNBOND event at seq K in the same block before `TierRedelegate` executes, `getValidatorEventLatestSeq` returns K and the position's `LastEventSeq` is set to K with `LastKnownBonded=true`. Because `getValidatorEventsSince` uses `StartExclusive(K)`, the UNBOND event itself is never replayed. `LastKnownBonded` is permanently stuck at `true`. When `dstVal` later re-bonds and a BOND event is appended at K+1, the next `ClaimTierRewards` call processes the BOND event with `bonded=true` (from `LastKnownBonded`), computes bonus for the entire unbonded interval `[blockTime_N, blockTime_M]`, and pays it out — a direct overclaim.

### Finding Description

**Step 1 — UNBOND event appended before redelegate (same block):**

`AfterValidatorBeginUnbonding` fires for `dstVal` and appends an UNBOND event at seq K. [1](#0-0) 

**Step 2 — `TierRedelegate` reads latest seq and sets checkpoints:** [2](#0-1) 

`getValidatorEventLatestSeq(dstVal)` returns K (the UNBOND event seq). Then `UpdateBonusCheckpoints(K, blockTime, true)` is called with a **hardcoded `true`** for `LastKnownBonded`, regardless of whether the event at K is an UNBOND.

**Step 3 — `getValidatorEventsSince` skips the UNBOND event:** [3](#0-2) 

The range uses `StartExclusive(startSeq)`, so events with seq ≤ K are never returned. The UNBOND event at K is permanently invisible to this position.

**Step 4 — Defensive check in `processEventsAndClaimBonus` only protects the immediate claim:** [4](#0-3) 

When `dstVal` is still unbonded and no new events exist, `bonded && val.IsBonded()` → `true && false` → no current-segment bonus. This looks safe. But: [5](#0-4) 

`pos.UpdateLastKnownBonded(bonded)` persists `bonded = true` because `bonded` was never set to `false` (the UNBOND event was skipped). `LastKnownBonded` remains `true` indefinitely.

**Step 5 — Re-bond triggers the overclaim:**

When `dstVal` re-bonds, `AfterValidatorBonded` appends a BOND event at seq K+1. The next `ClaimTierRewards`: [6](#0-5) 

- `bonded = pos.LastKnownBonded = true`
- `segmentStart = pos.LastBonusAccrual = blockTime_N` (redelegate block)
- Events since K: `[BOND at K+1, timestamp=blockTime_M]`
- Loop: `if bonded` → **true** → `computeSegmentBonus(segmentStart=blockTime_N, segmentEnd=blockTime_M)` → **bonus paid for the entire unbonded interval**

The validator was unbonded during `[blockTime_N, blockTime_M]`, but the position receives full bonus for that period.

### Impact Explanation

Direct bonus token overclaim from the `RewardsPoolName` module account. The attacker receives bonus rewards for a period during which their delegated validator was not bonded. The magnitude scales with the unbonded duration and the position's delegation size. This is a concrete fund loss from the rewards pool.

### Likelihood Explanation

The precondition — `dstVal` unbonding in the same block as `TierRedelegate` — is realistic. Validator state transitions (`AfterValidatorBeginUnbonding`) fire during `EndBlock` staking processing, which runs before transaction processing in the next block in Cosmos SDK. More precisely, in Cosmos SDK the staking `EndBlocker` runs at the end of a block, and hooks fire during that same block. A user can observe that `dstVal` is about to be kicked out (e.g., due to downtime) and submit `TierRedelegate` targeting that validator in the same block. No privileged access is required — only a standard `MsgTierRedelegate` transaction.

### Recommendation

In `TierRedelegate`, instead of hardcoding `true` for `LastKnownBonded`, inspect the actual current bonded state of `dstVal` at the time of the call:

```go
dstVal, err := ms.stakingKeeper.GetValidator(ctx, dstValAddr)
if err != nil {
    return nil, err
}
pos.UpdateBonusCheckpoints(latestSeq, sdkCtx.BlockTime(), dstVal.IsBonded())
```

This ensures that if `dstVal` is already unbonding/unbonded at redelegate time, `LastKnownBonded` is set to `false`, and the UNBOND event's effect is correctly reflected even though it is skipped by `getValidatorEventsSince`.

### Proof of Concept

```
Block N:
  EndBlocker: dstVal begins unbonding
    → AfterValidatorBeginUnbonding fires
    → appendValidatorEvent(dstVal, UNBOND) → seq K

  Tx: MsgTierRedelegate(positionId=1, dstValidator=dstVal)
    → claimRewards(pos) against srcVal (advances srcVal seq)
    → getValidatorEventLatestSeq(dstVal) = K
    → UpdateBonusCheckpoints(K, blockTime_N, true)
       pos.LastEventSeq = K
       pos.LastKnownBonded = true   ← BUG: should be false
       pos.LastBonusAccrual = blockTime_N

Block M (dstVal re-bonds):
  AfterValidatorBonded fires
    → appendValidatorEvent(dstVal, BOND) → seq K+1

Block M+1:
  Tx: MsgClaimTierRewards(positionId=1)
    → processEventsAndClaimBonus:
       bonded = true (LastKnownBonded)
       segmentStart = blockTime_N
       events = [BOND at K+1, ts=blockTime_M]
       loop: bonded=true → computeSegmentBonus(blockTime_N, blockTime_M)
             → OVERCLAIM: bonus for unbonded period paid out
```

Assert: `bonusCoins > 0` for the interval `[blockTime_N, blockTime_M]` despite `dstVal` being unbonded throughout.

func (k Keeper) getValidatorEventsSince(ctx context.Context, valAddr sdk.ValAddress, startSeq uint64) ([]EventEntry, error) {
	// Range from (valAddr, startSeq+1) to end of valAddr prefix.
	rng := collections.NewPrefixedPairRange[sdk.ValAddress, uint64](valAddr).
		StartExclusive(startSeq)

	var entries []EventEntry
	err := k.ValidatorEvents.Walk(ctx, rng, func(key collections.Pair[sdk.ValAddress, uint64], event types.ValidatorEvent) (bool, error) {
		entries = append(entries, EventEntry{Seq: key.K2(), Event: event})
		return false, nil
	})
	if err != nil {
		return nil, err
	}
	return entries, nil
}
