Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension` is documented as gating `addLiquidity` by depositor address, but its `beforeAddLiquidity` hook discards the `sender` parameter (the actual depositor, `msg.sender` of the pool call) and instead checks `owner` (the LP position recipient, a free caller-controlled argument). Any unprivileged caller can bypass the allowlist by supplying any allowlisted address as `owner`, completely defeating the pool admin's access control.

## Finding Description
In `MetricOmmPool.addLiquidity` (line 191), the pool calls `_beforeAddLiquidity(msg.sender, owner, ...)` where `msg.sender` is the actual depositor and `owner` is a free parameter supplied by the caller. [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both as `(sender, owner, ...)` to the extension via `abi.encodeCall`. [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` declares its first parameter as unnamed `address` (silently discarding `sender`) and checks `owner` against `allowedDepositor[msg.sender][owner]`. [3](#0-2) 

Since `owner` is freely chosen by the caller in `addLiquidity(address owner, ...)`, any caller sets `owner` to any allowlisted address and the guard passes unconditionally regardless of who the actual depositor (`sender`) is. By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (first parameter), confirming the asymmetry is a bug, not a design choice. [4](#0-3) 

## Impact Explanation
The pool admin's deposit allowlist is fully defeated by any unprivileged external caller with knowledge of one allowlisted address (readable from the public `allowedDepositor` mapping). Unauthorized liquidity enters the pool, breaking the core access-control invariant the extension is designed to enforce. Additionally, the allowlisted `owner` receives an unsolicited LP position they did not request, funded by the attacker's tokens — the allowlisted address can then call `removeLiquidity` to recover those tokens, constituting a forced transfer. This is a broken core pool functionality / admin-boundary break caused by an unprivileged path. [5](#0-4) 

## Likelihood Explanation
Exploitation requires no special privileges, flash loans, or oracle manipulation. The attacker only needs to: (a) read `allowedDepositor[pool][addr]` to find one allowlisted address (public state), and (b) call `pool.addLiquidity(owner=allowlisted_addr, ...)`. The condition is trivially satisfiable on any pool with at least one allowlisted depositor, and is repeatable indefinitely. [6](#0-5) 

## Recommendation
Replace the `owner` check with a `sender` check in `beforeAddLiquidity`:

```solidity
// FIXED
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This mirrors the correct pattern already used in `SwapAllowlistExtension.beforeSwap`. [4](#0-3) 

## Proof of Concept
```
Setup:
  Pool P has DepositAllowlistExtension configured.
  allowedDepositor[P][alice] = true
  bob is NOT in the allowlist.

Attack:
  1. bob calls P.addLiquidity(owner=alice, salt=S, deltas=D, callbackData, extensionData)
  2. Pool calls _beforeAddLiquidity(sender=bob, owner=alice, ...)
  3. Extension receives (_, alice, ...) — bob (sender) is silently dropped.
  4. Check: allowedDepositor[P][alice] == true → no revert.
  5. Pool credits LP position to alice at salt S.
  6. Pool calls bob.metricOmmSwapCallback(...) — bob's tokens are pulled into the pool.
  7. alice holds the LP position worth bob's deposited tokens.
  8. alice calls P.removeLiquidity(owner=alice, salt=S, ...) and recovers the tokens.

Result:
  - bob's deposit bypasses the allowlist entirely.
  - alice receives an unsolicited LP position (and can drain it).
  - The pool holds unauthorized liquidity, defeating the pool admin's access control.
``` [3](#0-2)
