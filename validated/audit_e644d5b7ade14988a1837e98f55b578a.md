### Title
Approve Without Prior Zero-Reset Bricks `creditDeposit()` for USDT-Like Tokens — (`core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` directly without first resetting the allowance to zero. For ERC20 tokens like USDT that revert on non-zero-to-non-zero allowance changes, any residual allowance left after a prior deposit cycle permanently bricks the `creditDeposit()` flow for that token.

---

### Finding Description

In `DirectDepositV1.creditDeposit()`, for each supported spot product, the contract approves the endpoint for the full `balance` (a `uint256`) and then calls `depositCollateralWithReferral` with `uint128(balance)`: [1](#0-0) 

The critical lines:

```solidity
token.approve(address(endpoint), balance);          // balance is uint256
endpoint.depositCollateralWithReferral(
    subaccount,
    productId,
    uint128(balance),   // truncated to uint128
    "-1"
);
``` [2](#0-1) 

The `approve` amount is `balance` (uint256), but the endpoint is only instructed to pull `uint128(balance)`. If `balance > type(uint128).max`, the endpoint's `transferFrom` consumes only `uint128(balance)` tokens, leaving a residual allowance of `balance - uint128(balance)` on the DDA contract. On the next invocation of `creditDeposit()` for a USDT-like token, `token.approve(address(endpoint), newBalance)` reverts because the current allowance is non-zero.

Contrast this with `ContractOwner.wrapVaultAsset()`, where the developers correctly applied the zero-reset pattern:

```solidity
assetToken.approve(tokenAddr, 0);
assetToken.approve(tokenAddr, assetBalance);
``` [3](#0-2) 

This shows awareness of the pattern in one place but it was missed in `DirectDepositV1`.

The `IIERC20Base` interface used inside `DirectDepositV1` exposes a raw `approve` with no safe wrapper: [4](#0-3) 

---

### Impact Explanation

If a residual allowance exists on the DDA contract for a USDT-like token, every subsequent call to `creditDeposit()` reverts at the `approve` line for that token. This permanently prevents users from depositing that collateral type through the `DirectDepositV1` mechanism. Funds sent to the DDA contract become stuck — they cannot be credited to the subaccount via the normal `creditDeposit()` path.

---

### Likelihood Explanation

The concrete trigger for the residual allowance is `balance > type(uint128).max`. For tokens with 6 decimals (USDT, USDC), this threshold is astronomically large and practically unreachable. For 18-decimal tokens it requires ~3.4 × 10^20 tokens in the DDA, also extremely unlikely. A secondary trigger exists if the endpoint's `depositCollateralWithReferral` does not consume the full approved amount (e.g., due to internal deposit caps or fee deductions), but this could not be confirmed from the available contract code. Likelihood is **low** under normal operating conditions, but the pattern is a latent correctness bug that becomes exploitable if token balances or endpoint behavior ever satisfy the trigger.

---

### Recommendation

Apply the zero-reset pattern before every `approve` call in `creditDeposit()`, consistent with what is already done in `wrapVaultAsset()`:

```solidity
token.approve(address(endpoint), 0);
token.approve(address(endpoint), balance);
```

Alternatively, use OpenZeppelin's `SafeERC20.safeIncreaseAllowance` or `forceApprove`.

---

### Proof of Concept

1. A USDT-like token is listed as a spot product.
2. A balance `B > type(uint128).max` accumulates in the `DirectDepositV1` contract (or the endpoint only consumes `uint128(B)` of the approved `B`).
3. `creditDeposit()` is called (permissionlessly via `ContractOwner.creditDepositV1(subaccount)`): `approve(endpoint, B)` succeeds; endpoint pulls `uint128(B)`; residual allowance = `B - uint128(B) > 0`.
4. Any subsequent call to `creditDeposit()` for that token hits `token.approve(endpoint, newBalance)` — USDT reverts because allowance ≠ 0.
5. The DDA is permanently bricked for that token; user funds sent to the DDA cannot be credited to the subaccount. [5](#0-4) [6](#0-5)

        uint32[] memory productIds = spotEngine.getProductIds();
        for (uint256 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            address tokenAddr = spotEngine.getToken(productId);
            require(tokenAddr != address(0), "Invalid productId.");
            IIERC20Base token = IIERC20Base(tokenAddr);
            uint256 balance = token.balanceOf(address(this));
            if (balance != 0) {
                token.approve(address(endpoint), balance);
                endpoint.depositCollateralWithReferral(
                    subaccount,
                    productId,
                    uint128(balance),
                    "-1"
                );
            }
        }
    }
