### Title
Signature-Domain Canonicalization Gap in `SnapshotHostInfo`: Malicious Relay Can Reorder `shards` to Invalidate Legitimate Signatures — (`chain/network/src/network_protocol/state_sync.rs`, `chain/network/src/network_protocol/proto_conv/peer_message.rs`)

### Summary

`SnapshotHostInfo::build_hash` hashes the `shards: Vec<ShardId>` field in its stored order with no canonicalization. The protobuf-to-Rust conversion `TryFrom<&proto::SnapshotHostInfo>` also preserves proto field order without sorting. A malicious relay peer can reorder the `shards` repeated field in a `SyncSnapshotHosts` protobuf message, producing a different Borsh-serialized byte sequence, a different `CryptoHash`, and a guaranteed `InvalidSignature` result in `verify()`. The legitimate peer's info is silently discarded, removing it from state sync peer discovery.

### Finding Description

**Signing path** (`state_sync.rs` lines 41–60):

`SnapshotHostInfo::new` calls `build_hash(&sync_hash, &epoch_height, &shards)` which calls `CryptoHash::hash_borsh((sync_hash, epoch_height, shards))`. The `shards` Vec is hashed in whatever order it was passed in — no sort, no dedup, no canonicalization. [1](#0-0) 

**Deserialization path** (`peer_message.rs` lines 127–138):

`TryFrom<&proto::SnapshotHostInfo>` reconstructs `shards` directly from the proto repeated field:
```rust
shards: x.shards.clone().into_iter().map(Into::into).collect(),
```
No sorting is applied. The proto wire order is preserved verbatim. [2](#0-1) 

**Verification path** (`state_sync.rs` lines 67–79):

`verify()` calls `self.hash()` → `build_hash(&self.sync_hash, &self.epoch_height, &self.shards)`, hashing `self.shards` in its stored (attacker-controlled) order. [3](#0-2) 

**No sorting anywhere**: A search for `sort.*shards` or `shards.*sort` across the entire production codebase returns zero matches. Neither the signing site nor the deserialization site enforces a canonical shard order.

**Concrete divergence**:
- Signed hash: `CryptoHash::hash_borsh((sync_hash, epoch_height, &[shard_0, shard_1]))`
- Verified hash after relay reorders: `CryptoHash::hash_borsh((sync_hash, epoch_height, &[shard_1, shard_0]))`

These are distinct byte sequences under Borsh (a `Vec<u64>` is length-prefixed then elements in order), so the hashes differ and `signature.verify(reordered_hash, public_key)` returns `false`.

**Propagation path**: The `SyncSnapshotHosts` message is flooded across the P2P network. Any intermediate peer that relays the message can reorder the `shards` repeated field in the protobuf encoding before forwarding. The receiving node calls `SnapshotHostsCache::insert` → `verify()` → `InvalidSignature`, and the info is dropped. [4](#0-3) 

### Impact Explanation

The affected peer's `SnapshotHostInfo` is discarded from the receiving node's `SnapshotHostsCache`. The receiving node will not discover that peer as a state snapshot host for the relevant shard(s), degrading or blocking state sync peer selection for those shards. The impact is scoped to **state sync liveness** (peer discovery failure), not consensus safety or ledger correctness. [5](#0-4) 

### Likelihood Explanation

Requires a malicious relay peer in the P2P network — an unprivileged position (no validator/admin privileges needed). Protobuf repeated field reordering is trivially achievable by any peer that parses and re-serializes the message. The attack is targeted (specific peer's info is dropped at specific receiving nodes) and silent (no error is surfaced beyond a metric/log).

### Recommendation

Canonicalize the `shards` field before hashing and signing. In `SnapshotHostInfo::new`, sort (and optionally dedup) the `shards` Vec before calling `build_hash`. In `TryFrom<&proto::SnapshotHostInfo>`, sort the deserialized shards before constructing the struct. This ensures `build_hash` always operates on the same canonical byte sequence regardless of wire order.

```rust
// In new():
let mut shards = shards;
shards.sort_unstable();
let hash = Self::build_hash(&sync_hash, &epoch_height, &shards);

// In TryFrom:
let mut shards: Vec<ShardId> = x.shards.clone().into_iter().map(Into::into).collect();
shards.sort_unstable();
```

### Proof of Concept

```rust

    async fn verify(
        &self,
        data: Vec<Arc<SnapshotHostInfo>>,
    ) -> (Vec<Arc<SnapshotHostInfo>>, Option<SnapshotHostInfoError>) {
        // Filter out any data which is invalid, outdated or which we already have.
        if data.iter().map(|d| d.peer_id.clone()).collect::<HashSet<_>>().len() != data.len() {
            return (vec![], Some(SnapshotHostInfoError::DuplicatePeerId));
        }
        let new_data = {
            let inner = self.0.lock();
            data.into_iter().filter(|d| !d.shards.is_empty() && inner.is_new(d)).collect_vec()
        };
        // Verify the signatures in parallel.
        // Verification will stop at the first encountered error.
        let (data, verification_result) = concurrency::rayon::run(move || {
            concurrency::rayon::try_map_result(new_data.into_iter().par_bridge(), |d| {
                match d.verify() {
                    Ok(()) => Ok(d),
                    Err(err) => Err(err),
                }
            })
        })
        .await;
        match verification_result {
            Ok(()) => (data, None),
            Err(err) => (data, Some(SnapshotHostInfoError::VerificationError(err))),
        }
```

```rust
        // Execute verification on the rayon threadpool.
        let (data, err) = self.verify(data).await;
```
