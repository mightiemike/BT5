### Title
`op_mod` (Opcode 61) Missing `DISABLE_OP` Guard Present on Sibling `op_modpow` (Opcode 60) — (`File: src/chia_dialect.rs`)

---

### Summary

`DISABLE_OP` is a `ClvmFlags` bit included in `MEMPOOL_MODE`. In the `ChiaDialect::op()` dispatch, opcode 60 (`op_modpow`) correctly gates itself behind this flag and returns `EvalErr::Unimplemented` when the flag is set. Opcode 61 (`op_mod`) — the sibling modular-arithmetic operator introduced alongside `op_modpow` — has no such guard. An attacker-controlled CLVM program can invoke `op_mod` in mempool mode even though the flag architecture was designed to prevent exactly that, producing a consensus divergence between mempool validation and block validation.

---

### Finding Description

In `src/chia_dialect.rs`, `MEMPOOL_MODE` is defined as:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)      // ← included
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [1](#0-0) 

The `DISABLE_OP` flag is defined generically (not as `DISABLE_MODPOW`), signalling it is a category-level gate:

```rust
const DISABLE_OP = 0x200;
``` [2](#0-1) 

In the operator dispatch, opcode 60 (`op_modpow`) correctly checks the flag:

```rust
60 => {
    if flags.contains(ClvmFlags::DISABLE_OP) {
        return Err(EvalErr::Unimplemented(o))?;
    }
    op_modpow
}
61 => op_mod,   // ← no DISABLE_OP check
``` [3](#0-2) 

`op_mod` (opcode 61) is the direct sibling of `op_modpow` (opcode 60) — both are modular-arithmetic operators introduced together, both appear together as GC candidates:

```rust
NodeVisitor::U32(
    2 | 7 | 9 | ... | 60 | 61 | 62 | 63,
) => true,
``` [4](#0-3) 

The missing guard on opcode 61 is structurally identical to the M-15 finding: a critical enforcement method that should be present on a contract/operator is absent, while the sibling correctly implements it.

---

### Impact Explanation

When the Chia mempool validates a spend bundle using `MEMPOOL_MODE`, it sets `DISABLE_OP`. A spend bundle containing a CLVM program that calls opcode 61 (`op_mod`) will:

- **Pass** mempool validation (because `op_mod` has no `DISABLE_OP` guard).
- **Potentially diverge** from any node or consensus path that enforces the flag differently, or from a future activation boundary where `op_mod` is supposed to be gated.

This is a **consensus divergence / flag wiring error**: the mempool accepts programs it should reject, or accepts them under different semantics than block validation, which can be exploited to craft transactions that are valid in the mempool but behave unexpectedly at the consensus layer.

---

### Likelihood Explanation

The entry path is fully attacker-controlled: any CLVM program submitted to the mempool can invoke opcode 61. No special privileges, keys, or social engineering are required. The attacker only needs to craft a spend bundle whose puzzle uses `(mod ...)` (opcode 61) while `MEMPOOL_MODE` is active.

---

### Recommendation

Add the same `DISABLE_OP` guard to opcode 61 that opcode 60 already has:

```rust
60 => {
    if flags.contains(ClvmFlags::DISABLE_OP) {
        return Err(EvalErr::Unimplemented(o))?;
    }
    op_modpow
}
61 => {
    if flags.contains(ClvmFlags::DISABLE_OP) {
        return Err(EvalErr::Unimplemented(o))?;
    }
    op_mod
}
``` [3](#0-2) 

If `op_mod` is intentionally exempt from `DISABLE_OP`, the flag should be renamed (e.g., `DISABLE_MODPOW`) and its inclusion in `MEMPOOL_MODE` documented explicitly to prevent future confusion.

---

### Proof of Concept

1. Construct a CLVM program that calls opcode 61: `(mod base modulus)` — e.g., `(61 (q . 10) (q . 3))`.
2. Submit a spend bundle containing this puzzle to a Chia full node running in mempool mode (`MEMPOOL_MODE` flags active).
3. Observe that the program executes and returns `1` (10 mod 3) instead of failing with `EvalErr::Unimplemented`.
4. Contrast with opcode 60 (`op_modpow`): submitting `(60 base exp mod)` under the same flags correctly returns `EvalErr::Unimplemented`.

The divergence is confirmed by the dispatch table at `src/chia_dialect.rs` lines 239–245: opcode 60 has the guard, opcode 61 does not. [3](#0-2)
