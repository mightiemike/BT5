### Title
Deprecated `verify_transaction_inclusion` Remains Callable, Bypassing Coinbase Merkle Proof Validation Against 64-Byte Transaction Forgery — (`contract/src/lib.rs`)

---

### Summary

The `verify_transaction_inclusion` (v1) function was superseded by `verify_transaction_inclusion_v2` specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability. However, v1 is not disabled or removed — it remains a fully reachable public NEAR contract method. Any unprivileged caller can invoke it directly, bypassing the coinbase proof check entirely and obtaining a `true` result for a forged transaction inclusion proof.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced to close the 64-byte transaction forgery attack (documented at https://www.bitmex.com/blog/64-Byte-Transactions). It does so by first verifying a coinbase Merkle proof before delegating to v1: [1](#0-0) 

The coinbase proof check is the security gate: [2](#0-1) 

However, v1 is only annotated with `#[deprecated]` — a Rust compiler hint that produces a warning at compile time but has **no runtime effect**. The function remains a live, publicly callable NEAR contract method: [3](#0-2) 

Its body computes the Merkle root directly from the caller-supplied `tx_id` with no coinbase validation: [4](#0-3) 

The `ProofArgs` struct accepted by v1 has no `coinbase_tx_id` or `coinbase_merkle_proof` fields, so the check cannot be performed even in principle: [5](#0-4) 

---

### Impact Explanation

The 64-byte attack allows an adversary to supply a 64-byte value as `tx_id` that is actually a serialized internal Merkle tree node rather than a real transaction. Without the coinbase proof anchor, the contract cannot distinguish a leaf hash from an internal node. `verify_transaction_inclusion` will return `true` for a transaction that does not exist in the block. Any NEAR contract consuming this result — e.g., a cross-chain bridge releasing funds, a proof-of-payment gate, or a settlement contract — will be deceived into accepting a fabricated Bitcoin transaction as confirmed.

---

### Likelihood Explanation

The entry path requires no privilege: any NEAR account can call `verify_transaction_inclusion` directly. The 64-byte forgery technique is well-documented and the necessary crafted inputs (a valid block hash on the mainchain plus a forged Merkle path) are constructible off-chain. The v1 function is not paused by default and carries no access-control role restriction beyond the shared `#[pause]` decorator. The analog to the original report is exact: a function that should have been overridden or disabled was left reachable, allowing the intended security mechanism to be bypassed entirely.

---

### Recommendation

Remove `verify_transaction_inclusion` as a callable NEAR contract method, or replace its body with an unconditional `env::panic_str("use verify_transaction_inclusion_v2")`. A Rust `#[deprecated]` attribute is insufficient — it only affects callers at compile time and has no effect on on-chain invocations. The function must be actively disabled at the contract dispatch level.

---

### Proof of Concept

1. Identify any block hash `B` present in `mainchain_header_to_height` (e.g., via `get_block_hash_by_height`).
2. Retrieve the stored `merkle_root` for block `B`.
3. Craft a 64-byte value `fake_tx_id` that is an internal Merkle node such that `compute_root_from_merkle_proof(fake_tx_id, index, proof_path) == merkle_root` for some attacker-chosen `proof_path`.
4. Call `verify_transaction_inclusion` with `tx_id = fake_tx_id`, `tx_block_blockhash = B`, `tx_index = index`, `merkle_proof = proof_path`, `confirmations = 1`.
5. The function returns `true`.
6. Any consuming contract that gates an action on this result (bridge release, settlement, etc.) executes as if the forged transaction were confirmed on-chain.

The same call to `verify_transaction_inclusion_v2` would fail at step 4 because the coinbase proof check would reject the forged path: [2](#0-1)

        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,

        );
