Audit Report

## Title
SwapAllowlistExtension Bypassed via Router — Sender Is Router, Not Original Trader - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`, so when a swap is routed through `MetricOmmSimpleRouter`, the `sender` seen by `SwapAllowlistExtension.beforeSwap` is the router's address, not the originating EOA. Any pool that allowlists the router to support router-mediated swaps therefore grants every user on the network the ability to bypass the per-address allowlist, defeating the extension's entire purpose.

## Finding Description
**Root cause — sender binding in `MetricOmmPool.swap`:**

```solidity
// metric-core/contracts/MetricOmmPool.sol:230-240
_beforeSwap(
    msg.sender,   // ← always the direct caller of pool.swap()
    recipient,
    ...
);
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point), the router calls `IMetricOmmPoolActions(params.pool).swap(...)` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol:72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The pool's `msg.sender` is the **router**, so `sender = router address` is forwarded to `SwapAllowlistExtension.beforeSwap`:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol:37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the router. The check resolves to `allowedSwapper[pool][router]`, not `allowedSwapper[pool][EOA]`.

**Exploit flow:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` and allowlists only specific EOAs (e.g., KYC'd addresses).
2. To support legitimate router-mediated swaps for those EOAs, the admin must also call `setAllowedToSwap(pool, router, true)`.
3. Once the router is allowlisted, any unprivileged EOA — including one explicitly not on the allowlist — calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
4. The pool receives `swap(...)` from `msg.sender = router`; `_beforeSwap` passes `sender = router` to the extension.
5. The extension checks `allowedSwapper[pool][router] == true` → passes. The unauthorized user's swap executes.

**Existing guards are insufficient:** The extension has no mechanism to recover the original EOA from the router call. There is no `tx.origin` check, no forwarded-caller field in `extensionData`, and no router-side identity injection. The `_requireExpectedCallbackCaller` check in the router only validates the pool calling back, not the user identity.

## Impact Explanation
The `SwapAllowlistExtension` is a core periphery security control designed to restrict swap access to approved addresses (e.g., for compliance, KYC, or permissioned pool operation). The bypass allows any unprivileged user to trade in a pool that the admin intended to restrict, directly violating the pool's access-control invariant. This constitutes broken core pool functionality — the allowlist gate — and can result in unauthorized parties draining liquidity or executing trades in restricted pools, causing direct loss of LP assets or protocol integrity.

## Likelihood Explanation
The condition is reachable by any unprivileged EOA with no special setup: the public router is a deployed, permissionless contract. The only prerequisite is that the pool admin has allowlisted the router (which is the expected operational configuration for any pool that supports router-mediated swaps). The attack requires a single `exactInputSingle` call and is fully repeatable.

## Recommendation
The `sender` forwarded to extensions must represent the economically relevant actor, not the intermediary. Two viable fixes:

1. **Router injects caller identity via `extensionData`:** The router encodes `msg.sender` (the original EOA) into `extensionData` and the extension decodes and checks it. This requires a convention between router and extension.
2. **Pool exposes a `swapWithSender` entry point:** Add an overload that accepts an explicit `sender` address, callable only by whitelisted routers, so the pool can pass the true originator to extensions.
3. **Extension checks `tx.origin` as a fallback:** Only viable if the pool is never called from other contracts; generally fragile.

The cleanest production fix is option 1 or 2, with the router always encoding the originating EOA and the extension verifying it.

## Proof of Concept
```solidity
// Foundry test sketch
function test_swapAllowlistBypassViaRouter() public {
    // Setup: pool with SwapAllowlistExtension; only `allowedUser` is allowlisted
    swapAllowlist.setAllowedToSwap(address(pool), allowedUser, true);
    // Admin also allowlists the router so legitimate users can use it
    swapAllowlist.setAllowedToSwap(address(pool), address(router), true);

    // Attacker: an EOA that is NOT on the allowlist
    address attacker = makeAddr("attacker");
    deal(token0, attacker, 1e18);
    vm.startPrank(attacker);
    IERC20(token0).approve(address(router), type(uint256).max);

    // Attacker routes through the public router — sender seen by extension = router
    // Extension check: allowedSwapper[pool][router] == true → passes
    router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        recipient: attacker,
        tokenIn: token0,
        zeroForOne: true,
        amountIn: 1e18,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        deadline: block.timestamp,
        extensionData: ""
    }));
    // Swap succeeds despite attacker not being on the allowlist
}
```
