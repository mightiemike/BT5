### Title
Difficulty Retarget Calculation for Fork Blocks Reads Reference Block from Main-Chain Index Instead of Fork Ancestor — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

When a fork block at a difficulty-retarget boundary is validated, `get_next_work_required` fetches the reference block at the start of the retarget interval via `get_header_by_height`. That function always reads from `mainchain_height_to_header` — the main-chain index — even when the block being validated belongs to a fork whose history diverged before that reference height. The contract therefore computes the expected difficulty using the wrong block's timestamp, allowing fork blocks with an incorrect difficulty to pass or fail validation, and corrupting the chainwork comparison that drives chain selection.

---

### Finding Description

`BlocksGetter::get_header_by_height` is implemented in `lib.rs` as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

It unconditionally resolves the height through `mainchain_height_to_header`, the index that maps heights to **main-chain** block hashes.

This function is called inside `get_next_work_required` for every chain variant whenever a retarget boundary is reached:

- **Bitcoin** (`bitcoin.rs` line 81): `blocks_getter.get_header_by_height(first_block_height)`
- **Litecoin** (`litecoin.rs` line 88): `blocks_getter.get_header_by_height(first_block_height)`
- **Dogecoin** (`dogecoin.rs` lines 292–295): `blocks_getter.get_header_by_height(height_first)` [2](#0-1) [3](#0-2) [4](#0-3) 

The developers themselves flagged this in a TODO comment:

```
// TODO: check if it is correct to get block header by height from mainchain
// without looping to find the ancestor
``` [5](#0-4) 

**Concrete trigger path:**

Consider Bitcoin with `difficulty_adjustment_interval = 2016`:

1. Main chain has blocks at heights 0–4031. Retarget boundaries are at heights 2016 and 4032.
2. A fork diverges at height 2015 (one block before the first retarget). The fork produces its own block at height 2016 with a different timestamp than the main-chain block at 2016.
3. The fork continues to height 4031. When the relayer submits the fork block at height 4032 (the second retarget boundary), `check_target` → `check_pow` → `get_next_work_required` is called.
4. `first_block_height = 4031 - 2015 = 2016`. The contract calls `get_header_by_height(2016)`.
5. This returns the **main-chain** block at height 2016, whose timestamp differs from the fork's actual ancestor at height 2016.
6. `calculate_next_work_required` uses this wrong timestamp, producing an incorrect `expected_bits`.
7. The fork block's `bits` field is validated against this wrong expected value. [6](#0-5) 

**Dogecoin is the most severe case.** Post-height 145,000, `difficulty_adjustment_interval = 1`, so `blocks_to_go_back = 1` and `first_block_height = prev_block_header.block_height - 1`. This means the wrong-reference bug fires for **every fork block** submitted after the first one, because the reference block is always one step back and is always resolved from the main-chain index rather than the fork's parent chain. [7](#0-6) 

The `check_target` call happens before `submit_block_header_inner` stores the block, so the state read during validation is the pre-submission main-chain state — exactly the desynchronization: the reference value is taken from a different chain branch than the one being validated. [8](#0-7) [9](#0-8) 

---

### Impact Explanation

The incorrect `expected_bits` corrupts the difficulty gate in two directions:

- **False acceptance**: If the wrong reference block produces a lower (easier) `expected_bits`, a fork block with that easier difficulty passes `require!(expected_bits == block_header.bits, ...)`. The block is stored with chainwork computed from the easier `bits`, inflating the fork's apparent accumulated work. If the inflated chainwork exceeds the main chain's, `submit_block_header_inner` triggers `reorg_chain`, replacing the canonical chain with the attacker's fork.
- **False rejection**: If the wrong reference block produces a harder `expected_bits`, a legitimately-mined fork block is rejected, preventing valid reorgs.

The first case directly corrupts `mainchain_tip_blockhash`, `mainchain_height_to_header`, and `mainchain_header_to_height` — the state that `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` rely on to confirm which blocks are canonical. A false reorg causes SPV proofs for transactions on the true chain to be rejected, and proofs for transactions on the attacker's fork to be accepted. [10](#0-9) [11](#0-10) 

---

### Likelihood Explanation

The entry point is `submit_blocks`, reachable by any trusted relayer (an explicitly supported production role). A malicious or compromised relayer can craft a fork with manipulated timestamps in the pre-retarget segment, then submit fork blocks at the retarget boundary with a `bits` value that matches the incorrectly computed `expected_bits`. No privileged key beyond the relayer role is required. For Dogecoin post-145,000, the bug fires on every fork block after the first, making it trivially reachable without any special timing. [12](#0-11) 

---

### Recommendation

Replace `get_header_by_height` calls inside `get_next_work_required` with a function that walks the fork's ancestor chain via `get_prev_header` to find the correct reference block at `first_block_height`. This mirrors what Bitcoin Core does: it traverses `pprev` pointers rather than looking up by height in a global index. The `get_prev_header` path already correctly follows the fork's parent chain through `headers_pool`, so the fix is to loop backward from `prev_block_header` for `blocks_to_go_back` steps instead of calling `get_header_by_height`.

---

### Proof of Concept

**Dogecoin (post-145,000, `difficulty_adjustment_interval = 1`):**

1. Main chain tip is at height 200,000. Main-chain block at height 199,999 has `time = T_main`.
2. Attacker submits a fork block at height 200,000 whose parent is the main-chain block at height 199,999 (fork diverges here). This block is stored as a fork block.
3. Attacker submits a second fork block at height 200,001 whose parent is the fork block at 200,000.
4. `check_target` → `get_next_work_required` is called. `blocks_to_go_back = 1`, `height_first = 199,999`.
5. `get_header_by_height(199,999)` returns the **main-chain** block at 199,999 with `time = T_main`.
6. But the fork's actual ancestor at 199,999 is the fork block at 200,000's parent — which is the same block here. In this step the bug does not fire yet.
7. Attacker submits fork block at height 200,002. `height_first = 200,000`. `get_header_by_height(200,000)` returns the **main-chain** block at 200,000 (with `time = T_main_200000`), but the fork's ancestor at 200,000 is the attacker's fork block (with `time = T_fork_200000 ≠ T_main_200000`).
8. `calculate_next_work_required` uses `T_main_200000` instead of `T_fork_200000`, producing wrong `expected_bits`.
9. The attacker crafts the fork block at 200,002 with `bits` matching the wrong `expected_bits`. It passes `require!(expected_bits == block_header.bits)`.
10. If the fork's cumulative chainwork (computed from the accepted `bits`) exceeds the main chain's, `reorg_chain` fires and the canonical chain is replaced. [13](#0-12) [14](#0-13)

                return config.proof_of_work_limit_bits;

                    blocks_getter.get_prev_header(&current_block_header.block_header);

            let last_bits = current_block_header.block_header.bits;
            return last_bits;
        }
        return prev_block_header.block_header.bits;

        prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);

    let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
    calculate_next_work_required(
        config,
        prev_block_header,
        interval_tail_extend_header.block_header.time.into(),
    )
}

```rust
fn get_next_work_required(
    config: &DogecoinConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    // Dogecoin: Special rules for minimum difficulty blocks with Digishield
    if allow_min_difficulty_for_block(config, block_header, prev_block_header) {
        // Special difficulty rule for testnet:
        // If the new block's timestamp is more than 2* nTargetSpacing minutes
        // then allow mining of a min-difficulty block.
        return config.proof_of_work_limit_bits;
    }

    // Only change once per difficulty adjustment interval
    let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
    let difficulty_adjustment_interval = if new_difficulty_protocol {
        1
    } else {
        config.difficulty_adjustment_interval
    };

    if (prev_block_header.block_height + 1) % difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            // Special difficulty rule for testnet:
            // If the new block's timestamp is more than 2* 10 minutes
            // then allow mining of a min-difficulty block.
            if block_header.time
                > prev_block_header.block_header.time + config.pow_target_spacing * 2
            {
                return config.proof_of_work_limit_bits;
            }

            // Return the last non-special-min-difficulty-rules-block
            let mut current_block_header = prev_block_header.clone();

            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            return current_block_header.block_header.bits;
        }

        return prev_block_header.block_header.bits;
    }

    // Litecoin: This fixes an issue where a 51% attack can change difficulty at will.
    // Go back the full period unless it's the first retarget after genesis. Code courtesy of Art Forz
    let mut blocks_to_go_back = difficulty_adjustment_interval - 1;
    if prev_block_header.block_height + 1 != difficulty_adjustment_interval {
        blocks_to_go_back = difficulty_adjustment_interval;
    }

    // Go back by what we want to be 14 days worth of blocks
    let height_first = prev_block_header
        .block_height
        .checked_sub(blocks_to_go_back)
        .unwrap_or_else(|| env::panic_str("Height underflow when calculating first block height"));

    // TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
    let first_block_time = blocks_getter
        .get_header_by_height(height_first)
        .block_header
        .time;

    calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))
