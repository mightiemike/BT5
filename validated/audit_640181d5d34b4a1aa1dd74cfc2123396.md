### Title
Trailing Bytes Silently Accepted in `deserialize_2026` — Non-Canonical Deserialization / Consensus Divergence Risk - (`src/serde_2026/de.rs`)

### Summary

`deserialize_2026` and `deser_auto` accept attacker-controlled serde_2026 blobs that contain trailing garbage bytes after a structurally valid CLVM object. The deserializer reads exactly the bytes it expects and returns `Ok(node)` without verifying that the input is fully consumed. Two distinct byte sequences — one canonical, one with appended garbage — silently produce the same `NodePtr`. This is the direct analog of the Oracle report's missing round-completeness check: data is consumed without verifying that the feed (blob) is complete and unambiguous.

---

### Finding Description

`deserialize_2026_body_from_stream` in `src/serde_2026/de.rs` drives the entire serde_2026 parse. It reads a `group_count` varint, all atom groups, an `instruction_count` varint, and exactly `instruction_count` instructions. When the stack collapses to a single node it returns:

```rust
if stack.len() != 1 {
    return Err(EvalErr::SerializationError);
}
Ok(stack[0])
```

There is no check that the underlying `reader` is exhausted. [1](#0-0) 

The slice-level entry point `deserialize_2026` wraps the blob in a `Cursor` and delegates:

```rust
pub fn deserialize_2026(allocator, blob, max_atom_len, strict) -> Result<NodePtr> {
    deserialize_2026_from_stream(allocator, &mut Cursor::new(blob), max_atom_len, strict)
}
```

After the call returns, the cursor's position is never compared to `blob.len()`. [2](#0-1) 

The Python-exposed `deser_auto` function in `wheel/src/api.rs` strips the magic prefix and calls `deserialize_2026_body_from_stream` directly on a `Cursor` wrapping the remaining body bytes, again without checking that the cursor reaches the end:

```rust
let node = if let Some(body) = blob.strip_prefix(SERDE_2026_MAGIC_PREFIX.as_slice()) {
    deserialize_2026_body_from_stream(&mut a, &mut Cursor::new(body), max_atom_len, strict)
        .map_err(eval_to_py)?
} else { ... };
``` [3](#0-2) 

The `strict` flag only governs overlong varint encodings; it does not enable trailing-byte rejection. [4](#0-3) 

---

### Impact Explanation

Any two blobs that differ only in bytes appended after the last valid instruction decode to the identical `NodePtr`. This breaks the canonical-serialization invariant that the serde_2026 format is designed to provide. Concretely:

- **Consensus divergence**: if a future Chia consensus rule gates on whether a serialized puzzle or solution is a valid serde_2026 blob, a node running clvm_rs will accept blobs with trailing bytes while a stricter implementation (or a future version of clvm_rs) rejects them. The two nodes reach different conclusions about the same transaction.
- **Non-canonical cache keys**: any component that uses the raw bytes as a cache or deduplication key (e.g., mempool, puzzle-hash cache) will treat the padded and unpadded blobs as distinct objects that map to the same tree, enabling cache-aliasing or double-spend-style confusion.
- **Python API surface**: `deser_auto` and `deser_2026` are directly callable from Python with attacker-supplied bytes, making the entry path trivially reachable. [5](#0-4) 

---

### Likelihood Explanation

The attacker-controlled entry path is the Python `deserialize` / `deser_auto` / `deser_2026` API, which accepts raw bytes from any caller. Any Chia full node or wallet that deserializes serde_2026-encoded puzzles or solutions from the network exposes this path. Crafting a blob with trailing bytes requires no cryptographic capability — appending a single `0x00` byte to any valid serde_2026 blob is sufficient to trigger the issue. The likelihood is **medium**: the serde_2026 format is new and may not yet be enforced in consensus, but the Python API is already publicly exposed.

---

### Recommendation

After `deserialize_2026_body_from_stream` returns in the slice-level entry points, verify that the reader is fully consumed. For the `Cursor`-based paths:

```rust
pub fn deserialize_2026(
    allocator: &mut Allocator,
    blob: &[u8],
    max_atom_len: usize,
    strict: bool,
) -> Result<NodePtr> {
    let mut cursor = Cursor::new(blob);
    let node = deserialize_2026_from_stream(allocator, &mut cursor, max_atom_len, strict)?;
    // Reject trailing bytes — analogous to checking round completeness
    if cursor.position() as usize != blob.len() {
        return Err(EvalErr::SerializationError);
    }
    Ok(node)
}
```

Apply the same guard in `deser_auto` (after stripping the magic prefix) and in `deser_2026`. The stream variant `deserialize_2026_from_stream` should remain unchecked, as stream callers intentionally read one object from a larger stream.

---

### Proof of Concept

```python
from clvm_rs import deser_2026, ser_2026, clvm_tree_to_lazy_node
from clvm import Program

# Serialize a simple atom
p = Program.to(b"hello")
canonical_blob = ser_2026(clvm_tree_to_lazy_node(p))

# Append trailing garbage
padded_blob = canonical_blob + b"\xde\xad\xbe\xef"

# Both succeed and produce the same tree — trailing bytes silently ignored
node_canonical = deser_2026(canonical_blob)
node_padded    = deser_2026(padded_blob)   # should raise, but does not

assert node_canonical.atom == node_padded.atom  # passes — same result, different bytes
```

The `deserialize_2026_body_from_stream` loop exits after processing `instruction_count` instructions and returns `Ok(stack[0])` without reading or rejecting the four trailing bytes. [6](#0-5)
