### Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is designed to gate `addLiquidity` by depositor address. However, its `beforeAddLiquidity` hook silently discards the `sender` parameter (the actual caller) and instead checks `owner` (the position recipient). Because `MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address with no requirement that `msg.sender == owner`, any non-allowlisted address can bypass the allowlist by supplying an allowlisted address as `owner`.

---

### Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` receives two address parameters: `sender` (the actual caller of `addLiquidity`, i.e. `msg.sender` of the pool call) and `owner` (the position recipient, an arbitrary address chosen by the caller). The function silently drops `sender` (it is unnamed) and performs the allowlist check exclusively on `owner`: [1](#0-0) 

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner`: [2](#0-1) 

Critically, unlike `removeLiquidity` which enforces `msg.sender == owner`, `addLiquidity` has **no such check**: [3](#0-2) 

The result: any non-allowlisted address `Bob` can call `pool.addLiquidity(alice, ...)` where `alice` is allowlisted. The extension checks `allowedDepositor[pool][alice]` → passes. Bob pays the tokens (via callback), Alice receives the position. The allowlist is completely defeated.

The same bypass is reachable through the periphery's `addLiquidityExactShares(pool, owner, ...)` and `addLiquidityWeighted(pool, owner, ...)` overloads that accept an explicit `owner`: [4](#0-3) 

The NatSpec for the extension explicitly states its purpose is to gate by **depositor** address: [5](#0-4) 

The `sender` is the depositor; `owner` is the beneficiary. Checking `owner` instead of `sender` is the root cause.

---

### Impact Explanation

1. **Allowlist completely ineffective**: Any non-allowlisted address can deposit into a restricted pool by naming any allowlisted address as `owner`. The pool admin's access control intent is entirely defeated.
2. **Forced position griefing**: An attacker can force unwanted LP positions onto allowlisted addresses without their consent, exposing them to impermanent loss or price risk in pools they did not choose to enter.
3. **Compliance/KYC bypass**: Pools configured with a deposit allowlist for regulatory or compliance reasons (e.g., permissioned LP pools) provide no actual restriction.

---

### Likelihood Explanation

- `addLiquidity` is a public, permissionless function with no `msg.sender == owner` guard.
- The bypass requires only knowing one allowlisted address (publicly readable from `allowedDepositor` mapping or from events).
- No special privileges, flash loans, or complex setup are required.
- The periphery `addLiquidityExactShares(pool, owner, ...)` makes the attack trivially accessible to EOAs.

---

### Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual depositor/payer) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Additionally, consider whether `addLiquidity` should enforce `msg.sender == owner` (as `removeLiquidity` does) or whether the allowlist should check both `sender` and `owner` depending on the intended semantics.

---

### Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` configured.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)` — Alice is allowlisted; Bob is not.
3. Bob calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
4. Pool calls `_beforeAddLiquidity(bob, alice, ...)` → extension checks `allowedDepositor[pool][alice]` = `true` → **passes**.
5. Bob's tokens are pulled via callback; Alice receives the LP position.
6. Bob has successfully deposited into a pool he is not allowlisted for, and Alice has an unwanted position. [1](#0-0) [6](#0-5)
