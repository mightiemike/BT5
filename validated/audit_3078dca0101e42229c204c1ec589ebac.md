### Title
`ChunkHeaderView` Round-Trip Destroys V6 (Spice) Chunk Hash Identity — (`core/primitives/src/views.rs`)

### Summary

`From<ChunkHeaderView> for ShardChunkHeader` has no arm for `ShardChunkHeaderInner::V6`. When a Spice chunk (inner discriminant `5`) is serialized to `ChunkHeaderView` and converted back, it is silently reconstructed as `ShardChunkHeaderInner::V3` (discriminant `2`). The recomputed `chunk_hash` is guaranteed to differ from the on-chain value because the Borsh byte sequences are structurally incompatible.

---

### Finding Description

**Step 1 — Original construction.**

`ShardChunkHeaderV3::new_for_spice` builds a header whose inner is `ShardChunkHeaderInner::V6(ShardChunkHeaderInnerV6SpiceTxOnly{…})` (Borsh discriminant `5`). [1](#0-0) 

The chunk hash is `combine_hash(sha256(borsh([5, prev_block_hash, encoded_merkle_root, encoded_length, height_created, shard_id, prev_outgoing_receipts_root, tx_root])), encoded_merkle_root)`. [2](#0-1) 

**Step 2 — Forward conversion to `ChunkHeaderView`.**

`From<ShardChunkHeader> for ChunkHeaderView` always writes `congestion_info: Some(inner.congestion_info().into())`. For V6, `congestion_info()` returns `CongestionInfo::default()`. [3](#0-2) 

It also writes `bandwidth_requests: None` (V6 returns `None`) and `proposed_split: None` (V6 `has_proposed_split_field()` returns `false`). [4](#0-3) [5](#0-4) 

**Step 3 — Reverse conversion from `ChunkHeaderView`.**

`From<ChunkHeaderView> for ShardChunkHeader` selects the inner version by pattern-matching `(proposed_split, bandwidth_requests, congestion_info)`. The V6 view produces `(None, None, Some(_))`, which maps to `ShardChunkHeaderInner::V3` (discriminant `2`). [6](#0-5) 

There is no `V6` arm. The code comment acknowledges this: `// TODO: store the version in ChunkHeaderView instead of guessing it from which fields are populated.` [7](#0-6) 

**Step 4 — Hash divergence.**

After reconstruction, `header.init()` recomputes the hash from the V3 inner. [8](#0-7) 

The Borsh encoding of V3 inner begins with discriminant byte `0x02` and includes fields absent from V6 (`prev_state_root`, `prev_outcome_root`, `prev_gas_used`, `gas_limit`, `prev_balance_burnt`, `prev_validator_proposals`, `congestion_info`). The Borsh encoding of V6 inner begins with `0x05` and has a completely different, shorter layout. [9](#0-8) 

The resulting `chunk_hash` of the reconstructed header is therefore a different `CryptoHash` than the on-chain value.

---

### Impact Explanation

Any production code that converts a `ChunkHeaderView` back to `ShardChunkHeader` and then calls `.chunk_hash()` will obtain a wrong hash for every Spice (V6) chunk. This includes:

- The nearcore indexer (`chain/indexer-primitives/src/lib.rs` uses `ChunkHeaderView`)
- Any RPC consumer or light client that verifies chunk identity by round-tripping through `ChunkHeaderView`

The `chunk_hash` field stored inside `ChunkHeaderView` is correct (it is copied verbatim from the original header at line 1134/1139), but the reconstructed `ShardChunkHeader` carries a recomputed, wrong hash. Any signature or Merkle verification performed against the reconstructed header will fail or produce a false result.

---

### Likelihood Explanation

This divergence is triggered by any Spice-era chunk once `ProtocolFeature::Spice` is enabled. It is not attacker-controlled in the sense of injecting malformed bytes; it is a structural loss of information in the `ChunkHeaderView` schema. Every V6 chunk that passes through the JSON RPC layer and is then re-converted will exhibit the wrong hash.

---

### Recommendation

Add an `is_spice_chunk: bool` (or a `chunk_header_inner_version: u8`) field to `ChunkHeaderView` so that `From<ChunkHeaderView> for ShardChunkHeader` can reconstruct the correct inner variant without guessing. The existing TODO comment at line 1174 already identifies this gap. Until fixed, any caller of `From<ChunkHeaderView> for ShardChunkHeader` must not rely on the resulting `chunk_hash()` for identity verification.

---

### Proof of Concept

```rust
// Demonstrates the hash divergence — no validator key needed.
let signer = EmptyValidatorSigner::default().into();
let original = ShardChunkHeader::V3(ShardChunkHeaderV3::new_for_spice(
    CryptoHash::default(), CryptoHash::default(), 0, 1,
    ShardId::new(0), CryptoHash::default(), CryptoHash::default(), &signer,
));
let original_hash = original.chunk_hash();

let view: ChunkHeaderView = original.into();
// view.congestion_info = Some(default), bandwidth_requests = None, proposed_split = None

let reconstructed: ShardChunkHeader = view.into();
// reconstructed inner is V3 (discriminant 2), not V6 (discriminant 5)
let reconstructed_hash = reconstructed.chunk_hash();

assert_eq!(original_hash, reconstructed_hash); // FAILS
```

The assertion fails because `borsh([5, …V6 fields…])` ≠ `borsh([2, …V3 fields…])`, so `combine_hash(sha256(inner_bytes), encoded_merkle_root)` produces different values.
