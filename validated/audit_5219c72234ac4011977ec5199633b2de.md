The vulnerability is real. Here is the complete trace:

**Execution path:**

1. EOA (non-allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle` — `msg.sender = EOA`
2. Router calls `pool.swap(recipient, ...)` — now `msg.sender = router` inside the pool
3. Pool's `swap` calls `_beforeSwap(msg.sender, ...)` — passes `router` as `sender`
4. `_beforeSwap` calls `extension.beforeSwap(sender=router, ...)`
5. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` = `allowedSwapper[pool][router]`

If the pool admin allowlisted the router (`allowedSwapper[pool][router] = true`), every EOA routing through it passes the check — the extension never sees the original EOA.

### Title
Router-as-sender collapses per-user swap allowlist in `SwapAllowlistExtension` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the pool's `msg.sender` — the router contract — not the originating EOA. Allowlisting the router address grants unrestricted swap access to every user of that router.

### Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` forwards it verbatim to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap` — i.e., the router: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly as the router contract, with no forwarding of the originating EOA: [4](#0-3) 

### Impact Explanation
A pool admin who wants to allow router-based swaps adds the router to the allowlist (`allowedSwapper[pool][router] = true`). This single entry grants every EOA that calls through `MetricOmmSimpleRouter` the ability to swap, regardless of whether they are individually allowlisted. The per-user gate is completely bypassed, allowing unauthorized token outflows from the pool for all router users.

### Likelihood Explanation
This is a natural and expected admin action: a pool operator who deploys `SwapAllowlistExtension` and also wants to support the canonical router will allowlist the router address. The flaw is structural — there is no way to allowlist the router without also allowlisting all its users. No malicious setup is required; the vulnerability is triggered by normal, intended usage.

### Recommendation
The extension must check the originating user, not the immediate caller. Options:
- Pass the originating EOA through `extensionData` and verify it in the extension (requires router cooperation).
- Add a separate `recipient` or `originator` field to the extension interface.
- Document clearly that allowlisting a router address grants access to all its users, and provide a router-aware extension variant that reads the payer from transient storage or `extensionData`.

### Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension`.
2. Pool admin calls `setAllowedToSwap(pool, address(router), true)` — does NOT allowlist any EOA.
3. Non-allowlisted EOA calls `router.exactInputSingle(...)`.
4. `pool.swap` is called with `msg.sender = router`; `beforeSwap` receives `sender = router`; `allowedSwapper[pool][router] = true` → check passes.
5. Swap executes; EOA receives token output. No `NotAllowedToSwap` revert.
