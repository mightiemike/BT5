### Title
CEI Violation in `withdrawCollateral` — Token Transfer Before Balance Update Enables Reentrancy — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` performs the external token transfer to the recipient via `handleWithdrawTransfer` **before** decrementing the sender's balance in the spot engine via `spotEngine.updateBalance`. This violates the Checks-Effects-Interactions pattern. If the collateral token supports transfer callbacks (e.g., ERC777 `tokensReceived`), the recipient gains execution control while the spot engine still reflects the pre-withdrawal balance, enabling a second withdrawal to pass the post-withdrawal health check with stale state.

---

### Finding Description

In `Clearinghouse.withdrawCollateral` (lines 391–421), the execution order is:

1. **Line 408** — `handleWithdrawTransfer(token, sendTo, amount, idx)` — external call chain that ultimately calls `token.safeTransfer(sendTo, amount)` inside `BaseWithdrawPool.handleWithdrawTransfer`.
2. **Line 412** — `spotEngine.updateBalance(productId, sender, amountRealized)` — balance decrement.
3. **Line 419** — `require(getHealth(sender, healthType) >= 0, ...)` — health check. [1](#0-0) 

The internal call chain for step 1 is:

- `Clearinghouse.handleWithdrawTransfer` (lines 377–385): first calls `token.safeTransfer(withdrawPool, amount)`, then calls `BaseWithdrawPool(withdrawPool).submitWithdrawal(...)`.
- `BaseWithdrawPool.submitWithdrawal` (lines 116–132): marks `markedIdxs[idx] = true`, then calls `handleWithdrawTransfer(token, sendTo, amount)`.
- `BaseWithdrawPool.handleWithdrawTransfer` (lines 184–190): calls `token.safeTransfer(to, uint256(amount))` — this is the external call that delivers tokens to the user-controlled `sendTo` address. [2](#0-1) [3](#0-2) [4](#0-3) 

At the moment `token.safeTransfer(sendTo, amount)` executes, `spotEngine.updateBalance` has **not yet been called**. The spot engine still holds the pre-withdrawal balance for `sender`. If `sendTo` is a contract with an ERC777 `tokensReceived` hook (or any token callback), it gains execution control with stale accounting state.

The `onlyEndpoint` modifier on `withdrawCollateral` is a partial guard — it prevents direct re-entry into `withdrawCollateral` itself. However, the Endpoint exposes publicly callable slow-mode processing paths (visible in `EndpointTx.processSlowModeTransactionImpl`, line 202) that route to `clearinghouse.withdrawCollateral`. An attacker who pre-queued a slow-mode withdrawal can call the slow-mode processor during the callback, triggering a second `withdrawCollateral` before the first one has updated the balance. [5](#0-4) 

---

### Impact Explanation

**Corrupted state delta**: `spotEngine` balance for `sender` is read as the full pre-withdrawal amount during the re-entrant call. The health check at line 419 passes because it evaluates against the stale (not-yet-decremented) balance. The attacker can withdraw the same collateral amount twice (or more), draining the protocol's token reserves held in `Clearinghouse` / `WithdrawPool` beyond what the subaccount is entitled to.

**Asset impact**: Direct theft of collateral tokens from the protocol. The `tvl`-equivalent accounting in the spot engine becomes permanently desynchronized from actual token balances. [6](#0-5) 

---

### Likelihood Explanation

**Medium-low.** Two conditions must hold simultaneously:

1. The collateral token for the targeted `productId` must support transfer callbacks (ERC777 or similar). Standard ERC20 tokens do not, but the protocol is designed to support multiple tokens via `spotEngine.getConfig(productId).token`, and future token listings could include ERC777-compatible assets.
2. The Endpoint's slow-mode processing must be callable by an unprivileged user during the callback window. Pre-queuing a slow-mode withdrawal is a standard user action requiring no privilege.

The `onlyEndpoint` guard on `withdrawCollateral` reduces but does not eliminate the attack surface, because the Endpoint itself is the re-entry vector via slow-mode processing.

---

### Recommendation

Apply the Checks-Effects-Interactions pattern: move `spotEngine.updateBalance` and `getHealth` **before** `handleWithdrawTransfer`. The corrected order in `withdrawCollateral` should be:

1. Compute `amountRealized`.
2. Call `spotEngine.updateBalance(productId, sender, amountRealized)`.
3. Call `spotEngine.assertUtilization(productId)`.
4. Call `require(getHealth(sender, healthType) >= 0, ...)`.
5. **Then** call `handleWithdrawTransfer(token, sendTo, amount, idx)`.

Additionally, add a `nonReentrant` modifier (OpenZeppelin `ReentrancyGuard`) to `withdrawCollateral` and to any Endpoint entry points that route to it. [7](#0-6) 

---

### Proof of Concept

**Preconditions**:
- `productId` maps to an ERC777-compatible token.
- Attacker controls subaccount `A` with balance `B` in the spot engine.
- Attacker pre-submits a slow-mode `WithdrawCollateral` transaction for amount `B` (nonce `N1`) via the Endpoint.
- Attacker also has a pending slow-mode `WithdrawCollateral` for amount `B` (nonce `N2`) queued.

**Attack steps**:

1. Sequencer processes `N1`. `Clearinghouse.withdrawCollateral` is called.
2. Execution reaches `handleWithdrawTransfer` (line 408). Tokens flow: `Clearinghouse → WithdrawPool → sendTo` (attacker contract).
3. ERC777 `tokensReceived` fires on the attacker contract. At this moment, `spotEngine` still shows balance `B` for subaccount `A`.
4. Attacker calls the Endpoint's slow-mode processor for `N2`. This calls `clearinghouse.withdrawCollateral` again for amount `B`.
5. The second call passes the health check (balance still `B` in spot engine), transfers another `B` tokens to the attacker.
6. Second call completes: `spotEngine.updateBalance` decrements balance to `0`.
7. Callback returns. First call resumes at line 412: `spotEngine.updateBalance` decrements balance from `0` to `-B` (underflow or revert depending on engine implementation).

Net result: attacker received `2B` tokens while only entitled to `B`, or the protocol state is corrupted with a negative balance. [8](#0-7) [9](#0-8)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
