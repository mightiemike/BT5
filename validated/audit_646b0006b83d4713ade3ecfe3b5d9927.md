### Title
`getCoreRisk` Hardcoded `amount=1` Causes Inflated Spread Health for Short Spot / Long Perp Positions — (`core/contracts/BaseEngine.sol`)

---

### Summary

`BaseEngine.getCoreRisk()` always passes the literal `1` as the `amount` argument to `RiskHelper._getWeightX18()`, so `CoreRisk.longWeight` is always the **long** weight regardless of the subaccount's actual position direction. When `Clearinghouse.getHealth()` computes the spread health adjustment for a **short spot / long perp** pair, it uses this field for both `existingWeight` and `spreadWeight`, producing a large positive health bonus instead of the correct small penalty. An attacker can exploit this to borrow more quote tokens than the protocol permits, creating bad debt.

---

So `CoreRisk.longWeight` is always `longWeightInitialX18` (or maintenance equivalent), even when the actual balance is negative (short).

### Impact Explanation

Concrete numerical trace (INITIAL health, `longWeightInitial = 0.9`, `shortWeightInitial = 1.1`, position size = 100, price = P):

| | **With bug** | **Correct** |
|---|---|---|
| `spotCoreRisk.longWeight` | 0.9 (long weight) | 1.1 (short weight) |
| `perpCoreRisk.longWeight` | 0.9 | 0.9 |
| `existingWeight` | 0.9 | 1.0 |
| `spreadWeight` | 0.98 | 0.99 (capped) |
| `spreadWeight − existingWeight` | **+0.08** | **−0.01** |
| Spread health adjustment | **+16P** | **−2P** |

The bug converts a `−2P` health penalty into a `+16P` health bonus — an **18P overstatement** per 100 units of spread at price P. The attacker can open a short spot + long perp spread, observe inflated health, and withdraw or borrow additional quote tokens that the protocol should not permit, leaving the protocol with bad debt when the position is eventually liquidated at a loss.

---

### Likelihood Explanation

The path is fully permissionless: any user can open a short spot position and a long perp position in a registered spread pair through the normal order/deposit flow. No admin access, sequencer compromise, or special privilege is required. The spread health path in `getHealth` is exercised on every health check (withdrawals, borrows, liquidation checks). The bug is deterministic and reproducible on an unmodified deployment.

---

### Recommendation

Pass the actual `amount` (not the literal `1`) to `_getWeightX18` inside `getCoreRisk`:

```solidity
// Before (buggy):
RiskHelper._getWeightX18(risk, 1, healthType)

// After (correct):
RiskHelper._getWeightX18(risk, amount, healthType)
``` [5](#0-4) 

This ensures `CoreRisk.longWeight` reflects the actual position direction's weight, matching the behavior of `_calculateProductHealth` which already passes the real `amount`. [6](#0-5)

### Proof of Concept

```
Setup:
  - Spread pair: spotId=1, perpId=2
  - longWeightInitialX18 = 0.9e18, shortWeightInitialX18 = 1.1e18
  - price = 1000e18

Attacker:
  1. Deposits collateral (e.g., 200 USDC quote)
  2. Opens short spot position: amount = -100 on spotId=1
  3. Opens long perp position:  amount = +100 on perpId=2

getHealth() call:
  - spotCoreRisk = getCoreRisk(subaccount, 1, INITIAL)
      → amount=-100, longWeight=_getWeightX18(risk, 1, INITIAL)=0.9e18  ← BUG
  - perpCoreRisk = getCoreRisk(subaccount, 2, INITIAL)
      → amount=+100, longWeight=0.9e18  ← correct
  - basisAmount = -max(-100, -100) = 100
  - existingWeight = (0.9e18 + 0.9e18)/2 = 0.9e18
  - spreadWeight = 1e18 - (1e18 - 0.9e18)/5 = 0.98e18
  - spread adjustment = 100 * 2000e18 * (0.98e18 - 0.9e18)
                      = 100 * 2000e18 * 0.08e18 = +16,000e18

Correct behavior (without bug):
  - spotCoreRisk.longWeight = shortWeightInitialX18 = 1.1e18
  - existingWeight = (1.1e18 + 0.9e18)/2 = 1.0e18
  - spreadWeight = min(1e18 - (1e18 - 1.1e18)/5, 0.99e18) = 0.99e18
  - spread adjustment = 100 * 2000e18 * (0.99e18 - 1.0e18) = -2,000e18

Attacker's health is overstated by 18,000e18 units.
Attacker borrows additional quote tokens up to this inflated health limit.
Protocol accrues bad debt when position is liquidated.
```
