### Title
Checkpoint Advanced Before Pool Check in `slashRedelegationPosition` Causes Permanent Bonus Loss on Partial Redelegation Slash — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

### Summary

`processEventsAndClaimBonus` advances `pos.LastBonusAccrual` (the accrual checkpoint) **before** checking whether the bonus pool has sufficient balance. When `slashRedelegationPosition` swallows `ErrInsufficientBonusPool` to avoid a chain halt, it then calls `setPositionWithState` with the already-advanced checkpoint. The accrued bonus is permanently forfeited — the position owner can never reclaim it.

### Finding Description

**Step 1 — Checkpoint advanced before pool check.**

In `processEventsAndClaimBonus`, the in-memory `pos` struct (passed by pointer) is mutated unconditionally before the pool balance is verified: [1](#0-0) 

`applyBonusAccrualCheckpoint` (line 215) and `pos.UpdateLastKnownBonded` (line 217) both write to `*pos` before `sufficientBonusPoolBalance` is called at line 230. If the pool is short, the function returns `ErrInsufficientBonusPool` with the checkpoint already advanced in the caller's `pos` variable.

**Step 2 — Error is swallowed in the slash hook.**

`slashRedelegationPosition` passes `&pos` (pointer) and explicitly ignores `ErrInsufficientBonusPool`: [2](#0-1) 

After the swallow, `pos.LastBonusAccrual` is already set to `blockTime` in memory.

**Step 3 — Mutated position is persisted.**

For a partial slash, execution falls through to: [3](#0-2) 

`setPositionWithState` is called with the `pos` that carries the advanced `LastBonusAccrual`. The checkpoint is written to the store. All future calls to `processEventsAndClaimBonus` for this position will start from `blockTime`, permanently skipping the pre-slash accrual window.

**Step 4 — Full-slash path also loses the bonus (but resets checkpoints).**

For a full slash, `pos.ClearBonusCheckpoints()` is called at line 70, which zeroes `LastBonusAccrual`. The bonus is still not paid, but since `pos.Delegation` is set to nil the position cannot earn future rewards anyway — the loss is the same. [4](#0-3) 

**Root cause in `applyBonusAccrualCheckpoint`:** [5](#0-4) 

The function mutates `pos.Position` in place with no rollback path. The pool check at line 230 of `claim_rewards.go` comes too late.

### Impact Explanation

A position owner who has an active redelegation (i.e., a `RedelegationMappings` entry exists for their `unbondingID`) permanently loses all bonus rewards accrued since their last checkpoint if:
- the source validator is slashed via `SlashRedelegation` (a normal chain event), **and**
- the `RewardsPoolName` module account balance is insufficient to cover the owed bonus at that moment.

The checkpoint is advanced, no coins are transferred, and there is no retry path — the accrual window is gone. This is a direct, permanent loss of user funds (bonus rewards denominated in the bond denom).

### Likelihood Explanation

The bonus pool can be drained to zero by normal user claims. The pool is replenished by governance or protocol mechanics, not automatically per-block. A window where the pool is empty or near-empty is realistic, especially under high claim volume or delayed replenishment. Validator slashes (downtime, double-sign) are routine chain events. The combination is not exotic.

### Recommendation

Move `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` to **after** the successful `SendCoinsFromModuleToAccount` call (or at minimum after `sufficientBonusPoolBalance` passes). In the `slashRedelegationPosition` error-swallow path, do **not** call `setPositionWithState` with the mutated `pos`; instead re-fetch or restore the original checkpoint values before persisting, so the position can retry the bonus claim once the pool is replenished.

### Proof of Concept

```
1. Fund the bonus pool with a small amount X.
2. Create a tier position P and redelegate it (creates RedelegationMappings[unbondingID] = positionID).
3. Advance block time by T days so bonus B > X accrues.
4. Drain the pool to zero via MsgClaimTierRewards on other positions.
5. Trigger SlashRedelegation on the source validator (e.g., via the slashing keeper's HandleEquivocationEvidence path).
6. BeforeRedelegationSlashed fires → slashRedelegationPosition → processEventsAndClaimBonus:
   - applyBonusAccrualCheckpoint advances pos.LastBonusAccrual to blockTime ✓
   - sufficientBonusPoolBalance returns ErrInsufficientBonusPool ✓
   - error swallowed ✓
   - setPositionWithState persists pos with advanced LastBonusAccrual ✓
7. Assert: pos.LastBonusAccrual == blockTime (checkpoint advanced).
8. Assert: owner's balance unchanged (no bonus paid).
9. Fund the pool and call MsgClaimTierRewards for position P.
10. Assert: bonus paid is zero (the pre-slash window is gone).
```

The existing test `TestProcessEvents_InsufficientPool_Error` confirms `ErrInsufficientBonusPool` is returned after the checkpoint mutation occurs, but no test covers the `slashRedelegationPosition` swallow path with a subsequent `setPositionWithState` call. [6](#0-5)

func applyBonusAccrualCheckpoint(pos *types.Position, blockTime time.Time) {
	accrualEnd := blockTime
	if pos.CompletedExitLockDuration(blockTime) {

	}
