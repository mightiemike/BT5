Audit Report

## Title
`SwapAllowlistExtension` gates the router address instead of the originating user, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is `msg.sender` of `MetricOmmPool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router contract is `msg.sender` to the pool, so the allowlist evaluates `allowedSwapper[pool][router]` — a single boolean covering every user of that router. Any unprivileged user can bypass the per-pool swap allowlist by calling the public router, completely defeating the access-control invariant the extension is designed to enforce.

## Finding Description

**Root cause — `SwapAllowlistExtension.beforeSwap`** checks `sender`, which is the direct caller of `pool.swap()`, not the originating economic actor:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (the extension is called by the pool via `CallExtension`). `sender` is the value forwarded from `MetricOmmPool.swap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // ← direct caller of pool.swap()
```

`ExtensionCalling._beforeSwap` passes this value unchanged to every configured extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L149-176
function _beforeSwap(address sender, ...) internal {
    _callExtensionsInOrder(BEFORE_SWAP_ORDER,
        abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...)));
}
```

**Exploit path — `MetricOmmSimpleRouter.exactInputSingle`** calls `pool.swap()` directly, making the router the `msg.sender` to the pool:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The same pattern applies to `exactInput` (L104), `exactOutputSingle` (L136), and `exactOutput` (L165).

**Why existing guards fail:** The allowlist check `allowedSwapper[pool][router]` is a single boolean. The pool admin must allowlist the router for any allowlisted user to use it. Once the router is allowlisted, every user — including those explicitly excluded — can call the router and pass the check, because the extension never sees the originating `msg.sender` (`bob`), only the router address.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of counterparties (e.g., KYC'd addresses, whitelisted market makers, protocol-owned accounts) is fully open to any user who routes through `MetricOmmSimpleRouter`. The attacker pays the pool's input token and receives the output token at oracle-derived prices — a complete, economically meaningful swap — without appearing on the allowlist. This breaks the core invariant the extension is designed to enforce and constitutes unauthorized extraction of pool liquidity at oracle prices. Severity is High: the access-control mechanism is completely bypassed with zero privileged access required.

## Likelihood Explanation

`MetricOmmSimpleRouter` is a public, permissionless periphery contract. No special role, token balance, or prior interaction is required beyond knowing the router address (a deployed, documented periphery contract). The bypass requires a single transaction. The precondition — the router being allowlisted — is a necessary operational condition for any allowlisted user to use the router, so it is expected to be true in any realistic deployment of this extension.

## Recommendation

Pass the originating user through the swap path so the allowlist gates the economic actor rather than the intermediary. The preferred fix is to add a `trustedForwarder` registry to `SwapAllowlistExtension`: when `sender` is a registered forwarder, the extension reads the real user from `extensionData` (encoded by the router at call time) and checks that address instead. Complementary approaches include having the router encode `msg.sender` into `extensionData` by convention, or documenting that pools using `SwapAllowlistExtension` must be called directly and are incompatible with router-mediated swaps (enforced by reverting when `sender` is a known router and no real-user override is present).

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension as beforeSwap hook.
  - Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is allowed
  - Pool admin calls setAllowedToSwap(pool, router, true)  // required for alice to use router
  - bob is NOT on the allowlist.

Attack:
  1. bob calls router.exactInputSingle({pool: pool, recipient: bob, ...})
  2. Router calls pool.swap(bob, ...) — msg.sender to pool = router address
  3. Pool calls _beforeSwap(sender=router, ...)
  4. ExtensionCalling forwards sender=router to SwapAllowlistExtension.beforeSwap
  5. Extension checks allowedSwapper[pool][router] → true
  6. Swap executes; bob receives output tokens at oracle price.

Result:
  bob, explicitly excluded from the allowlist, completes a full swap against
  the restricted pool via exactInputSingle, exactInput, exactOutputSingle, or
  exactOutput. The allowlist invariant is violated with a single transaction
  and zero privileged access.
```
