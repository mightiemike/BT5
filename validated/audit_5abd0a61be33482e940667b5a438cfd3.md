### Title
`burnNlp` Uses MAINTENANCE Health Check Instead of INITIAL, Allowing Subaccount to Drop Below Initial Health Threshold — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.burnNlp` enforces only a MAINTENANCE health check after burning NLP tokens, while every other collateral-reducing operation in the protocol (`mintNlp`, `withdrawCollateral`) enforces an INITIAL health check. This asymmetry allows a user with open leveraged perp positions to burn NLP and leave their subaccount below the initial health threshold while remaining above maintenance health, bypassing the protocol's primary risk buffer.

---

### Finding Description

`mintNlp` enforces: [1](#0-0) 

`withdrawCollateral` enforces: [2](#0-1) 

`burnNlp` enforces only: [3](#0-2) 

The in-code comment acknowledges the health-decrease risk but applies the wrong health type: [4](#0-3) 

**Health math for the burn:**

The burn removes `nlpAmount` of NLP and credits `quoteAmount = nlpAmount * price - burnFee` of quote. The net change in health for each type is:

```
ΔH_initial     = nlpAmount * price * (1 - w_nlp_initial)     - burnFee
ΔH_maintenance = nlpAmount * price * (1 - w_nlp_maintenance) - burnFee
```

The product configuration constraint enforces `longWeightInitial ≤ longWeightMaintenance ≤ 1e9`: [5](#0-4) 

Therefore `ΔH_initial ≥ ΔH_maintenance` — the burn improves initial health at least as much as maintenance health. However, because leveraged perp positions create a gap `H_m > H_i` before the burn, it is possible that:

```
H_i + ΔH_i < 0   (initial health goes negative)
H_m + ΔH_m ≥ 0   (maintenance check passes)
```

**Concrete scenario (NLP configured with `longWeightInitial = longWeightMaintenance = 1.0`):**

When both weights equal 1.0, `ΔH_initial = ΔH_maintenance = -burnFee`. The burn fee is `max(ONE, quoteAmount / 1000)`: [6](#0-5) 

If before the burn:
- `H_i = 0.5` (slightly positive initial health)
- `H_m = 1.5` (positive maintenance health, gap from leveraged perp positions)
- `burnFee = 1.0` (minimum fee, `ONE`)

After burn:
- `H_i' = 0.5 - 1.0 = -0.5 < 0` — initial health negative
- `H_m' = 1.5 - 1.0 = 0.5 ≥ 0` — maintenance check passes

The same scenario works for any NLP weight configuration where the pre-burn gap `H_m - H_i > ΔH_i - ΔH_m`, which is achievable with any leveraged perp position.

---

### Impact Explanation

A user can deliberately exit the initial health buffer zone through a `burnNlp` call. Once under initial health:
- The subaccount is in a state the protocol normally prevents through all other collateral-reducing paths
- Any adverse price movement on open perp positions can push the subaccount below maintenance health, triggering liquidation
- If the liquidation cannot fully cover the debt (e.g., due to rapid price movement or insufficient insurance), bad debt is created and socialized to the protocol

The protocol's initial health margin exists precisely to provide this buffer. Bypassing it via `burnNlp` removes the safety margin and increases the probability of bad debt creation.

---

### Likelihood Explanation

The path is fully reachable through the standard signed transaction flow:

`Endpoint → EndpointTx.processTransactionImpl (BurnNlp branch) → Clearinghouse.burnNlp` [7](#0-6) 

No special privileges are required. Any user who holds unlocked NLP and has open perp positions can trigger this. The minimum burn fee (`ONE`) is always applied, making the health decrease deterministic even for small burns. The scenario is reproducible on an unmodified local deployment.

---

### Recommendation

Replace the MAINTENANCE health check in `burnNlp` with an INITIAL health check, consistent with `mintNlp` and `withdrawCollateral`:

```solidity
// Replace:
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);

// With:
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
```

### Proof of Concept

```solidity
// Setup:
// - NLP configured with longWeightInitial = longWeightMaintenance = 1e9 (1.0)
// - Perp product with longWeightInitial = 0.8e9, longWeightMaintenance = 0.9e9

// 1. User deposits 100 USDC quote
// 2. User mints NLP worth 50 USDC → receives 50 NLP
//    Initial health after mint: 50*1.0 + 50*1.0 - 0 = 100 (passes INITIAL check)
// 3. User opens leveraged long perp position of size S at price P
//    Choose S such that:
//      H_i = 100 - S*P*(1-0.8) = 0.5  → S*P = 497.5
//      H_m = 100 - S*P*(1-0.9) = 1.5  → S*P = 985  (inconsistent, adjust)
//    More precisely: set perp position so H_i = 0.5, H_m = 1.5
//    (gap of 1.0 achievable with S*P = 10, weights 0.8/0.9 → gap = S*P*0.1 = 1.0)
// 4. User submits BurnNlp for 1 NLP at price 1.0
//    burnFee = max(ONE, 1.0/1000) = ONE = 1.0
//    quoteAmount = 1.0 - 1.0 = 0  (quoteAmount = 0, no quote credited)
//    NLP balance decreases by 1
//    ΔH_i = 0 - 1*1.0*1.0 = -1.0  (lost NLP health, no quote gained)
//    ΔH_m = 0 - 1*1.0*1.0 = -1.0
//    H_i' = 0.5 - 1.0 = -0.5 < 0
//    H_m' = 1.5 - 1.0 = 0.5 ≥ 0  → MAINTENANCE check passes
// 5. Assert: getHealth(subaccount, INITIAL) < 0  ✓
//            getHealth(subaccount, MAINTENANCE) >= 0  ✓
// 6. Subaccount is now under initial health with open leveraged perp position
//    Any adverse price move pushes it under maintenance → liquidation → potential bad debt
```
