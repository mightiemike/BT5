### Title
Protobuf `ValidResourceBounds` Converter Silently Downgrades `AllResources` to `L1Gas`, Causing Hash Preimage Divergence Across P2P Boundary - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf converter for `ValidResourceBounds` silently converts `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` to `L1Gas(X)` based solely on runtime field values. The `get_tip_resource_bounds_hash` function produces structurally different hash preimages for these two variants — `AllResources` includes a third `L1_DATA_GAS` element in the Poseidon chain while `L1Gas` omits it entirely. A V3 transaction accepted by the gateway under `AllResources` therefore carries a different canonical hash than the same transaction after a P2P protobuf round-trip, where it is silently re-classified as `L1Gas`. No deprecated-hash fallback exists for `InvokeTransaction::V3`, so `validate_transaction_hash` cannot reconcile the two values.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
