### Title
Equihash Solution Validity Result Silently Discarded — Invalid PoW Blocks Accepted into Canonical Chain - (File: `contract/src/zcash.rs`)

### Summary
In the Zcash build of the BTC light client, `equihash::is_valid_solution` returns `Result<bool, _>`. The contract handles the `Err` variant by panicking, but the `Ok(bool)` value is never bound or checked. When the library returns `Ok(false)` — a structurally valid call that found the solution to be cryptographically invalid — the result is silently discarded and the block passes PoW verification unconditionally.

### Finding Description
In `contract/src/zcash.rs`, inside `check_pow`, the Equihash validation is written as:

```rust
equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
```

`equihash::is_valid_solution` has the signature `fn(...) -> Result<bool, Error>`. The `.unwrap_or_else` closure fires only on `Err(e)`. On `Ok(false)` — the case where the library successfully evaluated the solution and determined it is **invalid** — `unwrap_or_else` returns `false` to the call site, but that `bool` is not bound to any variable and is not passed to `require!`. The statement is a no-op for the `Ok(false)` path. Execution continues and the block is stored in the canonical chain.

The correct pattern, used everywhere else in the contract for boolean guards, is:

```rust
require!(
    equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
        .unwrap_or_else(|e| env::panic_str(&format!("Invalid Equihash solution: {e}"))),
    "Invalid Equihash solution"
);
``` [1](#0-0) 

### Impact Explanation
Any unprivileged NEAR account can call `submit_blocks` with a Zcash block header whose `solution` field is cryptographically invalid. Because `check_pow` does not enforce the `false` return from `is_valid_solution`, the block passes all PoW checks and is stored in `headers_pool` and potentially promoted to the canonical mainchain via `submit_block_header_inner`. This corrupts the canonical chain state: the light client will treat blocks with no valid proof of work as confirmed mainchain blocks. Downstream contracts calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` against these blocks will receive `true` for transactions in fabricated blocks, enabling proof-verification forgery. [2](#0-1) [3](#0-2) 

### Likelihood Explanation
The entry point is `submit_blocks`, a public payable method reachable by any NEAR account. The only cost is the storage deposit for the submitted headers. No privileged role, leaked key, or social engineering is required. An attacker needs only to craft a Zcash header with a well-formed but cryptographically invalid Equihash solution — a trivial operation since the solution field is arbitrary bytes. [4](#0-3) 

### Recommendation
Wrap the `is_valid_solution` call in a `require!` so that `Ok(false)` causes the transaction to revert:

```rust
require!(
    equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
        .unwrap_or_else(|e| env::panic_str(&format!("Invalid Equihash solution: {e}"))),
    "Invalid Equihash solution"
);
```

### Proof of Concept
1. Build the Zcash variant of the contract (`make build-zcash`).
2. Initialize the contract with a valid Zcash genesis sequence.
3. Construct a `BlockHeader` for Zcash with a valid `bits`, valid `time`, valid `version`, but a `solution` field filled with 1344 zero bytes (an invalid Equihash solution).
4. Call `submit_blocks` with this header and the required storage deposit.
5. Observe that the call succeeds and `get_last_block_header` returns the submitted header — the block was accepted into the canonical chain despite having no valid proof of work.
6. Call `verify_transaction_inclusion` with a fabricated Merkle proof against this block; it returns `true`. [5](#0-4)

    #[pause]
    #[trusted_relayer]
    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
