### Title
Division by Zero in `PerpEngine.socializeSubaccount()` Permanently Freezes Subaccount Finalization — (File: `core/contracts/PerpEngine.sol`)

---

### Summary

When a subaccount is the sole participant in a perp market and closes its position (reducing `openInterest` to zero) while retaining a negative `vQuoteBalance`, the `socializeSubaccount` function reverts with a division-by-zero error. This permanently blocks the finalization path in `ClearinghouseLiq._finalizeSubaccount()`, leaving the subaccount's negative balance unresolvable and the protocol's loss-socialization mechanism permanently broken for that subaccount.

---

### Finding Description

`PerpEngine.socializeSubaccount()` is called during subaccount finalization to spread residual losses across remaining open-interest holders. The function divides the residual negative `vQuoteBalance` by `state.openInterest` to compute a per-share funding adjustment: [1](#0-0) 

```solidity
if (balance.vQuoteBalance < 0) {
    // socialize across all other participants
    int128 fundingPerShare = -balance.vQuoteBalance.div(
        state.openInterest   // ← reverts if == 0
    );
```

`state.openInterest` is the sum of absolute position amounts across all participants. It is decremented in `_updateBalance` when a position is closed: [2](#0-1) 

If the liquidatee was the only participant in a perp market and has already closed their position (`balance.amount == 0`), `openInterest` is zero. The `_finalizeSubaccount` path in `ClearinghouseLiq` explicitly requires all perp `amount` fields to be zero before calling `socializeSubaccount`: [3](#0-2) 

So the precondition for the revert (`amount == 0`, `openInterest == 0`) is enforced by the finalization check itself. The call chain is:

```
liquidateSubaccountImpl (productId == type(uint32).max)
  → _finalizeSubaccount
      → perpEngine.socializeSubaccount   ← division by zero here
``` [4](#0-3) 

The `MathSD21x18.div` function explicitly reverts on a zero denominator: [5](#0-4) 

---

### Impact Explanation

When the revert occurs, `liquidateSubaccountImpl` cannot complete. Because all perp positions are already at `amount == 0`, there is no other liquidation path available (non-finalization paths require a non-zero position to liquidate). The subaccount is permanently stuck in an unhealthy state:

- The negative `vQuoteBalance` cannot be socialized or written off.
- The insurance fund cannot be applied to cover the loss.
- The protocol's accounting for this subaccount is permanently broken.
- Any quote balance the liquidatee holds cannot be recovered through the normal finalization flow.

This is a direct analog to the external report: an arithmetic failure in the liquidation/finalization flow causes permanent asset lock.

---

### Likelihood Explanation

The trigger requires:
1. A subaccount is the sole (or last remaining) participant in a perp market.
2. The subaccount closes its position (`amount → 0`), retaining a negative `vQuoteBalance` from accumulated losses or funding payments.
3. The subaccount's quote balance is insufficient to cover the negative `vQuoteBalance`.
4. The insurance fund is also insufficient to fully cover it.
5. A liquidator submits a finalization transaction (`productId == type(uint32).max`).

This is realistic in low-liquidity or newly launched perp markets. It can occur organically (a user legitimately closes a losing position) or be deliberately engineered by an attacker who opens and closes a position in an otherwise empty market to create an unresolvable state.

---

### Recommendation

Add a guard in `socializeSubaccount` before dividing by `state.openInterest`. If `openInterest == 0` and `vQuoteBalance` is still negative after insurance coverage, the residual loss has no one to be socialized against. The protocol should handle this edge case explicitly — for example, by absorbing the residual loss into the insurance fund, emitting an event, or zeroing out the balance and adjusting the cumulative multiplier to reflect the loss:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No open interest to socialize against; absorb into insurance or zero out
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

---

### Proof of Concept

1. Deploy the protocol with a new perp market (productId = P).
2. User A (the only participant) opens a long position of size S at price X. `openInterest = S`.
3. Price drops. User A closes the position at price X' < X. `openInterest = 0`. `balance.vQuoteBalance = S * (X' - X) < 0`.
4. User A's quote balance is 0 (or insufficient). User A is now unhealthy (maintenance health < 0).
5. Liquidator calls `liquidateSubaccountImpl` with `txn.productId = type(uint32).max`.
6. `_finalizeSubaccount` passes the `require(balance.amount == 0)` check for perp P.
7. `perpEngine.socializeSubaccount(txn.liquidatee, insurance)` is called.
8. Inside, `balance.vQuoteBalance < 0` and `insurance` is insufficient.
9. `fundingPerShare = -balance.vQuoteBalance.div(state.openInterest)` → `div(0)` → revert `"DBZ"`.
10. The transaction reverts. The subaccount cannot be finalized. The negative `vQuoteBalance` is permanently unresolvable.
