### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas`, Producing Wrong Transaction Hash in P2P Sync Path — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

A hardcoded heuristic in the protobuf-to-`ValidResourceBounds` converter silently maps any `AllResources` variant whose `l1_data_gas` and `l2_gas` are both zero to the `L1Gas` variant. Because `get_tip_resource_bounds_hash` produces structurally different hash preimages for the two variants (two resource felts for `L1Gas`, three for `AllResources`), a V3 transaction submitted with zero `l1_data_gas` / `l2_gas` will carry a different hash after P2P-sync deserialization than the hash computed at ingress. Any downstream component that recomputes the hash from the deserialized struct — re-execution, simulation, or hash-based validation — will observe the wrong value.

---

### Finding Description

**Root cause — hardcoded variant-selection heuristic**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` contains:

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← wrong for AllResources txs
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

The `unwrap_or_default()` on `l1_data_gas` was introduced to tolerate 0.13.2 peers that never serialized that field. The subsequent zero-check was meant to distinguish old `L1Gas` transactions from new `AllResources` ones. However, a valid 0.13.3+ `AllResources` transaction whose `l1_data_gas` and `l2_gas` happen to be zero is indistinguishable from a 0.13.2 `L1Gas` transaction under this heuristic, so it is silently downgraded.

**Hash divergence**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` chains a different number of resource felts depending on the variant:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 felts total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 felts
    }
});
``` [2](#0-1) 

A transaction originally hashed as `AllResources` (3-felt preimage → hash H₁) will be rehashed as `L1Gas` (2-felt preimage → hash H₂ ≠ H₁) after the protobuf round-trip.

**Affected path**

The P2P sync path deserializes `DeployAccountTransactionV3` and `InvokeTransactionV3` through this converter:

```rust
let resource_bounds = ValidResourceBounds::try_from(
    value.resource_bounds.ok_or(missing("DeployAccountV3::resource_bounds"))?,
)?;
``` [3](#0-2) 

The consensus path is **not** affected because it uses the separate `AllResourceBounds::try_from` converter in `rpc_transaction.rs`, which requires `l1_data_gas` to be present and never performs the zero-check downgrade:

```rust
impl TryFrom<protobuf::ResourceBounds> for AllResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        Ok(Self {
            l1_gas: value.l1_gas.ok_or(missing("ResourceBounds::l1_gas"))?.try_into()?,
            l2_gas: value.l2_gas.ok_or(missing("ResourceBounds::l2_gas"))?.try_into()?,
            l1_data_gas: value.l1_data_gas.ok_or(missing("ResourceBounds::l1_data_gas"))?.try_into()?,
        })
    }
}
``` [4](#0-3) 

**Developer acknowledgement**

The consensus round-trip test explicitly works around this issue by injecting a non-zero `l2_gas` value before serialization:

```rust
// If all the fields of `AllResources` are 0 upon serialization,
// then the deserialized value will be interpreted as the `L1Gas` variant.
fn add_gas_values_to_transaction(transactions: &mut [ConsensusTransaction]) {
    ...
    resource_bounds.l2_gas.max_amount = GasAmount(1);
``` [5](#0-4) 

This confirms the bug is known but not fixed in the production converter.

---

### Impact Explanation

Any syncing node that receives a V3 `Invoke` or `DeployAccount` transaction with zero `l1_data_gas` and zero `l2_gas` via P2P sync will store it with the `L1Gas` variant. Downstream components that recompute the hash from the stored struct — including `starknet_simulateTransactions`, re-execution for state verification, or any call to `validate_transaction_hash` — will compute H₂ instead of H₁. This constitutes **wrong hash / wrong executable payload** bound to the transaction, matching:

> **High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

---

### Likelihood Explanation

Any unprivileged user can submit a V3 `Invoke` transaction with `l1_gas > 0`, `l2_gas = 0`, `l1_data_gas = 0`. The gateway accepts it (resource-bound validation only requires at least one non-zero bound). The transaction is stored internally as `AllResources`. When a peer syncs the block, the converter fires and the variant is silently downgraded. No special privilege or malicious peer is required.

---

### Recommendation

Remove the zero-check downgrade from `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Once support for 0.13.2 peers is dropped (as the `TODO` comment already anticipates), assert that `l1_data_gas` is present and always produce `AllResources`. Until then, add an explicit protocol-version gate so that the `L1Gas` path is only taken when the peer is known to be a pre-0.13.3 node, not based on field values alone.

---

### Proof of Concept

1. Submit an `InvokeV3` transaction to the gateway with `resource_bounds = { l1_gas: { max_amount: 1, max_price: 1 }, l2_gas: { max_amount: 0, max_price: 0 }, l1_data_gas: { max_amount: 0, max_price: 0 } }`.
2. The gateway computes hash H₁ using `AllResources` (3-felt resource preimage) and stores the transaction.
3. The transaction is included in a block and propagated via P2P sync.
4. A syncing peer deserializes the transaction through `ValidResourceBounds::try_from`: `l1_data_gas.is_zero() && l2_gas.is_zero()` → `L1Gas`.
5. Call `get_tip_resource_bounds_hash` on the deserialized struct: only 2 resource felts are chained → hash H₂ ≠ H₁.
6. Any RPC simulation or re-execution on the syncing node uses H₂, diverging from the canonical H₁ stored on the proposing node.
