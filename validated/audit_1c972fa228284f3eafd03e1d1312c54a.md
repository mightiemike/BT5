### Title
Missing Contract Existence Check in `ERC20Helper.safeTransfer` / `safeTransferFrom` Enables Silent Failed Transfers — (File: `core/contracts/libraries/ERC20Helper.sol`)

---

### Summary

`ERC20Helper` — the Nado analog of Uniswap's `TransferHelper` — performs all token transfers via low-level `.call()` without first verifying that the target token contract exists. Per EVM design, a low-level call to a non-existent (or self-destructed) address returns `(true, "")`, which passes the `require(success && (data.length == 0 || ...))` guard. This means a deposit of a destroyed token silently succeeds on-chain, crediting the depositor's protocol balance with no real asset received.

---

### Finding Description

`ERC20Helper.safeTransfer` and `ERC20Helper.safeTransferFrom` both issue a raw `.call()` to the token address:

```solidity
(bool success, bytes memory data) = address(self).call(
    abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
);
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [1](#0-0) 

The identical pattern is repeated for `safeTransferFrom`: [2](#0-1) 

Neither function checks `address(self).code.length > 0` before the call. If the token contract has been self-destructed, the EVM returns `success = true` and `data = ""`, satisfying the require and silently completing a no-op transfer.

The same unchecked low-level call pattern is independently duplicated in `DirectDepositV1.safeTransfer`: [3](#0-2) 

`ERC20Helper` is imported and used via `using ERC20Helper for IERC20Base` in both `EndpointStorage` and `Clearinghouse`: [4](#0-3) [5](#0-4) 

The critical deposit path in `EndpointStorage.handleDepositTransfer` calls both helpers back-to-back with no existence guard: [6](#0-5) 

The withdrawal path in `Clearinghouse.handleWithdrawTransfer` and `BaseWithdrawPool.handleWithdrawTransfer` also call `token.safeTransfer` through the same library: [7](#0-6) [8](#0-7) 

---

### Impact Explanation

**Deposit path (asset theft):** If a supported spot collateral token is self-destructed, an attacker can call `depositCollateral` through the `Endpoint`. `handleDepositTransfer` calls `safeTransferFrom` (no real tokens move) and then `safeTransferTo` (also a no-op), both returning success. `Clearinghouse.depositCollateral` then credits the attacker's on-chain balance with `amountRealized` units of that product. The attacker can subsequently use that phantom balance as collateral to borrow or withdraw a different, live token — draining real protocol assets. [9](#0-8) 

**Withdrawal path (user fund loss):** A legitimate user withdrawing a self-destructed token has their balance debited (`spotEngine.updateBalance` with a negative delta) while `handleWithdrawTransfer` silently no-ops. The user loses their recorded balance with no token received. [10](#0-9) 

---

### Likelihood Explanation

ERC-20 tokens can be self-destructed if they contain a `selfdestruct` opcode (common in upgradeable proxy patterns where the implementation is destroyed). The Nado protocol supports arbitrary spot collateral tokens registered via `SpotEngine`. Any registered token that is later destroyed — whether by a bug, a rug, or an upgrade gone wrong — immediately opens this path to any unprivileged caller who can submit a deposit transaction through the public `Endpoint` interface. No special privileges are required beyond being an ordinary depositor.

---

### Recommendation

**Short term:** Add a contract existence check at the top of both `ERC20Helper.safeTransfer` and `ERC20Helper.safeTransferFrom`:

```solidity
require(address(self).code.length > 0, ERR_TRANSFER_FAILED);
```

This mirrors the fix recommended for Uniswap's `TransferHelper` and ensures that a call to a non-existent address reverts rather than silently succeeding.

**Long term:** Replace the hand-rolled low-level call wrappers with OpenZeppelin's `SafeERC20` library, which already includes this existence check internally and is battle-tested. The `Airdrop.sol` contract already uses `SafeERC20.safeTransfer` correctly — the same pattern should be applied uniformly across all transfer sites. [11](#0-10) 

---

### Proof of Concept

1. A spot collateral token `T` is registered in the Nado `SpotEngine`.
2. Token `T`'s contract is self-destructed (e.g., via a buggy upgrade or malicious action by the token deployer).
3. Attacker calls `Endpoint.depositCollateral` with `productId` = T's product ID and `amount` = 1,000,000 units.
4. `EndpointStorage.handleDepositTransfer` executes:
   - `safeTransferFrom(token, attacker, 1_000_000)` → `address(T).call(...)` → T has no code → EVM returns `(true, "")` → `require` passes → no tokens move.
   - `safeTransferTo(token, clearinghouse, 1_000_000)` → same no-op.
5. `Clearinghouse.depositCollateral` calls `spotEngine.updateBalance(productId, attacker_subaccount, amountRealized)` — attacker's balance is credited.
6. Attacker calls `withdrawCollateral` for a different live token (e.g., USDC) using the phantom T balance as collateral, passing the health check.
7. `handleWithdrawTransfer` transfers real USDC to the attacker. Protocol's USDC reserves are drained. [12](#0-11) [6](#0-5) [9](#0-8)

        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
