### Title
Phantom-Index Merkle Proof Bypass via Duplicate-Leaf Aliasing in Odd-Length Blocks — (`merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

---

### Summary

`compute_root_from_merkle_proof` accepts an arbitrary `transaction_position` with no upper-bound check against the actual transaction count. Because Bitcoin's Merkle tree duplicates the last leaf when the list length is odd, the proof for the real last position `n-1` is mathematically identical to a proof for the phantom position `n`. Any unprivileged NEAR caller can therefore call `verify_transaction_inclusion` (or `verify_transaction_inclusion_v2`) with `tx_index = n` for a block that contains exactly `n` transactions (odd `n`), supply the proof for index `n-1`, and receive `true` — even though position `n` does not exist in the block.

---

### Finding Description

`compute_root_from_merkle_proof` iterates over the supplied proof hashes, using `current_position % 2` to decide left/right placement at each level, then divides by 2 to move up the tree. [1](#0-0) 

There is no guard that rejects a `transaction_position` ≥ the number of transactions in the block. The contract stores only the block header (merkle root + PoW fields); the transaction count is never recorded on-chain. [2](#0-1) 

**Why the proof for index `n-1` also satisfies index `n` (odd `n`):**

For a 3-tx block `[A, B, C]`, the tree is built over `[A, B, C, C]` (C duplicated):

```
Level 0:  A    B    C    C
Level 1:  H(A,B)   H(C,C)
Root:     H(H(A,B), H(C,C))
```

Proof for real index 2 (C): `proof = [C, H(A,B)]`

Verification with `compute_root_from_merkle_proof(C, 2, [C, H(A,B)])`:
- pos=2 (even) → `H(C, proof[0]=C)` = `H(C,C)`, pos→1
- pos=1 (odd)  → `H(proof[1]=H(A,B), H(C,C))` = root ✓

Verification with `compute_root_from_merkle_proof(C, 3, [C, H(A,B)])` (phantom):
- pos=3 (odd)  → `H(proof[0]=C, C)` = `H(C,C)`, pos→1
- pos=1 (odd)  → `H(proof[1]=H(A,B), H(C,C))` = root ✓

Both paths produce the canonical root. The function cannot distinguish them.

`verify_transaction_inclusion_v2` does not close this gap: its only addition is a coinbase proof length-equality check and a coinbase-at-index-0 validation; it then delegates directly to `verify_transaction_inclusion`. [3](#0-2) 

---

### Impact Explanation

`verify_transaction_inclusion` returns `true` for a transaction position that does not exist in the committed block. Any downstream NEAR contract or off-chain system that trusts this return value to confirm a Bitcoin transaction's inclusion can be deceived into accepting a fabricated inclusion claim. The attacker needs no special role; the function is publicly callable (only gated by `#[pause]`, which is not active under normal operation). [4](#0-3) 

---

### Likelihood Explanation

The precondition is simply that a canonical mainchain block with an odd transaction count exists in `headers_pool` and `mainchain_header_to_height` — a routine condition for any real Bitcoin block. The attacker needs only the real Merkle proof for the last transaction (publicly derivable from any Bitcoin full node or block explorer) and submits it with `tx_index` incremented by one. No privileged access, no key material, and no special chain state is required.

---

### Recommendation

1. **Enforce an upper-bound on `tx_index`**: require callers to supply the total transaction count for the block and validate `tx_index < tx_count`. Because the header does not store the count, this must be supplied as a proof argument and committed to (e.g., via a separate on-chain field or by requiring the coinbase proof length to imply a minimum tree depth).

2. **Alternatively, reject even-position proofs whose sibling equals the transaction hash itself**: at each level where `current_position` is even and `proof_hash == current_hash`, the node is a duplicate-leaf padding node; a valid real-transaction proof should never exhibit this at the leaf level for any position other than the last real leaf.

3. **Minimum fix**: add a `tx_count: u64` field to `ProofArgs`/`ProofArgsV2`, require `args.tx_index < args.tx_count`, and verify that the proof length equals `ceil(log2(tx_count))`.

---

```rust
// In merkle-tools, add to #[cfg(test)] mod tests:
#[test]
fn phantom_index_bypass() {
    // 3-tx block: odd length, last leaf C is duplicated
    let a = decode_hex("aaaa..."); // any 32-byte hash
    let b = decode_hex("bbbb...");
    let c = decode_hex("cccc...");
    let txs = vec![a.clone(), b.clone(), c.clone()];

    let root = merkle_root_calculator(&txs);

    // Legitimate proof for real index 2
    let proof_for_2 = merkle_proof_calculator(txs.clone(), 2);

    // Real index 2 verifies correctly
    assert_eq!(compute_root_from_merkle_proof(c.clone(), 2, &proof_for_2), root);

    // Phantom index 3 (does not exist) also verifies with the same proof and same tx hash
    assert_eq!(
        compute_root_from_merkle_proof(c.clone(), 3, &proof_for_2),
        root,
        "phantom index 3 must NOT equal root — but it does, confirming the bug"
    );
}
```

The second `assert_eq` passes, confirming that `compute_root_from_merkle_proof` returns the canonical root for a position that does not exist in the block. [5](#0-4) [6](#0-5) [7](#0-6)

```rust
pub fn merkle_proof_calculator(tx_hashes: Vec<H256>, transaction_position: usize) -> Vec<H256> {
    let mut transaction_position = transaction_position;
    let mut merkle_proof = Vec::new();
    let mut current_hashes = tx_hashes;

    while current_hashes.len() > 1 {
        if current_hashes.len() % 2 == 1 {
            current_hashes.push(current_hashes[current_hashes.len() - 1].clone());
        }

        if transaction_position % 2 == 1 {
            merkle_proof.push(current_hashes[transaction_position - 1].clone());
        } else {
            merkle_proof.push(current_hashes[transaction_position + 1].clone());
        }

        let mut new_hashes = Vec::new();

        for i in (0..current_hashes.len() - 1).step_by(2) {
            new_hashes.push(compute_hash(&current_hashes[i], &current_hashes[i + 1]));
        }

        current_hashes = new_hashes;
        transaction_position /= 2;
    }

    merkle_proof
}
```

```rust
#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
```
