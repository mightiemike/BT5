### Title
SwapAllowlistExtension Checks Router Address Instead of Actual User — Allowlist Fully Bypassed via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed by the pool, which is always `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the user. The extension therefore checks whether the **router** is allowlisted, not the actual user. Any pool admin who allowlists the router to support router-based trading inadvertently opens the allowlist to every user on the network.

---

### Finding Description

`MetricOmmPool.swap()` calls `ExtensionCalling._beforeSwap(msg.sender, recipient, ...)`, forwarding its own `msg.sender` as the `sender` argument to every configured extension. [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` receives that `sender` and checks it against the per-pool allowlist, where `msg.sender` inside the extension is the pool:

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
``` [2](#0-1) 

`pool.swap()` has no explicit `sender` parameter; the pool always uses `msg.sender`: [3](#0-2) 

When `MetricOmmSimpleRouter` calls `pool.swap(...)`, the pool's `msg.sender` is the router. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. [4](#0-3) 

The pool's `swap` interface and the extension's `beforeSwap` signature provide no mechanism to propagate the original EOA through the router hop.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` is a curated pool: only explicitly approved addresses may trade. The moment the pool admin allowlists `MetricOmmSimpleRouter` (the natural step to enable router-based trading), the allowlist is rendered meaningless. Every user on the network can call `router.exactInput(...)` or `router.exactOutput(...)` targeting that pool and the extension will pass because it sees `sender = router`, which is allowlisted. Unauthorized users can drain LP liquidity at oracle-fair prices, bypassing any KYC, compliance, or access-control intent the pool admin configured. This is a direct loss of the curated-pool invariant and constitutes broken core pool functionality for any pool relying on the swap allowlist.

---

### Likelihood Explanation

The trigger requires no special privilege. Any user can call the public router. The precondition — that the router is allowlisted — is the natural operational state for any pool that wants to support router-mediated multi-hop swaps. Pool admins have no documented warning that allowlisting the router opens the gate to all users. The bypass is therefore reachable on every production pool that combines `SwapAllowlistExtension` with router support.

---

### Recommendation

The pool should forward the original caller's identity through the swap interface, or the router should pass the originating user address as part of `extensionData` and the extension should decode and verify it. The cleanest fix is to add an explicit `sender` field to `pool.swap()` that the router populates with `msg.sender` before calling the pool, and to have the pool validate that only trusted periphery contracts may supply a non-`msg.sender` value for that field. Alternatively, the extension can be redesigned to check `recipient` or to decode the true originator from `extensionData`, with the router committing to a signed or authenticated user identity.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` as the `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-based trading.
3. Attacker (address not in the allowlist) calls `MetricOmmSimpleRouter.exactInput(...)` targeting the pool.
4. The router calls `pool.swap(recipient, ...)` — pool sees `msg.sender = router`.
5. Pool calls `extension.beforeSwap(router, ...)` — extension checks `allowedSwapper[pool][router]` → `true`.
6. Swap executes. Attacker successfully trades on a pool they were never authorized to access.
7. Repeat until LP position is drained or arbitrage profit is exhausted.

The allowlist check that should have blocked step 3 is silently bypassed because the identity gated by the extension (`router`) is not the identity the pool admin intended to gate (`attacker`). [2](#0-1) [1](#0-0)
