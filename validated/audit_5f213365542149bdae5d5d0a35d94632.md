### Title
Unprotected `initialize()` in `Endpoint`, `Clearinghouse`, `SpotEngine`, and `PerpEngine` Allows Front-Running to Seize Protocol Ownership and Corrupt Core State — (`core/contracts/Endpoint.sol`, `core/contracts/Clearinghouse.sol`, `core/contracts/SpotEngine.sol`, `core/contracts/PerpEngine.sol`)

---

### Summary

`Endpoint`, `Clearinghouse`, `SpotEngine`, and `PerpEngine` are upgradeable contracts whose `initialize()` functions carry no access control and whose constructors do not call `_disableInitializers()`. Any unprivileged caller can front-run the deployment transaction that calls `initialize()` on the proxy, supplying attacker-controlled addresses for every critical protocol parameter and claiming ownership of the contract. The legitimate deployer's subsequent `initialize()` call reverts because the `initializer` guard has already been consumed.

---

### Finding Description

All four contracts are deployed behind proxies and rely on a one-time `initialize()` call to configure themselves. None of them define a constructor that calls `_disableInitializers()`, and none of their `initialize()` functions check `msg.sender` against any expected deployer address.

**`Endpoint.initialize()`** — no constructor, no access control:

```solidity
function initialize(
    address _sanctions,
    address _sequencer,
    address _offchainExchange,
    IClearinghouse _clearinghouse,
    address _verifier,
    address _endpointTx          // ← delegatecall target
) external initializer {
    __Ownable_init();
    ...
    endpointTx = _endpointTx;
    ...
}
``` [1](#0-0) 

The `endpointTx` address is subsequently used as the target of a `delegatecall` inside `_delegatecallEndpointTx`:

```solidity
(bool success, bytes memory result) = endpointTx.delegatecall(callData);
``` [2](#0-1) 

**`Clearinghouse.initialize()`** — no constructor, no access control:

```solidity
function initialize(
    address _endpoint,
    address _quote,
    address _clearinghouseLiq,
    uint256 _spreads,
    address _withdrawPool
) external initializer {
    __Ownable_init();
    ...
}
``` [3](#0-2) 

**`SpotEngine.initialize()` and `PerpEngine.initialize()`** — no constructor, no access control on the outer function; the `initializer` guard lives only on the internal `BaseEngine._initialize()`:

```solidity
// SpotEngine
function initialize(...) external {          // ← no initializer modifier here
    _initialize(_clearinghouse, _offchainExchange, _endpoint, _admin);
    ...
}
``` [4](#0-3) 

```solidity
// BaseEngine
function _initialize(...) internal initializer {
    __Ownable_init();
    setEndpoint(_endpointAddr);
    transferOwnership(_admin);
    canApplyDeltas[_endpointAddr] = true;
    canApplyDeltas[_clearinghouseAddr] = true;
    canApplyDeltas[_offchainExchangeAddr] = true;
}
``` [5](#0-4) 

Contrast this with the contracts that **are** protected — `ContractOwner`, `Verifier`, `Airdrop`, and `BaseWithdrawPool` all call `_disableInitializers()` in their constructors and/or check `msg.sender`:

```solidity
// ContractOwner — protected
constructor() { _disableInitializers(); }
function initialize(...) external initializer {
    require(_deployer == msg.sender, "expected deployed to initialize");
    ...
}
``` [6](#0-5) 

The four vulnerable contracts have no equivalent protection.

---

### Impact Explanation

An attacker who wins the front-run on `Endpoint.initialize()` gains:

1. **Protocol ownership** — `__Ownable_init()` sets `msg.sender` (the attacker) as owner; all `onlyOwner` functions become attacker-controlled.
2. **Arbitrary `delegatecall` target** — `endpointTx` is set to an attacker-supplied address. Every subsequent call that routes through `_delegatecallEndpointTx` executes attacker-controlled bytecode in the storage context of `Endpoint`, enabling full state corruption or fund drainage.
3. **Sequencer substitution** — `sequencer` is set to an attacker address, allowing the attacker to submit arbitrary sequenced transactions.
4. **Clearinghouse and verifier substitution** — all downstream collateral accounting and signature verification is redirected to attacker-controlled contracts.

Front-running `SpotEngine.initialize()` or `PerpEngine.initialize()` additionally lets the attacker:
- Become owner of the engine (all `onlyOwner` product-management calls are attacker-controlled).
- Populate `canApplyDeltas` with attacker-controlled addresses, granting them the ability to call `updateBalance` and corrupt every user's position. [7](#0-6) 

---

### Likelihood Explanation

The attack window is the block gap between proxy deployment and the deployer's `initialize()` call. On any public EVM chain this is observable in the mempool. The attacker needs only to submit a higher-gas transaction calling `initialize()` with their own parameters. No special privilege, leaked key, or social engineering is required — only a standard EOA and knowledge of the proxy address (which is public once the deployment transaction is broadcast).

---

### Recommendation

For each affected contract, add a constructor that calls `_disableInitializers()` to lock the implementation contract itself, and add a deployer check inside `initialize()`:

```solidity
/// @custom:oz-upgrades-unsafe-allow constructor
constructor() {
    _disableInitializers();
}

function initialize(...) external initializer {
    require(msg.sender == expectedDeployer, "unauthorized initializer");
    ...
}
```

This mirrors the pattern already used correctly in `ContractOwner` and `Airdrop`. [8](#0-7) [9](#0-8) 

---

### Proof of Concept

```
Deploy Endpoint implementation → Deploy Endpoint proxy (points to implementation)
  ↓ (deployment tx visible in mempool)
Attacker observes proxy address, submits higher-gas tx:
  Endpoint(proxy).initialize(
      sanctions  = address(0),
      sequencer  = attacker,
      offchainExchange = maliciousExchange,
      clearinghouse    = maliciousClearinghouse,
      verifier         = maliciousVerifier,
      endpointTx       = maliciousEndpointTx   // delegatecall target
  )
  ↓
Attacker's tx mines first; initializer guard consumed.
Deployer's initialize() call reverts with "Initializable: contract is already initialized".
Protocol is now live with:
  - owner          = attacker
  - sequencer      = attacker
  - endpointTx     = maliciousEndpointTx  (arbitrary delegatecall)
  - clearinghouse  = maliciousClearinghouse
```

The same sequence applies independently to `Clearinghouse.initialize()`, `SpotEngine.initialize()`, and `PerpEngine.initialize()`. [10](#0-9) [3](#0-2) [4](#0-3) [11](#0-10)
