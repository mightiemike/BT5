### Title
`OffchainExchange.updateTierFeeRates()` Setting a Per-Product Fee Rate Unexpectedly Zeros Out the Effective Default Fee Rates for All Other Products in the Same Tier — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

When `updateTierFeeRates` is called with a specific `productId` (not `QUOTE_PRODUCT_ID`), it unconditionally marks the entire fee tier as "non-default" via `nonDefaultFeeTierMask`. This causes `getTierFeeRateX18` to return zero fee rates for every other product in that tier that has not been explicitly configured, silently replacing the hardcoded 2 bps taker default with `{0, 0}`.

---

### Finding Description

`getTierFeeRateX18` uses `nonDefaultFeeTierMask` as a gate:

```solidity
// OffchainExchange.sol lines 933–946
function getTierFeeRateX18(uint32 tier, uint32 productId)
    public view returns (FeeRates memory)
{
    if (nonDefaultFeeTierMask & (1 << tier) != 0) {
        return feeRates[tier][productId];   // returns storage slot, may be {0,0}
    }
    return FeeRates({
        makerRateX18: 0,
        takerRateX18: 200_000_000_000_000   // 2 bps default
    });
}
```

`updateTierFeeRates` has two branches, but `nonDefaultFeeTierMask |= uint128(1) << txn.tier` sits **outside** both branches and is always executed:

```solidity
// OffchainExchange.sol lines 962–991
function updateTierFeeRates(IEndpoint.UpdateTierFeeRates memory txn)
    external onlyEndpoint
{
    if (txn.productId == QUOTE_PRODUCT_ID) {
        // iterates ALL products and sets feeRates for each
        ...
    } else {
        // sets feeRates ONLY for txn.productId
        feeRates[txn.tier][txn.productId] = FeeRates(
            txn.makerRateX18,
            txn.takerRateX18
        );
    }
    nonDefaultFeeTierMask |= uint128(1) << txn.tier;  // ← always executed
}
```

When the `else` branch runs, only `feeRates[tier][txn.productId]` is populated. Every other product in that tier remains at its zero-initialized storage value `{0, 0}`. But because the tier bit is now set in `nonDefaultFeeTierMask`, `getTierFeeRateX18` will return `{0, 0}` for all those other products instead of the 2 bps default.

This is the direct analog to the reported vulnerability: a per-asset configuration write unexpectedly mutates the effective global default for all other assets sharing the same tier, in a way that is unfavorable to the protocol.

---

### Impact Explanation

Every user assigned to the affected fee tier will pay **0 taker fees** on all products except the one explicitly configured. The protocol's fee revenue for those products is silently zeroed out. This is a concrete accounting corruption: `market.collectedFees` will accumulate nothing for affected products, and `dumpFees` will transfer nothing to `X_ACCOUNT` for them. The impact scales with trading volume across all affected products.

---

### Likelihood Explanation

Medium. This is triggered by a routine, legitimate admin operation — setting a custom fee rate for a single product within a tier. It does not require any malicious intent. It would occur naturally the first time a per-product rate is set for any tier, or whenever a new product is listed after a tier has already been partially configured. The admin has no on-chain signal that other products in the tier have been affected.

---

### Recommendation

1. When the `else` branch runs, initialize all other products in the tier to the current default before setting the mask, **or**
2. Only set `nonDefaultFeeTierMask` for a tier when all products have been explicitly configured (e.g., via the `QUOTE_PRODUCT_ID` path), **or**
3. Change `getTierFeeRateX18` to fall back to the hardcoded default when `feeRates[tier][productId]` is zero/uninitialized, so that partial tier configuration does not silently zero out fees for unconfigured products.

---

### Proof of Concept

1. Protocol has products A (id=2), B (id=4), C (id=6). All users are on tier 0 (default: 0 maker, 2 bps taker).
2. Admin calls `updateTierFeeRates` with `tier=1, productId=2 (A), makerRate=0, takerRate=100_000_000_000_000` (1 bps).
3. `feeRates[1][2]` is set to `{0, 1bps}`. `feeRates[1][4]` and `feeRates[1][6]` remain `{0, 0}`.
4. `nonDefaultFeeTierMask |= 1 << 1` is set.
5. A user in tier 1 trades product B (id=4): `getTierFeeRateX18(1, 4)` hits the `nonDefaultFeeTierMask` branch and returns `feeRates[1][4]` = `{0, 0}`.
6. `applyFee` computes `feeRate = 0`, `keepRateX18 = ONE`, `orderInfo.fee = 0`. The user pays **zero taker fees** on product B.
7. `market.collectedFees` for product B receives nothing; `dumpFees` transfers nothing to `X_ACCOUNT` for product B. [1](#0-0) [2](#0-1) [3](#0-2)
