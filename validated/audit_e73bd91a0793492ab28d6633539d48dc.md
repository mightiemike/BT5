### Title
`only_query` Flag Hardcoded to `false` in `BroadcastedTransaction` → `ExecutableTransactionInput` Conversion Causes Wrong Transaction Hash and Execution Info in Fee Estimation and Simulation - (`File: crates/apollo_rpc/src/v0_8/api/mod.rs`)

---

### Summary

The RPC layer accepts query transactions (version bit 128 set, e.g., `0x100000000000000000000000000000003`) for `starknet_estimateFee` and `starknet_simulateTransactions`, but the conversion from `BroadcastedTransaction` to `ExecutableTransactionInput` unconditionally hardcodes `only_query = false`. This drops the query-version semantic entirely: the transaction hash fed into the blockifier is computed over the base version (`0x3`) rather than the query version (`0x100000000000000000000000000000003`), and the version reported to the account contract via `get_execution_info` is wrong. Any account contract that branches on the query bit receives incorrect execution context, making the simulation return an authoritative-looking but wrong fee estimate or trace.

---

### Finding Description

The Starknet protocol defines a "query version" for simulation: a transaction submitted with bit 128 set (e.g., `0x100000000000000000000000000000003` for V3) signals that it is a read-only simulation and must never be sequenced. The account contract can inspect this bit via `get_execution_info` to apply simulation-specific logic.

The RPC type system correctly models this:

```rust
// crates/apollo_rpc/src/v0_8/transaction.rs
pub enum TransactionVersion3 {
    #[serde(rename = "0x3")]
    Version3,
    #[serde(rename = "0x100000000000000000000000000000003")]
    Version3OnlyQuery,
}
``` [1](#0-0) 

However, the conversion from the accepted `BroadcastedTransaction` to the internal `ExecutableTransactionInput` unconditionally hardcodes `only_query = false` for every transaction type, with an unresolved TODO acknowledging the problem:

```rust
// crates/apollo_rpc/src/v0_8/api/mod.rs

    type Error = ErrorObjectOwned;
    fn try_from(value: BroadcastedTransaction) -> Result<Self, Self::Error> {

        match value {
            BroadcastedTransaction::Declare(tx) => Ok(tx.try_into()?),
            BroadcastedTransaction::DeployAccount(tx) => Ok(Self::DeployAccount(tx.into(), false)),
            BroadcastedTransaction::Invoke(tx) => Ok(Self::Invoke(tx.into(), false)),
        }
    }
}
``` [2](#0-1) 

The same pattern is repeated for `BroadcastedDeclareTransaction::V1`:

```rust
Ok(Self::DeclareV1(
    starknet_api::transaction::DeclareTransactionV0V1 { ... },
    sn_api_contract_class,

    // TODO(yair): pass the right value for only_query field.
    false,

``` [3](#0-2) 

The `only_query` flag is the sole carrier of the query-bit semantic after the RPC type is converted to the internal `starknet_api` type (which stores only the base version). It flows into two critical places:

**1. Transaction hash computation** — `calc_tx_hash` passes `only_query` directly to `get_transaction_hash` via `TransactionOptions`:

```rust
fn calc_tx_hash(self, chain_id: &ChainId) -> ExecutionResult<(Self, TransactionHash)> {
    match self.apply_on_transaction(|tx, only_query| {
        get_transaction_hash(tx, chain_id, &TransactionOptions { only_query })
    }) { ... }
}
``` [4](#0-3) 

`get_transaction_hash` calls `signed_tx_version_from_tx`, which adds `QUERY_VERSION_BASE` (2^128) to the version only when `only_query = true`:

```rust
pub fn signed_tx_version(
    tx_version: &TransactionVersion,
    transaction_options: &TransactionOptions,
) -> TransactionVersion {
    let query_only_bit = *QUERY_VERSION_BASE;
    if transaction_options.only_query {
        TransactionVersion(tx_version.0 + query_only_bit)
    } else {
        *tx_version
    }
}
``` [5](#0-4) 

With `only_query` hardcoded to `false`, the hash is computed over version `0x3` instead of `0x100000000000000000000000000000003`.

**2. Execution flags passed to the blockifier** — `to_blockifier_tx` propagates `only_query` into `ExecutionFlags`, which controls what version the contract sees via `get_execution_info`:

```rust
ExecutableTransactionInput::Invoke(invoke_tx, only_query) => {
    let execution_flags =
        ExecutionFlags { only_query, charge_fee, validate, strict_nonce_check };
    BlockifierTransaction::from_api(Transaction::Invoke(invoke_tx), tx_hash, None, None, None, execution_flags)
``` [6](#0-5) 

Because `only_query = false`, the blockifier reports version `0x3` to the contract's `get_execution_info` syscall instead of `0x100000000000000000000000000000003`.

---

### Impact Explanation

Any account contract that inspects the transaction version during `__validate__` or `__execute__` to distinguish simulation from real execution receives the wrong value. Concretely:

- **Wrong fee estimate**: If the account skips expensive validation logic when it detects the query bit, running with `only_query = false` forces that logic to execute, inflating the estimated gas. Conversely, if the account has simulation-only paths, they are never taken, deflating the estimate.
- **Wrong simulation trace**: The execution trace returned by `starknet_simulateTransactions` reflects a different code path than what would actually run on-chain for a query transaction.
- **Wrong transaction hash in simulation context**: The hash embedded in the simulation uses version `0x3`; any contract that hashes the transaction info internally (e.g., for replay protection checks during simulation) operates on a different preimage than the user intended.

This matches the allowed impact: **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

---

### Likelihood Explanation

The `TransactionVersion3::Version3OnlyQuery` variant is a first-class, documented RPC value (`0x100000000000000000000000000000003`). Any client that follows the Starknet spec and submits a query-version transaction to `starknet_estimateFee` or `starknet_simulateTransactions` triggers this path. The TODO comment confirms the bug is known and unresolved. No special privilege is required; any unprivileged RPC caller can reach it.

---

### Recommendation

Extract the `only_query` flag from the incoming transaction version before discarding the RPC-layer type. For example:

```rust
impl TryFrom<BroadcastedTransaction> for ExecutableTransactionInput {
    type Error = ErrorObjectOwned;
    fn try_from(value: BroadcastedTransaction) -> Result<Self, Self::Error> {
        match value {
            BroadcastedTransaction::DeployAccount(tx) => {
                let only_query = tx.is_query_version(); // extract before conversion
                Ok(Self::DeployAccount(tx.into(), only_query))
            }
            BroadcastedTransaction::Invoke(tx) => {
                let only_query = tx.is_query_version();
                Ok(Self::Invoke(tx.into(), only_query))
            }
            BroadcastedTransaction::Declare(tx) => Ok(tx.try_into()?),
        }
    }
}
```

The `is_query_version()` helper should check whether the version field is one of the `*OnlyQuery` enum variants before the base version is extracted by the `Into` conversion.

### Proof of Concept

**Step 1**: Submit `starknet_estimateFee` with a V3 invoke transaction carrying version `0x100000000000000000000000000000003`:

```json
{
  "method": "starknet_estimateFee",
  "params": {
    "request": [{
      "type": "INVOKE",
      "version": "0x100000000000000000000000000000003",
      "sender_address": "0xACCOUNT",
      ...
    }],
    "simulation_flags": [],
    "block_id": "latest"
  }
}
```

**Step 2**: The RPC layer deserializes the version as `TransactionVersion3::Version3OnlyQuery` — accepted without error.

**Step 3**: `TryFrom<BroadcastedTransaction> for ExecutableTransactionInput` fires at line 336 with `only_query = false` hardcoded. [7](#0-6) 

**Step 4**: `calc_tx_hash` computes the hash with `TransactionOptions { only_query: false }`, so `signed_tx_version` returns `0x3` instead of `0x100000000000000000000000000000003`. [4](#0-3) 

**Step 5**: The blockifier executes the transaction with `ExecutionFlags { only_query: false, ... }`. The account contract's `get_execution_info` syscall returns version `0x3`.

**Step 6**: If the account contract branches on the query bit (e.g., skips signature verification for simulation), it takes the wrong branch. The returned fee estimate and execution trace are wrong.

**Divergent value**: version field in `get_execution_info` = `0x3` (actual) vs `0x100000000000000000000000000000003` (expected); transaction hash preimage uses `0x3` instead of `0x100000000000000000000000000000003`.

pub fn signed_tx_version(
    tx_version: &TransactionVersion,
    transaction_options: &TransactionOptions,
) -> TransactionVersion {
    // If only_query is true, set the 128-th bit.
    let query_only_bit = *QUERY_VERSION_BASE;
    assert_eq!(
        tx_version.0.to_biguint() & query_only_bit.to_biguint(),
        BigUint::from(0_u8),
        "Requested signed tx version with version that already has query bit set: {tx_version:?}."
    );
    if transaction_options.only_query {
        TransactionVersion(tx_version.0 + query_only_bit)
    } else {
        *tx_version
    }
}
