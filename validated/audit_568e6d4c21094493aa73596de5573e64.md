### Title
Unbounded Second-Allocator Materialization in `serialize_2026` via `intern_tree` with `u32::MAX` Heap Limit — (File: `src/serde_2026/ser.rs`, `src/serde/intern.rs`)

---

### Summary

`serialize_2026` calls `intern_tree`, which creates a **second full `Allocator`** with `heap_limit = u32::MAX` (≈ 4 GiB) and three unbounded `HashMap`s, fully materializing the entire CLVM tree in memory before any size check is applied. The only post-materialization guard checks atom/pair *count*, not total heap bytes. An attacker who can cause a large CLVM tree to be serialized to the 2026 format can trigger OOM before any limit fires.

---

### Finding Description

**Root cause — `SerializerState::new` calls `intern_tree` with no pre-check:**

`src/serde_2026/ser.rs` line 94 calls `intern_tree(allocator, node)` unconditionally:

```rust
pub fn new(allocator: &Allocator, node: NodePtr) -> Result<Self> {
    let tree = intern_tree(allocator, node)?;          // ← full materialization here

    if tree.atoms.len() > MAX_INDEX || tree.pairs.len() > MAX_INDEX {  // ← post-check
        return Err(EvalErr::SerializationError);
    }
    ...
}
```

The size guard fires only after `intern_tree` has already allocated everything.

**`intern_tree` uses `u32::MAX` as the heap limit:**

`src/serde/intern.rs` line 139–141:

```rust
pub fn intern_tree(source: &Allocator, node: NodePtr) -> Result<InternedTree> {
    intern_tree_limited(source, node, u32::MAX as usize)   // ← effectively unlimited
}
```

Inside `intern_tree_limited`, three unbounded `HashMap`s are allocated before any atom is copied:

```rust
let mut node_to_interned: HashMap<NodePtr, NodePtr> = HashMap::new();
let mut atom_to_interned: HashMap<Atom, NodePtr> = HashMap::new();
let mut pair_to_interned: HashMap<(NodePtr, NodePtr), NodePtr> = HashMap::new();
```

These maps grow proportionally to the number of unique nodes in the source tree and are not bounded by any limit.

**Attacker-controlled entry path — `deser_legacy` / `deser_backrefs` have no size limits:**

`wheel/src/api.rs` lines 104–114:

```rust
fn deser_legacy(blob: &[u8]) -> PyResult<LazyNode> {
    let mut a = Allocator::new();                          // ← no heap limit
    let node = node_from_bytes(&mut a, blob).map_err(eval_to_py)?;
    Ok(LazyNode::new(Rc::new(a), node))
}

fn deser_backrefs(blob: &[u8]) -> PyResult<LazyNode> {
    let mut a = Allocator::new();                          // ← no heap limit
    let node = node_from_bytes_backrefs(&mut a, blob).map_err(eval_to_py)?;
    Ok(LazyNode::new(Rc::new(a), node))
}
```

Neither function accepts a `max_atom_len` or total-size cap. The only protection is the allocator's atom/pair count limits (`MAX_NUM_ATOMS`, `MAX_NUM_PAIRS`), not a byte budget.

**Multiplicative memory footprint:**

When `ser_2026(node)` is called on a large `LazyNode`:

1. Source allocator holds the deserialized tree (up to the allocator's atom/pair count limits × max atom size).
2. `intern_tree` creates a **second allocator** (heap_limit = `u32::MAX`) and copies all unique atoms and pairs into it.
3. Three `HashMap`s grow proportionally to the unique node count.
4. Total peak memory ≈ 2× source tree + HashMap overhead — all before the post-materialization count check fires.

The `intern_tree_limited` variant exists precisely to enforce a heap cap, but `serialize_2026` never uses it.

**The `Program.to_bytes_2026()` path doubles the problem:**

`wheel/python/clvm_rs/program.py` lines 78–81:

```python
def to_bytes_2026(self) -> bytes:
    lazy = clvm_tree_to_lazy_node(self)   # first intern pass (Allocator::new(), unlimited)
    return ser_2026(lazy)                 # second intern pass (intern_tree, u32::MAX)
```

Two full interning passes run sequentially, each creating a new allocator and three HashMaps, before any size check.

---

### Impact Explanation

An attacker who can supply a large CLVM blob to `deser_legacy` or `deser_backrefs` and then trigger `ser_2026` (or `Program.to_bytes_2026()`) on the result will cause the process to allocate at least 2× the source tree's byte footprint plus HashMap overhead before any limit is enforced. On a system where the source tree approaches the allocator's atom/pair count ceiling, this can exhaust available RAM and trigger OOM termination, denying service to all concurrent users.

---

### Likelihood Explanation

The Python API (`deser_legacy`, `deser_backrefs`, `ser_2026`) is the primary public surface. Any caller that deserializes an attacker-supplied blob and then re-serializes it to 2026 format (e.g., a node that accepts CLVM programs and converts them for storage or transmission) is vulnerable. The `Program.to_bytes_2026()` convenience method makes this a one-call trigger. No authentication or special privilege is required.

---

### Recommendation

1. **Replace `intern_tree` with `intern_tree_limited`** in `SerializerState::new` (`src/serde_2026/ser.rs` line 94), passing a caller-supplied or configurable heap cap.
2. **Add a pre-check** on the source allocator's current heap usage before calling `intern_tree`, rejecting inputs that would cause the interned copy to exceed a budget.
3. **Apply `max_atom_len` and total-size limits** to `deser_legacy` and `deser_backrefs` in `wheel/src/api.rs`, analogous to the limits already present on `deser_2026`.

---

### Proof of Concept

```python
from clvm_rs import deser_legacy, ser_2026

# Build a CLVM blob with many large unique atoms.
# Legacy format allows atoms up to ~17 GiB; even moderate sizes suffice.
# Each atom: 2-byte length prefix (0x82 = 2-byte atom) + 0x3FFF bytes of data.
atom = b"\x82" + b"\xAA" * 0x3FFF          # ~16 KiB atom
# Wrap N such atoms in a list (cons chain) to fill the allocator.
# N × 16 KiB atoms → source tree ≈ N × 16 KiB
# intern_tree creates a second allocator ≈ N × 16 KiB + HashMap overhead
# Total peak ≈ 2 × N × 16 KiB before any size check fires.

# Construct a cons chain of 50,000 distinct atoms (~800 MiB source tree):
blob = b""
for i in range(50_000):
    blob = b"\xff" + bytes([0x82]) + i.to_bytes(2, "big") + b"\x00" * (0x3FFD) + blob
blob += b"\x80"  # nil terminator

node = deser_legacy(blob)          # source allocator: ~800 MiB, no size limit
ser_2026(node)                     # intern_tree: second allocator ~800 MiB + HashMaps → OOM
```

The tiny crafted blob drives `intern_tree` to allocate a second full copy of the tree (heap_limit = `u32::MAX`) plus three unbounded `HashMap`s, exhausting memory before the post-materialization count check at `src/serde_2026/ser.rs` lines 96–98 can fire.
