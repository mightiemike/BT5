### Title
Contract with >64 parameters deployed at protocol v83 becomes permanently unexecutable after upgrade to v84 — (`runtime/near-vm-runner/src/prepare/prepare_v2.rs`)

### Summary
`prepare_v2` (active at protocol ≤83) has no `max_params_per_function` enforcement. `prepare_v3` (activated at protocol 84 when `reftypes_bulk_memory` and `vm_kind` both flip) enforces `max_params_per_function = 64`. A contract with 65+ parameters in any function, deployed at v83, is accepted with `Ok`, but after the upgrade to v84 every call to that contract fails with `Err(PrepareError::TooManyParamsPerFunction)`, making it permanently unexecutable.

### Finding Description
At protocol version 83, `reftypes_bulk_memory = false` and `vm_kind = NearVm`, so `prepare.rs` routes to `prepare_v2::prepare_contract`. The `PrepareContext` in `prepare_v2.rs` initialises limits for functions, locals, tables, table elements, and types — but has no `param_limit` field and performs no per-function parameter count check. `max_params_per_function` is `None` in the v83 config (it is introduced as a new field in `84.yaml` with no `old` value).

At protocol version 84, `reftypes_bulk_memory = true` and `vm_kind = Wasmtime`, so `prepare.rs` routes to `prepare_v3::prepare_contract`. `prepare_v3` passes `config.limit_config.max_params_per_function.unwrap_or(u64::MAX)` — now `Some(64)` — to `InstrumentContext::new()`. `InstrumentContext` iterates every function's type signature and returns `Error::TooManyParamsPerFunction` if any function exceeds 64 parameters, which is mapped to `PrepareError::TooManyParamsPerFunction`.

Because NEAR re-prepares contracts from the stored raw WASM bytes on every call (using the current epoch's `RuntimeConfig`), a contract stored at v83 is re-prepared under v84 rules on the first call after the upgrade. There is no migration that grandfathers pre-existing contracts.

The divergent value is concrete:
- v83: `prepare_contract(wasm_65_params) → Ok(instrumented_bytes)`
- v84: `prepare_contract(wasm_65_params) → Err(PrepareError::TooManyParamsPerFunction)`

### Impact Explanation
Any contract deployed before the v83→v84 epoch boundary that contains a function with more than 64 parameters becomes permanently unexecutable after the upgrade. All function calls to that contract will abort with a `CompilationError::PrepareError(TooManyParamsPerFunction)`. The contract's state is frozen and inaccessible. This affects any user who deployed such a contract in good faith (the limit did not exist at deploy time).

### Likelihood Explanation
The 64-parameter limit is low enough that real contracts could exceed it — for example, auto-generated ABI wrappers or contracts compiled from languages that pass many arguments by value. The window is the entire lifetime of protocol version 83 on mainnet/testnet. Any contract deployed during that window with >64 params is silently broken post-upgrade.

### Recommendation
Add a `max_params_per_function` check to `prepare_v2::PrepareContext` (reading `config.limit_config.max_params_per_function`) so that the same limit is enforced at deploy time under v83. Alternatively, introduce a migration pass that re-validates all stored contracts against the new limits before the epoch boundary activates v84 rules, and surfaces the error at deploy time rather than silently accepting the contract.

```rust
pub(crate) fn prepare_contract(
    original_code: &[u8],
    features: crate::features::WasmFeatures,
    config: &Config,
    kind: VMKind,
) -> Result<Vec<u8>, PrepareError> {
    let lightly_steamed = PrepareContext::new(original_code, features, config).run()?;

    let res = finite_wasm::Analysis::new()
        .with_stack(Box::new(SimpleMaxStackCfg))
        .with_gas(Box::new(SimpleGasCostCfg(u64::from(config.regular_op_cost))))
        .analyze(&lightly_steamed)
        .map_err(|err| {
            tracing::error!(?err, ?kind, "analysis failed");
            PrepareError::Deserialization
        })?
        // Make sure contracts can’t call the instrumentation functions via `env`.
        .instrument("internal", &lightly_steamed)
        .map_err(|err| {
            tracing::error!(?err, ?kind, "instrumentation failed");
            PrepareError::Serialization
        })?;
    if let Some(max_size) = config.limit_config.max_instrumented_code_size {
        if res.len() as u64 > max_size {
            tracing::debug!(target: "vm", size=res.len(), ?kind, "instrumented code too large");
            return Err(PrepareError::InstrumentedCodeTooLarge);
        }
    }
    Ok(res)
}
```
