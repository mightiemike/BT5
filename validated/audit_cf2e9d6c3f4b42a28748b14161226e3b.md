### Title
Permanent Loss of Accrued Bonus Rewards via Redelegation Slash When Rewards Pool Is Insufficient — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

### Summary
When `BeforeRedelegationSlashed` fires and the rewards pool balance is below the owed bonus, `processEventsAndClaimBonus` advances `LastBonusAccrual` to `blockTime` before checking pool sufficiency. The caller (`slashRedelegationPosition`) deliberately swallows `ErrInsufficientBonusPool` and persists the position with the advanced checkpoint. The legitimately accrued bonus is permanently unrecoverable.

### Finding Description

In `processEventsAndClaimBonus`, `applyBonusAccrualCheckpoint` is called at line 215 — unconditionally, before the pool balance check at line 230:

```go
// claim_rewards.go
applyBonusAccrualCheckpoint(&pos.Position, blockTime)   // line 215 — checkpoint advanced
pos.UpdateLastKnownBonded(bonded)                        // line 217

...

if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {  // line 230
    return nil, err   // returns ErrInsufficientBonusPool, but pos is already mutated
}
``` [1](#0-0) 

Because `pos` is passed as `*types.PositionState`, the caller sees the mutated `LastBonusAccrual`. In `slashRedelegationPosition`, the error is caught and swallowed:

```go
// slash.go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
    } else {
        return err
    }
}
// pos.LastBonusAccrual is now blockTime, no bonus was paid
...
return k.setPositionWithState(ctx, pos, ...)  // persists the advanced checkpoint
``` [2](#0-1) 

`applyBonusAccrualCheckpoint` itself unconditionally calls `pos.UpdateLastBonusAccrual(accrualEnd)`: [3](#0-2) 

And `UpdateLastBonusAccrual` writes directly to the struct field: [4](#0-3) 

### Impact Explanation

A position owner who has legitimately accrued bonus rewards over a bonded period permanently loses those rewards if:
1. The rewards pool balance is below the owed bonus at the moment `BeforeRedelegationSlashed` fires, AND
2. A redelegation slash occurs on the validator the position redelegated to.

Neither condition requires any action by the position owner. The slash is a third-party event (validator double-sign or downtime). The pool being low is a realistic operational state. The loss is permanent because `LastBonusAccrual` is now at `blockTime` — no future claim path can recover the skipped interval.

### Likelihood Explanation

- Redelegation slashes are a normal Cosmos SDK event triggered by validator misbehavior.
- The rewards pool can be depleted by normal bonus claims from other positions.
- No attacker action is required; the conditions can arise organically.
- The hook `BeforeRedelegationSlashed` is a standard production path registered via `Hooks()`. [5](#0-4) 

### Recommendation

Move `applyBonusAccrualCheckpoint` (and `UpdateLastKnownBonded`) to **after** the successful pool transfer, so the checkpoint is only advanced when the bonus is actually paid. Alternatively, if the pool is insufficient, either defer the checkpoint (do not advance `LastBonusAccrual`) so the owner can claim later when the pool is refilled, or revert the entire slash-hook path with an error rather than silently forgoing rewards.

### Proof of Concept

1. Create a position with an active redelegation to validator V2 (from V1).
2. Let the position accrue bonus rewards over several blocks.
3. Drain the rewards pool to zero (via legitimate bonus claims from other positions).
4. Trigger a redelegation slash on V2 (e.g., via `SlashRedelegation` in the staking keeper, which calls `BeforeRedelegationSlashed`).
5. Assert: `pos.LastBonusAccrual == blockTime` (checkpoint advanced).
6. Assert: owner's balance did not increase (no bonus paid).
7. Refill the pool and attempt to claim — the interval `[old LastBonusAccrual, slashBlockTime]` is gone; no bonus is recoverable for that period. [6](#0-5) [7](#0-6)
