### Title
`BurnNlp` Excluded from Slow-Mode Escape Hatch, Locking NLP Holders Out During Sequencer Unavailability — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

`BurnNlp` (the NLP liquidity-exit operation) is not handled in `processSlowModeTransactionImpl`. When the sequencer is offline or censoring, NLP holders have no on-chain path to redeem their tokens. The slow-mode queue — the protocol's own censorship-resistance mechanism — silently drops any `BurnNlp` submission, locking user funds indefinitely.

---

### Finding Description

Nado's slow-mode system is the canonical escape hatch that lets users bypass the sequencer after a 3-day delay. `submitSlowModeTransactionImpl` accepts any transaction type that is not explicitly blocked: `BurnNlp` is not in the `DepositCollateral` revert branch, not in the owner-only branch, so it falls into the `else` branch, charges the slow-mode fee, and is enqueued. [1](#0-0) 

When the queued transaction is later executed via `_executeSlowModeTransaction`, it calls `processSlowModeTransactionImpl`. That function handles `DepositCollateral`, `WithdrawCollateral`, `LinkSigner`, and a set of admin operations — but **`BurnNlp` is absent**. Execution falls through to the terminal `else { revert(); }` branch. [2](#0-1) 

Because `_executeSlowModeTransaction` wraps the call in a `try/catch` on non-hardhat chains, the revert is swallowed silently. [3](#0-2) 

The only code path that successfully executes `BurnNlp` is `processTransactionImpl`, which is exclusively reachable by the sequencer via `submitTransactionsChecked`. [4](#0-3) 

This means the sequencer is the sole gatekeeper for NLP redemptions. If the sequencer goes offline, is congested, or selectively withholds `BurnNlp` transactions during adverse market conditions, NLP holders have no on-chain recourse.

---

### Impact Explanation

NLP holders cannot redeem their tokens via the slow-mode escape hatch. During a sequencer outage or deliberate censorship — precisely the scenario where users most need to exit — their capital is locked. The `burnNlp` function in `Clearinghouse.sol` computes redemption value using an oracle price supplied by the sequencer; if the sequencer is offline, that price is also unavailable, compounding the lock. [5](#0-4) 

The corrupted invariant is: *the slow-mode queue guarantees any user-initiated transaction can be executed on-chain after the timeout*. `BurnNlp` violates this invariant — it is accepted into the queue but can never be executed through it.

---

### Likelihood Explanation

The sequencer is a single off-chain component. Any unplanned downtime, upgrade, or deliberate censorship during a market stress event (the exact scenario where NLP holders would want to exit) triggers this lock. No privileged attacker is required; ordinary sequencer unavailability is sufficient. The 3-day slow-mode delay means users cannot even detect the failure until after the timeout expires and the transaction silently drops.

---

### Recommendation

Add a `BurnNlp` handler inside `processSlowModeTransactionImpl` that mirrors the sequencer path but sources the oracle price from the on-chain `priceX18[NLP_PRODUCT_ID]` mapping (already maintained by the sequencer on each submission) rather than from a sequencer-supplied parameter. This allows NLP holders to exit at the last known on-chain price when the sequencer is unavailable, consistent with how `WithdrawCollateral` is handled in slow mode. [6](#0-5) 

---

### Proof of Concept

1. User holds NLP tokens and wants to exit during a sequencer outage.
2. User calls `submitSlowModeTransaction(abi.encodePacked(uint8(TransactionType.BurnNlp), abi.encode(burnNlpTx)))`.
3. `submitSlowModeTransactionImpl` does not match any explicit branch for `BurnNlp`; it charges the slow-mode fee and enqueues the transaction. [7](#0-6) 
4. After 3 days, user or anyone calls `executeSlowModeTransaction()`.
5. `_executeSlowModeTransaction` calls `this.processSlowModeTransaction(txn.sender, txn.tx)` inside a `try/catch`. [8](#0-7) 
6. `processSlowModeTransactionImpl` reaches `else { revert(); }` for `BurnNlp`. [9](#0-8) 
7. The revert is caught silently. The slow-mode fee is consumed, the transaction is dequeued, and the NLP burn never executes. The user's NLP tokens remain locked with no further recourse.

            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }

    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
