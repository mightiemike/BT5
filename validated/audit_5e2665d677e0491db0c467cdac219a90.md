### Title
Protobuf Deserialization of `ValidResourceBounds` Silently Mutates Variant, Producing a Different Transaction Hash Preimage - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary
The protobuf-to-Rust conversion for `ValidResourceBounds` uses a zero-value heuristic to decide between the `L1Gas` and `AllResources` variants. A post-0.13.3 transaction submitted with `AllResources` where both `l2_gas` and `l1_data_gas` are zero is silently re-classified as `L1Gas` after a protobuf round-trip. Because `get_tip_resource_bounds_hash` hashes a different number of resource elements depending on the variant (2 for `L1Gas`, 3 for `AllResources`), the transaction hash preimage changes, producing a divergent hash for the same transaction.

### Finding Description

**Conversion boundary (root cause):**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` applies a zero-check heuristic:

```rust
let l1_data_gas = value.l1_data_gas.unwrap_or_default();   // silently defaults to zero
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← variant changed
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

A post-0.13.3 transaction whose `AllResources` bounds happen to carry `l2_gas = 0` and `l1_data_gas = 0` is serialized with all three fields present (all zero). On deserialization the condition `l1_data_gas.is_zero() && l2_gas.is_zero()` is true, so the variant is silently changed to `L1Gas`.

**Hash preimage divergence (exact wrong value):**

`get_tip_resource_bounds_hash` branches on the variant to decide how many resource elements to include in the Poseidon hash:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 elements total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 elements total
    }
});
``` [2](#0-1) 

`L1Gas` → hash over `[tip, L1_GAS_packed, L2_GAS_packed]`  
`AllResources` (zero l1_data_gas) → hash over `[tip, L1_GAS_packed, L2_GAS_packed, L1_DATA_GAS_packed_zero]`

These are distinct Poseidon outputs. The transaction hash computed from the deserialized object therefore differs from the hash computed at submission time.

**Acknowledgement in test code:**

The consensus test helper explicitly works around this behavior with the comment:

```rust
// If all the fields of `AllResources` are 0 upon serialization,
// then the deserialized value will be interpreted as the `L1Gas` variant.
fn add_gas_values_to_transaction(transactions: &mut [ConsensusTransaction]) {
    ...
    resource_bounds.l2_gas.max_amount = GasAmount(1);
    ...
}
``` [3](#0-2) 

This confirms the variant mutation is a known, unguarded behavior at the serialization boundary.

**Contrast with the safe RPC path:**

The `TryFrom<protobuf::ResourceBounds> for AllResourceBounds` used in the RPC/consensus path always returns `AllResourceBounds` and requires all three fields:

```rust
impl TryFrom<protobuf::ResourceBounds> for AllResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        Ok(Self {
            l1_gas: value.l1_gas.ok_or(missing("ResourceBounds::l1_gas"))?.try_into()?,
            l2_gas: value.l2_gas.ok_or(missing("ResourceBounds::l2_gas"))?.try_into()?,
            l1_data_gas: value.l1_data_gas.ok_or(missing("ResourceBounds::l1_data_gas"))?.try_into()?,
        })
    }
}
``` [4](#0-3) 

The P2P sync path for historical `Transaction` objects uses the unsafe `ValidResourceBounds` converter, not this safe one.

### Impact Explanation

Any component that deserializes a `Transaction` from protobuf (P2P sync, block replay, re-execution for proving) and then recomputes the transaction hash will obtain a hash that differs from the canonical hash stored in the block. This binds the wrong hash to the executable payload, matching the impact category:

> **High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

Concretely:
- A syncing node that validates `validate_transaction_hash` after P2P deserialization will reject a valid, already-committed transaction.
- A prover or re-executor that recomputes the hash from the deserialized object will produce a mismatched hash, causing proof failure or wrong receipt.
- Fee enforcement mode also changes (`L1Gas` checks only L1 gas; `AllResources` checks all three resources), so L2 gas and L1 data gas bounds are silently skipped for the re-executed transaction. [5](#0-4) 

### Likelihood Explanation

The trigger is an unprivileged user submitting a V3 transaction with `AllResourceBounds` where `l2_gas = 0` and `l1_data_gas = 0`. This is a structurally valid transaction accepted by the gateway (the gateway validates resource bounds but does not require them to be non-zero). The condition is reachable without any special privilege. The only requirement is that both optional gas fields are left at their default zero values, which is a common pattern for transactions that only consume L1 gas.

### Recommendation

Replace the zero-value heuristic with an explicit version/protocol gate. Post-0.13.3 transactions always carry `AllResources`; the variant should be determined by the Starknet protocol version of the block being synced, not by inspecting field values. At minimum, the `unwrap_or_default()` fallback for `l1_data_gas` should be removed and replaced with an explicit error once 0.13.2 support is dropped (as the existing TODO comment already notes). Until then, the deserialization should preserve `AllResources` whenever all three fields are present in the protobuf message, regardless of their values:

```rust
// If l1_data_gas field is present (even if zero), preserve AllResources.
Ok(match value.l1_data_gas {
    Some(l1_data_gas_proto) => {
        let l1_data_gas: ResourceBounds = l1_data_gas_proto.try_into()?;
        ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
    }
    None => ValidResourceBounds::L1Gas(l1_gas),  // only for legacy 0.13.2 messages
})
```

### Proof of Concept

1. Submit an Invoke V3 transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` to the gateway. The gateway computes hash H₁ using `AllResources` (3-element Poseidon input including the zero L1_DATA_GAS element).

2. The transaction is included in a block. A peer node receives the block via P2P and deserializes the transaction through `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Because `l2_gas.is_zero() && l1_data_gas.is_zero()`, the variant becomes `L1Gas`.

3. The peer node recomputes the hash using `get_tip_resource_bounds_hash` with the `L1Gas` variant (2-element Poseidon input, no L1_DATA_GAS element). The result is H₂ ≠ H₁.

4. Any call to `validate_transaction_hash` on the peer node returns `false` for this transaction, causing the block to be rejected or the transaction to be flagged as invalid — despite it being a legitimately committed transaction. [6](#0-5) [7](#0-6)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
