Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of End User, Enabling Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `sender`, which resolves to `msg.sender` of `pool.swap()` — the direct pool caller (e.g., `MetricOmmSimpleRouter`), not the end user. When a pool admin allowlists the router to enable router-based swaps for their curated pool, every unprivileged user can bypass the per-user allowlist by routing through the router. This is structurally inconsistent with `DepositAllowlistExtension`, which correctly ignores `sender` and checks `owner` (the actual position owner).

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the first argument to `_beforeSwap()`: [1](#0-0) 

`_beforeSwap` forwards it verbatim as `sender` to the extension. `SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`: [2](#0-1) 

When `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`) calls `pool.swap(params.recipient, ...)`, `msg.sender` from the pool's perspective is the **router address**, not the end user. So `sender` = router, and the allowlist check becomes `allowedSwapper[pool][router]`.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` discards `sender` entirely (first parameter unnamed) and checks `owner` — the actual position owner passed explicitly: [3](#0-2) 

`MetricOmmPool.addLiquidity` passes the explicit `owner` parameter (the real user) as the second argument to `_beforeAddLiquidity`: [4](#0-3) 

The deposit allowlist correctly gates by user identity regardless of intermediary. The swap allowlist does not.

**Exploit path:**
1. Pool admin deploys a curated pool with `SwapAllowlistExtension` as `beforeSwap` extension.
2. Pool admin allowlists specific users: `setAllowedToSwap(pool, userA, true)`.
3. Pool admin allowlists the router so allowlisted users can use standard periphery: `setAllowedToSwap(pool, router, true)`.
4. Non-allowlisted `userB` calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the curated pool.
5. Router calls `pool.swap(userB /*recipient*/, ...)` — `msg.sender` from pool's view = router.
6. `_beforeSwap(router /*sender*/, userB /*recipient*/, ...)` is called; extension checks `allowedSwapper[pool][router]` = `true`.
7. Swap succeeds. `userB` has bypassed the per-user allowlist entirely.

The `recipient` parameter — which is the actual end user in all router flows — is silently discarded in `beforeSwap`'s signature: [5](#0-4) 

## Impact Explanation

A curated pool's swap allowlist fails open for any unprivileged user who routes through `MetricOmmSimpleRouter` once the router is allowlisted. The pool admin cannot simultaneously (a) allow allowlisted users to use the router and (b) block non-allowlisted users from using the router. Unauthorized users can execute swaps against LP-provided liquidity at oracle-quoted prices, causing direct loss of LP assets and curation failure. This is broken core pool functionality with direct loss of user principal — High severity.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the standard supported periphery swap path. A pool admin configuring `SwapAllowlistExtension` who wants allowlisted users to use the router will naturally call `setAllowedToSwap(pool, router, true)`. The structural inconsistency with `DepositAllowlistExtension` (which checks `owner`, not `sender`) makes this mistake highly likely — a pool admin observing that the deposit allowlist correctly gates by user identity will assume the swap allowlist behaves identically. There is no NatDoc warning that `sender` is the direct pool caller rather than the end user. [6](#0-5) 

## Recommendation

`SwapAllowlistExtension.beforeSwap` should check `recipient` (the second argument, currently discarded) instead of `sender`, consistent with how `MetricOmmSimpleRouter` passes the end user as `recipient` in all swap flows:

```solidity
// Current (wrong actor — checks router when called via router):
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    ...
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {

// Fixed (check recipient — the end user in all router flows):
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    ...
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
```

This mirrors the `DepositAllowlistExtension` pattern of ignoring the intermediary caller and checking the actual economic actor.

## Proof of Concept

```
Setup:
  pool = curated pool with SwapAllowlistExtension as beforeSwap extension
  admin calls setAllowedToSwap(pool, userA, true)       // allowlist userA
  admin calls setAllowedToSwap(pool, router, true)      // enable router for allowlisted users

Attack:
  userB (NOT allowlisted) calls:
    MetricOmmSimpleRouter.exactInputSingle({pool, recipient: userB, ...})

  Router internally calls (MetricOmmSimpleRouter.sol L72-80):
    pool.swap(userB /*recipient*/, zeroForOne, amount, priceLimit, "", extensionData)
    → msg.sender from pool's view = router

  Pool calls (MetricOmmPool.sol L230-240):
    _beforeSwap(router /*sender*/, userB /*recipient*/, ...)

  SwapAllowlistExtension.beforeSwap checks (SwapAllowlistExtension.sol L37):
    allowedSwapper[pool][router] == true  ✓  → no revert

  Swap executes. userB receives output tokens.

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds — allowlist bypassed
```
