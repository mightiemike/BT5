### Title
Deprecated `verify_transaction_inclusion` remains an unrestricted public entrypoint, allowing any caller to permanently bypass the coinbase Merkle proof guard introduced in v2 — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability by requiring a valid coinbase Merkle proof. However, the original `verify_transaction_inclusion` (v1) is still an unrestricted, publicly callable on-chain method. Any unprivileged NEAR caller can invoke v1 directly at any time, permanently bypassing the coinbase guard. This is the exact structural analog to the loan protocol bug: just as a borrower can always repay through the lenient fallback path in `forceLiquidateExpiredLoan()` and avoid any penalty, a proof submitter here can always route through the lenient v1 path and avoid the strict coinbase enforcement added in v2.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces the coinbase check:

```rust
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);
``` [1](#0-0) 

This guard exists solely to prevent the 64-byte transaction Merkle proof forgery attack (documented at https://www.bitmex.com/blog/64-Byte-Transactions). After passing this check, v2 delegates to v1 internally:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [2](#0-1) 

But v1 itself carries no access restriction — it is decorated only with `#[pause]` and `#[deprecated]`, neither of which prevents an unprivileged caller from invoking it directly:

```rust
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
``` [3](#0-2) 

The `#[deprecated]` attribute is a Rust compiler hint only — it has zero effect on the compiled NEAR contract ABI. The method remains a fully callable public entrypoint. The `#[pause]` attribute only blocks calls when the contract is administratively paused; in normal operation the function is open to all callers.

v1's own documentation acknowledges the vulnerability it carries:

> This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash. [4](#0-3) 

v1 performs no coinbase anchor check; it only computes a Merkle root from the caller-supplied `tx_id` and proof and compares it to the stored block Merkle root:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [5](#0-4) 

---

### Impact Explanation

An attacker who knows the Merkle tree structure of any mainchain block can:

1. Select an internal Merkle tree node whose 32-byte hash they know.
2. Construct a valid sibling proof that hashes up to the block's `merkle_root` using that internal node as the "leaf."
3. Call `verify_transaction_inclusion` directly with the internal node hash as `tx_id`.
4. Receive a `true` return value from the contract.

Any recipient NEAR contract that calls `verify_transaction_inclusion` and acts on a `true` result (e.g., releasing funds, minting tokens, unlocking a bridge transfer) will be deceived into treating a non-existent Bitcoin transaction as confirmed. The corrupted proof result is the return value of `verify_transaction_inclusion` — the canonical on-chain verification output that downstream contracts consume.

---

### Likelihood Explanation

High. The entry path requires no privilege, no staked role, and no special key. Any NEAR account can call `verify_transaction_inclusion` in a single transaction. The 64-byte forgery technique is publicly documented and the Merkle tree structure of Bitcoin blocks is fully public data. The only precondition is that the contract is not administratively paused, which is the normal operating state.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or gate it with a role (e.g., `Role::DAO`) so it cannot be called by unprivileged accounts. If backward compatibility with existing callers is required, apply the same coinbase Merkle proof check inside v1 before performing the leaf proof, making both paths equally strict. The deprecation annotation alone provides no on-chain enforcement.

---

### Proof of Concept

1. Identify any block hash `B` present in `mainchain_header_to_height` (readable via `get_block_hash_by_height`).
2. Obtain the full transaction list for block `B` from a Bitcoin node. Compute the Merkle tree. Select any internal node `N` at depth `d` with known left/right children.
3. Build a sibling proof of length `d` that, starting from `N` as the leaf at index `i`, reconstructs the block's `merkle_root`.
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node hash)
   - `tx_block_blockhash = B`
   - `tx_index = i`
   - `merkle_proof = <crafted sibling path>`
   - `confirmations = 1`
5. The contract returns `true`. No coinbase proof was required, no role was needed, and the 64-byte guard introduced in v2 was never invoked.
