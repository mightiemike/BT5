### Title
No Slippage Protection on `mintNlp()` and `burnNlp()`: `oraclePriceX18` Is Sequencer-Supplied and Not Covered by User Signature — (`core/contracts/Clearinghouse.sol`, `core/contracts/interfaces/IEndpoint.sol`, `core/contracts/Verifier.sol`)

---

### Summary

In `Clearinghouse.sol`, both `mintNlp()` and `burnNlp()` compute the user's output amount entirely from `oraclePriceX18`. This price is **not part of the user's EIP-712 signed payload** — it is appended by the sequencer at submission time. The user's signed transaction contains no `minNlpAmountOut` (for mint) or `minQuoteAmountOut` (for burn), so there is no on-chain mechanism for the user to enforce a minimum acceptable output. This is a direct analog to M-12: a user-initiated operation whose output quantity is determined by a price the user cannot constrain.

---

### Finding Description

The `MintNlp` struct that the user signs contains only `sender`, `quoteAmount`, and `nonce`: [1](#0-0) 

The `SignedMintNlp` wrapper that the sequencer submits on-chain adds `oraclePriceX18` and `nlpPoolRebalanceX18` — fields that are **outside** the user's signature: [2](#0-1) 

The EIP-712 digest in `Verifier.sol` confirms that `oraclePriceX18` is excluded from the signed hash: [3](#0-2) 

In `Clearinghouse.mintNlp()`, the NLP tokens credited to the user are computed as:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
``` [4](#0-3) 

The user spends `quoteAmount` (which they signed) but receives `nlpAmount` determined solely by `oraclePriceX18` (which they did not sign and cannot constrain). There is no `minNlpAmountOut` check anywhere in the function.

The same pattern applies to `burnNlp()`. The user signs only `nlpAmount`; the quote returned is:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
``` [5](#0-4) 

There is no `minQuoteAmountOut` check. The `BurnNlp` struct and its digest also exclude `oraclePriceX18`: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

**For `mintNlp`:** A user who signs a transaction to spend `quoteAmount` of quote tokens can receive arbitrarily fewer NLP tokens than expected if `oraclePriceX18` is set to a higher value than the price at signing time. The user's quote balance is debited by the full `quoteAmount` regardless.

**For `burnNlp`:** A user who signs a transaction to burn `nlpAmount` of NLP tokens can receive arbitrarily fewer quote tokens than expected if `oraclePriceX18` is set to a lower value than the price at signing time. The user's NLP balance is fully burned regardless.

In both cases, the corrupted state delta is the user's on-chain balance in `SpotEngine`: they lose more value than they consented to. The asset delta is concrete: quote tokens (for mint) or NLP tokens (for burn) are permanently transferred at an unfavorable rate with no recourse.

---

### Likelihood Explanation

The sequencer is a privileged but not necessarily malicious actor. However, the vulnerability is reachable under normal operating conditions: the oracle price can legitimately move between the moment a user signs a `MintNlp`/`BurnNlp` transaction and the moment the sequencer submits it on-chain. Because the user's signature does not commit to any price bound, the sequencer can use any `oraclePriceX18` value — stale, updated, or adversarially chosen — and the contract will accept it. No special access or key compromise is required beyond the sequencer's normal submission role.

---

### Recommendation

Add a `minNlpAmountOut` field to `MintNlp` and a `minQuoteAmountOut` field to `BurnNlp`, include them in the EIP-712 signed digest, and enforce them on-chain:

```diff
// IEndpoint.sol
 struct MintNlp {
     bytes32 sender;
     uint128 quoteAmount;
     uint64 nonce;
+    int128 minNlpAmountOut;
 }

 struct BurnNlp {
     bytes32 sender;
     uint128 nlpAmount;
     uint64 nonce;
+    int128 minQuoteAmountOut;
 }
```

```diff
// Clearinghouse.sol — mintNlp()
 int128 nlpAmount = quoteAmount.div(oraclePriceX18);
+require(nlpAmount >= txn.minNlpAmountOut, "slippage: nlp amount too low");
```

```diff
// Clearinghouse.sol — burnNlp()
 int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
 int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
 quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
+require(quoteAmount >= txn.minQuoteAmountOut, "slippage: quote amount too low");
```

Update the `Verifier.sol` digest computation for both transaction types to include the new fields.

---

### Proof of Concept

1. User signs a `MintNlp` transaction: `{ sender: alice, quoteAmount: 1000e18, nonce: 1 }` when NLP oracle price is `1.00`.
2. Expected NLP received: `1000e18 / 1.00 = 1000 NLP`.
3. Before sequencer submits, oracle price moves to `2.00` (or sequencer uses a stale/different price).
4. Sequencer submits `SignedMintNlp` with `oraclePriceX18 = 2e18`.
5. On-chain: `nlpAmount = 1000e18 / 2e18 = 500 NLP`.
6. Alice's quote balance is debited by `1000e18` but she receives only `500 NLP` — half the expected amount — with no on-chain protection to revert the transaction.

The same scenario applies symmetrically to `burnNlp` with a downward price move.
