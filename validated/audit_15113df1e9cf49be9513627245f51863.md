### Title
Duplicate Position IDs in `ClaimTierRewards` Enable Double-Claiming of Bonus Rewards — (`x/tieredrewards/keeper/msg_server.go`)

---

### Summary

`MsgClaimTierRewards` accepts a caller-supplied list of position IDs. All positions are loaded from chain state into a local slice **before** any of them are processed. If the same position ID appears more than once in the list, the position is loaded with identical stale state for each occurrence. When the second occurrence is processed, the bonus-reward calculation re-uses the original `LastEventSeq` checkpoint, re-pays the same bonus from the `RewardsPoolName` module account, and re-decrements validator-event reference counts — breaking internal accounting.

---

### Finding Description

In `ClaimTierRewards` (`msg_server.go` lines 429–468), positions are collected in a first pass:

```go
for _, posId := range msg.PositionIds {
    pos, err := ms.getPositionState(ctx, posId)   // reads from chain store
    ...
    positions = append(positions, pos)             // appended as value copy
}
```

Then processed in a second pass via `claimRewardsAndUpdatesPositions` (`claim_rewards.go` lines 106–135):

```go
for i := range positions {
    pos := &positions[i]
    ...
    bonus, err := k.processEventsAndClaimBonus(ctx, pos)   // pays out bonus
    ...
    if err := k.setPosition(ctx, pos.Position, nil); err != nil { ... }  // persists updated checkpoint
}
```

`processEventsAndClaimBonus` (`claim_rewards.go` lines 142–251) computes bonus by fetching validator events since `pos.LastEventSeq`, pays them out via `bankKeeper.SendCoinsFromModuleToAccount`, and decrements each event's reference count. It then advances `pos.LastEventSeq` in the **local** struct and calls `setPosition` to persist.

When the same position ID appears twice in `msg.PositionIds`:

1. Both slice entries hold the **same pre-processing** `LastEventSeq`.
2. The first iteration pays bonus, decrements ref counts, and persists the advanced checkpoint.
3. The second iteration still holds the **old** `LastEventSeq`. It calls `getValidatorEventsSince(ctx, valAddr, oldSeq)`, which reads from chain state. If any of those events still have ref count > 0 (because other positions delegated to the same validator also reference them), the events are returned again.
4. Bonus is computed and paid a second time from `RewardsPoolName`.
5. Ref counts are decremented a second time, corrupting the event-lifecycle accounting.

The condition "ref count > 0 after first decrement" is satisfied whenever at least one other position is delegated to the same validator — a routine production scenario.

---

### Impact Explanation

An attacker who owns one or more positions can submit a single `MsgClaimTierRewards` transaction with their position ID repeated N times. Each repetition beyond the first drains an additional `bonusCoins` from the `RewardsPoolName` module account. The maximum extractable amount per transaction is bounded by the pool balance and the number of unprocessed validator events, but the attack is repeatable across blocks as new events accumulate. In the worst case the entire bonus pool is drained, leaving legitimate position holders unable to claim their earned rewards.

The corrupted `decrementEventRefCount` calls also leave validator-event reference counts lower than they should be, potentially causing premature event deletion and depriving other positions of their correct bonus calculation.

---

### Likelihood Explanation

Any position owner can craft this transaction with no special privileges. The only precondition — that at least one other position is delegated to the same validator — is trivially met on any live network where multiple users participate in the tiered-rewards module. The attack requires no leaked keys, no governance action, and no social engineering.

---

### Recommendation

1. **Deduplicate `msg.PositionIds` in `MsgClaimTierRewards.Validate()`** — reject or deduplicate the list before processing.
2. **Apply Check-Effects-Interactions order**: persist the updated position state (advance `LastEventSeq`) to the chain store **before** calling `bankKeeper.SendCoinsFromModuleToAccount`, so that any re-entry or re-processing of the same position reads the already-advanced checkpoint.
3. Alternatively, re-read each position from chain state inside `claimRewardsAndUpdatesPositions` rather than operating on a pre-loaded slice, so that the second occurrence of a duplicate ID sees the already-updated checkpoint.

---

### Proof of Concept

1. Alice owns position ID `42`, delegated to validator V. Validator V also has other positions delegated to it (ref counts > 1).
2. Alice submits `MsgClaimTierRewards{Owner: alice, PositionIds: [42, 42, 42]}`.
3. First loop iteration loads position 42 three times into `positions[0]`, `positions[1]`, `positions[2]` — all with `LastEventSeq = S`.
4. `claimRewardsAndUpdatesPositions` processes `positions[0]`: pays bonus B, decrements ref counts for events since S, persists `LastEventSeq = S'` to chain store.
5. Processes `positions[1]`: local `LastEventSeq` is still S; `getValidatorEventsSince(ctx, V, S)` returns the same events (ref count still > 0); pays bonus B again; decrements ref counts again.
6. Processes `positions[2]`: same as step 5 — pays bonus B a third time.
7. Alice receives 3×B instead of B; `RewardsPoolName` is debited 3×B; validator-event ref counts are decremented 3× instead of 1×. [1](#0-0) [2](#0-1) [3](#0-2)

func (ms msgServer) ClaimTierRewards(ctx context.Context, msg *types.MsgClaimTierRewards) (*types.MsgClaimTierRewardsResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	positions := make([]types.PositionState, 0, len(msg.PositionIds))
	for _, posId := range msg.PositionIds {
		pos, err := ms.getPositionState(ctx, posId)
		if err != nil {
			return nil, err
		}

		if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
			return nil, err
		}

		positions = append(positions, pos)
	}

	totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventTierRewardsClaimed{
		Owner:        msg.Owner,
		PositionIds:  msg.PositionIds,
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
	}); err != nil {
		return nil, err
	}

	return &types.MsgClaimTierRewardsResponse{
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
		PositionIds:  msg.PositionIds,
	}, nil
}

func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
}

func (k Keeper) processEventsAndClaimBonus(ctx context.Context, pos *types.PositionState) (sdk.Coins, error) {
	// Rewards should have been claimed before undelegation
	if !pos.IsDelegated() {
		return sdk.NewCoins(), nil
	}

	valAddr, err := sdk.ValAddressFromBech32(pos.Delegation.ValidatorAddress)
	if err != nil {
		return nil, err
	}

	events, err := k.getValidatorEventsSince(ctx, valAddr, pos.LastEventSeq)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	blockTime := sdkCtx.BlockTime()

	totalBonus := math.ZeroInt()
	// Use the persisted bonded state from the last replay, not a hardcoded default.
	// This prevents overpaying bonus for unbonded gaps between claims.
	bonded := pos.LastKnownBonded
	segmentStart := pos.LastBonusAccrual

	tier, err := k.getTier(ctx, pos.TierId)
	if err != nil {
		return nil, err
	}

	for _, entry := range events {
		evt := entry.Event

		if bonded {
			// Compute bonus for the bonded segment [segmentStart, eventTime]
			// using the snapshot rate at the event.
			bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
			totalBonus = totalBonus.Add(bonus)
		}

		// Update bonded state based on event type.
		switch evt.EventType {
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_UNBOND:
			bonded = false
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_BOND:
			bonded = true
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_SLASH:
			// Slash doesn't change bonded state.
		}

		segmentStart = evt.Timestamp
		pos.UpdateLastEventSeq(entry.Seq)

		// Decrement reference count.
		if err := k.decrementEventRefCount(ctx, valAddr, entry.Seq); err != nil {
			return nil, err
		}
	}

	val, err := k.stakingKeeper.GetValidator(ctx, valAddr)
	if err != nil {
		return nil, err
	}
	// Defensive: validator bond status check
	if bonded && val.IsBonded() {
		currentRate, err := k.getTokensPerShare(ctx, valAddr)
		if err != nil {
			return nil, err
		}
		bonus := k.computeSegmentBonus(*pos, tier, segmentStart, blockTime, currentRate)
		totalBonus = totalBonus.Add(bonus)
	}

	applyBonusAccrualCheckpoint(&pos.Position, blockTime)
	// Persist the bonded state so the next replay starts correctly.
	pos.UpdateLastKnownBonded(bonded)

	if totalBonus.IsZero() {
		return sdk.NewCoins(), nil
	}

	bondDenom, err := k.stakingKeeper.BondDenom(ctx)
	if err != nil {
		return nil, err
	}

	bonusCoins := sdk.NewCoins(sdk.NewCoin(bondDenom, totalBonus))

	if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
		return nil, err
	}

	ownerAddr, err := sdk.AccAddressFromBech32(pos.Owner)
	if err != nil {
		return nil, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid owner address")
	}

	if err := k.bankKeeper.SendCoinsFromModuleToAccount(ctx, types.RewardsPoolName, ownerAddr, bonusCoins); err != nil {
		return nil, err
	}
