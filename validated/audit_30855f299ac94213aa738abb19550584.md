### Title
SwapAllowlistExtension Gating Bypassed via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

The `SwapAllowlistExtension.beforeSwap` hook receives `sender` = `msg.sender` of `pool.swap`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. The allowlist therefore checks the router's address, not the actual trader's address. Any non-allowlisted user can bypass the swap allowlist by calling the public router instead of calling the pool directly.

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it verbatim to every configured extension:

```solidity
// MetricOmmPool.sol
_beforeSwap(
    msg.sender,   // ← router address when routed
    recipient,
    zeroForOne,
    ...
);
```

`ExtensionCalling._beforeSwap` encodes and dispatches this to `SwapAllowlistExtension.beforeSwap`:

```solidity
// ExtensionCalling.sol
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, zeroForOne, ...)
)
```

The extension's `beforeSwap` receives `sender` = router. Its allowlist lookup is keyed on `(pool, sender)`. Because `MetricOmmSimpleRouter` is a public, permissionless contract, any user can call `exactInput` / `exactOutput` on it. The pool then calls the extension with `sender = router`. If the router is allowlisted (or if the allowlist is set to `allowAll` for the router), every non-allowlisted user gains unrestricted swap access.

The symmetric failure also exists: if the router is not allowlisted, legitimately allowlisted users who route through the router are incorrectly blocked.

### Impact Explanation

A pool operator who deploys `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, whitelisted market makers, or protocol-controlled addresses) loses that protection entirely. Any address can execute swaps against the pool's LP liquidity by routing through the public router. This constitutes a broken core access-control invariant with direct fund-impacting consequences: LP capital is exposed to unrestricted trading by actors the pool operator explicitly intended to exclude.

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap entry point. Any user who discovers the allowlist restriction on direct pool calls will naturally try the router. No special privileges, flash loans, or unusual token behavior are required — a standard `exactInput` call suffices.

### Recommendation

The `SwapAllowlistExtension` must gate the economically relevant actor, not the intermediary. Two sound approaches:

1. **Check `recipient` as a proxy for the user** — the router sets `recipient` to the actual end user. Checking `recipient` instead of `sender` in `beforeSwap` gates the output beneficiary, which is closer to the intended identity.
2. **Require direct pool interaction** — document that pools using `SwapAllowlistExtension` must not configure the public router as an allowed caller, and enforce this in the extension's `initialize` or via a separate router-exclusion check.

The cleanest fix is to change the allowlist lookup key from `sender` to `recipient` in `SwapAllowlistExtension.beforeSwap`, since `MetricOmmSimpleRouter` always sets `recipient` to the caller's intended output address (the actual user).

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured as a `beforeSwap` extension.
2. Add address `A` to the allowlist; leave address `B` (attacker) off the allowlist.
3. Confirm: `pool.swap(...)` called directly by `B` reverts with `NotAllowedToSwap`.
4. Call `MetricOmmSimpleRouter.exactInput(...)` from `B` with `recipient = B`.
   - Pool receives the call with `msg.sender = router`.
   - `_beforeSwap(router, B, ...)` is dispatched.
   - `SwapAllowlistExtension` checks `allowedSwapper[pool][router]` — if router is allowlisted or `allowAll[pool] == true`, the check passes.
   - Swap executes successfully for the non-allowlisted attacker `B`.
5. LP funds are now accessible to any address via the router, defeating the allowlist entirely.
