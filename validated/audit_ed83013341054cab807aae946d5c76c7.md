Audit Report

## Title
`_validateDrawdown` off-by-one allows `drawdownE6 = E6`, zeroing `floorMultiplier` and permanently disabling the stop-loss guard — (`File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`_validateDrawdown` uses a strict `>` comparison, so `drawdownE6 = 1e6` (= `E6`) passes validation without error. This causes `floorMultiplier = E6 − drawdown = 0` at line 234, making the breach condition in `_applyWatermark` (`metric < (hwm * 0) / E6`) permanently false for any `uint256` metric. The `OracleValueStopLossExtension` stop-loss guard is silently disabled while the config appears fully active, allowing unprivileged traders to drain LP bins without triggering a revert.

## Finding Description

**Root cause — validation boundary (lines 305–307):**
```solidity
function _validateDrawdown(uint256 drawdownE6) private pure {
    if (drawdownE6 > E6) revert OracleStopLossDrawdownTooLarge(drawdownE6);
}
```
`drawdownE6 = 1e6` satisfies `1e6 > 1e6 == false`, so it is stored without error. The same off-by-one exists in `_validateDecay` (line 309–311).

**Floor computation (line 234):**
```solidity
uint256 floorMultiplier = E6 - drawdown;  // 1e6 - 1e6 = 0
```

**Breach condition (line 334):**
```solidity
breached = metric < (hwm * floorMultiplier) / E6;
// = metric < (hwm * 0) / 1e6
// = metric < 0   ← always false for uint256
```

The intended disable path is `drawdown == 0`, which triggers an explicit early return at line 217. `drawdown = 1e6` bypasses that guard, stores a non-zero value that looks active, yet produces `floorMultiplier = 0` through arithmetic — a silent misconfiguration with no on-chain signal.

The pool admin can reach this state at initialization (factory call with `drawdownE6 = 1e6`) or via the timelocked `proposeOracleStopLossDrawdown` → `executeOracleStopLossDrawdown` flow, both of which call the same broken `_validateDrawdown`. Once stored, every subsequent `afterSwap` call computes `floorMultiplier = 0` and `_applyWatermark` always returns `breached = false`, regardless of how far the per-share metric has fallen.

The parallel `_validateDecay` defect (`decayPerSecondE8 = E8`) causes `_decayed` to return 0 after one second (line 322: `if (factor >= E8) return 0`), collapsing watermarks to zero and making `metric >= hwm` trivially true, also suppressing breach detection.

## Impact Explanation
The `OracleValueStopLossExtension` is the sole on-chain hook that blocks value-draining swaps past the configured drawdown floor. With `floorMultiplier = 0`, every swap — including those that reduce per-share bin value to zero — passes `afterSwap` without reverting. LP token0 and token1 balances in affected bins can be fully extracted by a sequence of unprivileged swaps while the guard silently reports no breach. This is direct, unbounded loss of LP principal, meeting the Critical/High threshold under the allowed impact gate.

## Likelihood Explanation
The pool admin sets `drawdownE6` at factory initialization and can update it via the timelocked propose/execute flow. `E6` (100%) is the natural boundary value a developer reaches for when intending maximum permissiveness or during testing; the validation passes it without error or warning. Because `drawdownE6 = 0` is the documented disable path, an admin who sets `E6` expecting a "100% tolerance" configuration has no indication the guard is broken. The timelock on subsequent corrections means the exposure window can span days. The broken cap (`>` instead of `>=`) is a code-level defect, not a malicious-setup scenario.

## Recommendation
Change both validation helpers to use `>=` so boundary values are rejected:
```solidity
function _validateDrawdown(uint256 drawdownE6) private pure {
    if (drawdownE6 >= E6) revert OracleStopLossDrawdownTooLarge(drawdownE6);
}

function _validateDecay(uint256 decayPerSecondE8) private pure {
    if (decayPerSecondE8 >= E8) revert OracleStopLossDecayTooLarge(decayPerSecondE8);
}
```
Additionally, add a defensive check in `_afterSwapOracleStopLoss` or `_checkAndUpdateWatermarks` that reverts if `floorMultiplier == 0` and `drawdown != 0`, to catch any future path that could produce a zero floor from a non-zero drawdown.
