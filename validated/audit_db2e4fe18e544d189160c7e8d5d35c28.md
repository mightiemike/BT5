### Title
`calculate_reward` Silently Ignores the Passed `_protocol_version` Parameter, Permanently Hardcoding `protocol_reward_rate` for Mainnet Regardless of Protocol Upgrades — (`chain/epoch-manager/src/reward_calculator.rs`)

---

### Summary

`RewardCalculator::calculate_reward` accepts a `_protocol_version: ProtocolVersion` argument (the current epoch's protocol version) but never uses it. Instead, it branches on `self.genesis_protocol_version` — a value frozen at node startup from the genesis config — to decide whether to apply a hardcoded `protocol_reward_rate` of `1/10`. For every mainnet epoch (`genesis_protocol_version == PROD_GENESIS_PROTOCOL_VERSION == 29`), the treasury split is permanently locked to `1/10` and cannot be altered by any future protocol upgrade, even one that explicitly changes `protocol_reward_rate` in the versioned `EpochConfig`.

---

### Finding Description

`RewardCalculator::new` captures `genesis_protocol_version` from the genesis config at startup and stores it immutably in the struct: [1](#0-0) 

`calculate_reward` then receives `_protocol_version` (the live epoch version) but prefixes it with `_`, signalling it is intentionally unused: [2](#0-1) 

The branch that selects the rate compares `self.genesis_protocol_version` — not the passed `_protocol_version` — against `PROD_GENESIS_PROTOCOL_VERSION`: [3](#0-2) 

The caller in `finalize_epoch` correctly derives `epoch_protocol_version` from the live epoch info and passes it, but the callee discards it: [4](#0-3) 

Meanwhile, `max_inflation_rate` — the other economic parameter in the same call — **is** taken from the versioned `epoch_config` and therefore **does** respond to protocol upgrades: [5](#0-4) 

This creates an asymmetry: the total reward pool can change with protocol upgrades (via `max_inflation_rate`), but the treasury/validator split is permanently frozen at `1/10` for mainnet, regardless of what any future `EpochConfig` version specifies for `protocol_reward_rate`.

The `PROD_GENESIS_PROTOCOL_VERSION` constant is 29: [6](#0-5) 

The mainnet genesis itself sets `protocol_reward_rate = [0, 1]` (zero), yet the hardcoded path overrides this to `1/10`: [7](#0-6) 

---

### Impact Explanation

Any on-chain governance decision to change the treasury reward fraction — encoded as a new `protocol_reward_rate` in a future versioned `EpochConfig` — would be silently ignored for mainnet. The divergent value (`1/10` hardcoded vs. whatever the epoch config specifies) would persist across every epoch boundary without error or log, producing incorrect token minting and treasury credit that cannot be corrected without a code-level patch and redeployment. This is a broken protocol-upgrade-boundary invariant: the function signature advertises version-awareness but delivers none.

**Impact: High** — incorrect, non-overridable reward distribution affecting every epoch on mainnet.

---

### Likelihood Explanation

The `max_inflation_rate` was already changed via protocol upgrade (halved in a recent version, as confirmed by `test_adjust_max_inflation`): [8](#0-7) 

A future upgrade to `protocol_reward_rate` follows the same governance path and is plausible. The bug is latent but reachable the moment such an upgrade is ratified.

**Likelihood: Medium**

---

### Recommendation

Remove the `genesis_protocol_version`-based branch and instead read `protocol_reward_rate` from the versioned `EpochConfig` that is already passed into `finalize_epoch` (just as `max_inflation_rate` and `online_min_threshold` are). The `_protocol_version` parameter should either be used to gate version-specific behaviour or removed from the signature entirely to avoid the misleading contract.

---

### Proof of Concept

1. Deploy a protocol upgrade that sets `protocol_reward_rate = [1, 5]` (20%) in the new `EpochConfig`.
2. After the upgrade epoch boundary, call `finalize_epoch`; it passes the new `epoch_protocol_version` to `calculate_reward`.
3. Inside `calculate_reward`, `use_hardcoded_value` evaluates to `true` because `self.genesis_protocol_version == 29 == PROD_GENESIS_PROTOCOL_VERSION`.
4. `protocol_reward_rate` is set to `1/10` (10%), not `1/5` (20%).
5. The treasury receives 10% of epoch rewards; validators receive 90% — the upgrade's intended 20%/80% split is silently discarded every epoch. [3](#0-2)
