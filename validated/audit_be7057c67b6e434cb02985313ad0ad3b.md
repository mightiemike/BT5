### Title
Missing `chargeFee()` Call in Slow-Mode `WithdrawCollateral` Path Allows Withdrawal Fee Bypass — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The `processTransactionImpl()` (fast/sequencer path) correctly charges the product-specific `withdrawFeeX18` before executing a collateral withdrawal. The `processSlowModeTransactionImpl()` path, which handles the same `WithdrawCollateral` transaction type, omits the `chargeFee()` call entirely. Any user who routes a withdrawal through the slow-mode queue avoids paying the `withdrawFeeX18` fee, causing a direct accounting discrepancy: the user's spot balance is not debited for the fee and `sequencerFee[productId]` is never incremented.

---

### Finding Description

`EndpointTx.processTransactionImpl()` handles `WithdrawCollateral` as follows: [1](#0-0) 

It calls `chargeFee(signedTx.tx.sender, spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18, signedTx.tx.productId)` before delegating to `clearinghouse.withdrawCollateral()`.

`chargeFee()` performs two state mutations: [2](#0-1) 

It debits `fee` from the sender's spot balance and credits `sequencerFee[productId]`.

`processSlowModeTransactionImpl()` handles the identical `WithdrawCollateral` transaction type: [3](#0-2) 

There is no `chargeFee()` call. The function decodes the transaction, validates the sender, and calls `clearinghouse.withdrawCollateral()` directly — skipping the fee deduction entirely.

`WithdrawCollateralV2` is not present in the slow-mode handler at all, so only V1 is exposed via this path, but V1 is the standard withdrawal type.

---

### Impact Explanation

**Medium.** Every user who submits a `WithdrawCollateral` transaction via the slow-mode queue avoids paying `withdrawFeeX18`. The concrete state delta is:

- `spotEngine` balance for `sender` at `productId` is **not** reduced by `withdrawFeeX18`.
- `sequencerFee[productId]` is **not** incremented by `withdrawFeeX18`.

The protocol loses fee revenue proportional to `withdrawFeeX18 × withdrawal_amount` for every slow-mode withdrawal. Users who discover this can systematically route all withdrawals through slow mode to extract this advantage. The `SLOW_MODE_FEE` (a flat native-token fee charged at submission time) does not substitute for the product-specific `withdrawFeeX18`.

---

### Likelihood Explanation

**Medium.** Any user can call `submitSlowModeTransactionImpl()` with a `WithdrawCollateral` transaction by paying the flat `SLOW_MODE_FEE`. After the `SLOW_MODE_TX_DELAY` (hardcoded to three days), the sequencer processes it via `processSlowModeTransactionImpl()`, which omits the fee. The path is fully permissionless at submission time and requires no special privileges. The only friction is the three-day delay, which is acceptable for users making large withdrawals where `withdrawFeeX18` is material.

---

### Recommendation

Add the `chargeFee()` call to the `WithdrawCollateral` branch of `processSlowModeTransactionImpl()`, mirroring the fast path:

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.WithdrawCollateral memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.WithdrawCollateral)
    );
    validateSender(txn.sender, sender);
+   chargeFee(
+       txn.sender,
+       spotEngine.getConfig(txn.productId).withdrawFeeX18,
+       txn.productId
+   );
    clearinghouse.withdrawCollateral(
        txn.sender,
        txn.productId,
        txn.amount,
        address(0),
        nSubmissions
    );
}
```

---

### Proof of Concept

1. User holds a spot balance of token `T` (productId `P`) with `withdrawFeeX18 = F`.
2. User calls `submitSlowModeTransactionImpl()` with a `WithdrawCollateral{sender, productId: P, amount: A}` transaction, paying only the flat `SLOW_MODE_FEE`.
3. After three days, the sequencer calls `processSlowModeTransactionImpl(sender, transaction)`.
4. The branch at line 217–229 executes: `clearinghouse.withdrawCollateral(sender, P, A, address(0), nSubmissions)` is called with no prior `chargeFee()`.
5. `Clearinghouse.withdrawCollateral()` transfers `A` tokens to the user and debits `A * multiplier` from their spot balance — but does **not** debit `F`.
6. `sequencerFee[P]` remains unchanged.
7. Compare: if the same user had submitted via the fast path (`processTransactionImpl`), `chargeFee(sender, F, P)` would have reduced their spot balance by `F` and incremented `sequencerFee[P]` by `F`. [3](#0-2) [4](#0-3) [2](#0-1)
