### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable and Vulnerable to Merkle Internal-Node Proof Forgery — (File: `contract/src/lib.rs`)

### Summary
The deprecated `verify_transaction_inclusion` function is still a live, unpermissioned public entry point. It lacks the coinbase-proof depth-anchoring that `verify_transaction_inclusion_v2` introduced. Any NEAR caller can supply an internal merkle-tree node hash as `tx_id` with a shortened proof and receive `true` for a transaction that was never included in the block.

### Finding Description

`verify_transaction_inclusion` (lines 288–323 of `contract/src/lib.rs`) computes the merkle root from caller-supplied `(tx_id, tx_index, merkle_proof)` and compares it to the stored `header.block_header.merkle_root`: [1](#0-0) 

The underlying `compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` iterates over the proof elements and hashes left/right based on `current_position % 2`, then divides the position by 2 each step: [2](#0-1) 

Critically, neither function validates that `transaction_position` falls within the leaf layer, nor that `merkle_proof.len()` equals the full tree depth. An attacker can supply an internal node hash at depth D with a proof of length `(tree_depth − D)` and the computation will still arrive at the correct merkle root.

The only guard in the deprecated function is: [3](#0-2) 

This blocks the trivial empty-proof case (`tx_id == merkle_root`) but does not block the shortened-proof internal-node attack.

`verify_transaction_inclusion_v2` mitigates this by requiring `coinbase_merkle_proof.len() == merkle_proof.len()`, anchoring the tree depth via the always-present coinbase at index 0: [4](#0-3) 

The deprecated v1 function bypasses this mitigation entirely and remains callable by any NEAR account because it carries only `#[pause]` (pausable by a role-holder, not restricted by default) and no access-control guard: [5](#0-4) 

### Impact Explanation

Any downstream NEAR contract or off-chain system that calls `verify_transaction_inclusion` can be deceived into accepting a forged inclusion proof. The function returns `true` for a hash that corresponds to an internal merkle node — a value that is not a real Bitcoin transaction. Downstream consumers (e.g., bridge contracts releasing funds, token-minting contracts) that trust this result would act on a fabricated proof, enabling theft or unauthorized minting.

### Likelihood Explanation

The attack requires no special privileges and no computational work beyond reading a public Bitcoin block:

1. The attacker is any NEAR account — no role or stake required.
2. The merkle tree structure of every Bitcoin block is publicly derivable from the block's transaction list.
3. The attacker only needs to identify one internal node hash and its sibling path to the root.

### Recommendation

Remove `verify_transaction_inclusion` from the contract entirely, or gate it with an `#[access_control]` restriction so it cannot be called by unprivileged accounts. If backward compatibility must be preserved, replicate the coinbase-proof depth-anchoring check from `verify_transaction_inclusion_v2` inside the deprecated function before delegating to `compute_root_from_merkle_proof`.

### Proof of Concept

Consider a block whose merkle tree has four transactions `T1, T2, T3, T4`:

```
merkle_root = hash(N12 || N34)
  N12 = hash(T1 || T2)   ← internal node at depth 1, position 0
  N34 = hash(T3 || T4)   ← internal node at depth 1, position 1
```

Attacker calls `verify_transaction_inclusion` with:

```
tx_id              = N12          // internal node hash, not a real tx
tx_block_blockhash = <any mainchain block hash>
tx_index           = 0            // position of N12 in the depth-1 layer
merkle_proof       = [N34]        // one element — proof is shorter than full depth
confirmations      = 1
```

Inside `compute_root_from_merkle_proof`:
- `current_position = 0`, `0 % 2 == 0` → `current_hash = hash(N12 || N34) = merkle_root`

The equality check passes and the function returns `true`, falsely asserting that `N12` is an included transaction. [6](#0-5) [7](#0-6)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
