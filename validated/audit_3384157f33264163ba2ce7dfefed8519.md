### Title
Sanctions Check Performed Only at Slow Mode Submission, Not at Execution — (File: `core/contracts/EndpointTx.sol`)

---

### Summary
The `submitSlowModeTransactionImpl` function enforces a sanctions check (`requireUnsanctioned`) at the time a slow mode transaction is queued. However, `processSlowModeTransactionImpl` — which executes the queued transaction up to three days later — performs no sanctions re-check. If a user is added to the sanctions list after submitting a slow mode withdrawal but before it executes, the withdrawal still completes, transferring funds to a sanctioned address.

---

### Finding Description

In `EndpointTx.submitSlowModeTransactionImpl`, after determining the transaction type and charging fees, the sanctions check is applied to the submitting sender:

```solidity
requireUnsanctioned(sender);
slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
    sender: sender,
    tx: transaction
});
``` [1](#0-0) 

This check gates entry into the slow mode queue. The transaction is then stored and becomes executable after `SLOW_MODE_TX_DELAY` (three days).

When the queued transaction is later executed — either by the sequencer via `ExecuteSlowMode` or by any caller via `Endpoint.executeSlowModeTransaction()` — it routes to `processSlowModeTransactionImpl`. For the `WithdrawCollateral` case:

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.WithdrawCollateral memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.WithdrawCollateral)
    );
    validateSender(txn.sender, sender);
    clearinghouse.withdrawCollateral(
        txn.sender,
        txn.productId,
        txn.amount,
        address(0),
        nSubmissions
    );
``` [2](#0-1) 

There is no `requireUnsanctioned` call here. The only check is `validateSender`, which verifies the caller matches the stored sender address — it has no awareness of the sanctions list. [3](#0-2) 

The `requireUnsanctioned` helper is defined in `EndpointStorage` and queries the external `ISanctionsList`:

```solidity
function requireUnsanctioned(address sender) internal view virtual {
    require(!sanctions.isSanctioned(sender), ERR_WALLET_SANCTIONED);
}
``` [4](#0-3) 

The same gap exists for the `DepositCollateral` path in `depositCollateralWithReferral`. The sanctions check occurs at deposit submission time:

```solidity
requireUnsanctioned(msg.sender);
requireUnsanctioned(sender);
``` [5](#0-4) 

But when the queued `DepositCollateral` slow mode transaction executes in `processSlowModeTransactionImpl`, no sanctions check is performed before crediting the balance:

```solidity
if (txType == IEndpoint.TransactionType.DepositCollateral) {
    IEndpoint.DepositCollateral memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.DepositCollateral)
    );
    validateSender(txn.sender, sender);
    _recordSubaccount(txn.sender);
    clearinghouse.depositCollateral(txn);
``` [6](#0-5) 

The three-day delay window is the structural gap. The sanctions list is a live, mutable external contract. Any sanctioning event that occurs after submission but before execution is invisible to the execution path.

---

### Impact Explanation

For the `WithdrawCollateral` slow mode path: a user who is sanctioned after submitting a slow mode withdrawal will still have their collateral transferred out of the protocol to their address when the slow mode transaction executes. This directly transfers assets to a sanctioned address, violating the protocol's compliance invariant and potentially creating regulatory liability.

For the `DepositCollateral` path: a user sanctioned after submitting a deposit (but before execution) will have their balance credited on-chain, enabling them to trade or initiate further withdrawals as a sanctioned party.

The corrupted state is: `clearinghouse` balance credited to a sanctioned subaccount, or ERC-20 funds transferred to a sanctioned address.

---

### Likelihood Explanation

The slow mode queue enforces a mandatory three-day delay. Sanctions list updates are an operational reality for any protocol integrating OFAC-style compliance. The window is wide and predictable. An attacker aware of an impending sanctioning could deliberately front-run it by submitting a slow mode withdrawal immediately before the sanctioning event, guaranteeing execution after the check would have blocked them. This is a realistic, low-privilege, externally-triggered scenario requiring no special access.

---

### Recommendation

Re-apply `requireUnsanctioned` at the execution stage inside `processSlowModeTransactionImpl` for both `WithdrawCollateral` and `DepositCollateral` transaction types. Specifically, before calling `clearinghouse.withdrawCollateral` or `clearinghouse.depositCollateral`, check:

```solidity
requireUnsanctioned(sender);
requireUnsanctioned(address(bytes20(txn.sender)));
```

This mirrors the pattern already used at submission time and closes the TOCTOU gap between queue entry and execution.

---

### Proof of Concept

1. User `Alice` (address `0xAlice`) calls `submitSlowModeTransaction` with a `WithdrawCollateral` payload for her full balance. `requireUnsanctioned(0xAlice)` passes. The transaction is stored with `executableAt = block.timestamp + 3 days`.
2. One day later, `0xAlice` is added to the `ISanctionsList` contract.
3. After three days, any caller invokes `Endpoint.executeSlowModeTransaction()`.
4. `_executeSlowModeTransaction` calls `this.processSlowModeTransaction(txn.sender, txn.tx)`. [7](#0-6) 

5. `processSlowModeTransactionImpl` decodes the `WithdrawCollateral` payload, calls `validateSender` (passes — caller matches stored sender), then calls `clearinghouse.withdrawCollateral` directly.
6. No `requireUnsanctioned` check is performed. The withdrawal executes and ERC-20 tokens are transferred to `0xAlice`, a now-sanctioned address.

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
