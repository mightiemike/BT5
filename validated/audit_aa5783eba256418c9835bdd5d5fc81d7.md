### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable and Accepts Internal Merkle Tree Nodes as Valid Transaction IDs, Enabling Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The `verify_transaction_inclusion` function is still a live, publicly callable NEAR contract method despite being deprecated. It performs no validation that the supplied `tx_id` is a leaf node (a real transaction) rather than an internal Merkle tree node. Any unprivileged NEAR caller can supply a crafted `tx_id` equal to an internal node hash and a matching sibling proof, causing the function to return `true` for a forged transaction inclusion claim. Consumer contracts that rely on this method for cross-chain verification are exposed to proof forgery with no on-chain defense.

---

### Finding Description

`verify_transaction_inclusion` accepts a caller-supplied `tx_id` (H256), a `tx_index`, and a `merkle_proof` vector, then delegates directly to `compute_root_from_merkle_proof`: [1](#0-0) 

`compute_root_from_merkle_proof` iterates over the proof hashes, using `current_position % 2` to decide left/right placement at each level, and returns whatever hash results: [2](#0-1) 

The only guard in `verify_transaction_inclusion` is `require!(!args.merkle_proof.is_empty(), ...)`. There is no check that `tx_id` is a leaf-level hash. The function's own documentation acknowledges this: [3](#0-2) 

The function is still decorated `#[pause]` and exposed as a public NEAR method, meaning any unprivileged account can call it directly: [4](#0-3) 

`verify_transaction_inclusion_v2` was introduced to mitigate this by requiring a valid coinbase proof of the same depth before delegating to the deprecated path: [5](#0-4) 

However, `verify_transaction_inclusion` was never removed, so the unguarded path remains reachable.

---

### Impact Explanation

A consumer contract (e.g., an atomic swap or bridged-asset release contract) that calls `verify_transaction_inclusion` to confirm a Bitcoin transaction occurred can be deceived. The attacker supplies a `tx_id` equal to an internal Merkle tree node `N` at depth `d`, sets `tx_index` to the correct position for `N`, and provides the single sibling needed to reach the Merkle root. `compute_root_from_merkle_proof` faithfully computes the root and the comparison succeeds. The consumer contract receives `true` for a transaction that does not exist as a leaf, enabling unauthorized fund release or false cross-chain state acceptance.

The corrupted value is the boolean proof result returned to the consumer contract, which maps directly to an authorization decision.

---

### Likelihood Explanation

- The entry point is a public, permissionless NEAR method — no role, stake, or privileged key is required.
- The 64-byte transaction Merkle forgery attack is publicly documented (referenced in the contract's own deprecation notice and at https://www.bitmex.com/blog/64-Byte-Transactions).
- The attacker only needs to identify a real Bitcoin block, compute an internal node hash, and craft the matching proof — all off-chain, with no mining or PoW required.
- Any consumer contract that has not migrated from `verify_transaction_inclusion` to `verify_transaction_inclusion_v2` is immediately exploitable.

---

### Recommendation

Remove `verify_transaction_inclusion` from the contract entirely. It is deprecated, its vulnerability is documented in-code, and `verify_transaction_inclusion_v2` is the correct replacement. Retaining a publicly callable, known-broken verification method creates an ongoing attack surface for every consumer contract that has not yet migrated.

If removal is not immediately possible, add an unconditional `env::panic_str` at the top of `verify_transaction_inclusion` to make it non-callable, preserving the ABI slot without exposing the broken logic.

---

### Proof of Concept

Given a real Bitcoin block with transactions `[T1, T2, T3, T4]` and Merkle tree:

```
Root = hash(N12, N34)
N12  = hash(T1, T2)
N34  = hash(T3, T4)
```

An attacker calls `verify_transaction_inclusion` with:

```
tx_id             = N12   // internal node, not a real transaction
tx_index          = 0
merkle_proof      = [N34]
tx_block_blockhash = <hash of the real block>
confirmations     = 1
```

`compute_root_from_merkle_proof(N12, 0, [N34])`:
- position 0 is even → `hash(N12, N34)` = `Root` ✓

The function returns `true`. The consumer contract is told that the 64-byte crafted "transaction" with hash `N12` is confirmed in the block, which is false. No PoW, no privileged role, and no relayer cooperation is required. [6](#0-5) [1](#0-0)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
