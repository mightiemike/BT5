### Title
Stored `storage_usage` Diverges from Actual Non-Contract Storage in `action_delete_account` for Global-Contract Accounts — (File: `runtime/runtime/src/actions.rs`)

---

### Summary

Before `ProtocolFeature::FixDeleteAccountGlobalContractStorageUsage` is enabled, `action_delete_account` computes the account's non-contract storage usage by subtracting only the local contract code length from the stored `account.storage_usage()`. For accounts with `AccountContract::Global` or `AccountContract::GlobalByAccount` contracts, the stored `storage_usage` includes the identifier overhead (32 bytes for `Global(CryptoHash)`, variable for `GlobalByAccount(AccountId)`), but the legacy code subtracts zero — because there is no local code. The `> MAX_ACCOUNT_DELETION_STORAGE_USAGE` guard therefore fires on the inflated value, permanently blocking deletion of accounts whose actual non-contract storage is at or below the limit.

---

### Finding Description

In `action_delete_account`, the legacy branch executes:

```rust
let account_storage_usage = account_ref.storage_usage();
let code_len = get_code_len_or_default(
    state_update,
    account_id.clone(),
    account_ref.local_contract_hash().unwrap_or_default(),
)?;
account_storage_usage.saturating_sub(code_len)
``` [1](#0-0) 

For `AccountContract::Global(_)` and `AccountContract::GlobalByAccount(_)`, `local_contract_hash()` returns `None` (global contracts carry no local code hash): [2](#0-1) 

`unwrap_or_default()` therefore yields `CryptoHash::default()`, and `get_code_len_or_default` returns 0 for the default hash, so `code_len = 0` and `account_storage_usage` equals the raw stored value unchanged.

However, `account.storage_usage()` was incremented by the identifier overhead when `use_global_contract` was called:

```rust
account.set_storage_usage(
    account.storage_usage().checked_add(contract_identifier.len() as u64)...
);
``` [3](#0-2) 

`identifier_storage_usage()` returns 32 bytes for `Global(CryptoHash)` and `account_id.len()` bytes for `GlobalByAccount(AccountId)`: [4](#0-3) 

The guard that follows:

```rust
if account_storage_usage > Account::MAX_ACCOUNT_DELETION_STORAGE_USAGE {
    result.result =
        Err(ActionErrorKind::DeleteAccountWithLargeState { account_id: account_id.clone() }
            .into());
    return Ok(());
}
``` [5](#0-4) 

therefore fires on the inflated value. An account with `storage_usage = MAX_ACCOUNT_DELETION_STORAGE_USAGE + 32` (i.e., `10_032`) and a `Global(CryptoHash)` contract has actual non-contract storage of exactly `10_000 = MAX`, which is not `> MAX`, yet the legacy path computes `10_032 > 10_000` and rejects the deletion.

The fix branch correctly reads the actual contract storage and subtracts it:

```rust
let contract_storage = get_contract_storage_usage(state_update, account_id, account_ref)?;
account_ref.storage_usage().saturating_sub(contract_storage)
``` [6](#0-5) 

where `get_contract_storage_usage` returns `identifier_storage_usage()` for global contracts: [7](#0-6) 

---

### Impact Explanation

Any account whose `storage_usage` falls in the range `(MAX_ACCOUNT_DELETION_STORAGE_USAGE, MAX_ACCOUNT_DELETION_STORAGE_USAGE + identifier_size]` — concretely `(10_000, 10_032]` for a `Global(CryptoHash)` contract — cannot be deleted. `DeleteAccountAction` returns `ActionErrorKind::DeleteAccountWithLargeState` on every attempt. The account owner cannot recover their storage stake. The exact divergent value is the identifier overhead: 32 Borsh bytes for `Global(CryptoHash)`, `account_id.len()` bytes for `GlobalByAccount(AccountId)`. [8](#0-7) 

---

### Likelihood Explanation

Any unprivileged user can trigger this by:
1. Calling `UseGlobalContractAction` to attach a global contract (enabled on networks that support `AccountV2`).
2. Adding contract-data entries until `storage_usage` lands in the narrow window `(10_000, 10_032]`.

The window is narrow (32 bytes for the common `Global(CryptoHash)` case) but fully user-controlled. The bug is present on every network where global contracts are enabled and `ProtocolFeature::FixDeleteAccountGlobalContractStorageUsage` has not yet been activated. The protocol-version guard in the production code means the legacy path remains live until the feature epoch boundary.

---

### Recommendation

Replace the legacy subtraction with `get_contract_storage_usage`, which correctly handles all three contract variants (`None`, `Local`, `Global`, `GlobalByAccount`). This is already implemented in the `FixDeleteAccountGlobalContractStorageUsage` branch and should be activated at the earliest feasible protocol upgrade.

---

### Proof of Concept

The existing regression test in `runtime/runtime/src/actions.rs` demonstrates the exact failure:

```rust
// Before the fix: the identifier is not subtracted, so `MAX + 32 > MAX`.
let before = test_delete_account_in_empty_trie(
    &account_id,
    AccountContract::Global(CryptoHash::default()),
    Account::MAX_ACCOUNT_DELETION_STORAGE_USAGE + 32,
    enabled - 1,
);
expect_delete_account_too_large(&before);

// From the fix onwards: the 32-byte identifier is subtracted, so
// `MAX + 32 - 32 == MAX`, which is not `> MAX`.
let after = test_delete_account_in_empty_trie(
    &account_id,
    AccountContract::Global(CryptoHash::default()),
    Account::MAX_ACCOUNT_DELETION_STORAGE_USAGE + 32,
    enabled,
);
assert!(after.result.is_ok());
``` [9](#0-8)

fn get_contract_storage_usage(
    state_update: &TrieUpdate,
    account_id: &AccountId,
    account: &Account,
) -> Result<StorageUsage, StorageError> {
    Ok(match account.contract().as_ref() {
        AccountContract::None => 0,
        AccountContract::Local(code_hash) => {
            get_code_len_or_default(state_update, account_id.clone(), *code_hash)?
        }
        AccountContract::Global(_) | AccountContract::GlobalByAccount(_) => {
            account.contract().identifier_storage_usage()
        }
    })
}
