### Title
ERC20 Approval Race Condition in `DirectDepositV1.creditDeposit()` Blocks USDT Deposits - (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` without first resetting the allowance to zero. For tokens with approval race protections (USDT, USDC on some chains), if a non-zero allowance already exists when `creditDeposit()` is called, the `approve()` call will revert, permanently bricking the deposit path for that token.

---

### Finding Description

In `DirectDepositV1.creditDeposit()`, for each collateral token held by the DDA, the function unconditionally calls `approve` to the full current balance and then calls `depositCollateralWithReferral`:

```solidity
// core/contracts/DirectDepositV1.sol
uint256 balance = token.balanceOf(address(this));
if (balance != 0) {
    token.approve(address(endpoint), balance);          // no prior reset to 0
    endpoint.depositCollateralWithReferral(
        subaccount, productId, uint128(balance), "-1"
    );
}
``` [1](#0-0) 

If the endpoint's `depositCollateralWithReferral` does not consume the **exact** approved amount — for example, because it queues a slow-mode transaction rather than immediately calling `transferFrom`, or because `uint128(balance)` truncates a very large balance — the allowance remains non-zero. On the next invocation of `creditDeposit()`, the bare `approve(endpoint, newBalance)` call will revert for USDT and any other token implementing approval race protection.

`creditDeposit()` has **no access control** — it is `external` with no modifier. Any unprivileged caller can invoke it directly, or through the equally unrestricted `ContractOwner.creditDepositV1()`:

```solidity
// core/contracts/ContractOwner.sol
function creditDepositV1(bytes32 subaccount) external {
    ...
    DirectDepositV1(directDepositV1).creditDeposit();
}
``` [2](#0-1) 

By contrast, `ContractOwner.wrapVaultAsset()` correctly resets the allowance to zero before re-approving:

```solidity
assetToken.approve(tokenAddr, 0);
assetToken.approve(tokenAddr, assetBalance);
``` [3](#0-2) 

`creditDeposit()` lacks this reset entirely.

---

### Impact Explanation

Once a non-zero allowance is stranded on the DDA for a USDT-like token, every subsequent call to `creditDeposit()` for that token reverts. The DDA accumulates USDT but can never forward it to the endpoint. Funds are not lost (they remain in the DDA), but the deposit path is permanently bricked until an owner manually calls `withdraw()` and re-routes funds — a non-trivial recovery path.

---

### Likelihood Explanation

`creditDeposit()` is callable by any address with no restriction. [4](#0-3)  The trigger condition (non-zero residual allowance) arises if `depositCollateralWithReferral` queues rather than immediately executes the `transferFrom`, or if `balance > type(uint128).max` causes truncation leaving a residual allowance. USDT is a primary collateral token in most DeFi protocols, making this a realistic integration scenario.

---

### Recommendation

Mirror the pattern already used in `ContractOwner.wrapVaultAsset()`: reset the allowance to zero before setting the new value.

```solidity
if (balance != 0) {
    token.approve(address(endpoint), 0);           // reset first
    token.approve(address(endpoint), balance);
    endpoint.depositCollateralWithReferral(
        subaccount, productId, uint128(balance), "-1"
    );
}
``` [5](#0-4) 

---

### Proof of Concept

1. Deploy a USDT-like token (reverts on `approve` if current allowance ≠ 0).
2. Register it as a collateral product in `SpotEngine`.
3. Send 100 USDT to the DDA.
4. Call `creditDeposit()` — `approve(endpoint, 100)` succeeds; assume `depositCollateralWithReferral` queues the deposit and does not immediately call `transferFrom`, leaving allowance = 100.
5. Send 50 more USDT to the DDA.
6. Call `creditDeposit()` again — `approve(endpoint, 150)` **reverts** because current allowance (100) ≠ 0.
7. All future `creditDeposit()` calls for this token revert; the DDA accumulates USDT indefinitely with no forwarding path. [6](#0-5)
