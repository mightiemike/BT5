### Title
Deprecated `verify_transaction_inclusion` Accepts Unverified Caller-Supplied `tx_id`, Enabling Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The publicly callable `verify_transaction_inclusion` function accepts a caller-supplied `tx_id` and verifies it against the stored Merkle root without any on-chain check that `tx_id` is a valid leaf-node transaction hash. An unprivileged NEAR caller can supply an internal Merkle tree node as `tx_id` and construct a proof that makes the function return `true` for a transaction that was never broadcast or mined. Downstream contracts consuming this result are deceived into treating a fabricated Bitcoin transaction as confirmed.

---

### Finding Description

`verify_transaction_inclusion` in `contract/src/lib.rs` computes the Merkle root from three caller-controlled inputs — `tx_id`, `tx_index`, and `merkle_proof` — and compares the result to the `merkle_root` stored in the accepted block header: [1](#0-0) 

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
```

Bitcoin's Merkle tree hashes pairs of 32-byte values at every level. An internal node at depth *d* is itself a 32-byte SHA256d digest — indistinguishable in format from a leaf transaction hash. If an attacker supplies an internal node as `tx_id` and provides the sibling path that leads from that node to the root, `compute_root_from_merkle_proof` will reproduce the correct `merkle_root` and the function returns `true`.

The contract's own inline documentation acknowledges this: [2](#0-1) 

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash."

The function is marked `#[deprecated(since = "0.5.0")]`, but a Rust deprecation attribute is a compiler-side lint only; it does not remove the method from the compiled WASM binary or restrict on-chain invocation. The function remains `pub`, carries only `#[pause]` (not `#[trusted_relayer]`), and is therefore callable by any unprivileged NEAR account whenever the contract is not paused. [3](#0-2) 

`verify_transaction_inclusion_v2` closes this gap by first validating a coinbase proof at index 0 with the same proof length, which forces the target transaction to occupy a genuine leaf position at the same tree depth: [4](#0-3) 

But because v1 remains independently callable, the mitigation in v2 is bypassed entirely by calling v1 directly.

**Analog mapping to H-01:** In the reference report, `offerAmount` is accepted from an off-chain signer without any on-chain check of collateral value; the protocol trusts the external input and acts on it. Here, `tx_id` is accepted from an unprivileged caller without any on-chain check that it is a valid leaf node; the contract trusts the external input and returns a verification result that downstream contracts act on. Both root causes are identical in class: externally supplied data is accepted and acted upon without on-chain validation of its semantic correctness.

---

### Impact Explanation

Any NEAR contract or off-chain integration that calls `verify_transaction_inclusion` and acts on a `true` result — releasing funds, minting tokens, recording a cross-chain event — can be made to act on a fabricated Bitcoin transaction. The attacker does not need to mine a Bitcoin block or control any privileged role; they only need a real block already accepted by the light client and knowledge of its Merkle tree structure (both publicly available from any Bitcoin node). The corrupted state is the verification result itself: a `true` return for a `tx_id` that does not correspond to any real transaction in the block.

---

### Likelihood Explanation

- Entry path is fully unprivileged: `verify_transaction_inclusion` is `pub` with no role gate.
- The 64-byte internal-node forgery technique is publicly documented (BitMEX research, referenced in the contract's own v2 deprecation note at line 268).
- The attacker needs only a block hash already in the `headers_pool` and the block's Merkle tree, both of which are public Bitcoin data.
- No key material, social engineering, or privileged access is required.

---

### Recommendation

Remove `verify_transaction_inclusion` from the contract's public interface entirely, or gate it so that it panics unconditionally (forcing all callers to migrate to v2). A soft approach — adding a `require!(false, "use verify_transaction_inclusion_v2")` at the top of the function body — would close the attack path without a contract upgrade that changes storage layout.

---

### Proof of Concept

1. Attacker selects any Bitcoin block whose hash is present in the contract's `headers_pool` (e.g., block at height 685452, already accepted during initialization).
2. From the public Bitcoin block data, attacker reads the Merkle tree and picks an internal node `N` at depth *k* (any non-leaf node).
3. Attacker constructs `merkle_proof` as the sibling path from `N` up to the Merkle root — a standard Merkle inclusion proof, but rooted at an internal node rather than a leaf.
4. Attacker calls `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node hash)
   - `tx_index` = the position of `N`'s subtree
   - `merkle_proof` = the sibling path constructed in step 3
   - `tx_block_blockhash` = the real block hash
   - `confirmations = 1`
5. `compute_root_from_merkle_proof(N, tx_index, merkle_proof)` reproduces the real `merkle_root`; the function returns `true`.
6. A downstream bridge contract that calls `verify_transaction_inclusion` and releases funds on a `true` result processes the fabricated transaction, transferring assets to the attacker.

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
