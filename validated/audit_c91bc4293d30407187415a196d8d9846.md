### Title
`SwapAllowlistExtension` Gates the Router Address Instead of the End User, Enabling Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument passed by the pool, which is always `msg.sender` of the `pool.swap` call. When a user swaps through `MetricOmmSimpleRouter`, `msg.sender` seen by the pool is the **router contract**, not the end user. The extension therefore checks whether the router is allowlisted, not whether the actual trader is allowlisted. If the router is added to the allowlist (a natural admin action to "enable router swaps"), every user — including those explicitly excluded — can bypass the allowlist by routing through the router.

---

### Finding Description

**Call chain:**

```
User → MetricOmmSimpleRouter.exactInputSingle
         → IMetricOmmPoolActions(pool).swap(recipient, ...)   // msg.sender = router
              → MetricOmmPool._beforeSwap(msg.sender, ...)    // sender = router
                   → SwapAllowlistExtension.beforeSwap(sender=router, ...)
                        → allowedSwapper[pool][router]  ← checked, NOT the user
```

In `MetricOmmPool.swap`, the pool passes its own `msg.sender` (the router) as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then gates on `sender` (the router address), not on the originating user: [3](#0-2) 

The router calls `pool.swap` without forwarding the original `msg.sender` in any way that the extension can recover: [4](#0-3) 

The same misidentification occurs for every router entry point — `exactInput` (all hops), `exactOutputSingle`, and `exactOutput`: [5](#0-4) 

---

### Impact Explanation

Two concrete fund-impacting outcomes:

1. **Allowlist bypass (primary):** A pool admin allowlists the router address (the natural step to "permit router-mediated swaps"). Because the extension checks `allowedSwapper[pool][router]`, every user — including those the admin explicitly excluded — can now swap on the restricted pool by calling the router. The allowlist guard is completely neutralised for router-routed swaps.

2. **Legitimate users blocked (secondary):** If the admin does *not* allowlist the router, every user who is individually allowlisted is blocked from using the router, breaking the core swap flow for the intended audience.

Both outcomes satisfy the allowed-impact gate: the first is an admin-boundary break where an unprivileged path bypasses a configured access control; the second is broken core pool functionality.

---

### Likelihood Explanation

The trigger requires no special privilege. Any user can call `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) on a pool that has `SwapAllowlistExtension` configured. The bypass is automatic whenever the router is on the allowlist — a configuration a pool admin would naturally make when deploying a router-accessible restricted pool. No malicious setup is required; the bug is structural.

---

### Recommendation

The extension must gate on the **originating user**, not on the intermediate caller. Two complementary fixes:

1. **Pass the real user through the router.** The router should encode the original `msg.sender` in `callbackData` or `extensionData` and the extension should decode it. This requires a trust relationship between the router and the extension.

2. **Gate on `recipient` instead of `sender` for router flows, or introduce a dedicated `originSender` field.** The cleanest fix is to have the pool (or the router) pass the true originator as a separate argument that extensions can rely on, similar to how Uniswap v4 passes `msgSender` through hooks.

At minimum, the `SwapAllowlistExtension` documentation must warn that `sender` is the direct pool caller, not the end user, so pool admins do not inadvertently allowlist the router and open the gate to all users.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured.
  - allowedSwapper[pool][alice] = true   (alice is the intended user)
  - allowedSwapper[pool][bob]   = false  (bob is explicitly excluded)
  - allowedSwapper[pool][router] = true  (admin enables router access)

Attack:
  bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
    → pool.swap(recipient, ...) with msg.sender = router
    → _beforeSwap(sender=router, ...)
    → SwapAllowlistExtension checks allowedSwapper[pool][router] == true  ✓
    → swap proceeds — bob's trade settles against the restricted pool

Result:
  bob, who is explicitly excluded from the allowlist, executes a swap on the
  restricted pool. The allowlist guard is bypassed entirely via the public router.
