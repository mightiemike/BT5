Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the end-user, allowing full allowlist bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension` is intended to gate swaps by per-user address per pool, but its `beforeSwap` hook receives the direct caller of `pool.swap()` as `sender`. When `MetricOmmSimpleRouter` is used, that direct caller is the router contract, not the end-user. Any pool admin who allowlists the router (a necessary step to allow router-based trading) inadvertently grants every unprivileged address the ability to bypass the per-user allowlist entirely.

## Finding Description
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (the pool calls the extension) and `sender` is the first argument the pool passes:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

In `MetricOmmPool.swap()`, the pool passes `msg.sender` — the direct caller of `pool.swap()` — as `sender` to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // ← router address when called via MetricOmmSimpleRouter
    ...
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly from the router contract:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

The full call chain resolves to:

```
end-user → router.exactInputSingle()
         → pool.swap()  [msg.sender = router]
         → _beforeSwap(sender = router, ...)
         → extension.beforeSwap(sender = router, ...)
         → allowedSwapper[pool][router]  ← checked, NOT the end-user
```

The same flaw applies to `exactInput`, `exactOutputSingle`, and `exactOutput` — all four entry points call `pool.swap()` from the router's address. Note that in `exactInput` for intermediate hops, `msg.sender` passed to the callback context is `address(this)` (the router itself), and in `_exactOutputIterateCallback`, recursive `pool.swap()` calls are also made from the router's address.

The existing guard (`allowAllSwappers[pool]`) is a separate bypass path and does not mitigate this issue. There is no secondary check on the actual end-user identity anywhere in the hook or pool code.

## Impact Explanation
The `SwapAllowlistExtension` is the sole mechanism for per-user swap access control in pools that use it. Once the router is allowlisted (a natural and necessary admin action to enable router-based trading for any allowlisted user), the extension's access control is completely nullified: any unprivileged address can swap in a pool intended to be restricted (e.g., KYC-gated, whitelist-only, or compliance-restricted pools). This breaks a core pool functionality — the allowlist gate — and allows unauthorized swaps in restricted pools.

## Likelihood Explanation
The precondition — the router being in `allowedSwapper[pool][router]` — is a natural and expected admin action. Any pool admin who wants allowlisted users to trade through the router must add the router. The moment they do, the bypass is active for all addresses. No special attacker capability is required beyond calling a public router function. The bypass is repeatable and permanent until the router is removed from the allowlist (which would also break legitimate router-based trading for allowlisted users).

## Recommendation
Pass the actual end-user address through the swap call chain so the extension can check it. One approach: include the originating user in `extensionData` and have the extension verify it, combined with a router-level signature or trusted-forwarder pattern. Alternatively, redesign `_beforeSwap` to accept both the direct caller and an optional originator address, and have the router populate the originator. The simplest fix is to have the router pass `msg.sender` (the end-user) in `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check it when the direct `sender` is a known router.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to allow router-based trading.
3. An unprivileged address (not in `allowedSwapper`) calls `router.exactInputSingle(...)` targeting the pool.
4. The pool calls `_beforeSwap(msg.sender=router, ...)`, the extension checks `allowedSwapper[pool][router] == true`, and the swap proceeds — the unprivileged user successfully bypasses the allowlist.

Minimal Foundry test:
```solidity
function test_allowlistBypass() public {
    // admin allowlists router (natural setup)
    extension.setAllowedToSwap(pool, address(router), true);

    // attacker (not allowlisted) swaps via router
    vm.prank(attacker);
    router.exactInputSingle(ExactInputSingleParams({pool: pool, ...}));
    // succeeds — allowlist bypassed
}
```
