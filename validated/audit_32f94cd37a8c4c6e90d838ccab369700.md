### Title
`get_tip_resource_bounds_hash` Produces Divergent Hashes for `ValidResourceBounds::L1Gas` vs `AllResources` with Zero L2/Data Gas Across Protobuf Deserialization Boundary — (`File: crates/starknet_api/src/transaction_hash.rs`, `crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `get_tip_resource_bounds_hash` function hashes a **different number of resource-bound elements** depending on whether `ValidResourceBounds` is the `L1Gas` variant (2 elements: L1 + L2) or the `AllResources` variant (3 elements: L1 + L2 + L1_DATA). The protobuf deserializer in `transaction.rs` silently **downgrades** any `AllResources` value whose `l2_gas` and `l1_data_gas` are both zero into `L1Gas`. A V3 transaction submitted via RPC with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` has its hash computed as `AllResources` (3-element Poseidon input) at admission time, but after round-tripping through protobuf block-sync, the same transaction is deserialized as `L1Gas` and its hash is recomputed from a 2-element Poseidon input — producing a **different hash** for the same signed payload.
