### Title
`burnNlp` Applies Maintenance Health Check Instead of Initial Health Check, Allowing Subaccounts to Hold Positions Below Initial Margin ŌĆö (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.burnNlp` enforces only a `HealthType.MAINTENANCE` health check after burning NLP tokens. Every other collateral-reducing operation in the same contract enforces `HealthType.INITIAL`. Because MAINTENANCE weights are strictly more lenient than INITIAL weights, a user can burn NLP and leave their subaccount in a state where INITIAL health is negative ŌĆö a condition that no other user-triggered operation is permitted to create.

---

### Finding Description

In `Clearinghouse.sol`, the post-burn guard is:

```solidity
// Burning NLP can decrease health if the burn fee exceeds the health improvement
// from the withdrawal. This check prevents malicious actors from deliberately
// creating unhealthy subaccounts through NLP burns.
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [1](#0-0) 

Every other collateral-reducing path in the same contract uses `HealthType.INITIAL`:

| Function | Health check used |
|---|---|
| `withdrawCollateral` | `INITIAL` (line 419) |
| `transferQuote` | `INITIAL` via `_isAboveInitial` (line 249) |
| `mintNlp` | `INITIAL` (line 480) |
| `nlpProfitShare` | `INITIAL` (line 564) |
| `forceRebalanceNlpPool` | `INITIAL` (line 544) |
| **`burnNlp`** | **`MAINTENANCE`** ŌåÉ inconsistency | [2](#0-1) [3](#0-2) [4](#0-3) 

The health model uses two distinct weight sets. `RiskHelper._getWeightX18` returns `longWeightInitialX18` / `shortWeightInitialX18` for `INITIAL` and `longWeightMaintenanceX18` / `shortWeightMaintenanceX18` for `MAINTENANCE`. Maintenance long-weights are higher and maintenance short-weights are lower than their initial counterparts, making MAINTENANCE health always Ōēź INITIAL health for the same position. [5](#0-4) 

The root cause is structurally identical to the reported bug: the wrong threshold is applied to the collateral check. In the reference report, `userTotalDebt.percentMul(liquidationThreshold)` is used where `collateralValue.percentMul(liquidationThreshold)` is required ŌĆö both mistakes make the guard weaker than intended by applying the discount factor to the wrong operand.

---

### Impact Explanation

A user with an open leveraged position (e.g., a short spot liability) whose INITIAL health is just above zero can sign a `BurnNlp` transaction. The burn removes NLP collateral and returns quote minus a burn fee. Because NLP's `longWeightInitialX18` is less than 1 and the burn fee is non-zero, the net effect on INITIAL health can be negative. The MAINTENANCE check passes while INITIAL health drops below zero.

After the burn the subaccount:
- Cannot open new positions (any new trade calls `isUnderInitial` ŌåÆ reverts).
- Cannot be liquidated (MAINTENANCE health Ōēź 0).
- Holds its existing leveraged exposure with less collateral than the protocol's initial margin requirement.

This is a direct solvency/accounting corruption: the protocol's invariant that no user-triggered action may leave a subaccount below INITIAL health is broken. The subaccount is now in a "gray zone" that the protocol's risk model does not account for, and any adverse price move from this under-margined state increases the probability of a bad-debt liquidation that the insurance fund must cover.

---

### Likelihood Explanation

Medium. The trigger requires a user to (1) hold NLP tokens, (2) have an open leveraged position, and (3) size the burn so that post-burn INITIAL health < 0 Ōēż MAINTENANCE health. All three conditions are reachable by any unprivileged user through the normal `BurnNlp` signed-transaction flow processed by `EndpointTx`. [6](#0-5) 

No privileged access, governance capture, or external dependency failure is required.

---

### Recommendation

Replace the MAINTENANCE check in `burnNlp` with an INITIAL check, consistent with every other collateral-reducing operation:

```diff
- require(
-     getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
-     ERR_SUBACCT_HEALTH
- );
+ require(
+     getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
+     ERR_SUBACCT_HEALTH
+ );
``` [7](#0-6) 

---

### Proof of Concept

1. User opens a short spot position. INITIAL health = +╬Ą (just above zero).
2. User mints NLP with surplus quote. INITIAL health rises.
3. User signs a `BurnNlp` transaction sized so that:
   - `nlpAmount * oraclePriceX18 - burnFee` < `nlpAmount * longWeightInitialX18 * oraclePriceX18`
   - i.e., the quote returned is worth less (under INITIAL weights) than the NLP removed.
4. Sequencer submits the transaction; `Clearinghouse.burnNlp` executes.
5. Post-burn state: MAINTENANCE health Ōēź 0 (check passes), INITIAL health < 0.
6. The subaccount now holds its short spot exposure with less collateral than the protocol's initial margin requirement ŌĆö a state unreachable through any other user-controlled path.

    function _getWeightX18(
        Risk memory risk,
        int128 amount,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }

        int128 weight;
        if (amount >= 0) {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.longWeightInitialX18
                : risk.longWeightMaintenanceX18;
        } else {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.shortWeightInitialX18
                : risk.shortWeightMaintenanceX18;
        }

        return weight;
    }
