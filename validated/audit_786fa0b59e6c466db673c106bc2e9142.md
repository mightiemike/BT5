### Title
Canonicality Validation Function Defined But Never Called in Deserialization Path — (`File: src/serde/tools.rs`)

### Summary

`is_canonical_serialization` is a complete, exported canonicality checker for CLVM serialized bytes. It is never invoked inside `node_from_bytes` or `node_from_bytes_backrefs`. As a result, non-canonical CLVM serializations — including overlong atom length prefixes and trailing bytes after a valid object — are silently accepted by every deserialization entry point. This is a direct structural analog to the Dexter bug: a validation function exists and is exported, but is never wired into the actual parsing path.

### Finding Description

`is_canonical_serialization` (src/serde/tools.rs:206–236) performs two distinct checks:

1. **Overlong length prefixes**: e.g., atom `0x01` encoded as `0x8101` (two bytes) instead of the canonical single byte `0x01`. The function computes `min_value` for each prefix length and rejects encodings that use more bytes than necessary.
2. **Trailing bytes**: the final check `f.get_ref().len() as u64 == f.position()` (line 235) rejects any input where bytes remain after the CLVM object ends.

The function is exported from `src/serde/mod.rs` (line 40) and is exercised by fuzz harnesses. However, the actual deserialization entry points never call it:

- `node_from_bytes` (src/serde/de.rs:44–47) calls `node_from_stream` and returns without checking canonicality or whether all bytes were consumed.
- `node_from_bytes_backrefs` (exported from src/serde/de_br.rs, re-exported at src/serde/mod.rs:28) follows the same pattern.

The test suite in `src/serde/tools.rs` (lines 588–616) explicitly documents non-canonical inputs that `is_canonical_serialization` rejects — `"8101"`, `"817f"`, `"ff808080"`, overlong multi-byte prefixes — but none of these inputs are tested against `node_from_bytes` to confirm rejection. They are accepted silently.

### Impact Explanation

An attacker-controlled CLVM byte sequence with a non-canonical encoding (e.g., `[0x81, 0x01]` for the atom `1`, or `[0xff, 0x80, 0x80, 0x80]` for `(NIL . NIL)` with a trailing byte) is accepted by `node_from_bytes` and produces a valid `NodePtr`. The deserialized tree is logically correct, but the raw bytes differ from the canonical form.

Concrete broken invariants:
- **Trailing-byte acceptance**: `node_from_bytes(alloc, &[0xff, 0x80, 0x80, 0x80])` succeeds and returns `(NIL . NIL)`, silently discarding the trailing `0x80`. `is_canonical_serialization` correctly returns `false` for this input.
- **Overlong prefix acceptance**: `node_from_bytes(alloc, &[0x81, 0x01])` succeeds and returns atom `[0x01]`. The canonical encoding is `[0x01]` (one byte). `is_canonical_serialization` returns `false` for `"8101"`.

For blockchain consensus, this means:
- Two distinct byte sequences deserialize to the same CLVM tree and produce the same tree hash and execution result.
- Any system that identifies or deduplicates programs by raw serialized bytes (rather than tree hash) will treat canonical and non-canonical forms as distinct objects, enabling mempool spam with logically identical programs.
- If any node or layer enforces canonicality at a different point in the pipeline than another, the two nodes can diverge on whether a given serialized program is valid — a consensus-divergence risk.

### Likelihood Explanation

Crafting a non-canonical CLVM serialization requires only replacing a single-byte atom encoding with its overlong equivalent, or appending a trailing byte. No special privileges or keys are needed. The attacker-controlled entry path is direct: any API that accepts raw CLVM bytes (Python wheel, RPC, mempool submission) feeds those bytes into `node_from_bytes` or `node_from_bytes_backrefs` without a prior canonicality check.

### Recommendation

Call `is_canonical_serialization` (or inline its logic) inside `node_from_bytes` and `node_from_bytes_backrefs` before or after deserialization, and return an error for non-canonical inputs. Alternatively, add a trailing-bytes check to `node_from_stream` by asserting that the cursor position equals the buffer length after parsing completes. The `strict` parameter pattern already used in `deserialize_2026_body_from_stream` (src/serde_2026/de.rs:43) is a good model: expose a `strict: bool` flag in the legacy deserializers so callers that need to enforce canonicality can do so without breaking existing permissive uses.

pub fn is_canonical_serialization(b: &[u8]) -> bool {
    let mut f = Cursor::new(b);
    let mut counter = 1;
    let mut b = [0; 1];
    while counter > 0 {
        counter -= 1;
        if f.read_exact(&mut b).is_err() {
            return false;
        }
        if b[0] == CONS_BOX_MARKER {
            // we expect to parse two more items from the stream
            // the left and right sub tree
            counter += 2;
        } else if b[0] == BACK_REFERENCE {
            // This is a back-ref. We don't actually need to resolve it, just
            // parse the path and move on
            if f.read_exact(&mut b).is_err() {
                return false;
            }
            if !is_canonical_atom(&mut f, b[0]) {
                return false;
            }
        } else if !is_canonical_atom(&mut f, b[0]) {
            return false;
        }
        if (f.get_ref().len() as u64) < f.position() {
            return false;
        }
    }
    f.get_ref().len() as u64 == f.position()
}
