Audit Report

## Title
SwapAllowlistExtension Bypassed via Router: Any User Can Swap in Allowlist-Gated Pool — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the pool receives `msg.sender = router` and passes that as `sender` to the extension. If the pool admin allowlists the router address to enable router-mediated swaps, every unprivileged user can bypass the per-user allowlist by routing through the public router contract, as the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs its identity check as:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension's caller) and `sender` is the first parameter forwarded by the pool. In `MetricOmmPool.swap`, the pool always passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // ← direct caller of pool.swap(), not the originating EOA
    recipient, ...
);
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly with no mechanism to forward the originating user's address:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
```

The originating user's address is stored only in transient storage for the payment callback and is never surfaced to the pool or the extension. Consequently, when a pool admin allowlists the router address (intending to permit router-mediated swaps for allowlisted users), the extension evaluates `allowedSwapper[pool][router]` — which passes for every user who calls the router, regardless of whether that user is individually allowlisted. The same structural gap exists in `exactInput` (multi-hop) and `exactOutputSingle`/`exactOutput` paths.

## Impact Explanation
The `SwapAllowlistExtension` is a core access-control hook for allowlist-gated pools. When the router is allowlisted, the per-user allowlist is entirely bypassed, allowing any unprivileged user to execute swaps in a pool that was designed to restrict swap access. This constitutes broken core pool functionality — the allowlist invariant fails to hold for router-mediated swaps, enabling unauthorized swap execution in restricted pools.

## Likelihood Explanation
The precondition is that the pool admin has allowlisted the router address. This is a realistic and expected configuration: admins who want to support router-mediated swaps for their allowlisted users must allowlist the router. Once the router is allowlisted, any user can exploit this by simply calling `exactInputSingle` or `exactInput` on the public router. The exploit is trivially repeatable with no special privileges required beyond being able to call the public router.

## Recommendation
The extension must verify the originating user, not the direct caller. Options include:
1. Require that swappers interact directly with the pool (no router intermediary) when the allowlist is active.
2. Pass the originating user's address through `extensionData` and verify it in `beforeSwap`, with the router explicitly encoding `msg.sender` into `extensionData` before calling `pool.swap`.
3. Add a trusted-forwarder pattern where the router encodes the originating user in a verifiable way that the extension can authenticate.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to allowlist the router.
3. Pool admin does NOT call `setAllowedToSwap(pool, attacker, true)`.
4. Attacker calls `MetricOmmSimpleRouter.exactInputSingle` with the allowlist-gated pool.
5. Pool calls `_beforeSwap(router, ...)` → extension checks `allowedSwapper[pool][router]` → passes → swap executes.
6. Assert: attacker successfully swapped despite not being individually allowlisted.
