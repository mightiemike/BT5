### Title
Unconstrained `fee_proposal` During Initiation Window Allows Permanent L2 Gas Price Manipulation — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

The `fee_proposal` bound check in `is_proposal_init_valid` is gated on `fee_actual` being `Some`. During the first `fee_proposal_window_size` (10) blocks after V0_14_3 activation, `fee_actual` is always `None`, so the bound check is completely skipped. A malicious proposer selected during this initiation window can set `fee_proposal_fri` to any arbitrary value. Those values are recorded into the sliding window and become the `fee_actual` floor for all subsequent blocks, permanently locking the L2 gas price at an extreme level.

### Finding Description

In `is_proposal_init_valid` the fee-proposal range check uses an `if let` pattern that only fires when both `fee_actual` and `fee_proposal` are `Some`:

```rust
// Validate fee_proposal is within the configured margin of fee_actual.
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    let (lower_bound, upper_bound) = fee_proposal_bounds(
        fee_actual,
        VersionedConstants::latest_constants().fee_proposal_margin_ppt,
    );
    if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
        return Err(ValidateProposalError::InvalidProposalInit(...));
    }
}
``` [1](#0-0) 

`fee_actual` is computed by `compute_fee_actual`, which returns `None` whenever the sliding window does not yet contain `window_size` complete entries:

```rust
let Some(start) = height.0.checked_sub(window_size) else {
    warn!("Cannot compute fee_actual for height {height}: height is below window_size ({window_size})");
    return None;
};
``` [2](#0-1) 

`fee_proposal_window_size` is 10 blocks across all deployed versioned-constants files: [3](#0-2) 

Both the proposer path (`build_proposal`) and the validator path (`validate_proposal`) derive `fee_actual` from the same window: [4](#0-3) 

Because both sides see `fee_actual = None` during the initiation window, both sides skip the bound check and accept any `fee_proposal_fri` value. The accepted value is then committed into the block header and fed back into the window via `record_fee_proposal`: [5](#0-4) 

After the window fills, `fee_actual` becomes the median of those seeded values. `calculate_next_l2_gas_price_for_fin` uses `fee_actual` as a hard floor:

```rust
let effective_min = match fee_actual {
    Some(fa) => GasPrice(max(config_min.0, fa.0)),
    None => config_min,
};
calculate_next_base_gas_price(current_l2_gas_price, l2_gas_used, gas_target, effective_min)
``` [6](#0-5) 

With `fee_proposal_margin_ppt = 2` (0.2 ‰ per block), once `fee_actual` is anchored at an extreme value the per-block adjustment cap makes recovery essentially impossible within any practical timeframe.

### Impact Explanation

A malicious proposer selected during the first 10 blocks after V0_14_3 activation can set `fee_proposal_fri = u128::MAX`. Validators accept this without any bound check. After 10 such blocks the median `fee_actual` becomes `u128::MAX`, which becomes the `effective_min` for every subsequent block's gas price. All user transactions then require a gas price of `u128::MAX` FRI, which is economically impossible to pay — a permanent network-wide freeze of funds. This matches the "Critical. Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact" scope.

### Likelihood Explanation

The attack window is the first `fee_proposal_window_size` (10) blocks after V0_14_3 activates. Any validator node that is selected as proposer during those blocks — a normal, rotating BFT role — can trigger this without any special privilege. The condition is deterministic and predictable (the window is always empty at protocol activation), so an adversary can time their participation accordingly.

### Recommendation

Enforce a bound on `fee_proposal` even when `fee_actual` is `None`. During the initiation window, clamp `fee_proposal` to a reasonable range anchored to the current `l2_gas_price` (the EIP-1559 base fee already in use) or to `[min_gas_price, some_configured_max]`. For example:

```rust
// Always enforce a sanity bound, even during initiation.
let reference = proposal_init_validation.fee_actual
    .unwrap_or(proposal_init_validation.l2_gas_price_fri);
if let Some(fee_proposal) = init_proposed.fee_proposal_fri {
    let (lower_bound, upper_bound) = fee_proposal_bounds(
        reference,
        VersionedConstants::latest_constants().fee_proposal_margin_ppt,
    );
    if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
        return Err(ValidateProposalError::InvalidProposalInit(...));
    }
}
```

This mirrors the fix recommended for the WithdrawHook bug: add the limit check inside the "reset" branch as well, so the first request in a new period cannot bypass the limit.

### Proof of Concept

1. Network activates Starknet V0_14_3. `fee_proposals_window` is empty; `compute_fee_actual` returns `None` for all heights `< window_size` (10).
2. Malicious validator is selected as proposer for block 5 (within the initiation window).
3. Proposer constructs `ProposalInit` with `fee_proposal_fri = Some(GasPrice(u128::MAX))`.
4. Validator node calls `is_proposal_init_valid`. `proposal_init_validation.fee_actual` is `None`.
5. The `if let (Some(fee_actual), Some(fee_proposal)) = ...` pattern does **not** match; the bound check is skipped entirely.
6. The proposal is accepted and committed. `fee_proposal_fri = u128::MAX` is written into the block header.
7. `record_fee_proposal(BlockNumber(5), Some(GasPrice(u128::MAX)))` stores the extreme value in the window.
8. After 10 blocks with extreme values, `compute_fee_actual` returns `Some(GasPrice(u128::MAX))`.
9. `calculate_next_l2_gas_price_for_fin` sets `effective_min = u128::MAX`; `calculate_next_base_gas_price` returns `u128::MAX`.
10. Every subsequent block has `l2_gas_price = u128::MAX`. All user transactions are rejected for insufficient gas price. The network is permanently frozen.
