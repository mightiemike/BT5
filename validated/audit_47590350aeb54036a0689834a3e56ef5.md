### Title
Duplicate Position ID in `MsgClaimTierRewards` Enables Repeated Bonus Reward Drain - (`x/tieredrewards/keeper/msg_server.go`)

### Summary

`ClaimTierRewards` loads all requested positions into a slice before processing them. If the same position ID appears more than once in `msg.PositionIds`, each duplicate entry carries a stale in-memory copy of the position's `LastBonusAccrual` checkpoint. The bonus computation for the final time-segment uses that stale checkpoint, so the same accrued bonus is paid out once per duplicate entry, draining the module's bonus rewards pool.

---

### Finding Description

`ClaimTierRewards` in `msg_server.go` first loads every requested position into a local slice, then hands the whole slice to `claimRewardsAndUpdatesPositions`:

```go
// msg_server.go – ClaimTierRewards
positions := make([]types.PositionState, 0, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    pos, err := ms.getPositionState(ctx, posId)   // reads from store
    ...
    positions = append(positions, pos)             // snapshot in memory
}
totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
``` [1](#0-0) 

`claimRewardsAndUpdatesPositions` then iterates over the slice, calling `processEventsAndClaimBonus` on each entry and persisting the result:

```go
// claim_rewards.go – claimRewardsAndUpdatesPositions
for i := range positions {
    pos := &positions[i]
    ...
    bonus, err := k.processEventsAndClaimBonus(ctx, pos)
    ...
    if err := k.setPosition(ctx, pos.Position, nil); err != nil { ... }
}
``` [2](#0-1) 

Inside `processEventsAndClaimBonus`, the bonus for the current open segment is computed using `pos.LastBonusAccrual` as the segment start, and `blockTime` as the end:

```go
segmentStart := pos.LastBonusAccrual          // from the in-memory copy
...
bonus := k.computeSegmentBonus(*pos, tier, segmentStart, blockTime, currentRate)
...
applyBonusAccrualCheckpoint(&pos.Position, blockTime)  // advances checkpoint in memory
``` [3](#0-2) 

`applyBonusAccrualCheckpoint` advances `LastBonusAccrual` to `blockTime` only on the in-memory struct that was passed in:

```go
func applyBonusAccrualCheckpoint(pos *types.Position, blockTime time.Time) {
    accrualEnd := blockTime
    if pos.CompletedExitLockDuration(blockTime) { accrualEnd = pos.ExitUnlockAt }
    pos.UpdateLastBonusAccrual(accrualEnd)
}
``` [4](#0-3) 

When `msg.PositionIds = [1, 1, 1, …]`, the slice contains N independent copies of position 1, all loaded with the same original `LastBonusAccrual = T0`. The first iteration advances `positions[0].LastBonusAccrual` to `blockTime` and persists it to the store. The second iteration operates on `positions[1]`, which still holds `LastBonusAccrual = T0`. `processEventsAndClaimBonus` therefore recomputes the full bonus for `[T0, blockTime]` and pays it out again from the module pool. This repeats for every duplicate entry.

Even if validator events were already decremented and deleted by the first iteration, the final open-segment computation (lines 206–212 of `claim_rewards.go`) does not depend on stored events — it only needs `pos.LastBonusAccrual` and the current validator rate, both of which are available from the stale copy. [5](#0-4) 

There is no deduplication of `PositionIds` anywhere in `ClaimTierRewards` or in the `Validate()` call on the message. [6](#0-5) 

---

### Impact Explanation

An attacker who owns any tiered-rewards position can drain the module's `RewardsPoolName` bonus pool by submitting a single `MsgClaimTierRewards` transaction with the same position ID repeated N times. Each repetition transfers `bonus_per_period` tokens from the pool to the attacker. The pool is a shared community resource that funds bonus APY for all stakers; draining it directly harms every other participant's expected yield.

The corrupted value is the balance of the `types.RewardsPoolName` module account, which is decremented N times instead of once per claim. [7](#0-6) 

---

### Likelihood Explanation

The entry path is a standard, unprivileged Cosmos SDK transaction (`MsgClaimTierRewards`) available to any position owner. No special role, leaked key, or social engineering is required. The only prerequisite is holding a tiered-rewards position with accrued bonus — the normal state for any long-term staker. The attack is executable in a single transaction.

---

### Recommendation

Deduplicate `msg.PositionIds` before loading positions, or re-read each position from the store inside `claimRewardsAndUpdatesPositions` rather than operating on a pre-loaded slice. The simplest fix is to add a uniqueness check in `MsgClaimTierRewards.Validate()` or at the top of `ClaimTierRewards`:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, id := range msg.PositionIds {
    if _, dup := seen[id]; dup {
        return nil, errorsmod.Wrapf(sdkerrors.ErrInvalidRequest, "duplicate position id: %d", id)
    }
    seen[id] = struct{}{}
}
```

Alternatively, re-fetch each position from the store at the start of each loop iteration in `claimRewardsAndUpdatesPositions` to ensure the checkpoint is always current.

---

### Proof of Concept

1. User holds position ID `42` with `LastBonusAccrual = T0` (e.g., 30 days ago), earning 4 % bonus APY on 10 000 CRO → accrued bonus ≈ 32.9 CRO.
2. User broadcasts:
   ```json
   { "type": "MsgClaimTierRewards",
     "owner": "cro1...",
     "position_ids": [42, 42, 42, 42, 42, 42, 42, 42, 42, 42] }
   ```
3. `ClaimTierRewards` loads 10 identical copies of position 42 (all with `LastBonusAccrual = T0`) into the `positions` slice.
4. Each of the 10 iterations calls `processEventsAndClaimBonus` with a stale copy, computes ≈ 32.9 CRO, and calls `bankKeeper.SendCoinsFromModuleToAccount` from `RewardsPoolName`.
5. User receives ≈ 329 CRO instead of ≈ 32.9 CRO; the pool is drained by 10× the legitimate amount. [8](#0-7) [9](#0-8)

func (ms msgServer) ClaimTierRewards(ctx context.Context, msg *types.MsgClaimTierRewards) (*types.MsgClaimTierRewardsResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	positions := make([]types.PositionState, 0, len(msg.PositionIds))
	for _, posId := range msg.PositionIds {
		pos, err := ms.getPositionState(ctx, posId)
		if err != nil {
			return nil, err
		}

		if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
			return nil, err
		}

		positions = append(positions, pos)
	}

	totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventTierRewardsClaimed{
		Owner:        msg.Owner,
		PositionIds:  msg.PositionIds,
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
	}); err != nil {
		return nil, err
	}

	return &types.MsgClaimTierRewardsResponse{
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
		PositionIds:  msg.PositionIds,
	}, nil
}

func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
}
