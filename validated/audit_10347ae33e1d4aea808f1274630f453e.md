### Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Caller to Bypass the Deposit Allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is documented as gating `addLiquidity` by **depositor address**. Its `beforeAddLiquidity` hook silently ignores the `sender` argument (the actual caller who provides tokens via callback) and instead checks the caller-controlled `owner` argument (the LP position recipient). Any unprivileged address can bypass the allowlist by supplying an allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` calls the extension hook with `sender = msg.sender` and `owner` = the caller-supplied LP-position owner: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both values positionally: [2](#0-1) 

Inside the extension, the first positional parameter (`sender`) is unnamed and discarded; the allowlist check is performed only on `owner`: [3](#0-2) 

The contract's own NatSpec ("Gates `addLiquidity` by depositor address") and the setter name `setAllowedToDeposit(pool_, depositor, allowed)` make the intent unambiguous: the check should be on the entity that calls the pool and funds the deposit, i.e., `sender`. [4](#0-3) 

Because `owner` is a free parameter supplied by the caller, any address can pass the guard by setting `owner` to any address that the pool admin has allowlisted.

---

### Impact Explanation

The deposit allowlist is the sole on-chain mechanism a pool admin has to restrict who may inject liquidity into a permissioned pool. With this bug the restriction is entirely ineffective: an unprivileged caller can deposit arbitrary token amounts into the pool, bypassing the intended access control. Concretely:

- The unauthorized caller provides tokens (via the `IMetricOmmModifyLiquidityCallback` callback) and the allowlisted address receives the LP position.
- The pool's token balances and bin accounting are updated as if a legitimate deposit occurred.
- The pool admin's invariant — "only allowlisted addresses may add liquidity" — is broken for every pool that relies on this extension.

Because `removeLiquidity` enforces `msg.sender == owner`, the deposited tokens are locked under the allowlisted address's position; the attacker cannot directly recover them. The primary harm is the broken access-control invariant and the ability to force liquidity into a pool that is supposed to be restricted (e.g., regulatory-compliance pools, curated LP sets, or pools where unauthorized liquidity dilutes or distorts existing LP positions).

---

### Likelihood Explanation

High. No special role, privilege, or precondition is required. Any externally-owned account or contract can call `pool.addLiquidity(allowlistedAddress, ...)` at any time. The only cost to the attacker is the tokens they supply, which they cannot recover without the allowlisted address's cooperation.

---

### Recommendation

Replace the ignored first parameter with `sender` and check it against the allowlist:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol

function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

---

### Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` attached and calls:
   ```solidity
   extension.setAllowedToDeposit(pool, alice, true);
   // bob is NOT allowlisted
   ```
2. Bob (unauthorized) constructs a callback contract that funds the deposit and calls:
   ```solidity
   pool.addLiquidity(
       alice,          // owner — allowlisted, passes the broken check
       salt,
       deltas,
       callbackData,   // Bob's contract funds the transfer here
       extensionData
   );
   ```
3. Inside `beforeAddLiquidity`, `msg.sender` is the pool, `owner` is `alice`:
   ```
   allowedDepositor[pool][alice] == true  →  guard passes
   ```
4. Bob's callback transfers tokens into the pool; Alice receives the LP shares.
5. Bob has deposited into a pool he is explicitly barred from, bypassing the allowlist entirely. [3](#0-2) [5](#0-4)
