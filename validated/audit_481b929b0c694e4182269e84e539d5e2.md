### Title
Negative `canSettle` in `settlePnl` Inflates Global `availableSettle` Pool, Corrupting Protocol Solvency — (`File: core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.settlePnl` does not guard against a negative `canSettle` value. When a subaccount holds a losing perp position (`positionPnl < 0`), calling `settlePnl` on it causes `state.availableSettle` to **increase** rather than decrease, inflating the global pool of settleable PnL. Other users can then drain more USDC from the system than actually exists as realized profit, corrupting protocol solvency.

---

### Finding Description

`getSettlementState` computes `canSettle` as:

```solidity
// PerpEngine.sol lines 135-138
availableSettle = MathHelper.min(
    calculatePositionPnl(balance, productId),
    state.availableSettle
);
```

`calculatePositionPnl` returns `priceX18.mul(balance.amount) + balance.vQuoteBalance`. For a losing position (e.g., a long entered at 1000 USDC when the current price is 900 USDC), this value is **negative**. Since `MathHelper.min(negative, non-negative) = negative`, `canSettle` becomes negative. [1](#0-0) 

`settlePnl` then applies this negative value without any guard:

```solidity
// PerpEngine.sol lines 94-97
state.availableSettle -= canSettle;   // -= negative → INCREASES availableSettle
balance.vQuoteBalance -= canSettle;   // -= negative → INCREASES vQuoteBalance (loss reduced)
totalSettled += canSettle;            // negative total returned
``` [2](#0-1) 

`Clearinghouse._settlePnl` then applies the negative `amountSettled` to the subaccount's USDC balance:

```solidity
// Clearinghouse.sol lines 617-627
int128 amountSettled = perpEngine.settlePnl(subaccount, productIds);
_spotEngine().updateBalance(QUOTE_PRODUCT_ID, subaccount, amountSettled);
``` [3](#0-2) 

The net accounting effect when `canSettle = -X` (X > 0):

| State | Change |
|---|---|
| `state.availableSettle` | **+X** (inflated — wrong) |
| `balance.vQuoteBalance` | **+X** (loss reduced) |
| Subaccount USDC balance | **-X** (correct deduction) |

The perp loss is correctly transferred to the USDC balance, but `state.availableSettle` is simultaneously inflated by the same amount. Because `state.availableSettle` is the global cap on how much positive PnL any user can settle, this inflation allows profitable users to extract X more USDC from the system than actually exists as realized profit. [4](#0-3) 

---

### Impact Explanation

`state.availableSettle` is the protocol-wide pool of USDC that backs positive PnL settlements. Inflating it by X means the system promises X more USDC to profitable traders than it actually holds. Each time a losing position is settled via this path, the pool grows by the full magnitude of the loss. Profitable users who subsequently call `settlePnl` drain real USDC from the system against this phantom pool, leaving the protocol insolvent. The corrupted invariant is: `state.availableSettle` must never exceed the sum of all realized positive PnL not yet settled. [5](#0-4) 

---

### Likelihood Explanation

`Clearinghouse.settlePnl` is `onlyEndpoint` and is dispatched by the sequencer via `submitTransactionsChecked`. The sequencer routinely batches `SettlePnl` transactions across many subaccounts. Because the contract imposes no check that `positionPnl >= 0` before settling, any subaccount with an open losing position that is included in a settlement batch triggers the corruption. The sequencer does not need to be compromised — it only needs to include a losing subaccount in a routine settlement batch, which is a normal operational scenario. The root cause is entirely in the Nado contract code, not in sequencer behavior. [6](#0-5) 

---

### Recommendation

In `getSettlementState`, clamp `availableSettle` to zero from below so that a losing position can never produce a negative settlement amount:

```solidity
availableSettle = MathHelper.min(
    MathHelper.max(0, calculatePositionPnl(balance, productId)),
    state.availableSettle
);
```

Alternatively, add an explicit guard at the top of the `settlePnl` loop body:

```solidity
if (canSettle <= 0) { productIds >>= 32; continue; }
```

This ensures `state.availableSettle` is only ever decreased (or unchanged) by a settlement call, preserving the solvency invariant. [7](#0-6)

### Proof of Concept

**Setup:**
- Product `perpId = 2` (even → perp product)
- User A opens a long: `balance.amount = 1e18`, `balance.vQuoteBalance = -1000e18` (entered at 1000 USDC)
- Current oracle price: 900 USDC → `positionPnl = 900e18 - 1000e18 = -100e18`
- `state.availableSettle = 50e18` (from prior profitable trades by other users)

**Step 1 — Sequencer submits `SettlePnl` for User A:**

```
canSettle = min(-100e18, 50e18) = -100e18
state.availableSettle = 50e18 - (-100e18) = 150e18   ← inflated
balance.vQuoteBalance = -1000e18 - (-100e18) = -900e18 ← loss reduced
totalSettled = -100e18
User A USDC balance -= 100e18                         ← correct deduction
```

**Step 2 — User B (profitable, `positionPnl = +50e18`) calls `settlePnl`:**

```
canSettle = min(50e18, 150e18) = 50e18
User B USDC balance += 50e18                          ← extracted from phantom pool
state.availableSettle = 150e18 - 50e18 = 100e18       ← still inflated by 100
```

**Result:** The system has paid out 50 USDC to User B against a pool that was inflated by 100 USDC. The remaining phantom 100 USDC in `availableSettle` can be drained by any future profitable settler, extracting real USDC that the system does not hold. [5](#0-4) [8](#0-7)

    function settlePnl(bytes32 subaccount, uint256 productIds)
        external
        returns (int128)
    {
        _assertInternal();
        int128 totalSettled = 0;

        while (productIds != 0) {
            uint32 productId = uint32(productIds & ((1 << 32) - 1));
            // otherwise it means the product is a spot.
            if (productId % 2 == 0) {
                (
                    int128 canSettle,
                    State memory state,
                    Balance memory balance
                ) = getSettlementState(productId, subaccount);

                state.availableSettle -= canSettle;
                balance.vQuoteBalance -= canSettle;

                totalSettled += canSettle;

                _setState(productId, state);
                _setBalanceAndUpdateBitmap(productId, subaccount, balance);
            }
            productIds >>= 32;
        }
        return totalSettled;
    }
