### Title
Insufficient Bonus Pool Balance Permanently Traps Delegated Position Principal — (`x/tieredrewards/keeper/claim_rewards.go`, `bonus_rewards.go`, `msg_server.go`)

---

### Summary

Every user-accessible exit path for a delegated position unconditionally calls `claimRewards`, which calls `processEventsAndClaimBonus`, which performs a hard balance check against `RewardsPoolName`. If the pool balance is below the accrued bonus, `ErrInsufficientBonusPool` is returned and the entire transaction reverts. There is no code path that allows a position owner to recover their principal delegation when the bonus pool is depleted.

---

### Finding Description

`ExitTierWithDelegation` at `msg_server.go:532` calls `claimRewards` unconditionally before performing any delegation transfer: [1](#0-0) 

`claimRewards` in `claim_rewards.go:97` calls `processEventsAndClaimBonus`: [2](#0-1) 

`processEventsAndClaimBonus` computes `totalBonus` by walking all validator events since `pos.LastEventSeq`, then at line 230 calls `sufficientBonusPoolBalance`: [3](#0-2) 

`sufficientBonusPoolBalance` in `bonus_rewards.go` returns a hard error if the pool is short: [4](#0-3) 

This error propagates up through `processEventsAndClaimBonus` → `claimRewards` → `ExitTierWithDelegation`, reverting the entire transaction. The delegation transfer at line 548 is never reached. [5](#0-4) 

**Every other delegated-position exit path has the same flaw:**

- `TierUndelegate` calls `claimRewards` at `msg_server.go:166`
- `TierRedelegate` calls `claimRewards` at `msg_server.go:229`
- `ClearPosition` calls `claimRewards` at `msg_server.go:406`
- `ForceFullExitWithDelegation` (migration helper, not user-accessible) also calls `claimRewards` at `force_exit.go:37` [6](#0-5) 

The only path that skips `claimRewards` is `WithdrawFromTier`, but `validateWithdrawFromTier` requires `!pos.IsDelegated()` — meaning the position must already be undelegated, which itself requires `TierUndelegate` to succeed first. [7](#0-6) 

There is no "exit-without-bonus" code path anywhere in the module.

---

### Impact Explanation

When `RewardsPoolName` balance falls below the accrued bonus for any position, the owner of that position cannot:
- Exit with delegation (`ExitTierWithDelegation`)
- Undelegate (`TierUndelegate`)
- Redelegate (`TierRedelegate`)
- Clear the position (`ClearPosition`)

The staked principal is permanently locked in the position's delegator sub-account until the pool is externally refilled. This is a permanent fund lock with no user-side remedy.

---

### Likelihood Explanation

The `RewardsPoolName` module account is a finite pool. It can be depleted by:
- A large number of simultaneous bonus claims
- A governance action that drains or fails to replenish the pool
- A bug elsewhere that over-distributes from the pool

Once depleted, every position with non-zero accrued bonus is simultaneously locked. The condition is reachable without any privileged access — it is a natural consequence of the pool being finite and the check being a hard gate on all exit paths.

---

### Recommendation

Decouple principal recovery from bonus payout. When `sufficientBonusPoolBalance` fails, the implementation should either:

1. **Skip the bonus payout and record the owed amount as a debt** — allow the exit to proceed, and let the owner claim the bonus later once the pool is refilled; or
2. **Cap the bonus to available pool balance** — pay out whatever is available and advance the checkpoint, so the exit is never blocked.

The `sufficientBonusPoolBalance` hard-error should never gate principal recovery. The check is appropriate for `ClaimTierRewards` (a pure reward claim), but must not block `ExitTierWithDelegation`, `TierUndelegate`, or `TierRedelegate`.

---

### Proof of Concept

```
1. Create a position with a non-zero BonusApy tier; let it accrue bonus over time.
2. Drain RewardsPoolName to zero (e.g., via a separate ClaimTierRewards call that
   exhausts the pool, or by directly setting pool balance to zero in a keeper test).
3. Call MsgExitTierWithDelegation (or MsgTierUndelegate) for the position.
4. Observe: transaction reverts with ErrInsufficientBonusPool (tieredrewards error 12).
5. Repeat indefinitely — the owner has no alternative exit path while the pool is empty.
```

The exact state delta: `RewardsPoolName.balance < pos.accruedBonus` → all exit messages for that position return error code 12, delegation shares remain in the position's sub-account, owner cannot recover principal.

func (k Keeper) claimRewards(ctx context.Context, pos types.PositionState) (types.PositionState, sdk.Coins, sdk.Coins, error) {
	if !pos.IsDelegated() {
		return pos, sdk.NewCoins(), sdk.NewCoins(), nil
	}

	base, err := k.claimBaseRewards(ctx, pos)
	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	bonus, err := k.processEventsAndClaimBonus(ctx, &pos)
	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	return pos, base, bonus, nil
}

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
