### Title
`allow_client_side_proving = false` Config Gate Enforced at Gateway but Absent from Consensus Conversion Path — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The `allow_client_side_proving` flag in `StatelessTransactionValidatorConfig` is checked at the Gateway RPC entry point via `StatelessTransactionValidator::validate_client_side_proving_allowed`, but the identical conversion function `convert_rpc_tx_to_internal` is called without that check in the consensus proposal-validation path. When an operator sets `allow_client_side_proving = false` to disable client-side proving, a proposer can include invoke-V3 transactions carrying non-empty `proof_facts` in a block proposal; the validator node will convert, forward to the batcher, and execute them — bypassing the intended config gate entirely.

### Finding Description

**Gate at the Gateway (RPC path)**

`GenericGateway::add_tx_inner` calls `self.stateless_tx_validator.validate(&tx)`, which invokes `validate_client_side_proving_allowed`:

```rust
fn validate_client_side_proving_allowed(
    &self,
    tx: &RpcInvokeTransaction,
) -> StatelessTransactionValidatorResult<()> {
    if self.config.allow_client_side_proving {
        return Ok(());
    }
    let RpcInvokeTransaction::V3(tx) = tx;
    let has_proof_data = !tx.proof_facts.is_empty() || !tx.proof.is_empty();
    if has_proof_data {
        return Err(StatelessTransactionValidatorError::ClientSideProvingNotAllowed);
    }
    Ok(())
}
``` [1](#0-0) 

This is wired into the gateway's `add_tx_inner` flow: [2](#0-1) 

**No equivalent gate in the consensus path**

When a validator receives a proposal, `handle_proposal_part` in `validate_proposal.rs` converts each `ConsensusTransaction` by calling:

```rust
transaction_converter.convert_consensus_tx_to_internal_consensus_tx(tx)
``` [3](#0-2) 

`convert_consensus_tx_to_internal_consensus_tx` delegates to the shared private helper `convert_rpc_tx_to_internal`: [4](#0-3) 

`convert_rpc_tx_to_internal` performs compiled-class-hash validation for Declare and contract-address derivation for DeployAccount, but for Invoke it simply strips the `proof` field and passes `proof_facts` through — **no `allow_client_side_proving` check is present**: [5](#0-4) 

The resulting `InternalRpcTransaction` (with `proof_facts` intact) is forwarded to the batcher via `send_txs_for_proposal`, and the blockifier executes it. The blockifier's own `validate_proof_facts` checks `allowed_proof_versions`, `allowed_virtual_os_program_hashes`, and config-hash — but it has no knowledge of the `allow_client_side_proving` flag: [6](#0-5) 

**Two entry points, one invariant, one missing check:**

| Entry point | `allow_client_side_proving` enforced? |
|---|---|
| HTTP server → Gateway → `add_tx_inner` | ✅ via `stateless_tx_validator.validate` |
| Consensus → `validate_proposal` → `handle_proposal_part` | ❌ absent |

### Impact Explanation

When an operator sets `allow_client_side_proving = false` (e.g., because the feature is considered unsafe, unaudited, or temporarily disabled), the intent is that **no** invoke-V3 transaction carrying `proof_facts` should be accepted or executed by the node. The gateway enforces this for user-submitted transactions. However, any proposer in the validator set can include such transactions in a block proposal. The validating node will convert them through `convert_rpc_tx_to_internal` without the config check, forward them to the batcher, and execute them. If the embedded proof is cryptographically valid, the blockifier accepts the transaction and commits its state changes — directly contradicting the operator's config intent.

This matches the "High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing" impact: the consensus-path admission step accepts transactions that the node's own config declares inadmissible.

### Likelihood Explanation

Any validator that is scheduled as a proposer can trigger this. Proposer rotation is a normal part of the Tendermint-based consensus protocol, so the trigger is reachable by any participant in the validator set without additional privilege escalation. The `allow_client_side_proving` flag defaults to `true` in production config, so the window is limited to deployments that explicitly set it to `false` — but that is precisely the scenario where the gate is supposed to be enforced. [7](#0-6) 

### Recommendation

Apply the same `allow_client_side_proving` check inside `convert_rpc_tx_to_internal` (or as a pre-conversion guard in `handle_proposal_part`) so that the invariant is enforced regardless of which entry point delivers the transaction. The `TransactionConverter` already holds `chain_id`; it can be extended to hold a `allow_client_side_proving` flag sourced from node config, or the check can be added as a stateless guard in `handle_proposal_part` before conversion:

```rust
// In handle_proposal_part, before conversion:
if !config.allow_client_side_proving {
    if let ConsensusTransaction::RpcTransaction(RpcTransaction::Invoke(
        RpcInvokeTransaction::V3(ref tx))) = tx {
        if !tx.proof_facts.is_empty() || !tx.proof.is_empty() {
            return HandledProposalPart::Failed("ClientSideProvingNotAllowed".into());
        }
    }
}
```

### Proof of Concept

1. Configure the node with `gateway_config.static_config.stateless_tx_validator_config.allow_client_side_proving = false`.
2. Construct an invoke-V3 `RpcTransaction` with non-empty `proof_facts` and a valid `proof`.
3. Submit it via the HTTP gateway → rejected with `ClientSideProvingNotAllowed` (gate works).
4. As a scheduled proposer, include the same transaction in a `ProposalPart::Transactions` batch and stream it to a validating peer.
5. The validator's `handle_proposal_part` calls `convert_consensus_tx_to_internal_consensus_tx`, which calls `convert_rpc_tx_to_internal` — no `allow_client_side_proving` check fires.
6. The transaction is forwarded to the batcher via `send_txs_for_proposal` and executed by the blockifier.
7. If the proof is cryptographically valid, the blockifier's `validate_proof_facts` passes and the transaction is committed — bypassing the operator's config gate. [8](#0-7) [1](#0-0)

    fn validate_proof_facts(
        &self,
        block_context: &BlockContext,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        // Only Invoke V3 transactions can carry proof facts.
        let Transaction::Invoke(invoke_tx) = &self.tx else {
            return Ok(());
        };
        if invoke_tx.version() < TransactionVersion::THREE {
            return Ok(());
        }

        // Parse proof facts.
        let proof_facts = invoke_tx.proof_facts();
        let snos_proof_facts = match ProofFactsVariant::try_from(&proof_facts)
            .map_err(|e| TransactionPreValidationError::InvalidProofFacts(e.to_string()))?
        {
            ProofFactsVariant::Empty => return Ok(()),
            ProofFactsVariant::Snos(snos_proof_facts) => snos_proof_facts,
        };
        let os_constants = &block_context.versioned_constants.os_constants;

        if !os_constants.allowed_proof_versions.contains(&snos_proof_facts.proof_version.as_felt())
        {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Proof version {} is not allowed under this protocol version.",
                snos_proof_facts.proof_version
            )));
        }

        // Validate the program hash.
        let allowed = &os_constants.allowed_virtual_os_program_hashes;
        if !allowed.contains(&snos_proof_facts.program_hash) {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS program hash {} is not allowed",
                snos_proof_facts.program_hash
            )));
        }

        // Validate the block hash and block number.
        let proof_block_hash = snos_proof_facts.block_hash.0;
        let proof_block_number = snos_proof_facts.block_number.0;
        Self::validate_proof_block_number(
            proof_block_number,
            block_context.block_info.block_number,
        )?;
        Self::validate_proof_block_hash(proof_block_hash, proof_block_number, os_constants, state)?;

        // Validate the config hash.
        let virtual_os_config_hash = block_context.virtual_os_config_hash();
        let proof_config_hash = snos_proof_facts.config_hash;
        if virtual_os_config_hash != proof_config_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS config hash mismatch. Computed virtual OS config hash: \
                 {virtual_os_config_hash}, expected virtual OS config hash: {proof_config_hash}."
            )));
        }

        Ok(())
    }
