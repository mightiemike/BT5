Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address as `sender` instead of the originating user, allowing any address to bypass the per-user swap allowlist on curated pools — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool populates with `msg.sender` of the `pool.swap()` call. When swaps are routed through `MetricOmmSimpleRouter`, `msg.sender` in the pool is the router contract, not the originating user. If a pool admin allowlists the router so that allowlisted users can trade through it, every non-allowlisted user can bypass the restriction by calling the router, completely defeating the allowlist invariant.

## Finding Description

**Step 1 — Pool passes `msg.sender` (the router) as `sender` to the extension.**

In `MetricOmmPool.swap`, the pool calls `_beforeSwap` with `msg.sender` as the first argument:

```solidity
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    ...
);
``` [1](#0-0) 

**Step 2 — The router is `msg.sender` of the pool call.**

`MetricOmmSimpleRouter.exactInputSingle` stores the originating user only in transient callback context (for payment settlement) and calls `pool.swap` directly, making the router the `msg.sender` inside the pool:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
``` [2](#0-1) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [3](#0-2) 

**Step 3 — The allowlist checks the router address, not the originating user.**

```solidity
function beforeSwap(address sender, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
``` [4](#0-3) 

`msg.sender` here is the pool (correct). `sender` is the router address (wrong). The mapping lookup becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Step 4 — The irreconcilable dilemma.**

| Router allowlist state | Effect on allowlisted users | Effect on non-allowlisted users |
|---|---|---|
| Router **not** allowlisted | Cannot use the router at all | Correctly blocked |
| Router **allowlisted** | Can use the router | **Bypass: also allowed through router** |

There is no configuration that simultaneously permits allowlisted users to trade via the router and blocks non-allowlisted users from doing the same.

## Impact Explanation
Any non-allowlisted address can swap on a curated pool by calling `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) instead of calling `pool.swap` directly. If the pool admin has allowlisted the router to support normal user flows, the allowlist is completely defeated. Unauthorized users can execute adverse-selection trades that the allowlist was designed to prevent, constituting a direct loss of LP principal. This is a broken core pool functionality causing loss of funds.

## Likelihood Explanation
The bypass requires no special privilege, no flash loan, and no multi-transaction setup. Any user who knows the pool address can call the public router with a single transaction. The router is a standard, documented periphery contract that users are expected to use. The likelihood is high.

## Recommendation
The `SwapAllowlistExtension` must gate on the originating user's address, not the intermediary caller. The preferred approach is to redesign the hook signature to include an explicit `originator` field that the pool populates from a verified source (e.g., a signed payload or a trusted router registry), so the extension can always check the true economic actor regardless of the call path. Alternatively, the router can encode `msg.sender` into `extensionData` before calling the pool, and the extension decodes and verifies it — though this reintroduces a trust assumption on the router correctly reporting the user.

## Proof of Concept

```
Setup:
  pool = deploy MetricOmmPool with SwapAllowlistExtension as extension1, beforeSwap order = 1
  admin calls swapExtension.setAllowedToSwap(pool, userA, true)
  // Admin also allowlists the router so userA can trade via the router:
  admin calls swapExtension.setAllowedToSwap(pool, address(router), true)

Attack (executed by userB, who is NOT in the allowlist):
  userB calls router.exactInputSingle({
      pool:      pool,
      tokenIn:   token0,
      tokenOut:  token1,
      zeroForOne: true,
      amountIn:  X,
      recipient: userB,
      ...
  })

Trace:
  router.exactInputSingle
    → pool.swap(recipient=userB, ...)   [msg.sender in pool = router]
    → _beforeSwap(sender=router, ...)
    → SwapAllowlistExtension.beforeSwap(sender=router, ...)
    → allowedSwapper[pool][router] == true  ✓  (no revert)
    → swap executes; userB receives token1

Result:
  userB, who is not in the allowlist, successfully swaps on the curated pool.
  The allowlist invariant is broken.
```
