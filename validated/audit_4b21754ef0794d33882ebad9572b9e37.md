### Title
Deprecated `verify_transaction_inclusion` Accepts Internal Merkle Tree Node Hashes as Valid Transaction IDs — (File: `contract/src/lib.rs`)

---

### Summary

The `verify_transaction_inclusion` function accepts any 32-byte hash as a `tx_id` without validating that it is a leaf-level transaction hash rather than an internal Merkle tree node. An unprivileged NEAR caller can supply an internal node hash as `tx_id` with a crafted proof path, causing the function to return `true` for a transaction that does not exist in the Bitcoin block. This is the direct analog of the external report's pattern: a cryptographic operation (here, Merkle proof verification) accepts arbitrary data that should be restricted, enabling forgery.

---

### Finding Description

`verify_transaction_inclusion` (lines 288–323 of `contract/src/lib.rs`) accepts a `ProofArgs` struct containing a caller-supplied `tx_id: H256` and `merkle_proof: Vec<H256>`. It passes these directly to `merkle_tools::compute_root_from_merkle_proof` and compares the result to the block's stored `merkle_root`: [1](#0-0) 

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` performs no validation on whether the starting hash is a leaf or an internal node — it simply iterates the proof path and hashes pairs: [2](#0-1) 

The function's own docstring acknowledges the broken invariant explicitly: [3](#0-2) 

The function is marked `#[deprecated]` but Rust's `deprecated` attribute is a compiler-only lint — it imposes no runtime restriction. The function remains publicly callable by any NEAR account via a signed transaction. The `#[pause]` attribute means it can be disabled by a `PauseManager`, but it is active by default. [4](#0-3) 

The `ProofArgs` struct places no constraint on `tx_id`: [5](#0-4) 

The v2 function (`verify_transaction_inclusion_v2`) was introduced specifically to close this gap by requiring a coinbase proof that anchors the tree depth, but it does so only for its own path — it then delegates to the deprecated v1 function for the target transaction check: [6](#0-5) 

---

### Impact Explanation

Any recipient contract that calls `verify_transaction_inclusion` to gate a cross-chain action (e.g., releasing bridged BTC, minting wrapped tokens, or settling a payment) can be deceived into believing a fraudulent Bitcoin transaction was confirmed in a real block. The attacker does not forge a block header — they exploit the fact that an internal Merkle tree node hash, when used as `tx_id` with a correctly constructed sibling path, produces the correct Merkle root. The on-chain state corrupted is the boolean proof result returned to the consuming contract, which treats it as authoritative confirmation of a Bitcoin event that never occurred.

---

### Likelihood Explanation

The attack requires only:
1. Knowledge of the Merkle tree structure of any real Bitcoin block already accepted by the light client (all data is public on-chain).
2. The ability to call a permissionless NEAR contract function.

No privileged role, leaked key, or social engineering is required. The 64-byte transaction Merkle forgery technique (referenced in the v2 docstring at line 326) is well-documented and tooling exists to construct such proofs. [7](#0-6) 

---

### Recommendation

**Short term:** Remove `verify_transaction_inclusion` from the public ABI entirely (not just deprecate it), or add a `require!(false, "use verify_transaction_inclusion_v2")` guard so it cannot be called at runtime. Deprecation alone does not prevent on-chain invocation.

**Long term:** Ensure all verification paths require the coinbase proof, which bounds the tree depth and prevents an internal node from being presented as a leaf. The coinbase transaction is always at index 0, so its proof path length equals the tree depth, making it impossible to pass an internal node at a shallower depth as a leaf-level `tx_id`.

---

### Proof of Concept

1. Select any Bitcoin block already stored in the light client's `headers_pool` with a known Merkle tree (e.g., a block with 4 transactions: `[T0, T1, T2, T3]`).
2. Compute the internal node `N = dSHA256(T0 || T1)` — this is a real internal node of the tree, not a transaction.
3. Observe that `N` sits at position 0 in the second level of the tree. Its sibling at that level is `M = dSHA256(T2 || T3)`, and `dSHA256(N || M)` equals the block's `merkle_root`.
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node hash, not a real transaction)
   - `tx_index = 0`
   - `merkle_proof = [M]` (one sibling, proving position 0 at depth 1)
   - `tx_block_blockhash` = the real block hash
   - `confirmations = 1`
5. `compute_root_from_merkle_proof(N, 0, [M])` computes `dSHA256(N || M)` = `merkle_root`. The function returns `true`.
6. A consuming contract receives `true` and releases funds for a Bitcoin transaction `N` that does not exist. [8](#0-7)

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
```

```rust
fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
```
