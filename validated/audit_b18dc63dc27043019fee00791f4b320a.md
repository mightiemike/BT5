### Title
`requireValidTxSignatures` Passes With Zero Signatures When `nSigner == 0`, Enabling Unauthorized Fast Withdrawal Drain — (`File: core/contracts/Verifier.sol`)

---

### Summary

`Verifier.requireValidTxSignatures` enforces the signature count with `require(nSignatures == nSigner)`. When `nSigner == 0` — which occurs during initialization or any key-rotation window where all pubkeys have been deleted before new ones are assigned — the check degenerates to `require(0 == 0)`, which always passes. An unprivileged caller can invoke `BaseWithdrawPool.submitFastWithdrawal` with an empty `signatures` array, bypass the only authorization gate, and drain the pool.

---

### Finding Description

`Verifier.requireValidTxSignatures` is the sole authorization check protecting `BaseWithdrawPool.submitFastWithdrawal`. Its terminal guard is:

```solidity
require(nSignatures == nSigner, "not enough signatures");
```

`nSigner` is a storage counter incremented by `_assignPubkey` and decremented by `deletePubkey`. It starts at `0` on deployment and returns to `0` whenever all registered keys are removed. [1](#0-0) 

When `nSigner == 0`, the loop body never executes (or counts zero non-empty entries), `nSignatures` stays `0`, and `0 == 0` is trivially true. No cryptographic material is checked at all.

`submitFastWithdrawal` is `public` with no `msg.sender` restriction. It calls `requireValidTxSignatures` and then immediately transfers tokens to the `sendTo` address decoded from the attacker-supplied `transaction` bytes: [2](#0-1) 

The `sendTo` address is derived entirely from the caller-controlled `transaction` argument: [3](#0-2) 

---

### Impact Explanation

An attacker who calls `submitFastWithdrawal` while `nSigner == 0` can craft a `WithdrawCollateral` or `WithdrawCollateralV2` transaction naming any `sendTo` address and any `amount`, pass the signature check with `signatures = []`, and receive the full token balance of the pool. Every supported collateral token held by the `WithdrawPool` is at risk. The impact is complete, direct theft of user funds.

---

### Likelihood Explanation

`nSigner == 0` arises in two realistic operational scenarios:

1. **Deployment window**: Between contract deployment and the first `assignPubKey` call, `nSigner` is `0`. Any block in this window is exploitable.
2. **Key-rotation window**: The owner calls `deletePubkey` for all current signers before calling `assignPubKey` for the replacement set. During this gap, `nSigner` is `0`. [4](#0-3) 

Key rotation is a standard operational procedure (e.g., signer compromise, hardware refresh). The window may be only one or two transactions wide, but it is observable on-chain and can be front-run by a bot watching for `DeletePubkey` events. [5](#0-4) 

---

### Recommendation

Replace the equality check with a two-part guard that explicitly requires at least one valid signature:

```solidity
require(nSigner > 0, "no signers registered");
require(nSignatures == nSigner, "not enough signatures");
```

Additionally, consider performing key rotation atomically — assign new keys before deleting old ones — to eliminate the zero-signer window entirely.

---

    function requireValidTxSignatures(
        bytes calldata txn,
        uint64 idx,
        bytes[] calldata signatures
    ) public view {
        require(signatures.length <= 256, "too many signatures");
        bytes32 data = keccak256(
            abi.encodePacked(uint256(block.chainid), uint256(idx), txn)
        );
        bytes32 hashedMsg = keccak256(
            abi.encodePacked("\x19Ethereum Signed Message:\n32", data)
        );

        uint256 nSignatures = 0;
        for (uint256 i = 0; i < signatures.length; i++) {
            if (signatures[i].length > 0) {
                nSignatures += 1;
                require(
                    checkIndividualSignature(
                        hashedMsg,
                        signatures[i],
                        uint8(i)
                    ),
                    "invalid signature"
                );
            }
        }
        require(nSignatures == nSigner, "not enough signatures");
    }

        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures

        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;
