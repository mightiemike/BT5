### Title
`burnNlp` Enforces Only Maintenance Health, Allowing Subaccounts to Bypass Initial Health Threshold and Withdraw Value - (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`burnNlp` in `Clearinghouse.sol` checks only `MAINTENANCE` health after burning NLP tokens, while every other value-reducing operation (`withdrawCollateral`, `transferQuote`, `mintNlp`) enforces the stricter `INITIAL` health check. This asymmetry allows a user to receive quote (effectively withdraw value) through `burnNlp` even when their subaccount is below `INITIAL` health, bypassing the protocol's primary safety margin.

---

### Finding Description

The protocol enforces two health tiers:
- **INITIAL**: The primary safety margin. Subaccounts below this cannot withdraw, transfer quote, or mint NLP.
- **MAINTENANCE**: The liquidation threshold. Subaccounts below this can be liquidated.

The gap between INITIAL and MAINTENANCE is the protocol's intended buffer zone — a subaccount in this zone is not yet liquidatable but is considered too risky to allow further value extraction.

**`withdrawCollateral`** enforces INITIAL health:

```solidity
// Clearinghouse.sol:415-419
IProductEngine.HealthType healthType = sender == X_ACCOUNT
    ? IProductEngine.HealthType.PNL
    : IProductEngine.HealthType.INITIAL;
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
```

**`mintNlp`** enforces INITIAL health:

```solidity
// Clearinghouse.sol:479-482
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
```

**`burnNlp`** only enforces MAINTENANCE health:

```solidity
// Clearinghouse.sol:526-529
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
```

The code comment itself acknowledges the risk: *"Burning NLP can decrease health if the burn fee exceeds the health improvement from the withdrawal."* Yet the check used is the weaker MAINTENANCE threshold.

**Concrete attack path:**

1. User deposits collateral and mints NLP — INITIAL health check passes.
2. User opens leveraged positions (via `matchOrders`), driving health below INITIAL but above MAINTENANCE.
3. At this point, `withdrawCollateral` and `transferQuote` would revert with `ERR_SUBACCT_HEALTH`.
4. User submits a `BurnNlp` transaction. `burnNlp` executes, transfers quote back to the user, and only checks MAINTENANCE health — which passes.
5. The user has extracted quote value from the protocol despite being below INITIAL health.

The `burnNlp` path is fully reachable by an unprivileged user: the user signs a `BurnNlp` transaction, the sequencer submits it via `submitTransactions`, `EndpointTx.processTransactionImpl` dispatches it to `clearinghouse.burnNlp`. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

A user can extract quote from the protocol (via `burnNlp`) while their subaccount is in the INITIAL–MAINTENANCE buffer zone — a state the protocol explicitly prohibits for all other value-extraction paths. This:

1. **Bypasses the withdrawal safety margin**: The INITIAL health check exists precisely to ensure a buffer before liquidation. `burnNlp` circumvents it.
2. **Can worsen protocol solvency**: If the burn fee causes health to decrease further (as the code comment acknowledges is possible), the user can deliberately push their subaccount closer to the MAINTENANCE threshold, increasing the risk of bad debt if prices move adversely.
3. **Creates an inconsistent invariant**: The protocol guarantees that no subaccount below INITIAL health can extract value — `burnNlp` breaks this invariant. [4](#0-3) 

---

### Likelihood Explanation

**Medium.** The precondition requires the user to hold NLP tokens and have health in the INITIAL–MAINTENANCE gap. This gap is a normal market condition during periods of price volatility. The `burnNlp` transaction type is a standard user-facing operation submitted through the sequencer fast path, requiring no special privileges. The user only needs to sign a valid `BurnNlp` transaction. [5](#0-4) 

---

### Recommendation

Change the health check in `burnNlp` from `MAINTENANCE` to `INITIAL`:

```solidity
// Clearinghouse.sol ~line 526
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
```

This aligns `burnNlp` with `withdrawCollateral` and `mintNlp`, closing the enforcement gap. If the protocol intends to allow NLP burns as a health-recovery mechanism for subaccounts below INITIAL, a separate, narrowly scoped check should be introduced (e.g., only allow burns that strictly improve health). [6](#0-5) 

---

### Proof of Concept

1. Alice deposits 10,000 USDC and mints NLP worth 5,000 USDC. INITIAL health check passes.
2. Alice opens a leveraged perp position. Her INITIAL health drops to `-100` (below zero), but MAINTENANCE health remains `+200` (above zero).
3. Alice attempts `withdrawCollateral` for 1,000 USDC → reverts at line 419 (`ERR_SUBACCT_HEALTH`).
4. Alice submits a `BurnNlp` transaction for her NLP tokens. `burnNlp` executes:
   - NLP balance decreases, quote balance increases by `quoteAmount - burnFee`.
   - Line 527 checks `getHealth(alice, MAINTENANCE) >= 0` → `200 >= 0` → passes.
5. Alice receives quote despite being below INITIAL health. The INITIAL health check that guards all other withdrawal paths has been bypassed. [7](#0-6) [6](#0-5)

    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
