Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Originating User, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps on the `sender` argument, which the pool sets to `msg.sender` of `MetricOmmPool.swap`. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` inside the pool, so the extension evaluates the router's allowlist status rather than the actual user's. A pool admin who allowlists the router to support router-mediated swaps simultaneously opens the pool to every address on-chain, completely defeating the allowlist's access control.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as `sender` to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the value above: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly — making the router the `msg.sender` inside the pool: [4](#0-3) 

The same applies to `exactInput` (multi-hop), `exactOutputSingle`, and `exactOutput`: [5](#0-4) 

The result: the extension receives `sender = address(router)` and checks `allowedSwapper[pool][router]`. The originating user is never inspected. If the pool admin allowlists the router (the natural step to let allowlisted users trade via the standard periphery), every unpermissioned user on-chain can bypass the allowlist by calling through `MetricOmmSimpleRouter`. Conversely, if only individual user addresses are allowlisted, those users cannot trade through the router at all because the extension sees the router address and reverts.

The existing test `test_allowedSwapSucceeds` masks this bug by allowlisting `address(callers[0])` — a `TestCaller` contract that calls the pool directly — and never routing through `MetricOmmSimpleRouter`: [6](#0-5) 

## Impact Explanation
A curated pool using `SwapAllowlistExtension` is designed to restrict trading to a specific set of addresses (e.g., KYC'd counterparties, institutional LPs, or protocol-controlled addresses). Once the router is allowlisted, any unpermissioned user can execute swaps at oracle-anchored prices, extracting value from LP positions through adverse selection or trading in a pool never intended to be public. LP principal is directly at risk because the pool's spread and stop-loss parameters were calibrated for a controlled counterparty set. This constitutes a direct loss of user principal and a broken core pool access-control invariant.

## Likelihood Explanation
Any pool admin who deploys a curated pool and also wants to support the standard periphery router will allowlist the router address — this is the expected operational pattern, as the router is the primary user-facing entry point. The bypass is triggered by normal, correct-seeming admin configuration, not an exotic edge case. It is repeatable by any on-chain address with no special privileges.

## Recommendation
The pool must pass the originating user's address to the extension, not the intermediary's. Two complementary fixes:

1. **Pool-side (preferred)**: Add an explicit `swapper` parameter to `IMetricOmmPoolActions.swap` that the pool passes through to `_beforeSwap`. Direct callers set it to themselves; the router sets it to `msg.sender` at the router level. The extension then checks `allowedSwapper[pool][swapper]`.

2. **Router-side**: `MetricOmmSimpleRouter` passes the originating user (`msg.sender` at the router level) as a verified field inside `extensionData`, and the extension decodes and verifies it. This is weaker unless the pool enforces the field's integrity.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls swapExtension.setAllowedToSwap(pool, address(router), true)
    (to allow allowlisted users to trade via the router)
  - Pool admin calls swapExtension.setAllowedToSwap(pool, alice, true)

Attack:
  - Bob (not allowlisted) calls:
      router.exactInputSingle({pool: pool, recipient: bob, ...})
  - Router calls pool.swap(bob, ...) — msg.sender inside pool = router
  - Pool calls _beforeSwap(router, bob, ...)
  - SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  - Bob's swap executes successfully despite not being on the allowlist

Result:
  - Bob trades at oracle-anchored prices in a pool intended only for allowlisted counterparties
  - LP funds are exposed to uncontrolled adverse selection
```

A Foundry test reproducing this should: (1) deploy a pool with `SwapAllowlistExtension`, (2) allowlist `address(router)` but not Bob, (3) call `router.exactInputSingle` from Bob's address, and (4) assert the swap succeeds — confirming the bypass.
