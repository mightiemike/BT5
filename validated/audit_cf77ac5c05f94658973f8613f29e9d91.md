### Title
`gas_consumed` Always Zeroed in Per-Call `ExecutionResources` Returned by `starknet_simulateTransactions` — (`crates/apollo_rpc_execution/src/objects.rs`)

### Summary

`vm_resources_to_execution_resources` unconditionally sets `gas_consumed` to `StarknetApiGasVector::default()` (all zeros) in every `FunctionInvocation.execution_resources` object returned by the RPC simulation/trace endpoints.

### Finding Description

In `vm_resources_to_execution_resources`, the function receives a `GasVector` argument (destructured as `{ l1_gas, l1_data_gas, l2_gas }`) but only uses those values to populate `da_gas_consumed`. The `gas_consumed` field is hardcoded to the default (zero):

```rust
Ok(ExecutionResources {
    steps: u64_from_usize(vm_resources.n_steps),
    builtin_instance_counter,
    memory_holes: u64_from_usize(vm_resources.n_memory_holes),
    da_gas_consumed: StarknetApiGasVector { l1_gas, l2_gas, l1_data_gas },
    gas_consumed: StarknetApiGasVector::default(),   // ← always zero
})
``` [1](#0-0) 

This function is called for every `FunctionInvocation` built from a `CallInfo` during simulation/tracing: [2](#0-1) 

The `ExecutionResources` struct has two distinct gas fields: [3](#0-2) 

`gas_consumed` is intended to carry the sierra/L2 execution gas for the call. It is always returned as zero regardless of actual sierra gas consumed.

### Impact Explanation

Any unprivileged caller of `starknet_simulateTransactions` or `starknet_traceTransaction` receives a trace where every `FunctionInvocation.execution_resources.gas_consumed` is `{l1_gas: 0, l1_data_gas: 0, l2_gas: 0}`, even for Sierra contracts that consumed nonzero L2 gas. This is an authoritative-looking wrong value in the RPC response.

**Important scope limitation:** The top-level `FeeEstimation` returned by `starknet_estimateFee` is computed separately and is **not** affected. It reads directly from `receipt.gas`: [4](#0-3) 

So the claim that this "causes clients to underestimate fees" is **overstated**. The fee estimation itself is correct. The wrong value is confined to the per-call trace breakdown (`FunctionInvocation.execution_resources.gas_consumed`), not the top-level fee output.

### Likelihood Explanation

The path is fully unprivileged — any user can call `starknet_simulateTransactions`. The divergent value is concrete and deterministic: `gas_consumed` is always `GasVector::ZERO` in every trace response, regardless of actual sierra gas consumed.

### Recommendation

In `vm_resources_to_execution_resources`, populate `gas_consumed` from the actual per-call sierra gas rather than defaulting to zero. The call site at line 352 passes `gas_vector` (which is `receipt.da_gas`); the correct sierra gas would need to come from `call_info.resources` (e.g., a sierra gas field if tracked per call), not from the DA gas vector.

### Proof of Concept

1. Submit a `starknet_simulateTransactions` call with a Sierra contract that consumes nonzero L2/sierra gas.
2. Extract `result[0].transaction_trace.execute_invocation.execution_resources.gas_consumed` from the response.
3. Assert all fields are zero — this holds unconditionally due to line 399.

---

**Verdict:** Valid finding. The exact divergent value is concrete (`gas_consumed = GasVector::ZERO` always), the path is unprivileged, and the RPC trace response returns an authoritative-looking wrong value. However, the fee estimation impact is narrower than claimed — only the per-call trace breakdown is wrong, not the top-level `FeeEstimation`.
