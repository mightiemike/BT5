### Title
Cross-Chain Merkle Proof Replay Allows Duplicate Claims on Every Chain — (`core/contracts/Airdrop.sol`)

### Summary
The leaf hash in `_verifyProof` commits only to `(sender, totalAmount)` with no chain ID or contract address. If the same merkle root is registered on multiple EVM deployments of `Airdrop`, a proof generated on chain A is cryptographically identical and valid on chain B, allowing an attacker to drain their allocation from every chain.

### Finding Description

In `_verifyProof`, the leaf is constructed as:

```solidity
bytes32 leaf = keccak256(
    bytes.concat(keccak256(abi.encode(sender, totalAmount)))
);
``` [1](#0-0) 

No `block.chainid`, no contract address, and no chain-specific nonce is included. The merkle root is registered per-week by the owner:

```solidity
merkleRoots[week] = merkleRoot;
``` [2](#0-1) 

The `claimed` mapping that prevents double-claiming is local to each contract instance:

```solidity
mapping(uint32 => mapping(address => uint256)) internal claimed;
``` [3](#0-2) 

Because `claimed[week][sender]` on chain B is independent of chain A, and the leaf hash is identical across chains, a proof that passes `MerkleProof.verify` on chain A passes identically on chain B.

### Impact Explanation

If the protocol deploys `Airdrop` on N EVM chains and registers the same weekly merkle root on each (the natural multi-chain airdrop pattern), an attacker submits their single valid proof to all N instances. Each instance independently sets `claimed[week][attacker] = totalAmount` and transfers `totalAmount` tokens. The attacker receives `N × totalAmount` instead of `totalAmount`, draining the airdrop treasury on every chain beyond the first.

### Likelihood Explanation

The precondition — same root on multiple chains — is the standard operational pattern for a multi-chain airdrop. The `registerMerkleRoot` function has no chain-binding logic, so the owner would naturally register the same computed root everywhere. No special privileges are needed by the attacker; `claim()` is a public function callable by any address. [4](#0-3) 

### Recommendation

Bind the leaf to the chain and contract at construction time:

```solidity
bytes32 leaf = keccak256(
    bytes.concat(
        keccak256(abi.encode(block.chainid, address(this), sender, totalAmount))
    )
);
```

This makes every leaf chain- and contract-specific, so a proof from chain A is invalid on chain B even when the merkle root is identical.

### Proof of Concept

1. Deploy two `Airdrop` instances (different chains / different Hardhat network IDs) with the same `token` and `merkleRoot` for week 1.
2. Call `registerMerkleRoot(1, root)` on both instances.
3. Generate a valid Merkle proof for `(attacker, 1000e18)` against `root`.
4. Call `instance_A.claim([{week:1, totalAmount:1000e18, proof}])` → succeeds, transfers 1000e18.
5. Call `instance_B.claim([{week:1, totalAmount:1000e18, proof}])` → succeeds, transfers another 1000e18.
6. Assert attacker balance = 2000e18 and both `claimed[1][attacker]` mappings = 1000e18.

The exploit is locally reproducible on an unmodified Hardhat setup by forking two network IDs with the same contract bytecode and root. [5](#0-4)

    function _verifyProof(
        uint32 week,
        address sender,
        uint256 totalAmount,
        bytes32[] calldata proof
    ) internal {
        require(claimed[week][sender] == 0, "Already claimed.");
        require(
            merkleRoots[week] != bytes32(0),
            "Week hasn't been registered."
        );
        require(
            !ISanctionsList(sanctions).isSanctioned(sender),
            "address is sanctioned."
        );
        bytes32 leaf = keccak256(
            bytes.concat(keccak256(abi.encode(sender, totalAmount)))
        );
        bool isValidLeaf = MerkleProof.verify(proof, merkleRoots[week], leaf);
        require(isValidLeaf, "Invalid proof.");
        claimed[week][sender] = totalAmount;
    }
```
