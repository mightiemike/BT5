### Title
NFT Denom Registration Front-Running Allows Attacker to Seize Creator Rights and Mint Arbitrary NFTs - (`x/nft/keeper/denom.go`, `x/nft/keeper/keeper.go`)

### Summary

Any unprivileged account can front-run a legitimate `MsgIssueDenom` transaction by submitting an identical denom ID with their own address as `sender`. Because `SetDenom` is strictly first-come-first-served with no reservation or ownership proof, the attacker becomes the permanent `Creator` of the denom. All subsequent `MsgMintNFT` calls from the victim fail the `IsDenomCreator` guard, while the attacker retains exclusive minting rights over the collection.

### Finding Description

**Entry point:** `MsgIssueDenom` → `msgServer.IssueDenom` → `keeper.IssueDenom` → `keeper.SetDenom`

`SetDenom` stores the denom with the caller's address as `Creator` using a simple existence check: [1](#0-0) 

There is no reservation, commit-reveal, or signer-binding mechanism. Any account that submits `MsgIssueDenom{id=X}` before the intended creator wins permanent ownership of denom `X`.

`MintNFT` then enforces an exclusive creator check: [2](#0-1) 

which calls: [3](#0-2) 

Once the attacker's `MsgIssueDenom` is committed, the victim's `MsgMintNFT{sender=victim, denom_id=X}` will always return `ErrUnauthorized`. The attacker can freely call `MintNFT` with `recipient` set to any address.

**Attack path (concrete):**

1. Victim broadcasts `MsgIssueDenom{id="mycollection", sender=victim}` (visible in mempool).
2. Attacker submits `MsgIssueDenom{id="mycollection", sender=attacker}` with a higher gas price.
3. Attacker's tx is ordered first in the block; `SetDenom` stores `attacker` as `denom.Creator`.
4. Victim's tx executes next; `SetDenom` returns `"denomID mycollection has already exists"` — victim's tx fails.
5. Attacker calls `MsgMintNFT{sender=attacker, denom_id="mycollection", recipient=<any>}` — succeeds unconditionally.

No privileged access, governance, or key compromise is required. The attacker only needs to pay a marginally higher fee.

### Impact Explanation

The attacker becomes the sole authorized minter of the targeted denom. They can:
- Mint NFTs to arbitrary recipients and sell them as the legitimate collection.
- Permanently block the victim from ever minting under their intended denom ID.
- Impersonate a marketed NFT collection, deceiving buyers and capturing its economic value.

The victim's gas fees are lost and their denom ID is permanently squatted. The attacker gains exclusive control of a collection's minting rights — a direct theft of economically valuable on-chain capability.

### Likelihood Explanation

Mempool transactions are publicly observable on Cosmos SDK chains. Front-running by fee-bumping is a well-known and mechanically straightforward attack. No special tooling beyond a standard Cosmos SDK client is needed. Any actor monitoring the mempool can execute this reliably.

### Recommendation

- **Bind denom ID to signer at `ValidateBasic` or ante-handler level** — e.g., require `denom_id` to be derived from (or include) the sender's address, making squatting impossible.
- Alternatively, implement a **commit-reveal** or **reservation** scheme where a denom ID can only be finalized by the address that reserved it.
- At minimum, emit a warning in documentation that denom IDs should be chosen to be unguessable (e.g., include a hash of the creator address), though this is not a protocol-level fix.

```go
func TestDenomFrontRun(t *testing.T) {
    // Setup: two accounts — victim, attacker
    // Block N: attacker submits MsgIssueDenom{id="mycollection", sender=attacker}
    //          victim  submits MsgIssueDenom{id="mycollection", sender=victim}
    //          attacker tx ordered first (higher gas price)

    // Assert: SetDenom stores attacker as Creator
    denom, _ := keeper.GetDenom(ctx, "mycollection")
    require.Equal(t, attacker.String(), denom.Creator)

    // Assert: victim's IssueDenom failed (denomID already exists)

    // Assert: attacker can mint
    err := keeper.MintNFT(ctx, "mycollection", "token1", "", "", "", attacker, recipient)
    require.NoError(t, err)

    // Assert: victim cannot mint
    err = keeper.MintNFT(ctx, "mycollection", "token2", "", "", "", victim, recipient)
    require.ErrorContains(t, err, "is not the creator of")
}
```

The root cause is in `SetDenom` at [1](#0-0)  — it unconditionally stores the first caller as creator with no binding to any pre-committed identity — and in `MintNFT` at [4](#0-3)  which enforces that stored creator as the exclusive minter forever.

```go
func (k Keeper) SetDenom(ctx sdk.Context, denom types.Denom) error {
	if k.HasDenomID(ctx, denom.Id) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denomID %s has already exists", denom.Id)
	}

	if k.HasDenomNm(ctx, denom.Name) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denomName %s has already exists", denom.Name)
	}

	store := ctx.KVStore(k.storeKey)
	bz := k.cdc.MustMarshal(&denom)
	store.Set(types.KeyDenomID(denom.Id), bz)
	store.Set(types.KeyDenomName(denom.Name), []byte(denom.Id))
	return nil
```

```go
func (k Keeper) IsDenomCreator(ctx sdk.Context, denomID string, address sdk.AccAddress) (types.Denom, error) {
	denom, err := k.GetDenom(ctx, denomID)
	if err != nil {
		return types.Denom{}, err
	}

	creator, err := sdk.AccAddressFromBech32(denom.Creator)
	if err != nil {
		panic(err)
	}

	if !creator.Equals(address) {
		return types.Denom{}, sdkerrors.Wrapf(types.ErrUnauthorized, "%s is not the creator of %s", address, denomID)
	}

	return denom, nil
```
