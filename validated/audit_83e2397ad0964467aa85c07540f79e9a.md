### Title
Partial Position Reduction Can Leave Dust Positions Below `minSize`, Causing Distorted Fee Accounting - (`core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.matchOrders` enforces `sizeIncrement` alignment on every fill but never validates that the **remaining** position size after a partial reduce-only fill stays at or above `minSize`. A trader can sign a reduce-only order that leaves a dust position below `minSize`, which the on-chain contract will execute without any guard. The `applyFee` function uses `minSize` as a fee-metering threshold, so closing a dust position charges fees calculated against `minSize` rather than the actual (smaller) trade value, potentially making fees exceed the trade's worth and rendering the position economically unliquidatable.

---

### Finding Description

`OffchainExchange.sol` stores a per-market `minSize` parameter: [1](#0-0) 

It is exposed via `getMinSize`: [2](#0-1) 

Inside `matchOrders`, the only size-related invariant enforced is `sizeIncrement` alignment: [3](#0-2) 

There is **no corresponding check** that the resulting position size after the fill is `>= minSize`. The `_validateOrder` function, which is the sole per-order gate, checks version, signature, expiration, and reduce-only clamping — but never `minSize`: [4](#0-3) 

After balances are updated and health is checked, `matchOrders` returns without ever asserting the post-fill position size: [5](#0-4) 

Meanwhile, `applyFee` uses `minSize` as a fee-metering floor for taker orders: [6](#0-5) 

When `alreadyMatched == 0`, `meteredQuote` is initialised to `±market.minSize`. Then `feeApplied = abs(alreadyMatched + matchQuote) - minSize`. If the dust trade's `matchQuote < minSize`, `feeApplied` is negative and the `if (feeApplied > 0)` branch is skipped, so the entire fee is computed against `minSize` — not the actual dust trade size. The user is charged a fee proportional to `minSize` for a trade worth less than `minSize`.

---

### Impact Explanation

1. **Distorted fee accounting**: Closing a dust perp position charges fees as if the trade were `minSize`, which can exceed the actual trade value. This corrupts the protocol's fee accounting and can drain the dust position's collateral entirely.
2. **Unprofitable / impossible liquidations**: A dust perp position below `minSize` generates liquidation fees proportional to its tiny notional value. Liquidators have no economic incentive to liquidate it, leaving bad debt in the system.
3. **Accounting corruption**: Dust positions accumulate `vQuoteBalance` entries in `PerpEngine` that are too small to settle cleanly, causing persistent rounding drift in the engine's state.

---

### Likelihood Explanation

The sequencer is expected to enforce `minSize` off-chain, but the on-chain contract provides no backstop. A trader can sign a reduce-only order with `amount = -(currentPosition - 1)` (leaving 1 unit of dust). If the sequencer has a bug, is permissive, or is operated by a party with an incentive to create dust positions, the on-chain `matchOrders` will execute the fill unconditionally. Because the signed order is valid (correct signature, non-zero amount, not expired, reduce-only clamped correctly), no on-chain check rejects it.

---

### Recommendation

In `matchOrders`, after computing `ordersInfo.taker.amountDelta` and before updating balances, check that the post-fill position size for each non-X_ACCOUNT party is either zero (full close) or `>= market.minSize`:

```solidity
// After computing amountDelta, before _updateBalances:
int128 takerNewAmount = callState.isPerp
    ? callState.perp.getBalance(callState.productId, taker.order.sender).amount
        + ordersInfo.taker.amountDelta
    : callState.spot.getBalance(callState.productId, taker.order.sender).amount
        + ordersInfo.taker.amountDelta;

require(
    takerNewAmount == 0 ||
    MathHelper.abs(takerNewAmount) >= market.minSize,
    ERR_BELOW_MIN_SIZE
);
// mirror check for maker
```

Apply the same guard symmetrically for the maker side.

---

### Proof of Concept

1. Market is configured with `minSize = 1e18` (1 unit).
2. Trader opens a perp position: signs an order with `amount = 2e18`. Sequencer submits `matchOrders`. Position is now `2e18`. No `minSize` check exists in `_validateOrder` or `matchOrders`.
3. Trader signs a reduce-only order with `amount = -(1e18 + 1)` (reduces position to `1e18 - 1 = dust`). `_isReduceOnly` clamps it to `min(order.amount, -currentBalance) = -(1e18 + 1)` clamped to `-(1e18 + 1)` since position is `2e18`. [7](#0-6) 

4. Sequencer submits `matchOrders`. `amountDelta` is aligned to `sizeIncrement` but never checked against `minSize`. Balances are updated. Trader's position is now `2e18 - (1e18 + 1) = 1e18 - 1`, which is below `minSize`.
5. When this dust position is later closed, `applyFee` computes `meteredQuote = minSize = 1e18` for a trade worth `< 1e18`, charging fees larger than the trade value. [8](#0-7)

        }

            return true;
        }

                return false;

                order.amount = 0;
