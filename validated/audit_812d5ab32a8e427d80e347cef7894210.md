### Title
Host-function gas-key fee uses wire-length instead of trie-id-length for ML-DSA-65 keys, causing gas exhaustion - (File: runtime/near-vm-runner/src/logic/logic.rs)

### Summary

The host-function implementations of `promise_batch_action_add_gas_key_with_full_access`, `promise_batch_action_add_gas_key_with_function_call`, and `promise_batch_action_transfer_to_gas_key` pass `public_key_len as usize` (the raw borsh wire length of the key as supplied by the contract) into `gas_key_add_key_exec_fee` / `gas_key_transfer_exec_fee`. For ML-DSA-65 keys this value is **1953** (1 tag + 1952 raw bytes), whereas the actual on-trie identifier is only **33** bytes (1 tag + 32-byte SHA3-256 hash). The transaction-execution path correctly calls `action.public_key.trie_id_len()` (= 33). The 1920-byte divergence is multiplied by `gas_key_byte` exec fee and by `num_nonces`, producing a per-call overcharge of hundreds of Tgas — enough to guarantee gas exhaustion for any contract that tries to add or fund an ML-DSA-65 gas key via a host function.

### Finding Description

`PublicKey::len()` returns the borsh-encoded wire size (1 + 1952 = 1953 for ML-DSA-65), while `PublicKey::trie_id_len()` returns the on-trie identifier size (1 + 32 = 33 for ML-DSA-65). The design document explicitly states: *"Every storage-stake and trie-byte-priced fee path was updated to call `trie_id_len()`."*

The transaction-execution path in `runtime/runtime/src/config.rs` honours this contract:

```rust
TransferToGasKey(action) => {
    gas_key_transfer_exec_fee(fees, receiver_id.len(), action.public_key.trie_id_len())
        .total()
}
WithdrawFromGasKey(action) => {
    gas_key_transfer_exec_fee(fees, receiver_id.len(), action.public_key.trie_id_len())
        .total()
}
``` [1](#0-0) 

But the host-function path does not. In `logic.rs`, `public_key_len` is the raw byte count passed by the Wasm contract — the borsh wire length — and it is forwarded directly into the fee helper:

```rust
// promise_batch_action_transfer_to_gas_key
let send = gas_key_transfer_send_fee(&self.fees_config, sir, public_key_len as usize);
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,   // ← wire length, not trie_id_len
);
``` [2](#0-1) 

```rust
// promise_batch_action_add_gas_key_with_full_access
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,   // ← wire length, not trie_id_len
    num_nonces,
);
``` [3](#0-2) 

The same pattern appears in `promise_batch_action_add_gas_key_with_function_call` and in the wasmtime runner mirror: [4](#0-3) [5](#0-4) [6](#0-5) 

`gas_key_add_key_exec_fee` uses `public_key_len` directly to compute the trie-key length:

```rust
let nonce_key_len =
    access_key_key_len(account_id_len, public_key_len) + size_of::<NonceIndex>();
let per_byte = cfg.fee(ActionCosts::gas_key_byte).exec_fee()
    .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64).unwrap()
    .checked_mul(num_nonces).unwrap();
``` [7](#0-6) 

For an ML-DSA-65 key the trie-key length is inflated by 1920 bytes per nonce slot.

The existing fee-parity test (`test_gas_key_fee_parity`) only exercises `KeyType::ED25519`, for which `len()` and `trie_id_len()` are identical (both 33), so the divergence is invisible in the test suite: [8](#0-7) 

### Impact Explanation

**Impact: High**

With `gas_key_byte` exec fee of 101,435,400 gas/byte and a modest 4-nonce gas key, the overcharge is approximately:

```
1920 bytes × 101,435,400 gas/byte × 4 nonces ≈ 779 Tgas
```

The maximum gas per receipt is ~300 Tgas. Any contract that calls `promise_batch_action_add_gas_key_with_full_access` or `promise_batch_action_transfer_to_gas_key` with an ML-DSA-65 public key will always exhaust gas and fail. The operation is completely non-functional via the host-function path for ML-DSA-65 keys, while the identical operation submitted as a transaction succeeds normally. This breaks the fee-parity invariant that the protocol explicitly guarantees and renders the host-function gas-key API dead for post-quantum keys.

### Likelihood Explanation

**Likelihood: Medium**

Both `ProtocolFeature::GasKeys` and `ProtocolFeature::PostQuantumSignatures` must be active. `PostQuantumSignatures` is stabilised at protocol version 85; `GasKeys` is behind the `spice` feature flag. Once both are live, any contract author who attempts to manage ML-DSA-65 gas keys programmatically will trigger the bug deterministically — no special attacker capability is required beyond submitting a normal transaction that calls the host function.

### Recommendation

After decoding the public key, derive the fee length from the decoded key's `trie_id_len()` rather than from the raw `public_key_len` parameter. For example, in `promise_batch_action_add_gas_key_with_full_access`:

```rust
let public_key = self.get_public_key(public_key_ptr, public_key_len)?;
let decoded_pk = public_key.decode()?;
// ...
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    decoded_pk.trie_id_len(),   // ← correct: on-trie length
    num_nonces,
);
// pass decoded_pk to append_action_* as well
```

Apply the same fix to `promise_batch_action_add_gas_key_with_function_call` and `promise_batch_action_transfer_to_gas_key` in both `logic/logic.rs` and `wasmtime_runner/logic.rs`. Add a fee-parity test that exercises `KeyType::MLDSA65` alongside the existing `ED25519` case in `test_gas_key_fee_parity`.

### Proof of Concept

1. Deploy a contract that calls `promise_batch_action_add_gas_key_with_full_access` with a borsh-encoded `PublicKey::MLDSA65` (1953 bytes) and `num_nonces = 1`.
2. The host function receives `public_key_len = 1953`.
3. `gas_key_add_key_exec_fee` computes `nonce_key_len = access_key_key_len(account_id_len, 1953) + 2 ≈ 1957`.
4. `per_byte = 101,435,400 × (1957 + NONCE_VALUE_LEN) × 1 ≈ 198 Tgas`.
5. The correct fee (using `trie_id_len = 33`) would be `101,435,400 × (37 + NONCE_VALUE_LEN) × 1 ≈ 4 Tgas`.
6. The receipt runs out of gas and the action fails with `GasExceeded`.
7. Submitting the identical `AddKey` action as a regular transaction (which routes through `runtime/runtime/src/config.rs` using `trie_id_len()`) succeeds and charges ~4 Tgas — confirming the divergence. [9](#0-8) [10](#0-9)

pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}

pub fn gas_key_add_key_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
    num_nonces: NonceIndex,
) -> GasKeyAddFee {
    let num_nonces = num_nonces as u64;
    let base =
        cfg.fee(ActionCosts::gas_key_nonce_write_base).exec_fee().checked_mul(num_nonces).unwrap();
    let nonce_key_len =
        access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64)
        .unwrap()
        .checked_mul(num_nonces)
        .unwrap();
    GasKeyAddFee { base, per_byte }
