### Title
`Role::UnrestrictedSubmitBlocks` Has No Mechanism to Bypass the Pause Gate on `submit_blocks` — (`File: contract/src/lib.rs`)

---

### Summary

`Role::UnrestrictedSubmitBlocks` is explicitly documented as "Allows to use contract API even after contract is paused," but `submit_blocks` is decorated with a bare `#[pause]` attribute that carries no role exception. The role is defined, can be granted, and is referenced in the `trusted_relayer` bypass list — yet it has zero effect on the pause enforcement for `submit_blocks`. The sister role `UnrestrictedRunGC` is correctly wired with `#[pause(except(roles(Role::UnrestrictedRunGC)))]`, proving the pattern is known and intentionally applied elsewhere but omitted here.

---

### Finding Description

`contract/src/lib.rs` defines the `Role` enum with two pause-bypass roles:

- `UnrestrictedSubmitBlocks` — "Allows to use contract API even after contract is paused" [1](#0-0) 
- `UnrestrictedRunGC` — "Allows to use `run_mainchain_gc` API on a paused contract" [2](#0-1) 

`run_mainchain_gc` correctly wires its bypass:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
``` [3](#0-2) 

`submit_blocks` does not:

```rust
#[payable]
#[pause]           // ← bare pause, no except(roles(...))
#[trusted_relayer]
pub fn submit_blocks(
``` [4](#0-3) 

The `#[trusted_relayer]` macro's `bypass_roles(Role::DAO, Role::UnrestrictedSubmitBlocks)` is a separate enforcement layer that only bypasses the trusted-relayer staking requirement — it does not interact with the `near-plugins` `#[pause]` gate. [5](#0-4) 

The `#[pausable(manager_roles(Role::PauseManager))]` attribute on the struct confirms the pause system is active and enforced at runtime. [6](#0-5) 

The result: `Role::UnrestrictedSubmitBlocks` is a dead role. Granting it to any account produces no observable difference in behavior when the contract is paused.

---

### Impact Explanation

When a `PauseManager` pauses the contract (a legitimate operational action, e.g., during an emergency or upgrade), every call to `submit_blocks` is rejected — including calls from accounts holding `UnrestrictedSubmitBlocks` or `DAO` roles. No new Bitcoin block headers can be ingested. The light client's canonical chain tip becomes stale. Downstream NEAR contracts that call `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` (both also `#[pause]`-gated) will be unable to verify Bitcoin transactions during the pause window, breaking any protocol that depends on SPV proofs from this contract. [7](#0-6) 

The intended operational invariant — that privileged relayers can keep the light client live even during a pause — is silently violated with no on-chain error indicating the bypass is non-functional.

---

### Likelihood Explanation

Medium. The contract is actively deployed and the pause mechanism is a documented operational control. Any pause event (emergency response, upgrade staging) immediately activates the broken invariant. The `UnrestrictedSubmitBlocks` role will have been granted to relayer accounts precisely because operators believe it provides the documented bypass — making the silent failure likely to go undetected until a pause is actually exercised.

---

### Recommendation

Change the `submit_blocks` pause attribute to mirror the pattern used by `run_mainchain_gc`:

```rust
#[pause(except(roles(Role::UnrestrictedSubmitBlocks)))]
```

This wires the role into the `near-plugins` pause gate so that accounts holding `UnrestrictedSubmitBlocks` can continue submitting headers while the contract is paused, matching the documented intent of the role.

---

### Proof of Concept

1. Deploy the contract and grant `Role::PauseManager` to account `pause_admin` and `Role::UnrestrictedSubmitBlocks` to account `privileged_relayer`.
2. Call `pa_pause("submit_blocks")` from `pause_admin`.
3. Attempt to call `submit_blocks` from `privileged_relayer` with a valid Bitcoin header batch.
4. Observe: the call panics with the `near-plugins` pause error, despite `privileged_relayer` holding the role explicitly documented to bypass this restriction.
5. Compare: call `run_mainchain_gc` from an account holding `Role::UnrestrictedRunGC` — it succeeds, confirming the `except(roles(...))` pattern works correctly when applied. [3](#0-2)
