### Title
Merkle Second-Preimage Attack Allows Forged Transaction Inclusion Proofs — (`contract/src/lib.rs`, `merkle-tools/`)

### Summary

`verify_transaction_inclusion` accepts any internal merkle-tree node hash as a valid `tx_id` because it never validates proof depth against the actual transaction count. An unprivileged NEAR caller can construct a proof that a non-existent transaction was included in any real Bitcoin block already stored in the contract, causing the function to return `true` for a fabricated `tx_id`.

### Finding Description

The `verify_transaction_inclusion` public entry point performs SPV verification by recomputing the merkle root from the supplied `tx_id` and proof path and comparing it to the stored block header's `merkle_root` field. [1](#0-0) 

Bitcoin block headers carry only the merkle root — not the transaction count or tree depth. Without knowing the number of leaves, the contract cannot distinguish a leaf node (real transaction hash) from an internal node (hash of two child hashes). Any internal node at depth D in the tree has a valid sibling-hash path of length D that recomputes to the root. An attacker who knows the merkle tree of any stored block can pick any internal node hash H, supply the D siblings above it, and the verifier will compute the correct root and return `true`, even though H is not a transaction.

This is the direct analog of the external report's bypass pattern: the required invariant (merkle root equality) is satisfied, but the prerequisite check (that the proven leaf is actually a transaction, not an intermediate node) is silently skipped — exactly as `adjustmentsMade == adjustmentsRequired` was satisfied in the Vault contract without the actual loss-inducing adjustments being performed. [2](#0-1) 

### Impact Explanation

Any NEAR smart contract that calls `verify_transaction_inclusion` and acts on its boolean result (e.g., releasing funds, minting tokens, updating state) can be deceived into processing a payment or state change for a Bitcoin transaction that never existed. The forged proof is constructed entirely from public on-chain data (the stored block's merkle tree) and requires no mining, no privileged access, and no key material.

### Likelihood Explanation

The attack requires only:
1. A real Bitcoin block already stored in the contract (always true in production).
2. Knowledge of that block's merkle tree (fully public Bitcoin data).
3. A single NEAR transaction calling `verify_transaction_inclusion` with the crafted arguments.

No special role, deposit beyond gas, or social engineering is needed. Any unprivileged NEAR account can execute this.

### Recommendation

The contract cannot fix this alone because block headers do not encode transaction count. The fix must be applied at the API boundary:

- **Document and enforce at the call site**: `verify_transaction_inclusion` must be treated as a necessary-but-not-sufficient check. Callers must independently verify that `tx_id` is a real transaction (e.g., by verifying the raw serialized transaction against the hash before calling the contract).
- **Optionally accept `tx_count` as a parameter**: If callers can supply the number of transactions in the block (verifiable off-chain), the contract can reject proofs whose path length does not equal `ceil(log2(tx_count))`, closing the depth ambiguity.
- **Emit a clear revert/error** rather than returning `true` when the proof path length is inconsistent with any plausible tree depth, as a defense-in-depth measure. [3](#0-2) 

### Proof of Concept

1. Identify any Bitcoin block B already stored in the contract (e.g., the genesis block submitted during `init`).
2. Obtain B's full transaction list (public Bitcoin RPC data). Suppose B has 4 transactions T0–T3. The merkle tree has internal nodes:
   - `I01 = SHA256d(T0 || T1)`
   - `I23 = SHA256d(T2 || T3)`
   - `root = SHA256d(I01 || I23)`
3. Construct a forged proof claiming `I01` is a "transaction":
   - `tx_id = I01`
   - proof path = `[I23]` (one sibling, depth = 1)
4. Call `verify_transaction_inclusion({ tx_id: I01, block_hash: B, proof: [I23], ... })`.
5. The contract computes `SHA256d(I01 || I23) == root` → returns `true`.
6. `I01` is not a transaction hash; it is an internal node. No such transaction exists on Bitcoin.

Any downstream NEAR contract that gates value transfer on this `true` result is now exploitable by any unprivileged caller. [4](#0-3)
