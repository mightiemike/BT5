### Title
Difficulty Retarget Uses Main-Chain Ancestor Instead of Fork Ancestor for Fork Blocks - (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

### Summary
When validating a fork block that crosses a difficulty retarget boundary, `get_next_work_required` fetches the retarget-interval boundary block via `get_header_by_height`, which always reads from `mainchain_height_to_header`. For fork blocks, the correct ancestor at that height is the fork's own ancestor, not the main chain's block. This is a cross-module desynchronization: the stored main-chain mapping diverges from the fork's actual chain state, causing the difficulty check to use the wrong block's timestamp and producing an incorrect expected `bits` value.

The code itself acknowledges this in a TODO comment in `dogecoin.rs`:
> `// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor`

### Finding Description

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

It unconditionally reads from `mainchain_height_to_header`, which maps heights to main-chain block hashes. This is correct for main-chain block submissions, but wrong for fork submissions.

In `dogecoin.rs` (post-height 145,000, where `difficulty_adjustment_interval = 1`), every block triggers a retarget. The retarget calculation fetches the boundary block:

```rust
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [2](#0-1) 

When the submitted block is a fork block at height `H+2` (fork diverged at `H`), `height_first = H+1 - 1 = H`. The contract fetches the **main chain's** block at height `H`, but the fork's block at height `H+1` (the previous fork block) is the correct ancestor to use as the retarget boundary. The main chain's block at `H+1` has a different timestamp than the fork's block at `H+1`, so the computed timespan `prev_block_header.time - first_block_time` is wrong.

The same pattern exists in `bitcoin.rs`:

```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
``` [3](#0-2) 

And in `litecoin.rs`:

```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
``` [4](#0-3) 

### Impact Explanation

The incorrect `first_block_time` produces a wrong `actual_timespan`, which feeds into `calculate_next_work_required` and yields a wrong `expected_bits`. The contract then enforces:

```rust
require!(
    expected_bits == block_header.bits,
    ...
);
``` [5](#0-4) 

Two concrete consequences:

1. **Invalid fork blocks accepted**: An attacker submits a fork block at `H+1` with a timestamp crafted so that `fork_H+1_time - mainchain_H_time` produces a favorable (lower) expected difficulty for `H+2`. The attacker then submits `H+2` with that lower difficulty, which passes the `expected_bits == block_header.bits` check even though it would fail against the correct fork-ancestor-based difficulty. For Dogecoin post-145,000, this applies to every fork block after the first.

2. **Valid fork blocks rejected**: A legitimate relayer submitting a valid fork block whose `bits` matches the correct fork-ancestor-based difficulty will be rejected because the contract computes a different `expected_bits` using the wrong main-chain ancestor.

For Dogecoin (post-145,000), the Digishield formula clamps the timespan to `[retarget_timespan - retarget_timespan/4, retarget_timespan + retarget_timespan/2]` = `[45s, 90s]`, so the maximum difficulty manipulation per block is a 1.5× reduction. Over multiple fork blocks, this compounds. [6](#0-5) 

### Likelihood Explanation

Any unprivileged NEAR caller can invoke `submit_blocks` with fork headers. Fork submissions are a normal, supported protocol flow — the contract explicitly handles them in `submit_block_header_inner`. No privileged keys or special roles are required. The Dogecoin case (post-145,000) is the most impactful because every block is a retarget boundary, meaning the desynchronization affects every fork block after the divergence point. [7](#0-6) 

### Recommendation

Replace `get_header_by_height` calls inside `get_next_work_required` with a loop that walks the fork's own ancestor chain via `get_prev_header` until reaching the target height. This mirrors how Bitcoin Core and Dogecoin Core actually traverse the chain: they walk `pprev` pointers from the current block, not a height-indexed map of the canonical chain.

```rust
// Instead of:
let first_block_time = blocks_getter.get_header_by_height(height_first).block_header.time;

// Walk the fork's own ancestor chain:
let mut cursor = prev_block_header.clone();
while cursor.block_height > height_first {
    cursor = blocks_getter.get_prev_header(&cursor.block_header);
}
let first_block_time = cursor.block_header.time;
```

### Proof of Concept

1. Deploy the Dogecoin light client contract (post-genesis height ≥ 145,000).
2. Submit a sequence of main-chain blocks up to height `H`. The main chain's block at `H` has timestamp `T_mc_H` and block at `H+1` has timestamp `T_mc_H1`.
3. Submit a fork block at height `H+1` with timestamp `T_fork_H1` chosen so that `T_fork_H1 - T_mc_H` (the desynchronized timespan) is maximally large (up to 2 hours ahead of local time), producing a lower expected difficulty for `H+2`.
4. Submit a fork block at height `H+2` with `bits` matching the incorrectly computed lower difficulty. The contract accepts it via `expected_bits == block_header.bits`, even though the correct difficulty (computed from `T_fork_H2 - T_fork_H1`) would be higher.
5. If this fork accumulates more chainwork than the main chain, `reorg_chain` promotes it, corrupting the canonical chain state with headers that passed an incorrect difficulty check. [8](#0-7)
