### Title
Silent `uint256`→`uint128` Truncation in `creditDeposit` Deposits Incorrect Amount — (`DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` reads the contract's full `uint256` token balance and silently truncates it to `uint128` before passing it to `depositCollateralWithReferral`. In Solidity 0.8.x, explicit narrowing casts truncate silently — they do not revert. If the accumulated balance exceeds `type(uint128).max`, the subaccount is credited a wrong (truncated) amount while the full balance is approved for transfer.

---

### Finding Description

In `DirectDepositV1.creditDeposit()`:

```solidity
uint256 balance = token.balanceOf(address(this));
if (balance != 0) {
    token.approve(address(endpoint), balance);
    endpoint.depositCollateralWithReferral(
        subaccount,
        productId,
        uint128(balance),   // <-- silent truncation, no bounds check
        "-1"
    );
}
``` [1](#0-0) 

`balance` is `uint256` from `token.balanceOf(address(this))`. The cast `uint128(balance)` silently drops the upper 128 bits if `balance > type(uint128).max`. There is no `require(balance <= type(uint128).max)` guard anywhere in this function. The `depositCollateralWithReferral` interface accepts `uint128 amount`, so the truncated value is accepted without error. [2](#0-1) 

By contrast, every other deposit path in the protocol explicitly guards against this: `Clearinghouse.depositCollateral` enforces `require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW)` before any cast. [3](#0-2) 

`DirectDepositV1.creditDeposit()` has no equivalent guard.

---

### Impact Explanation

If `balance > 2^128 - 1`, the amount passed to the endpoint is `balance mod 2^128` — a value that bears no correct relationship to the actual token balance held. The subaccount receives a credit that is far smaller than the tokens deposited. The excess tokens remain in the DDA contract. In the degenerate case where `balance` is an exact multiple of `2^128`, `uint128(balance) == 0`, so `depositCollateralWithReferral` is called with `amount = 0` and the entire token balance is permanently stuck in the DDA contract with no mechanism to recover it (the only other exit is `withdraw`, which is `onlyOwner`).

The corrupted state delta is: `spotEngine.balance[subaccount][productId]` is updated with a truncated `amountRealized` instead of the correct scaled amount, permanently desynchronising the on-chain credit from the actual token custody. [4](#0-3) 

---

### Likelihood Explanation

`type(uint128).max ≈ 3.4 × 10^38`. For a token with 18 decimals this requires `~3.4 × 10^20` tokens to accumulate in a single DDA contract — an extreme condition for most tokens. Likelihood is therefore **low** for standard ERC-20s. However, the DDA contract is permissionless: anyone can send tokens to it, and `creditDeposit()` is callable by anyone with no access control. A token with low decimals (e.g., 0 or 2) and a very large supply, or a rebasing/fee-on-transfer token that inflates balances, could reach this threshold more plausibly. The absence of any guard is the root cause regardless of likelihood. [5](#0-4) 

---

### Recommendation

Add an explicit bounds check before the cast, mirroring the pattern used in `Clearinghouse.depositCollateral`:

```solidity
require(balance <= type(uint128).max, "balance exceeds uint128");
endpoint.depositCollateralWithReferral(
    subaccount,
    productId,
    uint128(balance),
    "-1"
);
```

Alternatively, if partial deposits are acceptable, loop depositing in `type(uint128).max`-sized chunks.

---

### Proof of Concept

1. A token `LARGE_BAL` with 0 decimals and total supply `> 2^128` is listed as a collateral product.
2. A user (or attacker) transfers `2^128 + 1` units of `LARGE_BAL` to the DDA contract address.
3. Anyone calls `DirectDepositV1.creditDeposit()`.
4. `balance = 2^128 + 1`; `uint128(balance) = 1`.
5. The endpoint transfers only `1` token from the DDA contract and credits the subaccount with `1` unit (scaled by the multiplier).
6. The remaining `2^128` tokens stay in the DDA contract. The subaccount's on-chain balance reflects `1` token, not `2^128 + 1`.
7. The off-chain exchange engine reads the `ModifyCollateral` event with the truncated `amountRealized` and assigns the wrong deposit credit to the user. [1](#0-0) [6](#0-5)
