Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual Swapper, Allowing Any User to Bypass the Curated-Pool Swap Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` as seen by the pool — the router contract — not the end user initiating the swap. When a pool admin allowlists the router address (required for any router-mediated swap to succeed), every user on the internet can bypass the allowlist and swap on the curated pool. No configuration exists that allows only allowlisted users to swap via the router.

## Finding Description
The call chain when a user swaps via `MetricOmmSimpleRouter.exactInputSingle` is:

1. `MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — so `msg.sender` to the pool is the router address. [1](#0-0) 

2. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, recipient, ...)` passing the router as `sender`. [2](#0-1) 

3. `ExtensionCalling._beforeSwap` forwards `sender` (the router) verbatim to the extension via `abi.encodeCall`. [3](#0-2) 

4. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` — where `msg.sender` is the pool (correct key) and `sender` is the router (wrong actor). The lookup becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. [4](#0-3) 

This creates an inescapable dilemma for pool admins:
- **Do not allowlist the router** → allowlisted users cannot use the router at all.
- **Allowlist the router** → every user on the internet bypasses the allowlist.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly ignores the `sender` parameter and checks `owner` (the position owner), which is the economically relevant actor for deposits. [5](#0-4) 

## Impact Explanation
A curated pool deploying `SwapAllowlistExtension` to restrict trading to specific counterparties (e.g., whitelisted market makers, KYC'd users, or protocol-controlled addresses) is fully bypassed the moment the pool admin allowlists the router. Any unprivileged user can call `MetricOmmSimpleRouter.exactInputSingle` and execute swaps on the restricted pool. This enables unauthorized extraction of LP assets at oracle-derived prices from a pool intended for private use, circumvention of regulatory or risk-management controls, and direct loss of LP principal if the pool was designed to trade only with trusted counterparties. This matches the **Admin-boundary break** and **Broken core pool functionality causing loss of funds** impact categories.

## Likelihood Explanation
Likelihood is **High**: `MetricOmmSimpleRouter` is the primary user-facing interface for the protocol. Any pool admin who wants to allow even a single user to swap via the router must allowlist the router address, which simultaneously opens the pool to all users. No special knowledge or privileged access is required — any user calls the public router. The bypass is deterministic and requires zero preconditions beyond the router being allowlisted.

## Recommendation
The `beforeSwap` hook must gate the economically relevant actor. The most robust on-chain fix is to check `recipient` instead of `sender`, since `recipient` is always the address that receives value from the swap and cannot be spoofed by an intermediary:

```solidity
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

Alternatively, the router can encode `msg.sender` (the real user) into `extensionData` and the extension can decode and check it, but this requires router cooperation and is trust-dependent.

## Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; only `trustedUser` is allowlisted
swapExtension.setAllowedToSwap(address(pool), trustedUser, true);
// Admin also allowlists the router so trustedUser can use it
swapExtension.setAllowedToSwap(address(pool), address(router), true);

// Attack: attacker (not allowlisted) bypasses the guard via the router
address attacker = makeAddr("attacker");
vm.startPrank(attacker);
token0.approve(address(router), type(uint256).max);

// This should revert with NotAllowedToSwap — but it succeeds
// because the extension checks allowedSwapper[pool][router] = true
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        recipient: attacker,
        tokenIn: address(token0),
        zeroForOne: true,
        amountIn: 1000e18,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        deadline: block.timestamp + 1,
        extensionData: ""
    })
);
// Attacker successfully swaps on a pool they are not allowlisted for
```
