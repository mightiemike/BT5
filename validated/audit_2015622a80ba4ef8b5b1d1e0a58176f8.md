### Title
Migration Halt: `ForceFullExitWithDelegation` Propagates `ErrValidatorNotBonded` for Jailed Validators, Aborting v8 Upgrade — (`x/tieredrewards/migrations/v2/migrate.go`, `x/tieredrewards/keeper/force_exit.go`, `x/tieredrewards/keeper/transfer_delegation.go`)

---

### Summary

The v2 migration (`Migrate1to2`) iterates all vesting-owned tier positions and calls `ForceFullExitWithDelegation` on each. That function calls `transferDelegationFromPosition`, which contains a hard `IsBonded()` guard that returns `ErrValidatorNotBonded` for any jailed or unbonded validator. The error is propagated without any catch or continue logic at every layer, causing the entire migration — and therefore the v8 chain upgrade — to abort.

---

### Finding Description

**Exact call chain:**

`Migrate1to2` → `v2.Migrate` → `exitVestedAccountsPositions` → `ForceFullExitWithDelegation(posID)` → `transferDelegationFromPosition` → `ErrValidatorNotBonded` → error propagates → upgrade halts.

**Step 1 — `exitVestedAccountsPositions` calls `ForceFullExitWithDelegation` with no error tolerance:** [1](#0-0) 

**Step 2 — `ForceFullExitWithDelegation` propagates the error from `transferDelegationFromPosition` unconditionally:** [2](#0-1) 

The only graceful return in `ForceFullExitWithDelegation` is for non-delegated positions (returns `nil`). There is no analogous guard for jailed/unbonded validators: [3](#0-2) 

**Step 3 — `transferDelegationFromPosition` has a hard `IsBonded()` check:** [4](#0-3) 

**Step 4 — Error propagates through `v2.Migrate`:** [5](#0-4) 

**Step 5 — `Migrate1to2` returns the error to the upgrade handler:** [6](#0-5) 

---

### Impact Explanation

If any vesting account holds a tier position delegated to a validator that is jailed (or otherwise non-bonded) at the moment the v8 upgrade block is processed, `RunMigrations` returns an error, the upgrade handler panics or aborts, and the chain halts. All funds in all positions are locked indefinitely until a patched binary is deployed via an emergency governance upgrade — a severe operational and financial impact.

---

### Likelihood Explanation

Validators are routinely jailed for downtime or double-signing. The window between position creation and the upgrade block can span months. There is no on-chain mechanism that prevents a vesting account from holding a position on a validator that later becomes jailed. The precondition is realistic and requires no attacker action — it can occur through ordinary validator misbehavior.

---

### Recommendation

In `ForceFullExitWithDelegation`, check the validator's bonded status before calling `transferDelegationFromPosition`. If the validator is not bonded, fall back to the standard unbonding path (`TierUndelegate`-equivalent: unbond from the position delegator address and let the 21-day unbonding period complete), or at minimum skip the position with a logged warning and continue the migration loop rather than aborting. The `exitVestedAccountsPositions` loop should also be made resilient by catching per-position errors and continuing, rather than returning on the first failure.

---

### Proof of Concept

1. Create a vesting account `V`.
2. Have `V` call `LockTier` or `CommitDelegationToTier` on validator `val1`, creating position `P`.
3. Jail `val1` (e.g., via simulated downtime slash in an integration test).
4. Run `Migrate1to2` (simulating the v8 upgrade block).
5. Observe: `exitVestedAccountsPositions` calls `ForceFullExitWithDelegation(P.Id)`, which calls `transferDelegationFromPosition`, which returns `ErrValidatorNotBonded` at `transfer_delegation.go:132-134`. The error propagates through `force_exit.go:63`, `migrate.go:100`, `migrate.go:38`, and `migrations.go:18`. `Migrate1to2` returns a non-nil error, the upgrade handler aborts, and the chain halts.
