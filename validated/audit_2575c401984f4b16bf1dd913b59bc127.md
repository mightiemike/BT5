Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` validates LP recipient (`owner`) instead of actual depositor (`sender`), allowing any unprivileged caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension` is intended to gate `addLiquidity` by depositor address. Its `beforeAddLiquidity` hook silently discards the `sender` argument (the actual caller who pays tokens and triggers the callback) and instead validates `owner` (the LP-position recipient). Because `addLiquidity` allows the caller to freely specify `owner`, any address not on the allowlist can deposit tokens by nominating an allowlisted address as `owner`, fully bypassing the guard. The allowlisted address receives an unwanted LP position it cannot refuse, and can only unwind it by calling `removeLiquidity` itself.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the position recipient into the extension hook:

```solidity
// metric-core/contracts/MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both addresses to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L95-98
_callExtensionsInOrder(
  BEFORE_ADD_LIQUIDITY_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
);
```

Inside `DepositAllowlistExtension.beforeAddLiquidity`, however, the first parameter (`sender`) is unnamed and discarded; only `owner` is checked:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
  }
  return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

After the hook passes, `LiquidityLib.addLiquidity` credits shares to `owner` and then fires the token-collection callback on `msg.sender` (the actual caller, confirmed by the library's delegatecall context):

```solidity
// metric-core/contracts/libraries/LiquidityLib.sol L147-148
IMetricOmmModifyLiquidityCallback(msg.sender)
  .metricOmmModifyLiquidityCallback(amount0Added, amount1Added, callbackData);
```

So the entity that pays tokens is `msg.sender` (the actual caller, Bob), not `owner` (Alice). The allowlist check on `owner` is therefore checking the wrong actor. `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual caller), confirming the asymmetry is a defect:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

Additionally, `removeLiquidity` enforces `msg.sender == owner`, meaning Alice (who received the forced position) must actively call `removeLiquidity` herself to unwind it:

```solidity
// metric-core/contracts/MetricOmmPool.sol L206
if (msg.sender != owner) revert NotPositionOwner();
```

## Impact Explanation

An unprivileged caller bypasses the pool admin's deposit allowlist entirely: tokens from a non-allowlisted address enter the pool, violating the core access-control invariant the extension is designed to enforce. If the allowlist is used for regulatory compliance or to restrict liquidity to trusted counterparties, the bypass has direct protocol-level consequences. Additionally, the allowlisted address (Alice) receives an LP position she never requested and must actively unwind it, constituting a griefing vector. This is a broken core pool access-control functionality causing unauthorized fund flows into the pool.

## Likelihood Explanation

No special privilege is required. Any EOA or contract can call `addLiquidity` with an arbitrary `owner`. The only prerequisite is knowing at least one allowlisted address, which is publicly readable from `allowedDepositor` or `allowAllDepositors`. No admin cooperation, flash loan, oracle manipulation, or non-standard token behavior is needed. The attack is repeatable at will.

## Recommendation

Replace the unnamed first parameter with `sender` and validate it instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
  }
  return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate both the payer and the position owner, both `sender` and `owner` should be checked independently.

## Proof of Concept

```
Setup
─────
1. Pool P is deployed with DepositAllowlistExtension E.
2. Pool admin calls E.setAllowedToDeposit(P, alice, true).
   → allowedDepositor[P][alice] = true
   → allowedDepositor[P][bob]   = false  (bob is NOT allowed)

Attack
──────
3. Bob deploys a router contract R that:
   a. Calls P.addLiquidity(alice, salt, deltas, callbackData, "").
   b. Implements metricOmmModifyLiquidityCallback to transfer required tokens to P.

4. Inside P.addLiquidity:
   - _beforeAddLiquidity(R, alice, ...) is called.
   - Extension E receives: sender=R (ignored), owner=alice.
   - Check: allowedDepositor[P][alice] == true → passes.
   - LiquidityLib credits LP shares to alice (positionBinShares keyed by alice).
   - Callback fires on R (msg.sender); R transfers tokens to P.

Result
──────
5. Bob (via R) has deposited tokens into P without being on the allowlist.
6. Alice holds an LP position she never requested.
7. Alice must call removeLiquidity herself (msg.sender == owner check) to unwind.
8. Pool admin's deposit restriction is silently violated.

Foundry test sketch:
- Deploy pool with DepositAllowlistExtension.
- allowedDepositor[pool][alice] = true, bob not listed.
- Bob's router calls addLiquidity(alice, ...) and funds the callback.
- Assert: transaction succeeds, alice's positionBinShares > 0, bob's address never checked.
