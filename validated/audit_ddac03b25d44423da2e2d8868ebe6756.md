### Title
Accrued Bonus Rewards Permanently Lost When Redelegation Slash Occurs With Insufficient Pool — (`File: x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

In `slashRedelegationPosition`, when `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, the error is silently swallowed and the position is persisted with all reward checkpoints already advanced — but without the bonus ever being paid. The accrued bonus is permanently and irrecoverably forfeited.

---

### Finding Description

`processEventsAndClaimBonus` (`x/tieredrewards/keeper/claim_rewards.go`) modifies the `pos *types.PositionState` pointer in-place **before** checking whether the pool can cover the computed bonus:

1. **Line 193** — inside the event loop: `pos.UpdateLastEventSeq(entry.Seq)` advances the event sequence checkpoint for every processed event.
2. **Line 215** — after the loop: `applyBonusAccrualCheckpoint(&pos.Position, blockTime)` advances `LastBonusAccrual` to the current block time.
3. **Line 217** — `pos.UpdateLastKnownBonded(bonded)` updates the bonded-state checkpoint.
4. **Line 230** — only then: `k.sufficientBonusPoolBalance(ctx, bonusCoins)` checks the pool. If insufficient, it returns an error — but all three checkpoints on `pos` are already mutated. [1](#0-0) [2](#0-1) [3](#0-2) 

In `slashRedelegationPosition` (`x/tieredrewards/keeper/slash.go`), the caller passes `&pos` (a pointer to the local position copy), catches the error, and explicitly swallows `ErrInsufficientBonusPool` to avoid a chain halt:

```go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
    } else {
        return err
    }
}
``` [4](#0-3) 

Execution then falls through to `setPositionWithState`, which persists the position with the already-advanced `LastBonusAccrual`, `LastEventSeq`, and `LastKnownBonded` — but with zero bonus paid: [5](#0-4) 

Because `LastBonusAccrual` is now set to the current block time and `LastEventSeq` has consumed all pending events, there is no future path to recover the forfeited bonus. The next `MsgClaimTierRewards` call will compute zero bonus for the already-consumed time window.

---

### Impact Explanation

Any tier position holder whose active redelegation is slashed while the `RewardsPoolName` module account holds insufficient `bondDenom` permanently loses all bonus rewards accrued since their last claim. The corrupted value is the `LastBonusAccrual` timestamp and `LastEventSeq` stored in the `Position` proto — they are advanced to reflect a "paid" state that was never actually paid. The `RewardsPoolName` balance is not reduced (no payment occurred), but the position's claim window is closed forever.

---

### Likelihood Explanation

The bonus pool (`RewardsPoolName`) is funded externally (e.g., via `MsgSend` to the module address) and is continuously drained by both the `BeginBlocker` base-rewards top-up and by user `MsgClaimTierRewards` calls. The pool can be empty or near-empty at any block. A redelegation slash is a normal Cosmos SDK protocol event triggered whenever a validator is slashed while a redelegation from it is still in the unbonding window. Any unprivileged user who has called `MsgTierRedelegate` is exposed. The two conditions — pool insufficient AND redelegation slash — are independent and can coincide at any time.

---

### Recommendation

Move `applyBonusAccrualCheckpoint`, `pos.UpdateLastKnownBonded`, and `pos.UpdateLastEventSeq` to execute **after** the pool balance check succeeds (i.e., after line 230 in `claim_rewards.go`). This ensures that if the pool is insufficient, the position's checkpoints remain unchanged and the accrued bonus is preserved for a future claim once the pool is replenished.

Alternatively, in `slashRedelegationPosition`, when `ErrInsufficientBonusPool` is caught, explicitly restore the original checkpoint values on `pos` before calling `setPositionWithState`, so the position is persisted in its pre-call state.

---

### Proof of Concept

1. User calls `MsgLockTier` → creates position P on validator V1.
2. User calls `MsgTierRedelegate` → redelegates P from V1 to V2; a `RedelegationMapping[unbondingId → P.Id]` entry is created.
3. The bonus pool is drained to zero (e.g., by other users claiming, or simply never funded).
4. V1 is slashed while the redelegation is still in the unbonding window.
5. Staking fires `BeforeRedelegationSlashed` → `slashRedelegationPosition` is called.
6. `processEventsAndClaimBonus(ctx, &pos)` runs:
   - Advances `pos.LastEventSeq` past all pending events (line 193).
   - Sets `pos.LastBonusAccrual = blockTime` (line 215).
   - Returns `ErrInsufficientBonusPool` (line 230) — no coins sent.
7. Error is swallowed (slash.go line 56–63).
8. `setPositionWithState(ctx, pos, ...)` persists the position with the advanced checkpoints (slash.go line 71 or 77).
9. User later calls `MsgClaimTierRewards` for position P → `processEventsAndClaimBonus` computes zero bonus because `LastBonusAccrual` already equals the current time and no new events exist. The entire accrued bonus window is permanently lost. [6](#0-5) [7](#0-6)
