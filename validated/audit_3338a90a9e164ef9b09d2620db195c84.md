### Title
Missing Validation of `skip_pow_verification` During Initialization Permanently Disables PoW Verification — (`contract/src/lib.rs`)

---

### Summary

The `init()` function in `contract/src/lib.rs` accepts the `skip_pow_verification` flag from `InitArgs` and stores it directly without any on-chain enforcement that it must be `false` for production deployments. If the contract is initialized with `skip_pow_verification: true` — a realistic risk during redeployments on new networks or protocol versions — all Proof-of-Work verification is permanently disabled for every subsequent `submit_blocks` call. Any account holding the `UnrestrictedSubmitBlocks` role can then submit headers with arbitrary `bits` values and invalid PoW hashes, corrupting the canonical mainchain state and enabling forged SPV proof results.

---

### Finding Description

The `init()` function unconditionally stores the caller-supplied `skip_pow_verification` flag: [1](#0-0) 

```rust
pub fn init(args: InitArgs) -> Self {
    let mut contract = Self {
        ...
        skip_pow_verification: args.skip_pow_verification,  // no validation
        gc_threshold: args.gc_threshold,
        ...
    };
```

The `InitArgs` struct defines this as a plain `bool` with no constraints: [2](#0-1) 

The only guidance is a doc comment — not an enforced invariant: [3](#0-2) 

Once stored, `skip_pow_verification` is read directly in `submit_blocks()` and passed to `submit_block_header()`: [4](#0-3) 

Inside `submit_block_header()`, when the flag is `true`, both `check_target()` (difficulty rule) and the PoW hash comparison are entirely skipped: [5](#0-4) 

There is no setter for `skip_pow_verification` after initialization, and the `migrate()` function carries the flag forward from old state without re-validating it: [6](#0-5) 

---

### Impact Explanation

With `skip_pow_verification: true` set at init time, any account granted the `UnrestrictedSubmitBlocks` role can call `submit_blocks()` with headers containing:
- Arbitrary `bits` (difficulty target)
- Any `prev_block_hash` pointing to a known block
- A hash that does not satisfy any PoW target

These headers are accepted and stored as canonical mainchain blocks. The corrupted mainchain state then causes `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` to return `true` for transactions that were never confirmed on the real Bitcoin chain, breaking the trust-minimized bridge guarantee. [7](#0-6) 

---

### Likelihood Explanation

Low — identical in character to the original report. The `init()` function is `#[private]`, so only the contract account (deployer) can call it. However, the risk materializes during redeployments on new networks or new protocol versions, where deployment scripts may inadvertently carry over a `skip_pow_verification: true` flag used during testing. The relayer's `flow.sh` deployment script and the `InitArgs` struct impose no on-chain guard against this. [8](#0-7) 

---

### Recommendation

Add an explicit on-chain `require!` in `init()` that rejects `skip_pow_verification: true` unless a deliberate override is provided, or at minimum emit a prominent log warning. The check should mirror the pattern already used for other critical init parameters:

```rust
// In init(), before storing the flag:
if args.skip_pow_verification {
    near_sdk::log!("WARNING: skip_pow_verification is true — PoW is disabled");
}
// For strict production enforcement:
// require!(!args.skip_pow_verification, "skip_pow_verification must be false in production");
```

Additionally, the `migrate()` function should re-validate the carried-over `skip_pow_verification` value from old state to prevent a misconfigured flag from surviving a contract upgrade. [9](#0-8) 

---

### Proof of Concept

1. Deploy the contract with `InitArgs { skip_pow_verification: true, ... }` — this succeeds with no on-chain rejection.
2. Grant any account the `UnrestrictedSubmitBlocks` role via `acl_grant_role`.
3. Call `submit_blocks` with a header whose `bits` encodes an impossibly easy target and whose hash does not satisfy any real PoW target.
4. The header is accepted and stored as the new mainchain tip (`mainchain_tip_blockhash` updated).
5. Call `verify_transaction_inclusion_v2` with a crafted Merkle proof against the fake block — returns `true`.

The test suite itself demonstrates step 1 is accepted without error: [10](#0-9)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
