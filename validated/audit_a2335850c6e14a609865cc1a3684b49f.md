### Title
Gas-Key Host Functions Compute Exec Fees Using Wire Length Instead of Trie-ID Length for ML-DSA-65 Keys — (`runtime/near-vm-runner/src/logic/logic.rs`, `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`)

---

### Summary

The three gas-key host functions (`promise_batch_action_add_gas_key_with_full_access`, `promise_batch_action_add_gas_key_with_function_call`, `promise_batch_action_transfer_to_gas_key`) compute their exec-side gas fees using the raw `public_key_len` value supplied by the WASM guest — the borsh wire length of the public key. For ML-DSA-65 keys this is **1953 bytes**. The actual on-trie identifier for an ML-DSA-65 key is a SHA3-256 hash: **33 bytes**. The transaction-path fee helpers correctly call `public_key.trie_id_len()` (33), but the host-function path passes `public_key_len as usize` (1953) to the same fee helper, producing a fee that is ~1920 bytes × `gas_key_byte` exec rate × `num_nonces` too high per call.

---

### Finding Description

`PublicKey::len()` returns the borsh-encoded wire size (1 + 1952 = 1953 for ML-DSA-65). `PublicKey::trie_id_len()` returns the on-trie identifier size (1 + 32 = 33 for ML-DSA-65, because the trie stores a SHA3-256 hash, not the full key). The two values are identical for ed25519 and secp256k1, but diverge by 1920 bytes for ML-DSA-65. [1](#0-0) [2](#0-1) 

The **transaction path** in `permission_exec_fees` correctly uses `trie_id_len()`:

```rust
let nonce_fee = gas_key_add_key_exec_fee(
    fees,
    account_id.len(),
    public_key.trie_id_len(),   // ← correct: 33 for ML-DSA-65
    gas_key_info.num_nonces,
);
``` [3](#0-2) 

The **host-function path** in `promise_batch_action_add_gas_key_with_full_access` passes the raw guest-supplied wire length instead:

```rust
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,    // ← wrong: 1953 for ML-DSA-65
    num_nonces,
);
``` [4](#0-3) 

The same mismatch appears in `promise_batch_action_add_gas_key_with_function_call`: [5](#0-4) 

And in `promise_batch_action_transfer_to_gas_key` (exec fee path): [6](#0-5) 

The wasmtime runner duplicates the same bug: [7](#0-6) [8](#0-7) 

`gas_key_add_key_exec_fee` uses `public_key_len` to compute the trie nonce-key length:

```rust
let nonce_key_len =
    access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();
``` [9](#0-8) 

`access_key_key_len` is a pure byte-count function that treats its second argument as the on-trie key size: [10](#0-9) 

When `public_key_len` = 1953 instead of 33, every nonce entry is priced as if its trie key is 1920 bytes longer than it actually is.

The existing fee-parity test only exercises ed25519 keys, where `len() == trie_id_len()`, so the discrepancy is invisible to it: [11](#0-10) 

---

### Impact Explanation

Any contract that calls `promise_batch_action_add_gas_key_with_full_access` or `promise_batch_action_add_gas_key_with_function_call` with an ML-DSA-65 public key is overcharged by approximately:

```
1920 bytes × gas_key_byte_exec_fee × num_nonces
```

per invocation. For `promise_batch_action_transfer_to_gas_key` the overcharge is:

```
1920 bytes × gas_key_byte_exec_fee
```

This breaks the protocol invariant that the fee for an operation is identical whether it is submitted as a transaction or dispatched via a host function. The overcharge is deterministic and reproducible on every call. Contracts that add ML-DSA-65 gas keys programmatically pay significantly more gas than the equivalent transaction, making such operations economically inconsistent and potentially unviable at scale.

---

### Likelihood Explanation

`PostQuantumSignatures` is stabilized at protocol version 85 (the current stable version). Any deployed contract that calls the gas-key host functions with an ML-DSA-65 key triggers the overcharge immediately. The WASM ABI passes `public_key_len` as the borsh-encoded byte count of the key buffer, which for ML-DSA-65 is always 1953. No special attacker capability is required — any unprivileged user can deploy a contract and invoke these host functions.

---

### Recommendation

In all three host functions, after decoding the public key, replace `public_key_len as usize` with `public_key.decode()?.trie_id_len()` (or decode first, then use `trie_id_len()`) when calling the fee helpers. Concretely, for `promise_batch_action_add_gas_key_with_full_access`:

```rust
let public_key = self.get_public_key(public_key_ptr, public_key_len)?.decode()?;
// ...
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key.trie_id_len(),   // ← use trie length, not wire length
    num_nonces,
);
// ...
self.ext.append_action_add_gas_key_with_full_access(receipt_idx, public_key, num_nonces);
```

Apply the same fix to `promise_batch_action_add_gas_key_with_function_call` and `promise_batch_action_transfer_to_gas_key` in both `logic.rs` and `wasmtime_runner/logic.rs`. Add a fee-parity test that exercises ML-DSA-65 keys alongside ed25519 keys.

---

### Proof of Concept

1. Deploy a contract that calls `promise_batch_action_add_gas_key_with_full_access` with an ML-DSA-65 public key and `num_nonces = N`.
2. Submit the equivalent `AddKey` transaction with the same ML-DSA-65 key and `num_nonces = N`.
3. Compare `gas_burnt` on the action execution receipt for both paths.
4. The host-function path burns approximately `1920 × gas_key_byte_exec_fee × N` more gas than the transaction path, despite writing identical trie entries.

The existing test `test_gas_key_fee_parity` demonstrates the correct parity for ed25519 keys. Running the same test with `KeyType::MLDSA65` would expose the divergence. [12](#0-11)

```rust
pub fn gas_key_add_key_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
    num_nonces: NonceIndex,
) -> GasKeyAddFee {
    let num_nonces = num_nonces as u64;
    let base =
        cfg.fee(ActionCosts::gas_key_nonce_write_base).exec_fee().checked_mul(num_nonces).unwrap();
    let nonce_key_len =
        access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64)
        .unwrap()
        .checked_mul(num_nonces)
        .unwrap();
    GasKeyAddFee { base, per_byte }
```

/// Verify that adding, funding, and deleting a gas key via transaction vs host function
/// produces identical gas_burnt and tokens_burnt on the action execution receipt.
fn test_gas_key_fee_parity(mode: GasKeyKind) {
    let mut setup = setup_host_function_test();
    let account = setup.account.clone();

    let num_nonces: NonceIndex = 4;
    let gas_key_a_signer: Signer =
        InMemorySigner::from_seed(account.clone(), KeyType::ED25519, "gas_key_tx").into();
    let gas_key_b_signer: Signer =
        InMemorySigner::from_seed(account.clone(), KeyType::ED25519, "gas_key_host_fn").into();

    let public_key_b_base64 = near_primitives_core::serialize::to_base64(
        &borsh::to_vec(&gas_key_b_signer.public_key()).unwrap(),
    );

    // Add gas key A via transaction, B via host function
    let add_a_outcome = setup.run_actions(vec![Action::AddKey(Box::new(AddKeyAction {
        public_key: gas_key_a_signer.public_key(),
        access_key: mode.access_key(num_nonces, &account),
    }))]);
    let add_b_outcome = setup.run_call_promise(serde_json::json!([
        {"batch_create": {"account_id": account.as_str()}, "id": 0},
        mode.add_action_json(num_nonces, &account, &public_key_b_base64),
    ]));
    assert_eq!(add_a_outcome.gas_burnt, add_b_outcome.gas_burnt);
    assert_eq!(add_a_outcome.tokens_burnt, add_b_outcome.tokens_burnt);
