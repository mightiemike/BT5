### Title
Fee-on-Transfer Token Deposit Inflates Subaccount Collateral Beyond Actual Holdings — (`core/contracts/EndpointStorage.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

The Nado deposit flow uses the caller-supplied `amount` parameter throughout — for both token transfers and on-chain balance crediting — without ever measuring the actual tokens received. When a fee-on-transfer token is a supported collateral asset, the Clearinghouse credits a user's subaccount with the full nominal `amount` while physically holding fewer tokens, creating an unbacked collateral surplus that corrupts protocol solvency.

---

### Finding Description

The deposit path spans three steps, each propagating the same nominal `amount` without reconciliation against actual balances:

**Step 1 — `Endpoint.depositCollateralWithReferral`** calls `handleDepositTransfer` with the raw user-supplied `amount`, then enqueues a `SlowModeTx` encoding `DepositCollateral { amount: amount }` — the same nominal figure. [1](#0-0) 

**Step 2 — `EndpointStorage.handleDepositTransfer`** performs two sequential transfers, both using the nominal `amount`:

```
safeTransferFrom(token, from, amount);          // user → Endpoint
safeTransferTo(token, address(clearinghouse), amount); // Endpoint → Clearinghouse
``` [2](#0-1) 

For a fee-on-transfer token with fee rate `f`:
- Transfer 1: Endpoint receives `amount × (1 - f)`, not `amount`.
- Transfer 2: Endpoint attempts to forward the full `amount`. If Endpoint holds any pre-existing balance of that token (e.g., from slow-mode fees or prior deposits), this transfer succeeds — but Clearinghouse receives only `amount × (1 - f)` again.

**Step 3 — `Clearinghouse.depositCollateral`** reads `txn.amount` from the slow-mode queue (the original nominal value) and credits the user's subaccount with `amountRealized = int128(txn.amount) * int128(multiplier)`. No balance snapshot is taken; the actual received amount is never consulted. [3](#0-2) 

The result: the Clearinghouse holds `amount × (1 - f)²` of the token but records `amount` of collateral for the depositor — a permanent, compounding discrepancy on every deposit.

---

### Impact Explanation

The inflated subaccount balance is used directly by `SpotEngine` and `PerpEngine` for health checks, position sizing, and liquidation thresholds. A user depositing a fee-on-transfer token receives more collateral credit than the protocol physically holds. This allows:

- Opening perpetual or spot positions larger than actual backing.
- Withdrawing the full credited `amount` while the protocol only holds `amount × (1 - f)²`, draining reserves from other depositors.
- Repeated deposits amplifying the shortfall, degrading protocol solvency proportionally to the token's fee rate and deposit volume.

The corrupted state variable is the `SpotEngine` balance for the depositor's subaccount, which exceeds the real token balance held by the Clearinghouse.

---

### Likelihood Explanation

This requires a fee-on-transfer token to be listed as a supported collateral product. The Nado architecture does not restrict token types at the contract level — any ERC-20 can be registered via governance. If such a token is listed (common in DeFi ecosystems), any unprivileged user calling `depositCollateral` or `depositCollateralWithReferral` triggers the bug. The `DirectDepositV1.creditDeposit` path also reaches the same vulnerable `depositCollateralWithReferral` entry point. [4](#0-3) 

---

### Recommendation

Replace the nominal-amount forwarding in `handleDepositTransfer` with a balance-snapshot pattern:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
    uint256 actualReceived = token.balanceOf(address(clearinghouse)) - before;
    // pass actualReceived (not amount) into the SlowModeTx
}
```

The `DepositCollateral` slow-mode transaction must encode `actualReceived`, not the caller-supplied `amount`, so that `Clearinghouse.depositCollateral` credits only what was physically received.

---

### Proof of Concept

1. A fee-on-transfer token `FOT` (1% fee per transfer) is listed as `productId = 5`.
2. Endpoint holds a pre-existing balance of 10 FOT (from prior slow-mode fee charges).
3. Attacker calls `Endpoint.depositCollateral("default", 5, 1000e18)`.
4. `handleDepositTransfer` runs:
   - `safeTransferFrom(FOT, attacker, 1000e18)` → Endpoint receives `990e18`.
   - `safeTransferTo(FOT, clearinghouse, 1000e18)` → succeeds (Endpoint had 10 FOT buffer); Clearinghouse receives `990e18`.
5. `SlowModeTx` is enqueued with `amount = 1000e18`.
6. After the 3-day delay, sequencer executes the slow-mode tx.
7. `Clearinghouse.depositCollateral` computes `amountRealized = 1000e18 * multiplier` and calls `spotEngine.updateBalance(5, attacker_subaccount, amountRealized)`.
8. Attacker's subaccount now shows `1000e18` collateral; Clearinghouse holds only `990e18`.
9. Attacker withdraws `1000e18`, leaving the Clearinghouse `10e18` short — funded by other depositors' assets. [2](#0-1) [5](#0-4)
