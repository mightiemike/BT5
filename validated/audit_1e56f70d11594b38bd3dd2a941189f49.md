### Title
4-Byte Secp Opcode Dispatch Bypasses `ENABLE_SECP_OPS` Hard-Fork Flag — (`File: src/chia_dialect.rs`)

---

### Summary

In `ChiaDialect::op()`, the 1-byte secp opcodes (64 = `secp256k1_verify`, 65 = `secp256r1_verify`) are correctly gated behind `ClvmFlags::ENABLE_SECP_OPS`. However, the 4-byte opcode aliases for the same operations (`0x13d61f00` and `0x1c3a8f00`) are dispatched unconditionally — without checking `ENABLE_SECP_OPS` — allowing an attacker to invoke secp signature verification even when the hard-fork flag has not been activated.

---

### Finding Description

`ChiaDialect::op()` handles two distinct opcode lengths. For 4-byte opcodes, the dispatch is:

```rust
// src/chia_dialect.rs lines 175–182
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
```

No flag check precedes the dispatch. By contrast, the 1-byte opcode path for the same operations reads:

```rust
// src/chia_dialect.rs lines 248–249
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

The `ENABLE_SECP_OPS` flag is documented as a hard-fork gate:

```rust
// src/chia_dialect.rs line 63
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
```

The 4-byte path was inherited from the "unknown operators with assigned cost" mechanism (where the cost is encoded in the opcode bytes themselves), but the explicit mapping to `op_secp256k1_verify`/`op_secp256r1_verify` was added without the corresponding flag guard. The result is that the hard-fork toggle is enforced on one entry path and silently ignored on the other.

---

### Impact Explanation

Any caller that constructs a `ChiaDialect` without `ENABLE_SECP_OPS` (i.e., before the secp hard-fork activates) expects secp operations to be unavailable. A CLVM program using the 4-byte opcode form bypasses that expectation and executes a live secp signature verification. This produces a concrete, different execution result compared to what the flag-controlled path would produce (unknown-operator no-op vs. cryptographic verification success/failure). Nodes running pre-activation consensus rules would accept or reject a spend differently depending on which opcode encoding the attacker chose, creating a consensus-divergence vector.

---

### Likelihood Explanation

The entry path is fully attacker-controlled: the attacker supplies raw CLVM bytes and can freely choose the 4-byte opcode encoding. No special privilege, configuration, or social engineering is required. The only prerequisite is that the target node is running with `ENABLE_SECP_OPS` unset (i.e., in a pre-activation or non-secp-enabled context), which is the exact scenario the flag is designed to protect.

---

### Recommendation

Add the `ENABLE_SECP_OPS` guard to the 4-byte dispatch path, mirroring the 1-byte path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

When `ENABLE_SECP_OPS` is not set, the 4-byte secp opcodes should fall through to `unknown_operator`, consistent with the pre-activation behavior.

```rust
pub fn op_secp256k1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256K1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256k1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256k1_verify pubkey")?;
    let verifier = K1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256k1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is message
    let msg = atom(a, msg, "secp256k1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256k1_verify sig")?;
    let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
```
