### Title
NLP Lock Period Bypassed: `handleNlpLockedBalance` Allows Burning Locked NLP Without Checking Unlocked Balance — (File: `core/contracts/SpotEngine.sol`)

---

### Summary

The `handleNlpLockedBalance` function in `SpotEngine.sol` decrements `unlockedBalanceSum` when NLP is burned without verifying that the user has sufficient unlocked NLP. The `burnNlp` path in `Clearinghouse.sol` only checks the total NLP balance, not the unlocked portion. This allows any user to burn NLP that is still within the 4-day `NLP_LOCK_PERIOD`, fully bypassing the intended lock mechanism.

---

### Finding Description

When a user mints NLP, `handleNlpLockedBalance` is called with a positive `amountDelta`, which enqueues the minted amount with `unlockedAt = getOracleTime() + NLP_LOCK_PERIOD` (4 days). [1](#0-0) 

When a user burns NLP, `updateBalance` is called with `productId == NLP_PRODUCT_ID` and a negative `amountDelta`, which triggers `handleNlpLockedBalance` with the negative delta: [2](#0-1) 

The burn path simply decrements `unlockedBalanceSum` by `amountDelta` with no check that `unlockedBalanceSum.amount + amountDelta >= 0`. If the user has no unlocked NLP (e.g., all NLP was minted recently and is still locked), `unlockedBalanceSum` silently goes negative.

The only on-chain guard in the `burnNlp` flow is: [3](#0-2) 

This checks the **total** NLP balance (locked + unlocked), not the unlocked portion. There is no call to `getNlpUnlockedBalance` or any comparison against `unlockedBalanceSum` before the burn proceeds.

The `NLP_LOCK_PERIOD` constant confirms the protocol's intent to enforce a 4-day lock: [4](#0-3) 

---

### Impact Explanation

A user can exit their NLP position before the 4-day lock period expires. The NLP pool acts as the counterparty for all traders; the lock period is a core mechanism to prevent users from rapidly entering and exiting the pool to exploit short-term PnL swings. Bypassing it allows a user to:

- Mint NLP, collect fees or favorable rebalance proceeds, then immediately burn before any adverse pool movement.
- Front-run known bad events (e.g., large liquidations) by burning locked NLP and receiving quote collateral back before losses are socialized.

The `unlockedBalanceSum` going negative is not self-correcting in a way that prevents the burn — the user has already received quote tokens back. The negative value is only offset when future locked tranches expire, but by then the user has already exited.

---

### Likelihood Explanation

Any unprivileged user who has minted NLP can trigger this immediately. No special role, admin access, or sequencer compromise is required. The entry path is a standard `burnNlp` transaction submitted through the `Endpoint`. The sequencer batches and submits it to `Clearinghouse`, which calls `spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount)`, which calls `handleNlpLockedBalance` — the vulnerable function. [5](#0-4) 

---

### Recommendation

Add an explicit check inside `handleNlpLockedBalance` before decrementing `unlockedBalanceSum`:

```solidity
} else if (amountDelta < 0) {
    Balance memory balanceSum = nlpLockedBalanceQueues[subaccount].unlockedBalanceSum;
    require(
        balanceSum.amount + amountDelta >= 0,
        "hNLP: Insufficient unlocked NLP balance"
    );
    balanceSum.amount += amountDelta;
    nlpLockedBalanceQueues[subaccount].unlockedBalanceSum = balanceSum;
}
```

This mirrors the fix applied in the referenced Paladin report: guard the state mutation with an explicit check before allowing it to proceed.

---

### Proof of Concept

**Setup**: `NLP_LOCK_PERIOD = 4 days`. User has no prior NLP.

1. **Day 0**: User mints 100 NLP. `handleNlpLockedBalance(user, +100)` enqueues `{balance: 100, unlockedAt: Day 4}`. `unlockedBalanceSum = 0`.

2. **Day 1** (lock still active): User submits `burnNlp` with `nlpAmount = 100`.
   - `spotEngine.updateBalance(NLP_PRODUCT_ID, user, -100)` is called.
   - `handleNlpLockedBalance(user, -100)` runs: `tryUnlockNlpBalance` finds nothing to unlock (Day 4 not reached). Then `unlockedBalanceSum.amount = 0 + (-100) = -100`. No revert.
   - `getBalance(NLP_PRODUCT_ID, user).amount = 100 - 100 = 0 >= 0`. Health check passes.
   - User receives quote tokens back.

3. **Result**: User has fully exited their NLP position 3 days before the lock period expired. The `unlockedBalanceSum` is `-100`, which will only be offset when Day 4 arrives — but the user's NLP balance is already zero and the quote has been returned. [6](#0-5) [7](#0-6)

    function handleNlpLockedBalance(bytes32 subaccount, int128 amountDelta)
        internal
    {
        _assertInternal();

        // N_ACCOUNT is not limited by lock period
        if (subaccount == N_ACCOUNT) return;

        tryUnlockNlpBalance(subaccount);
        if (amountDelta > 0) {
            NlpLockedBalanceQueue storage queue = nlpLockedBalanceQueues[
                subaccount
            ];
            if (
                queue.balanceCount > 0 &&
                queue.balances[queue.balanceCount - 1].unlockedAt ==
                getOracleTime() + NLP_LOCK_PERIOD
            ) {
                queue
                    .balances[queue.balanceCount - 1]
                    .balance
                    .amount += amountDelta;
            } else {
                queue.balances[queue.balanceCount] = NlpLockedBalance({
                    balance: Balance({amount: amountDelta}),
                    unlockedAt: getOracleTime() + NLP_LOCK_PERIOD
                });
                queue.balanceCount++;
            }
        } else if (amountDelta < 0) {
            Balance memory balanceSum = nlpLockedBalanceQueues[subaccount]
                .unlockedBalanceSum;
            balanceSum.amount += amountDelta;
            nlpLockedBalanceQueues[subaccount].unlockedBalanceSum = balanceSum;
        }
    }
