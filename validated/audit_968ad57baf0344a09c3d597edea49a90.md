### Title
`gas_prices_to_hash` omits `l2_gas_consumed` and `next_l2_gas_price` for Starknet version ≥ 0.14.0, producing a wrong block hash - (`File: crates/starknet_api/src/block_hash/block_hash_calculator.rs`)

---

### Summary

The `gas_prices_to_hash` function contains an explicit TODO acknowledging that `l2_gas_consumed` and `next_l2_gas_price` must be added to the gas-prices sub-hash "after 0.14.0". The `BlockHashVersion` enum has only two variants (`V0_13_2`, `V0_13_4`), and the `TryFrom<StarknetVersion>` gate maps every Starknet version ≥ 0.13.4 — including all 0.14.x production versions — to `V0_13_4`. Consequently, the two EIP-1559 fee-market fields are never hashed, and every block produced at Starknet version ≥ 0.14.0 carries a structurally wrong block hash. The echonet validation service and the P2P protobuf layer already carry both fields, confirming the protocol expects them in the hash.

---

---

### Impact Explanation

Every block produced at Starknet version ≥ 0.14.0 (currently 0.14.0–0.14.4, the live production range) receives a block hash that is missing two fee-market fields from its Poseidon preimage. The wrong hash is:

- **Stored in the MDBX database** as the canonical block hash, poisoning all downstream lookups.
- **Returned by RPC** (`starknet_getBlockWithTxHashes`, `starknet_getBlockWithReceipts`, etc.) as an authoritative-looking but incorrect value — matching the "RPC execution … returns an authoritative-looking wrong value" impact.
- **Used as `previous_block_hash`** in the next block's hash chain, propagating the error forward.
- **Divergent from the echonet CLI and any external verifier** that implements the correct ≥ 0.14.0 hash, causing silent disagreement between the sequencer and the proving/verification layer.

This falls squarely under: *Critical — Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input* (wrong block hash stored and served) and *High — RPC execution … returns an authoritative-looking wrong value*.

---

### Likelihood Explanation

The defect is unconditional for all blocks at Starknet version ≥ 0.14.0. No special transaction, attacker input, or race condition is required — every normal block production path triggers it. The current production network runs at 0.14.4, so the condition is permanently satisfied.

---

### Recommendation

1. Add a `BlockHashVersion::V0_14_0` (or `V0_14_1`) variant to the enum and extend `TryFrom<StarknetVersion>` to map versions ≥ 0.14.0 to it.
2. Add `l2_gas_consumed: GasAmount` and `next_l2_gas_price: GasPrice` to `PartialBlockHashComponents`.
3. Update `gas_prices_to_hash` to accept and chain these two fields when `block_hash_version >= V0_14_0`, matching the echonet CLI's preimage.
4. Update `PartialBlockHashComponents::new` to populate the new fields from `BlockInfo` / `FeeMarketInfo`.

---

### Proof of Concept

**Step 1 — Confirm the version gate never fires for 0.14.x.**

`TryFrom<StarknetVersion> for BlockHashVersion` maps `V0_14_0` through `V0_14_4` to `BlockHashVersion::V0_13_4` (the `else` branch at line 80). The `gas_prices_to_hash` branch `>= V0_13_4` therefore executes, but it hashes only six price fields — never `l2_gas_consumed` or `next_l2_gas_price`.

**Step 2 — Confirm the fields exist in the protocol.**

`BlockHeaderWithoutHash.l2_gas_consumed` and `.next_l2_gas_price` are populated from the feeder gateway JSON (`l2_gas_consumed`, `next_l2_gas_price` top-level fields present in `block_post_0_14_0.json`, `block_post_0_14_2.json`, `block_post_0_14_3.json`), stored in `StorageBlockHeader`, and transmitted over P2P in `protobuf::SignedBlockHeader`.

**Step 3 — Confirm the echonet CLI expects them.**

`echonet/echo_center.py::BlobTransformer._compute_block_hash` passes `fee_market_info["l2_gas_consumed"]` and `fee_market_info["next_l2_gas_price"]` to the CLI. If the CLI implements the correct ≥ 0.14.0 hash, it will produce a different digest than the Rust `calculate_block_hash` for every 0.14.x block.

**Step 4 — Divergent value.**

For a 0.14.4 block with `l2_gas_consumed = X` and `next_l2_gas_price = Y`:

- **Rust sequencer hash preimage**: `Poseidon("STARKNET_GAS_PRICES0" ‖ l1_gas_wei ‖ l1_gas_fri ‖ l1_data_wei ‖ l1_data_fri ‖ l2_wei ‖ l2_fri)`
- **Correct ≥ 0.14.0 hash preimage**: `Poseidon("STARKNET_GAS_PRICES0" ‖ l1_gas_wei ‖ l1_gas_fri ‖ l1_data_wei ‖ l1_data_fri ‖ l2_wei ‖ l2_fri ‖ X ‖ Y)`

The two Poseidon outputs differ for any non-zero `X` or `Y`, producing a wrong `gas_prices_hash` element and therefore a wrong final `block_hash` for every live block.

impl TryFrom<StarknetVersion> for BlockHashVersion {
    type Error = StarknetApiError;

    fn try_from(value: StarknetVersion) -> StarknetApiResult<Self> {
        if value < Self::V0_13_2.into() {
            Err(StarknetApiError::BlockHashVersion { version: value.to_string() })
        } else if value < Self::V0_13_4.into() {
            // Starknet versions 0.13.2 and 0.13.3 both have the same block hash mechanism.
            Ok(Self::V0_13_2)
        } else {
            Ok(Self::V0_13_4)
        }
    }
}
