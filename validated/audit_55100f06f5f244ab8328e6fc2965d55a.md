### Title
Silent Slow-Mode Failure Permanently Locks User Deposit Funds With No Retry Path - (File: `core/contracts/Endpoint.sol`)

---

### Summary

In `Endpoint._executeSlowModeTransaction`, the slow-mode transaction record is **deleted and its index bumped before processing begins**. If processing reverts, the catch block silently discards the failure. For `DepositCollateral` slow-mode transactions, the user's tokens have already been physically transferred to the Clearinghouse contract before the slow-mode entry was queued. A silent failure leaves those tokens in the Clearinghouse with no balance credit and no on-chain retry path — a permanent fund lock.

---

### Finding Description

`_executeSlowModeTransaction` unconditionally deletes the queued transaction and increments `txUpTo` on line 194 **before** attempting to execute it:

```solidity
SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
delete slowModeTxs[_slowModeConfig.txUpTo++];   // deleted + index bumped first
``` [1](#0-0) 

Execution is then attempted inside a bare `try/catch`. Any revert that is not an out-of-gas condition is silently swallowed:

```solidity
try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
    if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
        assembly { invalid() }
    }
    // try return funds now removed
}
``` [2](#0-1) 

The comment `// try return funds now removed` is a direct admission that a fund-recovery path was deliberately deleted, leaving no fallback.

For `DepositCollateral` slow-mode transactions, the token custody transfer happens **at submission time** inside `depositCollateralWithReferral`, before the slow-mode entry is even queued:

```solidity
handleDepositTransfer(
    IERC20Base(spotEngine.getToken(productId)),
    msg.sender,
    uint256(amount)
);
// ... then queue slow-mode tx
slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({ ... });
``` [3](#0-2) 

`handleDepositTransfer` moves tokens from the user all the way to the Clearinghouse contract in the same call:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    safeTransferFrom(token, from, amount);           // user → Endpoint
    safeTransferTo(token, address(clearinghouse), amount); // Endpoint → Clearinghouse
}
``` [4](#0-3) 

When the slow-mode tx is later processed, `clearinghouse.depositCollateral` is called to credit the user's internal balance. If that call reverts (e.g., the product was delisted in the intervening 3-day window via a `DelistProduct` slow-mode tx), the inner call's state changes are rolled back — but the tokens already sitting in the Clearinghouse are **not** rolled back, because the outer frame already committed the custody transfer. The slow-mode entry is gone (`txUpTo` was bumped), so there is no on-chain mechanism to retry or recover.

The same pattern applies to `DepositInsurance` slow-mode transactions, where `handleDepositTransfer` is called inside `submitSlowModeTransactionImpl` at submission time. [5](#0-4) 

---

### Impact Explanation

User deposit tokens are permanently locked inside the Clearinghouse contract. The user's internal balance is never credited, and there is no `rescue`, `recoverFunds`, or retry function accessible to the user or even to the owner. The only path to recovery would require an emergency contract upgrade. The asset delta is the full deposited amount for every `DepositCollateral` or `DepositInsurance` slow-mode transaction that fails silently.

---

### Likelihood Explanation

The `SLOW_MODE_TX_DELAY` is hardcoded to 3 days. [6](#0-5) 

During that window, a `DelistProduct` slow-mode tx (owner-only) can be queued and processed before the deposit tx, causing `clearinghouse.depositCollateral` to revert on a now-invalid product. Additionally, any future revert condition introduced in `clearinghouse.depositCollateral` (e.g., a new sanction check, a product-state guard) would silently trigger this path for all in-flight deposits. The 3-day window and the existence of a `DelistProduct` admin action make this a realistic, non-theoretical scenario.

---

### Recommendation

1. **Process before deleting**: Read and attempt to process the slow-mode transaction before deleting it and bumping `txUpTo`. Only advance the index on success, or implement a separate "failed" queue.
2. **Restore a fund-return path**: The removed `// try return funds now removed` logic should be reinstated. On failure, the Clearinghouse should be called to return the already-transferred tokens to the original depositor.
3. **Emit a failure event**: At minimum, emit an indexed event on silent failure so that off-chain monitoring can detect and manually remediate stuck deposits.

---

### Proof of Concept

1. User calls `depositCollateralWithReferral(subaccount, productId=X, amount=1000e6)`.
2. `handleDepositTransfer` moves 1000 USDC: user → Endpoint → Clearinghouse. Slow-mode tx queued at index `N` with `executableAt = now + 3 days`.
3. Owner submits a `DelistProduct` slow-mode tx for product X. It is processed before the deposit tx (owner can execute slow-mode txs immediately via the sequencer path).
4. Three days pass. Anyone calls `executeSlowModeTransaction()` (or the sequencer includes `ExecuteSlowMode`).
5. `_executeSlowModeTransaction` runs: `delete slowModeTxs[N]`, `txUpTo = N+1`.
6. `try this.processSlowModeTransaction(sender, tx)` → `clearinghouse.depositCollateral` reverts because product X is delisted.
7. Catch block fires; `gasleft()` check passes; execution continues silently. `// try return funds now removed`.
8. 1000 USDC sits in the Clearinghouse. User's internal balance: 0. Slow-mode entry at index N: deleted. No retry possible. Funds permanently locked. [7](#0-6)

```text
    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
```
