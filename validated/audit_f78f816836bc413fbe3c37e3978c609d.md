### Title
Liquidation Discount Extraction Deepens Insolvency on Already-Insolvent Subaccounts - (`File: core/contracts/ClearinghouseLiq.sol`)

---

### Summary

When a liquidatee's **PNL health** (absolute total value = `spotAmount × oraclePrice + quoteBalance`) is already negative, `_handleLiquidationPayment` still applies a full liquidation discount to the price paid by the liquidator. The liquidator extracts the discount as profit, while only 50% of it is returned to the insurance fund as fees. Each liquidation event on an already-insolvent subaccount therefore deepens the protocol's net deficit by `(oraclePrice − liquidationPrice) × amount × 0.5`.

---

### Finding Description

The liquidation price is computed in `getLiqPriceX18`:

```
penaltyX18 = (maintenanceWeight − ONE) / 5
liquidationPrice = oraclePrice × (ONE + penaltyX18)
```

For a long spot position with `maintenanceWeight < 1` (e.g., 0.9), `penaltyX18` is negative, so `liquidationPrice < oraclePrice`. The liquidator buys the collateral below oracle price. [1](#0-0) 

In `_handleLiquidationPayment` (spot branch):

```
liquidationPayment = liquidationPriceX18 × amount   // liquidatee receives this
liquidationFees    = (oraclePriceX18 − liquidationPriceX18) × LIQUIDATION_FEE_FRACTION × amount
// liquidator pays: liquidationPayment + liquidationFees
// insurance gets: liquidationFees  (50% of the discount)
// liquidator keeps: the other 50% of the discount
``` [2](#0-1) [3](#0-2) 

The only eligibility gate is `isUnderMaintenance` (maintenance health `< 0`). There is **no check on PNL health** (i.e., whether `spotAmount × oraclePrice + quoteBalance` is already negative). [4](#0-3) 

For positive spot amounts, `_assertLiquidationAmount` sets `maxLiquidatable = spotAmount` with no constraint tied to the liquidatee's remaining total value: [5](#0-4) 

The post-liquidation check only prevents over-liquidation (health going above initial), not liquidation when PNL health is already negative: [6](#0-5) 

---

### Impact Explanation

**Concrete numerical scenario** (maintenanceWeight = 0.9, `MIN_NON_SPREAD_LIQ_PENALTY_X18 = 0.5%`):

| State | Spot (units) | Oracle Price | Quote Balance | PNL Health (Total Value) |
|---|---|---|---|---|
| Before liquidation | 100 | $0.84 | −$85 | **−$1.00** (already insolvent) |
| After full liquidation | 0 | — | −$85 + $82.32 = **−$2.68** | **−$2.68** |

- Liquidation price = $0.84 × (1 − 0.02) = **$0.8232**
- Liquidatee receives $82.32; deficit grows from $1.00 → $2.68
- Liquidator extracts $0.84 of discount; insurance receives $0.84 (50%), but the net system deficit still grows by **$0.84**

The insurance fund is credited `liquidationFees` at line 579, but this is sourced from the liquidator's payment — it does not offset the value destroyed by the discount applied to the liquidatee's collateral. [7](#0-6) 

Because maintenance health remains negative after a partial liquidation, **repeated liquidation calls are permitted**, each one extracting another discount tranche and driving the deficit deeper. The insurance fund is eventually drained via `_finalizeSubaccount` → `perpEngine.socializeSubaccount` / `spotEngine.socializeSubaccount`. [8](#0-7) 

---

### Likelihood Explanation

This condition arises in any fast-moving market where the asset price overshoots the maintenance-health threshold and enters the PNL-negative zone before a liquidator acts. The gap between "maintenance health = 0" and "PNL health = 0" is exactly `spotAmount × oraclePrice × (1 − maintenanceWeight)`. For a 10% maintenance haircut on a $10 000 position, this gap is only $1 000 — easily bridged in a single volatile candle. Any unprivileged liquidator can then call `liquidateSubaccountImpl` through the `Endpoint` → `Clearinghouse.liquidateSubaccount` → `delegatecall` to `ClearinghouseLiq` path and trigger the insolvency-deepening sequence. [9](#0-8) 

---

### Recommendation

Before executing `_handleLiquidationPayment` for positive spot amounts, compute the liquidatee's PNL health. If it is already negative, either:

1. **Skip the discount entirely** and transfer the spot at oracle price (no liquidator incentive), proceeding directly to socialization; or
2. **Cap the liquidation discount** so that `liquidationPrice × amount ≥ quoteBalance.abs()`, ensuring the liquidation does not worsen the net deficit beyond what the insurance fund already covers.

Additionally, add a protocol-level invariant assertion (analogous to `manualAssert`) that verifies the insurance fund balance is sufficient to cover the aggregate PNL-negative exposure across all subaccounts after each liquidation batch.

---

### Proof of Concept

**Attacker-controlled entry path:**

```
User (liquidator) signs LiquidateSubaccount tx
  → Endpoint.submitTransactions (onlySequencer, but sequencer is permissionless relay)
  → Clearinghouse.liquidateSubaccount (line 644)
      → delegatecall ClearinghouseLiq.liquidateSubaccountImpl (line 658)
          → isUnderMaintenance(liquidatee) == true  ✓  (price dropped past maintenance threshold)
          → _assertLiquidationAmount: maxLiquidatable = spotAmount (no PNL check)
          → _handleLiquidationPayment:
              liquidationPrice = oraclePrice × 0.98  (2% below oracle)
              liquidatee.quote += liquidationPrice × amount   // less than oracle value
              liquidator.quote -= liquidationPrice × amount + fees
              insurance        += fees                        // only 50% of discount
          → require(!isAboveInitial(liquidatee))  ✓  (still under initial health)
          // PNL health is now MORE negative than before — insolvency deepened
```

**State delta corrupted:** `insurance` (ClearinghouseStorage.sol line 23) is depleted faster than necessary because each liquidation on a PNL-negative subaccount extracts `(oraclePrice − liquidationPrice) × amount × 0.5` net value from the system, which the insurance fund must ultimately absorb during finalization.     function liquidateSubaccountImpl(IEndpoint.LiquidateSubaccount calldata txn)
        external
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.sender != txn.liquidatee, ERR_UNAUTHORIZED);
        require(isUnderMaintenance(txn.liquidatee), ERR_NOT_LIQUIDATABLE);
        require(
            txn.liquidatee != X_ACCOUNT && txn.liquidatee != N_ACCOUNT,
            ERR_NOT_LIQUIDATABLE
        );
        require(
            txn.productId != QUOTE_PRODUCT_ID,
            ERR_INVALID_LIQUIDATION_PARAMS
        );

        ISpotEngine spotEngine = ISpotEngine(
            address(engineByType[IProductEngine.EngineType.SPOT])
        );
        IPerpEngine perpEngine = IPerpEngine(
            address(engineByType[IProductEngine.EngineType.PERP])
        );

        if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
            if (RiskHelper.isIsolatedSubaccount(txn.liquidatee)) {
                IOffchainExchange(
                    IEndpoint(getEndpoint()).getOffchainExchange()
                ).tryCloseIsolatedSubaccount(txn.liquidatee);
            }
            return;
        }

        if (
            (txn.amount < 0) &&
            (txn.isEncodedSpread ||
                address(productToEngine[txn.productId]) == address(spotEngine))
        ) {
            // when it's spread or spot liquidation, we need to make sure the liquidatee has
            // enough quote to buyback the liquidated amount.
            _assertCanLiquidateLiability(txn, spotEngine, perpEngine);
            _settlePositivePerpPnl(txn, spotEngine, perpEngine);
        }

        _assertLiquidationAmount(txn, spotEngine, perpEngine);

        // beyond this point, we can be sure that we can liquidate the entire
        // liquidation amount knowing that the insurance fund will remain solvent
        // subsequently we can just blast the remainder of the liquidation and
        // cover the quote balance from the insurance fund at the end
        _handleLiquidationPayment(txn, spotEngine, perpEngine);
    }
