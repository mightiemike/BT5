### Title
Deprecated `verify_transaction_inclusion` Remains Callable as a NEAR Contract Method, Enabling 64-Byte Merkle Proof Forgery - (File: `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` was deprecated in favour of `verify_transaction_inclusion_v2` specifically to close the 64-byte transaction Merkle proof forgery vulnerability. However, Rust's `#[deprecated]` attribute is a compile-time lint only — it has no runtime effect and does not remove the function from the NEAR contract's ABI. Any unprivileged NEAR caller can invoke `verify_transaction_inclusion` directly, bypassing the coinbase-proof guard added in v2, and obtain a `true` result for a non-existent transaction.

### Finding Description
`verify_transaction_inclusion` is annotated with `#[deprecated]` and documented as vulnerable to the 64-byte transaction Merkle proof forgery attack. The fix was implemented in `verify_transaction_inclusion_v2`, which first validates a coinbase Merkle proof before delegating to v1. [1](#0-0) 

The `#[deprecated]` attribute only causes the Rust compiler to emit a warning when *Rust code* calls the function. It does not:
- Remove the function from the compiled WASM binary
- Remove the function from the contract's public ABI
- Prevent any external NEAR account from calling it via a transaction

Because the function is `pub` inside a `#[near]` impl block, it is unconditionally exported as a callable contract method. [2](#0-1) 

`verify_transaction_inclusion_v2` adds the coinbase proof check that closes the forgery window, but this guard is entirely skippable by calling v1 directly: [3](#0-2) 

The 64-byte attack works because Bitcoin's Merkle tree uses the same double-SHA256 hash for both leaf nodes (real transactions) and internal nodes. A 64-byte value can be interpreted as two concatenated 32-byte child hashes of an internal node. An attacker who knows the Merkle tree of any confirmed block can pick an internal node whose hash equals a value they control, supply a valid Merkle path to the block's `merkle_root`, and have the contract return `true` for a transaction that was never broadcast.

### Impact Explanation
Any consumer contract (bridge, atomic swap, cross-chain lending protocol — all listed as target consumers in the README) that calls `verify_transaction_inclusion` to gate a fund release or state transition can be deceived into accepting a forged proof. The attacker receives a `true` result for a fabricated `tx_id` that corresponds to an internal Merkle node, not a real transaction. This corrupts the proof-verification result that the entire SPV security model depends on. [4](#0-3) 

### Likelihood Explanation
The entry path requires no privilege: any NEAR account can call `verify_transaction_inclusion` with a crafted `ProofArgs`. The Merkle tree of every confirmed Bitcoin block is public. Computing a valid internal-node forgery requires only off-chain arithmetic over publicly available block data. The function is permanently reachable as long as the current WASM is deployed.

### Recommendation
Remove `verify_transaction_inclusion` from the public ABI entirely. The simplest correct fix is to change its visibility from `pub fn` to `pub(crate) fn` (or remove the `pub` qualifier), which prevents the NEAR SDK macro from exporting it as a callable contract method. Alternatively, add a hard `env::panic_str("use verify_transaction_inclusion_v2")` as the first statement so any direct call aborts immediately. Relying solely on `#[deprecated]` is insufficient because it has no runtime enforcement.

### Proof of Concept
1. Obtain any confirmed Bitcoin block whose headers are stored in the contract (e.g., block at height H).
2. Compute the full Merkle tree of that block's transactions off-chain.
3. Select any internal node `N` at depth `d` with left child `L` and right child `R`. The 64-byte concatenation `L || R` is the "fake transaction" whose double-SHA256 equals `N`.
4. Build a valid Merkle path from `N` up to `merkle_root`.
5. Call `verify_transaction_inclusion` with:
   - `tx_id` = `N` (the internal node hash, not a real txid)
   - `tx_block_blockhash` = the block hash at height H
   - `tx_index` = the index corresponding to `N`'s position in the tree
   - `merkle_proof` = the path from `N` to `merkle_root`
   - `confirmations` = 1
6. The contract returns `true`.

`verify_transaction_inclusion_v2` would reject this call at the coinbase-proof check (line 358–365), but that guard is entirely bypassed by calling v1 directly. [5](#0-4)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    #[pause]
    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
