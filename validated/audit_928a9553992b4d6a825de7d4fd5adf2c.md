### Title
Attacker-Controlled `_cached_sha256_treehash` on `CLVMStorage` Objects Is Trusted Unconditionally, Corrupting `Program.tree_hash()` - (File: `wheel/python/clvm_rs/tree_hash.py`, `wheel/python/clvm_rs/program.py`)

---

### Summary

The Python layer of `clvm_rs` caches tree hashes in a `_cached_sha256_treehash` attribute on `CLVMStorage` objects. Both `sha256_treehash()` and `Program.wrap()` read this attribute from any caller-supplied object and treat it as authoritative without recomputing or verifying it. An attacker who can supply a `CLVMStorage`-compatible Python object with a pre-set `_cached_sha256_treehash` can inject an arbitrary 32-byte value that `Program.tree_hash()` will return verbatim, corrupting puzzle-hash comparisons, coin-ID derivation, and curried-puzzle-hash computation.

---

### Finding Description

`CLVMStorage` is a structural Python `Protocol` defined in `wheel/python/clvm_rs/clvm_storage.py`. It requires only `.atom` and `.pair`; `_cached_sha256_treehash` is explicitly documented as an **optional** performance field:

```python
# optional fields used to speed implementations:
# `_cached_sha256_treehash: Optional[bytes]` is used by `sha256_treehash`
``` [1](#0-0) 

`Treehasher.sha256_treehash()` in `tree_hash.py` reads this attribute from every node it visits and, if present, **immediately returns it as the hash without any recomputation or validation**:

```python
r = getattr(obj, "_cached_sha256_treehash", None)
if r is not None:
    self.cache_hits += 1
    hash_stack.append(r)
    return
``` [2](#0-1) 

`Program.wrap()` propagates the same attribute from any caller-supplied `CLVMStorage` object directly into the new `Program` instance:

```python
o._cached_sha256_treehash = getattr(v, "_cached_sha256_treehash", None)
``` [3](#0-2) 

`Program.tree_hash()` then returns the cached value without ever calling `sha256_treehash`:

```python
def tree_hash(self) -> bytes:
    if self._cached_sha256_treehash is None:
        self._cached_sha256_treehash = sha256_treehash(self._unwrapped)
    return self._cached_sha256_treehash
``` [4](#0-3) 

The broken invariant is: `Program.tree_hash()` is supposed to equal `sha256tree(program_content)` for all inputs. Because the cache is seeded from an unverified external attribute, this invariant is violated whenever a caller-supplied object carries a wrong `_cached_sha256_treehash`.

---

### Impact Explanation

`Program.tree_hash()` is the canonical puzzle-hash function in the Chia wallet and validator stack. It is used in three security-critical paths:

1. **Equality / identity checks** — `Program.__eq__` compares programs exclusively via `tree_hash()`:

   ```python
   return self.tree_hash() == other_obj.tree_hash()
   ``` [5](#0-4) 

   A spoofed hash makes two structurally different programs compare equal, or two identical programs compare unequal.

2. **Curried puzzle-hash derivation** — `Program.curry_hash()` feeds `self.tree_hash()` into the curry-and-treehash pipeline:

   ```python
   quoted_mod_hash = curry_treehasher.calculate_hash_of_quoted_mod_hash(
       self.tree_hash()
   )
   return curry_treehasher.curry_and_treehash(quoted_mod_hash, *args)
   ``` [6](#0-5) 

   A wrong `tree_hash()` produces a wrong puzzle hash, which produces a wrong coin ID. Wallets that derive coin IDs this way will fail to locate or spend coins, or will compute coin IDs that match attacker-chosen values.

3. **Serialization-based execution** — `Program.__bytes__` serializes `self` and the result is passed to `run_serialized_chia_program`. If a caller uses `tree_hash()` to decide *which* program to run (e.g., to look up a cached compiled puzzle), a spoofed hash causes the wrong program to be executed.

---

### Likelihood Explanation

The `CLVMStorage` protocol is explicitly designed to be implemented by third-party code. Any application that:

- Accepts `CLVMStorage` objects from external sources (plugins, user-provided puzzles, deserialized JSON/YAML representations), **or**
- Passes objects through a pipeline where one stage sets `_cached_sha256_treehash` before the content is modified

is directly exposed. The `Program.to()` entry point is the standard conversion path and accepts any `CLVMStorage`-compatible object without restriction. No special privileges or internal access are required; a caller only needs to set a Python attribute on an object before passing it to `Program.to()` or `Program.wrap()`.

---

### Recommendation

1. **Remove unconditional cache trust in `sha256_treehash`.** Do not read `_cached_sha256_treehash` from externally supplied objects. Only write it after computing the correct hash internally.

2. **In `Program.wrap()`, do not copy `_cached_sha256_treehash` from the source object.** Leave `_cached_sha256_treehash = None` on the new `Program` so it is computed on first access.

3. **If caching from external objects is desired for performance**, verify the cached value against a freshly computed hash before accepting it, or restrict caching to objects that were produced by `clvm_rs` itself (e.g., instances of `Program` or `LazyNode`).

### Proof of Concept

```python
from clvm_rs.program import Program

# Craft a CLVMStorage-compatible object whose atom is b"foo"
# but whose _cached_sha256_treehash is an attacker-chosen value.
class MaliciousStorage:
    atom = b"foo"
    pair = None
    _cached_sha256_treehash = b"\xde\xad" * 16  # 32 bytes of attacker data

evil = MaliciousStorage()
p = Program.to(evil)

# Program.wrap() copies the attacker-supplied hash verbatim.
# Program.tree_hash() returns it without recomputing.
assert p.tree_hash() == b"\xde\xad" * 16  # attacker-controlled hash returned

# The correct hash for atom b"foo" is:
correct = Program.new_atom(b"foo").tree_hash()
assert p.tree_hash() != correct  # invariant violated

# Downstream: equality check is now spoofable
p2 = Program.new_atom(b"bar")
evil2 = MaliciousStorage()
evil2.atom = b"bar"
evil2._cached_sha256_treehash = b"\xde\xad" * 16  # same fake hash
p3 = Program.to(evil2)
# p and p3 have different content but compare equal
assert p == p3  # True — consensus identity check is broken
```
