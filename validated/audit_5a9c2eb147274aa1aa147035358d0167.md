### Title
Fork Difficulty Retargeting Uses Current Main-Chain Block Instead of Fork Ancestor — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

When validating a fork block at a difficulty retarget boundary, `get_next_work_required` fetches the retarget interval's first block via `get_header_by_height`, which always reads from `mainchain_height_to_header` — the **current main chain** index. If the fork diverged before the retarget interval's start height, the fork's ancestor at that height has a different timestamp than the main-chain block stored there. An unprivileged `submit_blocks` caller can exploit this by setting the fork ancestor's timestamp to be much later than the main-chain block's timestamp, causing the contract to compute a lower difficulty than the fork actually requires, and thereby accepting fork blocks with insufficient proof-of-work.

---

### Finding Description

**Root cause — `get_header_by_height` always reads the current main chain:** [1](#0-0) 

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // ← current main chain only
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

Fork blocks are stored in `headers_pool` but **never** inserted into `mainchain_height_to_header`. Therefore, for any fork that diverged before the retarget interval's start height, `get_header_by_height(first_block_height)` silently returns the **main-chain** block at that height instead of the fork's ancestor.

**Bitcoin / Litecoin retarget path:** [2](#0-1) 

```rust
let first_block_height =
    prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(config, prev_block_header,
    interval_tail_extend_header.block_header.time.into())
```

**How the exploit works (Dogecoin, `difficulty_adjustment_interval = 1`):**

Every Dogecoin block is a retarget boundary, so `height_first = prev_block_height − 1`. An attacker needs only two fork blocks:

1. Submit fork block at height H (diverges from main chain). Set its timestamp to `T_late ≈ now + 2 h` (the maximum allowed by `MAX_FUTURE_BLOCK_TIME_LOCAL`).
2. Submit fork block at height H+1 (the retarget boundary).

During step 2, the contract computes:

```
actual_time_taken = fork_block_H.time − mainchain_block[H-1].time
                  = T_late − T_main_early
```

`T_main_early` is the honest main-chain block's timestamp (mined ~1 minute before H). `T_late` can be hours later. The Dogecoin damping formula clamps `modulated_timespan` to `[45 s, 90 s]`, so the difficulty is reduced by up to **33 %** relative to what the fork's own ancestor would have produced.

For **Bitcoin / Litecoin** the clamp is `[pow_target_timespan/4, pow_target_timespan*4]`, allowing up to a **4× reduction** in the required target (75 % easier PoW) for fork blocks at every 2016-block retarget boundary. [4](#0-3) 

```rust
if actual_time_taken < config.pow_target_timespan / 4 {
    actual_time_taken = config.pow_target_timespan / 4;
}
if actual_time_taken > config.pow_target_timespan * 4 {
    actual_time_taken = config.pow_target_timespan * 4;
}
```

---

### Impact Explanation

The difficulty check is the primary PoW security gate. Accepting fork blocks with a lower-than-required difficulty target means an attacker can accumulate chain work faster than the honest chain with less real computation. Once the fork's `chain_work` exceeds the main chain's, `submit_block_header_inner` triggers `reorg_chain`, replacing the canonical chain. This corrupts `mainchain_tip_blockhash`, `mainchain_height_to_header`, and all downstream `verify_transaction_inclusion` results — the core security guarantee of the light client. [5](#0-4) 

---

### Likelihood Explanation

- **Dogecoin**: Every block is a retarget boundary. Only two fork blocks are needed. Exploitable by any caller who can reach `submit_blocks`.
- **Bitcoin / Litecoin**: Requires submitting a fork that spans a 2016-block retarget boundary. Still reachable by any relayer; the reduced difficulty (up to 4×) partially offsets the cost.
- Entry point is the public `submit_blocks` method; no privileged role is required beyond whatever the `trusted_relayer` staking mechanism demands (which is open to any staker). [6](#0-5) 

---

### Recommendation

Replace the `get_header_by_height` call inside `get_next_work_required` with an ancestor walk that follows `prev_block_hash` links through `headers_pool` back to `first_block_height`. This mirrors what Bitcoin Core does and ensures the retarget uses the block that is actually an ancestor of the fork tip, not whatever block the current main chain happens to have at that height.

```rust
// Walk back through headers_pool to find the true fork ancestor
let mut cursor = prev_block_header.clone();
while cursor.block_height > first_block_height {
    cursor = blocks_getter.get_prev_header(&cursor.block_header);
}
// cursor is now the fork's ancestor at first_block_height
let first_block_time = cursor.block_header.time;
```

This fix must be applied consistently in `bitcoin.rs`, `litecoin.rs`, and `dogecoin.rs`.

---

### Proof of Concept

**Dogecoin (minimal, 2 fork blocks):**

1. Main chain has blocks 0 … N with honest timestamps.
2. Attacker calls `submit_blocks` with fork block at height N (prev = main block N−1). Sets `time = now + 7200` (2 h ahead, within `MAX_FUTURE_BLOCK_TIME_LOCAL`).
3. Attacker calls `submit_blocks` with fork block at height N+1 (prev = fork block N).
4. Inside `get_next_work_required`, `height_first = N − 1`. `get_header_by_height(N−1)` returns the main-chain block mined ~1 minute before N, with timestamp `T_main`.
5. `modulated_timespan = (now + 7200) − T_main` → clamped to 90 s → difficulty reduced by 33 %.
6. The fork block at N+1 is accepted with a target 33 % easier than the fork's own ancestor would have required.
7. Repeat to accumulate chain work and eventually trigger `reorg_chain`. [7](#0-6) [8](#0-7)
