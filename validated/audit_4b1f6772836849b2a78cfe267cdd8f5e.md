### Title
`MsgUpdateTier` Permanently Blocked When Rewards Pool Is Insufficient to Cover All Tier Positions' Accrued Bonus — (`x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`MsgUpdateTier` calls `claimRewardsAndUpdateTierPositions`, which iterates over every delegated position in a tier and attempts to pay out each position's accrued bonus. If the rewards pool is insufficient to cover any single position's bonus, the entire loop reverts with `ErrInsufficientBonusPool`, causing the governance transaction to fail atomically. This creates a permanent deadlock: governance cannot lower the APY (to stop pool drain) because doing so requires settling all accrued bonuses first, but the pool is empty.

---

### Finding Description

In `MsgUpdateTier`, when `BonusApy` changes, the handler unconditionally calls `claimRewardsAndUpdateTierPositions` before writing the new tier: [1](#0-0) 

`claimRewardsAndUpdateTierPositions` iterates over every position ID in the tier and calls `processEventsAndClaimBonus` for each delegated position: [2](#0-1) 

Inside `processEventsAndClaimBonus`, before transferring bonus coins, `sufficientBonusPoolBalance` is checked and returns `ErrInsufficientBonusPool` if the pool cannot cover the position's accrued bonus: [3](#0-2) [4](#0-3) 

This error propagates back through `claimRewardsAndUpdateTierPositions` and out of `UpdateTier` without any special handling. The entire governance transaction reverts.

The contrast with the `slashRedelegationPosition` hook is instructive — that hook explicitly catches `ErrInsufficientBonusPool` and continues (to avoid chain halt): [5](#0-4) 

`claimRewardsAndUpdateTierPositions` has no equivalent guard.

The existing test `TestUpdateTier_BonusApyChange_InsufficientPool` confirms this failure mode is real and reproducible: [6](#0-5) 

---

### Impact Explanation

When the rewards pool is depleted and any position in the tier has accrued non-zero bonus, every `MsgUpdateTier` call that changes `BonusApy` will revert. This creates a circular deadlock:

- Pool is empty → governance wants to reduce APY to zero to stop further drain → reducing APY requires settling all accrued bonuses → pool is empty → governance cannot reduce APY.

Governance is permanently locked out of adjusting the tier's `BonusApy` until the pool is replenished to cover the **total** accrued bonus across all positions simultaneously. With many long-running positions, this threshold can be very large. The corrupted invariant is: governance authority over tier parameters is rendered inoperable by a pool-balance condition that governance itself cannot resolve without first exercising the blocked operation.

---

### Likelihood Explanation

The rewards pool is an externally funded module account. It is drained by:
1. The `BeginBlocker` (`topUpBaseRewards`) transferring shortfall amounts every block when `TargetBaseRewardsRate > 0`.
2. Users claiming bonus rewards via `MsgClaimTierRewards` and other settlement paths.

Pool depletion is a normal operational scenario, not an edge case. Once the pool is low and positions have accrued bonus (which grows continuously while delegated to a bonded validator), any governance proposal to change `BonusApy` will fail at execution. The governance proposal passes the vote but the execution reverts, leaving the tier's APY frozen.

---

### Recommendation

In `claimRewardsAndUpdateTierPositions`, handle `ErrInsufficientBonusPool` the same way `slashRedelegationPosition` does: log the error and skip the position rather than propagating the error. Alternatively, decouple the "settle all bonuses" step from the APY update so that governance can change `BonusApy` without requiring the pool to be solvent for all positions simultaneously. A separate `MsgSettleTierBonuses` message could allow incremental settlement before the APY change.

---

### Proof of Concept

1. Create N positions in tier T with large lock amounts; let bonus accrue over time.
2. Allow the rewards pool to drain (BeginBlocker drains it each block when `TargetBaseRewardsRate > 0`, or users claim rewards).
3. Submit a governance proposal: `MsgUpdateTier` with a new `BonusApy` (e.g., reduce from 6% to 0%).
4. The proposal passes the vote.
5. At execution, `UpdateTier` calls `claimRewardsAndUpdateTierPositions` → iterates positions → first position with non-zero accrued bonus hits `sufficientBonusPoolBalance` → returns `ErrInsufficientBonusPool` → entire transaction reverts.
6. The tier's `BonusApy` remains unchanged. Governance cannot retry successfully until the pool is replenished to cover the sum of all positions' accrued bonuses — which itself grows larger every block the old APY remains in effect.

```go
func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
```

```go
func (s *KeeperSuite) TestUpdateTier_BonusApyChange_InsufficientPool() {
	lockAmount := sdkmath.NewInt(sdk.DefaultPowerReduction.Int64())
	pos := s.setupNewTierPosition(lockAmount, false)
	valAddr := sdk.MustValAddressFromBech32(pos.Delegation.ValidatorAddress)

	s.setValidatorCommission(valAddr, sdkmath.LegacyZeroDec())

	// Advance time so bonus accrues.
	s.ctx = s.ctx.WithBlockHeight(s.ctx.BlockHeight() + 1)
	s.ctx = s.ctx.WithBlockTime(s.ctx.BlockTime().Add(30 * 24 * time.Hour))

	// Update tier with new BonusApy — should fail due to insufficient pool.
	tier := newTestTier(1)
	initialBonusApy := tier.BonusApy
	updatedBonusApy := sdkmath.LegacyNewDecWithPrec(8, 2)
	tier.BonusApy = updatedBonusApy
	msgServer := keeper.NewMsgServerImpl(s.keeper)
	_, err := msgServer.UpdateTier(s.ctx, &types.MsgUpdateTier{
		Authority: s.keeper.GetAuthority(),
		Tier:      tier,
	})
	s.Require().ErrorIs(err, types.ErrInsufficientBonusPool)

	// Tier should NOT have been updated (tx failed).
	got, err := s.keeper.GetTier(s.ctx, 1)
	s.Require().NoError(err)
	s.Require().True(initialBonusApy.Equal(got.BonusApy), "tier should still have old APY")
```
