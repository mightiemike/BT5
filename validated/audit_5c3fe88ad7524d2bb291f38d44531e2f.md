### Title
Linked Signer Can Overwrite Subaccount's Linked Signer Without Owner Consent тАФ (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`. This means the **current linked signer** тАФ a delegated, non-owner key тАФ can sign a `LinkSigner` transaction to replace itself with any arbitrary address. The subaccount owner's consent is not required. This is a broken access control: a delegated party can unilaterally escalate or reassign signing authority over a subaccount it does not own.

---

### Finding Description

**Root cause тАФ `processTransactionImpl`, lines 576тАУ590:**

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // тЖР allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
```

`validateSignedTx` with `allowLinkedSigner = true` resolves to `validateSignature(sender, digest, sig, getLinkedSigner(sender))`, which accepts a valid signature from **either** the subaccount owner **or** the current linked signer. Because the transaction being validated is itself a `LinkSigner` mutation, the current linked signer can sign a transaction that overwrites `linkedSigners[subaccount]` with any address тАФ including one the subaccount owner has never seen.

**Contrast with the slow-mode path тАФ `processSlowModeTransactionImpl`, lines 232тАУ239:**

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.LinkSigner)
    );
    validateSender(txn.sender, sender);   // тЖР msg.sender must equal subaccount address
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
```

The slow-mode path correctly enforces that only the subaccount owner (`msg.sender == address(uint160(bytes20(txn.sender)))`) can change the linked signer. The fast (sequencer) path has no equivalent ownership gate.

**Broken invariant:** Only the subaccount owner should be able to mutate `linkedSigners[subaccount]`. The fast path violates this by accepting the current linked signer's signature as sufficient authorization for that mutation.

---

### Impact Explanation

Once a linked signer replaces itself with an attacker-controlled address, the attacker's new key is accepted as a valid signer for every transaction type that passes `allowLinkedSigner = true`:

| Transaction | `allowLinkedSigner` | Consequence |
|---|---|---|
| `WithdrawCollateral` | `true` | Initiates withdrawals (funds go to owner address тАФ no direct theft, but drains collateral) |
| `WithdrawCollateralV2` (sendTo == 0) | `true` | Same as above |
| `MintNlp` / `BurnNlp` | `true` | Manipulates victim's NLP position; burn fee destroys value |
| `LiquidateSubaccount` | `true` | Attacker liquidates third parties using victim's subaccount as liquidator, collecting fees |
| `MatchOrders` (via `getLinkedSignerOrNlpSigner`) | `true` | Attacker submits adversarial orders on behalf of victim, causing trading losses |
| `LinkSigner` (again) | `true` | Attacker can re-link to a fresh address each time the owner revokes, maintaining persistent access |

The most severe consequence is persistent, irrevocable-in-practice control: every time the owner revokes via slow mode (3-day delay), the attacker re-links before the revocation is processed. The attacker can continuously sign `MatchOrders` to cause financial losses to the victim's subaccount through adversarial trades.

**Impact: High** тАФ unauthorized persistent signing authority over victim subaccount; financial loss through trading manipulation and NLP burn fees; no direct collateral theft but effective account takeover.

---

### Likelihood Explanation

**Precondition:** The attacker must already hold the current linked signer key for the target subaccount. This is realistic because:
- Linked signers are commonly set to automated trading bots, API keys, or third-party services.
- A compromised linked signer service (e.g., a hacked trading bot) immediately enables this attack across all subaccounts it serves.
- The attacker does not need any privileged protocol access тАФ only possession of a key the victim previously authorized.

The nonce required for the `LinkSigner` transaction is public on-chain state (`nonces[address]`), so no additional information is needed.

**Likelihood: Medium** тАФ requires prior linked signer compromise, but that is a realistic and common threat model for DeFi trading infrastructure.

---

### Recommendation

Set `allowLinkedSigner = false` for the `LinkSigner` transaction type in `processTransactionImpl`. Only the subaccount owner's key should be accepted as authorization to mutate `linkedSigners`. This aligns the fast path with the already-correct slow-mode path.

```solidity
// BEFORE (vulnerable)
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // linked signer can overwrite itself
);

// AFTER (fixed)
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only subaccount owner can change linked signer
);
```

### Proof of Concept

1. Victim (`0xVICTIM`) owns subaccount `0xVICTIM000000000000000000000000000000000000000000000000000000000001`.
2. Victim sets `linkedSigners[subaccount] = 0xATTACKER_KEY_1` (e.g., for automated trading).
3. Attacker, holding `0xATTACKER_KEY_1`, constructs a `SignedLinkSigner` transaction:
   - `tx.sender = 0xVICTIM...0001`
   - `tx.signer = 0xATTACKER_KEY_2` (fresh attacker-controlled address)
   - `tx.nonce = nonces[0xVICTIM]` (read from public chain state)
   - `signature` = EIP-712 signature over the digest, signed by `0xATTACKER_KEY_1`
4. Attacker submits this to the sequencer. The sequencer calls `processTransactionImpl`.
5. `validateSignedTx(..., true)` accepts the signature because `recovered == getLinkedSigner(subaccount) == 0xATTACKER_KEY_1`. тЬУ
6. `linkedSigners[subaccount]` is now `0xATTACKER_KEY_2`.
7. Victim revokes via slow mode (3-day delay). Before the slow-mode tx executes, attacker repeats step 3тАУ6 with `0xATTACKER_KEY_2 тЖТ 0xATTACKER_KEY_3`.
8. Attacker maintains persistent signing authority. Using `0xATTACKER_KEY_N`, attacker signs `MatchOrders` transactions that trade against the victim's subaccount at unfavorable prices, draining value. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
