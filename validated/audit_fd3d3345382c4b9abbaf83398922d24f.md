Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of actual swapper, allowing full allowlist bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which `MetricOmmPool.swap()` sets to `msg.sender` — the router contract — not the end user. Because the router must be allowlisted for any legitimate swap to occur, every caller of `MetricOmmSimpleRouter` passes the allowlist check regardless of whether they are individually permitted. Any pool using this extension for curated access control is fully bypassed.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as the first argument to `_beforeSwap()`: [1](#0-0) 

`ExtensionCalling._beforeSwap()` forwards that value unchanged as `sender` to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap(params.recipient, ...)`: [4](#0-3) 

So `sender` = router address, and the check becomes `allowedSwapper[pool][router]`. Since the router must be allowlisted for any legitimate user to trade through it, this condition is always true in any live curated pool, and the check passes for every caller of the router.

The structural asymmetry with `DepositAllowlistExtension` confirms the bug: `beforeAddLiquidity` correctly ignores `sender` (the LiquidityAdder contract) and gates on `owner` (the actual depositor): [5](#0-4) 

`SwapAllowlistExtension` does the opposite — it ignores `recipient` (the actual swap beneficiary) and checks `sender` (the routing contract).

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses is completely defeated. Any unpermitted user who calls `MetricOmmSimpleRouter.exactInputSingle()` (or `exactInput`, `exactOutputSingle`, `exactOutput`) will pass the allowlist check and receive output tokens they should have been denied. This is a direct, fund-impacting curation failure: the pool settles the swap normally, transferring output tokens to the unpermitted user. This matches the "broken core pool functionality causing loss of funds" and "admin-boundary break bypassed by an unprivileged path" allowed impacts.

## Likelihood Explanation
The bypass is reachable through the standard, documented periphery path (`MetricOmmSimpleRouter`). No special privileges, malicious setup, or non-standard tokens are required. The only precondition — that the router is allowlisted — is always satisfied in any live curated pool, since without it no legitimate user could trade. The exploit is repeatable by any user who knows the router address.

## Recommendation
Change `SwapAllowlistExtension.beforeSwap` to gate on `recipient` (the actual swap beneficiary) instead of `sender` (the calling contract), mirroring the correct actor choice made in `DepositAllowlistExtension`:

```solidity
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

This mirrors `DepositAllowlistExtension`, which ignores `sender` and checks `owner`. Update `setAllowedToSwap`, `isAllowedToSwap`, and the `allowedSwapper` mapping semantics accordingly so that pool admins allowlist the recipient address rather than the router.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured on `beforeSwap`.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — only `alice` is permitted.
3. Admin calls `setAllowedToSwap(pool, router, true)` — required so `alice` can trade through the router.
4. `bob` (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool, recipient: bob, ...})`.
5. Router calls `pool.swap(recipient=bob, ...)` → pool sets `sender = router` → calls `extension.beforeSwap(sender=router, recipient=bob, ...)`.
6. Extension checks `allowedSwapper[pool][router]` → `true` → passes.
7. `bob` receives output tokens despite never being individually permitted.

A Foundry integration test can confirm this by deploying the extension, configuring the allowlist as above, and asserting that `bob`'s swap through the router succeeds while a direct `pool.swap()` call from `bob` (where `sender = bob`) correctly reverts.
