Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address as `sender` Instead of Ultimate EOA Swapper, Enabling Allowlist Bypass via `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`MetricOmmPool.swap` passes `msg.sender` (the direct caller of the pool) as the `sender` argument to `_beforeSwap`, so when a swap is routed through `MetricOmmSimpleRouter`, the extension receives the router's address rather than the originating EOA. `SwapAllowlistExtension.beforeSwap` gates access by checking `allowedSwapper[pool][sender]`, meaning it checks whether the router is allowlisted, not whether the ultimate swapper is allowlisted. If the pool admin allowlists the router to permit router-mediated swaps for legitimate users, any unprivileged address can bypass the allowlist entirely by calling through the router.

## Finding Description
**Root cause — `sender` binding in the pool:**

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` at line 231. When the swap originates from `MetricOmmSimpleRouter.exactInputSingle` (or any `exact*` variant), `msg.sender` at the pool level is the router contract, not the originating EOA.

`ExtensionCalling._beforeSwap` forwards this value verbatim as the `sender` argument to every configured extension:
```
abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
```

**Root cause — wrong identity checked in the extension:**

`SwapAllowlistExtension.beforeSwap` checks:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert NotAllowedToSwap();
}
```
Here `msg.sender` is the pool and `sender` is the router address. The check passes if the router is in the allowlist, regardless of who called the router.

**Exploit path:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a set of known counterparties (e.g., `alice`).
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to permit router-mediated swaps for allowlisted users (a natural admin action, since without this, even allowlisted users cannot use the router).
3. Unprivileged attacker (`eve`, not in the allowlist) calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the restricted pool.
4. The router calls `pool.swap(...)` — `msg.sender` at the pool is the router.
5. `beforeSwap` receives `sender = router`, checks `allowedSwapper[pool][router] == true` → passes.
6. Eve's swap executes on a pool she was never authorized to access.

**Existing guards are insufficient:** The only guard is the `allowedSwapper` mapping keyed by the direct caller of `pool.swap()`. There is no mechanism to propagate the originating EOA through the router into the extension check.

## Impact Explanation
An unprivileged trader can execute swaps on pools explicitly configured to restrict swap access. The allowlist's entire purpose — controlling which counterparties can trade against the pool's liquidity — is nullified. Depending on the pool's design (e.g., private OTC pools, KYC-gated pools, or pools with asymmetric pricing for specific counterparties), this allows unauthorized extraction of value from LP positions at prices or terms not intended for the general public. This constitutes broken core pool functionality and a direct admin-boundary break by an unprivileged path.

## Likelihood Explanation
The bypass requires only that the pool admin has allowlisted the router address, which is a necessary and expected admin action for any pool that wants to support router-mediated swaps for its allowlisted users. Any unprivileged EOA can then exploit this by simply calling the public router. No special privileges, flash loans, or complex setup are required. The condition is reachable in any production deployment that uses both `SwapAllowlistExtension` and `MetricOmmSimpleRouter`.

## Recommendation
Pass the originating EOA through the router to the extension. One approach: have the router encode the originating `msg.sender` into `extensionData`, and have `SwapAllowlistExtension.beforeSwap` decode and check that address when the direct `sender` is a known router. A cleaner approach is for the pool to expose a separate `swapOnBehalfOf` entry point that accepts an explicit `originator` argument verified via a router-signed context, or for the extension to check `recipient` (which the router sets to the user-supplied address) as a proxy for the intended swapper identity. At minimum, document that allowlisting the router grants unrestricted swap access to all users.
